## WebSockets проти HTTP

- **HTTP** працює за моделлю запит-відповідь: клієнт надсилає запит, сервер надсилає відповідь, після чого з’єднання закривається.
- **WebSockets** створюють постійне двонапрямлене з’єднання між клієнтом і сервером, яке залишається відкритим, доки одна зі сторін не надішле сигнал закриття.
- HTTP добре підходить для завантаження сторінок або отримання даних, але стає повільним і неефективним для застосунків у реальному часі.
- Кожне WebSocket-з’єднання починається з **HTTP handshake**: клієнт запитує оновлення, а сервер відповідає `101 Switching Protocols`.

### Потік з’єднання WebSocket

1. Клієнт підключається до сервера через HTTP і запитує оновлення.
2. Сервер приймає і відповідає `101 Switching Protocols`.
3. З’єднання залишається відкритим.
4. Обидві сторони можуть обмінюватися даними у будь-який час, доки з’єднання не закриється.

### Спілкування HTTP проти WebSocket

- **HTTP**: `Client → Request → Server → Response` (один цикл, потім закривається)
- **WebSocket**: `Client ↔ Server` (постійне, двонапрямлене)

### Корисна аналогія

- **HTTP** — це як обмін листами: ви пишете один, чекаєте на відповідь, і розмова закінчується.
- **WebSockets** — це як телефонна розмова: обидві сторони говорять у реальному часі стільки, скільки потрібно.

### Реальні випадки використання WebSockets

- Чат-застосунки
- Онлайн багатокористувацькі ігри
- Оновлення спортивних результатів у реальному часі
- Панелі біржових котирувань
- Спільні інструменти (наприклад, Google Docs)

## Архітектура Pub/Sub

- **Pub/Sub** (Publish/Subscribe) — це архітектура обміну повідомленнями, яка дозволяє частинам системи спілкуватися, не знаючи одне про одного безпосередньо.
- Є три основні частини: **видавці**, **теми** та **підписники**.
  - **Видавець** надсилає повідомлення до теми.
  - **Тема** (також називається каналом або групою) виступає посередником.
  - **Підписник** отримує повідомлення з тем, на які він підписаний.

### Потік повідомлень у Pub/Sub

1. Підписник підписується на тему.
2. Видавець надсилає повідомлення до цієї теми.
3. Система доставки повідомлень доставляє повідомлення кожному підписнику цієї теми.

### Пряме спілкування проти Pub/Sub

- **Без Pub/Sub**: `Application A → Application B` — тісно пов’язано; зміна в B може порушити A.
- **З Pub/Sub**: `Application A → Topic → Subscribers` — A лише публікує; йому не потрібно знати, хто отримує.

### Реальні випадки використання Pub/Sub

- Чат-застосунки з кількома кімнатами
- Онлайн багатокористувацькі ігри
- Системи сповіщень
- Панелі в реальному часі
- Подієво-орієнтовані застосунки

### WebSockets проти Pub/Sub

- **WebSockets** забезпечують канал зв’язку (дорогу).
- **Pub/Sub** забезпечує шаблон обміну повідомленнями, який визначає, хто отримує конкретні повідомлення (систему руху).
- Багато застосунків у реальному часі використовують обидва разом: WebSockets тримають з’єднання відкритим, а Pub/Sub направляє повідомлення потрібним підписникам.

## Робота з WebSockets у Node.js

- У Node.js немає вбудованого WebSocket-сервера. Бібліотека `ws` — це стандартний зовнішній пакет для цього.
- Встановіть за допомогою: `npm install ws`

### Створення WebSocket-сервера

```js
const WebSocket = require("ws");

const server = new WebSocket.Server({ port: 3000 });
```

### Події WebSocket

| Подія | Тригер |
|---|---|
| `connection` | Новий клієнт підключається до сервера |
| `message` | Підключений клієнт надсилає дані |
| `close` | З’єднання закривається |
| `error` | Виникає помилка у з’єднанні |

### Надсилання повідомлення клієнту

```js
socket.send("Hello from the server");
```

### Трансляція всім підключеним клієнтам

- `server.clients` містить усіх підключених клієнтів.
- Перевіряйте `readyState === WebSocket.OPEN` перед надсиланням, щоб уникнути помилок із відключеними клієнтами.

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **Трансляція** — це шаблон надсилання повідомлення одночасно кільком підключеним клієнтам — використовується в чатах, живих сповіщеннях, оновленнях спорту та подіях багатокористувацьких ігор.
