# --description--

Reactでは、レンダリングとはコンポーネントがユーザーインターフェース（UI）、通常はブラウザに表示されるプロセスのことです。

ReactはすべてのJavaScript、JSX、CSSのコードを受け取り、どのように見えるべきかを判断し、それをユーザーインターフェースに表示します。

レンダリングのプロセスは、トリガー、レンダー、コミットの3つのステージで構成されています。これらを詳しく見ていきましょう。

トリガーステージは、Reactが何かが変わったことを検知し、ユーザーインターフェースの更新が必要かもしれないと判断したときに発生します。この変化は多くの場合、stateやプロップスの更新によるものです。

例えば、夕食の時間だと気づくことが、キッチンに行って料理を始めるきっかけになるようなものです。

以下の`Counter`の例では、増加または減少ボタンをクリックすると、Reactが新しい`count`の値を表示するトリガーになります。

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h1>{count}</h1>
      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```

トリガーが発生すると、Reactはレンダーステージに入ります。ここでReactはコンポーネントを再評価し、何を表示すべきかを決定します。

これを行うために、Reactは「本物の」DOMの軽量コピーを使います。これが仮想DOMと呼ばれるものです。仮想DOMを使うことで、Reactはコンポーネントのどこを変更すべきかを素早くチェックできます。

このステージは、キッチンで材料を集めて夕食を調理している段階と考えてください。

`Counter`コンポーネントの場合、レンダーステージはReactが新しい`count`の値で関数を再実行し、更新された`<h1>{count}</h1>`の値に基づいて`count`部分がどう見えるべきかを再計算する段階です。ただし、画面上での変化は次のステージであるコミットまで見えません。

コミットステージは、Reactが仮想DOMで準備した変更を本物のDOMに適用する段階です。つまり、画面上で最終結果が見えるのはこのステージです。

これを実現するために、Reactは仮想DOMと実際のDOMを比較し、更新が必要な部分だけを特定して本物のDOMに変更を適用し、ユーザーインターフェースを更新します。

このステージは、調理した料理を盛り付けて見えるようにする段階と考えられます。Reactが実際のDOMに更新をコミットするときと同じです。

`Counter`コンポーネントの場合、コミットステージは新しい`count`の値が`h1`要素に適用され、ページ上で変化が見える段階です。

これら3つのプロセスは非常に高速です。なぜならReactは仮想DOMで変更を計算し、実際のDOMの操作を最小限に抑え、変更が必要な部分だけを更新するからです。
