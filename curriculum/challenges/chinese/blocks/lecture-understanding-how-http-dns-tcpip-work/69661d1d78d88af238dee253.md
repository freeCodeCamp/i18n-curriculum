# --description--

我们今天所知的数字世界基于设备能够相互通信，根据需要发送和接收数据的概念。

但是你有没有想过这个 `process` 在幕后是如何工作的？

当你浏览互联网时，你会不断通过网络请求和发送数据。这得益于基于标准规则和过程的详细进程，确保信息被适当的设备正确发送和接收。

这组规则和过程被称为**协议**。

协议基本上允许两个或多个设备通过确保它们都遵循某些标准（例如将要交换的数据的形式）来相互通信。

协议确保所有参与数据交换进程的设备都能够相互理解。形象地说，它们都会知道如何“讲同一种语言”。

TCP/IP 的核心是使互联网成为可能的协议。

**TCP/IP** 是一套通信协议，决定数据如何在网络中被打包、发送、路由和接收。它确保信息被可靠地传输。

具体来说，**IP**，即 **Internet Protocol**，是一种协议，确保数据从源头传输到网络中的正确目的地。

为什么这很重要？请记住，互联网是一个由互联设备组成的全局网络。为了让你的设备发送请求，首先它必须知道将数据发送到哪里。这就是 IP 地址的主要用途，也是互联网协议（IP）的基础。它们唯一标识网络上的设备，因此源设备知道将数据发送到哪里。

互联网协议本身是无连接的，这意味着每个数据单元（称为“packet”）都是独立发送到网络中的目的地的，发送前不会建立连接，也不会确保其正确或按顺序传递。如果它们确实到达目的地，源头根本不会收到通知。

这就是像 TCP 这样的协议变得非常重要的地方。

**TCP** 协议，称为 **Transmission Control Protocol**，将数据分解成小段。这些基本的数据单元被编号，并在到达目的地时使用特定的算法重新组合，这些算法检测数据是否完整且正确。这些算法还会检查误差和丢失的数据包，因此如果需要，可以请求源重新发送，确保信息被正确接收。

传输控制协议和互联网协议共同制定了确保数据在互联网中可靠传输所需的规则和标准。

除了 `TCP/IP`，现代互联网还有另一个非常重要的协议：`HTTP`。

超文本传输协议工作在应用层。

**HTTP** 代表 **超文本传输协议**。它定义了客户（如你的浏览器）和服务器之间通信的规则和标准。

当你尝试在浏览器中访问一个网站时，浏览器会向服务器发送一个超文本传输协议请求，请求该特定资源及其相关资产。服务器接收超文本传输协议请求，执行其逻辑，并向客户返回一个超文本传输协议响应，包含请求的内容或一些指示请求未成功原因的信息。

超文本传输协议请求和超文本传输协议响应具有非常相似的结构，包括：

* 请求行（起始行）
    
* 头部
    
* 体（可选的）

## **超文本传输协议 请求**

HTTP 请求的**请求行**分为三个主要部分：

* 一种描述被请求操作的 方法。例如，`GET` 用于检索 数据，`POST` 用于提交 数据。其他常见的方法包括 `PUT`、`PATCH`、`DELETE` 和 `HEAD`。
    
* 一个 `path` 或 `URI`，用于描述客户正在请求的特定资源。
    
* 用于通信进程的超文本传输协议版本。例如，`HTTP/1.1`。

这是一个使用 `GET` 方法在根路径 `/` 上并且超文本传输协议版本为 1.1 的示例：

`GET / HTTP/1.1`

**头部** 包含有关该 `request` 的密钥-值点对信息。

例如：`Content-Type: application/json`。

在头部下方，有一空行表示元数据（关于请求的附加信息）已完成。

最后是**体**。这是可选的。当客户必须向服务器发送信息时会发送，比如使用 `POST` 方法提交新建数据时。

这是一个超文本传输协议请求的示例：

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

## **`HTTP` 响应**

它们具有非常相似的结构：

* 状态行
    
* 头部
    
* 体
    

首先，**状态行**包含有关请求结果的信息。你通常会在这里查找：

* 用于通信进程的超文本传输协议版本。例如，`HTTP/1.1`。
    
* 表示请求结果的状态码。
    
* 原因短语，是状态码的简短人类可读描述。这有助于理解发生了什么。
    

常见的状态代码包括：

* `200 OK`
    
* `201 CREATED`
    
* `400 BAD REQUEST`
    
* `404 NOT FOUND`
    
* `403 FORBIDDEN`
    
* `500 INTERNAL SERVER ERROR`
    

有许多状态代码。它们按从左到右的第一个数字进行分类。

* `1xx` 用于简介响应。
    
* `2xx` 表示成功的响应。
    
* `3xx` 用于重定向消息。
    
* `4xx` 表示客户错误响应。
    
* `5xx` 表示服务器错误响应。
    

在 HTTP 响应的状态行下方，我们可以找到包含响应元数据的**头部**。例如，`Content-Type: text/html`。这也可能包括内容长度、日期、位置以及客户可能觉得有用的其他信息。

最后，我们查找超文本传输协议响应的**体**，其中包含客户请求的数据，或者如果请求未成功，则包含发生情况的说明。这是可选的，但大多数响应中都会包含。

这是一个超文本传输协议响应的示例：

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

既然你已经了解了超文本传输协议，你可能会想了解更多关于 HTTPS 的内容。

你可以将 HTTPS 视为 HTTP 的“安全”版本，其中浏览器和服务器之间发送的 `data` 为了安全性而被加密。大多数现代网络流量使用 HTTPS。

像 TCP/IP 和 HTTP 这样的协议使万维网和整个互联网成为我们大多数人每天依赖的必备工具。多亏了它们，设备才能在网络上正确通信并在全球范围内交换数据。
