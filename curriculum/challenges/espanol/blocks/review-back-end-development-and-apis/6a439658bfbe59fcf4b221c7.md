## Trabajando con Node.js y arquitectura orientada a eventos

- **Node.js**: Un entorno de ejecución de JavaScript multiplataforma y de código abierto que permite ejecutar código JavaScript fuera de un navegador. Se usa ampliamente para construir servidores web, APIs y sistemas back-end.
- **Limitaciones del lado del cliente**: JavaScript del lado del cliente tiene acceso restringido al sistema de archivos, soporte limitado para lógica compleja de aplicaciones y riesgos potenciales de seguridad (por ejemplo, no debes exponer credenciales de base de datos en código del cliente).
- **Objeto global en navegador vs Node.js**: En el navegador, el objeto global es `window`, que expone manipulación del DOM, cookies y eventos del navegador. En Node, el objeto global es `global`, que expone módulos integrados para archivos, redes e interacción con el sistema operativo.
- **Acceso al DOM**: El navegador proporciona acceso a la API del DOM, pero Node no. En cambio, Node puede acceder a casi todos los recursos del sistema, incluido el sistema de archivos, mientras que el navegador no.
- **Control de versiones**: Puedes elegir qué versión de Node se ejecuta en tu servidor. No tienes control sobre qué versión del navegador usan tus usuarios.
- **Arquitectura no bloqueante y orientada a eventos**: Node usa un solo hilo y un ciclo de eventos que maneja eficientemente muchas solicitudes simultáneas y operaciones de E/S sin bloquear, ideal para aplicaciones en tiempo real.
- **Callbacks**: Funciones que definen qué sucede una vez que una operación asíncrona termina. La fuerte dependencia en ellas es parte de por qué el código asíncrono en Node puede ser más difícil de leer y depurar.
- **Ventajas de Node en el back-end**: Un solo lenguaje (JavaScript) para front-end y back-end, arquitectura no bloqueante adecuada para alta concurrencia, gran ecosistema npm, gratuito y de código abierto.
- **Desventajas de Node en el back-end**: Monohilo, por lo que tareas intensivas en CPU (procesamiento de imágenes, criptografía, matemáticas complejas) pueden bloquear el ciclo de eventos, la dependencia en código asíncrono/callback puede reducir la legibilidad, los paquetes npm varían en calidad y mantenimiento.
- **NVM (Node Version Manager)**: Una herramienta para gestionar y cambiar entre múltiples versiones de Node.js en la misma máquina. En macOS/Linux se instala con:

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **Instalar la versión LTS de Node**: `nvm install --lts`
- **Cambiar versiones de Node**: `nvm use 20`
- **Listar todas las versiones instaladas de Node**: `nvm ls`
- **Establecer una versión predeterminada de Node (macOS/Linux)**: `nvm alias default 22.20.0`
- **nvm-windows**: Un proyecto separado para gestionar versiones de Node en Windows (no WSL). Descarga el instalador del repositorio oficial y ejecuta `nvm install lts` luego `nvm use <version>`.
- **Ejecutar un archivo JavaScript con Node**: `node app.js`

## Trabajando con módulos centrales de Node

### El módulo `fs`

- **Propósito**: Proporciona métodos para crear, leer, escribir, agregar y eliminar archivos y directorios.
- **Tres estilos de uso**: Basado en callbacks (asíncrono), basado en promesas (`fs/promises`) y síncrono (sufijo `Sync`). Los métodos síncronos bloquean el ciclo de eventos y solo son adecuados para scripts.
- **`writeFile()`**: Escribe contenido en un archivo, creándolo si no existe:

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**: Agrega contenido a un archivo existente sin sobrescribirlo.
- **`readFile()`**: Lee el contenido de un archivo. Pasa `"utf8"` como codificación para obtener una cadena, o omítelo para obtener un `Buffer`.
- **`unlink()`**: Elimina un archivo. Lanza `ENOENT` si el archivo no existe.

### El módulo `Buffer`

- **Propósito**: Maneja datos binarios (archivos, imágenes, flujos de red) directamente en memoria.
- **`Buffer.from()`**: Crea un búfer a partir de una cadena:

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**: Crea un búfer de `n` bytes lleno de ceros:

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **Truncamiento**: Escribir más datos de los que el búfer puede contener trunca silenciosamente el exceso.
- **`Buffer.byteLength(str)`**: Devuelve el número de bytes necesarios para almacenar una cadena.

### El módulo `crypto`

- **Propósito**: Proporciona herramientas criptográficas como hashing, HMAC, encriptación/desencriptación, firmas digitales y generación segura de números aleatorios.
- **`createHash(algorithm)`**: Crea un hash unidireccional (irreversible). Usa `.update(data).digest("hex")` para proporcionar los datos a hashear y devolver el hash como cadena hexadecimal:

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**: Similar a `createHash` pero requiere una clave secreta, adecuado para verificar integridad de datos y autenticación.
- **`createCipheriv()` / `createDecipheriv()`**: Encriptan y desencriptan datos. Ambos requieren un algoritmo, una clave y un vector de inicialización (IV):

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**: Genera bytes aleatorios criptográficamente seguros, más seguro que `Math.random()` para tokens y claves.
- **`randomInt(min, max)`**: Genera un entero aleatorio seguro en el rango dado. Útil para OTPs.
- **`sign()` / `verify()`**: Crea y verifica firmas digitales usando un par de claves privada/pública.

### El módulo `os`

- **Propósito**: Expone información sobre el sistema operativo donde se ejecuta Node.
- **`os.platform()`**: Devuelve el identificador del SO, por ejemplo `'darwin'`, `'win32'`, `'linux'`.
- **`os.arch()`**: Devuelve la arquitectura de la CPU, por ejemplo `'x64'`, `'arm64'`.
- **`os.type()`**: Devuelve el nombre oficial del SO, por ejemplo `'Darwin'`.
- **`os.release()`**: Devuelve la cadena de versión del kernel del SO.
- **`os.cpus()`**: Devuelve un arreglo de objetos describiendo cada núcleo lógico de CPU (modelo, velocidad, tiempos).
- **`os.uptime()`**: Devuelve el tiempo de actividad del sistema en segundos.
- **`os.totalmem()` / `os.freemem()`**: Devuelven la memoria total y disponible del sistema en bytes.
- **`os.userInfo()`**: Devuelve un objeto con el `uid`, `gid`, `username`, `homedir` y `shell` del usuario actual.
- **`os.networkInterfaces()`**: Devuelve objetos de interfaces de red con dirección, familia, MAC y detalles CIDR.

### El módulo `path`

- **Propósito**: Proporciona utilidades para trabajar con rutas de archivos y directorios de forma independiente de la plataforma.
- **`__filename`**: Variable global, la ruta absoluta del archivo actual.
- **`__dirname`**: Variable global, la ruta absoluta del directorio que contiene el archivo actual.
- **`path.basename(p)`**: Devuelve el último componente de una ruta (el nombre del archivo).
- **`path.dirname(p)`**: Devuelve la parte de directorio de una ruta.
- **`path.extname(p)`**: Devuelve la extensión del archivo, por ejemplo `'.js'`.
- **`path.join(...segments)`**: Une segmentos de ruta en una ruta normalizada y corrige barras erróneas.
- **`path.resolve(...segments)`**: Resuelve segmentos en una ruta absoluta comenzando desde el directorio de trabajo actual.
- **`path.parse(p)`**: Devuelve un objeto con propiedades `root`, `dir`, `base`, `ext` y `name`.
- **`path.format(obj)`**: Construye una cadena de ruta a partir de un objeto con propiedades `dir`, `name` y `ext`.

### El módulo `process`

- **Propósito**: Expone información y control sobre el proceso actual de Node.js. Es un objeto global, por lo que no requiere importación.
- **`process.env`**: Objeto que contiene todas las variables de entorno. Accede a valores específicos como `process.env.NODE_ENV`.
- **`process.argv`**: Arreglo de argumentos de línea de comandos. El índice 0 es el ejecutable de Node, el índice 1 es la ruta del script, los índices 2+ son argumentos del usuario.
- **`process.cwd()`**: Devuelve el directorio de trabajo actual.
- **Eventos del proceso**:

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**: Dispara una advertencia personalizada que puede ser capturada por un listener de `warning`.

### El módulo `stream`

- **Propósito**: Procesa grandes cantidades de datos en fragmentos en lugar de cargar todo en memoria de una vez.
- **Cuatro tipos de streams**: Readable (lee datos en fragmentos), Writable (escribe datos en fragmentos), Duplex (ambos), Transform (un Duplex que modifica datos mientras fluyen).
- **Encadenar streams**:

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **Evento `finish`**: Se dispara en un stream writable cuando todos los datos han sido vaciados.
- **Evento `error`**: Se dispara cuando un stream encuentra un problema.

## Introducción a npm

- **npm**: Consta de tres partes: el sitio web `npmjs.com`, el registro público (una base de datos de paquetes) y la herramienta CLI instalada con Node.js.
- **Propósito**: Permite instalar, gestionar y publicar paquetes reutilizables de JavaScript. npm es el gestor de paquetes más usado.
- **Alternativas**: Yarn, PNPM, Bun.

### El archivo `package.json`

- **Propósito**: El manifiesto de configuración de npm. Registra el nombre del proyecto, versión, descripción, punto de entrada (`main`), autor, licencia, scripts y dependencias.
- **Crear `package.json` interactivamente**: `npm init`
- **Crear con valores predeterminados**: `npm init -y`
- **`dependencies`**: Paquetes necesarios para que la app funcione en producción (por ejemplo, `express`, `react`).
- **`devDependencies`**: Paquetes solo necesarios durante desarrollo y pruebas (por ejemplo, `jest`, `nodemon`, `eslint`).
- **`license`**: Términos legales para el uso. Valores comunes: `MIT`, `ISC`, `GPL`.

### Versionado semántico (SemVer)

- **Formato**: `MAJOR.MINOR.PATCH`, por ejemplo, `4.17.21`.
- **MAJOR**: Se incrementa por cambios incompatibles.
- **MINOR**: Se incrementa por nuevas funcionalidades compatibles hacia atrás.
- **PATCH**: Se incrementa por correcciones de bugs compatibles hacia atrás.
- **Caret `^`**: Permite actualizaciones a versiones minor y patch pero no major. Ejemplo: `^4.17.21` acepta `4.18.0` pero no `5.0.0`.
- **Tilde `~`**: Permite actualizaciones solo a la versión patch. Ejemplo: `~1.2.3` acepta `1.2.4` pero no `1.3.0`.
- **Asterisco `*`**: Coincide con cualquier versión, adecuado solo para pruebas, nunca producción.
- **Versión exacta**: Sin prefijo, por ejemplo `1.2.3` fija exactamente esa versión.

### Instalación y eliminación de dependencias

- **Instalar un paquete**: `npm install express` (abreviado: `npm i express`)
- **Instalar como devDependency**: `npm install nodemon -D`
- **Desinstalar un paquete**: `npm uninstall chalk`
- **`node_modules/`**: Carpeta creada tras la instalación que contiene el código real del paquete. No se sube al control de versiones.
- **`package-lock.json`**: Archivo autogenerado que bloquea las versiones exactas de todos los paquetes instalados (incluyendo dependencias hijas). Siempre se debe subir y no editar manualmente.
- **Instalar una versión específica**: `npm install express@4.21.2`
- **Ver versiones disponibles**: `npm view express versions`
- **Ver paquetes desactualizados**: `npm outdated`
- **Actualizar un solo paquete** (dentro del rango permitido): `npm update express`
- **Forzar instalación de la última versión**: `npm install express@latest`

### Publicar en el registro npm

- **Iniciar sesión**: `npm login`
- **Paquetes con scope**: Usa `@username/package-name` para evitar colisiones de nombres.
- **`.npmignore`**: Lista archivos a excluir del paquete publicado (similar a `.gitignore`).
- **Publicar un paquete sin scope**: `npm publish`
- **Publicar un paquete con scope públicamente**: `npm publish --access public`

## Trabajando con scripts npm

- **Scripts npm**: Comandos personalizados definidos en el objeto `"scripts"` de `package.json`. Automatizan tareas repetitivas como iniciar un servidor, correr pruebas, linting y compilación.
- **Definir un script**:

```json
"scripts": {
  "start": "node app.js"
}
```

- **Ejecutar un script**: `npm run start`
- **Atajo para scripts integrados** (`start`, `test`, `stop`, `restart`): `npm start`
- **Comandos secuenciales con `&&`** (el siguiente comando se ejecuta solo si el anterior tiene éxito):

```json
"start": "npm run build && node server.js"
```

- **Comandos secuenciales con `;`** (el siguiente comando siempre se ejecuta):

```json
"start": "npm run build; node server.js"
```

- **Comandos concurrentes con `&`**: `"dev": "npm run start:server & npm run start:client"`
- **Pasar argumentos a un script** (usa el separador `--`): `npm start -- 8000`; los argumentos son accesibles vía `process.argv[2]`.

### Módulos CommonJS

- **CommonJS**: El sistema original de módulos de Node.js. Carga módulos de forma síncrona.
- **Importar** con `require()`:

```js
const { multiply } = require('./math');
```

- **Exportar** con `module.exports`:

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- CommonJS es el predeterminado cuando `package.json` tiene `"type": "commonjs"` (o no tiene campo `"type"`), o el archivo tiene extensión `.cjs`.

### Módulos ES (ESM)

- **Módulos ES (ESM)**: El formato moderno y estandarizado de módulos. Se cargan de forma asíncrona.
- **Exportación nombrada**: `export function multiply(a, b) { return a * b; }`
- **Exportación por defecto**: `export default function multiply(a, b) { return a * b; }`
- **Importar**: `import { multiply } from './math.mjs';`
- ESM se usa cuando el archivo tiene extensión `.mjs` o `package.json` contiene `"type": "module"`.

## Entendiendo cómo funcionan HTTP, DNS y TCP/IP

### Servidores y el modelo cliente-servidor

- **Modelo cliente-servidor**: Arquitectura de sistema distribuido donde los componentes son clientes (procesos que envían solicitudes, por ejemplo, un navegador web) o servidores.
- **Servidor**: Computadora o programa que recibe solicitudes de clientes y devuelve datos o servicios a través de una red.
- **Tipos de servidores**: Servidores web (HTML/CSS/recursos), servidores de aplicaciones (lógica de negocio), servidores de base de datos, servidores de correo, servidores de archivos, servidores proxy.

### DNS

- **DNS (Domain Name System)**, a menudo llamado "la guía telefónica de internet": Traduce nombres de dominio legibles por humanos (por ejemplo, `freecodecamp.org`) en direcciones IP.
- **Jerarquía de nombres de dominio** (de derecha a izquierda): Dominio de nivel superior (TLD, por ejemplo `.org`) → Dominio de segundo nivel (por ejemplo `freecodecamp`) → Subdominio (por ejemplo `www`).
- **Dirección IP**: Identificador numérico único para un dispositivo en una red. Ejemplo IPv4: `127.0.0.1`. Ejemplo IPv6: `0:0:0:0:0:0:0:1`.
- **Proceso de resolución DNS**: Navegador → Resolver recursivo → Servidores raíz → Servidor TLD → Servidor de nombres autoritativo → devuelve dirección IP → Resolver recursivo cachea y entrega al cliente.

### TCP/IP y HTTP

- **IP (Internet Protocol)**: Enruta paquetes de datos desde la fuente al destino usando direcciones IP. Sin conexión, cada paquete se envía independientemente.
- **TCP (Transmission Control Protocol)**: Divide datos en segmentos numerados, los reensambla en el destino, verifica errores y solicita retransmisión si es necesario. Garantiza entrega confiable y ordenada.
- **HTTP (HyperText Transfer Protocol)**: Protocolo de capa de aplicación que define cómo clientes y servidores intercambian contenido web.
- **HTTPS**: Versión segura de HTTP donde los datos se encriptan usando SSL/TLS.

### Estructura de una solicitud HTTP

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **Línea de solicitud**: Método HTTP + ruta/URI + versión HTTP.
- **Encabezados**: Pares clave-valor de metadatos.
- **Cuerpo** (opcional): Enviado con `POST`, `PUT`, `PATCH`, `DELETE` para transportar datos.

### Estructura de una respuesta HTTP

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **Línea de estado**: Versión HTTP + código de estado + frase de razón.
- **Encabezados**: Metadatos sobre la respuesta (tipo de contenido, longitud, directivas de caché).
- **Cuerpo** (opcional): Los datos solicitados.

### Categorías de códigos de estado HTTP

- **1xx Informativos**: Solicitud recibida, procesamiento continúa. Ejemplo: `101 Switching Protocols`.
- **2xx Éxito**: `200 OK`, `201 Created`, `204 No Content`.
- **3xx Redirección**: `301 Moved Permanently`, `302 Found`, `304 Not Modified`.
- **4xx Errores del cliente**: `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.
- **5xx Errores del servidor**: `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable`.

## Entendiendo el modelo solicitud-respuesta HTTP

- **Métodos HTTP**: `GET` (recuperar), `POST` (crear), `PUT` (actualización completa), `PATCH` (actualización parcial), `DELETE` (eliminar).
- **Ciclo solicitud-respuesta**: Cliente envía solicitud → Servidor valida autenticación, verifica método y URL, consulta base de datos → Servidor construye respuesta (código de estado + cuerpo opcional) → Cliente procesa respuesta.

### Recursos devueltos en respuestas HTTP

Cada respuesta incluye un encabezado `Content-Type` que indica al navegador cómo manejar el recurso:

- **HTML**: `text/html`
- **CSS**: `text/css`
- **JavaScript**: `application/javascript`
- **JSON**: `application/json`
- **PNG/JPEG/WebP**: `image/png`, `image/jpeg`, `image/webp`
- **MP4/MP3**: `video/mp4`, `audio/mpeg`
- **PDF**: `application/pdf`
- **WOFF2**: `font/woff2`

### Envío de formularios HTML

- **Atributo `action`**: Especifica la URL a la que se envían los datos del formulario.
- **Atributo `method`**: Especifica el método HTTP (`GET` o `POST`). HTML no soporta nativamente `PUT`, `PATCH` o `DELETE`, usa Fetch API o el paquete `method-override` en Express.
- **Validación del lado cliente**: Usa atributos `required` y `pattern` para retroalimentación inmediata. Siempre valida también en el servidor.
- **Formulario `GET`**: Datos agregados a la URL como parámetros de consulta (por ejemplo, formularios de búsqueda).
- **Formulario `POST`**: Datos enviados en el cuerpo de la solicitud (por ejemplo, formularios de inicio de sesión y registro).

## Entendiendo el modelo de estándares web

### Organismos de estándares

- **WHATWG**: Mantiene HTML y el DOM como un "estándar vivo" continuamente actualizado. La toma de decisiones la lideran los principales equipos de motores de navegador (Chromium, Gecko, WebKit).
- **W3C (World Wide Web Consortium)**: Organismo principal de estándares. Supervisa CSS, SVG, pautas de accesibilidad (WCAG) y numerosas APIs web.
- **ECMA International / TC39**: Ecma publica la especificación oficial de ECMAScript. TC39 desarrolla el lenguaje mediante un proceso de propuesta de cinco etapas (Etapa 0 idea inicial → Etapa 4 especificación completa).
- **Khronos Group**: Se enfoca en gráficos, computación y medios. Produce WebGL y WebGL 2 para renderizado 3D acelerado por GPU en navegadores.

### Creación de estándares web

1. **Identificación de una necesidad**: Un desarrollador u organización encuentra una dificultad y la documenta.
2. **Propuesta y discusión inicial**: Se comparte un issue informal en GitHub o un documento explicativo para medir interés. En TC39 esto es Etapa 0/Etapa 1.
3. **Redacción de la especificación**: Documento formal preciso que cubre todos los casos límite, errores y consideraciones de seguridad.
4. **Revisión, objeción e iteración**: El borrador es revisado públicamente por proveedores de navegadores; características que un navegador importante se niega a implementar rara vez se convierten en estándar.
5. **Implementación y pruebas**: Los proveedores implementan detrás de una bandera de funcionalidad; un conjunto de pruebas compartido asegura que todos los navegadores se comporten igual.
6. **Finalización**: W3C lo llama Recomendación, TC39 lo llama Etapa 4.

### Ciclo de vida de una funcionalidad de estándares web

1. **Propuesta**: Se identifica y documenta el problema.
2. **Incubación**: Discusión abierta, muchas propuestas mueren aquí.
3. **Especificación**: Se escribe la especificación formal con reglas exactas.
4. **Pruebas de interoperabilidad**: El conjunto de pruebas W3C Web Platform Tests verifica consistencia entre navegadores.
5. **Finalización**: La funcionalidad se lanza sin banderas como parte de la plataforma web base.
6. **Deprecación**: Funcionalidades obsoletas o inseguras marcadas para eliminación, pero los navegadores las mantienen para compatibilidad hacia atrás.

### Principios clave de los estándares web

- **Apertura**: Los estándares se desarrollan abiertamente y se implementan sin regalías.
- **Accesibilidad**: Integrada en HTML, CSS y comportamiento del navegador; WCAG provee pautas formales.
- **Interoperabilidad**: Una página construida según el estándar debe comportarse idénticamente en todos los navegadores.
- **Compatibilidad hacia atrás**: Las páginas antiguas deben seguir funcionando en navegadores modernos.
- **Privacidad y seguridad**: Nuevas APIs que podrían exponer datos de usuarios enfrentan un escrutinio significativo.
- **Capas**: HTML para estructura, CSS para presentación, JavaScript para comportamiento. Cada capa puede adoptarse independientemente.
- **Mejora progresiva**: Las funcionalidades deben funcionar para todos primero, luego mejorar para usuarios con navegadores más capaces.

## Entendiendo REST APIs y servicios web

### Servicios web

- **API (Interfaz de programación de aplicaciones)**: Mecanismo general para que software se comunique con otro software.
- **Servicio web**: Tipo específico de API que usa protocolos web (HTTP/HTTPS) para intercambiar datos en red. Todo servicio web es una API, pero no toda API es un servicio web.
- **Características**: Se ejecuta en un servidor, permite comunicación máquina a máquina, independiente de lenguaje y plataforma, usa protocolos web estándar.
- **REST**: Transferencia de estado representacional, el estilo de servicio web más usado. Simple, flexible, funciona naturalmente con HTTP.
- **SOAP**: Protocolo simple de acceso a objetos, usa solo XML, común en sistemas empresariales que requieren estándares estrictos.
- **XML-RPC**: Usa XML para codificar solicitudes y respuestas ligeras.

### Arquitectura REST

- **Sin estado**: Cada solicitud se maneja independientemente. El servidor no guarda estado de sesión entre solicitudes. Cada solicitud debe incluir toda la información necesaria (por ejemplo, un token de autenticación en cada llamada).
- **Recursos**: Todo es un recurso identificado por una URL, por ejemplo `/products` para una colección, `/products/1` para un elemento específico.
- **Métodos HTTP en REST**: `GET` (recuperar), `POST` (crear), `PUT`/`PATCH` (actualizar), `DELETE` (eliminar).
- **Formatos de respuesta**: JSON es el más común pero también se soportan XML y texto plano. Los clientes especifican el formato vía el encabezado `Accept`:

```bash
Accept: application/json
```

### Microservicios

- **Monolito**: Una base de código grande donde UI, base de datos y API residen juntos. Difícil de mantener y escalar conforme crece la complejidad.
- **Microservicios**: Enfoque arquitectónico donde una aplicación grande se divide en servicios pequeños e independientes, cada uno responsable de una capacidad de negocio. Los servicios se comunican vía APIs o colas de mensajes.
- **Ventajas**: Desarrollo y despliegue independientes más rápidos, mejor aislamiento de fallos, diferentes servicios pueden usar distintos lenguajes o frameworks.
- **Desventajas**: Depuración más compleja (los problemas abarcan servicios), mayor sobrecarga de infraestructura, aumento de comunicación en red.

## Trabajando con Express

- **Express.js**: Un framework web minimalista y flexible construido sobre Node.js. Simplifica la construcción de servidores web y APIs con mucho menos código repetitivo que el módulo nativo `http`.
- **Un servidor básico con Express**:

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**: Función fábrica que crea una instancia de aplicación Express (`app`).
- **`app.listen(port, callback)`**: Inicia el servidor y comienza a escuchar solicitudes HTTP en el puerto especificado.
- **`req`**: El objeto solicitud, contiene URL, encabezados, parámetros, cuerpo y datos de consulta.
- **`res`**: El objeto respuesta, usado para enviar datos al cliente.
- **Middleware**: Código que se ejecuta entre recibir una solicitud y enviar una respuesta.

## Entendiendo el enrutamiento en Express.js

- **Enrutamiento**: Define cómo responde una aplicación cuando un cliente visita una URL específica con un método HTTP específico.
- **`app.get/post/put/delete(path, handler)`**: Registra manejadores para los métodos HTTP respectivos.
- **Ruta estática**: Un patrón de URL fijo, por ejemplo, `app.get("/home", handler)`.
- **Ruta dinámica con parámetros**: Usa el marcador `:paramName`. Se captura vía `req.params`:

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**: Parámetros de ruta (segmentos `:id`).
- **`req.query`**: Parámetros de cadena de consulta (`?key=value`).
- **`req.body`**: Cuerpo de la solicitud (requiere middleware para parsear).
- **`res.send(data)`**: Envía una respuesta (texto, HTML, JSON o binario).
- **`res.json(obj)`**: Envía una respuesta JSON, establece automáticamente `Content-Type: application/json`.
- **`res.status(code).send(msg)`**: Establece el código de estado y envía una respuesta.
- **`res.redirect(url)`**: Redirecciona al cliente a otra URL.
- **`res.render(view, data)`**: Renderiza una vista de motor de plantillas y envía HTML.

### Manejadores de ruta encadenables con `app.route()`

Agrupa múltiples métodos HTTP para la misma ruta para reducir repetición:

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

Crea una mini-aplicación modular para agrupar rutas relacionadas:

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

### Servir archivos estáticos

- **`express.static(directory)`**: Middleware integrado que sirve archivos desde un directorio especificado:

```js
app.use(express.static(path.join(__dirname, "public")));
```

- Servir desde múltiples directorios con una ruta de montaje: `app.use("/images", express.static(path.join(__dirname, "images")));`

## Entendiendo manejo de errores y chequeos de salud

### Manejo de errores en Express

- **Middleware de manejo de errores**: Identificado por cuatro parámetros `(err, req, res, next)`. Siempre colocado al final de la pila de middleware:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **Errores asíncronos en Express 4**: Deben atraparse manualmente con `try/catch` y reenviarse con `next(err)`:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **Errores asíncronos en Express 5**: Se reenvían automáticamente al middleware de manejo de errores, no se necesita `next(err)` manual.
- **Manejador 404**: Colócalo después de todas las rutas pero antes del manejador de errores:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Depuración y registro

- **Depuración integrada**: Habilita configurando la variable de entorno `DEBUG`: `DEBUG=express:* node index.js`
- **`morgan`**: Middleware popular de registro de solicitudes HTTP. Registra método, URL, código de estado y tiempo de respuesta:

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### Chequeos de salud y apagados controlados

- **Ruta de verificación de estado**: Devuelve `200 OK` para indicar que la app está funcionando. Usada por balanceadores de carga y orquestadores de contenedores:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**: Señal enviada cuando un proceso debe terminar. Manejala para permitir que las solicitudes en curso finalicen:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**: Señal enviada al presionar Ctrl+C durante el desarrollo local. Manejala igual que `SIGTERM`.
- **Marcar como no saludable antes de apagar**: Devuelve un `503` desde el endpoint de verificación de estado durante el apagado para evitar que el balanceador de carga dirija nuevo tráfico:

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## Middleware en Express

- **Middleware**: Una función que se ejecuta durante el ciclo de vida de una solicitud. Tiene acceso a `req`, `res` y `next`. Llama a `next()` para pasar el control al siguiente middleware, omítelo para terminar el ciclo.
- **Middleware a nivel de aplicación**: Vinculado a toda la app mediante `app.use()`. Se ejecuta en cada solicitud entrante.
- **Middleware a nivel de enrutador**: Vinculado a una instancia específica de `express.Router()` mediante `router.use()`. Se aplica solo a las rutas manejadas por ese enrutador.
- **Middleware para manejo de errores**: Identificado por cuatro parámetros `(err, req, res, next)`. Se coloca al final de la pila.
- **`express.json()`**: Interpreta solicitudes entrantes con cargas JSON. Hace los datos accesibles en `req.body`.
- **`express.urlencoded({ extended: true })`**: Interpreta datos de formularios codificados en URL.
- **`express.static('public')`**: Sirve archivos estáticos desde un directorio.
- **`cors`**: Habilita el intercambio de recursos entre orígenes (Cross-Origin Resource Sharing). Instálalo con `npm install cors`, úsalo con `app.use(cors())`.
- **`morgan`**: Registra solicitudes HTTP. Usa `app.use(morgan('tiny'))`.
- El middleware se aplica en el orden en que se registra con `app.use()`.

## Entendiendo WebSockets

- **WebSockets**: Un protocolo que crea una conexión persistente y full-duplex (bidireccional) entre un cliente y un servidor, permitiendo comunicación en tiempo real.
- **Problema con HTTP para tiempo real**: HTTP requiere que el cliente consulte repetidamente al servidor por nuevos datos, por lo que es lento e ineficiente para apps de chat, marcadores en vivo y juegos multijugador.
- **Handshake de WebSocket**: Cada conexión WebSocket comienza con una solicitud HTTP. El cliente solicita una actualización y el servidor responde con `101 Switching Protocols`. Después la conexión permanece abierta hasta que alguna de las partes envía una señal de cierre.
- **Casos de uso**: Aplicaciones de chat, juegos multijugador en línea, paneles deportivos en vivo, feeds del mercado bursátil, herramientas colaborativas.

### La biblioteca `ws`

- Node.js no tiene un servidor WebSocket integrado; la biblioteca `ws` es la forma común de implementarlo.

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**: Envía un mensaje a un cliente específico.
- **Transmisión a todos los clientes conectados**:

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**: Propiedad en cada conexión. `WebSocket.OPEN` significa que la conexión está activa.

### Arquitectura Pub/Sub

- **Pub/Sub (Publicar/Suscribir)**: Un patrón de mensajería donde los **publicadores** envían mensajes a **temas**, y los **suscriptores** reciben mensajes de los temas a los que se han suscrito. Publicadores y suscriptores no se comunican directamente.
- **Ventajas sobre comunicación directa**: Desacopla componentes, un publicador no necesita saber quiénes son sus suscriptores.
- **Pub/Sub vs. WebSockets**: WebSockets son el canal de comunicación (la carretera). Pub/Sub es el patrón de mensajería que determina quién recibe qué mensajes (el sistema de tráfico).

## Introducción a la autenticación y autorización

### Autenticación vs. autorización

- **Autenticación**: Verifica la identidad, responde a la pregunta "¿Quién eres?", debe ocurrir antes de la autorización.
- **Autorización**: Determina permisos, responde a la pregunta "¿Qué puedes hacer?".
- **Tres factores de autenticación**:
  - **Algo que sabes**: Contraseña, PIN, respuesta a pregunta de seguridad.
  - **Algo que tienes**: Smartphone que recibe un código SMS, token de app autenticadora.
  - **Algo que eres**: Huella digital, reconocimiento facial (biometría).
- **Roles comunes en autorización**: Usuario estándar (autoservicio limitado), editor (gestión de contenido), administrador (acceso completo al sistema).
- En el código back-end, la autorización generalmente se aplica mediante middleware que verifica roles o permisos antes de conceder acceso.

### Falsificación de solicitudes entre sitios (CSRF)

- **CSRF**: Un ataque donde un sitio web malicioso engaña al navegador de un usuario autenticado para hacer una solicitud no deseada a un servidor back-end, explotando la adjunción automática de cookies del navegador.
- **Flujo del ataque**: El usuario inicia sesión en `trusted-bank.com` → visita `evil-hacker-site.com` → el sitio malicioso envía un formulario oculto al banco → el servidor del banco ve la cookie de sesión válida y ejecuta la solicitud.
- **`SameSite=Strict`**: El navegador nunca envía la cookie si la solicitud proviene de un sitio de terceros.
- **`SameSite=Lax`**: El navegador permite cookies solo cuando el usuario navega de forma segura al sitio objetivo (haciendo clic en un enlace), pero las bloquea en envíos de formularios entre sitios.
- **Tokens CSRF**: El servidor genera un token aleatorio impredecible incrustado en formularios. El cliente debe incluir el token en cada solicitud que cambie estado. Un sitio malicioso no puede leer el token desde el frontend, por lo que sus solicitudes falsificadas fallarán la validación.

### Tokens web JSON (JWT)

- **JWT**: Un token firmado digitalmente emitido por el servidor tras un inicio de sesión exitoso. El cliente lo almacena y lo envía con cada solicitud posterior, permitiendo autenticación sin estado (no se necesita almacenamiento de sesión en el servidor).
- **Estructura del JWT**: Tres partes codificadas en Base64URL separadas por puntos:
  - **Encabezado**: Tipo de token (`JWT`) y algoritmo de firma (por ejemplo, `HS256`).
  - **Carga útil**: Reclamos sobre el usuario (id, nombre de usuario, rol). No está encriptada, así que nunca almacenes contraseñas o datos sensibles aquí.
  - **Firma**: Hash del encabezado + carga útil + clave secreta del servidor. Previene manipulaciones.

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

- Si un usuario manipula la carga útil, la firma no coincidirá y el servidor rechazará la solicitud con `401 Unauthorized`.

### Passport.js

- **Passport.js**: Middleware de autenticación para Node.js. Estandariza el flujo de autenticación y maneja detalles de bajo nivel para que los desarrolladores se enfoquen en la lógica de negocio.
- **Estrategias**: Plugins modulares que implementan métodos específicos de autenticación. Más de 500 estrategias disponibles: `passport-local` (usuario/contraseña), `passport-jwt` (JWTs), `passport-google-oauth20` (inicio con Google), `passport-facebook`, `passport-github2`.
- **Integración con Express**:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **Uso como middleware de ruta**:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- Tras una autenticación exitosa, Passport almacena un dato del usuario en la cookie de sesión y adjunta el objeto del usuario autenticado a `req.user`.

### Helmet.js

- **Helmet.js**: Una colección de middleware de seguridad que protege apps Express configurando, modificando o eliminando automáticamente encabezados HTTP de respuesta.

```js
const helmet = require('helmet');
app.use(helmet());
```

- **Elimina `X-Powered-By`**: Evita que atacantes conozcan la tecnología del servidor.
- **Establece `Content-Security-Policy`**: Restringe de dónde se pueden cargar scripts, imágenes y estilos, mitigando ataques XSS.
- **Establece `X-Frame-Options: SAMEORIGIN`**: Previene clickjacking bloqueando que tu sitio se incruste en iframes externos.
- **Establece `Strict-Transport-Security`**: Obliga a los navegadores a usar HTTPS, protegiendo datos de intercepciones.

## Entendiendo seguridad y privacidad en aplicaciones web

### Seguridad vs. privacidad

- **Seguridad**: Los mecanismos técnicos que protegen sistemas y datos contra accesos no autorizados. La seguridad es la cerradura en la puerta.
- **Privacidad**: La gobernanza sobre cómo se recopilan, usan y comparten datos personales. La privacidad es la cortina en la ventana.
- Puedes tener seguridad sin privacidad (una base de datos bloqueada puede vender tus datos), pero no puedes tener privacidad sin seguridad (datos sin protección no pueden mantenerse privados).

### Política de mismo origen y CORS

- **Origen**: Definido por tres componentes: el protocolo (por ejemplo, `https`), el dominio (por ejemplo, `example.com`) y el puerto (por ejemplo, `443`). Los tres deben coincidir para que dos URLs compartan el mismo origen.
- **Política de mismo origen**: Una regla de seguridad del navegador que bloquea que scripts de un origen lean datos de otro origen diferente, evitando que sitios maliciosos roben datos de otras pestañas abiertas.
- **CORS (Cross-Origin Resource Sharing)**: Un mecanismo que permite a los servidores relajar la política de mismo origen para orígenes confiables mediante el establecimiento de encabezados HTTP.
- **En Express (desarrollo)**: `app.use(cors())` permite todos los orígenes.
- **En Express (producción)**: Pasa un objeto `corsOptions` con una lista blanca en `origin`:

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**: HTTP con cifrado proporcionado por TLS (Transport Layer Security). TLS reemplazó al protocolo SSL más antiguo.
- **Tres garantías de HTTPS**:
  - **Cifrado**: Los datos enviados entre cliente y servidor son ilegibles para terceros.
  - **Autenticación**: Un certificado digital de una autoridad certificadora confiable confirma que el servidor es quien dice ser.
  - **Integridad de datos**: Los mensajes no pueden alterarse en tránsito sin ser detectados.
- HTTPS es obligatorio para APIs modernas del navegador como geolocalización, notificaciones push y acceso a cámara.

### Por qué ocurren problemas de seguridad

- **Presión por entrega rápida**: Equipos que se apresuran para cumplir plazos omiten revisiones de seguridad y lanzan vulnerabilidades.
- **Integración compleja de sistemas**: El riesgo en la cadena de suministro surge cuando bibliotecas de terceros o servicios en la nube introducen vulnerabilidades que el equipo de desarrollo no escribió.
- **Mala gestión de configuración**: Dejar configuraciones predeterminadas o de fábrica (contraseñas admin por defecto, puertos abiertos) expone sistemas innecesariamente.
- **Error humano**: Ataques de phishing explotan al personal que hace clic en enlaces o adjuntos maliciosos y entrega credenciales.
- **Fusiones organizacionales**: Combinar diferentes bases de código e infraestructura expone políticas de seguridad desalineadas y vulnerabilidades heredadas.

### Cookies: almacenamiento, banderas de seguridad y privacidad

- **Cómo se almacenan las cookies**: El servidor envía un encabezado `Set-Cookie`. El navegador guarda el archivo en disco y lo devuelve automáticamente en cada solicitud posterior a ese origen.
- **Bandera `HttpOnly`**: Bloquea que JavaScript lea el valor de la cookie, previniendo robo de sesión por XSS.
- **Bandera `Secure`**: Restringe la cookie a conexiones HTTPS, protegiéndola de intercepciones en redes públicas.
- **`SameSite=Strict`**: Evita que el navegador envíe la cookie en cualquier solicitud entre sitios, bloqueando completamente ataques CSRF.
- **`SameSite=Lax`**: Permite cookies en navegaciones de nivel superior (por ejemplo, clic en enlace) pero las bloquea en envíos de formularios y llamadas API entre sitios.
- **Cookies de primer partido**: Limitadas al sitio que visitas.
- **Cookies de terceros**: Establecidas por redes publicitarias en miles de sitios para rastrear comportamiento y crear perfiles. Navegadores modernos las bloquean por defecto.
- **Cumplimiento GDPR**: Las leyes de privacidad requieren banners de consentimiento de cookies que dan a los usuarios el derecho legal a rechazar cookies de seguimiento no esenciales.

### Amenazas comunes de seguridad y mitigaciones

- **Phishing**: Correos fraudulentos que se hacen pasar por empresas confiables para robar credenciales. Las mitigaciones incluyen capacitación en seguridad y protocolos de autenticación de correo para bloquear mensajes maliciosos antes de que lleguen a la bandeja.
- **Malware** (virus, spyware, ransomware): El ransomware encripta archivos y exige pago para desbloquearlos. Las mitigaciones incluyen actualizaciones automáticas y software antimalware.
- **Credential stuffing**: Atacantes prueban millones de combinaciones robadas de usuario y contraseña en diferentes sitios. La mitigación principal es la autenticación multifactor (MFA), que requiere un segundo paso de verificación como un OTP o código de app autenticadora.
- **DDoS (Denegación de servicio distribuida)**: Atacantes inundan un servidor con tráfico falso para colapsarlo y bloquear usuarios legítimos. Las mitigaciones incluyen servicios en la nube para filtrar tráfico como Cloudflare, Akamai y AWS Shield.
- **Amenazas internas**: Empleados maliciosos o descuidados que roban o filtran datos. Las mitigaciones incluyen aplicar el **principio de menor privilegio** (los trabajadores solo acceden a lo que su trabajo requiere) y monitorear logs del sistema para actividad inusual.

### Política de seguridad de contenido y Permissions-Policy

- **CSP (Content Security Policy)**: Un encabezado HTTP de respuesta que lista dominios aprobados para scripts, estilos e imágenes. Cuando un navegador recibe un encabezado CSP, bloquea cualquier recurso cargado desde un dominio no listado. La directiva `script-src` especifica qué dominios pueden ejecutar JavaScript, deteniendo ataques de inyección XSS.
- **Permissions-Policy**: Un encabezado HTTP que habilita o deshabilita funciones del navegador y hardware del dispositivo para una página y cualquier iframe incrustado. Reemplazó al antiguo encabezado `Feature-Policy`.
  - `camera=(self)` restringe el acceso a la cámara web solo al origen de la página.
  - `geolocation=()` deshabilita completamente la geolocalización para todos los usuarios.
- Combinar ambos encabezados ofrece protección en capas: CSP previene inyección de código y Permissions-Policy previene acceso no autorizado a hardware.

### Conceptos fundamentales de privacidad del usuario

- **PII (Información de identificación personal)**: Cualquier dato que pueda identificar a una persona directa o indirectamente.
  - **Identificadores directos**: Nombre completo, dirección, número de identificación nacional.
  - **Identificadores indirectos**: Dirección IP, número de serie del dispositivo, historial preciso de ubicación.
- **Confidencialidad**: Mantener la PII segura contra accesos no autorizados mediante cifrado y controles de acceso.
- **Minimización de datos**: Recopilar solo la mínima PII necesaria para completar una tarea específica.
- **Limitación de propósito**: Los datos recopilados para un motivo no pueden reutilizarse (por ejemplo, venderse a redes publicitarias) sin un nuevo consentimiento del usuario.
- **Rastreo**: Monitoreo continuo del comportamiento del usuario en línea usando scripts, huellas digitales de dispositivos y registros del historial del navegador.
- **Derecho al olvido (borrado de datos)**: Derecho legal a solicitar la eliminación permanente de todos los registros personales de las bases de datos de una empresa tras cerrar una cuenta.

### El archivo `.env` y `dotenv`

- **Archivo `.env`**: Un archivo de texto plano en la raíz del proyecto que almacena secretos de configuración como claves API, contraseñas de base de datos y números de puerto como pares clave-valor. Siempre agrégalo a `.gitignore` para mantener los secretos fuera del control de versiones.
- **Paquete `dotenv`**: Lee el archivo `.env` al iniciar y inyecta los valores en `process.env`:

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **Dos beneficios clave**: Las credenciales se mantienen fuera del código fuente (seguridad), y la misma base de código puede comportarse diferente en desarrollo, staging y producción cambiando el archivo `.env` en cada servidor (flexibilidad).

### Leyes regionales de privacidad y cumplimiento

- **GDPR (Reglamento general de protección de datos)**: Protege a individuos en la UE. Requiere consentimiento explícito antes del rastreo, sin casillas pre-marcadas. Otorga a los usuarios el derecho a ver y eliminar permanentemente sus datos. Impone fuertes multas por incumplimiento.
- **CCPA (Ley de privacidad del consumidor de California) / CPRA (Ley de derechos de privacidad de California)**: Protege a residentes de California. Se enfoca en derechos de exclusión más que en consentimiento previo. Requiere un enlace visible "No vender mi información personal" en la página principal.
- **COPPA (Ley de protección de privacidad infantil en línea)**: Ley federal de EE. UU. que protege a niños menores de 13 años. Requiere una política de privacidad clara y consentimiento parental verificable antes de recopilar datos personales de un niño.
- **DPA (Ley de protección de datos)**: Marco principal de privacidad de datos del Reino Unido. Funciona junto con el GDPR del Reino Unido y otorga a los usuarios el derecho a acceder y eliminar datos personales.
- Otros países tienen sus propias leyes: Nigeria tiene la NDPA (Ley de protección de datos de Nigeria) e India tiene la DPDP Act (Ley de protección de datos personales digitales).
- **Pasos de cumplimiento para equipos globales**:
  - Construir banners de consentimiento conscientes de la ubicación que muestren el aviso de cookies adecuado según la ubicación geográfica del usuario.
  - Implementar un sistema automatizado DSAR (Solicitud de acceso de sujeto de datos) para encontrar, exportar o eliminar permanentemente los datos de un usuario tras su solicitud legal.
  - Practicar minimización de datos y usar verificaciones de edad para evitar rastreo no autorizado de menores.
  - Designar un Oficial de Protección de Datos (DPO) para supervisar la arquitectura de privacidad y actuar como contacto principal para autoridades regulatorias.
