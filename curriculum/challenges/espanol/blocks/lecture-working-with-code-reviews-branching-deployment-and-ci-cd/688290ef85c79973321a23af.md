# --description--

En una lección anterior, aprendiste cómo hacer un pull request a tu propio repositorio. También aprendiste cómo fusionar ese pull request. Pero solo pudiste hacer eso porque el repositorio te pertenece: tienes acceso de mantenedor a tus propios repositorios, lo que significa que tienes permiso para fusionar pull requests.

Cuando contribuyes a un proyecto de código abierto de otra persona, sin embargo, normalmente no tendrás esos permisos. En cambio, tendrás que esperar a que los mantenedores revisen el código y acepten tu contribución. Luego ellos se encargarán de fusionarla.

¿Pero cómo es el proceso de revisión de código?

El proceso que siguen los mantenedores varía de persona a persona, y puede haber enfoques específicos para cada proyecto. Pero en general, típicamente hay tres pasos en una revisión de código.

La primera es confirmar que el CI, o integraciones continuas, pase. Aprenderás más sobre esto en una lección futura, así que por ahora puedes considerarlas como verificaciones para validar que el código cumpla con las pautas de estilo y pase un conjunto de pruebas.

Si el CI falla, los mantenedores normalmente te pedirán que actualices el pull request para solucionar los errores. Estos errores deberían incluir algún tipo de informe que puedas revisar para ver qué salió mal, pero los mantenedores también pueden proporcionarte una guía más específica para resolver el problema.

Una vez que las verificaciones pasen, los mantenedores revisarán los cambios reales en tu código. Pueden analizar la vista de diff para ver qué has modificado, agregado o eliminado. Pueden hacer preguntas sobre por qué tomaste enfoques específicos, por lo que es bueno ser deliberado y reflexivo cuando prepares las conversaciones.

También pueden pedirte que hagas modificaciones adicionales. Hablaremos de cómo es eso en un momento.

El último paso suele ser descargar tus cambios localmente. Los mantenedores pueden entonces ejecutar la aplicación con tus modificaciones, probar la funcionalidad y verificar que nada esté roto o se comporte de manera inesperada.

Si todo se ve bien, un mantenedor aprobará tu pull request, a veces con un comentario "LGTM", que significa "looks good to me". Después de que tu pull request haya sido aprobado, debería fusionarse cuando los mantenedores estén listos para integrar los cambios. Algunos proyectos pueden requerir más de una aprobación; freeCodeCamp, por ejemplo, requiere dos.

¿Pero qué pasa si algo no está correcto con tu pull request? Los mantenedores pueden realizar una revisión en la que solicitan cambios. Estas solicitudes pueden presentarse de un par de formas.

A veces, un mantenedor puede simplemente dejar un "comment" con su solicitud, tal vez indicando algo que deberías cambiar o actualizar, pero con más libertad sobre cómo lo implementas.

O, los mantenedores también pueden dejar un comentario oficial de "request changes". Normalmente, un comentario así bloquea que el pull request sea fusionado. Tendrás que hacer los cambios deseados, y luego el mismo mantenedor que solicitó los cambios tendrá que revisar de nuevo y pedir más cambios o aprobar tu trabajo.

Cuando los mantenedores tienen cambios específicos en mente, en realidad pueden proponer cambios a través de la interfaz de GitHub. Cuando usan una sugerencia directa como esta, GitHub te dará un botón para confirmar la sugerencia directamente en tu rama. Sin embargo, si hay múltiples sugerencias como esta, querrás cambiar a la vista de archivos.

Desde esta vista, puedes agregar todas las sugerencias (una por una) en un "batch" y confirmarlas todas de una vez. Esto es increíblemente útil cuando hay muchas sugerencias, ya que hacerlas individualmente resultaría en muchos commits. Todas las verificaciones de CI tendrían que ejecutarse en cada commit, y esto puede ralentizar el proceso para aceptar tu PR.

Si los cambios solicitados son significativos, probablemente querrás hacer los cambios localmente, confirmarlos en la misma rama que usaste para el PR y subirlos. GitHub actualizará automáticamente tu pull request para reflejar los nuevos commits.

Hay un par de otras notas que queremos cubrir.

Primero, recuerda que los mantenedores también son personas. Los proyectos de código abierto rara vez generan suficientes ingresos para que los mantenedores trabajen en ellos a tiempo completo, lo que significa que tienen que mantener un trabajo. Es importante tener paciencia con el proceso de revisión: no hagas cosas como molestar a los mantenedores repetidamente para una nueva revisión, ni intentes apresurarlos en su proceso de revisión.

¡Hacer contribuciones puede ser emocionante! Pero no te apresures en el proceso. En cambio, tómate tu tiempo. Asegúrate de probar tus cambios localmente: ejecuta cualquier suite de pruebas que el proyecto pueda tener, ejecuta el servidor de desarrollo y confirma manualmente que tus cambios funcionen, etc. Y siempre sigue las directrices de contribución para el proyecto específico en el que estás trabajando.

Si tienes todo eso en mente y pones tu mejor esfuerzo, tus contribuciones deberían ir sin problemas y la revisión de tu código debería ser una aprobación sólida.
