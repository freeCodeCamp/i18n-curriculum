# --interactive--

修飾子は「フラグ」とも呼ばれ、正規表現の動作を変更します。以前のレッスンの例を思い出しましょう。

:::interactive_editor

```js
const regex = /freeCodeCamp/;
console.log(regex.test("freeCodeCamp")); // true
console.log(regex.test("freeCodeCamp is great")); // true
console.log(regex.test("I love freeCodeCamp")); // true
console.log(regex.test("freecodecamp")); // false
console.log(regex.test("FREECODECAMP")); // false
console.log(regex.test("free")); // false
console.log(regex.test("code")); // false
console.log(regex.test("camp")); // false
```

:::

覚えているなら、すべて小文字とすべて大文字の`freeCodeCamp`文字列はパターンにマッチしませんでした。これは、デフォルトでは正規表現が大文字と小文字を区別するためです。

では、正規表現に大文字小文字を区別しないように指示できたらどうでしょう？そのための修飾子があります。`i`フラグは正規表現に大文字小文字を無視させます。どうやって使うのでしょう？フラグは正規表現の閉じスラッシュの後に付けます。

```js
const regex = /freeCodeCamp/i;
```

最初の行の正規表現の変化に注目してください。これでどう変わるか確認してみましょう。

:::interactive_editor

```js
const regex = /freeCodeCamp/i;

console.log(regex.test("freeCodeCamp")); // true
console.log(regex.test("freeCodeCamp is great")); // true
console.log(regex.test("I love freeCodeCamp")); // true
console.log(regex.test("freecodecamp")); // true
console.log(regex.test("FREECODECAMP")); // true
console.log(regex.test("free")); // false
console.log(regex.test("code")); // false
console.log(regex.test("camp")); // false
```

:::

正規表現が大文字小文字を区別しなくなったので、すべて小文字とすべて大文字の文字列もテストに「合格」しました。大文字と小文字が混ざった文字列でも同様に動作します。

:::interactive_editor

```js
const regex = /freeCodeCamp/i;

console.log(regex.test("dO yOu LoVe fReEcOdEcAmP?")); // true
```

:::

他にも使えるフラグはたくさんあります。`g`フラグ、つまりグローバル修飾子は、正規表現がパターンに複数回マッチすることを可能にします。

コードにどのような影響があるか見てみましょう。`i`フラグはそのまま残しています。正規表現は複数のフラグ（必要なだけ）を使って望む動作を実現できます。

```js
const regex = /freeCodeCamp/gi;
```

あれ？これは何でしょう？グローバル修飾子のせいで、本来合格すべき文字列が失敗しているようです。

:::interactive_editor

```js
const regex = /freeCodeCamp/gi;

console.log(regex.test("freeCodeCamp")); // true
console.log(regex.test("freeCodeCamp is great")); // false
console.log(regex.test("I love freeCodeCamp")); // true
console.log(regex.test("freecodecamp")); // false
console.log(regex.test("FREECODECAMP")); // true
console.log(regex.test("free")); // false
console.log(regex.test("code")); // false
console.log(regex.test("camp")); // false
```

:::

なぜでしょう？グローバル修飾子は正規表現をステートフルにします。つまり、以前にマッチした場所を記憶します。最初の`freeCodeCamp`文字列にマッチしたとき、インデックス`0`からマッチが見つかったことを覚えています。

次に`freeCodeCamp is great`に対してテストしますが、インデックス`0`から始まりません。正規表現はすでにインデックス`0`でマッチを見つけたことを「知っている」ので、たとえ異なる文字列でもマッチの終了インデックスから開始します。

`freeCodeCamp`は`12`文字の長さなので、インデックス`0`でのマッチはインデックス`11`で終わります。マッチはインデックス`12`から再開されます。そして`is great`が`freeCodeCamp`にマッチしないため、`false`を返します。

マッチが見つからなかったため、状態を「失い」、次のマッチはインデックス`0`から始まります。

もしログを入れ替えて、インデックス`0`でマッチする文字列の直後にインデックス`11`より後でマッチする文字列が続くようにしたら：

:::interactive_editor

```js
const regex = /freeCodeCamp/gi;

console.log(regex.test("freeCodeCamp")); // true
console.log(regex.test("I loooooooove freeCodeCamp")); // true
```

:::

正規表現がグローバルの場合、`lastIndex`という新しいプロパティを持ちます。前のコードを使って、このプロパティがどう動くか見てみましょう。

:::interactive_editor

```js
const regex = /freeCodeCamp/gi;

console.log(regex.lastIndex); // 0
console.log(regex.test("freeCodeCamp")); // true
console.log(regex.lastIndex); // 12
console.log(regex.test("freeCodeCamp is great")); // false
console.log(regex.lastIndex); // 0
console.log(regex.test("I love freeCodeCamp")); // true
console.log(regex.lastIndex); // 19
console.log(regex.test("freecodecamp")); // false
console.log(regex.lastIndex); // 0
console.log(regex.test("FREECODECAMP")); // true
console.log(regex.lastIndex); // 12
console.log(regex.test("free")); // false
console.log(regex.lastIndex); // 0
console.log(regex.test("code")); // false
console.log(regex.lastIndex); // 0
console.log(regex.test("camp")); // false
```

:::

この例を見ると、`lastIndex`を使ってテストごとに正規表現の状態がどのように変わるか分かります。

グローバルフラグは、1つの文字列から複数のマッチを取得したいときに便利です。しかし、複数の文字列を同じ正規表現でテストする場合は、`g`フラグはオフにしておくのがベストです。

次のフラグを学ぶ前に、アンカーについて知る必要があります。キャレット（`^`）アンカーは正規表現の先頭にあり、「文字列の先頭にマッチする」と言います。

```js
const start = /^freecodecamp/i;
```

ドル記号（`$`）アンカーは正規表現の末尾にあり、「文字列の末尾にマッチする」と言います。

```js
const end = /freecodecamp$/i;
```

右側の出力を比べてみてください。

:::interactive_editor

```js
const start = /^freecodecamp/i;
const end = /freecodecamp$/i;
console.log(start.test("freecodecamp")); // true
console.log(end.test("freecodecamp")); // true
console.log(start.test("freecodecamp is great")); // true
console.log(end.test("freecodecamp is great")); // false
console.log(start.test("i love freecodecamp")); // false
console.log(end.test("i love freecodecamp")); // true
console.log(start.test("have met freecodecamp's founder")); // false
console.log(end.test("have met freecodecamp's founder")); // false
```

:::

開始アンカーは文字列の先頭にのみマッチし、終了アンカーは文字列の末尾にのみマッチするのが分かりますね。では、複数行にまたがるマッチはどうでしょう？見てみましょう。

:::interactive_editor

```js
const start = /^freecodecamp/i;
const end = /freecodecamp$/i;
const string = `I really love
freecodecamp
it's my favorite`;
console.log(start.test(string)); // false
console.log(end.test(string)); // false
```

:::

`freecodecamp`が単独の行にあっても、両方のテストに失敗します。これは、デフォルトではアンカーが文字列全体の先頭と末尾を探すためです。

しかし、`m`フラグ、つまりマルチライン修飾子を使うと、正規表現が複数行を扱えるようになります。これを正規表現に追加してみましょう。

:::interactive_editor

```js
const start = /^freecodecamp/im;
const end = /freecodecamp$/im;
const string = `I really love
freecodecamp
it's my favorite`;
console.log(start.test(string)); // true
console.log(end.test(string)); // true
```

:::

今は両方マッチします！`freecodecamp`が完全に単独の行にあるため、開始アンカーはその行の先頭に、終了アンカーはその行の末尾にマッチします。

最後に`d`フラグ、つまりインデックス修飾子があります。`i`フラグは大文字小文字を区別しないため、インデックス修飾子は別のフラグが必要でした。

`d`フラグはマッチオブジェクトで得られる情報を拡張します。正規表現に追加してみましょう。

```js
const regex = /freecodecamp/di;
const string = "we love freecodecamp isn't freecodecamp great?";
console.log(string.match(regex));
```

結果はこうなります。

```js
// [
//   'freecodecamp',
//   index: 8,
//   input: "we love freecodecamp isn't freecodecamp great?",
//   groups: undefined,
//   indices: [
//     0: [8, 20],
//     groups: undefined
//   ]
// ]
```

マッチオブジェクトに新しい`indices`プロパティが追加されます！このプロパティは2つの数字の配列で、最初の数字は元の文字列でマッチが始まるインデックス、2番目はマッチが終わった直後のインデックスです。この配列には`groups`プロパティもあり、名前付きキャプチャグループ用です。

他にも知っておくべきフラグはありますが、一般的なコードではあまり使われません。

1つ目はユニコード修飾子、`u`フラグです。これにより正規表現が特殊なユニコード文字にマッチできるようになります。

文字クラスについては今後のレッスンで学びますが、`u`フラグは`Extended_Pictographic`のような特殊クラスにアクセスでき、ほとんどの絵文字にマッチします。

:::interactive_editor

```js
const regex = /🍎/u;

const str = "I have an apple 🍎";
console.log(regex.test(str)); // true
```

:::

また`v`フラグもあり、ユニコードマッチングの機能をさらに拡張します。

2つ目はスティッキー修飾子、`y`フラグです。スティッキー修飾子はグローバル修飾子に似ていますが、いくつか例外があります。

最大の違いは、グローバル正規表現は`lastIndex`から始まり文字列の残り全体を検索しますが、スティッキー正規表現は前回の`null`でマッチがすぐに見つからなければ`lastIndex`を返し、`0`を`lastIndex`にリセットします。

最後はシングルライン修飾子、`s`フラグです。マルチライン修飾子は開始と終了アンカーが行の先頭と末尾にマッチするようにしますが、

シングルライン修飾子は正規表現のワイルドカード文字（`.`で表されるピリオド）が改行にもマッチするようにし、文字列を1行のテキストとして扱います。

修飾子はたくさんありますが、`i`と`g`フラグが最もよく使われ、覚えておくべき重要なものです。
