# --description--

Hai già imparato come fare commit delle modifiche e inviarle al tuo repository remoto, ma tutto questo è stato fatto sul tuo branch predefinito `main`. Questo può andare bene per un progetto piccolo, ma man mano che contribuisci ad altri progetti open source o il tuo progetto cresce, i branch diventano essenziali.

Per esempio, se fai un fork di un repository e fai commit sul branch `main` del tuo fork, potresti ritrovarti con modifiche nel tuo fork che il repository originale non ha accettato. Questo può rendere molto difficile aggiornare il tuo branch con le modifiche del fork che sono state accettate. Oppure, quando lavori sul tuo progetto, passare da un'attività all'altra diventa molto più complicato se non usi i branch. Invece, devi tenere traccia manualmente di quali modifiche riguardano quale funzionalità o correzione.

Ma cos’è esattamente un branch? Considera il tuo branch `main` come un’autostrada, con ogni commit che è un’auto che entra in autostrada. Potresti decidere di voler dare un’occhiata a qualcosa, come creare una nuova funzionalità, quindi devi uscire dall’autostrada per un po’ per andare a vedere.

Un branch è essenzialmente una deviazione dalla tua storia principale, dove puoi sperimentare liberamente modifiche per una nuova funzionalità o una correzione di bug. Se sei soddisfatto delle modifiche, puoi tornare sull’autostrada unendo il branch in `main`. Se non sei soddisfatto, puoi eliminare il branch e tornare a `main` senza accettare le modifiche.

Prima di creare un nuovo branch, puoi vedere i tuoi branch con il comando `git branch`. L’output potrebbe essere così:

```sh
* main
```

Al momento abbiamo solo un branch, il nostro branch predefinito `main`. L’asterisco (`*`) indica che è il branch che hai attualmente "check-out". Ne imparerai di più tra poco.

Puoi creare un nuovo branch con il comando `git branch`, specificando come argomento il nome del nuovo branch:

```sh
git branch feature
```

Questo crea un nuovo branch chiamato `feature`. Diamo un’altra occhiata ai nostri branch con `git branch`:

```sh
  feature
* main
```

Ora vediamo il nostro nuovo branch, ma siamo ancora sul branch `main`. Dobbiamo passare al nuovo branch "facendo il check-out". Quando fai il check-out di un branch, dici a Git di caricare i file esattamente nello stato in cui si trovavano in quel momento.

Facciamo il check-out del nostro nuovo branch con:

```sh
git checkout feature
```

oppure

```sh
git switch feature
```

E eseguiamo di nuovo `git branch` per vedere:

```sh
* feature
  main
```

Nota che l’asterisco si è spostato sul branch che hai attualmente "check-out".
 
Spesso quando crei un nuovo branch, vuoi fare subito il check-out. Invece di creare il branch e poi fare il check-out con due comandi, puoi creare e passare a un branch con un solo comando:

```sh
git checkout -b feature
```

oppure

```sh
git switch -c feature
```

Ora abbiamo creato e fatto il check-out di un nuovo branch. Questo branch è una copia del branch `main`, ha tutto il codice che `main` aveva al momento della creazione del branch. Ora possiamo aggiungere qualsiasi codice a questo branch senza influenzare il branch `main`. In pratica siamo usciti dall’autostrada per creare una funzionalità. Se controlli il tuo `git status`, vedresti qualcosa del genere:

```sh
On branch feature
nothing to commit, working tree clean
```

Crea un nuovo file `feature.md` eseguendo questo comando:

```sh
echo "This is our new feature" > feature.md
```

E ora lo stato sarà così:

```sh
On branch feature
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	feature.md

nothing added to commit but untracked files present (use "git add" to track)
```

Adesso è il momento di fare il commit della tua nuova funzionalità. Prima, aggiungi il file all’area di staging:

```sh
git add feature.md
```

Poi, fai il commit delle modifiche:

```sh
git commit -m "my new feature"
```

Infine, invia il tuo branch `feature` al repository:

```sh
git push -u origin feature
```

La flag `-u` collega il tuo branch locale `feature` a quello remoto, così i push futuri possono essere fatti solo con `git push`.

Adesso impareremo come fare una pull request per la nostra nuova funzionalità!
