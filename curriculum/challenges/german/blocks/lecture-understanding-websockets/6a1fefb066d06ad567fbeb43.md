# --description--

WebSockets ermöglichen die Kommunikation in Echtzeit zwischen einem Client und einem Server.

Normalerweise verwenden Client-Anwendungen wie Web- und Mobil-Apps HTTP. Bei HTTP sendet der Client eine Anfrage, und der Server antwortet darauf. Dies nennt man den Anfrage-Antwort-Zyklus, wie Sie bereits gelernt haben.

Nachdem die Verbindung hergestellt und Daten ausgetauscht wurden, wird die Verbindung geschlossen. Das funktioniert gut für die meisten Dinge, wie das Laden einer Seite, das Abrufen oder Senden von Daten. Für Echtzeitanwendungen wird es jedoch langsam und unzuverlässig.

Denken Sie zum Beispiel an eine Chat-App. Wenn Sie sich für HTTP entscheiden, muss der Client den Server immer wieder nach neuen Daten fragen, wobei die Verbindung für jede Nachricht geöffnet, die Daten empfangen und die Verbindung danach wieder geschlossen wird.

Dieses Problem lösen WebSockets. Sie schaffen eine dauerhafte Verbindung zwischen Client und Server. Sobald die Verbindung geöffnet ist, können beide Seiten jederzeit Daten in einem Echtzeitkanal austauschen, bis eine der beiden Seiten ein Schlusssignal an die andere sendet.

Obwohl WebSockets sich von HTTP unterscheiden, beginnt jede WebSocket-Verbindung mit HTTP in einem Prozess, der Handshake genannt wird.

Der Client fordert dann, die Verbindung auf WebSocket zu upgraden. Sobald der Server zustimmt, sendet er eine `101 Switching Protocols`-Antwort, die die Verbindung auf WebSocket umstellt. Danach bleibt die Verbindung offen, und die Echtzeitkommunikation beginnt.

Der Ablauf sieht so aus:

1. Client verbindet sich über HTTP mit dem Server  
2. Server akzeptiert und stellt die Verbindung auf WebSocket um  
3. Die Verbindung bleibt offen  
4. Beide Seiten tauschen jederzeit Daten aus, bis die Verbindung geschlossen wird  

Dieser Ablauf ist bidirektional, nicht wie bei HTTP ein Anfrage-Antwort-Modell.

```md
Client <---> Server
```

Bei HTTP folgt der Ablauf typischerweise einem Anfrage-Antwort-Modell:

```md
Client ---> Request
Server ---> Response
```

Was den Unterschied zwischen WebSockets und HTTP angeht: HTTP ist wie das Austauschen von Briefen, und WebSocket ist wie ein Telefonat. Sie schreiben einen Brief, warten auf eine Antwort, und das Gespräch endet. Bei einem Telefonat sprechen Sie und der Empfänger in Echtzeit miteinander.

WebSockets treiben Echtzeitanwendungen an wie Chat-Apps, Online-Multiplayer-Spiele, Live-Sport-Updates, Börsendashboards und kollaborative Werkzeuge wie Google Docs.
