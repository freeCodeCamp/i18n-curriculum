## Introduzione al controllo di versione

- **Definizione**: Un sistema di controllo di versione ti permette di tracciare e gestire le modifiche nel tuo progetto. Esempi di sistemi di controllo di versione usati nel software sono Git, SVN o Mercurial.

## Provider di controllo di versione basati sul cloud

- **Elenco dei provider di controllo di versione basati sul cloud**: GitHub e GitLab sono esempi popolari di provider di controllo di versione basati sul cloud che permettono ai team di sviluppo di collaborare e gestire repository.

## Installazione e configurazione di Git

- **Installare Git**: Per verificare se Git è già installato sulla tua macchina puoi eseguire il seguente comando nel terminale:

```sh
git --version
```

Se vedi un numero di versione, significa che Git è installato. Altrimenti, dovrai installarlo.

Per i sistemi Linux, Git spesso è preinstallato nella maggior parte delle distribuzioni. Se non hai Git preinstallato, dovresti poterlo installare con i comandi del tuo gestore di pacchetti come `sudo apt-get install git` o `sudo pacman -S git`.

Per gli utenti Mac, puoi installare Git tramite Homebrew con `brew install git`, oppure puoi scaricare l’installer eseguibile dal sito di Git.

Per Windows, puoi scaricare l’installer eseguibile dal sito di Git. Oppure, se hai configurato Chocolatey, puoi eseguire `choco install git.install` in PowerShell. Nota che su Windows potresti anche voler scaricare Git Bash per avere un ambiente shell simile a Unix.

Per assicurarti che l’installazione abbia funzionato, esegui di nuovo il comando `git --version` nel terminale.

- **Configurazioni di Git**: `git config` serve per impostare variabili di configurazione che determinano come Git opera sulla tua macchina. Per vedere le variabili di configurazione attuali e dove sono memorizzate nel sistema, puoi eseguire il seguente comando:

```sh
git config --list --show-origin
```

Al momento dovresti vedere solo le configurazioni a livello di sistema se hai appena installato Git per la prima volta.

Per impostare il nome utente, puoi eseguire il seguente comando:

```sh
git config --global user.name "Jane Doe"
```

Il flag `--global` viene usato qui per impostare il nome utente per tutti i progetti sul tuo sistema che usano Git. Se vuoi sovrascrivere il nome utente per un progetto specifico, puoi eseguire il comando nella directory di quel progetto senza il flag `--global`.

Per impostare l’indirizzo email dell’utente, puoi eseguire il seguente comando:

```sh
git config --global user.email janedoe@example.com
```

Un’altra configurazione che puoi impostare è l’editor preferito che vuoi usare con Git. Ecco un esempio di come impostare l’editor preferito su Emacs:

```sh
git config --global core.editor emacs
```

Se scegli di non impostare un editor preferito, Git userà l’editor predefinito del sistema.

## Software open source vs. closed source

- **Definizione**: “Open-source” significa che le persone possono vedere il codice che pubblichi, proporre modifiche, segnalare problemi e persino eseguire una versione modificata. “Closed-source” significa che solo le persone che autorizzi esplicitamente possono vedere e interagire con il progetto.

## GitHub

- **Definizione**: GitHub è una soluzione basata sul cloud che offre l’archiviazione di progetti con controllo di versione in qualcosa chiamato “repository” e abilita funzionalità di collaborazione da usare con quei progetti.
- **GitHub CLI**: Questo strumento serve per svolgere attività specifiche di GitHub senza uscire dalla riga di comando. Se non lo hai installato, puoi trovare le istruzioni nella documentazione di GitHub — ma dovrebbe essere disponibile nel gestore di pacchetti del tuo sistema.
- **GitHub Pages**: GitHub Pages è un’opzione per pubblicare siti statici o applicazioni che non richiedono un server back-end per gestire la logica. Cioè, applicazioni che girano interamente lato client o nel browser dell’utente possono essere completamente pubblicate su questa piattaforma.
- **GitHub Actions**: GitHub Actions è una funzionalità che ti permette di automatizzare flussi di lavoro direttamente nel tuo repository GitHub, inclusi build, test e deploy del codice.

## Comandi Git comuni

- **`git init`**: Questo comando inizializza un repository Git vuoto così Git può iniziare a tracciare le modifiche per questo progetto. Quando inizializzi un repository Git vuoto in un progetto, viene aggiunta una nuova directory nascosta `.git`. Questa directory `.git` contiene informazioni importanti per Git per gestire il progetto.
- **`git status`**: Questo comando serve per mostrare lo stato attuale della tua directory di lavoro — userai molto questo comando nel tuo flusso di lavoro.
- **`git add`**: Questo comando serve per mettere in staging le modifiche. Tutto ciò che è nell’area di staging sarà aggiunto al prossimo commit. Se vuoi mettere in staging tutte le modifiche non ancora messe in staging, puoi usare `git add .`. Il punto (`.`) è un alias per la directory corrente in cui ti trovi.
- **`git commit`**: Questo comando serve per fare il commit delle modifiche. Un commit è un’istantanea dello stato del progetto in quel momento. Se esegui `git commit`, si aprirà l’editor preferito che hai impostato nella configurazione di Git. Una volta aperto l’editor, puoi scrivere un messaggio dettagliato delle modifiche. Puoi anche scegliere di fornire un messaggio più breve usando il comando `git commit -m` così:

```sh
git commit -m "short message goes here"
```

- **`git log`**: Questo comando elenca tutti i commit precedenti con informazioni utili come autore, data del commit, messaggio del commit e hash del commit. L’hash del commit è una stringa lunga che serve come identificatore univoco per un commit.
- **`git remote add`**: Questo comando serve per configurare la connessione remota al tuo repository remoto.
- **`git push`**: Questo comando serve per inviare le modifiche a un repository remoto.
- **`git pull`**: Questo comando serve per scaricare le ultime modifiche dal repository remoto nel repository locale.
- **`git clone`**: Questo comando clona un repository. Questo significa che avrai una copia del repository. Questa copia include la cronologia del repository, tutti i file/cartelle e i commit sul tuo dispositivo locale.
- **`git remote -v`**: Questo comando mostra la lista dei repository remoti associati al tuo repository Git locale.
- **`git branch`**: Questo comando elenca tutti i tuoi branch locali.
- **`git fetch upstream`**: Questo comando dice a Git di prendere le ultime modifiche che sono sul tuo remoto upstream (che è il repository originale).
- **`git merge upstream/main`**: Questo comando dice a Git di unire le ultime modifiche dal branch `main` nel remoto upstream nel branch corrente.
- **`git reset`**: Questo comando ti permette di resettare lo stato corrente di un branch. Passare il flag `--hard` dice a Git di forzare i file locali a corrispondere allo stato del branch. Questo assicura che tu abbia una base pulita da cui lavorare.
- **`git rebase`**: Un rebase in Git è un modo per spostare o combinare una sequenza di commit da un branch a un altro.

## Lavorare con i branch

- **Definizione**: Un branch in Git è un’area di lavoro separata dove puoi fare modifiche. Il branch `main` spesso rappresenta il branch principale o di produzione in un’applicazione reale. I team di sviluppo creano più branch per nuove funzionalità e correzioni di bug e poi uniscono quelle modifiche nel branch `main`.
- **Creare un nuovo branch**: Per creare un nuovo branch puoi eseguire il seguente comando:

```sh
git branch feature
```

Per passare a quel branch, puoi eseguire il seguente comando:

```sh
git checkout feature
```

La maggior parte degli sviluppatori usa il comando abbreviato per creare e passare a un branch, che è il seguente:

```sh
git checkout -b new-branch-name
```

Un comando più recente e alternativo è il comando `git switch`. Ecco un esempio per creare e passare a un nuovo branch:

```sh
git switch -c new-branch-name
```

- **Strategie di branching**: Il tuo branch `main` è il branch predefinito e di solito è abbastanza stabile. Quindi è meglio creare nuovi branch da lì per elementi come correzioni di bug, nuove funzionalità o altro lavoro vario.
- **Conflitti di merge**: Succede quando Git cerca di unire automaticamente modifiche da branch diversi ma non riesce a decidere quali modifiche mantenere. Questo succede di solito quando ci sono modifiche in conflitto per la stessa parte del file.

## Cinque stati per un file tracciato da Git

- **“Untracked”**: Significa che il file è nuovo nel repository e Git non l’ha “visto” prima.
- **“Modified”**: Questo file esisteva nel commit precedente e ha modifiche non ancora committate.
- **“Ignored”**: Probabilmente non vedrai i file ignorati in Git, ma il tuo IDE potrebbe segnalarli. I file ignorati sono esclusi dalle operazioni Git, tipicamente perché sono inclusi nel file `.gitignore`.
- **“Deleted”**: Un file eliminato è l’opposto di un file non tracciato — è un file che esisteva prima ed è stato rimosso.
- **“Renamed”**: Un file rinominato è un file il cui contenuto non è cambiato, ma il nome o la posizione del file è stata modificata. In alcuni casi, un file può essere considerato rinominato anche se ha una piccola quantità di modifiche.

## File `.gitignore`

- **Definizione**: Il file `.gitignore` è un tipo speciale di file legato alle operazioni di Git. Il nome suggerisce che questo file serve a dire a Git di ignorare certe cose, ed è l’uso più comune. Ma in realtà dice a Git di smettere di tracciare un file.

## Lavorare con i repository

- **Definizione**: Un repository è come un contenitore per un progetto — se stai lavorando su un’app, tieni insieme i file di quell’app in un repository. I repository possono essere locali sul tuo computer o remoti su un servizio come GitHub.
- **Repository pubblici vs. privati**: Un repository pubblico può essere visto e scaricato da chiunque. Un repository privato può essere consultato solo da te e da chi autorizzi esplicitamente.
- **Creare repository su GitHub**: Per creare un nuovo repository su GitHub, puoi cliccare sul pulsante `"New Repository"` e seguire l’interfaccia di GitHub per configurare un nuovo repository.
- **Caricare repository locali su GitHub**: Se hai un progetto locale sul tuo computer, puoi caricare quel repository su GitHub. Ecco una panoramica passo passo del processo:

1. Inizializza un repository git vuoto nella directory del progetto (`git init`).
2. Fai modifiche al progetto.
3. Esegui il comando `git status` per vedere tutte le modifiche tracciate da git.
4. Metti in staging le modifiche (`git add`).
5. Fai il commit delle modifiche (`git commit`).
6. Configura la connessione remota (`git remote add`).
7. Carica le modifiche su GitHub (`git push`).

## Pull request

- **Pull request**: Una pull request è una richiesta di unire le modifiche dal tuo branch nel branch di destinazione. Le pull request sono il flusso che usi quando vuoi contribuire con modifiche di codice a un progetto. Questo permette ai manutentori del progetto di rivedere le tue modifiche. Possono lasciare commenti, fare domande e suggerire modifiche. Una volta completata la revisione, può essere approvata e unita nel branch principale.

## Contribuire ad altri repository

- **Processo**: Ci sono migliaia di progetti a cui puoi contribuire. Ecco il processo base per contribuire a un altro repository:

1. Leggi la documentazione per contribuire
2. Trova un problema disponibile su cui lavorare
3. Fai il fork del repository
4. Clona la tua copia forkata del repository
5. Crea un nuovo branch
6. Fai le modifiche secondo il problema
7. Crea una PR (pull request)
8. Aspetta la revisione della PR

## Lavorare con chiavi SSH e GPG

- **Chiavi GPG**: Le chiavi GPG, o Gnu Privacy Guard, sono tipicamente usate per firmare file o commit. Qualcuno può usare la tua chiave pubblica GPG per verificare che la firma del file provenga dalla tua chiave e che il contenuto del file non sia stato modificato o manomesso.

Per generare una chiave GPG, devi eseguire:

```sh
gpg --full-generate-key
```

- **Chiavi SSH**: Le chiavi SSH, o Secure SHell, sono tipicamente usate per autenticare una connessione remota a un server — tramite l’utilità `ssh`. Puoi anche usare una chiave SSH per firmare i commit.

Per una chiave SSH, esegui:

```sh
ssh-keygen -t ed25519 -C "your_email@example.com"
```

`ed25519` è un algoritmo moderno di firma a chiave pubblica.

- **Firmare i commit con chiavi GPG**: Per firmare i commit con la tua chiave GPG, devi caricare la tua chiave pubblica, non quella privata, nel tuo account GitHub. Per elencare le chiavi pubbliche, devi eseguire:

```sh
gpg --list-secret-keys --keyid-format=long
```

Poi, per ottenere la chiave pubblica, usa:

```sh
gpg --armor --export "<key id>"
```

Prendi l’ID breve ottenuto dall’elenco delle chiavi ed esegui questo comando per impostarla come chiave di firma git:

```sh
git config --global user.signingkey <your_gpg_key_id>
```

Poi puoi passare il flag `-S` al comando `git commit` per firmare un commit specifico — dovrai fornire la tua passphrase. In alternativa, se vuoi firmare automaticamente ogni commit, puoi impostare la configurazione autosign su `true`:

```sh
git config --global commit.gpgsign true
```

- **Firmare i commit con chiavi SSH**: Per firmare con una chiave SSH, che è una funzionalità relativamente nuova su GitHub, devi prima caricare la chiave nel tuo account GitHub. Poi devi impostare la modalità di firma di git per usare SSH:

```sh
git config --global gpg.format ssh
```

Poi, per impostare la chiave di firma, passi il percorso del file invece di un ID:

```sh
git config --global user.signingkey <path_to_your_ssh_keys>
```
