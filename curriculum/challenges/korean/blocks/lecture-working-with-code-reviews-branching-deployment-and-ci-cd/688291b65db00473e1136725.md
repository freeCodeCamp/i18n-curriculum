# --description--

CI는 Continuous Integration(지속적 통합)의 약자이고, CD는 Continuous Delivery(지속적 전달) 또는 Continuous Deployment(지속적 배포)의 약자입니다. 이 두 프로세스를 함께 사용하면 변경 사항이 제대로 작동하는지 확인한 후 애플리케이션에 자동으로 배포할 수 있습니다.

CI 파이프라인에서는 린터와 테스트를 코드에 대해 실행할 수 있습니다. CI 파이프라인은 일반적으로 풀 리퀘스트에서 실행되어 변경 사항이 애플리케이션을 망가뜨리지 않는지 확인하고, `main` 브랜치에서도 실행되어 배포 준비가 되었는지 확인합니다.

CD 파이프라인은 미리보기 환경을 설정해 변경 사항을 실시간으로 확인할 수 있다면 풀 리퀘스트에서도 실행할 수 있습니다. 하지만 `main` 브랜치에서는 변경 사항이 승인되고 병합될 때마다 지속적으로 배포하기 위해 실행합니다.

그렇다면 GitHub Actions와 실제로 어떻게 작동할까요? 먼저 GitHub Actions가 무엇인지 이해해야 합니다. GitHub Actions는 특정 명령을 실행할 수 있는 임시 실행기(임시 컴퓨터처럼 생각하세요)를 제공합니다.

Actions는 YAML 파일로 구성합니다. 여러 파일을 만들어 서로 다른 작업을 지정할 수 있습니다. 예를 들어, 하나는 CI용, 다른 하나는 CD용일 수 있습니다.

기본 CI 예제를 살펴보겠습니다:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  lint:
    name: Lint and Test
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Source Files
        uses: actions/checkout@v4

      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10

      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

내용이 많으니 나눠서 살펴보겠습니다. 첫 번째 섹션을 보면:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```

`name`는 액션 자체의 이름을 정의합니다. 이 이름은 PR의 상태 검사 섹션 같은 영역에 표시됩니다. `on` 속성은 "워크플로 트리거"라고 불리는 것으로, 액션을 실행할 이벤트를 정의합니다.

이 예제에서는 커밋이 `main`에 푸시되거나 풀 리퀘스트가 `main`를 대상으로 할 때 액션을 실행합니다. 즉, 누군가 풀 리퀘스트를 만들 때와 병합할 때 CI가 실행됩니다.

`jobs` 섹션은 액션이 실제로 수행할 작업을 정의하는 곳입니다. `lint:` 키는 해당 작업의 내부 이름을 임의로 정의합니다. `name` 속성은 이 작업의 외부 이름을 정의합니다. `runs-on` 속성은 액션이 실행될 환경 종류를 정의하는데, 이 경우 최신 지원 버전의 Ubuntu에서 실행합니다.

`steps` 속성은 실제 작업이 이루어지는 곳입니다. 이 블록은 액션이 수행할 단계들을 정의합니다. 첫 번째 블록을 살펴보겠습니다:

```yml
      - name: Checkout Source Files
        uses: actions/checkout@v4
```

이 단계에는 `name`과 `uses` 속성이 있습니다. `uses` 속성은 특별한데, 액션 실행기에게 명령을 실행하는 대신 GitHub에서 공개 액션을 가져와 실행하라고 지시합니다. `actions/checkout` 액션은 실행기 내에서 저장소를 복제하고 설정하는 역할을 합니다.

```yml
      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10
```

다음 두 블록도 서드파티 액션을 사용합니다. 이 단계들은 Node와 pnpm 패키지 관리자를 설정합니다. 두 단계 모두 `with` 키가 있어 서드파티 액션에 인수를 전달할 수 있습니다. 이것은 함수에 인수를 전달하는 것과 비슷하게 생각하세요.

마지막 네 블록이 실제 핵심 작업이 이루어지는 부분입니다.

```yml
      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

`Install Dependencies` 블록은 npm 패키지를 설치합니다. `Lint Source Files` 블록은 린터를 실행하고, `Verify Build` 블록은 Typescript 코드가 컴파일되는지 확인하며, `Run Tests` 블록은 단위 테스트를 실행합니다.

액션에서 중요한 점은 기본적으로 이 단계들 중 하나라도 실패하면 나머지 단계는 건너뛰고 전체 실행이 실패로 표시된다는 것입니다.

이 YAML 파일들을 어디에 두어야 GitHub가 이 액션들을 실행할 수 있을까요? 저장소 내 `.github/workflows` 디렉터리에 있어야 GitHub가 찾아서 사용할 수 있습니다.

액션이 할 수 있는 추가 기능이 많으니 문서를 읽고 freeCodeCamp 저장소 같은 프로젝트에서 기존 액션을 탐색해 보길 권장합니다. 하지만 지금은 GitHub Actions와 CI/CD의 기본 개념을 잘 이해했을 것입니다!
