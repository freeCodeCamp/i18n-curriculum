## Codici di stato della risposta HTTP

- I **codici di stato** sono brevi messaggi numerici che indicano al client cosa è successo dopo aver effettuato una richiesta (per esempio, 200, 404 o 500).

### 1xx — Informativi

- **100 Continue** — il server dice al client di inviare il resto della richiesta.
- **101 Switching Protocols** — il server sta cambiando protocollo (comune con WebSocket).

### 2xx — Successo

- **200 OK** — la richiesta è andata a buon fine.
- **201 Created** — una nuova risorsa è stata creata con successo.
- **204 No Content** — la richiesta è andata a buon fine ma non c’è un corpo di risposta (spesso usato per `DELETE`).

### 3xx — Reindirizzamento

- **301 Moved Permanently** — la risorsa ha un nuovo URL permanente.
- **302 Found** — la risorsa si trova temporaneamente a un URL diverso.
- **304 Not Modified** — il client può usare la versione in cache; nulla è cambiato.

### 4xx — Errori del client

- **400 Bad Request** — la richiesta è malformata o non valida.
- **401 Unauthorized** — è richiesta l’autenticazione.
- **403 Forbidden** — autenticato, ma senza permessi.
- **404 Not Found** — la risorsa richiesta non esiste.

### 5xx — Errori del server

- **500 Internal Server Error** — errore generico lato server.
- **502 Bad Gateway** — un proxy ha ricevuto una risposta non valida da un altro server.
- **503 Service Unavailable** — il server è temporaneamente sovraccarico o non disponibile.

### Usare i codici di stato in Express

- `res.status(400).send("Bad Request")` — invia un 400 con un messaggio in testo semplice.
- `res.status(200).json({ message: "Success" })` — invia un 200 con un corpo JSON.
- `res.status(500).send("Internal Server Error")` — invia un 500 con un messaggio.

## Gestione degli errori in Express

- Il **middleware per la gestione degli errori** ha quattro parametri: `(err, req, res, next)`. La firma con quattro parametri segnala a Express che si tratta di un gestore di errori.
- Metti il middleware per la gestione degli errori **alla fine** dello stack di middleware, dopo tutte le rotte.

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Gestione asincrona degli errori in Express 4

- Gli errori asincroni **non vengono catturati automaticamente** in Express 4.
- Avvolgi la logica asincrona in `try/catch` e chiama `next(err)` per inoltrare l’errore al gestore degli errori.

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- Un wrapper `asyncHandler` evita di ripetere `try/catch` in ogni rotta:

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Gestione asincrona degli errori in Express 5

- In Express 5, gli errori asincroni e le promise rifiutate vengono **inoltrati automaticamente** al gestore degli errori — non serve più chiamare manualmente `next(err)`.

### Gestire gli errori 404

- Aggiungi un middleware catch-all **dopo le rotte ma prima del gestore degli errori** per gestire le rotte non corrispondenti:

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Buone pratiche per la gestione degli errori

- Metti sempre il middleware per la gestione degli errori per ultimo.
- Gestisci i 404 separatamente dagli altri errori.
- In produzione, registra internamente i dettagli degli errori — non inviare mai tracce dello stack agli utenti.

## Debugging e logging in Express

- **Debugging integrato**: imposta la variabile d’ambiente `DEBUG` per abilitare l’output di debug integrato di Express.
  - `DEBUG=express:* node index.js` — abilita tutti i messaggi di debug di Express.
  - `DEBUG=express:router node index.js` — mostra solo l’output di debug relativo al router.
  - PowerShell: `$env:DEBUG = "express:*"; node index.js`
  - CMD: `set DEBUG=express:* && node index.js`
- **`morgan`**: un middleware popolare per registrare le richieste HTTP (metodo, URL, codice di stato, tempo di risposta).

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- Il **debugging** traccia come Express gestisce le richieste e i middleware senza intasare i log di produzione.
- Il **logging** registra cosa succede durante le richieste — utile per individuare bug e capire il traffico.

## Health check e spegnimenti graduali

### Health check

- Una **rotta di health check** segnala se l’app funziona correttamente.
- I bilanciatori di carico e gli orchestratori (per esempio Kubernetes, AWS Elastic Beanstalk) interrogano questa rotta per decidere se il servizio è sano.
- Una risposta **200** indica che è sano; nessuna risposta fa scattare un riavvio automatico.

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### Spegnimento graduale

- Uno **spegnimento graduale** permette all’app di completare le richieste in corso prima di fermarsi, invece di interrompere subito le connessioni.
- Ascolta il segnale `SIGTERM` (inviato quando il processo deve terminare) e chiama `server.close()`:

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- Ascolta il segnale `SIGINT` (inviato quando lo sviluppatore preme Ctrl+C durante lo sviluppo locale) e chiama la stessa logica di spegnimento di `SIGTERM`.

### Segnalare che l’app non è sana prima dello spegnimento

- Ritorna un **503** dall’health check durante lo spegnimento così il bilanciatore di carico smette di indirizzare nuovo traffico all’app.

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
