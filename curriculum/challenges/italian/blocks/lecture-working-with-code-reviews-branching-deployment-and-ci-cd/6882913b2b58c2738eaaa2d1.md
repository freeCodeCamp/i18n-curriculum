# --description--

Abbiamo brevemente accennato al file `.gitignore` in una lezione precedente. Ma cos'è esattamente questo file e cosa dovresti metterci dentro?

Il file `.gitignore` è un tipo speciale di file legato alle operazioni Git. Il nome suggerisce che questo file serve a dire a Git di ignorare delle cose, ed è proprio questo l'uso comune. Ma in realtà ciò che fa è dire a Git di smettere di tracciare un file. Questa distinzione è importante e sarà chiarita più avanti.

Cosa dovresti effettivamente mettere in questo file? Beh, dovresti elencare le cose che non vuoi vengano committate.

La prima cosa sono i tuoi file segreti. Potrebbe essere qualcosa come un file `.env` dove hai memorizzato le tue chiavi API per la tua applicazione. Potrebbe essere un file di chiave GPG privata che la tua app deve usare per l'autenticazione. Qualsiasi cosa contenga informazioni sensibili che non vuoi vengano divulgate al pubblico dovrebbe essere ignorata con `.gitignore`. Anche se il tuo repository è privato, dovresti comportarti come se fosse pubblico — trattalo come se chiunque potesse vederlo e fai attenzione a non committare segreti.

Diamo un'occhiata a come potrebbe apparire il nostro file in questo momento:

```sh
# Secrets
.env
github.key
```

Abbiamo detto a Git di ignorare il file `.env`, dove teniamo i nostri segreti. Abbiamo anche detto a Git di ignorare un file `github.key`, che contiene una chiave GPG privata.

La cosa successiva da considerare sono i pacchetti, o dipendenze, che il tuo progetto usa. Per esempio, quando lavori con un progetto Node.js, probabilmente installi pacchetti da npm. Questi pacchetti finiscono in una cartella `node_modules` nel tuo progetto, così il tuo codice può importarli e usarli.

Ma la cartella `node_modules` diventa enormemente grande, e sicuramente non vuoi committare tutti quei pacchetti e tracciare i cambiamenti ogni volta che ne aggiorni uno. Invece, committi il tuo file `package.json` e il relativo lockfile (per esempio `package-lock.json`, `pnpm-lock.yaml`). Questi file dicono a npm quali pacchetti e versioni installare, così un collaboratore può installare le stesse dipendenze senza doverle committare nel repository.

Andiamo avanti e aggiungiamo `node_modules` al nostro file `.gitignore`:

```sh
# Secrets
.env
github.key

# Packages
node_modules
```

Ora, parliamo di una cosa importante riguardo la sintassi per i file `.gitignore`. Quando aggiungi una voce come `node_modules`, Git ignorerà tutto ciò che corrisponde a quel pattern. Questo significa che ignorerà la directory `node_modules` nella radice del tuo progetto, ma ignorerà anche `node_modules` nella directory `client` del tuo progetto.

Se vuoi ignorare qualcosa solo nella radice, puoi anteporre la riga con una barra (/). `node_modules` diventerebbe `/node_modules`. Questo ignorerebbe la directory `node_modules` nella radice, ma non la directory `client/node_modules`. Non vogliamo questo, quindi lasciamo la nostra voce solo come `node_modules`.

La cosa successiva da considerare sono gli output di build. Per esempio, compilare un progetto TypeScript potrebbe generare una cartella `dist` per l'output JavaScript. Non vuoi committare quell'output perché appesantisce la cronologia Git. Invece, vuoi ignorare quell'output — i collaboratori possono rigenerarlo da soli eseguendo lo stesso comando di build.

Ignoriamo la cartella `dist`:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
```

A volte, un file fuori posto può far fallire il compilatore TypeScript, causando che l'output JavaScript finisca sparso ovunque. Poiché scriviamo esclusivamente TypeScript, usiamo la sintassi glob per ignorare tutti i file JavaScript:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js
```

Altri output comuni che potresti voler ignorare sono la directory `coverage` per i risultati della copertura dei test unitari, una directory `.angular` se usi Angular, una directory `.next` se usi NextJS, e così via.

Per ora, possiamo limitarci a quello che abbiamo qui. Se usi un framework che genera file, dovresti verificare nella loro documentazione cosa devi aggiungere al `.gitignore`.

Il quarto gruppo di cose da ignorare sono i file IDE. I prodotti Jetbrains, per esempio, potrebbero usare una cartella `.idea` per configurazioni specifiche dell'IDE. Visual Studio Code usa una cartella `.vscode`.

Spesso è bene ignorarli, perché contengono file specifici del tuo ambiente locale che altri collaboratori non useranno (perché usano le loro configurazioni). Aggiungiamo la directory di Visual Studio Code, dato che è l'IDE che stiamo usando.

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
```

Ma la nostra cartella `.vscode` contiene un file `settings.json` che include alcune cose specifiche del workspace che potrebbero essere utili ai nostri collaboratori — questo file permette loro di configurare le estensioni e l'ambiente per rispecchiare meglio il nostro.

Quindi usiamo la sintassi di negazione per "non ignorare" quel file:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json
```

Se usi MacOS, potresti conoscere il file `.DS_STORE`. Questo file contiene metadati che Finder usa per determinare come visualizzare la directory (per esempio con miniature o come lista). Allo stesso modo, gli utenti Windows potrebbero vedere un file nascosto `Thumbs.db` che memorizza in cache le miniature per la directory.

In generale è una buona idea ignorare questi file, dato che sono impostazioni personali che non influenzano il progetto. Aggiungiamoli entrambi:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json

# System Files
.DS_STORE
Thumbs.db
```

Una nota veloce sull'ignorare file già committati. Supponiamo che tu abbia accidentalmente committato la directory `dist` in precedenza, ma ora vuoi rimuoverla e ignorarla. Non puoi semplicemente aggiungerla al tuo file `.gitignore` — questo non fa sparire il contenuto già committato, dice solo a Git di smettere di tracciare il file. Quindi se ignori un file già committato, Git non traccerà più i cambiamenti ma non traccerà nemmeno la cancellazione del file.

Invece, devi prima eliminare il file. Crea un commit con quella cancellazione. Poi puoi aggiungere l'elemento al tuo `.gitignore` e creare un secondo commit con quell'aggiornamento. Da quel momento in poi, quando quel file verrà creato di nuovo, Git lo ignorerà e non apparirà nei tuoi commit.

Infine, se stai lavorando su un progetto in un linguaggio diverso, come Python, gli elementi da ignorare saranno diversi da quelli che abbiamo visto qui. GitHub mantiene un repository molto utile di template `.gitignore` per vari linguaggi e framework. Puoi trovare quella lista su <a href="https://github.com/github/gitignore" target="_blank">https://github.com/github/gitignore</a>.

Con questo, sei pronto a evitare di committare file che non vuoi nei tuoi repository!
