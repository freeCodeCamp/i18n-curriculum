# --description--

Wenn Sie eine Website besuchen, geben Sie zuerst eine Webadresse in Ihren Browser ein.

Zum Beispiel würden Sie, um zu freeCodeCamp zu gelangen, `freecodecamp.org` in die Adressleiste Ihres Browsers eingeben.

Dies wird als Domainname bezeichnet. Es ist ein eindeutiger, für Menschen lesbarer Bezeichner für diese Ressource im Internet.

Domainnamen stehen in engem Zusammenhang mit dem Konzept von DNS, dem Hauptthema dieser Lektion. Beginnen wir damit, ein wenig über die Hierarchie der Domainnamen zu sprechen, damit Sie verstehen, wie ein DNS hinter den Kulissen funktioniert.

Die Hierarchie der Domainnamen wird von rechts nach links gelesen, wobei von der allgemeinsten zur spezifischsten Kategorie übergegangen wird.

Die Hauptstruktur eines Domainnamens ist von rechts nach links:

* **Top-Level-Domain (TLD):** der letzte Abschnitt des Domainnamens, ganz rechts. Im Fall von `freecodecamp.org` wäre das `.org`. Weitere Beispiele sind `.com` und `.edu`.
    
* **Second-Level-Domain (SLD):** der Name, der von der Person oder Organisation registriert wurde. Das wäre `freecodecamp` in `freecodecamp.org`. Dies ist das, was Ihnen normalerweise zuerst auffällt, wenn Sie eine Webadresse sehen.
    
* **Third-Level-Domain (Subdomain):** wird verwendet, um einen bestimmten Abschnitt der Hauptdomain zu identifizieren. Zum Beispiel `www`, was für „World Wide Web“ steht. Heutzutage übernehmen moderne Webserver automatisch das Domain-Routing, sodass Sie eine Website mit oder ohne Eingabe von `www` aufrufen können. Weitere Beispiele für Subdomains sind `blog.`, `shop.`, `support.` und `api.`.
    

Domainnamen sind für uns Menschen sehr hilfreich, weil sie leicht zu merken und in einen Browser einzugeben sind. Computer arbeiten jedoch mit Zahlen. Ihr Browser muss den eingegebenen Domainnamen in etwas umwandeln, das als IP-Adresse bekannt ist, um die gesuchte Website zu finden.

Eine **IP-Adresse** ist eine eindeutige Zahlenfolge, die einen Computer in einem Netzwerk wie dem Internet identifiziert.

IP-Adressen sind entscheidend, damit Clients Anfragen an die richtigen Server senden und die passenden Antworten erhalten können. Ein Beispiel für eine IPv4-Adresse ist `127.0.0.1`. Die entsprechende IP-Adresse im neueren Standard IPv6 wäre `0:0:0:0:0:0:0:1`.

Es gibt viel über IP-Adressen zu lernen, und Sie möchten sich vielleicht intensiver damit beschäftigen, während Sie weiterlernen. Für den Moment müssen Sie nur verstehen, dass es sich um eindeutige numerische Adressen handelt, mit denen Computer in einem Netzwerk identifiziert werden.

Aus den Konzepten von Domainnamen und IP-Adressen ergibt sich nun das Konzept von DNS.

DNS steht für **Domain Name System**. Es wird oft als das „Telefonbuch“ des Internets bezeichnet.

Das DNS übersetzt den Domainnamen, den Sie in Ihren Browser eingeben, in die entsprechende IP-Adresse. Es ist ein hierarchisches und dezentrales Namenssystem für Computer und Geräte, die mit dem Internet oder einem bestimmten Netzwerk verbunden sind.

Wenn der Benutzer einen Domainnamen im Browser eingibt, wie zum Beispiel `freecodecamp.org`, ordnet das DNS diesen Namen der IP-Adresse des entsprechenden Servers zu. Sobald der Server identifiziert ist, wird die Anfrage des Clients an den Server gesendet. Der Server bearbeitet diese Anfrage entsprechend seiner Programmierung und sendet schließlich eine Antwort an den Client. Damit ist der Anfrage-Antwort-Zyklus abgeschlossen.

Das ist alles dank des DNS möglich, das diese anfängliche Umwandlung in die Ziel-IP-Adresse durchgeführt hat.

Tauchen wir nun in die Details des DNS-Auflösungsprozesses ein.

## Der DNS-Auflösungsprozess

* Die DNS-Auflösung funktioniert, indem eine Abfrage durch eine Kette von DNS-Servern gesendet wird.

* Wenn Sie (der Benutzer) einen Domainnamen im Browser eingeben, sendet der **Client** eine Anfrage an einen lokalen DNS-Server. Dieser lokale DNS-Server wird als **Recursive Resolver** bezeichnet. Er wird normalerweise von Ihrem Internetdienstanbieter (ISP) verwaltet.
    
* Der Recursive Resolver leitet diese Anfrage an die **Root-Server** weiter. Dies sind allgemeine Server, die keine direkten Informationen über IP-Adressen haben. Sie arbeiten, indem sie den Recursive Resolver an einen spezifischeren Server weiterleiten, nämlich an den Top-Level-Domain-(TLD-)Server für die jeweilige Top-Level-Domain der Anfrage. Zum Beispiel den Server, der für alle `.org`-Domains zuständig ist.
    
* Der Recursive Resolver fragt dann den **Top-Level-Domain-(TLD-)Server** ab. Dieser Server gibt die Adresse des spezifischen Authoritative Name Servers zurück, der die Einträge für die angefragte Domain enthält (zum Beispiel `freecodecamp.org`).
    
* Der Recursive Resolver startet eine letzte Abfrage an den entsprechenden **Authoritative Name Server.** Wenn der **DNS-Eintrag** gefunden wird, wird er an den Recursive Resolver zurückgegeben.
    
* Wenn der Recursive Resolver die IP-Adresse erhält, liefert er sie an den Client, sodass Sie mit dem Surfen auf der Website beginnen können.

Der Authoritative Name Server liefert **DNS-Einträge** zurück, nicht nur IP-Adressen. DNS-Einträge sind strukturierte Texte, die wichtige Informationen über eine Domain enthalten, wie Subdomain-Informationen, wie lange die Informationen zwischengespeichert werden sollen, und Informationen, die zur Ausführung des angeforderten Dienstes erforderlich sind.

Um den Prozess zu optimieren, speichert der Recursive Resolver den DNS-Eintrag vorübergehend im Cache, um den Vorgang zu beschleunigen. So müssen Sie dieselbe Anfrage nicht mehrfach in kurzer Zeit stellen. Das macht den gesamten Prozess viel effizienter.

Wie Sie sehen, sind DNS-Server unverzichtbar, weil sie den ersten, entscheidenden Schritt der Übersetzung von Domainnamen in IP-Adressen übernehmen. Dieser Übersetzungsprozess ermöglicht es jedem Client, den richtigen Server zu finden und sich mit ihm zu verbinden. Ohne dieses Verzeichnis-System wäre das Client-Server-Modell, das Rückgrat des modernen Webs, nicht funktionsfähig.
