# --description--
  
動的計画法は、複雑な問題をより単純な部分問題に分解し、その結果を保存して重複した計算を避けるアルゴリズムの手法です。このアプローチにより、通常は指数時間がかかる問題を多項式時間で解けるように変換します。

## 動的計画法の基本原則

動的計画法は、問題に以下の2つの重要な条件がある場合に機能します。

- **重複部分問題**：大きな問題を解く際に同じ小さな問題が何度も現れます。これらの部分問題を繰り返し計算する代わりに、その解を保存します。
    
- **最適部分構造**：問題の最適解は、その部分問題の最適解を含みます。つまり、小さな部分の最適解を組み合わせて最良の解を作り上げることができます。
    

これらの概念を、古典的な「階段を登る」問題で見てみましょう。

## 素朴な再帰の問題点

階段を登る問題を考えます。階段は`n`段あり、一度に1段か2段登れます。頂上に到達する異なる方法は何通りあるでしょうか？

```js
function climbStairsRecursive(n) {
  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Recursive rule: to reach step n, you can come from step (n-1) or step (n-2)
  return climbStairsRecursive(n - 1) + climbStairsRecursive(n - 2);
}
```

この実装は指数時間の計算量を持ち、膨大な重複計算が発生します。`climbStairsRecursive(5)`を計算するとき、以下のようになります：

- `climbStairsRecursive(5)`は`climbStairsRecursive(4)`と`climbStairsRecursive(3)`を呼び出します
    
- `climbStairsRecursive(4)`は`climbStairsRecursive(3)`と`climbStairsRecursive(2)`を呼び出します
    
- `climbStairsRecursive(3)`は**2回**計算されます
    
- `climbStairsRecursive(3)`は`climbStairsRecursive(2)`と`climbStairsRecursive(1)`を呼び出します
    
- `climbStairsRecursive(2)`は合計で**3回**計算されます
    

`n=5`だけでも、必要な計算は5つなのに9回も関数呼び出しが行われます。`n`が大きくなるにつれて、この重複は指数的に増加し、`climbStairsRecursive(30)`では数百万回以上の関数呼び出しが必要になります！計算量は`O(2^n)`となり、大きな`n`の値には非効率で実用的ではありません。

## 動的計画法による解決策

動的計画法は、以下の2つの主要な方法でこの重複計算を排除します。

### メモ化（トップダウンアプローチ）

メモ化は高コストな関数呼び出しの結果を保存し、同じ入力が再度現れたときにキャッシュされた結果を返します：

```js
function climbStairsMemo(n, memo = {}) {
  // If already calculated, return cached value
  if (memo[n] !== undefined) {
    return memo[n];
  }

  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Calculate once and store in memo for future use
  memo[n] =
    climbStairsMemo(n - 1, memo) +
    climbStairsMemo(n - 2, memo);

  return memo[n];
}
```

メモ化は非常に効率的で、`1`から`n`までの各ユニークな値を一度だけ計算します。`climbStairsMemo(3)`が再度必要になった場合、再計算せずに`O(1)`の時間でメモ辞書から参照します。

トップダウンアプローチで`climbStairsMemo(5)`の実行を追って、メモ化がどのように重複作業を排除するか見てみましょう：

```md
Call: climbStairsMemo(5)
  memo = {} (empty)
  
  Call: climbStairsMemo(4) 
    memo = {} (empty)
    
    Call: climbStairsMemo(3)
      memo = {} (empty)
      
      Call: climbStairsMemo(2) → returns 2 (base case)
      Call: climbStairsMemo(1) → returns 1 (base case)
      
      Result: 2 + 1 = 3
      memo = {3: 3} (stored!)
    
    Call: climbStairsMemo(2) → returns 2 (base case)
    
    Result: 3 + 2 = 5
    memo = {3: 3, 4: 5} (stored!)
  
  Call: climbStairsMemo(3) → returns 3 (FROM MEMO - no recursion!)
  
  Result: 5 + 3 = 8
  memo = {3: 3, 4: 5, 5: 8}
```

**効率の比較**

- **素朴な再帰**：9回の関数呼び出しで重複計算あり
    
- **メモ化**：5回のユニークな計算のみで保存した結果を再利用
    
- **時間計算量**：`O(2^n)`から`O(n)`に減少し、ユニークな計算は`n`回のみ
    
- **空間計算量**：メモ保存と呼び出しスタックで`O(n)`
    
- **実際の効果**：`climbStairsMemo(30)`は数百万回の関数呼び出しから約30回のユニーク計算に激減！
    

### タビュレーション（ボトムアップアプローチ）

タビュレーションは、部分問題の解をテーブルに埋めながら下から順に解を構築します：

```js
function climbStairsTabulation(n) {
  if (n <= 2) {
    return n;
  }

  // Create an array to store solutions from 0 to n
  const dp = new Array(n + 1).fill(0);

  // Base cases
  dp[1] = 1; // 1 way to reach step 1
  dp[2] = 2; // 2 ways to reach step 2

  // Build up the solution iteratively
  for (let i = 3; i <= n; i++) {
    // Ways to reach step i = ways to reach (i-1) + ways to reach (i-2)
    dp[i] = dp[i - 1] + dp[i - 2];
  }

  return dp[n];
}
```

タビュレーションは再帰を完全に排除し、最小の部分問題からターゲットまで反復的に解を作ります。

ボトムアップアプローチで`climbStairsTabulation(5)`の解を段階的に構築する様子を見てみましょう：

```md
Initial state:
dp = [0, 1, 2, 0, 0, 0]
     [0, 1, 2, 3, 4, 5] ← indices (step numbers)

Step by step construction:

i = 3:
  dp[3] = dp[2] + dp[1] = 2 + 1 = 3
  dp = [0, 1, 2, 3, 0, 0]
  
i = 4:
  dp[4] = dp[3] + dp[2] = 3 + 2 = 5
  dp = [0, 1, 2, 3, 5, 0]
  
i = 5:
  dp[5] = dp[4] + dp[3] = 5 + 3 = 8
  dp = [0, 1, 2, 3, 5, 8]

Final result: dp[5] = 8
```

**タビュレーションの主な利点**

- **再帰のオーバーヘッドなし**：メモ化とは異なり、再帰呼び出しスタックがありません
    
- **予測可能な実行順序**：1、2、3、4、5…の順に計算
    
- **キャッシュに優しい**：連続した配列アクセスでメモリ効率が良い
    
- **最適化が簡単**：最後の2つの値だけを保持すれば空間計算量を`O(1)`に削減可能
    

```js
function climbStairsOptimized(n) {
  if (n <= 2) {
    return n;
  }

  // Store only the last two results to save space
  let prev2 = 1; // ways to reach step 1
  let prev1 = 2; // ways to reach step 2

  for (let i = 3; i <= n; i++) {
    const current = prev1 + prev2;

    // Shift values forward
    prev2 = prev1;
    prev1 = current;
  }

  return prev1;
}
```

**効率の比較**

- **素朴な再帰**：`n=5`に対して9回の関数呼び出し、指数的増加
    
- **タビュレーション**：`n=5`に対して3回の単純な加算、線形増加
    
- **時間計算量**：`O(n)`で`O(2^n)`の代わり
    
- **空間計算量**：配列で`O(n)`、最適化で`O(1)`
    
- **予測可能な性能**：大きな入力でもスタックオーバーフローのリスクなし
    

両アプローチは時間計算量を指数的な`O(2^n)`から線形の`O(n)`に削減し、ミリ秒で解ける問題と何年も待つ問題の差を生み出します。

## 実世界での応用例

動的計画法はコンピュータサイエンスだけでなく幅広く使われています：

- **ルート最適化**：GPSシステムは動的計画法のアルゴリズムで最短経路を見つけます
    
- **テキスト処理**：スペルチェッカーやオートコンプリートは単語間の編集距離計算に動的計画法を使います
    
- **金融モデリング**：投資戦略やポートフォリオ最適化で頻繁に利用されます
    
- **資源配分**：ナップサック問題やその変種はスケジューリング、予算管理、資源管理に現れます
    

## 実践例：コインチェンジ問題

コインチェンジ問題は、動的計画法の2つの重要な原則、最適部分構造と重複部分問題を示す古典的なコーディングチャレンジです。

「目標金額を作るのに必要な最小のコイン枚数は？」という問題です。

動的計画法を使った解法の一例：

```js
function minCoins(amount, coins) {
  // Create dp array from 0 to amount, filled with Infinity
  const dp = new Array(amount + 1).fill(Infinity);

  // Base case: 0 coins needed to make amount 0
  dp[0] = 0;

  // Build up from 1 to target amount
  for (let i = 1; i <= amount; i++) {
    // Try each coin denomination
    for (let coin of coins) {
      // Only use coin if it is not larger than current amount
      if (coin <= i) {
        // Choose the minimum: current value OR solution for remaining amount + 1 coin
        dp[i] = Math.min(dp[i], dp[i - coin] + 1);
      }
    }
  }

  // Return result if possible, -1 if impossible (infinity)
  return dp[amount] === Infinity ? -1 : dp[amount];
}

/*
 Example usage:
coins = [1, 3, 4], amount = 6

dp[0] = 0
dp[1] = dp[0] + 1 = 1
dp[2] = dp[1] + 1 = 2
dp[3] = min(dp[2]+1, dp[0]+1) = min(2+1, 0+1) = 1
dp[4] = min(dp[3]+1, dp[1]+1, dp[0]+1) = min(1+1, 1+1, 0+1) = 1
dp[5] = min(dp[4]+1, dp[2]+1, dp[1]+1) = min(1+1, 2+1, 1+1) = 2
dp[6] = min(dp[5]+1, dp[3]+1, dp[2]+1) = min(2+1, 1+1, 2+1) = 2

Result: 2 coins (3 + 3)
*/
```

そして、`coins = [1, 3, 4]`、`amount = 6`に対して動的計画法のコインチェンジアルゴリズムがどのように動くか段階的に示します：

```md
Initial state:
dp = [0, ∞, ∞, ∞, ∞, ∞, ∞]
     [0, 1, 2, 3, 4, 5, 6] ← amounts

Building up the solution:

For amount = 1:
  Try coin 1: dp[1] = min(∞, dp[0] + 1) = min(∞, 0 + 1) = 1
  dp = [0, 1, ∞, ∞, ∞, ∞, ∞]

For amount = 2:
  Try coin 1: dp[2] = min(∞, dp[1] + 1) = min(∞, 1 + 1) = 2
  dp = [0, 1, 2, ∞, ∞, ∞, ∞]

For amount = 3:
  Try coin 1: dp[3] = min(∞, dp[2] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[3] = min(3, dp[0] + 1) = min(3, 0 + 1) = 1
  dp = [0, 1, 2, 1, ∞, ∞, ∞]

For amount = 4:
  Try coin 1: dp[4] = min(∞, dp[3] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[4] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  Try coin 4: dp[4] = min(2, dp[0] + 1) = min(2, 0 + 1) = 1
  dp = [0, 1, 2, 1, 1, ∞, ∞]

For amount = 5:
  Try coin 1: dp[5] = min(∞, dp[4] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[5] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  Try coin 4: dp[5] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, ∞]

For amount = 6:
  Try coin 1: dp[6] = min(∞, dp[5] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[6] = min(3, dp[3] + 1) = min(3, 1 + 1) = 2
  Try coin 4: dp[6] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, 2]

Final result: dp[6] = 2 (achieved with coins 3 + 3)
```

この解法は動的計画法の2つの原則を示しています。重複部分問題があるのは、金額6の最小コイン数を求めるには金額5、3、2の解が必要で、これらの部分問題は他の金額の計算でも現れます。最適部分構造があるのは、任意の金額の最適解はより小さな金額の最適解を含むためです。例えば金額3の最小コイン数が1なら、金額6はその解に3のコイン1枚を加えた方法の1つです。

DPなしでは、すべてのコインの組み合わせを試す必要があり、指数的に増えます。DPでは体系的に解を構築します：

- **時間計算量**：指数的ではなく`O(amount × number of coins)`
    
- **空間計算量**：`O(amount)`の`dp`配列
    
- **重複作業なし**：各部分問題は一度だけ解かれます
    
- **結果の再利用**：金額3の最小コイン数が分かれば、それを利用できる大きな金額の計算に活用
    

## 動的計画法を使うべき場合

動的計画法は以下の場合に効果的です：

- 問題が重複部分問題に分解できる
    
- 問題に最適部分構造がある
    
- 素朴な再帰解法では計算が繰り返される
    
- 時間計算量を優先し、空間計算量を犠牲にしてもよい
    

よくある動的計画法のパターンには、最小値・最大値を求める最適化問題、達成方法の数を数える問題、小さな決定に分解できる意思決定問題があります。

動的計画法は複雑な問題を部分問題の解を保存・再利用しながら扱いやすく変換します。この手法を理解すると、多くの計算問題を効率的に解けるようになります。
