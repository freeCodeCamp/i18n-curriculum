# --description--

Já se perguntou como manter seu app Express estável e pronto para produção, especialmente quando está sob alta carga ou sendo encerrado?

É aí que entram as verificações de saúde e os encerramentos suaves. Esses dois padrões ajudam a manter seu app disponível e garantem que ele seja finalizado de forma limpa, sem perder requisições ou corromper dados.

Vamos explicar.

Primeiro, o que é uma "verificação de saúde" no Express?

Uma verificação de saúde é uma rota simples que indica se seu app está ativo e funcionando.

Por exemplo:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

Ferramentas como balanceadores de carga ou orquestradores de container — pense em Kubernetes ou AWS Elastic Beanstalk — podem fazer ping nessa rota para decidir se seu app está saudável.

Se a rota retornar um código de status 200, seu serviço é considerado saudável. Lembrete — um código HTTP 200 significa que está tudo certo!

Se não responder, o orquestrador pode reiniciar o serviço automaticamente.

E o que é um "encerramento suave"?

Um encerramento suave acontece quando seu app está prestes a parar, como quando um servidor está reiniciando ou uma implantação está ocorrendo.

Em vez de cortar as conexões imediatamente, você dá tempo para o app terminar o que está fazendo.

No Express, você pode escutar o `SIGTERM`, que é um sinal do sistema enviado quando um processo deve terminar.

Aqui está um exemplo:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

Isso diz ao Express para parar de aceitar novas conexões e esperar as existentes terminarem antes de sair.

Você também deve tratar o `SIGINT`, que é o sinal enviado quando você pressiona Ctrl+C para parar seu app durante o desenvolvimento local.

Enquanto o `SIGTERM` é normalmente usado por orquestradores como Kubernetes em produção, o `SIGINT` cobre o fluxo de trabalho cotidiano do desenvolvedor. Sem tratá-lo, pressionar Ctrl+C mata o processo imediatamente, pulando qualquer limpeza que você tenha configurado.

Como ambos os sinais devem disparar o mesmo comportamento de encerramento suave, os desenvolvedores costumam escutar os dois.

Agora, vamos juntar tudo

Imagine que seu app está rodando em um container. O Kubernetes envia um `SIGTERM` para desligá-lo para uma atualização.

Com um encerramento suave básico, seu app para de aceitar novas conexões e finaliza as requisições atuais. Quando todo o trabalho termina, ele desconecta do banco de dados e sai limpo. Sem perda de dados, sem estado corrompido.

Mas há um problema — o balanceador de carga não sabe que seu app está desligando, então pode continuar enviando tráfego novo para ele. É aí que entra o padrão bônus abaixo.

Agora, tenho um bônus para você. Vamos ver "marcar o app como não saudável antes do desligamento".

Antes de desligar, você pode atualizar sua verificação de saúde para retornar um status 503. Assim, o balanceador de carga vê a resposta não saudável e para de direcionar tráfego novo para seu app durante a janela de desligamento, como demonstrado no código acima.

Lembrete: um código `503` significa "Serviço indisponível". Ele sinaliza que o servidor está temporariamente incapaz de atender requisições, que é exatamente o que acontece durante um encerramento suave.

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
