## HTTP 响应状态码

- **状态码** 是简短的数字消息，用来告诉客户请求之后发生了什么（例如，200、404 或 500）。

### 1xx — 信息性状态码

- **100 Continue** — 服务器告诉客户继续发送剩余请求。
- **101 Switching Protocols** — 服务器正在切换协议（通常见于 WebSockets）。

### 2xx — 成功状态码

- **200 OK** — 请求成功。
- **201 Created** — 新资源创建成功。
- **204 No Content** — 请求成功但没有响应正文（通常用于 `DELETE`）。

### 3xx — 重定向状态码

- **301 Moved Permanently** — 资源永久移动到新 URL。
- **302 Found** — 资源临时位于不同 URL。
- **304 Not Modified** — 客户端可以使用缓存版本；内容未改变。

### 4xx — 客户端错误

- **400 Bad Request** — 请求格式错误或无效。
- **401 Unauthorized** — 需要认证。
- **403 Forbidden** — 已认证，但无权限。
- **404 Not Found** — 请求的资源不存在。

### 5xx — 服务器错误

- **500 Internal Server Error** — 通用服务器端错误。
- **502 Bad Gateway** — 代理从另一服务器收到无效响应。
- **503 Service Unavailable** — 服务器暂时过载或宕机。

### 在 Express 中使用状态码

- `res.status(400).send("Bad Request")` — 发送带有纯文本消息的 400 状态码。
- `res.status(200).json({ message: "Success" })` — 发送带有 JSON 正文的 200 状态码。
- `res.status(500).send("Internal Server Error")` — 发送带有消息的 500 状态码。

## Express 中的错误处理

- **错误处理中间件** 有四个参数：`(err, req, res, next)`。四参数签名告诉 Express 这是一个错误处理程序。
- 将错误处理中间件**放在中间件栈的末尾**，所有路由之后。

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Express 4 异步错误处理

- Express 4 不会**自动捕获**异步错误。
- 将异步逻辑包裹在 `try/catch` 中，并调用 `next(err)` 将错误转发给错误处理程序。

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- 使用 `asyncHandler` 包装器避免在每个路由中重复 `try/catch`：

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Express 5 异步错误处理

- 在 Express 5 中，异步错误和被拒绝的 Promise 会**自动转发**给错误处理程序，无需手动 `next(err)`。

### 处理 404 错误

- 在路由之后但错误处理程序之前添加一个捕获所有的中间件来处理未匹配的路由：

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### 错误处理最佳实践

- 始终将错误处理中间件放在最后。
- 分开处理 404 错误和其他错误。
- 在生产环境中，内部记录错误详情——绝不向用户发送堆栈追踪。

## Express 中的调试和日志记录

- **内置调试**：设置 `DEBUG` 环境变量以启用 Express 的内置调试输出。
  - `DEBUG=express:* node index.js` — 启用所有 Express 调试消息。
  - `DEBUG=express:router node index.js` — 仅显示路由相关的调试输出。
  - PowerShell：`$env:DEBUG = "express:*"; node index.js`
  - CMD：`set DEBUG=express:* && node index.js`
- **`morgan`**：一个流行的中间件，用于记录 HTTP 请求（方法、URL、状态码、响应时间）。

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- **调试** 追踪 Express 如何处理请求和中间件，不会使生产日志杂乱。
- **日志记录** 记录请求过程中的情况，有助于发现错误和理解流量。

## 健康检查和优雅关闭

### 健康检查

- **健康检查路由** 报告应用是否正常运行。
- 负载均衡器和编排器（例如 Kubernetes、AWS Elastic Beanstalk）会访问此路由以判断服务是否健康。
- 返回 **200** 表示健康；无响应则触发自动重启。

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### 优雅关闭

- **优雅关闭** 允许应用完成现有请求后再停止，而不是立即断开连接。
- 监听 `SIGTERM` 信号（进程应终止时发送），并调用 `server.close()`：

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- 监听 `SIGINT` 信号（开发时按 Ctrl+C 发送），调用与 `SIGTERM` 相同的关闭逻辑。

### 关闭前标记应用为不健康

- 在关闭期间从健康检查返回 **503**，让负载均衡器停止向应用路由新流量。

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
