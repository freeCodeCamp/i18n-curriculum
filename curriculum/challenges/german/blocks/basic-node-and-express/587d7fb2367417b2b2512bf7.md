# --description--

Neben GET gibt es ein weiteres häufig genutztes HTTP-Verb, nämlich POST. POST ist die Standardmethode, um Client-Daten mit HTML-Formularen zu senden. In der REST-Konvention wird POST verwendet, um Daten zu senden, mit denen neue Einträge in der Datenbank erstellt werden (ein neuer Benutzer oder ein neuer Blogbeitrag). In diesem Projekt haben Sie keine Datenbank, aber Sie lernen trotzdem, wie man POST-Anfragen verarbeitet.

Bei dieser Art von Anfragen erscheinen die Daten nicht in der URL, sondern sind im Anfrage-Body verborgen. Der Body ist ein Teil der HTTP-Anfrage, auch Nutzlast genannt. Obwohl die Daten nicht in der URL sichtbar sind, bedeutet das nicht, dass sie privat sind. Um das zu verstehen, sehen Sie sich den Rohinhalt einer HTTP-POST-Anfrage an:

```http
POST /path/subpath HTTP/1.0
From: john@example.com
User-Agent: someBrowser/1.0
Content-Type: application/x-www-form-urlencoded
Content-Length: 20

name=John+Doe&age=25
```

Wie Sie sehen, ist der Body wie der Query-String kodiert. Dies ist das Standardformat, das von HTML-Formularen verwendet wird. Mit Ajax können Sie auch JSON verwenden, um Daten mit komplexerer Struktur zu verarbeiten. Es gibt auch eine weitere Kodierungsart: multipart/form-data. Diese wird verwendet, um Binärdateien hochzuladen. In dieser Übung verwenden Sie einen URL-kodierten Body. Um die Daten aus POST-Anfragen zu parsen, müssen Sie das `body-parser`-Paket verwenden. Dieses Paket ermöglicht es Ihnen, eine Reihe von Middleware zu nutzen, die Daten in verschiedenen Formaten dekodieren kann.

# --instructions--

`body-parser` wurde bereits installiert und befindet sich in der `package.json`-Datei Ihres Projekts. `require` Sie es am Anfang der `myApp.js`-Datei und speichern Sie es in einer Variablen namens `bodyParser`. Die Middleware zur Verarbeitung von URL-kodierten Daten wird von `bodyParser.urlencoded({extended: false})` zurückgegeben. Übergeben Sie die von der vorherigen Methodenaufruf zurückgegebene Funktion an `app.use()`. Wie üblich muss die Middleware vor allen Routen eingebunden werden, die davon abhängen.

**Hinweis:** `extended` ist eine Konfigurationsoption, die `body-parser` mitteilt, welche Parsing-Methode verwendet werden soll. Wenn `extended=false` verwendet wird, nutzt es die klassische Kodierung der `querystring`-Bibliothek. Wenn `extended=true` verwendet wird, nutzt es die `qs`-Bibliothek zum Parsen.

Bei der Verwendung von `extended=false` können Werte nur Strings oder Arrays sein. Das Objekt, das bei Verwendung von `querystring` zurückgegeben wird, erbt nicht prototypisch vom Standard-JavaScript-`Object`, was bedeutet, dass Funktionen wie `hasOwnProperty` und `toString` nicht verfügbar sind. Die erweiterte Version erlaubt mehr Datenflexibilität, wird aber von JSON übertroffen.
