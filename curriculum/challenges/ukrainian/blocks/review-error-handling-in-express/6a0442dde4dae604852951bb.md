## HTTP статуси відповіді

- **Статуси** — це короткі числові повідомлення, які повідомляють клієнту, що сталося після надсилання запиту (наприклад, 200, 404 або 500).

### 1xx — Інформаційні

- **100 Continue** — сервер каже клієнту надіслати решту запиту.
- **101 Switching Protocols** — сервер змінює протоколи (часто зустрічається з WebSockets).

### 2xx — Успішні

- **200 OK** — запит успішний.
- **201 Created** — новий ресурс успішно створено.
- **204 No Content** — запит успішний, але тіло відповіді відсутнє (часто використовується для `DELETE`).

### 3xx — Перенаправлення

- **301 Moved Permanently** — ресурс має нову постійну URL-адресу.
- **302 Found** — ресурс тимчасово знаходиться за іншою URL-адресою.
- **304 Not Modified** — клієнт може використовувати кешовану версію; нічого не змінилося.

### 4xx — Помилки клієнта

- **400 Bad Request** — запит некоректний або недійсний.
- **401 Unauthorized** — потрібна автентифікація.
- **403 Forbidden** — автентифікований, але немає дозволу.
- **404 Not Found** — запитуваний ресурс не існує.

### 5xx — Помилки сервера

- **500 Internal Server Error** — загальна помилка на стороні сервера.
- **502 Bad Gateway** — проксі отримав недійсну відповідь від іншого сервера.
- **503 Service Unavailable** — сервер тимчасово перевантажений або недоступний.

### Використання статусів у Express

- `res.status(400).send("Bad Request")` — надсилає 400 з текстовим повідомленням.
- `res.status(200).json({ message: "Success" })` — надсилає 200 з JSON у тілі.
- `res.status(500).send("Internal Server Error")` — надсилає 500 з повідомленням.

## Обробка помилок у Express

- **Підпрограмне забезпечення для обробки помилок** має чотири параметри: `(err, req, res, next)`. Саме чотирипараметровий підпис сигналізує Express, що це обробник помилок.
- Розміщуйте підпрограмне забезпечення для обробки помилок **в кінці** стеку підпрограм, після всіх маршрутів.

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Асинхронна обробка помилок у Express 4

- Асинхронні помилки **автоматично не ловляться** в Express 4.
- Обгорніть асинхронну логіку в `try/catch` і викликайте `next(err)`, щоб передати помилку обробнику помилок.

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- Обгортка `asyncHandler` допомагає уникнути повторення `try/catch` у кожному маршруті:

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Асинхронна обробка помилок у Express 5

- В Express 5 асинхронні помилки та відхилені проміси **автоматично передаються** обробнику помилок — ручний `next(err)` не потрібен.

### Обробка помилок 404

- Додайте підпрограмне забезпечення для перехоплення всіх запитів **після маршрутів, але перед обробником помилок**, щоб обробляти невідповідні маршрути:

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Найкращі практики обробки помилок

- Завжди розміщуйте підпрограмне забезпечення для обробки помилок останнім.
- Обробляйте 404 окремо від інших помилок.
- У продакшені ведіть внутрішній журнал помилок — ніколи не надсилайте користувачам трасування стека.

## Налагодження та логування в Express

- **Вбудоване налагодження**: встановіть змінну середовища `DEBUG`, щоб увімкнути вбудований в Express вивід налагодження.
  - `DEBUG=express:* node index.js` — увімкнути всі повідомлення налагодження Express.
  - `DEBUG=express:router node index.js` — показувати лише налагодження маршрутизатора.
  - PowerShell: `$env:DEBUG = "express:*"; node index.js`
  - CMD: `set DEBUG=express:* && node index.js`
- **`morgan`**: популярне підпрограмне забезпечення для логування HTTP-запитів (метод, URL, статус, час відповіді).

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- **Налагодження** відслідковує, як Express обробляє запити та підпрограми, не засмічуючи журнали продакшену.
- **Логування** фіксує, що відбувається під час запитів — корисно для виявлення помилок і розуміння трафіку.

## Перевірки стану та коректне завершення роботи

### Перевірки стану

- **Маршрут перевірки стану** повідомляє, чи працює застосунок правильно.
- Балансувальники навантаження та оркестратори (наприклад, Kubernetes, AWS Elastic Beanstalk) опитують цей маршрут, щоб визначити, чи сервіс здоровий.
- Відповідь **200** означає, що сервіс здоровий; відсутність відповіді спричиняє автоматичний перезапуск.

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### Коректне завершення роботи

- **Коректне завершення роботи** дозволяє застосунку завершити поточні запити перед зупинкою, замість негайного розриву з’єднань.
- Слухайте сигнал `SIGTERM` (надсилається, коли процес потрібно завершити) і викликайте `server.close()`:

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- Слухайте сигнал `SIGINT` (надсилається, коли розробник натискає Ctrl+C під час локальної розробки) і виконуйте ту саму логіку завершення, що й `SIGTERM`.

### Позначення застосунку як нездорового перед завершенням

- Під час завершення роботи повертайте **503** з маршруту перевірки стану, щоб балансувальник припинив направляти новий трафік до застосунку.

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
