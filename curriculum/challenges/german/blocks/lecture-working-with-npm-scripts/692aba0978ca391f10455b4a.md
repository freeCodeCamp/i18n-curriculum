# --description--

Lernen wir die npm-Registry kennen, in der Sie Ihre JavaScript-Pakete mit der Welt teilen können.

Die **npm-Registry** ist eine Datenbank mit öffentlichen und privaten Paketen.

Entwickler und Organisationen können Pakete in dieser Registry veröffentlichen, um ihre Werkzeuge mit ihren Communities und Teammitgliedern zu teilen und in vielen Fällen mit allen anderen Node.js-Entwicklern.

Pakete können entweder öffentlich oder privat sein. Öffentliche Pakete können von jedem aufgerufen werden. Private Pakete können nur von bestimmten Benutzern oder Mitgliedern einer Organisation aufgerufen werden.

In der npm-Registry können Sie mit nur wenigen Klicks das, was Sie gebaut haben, mit der Welt teilen. Klingt großartig, oder?

Sehen wir uns an, wie Sie ein Paket Schritt für Schritt veröffentlichen können.

## Schritt 1: Erstellen Sie ein Konto

Der erste Schritt ist, ein kostenloses npm-Konto zu erstellen. Um Ihr Konto zu erstellen:

* Gehen Sie zu [https://www.npmjs.com/signup](https://www.npmjs.com/signup).
    
* Füllen Sie das Formular aus. Sie müssen Ihren Benutzernamen, Ihre E-Mail-Adresse und Ihr Passwort eingeben.
    

Bitte beachten Sie, dass laut der Anmeldeseite „Ihre E-Mail-Adresse zu den Metadaten der Pakete hinzugefügt wird, die Sie veröffentlichen, sodass sie öffentlich sichtbar sein kann.“

## Schritt 2: **Melden Sie sich an**

Sobald Sie ein npm-Konto haben, müssen Sie sich im Terminal mit diesem Befehl anmelden:

```bash
npm login
```

Dann müssen Sie sich mit Ihren Kontoinformationen authentifizieren.

## **Schritt 3: Erstellen Sie eine** `package.json`-Datei

Bevor Sie Ihr Paket veröffentlichen, müssen Sie eine `package.json`-Datei mit diesem Befehl erstellen:

```bash
npm init
```

(Falls Ihr Projekt bereits eine hat, können Sie direkt zu **Schritt 4** gehen).

Sie müssen diese Informationen für Ihre `package.json`-Datei eingeben:

* **name**: Dies ist normalerweise der Name Ihres Projekts/Verzeichnisses. Wenn Sie planen, Ihr Paket öffentlich zu machen, muss dieser Name eindeutig sein, es sei denn, Sie verwenden Scopes (darauf gehen wir gleich ein).
    
* **version**: Sie sollten mit `"1.0.0"` beginnen, dem Standardwert. Dies stellt die Anfangsversion Ihres Pakets dar, und Sie können sie aktualisieren, wenn Sie Änderungen am Paket vornehmen.
    
* **main**: Der Einstiegspunkt Ihres Projekts. Standardmäßig ist dies `index.js`.
    
* **author**: Sie können Ihren Namen oder bevorzugten Benutzernamen angeben.
    
* **license**: Wählen Sie eine bestimmte Lizenz. Beispiele sind MIT und ISC.
    
* **description**: Eine kurze Zusammenfassung Ihres Pakets.
    

Wenn Sie den Namen Ihres Pakets wählen, stellen Sie sicher, dass er in der npm-Registry noch nicht vergeben ist. Dazu können Sie die folgende URL aufrufen und `<your-package-name>` durch den Namen Ihres Pakets ersetzen:

```bash
https://www.npmjs.com/package/<your-package-name>
```

Wenn es unter dieser URL kein Paket gibt, ist der Paketname verfügbar und Sie können ihn verwenden.

Falls der Name bereits vergeben ist und Sie ihn unbedingt behalten möchten, können Sie das sogenannte **Scope** verwenden, um es zu veröffentlichen.

### **Scopes**

Wenn Sie ein Benutzer- oder Organisationskonto in der npm-Registry erstellen, erhalten Sie ein Scope mit Ihrem Benutzernamen oder dem Namen Ihrer Organisation.

Scopes helfen, Namenskonflikte zu vermeiden, weil Sie Ihr Paket unter Ihrem Scope veröffentlichen können, selbst wenn bereits ein anderes Paket mit demselben Namen existiert.

Der Name eines Scoped-Pakets beginnt mit dem Scope-Namen. Zum Beispiel finden Sie in `package.json` dieses Schlüssel-Wert-Paar:

```bash
"name": "@<scope-name>/<your-package-name>"
```

Sie können diese Änderung direkt in der Datei vornehmen, wenn nötig, oder Sie können die `package.json`-Datei von Anfang an mit dem Scope erstellen, indem Sie `--scope=@<scope-name>` zum `npm init`-Befehl hinzufügen.

```bash
npm init --scope=@<scope-name>
```

Der Scope-Name kann Ihr Benutzername sein, wenn Sie ein persönliches Konto verwenden, oder der Name Ihrer Organisation, wenn Sie ein Organisationskonto verwenden.

## Schritt 4: Erstellen Sie README.md

Das README Ihres Pakets ist eine Markdown-Datei, in der Sie beschreiben, was Ihr Paket macht und wie es verwendet werden soll.

Einige Beispiele für wichtige Inhalte, die Sie aufnehmen möchten, sind:

* Sein Zweck.
    
* Wie man es installiert.
    
* Wichtige Abhängigkeiten und Hinweise.
    
* Wie man anfängt.
    
* Beispiel-Code-Ausschnitte.
    
* Richtlinien für Beiträge (falls Sie Beiträge akzeptieren).
    
* Verknüpfung zu einer Changelog-Datei, in der Sie die Änderungen in verschiedenen Versionen verfolgen.
    

Die Struktur des README ist flexibel. Sie können es an Ihre Bedürfnisse anpassen.

## **Schritt 5:** `.npmignore`

Die `.npmignore`-Datei teilt der npm-Registry mit, welche Dateien beim Veröffentlichen Ihres Pakets einbezogen und welche ignoriert werden sollen. Es ist sehr wichtig, zu verhindern, dass sensible Informationen wie private Schlüssel, Passwörter und persönliche Daten geteilt werden.

In dieser Datei können Sie Muster schreiben, um anzugeben, welche Dateien, Verzeichnisse und Dateitypen ignoriert werden. Sie folgt denselben Mustererkennungsregeln wie die `.gitignore`-Dateien.

Wenn Sie mit Git gearbeitet haben, denken Sie vielleicht, dass diese Datei sehr ähnlich wie `.gitignore` klingt, oder? Das liegt daran, dass sie es ist. Laut der npm-Dokumentation gilt: „Wenn keine `.npmignore`-Datei vorhanden ist, aber eine `.gitignore`-Datei existiert, ignoriert npm die durch die `.gitignore`-Datei angegebenen Dateien.“

Sie sollten auch wissen, dass npm in allen Unterverzeichnissen Ihres Pakets nach `.npmignore`- und `.gitignore`-Dateien sucht, nicht nur im Stammverzeichnis.

Standardmäßig werden diese Pfade automatisch ignoriert, sodass Sie sie nicht in `.npmignore` hinzufügen müssen:

* `.*.swp`
    
* `._*`
    
* `.DS_Store`
    
* `.git`
    
* `.gitignore`
    
* `.hg`
    
* `.npmignore`
    
* `.npmrc`
    
* `.lock-wscript`
    
* `.svn`
    
* `.wafpickle-*`
    
* `config.gypi`
    
* `CVS`
    
* `npm-debug.log`
    

Das `node_modules`-Verzeichnis wird ebenfalls ignoriert, außer gebündelte Abhängigkeiten.

## **Schritt 6: Veröffentlichen Sie Ihr Paket**

Bevor Sie Ihr Paket in der npm-Registry veröffentlichen, sollten Sie es testen, indem Sie diesen Befehl ausführen:

```bash
npm install <full-path-to-package-directory>
```

Sobald Sie bereit sind, es zu veröffentlichen, hängt der Befehl vom Typ des Pakets ab: Scope oder unscoped.

Wenn Ihr Paket unscoped ist, führen Sie diesen Befehl aus:

```bash
npm publish
```

Wenn Ihr Paket scoped ist, wird es standardmäßig als privates Paket veröffentlicht (wenn diese Funktion in Ihrem Konto verfügbar ist). Wenn Sie es öffentlich machen möchten, müssen Sie `--access public` zum Befehl hinzufügen.

```bash
npm publish --access public
```

Um Ihr öffentliches Paket zu sehen, gehen Sie zu dieser URL:

```bash
https://npmjs.com/package/<your-package-name>
```

Sie sollten unter dem Paketnamen eine Angabe zur Sichtbarkeit Ihres Pakets (öffentlich oder privat) sehen.

## **Schritt 7 (Optional): Installieren Sie Ihr Paket**

Jetzt, da alles eingerichtet ist, können Sie Ihr Paket in einem Node.js-Projekt mit diesem Befehl installieren:

```bash
npm install <your-package-name>
```

## Aktualisieren Ihres Pakets

Wenn Sie nach der Veröffentlichung Änderungen an Ihrem Paket vornehmen müssen, können Sie die Paketversion in `package.json` aktualisieren.

Dies ist die Standardkonvention für die semantische Versionierung:

* **Erstveröffentlichung:** `1.0.0`.
    
* **Patch-Version:** Erhöhen Sie die dritte Ziffer. Zum Beispiel von `1.0.0` auf `1.0.1`.
    
* **Minor-Version:** Erhöhen Sie die mittlere Ziffer und setzen Sie die letzte Ziffer auf null zurück. Zum Beispiel von `1.0.0` auf `1.1.0`.
    
* **Major-Version:** Erhöhen Sie die erste Ziffer und setzen Sie die mittlere und letzte Ziffer auf null zurück. Zum Beispiel von `1.1.0` auf `2.0.0`.
    
Großartige Arbeit. Jetzt wissen Sie, wie Sie Ihr Paket in der npm-Registry veröffentlichen und wie Sie es aktualisieren, um Bugs zu beheben und neue Funktionen hinzuzufügen.
