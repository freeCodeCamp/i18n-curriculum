# --description--

과거에는 대규모든 소규모든 대부분의 애플리케이션이 모놀리식으로 구축되었습니다. 모놀리식은 웹 애플리케이션의 모든 것을 포함하는 하나의 거대한 코드베이스입니다.

이 모놀리식 시스템은 UI, 데이터베이스, API가 모두 하나의 코드베이스에 존재한다는 뜻입니다.

아마존 같은 전자상거래 매장을 상상해 보세요. 상품 페이지, 결제 시스템, 결제 처리, 인증과 권한 부여, 그리고 이를 지원하는 API와 데이터베이스가 모두 같은 코드베이스에 있습니다.

모놀리식 방식은 작은 애플리케이션이나 초기 스타트업에 적합합니다. 하지만 애플리케이션이 성장하고 사용자 수요가 증가하면 모놀리식은 유지보수가 어려워지고, 업데이트가 느려지며, 확장하기도 힘들어집니다. 한 영역의 작은 변경도 전체 시스템을 다시 배포해야 할 수 있고, 한 기능을 확장하려면 전체 애플리케이션을 확장해야 하는 경우가 많습니다.

이러한 문제를 해결하기 위해 마이크로서비스라는 아키텍처 방식이 등장했습니다. 마이크로서비스는 모놀리식 시스템의 한계를 극복하기 위해 설계된 구조입니다.

마이크로서비스 아키텍처에서는 큰 애플리케이션을 더 작고 독립적인 서비스들로 나눕니다. 각 서비스는 상품, 주문, 인증, 결제 등 특정 비즈니스 기능을 담당합니다. 이 서비스들은 각각 자체 코드베이스를 가질 수 있고, 심지어 자체 데이터베이스를 관리할 수도 있으며, 완전히 다른 팀이 각각 작업할 수도 있습니다.

서비스들은 API나 메시지 기반 시스템을 통해 서로 통신하며 하나의 애플리케이션처럼 작동합니다.

이 방식 덕분에 웹과 모바일 애플리케이션 모두 유지보수, 업데이트, 확장이 더 쉬워집니다. 예를 들어, 전자상거래 플랫폼은 상품, 장바구니, 사용자, 주문, 결제 등 별도의 서비스를 가질 수 있습니다. 각 서비스는 다른 서비스에 영향을 주지 않고 독립적으로 구축, 배포, 확장할 수 있습니다.

마이크로서비스는 더 빠른 개발, 더 나은 장애 격리, 더 큰 유연성 등 여러 장점을 제공합니다. 서로 다른 서비스가 서로 다른 프로그래밍 언어나 프레임워크를 사용할 수도 있습니다. 예를 들어, 검색 서비스는 Java를, 상품 서비스는 Node.js를, 인증 서비스는 .NET을 사용할 수 있습니다.

하지만 마이크로서비스는 새로운 도전 과제도 가져옵니다. 문제 해결이 더 복잡해지는데, 문제 원인이 여러 서비스에 걸쳐 있을 수 있기 때문입니다. 배포와 인프라 관리도 더 복잡해지고, 서비스 간 빈번한 통신으로 네트워크 부하가 증가합니다.
