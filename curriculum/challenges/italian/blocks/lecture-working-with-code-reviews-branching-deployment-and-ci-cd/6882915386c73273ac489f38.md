# --description--

Abbiamo parlato un po’ di branch nelle lezioni precedenti. Hai imparato come creare branch, perché non dovresti fare commit direttamente su `main` e cose simili. Ma quando e perché dovresti effettivamente creare dei branch?

Prima di tutto, parliamo del motivo per cui non dovresti semplicemente pushare una serie di commit su `main`. Il tuo branch predefinito dovrebbe essere affidabilmente stabile — cioè, in qualsiasi momento dovrei poter scaricare il tuo attuale branch `main` e buildare, eseguire e testare la tua applicazione. Se stai pushando funzionalità parzialmente completate o commit WIP, questo potrebbe non essere sempre vero.

Inoltre, è molto comune lavorare su più funzionalità contemporaneamente. Se stai pushando lavoro parziale su `main` per più funzionalità allo stesso tempo, diventa molto più difficile eseguire il debug di un problema potenziale perché non hai modifiche isolate. Inoltre, cambiare contesto (cioè passare da un’attività all’altra) non è mai così pulito.

Quindi, quando dovresti creare un branch? Una regola generale è creare un branch per ogni unità di lavoro isolata. Cioè, se stai lavorando su una nuova funzionalità OAuth, quel lavoro dovrebbe stare su un branch dedicato. Se poi devi correggere un bug nell’autenticazione tramite password, quel lavoro dovrebbe stare su un suo branch dedicato. Devi ridisegnare la pagina del profilo utente? Quello è un altro branch.

Nota che dovresti sempre creare i tuoi branch a partire da `main`. Quindi, se stai lavorando su un branch per la tua funzionalità OAuth, dovresti fare checkout di `main` prima di creare un nuovo branch per la correzione del bug sull’autenticazione tramite password. Altrimenti, se crei un branch per la correzione del bug direttamente dal tuo branch OAuth, finirai con i commit del lavoro OAuth e la storia sarà confusa.

Ma come dovresti chiamare i tuoi branch? Beh, dipende tutto da te (o dal progetto a cui contribuisci), ma ecco alcune convenzioni comuni:

`scope/description` — dove `scope` è qualcosa come `feat` o `fix`, e `description` è una spiegazione molto breve delle modifiche del branch. Per esempio: `feat/oauth-support`.

`scope/issue/description` — simile alla convenzione precedente, ma includendo il numero associato al problema correlato. Per esempio: `fix/25/password-auth`.

`username/scope/description` — per progetti in cui lavorate tutti su branch nel repository originale invece di fare fork (questo è comune con repository interni privati), può essere utile includere il tuo nome utente all’inizio del branch. Per esempio: `naomi-lgbt/feat/profile-redesign`.

Vediamo alcune altre note sulle strategie di branching.

Per lavori su funzionalità a lungo termine, come una funzionalità beta che richiede molte ore di sviluppo, potresti voler usare un feature flag per abilitare selettivamente la logica invece di un branch. Più a lungo un branch rimane non unito, più è probabile che si verifichino conflitti.

Potresti però voler mantenere branch dedicati per ambienti di distribuzione o release. Per esempio, il repository freeCodeCamp usa `prod-staging` e `prod-current` per attivare le distribuzioni rispettivamente ai server di staging e produzione. Questo permette al team di iterare più velocemente, unendo quando le cose sono pronte e approvate senza dover attivare una distribuzione per ogni merge. Invece, le distribuzioni vengono create unendo l’ultimo `main` in quei branch di produzione.

Potresti anche voler mantenere “branch di release”, cioè branch che rappresentano una versione principale del tuo prodotto. Questo ti permette di avere uno stato pulito per ogni release principale, così se devi applicare una patch di sicurezza critica a una versione precedente puoi farlo senza includere le modifiche della nuova versione.

E infine, dovresti evitare di unire i tuoi branch in `main` usando la CLI locale. Invece, crea una pull request con le tue modifiche e unisci tramite quel flusso. Questo ti dà la possibilità di assicurarti che tutti i test passino nel tuo CI (che imparerai in una lezione futura) e di ricontrollare la vista diff per assicurarti di aver incluso solo le modifiche che vuoi unire.

Con questi consigli, puoi iniziare a creare branch per arrivare al successo!
