## 手動テストと自動テスト

- **手動テスト**: 手動テストでは、テスターがアプリケーションの各部分を手動で操作し、さまざまな機能が正しく動作するかを確認します。テスト中にバグが見つかった場合、テスターはそれらのバグをソフトウェアチームに報告し、修正してもらいます。
- **自動テスト**: 自動テストでは、アプリケーションが期待通りに動作するかをチェックする別のプログラムを書いてテストを自動化できます。

## ユニットテスト

- **ユニットテスト**: ユニットテストでは、各関数をテストしてすべてが期待通りに動作しているかを確認します。ユニットテストはコードの期待される動作を表すため、アプリケーションのドキュメントの一種としても役立ちます。
- **単一責任の原則**: 単一責任の原則は、各関数を小さくし、一つの責任だけを持つようにすることを推奨します。
- **一般的なJavaScriptテストフレームワーク**: よく使われるテストフレームワークにはJest、Mocha、Vitestがあります。Jestはユニットテストに人気のあるテストフレームワークです。

以下はJestを使ったユニットテストの例です。

まず、新しくフォーマットされた文字列を返す関数を作成できます。

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

別の`getFormattedWord.test.js`ファイルで、その関数が期待通りに動作しているかを検証するテストを書けます。`getFormattedWord.test.js`ファイルは次のようになります。

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **`expect`関数**: `expect`関数は値をテストするために使われます。
- **マッチャー**: マッチャーは値が期待通りに動作しているかをチェックする関数です。上の例ではマッチャーは`toBe()`です。Jestにはさまざまなマッチャーがあります。

Jestを使うには、まず`jest`パッケージを`npm i jest`でインストールする必要があります。また、`package.json`ファイルに次のようなスクリプトを追加する必要があります。

```js
"scripts": {
  "test": "jest"
},
```

その後、`npm run test`コマンドを実行してテストを実行できます。

## ソフトウェア開発ライフサイクル

- **ソフトウェア開発ライフサイクルの各段階**:
  - **計画段階**: 開発チームがステークホルダーから提案された作業の要件を収集します。
  - **設計段階**: ソフトウェアチームが要件を分解し、最適な解決策を決定します。
  - **実装段階**: ソフトウェアチームが要件を管理可能なタスクに分解し、実装します。
  - **テスト段階**: 新しい作業に対して手動および自動テストを行います。場合によっては、開発全体を通じてアプリケーションをテストし、問題を早期に発見して修正します。
  - **デプロイ段階**: 新しい変更をビルドまたはテスト環境にデプロイします。
  - **メンテナンス段階**: 本番アプリケーションで発生した問題を修正します。

- **ソフトウェア開発ライフサイクルの異なるモデル**:
  - **ウォーターフォールモデル**: 各段階が完了してから次の段階に進むモデルです。
  - **アジャイルモデル**: 作業をスプリントに分けて反復的に開発するモデルです。

## BDDとTDD

- **TDD**: テスト駆動開発は、まずテストを書いてから機能を構築する手法です。機能を作る前にテストを書くことで、開発中にリアルタイムのフィードバックが得られます。
- **BDD**: 振る舞い駆動開発は、一連のテストをビジネス目標に合わせるアプローチです。BDDのテストシナリオは技術者と非技術者の両方が理解できる言語で書かれるべきです。Gherkinはそのような構文の例です。
- **BDDテストフレームワーク**: BDDテストフレームワークの例にはCucumber、JBehave、SpecFlowがあります。

## ユニットテストにおけるアサーション

- **アサーション**: コードが期待通りに動作しているかをテストするために使われます。
- **アサーションライブラリ**: Chaiはよく使われるアサーションライブラリです。他の一般的なJavaScriptアサーションライブラリには`should.js`や`expect.js`があります。

以下はChaiを使ったアサーションの例で、`addThreeAndFour`関数の戻り値が数値の7と等しいかをチェックしています。

```js
assert.equal(addThreeAndFour(), 7);
```

- **ベストプラクティス**: どのアサーションライブラリを使う場合でも、どのテストが失敗しているか、なぜ失敗しているかがわかるように明確なアサートメッセージと失敗メッセージを書くことをお勧めします。

## モッキング、フェイキング、スタビング

- **モッキング**: モッキングは実際のデータを偽のデータに置き換え、実際のコンポーネントの動作をシミュレートするプロセスです。例えば、テスト中にAPIレスポンスをモックして、継続的にAPIコールを行わずにデータを取得できます。
- **スタビング**: スタブはアプリケーションの期待される動作に対して事前定義されたレスポンスやダミーデータを返すオブジェクトです。例えば、テストで実際のデータベース接続に依存せずにデータベース接続の動作をスタブできます。
- **フェイキング**: フェイクは複雑さや副作用のない実際のコンポーネントの簡略版です。例えば、実際のデータベースとやり取りする代わりにメモリ内にデータを保存してデータベースをフェイクできます。これにより、メモリ内でデータベース操作を模倣でき、実際のデータベースよりも高速になります。

## 機能テスト

- **機能テスト**: 機能テストはアプリケーションの機能や機能が期待通りに動作するかをチェックします。機能テストの目的は、複数のシナリオに対してシステム全体をテストすることです。
- **非機能テスト**: 非機能テストはパフォーマンスや信頼性などに焦点を当てます。
- **スモークテスト**: スモークテストは、より広範なテストを始める前にシステムの基本的または重要な問題を予備的にチェックすることです。

## エンドツーエンドテスト

- **エンドツーエンドテスト**: エンドツーエンドテスト（E2E）はユーザーの視点から実際のシナリオをテストします。E2Eテストはアプリケーションが正しく動作し、ユーザーにとって予測可能であることを保証します。ただし、セットアップ、設計、メンテナンスに時間がかかります。
- **エンドツーエンドテストフレームワーク**: PlaywrightはMicrosoftが開発した人気のあるエンドツーエンドテストフレームワークです。他の例にはCypress、Selenium、Puppeteerがあります。

以下はfreeCodeCampのコードベースでPlaywrightを使ったE2Eテストの例です。`beforeEach`フックは各テストの前に実行されます。テストは、寄付者がメニューバーにサポーターリンクを持ち、アバターの周りに特別なスタイライズされたボーダーがあることをチェックします。

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## ユーザビリティテスト

- **ユーザビリティテスト**: ユーザビリティテストは実際のユーザーがアプリケーションとやり取りし、デザイン、ユーザー体験、機能の問題を発見するためのテストです。ユーザビリティテストはユーザーにとってのアプリケーションの直感的な使いやすさに焦点を当てます。
- **ユーザビリティテストの4つの一般的なタイプ**:
  - **探索的**: 探索的ユーザビリティテストは、ユーザーがアプリケーションのさまざまな機能を操作して動作を理解することを含みます。
  - **比較的**: 比較テストは、自分のアプリケーションのユーザー体験を市場の類似アプリケーションと比較します。
  - **評価**: 評価テストは、アプリケーションの使いやすさを調査します。
  - **検証**: 検証テストは、ユーザーがアプリケーションを効果的に使えなくする重大な問題を特定します。
- **ユーザビリティテストツール**: ユーザビリティテストに使われるツールの例にはLoop11、Maze、Userbrain、UserTesting、UXTweakがあります。

## 互換性テスト

- **互換性テスト**: 互換性テストの目的は、アプリケーションが異なるコンピューティング環境で動作することを保証することです。
- **互換性テストの異なるタイプ**:
  - **後方互換性**: 後方互換性は、ソフトウェアが以前のバージョンと互換性があることを指します。
  - **前方互換性**: 前方互換性は、ソフトウェアやシステムが将来のバージョンでも動作することを指します。
  - **ハードウェア互換性**: ハードウェア互換性は、ソフトウェアが異なるハードウェア構成で正常に動作する能力です。
  - **オペレーティングシステム互換性**: オペレーティングシステム互換性は、macOS、Windows、UbuntuやFedoraなどのLinuxディストリビューションのような異なるOSでソフトウェアが動作する能力です。
  - **ネットワーク互換性**: ネットワーク互換性は、異なるネットワーク速度、プロトコル、セキュリティ設定などのネットワーク条件でソフトウェアが動作できることを意味します。
  - **ブラウザ互換性**: ブラウザ互換性は、Google Chrome、Safari、Firefoxなどの異なるブラウザでウェブアプリケーションが一貫して動作することを意味します。
  - **モバイル互換性**: AndroidやiOSのさまざまなデバイス（スマートフォンやタブレット）でソフトウェアアプリケーションが動作することを確認することが重要です。

## パフォーマンステスト

- **パフォーマンステスト**: パフォーマンステストでは、異なるワークロード下でアプリケーションの速度、応答性、スケーラビリティ、安定性をテストします。目的はあらゆる種類のパフォーマンスのボトルネックを解消することです。
- **パフォーマンステストの異なるタイプ**:
  - **ロードテスト**: ロードテストは、通常およびピーク時の負荷時にシステムがどのように動作するかを判断します。
  - **ストレステスト**: ストレステストは、極端な負荷でアプリケーションをテストし、システムが高負荷にどのように対応するかを確認します。
  - **ソークテスト（耐久テスト）**: ソークテストまたは耐久テストは、長時間にわたり高負荷でシステムをテストするロードテストの一種です。
  - **スパイクテスト**: スパイクテストは負荷を劇的に増減させ、システムの反応を分析します。
  - **ブレークポイントテスト（容量テスト）**: ブレークポイントテストまたは容量テストは、負荷を徐々に増やしていき、システムが故障または劣化し始めるポイントを見つけます。

## セキュリティテスト

- **セキュリティテスト**: セキュリティテストは脆弱性や弱点を特定するのに役立ちます。
- **セキュリティの原則**:
  - **機密性**: 機密性は、意図された受信者以外に機密情報が漏れないように保護します。
  - **完全性**: 完全性は、悪意のあるユーザーがユーザー情報を改ざんするのを防ぎます。
  - **認証**: 認証は、ユーザーの身元を確認し、そのユーザーがシステムを使用する権限があることを保証します。
  - **認可**: 認可は、認証されたユーザーがどの操作を実行できるか、またはシステムのどの部分にアクセスできるかを決定するプロセスです。
  - **可用性**: 可用性は、情報やサービスが必要なときに認可されたユーザーに利用可能であることを保証します。
  - **否認防止**: 否認防止は、送信者と受信者の両方が配信の証明と送信者の身元の検証を持つことを保証します。送信者が情報を送ったことを否定するのを防ぎます。
- **一般的なセキュリティ脅威**:
  - **クロスサイトスクリプティング（XSS）**: XSS攻撃は、攻撃者が悪意のあるスクリプトをウェブページに注入し、被害者のブラウザのコンテキストで実行する場合に発生します。
  - **SQLインジェクション**: SQLインジェクションは、悪意のあるユーザーがデータベースに悪意のあるコードを注入することを可能にします。
  - **サービス拒否（DoS）攻撃**: DoS攻撃は、悪意のあるユーザーが大量のリクエストやトラフィックでウェブサイトを溢れさせ、サーバーを遅くしたりクラッシュさせたりして、サイトをユーザーが利用できなくする攻撃です。
- **セキュリティテストツールのカテゴリ**:
  - **静的アプリケーションセキュリティテスト**: これらのツールはアプリケーションのソースコードを評価し、セキュリティの脆弱性を特定します。
  - **動的アプリケーションセキュリティテスト**: これらのツールはアプリケーションのフロントエンドと連携して潜在的なセキュリティの弱点を発見します。DASTツールはソースコードにアクセスしません。
- **侵入テスト（ペンテスト）**: 侵入テストは、アプリケーションに対して模擬的なサイバー攻撃を行い、システムの脆弱性を特定するセキュリティテストの一種です。

## A/Bテスト

- **A/Bテスト**: A/Bテストは、ページやアプリケーションの2つのバージョンを比較し、どちらのバージョンがより良いパフォーマンスを示すかを調べる手法です。バケットテストやスプリットテストとも呼ばれます。A/Bテストにより、よりデータ駆動型の意思決定が可能になり、ユーザー体験を継続的に改善できます。
- **A/Bテスト用ツール**: A/Bテストに使われるツールの例にはGrowthBookやLaunchDarklyがあります。

## アルファテストとベータテスト

初期の開発とソフトウェアテストが完了したら、テスターや実際のユーザーによるテストが重要です。ここでアルファテストとベータテストが登場します。

- **アルファテスト**: アルファテストは、選ばれたテスターグループがアプリケーションを操作し、市場にリリースする前にバグがないかを確認するテストです。アルファテストは受け入れテストの一部であり、ホワイトボックステストとブラックボックステストの両方の手法を利用します。
- **ベータテスト**: ベータテストは、実際のユーザーにアプリケーションを提供し、ユーザーが操作してフィードバックを提供するテストです。ベータテストもユーザー受け入れテストの一形態です。
- **受け入れテスト**: 受け入れテストは、ソフトウェアアプリケーションがビジネス要件やユーザーのニーズを満たしていることをリリース前に確認します。
- **ブラックボックステスト**: ブラックボックステストは、アプリケーションの期待される動作にのみ焦点を当てます。
- **ホワイトボックステスト**: ホワイトボックステストは、テスターが内部コンポーネントを知り、それらに対してテストを行います。

## リグレッションテスト

- **リグレッション**: リグレッションは、新しい変更によって既存の機能が意図せず壊れる状況を指します。
- **リグレッションテスト**: リグレッションテストはリグレッション問題を検出するのに役立ちます。リグレッションテストでは、アプリケーションの一部に対して機能テストを再実行し、すべてが期待通りに動作していることを確認します。
- **リグレッションテスト用ツール**: リグレッションテストに使えるツールにはPuppeteer、Playwright、Selenium、Cypressがあります。
- **リグレッションテストの手法**:
  - **ユニットリグレッションテスト**: 主要な変更や修正が行われるたびにテストすべき項目のリストを持つ手法です。
  - **部分的リグレッションテスト**: 新しい変更が特定の機能を壊していないかを確認するターゲットを絞った手法です。
  - **完全リグレッションテスト**: コードベースのすべての機能に対してテストを実行します。最も時間がかかり詳細な方法です。
- **再テスト**: 再テストは既知の問題をチェックし、それらが解決されていることを確認するために使われます。これに対し、リグレッションテストはコードベースの最近の変更によって発生した未知の問題を探します。
