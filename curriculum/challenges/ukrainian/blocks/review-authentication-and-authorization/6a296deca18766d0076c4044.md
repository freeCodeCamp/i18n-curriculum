## Автентифікація проти авторизації

**Автентифікація** — це процес перевірки особи користувача. Вона відповідає на питання *«Чи є ця людина тим, за кого себе видає?»* Коли користувач входить у систему, сервер порівнює його облікові дані з записами, збереженими під час реєстрації.

Розробники бекенду використовують три основні категорії факторів автентифікації:

- **Те, що ви знаєте:** пароль, PIN-код або відповідь на контрольне питання.
- **Те, що у вас є:** SMS-код на смартфоні або тимчасовий токен з додатка-аутентифікатора.
- **Те, ким ви є:** біометричні дані, такі як відбиток пальця або скан обличчя.

**Авторизація** починається після автентифікації і визначає, що дозволено робити автентифікованому користувачу. Вона відповідає на питання *«Які права має ця особа?»* Загальні ролі включають:

- **Стандартний користувач:** може редагувати свій профіль, але не може переглядати приватні дані інших користувачів.
- **Редактор:** може писати та публікувати вміст, але не може видаляти кодову базу.
- **Адміністратор:** має повний доступ для зміни баз даних, видалення користувачів і зміни системних налаштувань.

**Автентифікація завжди має відбуватися першою, потім авторизація.** Ви не можете вирішити, які права надати, поки не знаєте, хто користувач.

## JSON Web Tokens (JWT)

JWT — це цифровий ключ, який сервер видає користувачу після успішного входу. Користувач пред’являє його при кожному наступному запиті, щоб підтвердити свою особу без повторного введення пароля.

Чотирикроковий потік JWT:

1. **Вхід:** користувач надсилає облікові дані на сервер.
2. **Видача:** сервер перевіряє облікові дані, генерує JWT і відправляє його назад.
3. **Збереження:** клієнт зберігає JWT, зазвичай у cookie браузера або локальному сховищі.
4. **Запит:** для кожного наступного запиту до захищеного маршруту клієнт надсилає JWT, а сервер його перевіряє.

JWT складається з трьох частин, розділених двома крапками:

- **Заголовок:** закодований JSON, що оголошує тип токена (JWT) і алгоритм хешування, наприклад HMAC SHA256.
- **Корисне навантаження:** закодований JSON із твердженнями про користувача, такими як ідентифікатор користувача, ім’я користувача або роль. Заголовок і корисне навантаження кодуються в Base64, а не шифруються, тому ніколи не зберігайте тут чутливі дані, як-от паролі.
- **Підпис:** створюється шляхом хешування закодованих заголовка і корисного навантаження разом із секретним ключем, відомим лише серверу. Якщо корисне навантаження змінено, підпис не співпаде, і сервер відхилить запит із відповіддю `401 Unauthorized`.

JWT дозволяють реалізувати **автентифікацію без стану**. Серверу не потрібно зберігати дані сесії в базі даних, бо підпис доводить, що токен створено сервером.

## Підробка міжсайтових запитів (CSRF)

CSRF — це атака, яка змушує автентифікованого користувача ненавмисно виконати небажані дії у веб-застосунку. Зловмисний сайт обманює браузер жертви, щоб той надіслав шахрайський запит на бекенд-сервер, використовуючи те, що браузери автоматично приєднують cookie до будь-якого запиту на цільовий сервер.

**Приклад потоку атаки:**

1. Ви входите, наприклад, на `trusted-bank.com`. Банк зберігає токен сесії у cookie вашого браузера.
2. Не виходячи з системи, ви відвідуєте `evil-hacker-site.com`.
3. Зловмисний сайт тихо надсилає запит на `trusted-bank.com/transfer?amount=1000&to=hacker`.
4. Ваш браузер автоматично додає cookie сесії банку, тому сервер обробляє запит.

Зловмисний сайт не може прочитати ваш cookie. Він просто користується тим, що браузер автоматично додає cookie до будь-якого запиту на цільовий сервер.

**Захист від CSRF:**

- **`SameSite=Strict`:** браузер ніколи не надсилає cookie, якщо запит походить із стороннього сайту.
- **`SameSite=Lax`:** дозволяє cookie при безпечній навігації (натисканні на посилання), але блокує їх при крос-сайтових відправленнях форм і API-запитах.
- **CSRF токени:** коли користувач завантажує сторінку, сервер генерує випадковий, унікальний і непередбачуваний токен і приєднує його до форми на фронтенді або сесії. Клієнт має включити його у заголовки запиту або дані форми. Оскільки зловмисний сайт не має доступу до фронтенду, щоб скопіювати токен, підроблені запити не проходять валідацію.

## Passport.js

Passport.js — це підпрограмне забезпечення автентифікації для Node.js, яке стандартизує спосіб автентифікації запитів за допомогою модульних плагінів, званих **стратегіями**. Існує понад 500 стратегій, тож ви можете додати будь-який метод автентифікації, встановивши відповідний пакет:

- `passport-local`: ім’я користувача і пароль проти вашої бази даних.
- `passport-jwt`: перевіряє JSON Web Tokens на захищених API-маршрутах.
- `passport-google-oauth20`: вхід через облікові записи Google.
- `passport-github2`: вхід через облікові записи GitHub.

Встановіть основний пакет і стратегію:

```bash
npm install passport passport-local
```

Підключіть Passport до вашого Express-застосунку перед визначенням маршрутів:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database look-up and password verification logic
}));
```

Захистіть маршрут входу за допомогою Passport як підпрограмного забезпечення:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

Після успішної автентифікації Passport автоматично керує **керуванням сесією** (зберігає дані користувача в cookie сесії) і **декорацією запиту** (приєднує об’єкт користувача до `req.user` для використання в наступних обробниках маршрутів).

## Helmet.js

Helmet.js — це колекція підпрограмного забезпечення безпеки для Node.js, яка захищає ваш застосунок, автоматично встановлюючи, змінюючи або видаляючи HTTP-заголовки відповіді, які Express відкриває за замовчуванням.

Встановіть і застосуйте його одним кроком:

```bash
npm install helmet
```

```js
const helmet = require('helmet');
app.use(helmet());
```

Розміщення `app.use(helmet())` на початку стеку підпрограмного забезпечення застосовує ці захисти до кожної відповіді:

- **Видалено X-Powered-By:** Express за замовчуванням рекламує себе через `X-Powered-By: Express`. Helmet видаляє цей заголовок, щоб запобігти атакам на вразливості конкретних версій.
- **Встановлено Content-Security-Policy:** обмежує, звідки браузер може завантажувати скрипти, зображення та стилі, захищаючи від атак Cross-Site Scripting (XSS).
- **Встановлено X-Frame-Options у SAMEORIGIN:** запобігає вбудовуванню вашого сайту в зовнішні iframe, блокуючи атаки clickjacking.
- **Встановлено Strict-Transport-Security:** змушує браузери спілкуватися з вашим сервером лише через HTTPS, захищаючи дані від перехоплення в ненадійних мережах.
