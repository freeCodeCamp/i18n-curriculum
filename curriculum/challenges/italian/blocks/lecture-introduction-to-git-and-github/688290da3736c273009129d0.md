# --description--

Nella lezione precedente, abbiamo creato un ramo feature, effettuato il commit di un nuovo file su quel ramo e lo abbiamo inviato al nostro repository remoto. Ma quali sono i passaggi successivi? Dobbiamo creare una pull request su GitHub.

Una pull request è, di fatto, una richiesta di unire le modifiche dal tuo ramo al ramo di destinazione. Le pull request sono il flusso che usi quando vuoi contribuire con modifiche al codice a un progetto open source. Questo approccio permette ai manutentori del progetto di revisionare le tue modifiche. Possono lasciare commenti, fare domande, suggerire modifiche e (si spera) approvare le tue modifiche e unirle!

Andiamo di nuovo al nostro repository su GitHub.

Ora, dopo aver inviato il tuo nuovo ramo, dovresti vedere un banner elegante che dice che hai inviato a un ramo, e un pulsante su cui puoi cliccare per creare una pull request. Questo pulsante salta un paio di passaggi e semplifica il processo, quindi per ora lo ignoreremo. In questo modo sarai preparato per i casi in cui il banner non appare.

Clicca sulla scheda pull requests in alto nella pagina del repository, poi ignora di nuovo il banner e clicca su "New Pull Request". Verrai portato a un'interfaccia per preparare la tua richiesta.

Ora, ci sono alcuni termini che dobbiamo spiegare qui. Prima di tutto, abbiamo i menu a tendina "base" e "compare". La "base" è il tuo obiettivo per l'unione. Poiché vogliamo unire in `main`, possiamo lasciarla così com'è.

Il "compare" è ciò che vuoi unire. Vogliamo unire il nostro ramo `feature`, quindi dovremmo cambiarlo in `feature`. Questo è anche noto come ramo head, un termine che ti conviene ricordare.

Una volta fatto questo, l'interfaccia si aggiornerà. Ora dovrebbe mostrare i commit su `feature` che non sono su `main`, indicando ciò che stai richiedendo di unire. Vedrai anche quella che si chiama "diff", una rappresentazione visiva delle modifiche effettive che hai fatto.

Poiché tutto ciò che abbiamo fatto è aggiungere un file, il diff è abbastanza pulito. Per modifiche più complesse, GitHub evidenzierà cose come modifiche di riga, aggiunte, cancellazioni e altro. Capire come leggere un diff può essere difficile, ma con un po' di pratica diventerai abbastanza abile.

Per ora, torniamo alla nostra pull request. C'è un piccolo collegamento in alto che dice "compare across forks". Se stessimo lavorando dal nostro fork del repository freeCodeCamp, questo ci darebbe la possibilità di cambiare la nostra PR per puntare al repository originale di freeCodeCamp, invece che solo al ramo `main` all'interno del nostro fork.

Poiché non stiamo lavorando con un fork, possiamo ignorare questo. Tuttavia, è molto importante capire che ciò che stiamo facendo qui è principalmente per imparare ed esplorare. Non dovresti aprire una PR su un progetto di qualcun altro che contiene solo modifiche di pratica — questo crea rumore extra per i manutentori.

Ora, dato che siamo soddisfatti delle nostre modifiche e tutto è configurato, possiamo cliccare sul pulsante "Create Pull Request". Questo ci porta a una nuova interfaccia.

Verso il basso e fuori vista ci sono la stessa lista di commit e la vista diff che abbiamo visto prima. Non fa mai male ricontrollare un'ultima volta prima di creare la PR, ma per questa esplorazione puoi andare avanti.

Vedrai anche un paio di nuovi input. Il primo input è il titolo della tua pull request. GitHub lo genererà automaticamente in base al commit che stai unendo, se ce n'è solo uno nella PR, o al nome del ramo, se hai più commit.

Il secondo input è per la tua descrizione. Nella maggior parte dei casi, questo non sarà vuoto. Invece, i manutentori di un progetto possono definire un modello che viene precompilato. Se ne forniscono uno, dovresti rispettare il loro flusso di lavoro e completare il modello.

In entrambi i casi, ti conviene leggere la documentazione per contribuire al progetto per assicurarti di capire e seguire le loro linee guida su come intitolare e compilare la tua PR.

Poiché possiedi questo repository, vedrai a destra che puoi impostare revisori, assegnatari, etichette, progetti e milestone. Non potrai modificare questi elementi, tranne per richiedere una revisione, quando contribuisci a un progetto di qualcun altro. Ma se vuoi esplorare queste funzionalità, puoi impostarle come preferisci ora che questo è il tuo repository.

Una volta che tutto è compilato e configurato, clicca su "Create pull request". Infine, vedrai il risultato effettivo!

Quando fai una pull request su un progetto di qualcun altro, deve passare attraverso una "code review". Questo è un processo in cui i manutentori possono valutare la tua pull request, richiedere modifiche, fare domande e così via. Una volta che i manutentori l'hanno approvata, possono unirla.

Poiché questo è il tuo progetto, non devi aspettare una revisione. Puoi invece ricontrollare la cronologia dei commit nella scheda Commits, le modifiche nella scheda Files changed, e se sei soddisfatto puoi unirla.

Dopo che è stata unita, vorrai scaricare queste modifiche nel tuo repository locale.

Torna al terminale e assicurati di essere ancora nella cartella del repository. Poi passa al ramo `main` con `git checkout main`.

Successivamente, scarica le modifiche più recenti con `git pull`.

Ora, se guardi la cronologia dei commit con `git log`, vedrai il tuo commit. Ma vedrai anche un messaggio `Merge pull request #1`. Cos'è?

Beh, quando unisci una pull request su GitHub ci sono tre strategie che puoi usare:

La predefinita è "Merge", che porta tutti i commit dal ramo head, poi crea un altro commit per processare e registrare l'unione.

Un'altra opzione è "Squash and Merge", che prende TUTTI i commit dal ramo head, li schiaccia in un singolo commit e unisce quel nuovo commit direttamente nel ramo base senza creare un commit di merge.

Infine, c'è "Rebase and Merge", che prende tutti i commit, li resetta per "venire dopo" l'ultimo commit sul ramo base, poi unisce il ramo head senza un commit di merge. Il rebase è un argomento complicato che imparerai in dettaglio più avanti, quindi va bene se oggi non lo capisci del tutto.

Ogni progetto open source sceglierà la strategia di merge che funziona meglio. Ma a causa di queste differenze di comportamento, dovresti SEMPRE creare un nuovo ramo su cui fare i commit dei tuoi contributi. Non fare mai commit direttamente sul ramo predefinito `main`, altrimenti le cose si complicano quando devi portare giù le modifiche dal repository originale.

Con tutto questo, hai creato e unito con successo la tua prima pull request!
