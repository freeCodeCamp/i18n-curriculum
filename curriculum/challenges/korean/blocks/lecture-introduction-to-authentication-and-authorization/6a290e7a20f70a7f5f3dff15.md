# --description--

지난 두 강의에서 사용자 신원을 확인하고 쿠키를 사용해 세션을 유지하는 방법을 배웠습니다. 하지만 모든 요청에 쿠키를 자동으로 보내면 크로스 사이트 요청 위조(CSRF)라는 고유한 보안 취약점이 생깁니다.

CSRF는 인증된 사용자가 현재 로그인한 웹 애플리케이션에서 원하지 않는 동작을 실행하도록 강제하는 공격입니다. CSRF 공격에서는 악성 웹사이트가 피해자 브라우저를 속여 백엔드 서버에 위조된 요청을 보내게 하며, 서버가 사용자 브라우저 쿠키를 신뢰하는 점을 악용합니다.

CSRF를 이해하려면 은행 애플리케이션의 단계별 예시를 살펴보세요:

1. **설정:** 예를 들어 `trusted-bank.com` 은행 웹사이트에 로그인합니다. 은행은 자격 증명을 확인하고 세션 토큰을 브라우저 쿠키에 저장합니다.

2. **함정:** 은행에서 로그아웃하지 않은 상태로 새 브라우저 탭을 열어 악성 웹사이트인 `evil-hacker-site.com`을 방문합니다.

3. **트리거:** 악성 웹사이트에는 은행 백엔드로 자동 요청을 보내는 숨겨진 스크립트나 폼이 포함되어 있습니다. 예를 들어 `POST trusted-bank.com/transfer?amount=1000&to=hacker` 같은 요청입니다.

4. **실행:** 은행에 여전히 로그인되어 있으므로 브라우저가 자동으로 은행 세션 쿠키를 해당 요청에 첨부합니다. 은행 서버는 유효한 쿠키를 보고 사용자가 의도적으로 요청했다고 판단해 돈을 이체합니다.

중요한 점은 악성 웹사이트가 쿠키를 읽거나 데이터를 직접 훔칠 수 없다는 것입니다. 대신 브라우저가 대상 서버로 가는 모든 요청에 쿠키를 자동으로 첨부한다는 사실을 이용하는 것입니다.

주요 방어책은 **SameSite 쿠키 속성**과 **CSRF 토큰** 두 가지입니다.

가장 간단한 방어는 쿠키에 `SameSite` 속성을 설정하는 것입니다. 이 속성은 브라우저가 교차 사이트 요청에 쿠키를 언제 보낼지 알려줍니다.

- **SameSite=Strict:** 요청이 제3자 웹사이트에서 시작되면 브라우저가 절대 쿠키를 보내지 않습니다.

- **SameSite=Lax:** 사용자가 안전하게 대상 사이트로 이동할 때(예: 일반 링크 클릭)만 쿠키를 허용하고, 교차 사이트 폼 제출이나 API 요청에는 쿠키를 차단합니다.

강력한 보호를 위해 백엔드 프레임워크는 CSRF 토큰을 사용합니다. 사용자가 페이지를 불러올 때 백엔드가 무작위이고 고유하며 예측 불가능한 토큰을 생성해 프런트엔드 폼이나 세션에 첨부합니다.

클라이언트가 요청을 제출할 때 이 토큰을 맞춤 요청 헤더나 폼 데이터에 포함해야 합니다. 백엔드는 제출된 토큰과 사용자 세션에 저장된 토큰을 비교합니다. 악성 웹사이트는 프런트엔드에 접근해 이 비밀 토큰을 복사할 수 없으므로, 위조된 교차 사이트 요청에는 이 토큰이 없어 유효성 검사에 실패합니다.

이 전략들을 구현하면 백엔드에 도착하는 요청이 실제 사용자가 의도한 동작임을 보장할 수 있으며, 악성 행위자가 강제한 무단 동작이 아닙니다.
