# --description--

Você já se perguntou o que acontece quando algo dá errado no seu app Express?

E se a conexão com o banco de dados falhar? Ou se um usuário acessar uma rota que não existe?

Entender como o tratamento de erros funciona no Express é fundamental se você quer construir aplicações web seguras e confiáveis.

Nesta lição, vamos explicar como o Express lida com erros e como você pode capturá-los e responder a eles usando exemplos reais de código.

## Como tratar erros simples no Express

No Express, o tratamento de erros é baseado em middleware.

Assim como as funções middleware comuns, o middleware de tratamento de erros tem acesso a `req`, `res` e `next`. Mas ele inclui um parâmetro extra no início: `err`.

Veja como é a sintaxe:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

Você define esse middleware no final da sua pilha de middleware.

O Express sabe que é um manipulador de erros porque ele tem quatro parâmetros — o novo, `err`, além dos parâmetros típicos `req`, `res` e `next` usados por outras funções middleware.

Em uma lição futura, você vai aprender mais sobre middleware de tratamento de erros e como ele se encaixa na pilha maior de middleware do Express.

## Como tratar erros assíncronos

O tratamento de erros assíncronos depende da versão do Express que você está usando.

Note que, independentemente da versão do Express, você ainda precisa definir seu middleware de tratamento de erros no final da pilha, depois de todas as suas rotas.

Aqui está o manipulador de erros novamente:

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Express 4

No Express 4 e versões anteriores, que muitos projetos ainda usam, erros assíncronos **não são capturados automaticamente**.

Se você estiver usando `async`/`await` e algo lançar um erro dentro de um manipulador de rota `async`, o Express não vai capturá-lo a menos que você o passe manualmente para `next()`.

Veja como tratar erros assíncronos corretamente no Express 4:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

Se você esquecer o bloco `try/catch` e `next(err)`, o erro pode:

* Derrubar seu app
* Ou resultar em um aviso de rejeição de promise não tratada

Por isso, envolver a lógica assíncrona em `try`/`catch` é essencial no Express 4.

Alguns desenvolvedores até criam uma função auxiliar para evitar repetir `try/catch` em todos os lugares:

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

Depois usam assim:

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

Isso faz com que você não precise usar blocos `try/catch` em todas as suas rotas `async`, mas ainda encaminha os erros adequadamente.

## Express 5

A versão atual do Express melhora esse comportamento.

A partir do Express 5, se uma rota `async` lançar um erro ou retornar uma promise rejeitada, o Express encaminha automaticamente para o middleware de tratamento de erros.

Ou seja, este exemplo trata erros sem um bloco `try/catch`:

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

Se `getUserFromDatabase()` lançar um erro, o Express 5 vai capturá-lo e passá-lo para seu manipulador de erros automaticamente.

Não é necessário chamar `next(err)` manualmente.

## Como tratar erros 404

Erros 404 são um pouco diferentes. Eles acontecem quando nenhuma rota corresponde à requisição.

Você pode tratá-los com este middleware:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**Importante**: isso deve ficar depois das suas rotas, mas antes do middleware de tratamento de erros.

E finalmente, aqui estão algumas melhores práticas para tratamento de erros no Express:

* Sempre coloque seu middleware de tratamento de erros por último.
* Trate erros 404 separadamente.
* Use `next(err)` para propagar erros de rotas ou lógica `async` no Express 4 e versões anteriores.

Em produção, não envie rastros de pilha para os usuários. Registre os detalhes internamente e retorne uma mensagem amigável para o usuário.

O tratamento de erros no Express é sobre configurar middleware que escuta problemas e responde de forma clara e segura. Seja um bug no seu código ou uma rota ausente, seu app nunca deve travar sem um plano B.
