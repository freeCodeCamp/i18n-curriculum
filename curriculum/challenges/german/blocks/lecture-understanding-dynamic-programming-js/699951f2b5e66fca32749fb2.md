# --description--
  
Dynamische Programmierung ist eine algorithmische Technik, die komplexe Probleme löst, indem sie diese in einfachere Teilprobleme zerlegt und die Ergebnisse speichert, um redundante Berechnungen zu vermeiden. Dieser Ansatz verwandelt Probleme, die normalerweise exponentielle Zeit benötigen, in solche, die in polynomialer Zeit gelöst werden können.

## Grundsätze der dynamischen Programmierung

Dynamische Programmierung funktioniert, wenn zwei wichtige Bedingungen bei einem Problem vorliegen.

- **Überlappende Teilprobleme**: Dieselben kleineren Probleme treten mehrfach auf, wenn das größere Problem gelöst wird. Anstatt diese Teilprobleme immer wieder neu zu berechnen, speichern wir ihre Lösungen.
    
- **Optimale Teilstruktur**: Die optimale Lösung des Problems enthält optimale Lösungen seiner Teilprobleme. Das bedeutet, wir können die beste Lösung aufbauen, indem wir die besten Lösungen der kleineren Teile kombinieren.
    

Lassen Sie uns diese Konzepte anhand des klassischen Problems „Treppe steigen“ betrachten.

## Das Problem mit naiver Rekursion

Betrachten Sie das Problem „Treppe steigen“: Sie steigen eine Treppe mit `n` Stufen hinauf und können jeweils 1 oder 2 Stufen auf einmal steigen. Wie viele verschiedene Wege gibt es, um ganz oben anzukommen?

```js
function climbStairsRecursive(n) {
  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Recursive rule: to reach step n, you can come from step (n-1) or step (n-2)
  return climbStairsRecursive(n - 1) + climbStairsRecursive(n - 2);
}
```

Diese Implementierung hat eine exponentielle Zeitkomplexität wegen massiver redundanter Berechnungen. Beim Berechnen von `climbStairsRecursive(5)` passiert Folgendes:

- `climbStairsRecursive(5)` ruft `climbStairsRecursive(4)` und `climbStairsRecursive(3)` auf
    
- `climbStairsRecursive(4)` ruft `climbStairsRecursive(3)` und `climbStairsRecursive(2)` auf
    
- Jetzt wird `climbStairsRecursive(3)` **zweimal** berechnet
    
- `climbStairsRecursive(3)` ruft `climbStairsRecursive(2)` und `climbStairsRecursive(1)` auf
    
- `climbStairsRecursive(2)` wird **insgesamt 3-mal** berechnet
    

Für nur `n=5` machen wir 9 Funktionsaufrufe, obwohl wir nur 5 eindeutige Berechnungen benötigen. Mit wachsendem `n` explodiert diese Redundanz exponentiell – `climbStairsRecursive(30)` würde über mehrere Millionen Funktionsaufrufe erfordern! Die Zeitkomplexität wird `O(2^n)`, was ineffizient und unpraktisch für größere Werte von `n` ist.

## Lösungen mit dynamischer Programmierung

Dynamische Programmierung beseitigt diese redundanten Berechnungen durch zwei Hauptansätze:

### Memoisierung (Top-Down-Ansatz)

Memoisierung speichert die Ergebnisse teurer Funktionsaufrufe und gibt das zwischengespeicherte Ergebnis zurück, wenn dieselben Eingaben erneut auftreten:

```js
function climbStairsMemo(n, memo = {}) {
  // If already calculated, return cached value
  if (memo[n] !== undefined) {
    return memo[n];
  }

  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Calculate once and store in memo for future use
  memo[n] =
    climbStairsMemo(n - 1, memo) +
    climbStairsMemo(n - 2, memo);

  return memo[n];
}
```

Memoisierung ist viel effizienter, weil jeder eindeutige Wert von `1` bis `n` genau einmal berechnet wird. Wenn wir `climbStairsMemo(3)` erneut benötigen, berechnen wir ihn nicht neu (was weitere rekursive Aufrufe auslösen würde), sondern schlagen ihn einfach in unserem Memo-Wörterbuch in `O(1)` Zeit nach.

Verfolgen wir die Ausführung von `climbStairsMemo(5)` mit dem Top-Down-Ansatz, um zu sehen, wie Memoisierung redundante Arbeit eliminiert:

```md
Call: climbStairsMemo(5)
  memo = {} (empty)
  
  Call: climbStairsMemo(4) 
    memo = {} (empty)
    
    Call: climbStairsMemo(3)
      memo = {} (empty)
      
      Call: climbStairsMemo(2) → returns 2 (base case)
      Call: climbStairsMemo(1) → returns 1 (base case)
      
      Result: 2 + 1 = 3
      memo = {3: 3} (stored!)
    
    Call: climbStairsMemo(2) → returns 2 (base case)
    
    Result: 3 + 2 = 5
    memo = {3: 3, 4: 5} (stored!)
  
  Call: climbStairsMemo(3) → returns 3 (FROM MEMO - no recursion!)
  
  Result: 5 + 3 = 8
  memo = {3: 3, 4: 5, 5: 8}
```

**Effizienzvergleich**

- **Naive Rekursion**: Macht 9 Funktionsaufrufe mit wiederholten Berechnungen
    
- **Memoisierung**: Macht nur 5 eindeutige Berechnungen und verwendet dann gespeicherte Ergebnisse
    
- **Zeitkomplexität**: Reduziert von `O(2^n)` auf `O(n)`, da nur `n` eindeutige Berechnungen nötig sind
    
- **Speicherkomplexität**: `O(n)` für den Memo-Speicher und den Aufrufstack
    
- **Tatsächliche Auswirkung**: `climbStairsMemo(30)` sinkt von Millionen Funktionsaufrufen auf etwa 30 eindeutige Berechnungen!
    

### Tabellierung (Bottom-Up-Ansatz)

Tabellierung baut die Lösung von unten nach oben auf, indem eine Tabelle mit Lösungen für Teilprobleme gefüllt wird:

```js
function climbStairsTabulation(n) {
  if (n <= 2) {
    return n;
  }

  // Create an array to store solutions from 0 to n
  const dp = new Array(n + 1).fill(0);

  // Base cases
  dp[1] = 1; // 1 way to reach step 1
  dp[2] = 2; // 2 ways to reach step 2

  // Build up the solution iteratively
  for (let i = 3; i <= n; i++) {
    // Ways to reach step i = ways to reach (i-1) + ways to reach (i-2)
    dp[i] = dp[i - 1] + dp[i - 2];
  }

  return dp[n];
}
```

Tabellierung eliminiert Rekursion vollständig, indem die Lösung iterativ von den kleinsten Teilproblemen bis zum Ziel aufgebaut wird.

Sehen wir uns den Bottom-Up-Ansatz in Aktion an, um zu sehen, wie wir die Lösung systematisch aufbauen. Hier ist der iterative Aufbau für `climbStairsTabulation(5)`:

```md
Initial state:
dp = [0, 1, 2, 0, 0, 0]
     [0, 1, 2, 3, 4, 5] ← indices (step numbers)

Step by step construction:

i = 3:
  dp[3] = dp[2] + dp[1] = 2 + 1 = 3
  dp = [0, 1, 2, 3, 0, 0]
  
i = 4:
  dp[4] = dp[3] + dp[2] = 3 + 2 = 5
  dp = [0, 1, 2, 3, 5, 0]
  
i = 5:
  dp[5] = dp[4] + dp[3] = 5 + 3 = 8
  dp = [0, 1, 2, 3, 5, 8]

Final result: dp[5] = 8
```

**Wichtige Vorteile der Tabellierung**

- **Kein Rekursions-Overhead**: Im Gegensatz zur Memoisierung gibt es keinen rekursiven Aufrufstack.
    
- **Vorhersehbare Ausführung**: Wir berechnen Werte in einer vorgegebenen Reihenfolge (1, 2, 3, 4, 5...).
    
- **Cache-freundlich**: Sequentieller Array-Zugriff optimiert die Speichernutzung.
    
- **Leicht zu optimieren**: Kann die Speicherkomplexität auf `O(1)` reduzieren, da nur die letzten zwei Werte benötigt werden.
    

```js
function climbStairsOptimized(n) {
  if (n <= 2) {
    return n;
  }

  // Store only the last two results to save space
  let prev2 = 1; // ways to reach step 1
  let prev1 = 2; // ways to reach step 2

  for (let i = 3; i <= n; i++) {
    const current = prev1 + prev2;

    // Shift values forward
    prev2 = prev1;
    prev1 = current;
  }

  return prev1;
}
```

**Effizienzvergleich**

- **Naive Rekursion**: 9 Funktionsaufrufe für `n=5`, exponentielles Wachstum.
    
- **Tabellierung**: 3 einfache Additionen für `n=5`, lineares Wachstum.
    
- **Zeitkomplexität**: `O(n)` statt `O(2^n)`.
    
- **Speicherkomplexität**: `O(n)` für das Array oder `O(1)` mit Optimierung.
    
- **Vorhersehbare Leistung**: Kein Risiko eines Stack-Überlaufs bei großen Eingaben.
    

Beide Ansätze reduzieren die Zeitkomplexität von exponentiellem `O(2^n)` auf lineares `O(n)`, eine dramatische Verbesserung, die den Unterschied zwischen Millisekunden und Jahren bei größeren Eingaben ausmacht.

## Anwendungen in der Praxis

Dynamische Programmierung findet breite Anwendung in der Informatik und darüber hinaus:

- **Routenoptimierung**: GPS-Systeme verwenden dynamische Programmieralgorithmen, um kürzeste Wege zwischen Orten zu finden.
    
- **Textverarbeitung**: Rechtschreibprüfer und Autovervollständigungsfunktionen basieren oft auf dynamischer Programmierung, um Editierabstände zwischen Wörtern zu berechnen.
    
- **Finanzmodellierung**: Anlagestrategien und Portfolio-Optimierung nutzen häufig Techniken der dynamischen Programmierung.
    
- **Ressourcenzuteilung**: Das Rucksackproblem und seine Varianten treten in Planung, Budgetierung und Ressourcenmanagement auf.
    

## Praktisches Beispiel: Münzwechselproblem

Das Münzwechselproblem ist eine klassische Programmieraufgabe, die bei Lösung mit dynamischer Programmierung beide Schlüsselprinzipien von DP zeigt: optimale Teilstruktur und überlappende Teilprobleme.

Das Münzwechselproblem fragt: „Wie viele Münzen werden mindestens benötigt, um einen Zielbetrag zu erreichen?“

Hier ist eine Lösung mit dynamischer Programmierung:

```js
function minCoins(amount, coins) {
  // Create dp array from 0 to amount, filled with Infinity
  const dp = new Array(amount + 1).fill(Infinity);

  // Base case: 0 coins needed to make amount 0
  dp[0] = 0;

  // Build up from 1 to target amount
  for (let i = 1; i <= amount; i++) {
    // Try each coin denomination
    for (let coin of coins) {
      // Only use coin if it is not larger than current amount
      if (coin <= i) {
        // Choose the minimum: current value OR solution for remaining amount + 1 coin
        dp[i] = Math.min(dp[i], dp[i - coin] + 1);
      }
    }
  }

  // Return result if possible, -1 if impossible (infinity)
  return dp[amount] === Infinity ? -1 : dp[amount];
}

/*
 Example usage:
coins = [1, 3, 4], amount = 6

dp[0] = 0
dp[1] = dp[0] + 1 = 1
dp[2] = dp[1] + 1 = 2
dp[3] = min(dp[2]+1, dp[0]+1) = min(2+1, 0+1) = 1
dp[4] = min(dp[3]+1, dp[1]+1, dp[0]+1) = min(1+1, 1+1, 0+1) = 1
dp[5] = min(dp[4]+1, dp[2]+1, dp[1]+1) = min(1+1, 2+1, 1+1) = 2
dp[6] = min(dp[5]+1, dp[3]+1, dp[2]+1) = min(2+1, 1+1, 2+1) = 2

Result: 2 coins (3 + 3)
*/
```

Und so funktioniert der dynamische Programmieralgorithmus für das Münzwechselproblem Schritt für Schritt für `coins = [1, 3, 4]`, `amount = 6`:

```md
Initial state:
dp = [0, ∞, ∞, ∞, ∞, ∞, ∞]
     [0, 1, 2, 3, 4, 5, 6] ← amounts

Building up the solution:

For amount = 1:
  Try coin 1: dp[1] = min(∞, dp[0] + 1) = min(∞, 0 + 1) = 1
  dp = [0, 1, ∞, ∞, ∞, ∞, ∞]

For amount = 2:
  Try coin 1: dp[2] = min(∞, dp[1] + 1) = min(∞, 1 + 1) = 2
  dp = [0, 1, 2, ∞, ∞, ∞, ∞]

For amount = 3:
  Try coin 1: dp[3] = min(∞, dp[2] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[3] = min(3, dp[0] + 1) = min(3, 0 + 1) = 1
  dp = [0, 1, 2, 1, ∞, ∞, ∞]

For amount = 4:
  Try coin 1: dp[4] = min(∞, dp[3] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[4] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  Try coin 4: dp[4] = min(2, dp[0] + 1) = min(2, 0 + 1) = 1
  dp = [0, 1, 2, 1, 1, ∞, ∞]

For amount = 5:
  Try coin 1: dp[5] = min(∞, dp[4] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[5] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  Try coin 4: dp[5] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, ∞]

For amount = 6:
  Try coin 1: dp[6] = min(∞, dp[5] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[6] = min(3, dp[3] + 1) = min(3, 1 + 1) = 2
  Try coin 4: dp[6] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, 2]

Final result: dp[6] = 2 (achieved with coins 3 + 3)
```

Diese Lösung zeigt beide Schlüsselprinzipien der dynamischen Programmierung. Es gibt überlappende Teilprobleme, weil das Finden der minimalen Münzen für Betrag 6 die Lösungen für Beträge 5, 3 und 2 benötigt. Dieselben Teilprobleme tauchen auch bei der Berechnung anderer Beträge auf. Es gibt optimale Teilstruktur, weil die optimale Lösung für jeden Betrag optimale Lösungen für kleinere Beträge enthält. Wenn wir wissen, dass die minimalen Münzen für Betrag 3 1 sind, dann ist eine Möglichkeit, Betrag 6 zu bilden, diese Lösung plus eine weitere Münze im Wert von 3.

Ohne DP müssten wir jede mögliche Kombination von Münzen ausprobieren – eine exponentielle Anzahl von Möglichkeiten. Mit DP bauen wir die Lösung systematisch auf:

- **Zeitkomplexität**: `O(amount × number of coins)` statt exponentiell.
    
- **Speicherkomplexität**: `O(amount)` für das `dp` Array.
    
- **Keine redundante Arbeit**: Jedes Teilproblem (minimale Münzen für jeden Betrag) wird genau einmal gelöst.
    
- **Wiederverwendbare Ergebnisse**: Sobald wir die minimalen Münzen für Betrag 3 kennen, verwenden wir dieses Wissen für alle größeren Beträge, die davon profitieren können.
    

## Wann sollte man dynamische Programmierung verwenden?

Dynamische Programmierung ist effektiv, wenn:

- Das Problem in überlappende Teilprobleme zerlegt werden kann.
    
- Das Problem optimale Teilstruktur aufweist.
    
- Eine naive rekursive Lösung wiederholte Berechnungen erfordern würde.
    
- Sie die Zeitkomplexität auf Kosten der Speicherkomplexität optimieren möchten.
    

Gängige Muster der dynamischen Programmierung umfassen Optimierungsprobleme (Minimum-/Maximumwerte finden), Zählprobleme (Anzahl der Möglichkeiten etwas zu erreichen) und Entscheidungsprobleme, die in kleinere Entscheidungen zerlegt werden können.

Dynamische Programmierung verwandelt komplexe Probleme in handhabbare, indem systematisch Lösungen für Teilprobleme gespeichert und wiederverwendet werden. Das Verständnis dieser Technik öffnet die Tür, eine Vielzahl von rechnerischen Herausforderungen effizient zu lösen.
