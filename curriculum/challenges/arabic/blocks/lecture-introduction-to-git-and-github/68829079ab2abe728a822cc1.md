# --description--

في درس سابق، تعلمت كيفية إنشاء مستودع جديد على GitHub. لكن ماذا لو كان لديك مشروع موجود محليًا وتريد إضافته إلى GitHub؟ في هذا الدرس، ستتعلم كيفية دفع المستودعات المحلية الموجودة إلى GitHub وسحب التغييرات من المستودعات البعيدة.

ابدأ بإنشاء مستودع جديد على GitHub بدون أي ملفات. عند الانتهاء بشكل صحيح، يجب أن تُعاد توجيهك إلى صفحة تحتوي على عنوان URL اللازم لاستنساخ المستودع الخاص بك.

الآن انتقل إلى الطرفية واختر مشروعًا موجودًا ترغب في إضافته إلى GitHub. في هذا الدرس، اخترت مشروعًا محليًا يسمى `super-awesome-game` يحتوي على الملفات التالية:

```sh
jessicawilkins super-awesome-game >> ls
README.md  index.html script.js  styles.css
```

لإعداد التحكم في الإصدارات لهذا المشروع باستخدام Git، تحتاج إلى تشغيل الأمر `git init` في دليل المشروع. سيُنشئ هذا مستودع Git فارغًا حتى يبدأ Git بتتبع التغييرات لهذا المشروع. عند الانتهاء بشكل صحيح، يجب أن ترى مخرجات مشابهة في الطرفية:

```sh
jessicawilkins super-awesome-game >> git init
Initialized empty Git repository in /Users/jessicawilkins/workspace/freeCodeCamp/super-awesome-game/.git/
```

عند تهيئة مستودع Git فارغ في مشروع، سيتم إضافة دليل مخفي جديد باسم `.git`. لعرض إضافة هذا الدليل، يمكنك تشغيل الأمر `ls -a` الذي يعرض جميع الملفات والدلائل، بما في ذلك المخفية:

```sh
jessicawilkins super-awesome-game >> ls -a
.          ..         .git       README.md  index.html script.js  styles.css
```

من المهم عدم حذف ذلك الدليل `.git`، وإلا ستفقد كل سجل Git الخاص بك.

الآن بعد تهيئة مستودع Git الفارغ، يمكنك تشغيل الأمر `git status`. يُستخدم هذا الأمر لعرض الحالة الحالية لدليل العمل الخاص بك - ستستخدم هذا الأمر كثيرًا في سير عملك. يجب أن ترى مخرجات مشابهة لهذا:

```sh
jessicawilkins super-awesome-game >> git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	README.md
	index.html
	script.js
	styles.css

nothing added to commit but untracked files present (use "git add" to track)
```

`On branch main` تعني أنك على الفرع `main`. الفرع في Git هو مساحة عمل منفصلة يمكنك إجراء تغييرات فيها. غالبًا ما يمثل الفرع `main` الفرع الأساسي أو فرع الإنتاج في تطبيق حقيقي. فرق المطورين تنشئ عدة فروع للميزات الجديدة وتصحيح الأخطاء ثم تدمج تلك التغييرات مرة أخرى في الفرع `main`. ستتعلم المزيد عن هذه العملية في درس لاحق.

`Untracked files` تعني أن Git اكتشف ملفات لا يتتبعها حاليًا.

هناك خمس حالات يمكن أن يكون عليها الملف:

1. "Untracked": هذا يعني أن الملف جديد على المستودع، وGit لم "يراه" من قبل.
2. "Modified": هذا الملف كان موجودًا في الالتزام السابق، وله تغييرات لم تُسجَّل بعد.
3. "Ignored": من المحتمل ألا ترى الملفات المهملة في Git، لكن بيئة التطوير المتكاملة قد تظهر مؤشرًا لها. تُستثنى الملفات المهملة من عمليات Git، عادة لأنها مدرجة في ملف `.gitignore`.
4. "Deleted": الملف المحذوف هو عكس الملف غير المتتبع - هو ملف كان موجودًا سابقًا وتمت إزالته.
5. "Renamed": الملف المعاد تسميته هو ملف لم تتغير محتوياته، لكن تم تعديل اسمه أو موقعه. في بعض الحالات، يمكن اعتبار الملف معاد التسمية حتى لو كان يحتوي على تغييرات بسيطة.

عندما يكون لديك تغييرات غير ملتزمة في المستودع، ستحتاج أولًا إلى "تجهيزها". "منطقة التجهيز" هي ملف خاص في دليل `.git` يحتوي على معلومات ستُدرج في الالتزام التالي. الالتزام هو في الأساس لقطة للحالة الحالية للمستودع.

إذا أردت تجهيز ملف واحد في كل مرة، يمكنك استخدام الأمر التالي:

```sh
git add name-of-file
```

إذا أردت تجهيز كل التغييرات غير المجّهزة، يمكنك استخدام `git add .` النقطة (`.`) هي اختصار للدليل الحالي الذي أنت فيه. بعض المطورين يحذرون من استخدام `git add .` لأنك قد تنتهي بتجهيز ملفات لم تقصدها. إذا حدث ذلك، يمكنك "إلغاء تجهيزها" باستخدام الأمر التالي:

```sh
git reset name-of-file
```

بشكل عام، طالما تراقب الملفات التي تُضاف باستخدام `git status` قبل تجهيزها، ستكون الأمور على ما يرام.

بمجرد تجهيز ملفاتك، ستحتاج إلى الالتزام بها. يمكنك الالتزام بتغييراتك بتشغيل الأمر `git commit`. سيفتح هذا محرر النصوص المفضل الذي اخترته في إعدادات Git. بمجرد فتح المحرر، يمكنك كتابة رسالة مفصلة عن تغييراتك. يمكنك أيضًا اختيار كتابة رسالة قصيرة باستخدام الأمر `git commit -m` كما يلي:

```sh
git commit -m "short message goes here"
```

إذا كنت ستستخدم العلامة `-m`، فلا بد من كتابة رسالة وصفية لتوضيح سياق التغييرات. إليك بعض الأمثلة على رسائل جيدة قد تستخدمها:

```sh
git commit -m "feat(api): implement JWT-based authentication"
git commit -m "fix(auth): correct token expiration time"
git commit -m "refactor: simplify login flow logic"
```

تتبع `feat(api)` و `fix(auth)` أسلوب الالتزامات الاصطلاحية (Conventional Commits) الذي يُستخدم لتوفير سياق حول أنواع التغييرات في ذلك الالتزام.

بعد أن تلتزم عملك، وتشغل `git status` مرة أخرى، يجب أن ترى المخرجات التالية في الطرفية:

```sh
On branch main
nothing to commit, working tree clean
```

لعرض كل الالتزامات السابقة لهذا المشروع، يمكنك تشغيل الأمر `git log`. سيعرض هذا كل الالتزامات السابقة مع معلومات مفيدة مثل المؤلف، وتاريخ الالتزام، ورسالة الالتزام، ورمز الالتزام (commit hash). رمز الالتزام هو سلسلة طويلة تُستخدم كمُعرف فريد للالتزام.

بعد الانتهاء من الالتزام بعملك، يمكنك الآن دفع تغييراتك إلى GitHub. ستحتاج أولاً إلى إعداد الاتصال البعيد بمستودعك البعيد. تعلمت عن هذا في الدرس السابق، لكن هنا تذكير باستخدام SSH:

```sh
git remote add origin git@github.com:your-github-username/name-of-repo.git
```

الخطوة الأخيرة هي دفع تغييراتك إلى GitHub باستخدام هذا الأمر:

```sh
git push -u origin main
```

العلم `-u` هو اختصار لـ `--set-upstream`. يربط فرع `main` المحلي بفرع `main` البعيد على `origin`. تحتاج لاستخدامه مرة واحدة فقط لكل فرع جديد. بعد ذلك، يمكنك ببساطة تشغيل `git push` دون تحديد اسم الفرع.

عند الانتهاء بشكل صحيح، يجب أن ترى نتيجة مشابهة في الطرفية:

```sh
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Delta compression using up to 8 threads
Compressing objects: 100% (2/2), done.
Writing objects: 100% (3/3), 253 bytes | 253.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0 (from 0)
To https://github.com/jdwilkin4/super-awesome-game.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

يجب أن ترى أيضًا التغييرات الجديدة على صفحة مستودع GitHub عند تحديث الصفحة.

إذا كنت تتعاون مع آخرين في مشروع وتحتاج إلى سحب تغييرات جديدة من المستودع، يمكنك استخدام الأمر `git pull`. ستتعلم المزيد عن العمل الجماعي والتعاون في درس لاحق.
