## مكتبات وأُطُر جافاسكريبت

- توفر مكتبات وأُطُر جافاسكريبت حلولًا سريعة للمشاكل الشائعة وتسريع التطوير من خلال توفير شفرة مدمجة مسبقًا.
- تركز المكتبات عادةً على تقديم حلول لمهام محددة، مثل التلاعب بـ DOM، التعامل مع الأحداث، أو إدارة طلبات AJAX.
- من أمثلة مكتبات جافاسكريبت jQuery وReact.
- من ناحية أخرى، توفر الأُطُر هيكلًا أكثر تحديدًا لبناء التطبيقات. غالبًا ما تأتي مع مجموعة من القواعد والاصطلاحات التي يجب على المطورين اتباعها.
- من أمثلة الأُطُر Angular وNext.js، وهو إطار عمل ميتا لـ React.
- **التطبيقات ذات الصفحة الواحدة** (SPAs) هي تطبيقات ويب تحمل صفحة HTML واحدة وتُحدّث تلك الصفحة ديناميكيًا أثناء تفاعل المستخدم مع التطبيق دون إعادة تحميل الصفحة بالكامل.
- تستخدم SPAs جافاسكريبت لإدارة حالة التطبيق وعرض المحتوى. غالبًا ما يتم ذلك باستخدام أُطُر توفر أدوات رائعة لبناء واجهات مستخدم معقدة.
- بعض المشكلات المتعلقة بـ SPAs تشمل:

- تواجه قارئات الشاشة صعوبة مع المحتوى الذي يُحدّث ديناميكيًا.
  - لا يتغير عنوان URL عند تنقل المستخدم داخل التطبيق، مما يصعب حفظ الصفحة في المفضلة، أو الرجوع للخلف، أو مشاركة صفحات محددة.
  - قد يكون وقت التحميل الأولي بطيئًا إذا كان التطبيق كبيرًا لأن جميع الموارد يجب تحميلها مقدمًا.

## React

- React هي مكتبة جافاسكريبت شهيرة لبناء واجهات المستخدم وتطبيقات الويب.
- المفهوم الأساسي في React هو إنشاء مكونات واجهة مستخدم قابلة لإعادة الاستخدام يمكنها التحديث والعرض بشكل مستقل عند تغير البيانات.
- يسمح React للمطورين بوصف كيف يجب أن تبدو واجهة المستخدم بناءً على حالة التطبيق. ثم يقوم React بتحديث وعرض المكونات الصحيحة عند تغير البيانات أو الحالة.

## مكونات React

- المكونات هي اللبنات الأساسية لتطبيقات React التي تسمح للمطورين بتقسيم واجهات المستخدم المعقدة إلى أجزاء أصغر يمكن إدارتها.
- تُوصف واجهة المستخدم باستخدام JSX، وهو امتداد لصيغة جافاسكريبت يسمح بكتابة شفرة تشبه HTML داخل جافاسكريبت.
- المكونات هي في الأساس دوال أو أصناف جافاسكريبت تُرجع جزءًا من واجهة المستخدم.

إليك مثال على مكون React بسيط يعرض رسالة ترحيب:

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

لاستخدام المكون، يمكنك ببساطة استدعاؤه:
    
```jsx
  <Greeting />
```

## استيراد وتصدير مكونات React

- التصدير يجعل المكون متاحًا لملفات أخرى. الاستيراد يسمح لملف باستخدام مكون تم تصديره من مكان آخر.
- هناك نوعان من التصدير في React: التصدير الافتراضي والتصدير المسمي.

**التصدير الافتراضي**

- يمكن أن يحتوي الملف على تصدير افتراضي واحد فقط. وهو مثالي للملفات التي تحتوي بشكل رئيسي على مكون واحد.
- يمكنك تصدير مكون كتصدير افتراضي هكذا:

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default City;
```

- يمكنك أيضًا التصدير في نفس السطر مع تعريف المكون:

```jsx
export default function City() {
  return <p>New York</p>;
}
```

- لاستيراد تصدير افتراضي، استخدم كلمة `import` بدون أقواس متعرجة:

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

**التصدير المسمي**

- يسمح التصدير المسمي لملف بمشاركة عدة مكونات. على عكس التصدير الافتراضي، يجب استيرادها باستخدام الاسم الدقيق الذي تم تصديرها به.
- يمكنك تصدير عدة مكونات من نفس الملف هكذا:

```jsx
// Animals.jsx
export function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- لاستيراد التصديرات المسماة، استخدم الأقواس المتعرجة مع الأسماء المصدرة بالضبط:

```jsx
import { Cat, Dog } from './Animals';
```

- يمكنك إعادة تسمية تصدير مسمي أثناء الاستيراد باستخدام كلمة `as`:

```jsx
import { Cat as Kitty } from './Animals';
```

**التصدير المختلط بين الافتراضي والمسمي**

- يمكن للملف أن يحتوي على تصدير افتراضي واحد وعدة تصديرات مسماة في نفس الوقت:

```jsx
// Animals.jsx
export default function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- عند استيراد التصديرات المختلطة، يأتي التصدير الافتراضي أولًا (بدون أقواس)، يليه التصديرات المسماة (داخل الأقواس):

```jsx
import Cat, { Dog } from './Animals';
```

## إعداد مشروع React باستخدام Vite

- توفر أدوات إعداد المشاريع وواجهات سطر الأوامر طريقة سريعة وسهلة لبدء مشاريع جديدة، مما يسمح للمطورين بالتركيز على كتابة الشفرة بدلاً من التعامل مع الإعداد.
- Vite هو أداة إعداد مشاريع شهيرة ويمكن استخدامها مع React.
- لإنشاء مشروع جديد باستخدام Vite، يمكنك استخدام الأمر التالي في الطرفية:

```bash
npm create vite@latest my-react-app -- --template react
```

ينشئ هذا الأمر مشروع React جديد باسم `my-react-app` باستخدام قالب React الخاص بـ Vite. في مجلد المشروع، سترى ملف `package.json` يحتوي على تبعيات المشروع والأوامر المدرجة فيه.

- لتشغيل المشروع، انتقل إلى مجلد المشروع وشغّل الأوامر التالية:

```bash
cd my-react-app # path to the project directory
npm install # installs the dependencies listed in the package.json file
```

- بعد تثبيت التبعيات، يجب أن تلاحظ مجلدًا جديدًا في مشروعك يسمى `node_modules`.

- مجلد `node_modules` هو المكان الذي تُخزن فيه جميع الحزم والمكتبات المطلوبة لمشروعك.

- لتشغيل مشروعك، استخدم الأمر التالي:

```bash
npm run dev
```

- بعد ذلك، افتح المتصفح وانتقل إلى `http://localhost:5173` لرؤية تطبيق React الخاص بك يعمل.

- لرؤية شفرة قالب البداية فعليًا، يمكنك الدخول إلى مجلد `src` داخل مشروعك وستجد ملف `App.jsx`.

## تمرير الخصائص (props) في مكونات React

- في React، الخصائص (props) هي طريقة لتمرير البيانات من مكون أب إلى مكون ابن. هذه الآلية ضرورية لإنشاء عناصر واجهة مستخدم قابلة لإعادة الاستخدام وديناميكية.
- يمكن أن تكون الخصائص أي قيمة جافاسكريبت. لتمرير الخصائص من الأب إلى الابن، تضيف الخصائص كسمات عند استخدام المكون الابن في JSX الخاص بالأب. إليك مثالًا بسيطًا:

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

يمكنك تمرير عدة خصائص باستخدام معامل النشر `(...)`، بعد تحويلها إلى كائن. إليك مثالًا:
    
```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

في هذا الكود، يقوم معامل النشر `{...person}` بتحويل كائن person إلى خصائص فردية تُمرر إلى مكون Child.

## العرض الشرطي في React

- يسمح العرض الشرطي في React بإنشاء واجهات مستخدم ديناميكية. يُستخدم لعرض محتوى مختلف بناءً على شروط أو حالات معينة داخل التطبيق.
- هناك عدة طرق لعرض المحتوى بشكل شرطي في React. أحد الأساليب الشائعة هو استخدام المعامل الثلاثي. إليك مثالًا:

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- طريقة أخرى للعرض الشرطي هي استخدام معامل AND المنطقي (`&&`). هذا مفيد عندما تريد عرض المحتوى فقط إذا تحقق شرط معين. إليك مثالًا:

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

في الكود أعلاه، يتم عرض عنصر `h1` فقط إذا كان كائن المستخدم صحيحًا (truthy).

يمكنك أيضًا استخدام جملة `if` مباشرة بهذه الطريقة:

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```

## عرض القوائم في React

- عرض القوائم في React مهمة شائعة عند بناء واجهات المستخدم.
- يمكن عرض القوائم باستخدام طريقة `map()` لمصفوفة جافاسكريبت للتكرار على عناصر المصفوفة وإرجاع مصفوفة جديدة من عناصر JSX.
- على سبيل المثال، إذا كان لديك مصفوفة أسماء تريد عرضها كقائمة، يمكنك فعل التالي:

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- تذكر دائمًا توفير مفتاح فريد لكل عنصر في القائمة لمساعدة React في إدارة التحديثات وأدوار العرض. باستخدام هذه التقنيات، يمكنك إنشاء قوائم مرنة وفعالة وديناميكية في تطبيقات React الخاصة بك.

## الأنماط المضمّنة في React

- تسمح الأنماط المضمّنة في React بتطبيق أنماط CSS مباشرة على عناصر JSX باستخدام كائنات جافاسكريبت.
- لتطبيق الأنماط المضمّنة في React، يمكنك استخدام خاصية style على عناصر JSX. تأخذ خاصية style كائنًا حيث تكون المفاتيح خصائص CSS بصيغة تنسيق سنام الجمل والقيم هي القيم المقابلة. إليك مثالًا:

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

يمكنك أيضًا استخراج الأنماط إلى كائن منفصل والرجوع إليه في خاصية `style` بهذه الطريقة:

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- تدعم الأنماط المضمّنة التخصيص الديناميكي من خلال السماح بتطبيق الأنماط شرطياً بناءً على الخصائص أو الحالة. إليك مثالًا على كيفية تطبيق الأنماط شرطياً بناءً على خاصية:

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- في الكود أعلاه، يتم تعيين أنماط `color` و `fontSize` شرطياً بناءً على خاصية `isImportant`.

## العمل مع الأحداث في React

- **نظام الأحداث الاصطناعية**: هذه هي طريقة React في التعامل مع الأحداث. تعمل كغلاف حول الأحداث الأصلية مثل أحداث `click`، `keydown`، و`submit`. تستخدم معالجات الأحداث في React اصطلاح تسمية تنسيق سنام الجمل. (مثال: `onClick`، `onSubmit`، إلخ)

إليك مثالًا على استخدام خاصية `onClick` لعنصر `button` في React:

```jsx
function handleClick() {
  console.log("Button clicked!");
}

<button onClick={handleClick}>Click Me</button>;
```

عادةً ما تبدأ دوال معالجات الأحداث في React بالبادئة `handle` للدلالة على مسؤوليتها عن التعامل مع الأحداث، مثل `handleClick` أو `handleSubmit`.

عندما يُفعّل حدث بفعل المستخدم، يمرر React كائن حدث اصطناعي إلى المعالج الخاص بك. يتصرف هذا الكائن مثل كائن الحدث الأصلي في جافاسكريبت العادي، ويوفر خصائص مثل `type`، `target`، و`currentTarget`.

لمنع السلوكيات الافتراضية مثل تحديث المتصفح أثناء حدث `onSubmit`، يمكنك استدعاء الدالة `preventDefault()`:

```jsx
function handleSubmit(event) {
  event.preventDefault();
  console.log("Form submitted!");
}

<form onSubmit={handleSubmit}>
  <input type="text" />
  <button>Submit</button>
</form>;
```

يمكنك أيضًا تغليف دالة المعالج داخل دالة سهمية هكذا:

```jsx
function handleDelete(id) {
  console.log("Deleting item:", id);
}

<button onClick={() => handleDelete(1)}>Delete Item</button>;
```

## العمل مع الحالة (State) وخطاف `useState`

- **تعريف الحالة**: في React، الحالة هي كائن يحتوي على بيانات لمكون. عند تحديث الحالة، يُعاد عرض المكون. يعامل React الحالة كغير قابلة للتغيير، مما يعني أنه لا يجب تعديلها مباشرة.
- **خطاف `useState()`**: خطاف `useState` هو دالة تتيح لك إعلان متغيرات الحالة في المكونات الوظيفية. إليك الصيغة الأساسية:

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

في متغير الحالة لديك ما يلي:

- `stateVariable` يحمل القيمة الحالية للحالة
- `setStateFunction` (دالة التعيين) تحدث متغير الحالة
- `initialValue` يحدد الحالة الابتدائية

إليك مثالًا كاملاً لمكون `Counter`:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>

      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```

## العرض ومكونات React

- **التعريف**: في React، العرض هو العملية التي تظهر بها المكونات في واجهة المستخدم (UI)، عادةً في المتصفح. تتكون عملية العرض من ثلاث مراحل: التفعيل، العرض، والالتزام.

تحدث مرحلة التفعيل عندما يكتشف React أن شيئًا ما قد تغير وقد تحتاج واجهة المستخدم إلى التحديث. غالبًا ما يكون هذا التغيير بسبب تحديث في الحالة أو الخصائص.

بمجرد حدوث التفعيل، يدخل React مرحلة العرض. هنا، يعيد React تقييم مكوناتك ويحدد ما يجب عرضه. للقيام بذلك، يستخدم React نسخة خفيفة من DOM الحقيقية تسمى DOM الافتراضي. مع DOM الافتراضي، يمكن لـ React بسرعة التحقق مما يحتاج إلى التغيير في المكون.

مرحلة الالتزام هي حيث يأخذ React التغييرات المحضرة من DOM الافتراضي ويطبقها على DOM الحقيقي. بمعنى آخر، هذه هي المرحلة التي ترى فيها النتيجة النهائية على الشاشة.

## تحديث الكائنات والمصفوفات في الحالة

- **تحديث الكائنات في الحالة**: إذا كنت بحاجة إلى تحديث كائن في الحالة، يجب عليك أولاً إنشاء كائن جديد أو نسخ كائن موجود، ثم تعيين الحالة لذلك الكائن الجديد. يجب اعتبار أي كائن موضوع في الحالة للقراءة فقط. إليك مثالًا على تعيين اسم المستخدم، العمر، والمدينة. تُستخدم دالة `handleChange` لمعالجة التحديثات على معلومات المستخدم:

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **تحديث المصفوفات في الحالة**: عند تحديث المصفوفات في الحالة، من المهم عدم تعديل المصفوفة مباشرة باستخدام طرق مثل `push()` أو `pop()`. بدلاً من ذلك، يجب إنشاء مصفوفة جديدة عند تحديث الحالة:

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

إذا أردت إزالة عناصر من مصفوفة، يجب استخدام طريقة `filter()`، التي تُرجع مصفوفة جديدة بعد تصفية ما تريد إزالته:

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```

## الإشارة إلى القيم باستخدام Refs

- **خاصية `ref`**: يمكنك الوصول إلى عقدة DOM في React باستخدام خاصية `ref`. إليك مثالًا يوضح استخدام `ref` لتركيز عنصر `input`. تُستخدم خاصية `current` للوصول إلى القيمة الحالية لذلك الـ `ref`:

```jsx
import { useRef } from "react";

const Focus = () => {
  const inputRef = useRef(null);

  const handleFocus = () => {
    if (inputRef.current) {
      inputRef.current.focus();
    }
  };

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Enter text" />
      <button onClick={handleFocus}>Focus Input</button>
    </div>
  );
};

export default Focus;
```

## العمل مع خطاف `useEffect`

- **خطاف `useEffect()`**: في React، التأثير هو أي شيء يحدث خارج عملية عرض المكون. أي شيء لا يتعامل معه React مباشرة كجزء من عرض واجهة المستخدم. أمثلة شائعة تشمل جلب البيانات، تحديث عنوان تبويب المتصفح، القراءة من أو الكتابة إلى التخزين المحلي للمتصفح، الحصول على موقع المستخدم، وأكثر. هذه العمليات تتفاعل مع العالم الخارجي وتُعرف بالتأثيرات الجانبية. يوفر React خطاف `useEffect` للسماح لك بالتعامل مع هذه التأثيرات. يتيح لك `useEffect` تشغيل دالة بعد عرض أو تحديث المكون.

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [dependencies]);
```

تعمل دالة التأثير بعد عرض المكون، بينما يتحكم الوسيط الاختياري `dependencies` في توقيت تشغيل التأثير.

لاحظ أن `dependencies` يمكن أن تكون مصفوفة من "القيم التفاعلية" (الحالة، الخصائص، الدوال، المتغيرات، وهكذا)، أو مصفوفة فارغة، أو تُحذف تمامًا. إليك كيف تتحكم كل هذه الخيارات في عمل `useEffect`:

- إذا كانت `dependencies` مصفوفة تحتوي على قيمة تفاعلية واحدة أو أكثر، فسيتم تشغيل التأثير كلما تغيرت تلك القيم.

- إذا كانت `dependencies` مصفوفة فارغة، يعمل `useEffect` مرة واحدة فقط عند العرض الأول للمكون.

- إذا حذفت `dependencies`، يعمل التأثير في كل مرة يُعرض أو يُحدّث فيها المكون.

## كيفية إنشاء الخطافات المخصصة

- **الخطافات المخصصة**: تسمح الخطافات المخصصة باستخلاص منطق قابل لإعادة الاستخدام من المكونات، مثل جلب البيانات، إدارة الحالة، التبديل، والتأثيرات الجانبية مثل تتبع حالة الاتصال بالإنترنت. في React، تبدأ جميع الخطافات المدمجة بكلمة `use`، لذا يجب أن يتبع خطافك المخصص نفس الاصطلاح.

إليك مثالًا على إنشاء خطاف `useDebounce`:

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```

## العمل مع النماذج في React

- **المدخلات المُتحكم بها**: هي عندما تخزن قيمة حقل الإدخال في الحالة وتُحدّثها عبر أحداث `onChange`. يمنحك هذا تحكمًا كاملاً في بيانات النموذج ويسمح بالتحقق الفوري والعرض الشرطي.

```jsx
import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **المدخلات غير المُتحكم بها**: بدلاً من التعامل مع المدخلات عبر خطاف `useState`، تحتفظ المدخلات غير المُتحكم بها في HTML بحالتها الداخلية بمساعدة DOM. بما أن DOM يتحكم في قيم الإدخال، تحتاج إلى جلب قيم حقول الإدخال باستخدام `ref`.

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```

## العمل مع خطاف `useActionState`

- **الإجراءات على الخادم**: هي دوال تعمل على الخادم للسماح بمعالجة النماذج مباشرة على الخادم دون الحاجة إلى نقاط نهاية API. إليك مثالًا من تطبيق Next.js:

```js
"use server";

async function submitForm(formData) {
 const name = formData.get("name");
 return { message: `Hello, ${name}!` };
}
```

تُشير التوجيهة `"use server"` إلى أن الدالة هي إجراء على الخادم.

- **خطاف `useActionState`**: يقوم هذا الخطاف بتحديث الحالة بناءً على نتيجة إرسال النموذج. إليك الصيغة الأساسية لخطاف `useActionState`:

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state` هي الحالة الحالية التي تُرجعها الإجراء.
- `action` هي الدالة التي تُشغّل الإجراء على الخادم.
- `isPending` قيمة منطقية تشير إلى ما إذا كان الإجراء قيد التشغيل حاليًا أم لا.
- معلمة `actionFunction` هي الإجراء على الخادم نفسه.
- `initialState` هي المعلمة التي تمثل نقطة البداية للحالة قبل تشغيل الإجراء.
- `permalink` هي سلسلة اختيارية تحتوي على عنوان URL الفريد للصفحة التي يعدلها النموذج.

## جلب البيانات في React

- **خيارات جلب البيانات**: هناك العديد من الطرق المختلفة لجلب البيانات في React. يمكنك استخدام Fetch API الأصلي، أو أدوات طرف ثالث مثل Axios أو SWR.
- **متغيرات الحالة الشائعة عند جلب البيانات**: بغض النظر عن الطريقة التي تختارها لجلب البيانات في React، هناك بعض متغيرات الحالة التي ستحتاج إلى تتبعها. الأول هو البيانات نفسها. الثاني يتتبع ما إذا كانت البيانات لا تزال تُجلب. الثالث هو متغير حالة يلتقط أي أخطاء قد تحدث أثناء عملية جلب البيانات.

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

بما أن جلب البيانات هو تأثير جانبي، فمن الأفضل استخدام `Fetch API` داخل خطاف `useEffect`.

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

بعد ذلك يمكنك عرض رسالة تحميل إذا لم تكتمل عملية جلب البيانات، أو رسالة خطأ إذا حدث خطأ أثناء الجلب، أو عرض النتائج.

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

## العمل مع خطاف `useOptimistic`

- **خطاف `useOptimistic`**: يُستخدم هذا الخطاف للحفاظ على استجابة واجهات المستخدم أثناء انتظار اكتمال إجراء غير متزامن في الخلفية. يساعد في إدارة "التحديثات المتفائلة" في واجهة المستخدم، وهي استراتيجية تقدم تحديثات فورية للواجهة بناءً على النتيجة المتوقعة لإجراء، مثل انتظار استجابة الخادم.

إليك الصيغة الأساسية:

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState` هو الحالة المؤقتة التي تتحدث فورًا لتحسين تجربة المستخدم.
- `addOptimistic` هي الدالة التي تطبق التحديث المتفائل قبل تغيّر الحالة الفعلي.
- `actualState` هو القيمة الحقيقية للحالة التي تأتي من نتيجة إجراء ما، مثل جلب البيانات من الخادم.
- `updateFunction` هي الدالة التي تحدد كيف يجب أن تتحدث الحالة المتفائلة عند استدعائها.

إليك مثالًا على استخدام الخطاف `useOptimistic` في مكوِّن `TaskList`:

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**: تُستخدم هذه الدالة لعرض جزء من واجهة المستخدم وتحديد تحديث الحالة كتحول غير عاجل. هذا يسمح لواجهة المستخدم بأن تبقى متجاوبة أثناء التحديثات المكلفة. إليك الصيغة الأساسية:

```js
startTransition(action)
```

يقوم `action` بتحديث الحالة أو تشغيل بعض المنطق المتعلق بالتحول. هذا يضمن عدم حجب التحديثات العاجلة لواجهة المستخدم (مثل الكتابة أو النقر).

## العمل مع الخطاف `useMemo`

- **التخزين المؤقت (Memoization)**: هي تقنية تحسين تُخزّن نتائج استدعاءات الدوال المكلفة بناءً على معلمات محددة. عند تقديم نفس المعلمات مرة أخرى، تُعاد النتيجة المخزنة بدلاً من إعادة حساب الدالة.
- **الخطاف `useMemo`**: يُستخدم هذا الخطاف لتخزين القيم المحسوبة مؤقتًا. إليك مثالًا على تخزين نتيجة فرز مصفوفة كبيرة. ستُنفذ الدالة `expensiveSortFunction` فقط عندما تتغير `largeArray`:

```js
const memoizedSortedArray = useMemo(
  () => expensiveSortFunction(largeArray),
  [largeArray]
);
```

## أدوات إدارة التبعيات

- **تعريف التبعية**: في البرمجيات، التبعية هي عندما يعتمد مكوِّن أو وحدة في التطبيق على مكوِّن أو وحدة أخرى لتعمل بشكل صحيح. التبعيات شائعة في التطبيقات البرمجية لأنها تسمح للمطوّرين باستخدام دوال أو أدوات مُعدة مسبقًا من قبل الآخرين. التبعيتان الأساسيتان اللازمتان لمشروع React هما حزمتا `react` و `react-dom`:

- **تعريف مدير الحزم**: لإدارة تبعيات البرمجيات في مشروع، تحتاج إلى استخدام مدير حزم. مدير الحزم هو أداة تُستخدم للتثبيت، والتحديث، وإزالة التبعيات. العديد من لغات البرمجة الشهيرة مثل JavaScript وPython وRuby وJava تستخدم مديري حزم. من مديري الحزم الشائعين لـ JavaScript: npm وYarn وpnpm.
- **ملف `package.json`**: هو ملف تكوين رئيسي في المشاريع يحتوي على بيانات وصفية عن مشروعك، بما في ذلك اسمه، وإصداره، وتبعياته. كما يحدد السكريبتات، ومعلومات الترخيص، وإعدادات أخرى تساعد في إدارة المشروع وتبعياته.

## React Router

- **مقدمة**: React Router هي مكتبة طرف ثالث تسمح لك بإضافة التوجيه إلى تطبيقات React الخاصة بك. للبدء، تحتاج إلى تثبيت React Router في مشروع React موجود كما يلي:

```bash
npm i react-router
```

ثم داخل ملف `main.jsx` أو `index.jsx`، تحتاج إلى إعداد هيكل المسارات كما يلي:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

يُستخدم `path` و`element` لربط عنوان URL بمكوِّنات واجهة المستخدم معًا. في هذه الحالة، نُعد مسارًا للصفحة الرئيسية يشير إلى مكوِّن `App`.

- **إعداد عدة عروض ومسارات**: من الشائع في التطبيقات الكبيرة إعداد عدة عروض ومسارات كما يلي:

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

الخاصية `index` في هذه الأمثلة تمثل المسار الافتراضي لجزء معين من المسار. لذا، سيُعرض مكوِّن `Home` عند المسار الجذري `/` بينما يُعرض مكوِّن `ProductsHome` عند المسار `/products`.

- **التسكين في المسارات**: يمكنك تضمين مسارات داخل مسارات أخرى، مما يؤدي إلى إلحاق مسار الطفل بمسار الأب.

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

في المثال أعلاه، سيكون مسار المنتجات الرائجة هو products/trending.

- **الأجزاء الديناميكية**: الجزء الديناميكي هو حيث يكون أي جزء من مسار URL ديناميكيًا.

```jsx
<Route path=":category" element={<Category />} />
```

في هذا المثال لدينا جزء ديناميكي يسمى `category`. عندما ينتقل المستخدم إلى عنوان URL مثل products/brass-instruments، يتغير العرض إلى مكوِّن `Category` ويمكنك جلب البيانات المناسبة ديناميكيًا بناءً على الجزء.

- **الخطاف `useParams`**: يُستخدم هذا الخطاف للوصول إلى المعلمات الديناميكية من مسار URL.

```js
import { useParams } from "react-router";

export default function Category() {
  let params = useParams();
   {/* Accessing the category param: params.category */}
   {/* rest of code goes here */}
}
```

## أُطُر عمل React

- **مقدمة**: توفر أُطُر عمل React ميزات مثل التوجيه، وتحسين الصور، وجلب البيانات، والتوثيق، والمزيد. هذا يعني أنك قد لا تحتاج إلى إعداد تطبيقات منفصلة للواجهة الأمامية والخلفية لبعض حالات الاستخدام. من أمثلة أُطُر عمل React: Next.js وRemix.
- **توجيه Next.js**: يشمل هذا النظام دعم المسارات الديناميكية، والمسارات المتوازية، ومعالجات المسارات، وإعادة التوجيه، والتدويل، والمزيد.

إليك مثالًا على إنشاء معالج طلب مخصص:

```js
export async function GET() {
  const res = await fetch("https://example-api.com");
  const data = await res.json();

  return Response.json({ data });
}
```

- **تحسين الصور في Next.js**: يوسع مكوِّن `Image` عنصر HTML الأصلي `img` ويسمح بتحميل الصفحات بشكل أسرع وتحسين حجم الصور. هذا يعني أن الصور ستُحمّل فقط عند دخولها إطار العرض، وسيقوم مكوِّن `Image` تلقائيًا بتقديم الصور بالحجم المناسب لكل جهاز.

```jsx
import Image from "next/image";

export default function Page() {
  return (
    <Image src="link-to-image-goes-here" alt="descriptive-title-goes-here" />
  );
}
``` 

## تمرير الخصائص (Prop Drilling)

- **تعريف**: تمرير الخصائص هو عملية تمرير الخصائص من مكوِّن أب إلى مكوِّنات فرعية متداخلة بعمق، حتى عندما لا تحتاج بعض المكوِّنات الفرعية إلى هذه الخصائص.

## إدارة الحالة

- **واجهة برمجة التطبيقات للسياق (Context API)**: يشير السياق إلى عندما يجعل مكوِّن أب المعلومات متاحة للمكوِّنات الفرعية دون الحاجة لتمريرها صراحةً عبر الخصائص. تُستخدم `createContext` لإنشاء كائن سياق يمثل السياق الذي ستقرأه المكوِّنات الأخرى. يُستخدم `Provider` لتوفير قيم السياق للمكوِّنات الفرعية.

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**: يدير Redux الحالة من خلال توفير مخزن مركزي وتحكم صارم في تحديثات الحالة. يستخدم نمطًا متوقعًا مع الإجراءات (actions)، والمخفضات (reducers)، والبرمجيات الوسيطة (middleware). الإجراءات هي حمولات معلومات ترسل البيانات من تطبيقك إلى مخزن Redux، غالبًا ما تُشغل بواسطة تفاعلات المستخدم. المخفضات هي دوال تحدد كيف يجب أن تتغير الحالة استجابةً لتلك الإجراءات، مع ضمان تحديث الحالة بطريقة غير قابلة للتغيير. البرمجيات الوسيطة تعمل كجسر بين إرسال الإجراءات والمخفض، مما يسمح لك بتوسيع وظائف Redux (مثل التسجيل، والتعامل مع العمليات غير المتزامنة) دون تعديل التدفق الأساسي.

- **Zustand**: حل إدارة الحالة هذا مثالي للتطبيقات الصغيرة إلى المتوسطة الحجم. يعمل باستخدام الخطاف `useStore` للوصول إلى الحالة مباشرة في المكوِّنات والصفحات. هذا يسمح لك بتعديل والوصول إلى البيانات دون الحاجة إلى إجراءات أو مخفضات أو مزود.

## تصحيح أخطاء مكوِّنات React باستخدام أدوات مطوِّر React

- **أدوات مطوِّر React**: هي امتداد للمتصفح يمكنك استخدامه في Chrome وFirefox وEdge لفحص مكوِّنات React وتحديد مشاكل الأداء. بالنسبة لـ Safari، تحتاج إلى تثبيت حزمة npm `react-devtools`. بعد تثبيت أدوات مطوِّر React وفتح تطبيق React في المتصفح، افتح أدوات مطوِّر المتصفح للوصول إلى علامتي التبويب الإضافيتين المخصصتين لتصحيح React – المكوِّنات وProfiler.
- **علامة تبويب المكوِّنات**: تعرض هذه العلامة كل مكوِّن في شكل شجرة. إليك بعض الأمور التي يمكنك القيام بها في هذه العلامة:
  - عرض تسلسل المكوِّنات في التطبيق
  - فحص وتعديل الخصائص، والحالات، وقيم السياق في الوقت الحقيقي
  - فحص شفرة المصدر لكل مكوِّن محدد
  - تسجيل بيانات المكوِّن في وحدة التحكم
  - فحص عناصر DOM للمكوِّن
- **علامة تبويب Profiler**: تساعدك هذه العلامة على تحليل أداء المكوِّنات. يمكنك تسجيل أداء المكوِّنات لتحديد عمليات إعادة العرض غير الضرورية، وعرض مدة الالتزام، ومن ثم تحسين المكوِّنات البطيئة.

## مكوِّنات خادم React

- **تعريف**: مكوِّنات خادم React هي مكوِّنات React تُعرض حصريًا على الخادم، وترسل فقط HTML النهائي إلى العميل. هذا يعني أن تلك المكوِّنات يمكنها الوصول مباشرة إلى موارد الخادم وتقليل كمية جافا سكريبت المرسلة إلى المتصفح بشكل كبير.

## الفروقات بين الأداء الحقيقي والأداء المدرك

- **الأداء المدرك**: هو كيف يدرك المستخدمون أداء الموقع. هو تقييمهم من حيث الاستجابة والموثوقية. هذا قياس ذاتي، لذا يصعب تحديده بدقة، لكنه مهم جدًا لأن تجربة المستخدم تحدد نجاح أو فشل الموقع.
- **الأداء الحقيقي**: هو الأداء الموضوعي والقابل للقياس للموقع. يُقاس باستخدام مؤشرات مثل وقت تحميل الصفحة، ووقت استجابة الخادم، ووقت العرض. تتأثر هذه القياسات بعوامل متعددة تتعلق بالشبكة وبالشفرة نفسها.

## تقنيات لتحسين الأداء المدرك

- **التحميل الكسول**: هذه التقنية تقلل وقت التحميل الأولي قدر الإمكان عن طريق تحميل الموارد غير الأساسية في الخلفية.
- **تقليل تأخيرات الخطوط**: إذا كان موقعك يحتوي على خطوط مخصصة، يجب أيضًا محاولة تقليل تأخيرات تحميل الخطوط، لأن ذلك قد يؤدي إلى وميض أو عرض الخط البديل أثناء تحميل الخط المخصص. اقتراح لذلك هو استخدام خط بديل مشابه للخط المخصص، بحيث يكون التغيير أكثر سلاسة إذا حدث ذلك.
- **استخدام مؤشرات التحميل**: عرض مؤشر تحميل لعملية طويلة بمجرد نقر المستخدم على عنصر يمكن أن يساعد المستخدم على الشعور بالاتصال والمشاركة مع العملية، مما يجعل وقت الانتظار يبدو أقصر.

## مفاهيم الأداء الأساسية

- **ترتيب المصدر**: يشير إلى كيفية تنظيم عناصر HTML في المستند. هذا يحدد ما يتم تحميله أولًا ويمكن أن يؤثر بشكل كبير على الأداء وإمكانية الوصول.

بعض أفضل الممارسات لترتيب المصدر تشمل:

- وضع المحتوى الحرج مثل العناوين، والتنقل، أو النص الرئيسي في مكان مرتفع في هيكل HTML.
- تأجيل السكريبتات غير الأساسية مثل تلك الخاصة بالتحليلات أو الأدوات الخارجية، حتى لا تعيق العرض.
- استخدام التحسين التدريجي، لضمان عمل التجربة الأساسية حتى قبل تحميل الأنماط والسكريبتات. التحسين التدريجي هو طريقة لبناء المواقع والتطبيقات على أساس جعل الصفحة تعمل بـ HTML أولًا.

إليك مثالًا على ترتيب مصدر جيد، باستخدام أفضل الممارسات التي استعرضناها:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **مسار العرض الحرج**: هو تسلسل الخطوات التي يتبعها المتصفح لتحويل الشفرة إلى بكسلات على الشاشة.
- **زمن الاستجابة**: هو الوقت الذي يستغرقه الطلب للسفر بين المتصفح والخادم. بمعنى آخر، زمن استجابة مرتفع يعني صفحات بطيئة.

بعض طرق تقليل زمن الاستجابة تشمل:

- استخدام شبكات توصيل المحتوى (CDNs) لخدمة الملفات من مواقع أقرب.
- تفعيل الضغط باستخدام أدوات مثل Gzip لتقليل حجم الملفات.
- تحسين الصور واستخدام التحميل الكسول.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## تحسين INP

- **تعريف**: INP (Interaction to Next Paint) يقيس استجابة الصفحة بشكل عام عن طريق قياس الوقت من تفاعل المستخدم، مثل النقر أو الضغط على مفتاح، إلى المرة التالية التي يحدث فيها تحديث للعرض في المتصفح. INP أقل يعني صفحة أكثر استجابة.

إليك بعض الطرق لتحسين INP:

- تقليل عمل الخيط الرئيسي بتقسيم مهام جافا سكريبت الطويلة.
- استخدام `requestIdleCallback()` للسكريبتات غير الحرجة. هذا سيضع دالة في قائمة الانتظار ليتم استدعاؤها خلال فترات خمول المتصفح.
- تأجيل أو تحميل الكسول للأصول الثقيلة التي تم تغطيتها سابقًا.
- تحسين معالجات الأحداث. إذا كانت هذه المعالجات تعمل بشكل متكرر جدًا أو تقوم بعمليات مكلفة، فقد تبطئ الصفحة وتزيد INP. الحل هو استخدام تقنية التمهيد (debouncing). التمهيد يضمن أن الدالة تعمل فقط بعد توقف المستخدم عن الكتابة لفترة قصيرة - مثلاً 300 مللي ثانية. هذا يمنع الحسابات غير الضرورية ويحسن الأداء.

## كيف يعمل العرض في المتصفح

- **كيف يعمل العرض**: أولًا يقوم المتصفح بتحليل HTML وبناء DOM. بعد ذلك، يعالج المتصفح CSS، وينشئ نموذج كائن CSS أو CSSOM. هذا هيكل شجري آخر يحدد كيف يجب تنسيق العناصر. أخيرًا، يرسم المتصفح البكسلات على الشاشة، معروضًا كل عنصر بناءً على الأنماط والتخطيط المحسوب. في الصفحات المعقدة، قد يشمل ذلك طبقات متعددة تُركب معًا لتشكيل المخرج البصري النهائي.

## كيف يؤثر الأداء على الاستدامة

- **معلومات خلفية**: الإنترنت يستهلك حوالي 2% من انبعاثات الكربون العالمية — وهو نفس مقدار صناعة الطيران! كل بايت يُنقل يحتاج إلى كهرباء، من مراكز البيانات إلى أجهزة المستخدمين. الملفات الكبيرة والسكريبتات غير الفعالة تعني استهلاكًا أكبر للطاقة. موقع عالي الأداء ليس أسرع فقط، بل يقلل أيضًا من المعالجة غير الضرورية واستهلاك الطاقة.

## طرق تقليل أوقات تحميل الصفحة

- **تحسين أصول الوسائط**: الصور والفيديوهات الكبيرة هي من الأسباب الشائعة لبطء التحميل. بتحسين هذه الأصول، يمكنك تسريع موقعك بشكل كبير. يشمل ذلك ضغط الصور، واستخدام صيغ حديثة مثل WebP، واستخدام التحميل الكسول للأصول.
- **الاستفادة من التخزين المؤقت للمتصفح**: يسمح التخزين المؤقت للمتصفحات بتخزين أجزاء من موقعك محليًا، مما يقلل أوقات التحميل للزوار العائدين.
- **تصغير وضغط الملفات**: تقليل حجم ملفاتك يؤدي إلى تنزيلات أسرع. يشمل ذلك تقليل حجم الملفات المنقولة وتصغير ملفات CSS وJavaScript.

## تحسين "الوقت للاستخدام"

- **تعريف**: "الوقت للاستخدام" هو الفترة من طلب المستخدم لصفحة إلى الوقت الذي يمكنه فيه التفاعل معها بشكل فعّال. لتحسين "الوقت للاستخدام" يمكنك تحميل الأصول بشكل كسول أو تقليل الموارد التي تعيق العرض.

## مؤشرات الأداء الرئيسية للقياس

- **الطلاء الأول للمحتوى (FCP)**: يقيس مدى سرعة ظهور أول قطعة محتوى — نص أو صورة — على الشاشة. يُعتبر FCP جيدًا إذا كان أقل من 1.8 ثانية، وسيئًا إذا تجاوز 3 ثوانٍ. يمكنك فحص FCP باستخدام أدوات مطوِّر Chrome في علامة تبويب الأداء.
- **إجمالي وقت الحظر (TBT)**: يظهر مدة حظر الخيط الرئيسي بسبب مهام جافا سكريبت الثقيلة. إذا كان TBT مرتفعًا، يعاني المستخدمون من تفاعلات بطيئة. لتحسين TBT، قسم المهام الطويلة وأجل السكريبتات غير الضرورية.
- **معدل الارتداد**: هو نسبة الزوار الذين يغادرون دون التفاعل. إذا كان موقعك يعاني من معدل ارتداد مرتفع، فقد يكون السبب بطء الصفحة.
- **المستخدمون الفريدون**: يقيس عدد الزوار الفرديين الذين يزورون موقعك. لعرض معدل الارتداد والمستخدمين الفريدين، يمكنك استخدام Google Analytics. سيساعدك ذلك على مراقبة هذه المؤشرات وتحسين التفاعل.

## أدوات قياس الأداء الشائعة

- **أدوات مطوِّر Chrome**: هي أداة مدمجة داخل Google Chrome تتيح لك تحليل وتصحيح الأداء في الوقت الحقيقي. تعرض أدوات المطوِّر أوقات التحميل، واستخدام وحدة المعالجة المركزية، وتأخيرات العرض. هي مفيدة بشكل خاص لقياس الطلاء الأول للمحتوى (FCP)، وهو مدى سرعة رؤية المستخدم لأول محتوى مرئي. إذا كان موقعك بطيئًا، ستساعدك أدوات المطوِّر على تحديد الاختناقات.
- **Lighthouse**: أداة آلية تفحص الأداء، وتحسين محركات البحث، وإمكانية الوصول.
- **WebPageTest**: تتيح لك هذه الأداة اختبار كيفية تحميل موقعك من مواقع وأجهزة مختلفة. تقدم تفصيلًا دقيقًا لمؤشرات مثل مؤشر السرعة، وإجمالي وقت الحظر، وغيرها من مؤشرات الأداء الرئيسية. إذا أردت معرفة كيف يختبر المستخدمون الحقيقيون موقعك عالميًا، WebPageTest هي الأداة المناسبة.
- **PageSpeed Insights**: تحلل هذه الأداة موقعك وتقترح تحسينات سريعة لكل من الأجهزة المحمولة وسطح المكتب. تخبرك بما يبطئ موقعك وتعطي توصيات محددة مثل تحسين الصور، وإزالة السكريبتات التي تعيق العرض، وتقليل أوقات استجابة الخادم. PageSpeed Insights طريقة سريعة وسهلة لمعرفة كيف ترى Google أداء موقعك.
- **مراقبة المستخدم الحقيقي (RUM)**: تتعقب أدوات RUM سلوك المستخدم الفعلي، موضحة كيف يختبر الزوار الحقيقيون موقعك. من أدوات RUM الشهيرة Google Analytics، التي تتعقب أوقات تحميل الصفحات ومعدلات الارتداد، وNew Relic أو Datadog، التي تراقب مشاكل الأداء في الوقت الحقيقي. إذا أردت بيانات من المستخدمين الفعليين، أدوات RUM ضرورية.

## العمل مع واجهات برمجة تطبيقات الويب للأداء

- **تعريف**: تتيح واجهات برمجة تطبيقات الويب للأداء للمطوّرين تتبع مدى كفاءة تحميل الصفحة واستجابتها مباشرة في الشفرة. تسمح هذه الواجهات بقياس أوقات تحميل الصفحة، وتتبع تأخيرات العرض والتفاعل، وتحليل وقت تنفيذ جافا سكريبت.
- **`performance.now()`**: توفر هذه الواجهة طوابع زمنية عالية الدقة (بالملي ثانية) لقياس مدة تحميل أجزاء مختلفة من موقعك.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **واجهة توقيت الأداء (Performance Timing API)**: تعطي هذه الواجهة تفصيلًا لكل مرحلة من مراحل تحميل الصفحة، من بحث DNS إلى `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: تستمع هذه الواجهة لأحداث الأداء مثل تحولات التخطيط، والمهام الطويلة، وتفاعلات المستخدم.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## تقنيات لتحسين أداء CSS

- **رسوم CSS المتحركة**: تحريك بعض خصائص CSS، مثل الأبعاد، والموقع، والتخطيط، يُشغل عملية تسمى "إعادة التدفق" (reflow)، حيث يعيد المتصفح حساب موقع وهندسة بعض العناصر في الصفحة. هذا يتطلب إعادة رسم (repaint)، وهي عملية مكلفة حسابيًا. لذلك، يُنصح بتقليل عدد الرسوم المتحركة في CSS قدر الإمكان أو على الأقل إعطاء المستخدم خيار تشغيلها أو إيقافها.

## تقنيات لتحسين أداء جافا سكريبت

- **تقسيم الشفرة**: تقسيم شفرة جافا سكريبت إلى وحدات تؤدي مهامًا حرجة وغير حرجة مفيد أيضًا. بهذه الطريقة، يمكنك تحميل الوحدات الحرجة مسبقًا بأسرع وقت ممكن وتأجيل الوحدات غير الحرجة لعرض الصفحة بأسرع ما يمكن.
- **التلاعب بـ DOM**: تذكر أن التلاعب بـ DOM يشير إلى عملية تغيير محتوى الصفحة ديناميكيًا باستخدام جافا سكريبت من خلال التفاعل مع نموذج كائن المستند (DOM). التلاعب بـ DOM مكلف حسابيًا. بتقليل كمية التلاعب بـ DOM في شفرتك، ستحسن الأداء.

## أُطُر عمل CSS

- **أُطُر عمل CSS**: يمكن لأُطُر عمل CSS تسريع سير عملك، وخلق نمط بصري موحد عبر الموقع، وجعل تصميمك يبدو متناسقًا عبر متصفحات متعددة، والحفاظ على شفرة CSS أكثر تنظيمًا.
- **أُطُر عمل CSS الشهيرة**: بعض أُطُر العمل الشهيرة هي Tailwind CSS، وBootstrap، وMaterialize، وFoundation.
- **العيوب المحتملة**:
  - قد تتعارض CSS المقدمة من الإطار مع CSS المخصصة لديك.
  - قد يبدو موقعك مشابهًا لمواقع أخرى تستخدم نفس الإطار.
  - الأُطُر الكبيرة قد تسبب مشاكل في الأداء.

## نوعان من أُطُر عمل CSS

- **أُطُر عمل CSS ذات الأولوية للأدوات (Utility-first)**: هذه الأُطُر تحتوي على أصناف صغيرة لأغراض محددة، مثل تعيين الهامش، أو الحشو، أو لون الخلفية. يمكنك تعيين هذه الأصناف الصغيرة مباشرة إلى عناصر HTML حسب الحاجة. يُصنف Tailwind CSS كإطار عمل CSS من نوع Utility-first.

إليك مثالًا على استخدام Tailwind CSS لتنسيق زر.

```html
<button class="bg-blue-500 text-white font-bold py-2 px-4 rounded-full hover:bg-blue-700">
  Button
</button>
```

- **أُطُر عمل CSS القائمة على المكوِّنات**: هذه الأُطُر تحتوي على مكوِّنات مُعدة مسبقًا بأنماط محددة يمكنك إضافتها إلى موقعك. المكوِّنات متاحة في الوثائق الرسمية لإطار العمل، ويمكنك نسخها ولصقها في مشروعك. يُصنف Bootstrap كإطار عمل CSS قائم على المكوِّنات.

إليك مثالًا على استخدام Bootstrap لإنشاء مجموعة قائمة. بدلاً من تطبيق أصناف صغيرة على عناصر HTML الخاصة بك، ستضيف المكوِّن بالكامل، بما في ذلك الهيكل HTML.

```html
<div class="card" style="width: 25rem;">
  <ul class="list-group list-group-flush">
    <li class="list-group-item">HTML</li>
    <li class="list-group-item">CSS</li>
    <li class="list-group-item">JavaScript</li>
  </ul>
</div>
```

## Tailwind CSS

Tailwind هو إطار عمل CSS من نوع Utility-first. بدلاً من كتابة قواعد CSS مخصصة، تبني تصميماتك بدمج أصناف أدوات صغيرة مباشرة في HTML الخاص بك.

### أدوات التصميم المتجاوب

يستخدم Tailwind بادئات مثل `sm:`, `md:`, و `lg:` لتطبيق الأنماط عند أحجام شاشة مختلفة.

```html
<div class="w-full md:w-1/2 lg:flex-row">
  Responsive layout
</div>
```

### أدوات Flexbox

أصناف مثل `flex`, `flex-col`, `justify-around`, و `items-center` تسهل إنشاء تخطيطات مرنة.

```html
<div class="flex flex-col md:flex-row justify-around items-center">
  <p>Column on small screens</p>
  <p>Row on medium and larger screens</p>
</div>
```

### أدوات الشبكة (Grid)

يشمل Tailwind أدوات لـ CSS Grid، مثل `grid`, `grid-cols-1`, و `md:grid-cols-3`.

```html
<div class="grid grid-cols-1 md:grid-cols-3 gap-8">
  <div class="bg-gray-100 p-4">Column 1</div>
  <div class="bg-gray-100 p-4">Column 2</div>
  <div class="bg-gray-100 p-4">Column 3</div>
</div>
```

### أدوات التباعد

أدوات مثل `mt-8`, `mx-auto`, `p-4`, و `gap-4` تساعد في إنشاء تباعد متناسق دون كتابة CSS.

```html
<div class="mt-8 p-4 bg-indigo-600 text-white">
  Spaced content
</div>
```

### أدوات الطباعة

أدوات مثل `uppercase`, `font-bold`, `font-semibold`, و `text-4xl` تتحكم في مظهر النص.

يمكنك تعيين أحجام الخطوط التي تتكيف عند نقاط التوقف، مثل `text-3xl` و `md:text-5xl`.

```html
<h1 class="text-3xl md:text-5xl font-semibold text-center">
  Responsive Heading
</h1>
```

### الألوان وحالات التحويم

يوفر Tailwind لوحة ألوان واسعة، مثل `text-red-700`, `bg-indigo-600`, و `bg-gray-100`.

أصناف مثل `hover:bg-pink-600` تجعل التأثيرات التفاعلية بسيطة.

```html
<a href="#" class="bg-pink-500 hover:bg-pink-600 text-white px-4 py-2 rounded-md">
  Hover Me
</a>
```

### الحدود، الحلقات، والتأثيرات

- **الحدود**: `border-2 border-red-300` تضيف حدودًا بسماكة وألوان محددة.
- **الحلقات**: `ring-1 ring-gray-300` تخلق تأثيرات تشبه الإطار تُستخدم غالبًا للتركيز أو البطاقات.
- **الزوايا المستديرة والتكبير**: أصناف مثل `rounded-md`, `rounded-xl`, و `scale-105` تضيف لمسة نهائية.

```html
<div class="p-6 rounded-xl ring-2 ring-fuchsia-500 scale-105">
  Highlighted card
</div>
```

### التدرجات

يدعم Tailwind أدوات التدرجات اللونية مثل `bg-gradient-to-r from-fuchsia-500 to-indigo-600`.

```html
<div class="p-4 text-white bg-gradient-to-r from-fuchsia-500 to-indigo-600">
  Gradient background
</div>
```

## معالجات CSS المسبقة

- **معالج CSS مسبق**: معالج CSS مسبق هو أداة توسع CSS القياسي. يقوم بترجمة الشفرة ذات الصياغة الموسعة إلى ملف CSS أصلي. يمكن أن يساعد في كتابة CSS أنظف، قابل لإعادة الاستخدام، أقل تكرارًا، وقابل للتوسع للمشاريع المعقدة.
- **الميزات**: بعض الميزات التي توفرها معالجات CSS المسبقة تشمل المتغيرات، الخلائط، التسكين، ووراثة المحددات.
- **معالجات CSS المسبقة الشهيرة**: من أشهر معالجات CSS المسبقة Sass و Less و Stylus.
- **العيوب المحتملة**:
  - ترجمة قواعد CSS إلى CSS قياسي قد تسبب عبء إضافي.
  - قد يكون من الصعب تصحيح الشفرة المترجمة.

## Sass

- **Sass**: هو واحد من أشهر معالجات CSS المسبقة. Sass تعني "أوراق الأنماط الرائعة نحويًا".
- **الميزات المدعومة في Sass**: يدعم Sass ميزات مثل المتغيرات، قواعد CSS المتداخلة، الوحدات، الخلائط، الوراثة، والمشغلين للعمليات الرياضية الأساسية.

## نوعان من الصيغ المدعومة في Sass

- **صيغة SCSS**: SCSS (Sassy CSS) توسع الصيغة الأساسية لـ CSS. هي الصيغة الأكثر استخدامًا في Sass. ملفات SCSS تحمل الامتداد `.scss`.

هنا مثال على تعريف واستخدام متغير في SCSS.

```scss
$primary-color: #3498eb;

header {
  background-color: $primary-color;
}
```

- **الصيغة المتداخلة (Indented syntax)**: كانت الصيغة المتداخلة هي الصيغة الأصلية لـ Sass وتعرف أيضًا باسم "صيغة Sass".

هنا مثال على تعريف واستخدام متغير في الصيغة المتداخلة.

```sass
$primary-color: #3498eb

header
  background-color: $primary-color
```

### الخلائط (Mixins)

- **الخلائط**: تتيح لك الخلائط تجميع عدة خصائص CSS وقيمها تحت اسم واحد وإعادة استخدام هذا الكتلة من الشفرة في جميع أنحاء ملف الأنماط.

هنا مثال على تعريف خليط في صيغة SCSS. في هذه الحالة، الخليط يسمى `center-flex`. يحتوي على ثلاث خصائص CSS لمركزة العناصر باستخدام flexbox.

```scss
@mixin center-flex {
  display: flex;
  justify-content: center;
  align-items: center;
}
```

هنا مثال على استخدام الخليط الذي عرّفته.

```scss
section {
  @include center-flex;
  height: 500px;
  background-color: #3289a8;
}
```

## الاختبار اليدوي والآلي

- **الاختبار اليدوي**: في الاختبار اليدوي، يقوم المختبر بفحص كل جزء من التطبيق يدويًا واختبار الميزات المختلفة للتأكد من عملها بشكل صحيح. إذا تم اكتشاف أخطاء برمجية أثناء الاختبار، يقوم المختبر بإبلاغ فريق البرمجيات لإصلاحها.
- **الاختبار الآلي**: في الاختبار الآلي، يمكنك أتمتة اختباراتك بكتابة برنامج منفصل يتحقق مما إذا كان التطبيق يتصرف كما هو متوقع.

## اختبار الوحدة

- **اختبار الوحدة**: في اختبار الوحدة، تختبر كل دالة للتأكد من أن كل شيء يعمل كما هو متوقع. يمكن لاختبارات الوحدة أن تكون أيضًا شكلًا من أشكال التوثيق لتطبيقك لأنها تمثل السلوك المتوقع للشفرة.
- **مبدأ المسؤولية الواحدة**: ينصح مبدأ المسؤولية الواحدة بالحفاظ على كل دالة صغيرة ومسؤولة عن شيء واحد.
- **أطر اختبار جافا سكريبت الشائعة**: من أطر الاختبار الشائعة Jest و Mocha و Vitest. Jest هو إطار اختبار شائع لاختبارات الوحدة.

هنا مثال على اختبارات الوحدة باستخدام Jest.

أولًا، يمكنك إنشاء دالة مسؤولة عن إرجاع سلسلة نصية مُنسقة حديثًا:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

في ملف منفصل `getFormattedWord.test.js`، يمكنك كتابة بعض الاختبارات للتحقق من أن الدالة تعمل كما هو متوقع. سيبدو ملف `getFormattedWord.test.js` كما يلي:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **دالة `expect`**: تُستخدم دالة `expect` لاختبار قيمة.
- **Matcher**: Matcher هي دالة تتحقق مما إذا كانت القيمة تتصرف كما هو متوقع. في المثال أعلاه، الـ matcher هو `toBe()`. لدى Jest مجموعة متنوعة من matchers.

لاستخدام Jest، تحتاج أولًا إلى تثبيت حزمة `jest` باستخدام الأمر `npm i jest`. ستحتاج أيضًا إلى إضافة سكريبت إلى ملف `package.json` الخاص بك كما يلي:

```js
"scripts": {
  "test": "jest"
},
```

بعد ذلك، يمكنك تشغيل الأمر `npm run test` لتشغيل اختباراتك.

## دورة حياة تطوير البرمجيات

- **مراحل مختلفة من دورة حياة تطوير البرمجيات**:
  - **مرحلة التخطيط**: يجمع فريق التطوير المتطلبات للعمل المقترح من أصحاب المصلحة.
  - **مرحلة التصميم**: يقوم فريق البرمجيات بتفصيل المتطلبات ويقرر أفضل الطرق للحلول.
  - **مرحلة التنفيذ**: يقوم فريق البرمجيات بتقسيم المتطلبات إلى مهام قابلة للإدارة وينفذها.
  - **مرحلة الاختبار**: تشمل الاختبار اليدوي والآلي للعمل الجديد. أحيانًا يختبر الفريق التطبيق طوال مرحلة التطوير للقبض على أي مشاكل وإصلاحها.
  - **مرحلة التشغيل**: ينشر الفريق التغييرات الجديدة إلى بيئة البناء أو الاختبار.
  - **مرحلة الصيانة**: تشمل إصلاح أي مشاكل تظهر من العملاء في التطبيق الإنتاجي.

- **نماذج مختلفة لدورة حياة تطوير البرمجيات**:
  - **نموذج الشلال**: حيث يجب إكمال كل مرحلة من دورة الحياة قبل بدء المرحلة التالية.
  - **نموذج أجايل**: يركز على التطوير التكراري بتقسيم العمل إلى دفعات (sprints).

## BDD و TDD

- **TDD**: التطوير القائم على الاختبار هو منهجية تركز على كتابة الاختبارات أولًا. كتابة الاختبارات قبل بناء الميزات توفر تغذية راجعة فورية للمطورين أثناء عملية التطوير.
- **BDD**: التطوير القائم على السلوك هو نهج لمواءمة سلسلة من الاختبارات مع أهداف العمل. يجب كتابة سيناريوهات الاختبار في BDD بلغة مفهومة لكل من الفنيين وغير الفنيين. مثال على هذه الصيغة هو Gherkin.
- **أطر اختبار BDD**: تشمل أمثلة أطر اختبار BDD Cucumber و JBehave و SpecFlow.

## التحققات في اختبار الوحدة

- **التحقق**: تُستخدم التحققات لاختبار أن الشفرة تتصرف كما هو متوقع.
- **مكتبات التحقق**: Chai هي مكتبة تحقق شائعة الاستخدام. مكتبات تحقق جافا سكريبت أخرى شائعة هي `should.js` و `expect.js`.

هنا مثال على تحقق باستخدام Chai يتأكد من أن قيمة الإرجاع من دالة `addThreeAndFour` تساوي الرقم 7:

```js
assert.equal(addThreeAndFour(), 7);
```

- **أفضل الممارسات**: بغض النظر عن مكتبة التحقق التي تستخدمها، يجب أن تكتب رسائل تحقق وفشل واضحة تساعدك على فهم أي الاختبارات تفشل ولماذا.

## المحاكاة، التزييف، والتثبيت

- **المحاكاة**: المحاكاة هي عملية استبدال البيانات الحقيقية ببيانات زائفة تحاكي سلوك المكونات الحقيقية. على سبيل المثال، يمكنك محاكاة استجابة API في الاختبار بدلاً من إجراء مكالمات API مستمرة لجلب البيانات.
- **التثبيت**: التثبيتات هي كائنات تعيد استجابات محددة مسبقًا أو بيانات وهمية لسلوك متوقع في التطبيق. على سبيل المثال، يمكنك تثبيت سلوك اتصال قاعدة بيانات في اختباراتك دون الاعتماد على اتصال قاعدة بيانات حقيقي.
- **التزييف**: التزييفات هي نسخ مبسطة من المكونات الحقيقية بدون التعقيد أو التأثيرات الجانبية. على سبيل المثال، يمكنك تزييف قاعدة بيانات بتخزين البيانات في الذاكرة بدلاً من التفاعل مع قاعدة البيانات الحقيقية. هذا يسمح لك بمحاكاة عمليات قاعدة البيانات في الذاكرة، مما يكون أسرع بكثير من التعامل مع قاعدة البيانات الحقيقية.

## الاختبار الوظيفي

- **الاختبار الوظيفي**: يتحقق الاختبار الوظيفي مما إذا كانت ميزات ووظائف التطبيق تعمل كما هو متوقع. الهدف من الاختبار الوظيفي هو اختبار النظام ككل ضد سيناريوهات متعددة.
- **الاختبار غير الوظيفي**: يركز على أشياء مثل الأداء والموثوقية.
- **اختبار الدخان**: هو فحص أولي للنظام للتحقق من وجود مشاكل أساسية أو حرجة قبل بدء اختبار أكثر شمولاً.

## اختبار من النهاية إلى النهاية

- **اختبار من النهاية إلى النهاية**: اختبار من النهاية إلى النهاية، أو E2E، يختبر سيناريوهات العالم الحقيقي من منظور المستخدم. تساعد اختبارات E2E على ضمان أن يتصرف تطبيقك بشكل صحيح ومتوقع للمستخدمين. ومع ذلك، فإن إعدادها وتصميمها وصيانتها يستغرق وقتًا.
- **أطر اختبار من النهاية إلى النهاية**: Playwright هو إطار اختبار من النهاية إلى النهاية شهير طورته مايكروسوفت. أمثلة أخرى لأدوات اختبار من النهاية إلى النهاية تشمل Cypress و Selenium و Puppeteer.

هنا مثال على اختبارات E2E من قاعدة شفرة freeCodeCamp باستخدام Playwright. سيعمل الـ `beforeEach` hook قبل كل اختبار. تتحقق الاختبارات من أن المتبرع لديه رابط داعم في شريط القائمة، بالإضافة إلى حدود خاصة مزخرفة حول الصورة الرمزية:

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## اختبار سهولة الاستخدام

- **اختبار سهولة الاستخدام**: هو عندما يتفاعل المستخدمون الحقيقيون مع التطبيق لاكتشاف ما إذا كانت هناك أي مشاكل في التصميم أو تجربة المستخدم أو الوظائف في التطبيق. يركز اختبار سهولة الاستخدام على مدى سهولة استخدام التطبيق من قبل المستخدمين.
- **أربعة أنواع شائعة من اختبار سهولة الاستخدام**:
  - **استكشافي**: يتضمن تفاعل المستخدمين مع الميزات المختلفة للتطبيق لفهم كيفية عملها بشكل أفضل.
  - **مقارن**: حيث تقارن تجربة المستخدم في تطبيقك مع تطبيقات مماثلة في السوق.
  - **تقييمي**: حيث تدرس مدى سهولة استخدام التطبيق.
  - **تحققي**: حيث تحدد أي مشاكل كبيرة قد تمنع المستخدم من استخدام التطبيق بفعالية.
- **أدوات اختبار سهولة الاستخدام**: تشمل أمثلة الأدوات Loop11 و Maze و Userbrain و UserTesting و UXTweak.

## اختبار التوافق

- **اختبار التوافق**: الهدف من اختبار التوافق هو التأكد من أن تطبيقك يعمل في بيئات حوسبة مختلفة.
- **أنواع مختلفة من اختبار التوافق**:
  - **التوافق مع الإصدارات السابقة**: يشير إلى أن البرنامج متوافق مع الإصدارات السابقة.
  - **التوافق مع الإصدارات المستقبلية**: يشير إلى أن البرنامج والأنظمة ستعمل مع الإصدارات المستقبلية.
  - **توافق الأجهزة**: قدرة البرنامج على العمل بشكل صحيح في تكوينات أجهزة مختلفة.
  - **توافق أنظمة التشغيل**: قدرة البرنامج على العمل على أنظمة تشغيل مختلفة مثل macOS و Windows وتوزيعات Linux مثل Ubuntu و Fedora.
  - **توافق الشبكة**: يعني أن البرنامج يمكن أن يعمل في ظروف شبكة مختلفة مثل سرعات الشبكة المختلفة، البروتوكولات، إعدادات الأمان، إلخ.
  - **توافق المتصفح**: يعني أن تطبيق الويب يمكن أن يعمل بشكل متسق عبر متصفحات مختلفة مثل Google Chrome و Safari و Firefox.
  - **التوافق مع الأجهزة المحمولة**: من المهم التأكد من أن تطبيقاتك البرمجية تعمل على مجموعة متنوعة من أجهزة Android و iOS، بما في ذلك الهواتف والأجهزة اللوحية.

## اختبار الأداء

- **اختبار الأداء**: في اختبار الأداء، تختبر سرعة التطبيق، استجابته، قابليته للتوسع، واستقراره تحت أحمال عمل مختلفة. الهدف هو حل أي عنق زجاجة في الأداء.
- **أنواع مختلفة من اختبار الأداء**:
  - **اختبار التحميل**: يحدد كيف يتصرف النظام خلال أوقات الحمل العادي والذروة.
  - **اختبار الضغط**: تختبر تطبيقك تحت أحمال قصوى وترى كيف يستجيب النظام للحمل العالي.
  - **اختبار التحمل (اختبار النقع)**: هو نوع من اختبار التحميل حيث تختبر النظام بحمل مرتفع لفترة ممتدة.
  - **اختبار الذروة**: حيث تزيد وتنقص الأحمال بشكل كبير وتحلل ردود فعل النظام على التغييرات.
  - **اختبار نقطة الانهيار (اختبار السعة)**: حيث تزيد الحمل ببطء مع مرور الوقت حتى يبدأ النظام في الفشل أو التدهور.

## اختبار الأمان

- **اختبار الأمان**: يساعد اختبار الأمان في تحديد الثغرات والضعف.
- **مبادئ الأمان**:
  - **السرية**: تحمي من تسريب المعلومات الحساسة إلى مستلمين غير مقصودين.
  - **السلامة**: تمنع المستخدمين الخبيثين من تعديل معلومات المستخدم.
  - **التوثيق**: يتحقق من هوية المستخدم لضمان السماح له باستخدام النظام.
  - **التفويض**: هي عملية تحديد الإجراءات التي يُسمح للمستخدمين الموثقين بأدائها أو الأجزاء التي يمكنهم الوصول إليها في النظام.
  - **التوفر**: يضمن توفر المعلومات والخدمات للمستخدمين المصرح لهم عند الحاجة.
  - **عدم التنصل**: يضمن أن كل من المرسل والمستلم لديهما إثبات التسليم والتحقق من هوية المرسل. يحمي من إنكار المرسل لإرسال المعلومات.
- **التهديدات الأمنية الشائعة**:
  - **البرمجة النصية عبر المواقع (XSS)**: تحدث هجمات XSS عندما يحقن المهاجمون نصوصًا خبيثة في صفحة ويب ثم ينفذونها في سياق متصفح الضحية.
  - **حقن SQL**: يسمح للمستخدمين الخبيثين بحقن شفرة خبيثة في قاعدة البيانات.
  - **هجوم حجب الخدمة (DoS)**: هو عندما يفيض المستخدمون الخبيثون موقع ويب بعدد كبير من الطلبات أو الحركة، مما يبطئ الخادم وربما يتسبب في تعطل الموقع، مما يجعله غير متاح للمستخدمين.
- **فئات أدوات اختبار الأمان**:
  - **اختبار أمان التطبيقات الثابت (SAST)**: تقيم هذه الأدوات شفرة المصدر لتطبيق لتحديد الثغرات الأمنية.
  - **اختبار أمان التطبيقات الديناميكي (DAST)**: تتفاعل هذه الأدوات مع واجهة التطبيق الأمامية لكشف نقاط الضعف الأمنية المحتملة. لا تمتلك أدوات DAST وصولًا إلى شفرة المصدر.
- **اختبار الاختراق (pentest)**: هو نوع من اختبار الأمان يتضمن إنشاء هجمات إلكترونية محاكاة على التطبيق لتحديد أي ثغرات في النظام.

## اختبار A/B

- **اختبار A/B**: يتضمن اختبار A/B مقارنة نسختين من صفحة أو تطبيق ودراسة أي نسخة تؤدي أداءً أفضل. يُعرف أيضًا باسم اختبار الدلو أو التقسيم. يسمح اختبار A/B باتخاذ قرارات أكثر اعتمادًا على البيانات وتحسين تجربة المستخدم باستمرار.
- **أدوات اختبار A/B**: تشمل أمثلة الأدوات GrowthBook و LaunchDarkly.

## اختبار ألفا وبيتا

بعد الانتهاء من التطوير الأولي واختبار البرمجيات، من المهم أن يتم اختبار التطبيق بواسطة مختبرين ومستخدمين حقيقيين. هنا يأتي دور اختبار ألفا وبيتا.

- **اختبار ألفا**: يتم بواسطة مجموعة مختارة من المختبرين الذين يراجعون التطبيق للتأكد من عدم وجود أخطاء قبل إصداره في السوق. اختبار ألفا هو جزء من اختبار القبول ويستخدم تقنيات الصندوق الأبيض والأسود.
- **اختبار بيتا**: حيث يتم إتاحة التطبيق للمستخدمين الحقيقيين. يمكن للمستخدمين التفاعل مع التطبيق وتقديم الملاحظات. اختبار بيتا هو أيضًا شكل من أشكال اختبار قبول المستخدم.
- **اختبار القبول**: يضمن أن تطبيق البرمجيات يلبي متطلبات العمل واحتياجات المستخدمين قبل إصداره.
- **اختبار الصندوق الأسود**: يركز فقط على السلوك المتوقع للتطبيق.
- **اختبار الصندوق الأبيض**: يتضمن معرفة المختبر بالمكونات الداخلية وإجراء اختبارات عليها.

## اختبار الانحدار

- **الانحدار**: يشير إلى الحالات التي تكسر فيها التغييرات الجديدة الوظائف الحالية عن غير قصد.
- **اختبار الانحدار**: يساعد اختبار الانحدار في اكتشاف مشاكل الانحدار. في اختبار الانحدار، تعيد تشغيل الاختبارات الوظيفية على أجزاء من تطبيقك للتأكد من أن كل شيء لا يزال يعمل كما هو متوقع.
- **أدوات اختبار الانحدار**: تشمل الأدوات التي يمكنك استخدامها لإجراء اختبار الانحدار Puppeteer و Playwright و Selenium و Cypress.
- **تقنيات اختبار الانحدار**:
  - **اختبار انحدار الوحدة**: حيث يكون لديك قائمة بالعناصر التي يجب اختبارها في كل مرة يتم فيها تنفيذ تغييرات أو إصلاحات كبيرة في التطبيق.
  - **اختبار انحدار جزئي**: يتضمن نهجًا مستهدفًا لضمان أن التغييرات الجديدة لم تكسر جوانب محددة من التطبيق.
  - **اختبار انحدار كامل**: يشغل اختبارات على جميع الوظائف في قاعدة الشفرة. هذا هو الخيار الأكثر استهلاكًا للوقت والأكثر تفصيلاً.
- **إعادة الاختبار**: تُستخدم للتحقق من المشكلات المعروفة والتأكد من حلها. بالمقابل، يبحث اختبار الانحدار عن مشكلات غير معروفة قد تكون حدثت بسبب تغييرات حديثة في قاعدة الشفرة.

## ما هو TypeScript

- **JavaScript**: جافا سكريبت هي لغة ذات نوع ديناميكي. هذا يعني أن المتغيرات يمكن أن تستقبل أي قيم أثناء وقت التشغيل. التحدي في لغة ذات نوع ديناميكي هو أن نقص أمان النوع يمكن أن يسبب أخطاء.

على سبيل المثال، حتى لو كانت دالة جافا سكريبت الخاصة بك تتوقع مصفوفة، يمكنك استدعاؤها برقم:

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

الناتج في `console` للمثال أعلاه سيكون `undefined`.

- **TypeScript**: TypeScript توسع لغة جافا سكريبت لتشمل النوع الثابت، مما يساعد في اكتشاف الأخطاء الناتجة عن عدم تطابق الأنواع قبل تشغيل الشفرة.

على سبيل المثال، يمكنك تعريف نوع للمعلمة `array` كما يلي:

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

تعريف النوع هذا يخبر TypeScript أن المعلمة `array` يجب أن تكون مصفوفة من سلاسل نصية. ثم، عندما تستدعي `getRandomValue` وتمرر لها رقمًا، تحصل على خطأ يسمى خطأ المترجم.

- **المترجم**: تحتاج أولًا إلى ترجمة شفرة TypeScript إلى جافا سكريبت عادية. عند تشغيل المترجم، يقوم TypeScript بتقييم الشفرة ويرمي خطأ لأي مشاكل في عدم تطابق الأنواع.

## أنواع البيانات في TypeScript

- **أنواع البيانات الأولية في TypeScript**: لأنواع البيانات الأولية `string` و `null` و `undefined` و `number` و `boolean` و `bigint`، يقدم TypeScript كلمات مفتاحية مقابلة للنوع.

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **المصفوفة**: يمكنك تعريف مصفوفة من نوع محدد بصيغتين مختلفتين.

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **الكائنات**: يمكنك تعريف الهيكل الدقيق لكائن.

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

إذا أردت كائنًا بأي مفاتيح، لكن حيث يجب أن تكون كل القيم سلاسل نصية، هناك طريقتان لتعريفه:

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **أنواع أخرى مفيدة في TypeScript**:
  - **`any`**: `any` تشير إلى أن القيمة يمكن أن تكون من أي نوع. تخبر المترجم بالتوقف عن الاهتمام بنوع ذلك المتغير.
  - **`unknown`**: `unknown` تخبر TypeScript بأنك تهتم بنوع القيمة، لكنك لا تعرف ما هو فعليًا. `unknown` مفضل عمومًا على `any`.
  - **`void`**: هذا نوع خاص تستخدمه عادة عند تعريف الدوال. الدوال التي لا ترجع قيمة تستخدم نوع إرجاع `void`.
  - **`never`**: يمثل نوعًا لن يوجد أبدًا.

- **كلمة `type` المفتاحية**: هذه الكلمة تشبه `const`، لكن بدلاً من إعلان متغير، يمكنك إعلان نوع.

مفيدة لإعلان أنواع مخصصة مثل أنواع الاتحاد أو الأنواع التي تشمل فقط قيمًا محددة:

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**: الواجهات تشبه الأصناف للأنواع. يمكنها تنفيذ أو توسيع واجهات أخرى، وهي أنواع كائنات بشكل خاص، وعادة ما تُفضل إلا إذا كنت بحاجة إلى ميزة محددة تقدمها إعلان `type`.

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **تعريف نوع الإرجاع**: يمكنك أيضًا تعريف _نوع الإرجاع_ للدالة.

المثال أدناه يحدد قيمة الإرجاع كسلسلة نصية. إذا حاولت إرجاع أي شيء آخر، سيعطي TypeScript خطأ مترجم.

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```

## الأنواع العامة (Generics)

- **تعريف النوع العام**: يمكنك تعريف نوع عام والإشارة إليه في دالتك. يمكن اعتباره معلمة خاصة تقدمها لدالة للتحكم في سلوك تعريف نوع الدالة.

هنا مثال على تعريف نوع عام لدالة:

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

صيغة `<T>` تخبر TypeScript بأنك تعرف نوعًا عامًا `T` للدالة. `T` هو اسم شائع للأنواع العامة، لكن يمكنك استخدام أي اسم.

ثم تخبر TypeScript أن المعلمة `array` هي مصفوفة من القيم التي تطابق النوع العام، وأن القيمة المرجعة هي عنصر واحد من نفس النوع.

- **تحديد وسيط النوع في استدعاء الدالة**: يمكنك تمرير وسيط نوع لاستدعاء دالة باستخدام الأقواس الزاوية بين اسم الدالة ومعلماتها.

هنا مثال على تمرير وسيط نوع لاستدعاء دالة:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

هذا يخبر TypeScript أن العنصر الذي تتوقع العثور عليه سيكون عنصر إدخال.

## تضييق النوع (Type Narrowing)

- **التضييق بالصدق**: في المثال أدناه، تحصل على خطأ مترجم عند محاولة الوصول إلى خاصية `value` لـ `email` لأن `email` قد تكون `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

يمكنك استخدام شرط للتأكد من أن `email` _صحيحة_ قبل الوصول إلى الخاصية:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

يمكن أن تعمل فحوصات الصدق أيضًا في الاتجاه المعاكس:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

رمي خطأ ينهي التنفيذ المنطقي لهذا الكود، مما يعني أنه عند الوصول إلى استدعاء `console.log()`، يعرف TypeScript أن `email` _لا يمكن أن تكون_ `null`.

- **التسلسل الاختياري**: التسلسل الاختياري `?.` هو أيضًا شكل من أشكال تضييق النوع، تحت نفس الفرضية بأن الوصول إلى الخاصية لا يمكن أن يحدث إذا كانت قيمة `email` هي `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **عامل `typeof`**: يمكنك استخدام شرط للتحقق من نوع المتغير باستخدام عامل `typeof`.

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **كلمة `instanceof` المفتاحية**: إذا كان الكائن يأتي من صنف، يمكنك استخدام كلمة `instanceof` لتضييق النوع.

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **التحويل (Casting)**: عندما لا يستطيع TypeScript تحديد نوع القيمة تلقائيًا، مثل نتيجة طريقة `request.json()` في المثال أدناه، ستحصل على خطأ مترجم. إحدى الطرق لحل هذا هي تحويل النوع، لكن هذا يضعف قدرة TypeScript على اكتشاف الأخطاء المحتملة.

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **حارس النوع (Type Guard)**: بدلاً من تحويل النوع، يمكنك كتابة حارس نوع:

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

صيغة `user is User` تشير إلى أن دالتك ترجع قيمة منطقية، والتي عندما تكون صحيحة تعني أن قيمة `user` تحقق واجهة `User`.

## ملف `tsconfig`

- **`tsconfig.json`**: إعدادات مترجم TypeScript موجودة في ملف `tsconfig.json` في الدليل الجذري لمشروعك.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

هنا أوصاف لخيارات المترجم المستخدمة في المثال أعلاه:

- **`compilerOptions`**: خاصية `compilerOptions` هي حيث تتحكم في سلوك مترجم TypeScript.
- **`rootDir` و `outDir`**: تخبر `rootDir` و `outDir` TypeScript أي دليل يحتوي على ملفات المصدر، وأي دليل يجب أن يحتوي على شفرة جافا سكريبت المترجمة.
- **`lib`**: تحدد خاصية `lib` تعريفات النوع التي يستخدمها المترجم، وتسمح لك بتضمين دعم لإصدارات ES محددة، و DOM، وأكثر.
- **`module` و `moduleResolution`**: تعمل `module` و `moduleResolution` معًا لإدارة كيفية استخدام حزمتك للوحدات - إما CommonJS أو ECMAScript.
- **`esModuleInterop`**: يسمح `esModuleInterop` بتشغيل أكثر سلاسة بين CommonJS و ES modules عن طريق إنشاء كائنات مساحة أسماء تلقائيًا للاستيراد.
- **`skipLibCheck`**: يتخطى خيار `skipLibCheck` التحقق من ملفات `.d.ts` التي لا يتم الإشارة إليها بواسطة الاستيرادات في شفرتك.
- **`strict`**: يفعّل العلم `strict` عدة فحوصات، مثل ضمان التعامل الصحيح مع الأنواع القابلة لأن تكون فارغة وتحذير عندما يعود TypeScript إلى `any`.
- **`exclude`**: خاصية `exclude` على المستوى الأعلى تخبر المترجم بتجاهل هذه الملفات TypeScript أثناء الترجمة.
