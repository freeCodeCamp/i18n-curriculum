# --description--

CI steht für Continuous Integration, und CD steht für Continuous Delivery oder Continuous Deployment. Zusammen ermöglichen diese Prozesse, Änderungen an Ihrer Anwendung automatisch bereitzustellen, nachdem sichergestellt wurde, dass diese Änderungen funktionieren.

In einer CI-Pipeline führen Sie möglicherweise Ihren Linter und Tests gegen den Code aus. CI-Pipelines laufen typischerweise bei Pull Requests, um sicherzustellen, dass die vorgenommenen Änderungen die Anwendung nicht beschädigen, und auch auf dem `main`-Branch, um sicherzustellen, dass dieser bereit für das Deployment ist.

Eine CD-Pipeline kann ebenfalls bei Pull Requests laufen, wenn Sie eine Vorschauumgebung einrichten, damit Sie die Änderungen live ansehen können. Aber Sie führen sie auch auf Ihrem `main`-Branch aus, um Änderungen kontinuierlich bereitzustellen, sobald sie akzeptiert und zusammengeführt werden.

Aber wie funktionieren diese Prozesse eigentlich mit GitHub Actions? Zuerst müssen wir verstehen, was GitHub Actions überhaupt sind. GitHub Actions stellt temporäre Runner bereit (denken Sie an einen temporären Computer), die es Ihnen ermöglichen, bestimmte Befehle auszuführen.

Actions werden mit einer YAML-Datei konfiguriert. Sie können mehrere Dateien haben, um verschiedene Aktionen auszuführen. Zum Beispiel könnten Sie eine Datei für Ihre CI und eine andere für Ihre CD haben.

Schauen wir uns ein einfaches CI-Beispiel an:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  lint:
    name: Lint and Test
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Source Files
        uses: actions/checkout@v4

      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10

      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

Das ist viel, also brechen wir es herunter. Schauen wir uns den ersten Abschnitt an:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```

Das `name` definiert den Namen der Aktion selbst. Dieser erscheint in Bereichen wie dem Statusprüfungsabschnitt eines PR. Die `on`-Eigenschaften definieren sogenannte „Workflow-Trigger“ – das sind die Ereignisse, die Ihre Aktion auslösen.

In diesem Beispiel führen wir die Aktion aus, wenn Commits auf `main` gepusht werden und wenn ein Pull Request auf `main` abzielt. Das bedeutet, die CI läuft, wenn jemand einen Pull Request erstellt, und erneut, wenn wir ihn zusammenführen.

Der `jobs`-Abschnitt ist der Ort, an dem Sie definieren, was Ihre Aktion tatsächlich macht. Der `lint:`-Schlüssel ist ein beliebiger Schlüssel, der den internen Namen für diesen Job definiert. Die `name`-Eigenschaft definiert den externen Namen für diesen speziellen Job. Und die `runs-on`-Eigenschaft definiert, in welcher Umgebung Ihre Aktion ausgeführt werden soll – in diesem Fall läuft sie in der neuesten unterstützten Version von Ubuntu.

Die `steps`-Eigenschaft ist der Ort, an dem die Magie passiert. Dieser Block definiert die tatsächlichen Schritte, die Ihre Aktion ausführen soll. Schauen wir uns den ersten Block dort an:

```yml
      - name: Checkout Source Files
        uses: actions/checkout@v4
```

Dieser Schritt hat zwei Eigenschaften, die `name`, die der Schritt haben soll, und eine `uses`-Eigenschaft. Diese `uses`-Eigenschaft ist besonders, da sie dem Action-Runner sagt, eine öffentliche Aktion von GitHub zu holen und auszuführen, anstatt einen Befehl auszuführen. Die `actions/checkout`-Aktion kümmert sich um das Klonen und Einrichten des Repositories innerhalb Ihres Runners.

```yml
      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10
```

Die nächsten zwei Blöcke verwenden ebenfalls Drittanbieter-Aktionen. Diese Schritte richten Node und den pnpm-Paketmanager ein. Beide haben einen `with`-Schlüssel, mit dem Sie Argumente an diese Drittanbieter-Aktionen übergeben können. Denken Sie daran wie das Übergeben von Argumenten an eine Funktion.

Die letzten vier Blöcke sind der Ort, an dem die Magie wirklich passiert.

```yml
      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

Wir haben einen `Install Dependencies`-Block, der die npm-Pakete installiert. Unser `Lint Source Files`-Block führt den Linter aus, der `Verify Build`-Block stellt sicher, dass der TypeScript-Code kompiliert werden kann, und der `Run Tests`-Block führt die Unit-Tests aus.

Wichtig zu wissen bei Actions ist, dass standardmäßig, wenn einer dieser Schritte fehlschlägt, die verbleibenden übersprungen werden und der gesamte Lauf als Fehler markiert wird.

Wo legen Sie Ihre YAML-Dateien ab, damit Sie diese Aktionen ausführen können? Sie müssen in einem `.github/workflows`-Verzeichnis in Ihrem Repository liegen, damit GitHub sie finden und verwenden kann.

Es gibt viele weitere Dinge, die Actions tun können, und ich empfehle Ihnen, die Dokumentation zu lesen und bestehende Actions in Projekten wie dem freeCodeCamp-Repository zu erkunden. Aber für den Moment sollten Sie ein gutes Verständnis der Grundlagen von GitHub Actions und CI/CD haben!
