## 보안과 개인정보 보호

**보안**은 무단 접근, 도난, 악의적 손상으로부터 데이터를 보호합니다. 방화벽, 암호화, 안티바이러스 소프트웨어 같은 기술적 방어 수단을 구축해 악성 행위자가 시스템에 침입하지 못하게 합니다.

**개인정보 보호**는 조직이 개인 데이터를 수집, 저장, 사용, 공유하는 방식을 통제합니다. 사용자가 자신의 정보가 누구에게 보여질지 결정할 수 있는 권한을 제공합니다.

이 두 개념은 밀접하게 관련되어 있지만 구분됩니다:

- 보안은 있지만 개인정보 보호가 없을 수 있습니다. 회사가 데이터를 매우 안전한 데이터베이스에 저장하면서도 사용자의 동의 없이 광고주에게 판매할 수 있습니다. 이 경우 보안은 유지되지만 개인정보 보호는 침해됩니다.
- 보안 없이는 진정한 개인정보 보호가 불가능합니다. 개인 데이터를 보유한 서버가 침해되면 개인정보 보호 약속은 즉시 깨집니다.

보안은 기술적 방어 메커니즘을 제공하고, 개인정보 보호는 데이터 사용에 관한 법적 권리와 정책을 관리합니다.

## HTTPS

**HTTP**는 데이터를 평문으로 전송해 가로채는 사람이 내용을 읽을 수 있습니다. **HTTPS**(Hypertext Transfer Protocol Secure)는 **TLS (Transport Layer Security)**를 사용해 브라우저와 서버 간 통신 채널을 암호화합니다. TLS는 이전의 **SSL (Secure Sockets Layer)** 프로토콜을 대체했지만, 많은 웹 호스팅 플랫폼은 여전히 설정에서 SSL을 언급합니다.

HTTPS의 주요 이점:

- **암호화:** 데이터를 읽을 수 없는 코드로 변환합니다. 복호화 키는 브라우저와 서버만 보유합니다.
- **인증:** 사이트 소유자가 신뢰할 수 있는 기관에서 디지털 인증서를 받아 웹사이트의 정당성을 증명해야 합니다.
- **데이터 무결성:** 전송 중 데이터 변경을 막아 서버가 보낸 그대로 받게 합니다.
- **공용 Wi-Fi 보호:** 개방된 네트워크를 통해 개인 터널을 만들어 인근 공격자로부터 트래픽을 숨깁니다.

최신 브라우저는 평문 HTTP 사이트에 경고를 표시하고, 지리 위치 및 푸시 알림 같은 고급 기능에 HTTPS를 요구합니다.

## 동일 출처 정책과 CORS

**동일 출처 정책(SOP)**은 브라우저 보안 규칙으로, 웹페이지가 다른 출처의 데이터를 읽지 못하게 합니다. **출처**는 세 부분이 모두 정확히 일치해야 합니다:

- **프로토콜** (예: `https`)
- **도메인 이름** (예: `example.com`)
- **포트 번호** (예: `443` HTTPS 기본값, `80` HTTP 기본값)

이 정책은 한 브라우저 탭에서 열린 악성 사이트가 다른 탭에서 열린 은행 사이트의 데이터를 훔치는 것을 막습니다.

**CORS (Cross-Origin Resource Sharing)**는 서버가 HTTP 헤더를 사용해 통제된 교차 출처 데이터 공유에 참여하도록 허용합니다:

- 브라우저는 모든 교차 출처 요청에 `Origin` 헤더를 보냅니다.
- 서버가 해당 출처를 신뢰하면 `Access-Control-Allow-Origin` 헤더로 응답합니다.
- 브라우저는 응답 데이터를 읽을 수 있게 허용하거나 차단합니다.

Express에서는 `cors` 패키지가 이를 처리합니다:

```javascript
// development: allow all origins
app.use(cors());

// production: restrict to trusted domains
app.use(cors({ origin: ['https://yourapp.com', 'https://admin.yourapp.com'], credentials: true }));
```

## 쿠키: 저장, 보안, 개인정보 보호

**쿠키**는 웹사이트가 방문 간 정보를 기억하기 위해 기기에 저장하는 작은 텍스트 파일입니다. 서버가 `Set-Cookie` 헤더를 보내면 브라우저가 파일을 저장하고 이후 모든 요청에 자동으로 다시 보냅니다.

쿠키 보호를 위한 보안 플래그:

- **`HttpOnly`:** JavaScript가 쿠키를 읽지 못하게 막아 XSS를 통한 세션 토큰 탈취를 방지합니다.
- **`Secure`:** 쿠키가 HTTPS를 통해서만 전송되도록 하여 공용 Wi-Fi에서 가로채기를 막습니다.
- **`SameSite=Strict`:** 브라우저가 모든 교차 사이트 요청에 쿠키를 보내지 못하게 하여 CSRF 공격을 차단합니다.
- **`SameSite=Lax`:** 안전한 최상위 탐색에서는 쿠키를 허용하지만 교차 사이트 폼 제출과 API 요청에서는 차단합니다.

**개인정보 보호 영향:**

- **퍼스트파티 쿠키**는 방문 중인 웹사이트 내에서만 사용자의 행동을 추적합니다.
- **서드파티 쿠키**는 광고 네트워크가 설치하며 전혀 다른 웹사이트들에서 사용자를 추적해 상세한 관심사 프로필을 만듭니다.

GDPR 같은 개인정보 보호법은 웹사이트가 쿠키 동의 배너를 표시하고 사용자가 추적 쿠키를 거부할 권리를 부여하도록 요구합니다. 최신 브라우저는 기본적으로 서드파티 추적 쿠키를 차단합니다.

## 보안 문제가 발생하는 상황

보안 취약점은 우연히 발생하는 경우가 드뭅니다. 흔한 유발 상황은 다음과 같습니다:

- **빠른 출시 압박:** 개발팀이 엄격한 마감일을 맞추기 위해 철저한 코드 검토와 보안 테스트를 건너뛰어 미패치 버그와 누락된 접근 제어가 남습니다.
- **복잡한 시스템 통합:** 외부 라이브러리, 서드파티 API, 마이크로서비스마다 공급망 위험이 추가됩니다. 한 의존성의 취약점이 메인 애플리케이션으로 확산됩니다.
- **부실한 구성 관리:** 기본 공장 설정으로 서버나 클라우드 데이터베이스를 배포하면 빈 관리자 비밀번호, 열린 포트, 공개 데이터 접근이 남아 자동 스캐너가 이를 찾아냅니다.
- **인간 오류:** 피싱 이메일, 사기 SMS, 속임수 전화에 노출된 직원이 자격 증명을 노출하거나 데이터를 공격자에게 전달할 수 있습니다.
- **조직 합병:** 서로 다른 기업 네트워크를 빠르게 연결하면 오래된 서버에 대한 가시성 상실, 잘못된 방화벽 설정, 과도한 접근 권한 같은 사각지대가 생깁니다.

## 일반적인 보안 위협과 대응책

| 위협 | 작동 방식 | 대응책 |
|---|---|---|
| **피싱** | 신뢰받는 브랜드를 가장한 사기 이메일로 자격 증명을 훔침 | 보안 인식 교육; 이메일 인증 프로토콜(SPF, DKIM, DMARC) |
| **악성코드 (랜섬웨어, 스파이웨어)** | 피해자 파일을 암호화하거나 훔치고 금전을 요구함 | 자동 패치; 안티멀웨어 소프트웨어 |
| **자격 증명 대입 공격** | 수백만 개의 도난당한 사용자명/비밀번호 조합을 자동으로 테스트함 | 다중 인증(MFA) 적용 |
| **DDoS** | 가짜 트래픽으로 서버를 과부하 시켜 다운시킴 | 클라우드 기반 트래픽 필터링 서비스(예: Cloudflare) |
| **내부자 위협** | 불만 또는 부주의한 직원이 데이터를 유출하거나 훔침 | 최소 권한 원칙; 비정상 접근에 대한 시스템 로그 모니터링 |

## 콘텐츠 보안 정책(CSP)과 권한 정책

**콘텐츠 보안 정책(CSP)**은 브라우저가 페이지에서 로드할 수 있는 외부 리소스를 제한하는 HTTP 응답 헤더입니다. 주로 **교차 사이트 스크립팅(XSS)** 공격을 막기 위해 스크립트, 스타일, 이미지에 대해 신뢰할 수 있는 출처 도메인을 화이트리스트로 지정합니다.

승인되지 않은 도메인의 악성 스크립트가 실행되면 브라우저가 이를 차단하고 보안 오류를 기록합니다.

**권한 정책**(이전 명칭 Feature-Policy)은 웹캠, 마이크, 지리 위치 같은 브라우저 기능과 장치 하드웨어 접근을 제어합니다. 기능을 자신의 도메인으로 제한하거나 완전히 비활성화할 수 있습니다:

```http
Permissions-Policy: camera=(self), geolocation=()
```

`camera=(self)`: 자신의 도메인만 카메라에 접근할 수 있습니다.
`geolocation=()`: 페이지 내 모든 사용자에 대해 지리 위치 기능이 비활성화됩니다.

CSP는 데이터 전달을 잠그고 코드 주입을 방지하며, 권한 정책은 장치 하드웨어를 보호하고 개인정보 보호 준수를 강제합니다.

## 사용자 개인정보의 기본 개념

| 개념 | 정의 |
|---|---|
| **PII (개인 식별 정보)** | 개인을 직접 또는 간접적으로 식별할 수 있는 모든 데이터. 직접 식별자: 이름, 주소, ID 번호. 간접 식별자: IP 주소, 장치 일련번호, 위치 기록. |
| **기밀성** | 암호화와 접근 제어를 통해 무단 당사자가 개인 데이터를 접근하지 못하게 보장함. |
| **데이터 최소화** | 특정 작업을 완료하는 데 필요한 최소한의 PII만 수집함. |
| **목적 제한** | 한 가지 이유로 수집한 데이터는 법적으로 다른 용도로 사용할 수 없음(예: 연락처 정보를 광고주에게 판매). |
| **추적** | 스크립트, 장치 지문, 브라우저 기록 로그를 사용해 온라인 활동을 지속적으로 모니터링하며 사용자 프로필을 구축함. |
| **정보에 입각한 동의** | 사용자가 데이터 수집에 적극적으로 동의해야 하며, 미리 체크된 박스는 유효한 동의가 아님. |
| **잊힐 권리** | 사용자가 조직에 자신의 모든 개인 데이터와 추적 기록을 영구 삭제하도록 법적으로 요청할 수 있음. |

## 지역별 개인정보 보호법

| 법률 | 관할 구역 | 주요 요구사항 |
|---|---|---|
| **GDPR** | 유럽 연합 | 엄격한 옵트인 동의 요구; 사용자는 데이터 접근 및 삭제 권리 보유; 위반 시 무거운 벌금 부과. |
| **CCPA / CPRA** | 미국 캘리포니아 | 옵트아웃 모델; 기업은 "내 개인 정보 판매 금지" 링크를 표시해야 함. |
| **COPPA** | 미국(연방) | 13세 미만 아동 데이터 수집 전 검증된 부모 동의 필요. |
| **DPA / UK GDPR** | 영국 | 합법적이고 공정하며 투명한 데이터 처리; 개인은 접근 및 삭제 권리 보유. |
| **NDPA** | 나이지리아 | 나이지리아 거주자의 개인 데이터 수집, 저장, 사용을 규제. |
| **DPDP** | 인도 | 인도 거주자의 개인 데이터 사용 방식을 규제. |

**실무 준수 단계:**

1. **위치 인식 동의 배너:** 사용자의 지리적 위치를 감지해 지역 규정에 맞는 쿠키 배너를 표시하세요.
2. **자동 DSAR 시스템:** 데이터 주체 접근 요청을 처리해 사용자의 전체 데이터 기록을 안전하게 내보내거나 삭제하세요.
3. **데이터 최소화 및 연령 확인:** 필요한 데이터만 수집하고 미성년자 보호를 위해 연령 확인을 사용하세요.
4. **개인정보 보호 책임자(DPO):** 개인정보 보호 아키텍처를 감독하고 규제 준수를 관리할 전문가를 지정하세요.

## `.env` 파일과 `dotenv` 라이브러리

**`.env` 파일**은 프로젝트 루트에 위치한 평문 텍스트 파일로, API 키, 데이터베이스 비밀번호, 포트 번호 같은 민감한 설정 데이터를 키-값 쌍으로 저장합니다. 소스 코드와 공개 저장소에서 이를 분리해 보관합니다.

```bash
PORT=3000
DB_HOST=localhost
API_KEY=xyz123secretkey
```

`.env` 파일을 `.gitignore`에 항상 추가해 버전 관리에 절대 커밋되지 않도록 하세요.

**`dotenv` 패키지**는 시작 시 `.env` 파일을 읽어 키를 `process.env`에 주입합니다:

```javascript
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

두 가지 주요 이점:

- **보안:** 민감한 자격 증명이 소스 코드나 버전 기록에 절대 나타나지 않습니다.
- **유연성:** 각 서버에서 `.env` 파일만 교체해 개발, 테스트, 운영 환경에서 동일한 코드베이스를 실행할 수 있습니다.
