# --description--

تُشغّل خدمات الويب وواجهات برمجة التطبيقات REST معظم التطبيقات والمواقع التي تستخدمها يوميًا. تتيح هذه الخدمات للتطبيقات المختلفة التواصل مع بعضها البعض ومشاركة البيانات عبر شبكة، وغالبًا ما تكون الإنترنت.

واجهة برمجة التطبيقات (API) هي طريقة عامة لتمكين البرامج من التواصل مع برامج أخرى. خدمة الويب هي نوع محدد من واجهات برمجة التطبيقات يستخدم تقنيات الويب، مثل HTTP، لتبادل البيانات عبر شبكة، عادةً الإنترنت. لذلك، كل خدمة ويب هي واجهة برمجة تطبيقات، لكن ليس كل واجهة برمجة تطبيقات هي خدمة ويب. تُعد واجهات برمجة التطبيقات REST نوعًا شائعًا من خدمات الويب تتبع مجموعة محددة من القواعد لكيفية طلب البيانات وإرجاعها.

يركز هذا الدرس على ماهية خدمات الويب بشكل عام، والدرس التالي سيتناول تفاصيل أكثر عن كيفية عمل واجهات برمجة التطبيقات REST.

يمكنك التفكير في خدمة الويب كجسر بين برنامج وآخر. لا تحتوي على واجهة مستخدم وليست مخصصة للاستخدام المباشر. بدلاً من ذلك، تركز بالكامل على تبادل البيانات بين التطبيقات.

لكي يُعتبر أي نظام خدمة ويب، يجب أن تتوفر فيه الخصائص التالية:

* **تشغيل على خادم**: يجب استضافة خدمة الويب عبر الإنترنت مثل موقع ويب، ولكن بدلاً من صفحات يراها المستخدمون، تُتيح بيانات أو بعض الوظائف
    
* **التواصل بين الآلات**: خدمة الويب ليست مخصصة لعرضها مباشرة للبشر، بل لتستهلكها تطبيقات الهاتف وتطبيقات الويب، بحيث تُتيح بعض البيانات أو الوظائف التي يمكن للمستخدم رؤيتها في النهاية
    
* **غير معتمدة على اللغة**: يمكن كتابة خدمة الويب بأي لغة مثل Java أو C#، ويجب أن يتمكن أي تطبيق من التفاعل معها، حتى لو كانت مكتوبة بلغة مختلفة
    
* **غير معتمدة على النظام الأساسي**: لا يهم النظام الذي تعمل عليه الأجهزة. على سبيل المثال، يجب ألا يمنع تشغيل خدمة ويب على Linux جهازًا يعمل بنظام Windows من التواصل معها
    
* **استخدام بروتوكولات الويب القياسية**: عادةً ما تستخدم خدمة الويب HTTP/HTTPS لنقل البيانات عبر الإنترنت، وFTP لنقل الملفات، وSMTP للبريد الإلكتروني
    

**REST** و**SOAP** هما من أشهر أنواع خدمات الويب، رغم وجود طرق أخرى مثل **XML-RPC**.

**REST**، التي تعني **نقل الحالة التمثيلية**، هي الأكثر استخدامًا اليوم. تعتمد معظم التطبيقات الحديثة على خدمات الويب RESTful لأنها بسيطة ومرنة وتعمل بشكل طبيعي مع الويب.

تُتيح واجهات برمجة التطبيقات REST الموارد، التي تمثل عادةً بمسارات URL مثل `/users` أو `/users/20`. عندما يطلب العميل موردًا، يعيد الخادم تمثيلًا لذلك المورد. غالبًا ما يكون هذا التمثيل بصيغة JSON، لكنه قد يكون أيضًا بصيغ مثل XML أو حتى نص عادي.

يمكن للعملاء تحديد الصيغة التي يريدونها باستخدام رؤوس HTTP، مثل الرأس `Accept`. على سبيل المثال، يمكن للعميل طلب استجابة JSON بتعيين القيمة المناسبة في `Accept`.

```bash
Accept: application/json
```

ويمكنك تحديد أنك تريد XML بهذه الطريقة:

```bash
Accept: application/xml
```

يمكن أن يبدو المورد الذي تستقبله هكذا إذا كان بصيغة JSON:

```json
{
  "id": 20,
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "createdAt": "2025-09-22T10:30:00Z"
}
```

وهكذا إذا كان بصيغة XML:

```xml
<user>
  <id>20</id>
  <name>John Doe</name>
  <email>john@example.com</email>
  <role>admin</role>
  <createdAt>2025-09-22T10:30:00Z</createdAt>
</user>
```

SOAP تعني **بروتوكول الوصول إلى الكائنات البسيط**. يستخدم فقط صيغة بيانات XML، وهو الأكثر شيوعًا في أنظمة المؤسسات التي تحتاج إلى معايير وأمان أكثر صرامة.

إليك كيف يمكن أن تبدو بيانات SOAP:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
   <soapenv:Body>
      <getUser>
         <userId>20</userId>
      </getUser>
   </soapenv:Body>
</soapenv:Envelope>
```

يستخدم XML-RPC XML لترميز طلباته واستجاباته. هذا يجعله خفيف الوزن لكنه أقل مرونة من REST.

أما UDDI، فهو ليس خدمة تبادل بيانات بل دليل قائم على XML حيث يمكن للشركات إدراج واكتشاف ووصف خدمات الويب المتاحة لديها.
