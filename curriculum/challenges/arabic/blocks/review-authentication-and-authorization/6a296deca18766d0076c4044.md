## التوثيق مقابل التفويض

**التوثيق** هو عملية التحقق من هوية المستخدم. يجيب على سؤال *"هل هذا الشخص هو من يدعي أنه؟"* عند تسجيل الدخول، يقارن خادم الواجهة الخلفية بيانات الاعتماد مع السجلات المخزنة أثناء التسجيل.

يستخدم مطورو الواجهة الخلفية ثلاث فئات رئيسية لعوامل التوثيق:

- **شيء تعرفه:** كلمة مرور، رمز PIN، أو إجابة سؤال أمني.
- **شيء تملكه:** رمز SMS على الهاتف الذكي، أو رمز مؤقت من تطبيق التوثيق.
- **شيء أنت عليه:** بيانات بيومترية مثل بصمة الإصبع أو مسح التعرف على الوجه.

**التفويض** يبدأ بعد التوثيق ويقرر ما يُسمح للمستخدم الموثق بفعله. يجيب على سؤال *"ما الصلاحيات التي يمتلكها هذا الهوية؟"* الأدوار الشائعة تشمل:

- **مستخدم عادي:** يمكنه تعديل ملفه الشخصي فقط، ولا يمكنه عرض بيانات المستخدمين الآخرين الخاصة.
- **محرر:** يمكنه كتابة ونشر المحتوى، لكنه لا يستطيع حذف شفرة المصدر.
- **مشرف:** لديه صلاحيات كاملة لتعديل قواعد البيانات، حذف المستخدمين، وتغيير إعدادات النظام العامة.

**يجب أن يحدث التوثيق أولًا، ثم التفويض.** لا يمكنك تحديد الصلاحيات قبل معرفة هوية المستخدم.

## رموز الويب JSON (JWTs)

JWT هو بطاقة مفتاح رقمية يصدرها الخادم للمستخدم بعد تسجيل دخول ناجح. يقدمها المستخدم في كل طلب لاحق لإثبات هويته دون إعادة إدخال كلمة المرور.

تدفق JWT في أربع خطوات:

1. **تسجيل الدخول:** يرسل المستخدم بيانات الاعتماد إلى الخادم.
2. **الإصدار:** يتحقق الخادم من البيانات وينشئ JWT، ثم يرسله مرة أخرى.
3. **التخزين:** يخزن العميل JWT، عادة في ملف تعريف الارتباط للمتصفح أو التخزين المحلي.
4. **الطلب:** في كل طلب مستقبلي لمسار محمي، يرسل العميل JWT ويتحقق الخادم منه.

يتكون JWT من ثلاثة أجزاء مفصولة بنقطتين:

- **القسم العلوي:** JSON مشفر يعلن نوع الرمز (JWT) وخوارزمية التجزئة المستخدمة، مثل HMAC SHA256.
- **الحمولة:** JSON مشفر يحتوي على مطالبات عن المستخدم، مثل معرف المستخدم، اسم المستخدم، أو الدور. القسم العلوي والحمولة مشفران بـ Base64، وليس مشفرين، لذا لا تخزن بيانات حساسة مثل كلمات المرور هنا.
- **التوقيع:** يُنشأ بتجزئة القسم العلوي والحمولة معًا باستخدام مفتاح سري يعرفه الخادم فقط. إذا تم التلاعب بالحمولة، لن يتطابق التوقيع ويرفض الخادم الطلب مع استجابة `401 Unauthorized`.

تمكّن JWTs **التوثيق عديم الحالة**. لا يحتاج الخادم إلى تخزين بيانات الجلسة في قاعدة بيانات لأن التوقيع يثبت أنه أنشأ الرمز.

## تزوير طلبات المواقع المتقاطعة (CSRF)

CSRF هو هجوم يجبر المستخدم الموثق على تنفيذ إجراءات غير مرغوب فيها دون علمه على تطبيق ويب. يخدع موقع خبيث متصفح الضحية لإرسال طلب مزور إلى خادم الواجهة الخلفية، مستغلاً أن المتصفحات ترفق ملفات تعريف الارتباط تلقائيًا بأي طلب يذهب إلى الخادم المستهدف.

**تدفق الهجوم النموذجي:**

1. تسجل الدخول إلى، مثلاً، `trusted-bank.com`. يخزن البنك رمز جلسة في ملف تعريف الارتباط للمتصفح.
2. دون تسجيل الخروج، تزور `evil-hacker-site.com`.
3. يرسل الموقع الخبيث طلبًا صامتًا إلى `trusted-bank.com/transfer?amount=1000&to=hacker`.
4. يضيف متصفحك تلقائيًا ملف تعريف الارتباط الخاص بجلسة البنك، فيعالج الخادم الطلب.

لا يستطيع الموقع الخبيث قراءة ملف تعريف الارتباط الخاص بك. هو فقط يستغل أن المتصفح يرفق ملفات تعريف الارتباط تلقائيًا بأي طلب يذهب إلى الخادم المستهدف.

**الدفاعات ضد CSRF:**

- **`SameSite=Strict`:** لا يرسل المتصفح ملف تعريف الارتباط إذا جاء الطلب من موقع طرف ثالث.
- **`SameSite=Lax`:** يسمح بملفات تعريف الارتباط عند التنقل الآمن (نقر رابط)، لكنه يمنعها في إرسال النماذج عبر المواقع وطلبات API.
- **رموز CSRF:** عند تحميل المستخدم صفحة، ينشئ الخادم رمزًا عشوائيًا وفريدًا وغير متوقع ويرفقه بالنموذج الأمامي أو الجلسة. يجب على العميل تضمينه في رؤوس الطلب أو بيانات النموذج. لأن الموقع الخبيث لا يمكنه الوصول إلى الواجهة الأمامية لنسخ الرمز، تفشل الطلبات المزورة في التحقق.

## Passport.js

Passport.js هو برمجيات وسيطة للتوثيق في Node.js توحد طريقة توثيق الطلبات باستخدام ملحقات معيارية تسمى **استراتيجيات**. مع أكثر من 500 استراتيجية متاحة، يمكنك إضافة أي طريقة توثيق بتثبيت الحزمة المناسبة:

- `passport-local`: اسم المستخدم وكلمة المرور مقابل قاعدة بياناتك الخاصة.
- `passport-jwt`: يتحقق من رموز الويب JSON على مسارات API المحمية.
- `passport-google-oauth20`: تسجيل الدخول بحسابات Google.
- `passport-github2`: تسجيل الدخول بحسابات GitHub.

ثبت الحزمة الأساسية واستراتيجية:

```bash
npm install passport passport-local
```

اربط Passport بتطبيق Express قبل تعريف المسارات:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database look-up and password verification logic
}));
```

احمِ مسار تسجيل الدخول باستخدام Passport كبرمجيات وسيطة:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

بعد التوثيق الناجح، يدير Passport تلقائيًا **إدارة الجلسة** (يخزن بيانات المستخدم في ملف تعريف الارتباط للجلسة) و**تزيين الطلب** (يرفق كائن المستخدم إلى `req.user` لاستخدامه في معالجات المسارات التالية).

## Helmet.js

Helmet.js هو مجموعة برمجيات وسيطة أمنية لـ Node.js تحمي تطبيقك بضبط أو تعديل أو إزالة رؤوس استجابة HTTP التي يكشف عنها Express افتراضيًا.

ثبتها وطبقها في خطوة واحدة:

```bash
npm install helmet
```

```js
const helmet = require('helmet');
app.use(helmet());
```

وضع `app.use(helmet())` في أعلى مكدس البرمجيات الوسيطة يطبق هذه الحمايات على كل استجابة:

- **إزالة X-Powered-By:** يعرض Express وجوده افتراضيًا عبر `X-Powered-By: Express`. يزيل Helmet هذا القسم العلوي، مما يمنع المهاجمين من استهداف ثغرات خاصة بالإصدار.
- **تعيين Content-Security-Policy:** يقيّد أماكن تحميل المتصفح للسكريبتات، الصور، والأنماط، مما يحمي من هجمات البرمجة النصية عبر المواقع (XSS).
- **تعيين X-Frame-Options إلى SAMEORIGIN:** يمنع تضمين موقعك في إطار iframe خارجي، مما يحجب هجمات النقر الاحتيالي.
- **تعيين Strict-Transport-Security:** يجبر المتصفحات على التواصل مع خادمك عبر HTTPS فقط، مما يحمي البيانات من الاعتراض على الشبكات غير الموثوقة.
