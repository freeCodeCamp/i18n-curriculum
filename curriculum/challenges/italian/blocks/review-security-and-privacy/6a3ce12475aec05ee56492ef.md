## Sicurezza vs privacy

**Sicurezza** protegge i dati da accessi non autorizzati, furti e danni dannosi. Costruisce difese tecniche come firewall, crittografia e software antivirus per tenere fuori dai sistemi gli attori malevoli.

**Privacy** controlla come le organizzazioni raccolgono, archiviano, usano e condividono i dati personali. Dà agli utenti il potere di decidere chi può vedere le loro informazioni.

I due concetti sono strettamente correlati ma distinti:

- Puoi avere sicurezza senza privacy. Un'azienda può archiviare i tuoi dati in un database altamente sicuro e comunque venderli agli inserzionisti senza il tuo consenso. In quel caso, la sicurezza è intatta ma la privacy è violata.
- Non puoi avere vera privacy senza sicurezza. Se un server che contiene dati privati viene violato, qualsiasi promessa di privacy viene immediatamente infranta.

La sicurezza fornisce il meccanismo tecnico di difesa, mentre la privacy regola i diritti legali e le politiche sull'uso dei dati.

## HTTPS

**HTTP** trasferisce dati in testo semplice, rendendoli leggibili a chiunque li intercetti. **HTTPS** (Hypertext Transfer Protocol Secure) usa **TLS (Transport Layer Security)** per crittografare il canale di comunicazione tra browser e server. TLS ha sostituito il vecchio protocollo **SSL (Secure Sockets Layer)**, anche se molte piattaforme di hosting web fanno ancora riferimento a SSL nelle loro impostazioni.

Vantaggi chiave di HTTPS:

- **Crittografia:** Trasforma i dati in codice illeggibile. Solo browser e server possiedono le chiavi per decrittarli.
- **Autenticazione:** Richiede al proprietario del sito di ottenere un certificato digitale da un'organizzazione affidabile, dimostrando che il sito è legittimo.
- **Integrità dei dati:** Impedisce a chiunque di modificare i dati in transito, ricevi esattamente ciò che il server ha inviato.
- **Protezione su Wi-Fi pubblico:** Crea un tunnel privato attraverso reti aperte, nascondendo il tuo traffico da eventuali attaccanti nelle vicinanze.

I browser moderni avvertono gli utenti quando un sito usa HTTP semplice e richiedono HTTPS per funzionalità avanzate come geolocalizzazione e notifiche push.

## Same-Origin Policy e CORS

La **Same-Origin Policy (SOP)** è una regola di sicurezza del browser che impedisce a una pagina web di leggere dati da un'origine diversa. Un'**origine** è composta da tre parti che devono corrispondere esattamente:

- **Protocollo** (es. `https`)
- **Nome di dominio** (es. `example.com`)
- **Numero di porta** (es. `443`, il predefinito HTTPS e `80`, il predefinito HTTP)

Questa politica impedisce a siti malevoli aperti in una scheda del browser di rubare dati da un sito bancario aperto in un'altra scheda.

**CORS (Cross-Origin Resource Sharing)** permette ai server di optare per una condivisione controllata di dati cross-origin usando intestazioni HTTP:

- Il browser invia un'intestazione `Origin` con ogni richiesta cross-origin.
- Se il server si fida di quell'origine, risponde con un'intestazione `Access-Control-Allow-Origin`.
- Il browser allora permette di leggere i dati della risposta; altrimenti li blocca.

In Express, il pacchetto `cors` gestisce questo:

```javascript
// development: allow all origins
app.use(cors());

// production: restrict to trusted domains
app.use(cors({ origin: ['https://yourapp.com', 'https://admin.yourapp.com'], credentials: true }));
```

## Cookie: archiviazione, sicurezza e privacy

**Cookie** sono piccoli file di testo che i siti web salvano sul tuo dispositivo per ricordare informazioni tra una visita e l'altra. Il server invia un'intestazione `Set-Cookie`; il browser archivia il file e lo invia automaticamente ad ogni richiesta successiva.

Flag di sicurezza per proteggere i cookie:

- **`HttpOnly`:** Blocca JavaScript dal leggere il cookie, prevenendo il furto del token di sessione tramite XSS.
- **`Secure`:** Garantisce che il cookie venga trasmesso solo su HTTPS, proteggendolo dall'intercettazione su Wi-Fi pubblico.
- **`SameSite=Strict`:** Impedisce al browser di inviare il cookie in qualsiasi richiesta cross-site, bloccando attacchi CSRF.
- **`SameSite=Lax`:** Permette i cookie nelle navigazioni di primo livello sicure ma li blocca nelle sottomissioni di moduli e richieste API cross-site.

**Implicazioni per la privacy:**

- **Cookie di prima parte** tracciano le tue azioni solo sul sito che stai visitando.
- **Cookie di terze parti** sono inseriti da reti pubblicitarie e ti tracciano su siti completamente diversi, costruendo profili dettagliati dei tuoi interessi.

Leggi sulla privacy come il GDPR richiedono ai siti di mostrare banner di consenso ai cookie e di dare agli utenti il diritto di rifiutare i cookie di tracciamento. I browser moderni bloccano ora per impostazione predefinita i cookie di terze parti per il tracciamento.

## Situazioni in cui si verificano problemi di sicurezza

Le vulnerabilità di sicurezza raramente accadono per caso. Le situazioni comuni che le scatenano includono:

- **Fretta di rilasciare:** I team di sviluppo saltano revisioni approfondite del codice e test di sicurezza per rispettare scadenze rigide, lasciando bug non corretti e controlli di accesso mancanti.
- **Integrazione di sistemi complessi:** Ogni libreria esterna, API di terze parti o microservizio aggiunge rischio alla catena di fornitura. Una vulnerabilità in una dipendenza si propaga nell'applicazione principale.
- **Gestione della configurazione scadente:** Distribuire server o database cloud con impostazioni di fabbrica predefinite spesso lascia password amministrative vuote, porte aperte e accesso pubblico ai dati abilitato. Scanner automatici cercano questi sistemi mal configurati.
- **Errore umano:** Dipendenti presi di mira da email di phishing, SMS fraudolenti o telefonate ingannevoli possono involontariamente esporre credenziali o trasferire dati agli attaccanti.
- **Fusioni aziendali:** Collegare rapidamente due reti aziendali diverse crea punti ciechi come perdita di visibilità su vecchi server, firewall mal configurati e permessi di accesso eccessivi.

## Minacce comuni alla sicurezza e mitigazioni

| Minaccia | Come funziona | Mitigazione |
|---|---|---|
| **Phishing** | Email fraudolente imitano marchi affidabili per rubare credenziali | Formazione sulla consapevolezza della sicurezza; protocolli di autenticazione email (SPF, DKIM, DMARC) |
| **Malware (ransomware, spyware)** | Cripta o ruba file della vittima; richiede un riscatto | Patch automatiche; software antimalware |
| **Credential stuffing** | Strumenti automatici testano milioni di coppie username/password rubate su siti diversi | Applicare l'autenticazione multifattore (MFA) |
| **DDoS** | Inonda un server con traffico falso per farlo crashare | Servizi di filtraggio del traffico basati su cloud (es. Cloudflare) |
| **Minacce interne** | Dipendenti scontenti o negligenti perdono o rubano dati | Principio del minimo privilegio; monitorare i log di sistema per accessi insoliti |

## Content Security Policy (CSP) e Permissions-Policy

**Content Security Policy (CSP)** è un'intestazione di risposta HTTP che limita quali risorse esterne un browser può caricare su una pagina. Difende principalmente dagli attacchi **Cross-Site Scripting (XSS)** permettendo solo domini di origine fidati per script, stili e immagini.

Se uno script malevolo da un dominio non approvato tenta di eseguirsi, il browser lo blocca e registra un errore di sicurezza.

**Permissions-Policy** (ex Feature-Policy) controlla l'accesso a funzionalità del browser e hardware del dispositivo come webcam, microfono e geolocalizzazione. Puoi limitare una funzionalità al tuo dominio o disabilitarla completamente:

```http
Permissions-Policy: camera=(self), geolocation=()
```

`camera=(self)`: solo il tuo dominio può accedere alla camera.
`geolocation=()`: la geolocalizzazione è disabilitata per tutti sulla pagina.

Insieme, CSP blocca la consegna di dati e previene l'iniezione di codice; Permissions-Policy protegge l'hardware del dispositivo e garantisce la conformità alla privacy.

## Concetti fondamentali nella privacy degli utenti

| Concetto | Definizione |
|---|---|
| **PII (Informazioni personali identificabili)** | Qualsiasi dato che può identificare direttamente o indirettamente una persona. Identificatori diretti: nome, indirizzo, numero ID. Identificatori indiretti: indirizzo IP, numero seriale del dispositivo, cronologia delle posizioni. |
| **Riservatezza** | Garantire che i dati privati rimangano inaccessibili a parti non autorizzate tramite crittografia e controlli di accesso. |
| **Minimizzazione dei dati** | Raccogliere solo la quantità minima di PII necessaria per completare un'attività specifica. |
| **Limitazione dello scopo** | I dati raccolti per una ragione non possono essere usati legalmente per un'altra (es. vendere informazioni di contatto agli inserzionisti). |
| **Tracciamento** | Monitoraggio continuo dell'attività online usando script, fingerprinting del dispositivo e registri della cronologia del browser per costruire profili utente. |
| **Consenso informato** | Gli utenti devono acconsentire attivamente alla raccolta dati. Le caselle preselezionate non costituiscono consenso valido. |
| **Diritto all'oblio** | Gli utenti possono richiedere legalmente che un'organizzazione elimini permanentemente tutti i loro dati personali e la cronologia di tracciamento. |

## Leggi regionali sulla privacy

| Legge | Giurisdizione | Requisiti chiave |
|---|---|---|
| **GDPR** | Unione Europea | Richiede consenso esplicito; gli utenti hanno diritto di consultare ed eliminare i propri dati; multe severe per violazioni. |
| **CCPA / CPRA** | California, USA | Modello opt-out; le aziende devono mostrare un collegamento "Non vendere le mie informazioni personali". |
| **COPPA** | USA (federale) | Richiede consenso verificabile dei genitori prima di raccogliere dati da bambini sotto i 13 anni. |
| **DPA / UK GDPR** | Regno Unito | Trattamento dati lecito, equo e trasparente; gli individui hanno diritti di consultazione ed eliminazione. |
| **NDPA** | Nigeria | Regola la raccolta, l'archiviazione e l'uso dei dati personali dei residenti nigeriani. |
| **DPDP** | India | Regola come vengono usati i dati personali dei residenti indiani. |

**Passaggi pratici per la conformità:**

1. **Banner di consenso consapevoli della posizione:** Rileva la posizione geografica dell'utente e mostra banner cookie conformi alle normative locali.
2. **Sistema DSAR automatizzato:** Gestisce le richieste di accesso ai dati per esportare o eliminare in sicurezza l'intera cronologia dati di un utente.
3. **Minimizzazione dei dati e controllo dell'età:** Raccogli solo ciò che è necessario e usa la verifica dell'età per proteggere i minori.
4. **Responsabile della protezione dei dati (DPO):** Nomina un esperto per supervisionare l'architettura della privacy e gestire la conformità normativa.

## Il file `.env` e la libreria `dotenv`

Un **file `.env`** è un file di testo semplice posizionato nella radice di un progetto che archivia dati di configurazione sensibili come coppie chiave-valore. Questi includono chiavi API, password del database e numeri di porta. Li tiene fuori dal codice sorgente e lontano dai repository pubblici.

```bash
PORT=3000
DB_HOST=localhost
API_KEY=xyz123secretkey
```

Aggiungi sempre `.env` a `.gitignore` così non viene mai commesso nel controllo versione.

Il **pacchetto `dotenv`** legge il file `.env` all'avvio e inietta le chiavi in `process.env`:

```javascript
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

Due vantaggi chiave:

- **Sicurezza:** Le credenziali sensibili non compaiono mai nel codice sorgente o nella cronologia delle versioni.
- **Flessibilità:** Lo stesso codice funziona in sviluppo, test e produzione semplicemente sostituendo il file `.env` su ogni server.
