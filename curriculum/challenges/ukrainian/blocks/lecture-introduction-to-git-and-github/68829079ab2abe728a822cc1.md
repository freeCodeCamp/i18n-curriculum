# --description--

У попередньому уроці ви дізналися, як створити новий репозиторій на GitHub. Але що робити, якщо у вас вже є існуючий проєкт локально, який ви хочете додати на GitHub? У цьому уроці ви навчитеся пушити існуючі локальні репозиторії на GitHub і завантажувати зміни з віддалених репозиторіїв.

Почніть зі створення нового репозиторію на GitHub без жодних файлів. Якщо все зроблено правильно, вас перенаправить на сторінку з URL, який потрібен для клонування вашого репозиторію.

Тепер перейдіть до терміналу і виберіть існуючий проєкт, який хочете додати на GitHub. У цьому уроці я вибираю локальний проєкт під назвою `super-awesome-game` з такими файлами всередині:

```sh
jessicawilkins super-awesome-game >> ls
README.md  index.html script.js  styles.css
```

Щоб налаштувати контроль версій для цього проєкту за допомогою Git, потрібно виконати команду `git init` у директорії проєкту. Це ініціалізує порожній Git-репозиторій, щоб Git міг почати відстежувати зміни для цього проєкту. Якщо все зроблено правильно, у терміналі ви побачите схожий вихід:

```sh
jessicawilkins super-awesome-game >> git init
Initialized empty Git repository in /Users/jessicawilkins/workspace/freeCodeCamp/super-awesome-game/.git/
```

Коли ви ініціалізуєте порожній Git-репозиторій у проєкті, буде додано нову приховану директорію `.git`. Щоб побачити цю директорію, можна виконати команду `ls -a`, яка виводить список усіх файлів і директорій, включно з прихованими:

```sh
jessicawilkins super-awesome-game >> ls -a
.          ..         .git       README.md  index.html script.js  styles.css
```

Важливо не видаляти цю директорію `.git`, інакше вся історія Git буде втрачена.

Тепер, коли порожній Git-репозиторій ініціалізовано, можна виконати команду `git status`. Ця команда показує поточний стан вашої робочої директорії — ви часто будете її використовувати у своєму робочому процесі. Ви повинні побачити приблизно такий вихід:

```sh
jessicawilkins super-awesome-game >> git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	README.md
	index.html
	script.js
	styles.css

nothing added to commit but untracked files present (use "git add" to track)
```

`On branch main` означає, що ви перебуваєте на гілці `main`. Гілка в Git — це окрема робоча область, де ви можете вносити зміни. Гілка `main` зазвичай є основною або продуктивною гілкою у реальному застосунку. Команди розробників створюють кілька гілок для нових функцій і виправлень помилок, а потім об’єднують ці зміни назад у гілку `main`. Ви дізнаєтеся більше про цей процес у майбутньому уроці.

`Untracked files` означає, що Git виявив файли, які наразі не відстежує.

Існує п’ять станів, у яких може перебувати файл:

1. "Untracked": це означає, що файл новий для репозиторію, і Git його ще не "бачив".
2. "Modified": цей файл існував у попередньому коміті і має зміни, які ще не зафіксовані.
3. "Ignored": ви, ймовірно, не побачите ігноровані файли в Git, але ваша IDE може їх позначати. Ігноровані файли виключені з операцій Git, зазвичай тому, що вони включені у файл `.gitignore`.
4. "Deleted": видалений файл — це протилежність не відстежуваному файлу — це файл, який раніше існував, але був видалений.
5. "Renamed": перейменований файл — це файл, вміст якого не змінився, але було змінено ім’я або розташування. Іноді файл вважають перейменованим навіть якщо в ньому є невеликі зміни.

Якщо у вашому репозиторії є незакомічені зміни, спочатку потрібно їх "підготувати" (stage). "Область підготовки" — це спеціальний файл у вашій директорії `.git`, який містить інформацію, що увійде до наступного коміту. Коміт — це фактично знімок поточного стану вашого репозиторію.

Якщо хочете підготувати по одному файлу, використовуйте таку команду:

```sh
git add name-of-file
```

Якщо хочете підготувати всі непідготовлені зміни, використовуйте `git add .`. Крапка (`.`) — це псевдонім для поточної директорії, в якій ви перебуваєте. Деякі розробники радять уникати `git add .`, бо можна випадково підготувати файли, які не планували. Якщо це станеться, ви можете "відмінити підготовку" за допомогою такої команди:

```sh
git reset name-of-file
```

Взагалі, якщо ви контролюєте, які файли додаються за допомогою `git status` перед підготовкою, все буде гаразд.

Після підготовки файлів потрібно їх закомітити. Ви можете зробити це, виконавши команду `git commit`. Вона відкриє ваш улюблений редактор, який ви налаштували у конфігурації Git. Коли редактор відкриється, ви можете написати детальне повідомлення про зміни. Також можна надати коротке повідомлення, використовуючи команду `git commit -m` так:

```sh
git commit -m "short message goes here"
```

Якщо ви використовуєте прапорець `-m`, все одно варто надати описове повідомлення, щоб пояснити контекст змін. Ось кілька прикладів хороших повідомлень, які можна використовувати:

```sh
git commit -m "feat(api): implement JWT-based authentication"
git commit -m "fix(auth): correct token expiration time"
git commit -m "refactor: simplify login flow logic"
```

`feat(api)` і `fix(auth)` відповідають стилю Conventional Commits, який використовується для пояснення типів змін у коміті.

Після того, як ви закомітили роботу і знову виконали `git status`, у терміналі має з’явитися такий вихід:

```sh
On branch main
nothing to commit, working tree clean
```

Щоб переглянути всі попередні коміти цього проєкту, виконайте команду `git log`. Вона виведе список усіх комітів із корисною інформацією, як-от автор, дата коміту, повідомлення коміту та хеш коміту. Хеш коміту — це довгий рядок, який служить унікальним ідентифікатором коміту.

Після завершення коміту ви можете пушити зміни на GitHub. Спочатку потрібно налаштувати віддалене підключення до вашого віддаленого репозиторію. Ви вивчали це у попередньому уроці, але ось нагадування з використанням SSH:

```sh
git remote add origin git@github.com:your-github-username/name-of-repo.git
```

Останній крок — запушити зміни на GitHub за допомогою такої команди:

```sh
git push -u origin main
```

Прапорець `-u` — це скорочення для `--set-upstream`. Він з’єднує вашу локальну гілку `main` з віддаленою гілкою `main` на `origin`. Його потрібно використовувати лише один раз для нової гілки. Після цього можна просто виконувати `git push` без вказання імені гілки.

Якщо все зроблено правильно, у терміналі ви побачите схожий результат:

```sh
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Delta compression using up to 8 threads
Compressing objects: 100% (2/2), done.
Writing objects: 100% (3/3), 253 bytes | 253.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0 (from 0)
To https://github.com/jdwilkin4/super-awesome-game.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

Також ви побачите нові зміни на сторінці вашого репозиторію на GitHub після оновлення сторінки.

Якщо ви працюєте в команді над проєктом і потрібно завантажити нові зміни з репозиторію, використовуйте команду `git pull`. Ви дізнаєтеся більше про роботу в команді та співпрацю у майбутньому уроці.
