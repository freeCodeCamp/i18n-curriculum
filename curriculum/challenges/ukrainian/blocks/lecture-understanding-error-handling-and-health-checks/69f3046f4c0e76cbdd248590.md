# --description--

Колись замислювалися, як утримати свій застосунок Express стабільним і готовим до продакшену, особливо коли він під великим навантаженням або завершує роботу?

Ось тут і допомагають health checks і graceful shutdowns. Ці два шаблони допомагають тримати застосунок доступним і забезпечують його чисте завершення без втрати запитів чи пошкодження даних.

Давайте розберемося.

Спершу, що таке "health check" в Express?

Health check — це простий маршрут, який повідомляє, чи ваш застосунок працює.

Наприклад:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

Інструменти, як балансувальники навантаження або оркестратори контейнерів — наприклад Kubernetes чи AWS Elastic Beanstalk — можуть опитувати цей маршрут, щоб визначити, чи застосунок здоровий.

Якщо маршрут повертає статус 200, ваш сервіс вважається здоровим. Нагадаю — статус HTTP 200 означає, що все гаразд!

Якщо він не відповідає, оркестратор може автоматично перезапустити сервіс.

А що таке "graceful shutdown"?

Graceful shutdown — це те, що відбувається, коли ваш застосунок збирається зупинитися, наприклад, коли сервер перезавантажується або відбувається розгортання.

Замість того, щоб одразу розривати з’єднання, ви даєте застосунку час завершити поточні справи.

В Express ви можете слухати `SIGTERM`, який є системним сигналом, що надсилається, коли процес має завершитися.

Ось приклад:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

Це каже Express припинити приймати нові з’єднання і чекати, поки існуючі завершаться, перш ніж виходити.

Ви також маєте обробляти `SIGINT`, сигнал, який надсилається, коли ви натискаєте Ctrl+C, щоб зупинити застосунок під час локальної розробки.

Хоча `SIGTERM` зазвичай використовують оркестратори, як Kubernetes у продакшені, `SIGINT` охоплює повсякденний робочий процес розробника. Без обробки цього сигналу натискання Ctrl+C миттєво завершує процес, пропускаючи будь-яке прибирання, яке ви налаштували.

Оскільки обидва сигнали мають викликати однакову поведінку graceful shutdown, розробники зазвичай слухають обидва.

Тепер зберемо все докупи.

Уявіть, що ваш застосунок працює в контейнері. Kubernetes надсилає `SIGTERM`, щоб завершити його для оновлення.

З базовим graceful shutdown ваш застосунок припиняє приймати нові з’єднання і завершує поточні запити. Коли вся робота виконана, він відключається від бази даних і чисто виходить. Ніяких втрачених даних, ніякого пошкодженого стану.

Але є проблема — балансувальник навантаження не знає, що застосунок завершує роботу, тому може продовжувати надсилати новий трафік. Ось тут і допоможе бонусний шаблон нижче.

Тепер у мене для вас невеликий бонус. Розглянемо «позначення застосунку як нездорового перед завершенням».

Перед завершенням ви можете оновити health check, щоб він повертав статус 503. Таким чином балансувальник бачить нездорову відповідь і припиняє направляти новий трафік до вашого застосунку під час вікна завершення, як показано в коді вище.

Нагадаю: статус `503` означає «Сервіс недоступний». Це сигналізує, що сервер тимчасово не може обробляти запити, що саме і відбувається під час graceful shutdown.

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
