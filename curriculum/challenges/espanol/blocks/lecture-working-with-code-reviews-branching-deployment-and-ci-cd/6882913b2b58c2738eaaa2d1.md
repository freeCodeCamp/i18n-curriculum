# --description--

Hemos tocado brevemente el archivo `.gitignore` en una lección anterior. Pero, ¿qué es exactamente este archivo y qué deberías poner en él?

El archivo `.gitignore` es un tipo especial de archivo relacionado con las operaciones de Git. El nombre sugiere que este archivo se usa para decirle a Git que ignore cosas, y ese es el caso común. Pero lo que realmente hace es indicarle a Git que deje de rastrear un archivo. Esta distinción es importante y se aclarará más adelante.

¿Qué deberías poner realmente en este archivo? Bueno, deberías listar las cosas que no quieres que se comprometan.

El primero serían tus archivos de secretos. Esto puede ser algo como un archivo `.env`, donde has almacenado tus claves API para que tu aplicación las use. Puede ser un archivo de clave privada GPG que tu app necesita para la autenticación. Cualquier cosa que contenga información sensible que no quieras que se filtre al público debe ser ignorada con `.gitignore`. Incluso si tu repositorio es privado, debes actuar como si fuera público: trátalo como si cualquiera pudiera verlo y ten cuidado de no hacer commit de secretos.

Echemos un vistazo a cómo podría verse nuestro archivo ahora mismo:

```sh
# Secrets
.env
github.key
```

Le hemos indicado a Git que ignore el archivo `.env`, donde guardamos nuestros secretos. También le hemos indicado a Git que ignore un archivo `github.key`, que contiene una clave privada GPG.

Lo siguiente a considerar son los paquetes, o dependencias, que tu proyecto utiliza. Por ejemplo, cuando trabajas con un proyecto de Node.js, probablemente instales paquetes desde npm. Estos paquetes se colocan en una carpeta `node_modules` dentro de tu proyecto, para que tu código pueda importarlos y usarlos.

Pero la carpeta `node_modules` se vuelve prohibitivamente enorme, y definitivamente no quieres hacer commit de todos esos paquetes ni rastrear los cambios cada vez que actualices uno. En cambio, haces commit de tu archivo `package.json` y del archivo de bloqueo relevante (por ejemplo, `package-lock.json`, `pnpm-lock.yaml`). Estos archivos le indican a npm qué paquetes y versiones instalar, para que un colaborador pueda instalar las mismas dependencias que tú usas sin tener que hacer commit de ellas en el repositorio.

Vamos a añadir `node_modules` a nuestro archivo `.gitignore`:

```sh
# Secrets
.env
github.key

# Packages
node_modules
```

Ahora, toquemos un aspecto importante de la sintaxis para los archivos `.gitignore`. Cuando agregas una entrada como `node_modules`, Git ignorará cualquier cosa que coincida con ese patrón. Esto significa que ignorará el directorio `node_modules` en la raíz de tu proyecto, pero también ignorará el `node_modules` en el directorio `client` de tu proyecto.

Si solo quieres ignorar algo en la raíz, puedes anteponer la línea con una barra diagonal. `node_modules` se convertiría en `/node_modules`. Esto ignoraría el directorio raíz `node_modules`, pero no el directorio `client/node_modules`. No queremos eso, así que dejemos nuestra entrada como solo `node_modules`.

Lo siguiente que quieres considerar ignorar son tus resultados de compilación. Por ejemplo, compilar un proyecto TypeScript podría generar una carpeta `dist` para la salida de JavaScript. No quieres hacer commit de esa salida, porque infla el historial de Git. En cambio, quieres ignorar esa salida: los colaboradores pueden reproducirla por sí mismos ejecutando el mismo comando de compilación.

Vamos a ignorar la carpeta `dist`:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
```

A veces, un archivo mal ubicado puede hacer que el compilador de TypeScript falle, lo que resulta en que nuestro output de JavaScript quede desordenado. Dado que estamos escribiendo exclusivamente en TypeScript, vamos a usar la sintaxis glob para ignorar todos los archivos JavaScript:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js
```

Algunas otras salidas comunes que podrías ignorar son el directorio `coverage`, para los resultados de la cobertura de pruebas unitarias, un directorio `.angular` si estás usando Angular, un directorio `.next` si estás usando NextJS, y así sucesivamente.

Por ahora, podemos quedarnos con lo que tenemos aquí. Si estás usando un framework que genera archivos, deberías confirmar en su documentación qué necesitas agregar al `.gitignore`.

El cuarto grupo de cosas para ignorar son los archivos de IDE. Los productos de Jetbrains, por ejemplo, pueden usar una carpeta `.idea` para configuraciones específicas del IDE. Visual Studio Code usa una carpeta `.vscode`.

A menudo es bueno ignorarlos, porque contendrán archivos específicos de tu entorno local que otros colaboradores no necesitarán (porque usan sus propias configuraciones). Vamos a agregar el directorio de Visual Studio Code, ya que es lo que estamos usando como nuestro IDE.

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
```

Pero nuestra carpeta `.vscode` sí contiene un archivo `settings.json`, que incluye algunas cosas específicas del espacio de trabajo que podrían ser útiles para nuestros colaboradores; este archivo les permite configurar sus extensiones y entorno para reflejar mejor el nuestro.

Así que vamos a usar la sintaxis de negación para "unignore" ese archivo:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json
```

Si estás en MacOS, puede que estés familiarizado con el archivo `.DS_STORE`. Este archivo contiene metadatos que Finder usa para determinar cosas como cómo mostrar el directorio (por ejemplo, con miniaturas o como una lista). De manera similar, los usuarios de Windows pueden ver un archivo oculto `Thumbs.db`, que almacena en caché las miniaturas del directorio.

Generalmente es buena idea ignorar estos archivos, ya que son una configuración personal que no afecta al proyecto. Vamos a añadir ambos:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json

# System Files
.DS_STORE
Thumbs.db
```

Una nota rápida sobre ignorar archivos previamente confirmados. Supongamos que accidentalmente confirmaste tu directorio `dist` antes, pero ahora quieres eliminarlo y ignorarlo. No puedes simplemente agregarlo a tu archivo `.gitignore`: esto no hace que el contenido previamente confirmado desaparezca, sino que le indica a Git que deje de rastrear el archivo. Así que si ignoras un archivo que ya está confirmado, Git no rastreará ningún cambio, pero tampoco rastreará la eliminación del archivo.

En cambio, primero necesitarás eliminar el archivo. Crea un commit con esa eliminación. Luego puedes agregar el ítem a tu `.gitignore` y crear un segundo commit con esa actualización. De ahora en adelante, cuando ese archivo se cree de nuevo, Git lo ignorará y no aparecerá en tus commits.

Finalmente, si estás trabajando en un proyecto en un idioma diferente, como Python, los elementos que necesitas ignorar serán diferentes a lo que hemos cubierto aquí. GitHub mantiene un repositorio muy útil de plantillas `.gitignore` para varios lenguajes y frameworks. Puedes encontrar esa lista en <a href="https://github.com/github/gitignore" target="_blank">https://github.com/github/gitignore</a>.

Con eso, ¡ya estás listo para evitar cometer archivos que no quieres en tus repositorios!
