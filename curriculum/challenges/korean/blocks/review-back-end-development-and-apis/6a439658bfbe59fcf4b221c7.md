## Node.js와 이벤트 기반 아키텍처 작업하기

- **Node.js**: 브라우저 외부에서 JavaScript 코드를 실행할 수 있게 하는 오픈 소스 크로스 플랫폼 JavaScript 런타임 환경입니다. 웹 서버, API, 백엔드 시스템 구축에 널리 사용됩니다.
- **클라이언트 측 제한 사항**: 클라이언트 측 JavaScript는 파일 시스템 접근이 제한되고, 복잡한 애플리케이션 로직 지원이 제한되며, 보안 위험(예: 데이터베이스 자격 증명을 클라이언트 코드에 노출하면 안 됨)이 있습니다.
- **브라우저와 Node.js의 전역 객체 차이**: 브라우저에서는 전역 객체가 `window`로 DOM 조작, 쿠키, 브라우저 이벤트를 노출합니다. Node에서는 전역 객체가 `global`로 파일, 네트워킹, OS 상호작용을 위한 내장 모듈을 노출합니다.
- **DOM 접근**: 브라우저는 DOM API에 접근할 수 있지만 Node는 불가능합니다. 반대로 Node는 파일 시스템을 포함한 거의 모든 시스템 자원에 접근할 수 있지만 브라우저는 그렇지 않습니다.
- **버전 관리**: 서버에서 실행할 Node 버전을 선택할 수 있지만, 사용자가 실행하는 브라우저 버전은 제어할 수 없습니다.
- **논블로킹 이벤트 기반 아키텍처**: Node는 단일 스레드와 이벤트 루프를 사용해 많은 동시 요청과 I/O 작업을 효율적으로 처리하며, 블로킹 없이 실시간 애플리케이션에 적합합니다.
- **콜백**: 비동기 작업이 완료된 후 실행할 동작을 정의하는 함수입니다. 비동기 Node 코드는 콜백에 크게 의존하기 때문에 읽기와 디버깅이 어려울 수 있습니다.
- **백엔드에서 Node의 장점**: 프론트엔드와 백엔드 모두에서 단일 언어(JavaScript) 사용, 높은 동시성에 적합한 논블로킹 아키텍처, 방대한 npm 생태계, 무료 오픈 소스.
- **백엔드에서 Node의 단점**: 단일 스레드라 CPU 집약적 작업(이미지 처리, 암호화, 복잡한 수학)이 이벤트 루프를 막을 수 있고, 비동기/콜백 기반 코드에 많이 의존해 가독성이 떨어질 수 있으며, npm 패키지 품질과 유지 관리가 다양합니다.
- **NVM (Node Version Manager)**: 같은 머신에서 여러 Node.js 버전을 관리하고 전환하는 도구입니다. macOS/Linux에서는 다음 명령어로 설치하세요:

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **Node LTS 버전 설치**: `nvm install --lts`
- **Node 버전 전환**: `nvm use 20`
- **설치된 모든 Node 버전 목록**: `nvm ls`
- **기본 Node 버전 설정(macOS/Linux)**: `nvm alias default 22.20.0`
- **nvm-windows**: Windows(WSL 아님)에서 Node 버전을 관리하는 별도 프로젝트입니다. 공식 저장소에서 설치 프로그램을 다운로드하고 `nvm install lts` 후 `nvm use <version>`을 실행하세요.
- **Node로 JavaScript 파일 실행**: `node app.js`

## Node 코어 모듈 작업하기

### `fs` 모듈

- **목적**: 파일과 디렉터리를 생성, 읽기, 쓰기, 추가, 삭제하는 메서드를 제공합니다.
- **세 가지 사용 방식**: 콜백 기반(비동기), 프로미스 기반(`fs/promises`), 동기(`Sync` 접미사). 동기 메서드는 이벤트 루프를 차단하므로 스크립트에만 적합합니다.
- **`writeFile()`**: 파일에 내용을 쓰며, 파일이 없으면 생성합니다:

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**: 기존 파일에 내용을 덮어쓰지 않고 추가합니다.
- **`readFile()`**: 파일 내용을 읽습니다. 인코딩으로 `"utf8"`을 전달하면 문자열을, 생략하면 `Buffer`를 반환합니다.
- **`unlink()`**: 파일을 삭제합니다. 파일이 없으면 `ENOENT` 오류를 발생시킵니다.

### `Buffer` 모듈

- **목적**: 메모리 내에서 이진 데이터(파일, 이미지, 네트워크 스트림)를 직접 처리합니다.
- **`Buffer.from()`**: 문자열로부터 버퍼를 생성합니다:

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**: `n` 바이트 크기의 0으로 채워진 버퍼를 생성합니다:

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **잘림 현상**: 버퍼가 저장할 수 있는 용량보다 더 많은 데이터를 쓰면 초과 부분이 조용히 잘립니다.
- **`Buffer.byteLength(str)`**: 문자열을 저장하는 데 필요한 바이트 수를 반환합니다.

### `crypto` 모듈

- **목적**: 해싱, HMAC, 암호화/복호화, 디지털 서명, 안전한 난수 생성 같은 암호화 도구를 제공합니다.
- **`createHash(algorithm)`**: 단방향 해시(복호 불가)를 생성합니다. `.update(data).digest("hex")`를 사용해 해시할 데이터를 제공하고 16진수 문자열로 해시를 반환합니다:

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**: `createHash`와 비슷하지만 비밀 키가 필요하며 데이터 무결성과 인증에 적합합니다.
- **`createCipheriv()` / `createDecipheriv()`**: 데이터를 암호화하고 복호화합니다. 둘 다 알고리즘, 키, 초기화 벡터(IV)가 필요합니다:

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**: 암호학적으로 안전한 난수 바이트를 생성합니다. 토큰과 키에 `Math.random()`보다 안전합니다.
- **`randomInt(min, max)`**: 주어진 범위 내에서 안전한 난수 정수를 생성합니다. OTP에 유용합니다.
- **`sign()` / `verify()`**: 개인/공개 키 쌍을 사용해 디지털 서명을 생성하고 검증합니다.

### `os` 모듈

- **목적**: Node가 실행 중인 운영체제 정보를 노출합니다.
- **`os.platform()`**: OS 식별자를 반환합니다. 예: `'darwin'`, `'win32'`, `'linux'`.
- **`os.arch()`**: CPU 아키텍처를 반환합니다. 예: `'x64'`, `'arm64'`.
- **`os.type()`**: 공식 OS 이름을 반환합니다. 예: `'Darwin'`.
- **`os.release()`**: OS 커널 버전 문자열을 반환합니다.
- **`os.cpus()`**: 각 논리 CPU 코어(모델, 속도, 시간)를 설명하는 객체 배열을 반환합니다.
- **`os.uptime()`**: 시스템 가동 시간을 초 단위로 반환합니다.
- **`os.totalmem()` / `os.freemem()`**: 시스템 전체 및 사용 가능한 메모리를 바이트 단위로 반환합니다.
- **`os.userInfo()`**: 현재 사용자 `uid`, `gid`, `username`, `homedir`, `shell` 정보를 담은 객체를 반환합니다.
- **`os.networkInterfaces()`**: 주소, 패밀리, MAC, CIDR 세부 정보가 포함된 네트워크 인터페이스 객체를 반환합니다.

### `path` 모듈

- **목적**: 플랫폼에 독립적인 방식으로 파일과 디렉터리 경로를 다루는 유틸리티를 제공합니다.
- **`__filename`**: 전역 변수로 현재 파일의 절대 경로입니다.
- **`__dirname`**: 전역 변수로 현재 파일이 포함된 디렉터리의 절대 경로입니다.
- **`path.basename(p)`**: 경로의 마지막 구성 요소(파일명)를 반환합니다.
- **`path.dirname(p)`**: 경로의 디렉터리 부분을 반환합니다.
- **`path.extname(p)`**: 파일 확장자를 반환합니다. 예: `'.js'`.
- **`path.join(...segments)`**: 경로 세그먼트를 결합해 정규화된 경로를 만들고 잘못된 슬래시를 수정합니다.
- **`path.resolve(...segments)`**: 현재 작업 디렉터리부터 시작해 절대 경로로 세그먼트를 해석합니다.
- **`path.parse(p)`**: `root`, `dir`, `base`, `ext`, `name` 속성을 가진 객체를 반환합니다.
- **`path.format(obj)`**: `dir`, `name`, `ext` 속성을 가진 객체로부터 경로 문자열을 만듭니다.

### `process` 모듈

- **목적**: 현재 Node.js 프로세스에 대한 정보와 제어를 제공합니다. 전역 객체이므로 가져올 필요가 없습니다.
- **`process.env`**: 모든 환경 변수를 담은 객체입니다. `process.env.NODE_ENV` 같은 특정 값을 접근할 수 있습니다.
- **`process.argv`**: 명령줄 인수 배열입니다. 인덱스 0은 Node 실행 파일, 1은 스크립트 경로, 2 이상은 사용자 인수입니다.
- **`process.cwd()`**: 현재 작업 디렉터리를 반환합니다.
- **프로세스 이벤트**:

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**: `warning` 리스너로 잡을 수 있는 사용자 정의 경고를 발생시킵니다.

### `stream` 모듈

- **목적**: 모든 데이터를 한꺼번에 메모리에 올리지 않고 청크 단위로 처리합니다.
- **네 가지 스트림 유형**: 읽기 가능(청크 단위로 데이터 읽기), 쓰기 가능(청크 단위로 데이터 쓰기), 듀플렉스(둘 다), 변환(데이터 흐름 중 수정하는 듀플렉스).
- **스트림 파이핑**:

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **`finish` 이벤트**: 모든 데이터가 플러시되면 쓰기 가능한 스트림에서 발생합니다.
- **`error` 이벤트**: 스트림에 문제가 생기면 발생합니다.

## npm 소개

- **npm**: 세 부분으로 구성됩니다: 웹사이트 `npmjs.com`, 공개 레지스트리(패키지 데이터베이스), Node.js와 함께 설치되는 CLI 도구.
- **목적**: 재사용 가능한 JavaScript 패키지를 설치, 관리, 게시할 수 있게 합니다. npm은 가장 널리 사용되는 패키지 관리자입니다.
- **대안**: Yarn, PNPM, Bun.

### `package.json` 파일

- **목적**: npm 구성 매니페스트입니다. 프로젝트 이름, 버전, 설명, 진입점(`main`), 작성자, 라이선스, 스크립트, 의존성을 추적합니다.
- **대화형으로 `package.json` 생성**: `npm init`
- **기본값으로 생성**: `npm init -y`
- **`dependencies`**: 앱 실행에 필요한 패키지(예: `express`, `react`).
- **`devDependencies`**: 개발 및 테스트에만 필요한 패키지(예: `jest`, `nodemon`, `eslint`).
- **`license`**: 사용에 관한 법적 조건. 일반적인 값: `MIT`, `ISC`, `GPL`.

### 의미 있는 버전 관리(SemVer)

- **형식**: `MAJOR.MINOR.PATCH`, 예: `4.17.21`.
- **MAJOR**: 호환되지 않는 변경 시 증가.
- **MINOR**: 이전 버전과 호환되는 새로운 기능 추가 시 증가.
- **PATCH**: 이전 버전과 호환되는 버그 수정 시 증가.
- **캐럿 `^`**: 마이너 및 패치 버전 업데이트 허용, 메이저 버전은 불가. 예: `^4.17.21`은 `4.18.0`은 허용하지만 `5.0.0`은 불가.
- **틸드 `~`**: 패치 버전만 업데이트 허용. 예: `~1.2.3`은 `1.2.4`는 허용하지만 `1.3.0`은 불가.
- **별표 `*`**: 모든 버전과 매칭, 테스트용이며 프로덕션에는 적합하지 않음.
- **정확한 버전**: 접두사 없이, 예: `1.2.3`은 정확히 그 버전에 고정.

### 의존성 설치 및 제거

- **패키지 설치**: `npm install express` (단축: `npm i express`)
- **개발 의존성으로 설치**: `npm install nodemon -D`
- **패키지 제거**: `npm uninstall chalk`
- **`node_modules/`**: 설치 후 실제 패키지 코드가 들어있는 폴더. 버전 관리에 포함하지 않습니다.
- **`package-lock.json`**: 설치된 모든 패키지(하위 의존성 포함)의 정확한 버전을 고정하는 자동 생성 파일. 항상 커밋하고 수동으로 수정하지 마세요.
- **특정 버전 설치**: `npm install express@4.21.2`
- **사용 가능한 버전 보기**: `npm view express versions`
- **업데이트가 필요한 패키지 확인**: `npm outdated`
- **단일 패키지 업데이트**(허용 범위 내): `npm update express`
- **최신 버전 강제 설치**: `npm install express@latest`

### npm 레지스트리에 게시하기

- **로그인**: `npm login`
- **스코프 패키지**: 이름 충돌 방지를 위해 `@username/package-name` 형식 사용.
- **`.npmignore`**: 게시할 패키지에서 제외할 파일 목록( `.gitignore`와 유사).
- **스코프 없는 패키지 게시**: `npm publish`
- **스코프 패키지를 공개로 게시**: `npm publish --access public`

## npm 스크립트 작업하기

- **npm 스크립트**: `package.json`의 `"scripts"` 객체에 정의된 사용자 지정 명령어입니다. 서버 시작, 테스트 실행, 린팅, 빌드 같은 반복 작업을 자동화합니다.
- **스크립트 정의 예시**:

```json
"scripts": {
  "start": "node app.js"
}
```

- **스크립트 실행**: `npm run start`
- **내장 스크립트 이름 단축형** (`start`, `test`, `stop`, `restart`): `npm start`
- **`&&`를 사용한 순차 명령** (이전 명령이 성공해야 다음 명령 실행):

```json
"start": "npm run build && node server.js"
```

- **`;`를 사용한 순차 명령** (항상 다음 명령 실행):

```json
"start": "npm run build; node server.js"
```

- **`&`를 사용한 병렬 명령**: `"dev": "npm run start:server & npm run start:client"`
- **스크립트에 인수 전달** (`--` 구분자 사용): `npm start -- 8000`; 인수는 `process.argv[2]`로 접근 가능.

### CommonJS 모듈

- **CommonJS**: 원래 Node.js 모듈 시스템입니다. 모듈을 동기적으로 로드합니다.
- **`require()`로 가져오기**:

```js
const { multiply } = require('./math');
```

- **`module.exports`로 내보내기**:

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- `package.json`에 `"type": "commonjs"`가 있거나(`"type"` 필드가 없을 때도 포함), 파일 확장자가 `.cjs`일 때 CommonJS가 기본입니다.

### ES 모듈 (ESM)

- **ES 모듈(ESM)**: 최신 표준화된 모듈 형식입니다. 비동기적으로 로드됩니다.
- **이름 있는 내보내기**: `export function multiply(a, b) { return a * b; }`
- **기본 내보내기**: `export default function multiply(a, b) { return a * b; }`
- **가져오기**: `import { multiply } from './math.mjs';`
- `.mjs` 확장자이거나 `package.json`에 `"type": "module"`이 있으면 ESM을 사용합니다.

## HTTP, DNS, TCP/IP 작동 방식 이해하기

### 서버와 클라이언트-서버 모델

- **클라이언트-서버 모델**: 구성 요소가 클라이언트(요청을 보내는 프로세스, 예: 웹 브라우저) 또는 서버인 분산 시스템 아키텍처입니다.
- **서버**: 클라이언트로부터 요청을 받고 네트워크를 통해 데이터나 서비스를 반환하는 컴퓨터 또는 소프트웨어 프로그램입니다.
- **서버 유형**: 웹 서버(HTML/CSS/자산), 애플리케이션 서버(비즈니스 로직), 데이터베이스 서버, 메일 서버, 파일 서버, 프록시 서버.

### DNS

- **DNS(도메인 이름 시스템)**: 흔히 '인터넷 전화번호부'라 불리며, 사람이 읽을 수 있는 도메인 이름(예: `freecodecamp.org`)을 IP 주소로 변환합니다.
- **도메인 이름 계층 구조**(오른쪽에서 왼쪽): 최상위 도메인(TLD, 예: `.org`) → 2차 도메인(예: `freecodecamp`) → 서브도메인(예: `www`).
- **IP 주소**: 네트워크상의 장치를 고유하게 식별하는 숫자 식별자입니다. IPv4 예: `127.0.0.1`. IPv6 예: `0:0:0:0:0:0:0:1`.
- **DNS 해석 과정**: 브라우저 → 재귀적 해석기 → 루트 서버 → TLD 서버 → 권한 있는 이름 서버 → IP 주소 반환 → 재귀적 해석기가 캐시 후 클라이언트에 전달.

### TCP/IP와 HTTP

- **IP(인터넷 프로토콜)**: IP 주소를 사용해 출발지에서 목적지로 데이터 패킷을 라우팅합니다. 연결 지향적이지 않고 각 패킷이 독립적으로 전송됩니다.
- **TCP(전송 제어 프로토콜)**: 데이터를 번호가 매겨진 세그먼트로 나누고 목적지에서 재조립하며 오류를 검사하고 재전송을 요청해 신뢰성 있고 순서가 보장된 전달을 보장합니다.
- **HTTP(하이퍼텍스트 전송 프로토콜)**: 클라이언트와 서버가 웹 콘텐츠를 교환하는 방식을 정의하는 애플리케이션 계층 프로토콜입니다.
- **HTTPS**: SSL/TLS로 데이터를 암호화하는 HTTP의 보안 버전입니다.

### HTTP 요청 구조

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **요청 라인**: HTTP 메서드 + 경로/URI + HTTP 버전.
- **헤더**: 키-값 형태의 메타데이터 쌍.
- **본문**(선택적): `POST`, `PUT`, `PATCH`, `DELETE` 요청 시 데이터를 담아 보냅니다.

### HTTP 응답 구조

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **상태 라인**: HTTP 버전 + 상태 코드 + 이유 구.
- **헤더**: 응답에 관한 메타데이터(콘텐츠 유형, 길이, 캐싱 지시자).
- **본문**(선택적): 요청한 데이터.

### HTTP 상태 코드 분류

- **1xx 정보성**: 요청 수신, 처리 중. 예: `101 Switching Protocols`.
- **2xx 성공**: `200 OK`, `201 Created`, `204 No Content`.
- **3xx 리다이렉션**: `301 Moved Permanently`, `302 Found`, `304 Not Modified`.
- **4xx 클라이언트 오류**: `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.
- **5xx 서버 오류**: `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable`.

## HTTP 요청-응답 모델 이해하기

- **HTTP 메서드**: `GET`(조회), `POST`(생성), `PUT`(전체 업데이트), `PATCH`(부분 업데이트), `DELETE`(삭제).
- **요청-응답 사이클**: 클라이언트가 요청 → 서버가 인증 검증, 메서드와 URL 확인, 데이터베이스 쿼리 → 서버가 응답 생성(상태 코드 + 선택적 본문) → 클라이언트가 응답 처리.

### HTTP 응답에 포함된 자산

모든 응답에는 브라우저가 자산을 처리하는 방법을 알려주는 `Content-Type` 헤더가 포함됩니다:

- **HTML**: `text/html`
- **CSS**: `text/css`
- **JavaScript**: `application/javascript`
- **JSON**: `application/json`
- **PNG/JPEG/WebP**: `image/png`, `image/jpeg`, `image/webp`
- **MP4/MP3**: `video/mp4`, `audio/mpeg`
- **PDF**: `application/pdf`
- **WOFF2**: `font/woff2`

### HTML 폼 제출

- **`action` 속성**: 폼 데이터가 전송될 URL을 지정합니다.
- **`method` 속성**: HTTP 메서드(`GET` 또는 `POST`)를 지정합니다. HTML은 기본적으로 `PUT`, `PATCH`, `DELETE`를 지원하지 않으므로 Fetch API나 Express의 `method-override` 패키지를 사용하세요.
- **클라이언트 측 유효성 검사**: 즉각적인 피드백을 위해 `required`와 `pattern` 속성을 사용하세요. 서버에서도 항상 검증해야 합니다.
- **`GET` 폼**: 데이터가 URL 쿼리 매개변수로 추가됩니다(예: 검색 폼).
- **`POST` 폼**: 데이터가 요청 본문에 포함되어 전송됩니다(예: 로그인 및 회원가입 폼).

## 웹 표준 모델 이해하기

### 표준 기구

- **WHATWG**: HTML과 DOM을 지속적으로 업데이트하는 "리빙 스탠다드"로 유지합니다. 주요 브라우저 엔진 팀(Chromium, Gecko, WebKit)이 의사결정을 주도합니다.
- **W3C(월드 와이드 웹 컨소시엄)**: 주요 표준 기구로 CSS, SVG, 접근성 지침(WCAG), 다양한 웹 API를 감독합니다.
- **ECMA International / TC39**: Ecma는 공식 ECMAScript 사양을 발행하며, TC39는 5단계 제안 과정을 통해 언어를 개발합니다(0단계 아이디어 → 4단계 완전 명세).
- **Khronos Group**: 그래픽, 컴퓨팅, 미디어에 집중하며, 브라우저에서 GPU 가속 3D 렌더링을 위한 WebGL과 WebGL 2를 만듭니다.

### 웹 표준 만들기

1. **필요성 확인**: 개발자나 조직이 문제를 발견하고 기록합니다.
2. **제안 및 초기 논의**: 비공식 GitHub 이슈나 설명서를 공유해 관심을 확인합니다. TC39에서는 0단계/1단계에 해당합니다.
3. **명세 작성**: 모든 예외, 오류 조건, 보안 문제를 포함한 정밀한 공식 문서를 작성합니다.
4. **검토, 이의 제기, 반복**: 초안은 공개적으로 브라우저 공급자가 검토하며, 주요 브라우저가 구현을 거부하는 기능은 표준이 되기 어렵습니다.
5. **구현 및 테스트**: 공급자는 기능 플래그 뒤에서 구현하며, 공유 테스트 스위트로 모든 브라우저가 동일하게 동작하는지 확인합니다.
6. **최종화**: W3C는 이를 권고안(Recommendation)이라 부르고, TC39는 4단계라 부릅니다.

### 웹 표준 기능의 수명 주기

1. **제안**: 문제를 확인하고 문서화합니다.
2. **인큐베이션**: 공개 토론 단계로, 많은 제안이 여기서 사라집니다.
3. **명세**: 정확한 규칙을 담은 공식 명세를 작성합니다.
4. **상호운용성 테스트**: W3C 웹 플랫폼 테스트 스위트로 크로스 브라우저 일관성을 검증합니다.
5. **최종화**: 기능이 플래그 없이 기본 웹 플랫폼의 일부로 출시됩니다.
6. **폐기**: 대체되거나 안전하지 않은 기능은 제거 대상으로 표시되지만, 브라우저는 하위 호환성을 위해 유지합니다.

### 웹 표준의 핵심 원칙

- **개방성**: 표준은 공개적으로 개발되고 로열티 없이 구현됩니다.
- **접근성**: HTML, CSS, 브라우저 동작에 내장되어 있으며 WCAG가 공식 지침을 제공합니다.
- **상호운용성**: 표준에 맞게 만든 페이지는 모든 브라우저에서 동일하게 동작해야 합니다.
- **하위 호환성**: 오래된 페이지도 최신 브라우저에서 계속 작동해야 합니다.
- **개인정보 보호 및 보안**: 사용자 데이터를 노출할 수 있는 새 API는 엄격한 검토를 받습니다.
- **계층화**: HTML은 구조, CSS는 표현, JavaScript는 동작을 담당하며 각 계층은 독립적으로 채택할 수 있습니다.
- **점진적 향상**: 기능은 모든 사용자에게 먼저 작동하고, 더 강력한 브라우저 사용자에게는 향상됩니다.

## REST API와 웹 서비스 이해하기

### 웹 서비스

- **API(애플리케이션 프로그래밍 인터페이스)**: 소프트웨어가 다른 소프트웨어와 통신하는 일반적인 메커니즘입니다.
- **웹 서비스**: HTTP/HTTPS 같은 웹 프로토콜을 사용해 네트워크를 통해 데이터를 교환하는 특정 유형의 API입니다. 모든 웹 서비스는 API이지만 모든 API가 웹 서비스는 아닙니다.
- **특징**: 서버에서 실행되고, 머신 간 통신을 가능하게 하며, 언어 및 플랫폼에 독립적이고, 표준 웹 프로토콜을 사용합니다.
- **REST**: 표현 상태 전송으로 가장 널리 사용되는 웹 서비스 스타일입니다. 단순하고 유연하며 HTTP와 자연스럽게 작동합니다.
- **SOAP**: 단순 객체 접근 프로토콜로 XML만 사용하며, 엄격한 표준이 필요한 기업 시스템에서 흔합니다.
- **XML-RPC**: 경량 요청과 응답을 XML로 인코딩합니다.

### REST 아키텍처

- **무상태성**: 각 요청은 독립적으로 처리됩니다. 서버는 요청 간 세션 상태를 저장하지 않습니다. 모든 요청은 필요한 모든 정보를 포함해야 합니다(예: 모든 호출에 인증 토큰).
- **자원**: 모든 것은 URL로 식별되는 자원입니다. 예: 컬렉션은 `/products`, 특정 항목은 `/products/1`.
- **REST의 HTTP 메서드**: `GET`(조회), `POST`(생성), `PUT`/`PATCH`(업데이트), `DELETE`(삭제).
- **응답 형식**: JSON이 가장 흔하지만 XML과 일반 텍스트도 지원됩니다. 클라이언트는 `Accept` 헤더로 형식을 지정합니다:

```bash
Accept: application/json
```

### 마이크로서비스

- **모놀리식**: UI, 데이터베이스, API가 모두 하나의 큰 코드베이스에 있는 구조입니다. 복잡도가 증가하면 유지 관리와 확장이 어렵습니다.
- **마이크로서비스**: 큰 애플리케이션을 작고 독립적인 서비스로 나누는 아키텍처 접근법으로, 각 서비스는 단일 비즈니스 기능을 담당합니다. 서비스들은 API나 메시지 큐를 통해 통신합니다.
- **장점**: 독립적 개발 및 배포가 빠르고, 장애 격리가 우수하며, 서비스별로 다른 언어나 프레임워크를 사용할 수 있습니다.
- **단점**: 디버깅이 복잡하고(문제가 여러 서비스에 걸쳐 있음), 인프라 비용이 높으며, 네트워크 통신이 증가합니다.

## Express 작업하기

- **Express.js**: Node.js 위에 구축된 최소한의 유연한 웹 프레임워크로, 기본 `http` 모듈보다 훨씬 적은 보일러플레이트로 웹 서버와 API를 쉽게 구축할 수 있습니다.
- **기본 Express 서버 예시**:

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**: Express 애플리케이션 인스턴스(`app`)를 생성하는 팩토리 함수입니다.
- **`app.listen(port, callback)`**: 서버를 시작하고 지정한 포트에서 HTTP 요청을 수신합니다.
- **`req`**: 요청 객체로 URL, 헤더, 파라미터, 본문, 쿼리 데이터를 포함합니다.
- **`res`**: 응답 객체로 클라이언트에 데이터를 보낼 때 사용합니다.
- **미들웨어**: 요청을 받고 응답을 보내기 전 사이에 실행되는 코드입니다.

## Express.js 라우팅 이해하기

- **라우팅**: 클라이언트가 특정 URL에 특정 HTTP 메서드로 방문할 때 애플리케이션이 어떻게 응답할지 정의합니다.
- **`app.get/post/put/delete(path, handler)`**: 각각의 HTTP 메서드에 대한 핸들러를 등록합니다.
- **정적 라우트**: 고정된 URL 패턴, 예: `app.get("/home", handler)`.
- **매개변수가 있는 동적 라우트**: `:paramName` 자리 표시자를 사용하며, `req.params`로 캡처합니다:

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**: 라우트 매개변수(`:id` 세그먼트).
- **`req.query`**: 쿼리 문자열 매개변수(`?key=value`).
- **`req.body`**: 요청 본문(본문 파싱 미들웨어 필요).
- **`res.send(data)`**: 응답을 보냅니다(텍스트, HTML, JSON, 바이너리).
- **`res.json(obj)`**: JSON 응답을 보내며 자동으로 `Content-Type: application/json`을 설정합니다.
- **`res.status(code).send(msg)`**: 상태 코드를 설정하고 응답을 보냅니다.
- **`res.redirect(url)`**: 클라이언트를 다른 URL로 리다이렉트합니다.
- **`res.render(view, data)`**: 템플릿 엔진 뷰를 렌더링해 HTML을 보냅니다.

### `app.route()`로 체인 가능한 라우트 핸들러

같은 경로에 대해 여러 HTTP 메서드를 그룹화해 중복을 줄입니다:

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

관련 라우트를 그룹화하는 모듈식 미니 애플리케이션을 생성합니다:

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

- **`express.static(directory)`**: 지정한 디렉터리에서 파일을 제공하는 내장 미들웨어입니다:

```js
app.use(express.static(path.join(__dirname, "public")));
```

## 오류 처리와 상태 점검 이해하기

### Express에서 오류 처리

- **오류 처리 미들웨어**: 네 개의 매개변수 `(err, req, res, next)`로 식별되며, 항상 미들웨어 스택의 마지막에 위치합니다:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **Express 4 비동기 오류**: `try/catch`로 수동으로 잡아 `next(err)`로 전달해야 합니다:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **Express 5 비동기 오류**: 자동으로 오류 처리 미들웨어로 전달되며 수동 `next(err)`가 필요 없습니다.
- **404 핸들러**: 모든 라우트 뒤, 오류 처리기 앞에 배치하세요:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### 디버깅과 로깅

- **내장 디버그**: `DEBUG` 환경 변수를 설정해 활성화합니다: `DEBUG=express:* node index.js`
- **`morgan`**: 인기 있는 서드파티 HTTP 요청 로거 미들웨어로, 메서드, URL, 상태 코드, 응답 시간을 기록합니다:

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### 상태 점검과 우아한 종료

- **헬스 체크 라우트**: 앱이 실행 중임을 알리기 위해 `200 OK`를 반환합니다. 로드 밸런서와 컨테이너 오케스트레이터에서 사용합니다:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**: 프로세스를 종료해야 할 때 보내는 신호입니다. 진행 중인 요청이 완료될 수 있도록 처리하세요:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**: 로컬 개발 중 Ctrl+C를 눌렀을 때 보내는 신호입니다. `SIGTERM`과 동일하게 처리하세요.
- **종료 전 비정상 상태 표시**: 종료 중에 헬스 체크 엔드포인트에서 `503`을 반환하여 로드 밸런서가 새 트래픽 라우팅을 중지하도록 하세요:

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## Express 미들웨어

- **미들웨어**: 요청 수명 주기 동안 실행되는 함수입니다. `req`, `res`, `next`에 접근할 수 있습니다. 다음 미들웨어로 제어를 넘기려면 `next()`를 호출하고, 호출하지 않으면 사이클이 종료됩니다.
- **애플리케이션 레벨 미들웨어**: `app.use()`를 통해 전체 앱에 바인딩되며 모든 들어오는 요청에 대해 실행됩니다.
- **라우터 레벨 미들웨어**: 특정 `express.Router()` 인스턴스에 `router.use()`로 바인딩되며 해당 라우터가 처리하는 라우트에만 적용됩니다.
- **에러 처리 미들웨어**: 네 개의 매개변수 `(err, req, res, next)`로 식별되며 스택의 맨 아래에 위치합니다.
- **`express.json()`**: JSON 페이로드가 포함된 요청을 파싱하여 `req.body`에서 데이터를 사용할 수 있게 합니다.
- **`express.urlencoded({ extended: true })`**: URL 인코딩된 폼 데이터를 파싱합니다.
- **`express.static('public')`**: 지정한 디렉터리에서 정적 파일을 제공합니다.
- **`cors`**: 교차 출처 리소스 공유를 활성화합니다. `npm install cors`로 설치하고 `app.use(cors())`로 사용하세요.
- **`morgan`**: HTTP 요청을 로깅합니다. `app.use(morgan('tiny'))`로 사용하세요.
- 미들웨어는 `app.use()`에 등록된 순서대로 적용됩니다.

## WebSockets 이해하기

- **WebSockets**: 클라이언트와 서버 간에 지속적이고 양방향(풀 듀플렉스) 연결을 생성하여 실시간 통신을 가능하게 하는 프로토콜입니다.
- **실시간에 HTTP가 문제인 이유**: HTTP는 클라이언트가 새 데이터를 받기 위해 서버를 반복적으로 폴링해야 하므로 채팅 앱, 실시간 점수판, 멀티플레이어 게임에 느리고 비효율적입니다.
- **WebSocket 핸드셰이크**: 모든 WebSocket 연결은 HTTP 요청으로 시작합니다. 클라이언트가 업그레이드를 요청하고 서버가 `101 Switching Protocols`로 응답합니다. 이후 연결은 어느 쪽이든 종료 신호를 보낼 때까지 유지됩니다.
- **사용 사례**: 채팅 애플리케이션, 온라인 멀티플레이어 게임, 실시간 스포츠 대시보드, 주식 시장 피드, 협업 도구.

### `ws` 라이브러리

- Node.js에는 내장 WebSocket 서버가 없으며, `ws` 라이브러리가 일반적으로 사용됩니다.

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**: 특정 클라이언트에게 메시지를 보냅니다.
- **모든 연결된 클라이언트에게 방송하기**:

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**: 모든 연결에 있는 속성입니다. `WebSocket.OPEN`은 연결이 활성 상태임을 의미합니다.

### Pub/Sub 아키텍처

- **Pub/Sub (발행/구독)**: **발행자**가 **주제**에 메시지를 보내고, **구독자**가 자신이 구독한 주제에서 메시지를 받는 메시징 패턴입니다. 발행자와 구독자는 직접 통신하지 않습니다.
- **직접 통신 대비 장점**: 컴포넌트 간 결합도를 낮추며, 발행자는 구독자가 누구인지 알 필요가 없습니다.
- **Pub/Sub와 WebSockets 비교**: WebSockets는 통신 채널(도로)이고, Pub/Sub는 누가 어떤 메시지를 받는지 결정하는 메시징 패턴(교통 시스템)입니다.

## 인증과 권한 부여 소개

### 인증과 권한 부여

- **인증**: 신원을 확인하며 “당신은 누구인가요?”라는 질문에 답합니다. 권한 부여 전에 반드시 수행해야 합니다.
- **권한 부여**: 권한을 결정하며 “무엇을 할 수 있나요?”라는 질문에 답합니다.
- **세 가지 인증 요소**:
  - **알고 있는 것**: 비밀번호, PIN, 보안 질문 답변.
  - **가지고 있는 것**: SMS 코드를 받는 스마트폰, 인증 앱 토큰.
  - **본인 자체**: 지문, 얼굴 인식(생체 인식).
- **권한 부여의 일반적인 역할**: 표준 사용자(제한된 셀프 서비스), 편집자(콘텐츠 관리), 관리자(전체 시스템 접근).
- 백엔드 코드에서는 보통 미들웨어가 사용자 역할이나 권한을 확인해 접근을 허용합니다.

### 교차 사이트 요청 위조(CSRF)

- **CSRF**: 악성 웹사이트가 인증된 사용자의 브라우저를 속여 백엔드 서버에 원하지 않는 요청을 하도록 하는 공격으로, 브라우저의 자동 쿠키 첨부 기능을 악용합니다.
- **공격 흐름**: 사용자가 `trusted-bank.com`에 로그인 → `evil-hacker-site.com` 방문 → 악성 사이트가 은밀한 폼을 은행에 제출 → 은행 서버가 유효한 세션 쿠키를 보고 요청을 실행.
- **`SameSite=Strict`**: 요청이 제3자 웹사이트에서 시작되면 브라우저가 쿠키를 절대 보내지 않습니다.
- **`SameSite=Lax`**: 사용자가 안전하게 대상 사이트로 이동할 때(링크 클릭)만 쿠키를 허용하지만, 교차 사이트 폼 제출 시에는 차단합니다.
- **CSRF 토큰**: 서버가 폼에 임베드하는 무작위 예측 불가능한 토큰입니다. 클라이언트는 상태 변경 요청마다 토큰을 포함해야 하며, 악성 사이트는 프런트엔드에서 토큰을 읽을 수 없어 위조 요청이 유효성 검사에 실패합니다.

### JSON 웹 토큰(JWT)

- **JWT**: 로그인 성공 후 서버가 발급하는 디지털 서명 토큰입니다. 클라이언트가 저장하고 이후 모든 요청에 보내 무상태 인증(서버 세션 저장 불필요)을 가능하게 합니다.
- **JWT 구조**: 점으로 구분된 세 개의 Base64URL 인코딩 부분:
  - **헤더**: 토큰 유형(`JWT`)과 서명 알고리즘(예: `HS256`).
  - **페이로드**: 사용자에 관한 클레임(사용자 ID, 사용자명, 역할). 암호화되지 않으므로 비밀번호나 민감한 데이터는 절대 저장하지 마세요.
  - **서명**: 헤더 + 페이로드 + 서버 비밀키의 해시로 변조를 방지합니다.

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

- 사용자가 페이로드를 변조하면 서명이 일치하지 않아 서버가 `401 Unauthorized`로 요청을 거부합니다.

### Passport.js

- **Passport.js**: Node.js용 인증 미들웨어입니다. 인증 흐름을 표준화하고 저수준 세부사항을 처리해 개발자가 비즈니스 로직에 집중할 수 있게 합니다.
- **전략(Strategies)**: 특정 인증 방식을 구현하는 모듈형 플러그인입니다. 500개 이상의 전략이 있으며, 예: `passport-local`(사용자명/비밀번호), `passport-jwt`(JWT), `passport-google-oauth20`(구글 로그인), `passport-facebook`, `passport-github2`.
- **Express에 연결하기**:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **라우트 미들웨어로 사용하기**:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- 인증 성공 후 Passport는 세션 쿠키에 사용자 데이터를 저장하고 인증된 사용자 객체를 `req.user`에 붙입니다.

### Helmet.js

- **Helmet.js**: Express 앱을 보호하는 보안 미들웨어 모음으로, HTTP 응답 헤더를 자동으로 설정, 수정 또는 제거합니다.

```js
const helmet = require('helmet');
app.use(helmet());
```

- **`X-Powered-By` 제거**: 공격자가 서버 기술을 알지 못하게 합니다.
- **`Content-Security-Policy` 설정**: 스크립트, 이미지, 스타일을 로드할 수 있는 출처를 제한해 XSS 공격을 완화합니다.
- **`X-Frame-Options: SAMEORIGIN` 설정**: 외부 iframe에 사이트가 임베드되는 것을 막아 클릭재킹을 방지합니다.
- **`Strict-Transport-Security` 설정**: 브라우저가 HTTPS를 사용하도록 강제해 데이터 가로채기를 방지합니다.

## 웹 애플리케이션의 보안과 개인정보 보호 이해하기

### 보안과 개인정보 보호

- **보안**: 시스템과 데이터를 무단 접근으로부터 보호하는 기술적 방어 메커니즘입니다. 보안은 문에 자물쇠를 겁니다.
- **개인정보 보호**: 개인 데이터가 어떻게 수집, 사용, 공유되는지 관리하는 것입니다. 개인정보 보호는 창문에 커튼을 칩니다.
- 보안은 있어도 개인정보 보호가 없을 수 있지만(잠긴 데이터베이스가 데이터를 팔 수 있음), 개인정보 보호는 보안 없이는 불가능합니다(보호되지 않은 데이터는 비공개로 유지될 수 없음).

### 동일 출처 정책과 CORS

- **출처(Origin)**: 프로토콜(예: `https`), 도메인(예: `example.com`), 포트(예: `443`) 세 가지 요소로 정의됩니다. 두 URL이 동일 출처를 가지려면 세 가지가 모두 일치해야 합니다.
- **동일 출처 정책**: 브라우저 보안 규칙으로, 한 출처의 스크립트가 다른 출처의 데이터를 읽는 것을 차단해 악성 사이트가 다른 탭의 데이터를 훔치는 것을 방지합니다.
- **CORS(교차 출처 리소스 공유)**: 서버가 HTTP 헤더를 설정해 신뢰할 수 있는 출처에 대해 동일 출처 정책을 완화하는 메커니즘입니다.
- **Express 개발 환경**: `app.use(cors())`로 모든 출처를 허용합니다.
- **Express 프로덕션 환경**: `corsOptions` 객체에 허용할 `origin` 목록을 전달하세요:

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**: TLS(전송 계층 보안)를 통해 암호화된 HTTP입니다. TLS는 이전의 SSL 프로토콜을 대체했습니다.
- **HTTPS의 세 가지 보장**:
  - **암호화**: 클라이언트와 서버 간 전송되는 데이터가 제3자에게 읽히지 않습니다.
  - **인증**: 신뢰할 수 있는 인증 기관에서 발급한 디지털 인증서가 서버의 신원을 확인합니다.
  - **데이터 무결성**: 전송 중 메시지가 변조되면 감지됩니다.
- HTTPS는 지리 위치, 푸시 알림, 카메라 접근 같은 최신 브라우저 API에 필수입니다.

### 보안 문제가 발생하는 이유

- **빠른 배포 압박**: 마감일을 맞추기 위해 너무 빨리 움직이는 팀은 보안 검토를 건너뛰고 취약점을 배포합니다.
- **복잡한 시스템 통합**: 서드파티 라이브러리나 클라우드 서비스가 개발팀이 작성하지 않은 취약점을 도입할 때 공급망 위험이 발생합니다.
- **부실한 구성 관리**: 기본 관리자 비밀번호, 열린 포트 등 기본 설정을 그대로 두면 시스템이 불필요하게 노출됩니다.
- **인간 오류**: 피싱 공격은 악성 링크나 첨부파일을 클릭하고 자격 증명을 넘기는 직원을 노립니다.
- **조직 합병**: 서로 다른 코드베이스와 인프라를 합치면서 보안 정책 불일치와 레거시 취약점이 드러납니다.

### 쿠키: 저장, 보안 플래그, 개인정보 보호

- **쿠키 저장 방식**: 서버가 `Set-Cookie` 헤더를 보내면 브라우저가 디스크에 저장하고 이후 같은 출처 요청에 자동으로 쿠키를 반환합니다.
- **`HttpOnly` 플래그**: JavaScript가 쿠키 값을 읽지 못하게 해 XSS 기반 세션 탈취를 방지합니다.
- **`Secure` 플래그**: 쿠키를 HTTPS 연결에서만 전송하도록 제한해 공용 네트워크에서 가로채기를 막습니다.
- **`SameSite=Strict`**: 모든 교차 사이트 요청에 쿠키 전송을 차단해 CSRF 공격을 완전히 막습니다.
- **`SameSite=Lax`**: 최상위 네비게이션(링크 클릭) 시 쿠키를 허용하지만 교차 사이트 폼 제출과 API 호출은 차단합니다.
- **퍼스트파티 쿠키**: 방문 중인 사이트에 한정됩니다.
- **서드파티 쿠키**: 광고 네트워크가 수천 개 사이트에서 사용자 행동을 추적하기 위해 설정합니다. 최신 브라우저는 기본적으로 차단합니다.
- **GDPR 준수**: 개인정보 보호법은 사용자가 비필수 추적 쿠키를 거부할 권리를 가진 쿠키 동의 배너를 요구합니다.

### 일반적인 보안 위협과 대응책

- **피싱**: 신뢰할 만한 회사를 사칭해 자격 증명을 훔치는 사기 이메일입니다. 대응책으로는 보안 인식 교육과 이메일 인증 프로토콜 배포가 있어 악성 메시지가 받은편지함에 도달하기 전에 차단합니다.
- **악성코드**(바이러스, 스파이웨어, 랜섬웨어): 랜섬웨어는 파일을 암호화하고 해제 대가를 요구합니다. 대응책은 자동 업데이트와 안티멀웨어 소프트웨어입니다.
- **자격 증명 채우기 공격**: 공격자가 도난당한 수백만 개 사용자명과 비밀번호 조합을 여러 사이트에서 시험합니다. 주요 대응책은 OTP나 인증 앱 코드 같은 2차 인증(MFA)입니다.
- **DDoS(분산 서비스 거부)**: 공격자가 가짜 트래픽으로 서버를 마비시켜 정상 사용자를 차단합니다. 대응책은 Cloudflare, Akamai, AWS Shield 같은 클라우드 기반 트래픽 필터링 서비스입니다.
- **내부자 위협**: 악의적이거나 부주의한 직원이 데이터를 훔치거나 유출합니다. 대응책은 최소 권한 원칙을 적용해 직원이 업무에 필요한 것만 접근하게 하고, 이상 활동을 모니터링하는 것입니다.

### 콘텐츠 보안 정책과 권한 정책

- **CSP(콘텐츠 보안 정책)**: 스크립트, 스타일, 이미지에 대해 허용된 출처 도메인을 화이트리스트로 지정하는 HTTP 응답 헤더입니다. 브라우저가 CSP 헤더를 받으면 목록에 없는 도메인에서 로드된 리소스를 차단합니다. `script-src` 지시문은 JavaScript 실행 도메인을 지정해 XSS 공격을 막습니다.
- **Permissions-Policy**: 페이지와 임베드된 iframe의 브라우저 기능 및 장치 하드웨어 사용을 활성화하거나 비활성화하는 HTTP 헤더입니다. 이전의 `Feature-Policy` 헤더를 대체했습니다.
  - `camera=(self)`는 웹캠 접근을 페이지 자신의 출처로 제한합니다.
  - `geolocation=()`는 모든 사용자에 대해 위치 정보를 완전히 비활성화합니다.
- 두 헤더를 함께 사용하면 다층 보호가 가능하며, CSP는 코드 주입을 막고 Permissions-Policy는 무단 하드웨어 접근을 방지합니다.

### 기본 사용자 개인정보 보호 개념

- **PII(개인 식별 정보)**: 개인을 직접 또는 간접적으로 식별할 수 있는 모든 데이터입니다.
  - **직접 식별자**: 전체 이름, 집 주소, 주민등록번호.
  - **간접 식별자**: IP 주소, 장치 일련번호, 정확한 위치 기록.
- **기밀성**: 암호화와 접근 제어를 통해 PII를 무단 접근으로부터 안전하게 지키는 것입니다.
- **데이터 최소화**: 특정 작업을 완료하는 데 꼭 필요한 최소한의 PII만 수집합니다.
- **목적 제한**: 한 가지 이유로 수집한 데이터는 새로운 사용자 동의 없이 다른 용도로 재사용할 수 없습니다(예: 광고 네트워크에 판매).
- **추적**: 스크립트, 장치 지문, 브라우저 기록 로그를 사용해 온라인 사용자 행동을 지속적으로 모니터링합니다.
- **잊힐 권리(데이터 삭제)**: 계정 종료 후 회사 데이터베이스에서 모든 개인 기록을 영구 삭제할 법적 권리입니다.

### `.env` 파일과 `dotenv`

- **`.env` 파일**: 프로젝트 루트에 위치한 평문 파일로, API 키, 데이터베이스 비밀번호, 포트 번호 같은 구성 비밀을 키-값 쌍으로 저장합니다. 항상 `.gitignore`에 추가해 버전 관리에서 비밀을 제외하세요.
- **`dotenv` 패키지**: 시작 시 `.env` 파일을 읽어 `process.env`에 값을 주입합니다:

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **두 가지 주요 이점**: 자격 증명이 소스 코드에 포함되지 않아 보안이 강화되고, 각 서버에서 `.env` 파일을 교체해 개발, 스테이징, 프로덕션 환경에서 동일한 코드베이스가 다르게 동작하도록 유연성을 제공합니다.

### 지역별 개인정보 보호법과 준수

- **GDPR(일반 데이터 보호 규정)**: EU 내 개인을 보호합니다. 추적 전에 명시적 옵트인 동의를 요구하며, 사전 선택된 박스는 허용하지 않습니다. 사용자가 자신의 데이터를 조회하고 영구 삭제할 권리를 부여하며, 위반 시 무거운 벌금을 부과합니다.
- **CCPA(캘리포니아 소비자 개인정보 보호법) / CPRA(캘리포니아 개인정보 권리법)**: 캘리포니아 주민을 보호합니다. 옵트인보다 옵트아웃 권리에 중점을 두며, 홈페이지에 "내 개인정보 판매 금지" 링크를 눈에 띄게 표시해야 합니다.
- **COPPA(아동 온라인 개인정보 보호법)**: 13세 미만 아동을 보호하는 미국 연방법입니다. 명확한 개인정보 보호 정책과 부모의 검증된 동의 없이는 아동의 개인정보 수집을 금지합니다.
- **DPA(데이터 보호법)**: 영국의 주요 개인정보 보호 프레임워크입니다. 영국 GDPR과 함께 작동하며 사용자가 개인정보 접근 및 삭제 권리를 가집니다.
- 다른 국가들도 자체 법률이 있습니다: 나이지리아는 NDPA(나이지리아 데이터 보호법), 인도는 DPDP법(디지털 개인정보 보호법)이 있습니다.
- **글로벌 팀을 위한 준수 단계**:
  - 사용자의 지리적 위치에 따라 적절한 쿠키 알림을 표시하는 위치 인식 동의 배너를 구축하세요.
  - 법적 요청 시 사용자의 데이터를 찾고, 내보내고, 영구 삭제하는 자동화된 DSAR(데이터 주체 접근 요청) 시스템을 구현하세요.
  - 데이터 최소화 원칙을 실천하고 미성년자 추적을 방지하기 위해 연령 확인 게이트를 사용하세요.
  - 개인정보 보호 아키텍처를 감독하고 규제 당국의 주요 연락 담당자인 데이터 보호 책임자(DPO)를 지정하세요.
