# --description--

Nella lezione precedente, hai imparato come un server backend identifica un utente durante il login. Ma cosa succede dopo il login? Come fa il server a ricordare chi è l'utente quando naviga in una nuova pagina? Qui entrano in gioco i JSON Web Token (JWT).

In parole semplici, un JWT è una tessera digitale che il server rilascia a un utente dopo un login riuscito. L'utente presenta questa tessera al server a ogni richiesta successiva per dimostrare la propria identità senza dover reinserire la password.

Invece di far ricordare al server ogni utente attivo nella sua memoria, la responsabilità viene spostata sul client tramite un semplice processo in quattro passaggi:

1. **Login:** L'utente invia il proprio nome utente e password, o altri metodi di accesso, al server.

2. **Rilascio:** Il server verifica le credenziali confrontandole con quelle memorizzate al momento della registrazione. Se sono corrette, il server genera un JWT e lo invia all'utente.

3. **Archiviazione:** Il client riceve il JWT e lo conserva, tipicamente in un cookie del browser o nel local storage.

4. **Richiesta:** Per ogni futura richiesta a una rotta protetta, il client invia automaticamente il JWT. Il server verifica il token e concede l'accesso.

Un JWT sembra una lunga stringa caotica di caratteri. È composto da tre parti distinte separate da due punti, così:

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

La prima parte è l'**intestazione**, la seconda è il **payload** e la terza è la **firma**.

L'**intestazione** è un oggetto JSON codificato che indica al server che tipo di token è (JWT) e quale algoritmo di hashing è stato usato per proteggerlo, come HMAC SHA256.

Il **payload** è la parte centrale del token. Contiene i dati effettivi, o "claims", sull'utente. Qui si memorizzano informazioni come l'id utente, il nome utente o il ruolo dell'account, ad esempio admin o user.

Una nota importante sulla sicurezza è che intestazione e payload sono solo codificati (Base64), non criptati. Questo significa che chiunque può facilmente decodificarli e leggere i dati al loro interno. Quindi, non memorizzare mai dati sensibili come password in chiaro o numeri di carta di credito nel payload di un JWT.

La **firma** è la parte finale e più cruciale del token. Il server prende l'intestazione codificata, il payload codificato e li combina con una chiave segreta conosciuta solo dal server. Poi li sottopone a hashing per creare la firma.

La firma impedisce manomissioni. Se un utente prova a decodificare il proprio token, cambiare il ruolo da "user" a "admin" e ricodificarlo, la firma non corrisponderà più ai dati alterati.

Quando il server riceve il token modificato, tenta di ricreare la firma usando la sua chiave segreta. Poiché il payload è cambiato, la firma calcolata non corrisponderà a quella del token e il server rifiuterà immediatamente la richiesta con un codice di stato 401 Unauthorized.

In breve, i JWT permettono un'autenticazione senza stato. Il server non deve memorizzare dati di sessione in un database, si fida semplicemente del token perché la firma dimostra che è stato creato dal server stesso.
