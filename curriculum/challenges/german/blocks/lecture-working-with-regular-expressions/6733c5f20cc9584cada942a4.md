# --interactive--

Eine Capturing Group ermöglicht es Ihnen, einen Teil des übereinstimmenden Strings „einzufangen“, um ihn nach Belieben zu verwenden. Capturing Groups werden durch runde Klammern definiert, die das zu erfassende Muster enthalten, ohne führende Zeichen wie eine Vorausschau.

Fangen wir den `code` aus unserem `freeCodeCamp`-regulären Ausdruck ein. Dazu schließen wir `code` in runde Klammern ein und definieren es als Capturing Group:

```js
const regex = /free(code)camp/i;
```

Um das Verhalten zu bestätigen, können wir es an einem `freecodecamp`-String testen:

:::interactive_editor

```js
const regex = /free(code)camp/i;
console.log(regex.test("freecodecamp")); // true
```

:::

Aber das nutzt unsere erfasste Gruppe noch nicht wirklich. Schauen wir uns stattdessen das Ergebnis der Verwendung von `match` an:

```js
const regex = /free(code)camp/i;
console.log("freecodecamp".match(regex));
// [
//   'freecodecamp',
//   'code', <--
//   index: 0,
//   input: 'freecodecamp',
//   groups: undefined
// ]
```

Hier sehen wir, dass unser `match`-Array ein zweites Element hat, das der Teil des Strings ist, der von unserer Capturing Group erfasst wurde.

Beachten Sie, wie die Capturing Group genau das Muster `code` erfasst, wobei eine Zeichenklasse ein einzelnes Zeichen aus der Liste `c`, `o`, `d` und `e` erfassen würde.

Aber wie können wir das tatsächlich nutzen? Capturing Groups werden oft verwendet, wenn Inhalte eines Strings ersetzt werden. Richten wir etwas Code dafür ein. Wir wollen `freecodecamp` in `paidcodeworld` umwandeln:

:::interactive_editor

```js
const regex = /free(code)camp/i;
console.log("freecodecamp".replace(regex, "paidcodeworld"));
```

:::

Das funktioniert für sich allein, aber was, wenn wir nicht wissen, wie viele `o` in `code` sind? Wenn wir einen Quantifizierer für ein oder mehrere `o` brauchen:

:::interactive_editor

```js
const regex = /free(co+de)camp/i;
console.log("freecoooooooodecamp".replace(regex, "paidcodeworld"));
```

:::

Wir erhalten `paidcodeworld` als Ergebnis. Wir wollen die Anzahl der `o` erhalten, also müssen wir das, was vom regulären Ausdruck erfasst wurde, wiederverwenden.

Hier kommt eine Backreference ins Spiel. Anstatt den `code`-Teil unseres Ersetzungsstrings fest zu codieren, können wir direkt auf die erfasste Gruppe verweisen.

In einem `replace`-Aufruf erreichen Sie eine Backreference, indem Sie ein Dollarzeichen (`$`) gefolgt von der Nummer der zu verwendenden Capturing Group verwenden. In unserem Fall wäre das `$1`, da `code` in der ersten Capturing Group erfasst wird:

:::interactive_editor

```js
const regex = /free(co+de)camp/i;
console.log("freecoooooooodecamp".replace(regex, "paid$1world")); // paidcoooooooodeworld
```

:::

Wir haben nun erfolgreich eine unbekannte Anzahl von `o`-Zeichen erhalten, wenn wir `freecodecamp` in `paidcodeworld` umwandeln. Aber Backreferences sind nicht nur auf den replace-Aufruf beschränkt. Sie können sie tatsächlich direkt in einem regulären Ausdruck verwenden.

Das würde es ermöglichen, ein zuvor erfasstes Muster später im regulären Ausdruck zu matchen.

Angenommen, wir wollen `freecodecamp` zweimal mit der gleichen Anzahl von `o` matchen, aber irgendwo im String.

Zuerst müssen wir sie mit unserem Wildcard-Zeichen trennen und erlauben, dass beliebig viele Zeichen dieses Wildcard-Zeichen matchen:

```js
const regex = /free(co+de)camp.*free(co+de)camp/i;
```

Dieser Ausdruck stellt jedoch nicht sicher, dass die Anzahl der `o`-Zeichen gleich ist. Um das zu erreichen, müssen wir die zweite Capturing Group durch eine Referenz auf die erste ersetzen.

Innerhalb eines regulären Ausdrucks wird eine Backreference mit einem Backslash gefolgt von der Nummer der Capturing Group bezeichnet:

:::interactive_editor

```js
const regex = /free(co+de)camp.*free\1camp/i;
console.log(regex.test("freecooooodecamp is great i love freecooooodecamp")); // true
console.log(regex.test("freecooooodecamp is great i love freecodecamp")); // false
```

:::

Und damit sehen wir, dass ein String mit der korrekten Anzahl von `o` matcht, während ein String mit zwei unterschiedlichen Zahlen von `o` nicht matcht.

Diese Syntax ist großartig, kann aber schnell verwirrend werden, wenn Sie mehrere Capturing Groups referenzieren. Zum Glück können Sie anstelle von Zahlen Ihren Gruppen Namen geben.

Um eine benannte Capturing Group zu definieren, fügen Sie ein Fragezeichen (`?`) gefolgt vom Namen in spitzen Klammern an den Anfang der Gruppe. Nennen wir unsere Capturing Group `code`:

```js
const regex = /free(?<code>co+de)camp.*free\1camp/i;
```

Jetzt können wir unsere Backreference im regulären Ausdruck aktualisieren, um auf diese Gruppe zu verweisen. Eine benannte Backreference beginnt mit einem Backslash gefolgt vom Buchstaben `k` in JavaScript. Dann fügen Sie den Namen hinzu, wieder in spitze (`<`) und spitze (`>`) Klammern eingeschlossen. Schauen wir uns das an:

```js
const regex = /free(?<code>co+de)camp.*free\k<code>camp/i;
```

Wenn wir nun unseren `test()`-Aufruf prüfen, sehen wir, dass wir immer noch bestehen:

:::interactive_editor

```js
const regex = /free(?<code>co+de)camp.*free\k<code>camp/i;
console.log(regex.test("freecooooodecamp is freecooooodecamp")); // true
```

:::

Um unsere benannte Capturing Group in einem `replace()`-Aufruf zu verwenden, fügen wir ein Dollarzeichen in den String ein, gefolgt vom Namen in spitzen Klammern:

:::interactive_editor

```js
const regex = /free(?<code>co+de)camp/i;
console.log("freecooooodecamp".replace(regex, "paid$<code>camp")); // paidcooooodecamp
```

:::

Manchmal möchten Sie schließlich eine Gruppe von Zeichen erstellen, brauchen das erfasste Ergebnis aber nicht.

Angenommen, wir wollen entweder `freecodecamp` oder `freecandycamp` matchen. Sie könnten zwei Muster mit einer ODER-Zusicherung trennen:

```js
const regex = /freecodecamp|freecandycamp/i;
```

Das kann bei größeren regulären Ausdrücken ziemlich lang werden. Stattdessen können Sie eine non-capturing Group um die Zeichen erstellen, die Sie mit ODER verbinden müssen:

```js
const regex = /free(?:code|candy)camp/i;
```

Eine non-capturing Group speichert die `code|candy`-Übereinstimmung nicht separat im Speicher. Aber sie kann hilfreich sein, um alternative Muster zu erstellen, ohne Lesbarkeit oder Leistung zu opfern.
