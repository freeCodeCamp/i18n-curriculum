# --description--

Immer wenn Sie eine Website besuchen oder eine API verwenden, kommunizieren Ihr Browser und der Server ständig miteinander.

Einer der wichtigsten Teile dieses Gesprächs? HTTP-Antwortstatuscodes.

Das sind kurze numerische Nachrichten – wie 200, 404 oder 500 – die dem Client (dem Endgerät des Nutzers wie Laptop oder Telefon) mitteilen, was nach einer Anfrage passiert ist.

Sie haben bereits einige davon kennengelernt, aber lassen Sie uns diese nach Kategorien aufschlüsseln, damit Sie genau wissen, was sie bedeuten und wann Sie sie verwenden sollten.

## 1xx – Informationsantworten

Codes, die mit 1 beginnen, sieht man in der täglichen Entwicklung selten, aber sie zeigen an, dass die Anfrage empfangen wurde und der Prozess weiterläuft, zum Beispiel beim Hochladen einer Datei.

Häufiges Beispiel:

* 100 Continue – Der Server sagt dem Client: „Okay, Sie können den Rest der Anfrage senden.“
* 101 Switching Protocols – Der Client hat einen `Upgrade` Header gesendet, daher wechselt der Server das Protokoll. Das sehen Sie am häufigsten bei der Verwendung von Websockets.

## 2xx – Erfolgsantworten

Diese bedeuten, dass die Anfrage erfolgreich empfangen, verstanden und verarbeitet wurde.

* 200 OK – Die Anfrage war erfolgreich. Dies ist die häufigste Erfolgsantwort.
* 201 Created – Eine neue Ressource wurde erfolgreich erstellt, zum Beispiel nach der Registrierung eines neuen Nutzers.
* 204 No Content – Die Anfrage war erfolgreich, aber es gibt keinen Antwortkörper zurückzusenden. Wird oft für `DELETE` Anfragen verwendet.

## 3xx – Umleitungsnachrichten

Diese sagen dem Client: „Die Ressource, die Sie suchen, befindet sich woanders.“

* 301 Moved Permanently – Die Ressource hat eine neue dauerhafte URL.
* 302 Found – Die Ressource befindet sich vorübergehend unter einer anderen URL.
* 304 Not Modified – Wird für Caching verwendet. Es sagt dem Client, dass sich nichts geändert hat, sodass er die zwischengespeicherte Version verwenden kann.

## 4xx – Client-Fehler

Diese treten auf, wenn der Client eine ungültige Anfrage sendet.

* 400 Bad Request – Die Anfrage ist fehlerhaft oder ungültig.
* 401 Unauthorized – Authentifizierung ist erforderlich (Sie müssen sich anmelden).
* 403 Forbidden – Die Authentifizierung war erfolgreich, aber Sie haben keine Berechtigung.
* 404 Not Found – Die angeforderte Ressource existiert nicht. Und ja, das ist der bekannteste Fehler im Web.

## 5xx – Server-Fehler

Diese bedeuten, dass auf der Serverseite etwas schiefgelaufen ist.

* 500 Internal Server Error – Eine allgemeine „etwas ist schiefgelaufen“-Meldung.
* 502 Bad Gateway – Ein Server, der als Proxy fungiert, hat eine ungültige Antwort von einem anderen Server erhalten.
* 503 Service Unavailable – Der Server ist vorübergehend überlastet oder wegen Wartung nicht erreichbar.

Jetzt, da Sie mit einigen der häufigsten HTTP-Statuscodes vertraut sind, schauen wir uns einige praktische Beispiele in Express an.

Angenommen, Sie bauen eine API. Wenn ein Nutzer ungültige Daten sendet, können Sie mit dem Statuscode 400 antworten, was „Bad Request“ bedeutet. Sie können auch den String „Bad Request“ an den Client zurücksenden, damit klar ist, was der 400-Antwortcode bedeutet:

```js
res.status(400).send("Bad Request");
```

Wenn alles gut läuft, können Sie eine 200-Antwort senden. Sie können eine Nachricht hinzufügen, wenn Sie möchten. Hier senden wir etwas JSON mit einer einfachen „Success“-Nachricht:

```js
res.status(200).json({ message: "Success" });
```

Und wenn auf dem Server etwas abstürzt, können Sie den Statuscode 500 senden, was bedeutet, dass ein interner Serverfehler aufgetreten ist, zusammen mit einer klaren Fehlermeldung:

```js
res.status(500).send("Internal Server Error");
```

Diese Statuscodes sind nicht nur für Maschinen. Sie helfen Entwicklern, Probleme zu debuggen, und helfen Nutzern (und anderen Entwicklern) zu verstehen, was gerade passiert.
