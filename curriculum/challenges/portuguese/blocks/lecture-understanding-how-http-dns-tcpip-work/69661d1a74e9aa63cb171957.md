# --description--

Quando você visita um site, a primeira coisa que faz é digitar um endereço web no seu navegador.

Por exemplo, para acessar o freeCodeCamp, você deve digitar `freecodecamp.org` na barra de endereços do seu navegador.

Isto é conhecido como um nome de domínio. É um identificador exclusivo legível por humanos para esse recurso na internet.

Nomes de domínio estão muito relacionados ao conceito de DNS, que é o tema principal desta lição. Vamos começar falando um pouco sobre a hierarquia de nomes de domínio, para que você possa entender como um DNS funciona nos bastidores.

A hierarquia do nome de domínio é lida da direita para a esquerda, movendo-se da categoria mais geral para a mais específica.

A estrutura principal de um nome de domínio é, da direita para a esquerda:

* **Domínio de Nível Superior (TLD):** o último segmento do nome de domínio, localizado à direita. No caso de `freecodecamp.org`, isso seria `.org`. Outros exemplos incluem `.com` e `.edu`.
    
* **Domínio de Segundo Nível (SLD):** o nome registrado pela pessoa ou organização. Esse seria `freecodecamp` em `freecodecamp.org`. Isso é o que você geralmente percebe primeiro quando vê um endereço web.
    
* **Domínio de Terceiro Nível (Subdomínio):** usado para identificar uma seção específica do domínio principal. Por exemplo, `www`, que significa “World Wide Web.” Hoje em dia, servidores web modernos lidam automaticamente com o roteamento de domínios, então você pode acessar um site com ou sem digitar `www`. Outros exemplos de subdomínios incluem `blog.`, `shop.`, `support.` e `api.`.
    

Nomes de domínio são muito úteis para nós, humanos, porque são fáceis de lembrar e digitar em um navegador. No entanto, como você sabe, computadores trabalham com números. Seu navegador precisa transformar o nome de domínio que você digitou em algo conhecido como um endereço IP para encontrar o site que você está procurando.

Um **endereço IP** é uma sequência única de números que identifica um computador em uma rede, como a internet.

Endereços IP são essenciais para permitir que clientes enviem solicitações aos servidores corretos e obtenham as respostas apropriadas. Um exemplo de endereço IPv4 é `127.0.0.1`. O endereço IP equivalente em um padrão mais recente chamado IPv6 seria `0:0:0:0:0:0:0:1`.

Há muito para aprender sobre endereços IP e você pode querer explorá-los com mais profundidade à medida que continuar aprendendo. Mas, por enquanto, você só precisa entender que eles são os endereços numéricos únicos usados para identificar computadores em uma rede.

A partir dos conceitos de nomes de domínio e endereços IP, agora podemos chegar ao conceito de DNS.

DNS significa **Domain Name System**. Isso geralmente é referido como a “lista telefônica” da internet.

O DNS traduz o nome de domínio que você digita no seu navegador para o seu endereço IP correspondente. É um sistema de nomes hierárquico e descentralizado para computadores e dispositivos que estão conectados à internet ou a uma rede específica.

Quando o usuário digita um nome de domínio no navegador, como `freecodecamp.org`, o DNS mapeia esse nome para o endereço IP do seu servidor correspondente. Uma vez que o servidor foi identificado, a solicitação do cliente é enviada para o servidor. O servidor processa essa solicitação com base em como foi programado e finalmente envia uma resposta para o cliente. Isso completa o ciclo de solicitação-resposta.

Tudo isso é possível graças ao DNS que realizou aquela conversão inicial para o endereço IP de destino.

Vamos mergulhar nos detalhes do processo de resolução de DNS.

## O Processo de Resolução DNS

* A resolução de DNS funciona enviando uma consulta através de uma cadeia de servidores DNS.

* Quando você (o usuário) digita um nome de domínio no navegador, o **client** envia uma solicitação para um servidor DNS local. Esse servidor DNS local é conhecido como o **Recursive Resolver**. Geralmente, ele é gerenciado pelo seu provedor de serviços de internet (ISP).
    
* O Recursive Resolver encaminha essa solicitação para os **Root Servers**. Estes são servidores gerais que não possuem informações diretas sobre endereços IP. Eles funcionam direcionando o Recursive Resolver para um servidor mais específico, especificamente para o Top-Level Domain (TLD) Server do domínio de nível superior da solicitação. Por exemplo, o servidor responsável por todos os domínios `.org`.
    
* O Recursive Resolver então consulta o **Servidor de Domínio de Nível Superior (TLD)**. Este servidor retorna o endereço do Servidor de Nomes Autoritativo específico que possui os registros para o domínio solicitado (por exemplo, `freecodecamp.org`).
    
* O Recursive Resolver inicia uma consulta final para o correspondente **Authoritative Name Server.** Se o **DNS record** for encontrado, ele é retornado para o Recursive Resolver.
    
* Quando o Recursive Resolver recebe o Endereço IP, ele o entrega ao cliente, para que você possa começar a navegar no site.

O Servidor de Nomes Autorizado retorna **Registros DNS**, não apenas endereços IP. Os Registros DNS são entradas de texto estruturadas que contêm informações importantes sobre um domínio, como informações de subdomínio, por quanto tempo a informação deve ser armazenada em cache e informações necessárias para executar o serviço solicitado.

Para otimizar o processo, o Recursive Resolver armazena em cache temporariamente o DNS Record para acelerar o processo. Dessa forma, você não fará a mesma solicitação repetidamente em um curto período de tempo. Isso torna o processo geral muito mais eficiente.

Como você pode ver, os Servidores DNS são essenciais porque fornecem a etapa inicial e crítica de traduzir nomes de domínio em endereços IP. Esse processo de tradução permite que cada cliente localize e se conecte ao servidor correto. Sem esse sistema de diretório, o modelo cliente-servidor, a espinha dorsal da web moderna, não seria funcional.
