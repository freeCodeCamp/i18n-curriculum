# --description--

在上一节课中，你学习了后端服务器如何在登录时识别用户。但用户登录后会发生什么？当他们导航到新页面时，服务器如何记住他们的身份？这就是 JSON Web Token（JWT）发挥作用的地方。

通俗地说，JWT 是服务器在用户成功登录后发给用户的一张数字钥匙卡。用户在每次后续请求时向服务器出示这张钥匙卡，以证明他们的身份，而无需重新输入密码。

服务器不再在自己的内存中记住每个活跃用户，而是通过一个简单的四步流程将责任转移给客户端：

1. **登录：** 用户将用户名和密码或其他登录方式发送给服务器。

2. **签发：** 服务器将凭证与注册时保存的凭证进行验证。如果正确，服务器生成一个 JWT 并发送给用户。

3. **保存：** 客户端接收 JWT 并保存它，通常保存在浏览器 cookie 或本地存储中。

4. **请求：** 对于每个未来对受保护路由的请求，客户端会自动携带 JWT。服务器验证令牌并授予访问权限。

JWT 看起来像一串长而混乱的字符。它由三个不同部分组成，之间用两个点分隔，如下所示：

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

第一部分是 **头部**，第二部分是 **体**，第三部分是 **签名**。

**头部** 是一个编码的 JSON 对象，告诉服务器令牌的类型（JWT）以及用于保护它的哈希算法，比如 HMAC SHA256。

**体** 是令牌的中间部分。它包含实际的数据，或者说是关于用户的“声明”。这里存储的信息包括用户 ID、用户名或账户角色，比如管理员或用户。

一个重要的安全提示是，头部和体只是编码（Base64），而非加密。这意味着任何人都可以轻松解码并读取其中的数据。所以，绝不要在 JWT 体中存储敏感数据，比如明文密码或信用卡号码。

**签名** 是令牌的最后也是最关键的部分。服务器将编码后的头部和体与只有服务器知道的密钥混合，然后进行哈希运算生成签名。

签名防止篡改。如果用户试图解码令牌，将角色从“user”改为“admin”，然后重新编码，签名将不再匹配被篡改的数据。

当服务器收到被修改的令牌时，会尝试用其密钥重新生成签名。由于体已更改，计算出的签名与令牌中的签名不匹配，服务器会立即以 401 未授权状态码拒绝请求。

简而言之，JWT 允许无状态认证。服务器无需在数据库中保存会话数据，只需信任令牌，因为签名证明了令牌是服务器自己创建的。
