# --description--

ソフトウェアにおいて、依存関係とは、アプリケーション内のあるコンポーネントやモジュールが、正しく動作するために別のものに頼っている状態を指します。依存関係はソフトウェアアプリケーションでよく見られます。なぜなら、開発者が他の人が作成した既成の関数やツールを利用できるからです。

Reactアプリケーションを構築したい場合、Reactの依存関係をインストールする必要があります。これらの依存関係がなければ、コードは正しく動作せず、アプリケーションはエラーのリストを表示します。

より複雑なプロジェクトで作業する場合、多くの依存関係に頼ることがよくあります。依存関係のセットがプロジェクト内で適切に管理または定義されていないと、いわゆる依存関係地獄に陥ることになります。

プロジェクト内のソフトウェア依存関係を管理するには、パッケージマネージャーを使う必要があります。パッケージマネージャーは依存関係のインストール、更新、削除に使われるツールです。JavaScript、Python、Ruby、Javaなど、多くの人気プログラミング言語がパッケージマネージャーを使用しています。

以前のレッスンで、npmという人気のパッケージマネージャーを簡単に紹介しました。

Viteとnpmを使って新しいReactプロジェクトを作成するには、次のコマンドを実行します。

```sh
npm create vite@latest my-react-app -- --template react
```

前のレッスンで覚えているように、これで新しいReactアプリケーションを起動するために必要なすべてのボイラープレートコードが作成されます。アプリケーションを起動する前に、`npm install`または短縮形の`npm i`を実行して依存関係をインストールする必要があります。

プロジェクトのルートディレクトリにある`package.json`ファイルで、すべての依存関係を確認できます。

`package.json`ファイルは、プロジェクトの名前、バージョン、依存関係などのメタデータを含む、プロジェクトの重要な設定ファイルです。スクリプト、ライセンス情報、その他の設定も定義されており、プロジェクトとその依存関係の管理に役立ちます。

依存関係をインストールすると、`node_modules`フォルダーがプロジェクトに追加されます。

`node_modules`フォルダーは、プロジェクトに必要なすべてのパッケージやライブラリが保存される場所です。このフォルダーには、`package.json`ファイルにリストされた依存関係の実際のコードが含まれており、プロジェクトの直接の依存関係と、それらの依存関係の依存関係の両方が含まれます。

Reactプロジェクトに必要な2つの主要な依存関係は、`react`と`react-dom`のパッケージです。

```json
"dependencies": {
  "react": "^18.3.1",
  "react-dom": "^18.3.1"
}
```

`package.json`には、これらのパッケージの現在使用しているバージョンが記載されています。ローカルでパッケージを更新する必要がある場合は、`npm update`コマンドを実行できます。すべてのパッケージをグローバルに更新するには、`npm update -g`コマンドを実行します。

`package.json`ファイルに加えて、`package-lock.json`ファイルもあります。このファイルは、プロジェクトが使用しているすべてのパッケージの正確なバージョンを固定します。パッケージを更新すると、新しいバージョンもロックファイルに反映されます。

`package.json`ファイルのもう一つの重要な側面は、開発用依存関係です。

```json
"devDependencies": {
  "@eslint/js": "^9.17.0",
  "@types/react": "^18.3.18",
  "@types/react-dom": "^18.3.5",
  "@vitejs/plugin-react": "^4.3.4",
  "eslint": "^9.17.0",
  "eslint-plugin-react": "^7.37.2",
  "eslint-plugin-react-hooks": "^5.0.0",
  "eslint-plugin-react-refresh": "^0.4.16",
  "globals": "^15.14.0",
  "vite": "^6.0.5"
}
```

開発用依存関係は、開発時にのみ使用され、本番環境では使われないパッケージです。例えば、Jestのようなテスト用ライブラリがこれにあたります。Jestはローカルでプロジェクトをテストするためだけにインストールし、本番環境でアプリケーションを動かす際には必要ありません。

このレッスンの大部分ではnpmに焦点を当ててきましたが、yarnやpnpmのような他のパッケージマネージャーもあります。では、どのパッケージマネージャーをプロジェクトで使うべきでしょうか？

短い答えは、状況によります。

既存のプロジェクトにチームで参加する場合は、その決定はすでにされており、既存のパッケージマネージャーを使います。新しいプロジェクトを一から作る場合は、それぞれのマネージャーの長所と短所を調べて、自分のニーズに合ったものを選ぶ必要があります。
