# --description--

npm потрібен спосіб зберігати важливі деталі про ваш проєкт — такі як пакунки, які він використовує, версії цих пакунків і які команди він може виконувати. Для цього він використовує файл `package.json`.

Думайте про файл `package.json` як про конфігураційний файл npm для проєкту. Це простий JSON-файл, який описує ваш проєкт за допомогою пар ключ-значення, щоб відстежувати такі речі, як:

* назва та версія проєкту
    
* автор
    
* головний файл, який слід запускати першим
    
* ліцензія
    
* репозиторій коду, ключові слова та описи
    
* будь-які залежності, необхідні проєкту
    

Ось як може виглядати файл `package.json`:

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "description": "A simple NPM project",
  "main": "index.js",
  "scripts": {
    "start": "node index.js",
    "test": "echo \"Running tests...\""
  },
  "repository": {
    "type": "git",
    "url": "git+https://github.com/username/project.git"
  },
  "keywords": ["fCC", "js"],
  "author": "Kolade Chris",
  "license": "ISC",
  "dependencies": {
    "react": "^19.2.1"
  },
  "devDependencies": {
    "cypress": "^15.7.1"
  }
}
```

Поле `main` часто встановлюється в `index.js` і вказує Node.js, який файл запускати першим.

Поле `license` використовується для визначення юридичних умов, за яких інші люди можуть використовувати, змінювати та поширювати код у вашому пакунку. Існує багато різних ліцензій, таких як `MIT`, `ISC` або `GPL`, які мають різні умови та дозволи. За замовчуванням використовується `ISC`, ліцензія Internet Systems Consortium, яка є дуже ліберальною ліцензією відкритого коду.

Коли ваш проєкт зростає і ви встановлюєте пакунки, npm додає їх у один із двох розділів:

* `dependencies`, які потрібні вашому застосунку для роботи в продакшені
    
* `devDependencies`, які потрібні лише під час розробки
    

Наприклад, якщо ваш проєкт використовує React для інтерфейсу, це потрапляє в `dependencies`, бо він потрібен для роботи проєкту після розгортання. Інструмент на кшталт Cypress, який використовується лише для end-to-end тестування перед розгортанням у продакшен, належить до `devDependencies`.

Існує кілька простих способів створити файл `package.json`. Найпоширеніший спосіб — виконати таку команду:

```bash
npm init
```

Виконуйте цю команду у кореневій папці вашого проєкту. Ця команда запускає інтерактивний процес налаштування, де npm ставить вам низку питань. Ваші відповіді використовуються для заповнення полів, таких як назва проєкту, версія, опис, скрипти, ключові слова, автор і ліцензія.

Якщо ви хочете пропустити питання і створити файл `package.json` з значеннями за замовчуванням, ви можете виконати:

```bash
npm init -y
```

або

```bash
npm init --yes
```

Це миттєво створить для вас файл `package.json` з типовими значеннями у багатьох полях.

Зазвичай краще використовувати `npm init` без прапорців `-y` або `--yes`, щоб пройтися по кожному полю і налаштувати інформацію про ваш проєкт. Якщо ви не відповідаєте на питання, npm просто використовує значення за замовчуванням. Наприклад, якщо залишити поле ліцензії порожнім, npm встановить його за замовчуванням у `ISC`.

Ви можете помітити, що `dependencies` і `devDependencies` ще не вказані. Це тому, що ці розділи додаються лише після встановлення першого пакунка.

Ви також помітите, що поле `repository` відсутнє. Хоча npm питав про нього під час `npm init`, він не заповнює його автоматично пізніше. Наприклад, коли ви виконуєте команди `git remote add origin` або `git push`, npm не оновить поле репозиторію за вас — його потрібно додати вручну, якщо ви хочете, щоб воно було включене.
