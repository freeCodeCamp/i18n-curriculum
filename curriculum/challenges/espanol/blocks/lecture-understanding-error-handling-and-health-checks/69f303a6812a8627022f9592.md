# --description--

Cada vez que visitas un sitio web o usas una API, tu navegador y el servidor están constantemente comunicándose.

¿Una de las partes más importantes de esa conversación? Los códigos de estado de respuesta HTTP.

Son mensajes numéricos cortos — como 200, 404 o 500 — que le indican al cliente (el dispositivo del usuario final, como una laptop o un teléfono) qué pasó después de hacer una solicitud.

Ya has aprendido algunos de estos, pero vamos a desglosarlos por categoría para que recuerdes exactamente qué significan y cuándo usarlos.

## 1xx – respuestas informativas

Los códigos que empiezan con 1 rara vez se ven en el desarrollo diario, pero indican que la solicitud fue recibida y el proceso continúa, como cuando se sube un archivo.

Ejemplo común:

* 100 Continue – El servidor le dice al cliente: "Está bien, puedes enviar el resto de la solicitud."
* 101 Switching Protocols – El cliente envió un encabezado `Upgrade` por lo que el servidor está cambiando de protocolo. Esto se ve más comúnmente al usar Websockets.

## 2xx – respuestas de éxito

Estos significan que la solicitud fue recibida, entendida y procesada con éxito.

* 200 OK – La solicitud tuvo éxito. Esta es la respuesta de éxito más común.
* 201 Created – Se creó un nuevo recurso con éxito, por ejemplo, después de registrar un nuevo usuario.
* 204 No Content – La solicitud tuvo éxito, pero no hay cuerpo de respuesta para enviar. Se usa a menudo para solicitudes `DELETE`.

## 3xx – mensajes de redirección

Estos le dicen al cliente: "El recurso que buscas está en otro lugar."

* 301 Moved Permanently – El recurso tiene una nueva URL permanente.
* 302 Found – El recurso está temporalmente en una URL diferente.
* 304 Not Modified – Se usa para caché. Le dice al cliente que nada ha cambiado, así que puede usar la versión almacenada.

## 4xx – errores del cliente

Estos ocurren cuando el cliente envía una solicitud inválida.

* 400 Bad Request – La solicitud está mal formada o es inválida.
* 401 Unauthorized – Se requiere autenticación (necesitas iniciar sesión).
* 403 Forbidden – La autenticación fue exitosa, pero no tienes permiso.
* 404 Not Found – El recurso solicitado no existe. Y sí, este es el error más icónico en la web.

## 5xx – errores del servidor

Estos significan que algo salió mal en el servidor.

* 500 Internal Server Error – Un mensaje genérico de "algo salió mal".
* 502 Bad Gateway – Un servidor que actúa como proxy recibió una respuesta inválida de otro servidor.
* 503 Service Unavailable – El servidor está temporalmente sobrecargado o en mantenimiento.

Ahora que conoces algunos de los códigos de estado HTTP más comunes, veamos algunos ejemplos prácticos en Express.

Supongamos que estás construyendo una API. Si un usuario envía datos inválidos, puedes responder con un código de estado 400, que significa "Bad Request". También puedes enviar la cadena "Bad Request" de vuelta al cliente para que quede claro qué significa el código 400:

```js
res.status(400).send("Bad Request");
```

Si todo va bien, puedes enviar una respuesta 200. Puedes incluir un mensaje si quieres. Aquí enviamos un JSON con un mensaje simple de "Success":

```js
res.status(200).json({ message: "Success" });
```

Y si algo falla en el servidor, puedes enviar un código 500 que significa que ocurrió un error interno del servidor, junto con un mensaje claro sobre el error:

```js
res.status(500).send("Internal Server Error");
```

Estos códigos de estado no son solo para máquinas. Ayudan a los desarrolladores a depurar problemas y ayudan a los usuarios (y otros desarrolladores) a entender qué está pasando.
