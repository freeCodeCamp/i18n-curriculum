# --description--

GET のほかに、もう一つよく使われる HTTP 動詞があります。それが POST です。POST は HTML フォームでクライアントのデータを送信する際のデフォルトのメソッドです。REST の慣例では、POST はデータベースに新しいアイテム（新しいユーザーや新しいブログ投稿など）を作成するためのデータを送るのに使われます。このプロジェクトにはデータベースはありませんが、それでも POST リクエストの扱い方を学びます。

この種のリクエストでは、データは URL に現れず、リクエストボディに隠されています。ボディは HTTP リクエストの一部で、ペイロードとも呼ばれます。データが URL に見えなくても、それがプライベートであるとは限りません。なぜかを理解するために、HTTP POST リクエストの生の内容を見てみましょう。

```http
POST /path/subpath HTTP/1.0
From: john@example.com
User-Agent: someBrowser/1.0
Content-Type: application/x-www-form-urlencoded
Content-Length: 20

name=John+Doe&age=25
```

ご覧の通り、ボディはクエリ文字列のようにエンコードされています。これは HTML フォームで使われるデフォルトの形式です。Ajax を使う場合は、より複雑な構造のデータを扱うために JSON を使うこともできます。ほかに multipart/form-data というエンコード形式もあり、これはバイナリファイルのアップロードに使われます。この演習では URL エンコードされたボディを使います。POST リクエストから送られてくるデータを解析するには、`body-parser` パッケージを使う必要があります。このパッケージを使うと、さまざまな形式のデータをデコードできる一連のミドルウェアを利用できます。

# --instructions--

`body-parser` はすでにインストールされており、プロジェクトの `package.json` ファイルにあります。`require` を `myApp.js` ファイルの先頭で行い、`bodyParser` という名前の変数に格納してください。URL エンコードされたデータを処理するミドルウェアは `bodyParser.urlencoded({extended: false})` から返されます。前のメソッド呼び出しで返された関数を `app.use()` に渡してください。いつものように、このミドルウェアはそれに依存するすべてのルートの前にマウントする必要があります。

**注意:** `extended` は `body-parser` にどの解析方法を使うかを指示する設定オプションです。`extended=false` の場合は、従来の `querystring` ライブラリのエンコードを使います。`extended=true` の場合は、解析に `qs` ライブラリを使います。

`extended=false` を使う場合、値は文字列か配列のみです。`querystring` を使った場合に返されるオブジェクトは、デフォルトの JavaScript `Object` からプロトタイプ継承されていないため、`hasOwnProperty` や `toString` といった関数は利用できません。拡張版はより柔軟なデータを扱えますが、JSON には及びません。
