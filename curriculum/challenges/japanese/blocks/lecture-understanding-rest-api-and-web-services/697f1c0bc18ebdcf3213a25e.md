# --description--

**REST**は表現状態転送の略です。ウェブのクライアントとサーバーがインターネット上でどのように通信すべきかを説明する一連のガイドラインと考えてください。

このレッスンでは、RESTとは何か、その主要な概念、そしてどのように機能するかを学びます。

RESTは、異なるシステムが標準的なウェブ技術を使って相互にやり取りする方法を定義しています。RESTでは、ウェブブラウザ、モバイルアプリ、APIなどのクライアントが、以前のレッスンで説明した`GET`、`POST`、`PUT`、`PATCH`、`DELETE`といった一般的なHTTPメソッドを使ってサーバーと通信します。

RESTのコア概念には、**ステートレス**、**リソース**、**HTTPメソッド**、**レスポンスやステータスコード**があります。

**ステートレス**とは、クライアントがサーバーに送る各リクエストが独立して処理されることを意味します。サーバーは以前のリクエストを記憶しません。そのため、すべてのリクエストには処理に必要な情報がすべて含まれている必要があります。例えば、ユーザーが認証されている場合は、認証トークンを毎回のリクエストに含めなければなりません。

RESTでは、すべてがリソースとして扱われます。リソースとはアクセスまたは変更できるデータの一部です。各リソースはURLで識別されます。例えば、すべての製品を表す`/products`や特定の製品を表す`/products/1`などです。

以下にHTTPメソッド、ステータスコードの概要と、RESTアーキテクチャがどのように機能するかのステップバイステップの例を示します。

## HTTPメソッド

HTTPメソッドはリソースに対して行う操作を定義します：

* `GET`: リソースを取得する
    
* `POST`: 新しいリソースを作成する
    
* `PUT / PATCH`: 既存のリソースを更新する
    
* `DELETE`: リソースを削除する
    

## HTTPステータスコード

リクエスト処理後、サーバーは結果を示す**ステータスコード**でレスポンスします：

* `200` **(OK)**: リクエストは成功しました
    
* `201` **(Created)**: 新しいリソースが作成されました
    
* `400` **(Bad Request)**: クライアントが無効なデータを送信しました
    
* `404` **(Not Found)**: リクエストされたリソースは存在しません
    
* `500` **(Internal Server Error)**: サーバーでエラーが発生しました
    

## RESTの動作：ステップバイステップの例

### 1\. クライアントがリクエストを送信

* クライアントは`GET`や`POST`のようなメソッドを使ってHTTPリクエストを送信します
    
* リクエストにはリソースを識別するURLが含まれます
    
* ヘッダー、メタデータ、リクエストボディが含まれる場合もあります
    

### 2\. サーバーがリクエストを受信

* サーバーはリクエストの形式や認証（必要な場合）を検証します
    
* URLから対象のリソースを特定します
    
* HTTPメソッドに基づいて要求された操作を判断します
    
* 権限やアクセス制御を確認します
    

### 3\. サーバーが操作を実行

* `GET`: リソースを取得
    
* `POST`: 新しいリソースを作成
    
* `PUT / PATCH`: リソースを更新
    
* `DELETE`: リソースを削除
    
* その他必要なロジックを実行
    

### 4\. サーバーがレスポンスを作成

* サーバーは適切なステータスコード（例：`200`、`201`、`404`）を選択します
    
* 通常、**JSON**形式でレスポンスデータを整形します
    
* `Content-Type`や`Cache-Control`などの関連ヘッダーを追加します
    

### 5\. サーバーがレスポンスを送信

* レスポンスにはステータスコード、ヘッダー、任意のボディが含まれます
    
* サーバー側にクライアントの状態は保存されません（ステートレス）
    

### 6\. クライアントがレスポンスを処理

* クライアントはステータスコードを確認して成功か失敗かを判断します
    
* サーバーから返されたデータを処理します
