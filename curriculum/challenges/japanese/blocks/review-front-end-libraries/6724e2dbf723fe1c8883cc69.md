## JavaScriptライブラリとフレームワーク

- JavaScriptのライブラリとフレームワークは、よくある問題に対する迅速な解決策を提供し、組み込みのコードを使うことで開発を加速します。
- ライブラリは一般的に、DOM操作、イベント処理、AJAXリクエストの管理など、特定のタスクに対する解決策を提供することに重点を置いています。
- JavaScriptライブラリの例としては、jQueryやReactがあります。
- 一方、フレームワークはアプリケーション構築のためのより明確な構造を提供します。開発者が従うべきルールや慣例のセットが付属していることが多いです。
- フレームワークの例にはAngularやReactのメタフレームワークであるNext.jsがあります。
- **シングルページアプリケーション**（SPA）は、単一のHTMLページを読み込み、ユーザーがアプリケーションとやり取りする際にページ全体をリロードせずにそのページを動的に更新するウェブアプリケーションです。
- SPAはJavaScriptを使ってアプリケーションの状態を管理し、コンテンツをレンダリングします。これは複雑なユーザーインターフェースを構築するための優れたツールを提供するフレームワークを使って行われることが多いです。
- SPAに関する問題点には以下があります：

  - スクリーンリーダーは動的に更新されるコンテンツに対応しづらいです。
  - ユーザーがアプリ内を移動してもURLが変わらないため、特定のページをブックマークしたり戻ったり共有したりするのが難しいです。
  - アプリケーションが大きい場合、すべてのアセットを最初に読み込む必要があるため初期読み込み時間が遅くなることがあります。

## React

- Reactはユーザーインターフェースやウェブアプリケーションを構築するための人気のJavaScriptライブラリです。
- Reactのコアコンセプトは、データが変わると独立して更新・レンダリングできる再利用可能なUIコンポーネントの作成です。
- Reactはアプリケーションの状態に基づいてUIがどのように見えるべきかを記述できるようにします。データや状態が変わるとReactが適切なコンポーネントを更新・レンダリングします。

## Reactコンポーネント

- コンポーネントはReactアプリケーションの構成要素で、複雑なユーザーインターフェースを小さく管理しやすい部分に分割できます。
- UIはJSXというJavaScript構文の拡張を使って記述され、JavaScript内にHTMLのようなコードを書くことができます。
- コンポーネントは基本的にUIの一部を返すJS関数またはクラスです。

以下は挨拶メッセージをレンダリングするシンプルなReactコンポーネントの例です：

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

コンポーネントを使うには、単に次のように呼び出します：

```jsx
  <Greeting />
```

## Reactコンポーネントのインポートとエクスポート

- Reactコンポーネントは、あるファイルからエクスポートして別のファイルにインポートすることができます。  
- 例えば、`City.js`というファイルに`City`というコンポーネントがあるとします。`export`キーワードを使ってコンポーネントをエクスポートできます：

- Reactコンポーネントはあるファイルからエクスポートして別のファイルにインポートできます。
- 例えば、`City`という名前のコンポーネントが`City.js`というファイルにあるとします。`export`キーワードを使ってコンポーネントをエクスポートできます：

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default `City`;
```

- 別のファイルに`City`コンポーネントをインポートするには、`import`キーワードを使います：

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

- `default`キーワードは`City.js`ファイルのデフォルトエクスポートとして使われます。

- コンポーネント定義と同じ行でエクスポートすることもできます：

```jsx
export default function City() {
  return <p>New York</p>;
}
```

## Reactコンポーネントへのpropsの渡し方

- Reactではprops（プロパティの略）は親コンポーネントから子コンポーネントへデータを渡す方法です。この仕組みは再利用可能で動的なUI要素を作るために必要です。
- Propsは任意のJavaScriptの値を渡せます。親から子にpropsを渡すには、親のJSX内で子コンポーネントを使う際に属性としてpropsを追加します。簡単な例はこちらです：

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

複数のpropsをスプレッド演算子`(...)`を使ってオブジェクトに変換して渡すこともできます。例はこちら：

```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

このコードではスプレッド演算子`{...person}`がpersonオブジェクトを個別のpropsに変換し、Childコンポーネントに渡しています。

## Reactでの条件付きレンダリング

- Reactの条件付きレンダリングは、アプリケーション内の特定の条件や状態に基づいて異なるコンテンツを表示するために使います。
- 条件付きレンダリングにはいくつかの方法があります。よく使われるのは三項演算子です。例はこちら：
- コンテンツを条件付きでレンダリングするもう一つの方法は論理積の`&&`演算子を使うことです。これは特定の条件が満たされた場合にのみコンテンツをレンダリングしたいときに便利です。例を示します：

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- もう一つの方法は論理AND（`&&`）演算子を使うことです。これは特定の条件が満たされた場合のみコンテンツをレンダリングしたいときに便利です。例はこちら：

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

上記のコードでは、userオブジェクトが真の場合のみ`h1`要素がレンダリングされます。

直接`if`文を使うこともできます：

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```

## Reactでのリストのレンダリング

- Reactでリストをレンダリングするのはユーザーインターフェース構築でよくある作業です。
- JSの配列`map()`メソッドを使って配列のアイテムを繰り返し処理し、新しいJSX要素の配列を返すことでリストをレンダリングできます。
- 例えば、名前の配列をリストとしてレンダリングしたい場合は次のようにします：

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- 各リストアイテムに一意のキーを必ず指定して、Reactが更新やレンダリングを適切に管理できるようにしてください。これらのテクニックで柔軟で効率的な動的リストをReactアプリで作れます。

## Reactでのインラインスタイル

- ReactのインラインスタイルはJavaScriptオブジェクトを使ってJSX要素に直接CSSスタイルを適用できます。
- JSX要素のstyle属性にオブジェクトを渡します。オブジェクトのキーはキャメルケースのCSSプロパティ名、値は対応する値です。例はこちら：

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

スタイルを別のオブジェクトに抽出して、`style`属性で参照することもできます：

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- インラインスタイルはpropsやstateに基づいて条件付きでスタイルを適用する動的スタイリングもサポートします。以下はpropsに基づいて条件付きでスタイルを適用する例です：

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- 上記のコードでは`color`と`fontSize`のスタイルが`isImportant`プロップに基づいて条件付きで設定されています。

## Stateと`useState`フックの操作

- **stateの定義**：Reactではstateはコンポーネントのデータを含むオブジェクトです。stateが更新されるとコンポーネントは再レンダリングされます。Reactはstateを不変として扱い、直接変更しないようにします。
- **`useState()`フック**：`useState`フックは関数コンポーネントでstate変数を宣言できる関数です。基本的な構文は以下の通りです：

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

state変数には以下が含まれます：`Counter`コンポーネントの完全な例はこちらです：

- `stateVariable`は現在のstate値を保持します

```jsx
function Counter() {

      <h2>{count}</h2>

      <button onClick={() => setCount(count + 1)}>Increment</button>

}
```

## レンダリングとReactコンポーネント

- **定義**：Reactでのレンダリングはコンポーネントがユーザーインターフェース（UI）、通常はブラウザに表示されるプロセスです。レンダリングはトリガー、レンダー、コミットの3段階で構成されます。

トリガー段階はReactが何かが変わったことを検知し、UIの更新が必要かもしれないと判断したときに発生します。この変化は通常stateやpropsの更新によるものです。

トリガーが起きるとReactはレンダーステージに入り、コンポーネントを再評価して何を表示するか決定します。これには「仮想DOM」と呼ばれる軽量コピーを使います。仮想DOMを使うことでReactはどこを変更すべきか素早くチェックできます。

- **Updating Objects in State**: 状態の中のオブジェクトを更新する必要がある場合は、まず新しいオブジェクトを作成するか既存のオブジェクトをコピーし、その新しいオブジェクトの状態を設定するべきです。状態に入れられたオブジェクトはすべて読み取り専用と見なされるべきです。ここではユーザーの名前、年齢、都市を設定する例を示します。ユーザー情報の更新を処理するために`handleChange`関数が使われます：

## State内のオブジェクトと配列の更新

- **State内のオブジェクトの更新**：state内のオブジェクトを更新する場合は、新しいオブジェクトを作成するか既存のオブジェクトをコピーしてから新しいオブジェクトでstateを設定してください。stateに入れるオブジェクトは読み取り専用と考えます。以下はユーザーの名前、年齢、都市を設定する例です。`handleChange`関数はユーザー情報の更新を処理します：

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **State内の配列の更新**：state内の配列を更新する際は、`push()`や`pop()`のようなメソッドで配列を直接変更しないようにしてください。代わりにstate更新時には新しい配列を作成します：

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

配列からアイテムを削除したい場合は、削除したいものを除外した新しい配列を返す`filter()`メソッドを使います：

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```

- **`useEffect()` フック**: Reactでは、エフェクトとはコンポーネントのレンダリングプロセスの外で起こるすべてのことを指します。つまり、ReactがUIのレンダリングの一部として直接処理しないものすべてです。一般的な例としては、データの取得、ブラウザタブのタイトルの更新、ブラウザのローカルストレージの読み書き、ユーザーの位置情報の取得などがあります。これらの操作は外部の世界とやり取りし、副作用として知られています。Reactはこれらの副作用を処理するために`useEffect`フックを提供しています。`useEffect`はコンポーネントがレンダリングまたは更新された後に関数を実行できるようにします。

## `useEffect`フックの操作

- **`useEffect()`フック**：Reactではエフェクトとはコンポーネントのレンダリングプロセス外で起こるすべてのことを指します。例えばデータ取得、ブラウザタブのタイトル更新、ローカルストレージの読み書き、ユーザーの位置情報取得などです。これらは副作用と呼ばれます。Reactはこれらの副作用を扱うために`useEffect`フックを提供しています。`useEffect`はコンポーネントのレンダリングや更新後に関数を実行できます。

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [`dependencies`]);
```

エフェクト関数はコンポーネントのレンダリング後に実行され、オプションの`dependencies`引数でエフェクトの実行タイミングを制御します。

`dependencies`は「リアクティブな値」（state、props、関数、変数など）の配列、空配列、または省略可能です。これらのオプションが`useEffect`の動作を制御します：

- `dependencies`が1つ以上のリアクティブな値を含む配列の場合、それらが変わるたびにエフェクトが実行されます。
- `dependencies`が空配列の場合、`useEffect`はコンポーネントの初回レンダリング時に一度だけ実行されます。
- `dependencies`を省略すると、コンポーネントのレンダリングや更新のたびにエフェクトが実行されます。

## カスタムフックの作り方

- **カスタムフック**：カスタムフックはデータ取得、状態管理、トグル、オンライン状態の追跡などの再利用可能なロジックをコンポーネントから抽出するためのものです。Reactの組み込みフックはすべて`use`で始まるので、カスタムフックも同じ慣例に従うべきです。

以下は`useDebounce`フックを作成する例です：

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```

```jsx
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **非制御入力**：`useState`フックで入力を扱う代わりに、HTMLの非制御入力はDOMの助けを借りて独自の内部状態を維持します。DOMが入力値を管理するため、`ref`を使って入力フィールドの値を取得する必要があります。

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```

- **サーバーアクション**：これはAPIエンドポイントを使わずにサーバー上でフォーム処理を可能にする関数です。Next.jsアプリケーションの例はこちら：

- **`useActionState`フック**：このフックはフォーム提出の結果に基づいてstateを更新します。`useActionState`フックの基本構文はこちら：

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state`はアクションが返す現在のstateです。
- `action`はサーバーアクションをトリガーする関数です。
- `isPending`はアクションが実行中かどうかを示すブール値です。
- `actionFunction`パラメータはサーバーアクション自体です。
- `initialState`はアクション実行前のstateの開始点を表すパラメータです。
- `permalink`はフォームが変更するユニークなページURLを含むオプションの文字列です。

## Reactでのデータ取得

- **データ取得の選択肢**：Reactでデータを取得する方法は多様です。ネイティブのFetch APIやAxios、SWRなどのサードパーティツールを使えます。
- **データ取得時によく使うstate変数**：どの方法を使うにしても、データ自体、データ取得中かどうかの状態、取得中に発生したエラーを保持するstate変数が必要です。

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

データ取得は副作用なので、`Fetch API`を`useEffect`フック内で使うのが最適です。

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

データ取得が完了していなければローディングメッセージを、エラーがあればエラーメッセージを、そうでなければ結果をレンダリングできます。

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

## `useOptimistic`フックの操作

- **`useOptimistic`フック**：このフックは非同期アクションの完了を待つ間にUIのレスポンスを保つために使います。UIに即時更新を提供する「楽観的更新」を管理するのに役立ちます。

基本構文はこちら：

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState`はより良いユーザー体験のために即時に更新される一時的なstateです。
- `addOptimistic`は実際のstateが変わる前に楽観的更新を適用する関数です。
- `actualState`はサーバーからのデータ取得結果などの実際のstate値です。
- `updateFunction`は呼び出されたときに楽観的stateをどう更新するか決める関数です。

以下は`useOptimistic`フックを`TaskList`コンポーネントで使う例です：

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**：UIの一部をレンダリングし、状態更新を非緊急の遷移としてマークするために使います。これにより高負荷の更新中もUIがレスポンシブになります。基本構文はこちら：

```js
startTransition(action)
```

`action`は状態更新や遷移関連のロジックをトリガーします。これにより入力やクリックなどの緊急のUI更新がブロックされません。

- **`useMemo`フック**：計算済みの値をメモ化するために使います。大きな配列のソート結果をメモ化する例です。`expensiveSortFunction`は`largeArray`が変わったときだけ実行されます：

```js

  () => expensiveSortFunction(largeArray),
  [largeArray]
);
```

## React Router

- **概要**：React RouterはReactアプリにルーティングを追加できるサードパーティライブラリです。既存のReactプロジェクトにReact Routerをインストールするには次のようにします：

```bash
npm i react-router
```

次に`main.jsx`または`index.jsx`ファイル内でルート構造を設定します：

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

`path`と`element`はURLとUIコンポーネントを結びつけます。この例ではホームページのルートを`App`コンポーネントに設定しています。

- **複数ビューとルート設定**：大規模アプリでは複数のビューとルートを次のように設定することが一般的です：

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

これらの例の`index`プロップは特定のパスセグメントのデフォルトルートを表します。つまり`Home`コンポーネントはルート`/`パスで表示され、`ProductsHome`コンポーネントは`/products`パスで表示されます。

- **ルートのネスト**：ルートを他のルート内にネストすると、子ルートのパスが親ルートのパスに追加されます。

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

上記の例では、トレンド商品へのパスはproducts/trendingになります。

- **動的セグメント**：URLパスの任意の部分が動的になることを指します。

```jsx
<Route path=":category" element={<Category />} />
```

この例では`category`という動的セグメントがあります。ユーザーがproducts/brass-instrumentsのようなURLに移動するとビューは`Category`コンポーネントに切り替わり、セグメントに基づいて適切なデータを動的に取得できます。

- **`useParams`フック**：URLパスの動的パラメータにアクセスするために使います。

```js
import { useParams } from "react-router";

export default function Category() {
  let params = useParams();
   {/* Accessing the category param: params.category */}
   {/* rest of code goes here */}
}
```

## 状態管理

- **Context API**：Contextは親コンポーネントがpropsを明示的に渡さずに子コンポーネントに情報を提供する仕組みです。`createContext`は他のコンポーネントが読むコンテキストを表すコンテキストオブジェクトを作成し、`Provider`は子コンポーネントにコンテキスト値を供給します。

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**：Reduxは中央のストアと厳格な状態更新管理を提供して状態管理を行います。アクション、リデューサー、ミドルウェアという予測可能なパターンを使います。アクションはユーザー操作などでアプリからReduxストアにデータを送る情報のペイロードです。リデューサーはアクションに応じて状態を不変に更新する関数です。ミドルウェアはアクションのディスパッチとリデューサーの間の橋渡しをし、ログ記録や非同期処理などReduxの機能拡張を可能にします。

- **Zustand**：小規模から中規模アプリに適した状態管理ソリューションです。`useStore`フックを使ってコンポーネントやページで直接状態にアクセスし、アクションやリデューサー、プロバイダーなしでデータを変更・取得できます。

## 実際のパフォーマンスと体感パフォーマンスの違い

- **体感パフォーマンス**：ユーザーがウェブサイトのレスポンスや信頼性をどのように感じるかです。主観的な評価で定量化は難しいですが、ユーザー体験の成功や失敗を決める重要な指標です。
- **実際のパフォーマンス**：ページ読み込み時間、サーバー応答時間、レンダリング時間などの客観的かつ測定可能なパフォーマンスです。ネットワークやコード自体に関連する複数の要因に影響されます。

## 体感パフォーマンスを改善する手法

- **遅延読み込み**：非必須リソースをバックグラウンドで読み込むことで初期読み込み時間をできるだけ短縮します。
- **フォント遅延の最小化**：カスタムフォントがある場合、フォント読み込み遅延を最小化することをお勧めします。遅延があるとフォントのちらつきやフォールバックフォントの表示が起こるためです。似たフォールバックフォントを使うと変化が目立ちにくくなります。
- **読み込みインジケーターの使用**：ユーザーが要素をクリックしたらすぐに長時間処理の読み込みインジケーターを表示すると、ユーザーが処理に関与している感覚を持ち、待ち時間が短く感じられます。

## パフォーマンスの基本概念

- **ソースオーダー**：HTML要素がドキュメント内でどのように構造化されているかを指します。これにより何が先に読み込まれるかが決まり、パフォーマンスやアクセシビリティに大きな影響を与えます。

ソースオーダーのベストプラクティスには以下があります：

- 見出し、ナビゲーション、メインテキストなど重要なコンテンツをHTML構造の上位に配置する。
- アナリティクスやサードパーティウィジェットなど非必須スクリプトはレンダリングをブロックしないよう遅延させる。
- プログレッシブエンハンスメントを使い、スタイルやスクリプトが読み込まれる前でも基本的な体験ができるようにする。プログレッシブエンハンスメントはHTMLを最初に動作させる考え方に基づくウェブサイトやアプリの構築方法です。

以下は先ほどのベストプラクティスを使った良いソースオーダーの例です：

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **クリティカルレンダリングパス**：ブラウザがコードを画面上のピクセルに変換するためにたどる一連のステップです。
- **レイテンシー**：ブラウザとサーバー間のリクエストの往復にかかる時間です。つまりレイテンシーが高いとページが遅くなります。

レイテンシーを減らす方法には以下があります：

- CDN（コンテンツデリバリーネットワーク）を使い、ファイルを近い場所から配信する。
- Gzipなどの圧縮を有効にしてファイルサイズを減らす。
- 画像を最適化し、遅延読み込みを使う。

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## INPの改善

- **定義**：INP（Interaction to Next Paint）はユーザーがクリックやキー入力などの操作をしてからブラウザが次に画面を更新するまでの時間を測り、ページ全体のレスポンスの良さを評価します。INPが低いほどレスポンスが良いページです。

INPを改善する方法は以下の通りです：

- 長いJavaScriptタスクを分割してメインスレッドの負荷を減らす。
- 非クリティカルなスクリプトには`requestIdleCallback()`を使う。これはブラウザのアイドル時間に関数をキューイングします。
- 先述の重いアセットの遅延読み込みや遅延実行を行う。
- イベントハンドラを最適化する。頻繁に実行されたり重い処理を行うとページが遅くなりINPが増加します。解決策はデバウンスで、ユーザーが入力を止めてから一定時間（例：300ms）経ってから関数を実行し、不要な計算を防ぎパフォーマンスを向上させます。

## ブラウザでのレンダリングの仕組み

- **レンダリングの仕組み**：まずブラウザはHTMLをパースしてDOMを構築します。次にCSSを処理してCSS Object Model（CSSOM）を作ります。これは要素のスタイルを決めるツリー構造です。最後に計算されたスタイルとレイアウトに基づいてピクセルを画面に描画します。複雑なページでは複数のレイヤーが合成されて最終的なビジュアルが形成されます。

## パフォーマンスが持続可能性に与える影響

- **背景情報**：インターネットは世界の炭素排出量の約2％を占めており、これは航空業界と同じ規模です。データセンターからユーザーのデバイスまでのすべてのバイト転送には電力が必要です。大きなファイルや非効率なスクリプトはより多くの電力消費を意味します。高パフォーマンスなウェブサイトは単に速いだけでなく、不要な処理やエネルギー消費を減らします。

## ページ読み込み時間を短縮する方法

- **メディアアセットの最適化**：大きな画像や動画は読み込み遅延の主な原因です。これらを最適化することでサイトの速度を大幅に向上できます。画像圧縮、WebPなどのモダンフォーマットの使用、遅延読み込みが含まれます。
- **ブラウザキャッシュの活用**：キャッシュによりブラウザはウェブサイトの一部をローカルに保存し、再訪問時の読み込み時間を短縮します。
- **ファイルの縮小と圧縮**：ファイルサイズを減らすことでダウンロード時間を短縮できます。送信ファイルのサイズ削減やCSS・JavaScriptの縮小が含まれます。

## 「使えるまでの時間」の改善

- **定義**：「使えるまでの時間」はユーザーがページをリクエストしてから意味のある操作ができるまでの時間です。これを改善するにはアセットの遅延読み込みやレンダーブロックするリソースの最小化が有効です。

## パフォーマンス測定の主要指標

- **First Contentful Paint（FCP）**：最初のテキストや画像などのコンテンツが画面に表示される速さを測ります。良いFCPは1.8秒未満、悪いFCPは3秒以上とされます。Chrome DevToolsのパフォーマンスタブで確認できます。
- **Total Blocking Time**：メインスレッドが重いJavaScriptタスクでブロックされる時間を示します。TBTが高いとユーザーは操作が鈍く感じます。改善には長いタスクの分割や非必須スクリプトの遅延実行が有効です。
- **直帰率**：ユーザーが操作せずに離脱する割合です。直帰率が高い場合はページが遅い可能性があります。
- **ユニークユーザー数**：サイトを訪れた個別の訪問者数を追跡します。直帰率やユニークユーザー数はGoogle Analyticsで確認でき、エンゲージメント改善に役立ちます。

## 一般的なパフォーマンス測定ツール

- **Chrome DevTools**：Google Chromeに組み込まれたツールでリアルタイムにパフォーマンスを分析・デバッグできます。読み込み時間、CPU使用率、レンダリング遅延を表示します。特にFCPの測定に便利です。サイトが遅いと感じたらボトルネックを特定できます。
- **Lighthouse**：パフォーマンス、SEO、アクセシビリティを自動でチェックするツールです。
- **WebPageTest**：異なる場所やデバイスからのサイトの読み込みをテストできます。Speed Index、Total Blocking Timeなどの詳細なパフォーマンス指標を提供します。実際のユーザーが世界中でどのようにサイトを体験しているか知りたい場合に最適です。
- **PageSpeed Insights**：モバイルとデスクトップの両方でサイトを分析し、改善点を提案します。画像最適化、レンダーブロックスクリプトの削除、サーバー応答時間の短縮など具体的な推奨を教えてくれます。Googleがサイトのパフォーマンスをどう見ているかを素早く簡単にチェックできます。
- **Real User Monitoring（RUM）**：実際のユーザーの行動を追跡し、ユーザーがサイトをどのように体験しているかを示します。代表的なRUMツールにはGoogle Analytics（ページ読み込み時間や直帰率を追跡）、New RelicやDatadog（リアルタイムのパフォーマンス問題を監視）があります。実ユーザーデータが欲しい場合は必須です。

## パフォーマンスWeb APIの活用

- **定義**：パフォーマンスWeb APIは開発者がコード内でページの読み込み効率や応答性を直接追跡できるAPIです。ページ読み込み時間、レンダリングやインタラクションの遅延、JavaScript実行時間を測定できます。
- **`performance.now()`**：ミリ秒単位の高精度タイムスタンプを提供し、サイトの各部分の読み込み時間を測れます。

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**：DNSルックアップから`DOMContentLoaded`までのページ読み込みの各段階を詳細に分解してくれます。

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**：レイアウトシフト、長時間タスク、ユーザー操作などのパフォーマンスイベントを監視します。

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## CSSパフォーマンス改善の手法

- **CSSアニメーション**：寸法、位置、レイアウトなど特定のCSSプロパティのアニメーションは「リフロー」と呼ばれる処理を引き起こし、ブラウザが要素の位置や形状を再計算します。これには再描画が伴い計算コストが高いです。したがってCSSアニメーションの数はできるだけ減らすか、ユーザーがオンオフ切り替えできるようにすることをお勧めします。

## JavaScriptパフォーマンス改善の手法

- **コード分割**：JavaScriptコードをクリティカルな処理と非クリティカルな処理に分割します。クリティカルなものはできるだけ早くプリロードし、非クリティカルなものは遅延させてページの高速レンダリングを実現します。
- **DOM操作**：DOM操作はJavaScriptでページの内容を動的に変更する処理で計算コストが高いです。DOM操作の量を減らすことでパフォーマンスが向上します。

## CSSフレームワーク

- **CSSフレームワーク**：作業効率を上げ、ウェブサイト全体で統一されたビジュアルスタイルを作り、複数ブラウザ間でデザインの一貫性を保ち、CSSコードを整理しやすくします。
- **人気のCSSフレームワーク**：Tailwind CSS、Bootstrap、Materialize、Foundationなどがあります。
- **潜在的なデメリット**：
  - フレームワークのCSSがカスタムCSSと競合する可能性があります。
  - 同じフレームワークを使う他のサイトと似た見た目になることがあります。
  - 大規模なフレームワークはパフォーマンス問題を引き起こすことがあります。

## 2種類のCSSフレームワーク

- **ユーティリティファーストCSSフレームワーク**：マージン、パディング、背景色など特定の目的の小さなクラスを持ち、必要に応じてHTML要素に直接割り当てます。Tailwind CSSはユーティリティファーストCSSフレームワークに分類されます。

以下はTailwind CSSでボタンをスタイルする例です。

```html
<button class="bg-blue-500 text-white font-bold py-2 px-4 rounded-full hover:bg-blue-700">
  Button
</button>
```

- **コンポーネントベースCSSフレームワーク**：事前に定義されたスタイル付きのコンポーネントをウェブサイトに追加できます。CSSフレームワークの公式ドキュメントにコンポーネントがあり、コピー＆ペーストで使えます。BootstrapはコンポーネントベースCSSフレームワークに分類されます。

以下はBootstrapでリストグループを作る例です。小さなクラスをHTML要素に適用する代わりに、HTML構造を含むコンポーネント全体を追加します。

```html
<div class="card" style="width: 25rem;">
  <ul class="list-group list-group-flush">
    <li class="list-group-item">HTML</li>
    <li class="list-group-item">CSS</li>
    <li class="list-group-item">JavaScript</li>
  </ul>
</div>
```

TailwindはユーティリティファーストCSSフレームワークです。カスタムCSSルールを書く代わりに、小さなユーティリティクラスをHTMLに直接組み合わせてデザインを作ります。

Tailwindは`text-red-700`、`bg-indigo-600`、`bg-gray-100`のような幅広いカラーパレットを提供します。

## CSSプリプロセッサ

- **CSSプリプロセッサ**：標準CSSを拡張するツールで、拡張構文のコードをネイティブCSSファイルにコンパイルします。複雑なプロジェクトでよりクリーンで再利用可能、繰り返しを減らしたスケーラブルなCSSを書くのに役立ちます。
- **特徴**：変数、ミックスイン、ネスト、セレクター継承などの機能を提供します。

- **ミックスイン**：複数のCSSプロパティと値をまとめて名前を付け、スタイルシート全体で再利用できる機能です。

以下はSCSS構文でミックスインを定義する例です。ここでは`center-flex`という名前のミックスインで、flexboxを使って要素を中央に配置する3つのCSSプロパティがあります。

## 手動テストと自動テスト

- **手動テスト**：テスターがアプリケーションの各部分を手動で操作し、機能が正しく動作するか確認します。バグが見つかればソフトウェアチームに報告し修正します。
- **自動テスト**：別のプログラムを書いてアプリケーションが期待通りに動作するか自動でチェックします。

## 単体テスト

- **単体テスト**：各関数をテストして期待通りに動作するか確認します。単体テストはコードの期待される振る舞いを表すドキュメントとしても役立ちます。
- **単一責任の原則**：各関数は小さく、1つの責任に集中することが推奨されます。
- **一般的なJavaScriptテストフレームワーク**：Jest、Mocha、Vitestなどがあります。Jestは単体テストで人気のフレームワークです。

以下はJestを使った単体テストの例です。

まず、新しいフォーマット済み文字列を返す関数を作成します：

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

別の`getFormattedWord.test.js`ファイルで関数が期待通り動作するか検証するテストを書きます。`getFormattedWord.test.js`ファイルは次のようになります：

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **`expect`関数**：値をテストするために使います。
- **マッチャー**：値が期待通りかチェックする関数です。上記の例では`toBe()`がマッチャーです。Jestには様々なマッチャーがあります。

Jestを使うにはまず`jest`パッケージを`npm i jest`でインストールし、`package.json`ファイルに次のようなスクリプトを追加します：

```js
"scripts": {
  "test": "jest"
},
```

その後、`npm run test`コマンドでテストを実行できます。

## ソフトウェア開発ライフサイクル

- **ソフトウェア開発ライフサイクルの各段階**：
  - **計画段階**：開発チームがステークホルダーから要件を収集します。
  - **設計段階**：ソフトウェアチームが要件を分解し、最適な解決策を決定します。
  - **実装段階**：要件を管理可能なタスクに分解し実装します。
  - **テスト段階**：手動・自動テストを行い、開発中に発生した問題を検出・修正します。
  - **デプロイ段階**：新しい変更をビルドやテスト環境にデプロイします。
  - **メンテナンス段階**：本番環境で発生した問題を修正します。

- **ソフトウェア開発ライフサイクルのモデル**：
  - **ウォーターフォールモデル**：各段階を完了してから次の段階に進みます。
  - **アジャイルモデル**：作業をスプリントに分けて反復的に開発します。

## BDDとTDD

- **TDD**：テスト駆動開発は最初にテストを書くことを重視する手法です。機能を作る前にテストを書くことで開発中にリアルタイムのフィードバックが得られます。
- **BDD**：振る舞い駆動開発はビジネス目標に沿ったテストシナリオを作成します。BDDのテストシナリオは技術者と非技術者の両方が理解できる言語で書かれます。Gherkinが例です。
- **BDDテストフレームワーク**：Cucumber、JBehave、SpecFlowなどがあります。

## 単体テストにおけるアサーション

- **アサーション**：コードが期待通りに動作しているかテストするために使います。
- **アサーションライブラリ**：Chaiがよく使われます。他の一般的なJavaScriptアサーションライブラリには`should.js`や`expect.js`があります。

以下はChaiを使ったアサーションの例で、`addThreeAndFour`関数の戻り値が7と等しいかチェックします：

```js
assert.equal(addThreeAndFour(), 7);
```

- **ベストプラクティス**：どのアサーションライブラリを使う場合でも、どのテストがなぜ失敗したか分かるように明確なアサートメッセージと失敗メッセージを書くべきです。

## モック、フェイク、スタブ

- **モック**：実際のデータを偽のデータに置き換え、実際のコンポーネントの振る舞いをシミュレートします。例えばテスト中にAPIレスポンスをモックして連続的なAPI呼び出しを避けます。
- **スタブ**：期待される振る舞いに対して事前定義されたレスポンスやダミーデータを返すオブジェクトです。例えばテストで実際のデータベース接続に頼らずにデータベース接続の振る舞いをスタブできます。
- **フェイク**：複雑さや副作用のない実際のコンポーネントの簡略版です。例えば実際のデータベースとやり取りする代わりにメモリ内にデータを保存してデータベース操作を模倣できます。これにより実際のデータベースより高速に動作します。

## 機能テスト

- **機能テスト**：アプリケーションの機能や特徴が期待通りに動作するかをチェックします。複数のシナリオに対してシステム全体をテストすることが目的です。
- **非機能テスト**：パフォーマンスや信頼性などに焦点を当てます。
- **スモークテスト**：より広範なテストを始める前に基本的または重要な問題がないかを簡単にチェックします。

## エンドツーエンドテスト

- **エンドツーエンドテスト**：E2Eとも呼ばれ、ユーザー視点で実際のシナリオをテストします。アプリケーションが正しく予測可能に動作することを保証しますが、セットアップ、設計、メンテナンスに時間がかかります。
- **エンドツーエンドテストフレームワーク**：PlaywrightはMicrosoftが開発した人気のフレームワークです。他にCypress、Selenium、Puppeteerなどがあります。

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {

  const menu = page.getByTestId("header-menu");

  await expect(menu).toBeVisible();

});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## ユーザビリティテスト

- **ユーザビリティテスト**：実際のユーザーがアプリケーションとやり取りし、デザイン、ユーザー体験、機能に問題がないかを発見します。ユーザビリティテストはユーザーの直感的な使いやすさに焦点を当てます。
- **4つの一般的なユーザビリティテストの種類**：
  - **探索的**：ユーザーがアプリの様々な機能を操作して理解を深めます。
  - **比較的**：市場の類似アプリと自分のアプリのユーザー体験を比較します。
  - **評価的**：アプリの使いやすさを調査します。
  - **検証的**：ユーザーがアプリを効果的に使えない重大な問題を特定します。
- **ユーザビリティテストツール**：Loop11、Maze、Userbrain、UserTesting、UXTweakなどがあります。

## 互換性テスト

- **互換性テスト**：アプリケーションが異なるコンピューティング環境で動作することを保証します。
- **互換性テストの種類**：
  - **後方互換性**：ソフトウェアが以前のバージョンと互換性があること。
  - **前方互換性**：ソフトウェアやシステムが将来のバージョンでも動作すること。
  - **ハードウェア互換性**：異なるハードウェア構成で正しく動作すること。
  - **OS互換性**：macOS、Windows、UbuntuやFedoraなどのLinuxディストリビューションで動作すること。
  - **ネットワーク互換性**：異なるネットワーク速度、プロトコル、セキュリティ設定などで動作すること。
  - **ブラウザ互換性**：Google Chrome、Safari、Firefoxなど異なるブラウザで一貫して動作すること。
  - **モバイル互換性**：AndroidやiOSの様々なデバイス（スマホやタブレット）で動作すること。

## パフォーマンステスト

- **パフォーマンステスト**：アプリケーションの速度、応答性、スケーラビリティ、安定性を異なる負荷でテストし、パフォーマンスのボトルネックを解消することが目的です。
- **パフォーマンステストの種類**：
  - **負荷テスト**：通常およびピーク時の負荷でシステムの挙動を調べます。
  - **ストレステスト**：極端な負荷でシステムの耐性を確認します。
  - **ソークテスト（耐久テスト）**：長時間高負荷をかけてシステムをテストします。
  - **スパイクテスト**：負荷を急激に増減させてシステムの反応を分析します。
  - **ブレークポイントテスト（容量テスト）**：負荷を徐々に増やし、システムが失敗または劣化し始めるポイントを特定します。

## セキュリティテスト

- **セキュリティテスト**：脆弱性や弱点を特定します。
- **セキュリティの原則**：
  - **機密性**：意図しない受信者への機密情報の漏洩を防ぎます。
  - **完全性**：悪意あるユーザーによる情報の改ざんを防ぎます。
  - **認証**：ユーザーの身元を確認し、システムの利用権限を保証します。
  - **認可**：認証されたユーザーが実行できる操作やアクセス可能なシステム部分を決定します。
  - **可用性**：情報やサービスが必要なときに認可されたユーザーに利用可能であることを保証します。
  - **否認防止**：送信者と受信者の双方が配信証明と送信者の身元確認を持ち、送信者が送信を否認できないようにします。
- **一般的なセキュリティ脅威**：
  - **クロスサイトスクリプティング（XSS）**：攻撃者が悪意あるスクリプトをウェブページに注入し、被害者のブラウザで実行させます。
  - **SQLインジェクション**：悪意あるユーザーがデータベースに悪意あるコードを注入します。
  - **サービス拒否（DoS）攻撃**：悪意あるユーザーが大量のリクエストやトラフィックを送り、サーバーを遅くしたりクラッシュさせたりしてサイトを利用不能にします。
- **セキュリティテストツールのカテゴリ**：
  - **静的アプリケーションセキュリティテスト**：ソースコードを評価して脆弱性を特定します。
  - **動的アプリケーションセキュリティテスト**：アプリのフロントエンドと対話して潜在的な弱点を見つけます。DASTツールはソースコードにアクセスしません。
- **侵入テスト（ペネトレーションテスト）**：サイバー攻撃を模擬してシステムの脆弱性を特定するセキュリティテストの一種です。

## A/Bテスト

- **A/Bテスト**：ページやアプリの2つのバージョンを比較し、どちらがより良いパフォーマンスを示すかを調べます。バケットテストやスプリットテストとも呼ばれます。A/Bテストはよりデータ駆動型の意思決定を可能にし、ユーザー体験を継続的に改善します。
- **A/Bテスト用ツール**：GrowthBookやLaunchDarklyなどがあります。

## アルファテストとベータテスト

初期の開発とソフトウェアテストが完了したら、テスターや実際のユーザーによるテストが重要です。これがアルファテストとベータテストです。

- **アルファテスト**：選ばれたテスターグループがアプリを操作し、市場にリリースする前にバグがないか確認します。アルファテストは受け入れテストの一部で、ホワイトボックスとブラックボックスの両方のテスト手法を使います。
- **ベータテスト**：実際のユーザーにアプリを提供し、操作してフィードバックをもらいます。ベータテストもユーザー受け入れテストの一形態です。
- **受け入れテスト**：ソフトウェアがビジネス要件やユーザーのニーズを満たしているかリリース前に確認します。
- **ブラックボックステスト**：アプリの期待される振る舞いにのみ焦点を当てます。
- **ホワイトボックステスト**：テスターが内部コンポーネントを理解し、それらに対してテストを行います。

## リグレッションテスト

- **リグレッション**：新しい変更が既存の機能を意図せず壊すことを指します。
- **リグレッションテスト**：リグレッション問題を検出するために、アプリの一部に対して機能テストを再実行します。
- **リグレッションテスト用ツール**：Puppeteer、Playwright、Selenium、Cypressなどがあります。
- **リグレッションテストの手法**：
  - **単体リグレッションテスト**：大きな変更や修正ごとにテストすべき項目のリストを持ちます。
  - **部分的リグレッションテスト**：新しい変更が特定の機能を壊していないかターゲットを絞って確認します。
  - **完全リグレッションテスト**：コードベースのすべての機能に対してテストを実行します。最も時間がかかり詳細です。
- **再テスト**：既知の問題が解決されているか確認します。リグレッションテストはコードベースの最近の変更による未知の問題を探します。

## TypeScriptとは

- **JavaScript**：JavaScriptは動的型付け言語で、変数は実行時に任意の値を受け取れます。動的型付けの課題は型安全性がなくエラーが入りやすいことです。

例えば、JavaScript関数が配列を期待していても数値で呼び出せます：

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

上記の`console`出力は`undefined`になります。

- **TypeScript**：TypeScriptはJavaScriptに静的型付けを追加し、型の不一致によるエラーをコード実行前に検出できます。

例えば、`array`パラメータの型を次のように定義できます：

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

この型定義は`array`パラメータが文字列の配列でなければならないことをTypeScriptに伝えます。`getRandomValue`を数値で呼び出すとコンパイラエラーになります。

- **コンパイラ**：TypeScriptコードを通常のJavaScriptにコンパイルする必要があります。コンパイラはコードを評価し、型が合わない箇所でエラーを出します。

## TypeScriptのデータ型

- **TypeScriptのプリミティブ型**：`string`、`null`、`undefined`、`number`、`boolean`、`bigint`に対応する型キーワードがあります。

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **配列**：特定の型の配列を2つの構文で定義できます。

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **オブジェクト**：オブジェクトの正確な構造を定義できます。

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

任意のキーを持ち、すべての値が文字列でなければならないオブジェクトを定義する方法は2つあります：

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **TypeScriptのその他の便利な型**：
  - **`any`**：`any`は値が任意の型であることを示し、コンパイラに型を気にしないよう指示します。
  - **`unknown`**：`unknown`は値の型を気にするが実際に何かは知らないことを示します。`unknown`は`any`より一般的に好まれます。
  - **`void`**：関数の戻り値がない場合に使う特別な型です。
  - **`never`**：存在しない型を表します。

- **`type`キーワード**：`const`に似ていますが、変数ではなく型を宣言できます。

カスタム型（ユニオン型や特定の値のみを含む型など）を宣言するのに便利です：

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**：インターフェースは型のためのクラスのようなもので、他のインターフェースを実装または拡張でき、オブジェクト型に特化しており、特定の機能が必要な場合を除き`type`宣言より一般的に好まれます。

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **戻り値の型定義**：関数の戻り値の型を定義できます。

以下の例は戻り値を文字列と定義しています。異なる値を返そうとするとコンパイラエラーになります。

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```

## ジェネリクス

- **ジェネリック型の定義**：ジェネリック型は関数の型定義の振る舞いを制御する特別なパラメータのようなものです。

以下は関数のジェネリック型を定義する例です：

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

`<T>`構文は関数のジェネリック型`T`を定義していることをTypeScriptに伝えます。`T`はジェネリック型の一般的な名前ですが任意の名前を使えます。

次に`array`パラメータがジェネリック型に一致する値の配列であり、戻り値が同じ型の単一要素であることを伝えます。

- **関数呼び出しでの型引数の指定**：関数名とパラメータの間に山括弧で型引数を渡せます。

以下は型引数を渡す例です：

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

これはTypeScriptに期待する要素が入力要素であることを伝えます。

## 型絞り込み

- **真偽値による絞り込み**：以下の例では`email`が`null`である可能性があるため、`email`の`value`プロパティにアクセスしようとするとコンパイラエラーになります。

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

条件文を使って`email`が真であることを確認してからプロパティにアクセスできます：

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

真偽値チェックは逆方向にも使えます：

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

エラーをスローするとこのコードの論理実行が終了し、`console.log()`呼び出し時にTypeScriptは`email`が`null`であり得ないことを認識します。

- **オプショナルチェイニング**：オプショナルチェイニング`?.`も型絞り込みの一種で、`email`が`null`の場合はプロパティアクセスが起こらないという前提に基づきます。

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **`typeof`演算子**：`typeof`演算子を使って変数の型を条件付きでチェックできます。

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **`instanceof`キーワード**：オブジェクトがクラス由来の場合、`instanceof`キーワードで型を絞り込めます。

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **キャスト**：TypeScriptが値の型を自動判別できない場合（以下の`request.json()`メソッドの結果など）、コンパイラエラーになります。これを解決する方法の一つがキャストですが、型チェック能力が弱まります。

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **型ガード**：キャストの代わりに型ガードを書くこともできます：

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

`user is User`構文は関数が真の場合に`user`値が`User`インターフェースを満たすことを示すブール値を返すことを示します。

## `tsconfig`ファイル

- **`tsconfig.json`**：TypeScriptのコンパイラ設定はプロジェクトのルートディレクトリにある`tsconfig.json`ファイルにあります。

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

上記の例で使われているコンパイラオプションの説明は以下の通りです：

- **`compilerOptions`**：`compilerOptions`プロパティはTypeScriptコンパイラの動作を制御します。
- **`rootDir`と`outDir`**：`rootDir`と`outDir`はソースファイルのディレクトリとトランスパイル後のJavaScriptコードのディレクトリを指定します。
- **`lib`**：`lib`プロパティはコンパイラが使う型定義を決め、特定のESリリースやDOMのサポートを含められます。
- **`module`と`moduleResolution`**：`module`と`moduleResolution`はCommonJSかECMAScriptモジュールかを管理します。
- **`esModuleInterop`**：`esModuleInterop`はCommonJSとESモジュールの相互運用をスムーズにし、インポートの名前空間オブジェクトを自動生成します。
- **`skipLibCheck`**：`skipLibCheck`オプションはコード内でインポートされていない`.d.ts`ファイルの検証をスキップします。
- **`strict`**：`strict`フラグはnullable型の適切な扱いやTypeScriptが`any`にフォールバックする際の警告など複数のチェックを有効にします。
- **`exclude`**：トップレベルの`exclude`プロパティはコンパイル時にこれらのTypeScriptファイルを無視するようコンパイラに指示します。
