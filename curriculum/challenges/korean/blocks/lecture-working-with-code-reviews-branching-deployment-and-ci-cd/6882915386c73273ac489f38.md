# --description--

이전 수업에서 브랜치에 대해 조금 이야기했습니다. 브랜치를 만드는 방법, `main`에 직접 커밋하지 말아야 하는 이유 등을 배웠죠. 그렇다면 언제, 왜 브랜치를 실제로 만들어야 할까요?

먼저, `main`에 그냥 커밋을 잔뜩 푸시하면 안 되는 이유부터 이야기해 보겠습니다. 기본 브랜치는 언제나 안정적이어야 합니다. 즉, 언제든 현재 `main` 브랜치를 내려받아 애플리케이션을 빌드, 실행, 테스트할 수 있어야 합니다. 만약 부분적으로 완성된 기능이나 작업 중인 커밋을 푸시한다면, 이 조건이 항상 충족되지 않을 수 있습니다.

또한, 여러 기능을 동시에 작업하는 경우가 매우 흔합니다. 여러 기능에 대해 부분 작업을 `main`에 푸시하면, 변경 사항이 분리되어 있지 않기 때문에 문제를 디버그하기가 훨씬 어려워집니다. 게다가 작업 전환(어떤 작업을 하고 있는지 바꾸는 것)도 깔끔하지 않습니다.

그렇다면 언제 브랜치를 만들어야 할까요? 일반적인 규칙은 작업 단위마다 브랜치를 만드는 것입니다. 예를 들어, 새로운 OAuth 기능을 작업한다면 전용 브랜치에서 작업해야 합니다. 그다음 비밀번호 인증 버그를 고쳐야 한다면, 그 작업도 별도의 전용 브랜치에서 해야 합니다. 사용자 프로필 페이지를 다시 디자인해야 한다면 또 다른 브랜치를 만드세요.

항상 `main`에서 브랜치를 만들어야 한다는 점을 기억하세요. 예를 들어 OAuth 기능 브랜치에서 작업 중이라면, 비밀번호 인증 수정용 새 브랜치를 만들기 전에 `main`를 체크아웃해야 합니다. 그렇지 않고 OAuth 브랜치에서 바로 비밀번호 수정 브랜치를 만들면 OAuth 작업 커밋까지 포함되어 기록이 복잡해집니다.

그렇다면 브랜치 이름은 어떻게 정해야 할까요? 전적으로 본인(또는 기여하는 프로젝트)에 달려 있지만, 몇 가지 일반적인 관례를 소개합니다:

`scope/description` - 여기서 `scope`은 `feat` 또는 `fix` 같은 것이고, `description`은 브랜치 변경 사항에 대한 아주 짧은 설명입니다. 예를 들어: `feat/oauth-support`.

`scope/issue/description` - 앞의 관례와 비슷하지만 관련 이슈 번호를 포함합니다. 예를 들어: `fix/25/password-auth`.

`username/scope/description` - 원본 저장소에서 직접 브랜치를 작업하는 프로젝트(내부 비공개 저장소에서 흔함)에서는 브랜치 이름 앞에 사용자 이름을 포함하는 것이 도움이 될 수 있습니다. 예를 들어: `naomi-lgbt/feat/profile-redesign`.

브랜칭 전략에 관한 몇 가지 추가 사항도 알아봅시다.

베타 기능처럼 개발 시간이 많이 필요한 장기 기능 작업에는 브랜치 대신 기능 플래그를 사용해 논리를 선택적으로 활성화하는 것이 좋습니다. 브랜치가 오래 병합되지 않고 남아 있을수록 충돌 가능성이 커집니다.

하지만 배포 환경이나 릴리즈용 전용 브랜치를 유지하는 것은 좋습니다. 예를 들어 freeCodeCamp 저장소는 `prod-staging`와 `prod-current` 브랜치를 사용해 각각 스테이징과 프로덕션 서버 배포를 트리거합니다. 이렇게 하면 팀이 더 빠르게 반복 작업할 수 있고, 준비되고 승인된 변경 사항만 병합해 배포를 매번 트리거할 필요가 없습니다. 대신 최신 `main`를 해당 프로덕션 브랜치에 병합해 배포를 만듭니다.

또한 주요 버전을 나타내는 "릴리즈 브랜치"를 유지할 수도 있습니다. 이렇게 하면 각 주요 릴리즈마다 깨끗한 상태를 유지할 수 있어, 이전 버전에 중요한 보안 패치를 백포트할 때 새 버전 변경 사항을 포함하지 않고도 작업할 수 있습니다.

마지막으로, 로컬 명령줄 인터페이스를 사용해 `main`에 브랜치를 병합하는 것은 피하세요. 대신 변경 사항을 포함한 풀 리퀘스트를 만들고 그 흐름을 통해 병합하세요. 이렇게 하면 CI에서 테스트가 통과하는지 확인할 기회가 생기고, 병합할 변경 사항만 포함했는지 diff 뷰를 다시 점검할 수 있습니다(이 내용은 다음 수업에서 배울 예정입니다).

이 팁들을 활용해 성공적인 브랜칭 전략을 시작하세요!
