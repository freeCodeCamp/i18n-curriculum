# --description--

L’iniezione SQL è una vulnerabilità di sicurezza web in cui gli attaccanti modificano o sfruttano le query che l’applicazione web esegue sul database. Per esempio, gli attaccanti possono inserire codice SQL dannoso nella query per eseguire quel codice sul back-end.

Questo può succedere quando dai agli utenti la possibilità di inserire dati tramite moduli di login, barre di ricerca o parametri URL e poi usi quell’input utente in interpolazione o concatenazione di stringhe per generare il comando SQL finale.

A seconda dell’obiettivo dell’attaccante, il codice SQL dannoso può causare gravi problemi al tuo database e alla tua applicazione web, come bypassare l’autenticazione, rubare dati dal database o modificare o cancellare record del database. Perciò, devi prendere questa minaccia molto sul serio.

Per esempio, supponiamo che la tua applicazione web usi una query SQL come questa per autenticare gli utenti, incorporando direttamente il nome utente e la password inseriti dall’utente:

```sql
'SELECT * FROM users WHERE username = ' + username_input + ' AND password = ' + password_input + ';'
```

Se un attaccante inserisce qualcosa del genere nel campo `username`:

```sql
'" " OR "1"="1" --'
```

E qualsiasi altra cosa nel campo `password`, dato che `"1"="1"` è sempre `true` e `--` commenta il resto della query, la clausola `WHERE` diventa:

```sql
WHERE username = " " OR TRUE
```

Che è sempre `true`.

A seconda dell’implementazione del database, questo può restituire i dati del primo utente o di qualsiasi utente nel database e permettere all’attaccante di accedere senza credenziali valide.

Come puoi prevenire tutto questo? Ci sono due modi principali per prevenire l’iniezione SQL:

- Non scrivere query dinamiche con concatenazione di stringhe.
- E impedire che input SQL dannosi vengano inclusi nelle query che saranno eseguite sul back-end.

In pratica, puoi usare istruzioni parametrizzate, note anche come query parametrizzate. Separano la struttura SQL dai dati inseriti dall’utente, impedendo al database di interpretare i dati di input come codice eseguibile.

Un’altra tecnica importante è la validazione dell’input. È consigliata come misura di sicurezza secondaria in tutti i casi, ma è particolarmente importante per le parti delle query SQL relative ai nomi delle tabelle, ai nomi delle colonne e all’ordine di ordinamento. Se possibile, è consigliato assegnare i nomi delle tabelle e delle colonne direttamente nel codice, non dall’input utente.

Se, nonostante tutte le misure preventive, subisci un attacco di iniezione SQL, puoi minimizzarne l’impatto sul database limitando i permessi di ogni account del database. Concedi a ogni utente del database solo i permessi necessari per le sue attività, ma niente di più. Se un account ha bisogno solo di leggere dati dal database, concedigli solo l’accesso in lettura. E come misura preventiva, non assegnare mai permessi di amministratore agli account della tua applicazione. È molto pericoloso farlo perché un attaccante potrebbe ottenere accesso completo e controllo totale sul database.

L’iniezione SQL è un attacco molto comune di cui tu, come sviluppatore, devi essere consapevole e prevenire nell’implementazione della tua applicazione web. Con queste tecniche puoi proteggere il tuo database e mantenere al sicuro i dati dei tuoi utenti.
