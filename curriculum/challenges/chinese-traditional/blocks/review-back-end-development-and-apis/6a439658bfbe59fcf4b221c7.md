## 使用 Node.js 與事件驅動架構

- **Node.js**：一個開源、跨平台的 JavaScript 執行環境，允許你在瀏覽器外執行 JavaScript 程式碼。廣泛用於建置網頁伺服器、API 和後端系統。
- **客戶端限制**：客戶端 JavaScript 對檔案系統的存取有限，對複雜應用邏輯的支援有限，且存在潛在安全風險（例如，絕不能在客戶端程式碼中暴露資料庫憑證）。
- **瀏覽器與 Node.js 的全域物件**：瀏覽器中全域物件是 `window`，提供 DOM 操作、Cookie 和瀏覽器事件。Node 中的全域物件是 `global`，提供檔案、網路和作業系統互動的內建模組。
- **DOM 存取**：瀏覽器提供 DOM API 存取，但 Node 不提供。相反地，Node 幾乎可以存取所有系統資源（包括檔案系統），瀏覽器則無法。
- **版本控制**：你可以選擇伺服器上執行的 Node 版本，但無法控制使用者執行的瀏覽器版本。
- **非阻塞、事件驅動架構**：Node 使用單一執行緒和事件迴圈，有效處理多個同時請求和 I/O 操作而不阻塞，非常適合即時應用。
- **回呼函式**：定義非同步操作完成後要執行的函式。大量使用回呼是非同步 Node 程式碼較難閱讀和偵錯的原因之一。
- **Node 在後端的優點**：前後端使用單一語言（JavaScript）、非阻塞架構適合高併發、大型 npm 生態系統、免費且開源。
- **Node 在後端的缺點**：單執行緒，CPU 密集型任務（影像處理、加密、複雜數學）會阻塞事件迴圈，過度依賴非同步/回呼程式碼降低可讀性，npm 套件品質和維護度參差不齊。
- **NVM（Node 版本管理器）**：一個在同一台機器上管理和切換多個 Node.js 版本的工具。在 macOS/Linux 上安裝指令：

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **安裝 Node 的 LTS 版本**：`nvm install --lts`
- **切換 Node 版本**：`nvm use 20`
- **列出所有已安裝的 Node 版本**：`nvm ls`
- **設定預設 Node 版本（macOS/Linux）**：`nvm alias default 22.20.0`
- **nvm-windows**：Windows（非 WSL）上管理 Node 版本的獨立專案。從官方倉庫下載安裝程式，執行 `nvm install lts`，接著 `nvm use <version>`。
- **使用 Node 執行 JavaScript 檔案**：`node app.js`

## 使用 Node 核心模組

### `fs` 模組

- **用途**：提供建立、讀取、寫入、附加和刪除檔案與目錄的方法。
- **三種使用方式**：基於回呼（非同步）、基於 promise（`fs/promises`）和同步（`Sync` 後綴）。同步方法會阻塞事件迴圈，僅適合腳本使用。
- **`writeFile()`**：將內容寫入檔案，若檔案不存在則建立：

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**：將內容附加到現有檔案，不會覆寫。
- **`readFile()`**：讀取檔案內容。傳入 `"utf8"` 編碼可取得字串，省略則取得 `Buffer`。
- **`unlink()`**：刪除檔案。若檔案不存在會丟出 `ENOENT`。

### `Buffer` 模組

- **用途**：直接在記憶體中處理二進位資料（檔案、影像、網路資料流）。
- **`Buffer.from()`**：從字串建立一個緩衝區：

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**：建立一個長度為 `n` 的零填充緩衝區：

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **截斷**：寫入超過緩衝區容量的資料會自動靜默截斷。
- **`Buffer.byteLength(str)`**：回傳儲存字串所需的位元組數。

### `crypto` 模組

- **用途**：提供加密工具，如雜湊、HMAC、加解密、數位簽章和安全隨機數產生。
- **`createHash(algorithm)`**：建立單向雜湊（不可逆）。使用 `.update(data).digest("hex")` 提供要雜湊的資料並以十六進位字串回傳：

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**：類似 `createHash`，但需密鑰，適合驗證資料完整性和身份認證。
- **`createCipheriv()` / `createDecipheriv()`**：加密和解密資料。兩者都需指定演算法、金鑰和初始化向量（IV）：

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**：產生密碼學安全的隨機位元組，比 `Math.random()` 更安全，適用於產生令牌和金鑰。
- **`randomInt(min, max)`**：產生指定範圍內的安全隨機整數，適合一次性密碼（OTP）。
- **`sign()` / `verify()`**：使用私鑰/公鑰配對建立和驗證數位簽章。

### `os` 模組

- **用途**：揭露 Node 執行所在作業系統的資訊。
- **`os.platform()`**：回傳作業系統識別字串，例如 `'darwin'`、`'win32'`、`'linux'`。
- **`os.arch()`**：回傳 CPU 架構，例如 `'x64'`、`'arm64'`。
- **`os.type()`**：回傳官方作業系統名稱，例如 `'Darwin'`。
- **`os.release()`**：回傳作業系統核心版本字串。
- **`os.cpus()`**：回傳描述每個邏輯 CPU 核心的物件陣列（型號、速度、時間）。
- **`os.uptime()`**：回傳系統運行秒數。
- **`os.totalmem()` / `os.freemem()`**：回傳系統總記憶體和可用記憶體（位元組）。
- **`os.userInfo()`**：回傳包含當前使用者 `uid`、`gid`、`username`、`homedir` 和 `shell` 的物件。
- **`os.networkInterfaces()`**：回傳網路介面物件，包含位址、家族、MAC 和 CIDR 詳細資訊。

### `path` 模組

- **用途**：提供跨平台的檔案和目錄路徑工具。
- **`__filename`**：全域變數，當前檔案的絕對路徑。
- **`__dirname`**：全域變數，包含當前檔案的目錄絕對路徑。
- **`path.basename(p)`**：回傳路徑的最後一部分（檔名）。
- **`path.dirname(p)`**：回傳路徑的目錄部分。
- **`path.extname(p)`**：回傳檔案副檔名，例如 `'.js'`。
- **`path.join(...segments)`**：將多個路徑片段合併為標準化路徑，修正錯誤斜線。
- **`path.resolve(...segments)`**：從當前工作目錄解析片段為絕對路徑。
- **`path.parse(p)`**：回傳包含 `root`、`dir`、`base`、`ext` 和 `name` 屬性的物件。
- **`path.format(obj)`**：從包含 `dir`、`name` 和 `ext` 屬性的物件組合路徑字串。

### `process` 模組

- **用途**：揭露並控制當前 Node.js 執行程序。為全域物件，無需匯入。
- **`process.env`**：包含所有環境變數的物件。可存取特定值如 `process.env.NODE_ENV`。
- **`process.argv`**：命令列引數陣列。索引 0 是 Node 執行檔，索引 1 是腳本路徑，索引 2 以後是使用者引數。
- **`process.cwd()`**：回傳當前工作目錄。
- **程序事件**：

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**：觸發可被 `warning` 監聽器捕捉的自訂警告。

### `stream` 模組

- **用途**：以區塊方式處理大量資料，而非一次載入全部到記憶體。
- **四種串流類型**：可讀（分段讀取資料）、可寫（分段寫入資料）、雙工（同時可讀寫）、轉換（雙工且可修改資料流）。
- **串流串接（Piping）**：

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **`finish` 事件**：當可寫入的資料流所有資料都已清理時觸發。
- **`error` 事件**：當資料流遇到問題時觸發。

## npm 介紹

- **npm**：由三部分組成：網站 `npmjs.com`、公共登錄檔（軟體包資料庫）和隨 Node.js 安裝的 CLI 工具。
- **用途**：讓你安裝、管理和發佈可重複使用的 JavaScript 軟體包。npm 是最廣泛使用的套件管理工具。
- **替代方案**：Yarn、PNPM、Bun。

### `package.json` 檔案

- **用途**：npm 的設定清單。追蹤專案名稱、版本、描述、入口點（`main`）、作者、授權、腳本和相依項。
- **互動式建立 `package.json`**：`npm init`
- **使用預設值建立**：`npm init -y`
- **`dependencies`**：應用程式在生產環境中執行所需的套件（例如 `express`、`react`）。
- **`devDependencies`**：只在開發和測試期間需要的套件（例如 `jest`、`nodemon`、`eslint`）。
- **`license`**：使用的法律條款。常見值有 `MIT`、`ISC`、`GPL`。

### 語義版本控制（SemVer）

- **格式**：`MAJOR.MINOR.PATCH`，例如 `4.17.21`。
- **MAJOR**：有破壞性變更時遞增。
- **MINOR**：新增向後相容功能時遞增。
- **PATCH**：向後相容的錯誤修正時遞增。
- **插入符號 `^`**：允許更新次版本和修補版本，但不包含主版本。例如：`^4.17.21` 接受 `4.18.0` 但不接受 `5.0.0`。
- **波浪號 `~`**：只允許更新修補版本。例如：`~1.2.3` 接受 `1.2.4` 但不接受 `1.3.0`。
- **星號 `*`**：匹配任何版本，只適合測試，絕不適用於生產。
- **精確版本**：無前置符號，例如 `1.2.3` 鎖定為完全相同版本。

### 安裝與移除相依項

- **安裝套件**：`npm install express`（簡寫：`npm i express`）
- **安裝為開發相依項**：`npm install nodemon -D`
- **移除套件**：`npm uninstall chalk`
- **`node_modules/`**：安裝後產生的資料夾，包含實際的套件程式碼。不會提交到版本控制。
- **`package-lock.json`**：自動產生的檔案，鎖定所有已安裝套件（包含子相依項）的精確版本。務必提交此檔案，且不要手動編輯。
- **安裝特定版本**：`npm install express@4.21.2`
- **查看可用版本**：`npm view express versions`
- **檢查過時套件**：`npm outdated`
- **更新單一套件**（在允許範圍內）：`npm update express`
- **強制安裝最新版本**：`npm install express@latest`

### 發佈到 npm 登錄檔

- **登入**：`npm login`
- **作用域套件**：使用 `@username/package-name` 避免名稱衝突。
- **`.npmignore`**：列出要從發佈套件中排除的檔案（類似 `.gitignore`）。
- **發佈無作用域套件**：`npm publish`
- **公開發佈作用域套件**：`npm publish --access public`

## 使用 npm 腳本

- **npm 腳本**：在 `package.json` 的 `"scripts"` 物件中定義的自訂指令。用來自動化重複性工作，例如啟動伺服器、執行測試、程式碼檢查和建置。
- **定義腳本**：

```json
"scripts": {
  "start": "node app.js"
}
```

- **執行腳本**：`npm run start`
- **內建腳本名稱的簡寫**（`start`、`test`、`stop`、`restart`）：`npm start`
- **使用 `&&` 的連續命令**（只有前一個成功時才執行下一個命令）：

```json
"start": "npm run build && node server.js"
```

- **使用 `;` 的連續命令**（無論前一個是否成功都執行下一個命令）：

```json
"start": "npm run build; node server.js"
```

- **使用 `&` 的並行命令**：`"dev": "npm run start:server & npm run start:client"`
- **傳遞引數給腳本**（使用 `--` 分隔符）：`npm start -- 8000`；引數可透過 `process.argv[2]` 存取。

### CommonJS 模組

- **CommonJS**：Node.js 最初的模組系統。同步載入模組。
- 使用 `require()` 進行匯入：

```js
const { multiply } = require('./math');
```

- 使用 `module.exports` 進行匯出：

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- 當 `package.json` 中有 `"type": "commonjs"`（或沒有 `"type"` 欄位），或檔案副檔名為 `.cjs` 時，CommonJS 是預設模組系統。

### ES 模組 (ESM)

- **ES 模組**：現代標準化的模組格式。非同步載入。
- **命名匯出**：`export function multiply(a, b) { return a * b; }`
- **預設匯出**：`export default function multiply(a, b) { return a * b; }`
- **匯入**：`import { multiply } from './math.mjs';`
- 當檔案副檔名為 `.mjs` 或 `package.json` 包含 `"type": "module"` 時，使用 ESM。

## 了解 HTTP、DNS 和 TCP/IP 的運作原理

### 伺服器與客戶端/伺服器架構

- **客戶端/伺服器架構**：一種分散式系統架構，組件分為客戶端（發送請求的程序，例如網頁瀏覽器）或伺服器。
- **伺服器**：接收客戶端請求並透過網路回傳資料或服務的電腦或軟體程式。
- **伺服器類型**：網頁伺服器（HTML/CSS/資源）、應用伺服器（商務邏輯）、資料庫伺服器、郵件伺服器、檔案伺服器、代理伺服器。

### DNS

- **DNS（網域名稱系統）**，常被稱為「網際網路的電話簿」：將人類可讀的網域名稱（例如 `freecodecamp.org`）轉換成 IP 位址。
- **網域名稱階層**（由右至左）：頂級網域（TLD，例如 `.org`）→ 次級網域（例如 `freecodecamp`）→ 子網域（例如 `www`）。
- **IP 位址**：網路中裝置的唯一數字識別碼。IPv4 範例：`127.0.0.1`。IPv6 範例：`0:0:0:0:0:0:0:1`。
- **DNS 決議過程**：瀏覽器 → 遞迴解析器 → 根伺服器 → TLD 伺服器 → 權威名稱伺服器 → 回傳 IP 位址 → 遞迴解析器快取並傳送給客戶端。

### TCP/IP 和 HTTP

- **IP（網際網路協定）**：使用 IP 位址將資料封包從來源路由到目的地。無連線，每個封包獨立傳送。
- **TCP（傳輸控制協定）**：將資料拆成編號區段，在目的地重新組合，檢查錯誤並在需要時要求重傳。確保可靠且有序的傳送。
- **HTTP（超文字傳輸協定）**：定義客戶端和伺服器如何交換網頁內容的應用層協定。
- **HTTPS**：HTTP 的安全版本，資料使用 SSL/TLS 加密。

### HTTP 請求結構

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **請求列**：HTTP 方法 + 路徑/URI + HTTP 版本。
- **標頭**：鍵值對的詮釋資料。
- **主體**（可選）：用於 `POST`、`PUT`、`PATCH`、`DELETE` 傳送資料。

### HTTP 回應結構

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **狀態列**：HTTP 版本 + 狀態碼 + 原因短語。
- **標頭**：關於回應的詮釋資料（內容類型、長度、快取指令）。
- **主體**（可選）：請求的資料。

### HTTP 狀態碼類別

- **1xx 資訊性**：請求已接收，處理中。範例：`101 Switching Protocols`。
- **2xx 成功**：`200 OK`、`201 Created`、`204 No Content`。
- **3xx 重導向**：`301 Moved Permanently`、`302 Found`、`304 Not Modified`。
- **4xx 用戶端錯誤**：`400 Bad Request`、`401 Unauthorized`、`403 Forbidden`、`404 Not Found`。
- **5xx 伺服器錯誤**：`500 Internal Server Error`、`502 Bad Gateway`、`503 Service Unavailable`。

## 理解 HTTP 請求-回應模型

- **HTTP 方法**：`GET`（取得）、`POST`（建立）、`PUT`（完整更新）、`PATCH`（部分更新）、`DELETE`（刪除）。
- **請求-回應流程**：客戶端發送請求 → 伺服器驗證授權、檢查方法與 URL、查詢資料庫 → 伺服器建構回應（狀態碼 + 可選主體）→ 客戶端處理回應。

### HTTP 回應中回傳的資源

每個回應都包含一個 `Content-Type` 標頭，告訴瀏覽器如何處理該資源：

- **HTML**: `text/html`
- **CSS**: `text/css`
- **JavaScript**: `application/javascript`
- **JSON**: `application/json`
- **PNG/JPEG/WebP**: `image/png`, `image/jpeg`, `image/webp`
- **MP4/MP3**: `video/mp4`, `audio/mpeg`
- **PDF**: `application/pdf`
- **WOFF2**: `font/woff2`

### HTML 表單提交

- **`action` 屬性**：指定表單資料送出的 URL。
- **`method` 屬性**：指定 HTTP 方法（`GET` 或 `POST`）。HTML 原生不支援 `PUT`、`PATCH` 或 `DELETE`，請改用 Fetch API 或 Express 的 `method-override` 套件。
- **客戶端驗證**：使用 `required` 和 `pattern` 屬性提供即時回饋。伺服器端也必須驗證。
- **`GET` 表單**：資料附加在 URL 查詢參數（例如搜尋表單）。
- **`POST` 表單**：資料放在請求主體（例如登入和註冊表單）。

## 理解網頁標準模型

### 標準組織

- **WHATWG**：維護 HTML 和 DOM，作為持續更新的「活標準」。決策由主要瀏覽器引擎團隊（Chromium、Gecko、WebKit）主導。
- **W3C（萬維網聯盟）**：主要標準組織。監督 CSS、SVG、無障礙指引（WCAG）和眾多網頁 API。
- **ECMA International / TC39**：Ecma 發布官方 ECMAScript 規範。TC39 透過五階段提案流程開發語言（階段 0 概念 → 階段 4 完整規範）。
- **Khronos Group**：專注於圖形、運算和媒體。製作 WebGL 和 WebGL 2，實現瀏覽器中的 GPU 加速 3D 呈現。

### 制定網頁標準

1. **識別需求**：開發者或組織遇到問題並記錄下來。
2. **提案與早期討論**：非正式的 GitHub 問題或說明文件分享以評估興趣。TC39 中為階段 0/階段 1。
3. **撰寫規範**：精確的正式文件，涵蓋所有邊界情況、錯誤條件和安全考量。
4. **審查、異議與迭代**：草案由瀏覽器廠商公開審查，主要瀏覽器拒絕實作的功能很少成為標準。
5. **實作與測試**：廠商在功能旗標下實作，共享測試套件確保所有瀏覽器行為一致。
6. **定案**：W3C 稱為推薦標準，TC39 稱為階段 4。

### 網頁標準功能的生命週期

1. **提案**：識別問題並撰寫。
2. **孵化**：公開討論，許多提案在此階段流產。
3. **規範**：撰寫正式規範，訂定精確規則。
4. **互通性測試**：W3C 網頁平台測試套件驗證跨瀏覽器一致性。
5. **定案**：功能無需旗標即成為基線網頁平台的一部分。
6. **棄用**：被取代或不安全的功能標記為移除，但瀏覽器為相容性保留。

### 網頁標準的關鍵原則

- **開放性**：標準在公開環境開發，且免版稅實作。
- **無障礙性**：內建於 HTML、CSS 和瀏覽器行為，WCAG 提供正式指引。
- **互通性**：符合標準的頁面應在所有瀏覽器表現一致。
- **向後相容性**：舊頁面必須在現代瀏覽器繼續運作。
- **隱私與安全**：可能暴露使用者資料的新 API 需經嚴格審查。
- **分層**：HTML 負責結構，CSS 負責呈現，JavaScript 負責行為。每層可獨立採用。
- **漸進增強**：功能必須先對所有人有效，再為更強瀏覽器使用者提供增強。

## 理解 REST API 和網路服務

### 網路服務

- **API（應用程式介面）**：軟體與軟體間通訊的一般機制。
- **網路服務**：一種使用網路協定（HTTP/HTTPS）交換資料的特定 API。所有網路服務都是 API，但非所有 API 都是網路服務。
- **特性**：運行於伺服器，支援機器對機器通訊，語言無關、平台無關，使用標準網路協定。
- **REST**：表述性狀態轉移，是最廣泛使用的網路服務風格。簡單、彈性，與 HTTP 自然搭配。
- **SOAP**：簡單物件存取協定，只使用 XML，常見於需要嚴格標準的企業系統。
- **XML-RPC**：使用 XML 編碼輕量級請求與回應。

### REST 架構

- **無狀態性**：每個請求獨立處理。伺服器不儲存請求間的會話狀態。每個請求必須包含所有必要資訊（例如每次呼叫都帶授權令牌）。
- **資源**：一切皆為資源，由 URL 識別，例如 `/products` 代表群集，`/products/1` 代表特定項目。
- **REST 中的 HTTP 方法**：`GET`（取得）、`POST`（建立）、`PUT`/`PATCH`（更新）、`DELETE`（刪除）。
- **回應格式**：JSON 最常見，但也支援 XML 和純文字。客戶端透過 `Accept` 標頭指定格式：

```bash
Accept: application/json
```

### 微服務

- **單體式**：一個大型程式碼庫，UI、資料庫和 API 全部集中。隨著複雜度增加，維護和擴展困難。
- **微服務**：將大型應用拆分成小型獨立服務，每個服務負責單一商務功能。服務間透過 API 或訊息佇列通訊。
- **優點**：獨立快速開發與部署、更佳故障隔離、不同服務可使用不同語言或框架。
- **缺點**：除錯更複雜（問題跨服務）、基礎建設額外負擔較高、網路通訊增加。

## 使用 Express

- **Express.js**：基於 Node.js 的極簡且彈性網頁框架。比原生 `http` 模組更簡化建置網頁伺服器和 API 的樣板程式碼。
- **一個基本的 Express 伺服器**：

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**：工廠函式，建立 Express 應用實例（`app`）。
- **`app.listen(port, callback)`**：啟動伺服器並開始監聽指定埠的 HTTP 請求。
- **`req`**：請求物件，包含 URL、標頭、參數、主體和查詢資料。
- **`res`**：回應物件，用於回傳資料給客戶端。
- **中介軟體**：介於接收請求和送出回應之間執行的程式碼。

## 理解 Express.js 路由

- **路由**：定義應用如何回應客戶端以特定 HTTP 方法訪問特定 URL。
- **`app.get/post/put/delete(path, handler)`**：註冊對應 HTTP 方法的處理常式。
- **靜態路由**：固定 URL 模式，例如 `app.get("/home", handler)`。
- **帶參數的動態路由**：使用 `:paramName` 佔位符。透過 `req.params` 捕捉：

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**：路由參數（`:id` 段）。
- **`req.query`**：查詢字串參數（`?key=value`）。
- **`req.body`**：請求主體（需中介軟體解析）。
- **`res.send(data)`**：送出回應（文字、HTML、JSON 或二進位）。
- **`res.json(obj)`**：送出 JSON 回應，自動設定 `Content-Type: application/json`。
- **`res.status(code).send(msg)`**：設定狀態碼並送出回應。
- **`res.redirect(url)`**：將客戶端重導向至另一 URL。
- **`res.render(view, data)`**：渲染模板引擎視圖並送出 HTML。

### 使用 `app.route()` 串鏈路由處理常式

將同一路徑的多個 HTTP 方法群組，減少重複：

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

建立模組化小型應用以群組相關路由：

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

### 提供靜態檔案服務

- **`express.static(directory)`**：內建中介軟體，從指定目錄提供檔案服務：

```js
app.use(express.static(path.join(__dirname, "public")));
```

- 使用掛載路徑從多個目錄提供服務：`app.use("/images", express.static(path.join(__dirname, "images")));`

## 理解錯誤處理與健康檢查

### Express 中的錯誤處理

- **錯誤處理中介軟體**：以四個參數 `(err, req, res, next)` 辨識。總是放在中介軟體堆疊的最後：

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **Express 4 非同步錯誤**：必須手動用 `try/catch` 捕捉並用 `next(err)` 傳遞：

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **Express 5 非同步錯誤**：自動傳遞給錯誤處理中介軟體，無需手動 `next(err)`。
- **404 處理器**：放在所有路由後、錯誤處理器前：

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### 偵錯與日誌記錄

- **內建偵錯**：設定環境變數 `DEBUG` 啟用：`DEBUG=express:* node index.js`
- **`morgan`**：流行的第三方 HTTP 請求日誌中介軟體。記錄方法、URL、狀態碼和回應時間：

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### 健康檢查與優雅關閉

- **健康檢查路由**：回傳 `200 OK` 以表示應用程式正在運行。由負載平衡器和容器編排器使用：

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**：當程序應該終止時發送的訊號。處理它以允許正在進行的請求完成：

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**：在本地開發時按下 Ctrl+C 發送的訊號。處理方式與 `SIGTERM` 相同。
- **在關閉前標記為不健康**：在關閉期間從健康檢查端點回傳 `503`，以阻止負載平衡器路由新流量：

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## Express 中介軟體

- **中介軟體**：在請求生命週期中執行的函式。它可以存取 `req`、`res` 和 `next`。呼叫 `next()` 將控制權傳遞給下一個中介軟體，省略則結束該循環。
- **應用程式層級中介軟體**：透過 `app.use()` 綁定到整個應用程式。對每個進入請求都會執行。
- **路由器層級中介軟體**：透過 `router.use()` 綁定到特定的 `express.Router()` 實例。只套用於該路由器處理的路由。
- **錯誤處理中介軟體**：以四個參數 `(err, req, res, next)` 辨識。放置於中介軟體堆疊底部。
- **`express.json()`**：解析帶有 JSON 載荷的進入請求。使資料可從 `req.body` 取得。
- **`express.urlencoded({ extended: true })`**：解析 URL 編碼的表單資料。
- **`express.static('public')`**：從目錄提供靜態檔案服務。
- **`cors`**：啟用跨來源資源共享。使用 `npm install cors` 安裝，並用 `app.use(cors())` 啟用。
- **`morgan`**：記錄 HTTP 請求。使用 `app.use(morgan('tiny'))`。
- 中介軟體依註冊順序由 `app.use()` 套用。

## 了解 WebSockets

- **WebSockets**：一種協定，建立客戶端與伺服器間持久且全雙工（雙向）連線，實現即時通訊。
- **HTTP 在即時通訊上的問題**：HTTP 需要客戶端反覆輪詢伺服器以取得新資料，因此對聊天應用、即時比分板和多人遊戲來說速度慢且效率低。
- **WebSocket 握手**：每個 WebSocket 連線都從 HTTP 請求開始。客戶端請求升級，伺服器回應 `101 Switching Protocols`。之後連線保持開啟，直到任一方發送關閉訊號。
- **使用案例**：聊天應用、線上多人遊戲、即時運動儀表板、股市資訊串流、協作工具。

### `ws` 函式庫

- Node.js 沒有內建 WebSocket 伺服器；`ws` 函式庫是實作的常用方式。

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**：向特定客戶端發送訊息。
- **向所有連線客戶端廣播**：

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**：每個連線的屬性。`WebSocket.OPEN` 表示連線是作用中的。

### 發佈/訂閱架構

- **Pub/Sub（發佈/訂閱）**：一種訊息模式，**發佈者** 將訊息送到 **主題**，**訂閱者** 從他們訂閱的主題接收訊息。發佈者和訂閱者不直接通訊。
- **相較於直接通訊的優點**：解耦組件，發佈者不需知道訂閱者是誰。
- **Pub/Sub 與 WebSockets**：WebSockets 是通訊管道（道路），Pub/Sub 是決定誰接收哪些訊息的訊息系統（交通系統）。

## 認識身份驗證與授權

### 身份驗證與授權的差異

- **身份驗證**：驗證身份，回答「你是誰？」的問題，必須在授權之前完成。
- **授權**：決定權限，回答「你被允許做什麼？」的問題。
- **三種身份驗證因素**：
  - **你知道的東西**：密碼、PIN 碼、安全問題答案。
  - **你擁有的東西**：接收簡訊碼的智慧型手機、驗證器應用程式的代碼。
  - **你是什麼**：指紋、臉部辨識（生物特徵）。
- **授權中的常見角色**：標準使用者（有限自助服務）、編輯者（內容管理）、管理員（完整系統存取）。
- 在後端程式碼中，授權通常透過中介軟體執行，檢查使用者角色或權限後才授予存取。

### 跨站請求偽造（CSRF）

- **CSRF**：一種攻擊，惡意網站誘騙已驗證使用者的瀏覽器向後端伺服器發出不想要的請求，利用瀏覽器自動附加的 Cookie。
- **攻擊流程**：使用者登入 `trusted-bank.com` → 訪問 `evil-hacker-site.com` → 惡意網站提交隱藏表單給銀行 → 銀行伺服器看到有效的會話 Cookie 並執行請求。
- **`SameSite=Strict`**：若請求來自第三方網站，瀏覽器絕不送出 Cookie。
- **`SameSite=Lax`**：瀏覽器只在使用者安全導向目標網站（點擊連結）時允許 Cookie，但阻擋跨站表單提交。
- **CSRF 令牌**：伺服器產生隨機且不可預測的令牌嵌入表單。客戶端必須在每個會改變狀態的請求中包含該令牌。惡意網站無法從前端讀取令牌，因此偽造請求會驗證失敗。

### JSON 網路語彙單元（JWT）

- **JWT**：伺服器在成功登入後簽發的數位簽章令牌。客戶端儲存並在後續每次請求中帶上，實現無狀態身份驗證（不需伺服器端會話存儲）。
- **JWT 結構**：三個以點分隔的 Base64URL 編碼部分：
  - **標頭**：令牌類型（`JWT`）和簽名演算法（例如 `HS256`）。
  - **有效載荷**：關於使用者的聲明（使用者 ID、使用者名稱、角色）。未加密，切勿存放密碼或敏感資料。
  - **簽名**：標頭 + 有效載荷 + 伺服器密鑰的雜湊。防止篡改。

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

- 若使用者篡改有效載荷，簽名將不匹配，伺服器會以 `401 Unauthorized` 拒絕請求。

### Passport.js

- **Passport.js**：Node.js 的身份驗證中介軟體。標準化身份驗證流程，處理底層細節，讓開發者專注於商務邏輯。
- **策略**：實作特定身份驗證方法的模組化外掛。超過 500 種策略可用：`passport-local`（使用者名稱/密碼）、`passport-jwt`（JWT）、`passport-google-oauth20`（Google 登入）、`passport-facebook`、`passport-github2`。
- **整合到 Express**：

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **作為路由中介軟體使用**：

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- 成功驗證後，Passport 將一段使用者資料存入會話 Cookie，並將驗證過的使用者物件附加到 `req.user`。

### Helmet.js

- **Helmet.js**：一組安全中介軟體，透過自動設定、修改或移除 HTTP 回應標頭來保護 Express 應用程式。

```js
const helmet = require('helmet');
app.use(helmet());
```

- **移除 `X-Powered-By`**：防止攻擊者得知伺服器技術。
- **設定 `Content-Security-Policy`**：限制腳本、圖片和樣式的載入來源，減輕 XSS 攻擊。
- **設定 `X-Frame-Options: SAMEORIGIN`**：防止點擊劫持，阻止網站被外部 iframe 嵌入。
- **設定 `Strict-Transport-Security`**：強制瀏覽器使用 HTTPS，保護資料不被攔截。

## 了解網路應用程式的安全與隱私

### 安全與隱私的差異

- **安全**：保護系統和資料免於未授權存取的技術防禦機制。安全是門上的鎖。
- **隱私**：管理個人資料如何被收集、使用和分享。隱私是窗戶上的窗簾。
- 你可以有安全卻沒有隱私（鎖住的資料庫仍可能販售你的資料），但沒有安全就不可能有隱私（未受保護的資料無法保持私密）。

### 同源政策與 CORS

- **來源**：由三個部分定義：協定（例如 `https`）、網域（例如 `example.com`）和埠（例如 `443`）。三者皆相符時，兩個 URL 才共享同一來源。
- **同源政策**：瀏覽器的安全規則，阻止一個來源的腳本讀取不同來源的資料，防止惡意網站竊取其他開啟分頁的資料。
- **CORS（跨來源資源共享）**：讓伺服器透過設定 HTTP 標頭放寬同源政策，允許受信任的來源存取。
- **在 Express（開發環境）**：`app.use(cors())` 允許所有來源。
- **在 Express（生產環境）**：傳入帶有 `origin` 白名單的 `corsOptions` 物件：

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**：由 TLS（傳輸層安全性）提供加密的 HTTP。TLS 取代了舊有的 SSL 協定。
- **HTTPS 的三大保證**：
  - **加密**：客戶端與伺服器間傳送的資料對第三方不可讀。
  - **身份驗證**：由受信任的憑證授權中心簽發的數位憑證，確認伺服器身份。
  - **資料完整性**：訊息在傳輸途中無法被未被偵測地竄改。
- 現代瀏覽器 API（如地理位置、推播通知和相機存取）都要求 HTTPS。

### 為何會發生安全問題

- **快速交付壓力**：團隊為趕期限而跳過安全審查，導致漏洞流出。
- **複雜系統整合**：供應鏈風險來自第三方函式庫或雲端服務引入開發團隊未撰寫的漏洞。
- **設定管理不善**：保留預設或出廠設定（預設管理員密碼、開放埠）導致系統暴露。
- **人為錯誤**：釣魚攻擊利用員工點擊惡意連結或附件並交出憑證。
- **組織合併**：合併不同程式碼庫和基礎建設，暴露不匹配的安全政策和舊有漏洞。

### Cookies：儲存、安全旗標與隱私

- **Cookies 如何儲存**：伺服器發送 `Set-Cookie` 標頭。瀏覽器將檔案存到磁碟，並在後續每次對該來源的請求自動帶回。
- **`HttpOnly` 旗標**：阻止 JavaScript 讀取 Cookie 值，防止基於 XSS 的會話竊取。
- **`Secure` 旗標**：限制 Cookie 僅在 HTTPS 連線中傳送，保護公共網路上的攔截。
- **`SameSite=Strict`**：阻止瀏覽器在任何跨站請求中送出 Cookie，完全阻擋 CSRF 攻擊。
- **`SameSite=Lax`**：允許 Cookie 在頂層導覽（如點擊連結）時送出，但阻擋跨站表單提交和 API 呼叫。
- **第一方 Cookie**：限定於你正在訪問的網站。
- **第三方 Cookie**：由廣告網路在數千個網站設置，用於追蹤行為和建立使用者檔案。現代瀏覽器預設封鎖。
- **GDPR 合規**：隱私法規要求顯示 Cookie 同意橫幅，賦予使用者拒絕非必要追蹤 Cookie 的合法權利。

### 常見安全威脅與緩解措施

- **釣魚攻擊**：冒充可信公司發送詐騙郵件以竊取憑證。緩解措施包括安全意識訓練和部署郵件驗證協定以阻擋惡意郵件進入收件匣。
- **惡意軟體**（病毒、間諜軟體、勒索軟體）：勒索軟體加密檔案並要求贖金解鎖。緩解措施包括自動更新和防毒軟體。
- **憑證填充攻擊**：攻擊者在不同網站測試數百萬組被竊取的使用者名稱和密碼組合。主要緩解是多因素驗證（MFA），需第二道驗證步驟，如一次性密碼或驗證器代碼。
- **DDoS（分散式阻斷服務）**：攻擊者以假流量淹沒伺服器，使其當機並阻擋合法使用者。緩解措施包括雲端流量過濾服務，如 Cloudflare、Akamai 和 AWS Shield。
- **內部威脅**：惡意或疏忽的員工竊取或洩漏資料。緩解措施包括執行**最小權限原則**（員工僅能存取工作所需資源）和監控系統日誌異常活動。

### 內容安全政策與權限政策

- **CSP（內容安全政策）**：HTTP 回應標頭，白名單允許的腳本、樣式和圖片來源。瀏覽器收到 CSP 標頭後，會阻擋任何非清單上的資源載入。`script-src` 指令指定可執行 JavaScript 的網域，阻止 XSS 注入攻擊。
- **Permissions-Policy**：HTTP 標頭，啟用或停用頁面及其嵌入 iframe 的瀏覽器功能和裝置硬體。取代舊的 `Feature-Policy` 標頭。
  - `camera=(self)` 限制網路攝影機存取僅限頁面自身來源。
  - `geolocation=()` 完全停用所有使用者的地理位置功能。
- 結合兩個標頭提供分層保護：CSP 防止程式碼注入，Permissions-Policy 防止未授權硬體存取。

### 基本使用者隱私概念

- **PII（個人可識別資訊）**：任何可直接或間接識別個人的資料。
  - **直接識別符**：全名、住址、國民身分證號碼。
  - **間接識別符**：IP 位址、裝置序號、精確位置歷史。
- **機密性**：透過加密和存取控制保護 PII 免於未授權存取。
- **資料最小化**：僅收集完成特定任務所需的最少 PII。
- **目的限制**：為一個理由收集的資料不得未經使用者同意而轉作他用（例如販售給廣告網路）。
- **追蹤**：持續監控使用者線上行為，使用腳本、裝置指紋和瀏覽器歷史紀錄。
- **被遺忘權（資料刪除）**：使用者關閉帳號後，有法律權利要求公司永久刪除所有個人紀錄。

### `.env` 檔案與 `dotenv`

- **`.env` 檔案**：專案根目錄下的純文字檔，儲存 API 金鑰、資料庫密碼和埠號等設定祕密，以鍵值對形式。務必加入 `.gitignore`，避免祕密被版本控制。
- **`dotenv` 套件**：啟動時讀取 `.env` 檔案，並將值注入 `process.env`：

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **兩大關鍵利益**：憑證不會出現在原始碼中（安全性），同一份程式碼可透過在不同伺服器替換 `.env` 檔案，在開發、測試和生產環境展現不同行為（彈性）。

### 地區性隱私法規與合規

- **GDPR（通用資料保護規範）**：保護歐盟個人。要求追蹤前明確同意，不允許預先勾選。賦予使用者查看和永久刪除資料的權利。違規罰款高額。
- **CCPA（加州消費者隱私法案）/ CPRA（加州隱私權法案）**：保護加州居民。重點是選擇退出權利而非選擇加入。首頁必須有明顯的「請勿販售我的個人資訊」連結。
- **COPPA（兒童線上隱私保護法案）**：美國聯邦法，保護 13 歲以下兒童。要求明確隱私政策和可驗證的家長同意，才能收集兒童個資。
- **DPA（資料保護法案）**：英國主要的資料隱私框架。與英國 GDPR 並行，賦予使用者存取和刪除個資的權利。
- 其他國家有各自法規：奈及利亞有 NDPA（奈及利亞資料保護法），印度有 DPDP Act（數位個人資料保護法）。
- **全球團隊合規步驟**：
  - 建立依使用者地理位置顯示適當 Cookie 通知的同意橫幅。
  - 實作自動化 DSAR（資料主體存取請求）系統，依法尋找、匯出或永久刪除使用者資料。
  - 實踐資料最小化，並使用年齡驗證機制防止未授權追蹤未成年者。
  - 指定資料保護專員（DPO）監督隱私架構，並作為監管機關的主要聯絡人。
