## WebSockets vs HTTP

- **HTTP** segue um modelo de requisição-resposta: o cliente envia uma requisição, o servidor envia uma resposta, e então a conexão é encerrada.
- **WebSockets** criam uma conexão permanente e bidirecional entre cliente e servidor que permanece aberta até que um dos lados envie um sinal de encerramento.
- HTTP é ideal para carregar páginas ou buscar dados, mas fica lento e ineficiente para aplicações em tempo real.
- Toda conexão WebSocket começa com um **handshake HTTP**: o cliente solicita uma atualização, e o servidor responde com `101 Switching Protocols`.

### Fluxo da conexão WebSocket

1. O cliente conecta ao servidor via HTTP e solicita uma atualização.
2. O servidor aceita e responde com `101 Switching Protocols`.
3. A conexão permanece aberta.
4. Ambos os lados trocam dados a qualquer momento até que a conexão seja encerrada.

### Comunicação HTTP vs WebSocket

- **HTTP**: `Client → Request → Server → Response` (um ciclo, depois fecha)
- **WebSocket**: `Client ↔ Server` (persistente, bidirecional)

### Uma analogia útil

- **HTTP** é como trocar cartas: você escreve uma, espera a resposta, e a conversa termina.
- **WebSockets** são como uma ligação telefônica: ambos falam em tempo real pelo tempo que for necessário.

### Usos reais dos WebSockets

- Aplicativos de chat
- Jogos multiplayer online
- Atualizações ao vivo de placares esportivos
- Painéis de mercado financeiro
- Ferramentas colaborativas (ex.: Google Docs)

## Arquitetura Pub/Sub

- **Pub/Sub** (Publicar/Assinar) é uma arquitetura de mensagens que permite que partes de um sistema se comuniquem sem se conhecerem diretamente.
- Existem três partes principais: **publicadores**, **tópicos** e **assinantes**.
  - Um **publicador** envia mensagens para um tópico.
  - Um **tópico** (também chamado de canal ou grupo) atua como intermediário.
  - Um **assinante** recebe mensagens dos tópicos aos quais está inscrito.

### Fluxo de mensagens Pub/Sub

1. Um assinante se inscreve em um tópico.
2. Um publicador envia uma mensagem para esse tópico.
3. O sistema de mensagens entrega a mensagem para todos os assinantes desse tópico.

### Comunicação direta vs Pub/Sub

- **Sem Pub/Sub**: `Application A → Application B` — fortemente acoplado; uma mudança em B pode quebrar A.
- **Com Pub/Sub**: `Application A → Topic → Subscribers` — A apenas publica; não precisa saber quem recebe.

### Usos reais do Pub/Sub

- Aplicativos de chat com várias salas
- Jogos multiplayer online
- Sistemas de notificações
- Painéis ao vivo
- Aplicações orientadas a eventos

### WebSockets vs Pub/Sub

- **WebSockets** fornecem o canal de comunicação (a estrada).
- **Pub/Sub** fornece o padrão de mensagens que determina quem recebe mensagens específicas (o sistema de trânsito).
- Muitas aplicações em tempo real usam ambos juntos: WebSockets mantêm a conexão aberta, enquanto Pub/Sub direciona as mensagens aos assinantes corretos.

## Trabalhando com WebSockets no Node.js

- Node.js não tem servidor WebSocket embutido. A biblioteca `ws` é o pacote externo padrão usado para isso.
- Instale com: `npm install ws`

### Criando um servidor WebSocket

```js
const WebSocket = require("ws");

const server = new WebSocket.Server({ port: 3000 });
```

### Eventos WebSocket

| Evento | Acionador |
|---|---|
| `connection` | Um novo cliente conecta ao servidor |
| `message` | Um cliente conectado envia dados |
| `close` | Uma conexão é encerrada |
| `error` | Algo dá errado na conexão |

### Enviando uma mensagem para um cliente

```js
socket.send("Hello from the server");
```

### Transmitindo para todos os clientes conectados

- `server.clients` mantém todos os clientes conectados.
- Verifique `readyState === WebSocket.OPEN` antes de enviar para evitar erros com clientes desconectados.

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **Transmissão (broadcasting)** é o padrão de enviar uma mensagem para vários clientes conectados ao mesmo tempo — usado em apps de chat, notificações ao vivo, atualizações esportivas e eventos de jogos multiplayer.
