# --description--
  
Dynamische Programmierung ist eine algorithmische Technik, die komplexe Probleme löst, indem sie diese in einfachere Teilprobleme zerlegt und die Ergebnisse speichert, um redundante Berechnungen zu vermeiden. Dieser Ansatz verwandelt Probleme, die normalerweise exponentielle Zeit benötigen, in solche, die in polynomialer Zeit gelöst werden können.

## Grundsätze der dynamischen Programmierung

Dynamische Programmierung funktioniert, wenn zwei wichtige Bedingungen bei einem Problem vorliegen.

- **Überlappende Teilprobleme**: Dieselben kleineren Probleme treten mehrfach auf, wenn das größere Problem gelöst wird. Anstatt diese Teilprobleme immer wieder neu zu berechnen, speichern wir ihre Lösungen.
    
- **Optimale Teilstruktur**: Die optimale Lösung des Problems enthält optimale Lösungen seiner Teilprobleme. Das bedeutet, wir können die beste Lösung aufbauen, indem wir die besten Lösungen für kleinere Teile kombinieren.
    

Lassen Sie uns diese Konzepte anhand des klassischen Problems „Treppensteigen“ betrachten.

## Das Problem mit naiver Rekursion

Betrachten Sie das Problem des Treppensteigens: Sie steigen eine Treppe mit `n` Stufen hinauf und können jeweils 1 oder 2 Stufen auf einmal steigen. Wie viele verschiedene Wege gibt es, um nach oben zu gelangen?

```python
def climb_stairs_recursive(n):
    """Recursive approach"""
    if n <= 2:
        return n  # Base cases: 1 way for 1 step, 2 ways for 2 steps
    # To reach step n, we can come from step (n-1) or step (n-2)
    return climb_stairs_recursive(n-1) + climb_stairs_recursive(n-2)
```

Diese Implementierung hat eine exponentielle Zeitkomplexität aufgrund massiver redundanter Berechnungen. Beim Berechnen von `climb_stairs(5)` passiert Folgendes:

- `climb_stairs(5)` ruft `climb_stairs(4)` und `climb_stairs(3)` auf
    
- `climb_stairs(4)` ruft `climb_stairs(3)` und `climb_stairs(2)` auf
    
- Jetzt wird `climb_stairs(3)` **zweimal** berechnet
    
- `climb_stairs(3)` ruft `climb_stairs(2)` und `climb_stairs(1)` auf
    
- `climb_stairs(2)` wird **insgesamt 3-mal** berechnet
    

Für nur `n=5` machen wir 9 Funktionsaufrufe, obwohl wir nur 5 eindeutige Berechnungen benötigen. Mit wachsendem `n` explodiert diese Redundanz exponentiell – `climb_stairs(30)` würde einige Millionen Funktionsaufrufe erfordern. Die Zeitkomplexität wird `O(2^n)`, was ineffizient und unpraktisch für größere Werte von `n` ist.

## Lösungen mit dynamischer Programmierung

Dynamische Programmierung beseitigt diese redundante Berechnung durch zwei Hauptansätze:

### Memoisierung (Top-Down-Ansatz)

Memoisierung speichert die Ergebnisse teurer Funktionsaufrufe und gibt das zwischengespeicherte Ergebnis zurück, wenn dieselben Eingaben erneut auftreten:

```python
def climb_stairs_memo(n, memo={}):
    """Dynamic programming with memoization"""
    # Check if we've already calculated this value
    if n in memo:
        return memo[n]  # Return cached result - O(1) lookup!
    
    # Base cases
    if n <= 2:
        return n
    
    # Calculate once and store in memo for future use
    memo[n] = climb_stairs_memo(n-1, memo) + climb_stairs_memo(n-2, memo)
    return memo[n]
```

Memoisierung ist viel effizienter, weil jeder eindeutige Wert von `1` bis `n` genau einmal berechnet wird. Wenn wir `climb_stairs(3)` erneut benötigen, schauen wir es einfach in unserem Memo-Wörterbuch nach, anstatt es neu zu berechnen (was weitere rekursive Aufrufe auslösen würde), und das in `O(1)` Zeit.

Verfolgen wir die Ausführung von `climb_stairs(5)` mit dem Top-Down-Ansatz, um zu sehen, wie Memoisierung redundante Arbeit eliminiert:

```md
Call: climb_stairs_memo(5)
  memo = {} (empty)
  
  Call: climb_stairs_memo(4) 
    memo = {} (empty)
    
    Call: climb_stairs_memo(3)
      memo = {} (empty)
      
      Call: climb_stairs_memo(2) → returns 2 (base case)
      Call: climb_stairs_memo(1) → returns 1 (base case)
      
      Result: 2 + 1 = 3
      memo = {3: 3} (stored!)
    
    Call: climb_stairs_memo(2) → returns 2 (base case)
    
    Result: 3 + 2 = 5
    memo = {3: 3, 4: 5} (stored!)
  
  Call: climb_stairs_memo(3) → returns 3 (FROM MEMO - no recursion!)
  
  Result: 5 + 3 = 8
  memo = {3: 3, 4: 5, 5: 8}
```

**Effizienzvergleich**

- **Naive Rekursion**: Macht 9 Funktionsaufrufe mit wiederholten Berechnungen
    
- **Memoisierung**: Macht nur 5 eindeutige Berechnungen und verwendet dann gespeicherte Ergebnisse
    
- **Zeitkomplexität**: Reduziert von `O(2^n)` auf `O(n)`, da wir nur `n` eindeutige Berechnungen machen
    
- **Speicherkomplexität**: `O(n)` für den Memo-Speicher und den Aufruf-Stack
    
- **Tatsächliche Auswirkung**: `climb_stairs(30)` sinkt von Millionen Funktionsaufrufen auf etwa 30 eindeutige Teilproblem-Berechnungen
    

### Tabellierung (Bottom-Up-Ansatz)

Tabellierung baut die Lösung von unten nach oben auf, indem eine Tabelle mit Lösungen für Teilprobleme gefüllt wird:

```python
def climb_stairs_tabulation(n):
    """Dynamic programming with tabulation"""
    if n <= 2:
        return n
    
    # Create array to store results for all steps from 0 to n
    dp = [0] * (n + 1)
    dp[1] = 1  # 1 way to reach step 1
    dp[2] = 2  # 2 ways to reach step 2
    
    # Build up the solution iteratively
    for i in range(3, n + 1):
        # Ways to reach step i = ways to reach (i-1) + ways to reach (i-2)
        dp[i] = dp[i-1] + dp[i-2]
    
    return dp[n]
```

Tabellierung eliminiert Rekursion vollständig, indem die Lösung iterativ von den kleinsten Teilproblemen bis zum Ziel aufgebaut wird.

Sehen wir uns den Bottom-Up-Ansatz in Aktion an, um zu sehen, wie wir die Lösung systematisch aufbauen. Hier ist der iterative Aufbau für `climb_stairs(5)`:

```md
Initial state:
dp = [0, 1, 2, 0, 0, 0]
     [0, 1, 2, 3, 4, 5] ← indices (step numbers)

Step by step construction:

i = 3:
  dp[3] = dp[2] + dp[1] = 2 + 1 = 3
  dp = [0, 1, 2, 3, 0, 0]
  
i = 4:
  dp[4] = dp[3] + dp[2] = 3 + 2 = 5
  dp = [0, 1, 2, 3, 5, 0]
  
i = 5:
  dp[5] = dp[4] + dp[3] = 5 + 3 = 8
  dp = [0, 1, 2, 3, 5, 8]

Final result: dp[5] = 8
```

**Wichtige Vorteile der Tabellierung**

- **Kein Rekursions-Overhead**: Im Gegensatz zur Memoisierung gibt es keinen rekursiven Aufruf-Stack.
    
- **Vorhersehbare Ausführung**: Wir berechnen Werte in einer vorgegebenen Reihenfolge (1, 2, 3, 4, 5...).
    
- **Cache-freundlich**: Sequenzieller Array-Zugriff optimiert die Speichernutzung.
    
- **Leicht zu optimieren**: Kann die Speicherkomplexität auf `O(1)` reduzieren, da nur die letzten zwei Werte benötigt werden.
    

```python
def climb_stairs_optimized(n):
    if n <= 2:
        return n
    
    prev2, prev1 = 1, 2  # Only store last two values
    for i in range(3, n + 1):
        current = prev1 + prev2
        prev2, prev1 = prev1, current
    return prev1
```

**Effizienzvergleich**

- **Naive Rekursion**: 9 Funktionsaufrufe für `n=5`, exponentielles Wachstum.
    
- **Tabellierung**: 3 einfache Additionen für `n=5`, lineares Wachstum.
    
- **Zeitkomplexität**: `O(n)` statt `O(2^n)`.
    
- **Speicherkomplexität**: `O(n)` für das Array oder `O(1)` mit Optimierung.
    
- **Vorhersehbare Leistung**: Kein Risiko eines Stack-Überlaufs bei großen Eingaben.
    

Beide Ansätze reduzieren die Zeitkomplexität von exponentiellem `O(2^n)` auf lineares `O(n)`, eine dramatische Verbesserung, die den Unterschied ausmacht zwischen der Lösung des Problems in Millisekunden und dem Warten über Jahre bei größeren Eingaben.

## Anwendungen in der Praxis

Dynamische Programmierung findet breite Anwendung in der Informatik und darüber hinaus:

- **Routenoptimierung**: GPS-Systeme verwenden dynamische Programmieralgorithmen, um kürzeste Wege zwischen Orten zu finden.
    
- **Textverarbeitung**: Rechtschreibprüfer und Autovervollständigungsfunktionen basieren oft auf dynamischer Programmierung, um Editierabstände zwischen Wörtern zu berechnen.
    
- **Finanzmodellierung**: Anlagestrategien und Portfolio-Optimierung nutzen häufig Techniken der dynamischen Programmierung.
    
- **Ressourcenzuteilung**: Das Rucksackproblem und seine Varianten treten in Planung, Budgetierung und Ressourcenmanagement auf.
    

## Praktisches Beispiel: Münzwechselproblem

Das Münzwechselproblem ist eine klassische Programmieraufgabe, die bei Lösung mit dynamischer Programmierung beide Schlüsselprinzipien von DP demonstriert: optimale Teilstruktur und überlappende Teilprobleme.

Das Münzwechselproblem fragt: „Wie viele Münzen werden mindestens benötigt, um einen Zielbetrag zu erreichen?“

Hier ist eine Lösung mit dynamischer Programmierung:

```python
def min_coins(amount, coins):
    """Find minimum number of coins needed to make the given amount"""
    # Initialize dp array with "infinity" - represents impossible to make
    dp = [float('inf')] * (amount + 1)
    dp[0] = 0  # Base case: 0 coins needed for amount 0
    
    # For each amount from 1 to target amount
    for i in range(1, amount + 1):
        # Try each coin denomination
        for coin in coins:
            if coin <= i:  # Can only use coin if it doesn't exceed current amount
                # Update minimum: current minimum vs (coins for remaining amount + 1)
                dp[i] = min(dp[i], dp[i - coin] + 1)
    
    # Return result if possible, -1 if impossible
    return dp[amount] if dp[amount] != float('inf') else -1

# Example usage:
# coins = [1, 3, 4], amount = 6
# dp[6] = min(dp[5]+1, dp[3]+1, dp[2]+1) = min(3+1, 1+1, 2+1) = 2
# Result: 2 coins (3 + 3)
```

Und so funktioniert der dynamische Programmieralgorithmus für das Münzwechselproblem Schritt für Schritt für `coins = [1, 3, 4]`, `amount = 6`:

```md
Initial state:
dp = [0, ∞, ∞, ∞, ∞, ∞, ∞]
     [0, 1, 2, 3, 4, 5, 6] ← amounts

Building up the solution:

For amount = 1:
  Try coin 1: dp[1] = min(∞, dp[0] + 1) = min(∞, 0 + 1) = 1
  dp = [0, 1, ∞, ∞, ∞, ∞, ∞]

For amount = 2:
  Try coin 1: dp[2] = min(∞, dp[1] + 1) = min(∞, 1 + 1) = 2
  dp = [0, 1, 2, ∞, ∞, ∞, ∞]

For amount = 3:
  Try coin 1: dp[3] = min(∞, dp[2] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[3] = min(3, dp[0] + 1) = min(3, 0 + 1) = 1
  dp = [0, 1, 2, 1, ∞, ∞, ∞]

For amount = 4:
  Try coin 1: dp[4] = min(∞, dp[3] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[4] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  Try coin 4: dp[4] = min(2, dp[0] + 1) = min(2, 0 + 1) = 1
  dp = [0, 1, 2, 1, 1, ∞, ∞]

For amount = 5:
  Try coin 1: dp[5] = min(∞, dp[4] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[5] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  Try coin 4: dp[5] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, ∞]

For amount = 6:
  Try coin 1: dp[6] = min(∞, dp[5] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[6] = min(3, dp[3] + 1) = min(3, 1 + 1) = 2
  Try coin 4: dp[6] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, 2]

Final result: dp[6] = 2 (achieved with coins 3 + 3)
```

Diese Lösung zeigt beide Schlüsselprinzipien der dynamischen Programmierung. Es gibt überlappende Teilprobleme, weil das Finden der minimalen Münzen für Betrag 6 die Lösungen für Beträge 5, 3 und 2 benötigt. Dieselben Teilprobleme tauchen auch bei der Berechnung anderer Beträge auf. Es gibt optimale Teilstruktur, weil die optimale Lösung für jeden Betrag optimale Lösungen für kleinere Beträge enthält. Wenn wir wissen, dass die minimalen Münzen für Betrag 3 gleich 1 sind, dann ist eine Möglichkeit, Betrag 6 zu erreichen, diese Lösung plus eine weitere Münze im Wert von 3 zu verwenden.

Ohne DP müssten wir jede mögliche Kombination von Münzen ausprobieren – eine exponentielle Anzahl von Möglichkeiten. Mit DP bauen wir die Lösung systematisch auf:

- **Zeitkomplexität**: `O(amount × number of coins)` statt exponentiell.
    
- **Speicherkomplexität**: `O(amount)` für das `dp` Array.
    
- **Keine redundante Arbeit**: Jedes Teilproblem (minimale Münzen für jeden Betrag finden) wird genau einmal gelöst.
    
- **Wiederverwendbare Ergebnisse**: Sobald wir die minimalen Münzen für Betrag 3 kennen, verwenden wir dieses Wissen für alle größeren Beträge, die davon profitieren können.
    

## Wann sollte man dynamische Programmierung verwenden?

Dynamische Programmierung ist effektiv, wenn:

- Das Problem in überlappende Teilprobleme zerlegt werden kann.
    
- Das Problem eine optimale Teilstruktur aufweist.
    
- Eine naive rekursive Lösung wiederholte Berechnungen erfordern würde.
    
- Sie die Zeitkomplexität auf Kosten der Speicherkomplexität optimieren möchten.
    

Gängige Muster der dynamischen Programmierung umfassen Optimierungsprobleme (Minimum-/Maximumwerte finden), Zählprobleme (Anzahl der Möglichkeiten, etwas zu erreichen) und Entscheidungsprobleme, die in kleinere Entscheidungen zerlegt werden können.

Dynamische Programmierung verwandelt komplexe Probleme in handhabbare, indem systematisch Lösungen für Teilprobleme gespeichert und wiederverwendet werden. Das Verständnis dieser Technik öffnet die Tür, eine Vielzahl von rechnerischen Herausforderungen effizient zu lösen.
