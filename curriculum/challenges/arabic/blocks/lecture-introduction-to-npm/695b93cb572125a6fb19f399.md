# --description--

يحتاج npm إلى طريقة لتخزين تفاصيل مهمة عن مشروعك – مثل الحزم التي يستخدمها، وإصدار تلك الحزم، والأوامر التي يمكنه تشغيلها. يفعل ذلك باستخدام ملف `package.json`.

فكر في ملف `package.json` كملف إعدادات npm للمشروع. إنه ملف JSON بسيط يصف مشروعك باستخدام أزواج المفتاح-القيمة لتتبع أشياء مثل:

* اسم المشروع وإصداره
    
* المؤلف
    
* الملف الرئيسي الذي يجب تشغيله أولًا
    
* الترخيص
    
* مستودع الشفرة، الكلمات المفتاحية، والوصف
    
* أي تبعيات يحتاجها المشروع
    

إليك كيف قد يبدو ملف `package.json`:

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "description": "A simple NPM project",
  "main": "index.js",
  "scripts": {
    "start": "node index.js",
    "test": "echo \"Running tests...\""
  },
  "repository": {
    "type": "git",
    "url": "git+https://github.com/username/project.git"
  },
  "keywords": ["fCC", "js"],
  "author": "Kolade Chris",
  "license": "ISC",
  "dependencies": {
    "react": "^19.2.1"
  },
  "devDependencies": {
    "cypress": "^15.7.1"
  }
}
```

غالبًا ما يُعيّن الحقل `main` إلى `index.js`، ويخبر Node.js أي ملف يجب تشغيله أولًا.

يُستخدم حقل `license` لتحديد الشروط القانونية التي يمكن للآخرين بموجبها استخدام الشفرة في حزمتك وتعديلها وتوزيعها. هناك تراخيص مختلفة مثل `MIT`، و`ISC`، و`GPL`، ولكل منها شروط وأذونات مختلفة. الافتراضي هو `ISC`، ترخيص اتحاد أنظمة الإنترنت، وهو ترخيص مفتوح المصدر متساهل جدًا.

مع نمو مشروعك وتثبيت الحزم، يضيف npm هذه الحزم تحت أحد القسمين التاليين:

* `dependencies`، وهي الحزم التي يحتاجها تطبيقك للتشغيل في بيئة الإنتاج
    
* `devDependencies`، وهي الحزم التي تحتاجها فقط أثناء التطوير
    

على سبيل المثال، إذا كان مشروعك يستخدم React لواجهة المستخدم، فتوضع في `dependencies` لأنها ضرورية لعمل المشروع بعد النشر. أما أداة مثل Cypress، التي تُستخدم فقط للاختبار الشامل قبل النشر، فتكون في `devDependencies`.

هناك طريقتان سهلتان لإنشاء ملف `package.json`. الطريقة الأكثر شيوعًا هي تشغيل الأمر التالي:

```bash
npm init
```

يجب تشغيل هذا في المجلد الجذري لمشروعك. يبدأ هذا الأمر عملية إعداد تفاعلية حيث يطرح npm عليك سلسلة من الأسئلة. تُستخدم إجاباتك لملء حقول مثل اسم المشروع، الإصدار، الوصف، السكربتات، الكلمات المفتاحية، المؤلف، والترخيص.

إذا فضلت تخطي الأسئلة وإنشاء ملف `package.json` بقيم افتراضية، يمكنك تشغيل:

```bash
npm init -y
```

أو

```bash
npm init --yes
```

ينشئ هذا على الفور ملف `package.json` بقيم افتراضية في العديد من الحقول.

من الأفضل عادة استخدام `npm init` بدون العلامة `-y` أو `--yes` حتى تتمكن من مراجعة كل حقل وتخصيص معلومات مشروعك. إذا لم تجب على سؤال، يستخدم npm ببساطة قيمة افتراضية. على سبيل المثال، إذا تركت سؤال الترخيص فارغًا، سيعيّن npm القيمة `ISC` افتراضيًا.

قد تلاحظ أن `dependencies` و`devDependencies` غير مدرجتين بعد. ذلك لأن هذه الأقسام تُضاف فقط بعد تثبيت أول حزمة لك.

ستلاحظ أيضًا أن حقل `repository` مفقود. رغم أن npm سأل عنه أثناء `npm init`، إلا أنه لا يملأه تلقائيًا لاحقًا. على سبيل المثال، عند تشغيل أوامر مثل `git remote add origin` أو `git push`، لن يحدث npm حقل المستودع لك — عليك إضافته يدويًا إذا أردت تضمينه.

في الدروس التالية، ستتعلم المزيد عن التبعيات في `package.json` وكيفية تثبيتها وتحديثها وحذفها.
