# --description--

이전 수업에서 `.gitignore` 파일에 대해 간단히 다뤘습니다. 그런데 이 파일이 정확히 무엇이며, 그 안에 무엇을 넣어야 할까요?

`.gitignore` 파일은 Git 작업과 관련된 특별한 유형의 파일입니다. 이름에서 알 수 있듯이, 이 파일은 Git에게 무언가를 무시하라고 지시하는 데 사용되며, 이것이 일반적인 용도입니다. 하지만 실제로 이 파일이 하는 일은 Git에게 파일 추적을 중단하라고 알리는 것입니다. 이 차이가 중요하며, 나중에 더 명확히 설명할 것입니다.

그럼 실제로 이 파일에 무엇을 넣어야 할까요? 커밋하고 싶지 않은 항목들을 나열해야 합니다.

첫 번째는 비밀 파일입니다. 예를 들어, 애플리케이션에서 사용할 API 키를 저장한 `.env` 파일일 수 있습니다. 인증에 필요한 개인 GPG 키 파일일 수도 있습니다. 공개되면 안 되는 민감한 정보가 포함된 모든 것은 `.gitignore`로 무시해야 합니다. 저장소가 비공개라도 공개된 것처럼 행동하세요 — 누구나 볼 수 있다고 생각하고 비밀을 커밋하지 않도록 주의하세요.

지금 우리의 파일이 어떻게 생겼는지 살펴보겠습니다:

```sh
# Secrets
.env
github.key
```

Git에게 비밀을 보관하는 `.env` 파일을 무시하라고 했습니다. 또한 개인 GPG 키가 들어 있는 `github.key` 파일도 무시하라고 했습니다.

다음으로 고려할 것은 프로젝트에서 사용하는 패키지나 의존성입니다. 예를 들어 Node.js 프로젝트를 작업할 때 npm에서 패키지를 설치하는 경우가 많습니다. 이 패키지들은 프로젝트 내 `node_modules` 폴더에 들어가서 코드가 가져와 사용할 수 있습니다.

하지만 `node_modules` 폴더는 너무 커져서 모든 패키지를 커밋하고 업데이트할 때마다 변경 사항을 추적하는 것은 불가능합니다. 대신 `package.json` 파일과 관련된 잠금 파일(e.g. `package-lock.json`, `pnpm-lock.yaml`)을 커밋합니다. 이 파일들은 npm에게 어떤 패키지와 버전을 설치할지 알려주어, 기여자가 저장소에 커밋하지 않고도 동일한 의존성을 설치할 수 있게 합니다.

이제 `node_modules`을 `.gitignore` 파일에 추가해 보겠습니다:

```sh
# Secrets
.env
github.key

# Packages
node_modules
```

이제 `.gitignore` 파일에서 중요한 구문에 대해 살펴보겠습니다. `node_modules` 같은 항목을 추가하면 Git은 해당 패턴과 일치하는 모든 것을 무시합니다. 즉, 프로젝트 루트의 `node_modules` 디렉터리를 무시하지만, 프로젝트 내 `node_modules` 디렉터리의 `client`도 무시합니다.

루트에서만 무시하려면 줄 앞에 슬래시를 붙일 수 있습니다. `node_modules`은 `/node_modules`가 됩니다. 이렇게 하면 루트의 `node_modules` 디렉터리만 무시하고 `client/node_modules` 디렉터리는 무시하지 않습니다. 우리는 그럴 필요가 없으니, 항목을 `node_modules`로 둡니다.

다음으로 무시할 것은 빌드 결과물입니다. 예를 들어 TypeScript 프로젝트를 컴파일하면 JavaScript 출력물이 `dist` 폴더에 생성될 수 있습니다. 이 출력물을 커밋하면 Git 기록이 불필요하게 커지므로 커밋하지 않는 것이 좋습니다. 대신 기여자가 동일한 빌드 명령을 실행해 직접 생성할 수 있도록 무시합니다.

이제 `dist` 폴더를 무시해 보겠습니다:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
```

가끔 잘못된 파일이 TypeScript 컴파일러를 멈추게 하여 JavaScript 출력물이 여기저기 흩어질 수 있습니다. 우리는 TypeScript만 작성하므로 glob 구문을 사용해 모든 JavaScript 파일을 무시해 보겠습니다:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js
```

다른 일반적인 무시 대상은 단위 테스트 커버리지 결과를 담는 `coverage` 디렉터리, Angular를 사용하는 경우 `.angular` 디렉터리, NextJS를 사용하는 경우 `.next` 디렉터리 등이 있습니다.

지금은 여기까지로 충분합니다. 프레임워크가 파일을 생성한다면, 해당 문서에서 `.gitignore`에 무엇을 추가해야 하는지 확인하세요.

네 번째로 무시할 것은 IDE 파일입니다. 예를 들어 Jetbrains 제품은 IDE 전용 설정을 위해 `.idea` 폴더를 사용합니다. Visual Studio Code는 `.vscode` 폴더를 사용합니다.

이들은 보통 무시하는 것이 좋습니다. 로컬 환경에 특화된 파일이어서 다른 기여자에게는 필요 없기 때문입니다(각자 자신의 설정을 사용하므로). 우리는 Visual Studio Code를 IDE로 사용하니 해당 디렉터리를 추가해 보겠습니다.

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
```

하지만 `.vscode` 폴더에는 `settings.json` 파일이 포함되어 있습니다. 이 파일은 작업 공간별 설정을 담고 있어 기여자에게 유용할 수 있습니다 — 확장 기능과 환경을 우리와 비슷하게 구성할 수 있게 해줍니다.

그래서 부정 구문을 사용해 해당 파일을 "무시하지 않도록" 하겠습니다:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json
```

MacOS를 사용한다면 `.DS_STORE` 파일을 익숙하게 느낄 수 있습니다. 이 파일은 Finder가 디렉터리를 어떻게 렌더링할지(예: 썸네일 또는 목록) 결정하는 메타데이터를 담고 있습니다. Windows 사용자라면 디렉터리 썸네일을 캐시하는 숨김 `Thumbs.db` 파일을 볼 수 있습니다.

이 파일들은 개인 설정이므로 프로젝트에 영향을 주지 않아 무시하는 것이 일반적으로 좋습니다. 두 파일을 모두 추가해 보겠습니다:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json

# System Files
.DS_STORE
Thumbs.db
```

이미 커밋한 파일을 무시하는 것에 대해 간단히 설명하겠습니다. 예를 들어 `dist` 디렉터리를 실수로 커밋했는데 이제 삭제하고 무시하려 한다고 합시다. `.gitignore` 파일에 추가하는 것만으로는 이전에 커밋한 내용이 사라지지 않습니다. 이 파일은 Git에게 추적을 중단하라고 알릴 뿐입니다. 따라서 이미 커밋된 파일을 무시하면 변경 사항은 추적하지 않지만 삭제도 추적하지 않습니다.

대신 먼저 파일을 삭제하고 그 삭제를 커밋해야 합니다. 그다음 `.gitignore`에 항목을 추가하고 두 번째 커밋을 만듭니다. 앞으로 해당 파일이 다시 생성되면 Git이 무시하여 커밋에 나타나지 않습니다.

마지막으로, Python 같은 다른 언어로 프로젝트를 작업한다면 무시해야 할 항목이 여기서 다룬 것과 다를 수 있습니다. GitHub는 다양한 언어와 프레임워크용 `.gitignore` 템플릿을 매우 유용하게 관리하고 있습니다. 목록은 <a href="https://github.com/github/gitignore" target="_blank">https://github.com/github/gitignore</a>에서 확인하세요.

이제 저장소에 커밋하지 않을 파일을 피할 준비가 되었습니다!
