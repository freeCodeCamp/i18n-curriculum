# --description--
  
البرمجة الديناميكية هي تقنية خوارزمية تحل المشكلات المعقدة بتقسيمها إلى مشكلات فرعية أبسط وتخزين النتائج لتجنب الحسابات المتكررة. هذه الطريقة تحول المشكلات التي عادةً ما تستغرق وقتًا أُسّيًا إلى مشكلات يمكن حلها في وقت كثير الحدود.

## المبادئ الأساسية للبرمجة الديناميكية

تعمل البرمجة الديناميكية عندما تتوفر شرطان رئيسيان في المشكلة.

- **المشكلات الفرعية المتداخلة**: تظهر نفس المشكلات الصغيرة عدة مرات عند حل المشكلة الأكبر. بدلاً من إعادة حساب هذه المشكلات الفرعية مرارًا، نخزن حلولها.
    
- **البنية المثلى الفرعية**: الحل الأمثل للمشكلة يحتوي على حلول مثلى لمشكلاتها الفرعية. هذا يعني أننا نستطيع بناء أفضل حل بدمج أفضل الحلول للأجزاء الأصغر.
    

دعنا نفحص هذه المفاهيم باستخدام مشكلة "تسلق الدرج" الكلاسيكية.

## المشكلة مع التكرار البسيط

فكر في مشكلة تسلق الدرج: أنت تتسلق سلمًا به `n` درجة ويمكنك تسلق درجة واحدة أو درجتين في كل مرة. كم عدد الطرق المختلفة التي يمكنك بها الوصول إلى القمة؟

```python
def climb_stairs_recursive(n):
    """Recursive approach"""
    if n <= 2:
        return n  # Base cases: 1 way for 1 step, 2 ways for 2 steps
    # To reach step n, we can come from step (n-1) or step (n-2)
    return climb_stairs_recursive(n-1) + climb_stairs_recursive(n-2)
```

هذا التنفيذ له تعقيد زمني أُسّي بسبب الحسابات المتكررة الضخمة. عند حساب `climb_stairs(5)`، يحدث ما يلي:

- `climb_stairs(5)` يستدعي `climb_stairs(4)` و `climb_stairs(3)`
    
- `climb_stairs(4)` يستدعي `climb_stairs(3)` و `climb_stairs(2)`
    
- الآن يتم حساب `climb_stairs(3)` **مرتين**
    
- `climb_stairs(3)` يستدعي `climb_stairs(2)` و `climb_stairs(1)`
    
- يتم حساب `climb_stairs(2)` **3 مرات إجمالاً**
    

لمجرد `n=5`، نجري 9 استدعاءات دالة بينما نحتاج فقط إلى 5 حسابات فريدة. مع زيادة `n`، تتفجر هذه التكرارات أُسّيًا - `climb_stairs(30)` سيحتاج إلى ملايين استدعاءات الدالة. يصبح التعقيد الزمني `O(2^n)`، مما يجعله غير فعال وغير عملي للقيم الأكبر من `n`.

## حلول البرمجة الديناميكية

تزيل البرمجة الديناميكية هذا الحساب المتكرر من خلال نهجين رئيسيين:

### التذكر (النهج من الأعلى إلى الأسفل)

التذكر يخزن نتائج استدعاءات الدوال المكلفة ويُرجع النتيجة المخزنة عند تكرار نفس المُدخلات:

```python
def climb_stairs_memo(n, memo={}):
    """Dynamic programming with memoization"""
    # Check if we've already calculated this value
    if n in memo:
        return memo[n]  # Return cached result - O(1) lookup!
    
    # Base cases
    if n <= 2:
        return n
    
    # Calculate once and store in memo for future use
    memo[n] = climb_stairs_memo(n-1, memo) + climb_stairs_memo(n-2, memo)
    return memo[n]
```

التذكر أكثر كفاءة بكثير لأن كل قيمة فريدة من `1` إلى `n` تُحسب مرة واحدة فقط. عندما نحتاج `climb_stairs(3)` مرة أخرى، بدلاً من إعادة حسابها (مما يؤدي إلى المزيد من الاستدعاءات التكرارية)، ننظر إليها ببساطة في قاموس التذكر في وقت `O(1)`.

دعنا نتتبع تنفيذ `climb_stairs(5)` باستخدام النهج من الأعلى إلى الأسفل لنرى كيف يلغي التذكر العمل المتكرر:

```md
Call: climb_stairs_memo(5)
  memo = {} (empty)
  
  Call: climb_stairs_memo(4) 
    memo = {} (empty)
    
    Call: climb_stairs_memo(3)
      memo = {} (empty)
      
      Call: climb_stairs_memo(2) → returns 2 (base case)
      Call: climb_stairs_memo(1) → returns 1 (base case)
      
      Result: 2 + 1 = 3
      memo = {3: 3} (stored!)
    
    Call: climb_stairs_memo(2) → returns 2 (base case)
    
    Result: 3 + 2 = 5
    memo = {3: 3, 4: 5} (stored!)
  
  Call: climb_stairs_memo(3) → returns 3 (FROM MEMO - no recursion!)
  
  Result: 5 + 3 = 8
  memo = {3: 3, 4: 5, 5: 8}
```

**مقارنة الكفاءة**

- **التكرار البسيط**: يجري 9 استدعاءات دالة مع حسابات متكررة
    
- **التذكر**: يجري فقط 5 حسابات فريدة، ثم يعيد استخدام النتائج المخزنة
    
- **التعقيد الزمني**: انخفض من `O(2^n)` إلى `O(n)` لأننا نجري فقط `n` حسابًا فريدًا
    
- **التعقيد المكاني**: `O(n)` لتخزين التذكر ومكدس الاستدعاءات
    
- **التأثير الحقيقي**: `climb_stairs(30)` ينخفض من ملايين استدعاءات الدالة إلى حوالي 30 حسابًا فرعيًا فريدًا
    

### الجدولة (النهج من الأسفل إلى الأعلى)

الجدولة تبني الحل من الأساس، بملء جدول بحلول المشكلات الفرعية:

```python
def climb_stairs_tabulation(n):
    """Dynamic programming with tabulation"""
    if n <= 2:
        return n
    
    # Create array to store results for all steps from 0 to n
    dp = [0] * (n + 1)
    dp[1] = 1  # 1 way to reach step 1
    dp[2] = 2  # 2 ways to reach step 2
    
    # Build up the solution iteratively
    for i in range(3, n + 1):
        # Ways to reach step i = ways to reach (i-1) + ways to reach (i-2)
        dp[i] = dp[i-1] + dp[i-2]
    
    return dp[n]
```

الجدولة تلغي التكرار تمامًا من خلال بناء الحل بشكل تكراري من أصغر المشكلات الفرعية حتى الهدف.

لنرَ النهج من الأسفل إلى الأعلى في العمل لنرى كيف نبني الحل بشكل منهجي. هذا هو البناء التكراري لـ `climb_stairs(5)`:

```md
Initial state:
dp = [0, 1, 2, 0, 0, 0]
     [0, 1, 2, 3, 4, 5] ← indices (step numbers)

Step by step construction:

i = 3:
  dp[3] = dp[2] + dp[1] = 2 + 1 = 3
  dp = [0, 1, 2, 3, 0, 0]
  
i = 4:
  dp[4] = dp[3] + dp[2] = 3 + 2 = 5
  dp = [0, 1, 2, 3, 5, 0]
  
i = 5:
  dp[5] = dp[4] + dp[3] = 5 + 3 = 8
  dp = [0, 1, 2, 3, 5, 8]

Final result: dp[5] = 8
```

**المزايا الرئيسية للجدولة**

- **لا عبء التكرار**: على عكس التذكر، لا يوجد مكدس استدعاءات تكرارية.
    
- **تنفيذ متوقع**: نحسب القيم بترتيب محدد مسبقًا (1، 2، 3، 4، 5...).
    
- **صديق لذاكرة التخزين المؤقت**: الوصول المتسلسل للمصفوفة يحسن استخدام الذاكرة.
    
- **سهل التحسين**: يمكن تقليل التعقيد المكاني إلى `O(1)` لأننا نحتاج فقط إلى آخر قيمتين.
    

```python
def climb_stairs_optimized(n):
    if n <= 2:
        return n
    
    prev2, prev1 = 1, 2  # Only store last two values
    for i in range(3, n + 1):
        current = prev1 + prev2
        prev2, prev1 = prev1, current
    return prev1
```

**مقارنة الكفاءة**

- **التكرار البسيط**: 9 استدعاءات دالة لـ `n=5`، نمو أُسّي.
    
- **الجدولة**: 3 جمعات بسيطة لـ `n=5`، نمو خطي.
    
- **التعقيد الزمني**: `O(n)` بدلاً من `O(2^n)`.
    
- **التعقيد المكاني**: `O(n)` للمصفوفة، أو `O(1)` مع التحسين.
    
- **أداء متوقع**: لا خطر من تجاوز مكدس الاستدعاءات للمدخلات الكبيرة.
    

كلا النهجين يقللان التعقيد الزمني من أُسّي `O(2^n)` إلى خطي `O(n)`، وهو تحسن كبير يصنع الفرق بين حل المشكلة في أجزاء من الثانية أو انتظار سنوات للمدخلات الأكبر.

## التطبيقات العملية

لدى البرمجة الديناميكية تطبيقات واسعة في علوم الحاسوب وما بعدها:

- **تحسين المسارات**: تستخدم أنظمة GPS خوارزميات البرمجة الديناميكية لإيجاد أقصر المسارات بين المواقع.
    
- **معالجة النصوص**: تعتمد مدققات الإملاء وميزات الإكمال التلقائي غالبًا على البرمجة الديناميكية لحساب مسافات التعديل بين الكلمات.
    
- **النمذجة المالية**: تستخدم استراتيجيات الاستثمار وتحسين معرض الأعمال تقنيات البرمجة الديناميكية بشكل متكرر.
    
- **تخصيص الموارد**: تظهر مشكلة حقيبة الظهر وأنواعها في الجدولة والميزانية وإدارة الموارد.
    

## مثال عملي: مشكلة تبديل العملات

مشكلة تبديل العملات هي تحدٍّ برمجي كلاسيكي يُظهر عند حلها باستخدام البرمجة الديناميكية كلا المبدأين الرئيسيين: البنية المثلى الفرعية والمشكلات الفرعية المتداخلة.

تسأل مشكلة تبديل العملات: "ما هو الحد الأدنى لعدد العملات المطلوبة لصنع مبلغ مستهدف؟"

إليك حلًا واحدًا باستخدام البرمجة الديناميكية:

```python
def min_coins(amount, coins):
    """Find minimum number of coins needed to make the given amount"""
    # Initialize dp array with "infinity" - represents impossible to make
    dp = [float('inf')] * (amount + 1)
    dp[0] = 0  # Base case: 0 coins needed for amount 0
    
    # For each amount from 1 to target amount
    for i in range(1, amount + 1):
        # Try each coin denomination
        for coin in coins:
            if coin <= i:  # Can only use coin if it doesn't exceed current amount
                # Update minimum: current minimum vs (coins for remaining amount + 1)
                dp[i] = min(dp[i], dp[i - coin] + 1)
    
    # Return result if possible, -1 if impossible
    return dp[amount] if dp[amount] != float('inf') else -1

# Example usage:
# coins = [1, 3, 4], amount = 6
# dp[6] = min(dp[5]+1, dp[3]+1, dp[2]+1) = min(3+1, 1+1, 2+1) = 2
# Result: 2 coins (3 + 3)
```

وهنا كيف تعمل خوارزمية تبديل العملات بالبرمجة الديناميكية خطوة بخطوة لـ `coins = [1, 3, 4]`، `amount = 6`:

```md
Initial state:
dp = [0, ∞, ∞, ∞, ∞, ∞, ∞]
     [0, 1, 2, 3, 4, 5, 6] ← amounts

Building up the solution:

For amount = 1:
  Try coin 1: dp[1] = min(∞, dp[0] + 1) = min(∞, 0 + 1) = 1
  dp = [0, 1, ∞, ∞, ∞, ∞, ∞]

For amount = 2:
  Try coin 1: dp[2] = min(∞, dp[1] + 1) = min(∞, 1 + 1) = 2
  dp = [0, 1, 2, ∞, ∞, ∞, ∞]

For amount = 3:
  Try coin 1: dp[3] = min(∞, dp[2] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[3] = min(3, dp[0] + 1) = min(3, 0 + 1) = 1
  dp = [0, 1, 2, 1, ∞, ∞, ∞]

For amount = 4:
  Try coin 1: dp[4] = min(∞, dp[3] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[4] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  Try coin 4: dp[4] = min(2, dp[0] + 1) = min(2, 0 + 1) = 1
  dp = [0, 1, 2, 1, 1, ∞, ∞]

For amount = 5:
  Try coin 1: dp[5] = min(∞, dp[4] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[5] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  Try coin 4: dp[5] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, ∞]

For amount = 6:
  Try coin 1: dp[6] = min(∞, dp[5] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[6] = min(3, dp[3] + 1) = min(3, 1 + 1) = 2
  Try coin 4: dp[6] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, 2]

Final result: dp[6] = 2 (achieved with coins 3 + 3)
```

يُظهر هذا الحل كلا المبدأين الرئيسيين للبرمجة الديناميكية. هناك مشكلات فرعية متداخلة لأن إيجاد الحد الأدنى من العملات للمبلغ 6 يتطلب معرفة الحلول للمبالغ 5، 3، و2. تظهر هذه المشكلات الفرعية نفسها عند حساب مبالغ أخرى. هناك بنية مثلى فرعية لأن الحل الأمثل لأي مبلغ يدمج الحلول المثلى للمبالغ الأصغر. إذا عرفنا أن الحد الأدنى للعملات للمبلغ 3 هو 1، فإن إحدى الطرق لصنع المبلغ 6 هي استخدام هذا الحل بالإضافة إلى عملة واحدة بقيمة 3.

بدون البرمجة الديناميكية، نحتاج إلى تجربة كل تركيبة ممكنة من العملات - عدد أُسّي من الاحتمالات. مع البرمجة الديناميكية، نبني الحل بشكل منهجي:

- **التعقيد الزمني**: `O(amount × number of coins)` بدلاً من أُسّي.
    
- **التعقيد المكاني**: `O(amount)` لمصفوفة `dp`.
    
- **لا عمل متكرر**: تُحل كل مشكلة فرعية (إيجاد الحد الأدنى من العملات لكل مبلغ) مرة واحدة بالضبط.
    
- **إعادة استخدام النتائج**: بمجرد معرفة الحد الأدنى للعملات للمبلغ 3، نستخدم هذه المعرفة لجميع المبالغ الأكبر التي يمكن أن تستفيد منها.
    

## متى تستخدم البرمجة الديناميكية

تكون البرمجة الديناميكية فعالة عندما:

- يمكن تقسيم المشكلة إلى مشكلات فرعية متداخلة.
    
- تظهر المشكلة بنية مثلى فرعية.
    
- يتضمن الحل التكراري البسيط حسابات متكررة.
    
- تحتاج إلى تحسين التعقيد الزمني على حساب التعقيد المكاني.
    

تشمل أنماط البرمجة الديناميكية الشائعة مشكلات التحسين (إيجاد القيم الحد الأدنى/الأقصى)، مشكلات العد (عدد الطرق لتحقيق شيء ما)، ومشكلات القرار التي يمكن تقسيمها إلى قرارات أصغر.

تحول البرمجة الديناميكية المشكلات المعقدة إلى مشكلات يمكن إدارتها بتخزين وإعادة استخدام الحلول للمشكلات الفرعية بشكل منهجي. فهم هذه التقنية يفتح الباب لحل مجموعة واسعة من التحديات الحاسوبية بكفاءة.
