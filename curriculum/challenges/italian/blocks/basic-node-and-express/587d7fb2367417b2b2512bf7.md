# --description--

Oltre a GET, c’è un altro verbo HTTP comune, ed è POST. POST è il metodo predefinito usato per inviare dati dal client con i moduli HTML. Nella convenzione REST, POST serve per inviare dati per creare nuovi elementi nel database (un nuovo utente o un nuovo post del blog). In questo progetto non hai un database, ma imparerai comunque come gestire le richieste POST.

In questo tipo di richieste, i dati non appaiono nell’URL, sono nascosti nel corpo della richiesta. Il corpo è una parte della richiesta HTTP, chiamata anche payload. Anche se i dati non sono visibili nell’URL, questo non significa che siano privati. Per capire perché, guarda il contenuto grezzo di una richiesta HTTP POST:

```http
POST /path/subpath HTTP/1.0
From: john@example.com
User-Agent: someBrowser/1.0
Content-Type: application/x-www-form-urlencoded
Content-Length: 20

name=John+Doe&age=25
```

Come vedi, il corpo è codificato come la stringa di query. Questo è il formato predefinito usato dai moduli HTML. Con Ajax puoi anche usare JSON per gestire dati con una struttura più complessa. Esiste anche un altro tipo di codifica: multipart/form-data. Questa viene usata per caricare file binari. In questo esercizio userai un corpo codificato in URL. Per analizzare i dati provenienti dalle richieste POST, devi usare il pacchetto `body-parser`. Questo pacchetto ti permette di usare una serie di middleware, che possono decodificare dati in formati diversi.

# --instructions--

`body-parser` è già stato installato ed è nel file `package.json` del tuo progetto. `require` all’inizio del file `myApp.js` e salvalo in una variabile chiamata `bodyParser`. Il middleware per gestire dati codificati in URL viene restituito da `bodyParser.urlencoded({extended: false})`. Passa la funzione restituita dalla chiamata al metodo precedente a `app.use()`. Come sempre, il middleware deve essere montato prima di tutte le rotte che ne dipendono.

**Nota:** `extended` è un’opzione di configurazione che indica a `body-parser` quale tipo di parsing usare. Quando `extended=false` usa la codifica classica della libreria `querystring`. Quando `extended=true` usa la libreria `qs` per il parsing.

Quando usi `extended=false`, i valori possono essere solo stringhe o array. L’oggetto restituito usando `querystring` non eredita prototipicamente dal `Object` JavaScript predefinito, il che significa che funzioni come `hasOwnProperty` e `toString` non saranno disponibili. La versione estesa permette più flessibilità nei dati, ma è superata da JSON.
