## 실제 성능과 체감 성능의 차이

- **체감 성능**: 사용자가 웹사이트의 성능을 어떻게 느끼는지를 말합니다. 반응성이나 신뢰성 측면에서 평가하는 주관적인 측정이며, 수치화하기 어렵지만 사용자 경험이 웹사이트의 성공 여부를 결정하기 때문에 매우 중요합니다.
- **실제 성능**: 웹사이트의 객관적이고 측정 가능한 성능입니다. 페이지 로드 시간, 서버 응답 시간, 렌더링 시간 같은 지표로 측정하며, 네트워크와 코드 자체와 관련된 여러 요인의 영향을 받습니다.

## 체감 성능 향상을 위한 기법

- **지연 로딩**: 필수적이지 않은 리소스를 백그라운드에서 로드하여 초기 로드 시간을 최대한 줄이는 기법입니다.
- **폰트 지연 최소화**: 웹사이트에 커스텀 폰트가 있다면, 폰트 로딩 지연을 최소화해야 합니다. 그렇지 않으면 깜빡임 현상이나 커스텀 폰트가 로드되는 동안 대체 폰트가 표시될 수 있습니다. 이를 위해 커스텀 폰트와 비슷한 대체 폰트를 사용하는 것이 좋으며, 이렇게 하면 변화가 더 미묘해집니다.
- **로딩 표시기 사용**: 사용자가 요소를 클릭하자마자 긴 작업에 대해 로딩 표시기를 보여주면, 사용자가 작업에 연결되어 있다고 느끼고 기다리는 시간이 더 짧게 느껴집니다.

## 핵심 성능 개념

- **소스 순서**: HTML 요소가 문서 내에서 구조화된 방식을 말합니다. 무엇이 먼저 로드되는지를 결정하며, 성능과 접근성에 큰 영향을 미칩니다.

소스 순서에 대한 모범 사례는 다음과 같습니다:

- 제목, 내비게이션, 주요 텍스트 같은 중요한 콘텐츠를 HTML 구조 상단에 배치합니다.
- 분석용 스크립트나 서드파티 위젯 같은 필수적이지 않은 스크립트는 렌더링을 차단하지 않도록 지연시킵니다.
- 점진적 향상을 사용하여 스타일과 스크립트가 로드되기 전에도 핵심 경험이 작동하도록 합니다. 점진적 향상은 HTML을 우선으로 페이지를 구축하는 방법입니다.

다음은 방금 살펴본 모범 사례를 적용한 좋은 소스 순서 예시입니다:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **중요 렌더링 경로**: 브라우저가 코드를 화면의 픽셀로 변환하는 일련의 단계입니다.
- **지연 시간**: 브라우저와 서버 간 요청이 왕복하는 데 걸리는 시간입니다. 즉, 지연 시간이 길면 페이지가 느려집니다.

지연 시간을 줄이는 방법은 다음과 같습니다:

- CDN(콘텐츠 전송 네트워크)을 사용해 가까운 위치에서 파일을 제공합니다.
- Gzip 같은 압축을 활성화해 파일 크기를 줄입니다.
- 이미지 최적화와 지연 로딩을 활용합니다.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## INP 개선하기

- **정의**: INP(Interaction to Next Paint)는 사용자가 클릭이나 키 입력 같은 상호작용을 한 시점부터 브라우저가 다음 화면 업데이트를 하는 시점까지 걸리는 시간을 측정해 페이지의 전반적인 반응성을 평가합니다. INP가 낮을수록 반응성이 좋은 페이지입니다.

INP를 개선하는 방법은 다음과 같습니다:

- 긴 JavaScript 작업을 나누어 메인 스레드 작업을 줄입니다.
- `requestIdleCallback()`를 비필수 스크립트에 사용하세요. 이 함수는 브라우저가 유휴 상태일 때 호출됩니다.
- 앞서 다룬 무거운 자산을 지연 로드하거나 미룹니다.
- 이벤트 핸들러를 최적화하세요. 너무 자주 실행되거나 무거운 작업을 수행하면 페이지가 느려지고 INP가 증가합니다. 이를 해결하려면 디바운싱을 사용하세요. 디바운싱은 사용자가 입력을 멈춘 후 일정 시간(예: 300ms) 후에만 함수가 실행되도록 하여 불필요한 계산을 방지하고 성능을 향상합니다.

## 브라우저에서 렌더링 작동 방식

- **렌더링 작동 방식**: 먼저 브라우저가 HTML을 파싱해 DOM을 만듭니다. 다음으로 CSS를 처리해 CSS 객체 모델(CSSOM)을 구성합니다. CSSOM은 요소의 스타일을 결정하는 또 다른 트리 구조입니다. 마지막으로 브라우저가 계산된 스타일과 레이아웃을 기반으로 픽셀을 화면에 그려 렌더링합니다. 복잡한 페이지에서는 여러 레이어를 합성해 최종 시각 출력을 만듭니다.

## 성능이 지속 가능성에 미치는 영향

- **배경 정보**: 인터넷은 전 세계 탄소 배출량의 약 2%를 차지하는데, 이는 항공 산업과 비슷한 수준입니다. 데이터 센터부터 사용자 기기까지 모든 바이트 전송에는 전력이 필요합니다. 큰 파일과 비효율적인 스크립트는 더 많은 전력 소비를 초래합니다. 고성능 웹사이트는 단순히 빠른 것뿐 아니라 불필요한 처리와 에너지 사용을 줄입니다.

## 페이지 로딩 시간 줄이는 방법

- **미디어 자산 최적화**: 큰 이미지와 비디오는 로딩 지연의 주요 원인입니다. 이를 최적화하면 사이트 속도를 크게 높일 수 있습니다. 이미지 압축, WebP 같은 최신 포맷 사용, 지연 로딩 적용 등이 포함됩니다.
- **브라우저 캐싱 활용**: 캐싱은 브라우저가 웹사이트 일부를 로컬에 저장해 재방문 시 로드 시간을 줄입니다.
- **파일 축소 및 압축**: 파일 크기를 줄이면 다운로드 속도가 빨라집니다. 전송 파일 크기 축소와 CSS, JavaScript 파일 축소가 포함됩니다.

## "사용 가능 시간" 개선하기

- **정의**: "사용 가능 시간"은 사용자가 페이지를 요청한 시점부터 의미 있게 상호작용할 수 있을 때까지 걸리는 시간입니다. 이를 개선하려면 자산을 지연 로드하거나 렌더링을 차단하는 리소스를 최소화하세요.

## 성능 측정의 주요 지표

- **첫 번째 콘텐츠풀 페인트(FCP)**: 화면에 첫 번째 콘텐츠(텍스트나 이미지)가 나타나는 속도를 측정합니다. 좋은 FCP는 1.8초 미만, 나쁜 FCP는 3초 이상으로 간주합니다. Chrome 개발자 도구의 성능 탭에서 FCP를 확인할 수 있습니다.
- **총 차단 시간(TBT)**: 무거운 JavaScript 작업으로 메인 스레드가 차단된 시간을 보여줍니다. TBT가 높으면 사용자가 느리게 반응하는 경험을 합니다. TBT를 개선하려면 긴 작업을 나누고 비필수 스크립트를 미루세요.
- **이탈률**: 상호작용 없이 사이트를 떠나는 방문자의 비율입니다. 이탈률이 높으면 페이지가 너무 느릴 수 있습니다.
- **고유 사용자 수**: 사이트를 방문한 개별 방문자 수를 추적합니다. 이탈률과 고유 사용자 수는 Google Analytics에서 확인할 수 있으며, 이를 통해 참여도를 높일 수 있습니다.

## 일반적인 성능 측정 도구

- **Chrome 개발자 도구**: Google Chrome 내장 도구로 실시간으로 성능을 분석하고 디버그할 수 있습니다. 로딩 시간, CPU 사용량, 렌더 지연을 보여주며, 특히 FCP 측정에 유용합니다. 웹사이트가 느리다면 병목 현상을 찾는 데 도움이 됩니다.
- **Lighthouse**: 성능, SEO, 접근성을 자동으로 검사하는 도구입니다.
- **WebPageTest**: 다양한 위치와 기기에서 사이트 로딩을 테스트할 수 있습니다. 사이트의 속도 지수, 총 차단 시간 등 주요 성능 지표를 상세히 제공합니다. 전 세계 실제 사용자가 사이트를 어떻게 경험하는지 알고 싶다면 이 도구가 적합합니다.
- **PageSpeed Insights**: 모바일과 데스크톱 모두에 대해 웹사이트를 분석하고 빠른 개선점을 제안합니다. 이미지 최적화, 렌더링 차단 스크립트 제거, 서버 응답 시간 단축 같은 구체적 권고를 제공합니다. Google이 사이트 성능을 어떻게 보는지 빠르고 쉽게 확인할 수 있습니다.
- **실제 사용자 모니터링(RUM)**: 실제 사용자 행동을 추적해 사이트 경험을 보여줍니다. Google Analytics는 페이지 로드 시간과 이탈률을 추적하며, New Relic이나 Datadog은 실시간 성능 문제를 모니터링합니다. 실제 사용자 데이터를 원한다면 RUM 도구가 필수입니다.

## 성능 웹 API 활용하기

- **정의**: 성능 웹 API는 개발자가 코드 내에서 웹페이지 로딩과 응답 효율을 직접 추적할 수 있게 합니다. 페이지 로드 시간, 렌더링 및 상호작용 지연, JavaScript 실행 시간을 측정할 수 있습니다.
- **`performance.now()`**: 이 API는 사이트의 각 부분이 로드되는 데 걸리는 시간을 밀리초 단위로 정밀하게 측정할 수 있는 타임스탬프를 제공합니다.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: DNS 조회부터 `DOMContentLoaded`까지 페이지 로딩의 모든 단계를 세분화해 보여줍니다.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: 레이아웃 이동, 긴 작업, 사용자 상호작용 같은 성능 이벤트를 감지하는 API입니다.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## CSS 성능 향상 기법

- **CSS 애니메이션**: 크기, 위치, 레이아웃 같은 특정 CSS 속성을 애니메이션하면 브라우저가 "리플로우" 과정을 거쳐 요소 위치와 기하학을 다시 계산합니다. 이 과정은 다시 칠하기를 요구하며 계산 비용이 큽니다. 따라서 CSS 애니메이션 수를 최대한 줄이거나, 최소한 사용자가 켜고 끌 수 있는 옵션을 제공하는 것이 좋습니다.

## JavaScript 성능 향상 기법

- **코드 분할**: JavaScript 코드를 핵심 작업과 비핵심 작업 모듈로 나누는 것도 도움이 됩니다. 이렇게 하면 핵심 모듈을 최대한 빨리 프리로드하고, 비핵심 모듈은 미뤄 페이지 렌더링 속도를 높일 수 있습니다.
- **DOM 조작**: DOM 조작은 JavaScript로 문서 객체 모델(DOM)을 동적으로 변경하는 과정입니다. DOM 조작은 계산 비용이 크므로, JavaScript 코드에서 DOM 조작을 줄이면 성능이 향상됩니다.
