# --description--

npm으로 의존성을 설치할 때, 특히 기존 프로젝트에서 npm은 `package.json` 파일을 확인하여 프로젝트에 필요한 패키지를 파악합니다.

`package.json`의 각 의존성에는 다음과 같은 버전 번호가 포함되어 있습니다:

```bash
"lodash": "^4.17.21"
```

앞서 배운 것처럼, **캐럿(^)**은 npm이 `4.18.0` 또는 `4.17.22` 같은 최신 마이너 및 패치 버전을 설치할 수 있게 해줍니다. 이런 유연성은 유용하지만 문제를 일으킬 수도 있습니다.

두 개발자가 서로 다른 시점에 `npm install`를 실행하면, npm은 같은 패키지라도 약간 다른 버전을 설치할 수 있습니다. 작은 차이도 버그나 예상치 못한 동작을 초래할 수 있습니다.

`package-lock.json` 파일이 이 문제를 해결합니다. 이 파일은 `npm install`를 처음 실행할 때 자동으로 생성되며, 설치된 모든 의존성의 정확한 버전을 고정합니다—여기에는 의존성의 의존성인 모든 하위 의존성도 포함됩니다.

이렇게 하면 프로젝트에 참여하는 모든 사람이 매번 같은 버전의 패키지를 받게 됩니다.

다음은 `package.json`를 의존성으로 가진 간단한 `lodash` 파일입니다:

```bash
{
  "name": "package-lock-json",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC",
  "type": "commonjs",
  "dependencies": {
    "lodash": "^4.17.21"
  }
}
```

그리고 `package-lock.json`에서 일치하는 섹션은 다음과 같습니다:

```bash
"node_modules/lodash": {
  "version": "4.17.21",
  "resolved": "https://registry.npmjs.org/lodash/-/lodash-4.17.21.tgz",
  "integrity": "sha512-v2kDEe57..."
}
```

차이점을 확인하세요:

* `package.json`에서는 `lodash`가 `^4.17.21`로 표시되어 있는데, 이는 유연한 규칙입니다.
    
* `package-lock.json`에서는 `lodash`가 `4.17.21` 버전으로 고정되어 있습니다.
    

누군가 `npm install`를 언제 실행하든, 정확히 그 버전을 받게 됩니다.

이렇게 해서 누군가 `npm install`를 언제 실행하든, Lodash의 정확히 같은 버전을 매번 받게 됩니다.

이것은 프로젝트에서 사용하는 의존성의 의존성인 하위 의존성에도 적용된다는 점을 유의하세요.

예를 들어, Express를 사용하는 프로젝트의 `package-lock.json` 파일 일부는 다음과 같습니다:

```json
{
  "name": "package-lock-json-file",
  "version": "1.0.0",
  "lockfileVersion": 3,
  "requires": true,
  "packages": {
    "": {
      "name": "package-lock-json-file",
      "version": "1.0.0",
      "license": "ISC",
      "dependencies": {
        "express": "^5.2.1"
      }
    },
    ...
    "node_modules/etag": {
      "version": "1.8.1",
      "resolved": "https://registry.npmjs.org/etag/-/etag-1.8.1.tgz",
      "integrity": "sha512-aIL5Fx7mawVa300al2BnEE4iNvo1qETxLrPI/o05L7z6go7fCw1J6EQmbK4FmJ2AS7kgVF/KEZWufBfdClMcPg==",
      "license": "MIT",
      "engines": {
        "node": ">= 0.6"
      }
    },
    "node_modules/express": {
      "version": "5.2.1",
      "resolved": "https://registry.npmjs.org/express/-/express-5.2.1.tgz",
      "integrity": "sha512-hIS4idWWai69NezIdRt2xFVofaF4j+6INOpJlVOLDO8zXGpUVEVzIYk12UUi2JzjEzWL3IOAxcTubgz9Po0yXw==",
      "license": "MIT",
      "dependencies": {
        "accepts": "^2.0.0",
        "body-parser": "^2.2.1",
        "content-disposition": "^1.0.0",
        "content-type": "^1.0.5",
        "cookie": "^0.7.1",
        "cookie-signature": "^1.2.1",
        "debug": "^4.4.0",
        "depd": "^2.0.0",

        "mime-types": "^3.0.0",
        "on-finished": "^2.4.1",

      },

  }
}
```

Express 자체 외에도 `accepts`, `mime-types`, `negotiator` 같은 다른 패키지가 있는 것을 볼 수 있습니다. 이들은 하위 의존성으로, Express가 내부적으로 작동하기 위해 의존하는 패키지들입니다.

또한 `dependencies`의 `node_modules/express`는 범위에 캐럿 문자를 사용하지만, `node_modules/etags` 같은 각 하위 의존성 항목은 특정 버전으로 고정되어 있다는 점을 확인하세요.

요약하면, `package-lock.json` 파일은 모든 개발자, CI 서버, 배포 및 기타 환경에서 일관성을 유지하도록 도와줍니다.

항상 Git에 `package-lock.json` 파일을 커밋하여 팀이 같은 설정을 공유하도록 하세요. 또한 `package-lock.json` 파일을 수동으로 편집하지 마세요—의존성이 변경될 때마다 npm이 자동으로 업데이트합니다.
