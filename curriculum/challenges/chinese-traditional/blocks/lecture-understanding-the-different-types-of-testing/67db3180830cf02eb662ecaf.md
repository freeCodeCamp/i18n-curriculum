# --description--

你在之前的課程中首次接觸過斷言和單元測試。但作為複習，單元測試是指你獨立檢查各個函式（程式）或組件的輸出，與測試整個程式的最終輸出形成對比。斷言用於測試程式碼是否按預期運作。

在本課程中，我們將更深入地了解單元測試中的斷言，並探討一些最佳實踐。

一個常用的斷言函式庫是 Chai。這是 freeCodeCamp 程式碼庫中使用的相同函式庫。每當你參加工作坊或實驗室時，都會有一個自動化測試套件對你的程式碼進行測試，以確保你的程式碼表現如預期。

讓我們來看看幾個來自 Full-Stack Developer Curriculum 的範例。

在 Build a Calculator 工作坊中，第 1 步是測試一個名為 `addTwoAndSeven` 的函式（程式）。

```js
assert.isFunction(addTwoAndSeven);
```

該斷言會檢查所提供的程式碼是否包含名為 `addTwoAndSeven` 的函式。如果使用者創建一個陣列、字串或任何其他非函式的東西並將其指定給 `addTwoAndSeven`，則測試將失敗。

讓我們來看看工作坊後面另一個斷言：

```js
assert.equal(addThreeAndFour(), 7);
```

此斷言會檢查從 `addThreeAndFour` 函式傳回的返回值是否相等於數字 `7`。如果返回值不相等於 `7`，則測試將失敗。

Chai 內建了更多的斷言，例如檢查值是否為布林值、物件是否具有特定屬性名稱，或值是否為 `null`。其他常見的 JavaScript 斷言函式庫有 should.js 和 expect.js。

無論你使用哪一個斷言函式庫，都有一些最佳實務需要注意。第一點是撰寫清晰的 assert 訊息。清晰且有意義的 assert 訊息應該描述其背後的意圖。否則，將難以定位哪些測試失敗。

另一個最佳做法是撰寫具描述性的失敗訊息。具描述性的失敗訊息應該說明哪個條件為真。這將幫助你找出為何特定的 assert 失敗。

當你持續學習寫程式時，熟悉測試並開始為你的個人專案撰寫測試非常重要。這樣做將大大幫助你為第一份開發人員工作做好準備。
