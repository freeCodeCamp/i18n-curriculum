# --description--

En la clase anterior, aprendiste cómo un servidor backend identifica a un usuario durante el inicio de sesión. Pero, ¿qué pasa después de que inician sesión? ¿Cómo recuerda el servidor quiénes son cuando navegan a una nueva página? Aquí es donde entran los JSON Web Tokens (JWT).

En términos sencillos, un JWT es una tarjeta digital que el servidor emite a un usuario después de un inicio de sesión exitoso. El usuario presenta esta tarjeta al servidor en cada solicitud posterior para probar su identidad sin tener que volver a ingresar su contraseña.

En lugar de que el servidor recuerde a cada usuario activo en su propia memoria, traslada la responsabilidad al cliente usando un proceso simple de cuatro pasos:

1. **Inicio de sesión:** El usuario envía su nombre de usuario y contraseña, u otro método de inicio de sesión, al servidor.

2. **Emisión:** El servidor verifica las credenciales contra las que se almacenaron cuando se registraron. Si son correctas, el servidor genera un JWT y se lo envía al usuario.

3. **Almacenamiento:** El cliente recibe el JWT y lo almacena, típicamente en una cookie del navegador o en el almacenamiento local.

4. **Solicitud:** Para cada solicitud futura a una ruta protegida, el cliente envía automáticamente el JWT. El servidor verifica el token y concede acceso.

Un JWT parece una cadena larga y caótica de caracteres. Consiste en tres partes distintas separadas por dos puntos, así:

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

La primera parte es el **encabezado**, la segunda es la **carga útil**, y la tercera parte es la **firma**.

El **encabezado** es un objeto JSON codificado que le dice al servidor qué tipo de token es (JWT) y qué algoritmo de hash se usó para asegurar el token, como HMAC SHA256.

La **carga útil** es la parte central del token. Contiene los datos reales, o "claims" sobre el usuario. Aquí se almacena información como el id del usuario, nombre de usuario o el rol de la cuenta, como admin o user.

Una nota importante de seguridad es que el encabezado y la carga útil solo están codificados (Base64), no encriptados. Esto significa que cualquiera puede decodificarlos fácilmente y leer los datos dentro. Por eso, nunca almacenes datos sensibles como contraseñas en texto plano o números de tarjeta de crédito dentro de la carga útil de un JWT.

La **firma** es la parte final y más crucial del token. El servidor toma el encabezado codificado, la carga útil codificada y los mezcla con una clave secreta conocida solo por el servidor. Luego los hashea para crear la firma.

La firma previene la manipulación. Si un usuario intenta decodificar su token, cambiar su rol de "user" a "admin" y volver a codificarlo, la firma ya no coincidirá con los datos alterados.

Cuando el servidor recibe el token modificado, intenta recrear la firma usando su clave secreta. Como la carga útil cambió, la firma calculada no coincidirá con la firma del token, y el servidor rechazará instantáneamente la solicitud con un código de estado 401 Unauthorized.

En resumen, los JWT permiten autenticación sin estado. El servidor no necesita almacenar datos de sesión en una base de datos, simplemente confía en el token porque la firma prueba que el servidor mismo lo creó.
