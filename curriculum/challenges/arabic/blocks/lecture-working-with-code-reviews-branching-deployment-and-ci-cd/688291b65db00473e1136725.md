# --description--

تشير CI إلى التكامل المستمر، وCD تشير إلى التسليم المستمر أو النشر المستمر. معًا، تتيح هذه العمليات نشر التغييرات على تطبيقك تلقائيًا بعد التأكد من أن تلك التغييرات تعمل بشكل صحيح.

في خط أنابيب CI، قد تُشغّل أداة التحقق من الشفرة والاختبارات على الشفرة. عادةً ما تُشغّل خطوط أنابيب CI على طلبات السحب لضمان أن التغييرات التي أُجريت لا تُسبب تعطل التطبيق، وأيضًا على فرع `main` لضمان جاهزيته للنشر.

قد يُشغّل خط أنابيب CD أيضًا على طلبات السحب، إذا أعددت بيئة معاينة لتتمكن من مشاهدة التغييرات مباشرة. لكنك ستشغّله أيضًا على فرع `main` لنشر التغييرات باستمرار بمجرد قبولها ودمجها.

لكن كيف تعمل هذه الأمور فعليًا مع GitHub Actions؟ أولًا، نحتاج لفهم ما هي GitHub Actions. توفر GitHub Actions مشغلات مؤقتة (فكر فيها كحاسوب مؤقت) تتيح لك تنفيذ أوامر محددة.

تُكوّن Actions باستخدام ملف YAML. يمكنك أن تملك ملفات متعددة لتحديد إجراءات مختلفة للتشغيل. على سبيل المثال، قد يكون لديك ملف واحد لـ CI وآخر لـ CD.

لنلقِ نظرة على مثال أساسي لـ CI:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  lint:
    name: Lint and Test
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Source Files
        uses: actions/checkout@v4

      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10

      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

هذا كثير، فلنقسّمه. ننظر إلى القسم الأول:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```

تعرف الخاصية `name` اسم الإجراء نفسه. يظهر هذا في مناطق مثل قسم فحوصات الحالة في طلب السحب. تحدد خاصية `on` ما يُسمى "محفزات سير العمل" - وهي الأحداث التي ستُشغّل الإجراء الخاص بك.

في هذا المثال، نشغّل الإجراء عند دفع الالتزامات إلى `main`، وعندما يستهدف طلب السحب فرع `main`. هذا يعني أن CI يعمل عندما يُنشأ طلب سحب، ومرة أخرى عند دمجه.

قسم `jobs` هو المكان الذي تحدد فيه ما يفعله الإجراء فعليًا. المفتاح `lint:` هو مفتاح عشوائي يحدد الاسم الداخلي لذلك العمل. خاصية `name` تحدد الاسم الخارجي لهذا العمل بالذات. و`runs-on` تحدد نوع بيئة العمل التي يجب أن يُشغّل فيها الإجراء - في هذه الحالة، نشغّله على أحدث إصدار مدعوم من أوبونتو.

خاصية `steps` هي حيث تحدث السحر. يحدد هذا القسم الخطوات الفعلية التي يجب أن يتخذها الإجراء. لنلقِ نظرة على أول قسم هناك:

```yml
      - name: Checkout Source Files
        uses: actions/checkout@v4
```

تحتوي هذه الخطوة على خاصيتين، `name` التي يجب أن تحملها الخطوة، و`uses` خاصية. خاصية `uses` هذه مميزة، إذ تخبر مشغل الإجراء بالذهاب لجلب إجراء عام من GitHub وتشغيله، بدلاً من تنفيذ أمر. إجراء `actions/checkout` يتولى استنساخ المستودع وإعداده داخل المشغل الخاص بك.

```yml
      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10
```

القسمان التاليان يستخدمان أيضًا إجراءات طرف ثالث. هذه الخطوات تُعد Node ومدير الحزم pnpm. كلاهما يحتوي على مفتاح `with`، يمكنك استخدامه لتمرير معلمات لتلك الإجراءات الخارجية. فكر في هذا كتمرير معلمات إلى دالة.

الأربعة أقسام الأخيرة هي حيث يحدث السحر حقًا.

```yml
      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

لدينا قسم `Install Dependencies` الذي يثبت حزم npm. قسم `Lint Source Files` يشغّل أداة التحقق من الشفرة، قسم `Verify Build` يتأكد من أن شفرة TypeScript يمكن تجميعها، وقسم `Run Tests` يشغّل اختبارات الوحدة.

الأمر المهم الذي يجب معرفته عن الإجراءات هو أنه، افتراضيًا، إذا فشلت إحدى هذه الخطوات، تُتخطى الخطوات المتبقية ويُعلَن تشغيل العملية كفشل.

أين تضع ملفات YAML الخاصة بك لتشغيل هذه الإجراءات؟ يجب أن توضع في مجلد `.github/workflows` داخل مستودعك حتى يجدها GitHub ويستخدمها.

هناك الكثير من الأشياء الإضافية التي يمكن للإجراءات فعلها، وأشجعك على قراءة الوثائق واستكشاف الإجراءات الموجودة في مشاريع مثل مستودع freeCodeCamp. لكن الآن، يجب أن تكون لديك فهم جيد لأساسيات GitHub Actions وCI/CD!
