## رموز حالة استجابة HTTP

- **رموز الحالة** هي رسائل رقمية قصيرة تخبر العميل بما حدث بعد إرسال الطلب (مثل 200، 404، أو 500).

### 1xx — معلوماتية

- **100 Continue** — يخبر الخادم العميل بإرسال بقية الطلب.
- **101 Switching Protocols** — يقوم الخادم بتبديل البروتوكولات (يُرى عادة مع WebSockets).

### 2xx — نجاح

- **200 OK** — نجح الطلب.
- **201 Created** — تم إنشاء مورد جديد بنجاح.
- **204 No Content** — نجح الطلب لكن لا يوجد جسم استجابة (يُستخدم غالبًا مع `DELETE`).

### 3xx — إعادة توجيه

- **301 Moved Permanently** — للمورد عنوان URL جديد دائم.
- **302 Found** — المورد مؤقتًا في عنوان URL مختلف.
- **304 Not Modified** — يمكن للعميل استخدام النسخة المخزنة مؤقتًا؛ لم يتغير شيء.

### 4xx — أخطاء العميل

- **400 Bad Request** — الطلب غير صحيح أو غير صالح.
- **401 Unauthorized** — مطلوب توثيق.
- **403 Forbidden** — مُوثَّق، لكن لا يملك الإذن.
- **404 Not Found** — المورد المطلوب غير موجود.

### 5xx — أخطاء الخادم

- **500 Internal Server Error** — فشل عام في جانب الخادم.
- **502 Bad Gateway** — استلم الوكيل استجابة غير صالحة من خادم آخر.
- **503 Service Unavailable** — الخادم محمّل مؤقتًا أو متوقف.

### استخدام رموز الحالة في Express

- `res.status(400).send("Bad Request")` — يرسل رمز 400 مع رسالة نصية عادية.
- `res.status(200).json({ message: "Success" })` — يرسل رمز 200 مع جسم JSON.
- `res.status(500).send("Internal Server Error")` — يرسل رمز 500 مع رسالة.

## معالجة الأخطاء في Express

- **برمجيات وسيطة لمعالجة الأخطاء** لها أربعة معلمات: `(err, req, res, next)`. وجود أربعة معلمات يشير إلى Express أن هذه برمجيات وسيطة لمعالجة الأخطاء.
- ضع برمجيات معالجة الأخطاء **في نهاية** مكدس البرمجيات الوسيطة، بعد جميع المسارات.

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### معالجة الأخطاء غير المتزامنة في Express 4

- الأخطاء غير المتزامنة **لا تُلتقط تلقائيًا** في Express 4.
- لفّ المنطق غير المتزامن بـ `try/catch` واستدعِ `next(err)` لتمرير الخطأ إلى معالج الأخطاء.

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- يغنيك غلاف `asyncHandler` عن تكرار `try/catch` في كل مسار:

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### معالجة الأخطاء غير المتزامنة في Express 5

- في Express 5، تُمرر الأخطاء غير المتزامنة والوعود المرفوضة **تلقائيًا** إلى معالج الأخطاء — لا حاجة لاستدعاء `next(err)` يدويًا.

### معالجة أخطاء 404

- أضف برمجيات وسيطة شاملة **بعد المسارات وقبل معالج الأخطاء** لمعالجة المسارات غير المطابقة:

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### أفضل الممارسات في معالجة الأخطاء

- ضع دائمًا برمجيات معالجة الأخطاء في النهاية.
- تعامل مع أخطاء 404 بشكل منفصل عن الأخطاء الأخرى.
- في بيئة الإنتاج، سجّل تفاصيل الأخطاء داخليًا — لا ترسل تتبع المكدس للمستخدمين.

## تصحيح الأخطاء وتسجيلها في Express

- **التصحيح المدمج**: عيّن متغير البيئة `DEBUG` لتمكين إخراج التصحيح المدمج في Express.
  - `DEBUG=express:* node index.js` — يفعّل جميع رسائل تصحيح Express.
  - `DEBUG=express:router node index.js` — يعرض فقط إخراج تصحيح الموجِّه.
  - PowerShell: `$env:DEBUG = "express:*"; node index.js`
  - CMD: `set DEBUG=express:* && node index.js`
- **`morgan`**: برمجيات وسيطة شهيرة لتسجيل طلبات HTTP (الطريقة، URL، رمز الحالة، زمن الاستجابة).

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- **تصحيح الأخطاء** يتتبع كيفية تعامل Express مع الطلبات والبرمجيات الوسيطة دون إرباك سجلات الإنتاج.
- **التسجيل** يسجل ما يحدث عبر الطلبات — مفيد لاكتشاف الأخطاء وفهم حركة المرور.

## فحوصات الصحة والإيقاف السلس

### فحوصات الصحة

- **مسار فحص الصحة** يبلغ ما إذا كان التطبيق يعمل بشكل صحيح.
- يقوم موازنات التحميل ومنظمو العمليات (مثل Kubernetes، AWS Elastic Beanstalk) باستدعاء هذا المسار لتحديد صحة الخدمة.
- استجابة **200** تعني أن الخدمة سليمة؛ عدم الاستجابة يؤدي إلى إعادة تشغيل تلقائية.

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### الإيقاف السلس

- يتيح **الإيقاف السلس** للتطبيق إكمال الطلبات الحالية قبل التوقف، بدلاً من قطع الاتصالات فورًا.
- استمع لإشارة `SIGTERM` (المرسلة عند وجوب إنهاء العملية) واستدعِ `server.close()`:

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- استمع لإشارة `SIGINT` (المرسلة عند ضغط المطوِّر Ctrl+C أثناء التطوير المحلي) واستدعِ نفس منطق الإيقاف كما في `SIGTERM`.

### تعليم التطبيق بأنه غير سليم قبل الإيقاف

- أرجع رمز **503** من فحص الصحة أثناء الإيقاف ليوقف موازن التحميل توجيه حركة مرور جديدة إلى التطبيق.

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
