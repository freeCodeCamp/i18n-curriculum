## 서버와 클라이언트-서버 모델

- **서버**: 다른 컴퓨터로부터 요청을 받고 네트워크를 통해 데이터나 서비스를 다시 보내는 컴퓨터나 프로그램입니다. 이 용어는 물리적 하드웨어와 그 위에서 실행되는 소프트웨어 모두를 가리킵니다.
- **클라이언트-서버 모델**: 클라이언트가 요청을 보내고 서버가 그 요청을 처리하여 응답을 반환하는 분산 아키텍처입니다.
- **클라이언트**: 서버에 요청을 시작하는 프로세스나 애플리케이션(예: 웹 브라우저)입니다.
- **요청-응답 주기**: 클라이언트가 요청을 보내면 서버가 이를 처리하고 응답을 반환합니다.

### 서버 유형

- **웹 서버**: 브라우저에 웹 콘텐츠(HTML, 이미지, CSS)를 전달합니다.
- **애플리케이션 서버**: 비즈니스 로직을 처리하고 사용자 입력을 처리합니다.
- **데이터베이스 서버**: 데이터를 저장, 검색, 관리합니다.
- **메일 서버**: 이메일을 보내고 받고 저장합니다.
- **파일 서버**: 네트워크를 통해 파일 저장과 공유를 관리합니다.
- **프록시 서버**: 클라이언트와 다른 서버 사이에서 중개자 역할을 합니다.

## DNS (도메인 이름 시스템)

- **DNS**: 사람이 읽을 수 있는 도메인 이름을 IP 주소로 변환하는 계층적이고 분산된 명명 시스템으로, 인터넷의 "전화번호부"라고도 불립니다.
- **IP 주소**: 네트워크상의 장치를 고유하게 식별하는 숫자 식별자입니다(예: IPv4의 `127.0.0.1`, IPv6의 `0:0:0:0:0:0:0:1`).
- **도메인 이름 계층 구조** (오른쪽에서 왼쪽으로 읽음):
  - **TLD (최상위 도메인)**: 가장 오른쪽 부분(예: `.org`, `.com`).
  - **SLD (2차 도메인)**: 등록된 이름(예: `freecodecamp` in `freecodecamp.org`).
  - **서브도메인**: 메인 도메인의 특정 섹션을 식별(예: `www`, `blog`, `api`).

### DNS 해석 과정

1. 브라우저가 도메인 이름을 **재귀적 해석기**(보통 ISP가 관리)에 보냅니다.
2. 재귀적 해석기가 **루트 서버**에 질의하여 관련 TLD 서버로 안내받습니다.
3. **TLD 서버**가 도메인의 **권한 있는 이름 서버** 주소를 반환합니다.
4. 권한 있는 이름 서버가 IP 주소를 포함한 **DNS 레코드**를 반환합니다.
5. 재귀적 해석기가 클라이언트에 IP 주소를 전달하고, 향후 조회 속도를 위해 레코드를 **캐시**합니다.

## TCP/IP

- **프로토콜**: 네트워크 상에서 장치들이 통신할 수 있도록 하는 규칙과 표준의 집합입니다.
- **TCP/IP**: 데이터가 어떻게 패키징되고, 전송되고, 라우팅되고, 수신되는지를 결정하는 프로토콜 모음입니다.
  - **IP (인터넷 프로토콜)**: IP 주소를 사용해 출발지에서 목적지로 데이터 패킷을 라우팅합니다. 연결 지향적이지 않아 각 패킷이 독립적으로 전달되며 전달 보장이 없습니다.
  - **TCP (전송 제어 프로토콜)**: 데이터를 번호가 매겨진 세그먼트로 나누고 목적지에서 재조립하며, 오류와 손실된 패킷을 검사하고 필요 시 재전송을 요청합니다. 신뢰성 있고 순서가 보장된 전달을 보장합니다.

## HTTP (하이퍼텍스트 전송 프로토콜)

- **HTTP**: 웹에서 클라이언트와 서버 간 통신 규칙을 정의하는 애플리케이션 계층 프로토콜입니다.

- **HTTPS**: 데이터를 전송 전에 SSL/TLS로 암호화하는 HTTP의 보안 버전입니다.

### HTTP 요청 구조

- **요청 라인**: `METHOD /path HTTP/version` (예: `GET / HTTP/1.1`)
- **헤더**: 키-값 형태의 메타데이터 쌍(예: `Content-Type: application/json`)
- **본문** (선택적): 서버로 보내는 데이터로, `POST`, `PUT`, `PATCH`와 함께 사용됩니다.

**일반적인 HTTP 메서드:**

| 메서드 | 목적 |
| --- | --- |
| `GET` | 데이터 조회 |
| `POST` | 새 데이터 제출 |
| `PUT` | 리소스 교체 |
| `PATCH` | 리소스 부분 업데이트 |
| `DELETE` | 리소스 삭제 |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### HTTP 응답 구조

- **상태 라인**: HTTP 버전, 상태 코드, 이유 구문(예: `HTTP/1.1 200 OK`)
- **헤더**: 응답에 관한 메타데이터(예: `Content-Type: text/html`)
- **본문** (선택적): 반환된 콘텐츠나 오류 설명

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## HTTP 상태 코드

상태 코드는 첫 번째 숫자에 따라 그룹화됩니다:

- **1xx** — 정보성 응답
- **2xx** — 성공 응답
- **3xx** — 리다이렉션 메시지
- **4xx** — 클라이언트 오류 응답
- **5xx** — 서버 오류 응답

### 주요 코드

| 코드 | 이름 | 의미 |
| --- | --- | --- |
| `200` | OK | 요청 성공; 콘텐츠 반환 |
| `201` | Created | 리소스가 성공적으로 생성됨 |
| `204` | No Content | 성공했으나 반환할 콘텐츠 없음 |
| `301` | Moved Permanently | 리소스가 새 URL로 영구 이동됨 |
| `302` | Found | 리소스가 임시로 다른 URL로 이동됨 |
| `304` | Not Modified | 리소스가 마지막 요청 이후 변경되지 않음; 캐시된 버전 사용 |
| `400` | Bad Request | 서버가 요청을 이해하지 못함(잘못된 구문 또는 데이터) |
| `401` | Unauthorized | 인증 자격 증명 필요 |
| `403` | Forbidden | 유효한 자격 증명에도 불구하고 클라이언트가 리소스에 대한 권한 없음 |
| `404` | Not Found | 서버에 리소스가 존재하지 않음 |
| `500` | Internal Server Error | 서버에서 예상치 못한 오류 발생 |
| `502` | Bad Gateway | 서버가 상위 서버로부터 잘못된 응답을 받음 |
| `503` | Service Unavailable | 서버가 일시적으로 요청을 처리할 수 없음 |

## HTTP 요청-응답 모델

요청-응답 주기는 다음 구성 요소가 함께 작동합니다:

- **클라이언트**: HTTP 요청을 보내는 브라우저, 모바일 앱, 또는 API 클라이언트
- **HTTP 메서드**: 수행할 작업(`GET`, `POST`, `PUT`, `PATCH`, `DELETE`)
- **URL**: 요청하는 리소스의 주소
- **헤더**: 요청 또는 응답에 대한 메타데이터(예: `Authorization`, `Content-Type`)
- **본문**: `POST`, `PUT`, `PATCH`, `DELETE` 요청과 함께 보내는 데이터(주로 JSON 또는 폼 데이터)
- **서버**: 요청을 처리하고 응답을 반환하는 백엔드 시스템
- **상태 코드**: 성공, 오류, 또는 리다이렉트를 나타내는 숫자 코드
- **응답 본문**: 실제 반환되는 데이터로, 보통 HTML, JSON, 또는 XML

### 주기의 여섯 단계

1. **클라이언트가 메서드, URL, 헤더, 선택적 본문과 함께 요청을 시작합니다.**
2. **요청이 인터넷을 통해 서버로 전달됩니다.**
3. **서버가 요청을 처리합니다**: 메서드, URL, 인증을 확인하고 필요 시 데이터베이스를 조회합니다.
4. **서버가 상태 코드, 헤더, 선택적 본문과 함께 응답을 준비합니다.**
5. **응답이 클라이언트로 전송됩니다.**
6. **클라이언트가 응답을 처리합니다**: 성공(2xx) 시 데이터를 렌더링하고, 실패(4xx/5xx) 시 오류 메시지를 표시합니다.

## HTTP 응답 자산과 콘텐츠 유형

`Content-Type` 헤더는 브라우저가 응답 내 자산을 어떻게 처리할지 알려줍니다.

| 자산 | 콘텐츠 유형 |
| --- | --- |
| HTML | `text/html` |
| CSS | `text/css` |
| JavaScript | `application/javascript` |
| JSON | `application/json` |
| XML | `application/xml` |
| JPEG | `image/jpeg` |
| PNG | `image/png` |
| GIF | `image/gif` |
| SVG | `image/svg+xml` |
| WebP | `image/webp` |
| MP4 | `video/mp4` |
| MP3 | `audio/mpeg` |
| PDF | `application/pdf` |
| ZIP | `application/zip` |
| WOFF2 | `font/woff2` |

## HTML 폼 제출

- 폼은 `<form>` 요소와 `<input>` 요소를 사용해 사용자 데이터를 수집하고 제출합니다.
- 각 입력에는 서버가 데이터를 식별할 수 있도록 `name` 속성이 필요합니다.
- **클라이언트 측 유효성 검사**: 즉각적인 피드백을 위해 `required`와 `pattern` 속성을 사용합니다.
- **서버 측 유효성 검사**: 클라이언트 측 검사와 상관없이 보안을 위해 항상 서버에서 검증하세요.
- `action` 속성은 폼 데이터를 보낼 위치를 지정합니다.
- `method` 속성은 전송 방식을 지정합니다(`GET` 또는 `POST`; `PUT`, `PATCH`, `DELETE`는 Fetch API나 Express의 `method-override` 같은 우회 방법이 필요합니다).

### 폼 제출 흐름

1. 사용자가 폼을 작성하고 제출합니다.
2. 브라우저가 클라이언트 측 유효성 검사를 실행하고, 유효하면 진행합니다.
3. 브라우저가 `action`에 지정된 URL로 `method` 방식으로 폼 데이터를 전송합니다.
4. 서버가 데이터를 수신하고 처리합니다(데이터베이스 저장, 사용자 인증 등).
5. 서버가 추가 서버 측 유효성 검사를 수행할 수 있습니다.
6. 서버가 성공 메시지, 오류, 또는 리다이렉트를 응답합니다.
7. 사용자가 서버 응답에 따른 피드백을 확인합니다.

## 웹 표준 모델과 표준 기구

- **웹 표준**: 브라우저가 페이지를 렌더링하고 웹 기능을 가능하게 하기 위해 따르는 규칙과 명세입니다.
- **WHATWG**: 주요 브라우저 엔진 팀이 주도하는 HTML과 DOM의 지속적으로 업데이트되는 "생활 표준"을 유지하는 표준 기구입니다.
- **W3C**: CSS, SVG, 접근성 지침(WCAG), 다양한 웹 API를 감독하는 주요 표준 기구입니다.
- **ECMA / TC39**: ECMA는 ECMAScript 명세(JavaScript 표준)를 게시하며, TC39는 5단계 제안 프로세스(0~4단계)를 통해 언어 기능을 발전시킵니다.
- **Khronos Group**: GPU 가속 3D 렌더링을 브라우저에 제공하는 WebGL과 WebGL 2를 담당합니다.

## 웹 표준 생성 과정

1. **필요성 확인**: 개발자나 브라우저 엔지니어가 실제로 널리 공유되는 문제를 문서화합니다.
2. **제안 및 초기 논의**: 아이디어를 비공식적으로 작성해 피드백을 받고 브라우저 벤더의 관심을 평가합니다.
3. **명세 작성**: 모든 예외 상황, 오류 조건, 보안 문제를 포함한 정확하고 공식적인 문서를 작성합니다.
4. **검토, 이의 제기, 반복**: 초안을 공개적으로 검토하고 벤더가 이의를 제기하면 합의가 이루어질 때까지 명세를 수정합니다.
5. **구현 및 테스트**: 브라우저 벤더가 기능을 플래그 뒤에 구현하고, 공유 테스트 스위트로 모든 구현이 동일하게 동작하는지 검증합니다.
6. **최종화**: 여러 독립 구현이 테스트를 통과하면 표준이 최종 확정됩니다(W3C 권고안, TC39 4단계, 또는 WHATWG 생활 표준 업데이트).

## 웹 표준 기능의 수명 주기

1. **제안**: 누군가 문제를 발견하고 다른 사람도 공감하는지 알아보기 위해 기록합니다.
2. **인큐베이션**: 아이디어가 공개 토론을 통해 검증되며, 너무 틈새이거나 모두를 만족시키지 못하면 많은 제안이 여기서 사라집니다.
3. **명세화**: 모호한 개념을 정확하고 구현 가능한 규칙으로 바꾸는 공식 명세를 작성합니다.
4. **상호 운용성 테스트**: W3C 웹 플랫폼 테스트 스위트가 모든 브라우저가 동일하게 기능을 구현하는지 확인합니다.
5. **최종화**: 여러 구현이 통과하면 기능이 기본 웹 플랫폼의 일부로 사용자에게 제공됩니다.
6. **사용 중단**: 대체되거나 문제가 있는 기능은 폐기 대상으로 표시되며, 하위 호환성 의무로 인해 브라우저에서 제거는 느리게 진행됩니다.

## 웹 표준의 핵심 원칙

- **개방성**: 표준은 공개적으로 개발되며 누구나 로열티 없이 구현할 수 있습니다.
- **접근성**: 웹은 모두가 사용할 수 있도록 설계되었으며, HTML, CSS, 브라우저 동작에 접근성이 내장되어 있습니다.
- **상호 운용성**: 표준에 맞게 구축된 페이지는 모든 브라우저와 운영체제에서 동일하게 동작해야 합니다.
- **하위 호환성**: 새로운 기능은 기존 기능 위에 쌓여 오래된 페이지도 최신 브라우저에서 계속 작동합니다.
- **개인정보 보호 및 보안**: 사용자 데이터를 노출할 수 있는 새 API는 엄격한 검토를 거치며, 플랫폼은 기본적으로 사용자를 보호합니다.
- **계층화**: HTML은 구조를, CSS는 표현을, JavaScript는 동작을 처리하며 각 계층은 독립적으로 작동합니다.
