# --description--

Web services e APIs REST alimentam a maioria dos apps e sites que você usa todos os dias. Eles permitem que diferentes aplicações se comuniquem entre si e compartilhem dados por uma rede, geralmente a internet.

Uma API (Application Programming Interface) é uma forma geral para software se comunicar com outro software. Um serviço web é um tipo específico de API que usa tecnologias web, como HTTP, para trocar dados por uma rede, geralmente a internet. Por causa disso, todo serviço web é uma API—mas nem toda API é um serviço web. APIs REST são um tipo comum de serviço web que seguem um conjunto específico de regras sobre como os dados são requisitados e retornados.

Esta lição terá foco no que são serviços web em um nível alto e a próxima entrará em mais detalhes sobre como as APIs REST funcionam.

Você pode pensar em um serviço web como uma ponte de software para software. Ele não tem interface de usuário e não é destinado a ser usado diretamente. Em vez disso, ele está totalmente focado em trocar dados entre aplicações.

Para qualquer sistema ser considerado um serviço web, ele deve ter as seguintes características:

* **Roda em um servidor**: um serviço web deve ser hospedado online como um site, mas em vez de páginas que os usuários podem ver, ele expõe dados ou alguma funcionalidade
    
* **Comunicação máquina a máquina**: um serviço web não é feito para humanos visualizarem diretamente, mas para apps móveis e apps web consumirem, para que possam expor alguns dados ou funcionalidades que o usuário finalmente possa visualizar
    
* **Independente da linguagem**: um serviço web pode ser escrito em qualquer linguagem como Java ou C#, e qualquer app deve ser capaz de interagir com ele, mesmo que tenha sido escrito em uma linguagem diferente
    
* **Independente da plataforma**: não importa em qual plataforma as máquinas executem. Por exemplo, um serviço web executando no Linux não deve impedir um dispositivo executando Windows de se comunicar com ele
    
* **Use protocolos web padrão**: um serviço web normalmente deve usar HTTP/HTTPS para transferir dados pela internet, FTP para transferência de arquivo e SMTP para e-mail
    

**REST** e **SOAP** são dois dos tipos de serviços web mais conhecidos, embora existam outras abordagens, como **XML-RPC**.

**REST**, que significa **Representational State Transfer**, é de longe o mais usado hoje. A maioria das aplicações modernas depende de serviços web RESTful porque são simples, flexíveis e funcionam naturalmente com a web.

APIs REST expõem recursos, que geralmente são representados por caminhos de URL como `/users` ou `/users/20`. Quando um cliente solicita um recurso, o servidor retorna uma representação desse recurso. Essa representação é frequentemente em JSON, mas também pode estar em formatos como XML ou até texto simples.

Os clientes podem especificar o formato desejado usando cabeçalhos HTTP, como o cabeçalho `Accept`. Por exemplo, um cliente pode solicitar uma resposta JSON definindo o `Accept` com o valor apropriado.

```bash
Accept: application/json
```

E você pode especificar que deseja XML desta forma:

```bash
Accept: application/xml
```

O recurso que você recebe pode parecer assim se for JSON:

```json
{
  "id": 20,
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "createdAt": "2025-09-22T10:30:00Z"
}
```

E isso, se for XML:

```xml
<user>
  <id>20</id>
  <name>John Doe</name>
  <email>john@example.com</email>
  <role>admin</role>
  <createdAt>2025-09-22T10:30:00Z</createdAt>
</user>
```

SOAP significa **Simple Object Access Protocol**. Ele usa apenas o formato de dados XML e é o mais comum em sistemas de nível empresarial que precisam de padrões e segurança mais rigorosos.

Veja como os dados SOAP podem ser:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
   <soapenv:Body>
      <getUser>
         <userId>20</userId>
      </getUser>
   </soapenv:Body>
</soapenv:Envelope>
```

XML-RPC usa XML para codificar suas `requests` e `responses`. Isso o torna leve mas menos flexível que o `REST`. 

Quanto ao UDDI, ele não é um serviço de troca de dados, mas um diretório baseado em XML onde empresas podem listar, descobrir e descrever seus serviços web disponíveis.
