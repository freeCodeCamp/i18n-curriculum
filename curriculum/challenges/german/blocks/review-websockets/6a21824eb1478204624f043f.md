## WebSockets vs. HTTP

- **HTTP** folgt einem Anfrage-Antwort-Modell: Der Client sendet eine Anfrage, der Server sendet eine Antwort, dann wird die Verbindung geschlossen.
- **WebSockets** erstellen eine dauerhafte, bidirektionale Verbindung zwischen einem Client und einem Server, die offen bleibt, bis eine Seite ein Schlusssignal sendet.
- HTTP eignet sich gut zum Laden von Seiten oder Abrufen von Daten, wird aber bei Echtzeitanwendungen langsam und ineffizient.
- Jede WebSocket-Verbindung beginnt mit einem **HTTP-Handshake**: Der Client fordert ein Upgrade an, und der Server antwortet mit `101 Switching Protocols`.

### Ablauf einer WebSocket-Verbindung

1. Der Client verbindet sich über HTTP mit dem Server und fordert ein Upgrade an.
2. Der Server akzeptiert und antwortet mit `101 Switching Protocols`.
3. Die Verbindung bleibt offen.
4. Beide Seiten tauschen jederzeit Daten aus, bis die Verbindung geschlossen wird.

### Kommunikation HTTP vs. WebSocket

- **HTTP**: `Client → Request → Server → Response` (ein Zyklus, dann Schließen)
- **WebSocket**: `Client ↔ Server` (persistent, bidirektional)

### Eine hilfreiche Analogie

- **HTTP** ist wie das Austauschen von Briefen: Sie schreiben einen, warten auf eine Antwort, und das Gespräch endet.
- **WebSockets** sind wie ein Telefonat: Beide Seiten sprechen in Echtzeit, solange es nötig ist.

### Anwendungsbeispiele für WebSockets

- Chat-Anwendungen
- Online-Multiplayer-Spiele
- Live-Sportergebnis-Updates
- Börsenticker-Dashboards
- Kollaborative Werkzeuge (z. B. Google Docs)

## Pub/Sub-Architektur

- **Pub/Sub** (Publish/Subscribe) ist eine Nachrichtenarchitektur, die es Systemteilen ermöglicht, miteinander zu kommunizieren, ohne direkt voneinander zu wissen.
- Es gibt drei Hauptbestandteile: **Publisher**, **Topics** und **Subscriber**.
  - Ein **Publisher** sendet Nachrichten an ein Topic.
  - Ein **Topic** (auch Kanal oder Gruppe genannt) fungiert als Vermittler.
  - Ein **Subscriber** empfängt Nachrichten von Topics, denen er abonniert ist.

### Nachrichtenfluss bei Pub/Sub

1. Ein Subscriber abonniert ein Topic.
2. Ein Publisher sendet eine Nachricht an dieses Topic.
3. Das Nachrichtensystem liefert die Nachricht an jeden Subscriber dieses Topics.

### Direkte Kommunikation vs. Pub/Sub

- **Ohne Pub/Sub**: `Application A → Application B` — eng gekoppelt; eine Änderung in B kann A zerstören.
- **Mit Pub/Sub**: `Application A → Topic → Subscribers` — A veröffentlicht nur; es muss nicht wissen, wer empfängt.

### Anwendungsbeispiele für Pub/Sub

- Chat-Anwendungen mit mehreren Räumen
- Online-Multiplayer-Spiele
- Benachrichtigungssysteme
- Live-Dashboards
- Ereignisgesteuerte Anwendungen

### WebSockets vs. Pub/Sub

- **WebSockets** stellen den Kommunikationskanal bereit (die Straße).
- **Pub/Sub** stellt das Nachrichtenmuster bereit, das bestimmt, wer welche Nachrichten erhält (das Verkehrssystem).
- Viele Echtzeitanwendungen nutzen beides zusammen: WebSockets halten die Verbindung offen, während Pub/Sub Nachrichten an die richtigen Subscriber weiterleitet.

## Arbeiten mit WebSockets in Node.js

- Node.js hat keinen eingebauten WebSocket-Server. Die `ws`-Bibliothek ist das standardmäßige externe Paket dafür.
- Installation mit: `npm install ws`

### Einen WebSocket-Server erstellen

```js
const WebSocket = require("ws");

const server = new WebSocket.Server({ port: 3000 });
```

### WebSocket-Ereignisse

| Ereignis | Auslöser |
|---|---|
| `connection` | Ein neuer Client verbindet sich mit dem Server |
| `message` | Ein verbundener Client sendet Daten |
| `close` | Eine Verbindung wird geschlossen |
| `error` | Auf der Verbindung tritt ein Fehler auf |

### Eine Nachricht an einen Client senden

```js
socket.send("Hello from the server");
```

### An alle verbundenen Clients senden

- `server.clients` enthält alle verbundenen Clients.
- Überprüfen Sie `readyState === WebSocket.OPEN`, bevor Sie senden, um Fehler bei getrennten Clients zu vermeiden.

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **Broadcasting** ist das Muster, eine Nachricht gleichzeitig an mehrere verbundene Clients zu senden — verwendet in Chat-Apps, Live-Benachrichtigungen, Sport-Updates und Multiplayer-Spielereignissen.
