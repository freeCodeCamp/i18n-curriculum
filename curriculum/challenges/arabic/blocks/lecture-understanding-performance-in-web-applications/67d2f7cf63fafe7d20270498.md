# --description--

بينما أدوات مثل Lighthouse وChrome DevTools رائعة، أحيانًا تحتاج إلى رؤى مخصصة تناسب موقعك.

هنا تأتي واجهات برمجة تطبيقات الويب للأداء!

في هذا الدرس، سنشرح كيفية استخدام ثلاث واجهات برمجة تطبيقات ويب رئيسية لقياس وتحليل سرعة موقعك.

لكن أولًا، ما هي واجهات برمجة تطبيقات الويب للأداء؟

تتيح واجهات برمجة تطبيقات الويب للأداء للمطوّرين تتبع مدى كفاءة تحميل الصفحة واستجابتها، مباشرة من الشفرة. تسمح لك هذه الواجهات بقياس أوقات تحميل الصفحة، وتتبع تأخيرات العرض والتفاعل، وتحليل وقت تنفيذ جافاسكريبت.

باستخدام هذه الواجهات، يمكنك بناء أدوات مراقبة الأداء الخاصة بك دون الاعتماد على برامج طرف ثالث!

لنستكشف ثلاث واجهات برمجة تطبيقات ويب قوية يمكنك استخدامها اليوم.

أولًا، `performance.now()`.

تمنحك هذه الواجهة طوابع زمنية عالية الدقة (بالملي ثانية) لقياس المدة التي تستغرقها أجزاء مختلفة من موقعك للتحميل.

لنفترض أنك تريد قياس سرعة تنفيذ دالة:

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

هذه أكثر دقة من استخدام `Date.now()` لأنها تقيس الوقت بالملي ثانية بدقة ميكروثانية، متجنبةً مشاكل انحراف الساعة. يمكنك استخدامها لتتبع وقت تنفيذ الشفرة، وتأخيرات استجابة الأحداث، وأداء الرسوم المتحركة.

التالي، واجهة توقيت الأداء (Performance Timing API).

تعطيك هذه الواجهة تفصيلًا لكل مرحلة من مراحل تحميل الصفحة، من استعلام DNS إلى `DOMContentLoaded`.

هل تريد قياس المدة التي تستغرقها صفحتك للتحميل الكامل؟

```js
let [navigationTiming] = performance.getEntriesByType("navigation");

if (navigationTiming instanceof PerformanceNavigationTiming) {
  // Calculate time from navigation start to DOM content loaded
  const pageLoadTime =
    navigationTiming.domContentLoadedEventEnd - navigationTiming.startTime;

  console.log("DOM Content Loaded Time:", pageLoadTime, "ms");
}
```

المقاييس الرئيسية التي يمكنك تتبعها بهذه الواجهة هي وقت استعلام DNS - أو بمعنى آخر سرعة الاتصال، ووقت الوصول لأول بايت (TTFB) - أو سرعة استجابة الخادم، و`DOMContentLoaded` - أو بمعنى آخر، متى تصبح الصفحة جاهزة للتفاعل.

إذا كانت أوقات تحميل صفحتك بطيئة، تحدد هذه الواجهة بالضبط مكان حدوث التأخير!

وأخيرًا، دعنا نتحدث عن `PerformanceObserver`.

تستمع هذه الواجهة لأحداث الأداء مثل تغييرات التخطيط، والمهام الطويلة، وتفاعلات المستخدم.

هل تريد مراقبة مهام جافاسكريبت طويلة الأمد؟

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

وماذا يمكن لهذه الواجهة تتبع؟ يمكنها تتبع المهام الطويلة - أو بمعنى آخر، جافاسكريبت التي تعيق العرض، وتغييرات التخطيط لاكتشاف اهتزاز واجهة المستخدم، وتأخير الإدخال الأول (FID) - أو مدى سرعة استجابة الصفحة لمُدخَل المستخدم.

إذا أردت تتبع الأداء في الوقت الحقيقي، فهذه الواجهة تُحدث فرقًا كبيرًا.

فأي واجهة برمجة تطبيقات يجب أن تستخدم؟ إليك مقارنة سريعة:

| واجهة برمجة تطبيقات الأداء | الأفضل لـ                                         |
| :--------------------- | :------------------------------------------------- |
| `performance.now()`    | توقيت دقيق للدوال والسكريبتات                      |
| Performance Timing API | قياس أداء تحميل الصفحة بالكامل                      |
| Performance Observer   | مراقبة التفاعلات والعرض في الوقت الحقيقي            |

بدمج هذه الواجهات، يمكنك بناء أدوات قياس الأداء الخاصة بك وتتبع ما يهم موقعك بالضبط.
