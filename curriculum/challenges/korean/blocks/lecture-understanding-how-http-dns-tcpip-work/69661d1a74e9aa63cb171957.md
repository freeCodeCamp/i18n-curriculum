# --description--

웹사이트를 방문할 때 가장 먼저 하는 일은 브라우저에 웹 주소를 입력하는 것입니다.

예를 들어, freeCodeCamp에 가려면 브라우저 주소 표시줄에 `freecodecamp.org`를 입력합니다.

이것을 도메인 이름이라고 합니다. 인터넷 상에서 해당 자원을 식별하는 고유한 사람이 읽을 수 있는 식별자입니다.

도메인 이름은 이 수업의 주요 주제인 DNS 개념과 매우 밀접하게 관련되어 있습니다. DNS가 어떻게 작동하는지 이해할 수 있도록 도메인 이름 계층 구조에 대해 먼저 간단히 이야기해 보겠습니다.

도메인 이름 계층 구조는 오른쪽에서 왼쪽으로 읽으며, 가장 일반적인 범주에서 가장 구체적인 범주로 이동합니다.

도메인 이름의 주요 구조는 오른쪽에서 왼쪽으로 다음과 같습니다:

* **최상위 도메인(TLD):** 도메인 이름의 마지막 부분으로 오른쪽에 위치합니다. `freecodecamp.org`의 경우 `.org`가 이에 해당합니다. 다른 예로는 `.com`와 `.edu`가 있습니다.
    
* **2단계 도메인(SLD):** 개인이나 조직이 등록한 이름입니다. `freecodecamp`에서 `freecodecamp.org`가 이에 해당합니다. 웹 주소를 볼 때 보통 가장 먼저 눈에 띄는 부분입니다.
    
* **3단계 도메인(서브도메인):** 주 도메인의 특정 섹션을 식별하는 데 사용됩니다. 예를 들어, “World Wide Web”을 의미하는 `www`가 있습니다. 요즘 현대 웹 서버는 도메인 라우팅을 자동으로 처리하므로 `www`를 입력하지 않아도 웹사이트에 접근할 수 있습니다. 다른 서브도메인 예로는 `blog.`, `shop.`, `support.`, `api.`가 있습니다.
    

도메인 이름은 사람이 기억하고 브라우저에 입력하기 쉽기 때문에 매우 유용합니다. 하지만 컴퓨터는 숫자로 작동한다는 것을 알고 있죠. 브라우저는 입력한 도메인 이름을 IP 주소라고 하는 것으로 변환해야 원하는 웹사이트를 찾을 수 있습니다.

**IP 주소**는 네트워크, 예를 들어 인터넷 상에서 컴퓨터를 식별하는 고유한 숫자 수열입니다.

IP 주소는 클라이언트가 올바른 서버에 요청을 보내고 적절한 응답을 받도록 하는 데 필수적입니다. IPv4 주소의 예는 `127.0.0.1`입니다. 최신 표준인 IPv6에서는 이에 해당하는 IP 주소가 `0:0:0:0:0:0:0:1`입니다.

IP 주소에 대해 배울 내용이 많으며, 학습을 계속하면서 더 깊이 탐구할 수도 있습니다. 하지만 지금은 네트워크 상에서 컴퓨터를 식별하는 고유한 숫자 주소라는 점만 이해하면 됩니다.

도메인 이름과 IP 주소 개념에서 이제 DNS 개념에 도달할 수 있습니다.

DNS는 **도메인 이름 시스템**을 의미합니다. 보통 인터넷의 “전화번호부”라고 불립니다.

DNS는 브라우저에 입력한 도메인 이름을 해당하는 IP 주소로 변환합니다. 이는 인터넷이나 특정 네트워크에 연결된 컴퓨터와 장치에 대한 계층적이고 분산된 명명 시스템입니다.

사용자가 `freecodecamp.org`와 같은 도메인 이름을 브라우저에 입력하면, DNS는 그 이름을 해당 서버의 IP 주소에 매핑합니다. 서버가 확인되면 클라이언트의 요청이 서버로 전송됩니다. 서버는 프로그래밍된 대로 요청을 처리하고 최종적으로 클라이언트에 응답을 보냅니다. 이것이 요청-응답 주기를 완성합니다.

이 모든 과정은 대상 IP 주소로의 초기 변환을 수행한 DNS 덕분에 가능합니다.

이제 DNS 해상도 과정의 세부 사항을 살펴보겠습니다.

## DNS 해상도 과정

* DNS 해상도는 DNS 서버 체인을 통해 쿼리를 보내는 방식으로 작동합니다.

* 사용자가 브라우저에 도메인 이름을 입력하면, **클라이언트**는 로컬 DNS 서버에 요청을 보냅니다. 이 로컬 DNS 서버를 **재귀 해결자(Recursive Resolver)**라고 합니다. 보통 인터넷 서비스 제공자(ISP)가 관리합니다.
    
* 재귀 해결자는 그 요청을 **루트 서버(Root Servers)**로 전달합니다. 이 서버들은 IP 주소에 대한 직접 정보를 가지고 있지 않은 일반 서버입니다. 재귀 해결자에게 더 구체적인 서버, 즉 요청된 최상위 도메인(TLD) 서버로 안내합니다. 예를 들어, 모든 `.org` 도메인을 담당하는 서버입니다.
    
* 재귀 해결자는 다음으로 **최상위 도메인(TLD) 서버**에 쿼리를 보냅니다. 이 서버는 요청된 도메인에 대한 레코드를 보유한 특정 권한 있는 이름 서버(예: `freecodecamp.org`)의 주소를 반환합니다.
    
* 재귀 해결자는 해당 **권한 있는 이름 서버(Authoritative Name Server)**에 최종 쿼리를 시작합니다. **DNS 레코드**가 발견되면 재귀 해결자에게 반환됩니다.
    
* 재귀 해결자가 IP 주소를 받으면 클라이언트에 전달하여 웹사이트를 탐색할 수 있게 합니다.

권한 있는 이름 서버는 단순히 IP 주소뿐 아니라 **DNS 레코드**를 반환합니다. DNS 레코드는 서브도메인 정보, 정보가 캐시되어야 하는 기간, 요청된 서비스를 수행하는 데 필요한 정보 등 도메인에 관한 중요한 정보를 담은 구조화된 텍스트 항목입니다.

과정을 최적화하기 위해 재귀 해결자는 DNS 레코드를 임시로 캐시하여 속도를 높입니다. 이렇게 하면 짧은 시간 내에 같은 요청을 반복하지 않게 되어 전체 과정이 훨씬 효율적입니다.

보시다시피 DNS 서버는 도메인 이름을 IP 주소로 변환하는 초기이자 중요한 단계를 제공하기 때문에 필수적입니다. 이 변환 과정 덕분에 모든 클라이언트가 올바른 서버를 찾아 연결할 수 있습니다. 이 디렉터리 시스템이 없으면 현대 웹의 근간인 클라이언트-서버 모델이 작동하지 않을 것입니다.
