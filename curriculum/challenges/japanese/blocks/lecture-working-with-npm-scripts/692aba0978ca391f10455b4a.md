# --description--

npm レジストリについて学びましょう。ここでは JavaScript のパッケージを世界中と共有できます。

**npm レジストリ**は、公開および非公開のパッケージのデータベースです。

開発者や組織は、このレジストリにパッケージを公開して、自分たちのツールをコミュニティやチームメンバー、そして多くの場合はすべての Node.js 開発者と共有できます。

パッケージは公開か非公開のどちらかです。公開パッケージは誰でもアクセスできます。非公開パッケージは特定のユーザーや組織のメンバーだけがアクセスできます。

npm レジストリでは、数回のクリックだけで作ったものを世界と共有できます。素晴らしいですよね？

パッケージを公開する方法をステップごとに見ていきましょう。

## Step 1: アカウントを作成する

最初のステップは無料の npm アカウントを作成することです。アカウントを作成するには：

* [https://www.npmjs.com/signup](https://www.npmjs.com/signup) にアクセスしてください。
    
* フォームに記入します。ユーザー名、メールアドレス、パスワードを入力する必要があります。
    

サインアップページによると、「あなたのメールアドレスは公開するパッケージのメタデータに追加されるため、公開される場合があります」のでご注意ください。

## Step 2: **ログインする**

npm アカウントを作成したら、次のコマンドを実行してターミナルでログインしてください：

```bash
npm login
```

その後、アカウント情報で認証が必要です。

## **Step 3: `package.json` ファイルを作成する**

パッケージを公開する前に、次のコマンドで `package.json` ファイルを作成する必要があります：

```bash
npm init
```

（すでにプロジェクトにある場合は、**step 4** に進んでください）。

`package.json` ファイルには次の情報を入力してください：

* **name**: 通常はプロジェクトやディレクトリの名前です。パッケージを公開する場合、この名前はユニークである必要があります。ただし、スコープを使う場合は例外です（後ほど説明します）。
    
* **version**: デフォルト値の `"1.0.0"` から始めます。これはパッケージの初期バージョンを表し、変更に応じて更新できます。
    
* **main**: プロジェクトのエントリーポイントです。デフォルトは `index.js` です。
    
* **author**: 名前や好みのハンドルネームを記入できます。
    
* **license**: MIT や ISC など、特定のライセンスを選択してください。
    
* **description**: パッケージの簡単な説明です。
    

パッケージ名を選ぶ際は、npm レジストリですでに使われていないか確認してください。次の URL にアクセスし、`<your-package-name>` をパッケージ名に置き換えます：

```bash
https://www.npmjs.com/package/<your-package-name>
```

その URL にパッケージがなければ、その名前は利用可能で使えます。

ただし、名前がすでに使われていてどうしても使いたい場合は、**スコープ**を使って公開できます。

### **スコープ**

npm レジストリでユーザーまたは組織アカウントを作成すると、ユーザー名や組織名のスコープが付与されます。

スコープは名前の衝突を避けるのに役立ちます。スコープを使えば、同じ名前のパッケージがあっても自分のスコープの下で公開できます。

スコープ付きパッケージの名前はスコープ名で始まります。例えば、`package.json` の中には次のキーと値のペアがあります：

```bash
"name": "@<scope-name>/<your-package-name>"
```

必要に応じてファイル内で直接この変更を行うか、`package.json` コマンドに `--scope=@<scope-name>` を追加して最初からスコープ付きの `npm init` ファイルを作成できます。

```bash
npm init --scope=@<scope-name>
```

スコープ名は個人アカウントならユーザー名、組織アカウントなら組織名になります。

## Step 4: README.md を作成する

パッケージの README はマークダウンファイルで、パッケージの機能や使い方を説明します。

含めると良い重要な内容の例は：

* 目的
    
* インストール方法
    
* 重要な依存関係や注意点
    
* 始め方
    
* サンプルコードスニペット
    
* 貢献ガイドライン（貢献を受け入れる場合）
    
* 変更履歴を追跡する changelog ファイルへのリンク
    

README の構成は柔軟で、必要に応じて調整できます。

## **Step 5:** `.npmignore`

`.npmignore` ファイルは、パッケージを公開するときにどのファイルを含め、どのファイルを無視するかを npm レジストリに伝えます。秘密鍵やパスワード、個人情報などの機密情報を共有しないために非常に重要です。

このファイルには、無視するファイルやディレクトリ、ファイルの種類を指定するパターンを書けます。`.gitignore` ファイルと同じパターンマッチングルールに従います。

Git を使ったことがあるなら、このファイルは `.gitignore` に似ていると思うかもしれません。実際、npm のドキュメントによると「`.npmignore` ファイルがなく `.gitignore` ファイルがある場合、npm は `.gitignore` ファイルでマッチしたものを無視します」とあります。

また、npm はパッケージのルートディレクトリだけでなく、すべてのサブディレクトリの `.npmignore` と `.gitignore` ファイルも探します。

デフォルトでは、次のパスは自動的に無視されるため、`.npmignore` に追加する必要はありません：

* `.*.swp`
    
* `._*`
    
* `.DS_Store`
    
* `.git`
    
* `.gitignore`
    
* `.hg`
    
* `.npmignore`
    
* `.npmrc`
    
* `.lock-wscript`
    
* `.svn`
    
* `.wafpickle-*`
    
* `config.gypi`
    
* `CVS`
    
* `npm-debug.log`
    

`node_modules` ディレクトリも無視されますが、バンドルされた依存関係は除きます。

## **Step 6: パッケージを公開する**

npm レジストリにパッケージを公開する前に、次のコマンドでテストしてください：

```bash
npm install <full-path-to-package-directory>
```

公開準備ができたら、パッケージの種類（スコープ付きか無スコープか）によってコマンドが異なります。

無スコープのパッケージなら、次のコマンドを実行します：

```bash
npm publish
```

スコープ付きパッケージはデフォルトで非公開パッケージとして公開されます（アカウントでこの機能が使える場合）。公開したい場合は、コマンドに `--access public` を追加してください。

```bash
npm publish --access public
```

公開パッケージを見るには、次の URL にアクセスします：

```bash
https://npmjs.com/package/<your-package-name>
```

パッケージ名の下に可視性（公開か非公開か）が表示されます。

## **Step 7 (オプション): パッケージをインストールする**

準備が整ったら、Node.js プロジェクトに次のコマンドでパッケージをインストールできます：

```bash
npm install <your-package-name>
```

## パッケージの更新

公開後にパッケージを変更する必要がある場合は、`package.json` のパッケージバージョンを更新してください。

セマンティックバージョニングの標準的な慣例は次の通りです：

* **最初のリリース:** `1.0.0`。
    
* **パッチリリース:** 3番目の数字を増やします。例：`1.0.0` から `1.0.1` へ更新。
    
* **マイナーリリース:** 真ん中の数字を増やし、最後の数字をゼロにリセットします。例：`1.0.0` から `1.1.0` へ更新。
    
* **メジャーリリース:** 最初の数字を増やし、真ん中と最後の数字をゼロにリセットします。例：`1.1.0` から `2.0.0` へ更新。
    
よくできました。これで npm レジストリにパッケージを公開し、バグ修正や新機能追加のために更新する方法がわかりました。
