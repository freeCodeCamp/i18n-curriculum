## サーバーとクライアント・サーバーモデル

- **サーバー**：他のコンピューターからのリクエストを受け取り、ネットワーク経由でデータやサービスを返すコンピューターまたはプログラム。この用語は物理的なハードウェアとその上で動作するソフトウェアの両方を指します。
- **クライアント・サーバーモデル**：クライアントがリクエストを送り、サーバーがそれを処理してレスポンスを返す分散型アーキテクチャです。
- **クライアント**：サーバーにリクエストを送るプロセスやアプリケーション（例：ウェブブラウザー）。
- **リクエスト-レスポンスサイクル**：クライアントがリクエストを送り、サーバーが処理してレスポンスを返します。

### サーバーの種類

- **ウェブサーバー**：ブラウザーにウェブコンテンツ（HTML、画像、CSS）を配信します。
- **アプリケーションサーバー**：ビジネスロジックを処理し、ユーザーの入力を扱います。
- **データベースサーバー**：データの保存、取得、管理を行います。
- **メールサーバー**：メールの送受信と保存を行います。
- **ファイルサーバー**：ネットワーク上でのファイルの保存と共有を管理します。
- **プロキシサーバー**：クライアントと他のサーバーの間の仲介役を果たします。

## DNS（ドメインネームシステム）

- **DNS**：人間が読みやすいドメイン名をIPアドレスに変換する階層的かつ分散型の名前解決システムで、インターネットの「電話帳」と呼ばれます。
- **IPアドレス**：ネットワーク上のデバイスを一意に識別する数値（例：`127.0.0.1`（IPv4）、`0:0:0:0:0:0:0:1`（IPv6））。
- **ドメイン名の階層構造**（右から左に読む）：
  - **TLD（トップレベルドメイン）**：最も右の部分（例：`.org`、`.com`）。
  - **SLD（セカンドレベルドメイン）**：登録された名前（例：`freecodecamp` in `freecodecamp.org`）。
  - **サブドメイン**：メインドメインの一部を識別（例：`www`、`blog`、`api`）。

### DNS解決プロセス

1. ブラウザーがドメイン名を**再帰的リゾルバー**（通常はISPが管理）に送信します。
2. 再帰的リゾルバーが**ルートサーバー**に問い合わせ、該当するTLDサーバーを指示されます。
3. **TLDサーバー**がドメインの**権威ネームサーバー**のアドレスを返します。
4. 権威ネームサーバーがIPアドレスを含む**DNSレコード**を返します。
5. 再帰的リゾルバーがクライアントにIPアドレスを返し、将来の検索を高速化するためにレコードを**キャッシュ**します。

## TCP/IP

- **プロトコル**：ネットワーク上でデバイスが通信するためのルールと標準のセット。
- **TCP/IP**：データのパッケージング、送信、ルーティング、受信方法を決めるプロトコル群。
  - **IP（インターネットプロトコル）**：IPアドレスを使ってデータパケットを送信元から宛先へルーティングします。コネクションレスで、各パケットは独立して送られ、配達保証はありません。
  - **TCP（トランスミッションコントロールプロトコル）**：データを番号付きのセグメントに分割し、宛先で再構築し、エラーや紛失をチェックし、必要に応じて再送を要求します。信頼性のある順序通りの配信を保証します。

## HTTP（ハイパーテキストトランスファープロトコル）

- **HTTP**：ウェブ上でクライアントとサーバー間の通信ルールを定義するアプリケーションレベルのプロトコル。

- **HTTPS**：データをSSL/TLSで暗号化して送信するHTTPの安全なバージョン。

### HTTPリクエストの構造

- **リクエストライン**：`METHOD /path HTTP/version`（例：`GET / HTTP/1.1`）
- **ヘッダー**：キーと値のペアのメタデータ（例：`Content-Type: application/json`）。
- **ボディ**（任意）：サーバーに送信するデータで、`POST`、`PUT`、`PATCH`で使用されます。

**一般的なHTTPメソッド：**

| メソッド | 目的 |
| --- | --- |
| `GET` | データの取得 |
| `POST` | 新しいデータの送信 |
| `PUT` | リソースの置換 |
| `PATCH` | リソースの部分更新 |
| `DELETE` | リソースの削除 |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### HTTPレスポンスの構造

- **ステータスライン**：HTTPバージョン、ステータスコード、理由フレーズ（例：`HTTP/1.1 200 OK`）。
- **ヘッダー**：レスポンスに関するメタデータ（例：`Content-Type: text/html`）。
- **ボディ**（任意）：返されたコンテンツやエラーの説明。

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## HTTPステータスコード

ステータスコードは最初の数字ごとに分類されます：

- **1xx** — 情報提供のレスポンス。
- **2xx** — 成功のレスポンス。
- **3xx** — リダイレクトのメッセージ。
- **4xx** — クライアントエラーのレスポンス。
- **5xx** — サーバーエラーのレスポンス。

### よく使われるコード

| コード | 名前 | 意味 |
| --- | --- | --- |
| `200` | OK | リクエスト成功、コンテンツ返却。 |
| `201` | Created | リソースが正常に作成された。 |
| `204` | No Content | 成功したが返すコンテンツなし。 |
| `301` | Moved Permanently | リソースが恒久的に新しいURLに移動。 |
| `302` | Found | リソースが一時的に別のURLに移動。 |
| `304` | Not Modified | リソースは前回リクエスト以降変更なし。キャッシュを使用。 |
| `400` | Bad Request | サーバーがリクエストを理解できなかった（構文やデータが無効）。 |
| `401` | Unauthorized | 認証情報が必要。 |
| `403` | Forbidden | 有効な認証情報があってもアクセス権がない。 |
| `404` | Not Found | サーバーにリソースが存在しない。 |
| `500` | Internal Server Error | サーバーで予期しないエラーが発生。 |
| `502` | Bad Gateway | 上流サーバーから無効なレスポンスを受信。 |
| `503` | Service Unavailable | サーバーが一時的にリクエストを処理できない。 |

## HTTPリクエスト-レスポンスモデル

リクエスト-レスポンスサイクルは以下のコンポーネントが連携して動作します：

- **クライアント**：HTTPリクエストを送るブラウザー、モバイルアプリ、APIクライアント。
- **HTTPメソッド**：実行するアクション（`GET`、`POST`、`PUT`、`PATCH`、`DELETE`）。
- **URL**：要求されるリソースのアドレス。
- **ヘッダー**：リクエストやレスポンスのメタデータ（例：`Authorization`、`Content-Type`）。
- **ボディ**：`POST`、`PUT`、`PATCH`、`DELETE`のリクエストに含まれるデータ（多くはJSONやフォームデータ）。
- **サーバー**：リクエストを処理しレスポンスを返すバックエンドシステム。
- **ステータスコード**：成功、エラー、リダイレクトを示す数値コード。
- **レスポンスボディ**：実際に返されるデータ（通常はHTML、JSON、XML）。

### サイクルの6つのステップ

1. **クライアントがメソッド、URL、ヘッダー、任意のボディでリクエストを開始**します。
2. **リクエストがインターネット経由でサーバーに届きます**。
3. **サーバーがリクエストを処理**：メソッド、URL、認証を確認し、必要に応じてデータベースを問い合わせます。
4. **サーバーがステータスコード、ヘッダー、任意のボディでレスポンスを準備**します。
5. **レスポンスがクライアントに送信されます**。
6. **クライアントがレスポンスを処理**：成功（2xx）ならデータをレンダリングし、失敗（4xx/5xx）ならエラーメッセージを表示します。

## HTTPレスポンスのアセットとContent-Type

`Content-Type`ヘッダーはブラウザーにレスポンス内のアセットの扱い方を指示します。

| アセット | Content-Type |
| --- | --- |
| HTML | `text/html` |
| CSS | `text/css` |
| JavaScript | `application/javascript` |
| JSON | `application/json` |
| XML | `application/xml` |
| JPEG | `image/jpeg` |
| PNG | `image/png` |
| GIF | `image/gif` |
| SVG | `image/svg+xml` |
| WebP | `image/webp` |
| MP4 | `video/mp4` |
| MP3 | `audio/mpeg` |
| PDF | `application/pdf` |
| ZIP | `application/zip` |
| WOFF2 | `font/woff2` |

## HTMLフォームの提出

- フォームは`<form>`要素と`<input>`要素を使ってユーザーデータを収集し提出します。
- 各入力にはサーバーがデータを識別できるように`name`属性が必要です。
- **クライアント側バリデーション**：即時フィードバックのために`required`と`pattern`属性を使います。
- **サーバー側バリデーション**：クライアント側のチェックに関わらず、セキュリティのため必ずサーバー側でも検証します。
- `action`属性はフォームデータの送信先を指定します。
- `method`属性は送信方法を指定します（`GET`または`POST`；`PUT`、`PATCH`、`DELETE`はFetch APIやExpressの`method-override`などの回避策が必要です）。

### フォーム提出の流れ

1. ユーザーがフォームに入力して提出します。
2. ブラウザーがクライアント側バリデーションを実行し、有効なら続行します。
3. ブラウザーが`action`のURLに指定された`method`でフォームデータを送信します。
4. サーバーがデータを受け取り処理します（データベースへの保存、ユーザー認証など）。
5. サーバーが追加のサーバー側バリデーションを行う場合があります。
6. サーバーが成功メッセージ、エラー、またはリダイレクトで応答します。
7. ユーザーはサーバーのレスポンスに基づくフィードバックを受け取ります。

## ウェブ標準モデルと標準化団体

- **ウェブ標準**：ブラウザーがページをレンダリングしウェブ機能を可能にするためのルールと仕様。
- **WHATWG**：主要なブラウザーエンジンチームが推進するHTMLとDOMの継続的に更新される「リビングスタンダード」を管理する標準化団体。
- **W3C**：CSS、SVG、アクセシビリティガイドライン（WCAG）、多数のウェブAPIを監督する主要な標準化組織。
- **ECMA / TC39**：ECMAはECMAScript仕様（JavaScript標準）を公開し、TC39は5段階の提案プロセス（ステージ0～4）で言語機能を進展させます。
- **Khronos Group**：GPUアクセラレーションによる3Dレンダリングをブラウザーに提供するWebGLとWebGL 2を担当。

## ウェブ標準作成のプロセス

1. **ニーズの特定**：開発者やブラウザーエンジニアが広く共有される問題を文書化します。
2. **提案と初期議論**：アイデアを非公式にまとめ、フィードバックを集めブラウザーベンダーの関心を探ります。
3. **仕様の作成**：すべてのエッジケース、エラー条件、セキュリティ影響を含む正確で正式な文書を作成します。
4. **レビュー、異議申し立て、反復**：ドラフトを公開レビューし、ベンダーの異議を受けて合意が得られるまで修正します。
5. **実装とテスト**：ベンダーがフラグ付きで機能を実装し、共通のテストスイートで動作の一貫性を検証します。
6. **最終化**：複数の独立した実装がテストに合格すると、標準が確定します（W3C勧告、TC39ステージ4、またはWHATWGリビングスタンダードの更新）。

## ウェブ標準機能のライフサイクル

1. **提案**：問題を特定し、他者の共感を得るために文書化します。
2. **インキュベーション**：オープンな議論でアイデアを検証し、多くの提案はニッチすぎるか解決策が不十分でここで消えます。
3. **仕様化**：曖昧な概念を正確で実装可能なルールに変える正式な仕様を作成します。
4. **相互運用性テスト**：W3C Web Platform Testsスイートで全ブラウザーが同一に実装しているか検証します。
5. **最終化**：複数の実装が合格し、機能が基盤ウェブプラットフォームの一部としてユーザーに提供されます。
6. **廃止**：置き換えられたか問題のある機能は廃止予定とされ、後方互換性のためブラウザーからの削除はゆっくり進みます。

## ウェブ標準の主要原則

- **オープン性**：標準は公開され、誰でもロイヤリティフリーで実装できます。
- **アクセシビリティ**：ウェブは誰でも使えるよう設計されており、HTML、CSS、ブラウザーの動作にアクセシビリティが組み込まれています。
- **相互運用性**：標準に準拠したページはすべてのブラウザーやOSで同じように動作します。
- **後方互換性**：新機能は既存のものに重ねて追加されるため、古いページも最新のブラウザーで動作し続けます。
- **プライバシーとセキュリティ**：ユーザーデータを公開する可能性のある新APIは厳しい審査を受け、プラットフォームはデフォルトでユーザーを保護します。
- **レイヤリング**：HTMLは構造、CSSは見た目、JavaScriptは動作を担当し、それぞれ独立して機能します。
- **プログレッシブエンハンスメント**：機能はまず全員に動作し、その後より高機能なブラウザーやデバイス向けに改善されます。
