# --description--

تتيح WebSockets التواصل في الوقت الحقيقي بين العميل والخادم.

عادةً، تستخدم تطبيقات العميل مثل تطبيقات الويب والهاتف المحمول بروتوكول HTTP. مع HTTP، يرسل العميل طلبًا، ويرد الخادم باستجابة. يُسمى هذا دورة الطلب والاستجابة، كما تعلمت.

بعد إنشاء الاتصال وتبادل البيانات، يُغلق الاتصال. هذا يعمل جيدًا لمعظم الأمور، مثل تحميل صفحة أو جلب أو إرسال بيانات. لكنه يصبح بطيئًا وغير موثوق للتطبيقات في الوقت الحقيقي.

على سبيل المثال، فكر في تطبيق دردشة. إذا قررت استخدام HTTP له، يجب على العميل الاستمرار في طلب البيانات من الخادم بين الحين والآخر، مع فتح الاتصال، وصول البيانات، ثم إغلاق الاتصال بعد كل رسالة.

هذه هي المشكلة التي تحلها WebSockets. فهي تنشئ اتصالًا دائمًا بين العميل والخادم. بمجرد فتح الاتصال، يمكن للطرفين تبادل البيانات في أي وقت عبر قناة في الوقت الحقيقي حتى يُرسل إشارة إغلاق من أحدهما للآخر.

على الرغم من اختلاف WebSockets عن HTTP، يبدأ كل اتصال WebSocket ببروتوكول HTTP في عملية تسمى المصافحة.

ثم يطلب العميل ترقية الاتصال إلى WebSocket. بمجرد موافقة الخادم، يرسل استجابة `101 Switching Protocols` التي ترفع الاتصال إلى WebSocket. بعد ذلك، يبقى الاتصال مفتوحًا وتبدأ الاتصالات في الوقت الحقيقي.

تبدو العملية كالتالي:

1. يتصل العميل بالخادم عبر HTTP
2. يقبل الخادم ويرفع الاتصال إلى WebSocket
3. يبقى الاتصال مفتوحًا
4. يتبادل الطرفان البيانات في أي وقت حتى يُغلق الاتصال

هذه العملية ثنائية الاتجاه، وليست طلب-استجابة مثل HTTP.

```md
Client <---> Server
```

مع HTTP، يتبع التدفق عادة نموذج الطلب والاستجابة:

```md
Client ---> Request
Server ---> Response
```

في الفرق بين WebSockets وHTTP، يشبه HTTP تبادل الرسائل، وWebSocket يشبه مكالمة هاتفية. تكتب رسالة، تنتظر الرد، وتنتهي المحادثة. لكن مع المكالمة الهاتفية، تتحدث أنت والمستلم في الوقت الحقيقي.

تشغل WebSockets تطبيقات الوقت الحقيقي مثل تطبيقات الدردشة، الألعاب متعددة اللاعبين على الإنترنت، تحديثات الرياضة الحية، لوحات بيانات سوق الأسهم، وأدوات التعاون مثل Google Docs.
