# --description--

GitHub Pages ist eine wirklich solide Option, um statische Seiten oder Anwendungen zu deployen, die keinen Back-End-Server zur Verarbeitung von Logik benötigen. Das heißt, Anwendungen, die vollständig clientseitig oder im Browser des Nutzers ausgeführt werden, können vollständig auf dieser Plattform bereitgestellt werden.

Außerdem sind GitHub Pages für Open-Source-Projekte komplett kostenlos. Es gibt viele gute Gründe, Ihr Projekt als Open Source zu veröffentlichen, und Pages bietet noch einen weiteren Vorteil.

Aber wie deployt man eigentlich auf GitHub Pages? Nun, das hängt davon ab, wie Ihre Anwendung funktioniert.

Beginnen wir mit einer einfachen HTML-Seite. Hier ist eine, die Sie gerne zu Ihrem bestehenden Repository hinzufügen können:

```html
<!DOCTYPE html>
<html>
  <head>
    <title>freeCodeCamp</title>
    <link rel="stylesheet" href="./style.css" />
  </head>
  <body>
    <h1>Welcome to freeCodeCamp's Lessons!</h1>
    <p>Let's have some fun learning about Git and GitHub!</p>
  </body>
</html>
```

Jetzt müssen wir unser Repository besuchen und auf den Tab „Settings“ klicken, um zu den Repository-Einstellungen zu gelangen. Dann wählen wir „Pages“ aus der Seitenleiste aus.

Anfangs gibt es nur wenige Optionen, aber das wird sich ändern, sobald wir die Einstellungen konfigurieren. Zuerst müssen wir GitHub mitteilen, was deployed werden soll.

Wir wollen das auf „Deploy from a branch“ belassen, weil wir kein Framework verwenden und keine GitHub Actions konfiguriert haben. Mehr darüber lernen Sie in einer kommenden Lektion.

Als Nächstes legen Sie den Branch fest, von dem wir deployen, auf `main`, um genau das zu deployen, was dort liegt.

Dann haben Sie die Wahl, entweder aus dem Stammverzeichnis Ihres Projekts oder aus einem `docs`-Ordner zu deployen. Diese Funktion wird von Projekten wie freeCodeCamp genutzt – wir haben früher Dokumentation mit unserer Anwendung gebündelt, sodass wir nur die Dokumentation über den `docs`-Ordner auf GitHub Pages deployen konnten. Für den Moment lassen Sie es auf dem `/`- (Stamm-)Ordner und klicken Sie auf die Schaltfläche „Save“ neben diesem Dropdown, um Ihre Änderungen zu speichern.

Nachdem Sie gespeichert haben, sehen Sie ein neues Eingabefeld, um eine benutzerdefinierte Domain zu konfigurieren. Wenn Sie bereits eine haben und diese verwenden möchten, können Sie dem Link zu den Dokumentationen folgen, um das einzurichten. Für diese Lektion lassen wir das Feld jedoch leer.

Sie können auch die Domain sehen, unter der Ihre neue GitHub Pages-Seite bereitgestellt wurde. Ohne eine benutzerdefinierte Domain ist das standardmäßig `<username>.github.io`, wobei `<username>` der Name Ihres Accounts ist (oder der Organisations-Account, wenn Sie aus einem org-eigenen Repository deployen). Und das Repository wird unter einem Pfad bereitgestellt, der dem Namen des Repositories entspricht, zum Beispiel `<username>.github.io/<repo-name>`. Wenn Sie die URL besuchen, sehen Sie Ihre Seite!

Wenn Sie nun neue Änderungen in Ihren `main`-Branch pushen, aktualisiert GitHub Pages automatisch Ihre bereitgestellte Seite mit den neuesten Änderungen. Dieser Prozess kann je nach Serverauslastung einige Minuten dauern – und Sie müssen möglicherweise einen Hard-Refresh durchführen, um den Cache zu löschen.

Damit haben wir erfolgreich eine einfache Webanwendung auf GitHub Pages bereitgestellt. Sie können auch CSS- und JavaScript-Dateien in Ihr Pages-Deployment einbinden, sodass Sie schöne und interaktive Seiten erstellen können. Und sobald Sie ein Framework wie React oder Angular lernen, können Sie diese ebenfalls auf GitHub Pages deployen!
