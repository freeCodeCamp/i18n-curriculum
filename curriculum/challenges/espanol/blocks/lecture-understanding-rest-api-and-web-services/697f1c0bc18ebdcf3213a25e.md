# --description--

**REST** significa representational state transfer. Piénsalo como un conjunto de directrices que describen cómo los clientes y servidores web deben comunicarse entre sí a través de internet.

En esta lección, aprenderás qué es REST, sus conceptos clave y cómo funciona.

REST define una forma para que diferentes sistemas interactúen usando tecnologías web estándar. Con REST, clientes como navegadores web, aplicaciones móviles y APIs se comunican con un servidor usando métodos HTTP comunes como `GET`, `POST`, `PUT`, `PATCH` y `DELETE`, que discutimos en una lección anterior.

Algunos conceptos clave de REST incluyen **statelessness**, **resources**, **HTTP methods** y **response or status codes**.

**La ausencia de estado** significa que cada solicitud que un cliente envía al servidor se maneja de forma independiente. El servidor no recuerda solicitudes anteriores. Debido a esto, cada solicitud debe incluir toda la información necesaria para procesarla. Por ejemplo, si un usuario está autenticado, su token de autenticación debe enviarse con cada solicitud.

En REST, todo se trata como un recurso. Un recurso es un fragmento de datos que puede ser accedido o modificado. Cada recurso se identifica mediante una URL, como `/products` para todos los productos, o `/products/1` para un producto específico.

A continuación, se presenta una visión general de los métodos HTTP, códigos de estado y un ejemplo paso a paso de cómo funciona la arquitectura REST.

## Métodos HTTP

Los métodos HTTP definen la acción a realizar sobre un recurso:

* `GET`: Recuperar un recurso
    
* `POST`: Crear un nuevo recurso
    
* `PUT / PATCH`: Actualiza un recurso existente
    
* `DELETE`: Elimina un recurso
    

## Códigos de Estado HTTP

Después de procesar una solicitud, el servidor responde con un **código de estado** que indica el resultado:

* `200` **(OK)**: La solicitud fue exitosa
    
* `201` **(Created)**: Se creó un nuevo recurso
    
* `400` **(Solicitud Incorrecta)**: El cliente envió datos inválidos
    
* `404` **(No Encontrado)**: El recurso solicitado no existe
    
* `500` **(Error Interno del Servidor)**: Ocurrió un error en el servidor
    

## Cómo funciona REST: un ejemplo paso a paso

### 1\. El cliente envía una solicitud

* El cliente envía una solicitud HTTP usando un método como `GET` o `POST`
    
* La solicitud incluye una URL que identifica el recurso
    
* También puede incluir encabezados, metadatos y un cuerpo de solicitud
    

### 2\. El servidor recibe la solicitud

* El servidor valida el formato de la solicitud y la autenticación (si es requerida)
    
* Identifica el recurso objetivo a partir de la URL
    
* Determina la acción solicitada según el método HTTP
    
* Verifica permisos y control de acceso
    

### 3\. El servidor realiza la acción

* `GET`: Recuperar el recurso
    
* `POST`: Crear un nuevo recurso
    
* `PUT / PATCH`: Actualiza el recurso
    
* `DELETE`: Elimina el recurso
    
* Cualquier otra lógica necesaria se ejecuta
    

### 4\. El servidor construye la respuesta

* El servidor selecciona un código de estado apropiado (por ejemplo, `200`, `201` o `404`)
    
* Formatea los datos de la respuesta, usualmente como **JSON**
    
* Agrega encabezados relevantes como `Content-Type` o `Cache-Control`
    

### 5\. El servidor envía la respuesta

* La respuesta incluye un código de estado, encabezados y un cuerpo opcional
    
* No se almacena estado del cliente en el servidor (sin estado)
    

### 6\. El cliente maneja la respuesta

* El cliente verifica el código de estado para determinar éxito o fracaso
    
* Procesa cualquier dato devuelto por el servidor
