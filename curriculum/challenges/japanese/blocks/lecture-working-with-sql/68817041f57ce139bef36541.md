# --description--

SQLインジェクションは、ウェブアプリケーションがデータベースに対して行うクエリを攻撃者が変更したり悪用したりするウェブセキュリティの脆弱性です。例えば、攻撃者は悪意のあるSQLコードをクエリに挿入し、バックエンドでその悪意のあるコードを実行させることができます。

これは、ログインフォーム、検索バー、URLパラメータなどを通じてユーザーにデータを入力させ、そのユーザー入力を文字列の補間や連結で最終的なSQLコマンドを生成する際に起こる場合があります。

攻撃者の目的によっては、悪意のあるSQLコードが認証の回避、データベースからのデータの盗難、データベースのレコードの変更や削除など、データベースやウェブアプリケーションに重大な障害を引き起こすことがあります。したがって、この脅威は非常に深刻に受け止める必要があります。

例えば、ウェブアプリケーションがユーザー認証のために次のようなSQLクエリを使い、ユーザーが入力したユーザー名とパスワードを直接埋め込んでいるとします。

```sql
'SELECT * FROM users WHERE username = ' + username_input + ' AND password = ' + password_input + ';'
```

攻撃者が`username`フィールドに次のようなものを入力した場合：

```sql
'" " OR "1"="1" --'
```

そして`password`フィールドに何かを入力すると、`"1"="1"`は常に`true`であり、`--`がクエリの残りをコメントアウトするため、`WHERE`節は次のようになります：

```sql
WHERE username = " " OR TRUE
```

これは常に`true`となります。

データベースの実装によっては、これが最初のユーザーまたはデータベース内の任意のユーザーのデータを返し、攻撃者が有効な資格情報なしでログインできるようにします。

では、これをどう防げばよいでしょうか？SQLインジェクションを防ぐ主な方法は2つあります：

- 文字列の連結で動的なクエリを書かないこと。
- バックエンドで実行されるクエリに悪意のあるSQL入力が含まれないようにすること。

実際には、パラメータ化されたステートメント、別名パラメータ化クエリを使うことができます。これにより、SQLの構造とユーザーが入力したデータが分離され、データベースが入力データを実行可能なコードとして解釈するのを防ぎます。

もう一つ重要な手法は入力バリデーションです。これはすべての場合において二次的なセキュリティ対策として推奨されますが、特にテーブル名、カラム名、並び順に関わるSQLクエリの部分に対して重要です。可能であれば、テーブル名やカラム名はユーザー入力からではなく、コード内で直接代入することをお勧めします。

すべての予防策を講じてもSQLインジェクション攻撃を受けてしまった場合は、各データベースアカウントの権限を制限することでデータベースへの影響を最小限に抑えられます。各データベースユーザーには、その作業に必要な権限だけを付与し、それ以上は与えないでください。もしアカウントがデータベースからデータを読み取るだけでよい場合は、読み取りアクセスのみを付与してください。そして予防策として、アプリケーションアカウントに管理者権限を絶対に割り当てないでください。攻撃者が完全なアクセス権とデータベースの完全な制御を得る可能性があるため非常に危険です。

SQLインジェクションは非常に一般的な攻撃であり、開発者としてウェブアプリケーションの実装時に認識し、防ぐ必要があります。これらの手法を使えば、データベースを保護し、ユーザーのデータを安全に保てます。
