## Node.jsとイベント駆動アーキテクチャの活用

- **Node.js**: ブラウザの外でJavaScriptコードを実行できる、オープンソースでクロスプラットフォームのJavaScriptランタイム環境です。ウェブサーバー、API、バックエンドシステムの構築に広く使われています。
- **クライアント側の制限**: クライアント側のJavaScriptはファイルシステムへのアクセスが制限されており、複雑なアプリケーションロジックのサポートも限られています。また、セキュリティリスクがあり（例：データベースの認証情報をクライアント側コードに公開してはいけません）。
- **ブラウザとNode.jsのグローバルオブジェクトの違い**: ブラウザではグローバルオブジェクトは`window`で、DOM操作、クッキー、ブラウザイベントを公開しています。Nodeではグローバルオブジェクトは`global`で、ファイル、ネットワーク、OS操作の組み込みモジュールを公開しています。
- **DOMアクセス**: ブラウザはDOM APIへのアクセスを提供しますが、Nodeは提供しません。逆にNodeはファイルシステムを含むほぼすべてのシステムリソースにアクセスできますが、ブラウザはできません。
- **バージョン管理**: サーバーで動作するNodeのバージョンは選べますが、ユーザーが使うブラウザのバージョンは制御できません。
- **ノンブロッキングでイベント駆動のアーキテクチャ**: Nodeはシングルスレッドとイベントループを使い、多数の同時リクエストやI/O操作を効率的に処理し、ブロックしません。リアルタイムアプリに最適です。
- **コールバック**: 非同期処理が完了したときに何が起こるかを定義する関数です。Nodeの非同期コードが読みづらくデバッグしにくい理由の一つです。
- **バックエンドでのNodeの利点**: フロントエンドとバックエンドで単一言語（JavaScript）を使えること、高い同時処理に適したノンブロッキングアーキテクチャ、大規模なnpmエコシステム、無料かつオープンソースであること。
- **バックエンドでのNodeの欠点**: シングルスレッドなのでCPU負荷の高い処理（画像処理、暗号化、複雑な数学計算）がイベントループをブロックすることがあること、非同期/コールバックベースのコードに依存するため可読性が下がること、npmパッケージの品質やメンテナンス状況がまちまちなこと。
- **NVM（Node Version Manager）**: 同じマシンで複数のNode.jsバージョンを管理・切り替えできるツールです。macOS/Linuxでは以下でインストールします：

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **LTS版Nodeのインストール**: `nvm install --lts`
- **Nodeバージョンの切り替え**: `nvm use 20`
- **インストール済みNodeバージョンの一覧表示**: `nvm ls`
- **デフォルトNodeバージョンの設定（macOS/Linux）**: `nvm alias default 22.20.0`
- **nvm-windows**: Windows（WSLではない）用のNodeバージョン管理プロジェクトです。公式リポジトリからインストーラーをダウンロードし、`nvm install lts`、続いて`nvm use <version>`を実行します。
- **NodeでJavaScriptファイルを実行する**: `node app.js`

## Nodeコアモジュールの活用

### `fs` モジュール

- **目的**: ファイルやディレクトリの作成、読み取り、書き込み、追記、削除のメソッドを提供します。
- **3つの使用スタイル**: コールバックベース（非同期）、Promiseベース（`fs/promises`）、同期（`Sync`サフィックス）。同期メソッドはイベントループをブロックするため、スクリプト向けです。
- **`writeFile()`**: ファイルに内容を書き込み、存在しなければ作成します：

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**: 既存ファイルに内容を上書きせず追記します。
- **`readFile()`**: ファイル内容を読み取ります。エンコーディングに`"utf8"`を渡すと文字列を取得し、省略すると`Buffer`を取得します。
- **`unlink()`**: ファイルを削除します。ファイルが存在しない場合は`ENOENT`エラーをスローします。

### `Buffer` モジュール

- **目的**: バイナリデータ（ファイル、画像、ネットワークストリーム）をメモリ上で直接扱います。
- **`Buffer.from()`**: 文字列からバッファを作成します：

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**: `n`バイトのゼロ埋めバッファを作成します：

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **切り捨て**: バッファの容量を超えて書き込むと余分な部分は静かに切り捨てられます。
- **`Buffer.byteLength(str)`**: 文字列を格納するのに必要なバイト数を返します。

### `crypto` モジュール

- **目的**: ハッシュ化、HMAC、暗号化/復号、デジタル署名、安全な乱数生成などの暗号化ツールを提供します。
- **`createHash(algorithm)`**: 一方向ハッシュ（不可逆）を作成します。`.update(data).digest("hex")`でハッシュ対象のデータを渡し、16進文字列で返します：

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**: `createHash`に似ていますが秘密鍵が必要で、データ整合性検証や認証に適しています。
- **`createCipheriv()` / `createDecipheriv()`**: データの暗号化と復号を行います。どちらもアルゴリズム、鍵、初期化ベクトル（IV）が必要です：

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**: 暗号学的に安全な乱数バイトを生成します。トークンや鍵に`Math.random()`より安全です。
- **`randomInt(min, max)`**: 指定範囲内の安全な乱数整数を生成します。OTPに便利です。
- **`sign()` / `verify()`**: 秘密鍵/公開鍵ペアを使ってデジタル署名の作成と検証を行います。

### `os` モジュール

- **目的**: Nodeが動作しているOSに関する情報を公開します。
- **`os.platform()`**: OS識別子を返します。例：`'darwin'`、`'win32'`、`'linux'`。
- **`os.arch()`**: CPUアーキテクチャを返します。例：`'x64'`、`'arm64'`。
- **`os.type()`**: 公式のOS名を返します。例：`'Darwin'`。
- **`os.release()`**: OSカーネルのバージョン文字列を返します。
- **`os.cpus()`**: 論理CPUコアごとの情報（モデル、速度、稼働時間）を含む配列を返します。
- **`os.uptime()`**: システムの稼働時間（秒）を返します。
- **`os.totalmem()` / `os.freemem()`**: システムの総メモリと空きメモリ（バイト単位）を返します。
- **`os.userInfo()`**: 現在のユーザーの`uid`、`gid`、`username`、`homedir`、`shell`を含むオブジェクトを返します。
- **`os.networkInterfaces()`**: アドレス、ファミリー、MAC、CIDR情報を含むネットワークインターフェースのオブジェクトを返します。

### `path` モジュール

- **目的**: プラットフォームに依存しないファイル・ディレクトリパス操作のユーティリティを提供します。
- **`__filename`**: グローバル変数で、現在のファイルの絶対パスです。
- **`__dirname`**: グローバル変数で、現在のファイルがあるディレクトリの絶対パスです。
- **`path.basename(p)`**: パスの最後の部分（ファイル名）を返します。
- **`path.dirname(p)`**: パスのディレクトリ部分を返します。
- **`path.extname(p)`**: ファイル拡張子を返します。例：`'.js'`。
- **`path.join(...segments)`**: パスのセグメントを結合し、正規化して誤ったスラッシュを修正します。
- **`path.resolve(...segments)`**: 現在の作業ディレクトリから絶対パスを解決します。
- **`path.parse(p)`**: `root`、`dir`、`base`、`ext`、`name`プロパティを持つオブジェクトを返します。
- **`path.format(obj)`**: `dir`、`name`、`ext`を持つオブジェクトからパス文字列を生成します。

### `process` モジュール

- **目的**: 現在のNode.jsプロセスに関する情報と制御を公開します。グローバルオブジェクトなのでインポート不要です。
- **`process.env`**: すべての環境変数を含むオブジェクト。`process.env.NODE_ENV`のように特定の値にアクセスできます。
- **`process.argv`**: コマンドライン引数の配列。インデックス0はNode実行ファイル、1はスクリプトパス、2以降はユーザー引数です。
- **`process.cwd()`**: 現在の作業ディレクトリを返します。
- **プロセスイベント**：

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**: カスタム警告を発生させ、`warning`リスナーでキャッチ可能にします。

### `stream` モジュール

- **目的**: 大量のデータを一度にメモリに読み込むのではなく、チャンク単位で処理します。
- **4種類のストリーム**: 読み取り可能（チャンク単位で読み取る）、書き込み可能（チャンク単位で書き込む）、デュプレックス（両方）、トランスフォーム（流れるデータを変換するデュプレックス）。
- **ストリームのパイプ処理**：

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **`finish` イベント**: 書き込みストリームで全データがフラッシュされたときに発火します。
- **`error` イベント**: ストリームで問題が発生したときに発火します。

## npmの紹介

- **npm**: 3つの部分から成り、ウェブサイト`npmjs.com`、パッケージの公開レジストリ（データベース）、Node.jsに付属するCLIツールです。
- **目的**: 再利用可能なJavaScriptパッケージのインストール、管理、公開を可能にします。npmは最も広く使われているパッケージマネージャーです。
- **代替ツール**: Yarn、PNPM、Bun。

### `package.json` ファイル

- **目的**: npmの設定マニフェストです。プロジェクト名、バージョン、説明、エントリーポイント（`main`）、作成者、ライセンス、スクリプト、依存関係を管理します。
- **`package.json`を対話的に作成する**: `npm init`
- **デフォルト設定で作成する**: `npm init -y`
- **`dependencies`**: 本番環境でアプリを動かすために必要なパッケージ（例: `express`、`react`）。
- **`devDependencies`**: 開発やテスト時のみ必要なパッケージ（例: `jest`、`nodemon`、`eslint`）。
- **`license`**: 使用に関する法的条件。一般的な値は `MIT`、`ISC`、`GPL` です。

### セマンティックバージョニング（SemVer）

- **形式**: `MAJOR.MINOR.PATCH`、例：`4.17.21`。
- **MAJOR**: 後方互換性を壊す変更時に増加。
- **MINOR**: 後方互換性のある新機能追加時に増加。
- **PATCH**: 後方互換性のあるバグ修正時に増加。
- **キャレット `^`**: マイナーとパッチの更新を許可し、メジャーは許可しません。例：`^4.17.21`は`4.18.0`を許可しますが`5.0.0`は許可しません。
- **チルダ `~`**: パッチバージョンのみ更新を許可します。例：`~1.2.3`は`1.2.4`を許可しますが`1.3.0`は許可しません。
- **アスタリスク `*`**: 任意のバージョンにマッチします。テスト用で、本番には使いません。
- **正確なバージョン**: プレフィックスなし。例：`1.2.3`は正確にそのバージョンに固定します。

### 依存関係のインストールと削除

- **パッケージのインストール**: `npm install express`（省略形：`npm i express`）
- **devDependencyとしてインストール**: `npm install nodemon -D`
- **パッケージのアンインストール**: `npm uninstall chalk`
- **`node_modules/`**: インストール後に作成される実際のパッケージコードを含むフォルダー。バージョン管理には含めません。
- **`package-lock.json`**: インストールされたすべてのパッケージ（子依存関係含む）の正確なバージョンを固定する自動生成ファイル。必ずコミットし、手動編集はしないでください。
- **特定バージョンのインストール**: `npm install express@4.21.2`
- **利用可能なバージョンの確認**: `npm view express versions`
- **古いパッケージの確認**: `npm outdated`
- **単一パッケージの更新**（許容範囲内）: `npm update express`
- **最新バージョンの強制インストール**: `npm install express@latest`

### npmレジストリへの公開

- **ログイン**: `npm login`
- **スコープ付きパッケージ**: 名前の衝突を避けるために`@username/package-name`を使います。
- **`.npmignore`**: 公開パッケージから除外するファイルをリストアップします（`.gitignore`に似ています）。
- **スコープなしパッケージの公開**: `npm publish`
- **スコープ付きパッケージの公開（公開設定）**: `npm publish --access public`

## npmスクリプトの活用

- **npmスクリプト**: `package.json`の`"scripts"`オブジェクトに定義するカスタムコマンドです。サーバー起動、テスト実行、リンティング、ビルドなどの繰り返し作業を自動化します。
- **スクリプトの定義例**：

```json
"scripts": {
  "start": "node app.js"
}
```

- **スクリプトの実行**: `npm run start`
- **組み込みスクリプト名の省略形**（`start`、`test`、`stop`、`restart`）: `npm start`
- **`&&`による連続コマンド**（前のコマンドが成功した場合のみ次を実行）：

```json
"start": "npm run build && node server.js"
```

- **`;`による連続コマンド**（常に次のコマンドを実行）：

```json
"start": "npm run build; node server.js"
```

- **`&`による同時実行コマンド**: `"dev": "npm run start:server & npm run start:client"`
- **スクリプトへの引数渡し**（`--`区切りを使う）: `npm start -- 8000`。引数は`process.argv[2]`でアクセス可能です。

### CommonJSモジュール

- **CommonJS**: 元々のNode.jsモジュールシステム。モジュールを同期的に読み込みます。
- **`require()`によるインポート**：

```js
const { multiply } = require('./math');
```

- **`module.exports`によるエクスポート**：

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- `package.json`に`"type": "commonjs"`（または`"type"`フィールドなし）、あるいはファイル拡張子が`.cjs`の場合、CommonJSがデフォルトです。

### ESモジュール（ESM）

- **ESモジュール（ESM）**: 最新の標準化されたモジュール形式。非同期に読み込みます。
- **名前付きエクスポート**: `export function multiply(a, b) { return a * b; }`
- **デフォルトエクスポート**: `export default function multiply(a, b) { return a * b; }`
- **インポート**: `import { multiply } from './math.mjs';`
- ファイル拡張子が`.mjs`か、`package.json`に`"type": "module"`がある場合にESMが使われます。

## HTTP、DNS、TCP/IPの仕組みの理解

### サーバーとクライアント・サーバーモデル

- **クライアント・サーバーモデル**: コンポーネントがクライアント（リクエストを送るプロセス、例：ウェブブラウザ）かサーバーのいずれかで構成される分散システムアーキテクチャです。
- **サーバー**: クライアントからのリクエストを受け取り、ネットワーク経由でデータやサービスを返すコンピューターまたはソフトウェアプログラムです。
- **サーバーの種類**: ウェブサーバー（HTML/CSS/アセット）、アプリケーションサーバー（ビジネスロジック）、データベースサーバー、メールサーバー、ファイルサーバー、プロキシサーバーなど。

### DNS

- **DNS（ドメインネームシステム）**: インターネットの「電話帳」と呼ばれ、人間が読めるドメイン名（例：`freecodecamp.org`）をIPアドレスに変換します。
- **ドメイン名の階層構造**（右から左へ）: トップレベルドメイン（TLD、例：`.org`）→セカンドレベルドメイン（例：`freecodecamp`）→サブドメイン（例：`www`）。
- **IPアドレス**: ネットワーク上のデバイスを一意に識別する数値。IPv4例：`127.0.0.1`。IPv6例：`0:0:0:0:0:0:0:1`。
- **DNS解決プロセス**: ブラウザ → 再帰的リゾルバ → ルートサーバー → TLDサーバー → 権威ネームサーバー → IPアドレス返却 → 再帰的リゾルバがキャッシュしてクライアントに返す。

### TCP/IPとHTTP

- **IP（インターネットプロトコル）**: IPアドレスを使ってデータパケットを送信元から宛先へルーティングします。コネクションレスで、各パケットは独立して送信されます。
- **TCP（トランスミッションコントロールプロトコル）**: データを番号付きセグメントに分割し、宛先で再構築、エラー検出、再送要求を行います。信頼性のある順序付き配信を保証します。
- **HTTP（ハイパーテキストトランスファープロトコル）**: クライアントとサーバーがウェブコンテンツを交換するためのアプリケーション層プロトコルです。
- **HTTPS**: HTTPの安全版で、SSL/TLSを使ってデータを暗号化します。

### HTTPリクエストの構造

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **リクエストライン**: HTTPメソッド + パス/URI + HTTPバージョン。
- **ヘッダー**: キーと値のメタデータペア。
- **ボディ**（任意）: `POST`、`PUT`、`PATCH`、`DELETE`でデータを送信するために使います。

### HTTPレスポンスの構造

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **ステータスライン**: HTTPバージョン + ステータスコード + 理由フレーズ。
- **ヘッダー**: レスポンスに関するメタデータ（コンテンツタイプ、長さ、キャッシュ指示など）。
- **ボディ**（任意）: リクエストされたデータ。

### HTTPステータスコードのカテゴリ

- **1xx 情報**: リクエスト受信、処理継続中。例：`101 Switching Protocols`。
- **2xx 成功**: `200 OK`、`201 Created`、`204 No Content`。
- **3xx リダイレクト**: `301 Moved Permanently`、`302 Found`、`304 Not Modified`。
- **4xx クライアントエラー**: `400 Bad Request`、`401 Unauthorized`、`403 Forbidden`、`404 Not Found`。
- **5xx サーバーエラー**: `500 Internal Server Error`、`502 Bad Gateway`、`503 Service Unavailable`。

## HTTPリクエスト・レスポンスモデルの理解

- **HTTPメソッド**: `GET`（取得）、`POST`（作成）、`PUT`（完全更新）、`PATCH`（部分更新）、`DELETE`（削除）。
- **リクエスト・レスポンスサイクル**: クライアントがリクエストを送信 → サーバーが認証を検証し、メソッドとURLをチェックし、データベースをクエリ → サーバーがレスポンス（ステータスコード＋任意のボディ）を作成 → クライアントがレスポンスを処理。

### HTTPレスポンスで返されるアセット

すべてのレスポンスには、ブラウザがアセットをどう扱うかを示す`Content-Type`ヘッダーが含まれます：

- **HTML**: `text/html`
- **CSS**: `text/css`
- **JavaScript**: `application/javascript`
- **JSON**: `application/json`
- **PNG/JPEG/WebP**: `image/png`, `image/jpeg`, `image/webp`
- **MP4/MP3**: `video/mp4`, `audio/mpeg`
- **PDF**: `application/pdf`
- **WOFF2**: `font/woff2`

### HTMLフォームの送信

- **`action`属性**: フォームデータを送信するURLを指定します。
- **`method`属性**: HTTPメソッド（`GET`または`POST`）を指定します。HTMLはネイティブで`PUT`、`PATCH`、`DELETE`をサポートしないため、代わりにFetch APIやExpressの`method-override`パッケージを使います。
- **クライアント側バリデーション**: `required`や`pattern`属性を使って即時フィードバックを提供します。サーバー側でも必ず検証してください。
- **`GET`フォーム**: データはURLのクエリパラメータとして追加されます（例：検索フォーム）。
- **`POST`フォーム**: データはリクエストボディに送信されます（例：ログインや登録フォーム）。

## ウェブ標準モデルの理解

### 標準化団体

- **WHATWG**: HTMLとDOMを継続的に更新される「リビングスタンダード」として維持しています。主要ブラウザエンジンチーム（Chromium、Gecko、WebKit）が意思決定を行います。
- **W3C（World Wide Web Consortium）**: 主要な標準化団体で、CSS、SVG、アクセシビリティガイドライン（WCAG）、多数のウェブAPIを監督しています。
- **ECMA International / TC39**: Ecmaは公式のECMAScript仕様を公開し、TC39は5段階の提案プロセス（Stage 0の粗案からStage 4の正式仕様まで）で言語を開発しています。
- **Khronos Group**: グラフィックス、計算、メディアに注力し、ブラウザでGPUアクセラレーションされた3Dレンダリングを可能にするWebGLとWebGL 2を提供しています。

### ウェブ標準の作成

1. **ニーズの特定**: 開発者や組織が問題に直面し、それを書き留めます。
2. **提案と初期議論**: 非公式のGitHubイシューや説明文書を共有して関心を探ります。TC39ではStage 0/Stage 1に相当します。
3. **仕様の作成**: すべてのエッジケース、エラー条件、セキュリティ影響を網羅した正確な正式文書を作成します。
4. **レビュー、異議申し立て、反復**: ブラウザベンダーが公開レビューを行い、主要ブラウザが実装を拒否した機能は標準になりにくいです。
5. **実装とテスト**: ベンダーは機能フラグの背後で実装し、共通のテストスイートで全ブラウザの挙動を検証します。
6. **最終化**: W3CはこれをRecommendationと呼び、TC39はStage 4と呼びます。

### ウェブ標準機能のライフサイクル

1. **提案**: 問題が特定され、文書化されます。
2. **インキュベーション**: オープンな議論が行われ、多くの提案がここで消えます。
3. **仕様**: 正確なルールを持つ正式仕様が書かれます。
4. **相互運用性テスト**: W3C Web Platform Testsスイートでクロスブラウザの一貫性を検証します。
5. **最終化**: フラグなしで標準ウェブプラットフォームの一部として機能が提供されます。
6. **非推奨化**: 置き換えられたか安全でない機能は削除予定とされますが、後方互換性のためブラウザは残します。

### ウェブ標準の主要原則

- **オープン性**: 標準は公開され、ロイヤリティフリーで実装されます。
- **アクセシビリティ**: HTML、CSS、ブラウザの挙動に組み込まれ、WCAGが正式なガイドラインを提供します。
- **相互運用性**: 標準に準拠したページはすべてのブラウザで同じように動作します。
- **後方互換性**: 古いページは最新ブラウザでも動作し続ける必要があります。
- **プライバシーとセキュリティ**: ユーザーデータを公開する可能性のある新APIは厳しい審査を受けます。
- **レイヤリング**: HTMLは構造、CSSは見た目、JavaScriptは動作を担当し、それぞれ独立して採用可能です。
- **プログレッシブエンハンスメント**: まず全員が使えることを保証し、より高機能なブラウザユーザー向けに機能を拡張します。

## REST APIとウェブサービスの理解

### ウェブサービス

- **API（アプリケーションプログラミングインターフェース）**: ソフトウェア同士が通信するための一般的な仕組みです。
- **ウェブサービス**: HTTP/HTTPSなどのウェブプロトコルを使ってネットワーク経由でデータを交換する特定のAPIです。すべてのウェブサービスはAPIですが、すべてのAPIがウェブサービスとは限りません。
- **特徴**: サーバー上で動作し、機械間通信を可能にし、言語やプラットフォームに依存せず、標準的なウェブプロトコルを使います。
- **REST**: 表現状態転送。最も広く使われるウェブサービススタイルで、シンプルで柔軟、HTTPと自然に連携します。
- **SOAP**: シンプルオブジェクトアクセスプロトコル。XMLのみを使い、厳格な標準が必要な企業システムで一般的です。
- **XML-RPC**: 軽量なリクエストとレスポンスをXMLでエンコードします。

### RESTアーキテクチャ

- **ステートレス**: 各リクエストは独立して処理されます。サーバーはリクエスト間でセッション状態を保持しません。すべてのリクエストは必要な情報（例：認証トークン）を含む必要があります。
- **リソース**: すべてはURLで識別されるリソースです。例：コレクションは`/products`、特定アイテムは`/products/1`。
- **RESTのHTTPメソッド**: `GET`（取得）、`POST`（作成）、`PUT`/`PATCH`（更新）、`DELETE`（削除）。
- **レスポンス形式**: JSONが最も一般的ですが、XMLやプレーンテキストもサポートされます。クライアントは`Accept`ヘッダーで形式を指定します：

```bash
Accept: application/json
```

### マイクロサービス

- **モノリス**: UI、データベース、APIが一つの大きなコードベースにまとまっている形態。複雑になるとメンテナンスやスケールが難しいです。
- **マイクロサービス**: 大規模アプリケーションを小さく独立したサービスに分割し、それぞれが単一のビジネス機能を担当します。サービス間はAPIやメッセージキューで通信します。
- **利点**: 独立した開発・デプロイが速い、障害の隔離が良い、異なる言語やフレームワークを使える。
- **欠点**: デバッグが複雑（問題がサービス間にまたがる）、インフラコストが高い、ネットワーク通信が増える。

## Expressの活用

- **Express.js**: Node.js上に構築された最小限で柔軟なウェブフレームワークです。ネイティブの`http`モジュールよりもはるかに少ないボイラープレートでウェブサーバーやAPIを構築できます。
- **基本的なExpressサーバー**：

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**: Expressアプリケーションインスタンス（`app`）を作成するファクトリ関数です。
- **`app.listen(port, callback)`**: サーバーを起動し、指定したポートでHTTPリクエストの待ち受けを開始します。
- **`req`**: リクエストオブジェクトで、URL、ヘッダー、パラメータ、ボディ、クエリデータを含みます。
- **`res`**: レスポンスオブジェクトで、クライアントにデータを返すために使います。
- **ミドルウェア**: リクエスト受信とレスポンス送信の間に実行されるコードです。

## Express.jsのルーティングの理解

- **ルーティング**: クライアントが特定のURLに特定のHTTPメソッドでアクセスしたときにアプリケーションがどう応答するかを定義します。
- **`app.get/post/put/delete(path, handler)`**: それぞれのHTTPメソッドに対するハンドラを登録します。
- **静的ルート**: 固定のURLパターン。例：`app.get("/home", handler)`。
- **パラメータ付き動的ルート**: `:paramName`プレースホルダーを使います。`req.params`で取得可能です：

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**: ルートパラメータ（`:id`など）。
- **`req.query`**: クエリ文字列パラメータ（`?key=value`）。
- **`req.body`**: リクエストボディ（ボディパースミドルウェアが必要）。
- **`res.send(data)`**: レスポンスを送信します（テキスト、HTML、JSON、バイナリ）。
- **`res.json(obj)`**: JSONレスポンスを送信し、自動的に`Content-Type: application/json`を設定します。
- **`res.status(code).send(msg)`**: ステータスコードを設定してレスポンスを送信します。
- **`res.redirect(url)`**: クライアントを別のURLにリダイレクトします。
- **`res.render(view, data)`**: テンプレートエンジンのビューをレンダリングしてHTMLを送信します。

### `app.route()`によるチェーン可能なルートハンドラ

同じパスに対して複数のHTTPメソッドをグループ化し、繰り返しを減らします：

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

関連するルートをまとめるモジュール化されたミニアプリケーションを作成します：

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

### 静的ファイルの提供

- **`express.static(directory)`**: 指定したディレクトリからファイルを提供する組み込みミドルウェアです：

```js
app.use(express.static(path.join(__dirname, "public")));
```

- マウントパスを使って複数ディレクトリから提供：`app.use("/images", express.static(path.join(__dirname, "images")));`

## エラーハンドリングとヘルスチェックの理解

### Expressのエラーハンドリング

- **エラーハンドリングミドルウェア**: 4つのパラメータ`(err, req, res, next)`で識別され、ミドルウェアスタックの最後に配置します：

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **Express 4の非同期エラー**: `try/catch`で手動キャッチし、`next(err)`で転送する必要があります：

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **Express 5の非同期エラー**: 自動的にエラーハンドリングミドルウェアに転送され、手動で`next(err)`を呼ぶ必要はありません。
- **404ハンドラ**: すべてのルートの後、エラーハンドラの前に配置します：

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### デバッグとログ記録

- **組み込みデバッグ**: `DEBUG`環境変数を設定して有効化します：`DEBUG=express:* node index.js`
- **`morgan`**: 人気のあるサードパーティのHTTPリクエストロガーミドルウェア。メソッド、URL、ステータスコード、レスポンスタイムをログに記録します：

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### ヘルスチェックとグレースフルシャットダウン

- **ヘルスチェックルート**: アプリが稼働中であることを示すために `200 OK` を返します。ロードバランサーやコンテナオーケストレーターで使用されます。

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**: プロセスを終了すべきときに送られるシグナルです。進行中のリクエストが完了できるように処理してください。

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**: ローカル開発中にCtrl+Cを押したときに送られるシグナルです。`SIGTERM` と同様に処理してください。
- **シャットダウン前の非正常状態マーク**: シャットダウン中にヘルスチェックエンドポイントから `503` を返して、ロードバランサーが新しいトラフィックをルーティングしないようにします。

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## Express ミドルウェア

- **ミドルウェア**: リクエストのライフサイクル中に実行される関数です。`req`、`res`、`next` にアクセスできます。`next()` を呼び出すと次のミドルウェアに制御が渡り、省略すると処理が終了します。
- **アプリケーションレベルミドルウェア**: `app.use()` でアプリ全体にバインドされ、すべてのリクエストで実行されます。
- **ルーター レベルミドルウェア**: 特定の `express.Router()` インスタンスに `router.use()` でバインドされ、そのルーターが処理するルートにのみ適用されます。
- **エラーハンドリングミドルウェア**: 4つのパラメータ `(err, req, res, next)` で識別され、スタックの最後に配置されます。
- **`express.json()`**: JSON ペイロードのリクエストをパースし、`req.body` でデータを利用可能にします。
- **`express.urlencoded({ extended: true })`**: URLエンコードされたフォームデータをパースします。
- **`express.static('public')`**: 指定ディレクトリから静的ファイルを配信します。
- **`cors`**: クロスオリジンリソースシェアリングを有効にします。`npm install cors` でインストールし、`app.use(cors())` で使用します。
- **`morgan`**: HTTPリクエストをログに記録します。`app.use(morgan('tiny'))` で利用します。
- ミドルウェアは `app.use()` で登録した順に適用されます。

## WebSockets の理解

- **WebSockets**: クライアントとサーバー間に持続的な全二重（双方向）接続を作り、リアルタイム通信を可能にするプロトコルです。
- **リアルタイムにおけるHTTPの問題点**: HTTPはクライアントがサーバーに繰り返しポーリングする必要があり、チャットアプリやライブスコアボード、多人数ゲームには遅く非効率です。
- **WebSocketハンドシェイク**: すべてのWebSocket接続はHTTPリクエストから始まります。クライアントがアップグレードを要求し、サーバーが `101 Switching Protocols` で応答します。その後、どちらかが切断信号を送るまで接続は開いたままです。
- **利用例**: チャットアプリ、オンライン多人数ゲーム、ライブスポーツダッシュボード、株式市場フィード、共同作業ツール。

### `ws` ライブラリ

- Node.jsには組み込みのWebSocketサーバーがないため、`ws` ライブラリが一般的な実装方法です。

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**: 特定のクライアントにメッセージを送信します。
- **接続中のすべてのクライアントへのブロードキャスト**:

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**: すべての接続にあるプロパティです。`WebSocket.OPEN` は接続がアクティブであることを示します。

### Pub/Sub アーキテクチャ

- **Pub/Sub（パブリッシュ/サブスクライブ）**: **パブリッシャー**がメッセージを**トピック**に送信し、**サブスクライバー**が購読したトピックからメッセージを受け取るメッセージングパターンです。パブリッシャーとサブスクライバーは直接通信しません。
- **直接通信との利点**: コンポーネントの結合を緩め、パブリッシャーはサブスクライバーを知らなくてもよいです。
- **Pub/SubとWebSocketsの違い**: WebSocketsは通信チャネル（道路）であり、Pub/Subはどのメッセージを誰が受け取るかを決めるメッセージングパターン（交通システム）です。

## 認証と認可の紹介

### 認証と認可の違い

- **認証**: 身元を確認し、「あなたは誰ですか？」という質問に答えます。認可の前に行う必要があります。
- **認可**: 権限を決定し、「何が許可されていますか？」という質問に答えます。
- **3つの認証要素**:
  - **知っているもの**: パスワード、PIN、秘密の質問の答え。
  - **持っているもの**: SMSコードを受け取るスマートフォン、認証アプリのトークン。
  - **本人であるもの**: 指紋、顔認証（生体認証）。
- **認可の一般的な役割**: 標準ユーザー（限定的なセルフサービス）、編集者（コンテンツ管理）、管理者（システム全体アクセス）。
- バックエンドコードでは、認可は通常、ユーザーの役割や権限をチェックするミドルウェアで強制されます。

### クロスサイトリクエストフォージェリ（CSRF）

- **CSRF**: 悪意のあるウェブサイトが認証済みユーザーのブラウザを騙して、バックエンドサーバーに望まないリクエストを送らせる攻撃です。ブラウザの自動クッキー送信を悪用します。
- **攻撃の流れ**: ユーザーが `trusted-bank.com` にログイン → `evil-hacker-site.com` を訪問 → 悪意のあるサイトが銀行に隠しフォームを送信 → 銀行サーバーは有効なセッションCookieを見てリクエストを実行。
- **`SameSite=Strict`**: リクエストが第三者サイト由来の場合、ブラウザはクッキーを送信しません。
- **`SameSite=Lax`**: ユーザーが安全に対象サイトに遷移した場合（リンククリックなど）にのみクッキーを許可し、クロスサイトフォーム送信ではブロックします。
- **CSRFトークン**: サーバーがランダムで予測不可能なトークンをフォームに埋め込みます。クライアントは状態変更リクエストに必ずトークンを含める必要があります。悪意のあるサイトはフロントエンドからトークンを読み取れないため、偽造リクエストはバリデーションに失敗します。

### JSON Web Token (JWT)

- **JWT**: ログイン成功後にサーバーが発行するデジタル署名付きトークンです。クライアントはそれを保存し、以降のリクエストに添付してステートレス認証を実現します（サーバー側のセッション保存不要）。
- **JWTの構造**: ドットで区切られた3つのBase64URLエンコード部分から成ります。
  - **ヘッダー**: トークンタイプ（`JWT`）と署名アルゴリズム（例：`HS256`）。
  - **ペイロード**: ユーザーに関するクレーム（ユーザーID、ユーザー名、役割）。暗号化されていないため、パスワードや機密情報は絶対に入れないでください。
  - **署名**: ヘッダー＋ペイロード＋サーバー秘密鍵のハッシュ。改ざん防止に使われます。

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

- ユーザーがペイロードを改ざんすると署名が一致せず、サーバーは `401 Unauthorized` でリクエストを拒否します。

### Passport.js

- **Passport.js**: Node.js用の認証ミドルウェアです。認証フローを標準化し、低レベルの処理を扱うため、開発者はビジネスロジックに集中できます。
- **ストラテジー**: 特定の認証方法を実装するモジュールプラグインです。500以上のストラテジーがあり、`passport-local`（ユーザー名/パスワード）、`passport-jwt`（JWT）、`passport-google-oauth20`（Googleログイン）、`passport-facebook`、`passport-github2`などがあります。
- **Expressへの組み込み**:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **ルートミドルウェアとしての使用**:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- 認証成功後、PassportはセッションCookieにユーザーデータの一部を保存し、認証済みユーザーオブジェクトを `req.user` に添付します。

### Helmet.js

- **Helmet.js**: Expressアプリを保護するセキュリティミドルウェアのコレクションで、HTTPレスポンスヘッダーを自動的に設定、変更、削除します。

```js
const helmet = require('helmet');
app.use(helmet());
```

- **`X-Powered-By` を削除**: 攻撃者にサーバー技術を知られないようにします。
- **`Content-Security-Policy` を設定**: スクリプト、画像、スタイルの読み込み元を制限し、XSS攻撃を軽減します。
- **`X-Frame-Options: SAMEORIGIN` を設定**: クリックジャッキングを防ぎ、外部iframeへの埋め込みをブロックします。
- **`Strict-Transport-Security` を設定**: ブラウザにHTTPS使用を強制し、データの傍受を防ぎます。

## ウェブアプリケーションにおけるセキュリティとプライバシーの理解

### セキュリティとプライバシーの違い

- **セキュリティ**: システムやデータを不正アクセスから守る技術的防御手段です。セキュリティはドアの鍵に例えられます。
- **プライバシー**: 個人データの収集、利用、共有の管理です。プライバシーは窓のカーテンに例えられます。
- セキュリティがあってもプライバシーがない場合があります（鍵のかかったデータベースがデータを販売するなど）が、プライバシーには必ずセキュリティが必要です（保護されていないデータはプライベートに保てません）。

### Same-Origin Policy と CORS

- **オリジン**: プロトコル（例: `https`）、ドメイン（例: `example.com`）、ポート（例: `443`）の3つの要素で定義されます。これら3つすべてが一致する場合にのみ、2つのURLは同じオリジンとみなされます。
- **同一オリジンポリシー**: ブラウザのセキュリティルールで、あるオリジンのスクリプトが別のオリジンのデータを読み取ることをブロックし、悪意のあるサイトが他のタブのデータを盗むのを防ぎます。
- **CORS（Cross-Origin Resource Sharing）**: サーバーがHTTPヘッダーを設定して信頼できるオリジンに対して同一オリジンポリシーを緩和できる仕組みです。
- **Express（開発環境）での設定**: `app.use(cors())` はすべてのオリジンを許可します。
- **Express（本番環境）での設定**: `corsOptions` オブジェクトに `origin` の許可リストを渡してください。

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**: TLS（Transport Layer Security）による暗号化を施したHTTPです。TLSは旧SSLプロトコルの後継です。
- **HTTPSの3つの保証**:
  - **暗号化**: クライアントとサーバー間のデータが第三者に読まれません。
  - **認証**: 信頼された認証局からのデジタル証明書により、サーバーの正当性が確認されます。
  - **データ整合性**: 途中でメッセージが改ざんされていないことを保証します。
- HTTPSは位置情報、プッシュ通知、カメラアクセスなどの最新ブラウザAPIに必須です。

### なぜセキュリティ問題が起こるのか

- **急速な納品プレッシャー**: 期限に間に合わせるためにセキュリティレビューを省略し、脆弱性を含んだままリリースするチームがあります。
- **複雑なシステム統合**: サードパーティのライブラリやクラウドサービスが開発チームの知らない脆弱性を持ち込みます。
- **設定管理の不備**: デフォルトの管理者パスワードや開放されたポートなど、初期設定のまま放置するとシステムが不用意に露出します。
- **人的ミス**: フィッシング攻撃でスタッフが悪意あるリンクや添付ファイルをクリックし、認証情報を渡してしまいます。
- **組織合併**: 異なるコードベースやインフラを統合すると、セキュリティポリシーの不整合やレガシー脆弱性が表面化します。

### クッキー：保存、セキュリティフラグ、プライバシー

- **クッキーの保存方法**: サーバーが `Set-Cookie` ヘッダーを送信し、ブラウザがディスクに保存して以降の同一オリジンへのリクエストに自動で返します。
- **`HttpOnly` フラグ**: JavaScriptによるクッキー値の読み取りをブロックし、XSSによるセッション盗難を防ぎます。
- **`Secure` フラグ**: HTTPS接続時のみクッキーを送信し、公共ネットワークでの傍受を防ぎます。
- **`SameSite=Strict`**: クロスサイトリクエスト時にクッキーを送信せず、CSRF攻撃を完全に防ぎます。
- **`SameSite=Lax`**: トップレベルのナビゲーション（リンククリックなど）ではクッキーを許可し、クロスサイトフォーム送信やAPI呼び出しではブロックします。
- **ファーストパーティクッキー**: 訪問中のサイトにスコープされます。
- **サードパーティクッキー**: 広告ネットワークが多数のサイトでユーザー行動を追跡しプロファイルを作成します。現代ブラウザはデフォルトでこれをブロックします。
- **GDPR準拠**: プライバシー法は、ユーザーが非必須の追跡クッキーを拒否できる同意バナーの表示を義務付けています。

### 一般的なセキュリティ脅威と対策

- **フィッシング**: 信頼された企業を装う詐欺メールで認証情報を盗みます。対策はセキュリティ意識向上トレーニングと、悪意あるメールを受信箱に届く前にブロックするメール認証プロトコルの導入です。
- **マルウェア**（ウイルス、スパイウェア、ランサムウェア）: ランサムウェアはファイルを暗号化し解除のための身代金を要求します。対策は自動アップデートとアンチマルウェアソフトの利用です。
- **クレデンシャルスタッフィング**: 盗まれた大量のユーザー名とパスワードの組み合わせを複数サイトで試す攻撃です。主な対策は多要素認証（MFA）で、OTPや認証アプリコードなどの二段階認証を要求します。
- **DDoS（分散型サービス拒否）**: 偽トラフィックでサーバーを過負荷にし、正規ユーザーをブロックします。対策はCloudflare、Akamai、AWS Shieldなどのクラウドベースのトラフィックフィルタリングサービスです。
- **内部脅威**: 悪意あるまたは不注意な従業員によるデータの盗難や漏洩。対策は**最小権限の原則**の徹底（業務に必要な範囲のみアクセス許可）と異常な活動を検知するログ監視です。

### コンテンツセキュリティポリシーとPermissions-Policy

- **CSP（Content Security Policy）**: スクリプト、スタイル、画像の許可されたソースドメインをホワイトリスト化するHTTPレスポンスヘッダーです。ブラウザはCSPヘッダーを受け取るとリスト外のドメインからのリソース読み込みをブロックします。`script-src` 指令はJavaScript実行を許可するドメインを指定し、XSS攻撃を防ぎます。
- **Permissions-Policy**: ページおよび埋め込みiframeのブラウザ機能やデバイスハードウェアの使用を有効化または無効化するHTTPヘッダーです。旧 `Feature-Policy` ヘッダーの後継です。
  - `camera=(self)` はウェブカメラアクセスをページ自身のオリジンに制限します。
  - `geolocation=()` はすべてのユーザーに対して位置情報を完全に無効化します。
- 両ヘッダーを組み合わせることで多層的な保護を実現します。CSPはコードインジェクションを防ぎ、Permissions-Policyは不正なハードウェアアクセスを防ぎます。

### 基本的なユーザープライバシーの概念

- **PII（個人識別情報）**: 個人を直接または間接的に特定できるあらゆるデータ。
  - **直接識別子**: 氏名、住所、国民ID番号。
  - **間接識別子**: IPアドレス、デバイスのシリアル番号、正確な位置履歴。
- **機密性**: 暗号化やアクセス制御によりPIIを不正アクセスから守ること。
- **データ最小化**: 特定のタスクを完了するために必要最低限のPIIのみを収集すること。
- **目的制限**: 収集したデータは別の目的（例：広告ネットワークへの販売）に無断で使えません。新たなユーザー同意が必要です。
- **トラッキング**: スクリプト、デバイスフィンガープリント、ブラウザ履歴ログを使ったユーザー行動の継続的監視。
- **忘れられる権利（データ消去）**: アカウント閉鎖後に企業のデータベースからすべての個人記録の永久削除を要求できる法的権利。

### `.env` ファイルと `dotenv`

- **`.env` ファイル**: プロジェクトルートにあるプレーンテキストファイルで、APIキー、データベースパスワード、ポート番号などの設定秘密情報をキーと値のペアで保存します。バージョン管理から秘密情報を除外するために必ず `.gitignore` に追加してください。
- **`dotenv` パッケージ**: 起動時に `.env` ファイルを読み込み、値を `process.env` に注入します。

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **2つの主な利点**: 資格情報がソースコードから分離されてセキュリティが向上し、開発、ステージング、本番など環境ごとに `.env` ファイルを切り替えることで同じコードベースが異なる動作をする柔軟性が得られます。

### 地域ごとのプライバシー法とコンプライアンス

- **GDPR（一般データ保護規則）**: EUの個人を保護します。追跡前に明示的なオプトイン同意を要求し、事前チェックボックスは禁止です。ユーザーにデータ閲覧と永久削除の権利を与え、違反には重い罰金があります。
- **CCPA（カリフォルニア消費者プライバシー法）/ CPRA（カリフォルニアプライバシー権法）**: カリフォルニア州民を保護します。オプトアウト権に重点を置き、ホームページに「個人情報を販売しない」リンクの表示を義務付けます。
- **COPPA（子どものオンラインプライバシー保護法）**: 13歳未満の子どもを保護する米国連邦法です。明確なプライバシーポリシーと保護者の検証可能な同意を要求します。
- **DPA（データ保護法）**: 英国の主要なデータプライバシーフレームワークで、英国GDPRと連携し、ユーザーに個人データのアクセスと削除の権利を与えます。
- 他国にも独自の法律があります：ナイジェリアのNDPA（ナイジェリアデータ保護法）、インドのDPDP法（デジタル個人データ保護法）など。
- **グローバルチームのコンプライアンス手順**:
  - ユーザーの地理的位置に基づいて適切なクッキーノーティスを表示する位置認識型同意バナーを構築する。
  - 法的要求に応じてユーザーデータを検索、エクスポート、永久消去する自動DSAR（データ主体アクセス要求）システムを実装する。
  - データ最小化を実践し、未成年者の無断追跡を防ぐ年齢認証ゲートを設ける。
  - プライバシーアーキテクチャを監督し、規制当局の主要な連絡先となるデータ保護責任者（DPO）を任命する。
