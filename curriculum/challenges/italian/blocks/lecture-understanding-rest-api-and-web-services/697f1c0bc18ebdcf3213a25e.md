# --description--

**REST** sta per trasferimento di stato rappresentazionale. Pensalo come un insieme di linee guida che descrivono come client e server web dovrebbero comunicare tra loro su internet.

In questa lezione, imparerai cos’è REST, i suoi concetti chiave e come funziona.

REST definisce un modo per far interagire sistemi diversi usando tecnologie web standard. Con REST, client come browser web, app mobili e API comunicano con un server usando metodi HTTP comuni come `GET`, `POST`, `PUT`, `PATCH` e `DELETE`, di cui abbiamo parlato in una lezione precedente.

Alcuni concetti fondamentali di REST includono **assenza di stato**, **risorse**, **metodi HTTP** e **codici di risposta o di stato**.

L’**assenza di stato** significa che ogni richiesta che un client invia al server viene gestita in modo indipendente. Il server non ricorda le richieste precedenti. Per questo motivo, ogni richiesta deve includere tutte le informazioni necessarie per essere elaborata. Per esempio, se un utente è autenticato, il suo token di autenticazione deve essere inviato con ogni richiesta.

In REST, tutto è trattato come una risorsa. Una risorsa è un pezzo di dati che può essere consultato o modificato. Ogni risorsa è identificata da un URL, come `/products` per tutti i prodotti, o `/products/1` per un prodotto specifico.

Qui sotto trovi una panoramica dei metodi HTTP, dei codici di stato e un esempio passo dopo passo di come funziona l’architettura REST.

## Metodi HTTP

I metodi HTTP definiscono l’azione da eseguire su una risorsa:

* `GET`: Recuperare una risorsa
    
* `POST`: Creare una nuova risorsa
    
* `PUT / PATCH`: Aggiornare una risorsa esistente
    
* `DELETE`: Rimuovere una risorsa
    

## Codici di stato HTTP

Dopo aver elaborato una richiesta, il server risponde con un **codice di stato** che indica il risultato:

* `200` **(OK)**: La richiesta è stata eseguita con successo
    
* `201` **(Created)**: È stata creata una nuova risorsa
    
* `400` **(Bad Request)**: Il client ha inviato dati non validi
    
* `404` **(Not Found)**: La risorsa richiesta non esiste
    
* `500` **(Internal Server Error)**: Si è verificato un errore sul server
    

## Come funziona REST: un esempio passo dopo passo

### 1\. Il client invia una richiesta

* Il client invia una richiesta HTTP usando un metodo come `GET` o `POST`
    
* La richiesta include un URL che identifica la risorsa
    
* Può includere anche intestazioni, metadati e un corpo della richiesta
    

### 2\. Il server riceve la richiesta

* Il server verifica il formato della richiesta e l’autenticazione (se richiesta)
    
* Identifica la risorsa obiettivo dall’URL
    
* Determina l’azione richiesta in base al metodo HTTP
    
* Controlla i permessi e il controllo degli accessi
    

### 3\. Il server esegue l’azione

* `GET`: Recupera la risorsa
    
* `POST`: Crea una nuova risorsa
    
* `PUT / PATCH`: Aggiorna la risorsa
    
* `DELETE`: Rimuove la risorsa
    
* Esegue qualsiasi altra logica necessaria
    

### 4\. Il server costruisce la risposta

* Il server seleziona un codice di stato appropriato (per esempio `200`, `201` o `404`)
    
* Formattta i dati della risposta, di solito in **JSON**
    
* Aggiunge intestazioni rilevanti come `Content-Type` o `Cache-Control`
    

### 5\. Il server invia la risposta

* La risposta include un codice di stato, intestazioni e un corpo opzionale
    
* Nessuno stato del client viene archiviato sul server (assenza di stato)
    

### 6\. Il client gestisce la risposta

* Il client controlla il codice di stato per determinare successo o fallimento
    
* Elabora eventuali dati restituiti dal server
