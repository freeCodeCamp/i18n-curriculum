# --description--

Давайте навчимося знаходити або замінювати всі входження шаблону в рядку.

Ви вже вивчали методи `replace()` та `match()`, а також глобальний модифікатор `g`. Тепер ви можете поєднати ці знання, щоб працювати з усіма шаблонами в рядку. Нагадаємо наш початковий код для пошуку:

```js
const regex = /freeCodeCamp/;
const match = "freeCodeCamp".match(regex);
console.log(match);
```

І наш об’єкт результату пошуку:

```js
// [
//   'freeCodeCamp',
//   index: 0,
//   input: 'freeCodeCamp',
//   groups: undefined
// ]
```

Але що, якщо у нас є рядок із кількома входженнями `freecodecamp`, які потрібно знайти? Подивімося, як поводиться `match()` у такому випадку. Також додамо наш старий приклад `replace()` для порівняння:

```js
const regex = /freecodecamp/;
const str = "freecodecamp is the best we love freecodecamp";
const matched = str.match(regex);
const replaced = str.replace(regex, "freeCodeCamp");
console.log(matched);
console.log(replaced);
```

І ось результат:

```js
// [
//   'freecodecamp',
//   index: 0,
//   input: 'freecodecamp is the best we love freecodecamp',
//   groups: undefined
// ]
// freeCodeCamp is the best we love freecodecamp
```

О ні! `match()` повернув лише перше входження, а `replace()` замінив лише перше входження. Це тому, що за замовчуванням `match()` та `replace()` працюють лише з першим входженням шаблону.

На щастя, ви можете уникнути цього, використовуючи глобальний модифікатор у вашому регулярному виразі. Додамо його до нашого:

```js
const regex = /freecodecamp/g;
const str = "freecodecamp is the best we love freecodecamp";
const matched = str.match(regex);
const replaced = str.replace(regex, "freeCodeCamp");
console.log(matched);
console.log(replaced);
```

І перевіримо результат:

```js
// [ 'freecodecamp', 'freecodecamp' ]
// freeCodeCamp is the best we love freeCodeCamp
```

Це спрацювало! Наш виклик replace замінив усі рядки з нижнім регістром `freecodecamp`, а метод `match()` знайшов обидва.

Цікаво, що коли ви використовуєте глобальний модифікатор з `match()`, ви втрачаєте додаткову інформацію про групи захоплення та індекси рядка, яка зазвичай є в масиві результатів пошуку.

На щастя, оновлення ECMAScript 2019 принесло нам два нові методи: `matchAll()` та `replaceAll()`. Як і їхні одиничні аналоги, ці методи приймають рядок або регулярний вираз, а `replaceAll()` також приймає другий аргумент — рядок для заміни.

Але на відміну від попередніх методів, `replaceAll()` та `matchAll()` викинуть помилку, якщо ви передасте їм регулярний вираз без глобального модифікатора. Оновимо наш код, щоб використати ці нові методи:

```js
const pattern = "freecodecamp";
const str = "freecodecamp is the best we love freecodecamp";
const matched = str.matchAll(pattern);
const replaced = str.replaceAll(pattern, "freeCodeCamp");
console.log(matched);
console.log(replaced);
```

І наш результат:

```js
// {}
// freeCodeCamp is the best we love freeCodeCamp
```

Добрі новини! Наш `replaceAll()` спрацював саме так, як ми хотіли — він замінив усі входження рядка з нижнім регістром `freecodecamp` на правильну версію у верблюжому регістрі.

Але що це за порожній об’єкт? Насправді `matchAll()` повертає спеціальний тип об’єкта, який називається `Iterator`, з яким консоль freeCodeCamp не готова працювати.

Якщо заглянути в консоль браузера, `Iterator` має метод `next()`, який ми можемо викликати, щоб отримати наступне значення:

```js
// RegExpStringIterator {  }
//  <prototype>: RegExp String Iterator {
//    next: ƒ next(),
//    Symbol(Symbol.toStringTag): "RegExp String Iterator"
//    <prototype>: Object { ... }
//  }
```

Давайте викличемо `matched.next()` і виведемо результат:

```js
// {
//   "done": false,
//   "value": [
//     0: "freecodecamp"
//     groups: undefined
//     index: 0
//     input: "freecodecamp is the best we love freecodecamp"
//   ]
// }
```

Ось наш масив результатів пошуку! `next()` повертає об’єкт із двома значеннями: `done`, яке є `false`, коли в ітераторі є ще елементи, і `value`, що є значенням, яке ми щойно отримали. Тож, якщо викликати його ще раз:

```js
const regex = /freecodecamp/g;
const str = "freecodecamp is the best we love freecodecamp";
const matched = str.matchAll(regex);
const replaced = str.replaceAll(regex, "freeCodeCamp");
console.log(matched);
console.log(replaced);
console.log(matched.next());
console.log(matched.next());
```

```js
// {
//   "done": false,
//   "value": Array [ "freecodecamp"]
// }
//
// {
//   "done": false,
//   "value": Array [ "freecodecamp"]
// }
```

Чому ж тоді `done` все ще `false`? Адже у масиві має бути лише два входження, правда? Викличемо його втретє і подивимося, що отримаємо:

```js
// {
//   "done": false,
//   "value": Array [ "freecodecamp"]
// }
//
// {
//   "done": false,
//   "value": Array [ "freecodecamp"]
// }
//
// {
//   "done": true,
//   "value": undefined
// }
```

`done` нарешті `true`, але чому це значення `undefined`? Виявляється, ітератор `matchAll()` є лінивим. Він не знаходить усі входження одразу. Він знаходить входження лише тоді, коли ви викликаєте `next()`.

Поки він знаходить входження, він не є `done`. Коли ж не може знайти входження і повертає `undefined`, він стає `done`. Це може здатися незручним, але дуже корисно, коли ваш регулярний вираз є обчислювально дорогим.

Якщо ваш приклад менш складний, як наш, ви можете пропустити цю функцію і отримати всі входження одразу, перетворивши ітератор на масив. Це робиться викликом `Array.from()` з вашим ітератором як аргументом.

Оновимо наш код, щоб використати це — приберемо зайві виклики `replaceAll`, бо ми знаємо, що це працює:

```js
const regex = /freecodecamp/g;
const str = "freecodecamp is the best we love freecodecamp";
const matched = str.matchAll(regex);
console.log(Array.from(matched));
```

І нарешті отримаємо масив входжень:

```js
// [
//   'freecodecamp',
//   index: 0,
//   input: 'freecodecamp is the best we love freecodecamp',
//   groups: undefined
// ]
//
// [
//   'freecodecamp',
//   index: 33,
//   input: 'freecodecamp is the best we love freecodecamp',
//   groups: undefined
// ]
```

Ці потужні методи допоможуть вам маніпулювати та витягувати дані з рядків, не жертвуючи продуктивністю чи читабельністю.
