# --description--

前のレッスンでは、自分のリポジトリにプルリクエストを作成する方法を学びました。また、そのプルリクエストをマージする方法も学びました。しかし、それができたのはリポジトリが自分のものであり、自分のリポジトリにはメンテナ―アクセス権があるため、プルリクエストをマージする権限があったからです。

しかし、他の人のオープンソースプロジェクトに貢献する場合は、通常そのような権限はありません。代わりに、メンテナ―がコードレビューのプロセスを経てあなたの貢献を受け入れるのを待つ必要があります。そして、マージはメンテナ―が行います。

では、コードレビューのプロセスはどのようなものなのでしょうか？

メンテナ―によってプロセスは異なり、プロジェクトごとに特有の方法がある場合もあります。しかし一般的には、コードレビューには通常3つのステップがあります。

最初のステップは、CI（継続的インテグレーション）が通っているかを確認することです。これは今後のレッスンで詳しく学びますが、今はコードがスタイルガイドに準拠しているか、テストスイートを通過しているかを検証するチェックだと考えてください。

もしCIが失敗した場合、メンテナ―は通常、失敗を解決するためにプルリクエストの更新を依頼します。失敗には何が問題だったかを確認できるレポートが含まれているはずですが、メンテナ―がより具体的な解決方法を教えてくれる場合もあります。

チェックが通ったら、メンテナ―は実際のコード変更を確認します。差分ビューを見て、どこを修正、追加、削除したかを分析します。なぜその方法を選んだのか質問されることもあるので、会話の準備は慎重かつ丁寧に行うのが良いです。

追加の修正を依頼されることもあります。これについては後ほど説明します。

最後のステップは、メンテナ―があなたの変更をローカルに取り込みます。変更を反映したアプリケーションを実行し、機能をテストし、何も壊れていないか、予期しない動作がないかを確認します。

すべて問題なければ、メンテナ―がプルリクエストを承認します。時には「LGTM」というコメントが付くこともあります。これは「looks good to me」の略で「問題ない」という意味です。プルリクエストが承認されたら、メンテナ―が準備でき次第マージされます。プロジェクトによっては複数の承認が必要な場合もあります。例えば freeCodeCamp では2つの承認が必要です。

もしプルリクエストに問題があった場合はどうなるでしょうか？メンテナ―は変更を依頼するレビューを行うことがあります。これにはいくつかの形態があります。

時には、メンテナ―が「コメント」として変更を求めることがあります。これは何を変えるべきかを示すもので、実装方法はある程度自由です。

また、メンテナ―が正式に「変更をリクエスト」するコメントを残すこともあります。通常、このコメントがあるとプルリクエストはマージできません。望ましい変更を行い、変更をリクエストした同じメンテナ―が再度レビューし、さらに変更を求めるか承認します。

メンテナ―が具体的な変更案をGitHubのUIを通じて提案することもあります。こうした直接的な提案がある場合、GitHubは提案を自分のブランチに直接コミットするボタンを提供します。しかし、複数の提案がある場合はファイルビューに切り替えるのがお勧めです。

ファイルビューでは、提案を一つずつ「バッチ」に追加し、一括でコミットできます。提案が多い場合に非常に便利です。個別にコミットするとコミット数が増え、CIチェックがすべてのコミットで実行されるため、プルリクエストの承認が遅くなります。

もし依頼された変更が大きい場合は、ローカルで変更を行い、プルリクエストで使った同じブランチにコミットしてプッシュするのが良いでしょう。GitHubは自動的にプルリクエストを新しいコミットに更新します。

他にいくつか注意点があります。

まず、メンテナ―も人間であることを忘れないでください。オープンソースプロジェクトはメンテナ―がフルタイムで取り組めるほどの収益を生み出すことは稀で、多くは別の仕事を持っています。レビューの過程で何度もメンテナ―に再レビューを催促したり、急かしたりしないようにしましょう。

貢献はワクワクする体験です！でも急がずにじっくり進めてください。ローカルで変更をテストし、プロジェクトのテストスイートを実行し、開発サーバーを起動して手動で動作を確認するなど、しっかり確認しましょう。そして、必ず作業しているプロジェクトの貢献ガイドラインに従ってください。

これらを心に留めて最善を尽くせば、貢献はスムーズに進み、コードレビューも確実に承認されるはずです！
