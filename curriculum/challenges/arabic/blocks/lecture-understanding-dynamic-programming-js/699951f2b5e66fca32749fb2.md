# --description--
  
البرمجة الديناميكية هي تقنية خوارزمية تحل المشكلات المعقدة بتقسيمها إلى مشكلات فرعية أبسط وتخزين النتائج لتجنب الحسابات المتكررة. هذا الأسلوب يحول المشكلات التي عادةً ما تستغرق وقتًا أُسّيًا إلى مشكلات يمكن حلها في وقت كثير الحدود.

## المبادئ الأساسية للبرمجة الديناميكية

تعمل البرمجة الديناميكية عندما تتوفر شرطان رئيسيان في المشكلة.

- **المشكلات الفرعية المتداخلة**: تظهر نفس المشكلات الصغيرة عدة مرات عند حل المشكلة الأكبر. بدلاً من إعادة حساب هذه المشكلات الفرعية مرارًا، نخزن حلولها.
    
- **البنية الفرعية المثلى**: الحل الأمثل للمشكلة يحتوي على حلول مثلى لمشكلاتها الفرعية. هذا يعني أننا نستطيع بناء أفضل حل من خلال دمج أفضل الحلول للأجزاء الأصغر.
    

دعنا نفحص هذه المفاهيم باستخدام مشكلة "تسلق الدرج" الكلاسيكية.

## المشكلة مع الاستدعاء العودي البسيط

فكر في مشكلة تسلق الدرج: أنت تتسلق سلمًا به `n` درجة ويمكنك تسلق درجة واحدة أو درجتين في كل مرة. كم عدد الطرق المختلفة التي يمكنك بها الوصول إلى القمة؟

```js
function climbStairsRecursive(n) {
  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Recursive rule: to reach step n, you can come from step (n-1) or step (n-2)
  return climbStairsRecursive(n - 1) + climbStairsRecursive(n - 2);
}
```

هذا التنفيذ له تعقيد زمني أُسّي بسبب الحسابات المتكررة الضخمة. عند حساب `climbStairsRecursive(5)`، يحدث ما يلي:

- `climbStairsRecursive(5)` يستدعي `climbStairsRecursive(4)` و `climbStairsRecursive(3)`
    
- `climbStairsRecursive(4)` يستدعي `climbStairsRecursive(3)` و `climbStairsRecursive(2)`
    
- الآن يتم حساب `climbStairsRecursive(3)` **مرتين**
    
- `climbStairsRecursive(3)` يستدعي `climbStairsRecursive(2)` و `climbStairsRecursive(1)`
    
- يتم حساب `climbStairsRecursive(2)` **3 مرات إجمالاً**
    

لمجرد `n=5`، نجري 9 استدعاءات دالة بينما نحتاج فقط إلى 5 حسابات فريدة. مع زيادة `n`، تتضاعف هذه التكرارات أُسّيًا - `climbStairsRecursive(30)` سيحتاج إلى ملايين الاستدعاءات! يصبح التعقيد الزمني `O(2^n)`، مما يجعله غير فعال وغير عملي للقيم الأكبر من `n`.

## حلول البرمجة الديناميكية

تُزيل البرمجة الديناميكية هذا الحساب المتكرر من خلال نهجين رئيسيين:

### التذكر (النهج من الأعلى إلى الأسفل)

التذكر يخزن نتائج استدعاءات الدوال المكلفة ويُعيد النتيجة المخزنة عند تكرار نفس المُدخلات:

```js
function climbStairsMemo(n, memo = {}) {
  // If already calculated, return cached value
  if (memo[n] !== undefined) {
    return memo[n];
  }

  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Calculate once and store in memo for future use
  memo[n] =
    climbStairsMemo(n - 1, memo) +
    climbStairsMemo(n - 2, memo);

  return memo[n];
}
```

التذكر أكثر كفاءة بكثير لأن كل قيمة فريدة من `1` إلى `n` تُحسب مرة واحدة فقط. عندما نحتاج `climbStairsMemo(3)` مرة أخرى، بدلاً من إعادة حسابها (مما يؤدي إلى المزيد من الاستدعاءات العودية)، ننظر إليها ببساطة في قاموس التذكر لدينا في وقت `O(1)`.

دعنا نتتبع تنفيذ `climbStairsMemo(5)` باستخدام النهج من الأعلى إلى الأسفل لنرى كيف يُزيل التذكر العمل المتكرر:

```md
Call: climbStairsMemo(5)
  memo = {} (empty)
  
  Call: climbStairsMemo(4) 
    memo = {} (empty)
    
    Call: climbStairsMemo(3)
      memo = {} (empty)
      
      Call: climbStairsMemo(2) → returns 2 (base case)
      Call: climbStairsMemo(1) → returns 1 (base case)
      
      Result: 2 + 1 = 3
      memo = {3: 3} (stored!)
    
    Call: climbStairsMemo(2) → returns 2 (base case)
    
    Result: 3 + 2 = 5
    memo = {3: 3, 4: 5} (stored!)
  
  Call: climbStairsMemo(3) → returns 3 (FROM MEMO - no recursion!)
  
  Result: 5 + 3 = 8
  memo = {3: 3, 4: 5, 5: 8}
```

**مقارنة الكفاءة**

- **الاستدعاء العودي البسيط**: يجري 9 استدعاءات دالة مع حسابات متكررة
    
- **التذكر**: يجري فقط 5 حسابات فريدة، ثم يعيد استخدام النتائج المخزنة
    
- **التعقيد الزمني**: انخفض من `O(2^n)` إلى `O(n)` لأننا نجري فقط `n` حسابًا فريدًا
    
- **التعقيد المكاني**: `O(n)` لتخزين التذكر ومكدس الاستدعاءات
    
- **التأثير الحقيقي**: `climbStairsMemo(30)` ينخفض من ملايين الاستدعاءات إلى حوالي 30 حسابًا فريدًا!
    

### الجدولة (النهج من الأسفل إلى الأعلى)

الجدولة تبني الحل من القاعدة، مع ملء جدول بحلول المشكلات الفرعية:

```js
function climbStairsTabulation(n) {
  if (n <= 2) {
    return n;
  }

  // Create an array to store solutions from 0 to n
  const dp = new Array(n + 1).fill(0);

  // Base cases
  dp[1] = 1; // 1 way to reach step 1
  dp[2] = 2; // 2 ways to reach step 2

  // Build up the solution iteratively
  for (let i = 3; i <= n; i++) {
    // Ways to reach step i = ways to reach (i-1) + ways to reach (i-2)
    dp[i] = dp[i - 1] + dp[i - 2];
  }

  return dp[n];
}
```

الجدولة تلغي الاستدعاء العودي تمامًا من خلال بناء الحل بشكل تكراري من أصغر المشكلات الفرعية حتى الهدف.

لنرَ النهج من الأسفل إلى الأعلى قيد التنفيذ لنرى كيف نبني الحل بشكل منهجي. هذا هو البناء التكراري لـ `climbStairsTabulation(5)`:

```md
Initial state:
dp = [0, 1, 2, 0, 0, 0]
     [0, 1, 2, 3, 4, 5] ← indices (step numbers)

Step by step construction:

i = 3:
  dp[3] = dp[2] + dp[1] = 2 + 1 = 3
  dp = [0, 1, 2, 3, 0, 0]
  
i = 4:
  dp[4] = dp[3] + dp[2] = 3 + 2 = 5
  dp = [0, 1, 2, 3, 5, 0]
  
i = 5:
  dp[5] = dp[4] + dp[3] = 5 + 3 = 8
  dp = [0, 1, 2, 3, 5, 8]

Final result: dp[5] = 8
```

**المزايا الرئيسية للجدولة**

- **لا عبء استدعاء عودي**: على عكس التذكر، لا يوجد مكدس استدعاءات عودي.
    
- **تنفيذ متوقع**: نحسب القيم بترتيب محدد مسبقًا (1، 2، 3، 4، 5...).
    
- **صديق لذاكرة التخزين المؤقت**: الوصول المتسلسل للمصفوفة يحسن استخدام الذاكرة.
    
- **سهل التحسين**: يمكن تقليل التعقيد المكاني إلى `O(1)` لأننا نحتاج فقط إلى آخر قيمتين.
    

```js
function climbStairsOptimized(n) {
  if (n <= 2) {
    return n;
  }

  // Store only the last two results to save space
  let prev2 = 1; // ways to reach step 1
  let prev1 = 2; // ways to reach step 2

  for (let i = 3; i <= n; i++) {
    const current = prev1 + prev2;

    // Shift values forward
    prev2 = prev1;
    prev1 = current;
  }

  return prev1;
}
```

**مقارنة الكفاءة**

- **الاستدعاء العودي البسيط**: 9 استدعاءات دالة لـ `n=5`، نمو أُسّي.
    
- **الجدولة**: 3 جمعات بسيطة لـ `n=5`، نمو خطي.
    
- **التعقيد الزمني**: `O(n)` بدلاً من `O(2^n)`.
    
- **التعقيد المكاني**: `O(n)` للمصفوفة، أو `O(1)` مع التحسين.
    
- **أداء متوقع**: لا خطر من تجاوز مكدس الاستدعاءات للمدخلات الكبيرة.
    

كلا النهجين يقللان التعقيد الزمني من أُسّي `O(2^n)` إلى خطي `O(n)`، وهو تحسن كبير يصنع الفرق بين حل المشكلة في أجزاء من الثانية أو انتظار سنوات للمدخلات الأكبر.

## التطبيقات العملية

لدى البرمجة الديناميكية تطبيقات واسعة في علوم الحاسوب وما بعدها:

- **تحسين المسارات**: تستخدم أنظمة GPS خوارزميات البرمجة الديناميكية لإيجاد أقصر المسارات بين المواقع.
    
- **معالجة النصوص**: تعتمد مدققات الإملاء وميزات الإكمال التلقائي غالبًا على البرمجة الديناميكية لحساب مسافات التحرير بين الكلمات.
    
- **النمذجة المالية**: تستخدم استراتيجيات الاستثمار وتحسين معرض الأعمال تقنيات البرمجة الديناميكية بشكل متكرر.
    
- **تخصيص الموارد**: تظهر مشكلة حقيبة الظهر وأنواعها في الجدولة والميزانية وإدارة الموارد.
    

## مثال عملي: مشكلة تبديل العملات

مشكلة تبديل العملات هي تحدٍّ برمجي كلاسيكي يُظهر عند حلها باستخدام البرمجة الديناميكية كلا المبدأين الرئيسيين: البنية الفرعية المثلى والمشكلات الفرعية المتداخلة.

تسأل مشكلة تبديل العملات: "ما هو الحد الأدنى لعدد العملات اللازمة لصنع مبلغ مستهدف؟"

إليك حلًا واحدًا باستخدام البرمجة الديناميكية:

```js
function minCoins(amount, coins) {
  // Create dp array from 0 to amount, filled with Infinity
  const dp = new Array(amount + 1).fill(Infinity);

  // Base case: 0 coins needed to make amount 0
  dp[0] = 0;

  // Build up from 1 to target amount
  for (let i = 1; i <= amount; i++) {
    // Try each coin denomination
    for (let coin of coins) {
      // Only use coin if it is not larger than current amount
      if (coin <= i) {
        // Choose the minimum: current value OR solution for remaining amount + 1 coin
        dp[i] = Math.min(dp[i], dp[i - coin] + 1);
      }
    }
  }

  // Return result if possible, -1 if impossible (infinity)
  return dp[amount] === Infinity ? -1 : dp[amount];
}

/*
 Example usage:
coins = [1, 3, 4], amount = 6

dp[0] = 0
dp[1] = dp[0] + 1 = 1
dp[2] = dp[1] + 1 = 2
dp[3] = min(dp[2]+1, dp[0]+1) = min(2+1, 0+1) = 1
dp[4] = min(dp[3]+1, dp[1]+1, dp[0]+1) = min(1+1, 1+1, 0+1) = 1
dp[5] = min(dp[4]+1, dp[2]+1, dp[1]+1) = min(1+1, 2+1, 1+1) = 2
dp[6] = min(dp[5]+1, dp[3]+1, dp[2]+1) = min(2+1, 1+1, 2+1) = 2

Result: 2 coins (3 + 3)
*/
```

وهنا كيف تعمل خوارزمية تبديل العملات بالبرمجة الديناميكية خطوة بخطوة لـ `coins = [1, 3, 4]`، `amount = 6`:

```md
Initial state:
dp = [0, ∞, ∞, ∞, ∞, ∞, ∞]
     [0, 1, 2, 3, 4, 5, 6] ← amounts

Building up the solution:

For amount = 1:
  Try coin 1: dp[1] = min(∞, dp[0] + 1) = min(∞, 0 + 1) = 1
  dp = [0, 1, ∞, ∞, ∞, ∞, ∞]

For amount = 2:
  Try coin 1: dp[2] = min(∞, dp[1] + 1) = min(∞, 1 + 1) = 2
  dp = [0, 1, 2, ∞, ∞, ∞, ∞]

For amount = 3:
  Try coin 1: dp[3] = min(∞, dp[2] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[3] = min(3, dp[0] + 1) = min(3, 0 + 1) = 1
  dp = [0, 1, 2, 1, ∞, ∞, ∞]

For amount = 4:
  Try coin 1: dp[4] = min(∞, dp[3] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[4] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  Try coin 4: dp[4] = min(2, dp[0] + 1) = min(2, 0 + 1) = 1
  dp = [0, 1, 2, 1, 1, ∞, ∞]

For amount = 5:
  Try coin 1: dp[5] = min(∞, dp[4] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[5] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  Try coin 4: dp[5] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, ∞]

For amount = 6:
  Try coin 1: dp[6] = min(∞, dp[5] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[6] = min(3, dp[3] + 1) = min(3, 1 + 1) = 2
  Try coin 4: dp[6] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, 2]

Final result: dp[6] = 2 (achieved with coins 3 + 3)
```

هذا الحل يوضح كلا المبدأين الرئيسيين للبرمجة الديناميكية. هناك مشكلات فرعية متداخلة لأن إيجاد الحد الأدنى من العملات للمبلغ 6 يتطلب معرفة الحلول للمبالغ 5 و3 و2. تظهر هذه المشكلات الفرعية نفسها عند حساب مبالغ أخرى. هناك بنية فرعية مثلى لأن الحل الأمثل لأي مبلغ يدمج الحلول المثلى للمبالغ الأصغر. إذا عرفنا أن الحد الأدنى من العملات للمبلغ 3 هو 1، فإحدى الطرق لصنع المبلغ 6 هي استخدام هذا الحل بالإضافة إلى عملة واحدة بقيمة 3.

بدون البرمجة الديناميكية، نحتاج إلى تجربة كل تركيبة ممكنة من العملات - عدد أُسّي من الاحتمالات. مع البرمجة الديناميكية، نبني الحل بشكل منهجي:

- **التعقيد الزمني**: `O(amount × number of coins)` بدلاً من أُسّي.
    
- **التعقيد المكاني**: `O(amount)` لمصفوفة `dp`.
    
- **لا عمل متكرر**: تُحل كل مشكلة فرعية (إيجاد الحد الأدنى من العملات لكل مبلغ) مرة واحدة بالضبط.
    
- **إعادة استخدام النتائج**: بمجرد معرفة الحد الأدنى من العملات للمبلغ 3، نستخدم هذه المعرفة لكل المبالغ الأكبر التي يمكن أن تستفيد منها.
    

## متى تستخدم البرمجة الديناميكية

البرمجة الديناميكية فعالة عندما:

- يمكن تقسيم المشكلة إلى مشكلات فرعية متداخلة.
    
- تظهر المشكلة بنية فرعية مثلى.
    
- يتضمن الحل العودي البسيط حسابات متكررة.
    
- تحتاج إلى تحسين التعقيد الزمني على حساب التعقيد المكاني.
    

تشمل أنماط البرمجة الديناميكية الشائعة مشاكل التحسين (إيجاد القيم الدنيا/العليا)، مشاكل العد (عدد الطرق لتحقيق شيء ما)، ومشاكل القرار التي يمكن تقسيمها إلى قرارات أصغر.

تحول البرمجة الديناميكية المشكلات المعقدة إلى مشكلات يمكن إدارتها من خلال تخزين وإعادة استخدام الحلول للمشكلات الفرعية بشكل منهجي. فهم هذه التقنية يفتح الباب لحل مجموعة واسعة من التحديات الحاسوبية بكفاءة.
