# --description--

أثناء تصميم تطبيقاتك، غالبًا ما يكون لديك سلسلة من الدوال الصغيرة المسؤولة عن شيء واحد. يُعرف هذا بمبدأ "المسؤولية الواحدة". عندما يكون لديك سلسلة من الدوال الصغيرة، من الأفضل اختبار هذه الدوال للتأكد من أن كل شيء يعمل كما هو متوقع. في هذا الدرس، سنلقي نظرة على كيفية إنشاء اختبار وحدة باستخدام إطار العمل الشائع Jest.

في هذا المثال، سننشئ دالة مسؤولة عن إرجاع سلسلة نصية مُنسقة حديثًا:

```js
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

تحتوي دالة `getFormattedWord` على معلمة تسمى `str` وستتحقق أولًا مما إذا كانت `str` فارغة. إذا كانت كذلك، فستُرجع سلسلة نصية فارغة. وإلا، فستُرجع سلسلة جديدة حيث تُكتب الحرف الأول بحرف كبير. نحن نصدر هذه الدالة حتى نتمكن من استخدامها في ملف اختبار.

في ملف منفصل باسم `getFormattedWord.test.js`، يمكننا كتابة بعض الاختبارات للتحقق من أن الدالة تقوم بما يُفترض بها القيام به.

في أعلى ملف `getFormattedWord.test.js`، نحتاج أولًا إلى استيراد الدالة بهذه الطريقة:

```js
import { getFormattedWord } from "./getFormattedWord.js";
```

ثم نحتاج إلى تثبيت حزمة Jest باستخدام الأمر `npm i jest`. بعد ذلك يمكننا إضافة اختبار للتحقق من أن كلمة مكونة من أحرف صغيرة، `hello`، تُنسق بشكل صحيح إلى `Hello`، التي تبدأ بحرف كبير `H`:

```js
test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

تُستخدم دالة `expect` لاختبار قيمة. تُدمج مع matcher، وهي دالة تتحقق مما إذا كانت القيمة تتصرف كما هو متوقع. في هذه الحالة، الـ matcher هو `toBe()`. لدى Jest مجموعة متنوعة من matchers لمساعدتك في اختبار القيم الحقيقية، والسلاسل النصية، والأرقام، والمزيد.

إحدى الطرق لتشغيل اختباراتك هي إضافة سكربت npm إلى ملف `package.json` الخاص بك. إليك مثالًا على سكربت لإطار العمل Jest:

```js
"scripts": {
  "test": "jest"
},
```

ثم فقط شغّل الأمر `npm run test` في الطرفية لتشغيل اختباراتك. إذا نجحت اختباراتك، سترى شيئًا مثل هذا في الطرفية:

```sh
PASS ./getFormattedWord.test.js
  ✓ capitalizes the first letter of a word (1 ms)

Test Suites: 1 passed, 1 total
Tests:       1 passed, 1 total
Snapshots:   0 total
Time:        0.103 s, estimated 1 s
Ran all test suites.
```

لرؤية كيف يبدو اختبار فاشل، يمكننا عمدًا كسر الاختبار بتحديث الدالة إلى ما يلي:

```js
export function getFormattedWord(str) {
  if (!str) return '';
  return "This is incorrect";
}
```

الآن عند تشغيل الأمر `npm run test`، ستظهر رسالة خطأ لأن الاختبار كان يتوقع نتيجة مختلفة:

```sh
FAIL ./getFormattedWord.test.js
  ✕ capitalizes the first letter of a word (1 ms)

  ● capitalizes the first letter of a word

  expect(received).toBe(expected) // Object.is equality

  Expected: "Hello"
  Received: "This is incorrect"

    2 |
    3 | test('capitalizes the first letter of a word', () => {
    4 |   expect(getFormattedWord('hello')).toBe('Hello');
                                            ^
    5 | });
    6 |

    at Object.toBe (getFormattedWord.test.js:4:37)

Test Suites: 1 failed, 1 total
Tests:       1 failed, 1 total
Snapshots:   0 total
Time:        0.121 s, estimated 1 s
Ran all test suites.
```

الآن يمكننا تحديث دالتنا مرة أخرى إلى الأصلية هنا:

```js
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

أضف اختبارًا آخر للتحقق من الحالات التي لا يُقدم فيها أي معلمة إلى `getFormattedWord`:

```js
test("returns an empty string when no argument is provided", () => {
  expect(getFormattedWord()).toBe("");
});
```

هناك اختبارات أخرى يمكنك إضافتها لجعل الأمور أكثر متانة، لكن هذين الاختبارين الأولين مقدمة جيدة لاختبار الوحدة.

عندما يتعلق الأمر باختبار تطبيقات JavaScript، تشمل أطر العمل الشائعة للاختبار Jest وMocha وVitest.

اختبار الوحدة مهم لأنه يساعدك على اكتشاف المزيد من الأخطاء البرمجية في برامجك، ويضمن أن كل شيء يعمل كما هو متوقع. كما يمكن أن يكون شكلًا من أشكال التوثيق لتطبيقك لأنه يهدف إلى تمثيل السلوك المتوقع لشفراتك.
