# --description--

前のレッスンでは、機能ブランチを作成し、そのブランチに新しいファイルをコミットしてリモートリポジトリにプッシュしました。でも次に何をすればいいのでしょうか？GitHubでプルリクエストを作成する必要があります。

プルリクエストとは、あなたのブランチからターゲットブランチに変更を取り込むように依頼するものです。プルリクエストは、オープンソースプロジェクトにコード変更を貢献したいときに使う流れです。この方法により、プロジェクトのメンテナがあなたの変更をレビューできます。コメントを残したり、質問したり、修正を提案したり、そして（うまくいけば）変更を承認してマージしてくれます！

それでは、もう一度GitHubのリポジトリに移動しましょう。

新しいブランチをプッシュした後、ブランチにプッシュしたことを知らせるかっこいいバナーと、プルリクエストを作成するためのボタンが表示されるはずです。このボタンはいくつかのステップを省略してプロセスを簡略化するので、今回は無視します。バナーが表示されない場合に備えて準備しておきましょう。

リポジトリページの上部にあるプルリクエストタブをクリックし、バナーを無視して「New Pull Request」をクリックしてください。プルリクエストを準備するためのUIに移動します。

ここでいくつかの用語を説明します。まず、「base」と「compare」のドロップダウンがあります。「base」はマージのターゲットです。`main`にマージしたいので、そのままにしておきます。

「compare」はマージしたいものです。`feature`ブランチをマージしたいので、これを`feature`に変更します。これはヘッドブランチとも呼ばれ、覚えておくとよい用語です。

これを設定すると、UIが更新されます。`feature`にあって`main`にはないコミットが表示され、マージをリクエストしている内容が示されます。また「diff」と呼ばれる、実際に行った変更の視覚的な表現も見られます。

今回はファイルを追加しただけなので、diffはかなりシンプルです。もっと複雑な変更の場合、GitHubは行の修正、追加、削除などをハイライトします。diffの読み方は難しいこともありますが、練習すればかなり上達します。

さて、プルリクエストに戻りましょう。上部に「compare across forks」という小さなリンクがあります。もしfreeCodeCampリポジトリのフォークから作業している場合、これを使ってプルリクエストのターゲットを自分のフォーク内の`main`ブランチではなく、元のfreeCodeCampリポジトリに変更できます。

今回はフォークを使っていないので無視してかまいません。ただし、ここで行っていることは主に学習と探求のためであることを理解することが重要です。練習用の変更だけを含むプルリクエストを他人のプロジェクトに送るべきではありません。メンテナに余計な負担をかけてしまいます。

変更に満足し、すべて設定が完了したら、「Create Pull Request」ボタンをクリックしてください。新しいUIに移動します。

画面の下のほうには、先ほど見たコミットリストとdiffビューがあります。プルリクエストを作成する前にもう一度確認しても損はありませんが、この探求ではそのまま進んで大丈夫です。

また、いくつか新しい入力欄が表示されます。最初の入力欄はプルリクエストのタイトルです。GitHubは、プルリクエストにコミットが1つだけならそのコミットに基づいて、複数ある場合はブランチ名に基づいて自動生成します。

2つ目の入力欄は説明用です。ほとんどの場合、空欄にはなりません。プロジェクトのメンテナがテンプレートを定義している場合、それが自動で入力されます。テンプレートがある場合は、そのワークフローを尊重して記入してください。

どちらの場合も、プルリクエストのタイトルや説明の書き方については、プロジェクトのcontributingドキュメントを読んでルールを理解し、従うことをお勧めします。

このリポジトリの所有者なので、右側でレビュー担当者、担当者、ラベル、プロジェクト、マイルストーンを設定できます。他人のプロジェクトに貢献する場合はレビュー依頼以外は編集できませんが、ここでは自由に設定してみてください。

すべて入力と設定が完了したら、「Create pull request」をクリックします。ついに結果が表示されます！

他人のプロジェクトにプルリクエストを送る場合、「コードレビュー」を経る必要があります。これはメンテナがプルリクエストを評価し、変更を求めたり質問したりするプロセスです。承認されるとマージされます。

自分のプロジェクトなのでレビューを待つ必要はありません。コミット履歴はCommitsタブで、変更はFiles changedで確認し、満足したらマージできます。

マージしたら、これらの変更をローカルリポジトリに取り込みたいです。

ターミナルに戻り、まだリポジトリのフォルダにいることを確認してください。次に`main`ブランチに切り替えます。`git checkout main`を使ってください。

次に`git pull`で最新の変更をプルしてください。

コミット履歴を`git log`で見ると、自分のコミットが表示されますが、`Merge pull request #1`というメッセージも見えます。これは何でしょうか？

GitHubでプルリクエストをマージするとき、3つの戦略があります：

デフォルトは「Merge」で、ヘッドブランチのすべてのコミットを取り込み、さらにマージを処理・記録するためのコミットを作成します。

もう1つは「Squash and Merge」で、ヘッドブランチのすべてのコミットを1つにまとめて、その新しいコミットをベースブランチに直接マージします。マージコミットは作成されません。

最後に「Rebase and Merge」があり、すべてのコミットをベースブランチの最新コミットの後にリセットし、マージコミットなしでヘッドをマージします。リベースは複雑なトピックなので、後で詳しく学びます。今日完全に理解できなくても問題ありません。

オープンソースプロジェクトはそれぞれ最適なマージ戦略を選びます。しかしこれらの違いがあるため、必ず新しいブランチを作ってそこにコミットすることが重要です。デフォルトの`main`ブランチに直接コミットすると、元のリポジトリから変更を取り込むときに混乱します。

これで、最初のプルリクエストを無事に作成し、マージできました！
