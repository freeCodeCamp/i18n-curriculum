# --description--

SQL-Injection ist eine Sicherheitslücke im Web, bei der Angreifer die Abfragen verändern oder ausnutzen, die die Webanwendung an die Datenbank stellt. Zum Beispiel können Angreifer bösartigen SQL-Code in die Abfrage einfügen, um diesen schädlichen Code im Backend auszuführen.

Dies kann passieren, wenn Sie Nutzern die Möglichkeit geben, Daten über Login-Formulare, Suchfelder oder URL-Parameter einzugeben und diese Nutzereingaben dann in der String-Interpolation oder -Konkatenation verwenden, um den finalen SQL-Befehl zu erzeugen.

Je nach Ziel des Angreifers kann der bösartige SQL-Code erhebliche Störungen in Ihrer Datenbank und Webanwendung verursachen, wie das Umgehen der Authentifizierung, das Stehlen von Daten aus der Datenbank oder das Ändern oder Löschen von Datensätzen. Deshalb sollten Sie diese Bedrohung sehr ernst nehmen.

Zum Beispiel nehmen wir an, Ihre Webanwendung verwendet eine SQL-Abfrage wie diese, um Benutzer zu authentifizieren, wobei der vom Benutzer eingegebene Benutzername und das Passwort direkt eingebettet werden:

```sql
'SELECT * FROM users WHERE username = ' + username_input + ' AND password = ' + password_input + ';'
```

Wenn ein Angreifer etwas wie Folgendes in das `username`-Feld eingibt:

```sql
'" " OR "1"="1" --'
```

Und irgendetwas anderes im `password`-Feld, da `"1"="1"` immer `true` ist und `--` den Rest der Abfrage auskommentiert, wird die `WHERE`-Klausel zu:

```sql
WHERE username = " " OR TRUE
```

Was immer `true` ist.

Je nach Datenbank-Implementierung kann dies die Daten des ersten Benutzers oder eines beliebigen Benutzers in der Datenbank zurückgeben und dem Angreifer erlauben, sich ohne gültige Zugangsdaten einzuloggen.

Wie können Sie das also verhindern? Es gibt zwei Hauptwege, um SQL-Injection zu verhindern:

- Schreiben Sie keine dynamischen Abfragen mit String-Konkatenation.
- Und verhindern Sie, dass bösartige SQL-Eingaben in Abfragen gelangen, die im Backend ausgeführt werden.

In der Praxis können Sie parametrisierte Statements verwenden, auch bekannt als parametrisierte Abfragen. Diese trennen die SQL-Struktur von den vom Benutzer eingegebenen Daten, was verhindert, dass die Datenbank die Eingabedaten als ausführbaren Code interpretiert.

Eine weitere wichtige Technik ist die Validierung der Eingaben. Diese wird in allen Fällen als sekundäre Sicherheitsmaßnahme empfohlen, ist aber besonders wichtig für Teile von SQL-Abfragen, die sich auf Tabellennamen, Spaltennamen und Sortierreihenfolge beziehen. Wenn möglich, sollten Tabellennamen und Spaltennamen direkt im Code zugewiesen werden, nicht über Nutzereingaben.

Falls trotz aller Präventivmaßnahmen doch ein SQL-Injection-Angriff erfolgt, können Sie dessen Auswirkungen auf die Datenbank minimieren, indem Sie die Berechtigungen jedes Datenbankkontos einschränken. Gewähren Sie jedem Datenbankbenutzer nur die Berechtigungen, die er für seine Aufgaben benötigt, und nicht mehr. Wenn ein Konto nur Daten aus der Datenbank lesen muss, gewähren Sie ihm nur Leserechte. Und als vorbeugende Maßnahme sollten Sie Ihren Anwendungskonten niemals Administratorrechte zuweisen. Das ist sehr gefährlich, weil ein Angreifer so vollständigen Zugriff und volle Kontrolle über die Datenbank erhalten könnte.

SQL-Injection ist ein sehr häufiger Angriff, dessen Sie sich als Entwickler bewusst sein und den Sie bei der Implementierung Ihrer Webanwendung verhindern sollten. Mit diesen Techniken können Sie Ihre Datenbank schützen und die Daten Ihrer Nutzer sicher halten.
