# --description--

In una lezione precedente, hai imparato come creare un nuovo repository su GitHub. Ma cosa succede se hai un progetto esistente in locale che vuoi aggiungere a GitHub? In questa lezione, imparerai come fare il push di repository locali esistenti su GitHub e come scaricare le modifiche dai repository remoti.

Inizia creando un nuovo repository su GitHub senza alcun file. Se fatto correttamente, verrai reindirizzato a una pagina che include l’URL necessario per clonare il tuo repository.

Ora vai al terminale e seleziona un progetto esistente che vuoi aggiungere a GitHub. In questa lezione, scelgo un progetto locale chiamato `super-awesome-game` con i seguenti file al suo interno:

```sh
jessicawilkins super-awesome-game >> ls
README.md  index.html script.js  styles.css
```

Per configurare il controllo di versione per questo progetto usando Git, dovrai eseguire il comando `git init` nella directory del progetto. Questo inizializzerà un repository Git vuoto così Git potrà iniziare a tracciare le modifiche per questo progetto. Se fatto correttamente, dovresti vedere un output simile nel terminale:

```sh
jessicawilkins super-awesome-game >> git init
Initialized empty Git repository in /Users/jessicawilkins/workspace/freeCodeCamp/super-awesome-game/.git/
```

Quando inizializzi un repository Git vuoto in un progetto, verrà aggiunta una nuova directory nascosta `.git`. Per vedere l’aggiunta della directory puoi eseguire il comando `ls -a` che elenca tutti i file e le directory, comprese quelle nascoste:

```sh
jessicawilkins super-awesome-game >> ls -a
.          ..         .git       README.md  index.html script.js  styles.css
```

È importante non eliminare quella directory `.git`, altrimenti perderai tutta la cronologia di Git.

Ora che il repository Git vuoto è stato inizializzato, puoi eseguire il comando `git status`. Questo comando serve a mostrare lo stato attuale della tua directory di lavoro — userai molto questo comando nel tuo flusso di lavoro. Dovresti vedere un output simile a questo:

```sh
jessicawilkins super-awesome-game >> git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	README.md
	index.html
	script.js
	styles.css

nothing added to commit but untracked files present (use "git add" to track)
```

`On branch main` significa che sei sul ramo `main`. Un ramo in Git è un’area di lavoro separata dove puoi fare modifiche. Il ramo `main` rappresenta spesso il ramo principale o di produzione in un’applicazione reale. I team di sviluppatori creeranno più rami per nuove funzionalità e correzioni di bug e poi uniranno quelle modifiche nel ramo `main`. Imparerai di più su questo processo in una lezione futura.

`Untracked files` significa che Git ha rilevato file che attualmente non sta tracciando.

Ci sono cinque stati in cui un file può trovarsi:

1. "Untracked": significa che il file è nuovo nel repository e Git non l’ha ancora "visto".
2. "Modified": questo file esisteva nel commit precedente e ha modifiche non ancora commesse.
3. "Ignored": probabilmente non vedrai i file ignorati in Git, ma il tuo IDE potrebbe segnalarli. I file ignorati sono esclusi dalle operazioni Git, di solito perché sono inclusi nel file `.gitignore`.
4. "Deleted": un file eliminato è l’opposto di un file non tracciato — è un file che esisteva prima ed è stato rimosso.
5. "Renamed": un file rinominato è un file il cui contenuto non è cambiato, ma il nome o la posizione è stata modificata. In alcuni casi, un file può essere considerato rinominato anche se ha piccole modifiche.

Quando hai modifiche non commesse nel tuo repo, devi prima "staggiarle". Un "area di staging" è un file speciale nella tua directory `.git` che contiene informazioni che andranno nel tuo prossimo commit. Un commit è di fatto un’istantanea dello stato attuale del tuo repository.

Se vuoi staggiare un file alla volta, puoi usare questo comando:

```sh
git add name-of-file
```

Se vuoi staggiare tutte le modifiche non ancora staggiate, puoi usare `git add .` Il punto (`.`) è un alias per la directory corrente in cui ti trovi. Alcuni sviluppatori sconsigliano di usare `git add .` perché potresti finire per staggiare file che non volevi. Se succede, puoi "unstaggiare" i file con questo comando:

```sh
git reset name-of-file
```

In generale, però, finché controlli quali file stai aggiungendo con `git status` prima di staggiarli, dovresti stare tranquillo.

Una volta staggati i file, devi commetterli. Puoi fare il commit delle modifiche eseguendo il comando `git commit`. Questo aprirà l’editor preferito che hai impostato nella configurazione di Git. Quando l’editor si apre, puoi scrivere un messaggio dettagliato sulle modifiche. Puoi anche scegliere di fornire un messaggio più breve usando il comando `git commit -m` così:

```sh
git commit -m "short message goes here"
```

Se usi il flag `-m`, dovresti comunque fornire un messaggio descrittivo per dare contesto alle modifiche. Ecco alcuni esempi di buoni messaggi che potresti usare:

```sh
git commit -m "feat(api): implement JWT-based authentication"
git commit -m "fix(auth): correct token expiration time"
git commit -m "refactor: simplify login flow logic"
```

I flag `feat(api)` e `fix(auth)` seguono lo stile Conventional Commits, usato per dare contesto sul tipo di modifiche in quel commit.

Dopo aver commesso il lavoro, e aver eseguito di nuovo `git status`, dovresti vedere questo output nel terminale:

```sh
On branch main
nothing to commit, working tree clean
```

Per vedere tutti i commit precedenti di questo progetto puoi eseguire il comando `git log`. Questo elencherà tutti i commit precedenti con informazioni utili come autore, data del commit, messaggio e hash del commit. L’hash del commit è una lunga stringa che serve come identificatore unico per un commit.

Quando hai finito di commettere il lavoro, puoi fare il push delle modifiche su GitHub. Prima devi configurare la connessione remota al tuo repo remoto. Ne hai già parlato nella lezione precedente, ma ecco un promemoria usando SSH:

```sh
git remote add origin git@github.com:your-github-username/name-of-repo.git
```

L’ultimo passaggio è fare il push delle modifiche su GitHub con questo comando:

```sh
git push -u origin main
```

Il flag `-u` è un’abbreviazione per `--set-upstream`. Collega il tuo ramo locale `main` al ramo remoto `main` su `origin`. Devi usarlo solo una volta per ogni nuovo ramo. Dopo puoi semplicemente eseguire `git push` senza specificare il nome del ramo.

Se fatto correttamente, dovresti vedere un risultato simile nel terminale:

```sh
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Delta compression using up to 8 threads
Compressing objects: 100% (2/2), done.
Writing objects: 100% (3/3), 253 bytes | 253.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0 (from 0)
To https://github.com/jdwilkin4/super-awesome-game.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

Dovresti anche vedere le nuove modifiche sulla pagina del tuo repo GitHub quando aggiorni la pagina.

Se stai collaborando con altri su un progetto e devi scaricare nuove modifiche dal repository, puoi usare il comando `git pull`. Imparerai di più sul lavoro di squadra e la collaborazione in una lezione futura.
