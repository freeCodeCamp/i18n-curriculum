# --description--

Раніше ви вивчали, як робити коміти змін і відправляти їх у свій віддалений репозиторій, але все це відбувалося у вашій гілці за замовчуванням `main`. Це може працювати для невеликого проєкту, але коли ви робите внесок у інші проєкти з відкритим кодом або ваш проєкт зростає, гілки стають необхідними.

Наприклад, якщо ви форкнули репозиторій і робите коміти у гілці `main` свого форку, у вас можуть з’явитися зміни у форку, які оригінальний репозиторій не прийняв. Це може дуже ускладнити оновлення вашої гілки змінами з форку, які були прийняті. Або, коли ви працюєте над власним проєктом, перемикання між різними завданнями стає набагато складнішим, якщо не використовувати гілки. Замість цього доводиться вручну відстежувати, які зміни стосуються якої функції чи виправлення.

Але що таке гілка? Уявіть вашу гілку `main` як автомагістраль, а кожен коміт — це автомобіль, який виїжджає на цю автомагістраль. Ви можете захотіти подивитися щось інше, наприклад створити нову функцію, тому вам потрібно ненадовго з’їхати з автомагістралі, щоб це зробити.

Гілка — це, по суті, відгалуження від основної історії, де ви можете вільно експериментувати зі змінами для нової функції або виправлення помилки. Якщо ви задоволені своїми змінами, ви можете повернутися на автомагістраль, об’єднавши гілку з `main`. Якщо ні — ви можете видалити гілку і повернутися до `main` без прийняття змін.

Перед створенням нової гілки перегляньте свої гілки за допомогою команди `git branch`. Вихідні дані можуть виглядати так:

```sh
* main
```

Зараз у нас є лише одна гілка — наша гілка за замовчуванням `main`. Зірочка (`*`) означає, що це гілка, яку ви зараз "перевірили". Про це ви дізнаєтеся трохи пізніше.

Ви можете створити нову гілку командою `git branch` з аргументом, який вказує ім’я нової гілки:

```sh
git branch feature
```

Це створює нову гілку з ім’ям `feature`. Давайте знову подивимося на наші гілки за допомогою `git branch`:

```sh
  feature
* main
```

Тепер ми бачимо нашу нову гілку, але все ще перебуваємо у гілці `main`. Потрібно переключитися на нову гілку, "перевіривши" її. Коли ви перевіряєте нову гілку, ви кажете Git завантажити файли у тому стані, в якому вони були на той момент.

Перевіримо нашу нову гілку командою:

```sh
git checkout feature
```

або

```sh
git switch feature
```

І знову виконаємо `git branch`, щоб побачити:

```sh
* feature
  main
```

Зверніть увагу, що зірочка тепер перемістилася на гілку, яку ви зараз "перевірили".

Часто, створюючи нову гілку, ви захочете одразу її перевірити. Замість того, щоб створювати гілку, а потім перевіряти її двома командами, можна створити і переключитися на гілку однією командою:

```sh
git checkout -b feature
```

або

```sh
git switch -c feature
```

Отже, ми створили і перевірили нову гілку. Ця гілка є копією гілки `main`, вона містить увесь той самий код, що й `main` на момент створення гілки. Тепер ми можемо додавати будь-який код у цю гілку, не впливаючи на гілку `main`. По суті, ми з’їхали з автомагістралі, щоб створити функцію. Якщо ви перевірите свій `git status`, побачите щось на кшталт цього:

```sh
On branch feature
nothing to commit, working tree clean
```

Створіть новий файл `feature.md`, виконавши:

```sh
echo "This is our new feature" > feature.md
```

І тепер статус виглядатиме так:

```sh
On branch feature
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	feature.md

nothing added to commit but untracked files present (use "git add" to track)
```

Далі час закомітити вашу нову функцію. Спочатку додайте файл у стадію:

```sh
git add feature.md
```

Потім зробіть коміт змін:

```sh
git commit -m "my new feature"
```

Нарешті, відправте вашу гілку `feature` у репозиторій:

```sh
git push -u origin feature
```

Прапорець `-u` пов’язує вашу локальну гілку `feature` з віддаленою, тож у майбутньому можна буде просто виконувати `git push`.

Далі ми навчимося створювати запит на злиття для нашої нової функції!
