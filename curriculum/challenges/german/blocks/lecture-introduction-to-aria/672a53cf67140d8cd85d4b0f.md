# --interactive--

Statischen Inhalt zugänglich zu machen, kann relativ einfach sein, aber dynamischer Inhalt kann eine größere Herausforderung darstellen. Hier kommt WAI-ARIA ins Spiel.

Schauen wir uns an, was WAI-ARIA ist, welchen Zweck es hat, wie es funktioniert und einige Beispiele.

WAI-ARIA steht für Web Accessibility Initiative – Accessible Rich Internet Applications. Es ist eine Spezifikation, die die Barrierefreiheit für dynamische Inhalte und UI-Komponenten verbessert.

Beachten Sie, dass WCAG und WAI-ARIA nicht dasselbe sind. WCAG bietet allgemeine Richtlinien für Barrierefreiheit im Web, während WAI-ARIA spezifische Regeln bereitstellt, um dynamische und interaktive Inhalte für Nutzer von Hilfstechnologien zugänglich zu machen.

Der Hauptzweck von WAI-ARIA besteht also darin, die Barrierefreiheit für dynamische Inhalte und UI-Komponenten zu verbessern, die keine nativen HTML-Entsprechungen haben.

WAI-ARIA funktioniert, indem es eine Reihe von Attributen einführt, die Sie zu HTML-Elementen hinzufügen können, um zusätzliche semantische Informationen bereitzustellen. Diese Attribute werden in Rollen, Zustände und Eigenschaften unterteilt.

Die ARIA-Rolle definiert den Zweck eines Elements innerhalb einer Website oder Webanwendung. Hier ist ein Beispiel, wie die Rolle für ein `button`-Element auf `div` gesetzt wird.

```html
<div role="button">Click Me</div>
```

Damit zeigen Sie der Hilfstechnologie an, dass das Element ein Button ist. Rollen bieten jedoch keine Funktionalität. Nur weil Sie diesem `div` eine `role` von `button` geben, verhält es sich nicht wie ein Button. Um es wie einen Button aussehen und funktionieren zu lassen, müssten Sie CSS und JavaScript verwenden, um das gewünschte Ergebnis zu erzielen.

Hier ist ein Beispiel, wie Sie mit HTML, CSS und JavaScript ein benutzerdefiniertes `button`-Element erstellen.

**HINWEIS**: Machen Sie sich keine Sorgen, den CSS- und JavaScript-Code zu verstehen. Diese Sprachen lernen Sie in zukünftigen Modulen.

:::interactive_editor

```html
<link href="styles.css" rel="stylesheet">
<div id="custom-btn" role="button">Click Me</div>
<script src="index.js"></script>
```

```css
#custom-btn {
  display: inline-block;
  padding: 0.4em 1em;
  font-size: 1rem;
  font-family: sans-serif;
  color: buttontext;
  background-color: buttonface;
  border: 1px solid #ccc;
  border-radius: 4px;
  cursor: pointer;
  user-select: none;
  text-align: center;
}

#custom-btn:focus {
  outline: 2px solid Highlight;
  outline-offset: 2px;
}

#custom-btn:active {
  background-color: #ddd;
}
```

```js
const button = document.getElementById("custom-btn");

button.addEventListener("click", () => {
  alert("Button clicked!");
});

button.addEventListener("keydown", (e) => {
  if (e.key === "Enter" || e.key === " ") {
    e.preventDefault(); 
    button.click();
  }
});
```

:::

Es ist immer besser, das native `button`- oder `input`-Element mit `type="button"` zu verwenden.

ARIA-Eigenschaften liefern zusätzliche Details zu Elementen. Zum Beispiel ermöglicht Ihnen die `aria-labelledby`-Eigenschaft, ein Element mit einer bestimmten Bezeichnung zu verbinden:

```html
<h2 id="header-id">About freeCodeCamp</h2>
<button id="button-id" aria-labelledby="header-id button-id">Learn More</button>
```

Dies macht die Elemente für Nutzer von Hilfstechnologien verständlich und navigierbar.

Um das Beste aus WAI-ARIA herauszuholen, sollten Sie möglichst native HTML-Elemente verwenden, da diese in der Regel von Haus aus mehr Barrierefreiheit bieten.

Verwenden Sie WAI-ARIA nur, wenn HTML nicht ausreicht, und vergessen Sie nicht, mit Hilfstechnologien wie Screenreadern zu testen oder Menschen mit Behinderungen Ihre Arbeit testen zu lassen. Stellen Sie außerdem sicher, dass Ihre WAI-ARIA-Zustände und -Eigenschaften sich in Echtzeit mit dem Inhalt aktualisieren. Vermeiden Sie eine Übernutzung von ARIA, da dies oft verwirrend sein kann.
