# --description--

Sempre que você visita um site ou usa uma API, seu navegador e o servidor estão constantemente se comunicando.

Uma das partes mais importantes dessa conversa? Os códigos de status de resposta HTTP.

São mensagens numéricas curtas — como 200, 404 ou 500 — que dizem ao cliente (o dispositivo do usuário final, como um laptop ou celular) o que aconteceu depois que uma requisição foi feita.

Você já aprendeu alguns deles, mas vamos dividi-los por categoria para que você possa lembrar exatamente o que significam e quando usá-los.

## 1xx – respostas informativas

Códigos que começam com 1 são raramente vistos no desenvolvimento do dia a dia, mas indicam que a requisição foi recebida e o processo está continuando, como quando se faz upload de um arquivo.

Exemplo comum:

* 100 Continue – O servidor diz ao cliente: "Ok, você pode enviar o restante da requisição."
* 101 Switching Protocols – O cliente enviou um cabeçalho `Upgrade` então o servidor está trocando de protocolo. Você verá isso mais comumente ao usar Websockets.

## 2xx – respostas de sucesso

Significam que a requisição foi recebida, entendida e processada com sucesso.

* 200 OK – A requisição teve sucesso. Essa é a resposta de sucesso mais comum.
* 201 Created – Um novo recurso foi criado com sucesso, por exemplo, após registrar um novo usuário.
* 204 No Content – A requisição teve sucesso, mas não há corpo de resposta para enviar. Frequentemente usado para requisições `DELETE`.

## 3xx – mensagens de redirecionamento

Elas dizem ao cliente: "O recurso que você está procurando está em outro lugar."

* 301 Moved Permanently – O recurso tem uma nova URL permanente.
* 302 Found – O recurso está temporariamente em uma URL diferente.
* 304 Not Modified – Usado para cache. Diz ao cliente que nada mudou, então ele pode usar a versão em cache.

## 4xx – erros do cliente

Acontecem quando o cliente envia uma requisição inválida.

* 400 Bad Request – A requisição está malformada ou inválida.
* 401 Unauthorized – É necessária autenticação (você precisa fazer login).
* 403 Forbidden – A autenticação foi bem-sucedida, mas você não tem permissão.
* 404 Not Found – O recurso solicitado não existe. E sim, esse é o erro mais icônico da web.

## 5xx – erros do servidor

Significam que algo deu errado no lado do servidor.

* 500 Internal Server Error – Uma mensagem genérica de "algo deu errado".
* 502 Bad Gateway – Um servidor atuando como proxy recebeu uma resposta inválida de outro servidor.
* 503 Service Unavailable – O servidor está temporariamente sobrecarregado ou em manutenção.

Agora que você está familiarizado com alguns dos códigos de status HTTP mais comuns, vamos ver alguns exemplos práticos no Express.

Suponha que você esteja construindo uma API. Se um usuário enviar dados inválidos, você pode responder com o código de status 400, que significa "Bad Request". Você também pode enviar a string "Bad Request" de volta ao cliente para deixar claro o que o código 400 significa:

```js
res.status(400).send("Bad Request");
```

Se tudo correr bem, você pode enviar uma resposta 200. Pode incluir uma mensagem se quiser. Aqui estamos enviando um JSON com uma mensagem simples "Success":

```js
res.status(200).json({ message: "Success" });
```

E se algo travar no servidor, você pode enviar o código 500, que significa que ocorreu um erro interno no servidor, junto com uma mensagem clara sobre o erro:

```js
res.status(500).send("Internal Server Error");
```

Esses códigos de status não são só para máquinas. Eles ajudam desenvolvedores a depurar problemas e ajudam usuários (e outros desenvolvedores) a entender o que está acontecendo.
