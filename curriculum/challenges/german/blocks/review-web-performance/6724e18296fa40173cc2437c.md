## Unterschiede zwischen realer und wahrgenommener Performance

- **Wahrgenommene Performance**: So nehmen Nutzer die Performance einer Website wahr. Es geht darum, wie sie die Reaktionsfähigkeit und Zuverlässigkeit bewerten. Dies ist eine subjektive Messung, daher schwer zu quantifizieren, aber sehr wichtig, da die Nutzererfahrung über Erfolg oder Misserfolg einer Website entscheidet.
- **Reale Performance**: Dies ist die objektive und messbare Performance der Website. Sie wird mit Metriken wie Seitenladezeit, Serverantwortzeit und Renderzeit gemessen. Diese Messungen werden von verschiedenen Faktoren beeinflusst, die mit dem Netzwerk und dem Code selbst zusammenhängen.

## Techniken zur Verbesserung der wahrgenommenen Performance

- **Lazy Loading**: Diese Technik reduziert die anfängliche Ladezeit so weit wie möglich, indem nicht wesentliche Ressourcen im Hintergrund geladen werden.
- **Minimieren von Schriftart-Verzögerungen**: Wenn Ihre Website benutzerdefinierte Schriftarten verwendet, sollten Sie auch versuchen, Verzögerungen beim Laden der Schriftarten zu minimieren, da dies zu Flackern oder zur Anzeige der Fallback-Schriftart führen kann, während die benutzerdefinierte Schriftart geladen wird. Ein Vorschlag ist die Verwendung einer Fallback-Schriftart, die der benutzerdefinierten ähnlich ist, sodass die Änderung subtiler ausfällt.
- **Verwendung von Ladeanzeigen**: Das Anzeigen eines Ladeindikators bei einem lang andauernden Prozess, sobald der Nutzer auf ein Element klickt, kann dem Nutzer das Gefühl geben, verbunden und eingebunden zu sein, wodurch die Wartezeit kürzer erscheint.

## Grundlegende Performance-Konzepte

- **Quellreihenfolge**: Dies bezieht sich darauf, wie HTML-Elemente im Dokument strukturiert sind. Sie bestimmt, was zuerst geladen wird und kann die Performance und Barrierefreiheit erheblich beeinflussen.

Einige bewährte Vorgehensweisen für die Quellreihenfolge sind:

- Kritische Inhalte wie Überschriften, Navigation oder Haupttext höher in der HTML-Struktur platzieren.
- Nicht wesentliche Skripte wie solche für Analytics oder Drittanbieter-Widgets aufschieben, damit sie das Rendern nicht blockieren.
- Progressive Enhancement verwenden, um sicherzustellen, dass das Kernerlebnis bereits vor dem Laden von Styles und Skripten funktioniert. Progressive Enhancement ist eine Technik zum Erstellen von Websites und Anwendungen, die darauf basiert, dass die Seite zuerst mit HTML funktioniert.

Hier ist ein Beispiel für eine gute Quellreihenfolge unter Verwendung der gerade besprochenen bewährten Vorgehensweisen:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **Critical Rendering Path**: Dies ist die Folge von Schritten, die der Browser durchläuft, um Code in Pixel auf dem Bildschirm umzuwandeln.
- **Latenz**: Dies ist die Zeit, die eine Anfrage benötigt, um zwischen Browser und Server zu reisen. Mit anderen Worten: hohe Latenz bedeutet langsame Seiten.

Einige Möglichkeiten zur Reduzierung der Latenz sind:

- Verwendung von CDNs (Content Delivery Networks), um Dateien von näheren Standorten auszuliefern.
- Aktivierung von Kompression, z. B. mit Gzip, um Dateigrößen zu reduzieren.
- Optimierung von Bildern und Verwendung von Lazy Loading.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## Verbesserung von INP

- **Definition**: INP (Interaction to Next Paint) bewertet die Gesamtreaktionsfähigkeit einer Seite, indem die Zeit vom Nutzerinteraktionsereignis, wie einem Klick oder Tastendruck, bis zum nächsten Browser-Update gemessen wird. Ein niedriger INP zeigt eine reaktionsschnellere Seite an.

Hier einige Möglichkeiten zur Verbesserung von INP:

- Reduzieren Sie die Arbeit des Hauptthreads, indem Sie lange JavaScript-Aufgaben aufteilen.
- Verwenden Sie `requestIdleCallback()` für nicht-kritische Skripte. Dies fügt eine Funktion in die Warteschlange ein, die während der Leerlaufzeiten des Browsers aufgerufen wird.
- Schieben Sie schwere Assets auf oder laden Sie sie lazy, wie zuvor beschrieben.
- Optimieren Sie Ereignishandler. Wenn diese zu häufig ausgeführt werden oder schwere Operationen enthalten, können sie die Seite verlangsamen und INP erhöhen. Die Lösung ist Debouncing. Debouncing sorgt dafür, dass die Funktion erst ausgeführt wird, wenn der Nutzer für eine kurze Verzögerung aufgehört hat zu tippen – zum Beispiel 300 ms. Dies verhindert unnötige Berechnungen und verbessert die Performance.

## Wie Rendering im Browser funktioniert

- **Wie Rendering funktioniert**: Zuerst parst der Browser das HTML und baut das DOM auf. Danach verarbeitet der Browser das CSS und erstellt das CSS Object Model (CSSOM). Dies ist eine weitere Baumstruktur, die bestimmt, wie Elemente gestylt werden sollen. Schließlich malt der Browser die Pixel auf den Bildschirm und rendert jedes Element basierend auf den berechneten Styles und dem Layout. Bei komplexen Seiten können mehrere Ebenen zusammengesetzt werden, um die endgültige visuelle Ausgabe zu erzeugen.

## Wie Performance die Nachhaltigkeit beeinflusst

- **Hintergrundinformationen**: Das Internet verursacht etwa 2 % der weltweiten CO2-Emissionen – das entspricht der Luftfahrtindustrie! Jedes übertragene Byte benötigt Strom, von Rechenzentren bis zu Nutzergeräten. Größere Dateien und ineffiziente Skripte bedeuten mehr Energieverbrauch. Eine leistungsstarke Website ist nicht nur schneller, sondern reduziert auch unnötige Verarbeitung und Energieverbrauch.

## Möglichkeiten zur Reduzierung der Seitenladezeiten

- **Optimieren von Medienassets**: Große Bilder und Videos sind häufige Ursachen für langsame Ladezeiten. Durch die Optimierung dieser Assets können Sie Ihre Seite deutlich beschleunigen. Dazu gehören Komprimierung von Bildern, Verwendung moderner Formate wie WebP und Lazy Loading für Assets.
- **Nutzung von Browser-Caching**: Caching ermöglicht es Browsern, Teile Ihrer Website lokal zu speichern, wodurch die Ladezeiten für wiederkehrende Besucher reduziert werden.
- **Minimieren und Komprimieren von Dateien**: Die Reduzierung der Dateigröße führt zu schnelleren Downloads. Dazu gehört die Verringerung der übertragenen Dateigrößen sowie das Minifizieren von CSS- und JavaScript-Dateien.

## Verbesserung der „time to usable“

- **Definition**: „time to usable“ ist das Intervall vom Zeitpunkt der Seitenanforderung bis zu dem Moment, an dem der Nutzer sinnvoll mit der Seite interagieren kann. Um „time to usable“ zu verbessern, können Sie Assets lazy laden oder renderblockierende Ressourcen minimieren.

## Wichtige Metriken zur Messung der Performance

- **First Contentful Paint oder FCP**: Misst, wie schnell das erste sichtbare Inhaltselement – Text oder Bild – auf dem Bildschirm erscheint. Ein guter FCP liegt unter 1,8 Sekunden, ein schlechter über 3 Sekunden. Sie können Ihren FCP mit Chrome DevTools im Performance-Tab überprüfen.
- **Total Blocking Time**: Zeigt, wie lange der Hauptthread durch schwere JavaScript-Aufgaben blockiert ist. Ist die Total Blocking Time (TBT) hoch, erleben Nutzer träge Interaktionen. Zur Verbesserung von TBT sollten lange Aufgaben aufgeteilt und nicht wesentliche Skripte aufgeschoben werden.
- **Absprungrate**: Der Prozentsatz der Besucher, die die Seite verlassen, ohne zu interagieren. Eine hohe Absprungrate kann darauf hindeuten, dass Ihre Seite zu langsam ist.
- **Eindeutige Nutzer**: Diese Metrik verfolgt, wie viele einzelne Besucher Ihre Seite besuchen. Um Absprungrate und eindeutige Nutzer zu sehen, können Sie Google Analytics verwenden. Es ermöglicht Ihnen, diese Metriken zu überwachen und die Nutzerbindung zu verbessern.

## Übliche Tools zur Performance-Messung

- **Chrome DevTools**: Chrome DevTools ist ein eingebautes Tool in Google Chrome, mit dem Sie Performance in Echtzeit analysieren und debuggen können. DevTools zeigt Ladezeiten, CPU-Auslastung und Render-Verzögerungen an. Es ist besonders nützlich, um First Contentful Paint (FCP) zu messen, also wie schnell ein Nutzer den ersten sichtbaren Inhalt sieht. Wenn Ihre Website langsam wirkt, hilft DevTools, Engpässe zu erkennen.
- **Lighthouse**: Ein automatisiertes Tool, das Performance, SEO und Barrierefreiheit überprüft.
- **WebPageTest**: Dieses Tool testet, wie Ihre Seite von verschiedenen Standorten und Geräten geladen wird. Es liefert eine detaillierte Aufschlüsselung des Speed Index, der Total Blocking Time und anderer wichtiger Performance-Metriken. Wenn Sie wissen möchten, wie echte Nutzer Ihre Seite weltweit erleben, ist WebPageTest das richtige Tool.
- **PageSpeed Insights**: Dieses Tool analysiert Ihre Website und gibt schnelle Verbesserungsvorschläge für Mobilgeräte und Desktop. Es zeigt, was Ihre Seite verlangsamt, und gibt konkrete Empfehlungen wie Bildoptimierung, Entfernen renderblockierender Skripte und Reduzierung der Serverantwortzeiten. PageSpeed Insights ist eine schnelle und einfache Möglichkeit, zu prüfen, wie Google die Performance Ihrer Seite sieht.
- **Real User Monitoring**: RUM-Tools verfolgen das tatsächliche Nutzerverhalten und zeigen, wie echte Besucher Ihre Seite erleben. Beliebte RUM-Tools sind Google Analytics, das Seitenladezeiten und Absprungraten misst, sowie New Relic oder Datadog, die Performance-Probleme in Echtzeit überwachen. Wenn Sie Daten von echten Nutzern möchten, sind RUM-Tools unverzichtbar.

## Arbeiten mit Performance Web APIs

- **Definition**: Performance Web APIs ermöglichen Entwicklern, die Effizienz des Ladens und der Reaktion einer Webseite direkt im Code zu verfolgen. Diese APIs erlauben es, Seitenladezeiten zu messen, Render- und Interaktionsverzögerungen zu verfolgen und die Ausführungszeit von JavaScript zu analysieren.
- **`performance.now()`**: Diese API liefert hochpräzise Zeitstempel (in Millisekunden), um zu messen, wie lange verschiedene Teile Ihrer Seite zum Laden benötigen.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: Diese API gibt eine Aufschlüsselung jeder Phase des Seitenladens, vom DNS-Lookup bis zu `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: Diese API hört auf Performance-Ereignisse wie Layout-Verschiebungen, lange Aufgaben und Nutzerinteraktionen.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## Techniken zur Verbesserung der CSS-Performance

- **CSS-Animationen**: Das Animieren bestimmter CSS-Eigenschaften wie Dimensionen, Position und Layout löst einen Prozess namens „Reflow“ aus, bei dem der Browser die Position und Geometrie bestimmter Elemente auf der Seite neu berechnet. Dies erfordert ein Neuzeichnen, das rechenintensiv ist. Daher wird empfohlen, die Anzahl der CSS-Animationen so weit wie möglich zu reduzieren oder dem Nutzer zumindest die Möglichkeit zu geben, sie ein- oder auszuschalten.

## Techniken zur Verbesserung der JavaScript-Performance

- **Code-Splitting**: Das Aufteilen Ihres JavaScript-Codes in Module, die kritische und nicht-kritische Aufgaben ausführen, ist ebenfalls hilfreich. So können Sie die kritischen Module so früh wie möglich vorladen und die nicht-kritischen aufschieben, um die Seite so schnell wie möglich zu rendern.
- **DOM-Manipulation**: Denken Sie daran, dass DOM-Manipulation den Prozess bezeichnet, den Inhalt einer Seite dynamisch mit JavaScript zu ändern, indem mit dem Document Object Model (DOM) interagiert wird. Die Manipulation des DOM ist rechenintensiv. Durch die Reduzierung der DOM-Manipulation in Ihrem JavaScript-Code verbessern Sie die Performance.
