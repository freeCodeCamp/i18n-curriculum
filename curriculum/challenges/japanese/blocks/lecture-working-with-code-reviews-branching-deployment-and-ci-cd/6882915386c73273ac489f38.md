# --description--

前のレッスンでブランチについて少し話しました。ブランチの作成方法や、なぜ`main`に直接コミットすべきでないかなどを学びました。でも、実際にいつ、なぜブランチを作成すべきなのでしょうか？

まず、なぜ`main`にただコミットをたくさんプッシュしてはいけないのか話しましょう。デフォルトのブランチは常に安定している必要があります。つまり、いつでも現在の`main`ブランチをプルして、アプリケーションをビルド、実行、テストできる状態でなければなりません。もし未完成の機能や作業途中のコミットをプッシュしていると、これが常に成り立つとは限りません。

さらに、複数の機能を同時に進めることは非常に一般的です。複数の機能の部分的な作業を`main`に同時にプッシュしていると、変更が分離されていないため、問題のデバッグが非常に難しくなります。その上、作業内容を切り替える（コンテキストスイッチング）こともスムーズではありません。

では、いつブランチを作成すべきでしょうか？一般的な目安としては、作業の単位ごとにブランチを作成することです。例えば、新しいOAuth機能に取り組んでいるなら、その作業は専用のブランチで行うべきです。次にパスワード認証のバグを修正する必要があれば、その作業は別の専用ブランチで行います。ユーザープロフィールページの再設計が必要なら、それも別のブランチです。

ブランチは常に`main`から作成することに注意してください。つまり、OAuth機能用のブランチで作業している場合、パスワード認証の修正用の新しいブランチを作成する前に`main`をチェックアウトします。そうしないと、OAuthの作業コミットがパスワード修正ブランチに含まれてしまい、履歴が混乱します。

では、ブランチ名はどう付けるべきでしょうか？これはあなたや参加しているプロジェクト次第ですが、よく使われる慣例をいくつか紹介します。

`scope/description` - ここで`scope`は`feat`や`fix`のようなもので、`description`はブランチの変更内容を簡潔に説明したものです。例えば、`feat/oauth-support`のように。

`scope/issue/description` - 前の慣例に似ていますが、関連するIssue番号を含めます。例えば、`fix/25/password-auth`のように。

`username/scope/description` - フォークせずに元のリポジトリのブランチで作業するプロジェクト（プライベートな内部リポジトリでよくある）では、ブランチ名の先頭にユーザー名を入れると便利です。例えば、`naomi-lgbt/feat/profile-redesign`のように。

ブランチ戦略について他にもいくつか注意点を挙げます。

長期間続く機能開発、例えば多くの開発時間を要するベータ機能の場合、ブランチではなく機能フラグを使ってロジックを選択的に有効化することをお勧めします。ブランチが長くマージされずにいるほど、コンフリクトが起きやすくなります。

ただし、デプロイ環境やリリース用に専用ブランチを維持することはあります。例えば、freeCodeCampのリポジトリでは`prod-staging`と`prod-current`がそれぞれステージングと本番サーバーへのデプロイをトリガーします。これにより、チームは準備と承認ができたものをマージしながら、毎回デプロイをトリガーせずに素早く反復できます。代わりに、最新の`main`をこれらの本番ブランチにマージしてデプロイを作成します。

また、「リリースブランチ」、つまり製品のメジャーバージョンを表すブランチを維持することもあります。これにより、各メジャーリリースのクリーンな状態を保てるため、重要なセキュリティパッチを過去のリリースにバックポートする際に、新しいバージョンの変更を含めずに済みます。

最後に、ローカルのコマンドラインインターフェースを使って`main`に直接ブランチをマージするのは避けてください。代わりに、変更を含むプルリクエストを作成し、そのフローでマージしてください。これにより、CIでテストが通るか確認でき（これは今後のレッスンで学びます）、差分ビューでマージしたい変更だけが含まれているかを再確認できます。

これらのヒントを活用して、ブランチをうまく使いこなしていきましょう！
