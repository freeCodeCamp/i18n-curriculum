# --description--

Ми вже трохи говорили про Git і GitHub, тож тепер можемо почати все це поєднувати. Для цього вам потрібен репозиторій. Але що таке репозиторій? Ви можете уявити репозиторій як контейнер для проєкту — якщо ви працюєте над застосунком, ви зберігатимете файли цього застосунку разом у репозиторії.

Репозиторії можуть бути локальними на вашому комп’ютері або віддаленими на сервісі, як-от GitHub. Щоб створити репозиторій, ви можете почати або з локального, або з віддаленого — результат буде однаковим, тож обирайте той підхід, який найкраще підходить для вашого потоку роботи. Якщо ви починаєте з віддаленого репозиторію, ви можете створити його безпосередньо через вебсайт GitHub, дотримуючись цих інструкцій:

Відвідайте <a href="https://github.com" target="_blank">вебсайт GitHub</a>, переконайтеся, що ви увійшли в систему, а потім натисніть на іконку плюса у верхньому правому куті навігаційної панелі. Ви побачите кілька варіантів, але той, що вам потрібен, — це «New Repository». Виберіть його, щоб відкрити інтерфейс створення нового репозиторію.

Першою опцією, яку ви побачите, буде можливість вибрати шаблон. Ймовірно, у вас ще немає налаштованих шаблонів, але шаблон — це особливий тип репозиторію, який можна використовувати як стартову точку для нових репозиторіїв. Вони корисні для включення документації або шаблонів для проблем у всіх ваших проєктах.

Наступна опція — вибрати власника, ви можете обрати свій обліковий запис користувача або будь-які організації, якими ви володієте або до яких маєте доступ. Потім ви можете назвати репозиторій — загалом, бажано використовувати ім’я, яке лаконічно описує проєкт, над яким ви працюєте. Як загальна найкраща практика, у назві репозиторію слід уникати пробілів. GitHub автоматично замінить їх на дефіси.

Далі ви можете за бажанням додати опис, який з’явиться на домашній сторінці репозиторію.

Також можна вибрати, чи буде репозиторій публічним, чи приватним. Публічний репозиторій може переглядати і скачувати будь-хто. Приватний репозиторій доступний лише вам і тим, кому ви явно надасте доступ.

Нарешті, якщо ви не виберете шаблон, ви побачите опції для автоматичного створення файлів у репозиторії. Файл README з’явиться на домашній сторінці репозиторію і є чудовим способом надати документацію та описи вашого проєкту. Файл `.gitignore` дозволяє вказати файли, які Git буде «ігнорувати» — це корисно для таких речей, як залежності пакетів, файли середовища та інші, які ви не хочете зберігати. GitHub запропонує шаблони для цього файлу, які виключають загальні елементи для вказаної мови або середовища виконання. А ліцензія — це юридичний документ, який визначає, хто може використовувати ваше програмне забезпечення і що вони можуть з ним робити.

Коли всі налаштування будуть встановлені так, як вам подобається, натисніть кнопку «Create Repository», щоб завершити процес. Вас автоматично перенаправлять на посадкову сторінку вашого нового репозиторію!

Звідси ви можете натиснути кнопку «Code», щоб отримати варіанти клонування репозиторію на свій комп’ютер.

Для цього уроку можна ігнорувати вкладки «Codespaces» і «Copilot». Замість цього ми зосередимося на трьох варіантах у вкладці «Local». Перші два, HTTPS і SSH, використовуються однаково. Ви використовуєте команду `git clone` з URL, що надається в блоці, щоб клонувати віддалений репозиторій на свій комп’ютер. Різниця між ними полягає у методах з’єднання та автентифікації.

Для клонування публічного репозиторію вам фактично не потрібно автентифікуватися. Але щоб клонувати приватний репозиторій або відправляти зміни до публічних чи приватних репозиторіїв, потрібно бути автентифікованим. Віддалений URL HTTPS використовувався для автентифікації через ваше ім’я користувача GitHub і пароль. Однак через проблеми з безпекою це більше не підтримується, і тепер потрібно використовувати ім’я користувача та токен доступу. Навіть тоді це значно менш безпечний метод, ніж SSH, який використовує ваші приватні та публічні ключі. Це має додаткову перевагу автоматичного входу, замість того, щоб щоразу вводити токен. Окремий урок детальніше розгляне SSH і як він працює, але ви також можете ознайомитися з документацією GitHub про налаштування SSH.

Щоб клонувати репозиторій, ви можете виконати це в терміналі:

```sh
git clone git@github.com:your-username/your-repository-name
```

Це створить папку на вашому комп’ютері з назвою вашого репозиторію і міститиме всі файли з вашого репозиторію. Звідти ви можете `cd` (змінити директорію) на ваш клонований репозиторій за допомогою:

```sh
cd name-of-cloned-repo
```

Крім того, локальна копія вашого репозиторію матиме віддалений вказівник на копію на GitHub. Щоб перевірити ваші віддалені підключення, можна виконати таку команду:

```sh
git remote -v
```

Ще один варіант клонування репозиторію — використати GitHub CLI. Цей інструмент дозволяє виконувати специфічні для GitHub завдання, не виходячи з командного рядка. Якщо він у вас не встановлений, інструкції з встановлення можна знайти в документації GitHub — але він має бути доступний у менеджері пакетів вашої системи.

Щоб клонувати репозиторій за допомогою CLI, я б використав:

```sh
gh repo clone your-username/your-repository-name
```

Кінцевий результат цих команд однаковий: у вас буде локальна копія репозиторію.

Ви також можете використовувати GitHub CLI для створення репозиторію:

```sh
gh repo create
```

Якщо не передавати жодних аргументів, CLI запропонує інтерактивний майстер для налаштування репозиторію. Цей майстер надасть усі ті ж опції, що й вебінтерфейс. За допомогою цього інструменту ви також отримаєте можливість одразу клонувати новий репозиторій, тож не доведеться виконувати ручні кроки, описані раніше.

Тепер ви маєте базове розуміння того, як працюють репозиторії і як створити один, щоб почати правильно версіонувати свою роботу. У майбутніх уроках ви навчитеся, як відправляти існуючий локальний репозиторій на GitHub і як відправляти зміни до віддалених репозиторіїв.
