## 인증과 권한

**인증**은 사용자의 신원을 확인하는 과정입니다. *"이 사람이 주장하는 그 사람인가?"*라는 질문에 답합니다. 사용자가 로그인하면 백엔드 서버가 등록 시 저장된 기록과 자격 증명을 비교합니다.

백엔드 개발자는 세 가지 주요 인증 요소 범주를 사용합니다:

- **알고 있는 것:** 비밀번호, PIN, 또는 보안 질문 답변.
- **가지고 있는 것:** 스마트폰의 SMS 코드 또는 인증 앱에서 발급된 임시 토큰.
- **본인인 것:** 지문이나 얼굴 인식 스캔 같은 생체 정보.

**권한**은 인증 후에 이어서 인증된 사용자가 무엇을 할 수 있는지 결정합니다. *"이 신원에 어떤 권한이 있는가?"*라는 질문에 답합니다. 일반적인 역할은 다음과 같습니다:

- **표준 사용자:** 자신의 프로필을 수정할 수 있지만 다른 사용자의 개인 데이터를 볼 수 없습니다.
- **편집자:** 콘텐츠를 작성하고 게시할 수 있지만 코드베이스를 삭제할 수 없습니다.
- **관리자:** 데이터베이스 수정, 사용자 삭제, 시스템 전체 설정 변경 등 모든 권한을 가집니다.

**항상 먼저 인증이 이루어져야 하며, 그 다음에 권한이 결정됩니다.** 사용자가 누구인지 알기 전에는 어떤 권한을 부여할지 결정할 수 없습니다.

## JSON 웹 토큰(JWT)

JWT는 서버가 로그인 성공 후 사용자에게 발급하는 디지털 키카드입니다. 사용자는 이후 모든 요청에 이 토큰을 제시하여 비밀번호를 다시 입력하지 않고도 신원을 증명합니다.

JWT의 네 단계 흐름:

1. **로그인:** 사용자가 자격 증명을 서버에 보냅니다.
2. **발급:** 서버가 자격 증명을 확인하고 JWT를 생성해 반환합니다.
3. **저장:** 클라이언트는 JWT를 보통 브라우저 쿠키나 로컬 저장소에 저장합니다.
4. **요청:** 보호된 라우트에 대한 모든 이후 요청에 클라이언트가 JWT를 보내고 서버가 이를 검증합니다.

JWT는 두 개의 점으로 구분된 세 부분으로 구성됩니다:

- **헤더:** 토큰 유형(JWT)과 HMAC SHA256 같은 해싱 알고리즘을 선언하는 인코딩된 JSON.
- **페이로드:** 사용자 ID, 사용자 이름, 역할 같은 클레임을 담은 인코딩된 JSON. 헤더와 페이로드는 Base64로 인코딩되며 암호화되지 않으므로 비밀번호 같은 민감한 데이터는 절대 저장하지 마세요.
- **서명:** 인코딩된 헤더와 페이로드를 서버만 아는 비밀 키로 해싱해 생성합니다. 페이로드가 변조되면 서명이 일치하지 않아 서버가 `401 Unauthorized` 응답으로 요청을 거부합니다.

JWT는 **무상태 인증**을 가능하게 합니다. 서명이 토큰을 생성한 것을 증명하므로 서버는 데이터베이스에 세션 데이터를 저장할 필요가 없습니다.

## 교차 사이트 요청 위조(CSRF)

CSRF는 인증된 사용자가 모르는 사이에 웹 애플리케이션에서 원하지 않는 동작을 실행하도록 강요하는 공격입니다. 악성 웹사이트가 피해자의 브라우저를 속여 백엔드 서버에 위조된 요청을 보내게 만드는데, 브라우저가 대상 서버로 가는 모든 요청에 쿠키를 자동으로 첨부하는 점을 악용합니다.

**공격 예시 흐름:**

1. 예를 들어 `trusted-bank.com`에 로그인합니다. 은행은 세션 토큰을 브라우저 쿠키에 저장합니다.
2. 로그아웃하지 않고 `evil-hacker-site.com`을 방문합니다.
3. 악성 사이트가 조용히 `trusted-bank.com/transfer?amount=1000&to=hacker`로 요청을 보냅니다.
4. 브라우저가 자동으로 은행 세션 쿠키를 첨부해 서버가 요청을 처리합니다.

악성 사이트는 쿠키를 읽을 수 없습니다. 단지 브라우저가 대상 서버로 가는 모든 요청에 쿠키를 자동으로 첨부한다는 점을 이용할 뿐입니다.

**CSRF 방어 방법:**

- **`SameSite=Strict`:** 요청이 제3자 웹사이트에서 시작되면 브라우저가 쿠키를 절대 보내지 않습니다.
- **`SameSite=Lax`:** 안전한 탐색(링크 클릭) 시 쿠키를 허용하지만, 교차 사이트 폼 제출과 API 요청에서는 차단합니다.
- **CSRF 토큰:** 사용자가 페이지를 로드할 때 서버가 무작위, 고유, 예측 불가능한 토큰을 생성해 프런트엔드 폼이나 세션에 첨부합니다. 클라이언트는 요청 헤더나 폼 데이터에 이 토큰을 포함해야 합니다. 악성 사이트는 프런트엔드에 접근해 토큰을 복사할 수 없으므로 위조 요청은 유효성 검사에 실패합니다.

## Passport.js

Passport.js는 Node.js용 인증 미들웨어로, **전략(Strategies)**이라는 모듈식 플러그인을 사용해 요청 인증 방식을 표준화합니다. 500개 이상의 전략이 있어 관련 패키지를 설치하면 어떤 인증 방법도 추가할 수 있습니다:

- `passport-local`: 자체 데이터베이스를 이용한 사용자 이름과 비밀번호.
- `passport-jwt`: 보호된 API 라우트에서 JSON 웹 토큰 검증.
- `passport-google-oauth20`: 구글 계정으로 로그인.
- `passport-github2`: GitHub 계정으로 로그인.

핵심 패키지와 전략을 설치하세요:

```bash
npm install passport passport-local
```

라우트를 정의하기 전에 Passport를 Express 애플리케이션에 연결하세요:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database look-up and password verification logic
}));
```

Passport를 미들웨어로 사용해 로그인 라우트를 보호하세요:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

인증 성공 후 Passport는 자동으로 **세션 관리**(사용자 데이터를 세션 쿠키에 저장)와 **요청 장식**(`req.user`에 사용자 객체 첨부, 이후 라우트 핸들러에서 사용 가능)을 처리합니다.

## Helmet.js

Helmet.js는 Node.js용 보안 미들웨어 모음으로, Express가 기본적으로 노출하는 HTTP 응답 헤더를 자동으로 설정, 수정, 제거해 애플리케이션을 보호합니다.

한 번에 설치하고 적용하세요:

```bash
npm install helmet
```

```js
const helmet = require('helmet');
app.use(helmet());
```

미들웨어 스택 최상단에 `app.use(helmet())`를 배치하면 모든 응답에 다음 보호 기능이 적용됩니다:

- **X-Powered-By 제거:** Express는 기본적으로 `X-Powered-By: Express` 헤더로 존재를 알립니다. Helmet은 이 헤더를 제거해 공격자가 버전별 취약점을 노리지 못하게 합니다.
- **Content-Security-Policy 설정:** 브라우저가 스크립트, 이미지, 스타일을 로드할 수 있는 출처를 제한해 교차 사이트 스크립팅(XSS) 공격을 방어합니다.
- **X-Frame-Options를 SAMEORIGIN으로 설정:** 사이트가 외부 iframe에 임베드되는 것을 막아 클릭재킹 공격을 차단합니다.
- **Strict-Transport-Security 설정:** 브라우저가 서버와 HTTPS로만 통신하도록 강제해 신뢰할 수 없는 네트워크에서 데이터 가로채기를 방지합니다.
