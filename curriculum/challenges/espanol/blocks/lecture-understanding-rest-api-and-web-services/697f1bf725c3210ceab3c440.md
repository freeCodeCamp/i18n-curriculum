# --description--

Los servicios web y las APIs REST impulsan la mayoría de las aplicaciones y sitios web que usas todos los días. Permiten que diferentes aplicaciones se comuniquen entre sí y compartan datos a través de una red, comúnmente internet.

Una API (Interfaz de Programación de Aplicaciones) es una forma general para que el software se comunique con otro software. Un servicio web es un tipo específico de API que utiliza tecnologías web, como HTTP, para intercambiar datos a través de una red, usualmente internet. Por esto, todo servicio web es una API, pero no toda API es un servicio web. Las APIs REST son un tipo común de servicio web que siguen un conjunto específico de reglas sobre cómo se solicitan y devuelven los datos.

Esta lección se centrará en qué son los servicios web a un nivel general, y la siguiente profundizará más en cómo funcionan las APIs REST.

Puedes pensar en un servicio web como un puente de software a software. No tiene interfaz de usuario y no está diseñado para ser usado directamente. En cambio, se centra completamente en el intercambio de datos entre aplicaciones.

Para que cualquier sistema sea considerado un servicio web, debe tener las siguientes características:

* **Se ejecuta en un servidor**: un servicio web debe estar alojado en línea como un sitio web, pero en lugar de páginas que los usuarios puedan ver, expone datos o alguna funcionalidad
    
* **Comunicación máquina a máquina**: un servicio web no está hecho para que los humanos lo vean directamente, sino para que las aplicaciones móviles y web lo consuman, de modo que puedan exponer algunos datos o funcionalidades que el usuario finalmente pueda ver
    
* **Independiente del lenguaje**: un servicio web puede estar escrito en cualquier lenguaje como Java o C#, y cualquier aplicación debería poder interactuar con él, incluso si está escrito en un lenguaje diferente
    
* **Independiente de la plataforma**: no importa en qué plataforma se ejecuten las máquinas. Por ejemplo, un servicio web que se ejecuta en Linux no debería impedir que un dispositivo con Windows se comunique con él
    
* **Usa protocolos web estándar**: un servicio web normalmente debe usar HTTP/HTTPS para transferir datos por internet, FTP para transferencia de archivos y SMTP para correo electrónico
    

**REST** y **SOAP** son dos de los tipos de servicios web más conocidos, aunque existen otros enfoques, como **XML-RPC**.

**REST**, que significa **Representational State Transfer**, es con mucho el más utilizado hoy en día. La mayoría de las aplicaciones modernas dependen de servicios web RESTful porque son simples, flexibles y funcionan de manera natural con la web.

Las APIs REST exponen recursos, que generalmente se representan mediante rutas URL como `/users` o `/users/20`. Cuando un cliente solicita un recurso, el servidor devuelve una representación de ese recurso. Esta representación suele estar en JSON, pero también puede estar en formatos como XML o incluso texto plano.

Los clientes pueden especificar el formato que desean usando encabezados HTTP, como el encabezado `Accept`. Por ejemplo, un cliente puede solicitar una respuesta JSON configurando el valor adecuado de `Accept`.

```bash
Accept: application/json
```

Y puedes especificar que quieres XML de esta manera:

```bash
Accept: application/xml
```

El recurso que recibes puede verse así si es JSON:

```json
{
  "id": 20,
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "createdAt": "2025-09-22T10:30:00Z"
}
```

Y esto, si es XML:

```xml
<user>
  <id>20</id>
  <name>John Doe</name>
  <email>john@example.com</email>
  <role>admin</role>
  <createdAt>2025-09-22T10:30:00Z</createdAt>
</user>
```

SOAP significa **Simple Object Access Protocol**. Solo usa el formato de datos XML, y es el más común en sistemas a nivel empresarial que necesitan estándares y seguridad más estrictos.

Así es como podrían verse los datos SOAP:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
   <soapenv:Body>
      <getUser>
         <userId>20</userId>
      </getUser>
   </soapenv:Body>
</soapenv:Envelope>
```

XML-RPC usa XML para codificar sus solicitudes y respuestas. Esto lo hace ligero pero menos flexible que REST. 

En cuanto a UDDI, no es un servicio de intercambio de datos sino un directorio basado en XML donde las empresas pueden listar, descubrir y describir sus servicios web disponibles.
