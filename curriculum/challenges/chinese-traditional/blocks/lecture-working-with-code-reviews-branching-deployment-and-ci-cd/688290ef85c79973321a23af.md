# --description--

在之前的課程中，你學會了如何對你自己的儲存庫提出 pull request。你也學會了如何合併該 pull request。但你只能這麼做，是因為該儲存庫屬於你－你對你自己的儲存庫擁有維護者存取權，這表示你有權限合併 pull request。

然而，當你為別人的開源專案做出貢獻時，你通常不會擁有那些權限。相反地，你需要等待維護者通過程式碼審查流程並接受你的貢獻。然後他們會控制代碼合併。

但是程式碼審查的處理程序是什麼樣子？

維護者採取的處理程序會因人而異，且可能有專案特定的方法。但一般來說，程式碼審查通常有三個步驟。

第一點是確認 CI，或稱持續整合，是否通過。你將在未來的課程中學到更多相關內容，所以目前你可以將它們視為用來驗證程式碼是否符合風格準則並通過測試套件的檢查。

如果 CI 失敗，維護者通常會要求你更新 pull request 以解決失敗問題。這些失敗應該會包含某種報告，讓你可以查看錯誤原因，但維護者也可能會提供你更具體的指導來決議問題。

一旦檢查通過，維護者將會查看你實際的程式碼變更。他們可以分析差異檢視來了解你修改、添加或刪除了什麼。他們可能會詢問你為何採取特定的方法，因此在準備討論時，謹慎且深思熟慮是很重要的。

他們也可能會要求你做額外的修改。我們稍後會討論那是什麼樣子。

最後一步通常是將你的變更拉取到區域。維護者接著可以使用你的修改來執行應用程式、測試功能，並確認沒有任何東西損壞或出現異常行為。

如果一切看起來沒問題，維護者會批准你的 pull request——有時會附帶「LGTM」評論，代表「looks good to me」。在你的 pull request 被批准後，應該在維護者準備好合併變更時進行合併。有些專案可能需要超過一次的批准——例如 freeCodeCamp 需要兩次。

但是如果你的 pull request 有錯誤怎麼辦？維護者可以進行審查，並要求變更。這些請求可以有幾種形式。

有時候，維護者可能只會留下帶有他們請求的「comment」，可能指出你應該更改或更新的內容，但在如何實作上給予你更多自由。

或者，維護者也可能會留下官方的「request changes」評論。通常，這類評論會阻止 pull request 被合併。你需要進行所需的更改，然後同一位要求更改的維護者將需要重新審查並要求更多更改或批准你的工作。

當維護者有具體的變更想法時，他們實際上可以透過 GitHub 的 UI 提出變更。當他們使用像這樣的直接建議時，GitHub 會給你一個按鈕，讓你可以直接將建議提交到你的分支。然而，如果有多個這樣的建議，你會想切換到檔案檢視。

從此視圖中，你可以將所有建議（一個接一個）添加到一個「批次」中，並一次性提交它們。當有許多建議時，這非常有用，因為逐一處理會導致許多提交。所有的 CI 檢查都必須在每一次提交上執行，這會減慢接受你的 PR 的處理程序。

如果請求的變更幅度很大，你很可能會想要在區域的進行變更，將它們提交到你用於 PR 的相同分支，然後推送上去。GitHub 會自動更新你的 pull request，以反映新的提交。

還有幾個其他的注意事項我們想要涵蓋。

首先，請記住維護者也是人。開放原始碼專案很少會產生足夠的收入讓維護者全職投入，這表示他們必須同時維持一份工作。對於審查流程保持耐心非常重要——不要反覆提醒維護者重新審查，或試圖催促他們加快審查進度。

參與貢獻可以很令人興奮！但不要急於完成這個過程。相反地，請花時間。確保在區域的環境中測試你的變更－執行專案可能有的任何測試套件、執行開發伺服器並手動確認你的變更是否有效，等等。並且始終遵循你正在參與的特定專案的貢獻指南。

如果你牢記這些並全力以赴，你的貢獻應該會順利進行，你的程式碼審查也應該會獲得堅實的批准！
