# --description--

Haben Sie sich schon einmal gefragt, wie Sie Ihre Express-App stabil und produktionsbereit halten, besonders wenn sie stark belastet wird oder heruntergefahren wird?

Hier kommen Health Checks und Graceful Shutdowns ins Spiel. Diese beiden Muster helfen Ihnen, Ihre App verfügbar zu halten und sicherzustellen, dass sie sauber beendet wird, ohne Anfragen zu verlieren oder Daten zu beschädigen.

Lassen Sie uns das genauer ansehen.

Zuerst: Was ist überhaupt ein „Health Check“ in Express?

Ein Health Check ist eine einfache Route, die angibt, ob Ihre App läuft.

Zum Beispiel:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

Werkzeuge wie Load Balancer oder Container-Orchestratoren – denken Sie an Kubernetes oder AWS Elastic Beanstalk – können diese Route anpingen, um zu entscheiden, ob Ihre App gesund ist.

Wenn die Route einen Statuscode 200 zurückgibt, gilt Ihr Dienst als gesund. Zur Erinnerung – ein HTTP-Statuscode 200 bedeutet, dass alles in Ordnung ist!

Wenn keine Antwort kommt, kann der Orchestrator den Dienst automatisch neu starten.

Und was ist ein „Graceful Shutdown“?

Ein Graceful Shutdown ist das, was passiert, wenn Ihre App bald gestoppt wird, zum Beispiel wenn ein Server neu startet oder ein Deployment durchgeführt wird.

Anstatt Verbindungen sofort abzubrechen, geben Sie der App Zeit, ihre laufenden Aufgaben zu beenden.

In Express können Sie auf `SIGTERM` hören, ein Systemsignal, das gesendet wird, wenn ein Prozess beendet werden soll.

Hier ein Beispiel:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

Das sagt Express, keine neuen Verbindungen mehr anzunehmen und auf die Fertigstellung der bestehenden zu warten, bevor es beendet wird.

Sie sollten auch `SIGINT` behandeln, das Signal, das gesendet wird, wenn Sie während der lokalen Entwicklung Strg+C drücken, um Ihre App zu stoppen.

Während `SIGTERM` typischerweise von Orchestratoren wie Kubernetes in der Produktion verwendet wird, deckt `SIGINT` den alltäglichen Entwickler-Workflow ab. Ohne Behandlung wird durch Strg+C der Prozess sofort beendet, ohne dass eine Aufräumaktion ausgeführt wird.

Da beide Signale dasselbe Verhalten für einen Graceful Shutdown auslösen sollten, hören Entwickler üblicherweise auf beide.

Jetzt fassen wir alles zusammen

Stellen Sie sich vor, Ihre App läuft in einem Container. Kubernetes sendet ein `SIGTERM`, um sie für ein Update herunterzufahren.

Mit einem einfachen Graceful Shutdown hört Ihre App auf, neue Verbindungen anzunehmen, und beendet die laufenden Anfragen. Sobald alle Aufgaben erledigt sind, trennt sie die Verbindung zur Datenbank und beendet sich sauber. Keine verlorenen Daten, kein beschädigter Zustand.

Aber es gibt ein Problem – der Load Balancer weiß nicht, dass Ihre App herunterfährt, und könnte weiterhin neuen Traffic an Ihre App senden. Hier kommt das Bonus-Muster unten ins Spiel.

Jetzt habe ich noch ein kleines Extra für Sie. Wir schauen uns an, wie man die App vor dem Herunterfahren als „ungesund“ markiert.

Vor dem Herunterfahren können Sie Ihren Health Check so anpassen, dass er einen Status 503 zurückgibt. So sieht der Load Balancer die ungesunde Antwort und stoppt die Weiterleitung von neuem Traffic an Ihre App während des Shutdown-Zeitraums, wie im obigen Code gezeigt.

Zur Erinnerung: Ein `503`-Statuscode bedeutet „Service Unavailable“. Er signalisiert, dass der Server vorübergehend keine Anfragen bearbeiten kann – genau das passiert während eines Graceful Shutdowns.

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
