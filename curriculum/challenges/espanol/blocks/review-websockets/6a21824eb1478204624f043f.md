## WebSockets vs HTTP

- **HTTP** sigue un modelo de solicitud-respuesta: el cliente envía una solicitud, el servidor envía una respuesta y luego la conexión se cierra.
- **WebSockets** crean una conexión permanente y bidireccional entre un cliente y un servidor que permanece abierta hasta que alguna de las partes envía una señal de cierre.
- HTTP es adecuado para cargar páginas o obtener datos, pero se vuelve lento e ineficiente para aplicaciones en tiempo real.
- Cada conexión WebSocket comienza con un **handshake HTTP**: el cliente solicita una actualización y el servidor responde con `101 Switching Protocols`.

### Flujo de conexión WebSocket

1. El cliente se conecta al servidor vía HTTP y solicita una actualización.
2. El servidor acepta y responde con `101 Switching Protocols`.
3. La conexión permanece abierta.
4. Ambas partes intercambian datos en cualquier momento hasta que la conexión se cierra.

### Comunicación HTTP vs WebSocket

- **HTTP**: `Client → Request → Server → Response` (un ciclo, luego se cierra)
- **WebSocket**: `Client ↔ Server` (persistente, bidireccional)

### Una analogía útil

- **HTTP** es como intercambiar cartas: escribes una, esperas una respuesta y la conversación termina.
- **WebSockets** son como una llamada telefónica: ambas partes hablan en tiempo real durante el tiempo que sea necesario.

### Usos reales de WebSockets

- Aplicaciones de chat
- Juegos multijugador en línea
- Actualizaciones en vivo de resultados deportivos
- Paneles de mercado bursátil
- Herramientas colaborativas (por ejemplo, Google Docs)

## Arquitectura Pub/Sub

- **Pub/Sub** (Publicar/Suscribir) es una arquitectura de mensajería que permite que partes de un sistema se comuniquen sin conocerse directamente.
- Hay tres partes principales: **publicadores**, **temas** y **suscriptores**.
  - Un **publicador** envía mensajes a un tema.
  - Un **tema** (también llamado canal o grupo) actúa como intermediario.
  - Un **suscriptor** recibe mensajes de los temas a los que se ha suscrito.

### Flujo de mensajes Pub/Sub

1. Un suscriptor se suscribe a un tema.
2. Un publicador envía un mensaje a ese tema.
3. El sistema de mensajería entrega el mensaje a todos los suscriptores de ese tema.

### Comunicación directa vs Pub/Sub

- **Sin Pub/Sub**: `Application A → Application B` — acoplamiento fuerte; un cambio en B puede romper a A.
- **Con Pub/Sub**: `Application A → Topic → Subscribers` — A solo publica; no necesita saber quién recibe.

### Usos reales de Pub/Sub

- Aplicaciones de chat con múltiples salas
- Juegos multijugador en línea
- Sistemas de notificaciones
- Paneles en vivo
- Aplicaciones orientadas a eventos

### WebSockets vs Pub/Sub

- **WebSockets** proveen el canal de comunicación (la carretera).
- **Pub/Sub** provee el patrón de mensajería que determina quién recibe mensajes específicos (el sistema de tráfico).
- Muchas aplicaciones en tiempo real usan ambos juntos: WebSockets mantienen la conexión abierta, mientras Pub/Sub enruta los mensajes a los suscriptores correctos.

## Trabajando con WebSockets en Node.js

- Node.js no tiene un servidor WebSocket integrado. La biblioteca `ws` es el paquete externo estándar para esto.
- Instálalo con: `npm install ws`

### Crear un servidor WebSocket

```js
const WebSocket = require("ws");

const server = new WebSocket.Server({ port: 3000 });
```

### Eventos WebSocket

| Evento | Desencadenante |
|---|---|
| `connection` | Un nuevo cliente se conecta al servidor |
| `message` | Un cliente conectado envía datos |
| `close` | Una conexión se cierra |
| `error` | Algo falla en la conexión |

### Enviar un mensaje a un cliente

```js
socket.send("Hello from the server");
```

### Transmitir a todos los clientes conectados

- `server.clients` contiene todos los clientes conectados.
- Verifica `readyState === WebSocket.OPEN` antes de enviar para evitar errores con clientes desconectados.

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **Transmitir** es el patrón de enviar un mensaje a varios clientes conectados a la vez — se usa en aplicaciones de chat, notificaciones en vivo, actualizaciones deportivas y eventos de juegos multijugador.
