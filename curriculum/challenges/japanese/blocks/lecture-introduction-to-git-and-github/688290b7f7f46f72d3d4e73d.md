# --description--

コミットを作成して自分が管理するリポジトリにプッシュする方法を学びました。管理していない他のプロジェクトにも同じことができますが、いくつか追加のステップがあります。

まず第一に、一般的に他人のプロジェクトには「書き込みアクセス権」がありません。つまり、直接コミットをプッシュすることはできません。その代わりに、作業用にリポジトリのコピーを作成する必要があります。このプロセスを「フォーク」と呼びます。

リポジトリをフォークするには、GitHubで貢献したいリポジトリにアクセスしてください。上部に「Fork」ボタンが表示されます。これをクリックすると、フォーク作成のUIに移動します。

自分のリポジトリを作成したときと似た画面が表示されますが、オプションは少なめです。所有者として自分の個人アカウントかアクセス権のある組織を選択でき、フォークの新しい名前や説明を設定できます。

また、`main` ブランチ（またはリポジトリでデフォルト設定されているブランチ）だけをコピーするオプションもあります。特定の機能ブランチに貢献する必要がない限り、これは一般的に良い選択です。ブランチについては今後のレッスンで詳しく学びます。今はそのボックスにチェックを入れたままにしてください。

「Create fork」をクリックすると新しいリポジトリが作成され、自動的にそのページに移動します。

次に、自分のリポジトリと同じようにSSHやGitHub CLIを使ってローカルコンピューターにクローンしてください。

では、ローカルクローンに設定されているリモートを見てみましょう。ローカルリポジトリのディレクトリでターミナルを開き、`git remote -v` を実行すると、次のようなリストが表示されるはずです。

```sh
origin git@github.com:your-username/your-repo-name.git (fetch)
origin git@github.com:your-username/your-repo-name.git (pull)
```

GitHubからリポジトリをクローンすると、Gitは自動的にフォークを`origin`リモートとして設定します。しかし、フォークから作業するときは、元のリポジトリから変更を取り込むために自分のコピーを更新する必要がよくあります。

そのために、元のリポジトリを別のリモートとして追加しましょう。フォーク元のページに戻り、「Code」ボタンをクリックしてSSHのURLを取得します。次に、新しいアップストリームリモートを作成するコマンドを実行します。

```sh
git remote add upstream git@github.com:original-repo-username/repo-name.git
```

`upstream` はリモートの名前です。別の名前にしても構いませんが、`upstream` が慣例としてよく使われます。もう一度リモートを確認すると、新しいアップストリームリモートが表示されているはずです。

```sh
origin git@github.com:your-username/your-repo-name.git (fetch)
origin git@github.com:your-username/your-repo-name.git (pull)
upstream git@github.com:original-repo-username/your-repo-name.git (fetch)
upstream git@github.com:original-repo-username/your-repo-name.git (pull)
```

`upstream` リモートを使うと、`git fetch upstream` や `git merge upstream/main` のように元のプロジェクトから最新の変更をローカルのフォークに取り込むことができます。

フォークはリポジトリの一種なので、コミットを作成して変更をプッシュする方法は同じです。ただし、他人のプロジェクトに変更を加える際には守るべき戦略があります。これらの戦略については今後のレッスンで詳しく学びます。

ソースコードを公開し、コードの修正を許可し、コードの貢献を受け入れるプロジェクトは「オープンソース」プロジェクトと呼ばれます。オープンソースには、コミュニティがバグの発見や修正を手伝ってくれたり、他の人の貢献から学べたり、開発者のコミュニティを育ててネットワークを作れるなど、多くの利点があります。

他人のプロジェクトに変更を加えると、「オープンソースコントリビューター」と見なされます。オープンソースプロジェクトに貢献することは、プロの開発チームで働く体験の良い予行演習になります。ここで学ぶ戦略は、その後の最初の有給の役割にも役立ちます。

もし貢献方法について質問があれば、ほとんどのオープンソースプロジェクトには貢献ガイドラインがあり、必要な情報がまとめられています。それでもわからなければ、リポジトリにイシューを立てて質問したり、Discordのようなプラットフォームにコミュニティスペースがあれば参加して質問してみましょう。
