## Maktaba na mifumo ya JavaScript

- Maktaba na mifumo ya JavaScript hutoa suluhisho za haraka kwa matatizo ya kawaida na huongeza kasi ya ukuzaji kwa kutoa msimbo uliokwisha andaliwa.
- Maktaba kwa kawaida huzingatia zaidi kutoa suluhisho kwa kazi maalum, kama vile kubadilisha DOM, kushughulikia matukio, au kusimamia maombi ya AJAX.
- Mifano kadhaa ya maktaba za JavaScript ni jQuery na React.
- Kwa upande mwingine, mifumo hutoa muundo ulioelezwa zaidi kwa ajili ya kujenga programu. Mara nyingi huja na seti ya kanuni na mikataba ambayo wasanidi programu wanapaswa kufuata.
- Mifano ya mifumo ni pamoja na Angular na Next.js, ambayo ni meta framework kwa React.
- **Single-page applications** (SPAs) ni programu za mtandao ambazo hupakia ukurasa mmoja wa HTML na kusasisha ukurasa huo kwa njia ya kidinamikia kadri mtumiaji anavyoingiliana na programu bila kupakia upya ukurasa mzima.
- SPAs hutumia JavaScript kusimamia hali ya programu na kuonyesha maudhui. Hii mara nyingi hufanyika kwa kutumia mifumo ambayo hutoa zana bora za kujenga miingiliano changamano ya mtumiaji.
- Baadhi ya changamoto zinazohusiana na SPAs ni:

  - Visomaji cha skrini hupata ugumu na maudhui yanayosasishwa kidinamikia.
  - URL hubaki vilevile mtumiaji anaposafiri ndani ya programu, jambo ambalo linaweza kufanya iwe vigumu kuweka alama, kurudi nyuma au kushiriki kurasa maalum.
  - Muda wa awali wa kupakia unaweza kuwa mrefu ikiwa programu ni kubwa kwa sababu rasilimali zote zinahitaji kupakiwa mwanzo.

## React

- React ni maktaba maarufu ya JavaScript kwa ajili ya kujenga miingiliano ya mtumiaji na programu za mtandao.
- Dhana kuu ya React ni uundaji wa sehemu za UI zinazoweza kutumika tena ambazo zinaweza kusasishwa na kuonyeshwa upya kwa kujitegemea kadri data inavyobadilika.
- React inaruhusu wasanidi kuelezea jinsi UI inavyopaswa kuonekana kulingana na hali ya programu. React kisha inasasisha na kuonyesha sehemu sahihi wakati data au hali inabadilika.

## Sehemu za React

- Sehemu ni vipande vya msingi vya programu za React vinavyoruhusu wasanidi kugawanya miingiliano changamano ya mtumiaji kuwa vipande vidogo vinavyoweza kudhibitiwa.
- UI inaelezwa kwa kutumia JSX, ambayo ni upanuzi wa sintaksia ya JavaScript, unaoruhusu wasanidi kuandika msimbo unaofanana na HTML ndani ya JavaScript.
- Sehemu kimsingi ni kazi za JS au madarasa yanayorejesha kipande cha UI.

Hapa kuna mfano wa sehemu rahisi ya React inayotoa ujumbe wa salamu:

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

Ili kutumia sehemu hii, unaweza kuita tu:

```jsx
  <Greeting />
```

## Kulete na kusafirisha sehemu za React

- Kusafirisha (export) kunafanya sehemu ipatikane kwa mafaili mengine. Kulete (import) kunaruhusu faili kutumia sehemu iliyosafirishwa mahali pengine.
- Kuna aina mbili za usafirishaji katika React: usafirishaji chaguo-msingi (default exports) na usafirishaji wa majina (named exports).

**Usafirishaji Chaguo-msingi**

- Faili inaweza kuwa na usafirishaji mmoja tu wa chaguo-msingi. Hii inafaa kwa faili inayobeba sehemu moja kuu.
- Unaweza kusafirisha sehemu kama usafirishaji chaguo-msingi kwa njia hii:

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default City;
```

- Pia unaweza kusafirisha kwenye mstari mmoja na ufafanuzi wa sehemu:

```jsx
export default function City() {
  return <p>New York</p>;
}
```

- Ili kuleta usafirishaji chaguo-msingi, tumia neno `import` bila mabano ya mviringo:

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

**Usafirishaji wa Majina**

- Usafirishaji wa majina huruhusu faili kushiriki sehemu nyingi. Tofauti na usafirishaji chaguo-msingi, lazima ziletwe kwa jina lilelile zililosafirishwa nalo.
- Unaweza kusafirisha sehemu nyingi kutoka faili moja kwa njia hii:

```jsx
// Animals.jsx
export function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- Ili kuleta usafirishaji wa majina, tumia mabano ya mviringo na majina kamili yaliyosafirishwa:

```jsx
import { Cat, Dog } from './Animals';
```

- Unaweza kubadilisha jina la usafirishaji wa majina wakati wa kuleta kwa kutumia neno `as`:

```jsx
import { Cat as Kitty } from './Animals';
```

**Mchanganyiko wa Usafirishaji Chaguo-msingi na wa Majina**

- Faili inaweza kuwa na usafirishaji mmoja wa chaguo-msingi na usafirishaji wa majina kadhaa kwa wakati mmoja:

```jsx
// Animals.jsx
export default function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- Unapolete mchanganyiko wa usafirishaji, usafirishaji chaguo-msingi huja kwanza (bila mabano), ukifuatiwa na usafirishaji wa majina (ndani ya mabano):

```jsx
import Cat, { Dog } from './Animals';
```

## Kuweka mradi wa React kwa kutumia Vite

- Zana za kuanzisha miradi na kiolesura cha mstari wa amri (CLI) hutoa njia rahisi na ya haraka ya kuanzisha miradi mipya, zikikuruhusu uzingatie kuandika msimbo badala ya kushughulika na mipangilio.
- Vite ni zana maarufu ya kuanzisha miradi na inaweza kutumika na React.
- Ili kuunda mradi mpya kwa Vite, unaweza kutumia amri ifuatayo kwenye terminal yako:

```bash
npm create vite@latest my-react-app -- --template react
```

Amri hii inaunda mradi mpya wa React unaoitwa `my-react-app` kwa kutumia kiolezo cha React cha Vite. Ndani ya folda ya mradi, utaona faili ya `package.json` yenye utegemezi wa mradi na amri zilizoorodheshwa ndani yake.

- Ili kuendesha mradi, nenda kwenye folda ya mradi na endesha amri zifuatazo:

```bash
cd my-react-app # path to the project directory
npm install # installs the dependencies listed in the package.json file
```

- Mara utegemezi utakapowekwa, utagundua folda mpya kwenye mradi wako inayoitwa `node_modules`.

- Folda ya `node_modules` ndiyo inahifadhi vifurushi na maktaba zote zinazohitajika na mradi wako.

- Ili kuendesha mradi wako, tumia amri ifuatayo:

```bash
npm run dev
```

- Baada ya hapo, fungua kivinjari chako na nenda kwenye `http://localhost:5173` ili kuona programu yako ya React ikiendesha.

- Ili kuona msimbo wa kiolezo cha kuanzia, unaweza kuingia kwenye mradi wako ndani ya folda ya `src` na utaona faili ya `App.jsx`.

## Kupitisha props kwenye sehemu za React

- Katika React, props (kifupi cha properties) ni njia ya kupitisha data kutoka sehemu mzazi kwenda sehemu mtoto. Utaratibu huu ni muhimu ili kuunda vipengele vya UI vinavyoweza kutumika tena na kubadilika.
- Props zinaweza kuwa thamani yoyote ya JavaScript. Ili kupitisha props kutoka sehemu mzazi kwenda sehemu mtoto, ongeza props kama sifa unapoitumia sehemu mtoto kwenye JSX ya mzazi. Huu hapa ni mfano rahisi:

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

Unaweza kupitisha props nyingi kwa kutumia opereta ya spread `(...)`, baada ya kugeuza kuwa kitu. Huu hapa ni mfano:

```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

Katika msimbo huu, opereta ya spread `{...person}` inageuza kitu person kuwa props binafsi zinazopitishwa kwa sehemu ya Child.

## Uonyeshaji wa masharti (conditional rendering) katika React

- Uonyeshaji wa masharti katika React hukuwezesha kuunda miingiliano ya mtumiaji inayobadilika. Inatumika kuonyesha maudhui tofauti kulingana na hali fulani au state ndani ya programu yako.
- Kuna njia kadhaa za kuonyesha maudhui kwa masharti katika React. Njia moja ya kawaida ni kutumia opereta ya ternary. Huu hapa ni mfano:

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- Njia nyingine ni kutumia opereta ya AND ya kimantiki (`&&`). Hii inafaa unapopenda kuonyesha maudhui tu ikiwa hali fulani imetimia. Huu hapa ni mfano:

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

Katika msimbo hapo juu, kipengele cha `h1` kitaonyeshwa tu ikiwa kitu user ni kweli.

Unaweza pia kutumia tamko la moja kwa moja la `if` kwa njia hii:

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```

## Kuonyesha orodha katika React

- Kuonyesha orodha katika React ni kazi ya kawaida unapojenga miingiliano ya mtumiaji.
- Orodha zinaweza kuonyeshwa kwa kutumia njia ya safu ya data ya JS `map()` ili kupita juu ya safu ya vitu na kurejesha safu mpya ya vipengele vya JSX.
- Kwa mfano, ikiwa una safu ya majina unayotaka kuonyesha kama orodha, unaweza kufanya hivi:

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- Kumbuka kila wakati kutoa ufunguo wa kipekee kwa kila kipengele cha orodha ili kusaidia React kusimamia majukumu ya kusasisha na kuonyesha. Kwa mbinu hizi, unaweza kuunda orodha zinazobadilika, bora na zenye ufanisi katika programu zako za React.

## Mitindo ya ndani ya mstari (inline styles) katika React

- Mitindo ya ndani ya mstari katika React hukuwezesha kutumia mitindo ya CSS moja kwa moja kwenye vipengele vya JSX kwa kutumia vitu vya JavaScript.
- Ili kutumia mitindo ya ndani ya mstari katika React, tumia sifa ya style kwenye vipengele vya JSX. Sifa ya style inachukua kitu ambapo funguo ni sifa za CSS kwa camelCase na thamani ni zile zinazolingana. Huu hapa ni mfano:

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

Unaweza pia kutoa mitindo kwenye kitu tofauti na kukitaja kwenye sifa ya `style` kwa njia hii:

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Mitindo ya ndani ya mstari inasaidia mitindo ya kidinamikia kwa kukuwezesha kutumia mitindo kulingana na props au state. Huu hapa ni mfano wa jinsi unavyoweza kutumia mitindo kulingana na prop:

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Katika msimbo hapo juu, mitindo ya `color` na `fontSize` imewekwa kulingana na prop ya `isImportant`.

## Kufanya kazi na matukio katika React

- **Mfumo wa Tukio Bandia (Synthetic Event System)**: Hii ni njia ya React ya kushughulikia matukio. Inafanya kazi kama kifuniko juu ya matukio asilia kama matukio ya `click`, `keydown`, na `submit`. Washughulikiaji wa matukio katika React hutumia mkataba wa majina wa camelCase. (Mf. `onClick`, `onSubmit`, n.k.)

Huu hapa ni mfano wa kutumia sifa ya `onClick` kwa kipengele cha `button` katika React:

```jsx
function handleClick() {
  console.log("Button clicked!");
}

<button onClick={handleClick}>Click Me</button>;
```

Katika React, kazi za kushughulikia matukio mara nyingi huanza na kiambishi awali `handle` kuonyesha kwamba zinashughulikia matukio, kama `handleClick` au `handleSubmit`.

Wakati kitendo cha mtumiaji kinachochea tukio, React hupitisha kitu cha Synthetic Event kwenye mshughulikiaji wako. Kitu hiki hutenda kama kitu cha Event asilia katika JavaScript ya kawaida, kikitoa sifa kama `type`, `target`, na `currentTarget`.

Ili kuzuia tabia chaguo-msingi kama vile kuonyesha upya kivinjari wakati wa tukio la `onSubmit`, kwa mfano, unaweza kuita njia ya `preventDefault()`:

```jsx
function handleSubmit(event) {
  event.preventDefault();
  console.log("Form submitted!");
}

<form onSubmit={handleSubmit}>
  <input type="text" />
  <button>Submit</button>
</form>;
```

Unaweza pia kufunga kazi ya mshughulikiaji kwenye kitendakazi cha mshale kama hivi:

```jsx
function handleDelete(id) {
  console.log("Deleting item:", id);
}

<button onClick={() => handleDelete(1)}>Delete Item</button>;
```

## Kufanya kazi na state na `useState` Hook

- **Ufafanuzi wa state**: Katika React, state ni kitu kinachobeba data ya sehemu. State inaposasishwa, sehemu itajionyesha upya. React inachukulia state kama isiyoweza kubadilishwa moja kwa moja, hivyo hupaswi kuibadilisha moja kwa moja.
- **`useState()` Hook**: `useState` ni kitendakazi kinachokuwezesha kutangaza vigezo vya state kwenye sehemu za kitendakazi. Huu hapa ni muundo wa msingi:

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

Katika kigezo cha state una yafuatayo:

- `stateVariable` inashikilia thamani ya sasa ya state
- `setStateFunction` (kazi ya kusasisha) inasasisha kigezo cha state
- `initialValue` inaweka state ya awali

Huu hapa ni mfano kamili kwa sehemu ya `Counter`:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>

      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```

## Uonyeshaji na sehemu za React

- **Ufafanuzi**: Katika React, uonyeshaji ni mchakato ambao sehemu zinaonekana kwenye kiolesura cha mtumiaji (UI), kwa kawaida kwenye kivinjari. Mchakato wa uonyeshaji una hatua tatu: kuchochea, kuonyesha, na kutekeleza.

Hatua ya kuchochea hutokea React inapogundua kwamba kitu kimebadilika na UI inaweza kuhitaji kusasishwa. Mabadiliko haya mara nyingi hutokana na sasisho la state au props.

Mara tu kuchochea kunapotokea, React inaingia kwenye hatua ya uonyeshaji. Hapa, React inatathmini upya sehemu zako na kuamua nini cha kuonyesha. Ili kufanya hivi, React hutumia nakala nyepesi ya DOM halisi inayoitwa virtual DOM. Kwa kutumia virtual DOM, React inaweza kuchunguza haraka ni nini kinahitaji kubadilika kwenye sehemu.

Hatua ya kutekeleza ni pale ambapo React inachukua mabadiliko yaliyotayarishwa kutoka virtual DOM na kuyaweka kwenye DOM halisi. Kwa maneno mengine, hapa ndipo unaona matokeo ya mwisho kwenye skrini.

## Kusasisha vitu na safu ya data kwenye state

- **Kusasisha vitu kwenye state**: Ikiwa unahitaji kusasisha kitu kwenye state, unapaswa kuunda kitu kipya au kunakili kitu kilichopo kwanza, kisha uweke state kwa kitu hicho kipya. Kitu chochote kilichowekwa kwenye state kinapaswa kuchukuliwa kama kisichoweza kubadilishwa moja kwa moja. Huu hapa ni mfano wa kuweka jina la mtumiaji, umri na jiji. Kazi ya `handleChange` inatumika kushughulikia masasisho ya taarifa za mtumiaji:

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **Kusasisha safu ya data kwenye state**: Unaposasisha safu ya data kwenye state, ni muhimu kutoibadilisha moja kwa moja kwa kutumia njia kama `push()` au `pop()`. Badala yake, unapaswa kuunda safu mpya unaposasisha state:

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

Ikiwa unataka kuondoa vitu kwenye safu ya data, tumia njia ya `filter()`, ambayo inarejesha safu mpya baada ya kuchuja kile unachotaka kuondoa:

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```

## Kurejelea thamani kwa kutumia refs

- **Sifa ya `ref`**: Unaweza kupata nodi ya DOM katika React kwa kutumia sifa ya `ref`. Huu hapa ni mfano wa kuonyesha `ref` ili kuweka makini kwenye kipengele cha `input`. Sifa ya `current` inatumika kupata thamani ya sasa ya `ref` hiyo:

```jsx
import { useRef } from "react";

const Focus = () => {
  const inputRef = useRef(null);

  const handleFocus = () => {
    if (inputRef.current) {
      inputRef.current.focus();
    }
  };

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Enter text" />
      <button onClick={handleFocus}>Focus Input</button>
    </div>
  );
};

export default Focus;
```

## Kufanya kazi na `useEffect` Hook

- **`useEffect()` Hook**: Katika React, effect ni chochote kinachotokea nje ya mchakato wa uonyeshaji wa sehemu. Hii ni, chochote ambacho React haitashughulikia moja kwa moja kama sehemu ya uonyeshaji wa UI. Mifano ya kawaida ni pamoja na kuchukua data, kusasisha kichwa cha tab ya kivinjari, kusoma au kuandika kwenye hifadhi ya ndani ya kivinjari, kupata eneo la mtumiaji, na mengine mengi. Operesheni hizi huingiliana na ulimwengu wa nje na zinajulikana kama side effects. React inatoa hook ya `useEffect` ili kukuwezesha kushughulikia side effects hizo. `useEffect` hukuwezesha kuendesha kazi baada ya sehemu kuonyeshwa au kusasishwa.

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [dependencies]);
```

Kazi ya effect inaendeshwa baada ya sehemu kuonyeshwa, wakati hoja ya hiari ya `dependencies` inadhibiti lini effect itaendeshwa.

Kumbuka kwamba `dependencies` inaweza kuwa safu ya "thamani tendaji" (state, props, kazi, vigezo, n.k.), safu tupu, au ikosekane kabisa. Hivi ndivyo chaguzi hizo zinavyodhibiti jinsi `useEffect` inavyofanya kazi:

- Ikiwa `dependencies` ni safu inayojumuisha angalau thamani moja tendaji, effect itaendeshwa kila zinapobadilika.

- Ikiwa `dependencies` ni safu tupu, `useEffect` itaendeshwa mara moja tu sehemu inaponyeshwa mara ya kwanza.

- Ukikosa `dependencies`, effect itaendeshwa kila sehemu inaponyeshwa au kusasishwa.

## Jinsi ya kuunda hooks maalum

- **Hooks maalum**: Hook maalum hukuwezesha kutoa mantiki inayoweza kutumika tena kutoka kwenye sehemu, kama vile kuchukua data, usimamizi wa state, kubadilisha hali, na side effects kama kufuatilia hali ya mtandaoni. Katika React, hooks zote zilizojengwa huanza na neno `use`, hivyo hook yako maalum inapaswa kufuata mkataba huo.

Huu hapa ni mfano wa kuunda hook ya `useDebounce`:

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```

## Kufanya kazi na fomu katika React

- **Ingizo linalodhibitiwa (Controlled Inputs)**: Hii ni pale unapohifadhi thamani ya sehemu ya ingizo kwenye state na kuisasisha kupitia matukio ya `onChange`. Hii inakupa udhibiti kamili wa data ya fomu na hukuwezesha uthibitisho wa papo hapo na uonyeshaji wa masharti.

```jsx
import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **Ingizo lisilodhibitiwa (Uncontrolled Inputs)**: Badala ya kushughulikia ingizo kupitia hook ya `useState`, ingizo lisilodhibitiwa katika HTML huhifadhi state yake ya ndani kwa msaada wa DOM. Kwa kuwa DOM inadhibiti thamani za ingizo, unahitaji kuchukua thamani za sehemu za ingizo kwa kutumia `ref`.

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```

## Kufanya kazi na `useActionState` Hook

- **Vitendo vya seva (Server Actions)**: Hizi ni kazi zinazotekelezwa kwenye seva ili kuruhusu usimamizi wa fomu moja kwa moja kwenye seva bila hitaji la endpoints za API. Huu hapa ni mfano kutoka kwenye programu ya Next.js:

```js
"use server";

async function submitForm(formData) {
 const name = formData.get("name");
 return { message: `Hello, ${name}!` };
}
```

Maelekezo ya `"use server"` yanaweka alama kazi hiyo kama server action.

- **`useActionState` Hook**: Hook hii inasasisha state kulingana na matokeo ya uwasilishaji wa fomu. Huu hapa ni muundo wa msingi wa hook ya `useActionState`:

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state` ni state ya sasa inayorejeshwa na action.
- `action` ni kazi inayochochea server action.
- `isPending` ni boolean inayoonyesha kama action inaendeshwa kwa sasa au la.
- Kigezo cha `actionFunction` ni server action yenyewe.
- `initialState` ni kigezo kinachowakilisha hali ya mwanzo ya state kabla action haijaendeshwa.
- `permalink` ni mfuatano wa herufi wa hiari unaobeba URL ya kipekee ya ukurasa ambao fomu inabadilisha.

## Kuchukua data katika React

- **Chaguzi za kuchukua data**: Kuna njia nyingi tofauti za kuchukua data katika React. Unaweza kutumia Fetch API asilia, au zana za wahusika wengine kama Axios au SWR.
- **Vigezo vya state vinavyotumika sana wakati wa kuchukua data**: Bila kujali njia utakayochagua kuchukua data katika React, kuna baadhi ya vigezo vya state unavyopaswa kufuatilia. Cha kwanza ni data yenyewe. Cha pili ni kufuatilia kama data bado inachukuliwa. Cha tatu ni kigezo cha state kitakachoshika makosa yoyote yanayoweza kutokea wakati wa kuchukua data.

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

Kwa kuwa kuchukua data ni side effect, ni bora kutumia `Fetch API` ndani ya hook ya `useEffect`.

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

Kisha unaweza kuonyesha ujumbe wa kupakia ikiwa kuchukua data hakujakamilika, ujumbe wa kosa ikiwa kulikuwa na kosa, au matokeo.

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

```bash
npm i axios
```

```js
import axios from "axios";
```

Kisha unaweza kuchukua data kwa kutumia `axios.get`:

```js

useEffect(() => {

      setError(err);
    } finally {
      setLoading(false);
    }
```

## Kufanya kazi na `useOptimistic` Hook

- **`useOptimistic` Hook**: Hook hii hutumika kuweka UI iendelee kujibu wakati unasubiri kitendo cha async kukamilika nyuma ya pazia. Inasaidia kusimamia "optimistic updates" kwenye UI, mkakati ambapo unatoa masasisho ya papo hapo kwenye UI kulingana na matokeo yanayotarajiwa ya kitendo, kama vile kusubiri jibu la seva.

Huu hapa ni muundo wa msingi:

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState` ni state ya muda inayosasishwa mara moja kwa uzoefu bora wa mtumiaji.
- `addOptimistic` ni kazi inayotumika kusasisha optimistic state kabla state halisi haijabadilika.
- `actualState` ni thamani halisi ya state inayotokana na matokeo ya kitendo, kama vile kuchukua data kutoka seva.
- `updateFunction` ni kazi inayodhibiti jinsi optimistic state inavyopaswa kusasishwa inapoitiwa.

Huu hapa ni mfano wa kutumia hook ya `useOptimistic` kwenye sehemu ya `TaskList`:

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**: Hii hutumika kuonyesha sehemu ya UI na kuweka sasisho la state kama transition isiyo ya dharura. Hii inaruhusu UI iendelee kujibu wakati wa masasisho mazito. Huu hapa ni muundo wa msingi:

```js
startTransition(action)
```

`action` hufanya sasisho la state au kuchochea mantiki inayohusiana na transition. Hii inahakikisha kwamba masasisho ya UI ya dharura (kama kuandika au kubofya) hayazuiliwi.

## React Router

- **Utangulizi**: React Router ni maktaba ya wahusika wengine inayokuwezesha kuongeza routing kwenye programu zako za React. Kuanza, utahitaji kusakinisha React Router kwenye mradi wa React uliopo kwa njia hii:

```bash
npm i react-router
```

Kisha ndani ya faili ya `main.jsx` au `index.jsx`, utahitaji kuweka muundo wa njia kama hivi:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

`path` na `element` hutumika kuunganisha URL na vipengele vya UI pamoja. Katika mfano huu, tunaweka njia ya ukurasa wa mwanzo inayomwelekeza kwenye sehemu ya `App`.

- **Maoni mengi na usanidi wa njia**: Ni kawaida kwenye programu kubwa kuwa na maoni mengi na usanidi wa njia kama hivi:

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

Prop ya `index` kwenye mifano hii inawakilisha njia chaguo-msingi kwa sehemu ya path. Kwa hiyo sehemu ya `Home` itaonyeshwa kwenye path ya mizizi ya `/` wakati sehemu ya `ProductsHome` itaonyeshwa kwenye path ya `/products`.

- **Kuweka njia ndani ya nyingine (Nesting Routes)**: Unaweza kuweka njia ndani ya njia nyingine, jambo linalosababisha path ya njia mtoto kuongezwa kwenye path ya mzazi.

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

Katika mfano hapo juu, path ya bidhaa zinazotrend itakuwa products/trending.

- **Vipande vya kidinamikia (Dynamic Segments)**: Kipande cha kidinamikia ni pale sehemu yoyote ya path ya URL ni ya kidinamikia.

```jsx
<Route path=":category" element={<Category />} />
```

Katika mfano huu tuna kipande cha kidinamikia kinachoitwa `category`. Mtumiaji anapoenda kwenye URL kama products/brass-instruments, mtazamo utabadilika kuwa sehemu ya `Category` na unaweza kuchukua data inayofaa kulingana na kipande hicho.

- **`useParams` Hook**: Hook hii hutumika kupata vigezo vya kidinamikia kutoka kwenye path ya URL.

```js
import { useParams } from "react-router";

export default function Category() {
  let params = useParams();
   {/* Accessing the category param: params.category */}
   {/* rest of code goes here */}
}
```

## Prop Drilling

- **Ufafanuzi**: Prop drilling ni mchakato wa kupitisha props kutoka sehemu mzazi hadi sehemu za mtoto zilizopangwa ndani sana, hata kama baadhi ya sehemu za mtoto hazihitaji props hizo.

## Usimamizi wa state

- **Context API**: Context inamaanisha pale ambapo sehemu mzazi inafanya taarifa zipatikane kwa sehemu za mtoto bila kuzipitisha moja kwa moja kupitia props. `createContext` hutumika kuunda kitu cha context kinachowakilisha context ambayo sehemu nyingine zitasoma. `Provider` hutumika kusambaza thamani za context kwa sehemu za mtoto.

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**: Redux hushughulikia usimamizi wa state kwa kutoa hifadhi kuu na udhibiti mkali wa masasisho ya state. Inatumia muundo unaotabirika na actions, reducers, na middleware. Actions ni mizigo ya taarifa inayotuma data kutoka kwenye programu yako kwenda kwenye hifadhi ya Redux, mara nyingi huchochewa na mwingiliano wa mtumiaji. Reducers ni kazi zinazobainisha jinsi state inavyopaswa kubadilika kutokana na actions hizo, zikihakikisha state inasasishwa kwa njia isiyoweza kubadilishwa moja kwa moja. Middleware, kwa upande mwingine, hufanya kazi kama daraja kati ya kutuma action na reducer, ikikuruhusu kuongeza utendaji wa Redux (mf. kurekodi matukio, kushughulikia operesheni za async) bila kubadilisha mtiririko wa msingi.

- **Zustand**: Suluhisho hili la usimamizi wa state linafaa kwa programu ndogo hadi za kati. Inafanya kazi kwa kutumia hook ya `useStore` kupata state moja kwa moja kwenye sehemu na kurasa. Hii inakuwezesha kubadilisha na kupata data bila hitaji la actions, reducers, au provider.

## Tofauti kati ya utendaji halisi na unaoonekana

- **Utendaji unaoonekana**: Hii ni jinsi watumiaji wanavyohisi utendaji wa tovuti. Ni jinsi wanavyoipima kwa kujibu na kutegemewa. Hii ni kipimo cha hisia, hivyo ni vigumu kukipima, lakini ni muhimu sana, kwa kuwa uzoefu wa mtumiaji huamua mafanikio au kushindwa kwa tovuti.
- **Utendaji halisi**: Huu ni utendaji wa tovuti unaoweza kupimika na ni wa kweli. Unapimwa kwa kutumia vipimo kama muda wa kupakia ukurasa, muda wa jibu la seva, na muda wa uonyeshaji. Vipimo hivi vinaathiriwa na mambo mengi yanayohusiana na mtandao na msimbo wenyewe.

## Mbinu za kuboresha utendaji unaoonekana

- **Kupakia kwa uvivu (Lazy Loading)**: Mbinu hii inapunguza muda wa awali wa kupakia kwa kupakia rasilimali zisizo muhimu nyuma ya pazia.
- **Punguza ucheleweshaji wa fonti**: Ikiwa tovuti yako ina fonti maalum, unapaswa pia kupunguza ucheleweshaji wa kupakia fonti, kwani hii inaweza kusababisha kurukaruka au kuonyesha fonti mbadala wakati fonti maalum inachukuliwa. Pendekezo ni kutumia fonti mbadala inayofanana na fonti maalum, ili mabadiliko yawe madogo.
- **Tumia viashiria vya kupakia**: Kuonyesha kiashiria cha kupakia kwa mchakato unaochukua muda mrefu mara tu mtumiaji anapobofya kipengele kunaweza kumfanya mtumiaji ajisikie ameunganishwa na mchakato, na kufanya muda wa kusubiri uonekane mfupi.

## Dhana kuu za utendaji

- **Mpangilio wa chanzo (Source order)**: Hii inahusu jinsi vipengele vya HTML vinavyopangwa kwenye hati. Hii huamua ni nini kinapakia kwanza na inaweza kuathiri sana utendaji na Ufikikaji.

Baadhi ya mbinu bora za mpangilio wa chanzo ni pamoja na:

- Kuweka maudhui muhimu kama vichwa vya sehemu, urambazaji au maandishi makuu juu kwenye muundo wa HTML.
- Kuchelewesha scripts zisizo muhimu kama zile za analytics, au widgets za wahusika wengine, ili zisiwazuie uonyeshaji.
- Kutumia uboreshaji wa hatua kwa hatua (progressive enhancement), ili kuhakikisha uzoefu wa msingi unafanya kazi hata kabla ya mitindo na scripts kupakiwa. Progressive enhancement ni njia ya kujenga tovuti na programu kwa msingi wa kufanya ukurasa ufanye kazi na HTML kwanza.

Huu hapa ni mfano wa mpangilio mzuri wa chanzo, ukitumia mbinu bora tulizopitia:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **Njia muhimu ya uonyeshaji (Critical Rendering Path)**: Hii ni mfululizo wa hatua ambazo kivinjari hufuata kubadilisha msimbo kuwa picha kwenye skrini.
- **Ucheleweshaji (Latency)**: Huu ni muda unaochukua ombi kusafiri kati ya kivinjari na seva. Kwa maneno mengine, ucheleweshaji mkubwa ni sawa na kurasa polepole.

Baadhi ya njia za kupunguza ucheleweshaji ni pamoja na:

- Kutumia CDNs, au Content Delivery Networks, kutoa mafaili kutoka maeneo ya karibu.
- Kuwezesha compression kwa kutumia Gzip ili kupunguza ukubwa wa mafaili.
- Kuboresha picha na kutumia lazy loading.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## Kuboresha INP

- **Ufafanuzi**: INP (Interaction to Next Paint) hupima jinsi ukurasa unavyojibu kwa ujumla kwa kupima muda kutoka mtumiaji anapofanya kitendo, kama kubofya au kubonyeza kitufe, hadi wakati kivinjari kinapoboresha uonyeshaji. INP ndogo inaonyesha ukurasa unaojibu haraka zaidi.

Hizi hapa ni njia za kuboresha INP:

- Punguza kazi kwenye thread kuu kwa kugawanya kazi ndefu za JavaScript.
- Tumia `requestIdleCallback()` kwa scripts zisizo muhimu. Hii itaweka kazi kwenye foleni ili iitwe wakati kivinjari kipo kwenye kipindi cha kupumzika.
- Chelewesha au pakia kwa uvivu rasilimali nzito kama ilivyoelezwa awali.
- Boresha washughulikiaji wa matukio. Ikiwa washughulikiaji hawa wanaendeshwa mara nyingi au wanafanya kazi nzito, wanaweza kupunguza kasi ya ukurasa na kuongeza INP. Suluhisho ni kutumia debouncing. Debouncing huhakikisha kwamba kazi inaendeshwa tu baada ya mtumiaji kuacha kuandika kwa kuchelewa kidogo - kwa mfano 300ms. Hii huzuia mahesabu yasiyo ya lazima na kuboresha utendaji.

## Jinsi uonyeshaji unavyofanya kazi kwenye kivinjari

- **Jinsi uonyeshaji unavyofanya kazi**: Kwanza kivinjari huchanganua HTML na kujenga DOM. Kisha, kivinjari hushughulikia CSS, kikijenga CSS Object Model, au CSSOM. Hii ni muundo mwingine wa mti unaoamua jinsi vipengele vinavyopaswa kupambwa. Hatimaye, kivinjari hupaka rangi pikseli kwenye skrini, kikiwasilisha kila kipengele kulingana na mitindo na mpangilio uliokokotolewa. Katika kurasa changamano, hii inaweza kuhusisha tabaka nyingi zinazounganishwa pamoja ili kutoa matokeo ya mwisho ya kuona.

## Jinsi utendaji unavyoathiri uendelevu

- **Maelezo ya msingi**: Mtandao unachangia takriban 2% ya uzalishaji wa kaboni duniani kote—hii ni sawa na sekta ya ndege! Kila baiti inayohamishwa inahitaji umeme, kutoka vituo vya data hadi vifaa vya watumiaji. Mafaili makubwa na scripts zisizofaa humaanisha matumizi zaidi ya nguvu. Tovuti yenye utendaji wa juu siyo tu haraka, bali pia inapunguza usindikaji usio wa lazima na matumizi ya nishati.

## Njia za kupunguza muda wa kupakia ukurasa

- **Boresha rasilimali za media**: Picha na video kubwa ni sababu kuu za muda mrefu wa kupakia. Kwa kuboresha rasilimali hizi, unaweza kuongeza kasi ya tovuti yako kwa kiasi kikubwa. Hii ni pamoja na mambo kama kubana picha, kutumia fomati za kisasa kama WebP na kutumia lazy loading kwa rasilimali.
- **Tumia cache ya kivinjari**: Cache inaruhusu vivinjari kuhifadhi sehemu za tovuti yako ndani ya nchi, hivyo kupunguza muda wa kupakia kwa watembeleaji wanaorudi.
- **Punguza na banisha mafaili**: Kupunguza ukubwa wa mafaili yako kunaweza kuongeza kasi ya upakuaji. Hii ni pamoja na kupunguza ukubwa wa mafaili yanayotumwa na kubanisha mafaili ya CSS na JavaScript.

## Kuboresha "time to usable"

- **Ufafanuzi**: "time to usable" ni muda kutoka mtumiaji anapoomba ukurasa hadi anapoweza kuingiliana nao kikamilifu. Ili kuboresha "time to usable" unaweza kupakia rasilimali zako kwa uvivu au kupunguza rasilimali zinazozuia uonyeshaji.

## Vipimo muhimu vya kupima utendaji

- **First Contentful Paint au FCP**: Hupima jinsi kipande cha kwanza cha maudhui—maandishi au picha—kinavyoonekana haraka kwenye skrini. FCP nzuri ni chini ya sekunde 1.8, na FCP mbaya ni zaidi ya sekunde 3. Unaweza kuangalia FCP yako kwa kutumia Chrome DevTools na kuangalia tab ya performance.
- **Total Blocking Time**: Hii inaonyesha muda ambao thread kuu imezuiwa na kazi nzito za JavaScript. Ikiwa Total Blocking Time (TBT) ni kubwa, watumiaji hupata mwingiliano wa polepole. Ili kuboresha TBT, gawanya kazi ndefu na chelewesha scripts zisizo muhimu.
- **Bounce Rate**: Hii ni asilimia ya watembeleaji wanaoondoka bila kuingiliana. Ikiwa tovuti yako ina bounce rate kubwa huenda ni kwa sababu ukurasa wako ni polepole.
- **Unique Users**: Kipimo hiki hufuatilia ni watumiaji wangapi wa kipekee wanatembelea tovuti yako. Ili kuona Bounce Rate na Unique Users, unaweza kutumia Google Analytics. Itakuruhusu kufuatilia vipimo hivi na kuboresha ushiriki.

## Zana za kawaida za kupima utendaji

- **Chrome DevTools**: Chrome DevTools ni zana iliyojengwa ndani ya Google Chrome inayokuwezesha kuchambua na kurekebisha utendaji papo hapo. DevTools itaonyesha muda wa kupakia, matumizi ya CPU, na ucheleweshaji wa uonyeshaji. Inafaa sana kwa kupima First Contentful Paint, au FCP, ambayo ni jinsi mtumiaji anavyoona maudhui ya kwanza. Ikiwa tovuti yako inaonekana polepole, DevTools itakusaidia kubaini vikwazo.
- **Lighthouse**: Hii ni zana ya kiotomatiki inayokagua utendaji, SEO, na Ufikikaji.
- **WebPageTest**: Zana hii inakuwezesha kupima jinsi tovuti yako inavyopakia kutoka maeneo na vifaa tofauti. Inakupa muhtasari wa kina wa Speed Index ya tovuti yako, Total Blocking Time, na vipimo vingine muhimu vya utendaji. Ikiwa unataka kujua jinsi watumiaji halisi wanavyopata tovuti yako duniani kote, WebPageTest ndiyo zana kwa hilo.
- **PageSpeed Insights**: Zana hii inachambua tovuti yako na kupendekeza maboresho ya haraka kwa simu na kompyuta. Itakuambia nini kinapunguza kasi ya tovuti yako na kutoa mapendekezo maalum kama kuboresha picha, kuondoa scripts zinazozuia uonyeshaji, na kupunguza muda wa jibu la seva. PageSpeed Insights ni njia ya haraka na rahisi ya kuangalia jinsi Google inavyoona utendaji wa tovuti yako.
- **Real User Monitoring**: Zana za RUM hufuatilia tabia halisi ya watumiaji, zikionyesha jinsi watembeleaji halisi wanavyopata tovuti yako. Zana maarufu za RUM ni pamoja na Google Analytics, inayofuatilia muda wa kupakia ukurasa na bounce rates, na New Relic au Datadog, zinazofuatilia masuala ya utendaji kwa wakati halisi. Ikiwa unataka data kutoka kwa watumiaji halisi, zana za RUM ni muhimu.

## Kufanya kazi na Performance Web APIs

- **Ufafanuzi**: Performance Web APIs zinawawezesha wasanidi kufuatilia jinsi ukurasa wa mtandao unavyopakia na kujibu moja kwa moja kwenye msimbo. APIs hizi hukuwezesha kupima muda wa kupakia ukurasa, kufuatilia ucheleweshaji wa uonyeshaji na mwingiliano na kuchambua muda wa utekelezaji wa JavaScript.
- **`performance.now()`**: API hii inakupa timestamps zenye usahihi wa juu (katika milisekunde) kupima muda unaochukuliwa na sehemu tofauti za tovuti yako kupakia.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: API hii inakupa muhtasari wa kila hatua ya kupakia ukurasa, kutoka kwenye DNS lookup hadi `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: API hii inasikiliza matukio ya utendaji kama layout shifts, kazi ndefu, na mwingiliano wa mtumiaji.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

Hii inahitaji repaint, ambayo ni ghali kimahesabu. Kwa hiyo, inashauriwa kupunguza idadi ya uhuishaji wa CSS kadri inavyowezekana au angalau kumpa mtumiaji chaguo la kuwasha au kuzima.

- **Kugawanya msimbo (Code Splitting)**: Kugawanya msimbo wako wa JavaScript kwenye moduli zinazofanya kazi muhimu na zisizo muhimu pia ni muhimu. Kwa njia hii, utaweza kupakia mapema zile muhimu haraka iwezekanavyo na kuchelewesha zisizo muhimu ili kurasa zionyeshwe haraka iwezekanavyo.
- **Kubadilisha DOM**: Kumbuka kwamba kubadilisha DOM kunamaanisha mchakato wa kubadilisha maudhui ya ukurasa kwa njia ya kidinamikia kwa JavaScript kwa kuingiliana na Document Object Model (DOM). Kubadilisha DOM ni ghali kimahesabu. Kupunguza kiasi cha kubadilisha DOM kwenye msimbo wako wa JavaScript kutaboresha utendaji.

## Mifumo ya CSS

- **Mifumo ya CSS**: Mifumo ya CSS inaweza kuongeza kasi ya kazi yako, kuunda mtindo wa kuona unaofanana kwenye tovuti, kufanya muundo wako uonekane sawa kwenye vivinjari vingi, na kuweka msimbo wako wa CSS ukiwa na mpangilio mzuri.
- **Mifumo maarufu ya CSS**: Baadhi ya mifumo maarufu ya CSS ni Tailwind CSS, Bootstrap, Materialize, na Foundation.
- **Hasara zinazoweza kutokea**:

  - Tovuti yako inaweza kuonekana sawa na tovuti nyingine zinazotumia mfumo huo huo.
  - Mifumo mikubwa inaweza kusababisha matatizo ya utendaji.

## Aina mbili za mifumo ya CSS

- **Mifumo ya CSS ya utility-first**: Mifumo hii ina madarasa madogo yenye madhumuni maalum, kama kuweka margin, padding, au rangi ya nyuma. Unaweza kuweka madarasa haya madogo moja kwa moja kwenye vipengele vya HTML kadri inavyohitajika. Tailwind CSS inachukuliwa kama mfumo wa utility-first CSS.

Huu hapa ni mfano wa kutumia Tailwind CSS kupamba kitufe.

```html
<button class="bg-blue-500 text-white font-bold py-2 px-4 rounded-full hover:bg-blue-700">
  Button
```

- **Mifumo ya CSS inayotegemea sehemu**: Mifumo hii ina sehemu zilizojengwa tayari zenye mitindo iliyofafanuliwa tayari unayoweza kuongeza kwenye tovuti yako. Sehemu zinapatikana kwenye nyaraka rasmi za mfumo wa CSS, na unaweza kunakili na kubandika kwenye mradi wako. Bootstrap inachukuliwa kama mfumo wa CSS unaotegemea sehemu.

Huu hapa ni mfano wa kutumia Bootstrap kuunda list group. Badala ya kutumia madarasa madogo kwenye vipengele vya HTML, utaongeza sehemu nzima, ikiwa ni pamoja na muundo wa HTML.

```html
<div class="card" style="width: 25rem;">

    <li class="list-group-item">HTML</li>

    <li class="list-group-item">JavaScript</li>

</div>
```

## Tailwind CSS

Tailwind ni mfumo wa utility-first CSS. Badala ya kuandika sheria maalum za CSS, unajenga miundo yako kwa kuchanganya madarasa madogo ya utility moja kwa moja kwenye HTML yako.

### Utilities za muundo unaobadilika (Responsive Design)

Tailwind hutumia viambishi awali kama `sm:`, `md:`, na `lg:` kutumia mitindo kwenye ukubwa tofauti wa skrini.

Madarasa kama `flex`, `flex-col`, `justify-around`, na `items-center` hufanya iwe rahisi kuunda miundo inayobadilika.

```html
</div>
```

Tailwind inajumuisha utilities za CSS Grid, kama `grid`, `grid-cols-1`, na `md:grid-cols-3`.

```html
<div class="grid grid-cols-1 md:grid-cols-3 gap-8">
```

Utilities kama `mt-8`, `mx-auto`, `p-4`, na `gap-4` husaidia kuunda nafasi thabiti bila kuandika CSS.

Utilities kama `uppercase`, `font-bold`, `font-semibold`, na `text-4xl` hudhibiti mwonekano wa maandishi.

Tailwind hutoa rangi nyingi, kama `text-red-700`, `bg-indigo-600`, na `bg-gray-100`.

Tailwind inasaidia utilities za gradient kama `bg-gradient-to-r from-fuchsia-500 to-indigo-600`.

### Mixins

- **Mixins**: Mixins hukuwezesha kuweka pamoja sifa nyingi za CSS na thamani zake chini ya jina na kutumia tena block hiyo ya msimbo wa CSS kwenye stylesheet yako.

```scss
@mixin center-flex {
```

```scss
section {
```

## Majaribio ya mwongozo na ya kiotomatiki

- **Majaribio ya mwongozo**: Katika majaribio ya mwongozo, mjaribu hupitia kila sehemu ya programu na kujaribu vipengele tofauti ili kuhakikisha inafanya kazi ipasavyo. Ikiwa hitilafu yoyote itagunduliwa kwenye mchakato wa majaribio, mjaribu ataripoti hitilafu hizo kwa timu ya programu ili zirekebishwe.
- **Majaribio ya kiotomatiki**: Katika majaribio ya kiotomatiki, unaweza kujiandikia programu tofauti inayokagua kama programu yako inafanya kazi kama inavyotarajiwa.

## Majaribio ya kitengo (Unit Testing)

- **Unit Testing**: Katika unit testing, unajaribu kila kitendakazi ili kuhakikisha kila kitu kinafanya kazi kama inavyotarajiwa. Unit tests pia zinaweza kutumika kama aina ya nyaraka kwa programu yako kwa sababu zinawakilisha tabia inayotarajiwa ya msimbo wako.
- **Kanuni ya Wajibu Mmoja (Single Responsibility Principle)**: Kanuni hii inapendekeza kila kitendakazi kiwe kidogo na kiwajibike kwa jambo moja tu.
- **Mifumo ya kawaida ya majaribio ya JavaScript**: Baadhi ya mifumo ya kawaida ya majaribio ni Jest, Mocha, na Vitest. Jest ni mfumo maarufu wa majaribio ya kitengo.

Huu hapa ni mfano wa majaribio ya kitengo kwa kutumia Jest.

Kwanza, unaweza kuunda kitendakazi kinachorejesha mfuatano mpya wa herufi:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

Katika faili tofauti ya `getFormattedWord.test.js`, unaweza kuandika majaribio ili kuthibitisha kuwa kitendakazi kinafanya kazi kama inavyotarajiwa. Faili ya `getFormattedWord.test.js` itaonekana hivi:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **Kitendakazi cha `expect`**: Kitendakazi cha `expect` kinatumika kujaribu thamani.
- **Matcher**: Matcher ni kitendakazi kinachokagua kama thamani inatenda kama inavyotarajiwa. Katika mfano hapo juu, matcher ni `toBe()`. Jest ina matchers mbalimbali.

Ili kutumia Jest, kwanza unahitaji kusakinisha kifurushi cha `jest` kwa kutumia `npm i jest`. Pia utahitaji kuongeza script kwenye faili yako ya `package.json` kama hivi:

```js
"scripts": {
  "test": "jest"
},
```

Kisha, unaweza kuendesha amri ya `npm run test` kuendesha majaribio yako.

## Mzunguko wa maisha ya ukuzaji wa programu

- **Hatua tofauti za mzunguko wa maisha ya ukuzaji wa programu**:
  - **Hatua ya kupanga**: Timu ya ukuzaji hukusanya mahitaji ya kazi inayopendekezwa kutoka kwa wadau.
  - **Hatua ya kubuni**: Timu ya programu hugawanya mahitaji na kuamua mbinu bora za suluhisho.
  - **Hatua ya utekelezaji**: Timu ya programu hugawanya mahitaji kuwa kazi ndogo na kuzitekeleza.
  - **Hatua ya majaribio**: Hii inahusisha majaribio ya mwongozo na ya kiotomatiki kwa kazi mpya. Wakati mwingine, timu hujaribu programu katika hatua nzima ya ukuzaji ili kugundua na kurekebisha matatizo yoyote yanayotokea.
  - **Hatua ya usambazaji**: Timu husambaza mabadiliko mapya kwenye mazingira ya kujenga au ya majaribio.
  - **Hatua ya matengenezo**: Hii inahusisha kurekebisha matatizo yoyote yanayotokea kutoka kwa wateja kwenye programu ya uzalishaji.

- **Mifano tofauti ya mzunguko wa maisha ya ukuzaji wa programu**:
  - **Mfano wa Waterfall**: Waterfall ni pale ambapo kila hatua ya mzunguko lazima ikamilike kabla hatua inayofuata haijaanza.
  - **Mfano wa Agile**: Agile inazingatia ukuzaji wa mfululizo kwa kugawanya kazi kwenye sprints.

## BDD na TDD

- **TDD**: Test-driven development ni mbinu inayosisitiza kuandika majaribio kwanza. Kuandika majaribio kabla ya kujenga vipengele hutoa mrejesho wa papo hapo kwa wasanidi wakati wa mchakato wa ukuzaji.
- **BDD**: Behavior-driven development ni mbinu ya kulinganisha mfululizo wa majaribio na malengo ya biashara. Matukio ya majaribio kwenye BDD yanapaswa kuandikwa kwa lugha inayoeleweka na watu wa kiufundi na wasio wa kiufundi. Mfano wa sintaksia hiyo ni Gherkin.
- **Mifumo ya majaribio ya BDD**: Mifano ya mifumo ya majaribio ya BDD ni pamoja na Cucumber, JBehave, na SpecFlow.

## Uthibitisho (Assertions) kwenye majaribio ya kitengo

- **Uthibitisho**: Uthibitisho hutumika kujaribu kama msimbo unafanya kazi kama inavyotarajiwa.
- **Maktaba za uthibitisho**: Chai ni maktaba maarufu ya uthibitisho. Maktaba nyingine za uthibitisho za JavaScript ni `should.js` na `expect.js`.

Huu hapa ni mfano wa uthibitisho kwa kutumia Chai unaokagua kama thamani ya return kutoka kitendakazi cha `addThreeAndFour` ni sawa na nambari 7:

```js
assert.equal(addThreeAndFour(), 7);
```

- **Mbinu bora**: Bila kujali maktaba ya uthibitisho unayotumia, unapaswa kuandika ujumbe wa uthibitisho na wa makosa ulio wazi utakao kusaidia kuelewa ni majaribio gani yanashindwa na kwa nini.

## Mocking, Faking, na Stubbing

- **Mocking**: Mocking ni mchakato wa kubadilisha data halisi na data ya uongo inayofanana na tabia ya vipengele halisi. Kwa mfano, unaweza ku-mock jibu la API kwenye majaribio badala ya kufanya maombi ya API mara kwa mara kuchukua data.
- **Stubbing**: Stubs ni vitu vinavyorejesha majibu yaliyofafanuliwa awali au data ya mfano kwa tabia inayotarajiwa kwenye programu. Kwa mfano, unaweza kufanya stub ya tabia ya muunganisho wa hifadhidata kwenye majaribio yako bila kutegemea muunganisho halisi wa hifadhidata.
- **Faking**: Fakes ni matoleo mepesi ya vipengele halisi bila ugumu au athari zake. Kwa mfano, unaweza fake hifadhidata kwa kuhifadhi data kwenye memory badala ya kuingiliana na hifadhidata halisi. Hii itakuwezesha kuiga operesheni za hifadhidata kwenye memory, ambayo itakuwa haraka zaidi kuliko kushughulika na hifadhidata halisi.

## Majaribio ya utendaji (Functional Testing)

- **Functional Testing**: Functional testing inakagua kama vipengele na kazi za programu vinafanya kazi kama inavyotarajiwa. Lengo la functional testing ni kujaribu mfumo mzima dhidi ya matukio mengi.
- **Non-Functional Testing**: Non-functional testing inazingatia mambo kama utendaji na kutegemewa.
- **Smoke testing**: Smoke testing ni ukaguzi wa awali wa mfumo kwa matatizo ya msingi au muhimu kabla ya kuanza majaribio ya kina zaidi.

## Majaribio ya mwisho hadi mwisho (End-to-End Testing)

- **End-to-End Testing**: End-to-end testing, au E2E, hujaribu matukio halisi kutoka mtazamo wa mtumiaji. Majaribio ya mwisho hadi mwisho husaidia kuhakikisha programu yako inafanya kazi ipasavyo na inatabirika kwa watumiaji. Hata hivyo, inachukua muda mwingi kuanzisha, kubuni na kudumisha.
- **Mifumo ya majaribio ya mwisho hadi mwisho**: Playwright ni mfumo maarufu wa majaribio ya mwisho hadi mwisho uliotengenezwa na Microsoft. Mifano mingine ya zana za majaribio ya mwisho hadi mwisho ni pamoja na Cypress, Selenium, na Puppeteer.

Huu hapa ni mfano wa majaribio ya E2E kutoka kwenye msimbo wa freeCodeCamp kwa kutumia Playwright. Hook ya `beforeEach` itaendeshwa kabla ya kila jaribio. Majaribio yanakagua kama mfadhili ana kiungo cha supporter kwenye menu bar, pamoja na border maalum iliyopambwa kwenye avatar yao:

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## Majaribio ya matumizi (Usability Testing)

- **Usability Testing**: Usability testing ni pale unapowapa watumiaji halisi watumie programu ili kugundua kama kuna matatizo ya muundo, uzoefu wa mtumiaji, au utendaji kwenye programu. Usability testing inazingatia jinsi programu inavyoweza kutumika kwa urahisi na watumiaji.
- **Aina nne za kawaida za usability testing**:
  - **Explorative**: Explorative usability testing inahusisha watumiaji kuingiliana na vipengele tofauti vya programu ili kuelewa jinsi vinavyofanya kazi.
  - **Comparative**: Comparative testing ni pale unapolinganishwa uzoefu wa mtumiaji wa programu yako na programu nyingine zinazofanana sokoni.
  - **Assessment**: Assessment testing ni pale unapochunguza jinsi programu ilivyo rahisi kutumia.
  - **Validation**: Validation testing ni pale unapobaini matatizo makubwa yatakayomzuia mtumiaji kutumia programu ipasavyo.
- **Zana za usability testing**: Mifano ya zana za usability testing ni Loop11, Maze, Userbrain, UserTesting, na UXTweak.

## Majaribio ya ulinganifu (Compatibility Testing)

- **Compatibility Testing**: Lengo la compatibility testing ni kuhakikisha programu yako inafanya kazi kwenye mazingira tofauti ya kompyuta.
- **Aina tofauti za compatibility testing**:
  - **Backwards Compatibility**: Backwards compatibility ni pale programu inafanya kazi na matoleo ya awali.
  - **Forwards Compatibility**: Forwards compatibility ni pale programu na mifumo itafanya kazi na matoleo yajayo.
  - **Hardware Compatibility**: Hardware compatibility ni uwezo wa programu kufanya kazi ipasavyo kwenye usanidi tofauti wa vifaa.
  - **Operating Systems Compatibility**: Operating systems compatibility ni uwezo wa programu kufanya kazi kwenye mifumo tofauti ya uendeshaji, kama macOS, Windows, na Linux kama Ubuntu na Fedora.
  - **Network Compatibility**: Network compatibility inamaanisha programu inaweza kufanya kazi kwenye hali tofauti za mtandao, kama kasi tofauti za mtandao, itifaki, mipangilio ya usalama, n.k.
  - **Browser Compatibility**: Browser compatibility inamaanisha programu ya mtandao inaweza kufanya kazi kwa uthabiti kwenye vivinjari tofauti, kama Google Chrome, Safari, Firefox, n.k.
  - **Mobile Compatibility**: Ni muhimu kuhakikisha programu zako zinafanya kazi kwenye vifaa mbalimbali vya Android na iOS, ikiwa ni pamoja na simu na tableti.

## Majaribio ya utendaji (Performance Testing)

- **Performance Testing**: Katika performance testing, unajaribu kasi, majibu, uwezo wa kupanuka, na uthabiti wa programu chini ya mizigo tofauti ya kazi. Lengo ni kutatua aina yoyote ya vikwazo vya utendaji.
- **Aina tofauti za performance testing**:
  - **Load Testing**: Load testing huamua jinsi mfumo unavyofanya kazi wakati wa mizigo ya kawaida na ya juu.
  - **Stress Testing**: Stress testing ni pale unapojaribu programu yako chini ya mizigo mikubwa na kuona jinsi mfumo wako unavyoshughulikia mzigo huo.
  - **Soak Testing (Endurance Testing)**: Soak testing au endurance testing ni aina ya load testing ambapo unajaribu mfumo na mzigo mkubwa kwa muda mrefu.
  - **Spike Testing**: Spike testing ni pale unapoongeza na kupunguza mizigo kwa ghafla na kuchambua jinsi mfumo unavyojibu mabadiliko hayo.
  - **Breakpoint Testing (Capacity Testing)**: Breakpoint testing au capacity testing ni pale unapoongeza mzigo polepole hadi mfumo uanze kushindwa au kupungua utendaji.

## Majaribio ya usalama (Security Testing)

- **Security Testing**: Security testing husaidia kubaini udhaifu na mapungufu.
- **Kanuni za usalama**:
  - **Usiri (Confidentiality)**: Hii inalinda dhidi ya utoaji wa taarifa nyeti kwa wapokeaji wasiotarajiwa.
  - **Uadilifu (Integrity)**: Hii inahusisha kuzuia watumiaji wabaya kubadilisha taarifa za mtumiaji.
  - **Uthibitisho wa utambulisho (Authentication)**: Hii inahusisha kuthibitisha utambulisho wa mtumiaji ili kuhakikisha anaruhusiwa kutumia mfumo huo.
  - **Utoaji ruhusa (Authorization)**: Hii ni mchakato wa kubaini ni vitendo gani watumiaji waliothibitishwa wanaruhusiwa kufanya au sehemu gani za mfumo wanaruhusiwa kupata.
  - **Upatikanaji (Availability)**: Hii inahakikisha taarifa na huduma zinapatikana kwa watumiaji waliothibitishwa wakati wanapozihitaji.
  - **Kukanusha (Non-Repudiation)**: Hii inahakikisha kwamba mtumaji na mpokeaji wote wana ushahidi wa uwasilishaji na uthibitisho wa utambulisho wa mtumaji. Inalinda dhidi ya mtumaji kukana kuwa ametuma taarifa.
- **Vitisho vya usalama vya kawaida**:
  - **Cross-Site Scripting (XSS)**: Mashambulizi ya XSS hutokea pale mdukuzi anapochomeka scripts mbaya kwenye ukurasa wa mtandao na kuzitekeleza kwenye kivinjari cha mhanga.
  - **SQL Injection**: SQL injection inaruhusu watumiaji wabaya kuchomeka msimbo mbaya kwenye hifadhidata.
  - **Denial-of-Service (DoS) Attack**: Shambulio la DoS ni pale watumiaji wabaya wanapofurika tovuti na maombi mengi au trafiki, na kusababisha seva kupungua kasi au hata kuanguka, hivyo kufanya tovuti isipatikane kwa watumiaji.
- **Makundi ya zana za majaribio ya usalama**:
  - **Static Application Security Testing**: Zana hizi zinakagua msimbo wa chanzo wa programu ili kubaini udhaifu wa usalama.
  - **Dynamic Application Security Testing**: Zana hizi huingiliana na sehemu ya mbele ya programu ili kugundua mapungufu ya usalama. DAST hazina ufikiaji wa msimbo wa chanzo.
- **Penetration Testing (pentest)**: Penetration testing ni aina ya majaribio ya usalama inayohusisha kuunda mashambulizi ya kimtandao yaliyosimuliwa kwenye programu ili kubaini udhaifu wowote kwenye mfumo.

## A/B Testing

- **A/B Testing**: A/B testing inahusisha kulinganisha matoleo mawili ya ukurasa au programu na kuchunguza toleo lipi linafanya vizuri zaidi. Inajulikana pia kama bucket au split testing. A/B testing inakuwezesha kufanya maamuzi yanayoongozwa na data na kuboresha uzoefu wa mtumiaji kila wakati.
- **Zana za A/B Testing**: Mifano ya zana za kutumia kwa A/B testing ni GrowthBook na LaunchDarkly.

## Majaribio ya Alpha na Beta

Baada ya ukuzaji wa awali na majaribio ya programu kukamilika, ni muhimu programu ijazwe na majaribio na watumiaji halisi. Hapa ndipo majaribio ya alpha na beta yanapokuja.

- **Alpha Testing**: Alpha testing hufanywa na kundi maalum la wajaribu wanaopitia programu kuhakikisha hakuna hitilafu kabla haijaachiwa sokoni. Alpha testing ni sehemu ya acceptance testing na hutumia mbinu za white na black box testing.
- **Beta Testing**: Beta testing ni pale programu inapotolewa kwa watumiaji halisi. Watumiaji wanaweza kuingiliana na programu na kutoa mrejesho. Beta testing pia ni aina ya user acceptance testing.
- **Acceptance Testing**: Acceptance testing inahakikisha programu inakidhi mahitaji ya biashara na watumiaji kabla ya kutolewa.
- **Black Box Testing**: Black box testing inazingatia tu tabia inayotarajiwa ya programu.
- **White Box Testing**: White box testing inahusisha mjaribu kujua vipengele vya ndani na kufanya majaribio juu yake.

## Majaribio ya regression

- **Regression**: Regression inahusu hali ambapo mabadiliko mapya yanaharibu kwa bahati mbaya utendaji uliopo.
- **Regression Testing**: Regression testing husaidia kugundua matatizo ya regression. Katika regression testing, unarudia majaribio ya utendaji dhidi ya sehemu za programu yako ili kuhakikisha kila kitu bado kinafanya kazi kama inavyotarajiwa.
- **Zana za Regression Testing**: Zana unazoweza kutumia kufanya regression testing ni Puppeteer, Playwright, Selenium, na Cypress.
- **Mbinu za Regression Testing**:
  - **Unit Regression Testing**: Hii ni pale unapokuwa na orodha ya vitu vinavyopaswa kujaribiwa kila wakati mabadiliko makubwa au marekebisho yanafanywa kwenye programu.
  - **Partial Regression Testing**: Hii inahusisha mbinu za kulenga kuhakikisha mabadiliko mapya hayajaharibu vipengele maalum vya programu.
  - **Complete Regression Testing**: Hii inaendesha majaribio dhidi ya vipengele vyote kwenye msimbo. Hii ndiyo chaguo linalochukua muda mwingi na ni la kina zaidi.
- **Retesting**: Retesting hutumika kukagua matatizo yanayojulikana na kuhakikisha yamesuluhishwa. Tofauti na regression testing, retesting inatafuta matatizo yanayojulikana, wakati regression testing inatafuta matatizo mapya yaliyotokea kutokana na mabadiliko ya hivi karibuni kwenye msimbo.

## TypeScript ni nini

- **JavaScript**: JavaScript ni lugha yenye aina za data zinazobadilika. Hii ina maana kwamba vigezo vinaweza kupokea thamani yoyote wakati wa utekelezaji. Changamoto ya lugha yenye aina zinazobadilika ni kwamba ukosefu wa usalama wa aina unaweza kuleta makosa.

Kwa mfano, hata kama kitendakazi chako cha JavaScript kinatarajia safu ya data, bado unaweza kukiita na nambari:

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

`console` ya mfano hapo juu itakuwa `undefined`.

- **TypeScript**: TypeScript inapanua lugha ya JavaScript kwa kuongeza aina za data tuli, ambazo husaidia kugundua makosa yanayosababishwa na kutofautiana kwa aina kabla ya kuendesha msimbo wako.

Kwa mfano, unaweza kufafanua aina ya kigezo cha `array` kama ifuatavyo:

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

Ufafanuzi huu wa aina unaiambia TypeScript kwamba kigezo cha `array` lazima kiwe safu ya mfuatano wa herufi. Kisha, unapoiita `getRandomValue` na kuipatia nambari, utapata kosa linaloitwa kosa la compiler.

- **Compiler**: Kwanza unahitaji kutafsiri msimbo wa TypeScript kuwa JavaScript ya kawaida. Unapoendesha compiler, TypeScript itatathmini msimbo wako na kutoa kosa kwa matatizo yoyote ambapo aina hazilingani.

## Aina za data katika TypeScript

- **Aina za data msingi katika TypeScript**: Kwa aina za data msingi `string`, `null`, `undefined`, `number`, `boolean`, na `bigint`, TypeScript hutoa maneno muhimu ya aina.

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **Safu ya data**: Unaweza kufafanua safu ya aina maalum kwa sintaksia mbili tofauti.

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **Vitu**: Unaweza kufafanua muundo kamili wa kitu.

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

Ikiwa unataka kitu chenye funguo zozote, lakini thamani zote lazima ziwe mfuatano wa herufi, kuna njia mbili za kufafanua:

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **Aina nyingine muhimu katika TypeScript**:
  - **`any`**: `any` inaonyesha kwamba thamani inaweza kuwa aina yoyote. Inaambia compiler kuacha kujali aina ya kigezo hicho.
  - **`unknown`**: `unknown` inaiambia TypeScript kwamba _unajali_ aina ya thamani, lakini hujui ni ipi. `unknown` inapendekezwa zaidi kuliko `any`.
  - **`void`**: Hii ni aina maalum utakayotumia tu unapofafanua kazi. Kazi zisizo na thamani ya return hutumia aina ya return ya `void`.
  - **`never`**: Inawakilisha aina ambayo haitakuwepo kamwe.

- **Neno muhimu la `type`**: Neno hili ni kama `const`, lakini badala ya kutangaza kigezo, unaweza kutangaza aina.

Inafaa kwa kutangaza aina maalum kama union types au aina zinazojumuisha thamani maalum tu:

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**: Interfaces ni kama madarasa kwa aina. Zinaweza kutekeleza au kupanua interfaces nyingine, ni maalum kwa aina za vitu, na kwa kawaida hupendekezwa isipokuwa unahitaji kipengele maalum kinachotolewa na tamko la `type`.

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **Kufafanua aina ya return**: Unaweza pia kufafanua _aina ya return_ ya kitendakazi.

Mfano hapa chini unafafanua thamani ya return kama mfuatano wa herufi. Ukijaribu kurejesha kitu kingine, TypeScript itatoa kosa la compiler.

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```

## Generics

- **Kufafanua aina generic**: Unaweza kufafanua aina generic na kuitaja kwenye kitendakazi chako. Inaweza kufikiriwa kama kigezo maalum unachotoa kwenye kitendakazi kudhibiti tabia ya ufafanuzi wa aina ya kitendakazi.

Huu hapa ni mfano wa kufafanua aina generic kwa kitendakazi:

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

Sintaksia ya `<T>` inaambia TypeScript kwamba unafafanua aina generic `T` kwa kitendakazi. `T` ni jina la kawaida kwa aina generic, lakini unaweza kutumia jina lolote.

Kisha, unaambia TypeScript kwamba kigezo cha `array` ni safu ya thamani zinazolingana na aina generic, na kwamba thamani inayorejeshwa ni kipengele kimoja cha aina hiyo hiyo.

- **Kufafanua hoja ya aina kwenye mwito wa kitendakazi**: Unaweza kupitisha hoja ya aina kwenye mwito wa kitendakazi kwa kutumia mabano ya pembe kati ya jina la kitendakazi na vigezo vyake.

Huu hapa ni mfano wa kupitisha hoja ya aina kwenye mwito wa kitendakazi:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Hii inaambia TypeScript kwamba kipengele unachotarajia kupata kitakuwa kipengele cha ingizo.

## Type Narrowing

- **Kukaza kwa ukweli (Narrowing by Truthiness)**: Katika mfano hapa chini, utapata kosa la compiler ukijaribu kupata sifa ya `value` ya `email` kwa sababu `email` _inaweza_ kuwa `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Unaweza kutumia tamko la masharti kuthibitisha `email` ni _kweli_ kabla ya kupata sifa:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

Ukaguzi wa ukweli pia unaweza kufanya kazi kwa njia tofauti:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

Kutupa kosa kunamaliza utekelezaji wa mantiki ya msimbo huu, hivyo unapofikia mwito wa `console.log()`, TypeScript inajua `email` _haiwezi_ kuwa `null`.

- **Optional Chaining**: Optional chaining `?.` pia ni aina ya type narrowing, kwa msingi uleule kwamba kupata sifa hakuwezi kutokea ikiwa thamani ya `email` ni `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **Opereta ya `typeof`**: Unaweza kutumia masharti kukagua _aina ya_ kigezo kwa kutumia opereta ya `typeof`.

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **Neno muhimu la `instanceof`**: Ikiwa kitu kinatoka kwenye darasa, unaweza kutumia neno muhimu la `instanceof` kukaza aina.

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **Casting**: Wakati TypeScript haiwezi kubaini moja kwa moja aina ya thamani, kama matokeo kutoka njia ya `request.json()` kwenye mfano hapa chini, utapata kosa la compiler. Njia moja ya kutatua hili ni kwa kufanya casting ya aina, lakini kufanya hivyo kunapunguza uwezo wa TypeScript kugundua makosa yanayoweza kutokea.

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **Type Guard**: Badala ya kufanya casting ya aina, unaweza kuandika type guard:

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

Sintaksia ya `user is User` inaonyesha kwamba kitendakazi chako kinarejesha thamani ya kweli au siyo kweli, ambayo ikiwa ni kweli inamaanisha thamani ya `user` inakidhi interface ya `User`.

## Faili ya `tsconfig`

- **`tsconfig.json`**: Mipangilio ya compiler ya TypeScript ipo kwenye faili ya `tsconfig.json` kwenye mzizi wa mradi wako.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

Hizi hapa ni maelezo ya chaguo za compiler zilizotumika kwenye mfano hapo juu:

- **`compilerOptions`**: Sifa ya `compilerOptions` ndiyo unayodhibiti jinsi compiler ya TypeScript inavyofanya kazi.
- **`rootDir` na `outDir`**: `rootDir` na `outDir` zinaambia TypeScript ni folda gani inabeba mafaili yako ya chanzo, na ni folda gani inapaswa kubeba msimbo wa JavaScript uliotafsiriwa.
- **`lib`**: Sifa ya `lib` inaamua ni ufafanuzi gani wa aina compiler itumie, na inakuwezesha kujumuisha msaada kwa matoleo maalum ya ES, DOM, na zaidi.
- **`module` na `moduleResolution`**: `module` na `moduleResolution` hufanya kazi pamoja kudhibiti jinsi kifurushi chako kinavyotumia moduli - aidha CommonJS au ECMAScript.
- **`esModuleInterop`**: `esModuleInterop` inaruhusu ulinganifu bora kati ya CommonJS na ES modules kwa kuunda moja kwa moja vitu vya namespace kwa imports.
- **`skipLibCheck`**: Chaguo la `skipLibCheck` linaruka kuthibitisha mafaili ya `.d.ts` ambayo hayarejelewi na imports kwenye msimbo wako.
- **`strict`**: Bendera ya `strict` inawasha ukaguzi kadhaa, kama kuhakikisha utunzaji sahihi wa aina zinazoweza kuwa tupu na kutoa onyo TypeScript inaposhindwa na `any`.
- **`exclude`**: Sifa ya `exclude` kwenye ngazi ya juu inaambia compiler kupuuza mafaili haya ya TypeScript wakati wa kutafsiri.
