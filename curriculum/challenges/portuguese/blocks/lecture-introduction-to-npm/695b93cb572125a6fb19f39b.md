# --description--

Versionamento semântico, ou SemVer, é um sistema que dá aos números de versão um significado claro, para que os desenvolvedores possam entender rapidamente que tipo de mudanças foram feitas no software.

SemVer usa este formato:

```bash
MAJOR.MINOR.PATCH
```

Por exemplo, a biblioteca popular `lodash` está atualmente na versão `4.17.21`.

Aqui está o que cada parte significa:

* MAIOR — Grandes mudanças que quebram código existente. Exemplo: remover ou renomear uma função da qual o código existente possa depender.
    
* MENOR — Novos recursos que não quebram o código existente. Exemplo: adicionar uma nova função ou parâmetro opcional.
    
* PATCH — Pequenas correções e melhorias. Exemplo: corrigir um bug ou erro de digitação sem alterar como algo funciona.
    

<img src="https://cdn.freecodecamp.org/curriculum/lecture-transcripts/what-is-semantic-versioning-1.png" alt="In the version number 4.17.21, 4 is the Major Version, it's changed for breaking changes, the 17 is the minor version, it's changed for non-breaking changes, and the 21 is the patch, it's changed for bug fixes and small features">

Vamos supor que você tenha um pacote chamado `my-package`. Aqui está um exemplo de como o número da versão pode mudar ao longo do tempo usando versionamento semântico:

| Version | What Changed |
| --- | --- |
| 1.0.0 | Initial stable release |
| 1.1.0 | New optional feature |
| 1.1.1 | Small bug fix or feature |
| 2.0.0 | Breaking changes introduced |

Mas há mais sobre números de versão no npm. Quando você instala um pacote, a versão no seu arquivo `package.json` pode parecer assim:

```json
"lodash": "^4.17.21" 
```

Você já aprendeu que `4` é a versão major, `17` é a versão minor e `21` é a versão patch. Mas o que significa o caret (`^`) na frente?

O símbolo caret permite que o npm instale automaticamente versões menores e de correção mais recentes (`4.x.x`), mas não versões principais mais recentes (`5.x.x`), porque atualizações principais podem incluir mudanças incompatíveis.

Quando a versão major é `0`, o caret se torna mais restritivo. Por exemplo, `^0.2.3` permitiria atualizações como `0.2.4` mas não `0.3.0`, já que versões major começando com `0` são consideradas menos estáveis.

O acento circunflexo não é o único símbolo que `npm` e outros gerenciadores de pacotes JavaScript usam para gerenciar versões de dependências.

Existe o símbolo til (`~`) que permite atualizações apenas na versão patch. Então, `~1.2.3` aceitaria `1.2.4` e `1.2.9`, mas nunca `1.3.0`.

Você também pode usar os símbolos maior que (`>`), menor que (`<`) e seus equivalentes "ou igual" para aceitar versões dentro de um intervalo. Por exemplo, `>=1.2.3 <2.0.0` significa qualquer versão de `1.2.3` até, mas não incluindo, `2.0.0`.

O símbolo asterisco (`*`), corresponde a qualquer versão, então ele aceita tudo. Isso é ideal apenas para testes e não é recomendado para produção.

Finalmente, você pode travar uma versão exata não prefixando o número da versão com nenhum símbolo. Por exemplo, a versão `1.2.3` aceita apenas `1.2.3`.

Aqui está uma tabela de resumo rápido:

| Symbol | Meaning | Example Range |
| --- | --- | --- |
| `^` | Allow minor + patch updates | `^1.2.3` → `1.x.x` |
| `~` | Allow patch updates only | `~1.2.3` → `1.2.x` |
| `>`, `<`, `>=`, `<=` | Custom range | `>=1.2.3` or `<2.0.0` |
| **\*** | Any version | all versions |
| none | Exact version | `1.2.3` |
