# --description--

Bevor Sie Ihr Repository zu GitHub pushen können, müssen Sie authentifiziert sein. Sowohl SSH- als auch GPG-Schlüssel verwenden ein öffentlich-privates Schlüsselpaar. Das bedeutet, dass Sie einen privaten Schlüssel auf Ihrem lokalen Rechner behalten, mit dem Sie sich authentifizieren. Den öffentlichen Schlüssel teilen Sie mit den Personen oder Diensten, die Ihre Authentifizierung überprüfen müssen. Anders gesagt: Der private Schlüssel ist erforderlich, um eine Aktion auszuführen, und der öffentliche Schlüssel wird verwendet, um diese Aktion zu verifizieren.

GPG- oder GNU Privacy Guard-Schlüssel werden typischerweise verwendet, um Dateien oder Commits zu signieren. Jemand kann dann Ihren öffentlichen GPG-Schlüssel verwenden, um zu überprüfen, dass die Dateisignatur von Ihrem Schlüssel stammt und dass der Inhalt der Datei nicht verändert oder manipuliert wurde. SSH- oder Secure SHell-Schlüssel werden normalerweise verwendet, um eine Remote-Verbindung zu einem Server über das `ssh`-Dienstprogramm zu authentifizieren. Wie Sie in dieser Lektion lernen werden, können Sie jedoch auch einen SSH-Schlüssel verwenden, um Commits zu signieren.

Um einen GPG-Schlüssel zu generieren, führen Sie aus:

```sh
gpg --full-generate-key
```

Für einen SSH-Schlüssel führen Sie aus:

```sh
ssh-keygen -t ed25519 -C "your_email@example.com"
```

Geben Sie unbedingt Ihre tatsächliche E-Mail-Adresse an.

Beide Befehle sollten nach einer Passphrase fragen, um den Schlüssel zu sichern. Diese ist optional, wird aber dringend empfohlen.

Für SSH-Schlüssel erhalten Sie zwei Dateien – eine für Ihren privaten Schlüssel und eine für den öffentlichen. Die Dateinamen der öffentlichen Schlüssel enden mit `.pub`, damit Sie nicht versehentlich Ihren privaten Schlüssel teilen. Diese Dateien befinden sich in `~/.ssh`.

GPG-Schlüssel hingegen befinden sich in einem versteckten und verschlüsselten Verzeichnis. Um stattdessen Ihre öffentlichen Schlüssel aufzulisten, führen Sie aus:

```sh
gpg --list-secret-keys --keyid-format=long
```

Dann, um den öffentlichen Schlüssel zu erhalten, verwenden Sie:

```sh
gpg --armor --export "<key id>"
```

Aber was können Sie mit diesen neuen Schlüsseln tun? Nun, wir haben darüber gesprochen, Ihre Commits zu signieren, und das können Sie mit beiden Schlüsseln tun.

Um Ihre Commits mit Ihrem GPG-Schlüssel zu signieren, müssen Sie Ihren öffentlichen Schlüssel – nicht den privaten – in Ihrem GitHub-Konto hochladen. Dann nehmen Sie die kurze ID, die Sie beim Auflisten der Schlüssel erhalten haben, und führen diesen Befehl aus, um sie als Ihren git-Signaturschlüssel festzulegen:

```sh
git config --global user.signingkey <your_gpg_key_id>
```

Dann können Sie das `-S`-Flag an Ihren `git commit`-Befehl übergeben, um einen bestimmten Commit zu signieren – Sie müssen Ihre Passphrase eingeben. Alternativ können Sie die Autosign-Konfiguration auf `true` setzen, um jeden Commit automatisch zu signieren:

```sh
git config --global commit.gpgsign true
```

Um mit einem SSH-Schlüssel zu signieren, was eine relativ neue Funktion bei GitHub ist, müssen Sie zunächst den Schlüssel in Ihrem GitHub-Konto hochladen. Dann müssen Sie den Signiermodus für git auf SSH setzen:

```sh
git config --global gpg.format ssh
```

Anschließend setzen Sie den Signierschlüssel, indem Sie den Dateipfad anstelle einer ID übergeben:

```sh
git config --global user.signingkey <path_to_your_ssh_keys>
```

Schließlich können Sie die automatische Signierung aktivieren, wie Sie es auch für einen GPG-Schlüssel tun würden.

Mit einer dieser beiden Methoden sind Sie nun bereit, verifizierte Commits zu erstellen, die auf GitHub ein spezielles Abzeichen erhalten.
