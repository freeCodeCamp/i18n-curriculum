## 安全性与隐私

**安全性** 保护数据免受未经授权的访问、盗窃和恶意破坏。它构建技术防御，如防火墙、加密和杀毒软件，以阻止恶意行为者进入你的系统。

**隐私** 控制组织如何收集、存储、使用和共享个人数据。它赋予用户决定谁可以查看其信息的权力。

这两个概念密切相关但又不同：

- 你可以有安全性而没有隐私。一家公司可以将你的数据存储在高度安全的数据库中，但仍在未经你同意的情况下将其出售给广告商。在这种情况下，安全性完好，但隐私被侵犯。
- 没有安全性就不可能有真正的隐私。如果存储私人数据的服务器被攻破，任何隐私承诺都会立即被打破。

安全性提供技术防御机制，而隐私则管理围绕数据使用的法律权利和政策。

## HTTPS

**HTTP** 以明文传输数据，使任何拦截者都能读取。**HTTPS**（超文本传输协议安全）使用 **TLS（传输层安全）** 加密浏览器和服务器之间的通信通道。TLS 取代了较旧的 **SSL（安全套接字层）** 协议，尽管许多网站托管平台今天仍在设置中引用 SSL。

HTTPS 的主要优势：

- **加密：** 将数据混淆成不可读的代码。只有浏览器和服务器持有解密密钥。
- **认证：** 要求网站所有者从受信任机构获取数字证书，证明网站的合法性。
- **数据完整性：** 防止任何人在传输过程中篡改数据，你收到的正是服务器发送的内容。
- **公共 Wi-Fi 保护：** 在开放网络中创建私密通道，隐藏你的流量免受附近攻击者窥视。

现代浏览器会在网站使用纯 HTTP 时警告用户，并要求 HTTPS 才能启用地理定位和推送通知等高级功能。

## 同源策略和 CORS

**同源策略（SOP）** 是浏览器的安全规则，防止网页读取不同源的数据。一个 **源** 由三部分组成，必须完全匹配：

- **协议**（例如 `https`）
- **域名**（例如 `example.com`）
- **端口号**（例如 `443`，HTTPS 默认端口，和 `80`，HTTP 默认端口）

该策略阻止恶意网站在一个浏览器标签页中打开时，窃取另一个标签页中银行网站的数据。

**CORS（跨源资源共享）** 允许服务器通过 HTTP 头部选择性地进行受控的跨源数据共享：

- 浏览器在每个跨源请求中发送 `Origin` 头。
- 如果服务器信任该源，则回复 `Access-Control-Allow-Origin` 头。
- 浏览器随后允许读取响应数据，否则阻止读取。

在 Express 中，`cors` 包处理此功能：

```javascript
// development: allow all origins
app.use(cors());

// production: restrict to trusted domains
app.use(cors({ origin: ['https://yourapp.com', 'https://admin.yourapp.com'], credentials: true }));
```

## Cookies：存储、安全性和隐私

**Cookies** 是网站保存在你设备上的小型文本文件，用于记住访问间的信息。服务器发送 `Set-Cookie` 头；浏览器存储该文件，并在每次后续请求时自动发送回去。

保护 cookies 的安全标记：

- **`HttpOnly`：** 阻止 JavaScript 读取 cookie，防止通过 XSS 窃取会话令牌。
- **`Secure`：** 确保 cookie 仅通过 HTTPS 传输，防止在公共 Wi-Fi 上被截获。
- **`SameSite=Strict`：** 阻止浏览器在任何跨站请求中发送 cookie，防止 CSRF 攻击。
- **`SameSite=Lax`：** 允许在安全的顶层导航中发送 cookie，但阻止跨站表单提交和 API 请求。

**隐私影响：**

- **第一方 cookies** 仅跟踪你在访问的网站上的行为。
- **第三方 cookies** 由广告网络放置，跨不同网站跟踪你，构建详细的兴趣档案。

像 GDPR 这样的隐私法律要求网站显示 cookie 同意横幅，并赋予用户拒绝跟踪 cookie 的权利。现代浏览器默认阻止第三方跟踪 cookie。

## 发生安全问题的情况

安全漏洞很少偶然发生。常见触发情况包括：

- **赶工发布：** 开发团队跳过彻底的代码审查和安全测试以赶时间，留下未修补的错误和缺失的访问控件。
- **复杂系统集成：** 每个外部库、第三方 API 或微服务都增加供应链风险。一个依赖的漏洞会传播到主应用。
- **配置管理不善：** 使用默认出厂设置部署服务器或云数据库，常留下空白管理员密码、开放端口和公共数据访问。自动扫描器会寻找这些错误配置的系统。
- **人为错误：** 员工被钓鱼邮件、欺诈短信或欺骗电话针对，可能无意中泄露凭证或向攻击者传输数据。
- **组织合并：** 快速连接两个不同的企业网络会产生盲点，如旧服务器失去可见性、防火墙配置错误和过度访问权限。

## 常见安全威胁及缓解措施

| 威胁 | 工作原理 | 缓解措施 |
|---|---|---|
| **钓鱼** | 伪造邮件模仿可信品牌窃取凭证 | 安全意识培训；邮件认证协议（SPF、DKIM、DMARC） |
| **恶意软件（勒索软件、间谍软件）** | 加密或窃取受害者文件；勒索赎金 | 自动打补丁；反恶意软件 |
| **凭证填充** | 自动工具测试数百万被盗用户名/密码组合 | 强制多因素认证（MFA） |
| **DDoS** | 用假流量淹没服务器使其崩溃 | 基于云的流量过滤服务（如 Cloudflare） |
| **内部威胁** | 不满或粗心员工泄露或窃取数据 | 最小权限原则；监控系统日志异常访问 |

## 内容安全策略（CSP）和权限策略

**内容安全策略（CSP）** 是一个 HTTP 响应头，限制浏览器在页面上加载哪些外部资源。它主要通过为脚本、样式和图像白名单可信源域来防御 **跨站脚本（XSS）** 攻击。

如果来自未批准域的恶意脚本尝试运行，浏览器会阻止它并记录安全错误。

**权限策略**（前称功能策略）控制对浏览器功能和设备硬件（如摄像头、麦克风和地理位置）的访问。你可以限制功能仅限自己域名或完全禁用：

```http
Permissions-Policy: camera=(self), geolocation=()
```

`camera=(self)`：只有你自己的域可以访问摄像头。
`geolocation=()`：页面上所有人都禁用地理位置。

CSP 锁定数据传输并防止代码注入；权限策略保护设备硬件并执行隐私合规。

## 用户隐私的基本概念

| 概念 | 定义 |
|---|---|
| **PII（个人身份信息）** | 任何能直接或间接识别个人的数据。直接标识符：姓名、地址、身份证号。间接标识符：IP 地址、设备序列号、位置历史。 |
| **机密性** | 通过加密和访问控件确保私人数据不被未授权方访问。 |
| **数据最小化** | 仅收集完成特定任务所需的最少 PII。 |
| **目的限制** | 为一个目的收集的数据不得非法用于其他目的（如将联系信息出售给广告商）。 |
| **跟踪** | 使用脚本、设备指纹和浏览器历史日志持续监控在线活动以构建用户档案。 |
| **知情同意** | 用户必须主动选择加入数据收集。预先勾选的框不构成有效同意。 |
| **被遗忘权** | 用户有权合法要求组织永久删除其所有个人数据和跟踪历史。 |

## 区域隐私法律

| 法律 | 司法管辖区 | 主要要求 |
|---|---|---|
| **GDPR** | 欧盟 | 严格的选择加入同意；用户有权访问和删除数据；违规罚款严厉。 |
| **CCPA / CPRA** | 美国加州 | 选择退出模式；企业必须显示“请勿出售我的个人信息”链接。 |
| **COPPA** | 美国（联邦） | 收集 13 岁以下儿童数据前需可验证的家长同意。 |
| **DPA / UK GDPR** | 英国 | 合法、公平和透明的数据处理；个人有访问和删除权。 |
| **NDPA** | 尼日利亚 | 规范尼日利亚居民个人数据的收集、存储和使用。 |
| **DPDP** | 印度 | 规范印度居民个人数据的使用。 |

**实用合规步骤：**

1. **基于位置的同意横幅：** 检测用户地理位置，显示符合当地法规的 cookie 横幅。
2. **自动化 DSAR 系统：** 处理数据主体访问请求，安全导出或删除用户的全部数据历史。
3. **数据最小化和年龄门控：** 仅收集必要数据，使用年龄验证保护未成年人。
4. **数据保护官（DPO）：** 指定专家监督隐私架构并管理法规合规。

## `.env` 文件和 `dotenv` 库

**`.env` 文件** 是放在项目根目录的纯文本文件，存储敏感配置数据的键值对，包括 API 密钥、数据库密码和端口号。它将这些信息从源代码和公共仓库中隔离开。

```bash
PORT=3000
DB_HOST=localhost
API_KEY=xyz123secretkey
```

务必将 `.env` 添加到 `.gitignore`，确保它永远不会被提交到版本控制。

**`dotenv` 包** 在启动时读取 `.env` 文件，并将密钥注入到 `process.env` 中：

```javascript
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

两个主要好处：

- **安全性：** 敏感凭证不会出现在源代码或版本历史中。
- **灵活性：** 只需在每台服务器上替换 `.env` 文件，同一代码库即可在开发、测试和生产环境运行。
