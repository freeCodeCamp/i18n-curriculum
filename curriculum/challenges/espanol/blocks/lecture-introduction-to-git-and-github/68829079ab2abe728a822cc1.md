# --description--

En una lección anterior, aprendiste cómo crear un nuevo repositorio en GitHub. Pero, ¿qué pasa si tienes un proyecto existente localmente que quieres agregar a GitHub? En esta lección, aprenderás cómo hacer push de repositorios locales existentes a GitHub y cómo hacer pull de cambios desde repositorios remotos.

Comienza creando un nuevo repositorio en GitHub sin ningún archivo. Cuando lo hagas correctamente, deberías ser redirigido a una página que incluye la URL necesaria para clonar tu repositorio.

Ahora navega a tu terminal y selecciona un proyecto existente que desees agregar a GitHub. En esta lección, estoy eligiendo un proyecto local llamado `super-awesome-game` con los siguientes archivos dentro:

```sh
jessicawilkins super-awesome-game >> ls
README.md  index.html script.js  styles.css
```

Para configurar el control de versiones de este proyecto usando Git, necesitarás ejecutar el comando `git init` en el directorio del proyecto. Esto inicializará un repositorio Git vacío para que Git pueda comenzar a rastrear los cambios de este proyecto. Cuando lo hagas correctamente, deberías ver una salida similar en la terminal:

```sh
jessicawilkins super-awesome-game >> git init
Initialized empty Git repository in /Users/jessicawilkins/workspace/freeCodeCamp/super-awesome-game/.git/
```

Cuando inicializas un repositorio Git vacío en un proyecto, se añadirá un nuevo directorio oculto `.git`. Para ver la adición del directorio puedes ejecutar el comando `ls -a`, que lista todos los archivos y directorios, incluidos los ocultos:

```sh
jessicawilkins super-awesome-game >> ls -a
.          ..         .git       README.md  index.html script.js  styles.css
```

Es importante no eliminar ese directorio `.git`, de lo contrario se perderá todo tu historial de Git.

Ahora que el repositorio Git vacío ha sido inicializado, puedes ejecutar el comando `git status`. Este comando se usa para mostrar el estado actual de tu directorio de trabajo; usarás este comando mucho en tu flujo de trabajo. Deberías ver una salida similar a esta:

```sh
jessicawilkins super-awesome-game >> git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	README.md
	index.html
	script.js
	styles.css

nothing added to commit but untracked files present (use "git add" to track)
```

`On branch main` significa que estás en la rama `main`. Una rama en Git es un espacio de trabajo separado donde puedes hacer cambios. La rama `main` a menudo representa la rama principal o de producción en una aplicación del mundo real. Los equipos de desarrollo crearán múltiples ramas para nuevas funciones y correcciones de errores, y luego fusionarán esos cambios de nuevo en la rama `main`. Aprenderás más sobre ese proceso en una lección futura.

`Untracked files` significa que Git ha detectado archivos que actualmente no está rastreando.

Hay cinco estados en los que un archivo puede estar:

1. "Untracked": Esto significa que el archivo es nuevo en el repositorio y que Git no lo ha "visto" antes.
2. "Modified": Este archivo existía en el commit anterior y tiene cambios que no han sido confirmados.
3. "Ignored": Probablemente no verás archivos ignorados en Git, pero tu IDE podría tener un indicador para ellos. Los archivos ignorados están excluidos de las operaciones de Git, normalmente porque están incluidos en el archivo `.gitignore`.
4. "Deleted": Un archivo eliminado es lo opuesto a un archivo no rastreado: es un archivo que existía previamente y ha sido eliminado.
5. "Renombrado": Un archivo renombrado es un archivo cuyo contenido no ha cambiado, pero el nombre o la ubicación del archivo fue modificada. En algunos casos, un archivo puede considerarse renombrado incluso si tiene una pequeña cantidad de cambios.

Cuando tienes cambios no confirmados en tu repo, primero necesitarás "stage"arlos. Un "staging area" es un archivo especial en tu directorio `.git` que contiene información que irá en tu próximo commit. Un commit es efectivamente una instantánea del estado actual de tu repositorio.

Si quieres preparar un archivo a la vez, entonces puedes usar el siguiente comando:

```sh
git add name-of-file
```

Si quieres preparar todos los cambios no preparados, entonces puedes usar `git add .` El punto (`.`) es un alias para el directorio actual en el que estás. Algunos desarrolladores advierten en contra de usar `git add .` porque podrías terminar preparando archivos que no querías. Si eso sucede, puedes "desprepararlos" usando el siguiente comando:

```sh
git reset name-of-file
```

En general, mientras estés monitoreando qué archivos se están agregando usando `git status` antes de prepararlos, entonces deberías estar bien.

Una vez que prepares tus archivos, necesitarás confirmarlos. Puedes confirmar tus cambios ejecutando el comando `git commit`. Esto abrirá el editor que hayas configurado como preferido en la configuración de Git. Una vez que el editor esté abierto, puedes proporcionar un mensaje detallado de tus cambios. También puedes optar por proporcionar un mensaje más corto usando el comando `git commit -m` así:

```sh
git commit -m "short message goes here"
```

Si vas a usar el flag `-m`, aún querrás proporcionar un mensaje descriptivo para dar contexto a los cambios. Aquí tienes algunos ejemplos de buenos mensajes que podrías usar:

```sh
git commit -m "feat(api): implement JWT-based authentication"
git commit -m "fix(auth): correct token expiration time"
git commit -m "refactor: simplify login flow logic"
```

El `feat(api)` y `fix(auth)` siguen el estilo Conventional Commits que se usa para proporcionar contexto sobre los tipos de cambios en ese commit.

Después de que hayas confirmado tu trabajo, y ejecutes `git status` de nuevo, deberías ver la siguiente salida en el terminal:

```sh
On branch main
nothing to commit, working tree clean
```

Para ver todos tus commits anteriores de este proyecto puedes ejecutar el comando `git log`. Esto listará todos los commits previos con información útil como el autor, la fecha del commit, el mensaje del commit y el hash del commit. El hash del commit es una cadena larga que sirve como identificador único para un commit.

Una vez que hayas terminado de hacer commit a tu trabajo, ahora puedes hacer push de tus cambios a GitHub. Primero necesitarás configurar la conexión remota a tu repositorio remoto. Aprendiste sobre esto en la lección anterior, pero aquí tienes un recordatorio usando SSH:

```sh
git remote add origin git@github.com:your-github-username/name-of-repo.git
```

El último paso es subir tus cambios a GitHub usando este comando:

```sh
git push -u origin main
```

La bandera `-u` es una abreviatura de `--set-upstream`. Conecta tu rama local `main` con la rama remota `main` en `origin`. Solo necesitas usarla una vez por cada rama nueva. Después de eso, puedes simplemente ejecutar `git push` sin especificar el nombre de la rama.

Cuando se hace correctamente, deberías ver un resultado similar en el terminal:

```sh
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Delta compression using up to 8 threads
Compressing objects: 100% (2/2), done.
Writing objects: 100% (3/3), 253 bytes | 253.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0 (from 0)
To https://github.com/jdwilkin4/super-awesome-game.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

También deberías ver los nuevos cambios en la página de tu repositorio de GitHub cuando refresques la página.

Si estás colaborando con otros en un proyecto y necesitas descargar nuevos cambios del repositorio, puedes usar el comando `git pull`. Aprenderás más sobre trabajar en equipo y colaboración en una lección futura.
