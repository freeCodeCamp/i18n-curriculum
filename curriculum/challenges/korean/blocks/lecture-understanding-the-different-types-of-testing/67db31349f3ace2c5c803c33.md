# --description--

소프트웨어 개발에서 알아야 할 두 가지 테스트 접근법이 있습니다. 이 접근법들은 TDD(테스트 주도 개발)와 BDD(행동 주도 개발)입니다.

먼저 TDD를 살펴보겠습니다. 테스트 주도 개발은 애플리케이션의 기능을 만들기 전에 테스트를 먼저 작성하는 방법론입니다.

TDD의 일반적인 과정은 다음과 같습니다:

1. 테스트를 작성하세요

2. 테스트가 실패하는지 확인하세요

3. 해당 테스트를 위한 코드를 작성하세요

4. 이번에는 테스트가 통과하는지 확인하세요

5. 코드에 필요한 리팩터링을 하세요

1~5단계를 반복하세요.

기능을 만들기 전에 테스트를 작성하면 개발 과정 중에 개발자에게 실시간 피드백을 제공합니다. 또한 개발 초기 단계에서 버그를 더 빨리 잡는 데 도움이 됩니다.

일부 소프트웨어 팀은 TDD 방식을 선택하지만, TDD는 업계에서 논란이 많은 주제입니다. TDD를 옹호하거나 비판하는 수많은 글과 영상이 있습니다. 이 접근법에 대해 어떻게 생각하든, 실제 상황에서 마주칠 수 있으니 알고 있는 것이 중요합니다.

두 번째로 살펴볼 테스트 접근법은 행동 주도 개발(BDD)입니다. 행동 주도 개발은 일련의 테스트를 비즈니스 목표와 일치시키는 방법입니다. 이 테스트들은 기술자와 비기술자 모두가 이해할 수 있는 언어로 작성되어야 합니다.

소프트웨어 애플리케이션에서 특정 행동을 테스트할 때, BDD는 일반적으로 비즈니스에서 행동과 시나리오를 설명하는 데 사용하는 Gherkin 언어를 사용합니다. Gherkin 언어는 시나리오를 설명할 때 Given, When, Then 문장을 사용합니다.

예를 들어:

**시나리오:** 새 블로그 게시물 작성하기

**Given** 사용자가 자신의 계정에 로그인되어 있고 "새 게시물 작성" 페이지에 있다면,

**When** 사용자가 게시물의 제목과 내용을 입력하고 "게시" 버튼을 클릭하면,

**Then** 블로그 게시물이 올바른 제목, 내용, 타임스탬프와 함께 홈페이지에 저장되고 표시되어야 합니다.

먼저 시나리오를 정의하세요. 예를 들어 사용자가 애플리케이션에 로그인하거나 웹사이트에서 주문하는 경우가 시나리오가 될 수 있습니다. 이 예에서는 사용자가 사이트에서 새 블로그 게시물을 작성하는 시나리오입니다.

시나리오가 정의되면 Given 문장부터 시작하세요. 이는 사용자가 행동을 수행하기 전에 어디에 있는지에 대한 맥락을 제공합니다. When 문장은 사용자가 취하는 행동을 나타냅니다.

마지막으로 Then 문장은 사용자가 이전에 취한 행동으로부터 기대하는 결과를 나타냅니다. 모든 시나리오를 정의한 후 각 시나리오에 대한 테스트를 작성할 수 있습니다.

BDD 테스트 프레임워크의 예로는 Cucumber, JBehave, SpecFlow가 있습니다.
