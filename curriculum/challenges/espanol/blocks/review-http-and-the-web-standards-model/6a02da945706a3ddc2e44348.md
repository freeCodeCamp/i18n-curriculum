## Servidores y el modelo cliente-servidor

- **Servidor**: Una computadora o programa que recibe solicitudes de otras computadoras y envía datos o servicios de vuelta a través de la red. El término se refiere tanto al hardware físico como al software que se ejecuta en él.
- **Modelo cliente-servidor**: Una arquitectura distribuida donde los clientes envían solicitudes y los servidores las manejan y devuelven respuestas.
- **Cliente**: El proceso o aplicación (por ejemplo, un navegador web) que inicia solicitudes a un servidor.
- **Ciclo solicitud-respuesta**: El cliente envía una solicitud; el servidor la procesa y devuelve una respuesta.

### Tipos de servidores

- **Servidores web**: Entregan contenido web (HTML, imágenes, CSS) a los navegadores.
- **Servidores de aplicaciones**: Manejan la lógica de negocio y procesan la entrada del usuario.
- **Servidores de base de datos**: Almacenan, recuperan y gestionan datos.
- **Servidores de correo**: Envían, reciben y almacenan correos electrónicos.
- **Servidores de archivos**: Gestionan el almacenamiento y compartición de archivos en una red.
- **Servidores proxy**: Actúan como intermediarios entre clientes y otros servidores.

## DNS (Sistema de nombres de dominio)

- **DNS**: Un sistema de nombres jerárquico y descentralizado que traduce nombres de dominio legibles por humanos en direcciones IP, a menudo llamado la "guía telefónica" de internet.
- **Dirección IP**: Un identificador numérico único para un dispositivo en una red (por ejemplo, `127.0.0.1` en IPv4, `0:0:0:0:0:0:0:1` en IPv6).
- **Jerarquía de nombres de dominio** (leer de derecha a izquierda):
  - **TLD (dominio de nivel superior)**: El segmento más a la derecha (por ejemplo, `.org`, `.com`).
  - **SLD (dominio de segundo nivel)**: El nombre registrado (por ejemplo, `freecodecamp` en `freecodecamp.org`).
  - **Subdominio**: Identifica una sección del dominio principal (por ejemplo, `www`, `blog`, `api`).

### Proceso de resolución DNS

1. El navegador envía el nombre de dominio al **resolutor recursivo** (usualmente gestionado por el ISP).
2. El resolutor recursivo consulta a los **servidores raíz**, que lo dirigen al servidor TLD correspondiente.
3. El **servidor TLD** devuelve la dirección del **servidor de nombres autoritativo** para el dominio.
4. El servidor de nombres autoritativo devuelve el **registro DNS** que contiene la dirección IP.
5. El resolutor recursivo entrega la dirección IP al cliente y **almacena en caché** el registro para acelerar futuras consultas.

## TCP/IP

- **Protocolo**: Un conjunto de reglas y estándares que permiten la comunicación entre dispositivos en una red.
- **TCP/IP**: Un conjunto de protocolos que determina cómo se empaquetan, envían, enrutan y reciben los datos.
  - **IP (Protocolo de Internet)**: Enruta paquetes de datos desde la fuente hasta el destino usando direcciones IP. Es sin conexión — cada paquete viaja de forma independiente sin garantía de entrega.
  - **TCP (Protocolo de control de transmisión)**: Divide los datos en segmentos numerados, los reensambla en el destino, verifica errores y paquetes perdidos, y solicita retransmisión si es necesario. Garantiza entrega confiable y ordenada.

## HTTP (Protocolo de transferencia de hipertexto)

- **HTTP**: Un protocolo a nivel de aplicación que define las reglas para la comunicación entre clientes y servidores en la web.

- **HTTPS**: Una versión segura de HTTP donde los datos se encriptan usando SSL/TLS antes de la transmisión.

### Estructura de una solicitud HTTP

- **Línea de solicitud**: `METHOD /path HTTP/version` (por ejemplo, `GET / HTTP/1.1`)
- **Encabezados**: Pares clave-valor de metadatos (por ejemplo, `Content-Type: application/json`).
- **Cuerpo** (opcional): Datos enviados al servidor, usados con `POST`, `PUT` y `PATCH`.

**Métodos HTTP comunes:**

| Método | Propósito |
| --- | --- |
| `GET` | Recuperar datos |
| `POST` | Enviar nuevos datos |
| `PUT` | Reemplazar un recurso |
| `PATCH` | Actualizar parcialmente un recurso |
| `DELETE` | Eliminar un recurso |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### Estructura de una respuesta HTTP

- **Línea de estado**: Versión HTTP, código de estado y frase de razón (por ejemplo, `HTTP/1.1 200 OK`).
- **Encabezados**: Metadatos sobre la respuesta (por ejemplo, `Content-Type: text/html`).
- **Cuerpo** (opcional): El contenido devuelto o explicación del error.

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## Códigos de estado HTTP

Los códigos de estado se agrupan según su primer dígito:

- **1xx** — Respuestas informativas.
- **2xx** — Respuestas exitosas.
- **3xx** — Mensajes de redirección.
- **4xx** — Errores del cliente.
- **5xx** — Errores del servidor.

### Códigos comunes

| Código | Nombre | Significado |
| --- | --- | --- |
| `200` | OK | Solicitud exitosa; contenido devuelto. |
| `201` | Created | Recurso creado con éxito. |
| `204` | No Content | Éxito pero sin contenido para devolver. |
| `301` | Moved Permanently | Recurso movido permanentemente a una nueva URL. |
| `302` | Found | Recurso movido temporalmente a otra URL. |
| `304` | Not Modified | Recurso sin cambios desde la última solicitud; usar versión en caché. |
| `400` | Bad Request | El servidor no pudo entender la solicitud (sintaxis o datos inválidos). |
| `401` | Unauthorized | Se requieren credenciales de autenticación. |
| `403` | Forbidden | El cliente no tiene permiso para el recurso, incluso con credenciales válidas. |
| `404` | Not Found | El recurso no existe en el servidor. |
| `500` | Internal Server Error | Error inesperado en el servidor. |
| `502` | Bad Gateway | El servidor recibió una respuesta inválida de un servidor ascendente. |
| `503` | Service Unavailable | El servidor no puede manejar solicitudes temporalmente. |

## El modelo de solicitud-respuesta HTTP

El ciclo de solicitud-respuesta consta de estos componentes trabajando juntos:

- **Cliente**: Navegador, app móvil o cliente API que envía solicitudes HTTP.
- **Método HTTP**: La acción a realizar (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`).
- **URL**: La dirección del recurso solicitado.
- **Encabezados**: Metadatos para la solicitud o respuesta (por ejemplo, `Authorization`, `Content-Type`).
- **Cuerpo**: Datos enviados con solicitudes `POST`, `PUT`, `PATCH` o `DELETE` (a menudo JSON o datos de formulario).
- **Servidor**: El sistema backend que procesa solicitudes y devuelve respuestas.
- **Código de estado**: Código numérico que indica éxito, error o redirección.
- **Cuerpo de respuesta**: Los datos reales devueltos, típicamente HTML, JSON o XML.

### Los seis pasos del ciclo

1. **El cliente inicia una solicitud** con un método, URL, encabezados y cuerpo opcional.
2. **La solicitud viaja al servidor** a través de internet.
3. **El servidor procesa la solicitud**: verifica método, URL, autenticación y consulta una base de datos si es necesario.
4. **El servidor prepara una respuesta** con código de estado, encabezados y cuerpo opcional.
5. **La respuesta se envía de vuelta** al cliente.
6. **El cliente maneja la respuesta**: renderiza datos en caso de éxito (2xx) o muestra un mensaje de error en caso de fallo (4xx/5xx).

## Envío de formularios HTML

- Los formularios recopilan y envían datos de usuario usando el elemento `<form>` con elementos `<input>`.
- Cada entrada necesita un atributo `name` para que el servidor pueda identificar los datos.
- **Validación del lado cliente**: Usa los atributos `required` y `pattern` para retroalimentación inmediata.
- **Validación del lado servidor**: Siempre valida en el servidor por seguridad, sin importar las comprobaciones del cliente.
- El atributo `action` especifica dónde enviar los datos del formulario.
- El atributo `method` especifica cómo enviarlos (`GET` o `POST`; `PUT`, `PATCH`, `DELETE` requieren soluciones alternativas como la API Fetch o `method-override` en Express).

### Flujo de envío de formulario

1. El usuario completa y envía el formulario.
2. El navegador ejecuta la validación del lado cliente; continúa si es válido.
3. El navegador envía los datos del formulario a la URL en `action` usando el método `method` especificado.
4. El servidor recibe y procesa los datos (almacena en base de datos, autentica usuario, etc.).
5. El servidor puede ejecutar validación adicional del lado servidor.
6. El servidor responde con un mensaje de éxito, error o redirección.
7. El usuario recibe retroalimentación basada en la respuesta del servidor.

## Modelo de estándares web y organismos de estándares

- **Estándares web**: Las reglas y especificaciones que los navegadores siguen para renderizar páginas y habilitar funcionalidades web.
- **WHATWG**: El organismo que mantiene HTML y el DOM como un "estándar vivo" continuamente actualizado impulsado por los principales equipos de motores de navegador.
- **W3C**: La organización principal de estándares, supervisando CSS, SVG, pautas de accesibilidad (WCAG) y numerosas APIs web.
- **ECMA / TC39**: ECMA publica la especificación ECMAScript (el estándar de JavaScript); TC39 avanza las funcionalidades del lenguaje mediante un proceso de propuestas de cinco etapas (etapas 0–4).
- **Khronos Group**: Responsable de WebGL y WebGL 2, que llevan renderizado 3D acelerado por GPU al navegador.

## Proceso para crear estándares web

1. **Identificar una necesidad**: Un desarrollador o ingeniero de navegador documenta un problema real y ampliamente compartido.
2. **Propuesta y discusión inicial**: La idea se redacta informalmente para recopilar opiniones y medir el interés de los proveedores de navegador.
3. **Redacción de la especificación**: Se elabora un documento preciso y formal que cubre todos los casos límite, condiciones de error e implicaciones de seguridad.
4. **Revisión, objeciones e iteración**: El borrador se revisa públicamente; los proveedores plantean objeciones y la especificación se revisa hasta alcanzar consenso.
5. **Implementación y pruebas**: Los proveedores implementan la funcionalidad detrás de una bandera; un conjunto de pruebas compartidas verifica que todas las implementaciones se comporten idénticamente.
6. **Finalización**: Una vez que múltiples implementaciones independientes pasan las pruebas, el estándar se finaliza (Recomendación W3C, etapa 4 de TC39 o actualización del estándar vivo de WHATWG).

## Ciclo de vida de las funcionalidades de estándares web

1. **Propuesta**: Alguien identifica un problema y lo documenta para saber si otros lo comparten.
2. **Incubación**: La idea se prueba mediante discusión abierta; muchas propuestas mueren aquí si son demasiado específicas o ninguna solución satisface a todos.
3. **Especificación**: Se escribe una especificación formal que convierte conceptos vagos en reglas exactas e implementables.
4. **Pruebas de interoperabilidad**: El conjunto de pruebas W3C Web Platform Tests verifica que cada navegador implemente la funcionalidad de forma idéntica.
5. **Finalización**: Varias implementaciones exitosas significan que la funcionalidad se lanza a los usuarios como parte de la plataforma web base.
6. **Deprecación**: Las funcionalidades obsoletas o problemáticas se marcan para descontinuación; su eliminación de los navegadores es lenta debido a obligaciones de compatibilidad hacia atrás.

## Principios clave de los estándares web

- **Apertura**: Los estándares se desarrollan de forma abierta y se implementan sin regalías por cualquiera.
- **Accesibilidad**: La web está diseñada para ser usable por todos, con accesibilidad integrada en HTML, CSS y el comportamiento del navegador.
- **Interoperabilidad**: Una página construida según el estándar debe comportarse idénticamente en todos los navegadores y sistemas operativos.
- **Compatibilidad hacia atrás**: Las nuevas capacidades se agregan sobre las existentes para que las páginas antiguas sigan funcionando en navegadores modernos.
- **Privacidad y seguridad**: Las nuevas APIs que podrían exponer datos de usuario enfrentan un escrutinio significativo; la plataforma protege a los usuarios por defecto.
- **Capas**: HTML maneja la estructura, CSS la presentación y JavaScript el comportamiento — cada capa funciona de forma independiente.
- **Mejora progresiva**: Las funcionalidades deben funcionar primero para todos y luego mejorar para usuarios con navegadores o dispositivos más capaces.
