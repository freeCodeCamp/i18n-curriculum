# --description--

Haben Sie sich schon einmal gefragt, was passiert, wenn in Ihrer Express-Anwendung etwas schiefgeht?

Was, wenn eine Datenbankverbindung fehlschlägt? Oder ein Benutzer eine Route besucht, die nicht existiert?

Zu verstehen, wie das Fehlerhandling in Express funktioniert, ist entscheidend, wenn Sie sichere und zuverlässige Webanwendungen erstellen möchten.

In dieser Lektion zeigen wir Ihnen, wie Express Fehler behandelt und wie Sie sie mit echten Codebeispielen abfangen und darauf reagieren können.

## Wie man einfache Fehler in Express behandelt

In Express basiert das Fehlerhandling auf Middleware.

Genau wie reguläre Middleware-Funktionen hat Fehlerbehandlungs-Middleware Zugriff auf `req`, `res` und `next`. Aber sie enthält einen zusätzlichen Parameter am Anfang: `err`.

So sieht die Syntax aus:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

Sie definieren diese Middleware am Ende Ihres Middleware-Stacks.

Express erkennt sie als Fehler-Handler, weil sie vier Parameter hat – den neuen `err` zusätzlich zu den üblichen `req`, `res` und `next` Parametern für andere Middleware-Funktionen.

In einer kommenden Lektion lernen Sie mehr über Fehlerbehandlungs-Middleware und wie sie in den größeren Express-Middleware-Stack passt.

## Wie man asynchrone Fehler behandelt

Die Behandlung asynchroner Fehler hängt davon ab, welche Version von Express Sie verwenden.

Beachten Sie, dass Sie unabhängig von der Express-Version Ihre Fehlerbehandlungs-Middleware immer am Ende Ihres Stacks definieren müssen, also nach allen Ihren Routen.

Hier ist der Fehler-Handler noch einmal:

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Express 4

In Express 4 und älteren Versionen, die viele Projekte noch verwenden, werden asynchrone Fehler **nicht automatisch abgefangen**.

Wenn Sie `async`/`await` verwenden und etwas in einem `async` Route-Handler einen Fehler wirft, fängt Express diesen nicht ab, es sei denn, Sie geben ihn manuell an `next()` weiter.

So behandeln Sie asynchrone Fehler richtig in Express 4:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

Wenn Sie den `try/catch` Block und `next(err)` vergessen, kann der Fehler:

* Ihre Anwendung zum Absturz bringen
* Oder eine Warnung wegen einer unbehandelten Promise-Ablehnung verursachen

Deshalb ist es in Express 4 essenziell, asynchrone Logik in `try`/`catch` einzuwickeln.

Manche Entwickler erstellen sogar eine Hilfsfunktion, um zu vermeiden, `try/catch` überall zu wiederholen:

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

Dann verwenden sie sie so:

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

So müssen Sie in all Ihren `try/catch` Routen keine `async` Blöcke verwenden und leiten Fehler trotzdem korrekt weiter.

## Express 5

Die aktuelle Version von Express verbessert dieses Verhalten.

Ab Express 5, wenn eine `async` Route einen Fehler wirft oder ein abgelehntes Promise zurückgibt, leitet Express diesen automatisch an die Fehlerbehandlungs-Middleware weiter.

Das bedeutet, dass dieses Beispiel Fehler ohne einen `try/catch` behandelt:

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

Wenn `getUserFromDatabase()` einen Fehler wirft, fängt Express 5 ihn ab und gibt ihn automatisch an Ihren Fehler-Handler weiter.

Ein manueller `next(err)` Aufruf ist nicht mehr nötig.

## Wie man 404-Fehler behandelt

404-Fehler sind etwas anders. Sie treten auf, wenn keine Route zur Anfrage passt.

Sie können sie mit dieser Middleware behandeln:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**Wichtig**: Diese Middleware sollte nach Ihren Routen, aber vor Ihrer Fehlerbehandlungs-Middleware stehen.

Und schließlich hier einige bewährte Vorgehensweisen für das Fehlerhandling in Express:

* Platzieren Sie Ihre Fehlerbehandlungs-Middleware immer zuletzt.
* Behandeln Sie 404-Fehler separat.
* Verwenden Sie `next(err)`, um Fehler aus Routen oder `async` Logik in Express 4 und älteren Versionen weiterzuleiten.

In der Produktion sollten Sie keine Stack-Traces an Benutzer senden. Protokollieren Sie die Details intern und geben Sie stattdessen eine benutzerfreundliche Nachricht zurück.

Fehlerhandling in Express bedeutet, Middleware einzurichten, die auf Probleme hört und klar sowie sicher darauf reagiert. Egal ob ein Bug in Ihrem Code oder eine fehlende Route – Ihre Anwendung sollte niemals ohne Backup-Plan abstürzen.
