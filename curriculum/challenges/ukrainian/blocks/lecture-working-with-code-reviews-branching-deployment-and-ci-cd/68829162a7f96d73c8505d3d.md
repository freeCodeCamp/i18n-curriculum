# --description--

Якщо ви спілкувалися з досвідченими розробниками, чи то в онлайн-спільноті, чи особисто, ви, мабуть, чули про «страшні конфлікти злиття». Але що це насправді?

Отже, коли ви використовуєте правильну стратегію гілкування, як ми обговорювали в попередніх уроках, ви можете одночасно працювати над двома різними функціями в двох різних гілках. Або, можливо, ви працюєте над чимось, а колега — над іншою гілкою. У будь-якому разі, у вас є дві незалежні зміни. Але іноді обидві ці гілки редагують один і той самий файл. І якщо інша гілка буде об’єднана в `main` першою, зміни вашої гілки опиняються позаду.

Зазвичай це можна виправити, об’єднавши `main` назад у вашу функціональну гілку. Але коли зміни з іншої гілки конфліктують із вашими, Git не може визначити, які зміни ви хочете зберегти під час злиття. Це називається «конфліктом злиття».

А як це виправити? Для менших і простіших конфліктів GitHub надасть вам веб-інтерфейс для їх виправлення. Кнопка «Вирішити конфлікти» з’явиться внизу запиту на злиття, якщо є конфлікти. Якщо ви натиснете на неї, ви побачите файл із конфліктами, який містить ваш код і конфліктний код. Git додасть «маркувальники конфлікту злиття» навколо вашого коду та конфліктного коду, щоб допомогти ідентифікувати проблему. Це виглядає приблизно так:

```sh
1 <<<<<<< feat/conflict
2 Here's some code from my first PR :)
3 =======
4 Here's some code that was merged to main that conflicts with the code on my PR
5 >>>>>>> main
```

Відкриваючий маркер — це `<<<<<<< feat/conflict`, і він позначає початок змін, які робить наша поточна гілка. Потім ми бачимо `Here's some code from my first PR :)`. Після цього йде маркер `=======`, який позначає кінець наших змін і початок змін гілки `main`. Ми бачимо, що зміни, внесені в `main`, — це рядок `Here's some code that was merged to main that conflicts with the code on my PR`. А після цього йде закриваючий маркер `>>>>>> main`, який показує кінець конфліктів.

А як це виправити? Вам потрібно вирішити, які зміни ви хочете зберегти в рамках конфліктів злиття. Якщо ви хочете зберегти обидві зміни, можна просто видалити маркувальники конфлікту і залишити зміни без змін. Якщо ж ви хочете зберегти зміни лише з однієї гілки, потрібно видалити маркувальники конфлікту та зміни з іншої гілки (ті, які ви не хочете зберігати). І нарешті, якщо вам не подобається жодна зі змін, можна видалити все від відкриваючого маркера до закриваючого (включно).

Давайте видалимо зміни гілки `main`, але залишимо наші. Ось що ми маємо тепер:

```sh
1 Here's some code from my first PR :)
```

Натисніть кнопку «Позначити як вирішене», щоб повідомити GitHub, що ви задоволені своїм рішенням. Якщо були інші файли, GitHub переключився б на наступний. Але оскільки це наш єдиний файл із конфліктом, з’явиться кнопка «Зробити коміт злиття». Натисніть цю кнопку, і GitHub створить коміт злиття для вас. Вас повернуть до запиту на злиття, де ви побачите свій новий коміт злиття і вирішені конфлікти. І все готово!

А що, якщо ви не можете редагувати конфлікти безпосередньо на GitHub, бо вони надто складні?

Якщо у вашій гілці небагато комітів або лише кілька комітів змінюють цей файл, ви можете спробувати об’єднати `main` у вашу гілку:

```sh
git checkout feat/conflict
git fetch origin
git merge origin/main
```

Під час обробки цього злиття ви отримаєте конфлікти, які потрібно вирішити вручну. Git повідомить, які файли потрібно виправити. Коли ми відкриваємо файли з конфліктами в редакторі, ми бачимо ті самі маркувальники конфлікту, з деякими корисними доповненнями інтерфейсу завдяки VSCode. Виправте конфлікти у файлі так само. Але цього разу нам потрібно вручну створити коміт. Це так само, як створення будь-якого іншого коміту:

```sh
git add .
git commit -m "chore: resolve conflicts"
```

Після виконання цих команд ми бачимо, що Git прийняв наше рішення.

Іноді, коли у вас багато комітів або конфліктів, корисним може бути rebase. Замість того, щоб об’єднувати зміни, rebase бере всі коміти вашої гілки і повторно застосовує їх так, щоб вони йшли після останніх змін на `main`. У термінах Git це фактично «скидає» базу вашої гілки до найновішого коміту на `main`.

Щоб попрактикуватися з цим, нам потрібна нова гілка:

```sh
git checkout main
git pull
git checkout -b feat/rebase
```

Давайте створимо багато комітів для нашої гілки, щоб потренуватися з rebase:

```sh
echo "change one" > README.md
git add .
git commit -m "change one"
echo "change two" > README.md
git add .
git commit -m "change two"
echo "change three" > README.md
git add .
git commit -m "change three"
```

Тепер зробимо ще одну зміну на `main`, щоб створити конфлікт:

```sh
git checkout main
echo "change four" > README.md
git add .
git commit -m "change four"
```

І переключимося на нашу гілку для rebase:

```sh
git checkout feat/rebase
```

Далі виконайте `git rebase -i main`. Це зробить rebase нашої гілки на поточний стан гілки `main`. Прапорець `-i` каже Git виконати rebase в інтерактивному режимі. Ви побачите свої коміти з написом `pick` поруч із ними. `pick` каже Git використовувати коміт без змін для rebase. Збережіть і закрийте файл, і ми отримаємо повідомлення про помилку, що є конфлікти.

Наш коміт `change one` конфліктує з комітом `change four`, і нам потрібно це вирішити. Відкрийте файл у редакторі, і ви побачите той самий інтерфейс, що й раніше з конфліктами. Залишимо наші зміни `change one`. Потім потрібно виконати `git add .`, щоб додати зміни, і `git rebase --continue`, щоб повідомити Git, що ми готові продовжити rebase. Оскільки ми вирішили конфлікт у першому коміті, Git автоматично обробить rebase решти комітів.

Тепер, для останнього очищення, давайте об’єднаємо наші три коміти змін в один для чистішої історії. Знову виконайте `git rebase -i main`, щоб відкрити редактор. Потім залиште першу команду як `pick`, а наступні дві змініть на `squash`. Можливо, ви знову зіткнетеся з тим самим конфліктом. Якщо так, вирішіть його так само, як раніше. Після того, як ви вирішите і продовжите rebase (або одразу, якщо конфліктів не було), з’явиться редактор повідомлення нового коміту. Це повідомлення буде вашим новим комітом, що містить зміни з трьох об’єднаних комітів. Назвемо його `change three complete`. Збережіть і закрийте редактор, і Git завершить rebase.

Якщо зараз перевірити історію ваших комітів, ми побачимо, що тепер у нас є один коміт `change three complete`, і він правильно базується на коміті `change four`. Це означає, що ви успішно виконали rebase з об’єднаними комітами! І тепер у вас є інструменти, щоб стати чемпіоном з вирішення конфліктів злиття!
