## 수동 테스트와 자동 테스트

- **수동 테스트**: 수동 테스트에서는 테스터가 애플리케이션의 각 부분을 직접 확인하며 다양한 기능이 제대로 작동하는지 테스트합니다. 테스트 과정에서 버그가 발견되면 테스터가 소프트웨어 팀에 보고하여 수정할 수 있게 합니다.
- **자동 테스트**: 자동 테스트에서는 애플리케이션이 예상대로 작동하는지 확인하는 별도의 프로그램을 작성하여 테스트를 자동화할 수 있습니다.

## 단위 테스트

- **단위 테스트**: 단위 테스트에서는 각 함수를 테스트하여 모든 것이 예상대로 작동하는지 확인합니다. 단위 테스트는 코드의 예상 동작을 나타내기 때문에 애플리케이션 문서 역할도 할 수 있습니다.
- **단일 책임 원칙**: 단일 책임 원칙은 각 함수가 작고 하나의 책임만 가지도록 유지할 것을 권장합니다.
- **일반적인 JavaScript 테스트 프레임워크**: Jest, Mocha, Vitest 등이 일반적인 테스트 프레임워크입니다. Jest는 단위 테스트에 널리 사용되는 프레임워크입니다.

다음은 Jest를 사용한 단위 테스트 예시입니다.

먼저, 새로 포맷된 문자열을 반환하는 함수를 만들 수 있습니다:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

별도의 `getFormattedWord.test.js` 파일에서 함수가 예상대로 작동하는지 확인하는 테스트를 작성할 수 있습니다. `getFormattedWord.test.js` 파일은 다음과 같습니다:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **`expect` 함수**: `expect` 함수는 값을 테스트하는 데 사용됩니다.
- **매처**: 매처는 값이 예상대로 동작하는지 확인하는 함수입니다. 위 예시에서 매처는 `toBe()`입니다. Jest에는 다양한 매처가 있습니다.

Jest를 사용하려면 먼저 `jest` 패키지를 `npm i jest`로 설치해야 합니다. 또한 `package.json` 파일에 다음과 같은 스크립트를 추가해야 합니다:

```js
"scripts": {
  "test": "jest"
},
```

그런 다음 `npm run test` 명령어로 테스트를 실행할 수 있습니다.

## 소프트웨어 개발 생명주기

- **소프트웨어 개발 생명주기의 다양한 단계**:
  - **기획 단계**: 개발 팀이 이해관계자로부터 작업 요구사항을 수집합니다.
  - **설계 단계**: 소프트웨어 팀이 요구사항을 분해하고 최적의 해결책을 결정합니다.
  - **구현 단계**: 소프트웨어 팀이 요구사항을 관리 가능한 작업으로 나누어 구현합니다.
  - **테스트 단계**: 새 작업에 대해 수동 및 자동 테스트를 수행합니다. 때로는 개발 전 과정에서 애플리케이션을 테스트하여 문제를 발견하고 수정합니다.
  - **배포 단계**: 팀이 새 변경사항을 빌드 또는 테스트 환경에 배포합니다.
  - **유지 관리 단계**: 운영 중인 애플리케이션에서 발생하는 문제를 수정합니다.

- **소프트웨어 개발 생명주기의 다양한 모델**:
  - **워터폴 모델**: 각 단계가 완료되어야 다음 단계로 넘어갈 수 있는 모델입니다.
  - **애자일 모델**: 작업을 스프린트로 나누어 반복적으로 개발하는 모델입니다.

## BDD와 TDD

- **TDD**: 테스트 주도 개발은 먼저 테스트를 작성하는 방법론입니다. 기능을 구현하기 전에 테스트를 작성하면 개발 과정에서 실시간 피드백을 받을 수 있습니다.
- **BDD**: 행동 주도 개발은 테스트 시나리오를 비즈니스 목표에 맞추는 접근법입니다. BDD의 테스트 시나리오는 기술자와 비기술자 모두 이해할 수 있는 언어로 작성되어야 합니다. Gherkin 구문이 그 예입니다.
- **BDD 테스트 프레임워크**: Cucumber, JBehave, SpecFlow 등이 있습니다.

## 단위 테스트에서의 어서션

- **어서션**: 어서션은 코드가 예상대로 동작하는지 테스트하는 데 사용됩니다.
- **어서션 라이브러리**: Chai는 널리 사용되는 어서션 라이브러리입니다. 다른 일반적인 JavaScript 어서션 라이브러리는 `should.js`와 `expect.js`입니다.

다음은 Chai를 사용해 `addThreeAndFour` 함수의 반환값이 숫자 7과 같은지 확인하는 어서션 예시입니다:

```js
assert.equal(addThreeAndFour(), 7);
```

- **모범 사례**: 어떤 어서션 라이브러리를 사용하든, 어떤 테스트가 실패하는지와 그 이유를 이해할 수 있도록 명확한 어서션 및 실패 메시지를 작성해야 합니다.

## 목킹, 페이킹, 스터빙

- **목킹**: 목킹은 실제 데이터를 가짜 데이터로 대체하여 실제 컴포넌트의 동작을 시뮬레이션하는 과정입니다. 예를 들어, 테스트 중에 API를 계속 호출하는 대신 API 응답을 목킹할 수 있습니다.
- **스터빙**: 스터브는 애플리케이션에서 예상되는 동작에 대해 미리 정의된 응답이나 더미 데이터를 반환하는 객체입니다. 예를 들어, 실제 데이터베이스 연결 없이 테스트에서 데이터베이스 연결 동작을 스터빙할 수 있습니다.
- **페이킹**: 페이크는 복잡성이나 부작용 없이 실제 컴포넌트의 단순화된 버전입니다. 예를 들어, 실제 데이터베이스 대신 메모리에 데이터를 저장하여 데이터베이스 작업을 모방할 수 있습니다. 이렇게 하면 실제 데이터베이스를 다루는 것보다 훨씬 빠릅니다.

## 기능 테스트

- **기능 테스트**: 기능 테스트는 애플리케이션의 기능과 특징이 예상대로 작동하는지 확인합니다. 기능 테스트의 목표는 여러 시나리오에 대해 시스템 전체를 테스트하는 것입니다.
- **비기능 테스트**: 비기능 테스트는 성능과 신뢰성 같은 부분에 집중합니다.
- **스모크 테스트**: 스모크 테스트는 더 광범위한 테스트를 시작하기 전에 시스템의 기본적이거나 중요한 문제를 예비 점검하는 것입니다.

## 종단 간 테스트

- **종단 간 테스트**: 종단 간 테스트(E2E)는 사용자의 관점에서 실제 시나리오를 테스트합니다. E2E 테스트는 애플리케이션이 올바르게 동작하고 사용자에게 예측 가능하도록 돕습니다. 하지만 설정, 설계, 유지 관리에 시간이 많이 걸립니다.
- **종단 간 테스트 프레임워크**: Playwright는 Microsoft에서 개발한 인기 있는 E2E 테스트 프레임워크입니다. 그 밖에 Cypress, Selenium, Puppeteer 등이 있습니다.

다음은 freeCodeCamp 코드베이스에서 Playwright를 사용한 E2E 테스트 예시입니다. `beforeEach` 훅은 각 테스트 전에 실행됩니다. 테스트는 기부자가 메뉴 바에 후원자 링크가 있고 아바타 주위에 특별한 스타일의 테두리가 있는지 확인합니다:

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## 사용성 테스트

- **사용성 테스트**: 사용성 테스트는 실제 사용자가 애플리케이션과 상호작용하며 디자인, 사용자 경험, 기능상의 문제를 발견하는 과정입니다. 사용성 테스트는 사용자의 직관성을 평가합니다.
- **사용성 테스트의 네 가지 일반 유형**:
  - **탐색적**: 사용자가 애플리케이션의 다양한 기능을 사용해 보며 작동 방식을 이해합니다.
  - **비교적**: 애플리케이션의 사용자 경험을 시장 내 유사 애플리케이션과 비교합니다.
  - **평가적**: 애플리케이션 사용의 직관성을 연구합니다.
  - **검증적**: 사용자가 애플리케이션을 효과적으로 사용하는 데 방해가 되는 주요 문제를 식별합니다.
- **사용성 테스트 도구**: Loop11, Maze, Userbrain, UserTesting, UXTweak 등이 있습니다.

## 호환성 테스트

- **호환성 테스트**: 호환성 테스트의 목표는 애플리케이션이 다양한 컴퓨팅 환경에서 작동하는지 확인하는 것입니다.
- **호환성 테스트의 다양한 유형**:
  - **하위 호환성**: 소프트웨어가 이전 버전과 호환되는 경우를 말합니다.
  - **상위 호환성**: 소프트웨어와 시스템이 미래 버전과도 작동하는 경우를 말합니다.
  - **하드웨어 호환성**: 소프트웨어가 다양한 하드웨어 구성에서 제대로 작동하는 능력입니다.
  - **운영체제 호환성**: macOS, Windows, Ubuntu, Fedora 같은 다양한 운영체제에서 소프트웨어가 작동하는 능력입니다.
  - **네트워크 호환성**: 다양한 네트워크 속도, 프로토콜, 보안 설정 등에서 소프트웨어가 작동하는 능력입니다.
  - **브라우저 호환성**: Google Chrome, Safari, Firefox 등 다양한 브라우저에서 웹 애플리케이션이 일관되게 작동하는 능력입니다.
  - **모바일 호환성**: Android와 iOS 기기(휴대폰, 태블릿 등)에서 소프트웨어 애플리케이션이 작동하는지 확인하는 것이 중요합니다.

## 성능 테스트

- **성능 테스트**: 성능 테스트에서는 애플리케이션의 속도, 반응성, 확장성, 안정성을 다양한 작업 부하에서 테스트합니다. 목표는 성능 병목 현상을 해결하는 것입니다.
- **성능 테스트의 다양한 유형**:
  - **부하 테스트**: 정상 및 최대 부하 시 시스템의 동작을 확인합니다.
  - **스트레스 테스트**: 극한 부하에서 애플리케이션을 테스트하여 시스템이 어떻게 반응하는지 봅니다.
  - **지속 테스트(내구성 테스트)**: 장시간 높은 부하로 시스템을 테스트하는 부하 테스트 유형입니다.
  - **스파이크 테스트**: 부하를 급격히 증가 및 감소시키며 시스템 반응을 분석합니다.
  - **한계점 테스트(용량 테스트)**: 부하를 점진적으로 증가시켜 시스템이 실패하거나 성능이 저하되는 시점을 찾습니다.

## 보안 테스트

- **보안 테스트**: 보안 테스트는 취약점과 약점을 식별하는 데 도움을 줍니다.
- **보안 원칙**:
  - **기밀성**: 의도된 수신자가 아닌 다른 사람에게 민감한 정보가 노출되는 것을 방지합니다.
  - **무결성**: 악의적인 사용자가 사용자 정보를 변경하지 못하도록 합니다.
  - **인증**: 사용자의 신원을 확인하여 시스템 사용 권한이 있는지 검증합니다.
  - **권한 부여**: 인증된 사용자가 수행할 수 있는 작업이나 접근 가능한 시스템 부분을 결정하는 과정입니다.
  - **가용성**: 권한 있는 사용자가 필요할 때 정보와 서비스를 이용할 수 있도록 보장합니다.
  - **부인 방지**: 송신자와 수신자가 전달 증명과 송신자 신원 확인을 갖도록 하여 송신자가 전송을 부인하지 못하게 합니다.
- **일반적인 보안 위협**:
  - **교차 사이트 스크립팅(XSS)**: 공격자가 악성 스크립트를 웹 페이지에 삽입해 피해자의 브라우저에서 실행하는 공격입니다.
  - **SQL 인젝션**: 악의적인 사용자가 데이터베이스에 악성 코드를 삽입하는 공격입니다.
  - **서비스 거부(DoS) 공격**: 악의적인 사용자가 대량의 요청이나 트래픽을 보내 서버를 느리게 하거나 다운시켜 사이트를 사용할 수 없게 만드는 공격입니다.
- **보안 테스트 도구 분류**:
  - **정적 애플리케이션 보안 테스트**: 애플리케이션 소스 코드를 평가하여 보안 취약점을 식별합니다.
  - **동적 애플리케이션 보안 테스트**: 애플리케이션 프런트엔드와 인터페이스하여 잠재적 보안 약점을 찾아냅니다. DAST 도구는 소스 코드에 접근하지 않습니다.
- **침투 테스트(펜테스트)**: 애플리케이션에 모의 사이버 공격을 수행하여 시스템의 취약점을 식별하는 보안 테스트 유형입니다.

## A/B 테스트

- **A/B 테스트**: A/B 테스트는 페이지나 애플리케이션의 두 버전을 비교하여 어느 쪽이 더 나은 성능을 내는지 연구하는 방법입니다. 버킷 테스트 또는 분할 테스트라고도 합니다. A/B 테스트는 데이터 기반 의사결정을 가능하게 하여 사용자 경험을 지속적으로 개선합니다.
- **A/B 테스트 도구**: GrowthBook, LaunchDarkly 등이 있습니다.

## 알파 및 베타 테스트

초기 개발과 소프트웨어 테스트가 완료되면 테스터와 실제 사용자가 애플리케이션을 테스트하는 것이 중요합니다. 이때 알파 테스트와 베타 테스트가 필요합니다.

- **알파 테스트**: 알파 테스트는 선택된 테스터 그룹이 애플리케이션을 점검하여 출시 전에 버그가 없는지 확인하는 과정입니다. 알파 테스트는 인수 테스트의 일부이며 화이트 박스와 블랙 박스 테스트 기법을 모두 사용합니다.
- **베타 테스트**: 베타 테스트는 애플리케이션을 실제 사용자에게 공개하여 사용자가 상호작용하고 피드백을 제공하는 과정입니다. 베타 테스트 역시 사용자 인수 테스트의 한 형태입니다.
- **인수 테스트**: 인수 테스트는 소프트웨어 애플리케이션이 비즈니스 요구사항과 사용자 요구를 충족하는지 확인하는 과정입니다.
- **블랙 박스 테스트**: 블랙 박스 테스트는 애플리케이션의 예상 동작에만 집중합니다.
- **화이트 박스 테스트**: 화이트 박스 테스트는 테스터가 내부 구성 요소를 알고 테스트를 수행합니다.

## 회귀 테스트

- **회귀**: 회귀는 새로운 변경사항이 기존 기능을 의도치 않게 망가뜨리는 상황을 말합니다.
- **회귀 테스트**: 회귀 테스트는 회귀 문제를 발견하는 데 도움을 줍니다. 회귀 테스트에서는 애플리케이션의 일부에 대해 기능 테스트를 다시 실행하여 모든 것이 여전히 예상대로 작동하는지 확인합니다.
- **회귀 테스트 도구**: Puppeteer, Playwright, Selenium, Cypress 등이 있습니다.
- **회귀 테스트 기법**:
  - **단위 회귀 테스트**: 주요 변경이나 수정이 있을 때마다 테스트해야 할 항목 목록을 갖고 테스트하는 방법입니다.
  - **부분 회귀 테스트**: 새 변경사항이 애플리케이션의 특정 부분을 망가뜨리지 않았는지 확인하는 타깃 접근법입니다.
  - **전체 회귀 테스트**: 코드베이스의 모든 기능에 대해 테스트를 실행하는 방법으로 가장 시간이 많이 걸리고 상세한 옵션입니다.
- **재테스트**: 재테스트는 알려진 문제를 확인하고 해결되었는지 확인하는 데 사용됩니다. 반면 회귀 테스트는 최근 코드 변경으로 발생할 수 있는 알려지지 않은 문제를 찾습니다.
