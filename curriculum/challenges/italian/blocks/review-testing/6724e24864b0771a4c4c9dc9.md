## Test manuali e automatizzati

- **Test manuale**: Nel test manuale, un tester esamina manualmente ogni parte dell’applicazione e prova diverse funzionalità per assicurarsi che funzionino correttamente. Se durante il test vengono scoperti bug, il tester li segnala al team di sviluppo software affinché possano essere risolti.
- **Test automatizzato**: Nel test automatizzato, puoi automatizzare i test scrivendo un programma separato che verifica se la tua applicazione si comporta come previsto.

## Test unitari

- **Test unitario**: Nel test unitario, testi ogni funzione per assicurarti che tutto funzioni come previsto. I test unitari possono anche servire come forma di documentazione per la tua applicazione perché rappresentano il comportamento atteso del codice.
- **Principio di responsabilità singola**: Il principio di responsabilità singola consiglia di mantenere ogni funzione piccola e responsabile di una sola cosa.
- **Framework comuni per test JavaScript**: Alcuni framework comuni per i test includono Jest, Mocha e Vitest. Jest è un framework popolare per i test unitari.

Ecco un esempio di test unitari usando Jest.

Per prima cosa, puoi creare una funzione responsabile di restituire una stringa formattata di recente:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

In un file `getFormattedWord.test.js` separato, puoi scrivere alcuni test per verificare che la funzione funzioni come previsto. Il file `getFormattedWord.test.js` sarà così:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **Funzione `expect`**: La funzione `expect` viene usata per testare un valore.
- **Matcher**: Il matcher è una funzione che verifica se il valore si comporta come previsto. Nell’esempio sopra, il matcher è `toBe()`. Jest offre una varietà di matcher.

Per usare Jest, devi prima installare il pacchetto `jest` usando `npm i jest`. Dovrai anche aggiungere uno script al file `package.json` così:

```js
"scripts": {
  "test": "jest"
},
```

Poi puoi eseguire il comando `npm run test` per far partire i test.

## Ciclo di vita dello sviluppo software

- **Diverse fasi del ciclo di vita dello sviluppo software**:
  - **Fase di pianificazione**: Il team di sviluppo raccoglie i requisiti per il lavoro proposto dagli stakeholder.
  - **Fase di progettazione**: Il team software suddivide i requisiti e decide le migliori soluzioni.
  - **Fase di implementazione**: Il team software suddivide i requisiti in attività gestibili e le implementa.
  - **Fase di test**: Questa fase include test manuali e automatizzati per il nuovo lavoro. A volte il team testa l’applicazione durante tutta la fase di sviluppo per individuare e risolvere eventuali problemi.
  - **Fase di distribuzione**: Il team distribuisce le nuove modifiche in un ambiente di build o test.
  - **Fase di manutenzione**: Questa fase prevede la risoluzione di problemi segnalati dagli utenti nell’applicazione in produzione.

- **Diversi modelli del ciclo di vita dello sviluppo software**:
  - **Modello a cascata**: Nel modello a cascata ogni fase del ciclo di vita deve essere completata prima che inizi la successiva.
  - **Modello agile**: Il modello agile si concentra sullo sviluppo iterativo suddividendo il lavoro in sprint.

## BDD e TDD

- **TDD**: Lo sviluppo guidato dai test è una metodologia che enfatizza la scrittura dei test prima di tutto. Scrivere i test prima di implementare le funzionalità fornisce un feedback in tempo reale agli sviluppatori durante lo sviluppo.
- **BDD**: Lo sviluppo guidato dal comportamento è l’approccio che allinea una serie di test agli obiettivi di business. Gli scenari di test in BDD dovrebbero essere scritti in un linguaggio comprensibile sia da persone tecniche sia non tecniche. Un esempio di sintassi è Gherkin.
- **Framework per test BDD**: Esempi di framework per test BDD includono Cucumber, JBehave e SpecFlow.

## Asserzioni nei test unitari

- **Asserzione**: Le asserzioni servono a verificare che il codice si comporti come previsto.
- **Librerie di asserzioni**: Chai è una libreria di asserzioni comunemente usata. Altre librerie comuni per asserzioni JavaScript sono `should.js` e `expect.js`.

Ecco un esempio di asserzione con Chai che verifica che il valore di ritorno della funzione `addThreeAndFour` sia uguale al numero 7:

```js
assert.equal(addThreeAndFour(), 7);
```

- **Buone pratiche**: Qualunque libreria di asserzioni tu usi, dovresti scrivere messaggi chiari di asserzione e di errore che ti aiutino a capire quali test stanno fallendo e perché.

## Mocking, faking e stubbing

- **Mocking**: Il mocking è il processo di sostituire dati reali con dati falsi che simulano il comportamento di componenti reali. Per esempio, potresti simulare la risposta di un’API durante i test invece di fare chiamate continue all’API per recuperare i dati.
- **Stubbing**: Gli stub sono oggetti che restituiscono risposte predefinite o dati fittizi per un comportamento atteso in un’applicazione. Per esempio, puoi stubbare il comportamento di una connessione al database nei test senza dover dipendere da una connessione reale.
- **Faking**: I fake sono versioni semplificate di componenti reali senza la complessità o gli effetti collaterali. Per esempio, puoi simulare un database memorizzando i dati in memoria invece di interagire con il database reale. Questo ti permette di imitare le operazioni del database in memoria, molto più velocemente che con il database reale.

## Test funzionali

- **Test funzionale**: Il test funzionale verifica se le funzionalità e le funzioni dell’applicazione funzionano come previsto. L’obiettivo è testare il sistema nel suo insieme contro scenari multipli.
- **Test non funzionale**: Il test non funzionale si concentra su aspetti come prestazioni e affidabilità.
- **Smoke testing**: Lo smoke testing è un controllo preliminare sul sistema per individuare problemi di base o critici prima di iniziare test più approfonditi.

## Test end-to-end

- **Test end-to-end**: Il test end-to-end, o E2E, verifica scenari reali dal punto di vista dell’utente. I test end-to-end aiutano a garantire che la tua applicazione si comporti correttamente e sia prevedibile per gli utenti. Tuttavia, richiede molto tempo per essere configurato, progettato e mantenuto.
- **Framework per test end-to-end**: Playwright è un framework popolare per test end-to-end sviluppato da Microsoft. Altri esempi di strumenti per test end-to-end includono Cypress, Selenium e Puppeteer.

Ecco un esempio di test E2E dal codice freeCodeCamp usando Playwright. Il hook `beforeEach` verrà eseguito prima di ogni test. I test verificano che il donatore abbia un collegamento per i sostenitori nella barra del menu, oltre a un bordo stilizzato speciale intorno all’avatar:

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## Test di usabilità

- **Test di usabilità**: Il test di usabilità prevede che utenti reali interagiscano con l’applicazione per scoprire se ci sono problemi di design, esperienza utente o funzionalità. Il test di usabilità si concentra sull’intuitività dell’applicazione per gli utenti.
- **Quattro tipi comuni di test di usabilità**:
  - **Esplorativo**: Il test di usabilità esplorativo coinvolge utenti che interagiscono con le diverse funzionalità dell’applicazione per capire meglio come funzionano.
  - **Comparativo**: Il test comparativo confronta l’esperienza utente della tua applicazione con applicazioni simili sul mercato.
  - **Valutativo**: Il test valutativo studia quanto l’applicazione sia intuitiva da usare.
  - **Di validazione**: Il test di validazione identifica eventuali problemi gravi che impedirebbero all’utente di usare efficacemente l’applicazione.
- **Strumenti per test di usabilità**: Esempi di strumenti per test di usabilità includono Loop11, Maze, Userbrain, UserTesting e UXTweak.

## Test di compatibilità

- **Test di compatibilità**: L’obiettivo del test di compatibilità è assicurarsi che la tua applicazione funzioni in diversi ambienti informatici.
- **Diversi tipi di test di compatibilità**:
  - **Compatibilità retroattiva**: La compatibilità retroattiva indica che il software è compatibile con versioni precedenti.
  - **Compatibilità futura**: La compatibilità futura indica che il software e i sistemi funzioneranno con versioni future.
  - **Compatibilità hardware**: La compatibilità hardware è la capacità del software di funzionare correttamente in diverse configurazioni hardware.
  - **Compatibilità sistemi operativi**: La compatibilità con i sistemi operativi è la capacità del software di funzionare su diversi sistemi operativi, come macOS, Windows e distribuzioni Linux come Ubuntu e Fedora.
  - **Compatibilità di rete**: La compatibilità di rete significa che il software può funzionare in diverse condizioni di rete, come velocità diverse, protocolli, impostazioni di sicurezza ecc.
  - **Compatibilità browser**: La compatibilità browser significa che l’applicazione web funziona in modo coerente su diversi browser, come Google Chrome, Safari, Firefox ecc.
  - **Compatibilità mobile**: È importante assicurarsi che le applicazioni software funzionino su una varietà di dispositivi Android e iOS, inclusi telefoni e tablet.

## Test di prestazioni

- **Test di prestazioni**: Nel test di prestazioni si verifica la velocità, la reattività, la scalabilità e la stabilità di un’applicazione sotto diversi carichi di lavoro. L’obiettivo è risolvere qualsiasi tipo di collo di bottiglia nelle prestazioni.
- **Diversi tipi di test di prestazioni**:
  - **Load testing**: Il load testing determina come si comporta un sistema durante carichi normali e di picco.
  - **Stress testing**: Lo stress testing verifica come l’applicazione risponde a carichi estremi.
  - **Soak testing (test di resistenza)**: Il soak testing o test di resistenza è un tipo di load testing in cui si testa il sistema con un carico elevato per un periodo prolungato.
  - **Spike testing**: Lo spike testing consiste nell’aumentare e diminuire drasticamente i carichi e analizzare le reazioni del sistema ai cambiamenti.
  - **Breakpoint testing (test di capacità)**: Il breakpoint testing o test di capacità consiste nell’aumentare lentamente il carico nel tempo fino al punto in cui il sistema inizia a fallire o degradarsi.

## Test di sicurezza

- **Test di sicurezza**: Il test di sicurezza aiuta a identificare vulnerabilità e debolezze.
- **Principi di sicurezza**:
  - **Riservatezza**: Protegge contro la divulgazione di informazioni sensibili a destinatari non autorizzati.
  - **Integrità**: Previene che utenti malintenzionati modifichino informazioni degli utenti.
  - **Autenticazione**: Verifica l’identità dell’utente per assicurarsi che possa usare il sistema.
  - **Autorizzazione**: Determina quali azioni gli utenti autenticati possono eseguire o quali parti del sistema possono consultare.
  - **Disponibilità**: Garantisce che informazioni e servizi siano disponibili agli utenti autorizzati quando necessario.
  - **Non ripudio**: Assicura che mittente e destinatario abbiano prova della consegna e verifica dell’identità del mittente. Protegge contro la negazione da parte del mittente di aver inviato le informazioni.
- **Minacce comuni alla sicurezza**:
  - **Cross-Site Scripting (XSS)**: Gli attacchi XSS avvengono quando un attaccante inietta script malevoli in una pagina web ed esegue tali script nel contesto del browser della vittima.
  - **SQL Injection**: L’iniezione SQL permette a utenti malintenzionati di inserire codice malevolo in un database.
  - **Denial-of-Service (DoS)**: Un attacco DoS si verifica quando utenti malintenzionati inondano un sito web con un alto numero di richieste o traffico, causando rallentamenti o crash del server e rendendo il sito non disponibile agli utenti.
- **Categorie di strumenti per test di sicurezza**:
  - **Static Application Security Testing**: Questi strumenti analizzano il codice sorgente di un’applicazione per identificare vulnerabilità di sicurezza.
  - **Dynamic Application Security Testing**: Questi strumenti interagiscono con il front-end dell’applicazione per scoprire potenziali debolezze di sicurezza. Gli strumenti DAST non hanno accesso al codice sorgente.
- **Test di penetrazione (pentest)**: Il test di penetrazione è un tipo di test di sicurezza che simula attacchi informatici sull’applicazione per identificare vulnerabilità nel sistema.

## Test A/B

- **Test A/B**: Il test A/B consiste nel confrontare due versioni di una pagina o applicazione e studiare quale versione funziona meglio. È anche noto come bucket o split testing. Il test A/B ti permette di prendere decisioni più basate sui dati e migliorare continuamente l’esperienza utente.
- **Strumenti per test A/B**: Esempi di strumenti per test A/B includono GrowthBook e LaunchDarkly.

## Test alpha e beta

Una volta completato lo sviluppo iniziale e i test software, è importante far testare l’applicazione da tester e utenti reali. Qui entrano in gioco i test alpha e beta.

- **Test alpha**: Il test alpha viene eseguito da un gruppo selezionato di tester che esaminano l’applicazione per assicurarsi che non ci siano bug prima del rilascio sul mercato. Il test alpha fa parte del test di accettazione e utilizza tecniche di test sia white box sia black box.
- **Test beta**: Il test beta rende l’applicazione disponibile agli utenti reali. Gli utenti possono interagire con l’applicazione e fornire feedback. Il test beta è anche una forma di test di accettazione utente.
- **Test di accettazione**: Il test di accettazione verifica che l’applicazione software soddisfi i requisiti di business e le esigenze degli utenti prima del rilascio.
- **Test black box**: Il test black box si concentra solo sul comportamento atteso dell’applicazione.
- **Test white box**: Il test white box prevede che il tester conosca i componenti interni e li testi.

## Test di regressione

- **Regressione**: La regressione indica situazioni in cui nuove modifiche rompono involontariamente funzionalità esistenti.
- **Test di regressione**: Il test di regressione aiuta a individuare problemi di regressione. Nel test di regressione si rieseguono test funzionali su parti dell’applicazione per assicurarsi che tutto funzioni ancora come previsto.
- **Strumenti per test di regressione**: Strumenti che puoi usare per il test di regressione includono Puppeteer, Playwright, Selenium e Cypress.
- **Tecniche per test di regressione**:
  - **Test di regressione unitari**: Si tratta di una lista di elementi da testare ogni volta che si implementano modifiche o correzioni importanti nell’app.
  - **Test di regressione parziali**: Approcci mirati per assicurarsi che le nuove modifiche non abbiano rotto aspetti specifici dell’applicazione.
  - **Test di regressione completi**: Esegue test su tutte le funzionalità nel codice. È l’opzione più dettagliata e dispendiosa in termini di tempo.
- **Ritest**: Il ritest serve a verificare problemi noti e assicurarsi che siano stati risolti. Al contrario, il test di regressione cerca problemi sconosciuti che potrebbero essere comparsi con modifiche recenti nel codice.
