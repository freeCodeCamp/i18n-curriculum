## Autenticação vs autorização

**Autenticação** é o processo de verificar a identidade de um usuário. Responde à pergunta *"Esta pessoa é quem diz ser?"* Quando um usuário faz login, o servidor backend compara suas credenciais com os registros armazenados durante o cadastro.

Desenvolvedores backend usam três categorias principais de fatores de autenticação:

- **Algo que você sabe:** Uma senha, PIN ou resposta a uma pergunta de segurança.
- **Algo que você tem:** Um código SMS no smartphone ou um token temporário de um app autenticador.
- **Algo que você é:** Dados biométricos como impressão digital ou reconhecimento facial.

**Autorização** acontece após a autenticação e decide o que o usuário autenticado tem permissão para fazer. Responde à pergunta *"Quais permissões essa identidade possui?"* Papéis comuns incluem:

- **Usuário padrão:** Pode editar seu próprio perfil, mas não pode ver dados privados de outros usuários.
- **Editor:** Pode escrever e publicar conteúdo, mas não pode excluir o código.
- **Admin:** Tem permissão total para modificar bancos de dados, excluir usuários e alterar configurações do sistema.

**A autenticação deve sempre acontecer primeiro, depois a autorização.** Você não pode decidir quais permissões conceder até saber quem é o usuário.

## JSON Web Tokens (JWTs)

Um JWT é um cartão-chave digital que o servidor emite para o usuário após um login bem-sucedido. O usuário o apresenta em cada requisição subsequente para provar sua identidade sem precisar digitar a senha novamente.

O fluxo de quatro passos do JWT:

1. **Login:** O usuário envia as credenciais para o servidor.
2. **Emissão:** O servidor verifica as credenciais e gera um JWT, então o envia de volta.
3. **Armazenamento:** O cliente armazena o JWT, normalmente em um cookie do navegador ou no armazenamento local.
4. **Requisição:** Para cada requisição futura a uma rota protegida, o cliente envia o JWT e o servidor o verifica.

Um JWT consiste em três partes separadas por dois pontos:

- **Cabeçalho:** JSON codificado declarando o tipo do token (JWT) e o algoritmo de hash usado, como HMAC SHA256.
- **Payload:** JSON codificado contendo declarações sobre o usuário, como id, nome de usuário ou papel. O cabeçalho e o payload são codificados em Base64, não criptografados, então nunca armazene dados sensíveis como senhas aqui.
- **Assinatura:** Criada ao fazer hash do cabeçalho e payload codificados junto com uma chave secreta conhecida apenas pelo servidor. Se o payload for alterado, a assinatura não corresponderá e o servidor rejeitará a requisição com uma resposta `401 Unauthorized`.

JWTs permitem **autenticação sem estado**. O servidor não precisa armazenar dados de sessão no banco de dados porque a assinatura prova que ele criou o token.

## Falsificação de requisição entre sites (CSRF)

CSRF é um ataque que força um usuário autenticado a executar ações indesejadas em uma aplicação web sem saber. Um site malicioso engana o navegador da vítima para enviar uma requisição fraudulenta a um servidor backend, explorando o fato de que navegadores anexam automaticamente cookies a qualquer requisição para o servidor alvo.

**Fluxo de ataque exemplo:**

1. Você faz login, por exemplo, em `trusted-bank.com`. O banco armazena um token de sessão no cookie do seu navegador.
2. Sem fazer logout, você visita `evil-hacker-site.com`.
3. O site malicioso envia silenciosamente uma requisição para `trusted-bank.com/transfer?amount=1000&to=hacker`.
4. Seu navegador automaticamente adiciona o cookie de sessão do banco, então o servidor processa a requisição.

O site malicioso não pode ler seu cookie. Ele apenas se aproveita do fato de que seu navegador anexa cookies automaticamente a qualquer requisição para o servidor alvo.

**Defesas contra CSRF:**

- **`SameSite=Strict`:** O navegador nunca envia o cookie se a requisição vier de um site de terceiros.
- **`SameSite=Lax`:** Permite cookies em navegação segura (clicando em um link), mas bloqueia em envios de formulários e requisições API cross-site.
- **Tokens CSRF:** Quando o usuário carrega uma página, o servidor gera um token aleatório, único e imprevisível e o anexa ao formulário frontend ou sessão. O cliente deve incluí-lo nos cabeçalhos da requisição ou dados do formulário. Como um site malicioso não pode acessar o frontend para copiar o token, requisições forjadas falham na validação.

## Passport.js

Passport.js é um middleware de autenticação para Node.js que padroniza como você autentica requisições usando plugins modulares chamados **Strategies**. Com mais de 500 estratégias disponíveis, você pode adicionar qualquer método de autenticação instalando o pacote relevante:

- `passport-local`: nome de usuário e senha contra seu próprio banco de dados.
- `passport-jwt`: valida JSON Web Tokens em rotas API protegidas.
- `passport-google-oauth20`: login com contas Google.
- `passport-github2`: login com contas GitHub.

Instale o pacote principal e uma estratégia:

```bash
npm install passport passport-local
```

Integre o Passport no seu app Express antes de definir as rotas:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database look-up and password verification logic
}));
```

Proteja uma rota de login com Passport como middleware:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

Após a autenticação bem-sucedida, o Passport gerencia automaticamente **a sessão** (armazena dados do usuário no cookie de sessão) e **decora a requisição** (anexa o objeto do usuário em `req.user` para uso nos manipuladores de rota seguintes).

## Helmet.js

Helmet.js é uma coleção de middleware de segurança para Node.js que protege sua aplicação configurando, modificando ou removendo automaticamente cabeçalhos HTTP de resposta que o Express expõe por padrão.

Instale e aplique em um único passo:

```bash
npm install helmet
```

```js
const helmet = require('helmet');
app.use(helmet());
```

Colocar `app.use(helmet())` no topo da sua pilha de middleware aplica essas proteções a toda resposta:

- **X-Powered-By removido:** O Express anuncia sua presença por padrão com `X-Powered-By: Express`. O Helmet remove esse cabeçalho, impedindo que atacantes explorem vulnerabilidades específicas de versão.
- **Content-Security-Policy configurado:** Restringe de onde o navegador pode carregar scripts, imagens e estilos, defendendo contra ataques Cross-Site Scripting (XSS).
- **X-Frame-Options configurado para SAMEORIGIN:** Impede que seu site seja incorporado em um iframe externo, bloqueando ataques de clickjacking.
- **Strict-Transport-Security configurado:** Obriga navegadores a se comunicarem com seu servidor apenas via HTTPS, protegendo dados contra interceptação em redes não confiáveis.
