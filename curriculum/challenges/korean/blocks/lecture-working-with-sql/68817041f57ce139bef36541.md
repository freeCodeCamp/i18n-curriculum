# --description--

SQL 인젝션은 공격자가 웹 애플리케이션이 데이터베이스에 보내는 쿼리를 변경하거나 악용하는 웹 보안 취약점입니다. 예를 들어, 공격자는 쿼리에 악성 SQL 코드를 삽입하여 백엔드에서 악성 코드를 실행할 수 있습니다.

이는 사용자가 로그인 폼, 검색창, URL 매개변수를 통해 데이터를 입력할 수 있게 하고, 그 사용자 입력을 문자열 보간이나 연결에 사용해 최종 SQL 명령을 생성할 때 발생할 수 있습니다.

공격자의 목표에 따라, 악성 SQL 코드는 인증 우회, 데이터베이스에서 데이터 탈취, 데이터베이스 레코드 수정 또는 삭제 등 데이터베이스와 웹 애플리케이션에 심각한 장애를 일으킬 수 있습니다. 따라서 이 위협을 매우 심각하게 받아들여야 합니다.

예를 들어, 웹 애플리케이션이 사용자를 인증하기 위해 사용자 이름과 비밀번호를 직접 임베드하는 다음과 같은 SQL 쿼리를 사용한다고 가정해 보겠습니다:

```sql
'SELECT * FROM users WHERE username = ' + username_input + ' AND password = ' + password_input + ';'
```

공격자가 `username` 필드에 다음과 같은 값을 입력한다면:

```sql
'" " OR "1"="1" --'
```

그리고 `password` 필드에는 아무 값이나 입력해도, `"1"="1"`는 항상 `true`이고 `--`는 쿼리의 나머지 부분을 주석 처리하므로, `WHERE` 절은 다음과 같이 됩니다:

```sql
WHERE username = " " OR TRUE
```

이는 항상 `true`이 됩니다.

데이터베이스 구현에 따라, 이 쿼리는 데이터베이스 내 첫 번째 사용자 또는 임의의 사용자의 데이터를 반환하여 공격자가 유효한 자격 증명 없이 로그인할 수 있게 합니다.

그렇다면 이를 어떻게 방지할 수 있을까요? SQL 인젝션을 방지하는 주요 방법은 두 가지입니다:

- 문자열 연결로 동적 쿼리를 작성하지 마세요.
- 백엔드에서 실행될 쿼리에 악성 SQL 입력이 포함되지 않도록 하세요.

실제로는 매개변수화된 문장, 즉 매개변수화된 쿼리를 사용할 수 있습니다. 이 방법은 SQL 구조와 사용자가 입력한 데이터를 분리하여 데이터베이스가 입력 데이터를 실행 가능한 코드로 해석하지 못하게 합니다.

또 다른 중요한 기법은 입력 유효성 검사입니다. 이는 모든 경우에 보조 보안 수단으로 권장되지만, 특히 테이블 이름, 열 이름, 정렬 순서와 관련된 SQL 쿼리 부분에 중요합니다. 가능하다면 테이블 이름과 열 이름은 사용자 입력이 아닌 코드 내에서 직접 할당하는 것이 좋습니다.

모든 예방 조치에도 불구하고 SQL 인젝션 공격을 받았다면, 각 데이터베이스 계정의 권한을 제한하여 데이터베이스에 미치는 영향을 최소화할 수 있습니다. 각 데이터베이스 사용자에게 작업에 필요한 권한만 부여하고 그 이상은 부여하지 마세요. 계정이 데이터베이스에서 데이터를 읽기만 하면 된다면 읽기 권한만 부여하세요. 그리고 예방 차원에서 애플리케이션 계정에 관리자 권한을 절대 할당하지 마세요. 이는 매우 위험한데, 공격자가 데이터베이스에 완전한 접근과 제어 권한을 얻을 수 있기 때문입니다.

SQL 인젝션은 매우 흔한 공격으로, 개발자로서 반드시 인지하고 웹 애플리케이션 구현 시 방지해야 합니다. 이러한 기법들을 통해 데이터베이스를 보호하고 사용자의 데이터를 안전하게 지킬 수 있습니다.
