# --description--

Expressアプリで何か問題が起きたとき、どうなるのか気になったことはありますか？

データベース接続が失敗したら？存在しないルートにユーザーがアクセスしたら？

Expressのエラー処理の仕組みを理解することは、安全で信頼性の高いウェブアプリケーションを作るために非常に重要です。

このレッスンでは、Expressがエラーをどのように処理するか、そして実際のコード例を使ってエラーをキャッチしレスポンスする方法を解説します。

## Expressで簡単なエラーを処理する方法

Expressでは、エラー処理はミドルウェアをベースにしています。

通常のミドルウェア関数と同様に、エラー処理ミドルウェアは`req`、`res`、`next`にアクセスできます。ただし、先頭にもう1つパラメータが追加されます：`err`です。

構文は次のようになります：

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

このミドルウェアは、ミドルウェアスタックの最後に定義します。

Expressは4つのパラメータがあるため、これはエラーハンドラだと認識します。通常の`err`、`req`、`res`に加えて、新しい`next`があるからです。

今後のレッスンで、エラー処理ミドルウェアがExpressのミドルウェアスタック全体でどのように機能するかをさらに学びます。

## 非同期エラーの処理方法

非同期エラーの処理は、使用しているExpressのバージョンによって異なります。

どのバージョンのExpressを使っていても、すべてのルートの後、スタックの最後にエラー処理ミドルウェアを定義する必要があることに注意してください。

もう一度エラーハンドラを示します：

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Express 4

多くのプロジェクトでまだ使われているExpress 4およびそれ以前のバージョンでは、非同期エラーは**自動的にはキャッチされません**。

`async`/`await`を使い、`async`ルートハンドラ内で何かがスローされた場合、手動で`next()`に渡さない限りExpressはそれをキャッチしません。

Express 4で非同期エラーを正しく処理する方法は次の通りです：

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

`try/catch`ブロックと`next(err)`を忘れると、エラーは次のようになる可能性があります：

* アプリがクラッシュする
* または未処理のPromise拒否警告が出る

だからこそ、Express 4では非同期ロジックを`try`/`catch`でラップすることが必須です。

繰り返し`try/catch`を書くのを避けるために、ヘルパー関数を作る開発者もいます：

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

そしてこのように使います：

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

これにより、すべての`try/catch`ルートで`async`ブロックを使わなくても、エラーを適切に転送できます。

## Express 5

Expressの最新バージョンでは、この挙動が改善されています。

Express 5以降では、`async`ルートでエラーがスローされたり、拒否されたPromiseが返された場合、自動的にエラー処理ミドルウェアに転送されます。

つまり、この例は`try/catch`なしでエラーを処理できます：

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

`getUserFromDatabase()`がエラーをスローしても、Express 5はそれをキャッチして自動的にエラーハンドラに渡します。

手動で`next(err)`を呼び出す必要はありません。

## 404エラーの処理方法

404エラーは少し異なります。リクエストにマッチするルートがない場合に発生します。

次のミドルウェアで処理できます：

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**重要**：これはルートの後、エラー処理ミドルウェアの前に配置してください。

最後に、Expressでのエラー処理のベストプラクティスをいくつか紹介します：

* エラー処理ミドルウェアは必ず最後に置くこと。
* 404エラーは別に処理すること。
* Express 4以前では、ルートや`next(err)`ロジックからのエラーを`async`でバブルアップさせること。

本番環境では、スタックトレースをユーザーに送らないでください。詳細は内部でログに記録し、ユーザーにはわかりやすいメッセージを返しましょう。

Expressのエラー処理は、問題を検知して明確かつ安全に対応するミドルウェアを設定することです。コードのバグでもルートの欠落でも、アプリがバックアッププランなしにクラッシュすることはあってはなりません。
