# --description--

我們今天所知的數位世界是基於裝置能夠彼此通訊，並根據需要傳送和接收資料的概念。

但是你有沒有想過這個處理程序在幕後是如何運作的？

當你瀏覽網際網路時，你會不斷透過網路請求和傳送資料。這是因為有一套基於標準規則和程序的詳細處理程序，使資訊能夠正確地由適當的裝置發送和接收。

那組規則和程序稱為**協定**。

協定基本上允許兩個或多個裝置彼此通訊，方法是確保它們都遵循某些標準，例如將要交換的資料格式。

協定確保所有參與資料交換處理程序的裝置都能夠互相理解。比喻來說，他們都會知道如何「說同一種語言」。

TCP/IP 的核心是使網際網路成為可能的協定。

**TCP/IP** 是一組通訊協定，決定資料如何在網路中被封裝、傳送、路由和接收。它確保資訊能可靠地傳輸。

具體來說，**IP**，其中 IP 代表 **Internet Protocol**，是一種協定，確保資料從原始碼傳送到網路中正確的目的地。

為什麼這很重要？請記住，網際網路是一個由相互連接裝置組成的全域網路。為了讓你的裝置發送請求，首先它必須知道要將那些資料送到哪裡。這就是 IP 位址的主要用途，也是網際網路協定（IP）的基礎。它們能唯一識別網路上的裝置，讓原始裝置知道要將資料送到哪裡。

網際網路協定本身是無連線的，這表示每個資料單位（稱為「封包」）會獨立地透過網路傳送到目的地，而不會事先建立連線或確保它會正確或依序送達。如果封包確實抵達目的地，原始端完全不會收到通知。

這就是像 TCP 這類通訊協定變得非常重要的地方。

**TCP** 協定，稱為 **Transmission Control Protocol**，將資料拆分成小段。這些基本的資料單位會被編號，並在到達目的地時使用特定的演算法重新組合，這些演算法會檢測資料是否完整且正確。這些演算法也會檢查錯誤和遺失的封包，因此如果需要，可以要求原始碼重新傳送，確保資訊正確接收。

傳輸控制協定和網際網路協定共同制定了確保資料能夠可靠地透過網際網路傳輸所需的規則和標準。

除了 TCP/IP，現代網際網路還有另一個非常重要的協定：HTTP。

HTTP 在應用層次運作。

**HTTP** 代表 **HyperText Transfer Protocol**。它定義了客戶端（例如你的瀏覽器）與伺服器之間通訊的規則和標準。

當你嘗試在瀏覽器中存取網站時，瀏覽器會向伺服器發送 HTTP 請求，要求該特定資源及其相關資產。伺服器接收 HTTP 請求，執行其邏輯，並傳回包含所請求內容或說明請求失敗原因的 HTTP 回應給客戶端。

HTTP 請求和 HTTP 回應具有非常相似的結構，包括：

* 請求行（起始行）
    
* 標題
    
* 主體（選用）

## **HTTP 請求**

HTTP 請求的**請求列**分為三個主要零件：

* 一種描述所請求動作的方法。 例如，`GET` 用於擷取資料，`POST` 用於提交資料。其他常見的方法包括 `PUT`、`PATCH`、`DELETE` 和 `HEAD`。
    
* 描述客戶端所請求之特定資源的路徑或 URI。
    
* 用於通訊處理程序的 HTTP 通訊協定版本。例如，`HTTP/1.1`。

這是一個使用 `GET` 方法於根目錄 `/` 路徑及 HTTP 版本 1.1 的範例：

`GET / HTTP/1.1`

**header** 包含關於請求的鍵值對組資訊。

例如：`Content-Type: application/json`。

標頭下方有一個空行，顯示詮釋資料（關於請求的額外資訊）已完成。

最後是**body**。這是選用的。當客戶端必須將資訊傳送到伺服器時會送出，例如使用 `POST` 方法提交新資料時。

這是一個 HTTP 請求的範例：

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

## **HTTP 回應**

它們有非常相似的結構：

* 狀態列
    
* 標題
    
* 主體
    

首先，**狀態列**包含有關請求結果的資訊。你通常會在這裡找到：

* 用於通訊處理程序的 HTTP 版本。例如，`HTTP/1.1`。
    
* 狀態碼，顯示請求的結果。
    
* 原因短語，是狀態碼的簡短可讀描述。這有助於理解發生了什麼。
    

常見的狀態碼包括：

* `200 OK`
    
* `201 CREATED`
    
* `400 BAD REQUEST`
    
* `404 NOT FOUND`
    
* `403 FORBIDDEN`
    
* `500 INTERNAL SERVER ERROR`
    

有許多狀態碼。它們依據從左到右的第一個數字進行分類。

* `1xx` 表示資訊性回應。
    
* `2xx` 表示成功的回應。
    
* `3xx` 用於重新導向訊息。
    
* `4xx` 表示客戶端錯誤回應。
    
* `5xx` 代表伺服器錯誤回應。
    

在 HTTP 回應的狀態列下方，我們會找到包含回應詮釋資料的**標頭**。例如，`Content-Type: text/html`。這也可能包含內容長度、日期、位置，以及客戶端可能覺得有用的其他資訊。

最後，我們會找到 HTTP 回應的 **body**，其中包含客戶端所請求的資料，或是在請求未成功時對發生狀況的說明。這是選用的，但大多數回應中都會包含。

這是一個 HTTP 回應的範例：

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

既然你已經了解了 HTTP，你可能會想知道更多關於 HTTPS 的資訊。

你可以將 HTTPS 視為 HTTP 的「安全」版本，其中瀏覽器與伺服器之間傳送的資料會為了安全目的而加密。大多數現代網路流量都使用 HTTPS。

像 TCP/IP 和 HTTP 這樣的通訊協定使得全球資訊網和整個網際網路成為我們大多數人每天都依賴的重要工具。多虧了它們，裝置才能在網路上正確通訊並在全球交換資料。
