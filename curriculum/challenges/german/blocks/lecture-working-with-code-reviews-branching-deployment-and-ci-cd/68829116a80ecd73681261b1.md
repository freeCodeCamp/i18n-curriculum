# --description--

Wir haben bereits kurz darüber gesprochen, wie man Branches verwendet, um isolierte Änderungen vorzunehmen und so zu vermeiden, dass Ihr `main` Branch verschmutzt wird. Aber was bedeutet das genau, und warum ist das wichtig?

Wenn Sie Änderungen direkt in Ihrem `main` Branch committen und das Upstream-Repository andere Änderungen akzeptiert, die nicht von Ihnen stammen, endet das in einer sogenannten „divergenten Historie“. Das bedeutet, dass die Commits in Ihrem `main` Branch nicht mehr mit den Commits im Upstream-`main` Branch übereinstimmen. Das macht das Synchronisieren der Upstream-Änderungen zur Qual.

Angenommen, Sie haben gute Branching-Praktiken befolgt, und dieses Problem tritt nicht auf. Wenn Sie unseren vorherigen Lektionen gefolgt sind, sollten Sie eine Fork eines Repositories haben, die Sie lokal geklont haben, und Sie sollten ein Upstream-Remote haben, das auf das Original-Repository zeigt.

Falls Sie das nicht gemacht haben, hier eine kurze Erinnerung: Sie sollten ein Repository forken, Ihre Fork lokal klonen und diesen Befehl ausführen:

```sh
git remote add upstream <url>
```

Ersetzen Sie `<url>` durch die SSH-URL des Original-Repositories, NICHT Ihrer Fork.

Mit einem richtig konfigurierten Upstream können Sie die folgende Befehlsfolge ausführen, um die neuesten Upstream-Änderungen zu holen:

```sh
git checkout main
git fetch upstream
git merge upstream/main
```

Was bewirken diese Befehle? Nun, Sie haben bereits etwas über `git checkout` gelernt, daher stellt der erste Befehl sicher, dass Sie sich lokal auf Ihrem `main` Branch befinden.

Der zweite Befehl, `git fetch upstream`, sagt Git, dass es die neuesten Änderungen vom Upstream-Remote holen soll (also vom Original-Repo). Im Gegensatz zu einem Pull aktualisiert Git jedoch nicht automatisch Ihren Branch. Das ist wichtig, denn Ihr lokaler `main` Branch ist mit dem `main` Branch Ihrer Fork verbunden – nicht mit dem `main` Branch des Original-Repos! Ein git pull kann unerwartetes Verhalten verursachen.

Der letzte Befehl, `git merge upstream/main`, sagt Git, die neuesten Änderungen vom `main` Branch im Upstream-Remote in Ihren aktuellen Branch zu mergen – das ist Ihr lokaler `main` Branch, weil Sie diesen ausgecheckt haben.

Aber was, wenn der Merge fehlschlägt? Was, wenn Sie versehentlich in `main` committen und Git die divergenten Historien nicht zusammenführen kann? Nun... vorausgesetzt, Sie haben keine Pull Requests von Ihrem `main` Branch gemacht, können Sie ein Hard Reset durchführen, um Git zu zwingen, Ihren Branch zu aktualisieren.

```sh
git reset --hard upstream/main
```

Der Befehl `git reset` erlaubt es Ihnen, den aktuellen Zustand eines Branches zurückzusetzen. Das Flag `--hard` sagt Git, die lokalen Dateien zwangsweise an den Branch-Zustand anzupassen. So stellen Sie sicher, dass Sie mit einer sauberen Basis arbeiten. Und das Flag `upstream/main` sagt Git, dass Ihr aktueller Branch (also `main`) mit dem `main` Branch des Upstream-Remotes übereinstimmen soll. Seien Sie vorsichtig mit diesem Befehl – wann immer Sie eine Git-Historie zwangsweise aktualisieren, können merkwürdige und unerwünschte Dinge passieren.

Sobald Ihr lokaler `main` Branch korrekt aktualisiert ist, ist es wichtig, diesen Zustand zu Ihrem Remote-Fork hochzuschieben (der, wenn Sie unseren Lektionen gefolgt sind, `origin` heißen sollte). Dafür können Sie diesen Befehl ausführen:

```sh
git push
```

Das pusht den aktuellen Zustand zu Ihrem Remote-Fork. Wenn Sie jedoch ein Hard Reset gemacht haben, um die Historie zu bereinigen, kann es sein, dass Ihr Remote-Fork ebenfalls eine divergente Historie hat und das Push fehlschlägt.

Das können Sie beheben, indem Sie das Pushen erzwingen:

```sh
git push --force
```

**Seien Sie vorsichtig mit diesem Befehl**. Ein Force Push schreibt Ihre Remote-Historie effektiv neu, und wenn Sie etwas falsch gemacht haben, wird das Entwirren zu einem Albtraum.

Das Hochladen der synchronisierten Änderungen zu Ihrem Remote-Fork ist wichtig, weil Sie so sicherstellen, dass Sie die Fork mit den neuesten Änderungen sicher klonen können, falls Sie Ihr lokales Repository verlieren.

Und sobald Ihr `main` Branch synchronisiert ist, können Sie frei einen neuen Branch von diesem Zustand aus erstellen, um an einem weiteren Beitrag zu arbeiten – mit der neuesten Version des Original-Codebases!
