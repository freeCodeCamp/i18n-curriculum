# --description--

Hai imparato a creare commit e a inviarli al tuo repository che controlli. Puoi fare lo stesso per altri progetti che non mantieni, ma ci sono alcuni passaggi in più.

Prima di tutto, generalmente non avrai il "permesso di scrittura" su un progetto di qualcun altro. Questo significa che non puoi inviare commit direttamente a quel progetto. Invece, dovrai creare una copia del repository da cui lavorare. Questo processo si chiama "fork".

Per fare il fork di un repository, visita il repository a cui vuoi contribuire su GitHub. In alto, vedrai un pulsante "Fork". Cliccalo per accedere all'interfaccia per creare il tuo fork.

Vedrai una schermata simile a quella in cui hai creato il tuo repository, ma con meno opzioni. Puoi scegliere un proprietario, che può essere il tuo account personale o un'organizzazione a cui hai accesso, un nuovo nome per il fork e una nuova descrizione.

Hai anche l'opzione di copiare solo il ramo `main` (o quello predefinito configurato per il repository). Di solito è una buona idea, a meno che tu non debba contribuire a un ramo funzionalità specifico. Imparerai di più sui rami in una lezione futura. Per ora, lascia quella casella selezionata.

Clicca su "Create fork" e il tuo nuovo repository verrà creato, e verrai automaticamente portato lì.

Poi, clonalo sul tuo computer locale proprio come faresti con il tuo repository, usando SSH o l'interfaccia a riga di comando di GitHub.

Ora, diamo un’occhiata ai remote configurati nella tua copia locale. Apri un terminale nella directory del repository locale ed esegui `git remote -v` per ottenere una lista completa, dovrebbe apparire qualcosa del genere:

```sh
origin git@github.com:your-username/your-repo-name.git (fetch)
origin git@github.com:your-username/your-repo-name.git (pull)
```

Clonando un repository da GitHub, Git ha configurato automaticamente il tuo fork come remote `origin`. Tuttavia, quando lavori da un fork, spesso dovrai aggiornare la tua copia per portare dentro le modifiche dal repository originale.

Per farlo, aggiungiamo il repository originale come un altro remote. Torna alla pagina da cui hai fatto il fork, clicca sul pulsante "Code" e prendi l’URL SSH. Poi esegui il comando per creare un nuovo remote upstream:

```sh
git remote add upstream git@github.com:original-repo-username/repo-name.git
```

`upstream` è il nome del remote — puoi chiamarlo in altro modo, ma `upstream` tende a essere la convenzione. Se guardi di nuovo i tuoi remote, vedrai il nuovo remote upstream:

```sh
origin git@github.com:your-username/your-repo-name.git (fetch)
origin git@github.com:your-username/your-repo-name.git (pull)
upstream git@github.com:original-repo-username/your-repo-name.git (fetch)
upstream git@github.com:original-repo-username/your-repo-name.git (pull)
```

Con il remote `upstream` potrai fare cose come `git fetch upstream` e `git merge upstream/main` per ottenere le ultime modifiche dal progetto originale nel tuo fork locale.

Un fork è comunque un repository, e crei commit e invii modifiche allo stesso modo. Tuttavia, prima di iniziare, dovresti capire che ci sono alcune strategie da seguire quando modifichi il progetto di qualcun altro. Imparerai di più su queste strategie nelle prossime lezioni.

I progetti che rendono il loro codice sorgente disponibile al pubblico, permettono di modificarlo e accettano contributi di codice si chiamano progetti "open source". Ci sono molti vantaggi nell’open source, come permettere alla comunità di aiutarti a identificare e correggere bug, imparare dai contributi che le persone fanno al tuo lavoro e creare una comunità di sviluppatori che può diventare una rete di contatti.

Quando modifichi il progetto di qualcun altro, sei considerato un "contributore open source". Contribuire a progetti open source può essere un’ottima anteprima di cosa significa lavorare in un team di sviluppatori professionisti. E alcune delle strategie qui ti aiuteranno proprio in questo, e ti saranno utili nel tuo primo ruolo retribuito.

E se hai domande su come contribuire, la maggior parte dei progetti open source ha linee guida per i contributi che contengono una panoramica di tutto ciò che devi sapere. Ma se ancora non sei sicuro, non fa mai male aprire un problema nel repository per fare domande — oppure, se hanno uno spazio comunitario su una piattaforma come Discord, entra e chiedi lì.
