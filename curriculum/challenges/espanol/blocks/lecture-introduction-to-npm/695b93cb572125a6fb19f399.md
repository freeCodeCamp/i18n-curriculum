# --description--

npm necesita una forma de almacenar detalles importantes sobre tu proyecto, como qué paquetes usa, la versión de esos paquetes y qué comandos puede ejecutar. Esto se hace con el archivo `package.json`.

Piensa en el archivo `package.json` como un archivo de configuración npm para el proyecto. Es un archivo JSON simple que describe tu proyecto usando pares clave-valor para llevar un registro de cosas como:

* el nombre y la versión del proyecto
    
* el autor
    
* el archivo principal que debe ejecutarse primero
    
* la licencia
    
* el repositorio de código, palabras clave y descripciones
    
* cualquier dependencia que el proyecto necesite
    

Así es como podría verse un archivo `package.json`:

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "description": "A simple NPM project",
  "main": "index.js",
  "scripts": {
    "start": "node index.js",
    "test": "echo \"Running tests...\""
  },
  "repository": {
    "type": "git",
    "url": "git+https://github.com/username/project.git"
  },
  "keywords": ["fCC", "js"],
  "author": "Kolade Chris",
  "license": "ISC",
  "dependencies": {
    "react": "^19.2.1"
  },
  "devDependencies": {
    "cypress": "^15.7.1"
  }
}
```

El campo `main` a menudo se establece en `index.js` y le indica a Node.js qué archivo ejecutar primero.

El campo `license` se usa para especificar los términos legales bajo los cuales otras personas pueden usar, modificar y distribuir el código en tu paquete. Hay muchas licencias diferentes como `MIT`, `ISC` o `GPL`, que tienen distintos términos y permisos. La predeterminada es `ISC`, la Licencia del Internet Systems Consortium, que es una licencia de código abierto muy permisiva.

A medida que tu proyecto crece y instalas paquetes, npm los añade bajo una de dos secciones:

* `dependencies`, que son paquetes que tu app necesita para ejecutarse en producción
    
* `devDependencies`, que son paquetes que solo necesitas durante el desarrollo
    

Por ejemplo, si tu proyecto usa React para su UI, eso va en `dependencies` porque es necesario para que tu proyecto funcione una vez desplegado. Una herramienta como Cypress, que solo se usa para pruebas end-to-end antes de desplegar a producción, pertenece en `devDependencies`.

Hay un par de formas fáciles de crear un archivo `package.json`. La forma más común es ejecutar el siguiente comando:

```bash
npm init
```

Debes ejecutar esto en la carpeta raíz de tu proyecto. Este comando inicia un proceso de configuración interactivo donde npm te hace una serie de preguntas. Tus respuestas se usan para completar campos como el nombre del proyecto, versión, descripción, scripts, palabras clave, autor y licencia.

Si prefieres saltarte las preguntas y crear un archivo `package.json` con valores predeterminados, puedes ejecutar:

```bash
npm init -y
```

o

```bash
npm init --yes
```

Esto crea instantáneamente un archivo `package.json` para ti con valores predeterminados en muchos campos.

Por lo general, es mejor usar `npm init` sin la bandera `-y` o `--yes` para que puedas revisar cada campo y personalizar la información de tu proyecto. Si no respondes una pregunta, npm simplemente usa un valor predeterminado. Por ejemplo, si dejas vacía la pregunta de la licencia, npm la establecerá en `ISC` por defecto.

Puede que notes que `dependencies` y `devDependencies` aún no están listados. Eso es porque estas secciones solo se agregan después de que instalas tu primer paquete.

También notarás que falta el campo `repository`. Aunque npm preguntó por él durante `npm init`, no lo completa automáticamente después. Por ejemplo, cuando ejecutas comandos como `git remote add origin` o `git push`, npm no actualizará el campo repository por ti; tienes que agregarlo manualmente si quieres que se incluya.

En las próximas lecciones, aprenderás más sobre las dependencias en `package.json` y cómo instalarlas, actualizarlas y eliminarlas.
