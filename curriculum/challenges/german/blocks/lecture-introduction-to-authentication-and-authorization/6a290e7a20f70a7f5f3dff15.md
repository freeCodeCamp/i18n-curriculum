# --description--

In den letzten zwei Vorlesungen haben Sie gelernt, wie Sie die Benutzeridentität überprüfen und deren Sitzung mithilfe von Cookies aufrechterhalten. Das automatische Senden von Cookies mit jeder Anfrage führt jedoch zu einer einzigartigen Sicherheitslücke namens Cross-Site Request Forgery, CSRF.

CSRF ist ein Angriff, der einen authentifizierten Benutzer dazu zwingt, unerwünschte Aktionen auf einer Web-Anwendung auszuführen, bei der er gerade angemeldet ist. Bei einem CSRF-Angriff täuscht eine bösartige Website den Browser des Opfers, eine betrügerische Anfrage an Ihren Backend-Server zu senden, indem sie das Vertrauen des Servers in das Benutzer-Cookie ausnutzt.

Um CSRF zu verstehen, betrachten Sie dieses Schritt-für-Schritt-Beispiel einer Banking-Anwendung:

1. **Die Einrichtung:** Sie melden sich bei Ihrer Bank-Website an, zum Beispiel `trusted-bank.com`. Die Bank überprüft Ihre Zugangsdaten und speichert ein Sitzungstoken in Ihrem Browser-Cookie.

2. **Die Falle:** Ohne sich bei Ihrer Bank abzumelden, öffnen Sie einen neuen Browser-Tab und besuchen eine bösartige Website, zum Beispiel `evil-hacker-site.com`.

3. **Der Auslöser:** Die bösartige Website enthält ein verstecktes Skript oder ein Formular, das automatisch eine Anfrage an das Bank-Backend sendet, etwa `POST trusted-bank.com/transfer?amount=1000&to=hacker`.

4. **Die Ausführung:** Da Sie noch bei Ihrer Bank angemeldet sind, fügt Ihr Browser automatisch Ihr Bank-Sitzungscookie zu dieser Anfrage hinzu. Der Bank-Server sieht ein gültiges Cookie, geht davon aus, dass Sie die Anfrage absichtlich gestellt haben, und überweist Ihr Geld.

Die entscheidende Erkenntnis ist, dass die bösartige Website Ihr Cookie nicht lesen oder Ihre Daten direkt stehlen kann. Stattdessen nutzt sie einfach aus, dass Ihr Browser Cookies automatisch an jede Anfrage an den Zielserver anhängt.

Es gibt zwei Hauptabwehrmaßnahmen: das **SameSite-Cookie-Attribut** und **CSRF-Tokens**.

Die einfachste Abwehr ist das Setzen des `SameSite`-Attributs auf Ihren Cookies. Dies teilt dem Browser mit, wann er das Cookie bei Cross-Site-Anfragen mitsenden darf.

- **SameSite=Strict:** Der Browser sendet das Cookie niemals, wenn die Anfrage von einer Drittanbieter-Website stammt.

- **SameSite=Lax:** Der Browser erlaubt Cookies nur, wenn ein Benutzer sicher zur Zielseite navigiert, etwa durch Klicken auf einen normalen Link, blockiert sie jedoch bei Cross-Site-Formularübermittlungen oder API-Anfragen.

Für einen robusten Schutz verwenden Backend-Frameworks CSRF-Tokens. Wenn ein Benutzer eine Seite lädt, generiert das Backend ein zufälliges, einzigartiges und unvorhersehbares Token und hängt es an das Frontend-Formular oder die Sitzung an.

Wenn der Client eine Anfrage absendet, muss er dieses Token in den benutzerdefinierten Anfrage-Headern oder Formulardaten mitsenden. Das Backend vergleicht das übermittelte Token mit dem in der Benutzersitzung gespeicherten Token. Da eine bösartige Website nicht auf Ihr Frontend zugreifen kann, um dieses geheime Token zu kopieren, fehlt jeder von ihr gefälschten Cross-Site-Anfrage dieses Token und die Validierung schlägt fehl.

Indem Sie diese Strategien implementieren, stellen Sie sicher, dass Anfragen, die bei Ihrem Backend ankommen, absichtliche Aktionen des tatsächlichen Benutzers sind und keine unautorisierten Aktionen, die von einem bösartigen Akteur erzwungen werden.
