## バージョン管理の紹介

- **定義**: バージョン管理システムは、プロジェクトの変更を追跡・管理するためのものです。ソフトウェアで使われるバージョン管理システムの例には、Git、SVN、Mercurialがあります。

## クラウドベースのバージョン管理プロバイダー

- **クラウドベースのバージョン管理プロバイダーの例**: GitHubやGitLabは、ソフトウェアチームが協力してリポジトリを管理できる人気のクラウドベースのバージョン管理プロバイダーです。

## Gitのインストールとセットアップ

- **Gitのインストール**: 端末で次のコマンドを実行して、Gitがすでにインストールされているか確認できます。

```sh
git --version
```

バージョン番号が表示されればGitがインストールされています。表示されなければ、インストールが必要です。

Linuxシステムでは、多くのディストリビューションにGitがプリインストールされていることが多いです。プリインストールされていない場合は、`sudo apt-get install git`や`sudo pacman -S git`などのパッケージマネージャーのコマンドでインストールできます。

Macユーザーは、Homebrewを使って`brew install git`でGitをインストールするか、Gitのウェブサイトから実行可能インストーラーをダウンロードできます。

Windowsでは、Gitのウェブサイトから実行可能インストーラーをダウンロードできます。また、Chocolateyを設定している場合は、PowerShellで`choco install git.install`を実行できます。Windowsでは、Unix風のシェル環境を使うためにGit Bashをダウンロードすることもお勧めします。

インストールが成功したか確認するには、端末で再度`git --version`コマンドを実行してください。

- **Gitの設定**: `git config`は、Gitがマシン上でどのように動作するかを決める設定変数を設定するために使います。現在の設定変数とそれがどこに保存されているかを確認するには、次のコマンドを実行してください。

```sh
git config --list --show-origin
```

Gitを初めてインストールした場合は、システムレベルの設定だけが表示されるはずです。

ユーザー名を設定するには、次のコマンドを実行します。

```sh
git config --global user.name "Jane Doe"
```

ここで`--global`フラグは、Gitを使うすべてのプロジェクトに対してユーザー名を設定するために使います。特定のプロジェクトだけでユーザー名をオーバーライドしたい場合は、そのプロジェクトのディレクトリで`--global`フラグなしでコマンドを実行してください。

ユーザーのメールアドレスを設定するには、次のコマンドを実行します。

```sh
git config --global user.email janedoe@example.com
```

Gitに使う好みのエディターを設定することもできます。例えば、好みのエディターをEmacsに設定する例は以下の通りです。

```sh
git config --global core.editor emacs
```

好みのエディターを設定しなければ、Gitはシステムのデフォルトエディターを使います。

## オープンソースとクローズドソースソフトウェア

- **定義**: 「オープンソース」とは、公開したコードを誰でも見られ、変更提案や問題報告、改変版の実行ができることを意味します。「クローズドソース」とは、明示的に許可した人だけがプロジェクトを見たり操作したりできることを意味します。

## GitHub

- **定義**: GitHubはクラウドベースのサービスで、バージョン管理されたプロジェクトを「リポジトリ」と呼ばれる場所に保存し、それらのプロジェクトで使う協力機能を提供します。
- **GitHub CLI**: コマンドラインからGitHub特有の作業を行うためのツールです。インストールしていなければ、GitHubのドキュメントでインストール方法を確認できますが、システムのパッケージマネージャーで利用可能なはずです。
- **GitHub Pages**: 静的サイトや、バックエンドサーバーを必要としないアプリケーションをデプロイするためのオプションです。つまり、完全にクライアント側やユーザーのブラウザで動作するアプリケーションをこのプラットフォームで公開できます。
- **GitHub Actions**: GitHubリポジトリ内でワークフローを自動化できる機能で、コードのビルド、テスト、デプロイなどを行えます。

## よく使うGitコマンド

- **`git init`**: 空のGitリポジトリを初期化し、Gitがこのプロジェクトの変更を追跡できるようにします。空のGitリポジトリを初期化すると、新しい`.git`という隠しディレクトリが追加されます。この`.git`ディレクトリには、Gitがプロジェクトを管理するための重要な情報が含まれています。
- **`git status`**: 作業ディレクトリの現在の状態を表示するコマンドで、ワークフローで頻繁に使います。
- **`git add`**: 変更をステージングするコマンドです。ステージングエリアにあるものは次のコミットに含まれます。すべての未ステージ変更をステージングしたい場合は、`git add .`を使えます。ピリオド（`.`）は現在のディレクトリのエイリアスです。
- **`git commit`**: 変更をコミットするコマンドです。コミットはその時点のプロジェクトのスナップショットです。`git commit`を実行すると、Git設定で指定した好みのエディターが開きます。エディターが開いたら、変更内容の詳細なメッセージを入力できます。短いメッセージを使いたい場合は、次のように`git commit -m`コマンドを使えます。

```sh
git commit -m "short message goes here"
```

- **`git log`**: これまでのすべてのコミットを、作者、コミット日時、メッセージ、コミットハッシュなどの情報とともに一覧表示します。コミットハッシュはコミットの一意識別子となる長い文字列です。
- **`git remote add`**: リモートリポジトリへの接続を設定するコマンドです。
- **`git push`**: 変更をリモートリポジトリにプッシュするコマンドです。
- **`git pull`**: リモートリポジトリから最新の変更をローカルリポジトリにプルするコマンドです。
- **`git clone`**: リポジトリをクローンするコマンドです。これにより、リポジトリの履歴、すべてのファイルやフォルダー、コミットを含むコピーがローカルに作成されます。
- **`git remote -v`**: ローカルGitリポジトリに関連付けられたリモートリポジトリの一覧を表示します。
- **`git branch`**: ローカルのすべてのブランチを一覧表示します。
- **`git fetch upstream`**: アップストリームリモート（元のリポジトリ）にある最新の変更を取得するようGitに指示します。
- **`git merge upstream/main`**: アップストリームリモートの`main`ブランチから最新の変更を現在のブランチにマージします。
- **`git reset`**: ブランチの現在の状態をリセットするコマンドです。`--hard`フラグを渡すと、ローカルファイルを強制的にブランチの状態に合わせます。これにより、作業をクリーンな状態から始められます。
- **`git rebase`**: Gitのリベースは、一連のコミットをあるブランチから別のブランチに移動または結合する方法です。

## ブランチの操作

- **定義**: Gitのブランチは、変更を加えられる別の作業スペースです。`main`ブランチは、実際のアプリケーションでは主にプライマリまたは本番用のブランチを表すことが多いです。開発チームは新機能やバグ修正のために複数のブランチを作成し、それらの変更を`main`ブランチにマージします。
- **新しいブランチの作成**: 新しいブランチを作成するには、次のコマンドを実行します。

```sh
git branch feature
```

そのブランチに切り替えるには、次のコマンドを実行します。

```sh
git checkout feature
```

多くの開発者は、ブランチ作成と切り替えを同時に行う短縮コマンドを使います。

```sh
git checkout -b new-branch-name
```

新しくて代替のコマンドとして`git switch`があります。新しいブランチを作成して切り替える例は以下の通りです。

```sh
git switch -c new-branch-name
```

- **ブランチ戦略**: `main`ブランチはデフォルトのブランチで、通常はかなり安定しています。バグ修正や新機能、その他の作業のために新しいブランチを作る際は、ここから分岐するのが最適です。
- **マージコンフリクト**: これはGitが異なるブランチの変更を自動でマージしようとしたときに、どの変更を残すべきか判断できない場合に起こります。通常、同じファイルの同じ部分に対して矛盾する変更があると発生します。

## Gitで追跡されるファイルの5つの状態

- **「Untracked」**: これはファイルがリポジトリに新しく、Gitがまだ「見て」いない状態を意味します。
- **「Modified」**: このファイルは前のコミットに存在し、まだコミットされていない変更があります。
- **「Ignored」**: Gitでは無視されたファイルは通常見えませんが、IDEには表示されることがあります。無視されたファイルはGitの操作から除外されており、通常は`.gitignore`ファイルに含まれています。
- **「Deleted」**: 削除されたファイルは、未追跡ファイルの逆で、以前存在していたファイルが削除された状態です。
- **「Renamed」**: 名前や場所が変更されたファイルで、中身は変わっていません。場合によっては、少しだけ変更があってもファイルはリネームされたとみなされることがあります。

## `.gitignore`ファイル

- **定義**: `.gitignore`ファイルはGit操作に関連する特別な種類のファイルです。名前からはGitに無視させるためのファイルと思われがちですが、実際にはGitにファイルの追跡をやめさせるために使われます。

## リポジトリの操作

- **定義**: リポジトリはプロジェクトのコンテナのようなもので、アプリのファイルをまとめて管理します。リポジトリはローカルのコンピューター上にあったり、GitHubのようなサービス上にリモートで存在したりします。
- **公開リポジトリと非公開リポジトリ**: 公開リポジトリは誰でも閲覧・ダウンロードできます。非公開リポジトリは自分と明示的にアクセス権を与えた人だけがアクセスできます。
- **GitHubでのリポジトリ作成**: GitHubで新しいリポジトリを作成するには、`"New Repository"`ボタンをクリックし、GitHubのUIに従って新しいリポジトリを設定してください。
- **ローカルリポジトリをGitHubにプッシュする**: コンピューター上のローカルプロジェクトをGitHubにプッシュする手順は以下の通りです。

1. プロジェクトディレクトリで空のgitリポジトリを初期化する（`git init`）。
2. プロジェクトに変更を加える。
3. `git status`コマンドを実行して、Gitが追跡しているすべての変更を確認する。
4. 変更をステージングする（`git add`）。
5. 変更をコミットする（`git commit`）。
6. リモート接続を設定する（`git remote add`）。
7. 変更をGitHubにプッシュする（`git push`）。

## プルリクエスト

- **プルリクエスト**: プルリクエストは、自分のブランチの変更をターゲットブランチに取り込むよう依頼するものです。コード変更をプロジェクトに貢献したいときに使う流れで、プロジェクトの管理者が変更をレビューできます。コメントを残したり質問したり修正を提案したりできます。レビューが完了すると承認され、メインブランチにマージされます。

## 他のリポジトリへの貢献

- **手順**: 貢献できるプロジェクトは数千あります。基本的な貢献の流れは以下の通りです。

1. 貢献に関するドキュメントを読む
2. 作業可能なIssueを見つける
3. リポジトリをフォークする
4. フォークしたリポジトリをクローンする
5. 新しいブランチを作成する
6. Issueに沿って変更を加える
7. PR（プルリクエスト）を作成する
8. PRのレビューを待つ

## SSHキーとGPGキーの操作

- **GPGキー**: GPG（Gnu Privacy Guard）キーは、ファイルやコミットに署名するために使われます。公開GPGキーを使うことで、ファイルの署名があなたのキーによるものであり、内容が改ざんされていないことを検証できます。

GPGキーを生成するには、次のコマンドを実行します。

```sh
gpg --full-generate-key
```

- **SSHキー**: SSH（Secure SHell）キーは、`ssh`ユーティリティを使ってサーバーへのリモート接続を認証するために使われます。SSHキーはコミットの署名にも使えます。

SSHキーを生成するには、次のコマンドを実行します。

```sh
ssh-keygen -t ed25519 -C "your_email@example.com"
```

`ed25519`は最新の公開鍵署名アルゴリズムです。

- **GPGキーでコミットに署名する**: コミットにGPGキーで署名するには、秘密鍵ではなく公開鍵をGitHubアカウントにアップロードする必要があります。公開鍵の一覧を表示するには、次のコマンドを実行します。

```sh
gpg --list-secret-keys --keyid-format=long
```

公開鍵を取得するには、次のコマンドを使います。

```sh
gpg --armor --export "<key id>"
```

一覧で得た短いIDを使って、Gitの署名キーとして設定するには次のコマンドを実行します。

```sh
git config --global user.signingkey <your_gpg_key_id>
```

その後、特定のコミットに署名するには`-S`フラグを`git commit`コマンドに渡します。パスフレーズの入力が必要です。すべてのコミットに自動で署名したい場合は、autosign設定を`true`にしてください。

```sh
git config --global commit.gpgsign true
```

- **SSHキーでコミットに署名する**: GitHubで比較的新しい機能であるSSHキー署名を使うには、まずキーをGitHubアカウントにアップロードします。次にGitの署名モードをSSHに設定します。

```sh
git config --global gpg.format ssh
```

署名キーを設定する際は、IDの代わりにファイルパスを渡します。

```sh
git config --global user.signingkey <path_to_your_ssh_keys>
```
