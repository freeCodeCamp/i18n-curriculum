## الخوادم ونموذج العميل-الخادم

- **الخادم**: جهاز كمبيوتر أو برنامج يستقبل الطلبات من أجهزة أخرى ويرسل البيانات أو الخدمات عبر الشبكة. يشير المصطلح إلى كل من الأجهزة المادية والبرمجيات التي تعمل عليها.
- **نموذج العميل-الخادم**: بنية موزعة حيث يرسل العملاء الطلبات ويتولى الخوادم معالجتها وإرجاع الاستجابات.
- **العميل**: العملية أو التطبيق (مثل متصفح الويب) الذي يبدأ الطلبات إلى الخادم.
- **دورة الطلب والاستجابة**: يرسل العميل طلبًا؛ يعالجه الخادم ويرد باستجابة.

### أنواع الخوادم

- **خوادم الويب**: توصل محتوى الويب (HTML، الصور، CSS) إلى المتصفحات.
- **خوادم التطبيقات**: تتولى منطق الأعمال وتعالج مُدخلات المستخدم.
- **خوادم قواعد البيانات**: تخزن البيانات وتسترجعها وتديرها.
- **خوادم البريد**: ترسل وتستقبل وتخزن البريد الإلكتروني.
- **خوادم الملفات**: تدير تخزين الملفات ومشاركتها عبر الشبكة.
- **خوادم البروكسي**: تعمل كوسطاء بين العملاء وخوادم أخرى.

## نظام أسماء النطاقات (DNS)

- **DNS**: نظام تسمية هرمي ولا مركزي يترجم أسماء النطاقات القابلة للقراءة البشرية إلى عناوين IP، ويُطلق عليه غالبًا "دليل الهاتف" للإنترنت.
- **عنوان IP**: معرف رقمي فريد لجهاز على الشبكة (مثل `127.0.0.1` في IPv4، و`0:0:0:0:0:0:0:1` في IPv6).
- **هيكلية أسماء النطاقات** (اقرأ من اليمين إلى اليسار):
  - **النطاق الأعلى (TLD)**: الجزء الأيمن (مثل `.org`، `.com`).
  - **النطاق الثاني (SLD)**: الاسم المسجل (مثل `freecodecamp` في `freecodecamp.org`).
  - **النطاق الفرعي**: يحدد قسمًا من النطاق الرئيسي (مثل `www`، `blog`، `api`).

### عملية حل DNS

1. يرسل المتصفح اسم النطاق إلى **المُحلل التكراري** (عادةً ما يديره مزود خدمة الإنترنت).
2. يستعلم المُحلل التكراري **خوادم الجذر** التي توجهه إلى خادم النطاق الأعلى المناسب.
3. يعيد **خادم النطاق الأعلى** عنوان **خادم الأسماء المخول** للنطاق.
4. يعيد خادم الأسماء المخول **سجل DNS** يحتوي على عنوان IP.
5. يسلم المُحلل التكراري عنوان IP إلى العميل ويُخزن السجل مؤقتًا لتسريع عمليات البحث المستقبلية.

## TCP/IP

- **البروتوكول**: مجموعة قواعد ومعايير تسمح للأجهزة بالتواصل عبر الشبكة.
- **TCP/IP**: مجموعة بروتوكولات تحدد كيفية تغليف البيانات وإرسالها وتوجيهها واستقبالها.
  - **IP (بروتوكول الإنترنت)**: يوجه حزم البيانات من المصدر إلى الوجهة باستخدام عناوين IP. هو بدون اتصال — كل حزمة تسافر بشكل مستقل دون ضمان التسليم.
  - **TCP (بروتوكول التحكم بالنقل)**: يقسم البيانات إلى أجزاء مرقمة، يعيد تجميعها في الوجهة، يتحقق من الأخطاء والحزم المفقودة، ويطلب إعادة الإرسال إذا لزم الأمر. يضمن التسليم الموثوق والمرتب.

## HTTP (بروتوكول نقل النص الفائق)

- **HTTP**: بروتوكول على مستوى التطبيق يحدد قواعد التواصل بين العملاء والخوادم على الويب.

- **HTTPS**: نسخة آمنة من HTTP حيث تُشفّر البيانات باستخدام SSL/TLS قبل الإرسال.

### هيكل طلب HTTP

- **سطر الطلب**: `METHOD /path HTTP/version` (مثل `GET / HTTP/1.1`)
- **الرؤوس**: أزواج بيانات وصفية من نوع المفتاح والقيمة (مثل `Content-Type: application/json`).
- **الجسم** (اختياري): البيانات المرسلة إلى الخادم، تُستخدم مع `POST`، `PUT`، و`PATCH`.

**طرق HTTP الشائعة:**

| الطريقة | الغرض |
| --- | --- |
| `GET` | استرجاع البيانات |
| `POST` | إرسال بيانات جديدة |
| `PUT` | استبدال مورد |
| `PATCH` | تحديث جزئي لمورد |
| `DELETE` | حذف مورد |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### هيكل استجابة HTTP

- **سطر الحالة**: إصدار HTTP، رمز الحالة، وعبارة السبب (مثل `HTTP/1.1 200 OK`).
- **الرؤوس**: بيانات وصفية عن الاستجابة (مثل `Content-Type: text/html`).
- **الجسم** (اختياري): المحتوى المُعاد أو شرح الخطأ.

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## رموز حالة HTTP

تُصنف رموز الحالة حسب الرقم الأول:

- **1xx** — استجابات معلوماتية.
- **2xx** — استجابات ناجحة.
- **3xx** — رسائل إعادة توجيه.
- **4xx** — أخطاء العميل.
- **5xx** — أخطاء الخادم.

### الرموز الشائعة

| الرمز | الاسم | المعنى |
| --- | --- | --- |
| `200` | OK | نجح الطلب؛ تم إرجاع المحتوى. |
| `201` | Created | تم إنشاء المورد بنجاح. |
| `204` | No Content | نجح الطلب لكن لا يوجد محتوى للإرجاع. |
| `301` | Moved Permanently | تم نقل المورد نهائيًا إلى عنوان جديد. |
| `302` | Found | تم نقل المورد مؤقتًا إلى عنوان آخر. |
| `304` | Not Modified | لم يتغير المورد منذ آخر طلب؛ استخدم النسخة المخزنة.
| `400` | Bad Request | لم يستطع الخادم فهم الطلب (صيغة أو بيانات غير صحيحة). |
| `401` | Unauthorized | مطلوب بيانات توثيق.
| `403` | Forbidden | لا يملك العميل إذن الوصول للمورد حتى مع بيانات توثيق صحيحة.
| `404` | Not Found | المورد غير موجود على الخادم.
| `500` | Internal Server Error | خطأ غير متوقع في الخادم.
| `502` | Bad Gateway | تلقى الخادم استجابة غير صالحة من خادم أعلى.
| `503` | Service Unavailable | الخادم غير قادر مؤقتًا على معالجة الطلبات.

## نموذج طلب واستجابة HTTP

تتكون دورة الطلب والاستجابة من هذه المكونات التي تعمل معًا:

- **العميل**: متصفح، تطبيق جوال، أو عميل API يرسل طلبات HTTP.
- **طريقة HTTP**: الإجراء المراد تنفيذه (`GET`، `POST`، `PUT`، `PATCH`، `DELETE`).
- **URL**: عنوان المورد المطلوب.
- **الرؤوس**: بيانات وصفية للطلب أو الاستجابة (مثل `Authorization`، `Content-Type`).
- **الجسم**: البيانات المرسلة مع طلبات `POST`، `PUT`، `PATCH`، أو `DELETE` (غالبًا JSON أو بيانات نموذج).
- **الخادم**: النظام الخلفي الذي يعالج الطلبات ويرد بالاستجابات.
- **رمز الحالة**: رمز رقمي يشير إلى النجاح أو الخطأ أو إعادة التوجيه.
- **جسم الاستجابة**: البيانات الفعلية المعادة، عادة HTML، JSON، أو XML.

### الخطوات الست للدورة

1. **يبدأ العميل طلبًا** بطريقة، URL، رؤوس، وجسم اختياري.
2. **يسافر الطلب إلى الخادم** عبر الإنترنت.
3. **يعالج الخادم الطلب**: يتحقق من الطريقة، URL، التوثيق، ويستعلم قاعدة بيانات إذا لزم الأمر.
4. **يُعد الخادم استجابة** برمز حالة، رؤوس، وجسم اختياري.
5. **تُرسل الاستجابة إلى العميل**.
6. **يتعامل العميل مع الاستجابة**: يعرض البيانات عند النجاح (2xx) أو يظهر رسالة خطأ عند الفشل (4xx/5xx).

## أصول استجابة HTTP وأنواع المحتوى

يخبر رأس `Content-Type` المتصفح بكيفية التعامل مع الأصل في الاستجابة.

| الأصل | نوع المحتوى |
| --- | --- |
| HTML | `text/html` |
| CSS | `text/css` |
| JavaScript | `application/javascript` |
| JSON | `application/json` |
| XML | `application/xml` |
| JPEG | `image/jpeg` |
| PNG | `image/png` |
| GIF | `image/gif` |
| SVG | `image/svg+xml` |
| WebP | `image/webp` |
| MP4 | `video/mp4` |
| MP3 | `audio/mpeg` |
| PDF | `application/pdf` |
| ZIP | `application/zip` |
| WOFF2 | `font/woff2` |

## إرسال نماذج HTML

- تجمع النماذج بيانات المستخدم وترسلها باستخدام عنصر `<form>` مع عناصر `<input>`.
- يحتاج كل مُدخل إلى خاصية `name` ليتمكن الخادم من التعرف على البيانات.
- **التحقق من جانب العميل**: استخدم خاصيتي `required` و`pattern` لتقديم ردود فورية.
- **التحقق من جانب الخادم**: تحقق دائمًا على الخادم لأسباب أمنية، بغض النظر عن التحقق على العميل.
- تحدد خاصية `action` مكان إرسال بيانات النموذج.
- تحدد خاصية `method` كيفية الإرسال (`GET` أو `POST`؛ تتطلب `PUT`، `PATCH`، `DELETE` حلولًا بديلة مثل Fetch API أو `method-override` في Express).

### سير إرسال النموذج

1. يملأ المستخدم النموذج ويرسله.
2. يتحقق المتصفح من صحة البيانات على جهة العميل؛ ويستمر إذا كانت صحيحة.
3. يرسل المتصفح بيانات النموذج إلى عنوان URL الموجود في `action` باستخدام `method` المحددة.
4. يستقبل الخادم البيانات ويعالجها (تخزينها في قاعدة بيانات، التحقق من هوية المستخدم، إلخ).
5. قد يجري الخادم تحققًا إضافيًا على جهة الخادم.
6. يرد الخادم برسالة نجاح، أو خطأ، أو إعادة توجيه.
7. يرى المستخدم رد الفعل بناءً على استجابة الخادم.

## نموذج معايير الويب والهيئات المعنية

- **معايير الويب**: القواعد والمواصفات التي تتبعها المتصفحات لعرض الصفحات وتمكين قدرات الويب.
- **WHATWG**: الهيئة التي تحافظ على HTML وDOM كـ"معيار حي" محدث باستمرار تديره فرق محركات المتصفحات الكبرى.
- **W3C**: المنظمة الرئيسية للمعايير، تشرف على CSS، SVG، إرشادات إمكانية الوصول (WCAG)، والعديد من واجهات برمجة التطبيقات للويب.
- **ECMA / TC39**: تنشر ECMA مواصفة ECMAScript (معيار JavaScript)؛ يدفع TC39 ميزات اللغة عبر عملية اقتراح من خمس مراحل (المرحلة 0–4).
- **مجموعة Khronos**: مسؤولة عن WebGL وWebGL 2، التي توفر عرض ثلاثي الأبعاد معزز بوحدة معالجة الرسوميات في المتصفح.

## عملية إنشاء معايير الويب

1. **تحديد الحاجة**: يوثق مطوِّر أو مهندس متصفح مشكلة حقيقية وشائعة.
2. **الاقتراح والنقاش المبكر**: تُكتب الفكرة بشكل غير رسمي لجمع الملاحظات وقياس اهتمام بائعي المتصفحات.
3. **كتابة المواصفة**: يُصاغ مستند دقيق ورسمي يغطي كل الحالات الخاصة، وأخطاء الأمان، والاعتبارات الأمنية.
4. **المراجعة والاعتراض والتكرار**: تُراجع المسودة علنًا؛ يطرح البائعون اعتراضاتهم وتُعدل المواصفة حتى يتم التوافق.
5. **التنفيذ والاختبار**: ينفذ بائعو المتصفحات الميزة خلف علم؛ تتحقق مجموعة اختبارات مشتركة من تطابق جميع التنفيذات.
6. **الإنهاء**: بمجرد اجتياز عدة تنفيذات مستقلة للاختبارات، تُنهى المواصفة (توصية W3C، المرحلة 4 لـ TC39، أو تحديث معيار حي لـ WHATWG).

## دورة حياة ميزات معايير الويب

1. **الاقتراح**: يحدد شخص ما مشكلة ويكتبها لمعرفة ما إذا كان الآخرون يشاركونه الرأي.
2. **الحضانة**: تُختبر الفكرة من خلال نقاش مفتوح؛ تموت العديد من الاقتراحات هنا إذا كانت متخصصة جدًا أو لم ترضِ الجميع.
3. **المواصفة**: تُكتب مواصفة رسمية تحول المفاهيم الغامضة إلى قواعد دقيقة وقابلة للتنفيذ.
4. **اختبار التوافق**: تتحقق مجموعة اختبارات منصة الويب لـ W3C من أن كل متصفح ينفذ الميزة بشكل متطابق.
5. **الإنهاء**: تعني عدة تنفيذات ناجحة أن الميزة تُطرح للمستخدمين كجزء من منصة الويب الأساسية.
6. **الإهمال**: تُعلَم الميزات المتجاوزة أو الإشكالية بوقف الاستخدام؛ إزالة الدعم من المتصفحات بطيئة بسبب التزامات التوافق العكسي.

## المبادئ الأساسية لمعايير الويب

- **الانفتاح**: تُطوَّر المعايير علنًا وتُنفذ بدون رسوم ترخيص من أي جهة.
- **إمكانية الوصول**: صُمم الويب ليكون قابلًا للاستخدام من الجميع، مع إمكانية الوصول مدمجة في HTML، CSS، وسلوك المتصفح.
- **التوافقية**: يجب أن تتصرف الصفحة المبنية وفق المعيار بشكل متطابق عبر جميع المتصفحات وأنظمة التشغيل.
- **التوافق العكسي**: تُضاف القدرات الجديدة فوق القديمة حتى تستمر الصفحات القديمة في العمل على المتصفحات الحديثة.
- **الخصوصية والأمان**: تواجه واجهات برمجة التطبيقات الجديدة التي قد تكشف بيانات المستخدم تدقيقًا كبيرًا؛ تحمي المنصة المستخدمين افتراضيًا.
- **التدرج الطبقي**: يتولى HTML الهيكل، وCSS العرض، وJavaScript السلوك — كل طبقة تعمل بشكل مستقل.
- **التحسين التدريجي**: يجب أن تعمل الميزات للجميع أولًا، ثم تتحسن للمستخدمين ذوي المتصفحات أو الأجهزة الأكثر قدرة.
