# --interactive--

Модифікатори, які часто називають "прапорцями", змінюють поведінку регулярного виразу. Згадаймо наш приклад з попереднього уроку:

:::interactive_editor

```js
const regex = /freeCodeCamp/;
console.log(regex.test("freeCodeCamp")); // true
console.log(regex.test("freeCodeCamp is great")); // true
console.log(regex.test("I love freeCodeCamp")); // true
console.log(regex.test("freecodecamp")); // false
console.log(regex.test("FREECODECAMP")); // false
console.log(regex.test("free")); // false
console.log(regex.test("code")); // false
console.log(regex.test("camp")); // false
```

:::

Якщо пам’ятаєте, рядки `freeCodeCamp` у нижньому та верхньому регістрах не відповідали шаблону. Це тому, що за замовчуванням регулярні вирази чутливі до регістру.

Але що, якби ми могли сказати регулярному виразу ігнорувати регістр? Для цього є модифікатор. Прапорець `i` змушує регулярний вираз ігнорувати регістр. Як його використати? Прапорці ставляться після закриваючого слеша в регулярному виразі:

```js
const regex = /freeCodeCamp/i;
```

Зверніть увагу на зміну в регулярному виразі в першому рядку. Тепер перевіримо, як це змінює результати:

:::interactive_editor

```js
const regex = /freeCodeCamp/i;

console.log(regex.test("freeCodeCamp")); // true
console.log(regex.test("freeCodeCamp is great")); // true
console.log(regex.test("I love freeCodeCamp")); // true
console.log(regex.test("freecodecamp")); // true
console.log(regex.test("FREECODECAMP")); // true
console.log(regex.test("free")); // false
console.log(regex.test("code")); // false
console.log(regex.test("camp")); // false
```

:::

Оскільки наш регулярний вираз тепер нечутливий до регістру, рядки у нижньому та верхньому регістрах "пройшли" тест. Це також працює для рядка з випадковим поєднанням верхнього та нижнього регістрів:

:::interactive_editor

```js
const regex = /freeCodeCamp/i;

console.log(regex.test("dO yOu LoVe fReEcOdEcAmP?")); // true
```

:::

Існує ще кілька прапорців, які ви можете використовувати. Прапорець `g`, або глобальний модифікатор, дозволяє регулярному виразу знаходити шаблон більше одного разу.

Давайте подивимось, як це впливає на наш код. Ви помітите, що ми залишили прапорець `i` — регулярний вираз може використовувати кілька прапорців (стільки, скільки потрібно) для досягнення бажаної поведінки:

```js
const regex = /freeCodeCamp/gi;
```

Почекайте... що це? Здається, що глобальний модифікатор змушує деякі рядки, які мали проходити, тепер не проходити:

:::interactive_editor

```js
const regex = /freeCodeCamp/gi;

console.log(regex.test("freeCodeCamp")); // true
console.log(regex.test("freeCodeCamp is great")); // false
console.log(regex.test("I love freeCodeCamp")); // true
console.log(regex.test("freecodecamp")); // false
console.log(regex.test("FREECODECAMP")); // true
console.log(regex.test("free")); // false
console.log(regex.test("code")); // false
console.log(regex.test("camp")); // false
```

:::

Чому? Глобальний модифікатор робить регулярний вираз зі станом. Це означає, що він пам’ятає, де раніше знайшов відповідність шаблону. Тож коли він знаходить перший рядок `freeCodeCamp`, він запам’ятовує, що відповідність починається з індексу `0`.

Потім ми тестуємо його на `freeCodeCamp is great`, але він не починається з індексу `0`. Регулярний вираз "знає", що вже знайшов відповідність на індексі `0`, тому, хоч це інший рядок, він починає пошук з кінця попередньої відповідності.

`freeCodeCamp` має довжину `12` символів, тому відповідність, що починається на `0`, закінчується на індексі `11`. Пошук продовжується з індексу `12`. І оскільки `is great` не відповідає `freeCodeCamp`, повертається `false`.

Потім, оскільки відповідність не знайдена, він "втрачає" свій стан і починає наступний пошук з індексу `0`.

Якщо поміняти місцями логи так, щоб рядок з відповідністю на `0` йшов одразу перед рядком з відповідністю пізніше за індекс `11`:

:::interactive_editor

```js
const regex = /freeCodeCamp/gi;

console.log(regex.test("freeCodeCamp")); // true
console.log(regex.test("I loooooooove freeCodeCamp")); // true
```

:::

Коли регулярний вираз глобальний, він отримує нову властивість `lastIndex`. Візьмемо наш попередній код і подивимось, як ця властивість працює:

:::interactive_editor

```js
const regex = /freeCodeCamp/gi;

console.log(regex.lastIndex); // 0
console.log(regex.test("freeCodeCamp")); // true
console.log(regex.lastIndex); // 12
console.log(regex.test("freeCodeCamp is great")); // false
console.log(regex.lastIndex); // 0
console.log(regex.test("I love freeCodeCamp")); // true
console.log(regex.lastIndex); // 19
console.log(regex.test("freecodecamp")); // false
console.log(regex.lastIndex); // 0
console.log(regex.test("FREECODECAMP")); // true
console.log(regex.lastIndex); // 12
console.log(regex.test("free")); // false
console.log(regex.lastIndex); // 0
console.log(regex.test("code")); // false
console.log(regex.lastIndex); // 0
console.log(regex.test("camp")); // false
```

:::

Дивлячись на цей приклад, ви бачите, як стан регулярного виразу змінюється з кожним викликом тесту, використовуючи `lastIndex` для відстеження попередніх відповідностей.

Глобальний прапорець чудово підходить, коли потрібно знайти кілька відповідностей у одному рядку. Але якщо ви тестуєте кілька рядків одним регулярним виразом, краще вимкнути прапорець `g`.

Перед тим, як вивчати наступний прапорець, потрібно знати про якірці. Якір карет (`^`) на початку регулярного виразу означає "збіг на початку рядка":

```js
const start = /^freecodecamp/i;
```

Якір долара (`$`) в кінці регулярного виразу означає "збіг в кінці рядка":

```js
const end = /freecodecamp$/i;
```

Зробіть паузу і порівняйте вихідні дані праворуч:

:::interactive_editor

```js
const start = /^freecodecamp/i;
const end = /freecodecamp$/i;
console.log(start.test("freecodecamp")); // true
console.log(end.test("freecodecamp")); // true
console.log(start.test("freecodecamp is great")); // true
console.log(end.test("freecodecamp is great")); // false
console.log(start.test("i love freecodecamp")); // false
console.log(end.test("i love freecodecamp")); // true
console.log(start.test("have met freecodecamp's founder")); // false
console.log(end.test("have met freecodecamp's founder")); // false
```

:::

Бачите, як якір початку збігається лише на початку рядка, а якір кінця — лише в кінці рядка? А як щодо пошуку по кількох рядках? Подивимось:

:::interactive_editor

```js
const start = /^freecodecamp/i;
const end = /freecodecamp$/i;
const string = `I really love
freecodecamp
it's my favorite`;
console.log(start.test(string)); // false
console.log(end.test(string)); // false
```

:::

Хоча `freecodecamp` знаходиться на окремому рядку, він не проходить обидва тести. Це тому, що за замовчуванням якірці шукають початок і кінець усього рядка.

Але ви можете зробити регулярний вираз багаторядковим за допомогою прапорця `m`, або мульти-рядкового модифікатора. Додамо його до наших регулярних виразів і подивимось, що отримаємо:

:::interactive_editor

```js
const start = /^freecodecamp/im;
const end = /freecodecamp$/im;
const string = `I really love
freecodecamp
it's my favorite`;
console.log(start.test(string)); // true
console.log(end.test(string)); // true
```

:::

Тепер обидва збігаються! Оскільки `freecodecamp` повністю на своєму рядку, якір початку збігається з початком цього рядка, а якір кінця — з кінцем цього рядка.

Нарешті, є прапорець `d`, або індексний модифікатор. Пам’ятайте, що прапорець `i` відповідає за нечутливість до регістру, тому індексний модифікатор отримав інший прапорець.

Прапорець `d` розширює інформацію, яку ви отримуєте в об’єкті відповідності. Додамо його до нашого регулярного виразу:

```js
const regex = /freecodecamp/di;
const string = "we love freecodecamp isn't freecodecamp great?";
console.log(string.match(regex));
```

І результат:

```js
// [
//   'freecodecamp',
//   index: 8,
//   input: "we love freecodecamp isn't freecodecamp great?",
//   groups: undefined,
//   indices: [
//     0: [8, 20],
//     groups: undefined
//   ]
// ]
```

Наш об’єкт відповідності отримує нову властивість `indices`! Ця властивість — масив із двох чисел: перше — індекс у початковому рядку, де починається відповідність, друге — індекс після кінця відповідності. Цей масив також має додаткову властивість `groups`, яка використовується для іменованих груп захоплення.

Існують ще кілька прапорців, які варто знати, але вони менш поширені у звичайному коді.

Перший — модифікатор unicode, або прапорець `u`. Він розширює функціональність регулярного виразу, дозволяючи знаходити спеціальні символи unicode.

Ви дізнаєтесь більше про класи символів у майбутньому уроці, але прапорець `u` дає доступ до спеціальних класів, як `Extended_Pictographic`, для пошуку більшості емодзі:

:::interactive_editor

```js
const regex = /🍎/u;

const str = "I have an apple 🍎";
console.log(regex.test(str)); // true
```

:::

Також існує прапорець `v`, який ще більше розширює можливості unicode-пошуку.

Другий — модифікатор sticky, або прапорець `y`. Sticky поводиться дуже схоже на глобальний модифікатор, але з кількома відмінностями.

Найголовніша — глобальний регулярний вираз починає пошук з `lastIndex` і шукає всю решту рядка для іншої відповідності, а sticky регулярний вираз повертає `null` і скидає `lastIndex` до `0`, якщо одразу немає відповідності на попередньому `lastIndex`.

І останній — модифікатор single-line, або прапорець `s`. Пам’ятайте, що мульти-рядковий модифікатор дозволяє якірцям початку і кінця збігатися з початком і кінцем рядка, а не всього рядка.

Модифікатор single-line дозволяє символу-джокеру, який у regex позначається крапкою (`.`), збігатися з розривами рядків — фактично розглядаючи рядок як один рядок тексту.

Існує багато таких модифікаторів, але прапорці `i` і `g` — це ті, які ви використовуватимете найчастіше і які найважливіше запам’ятати.
