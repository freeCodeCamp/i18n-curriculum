# --description--

有没有想过如何让你的 Express 应用保持稳定并适合生产环境，尤其是在负载很重或正在关闭时？

这就是健康检查和优雅关闭发挥作用的地方。这两种模式帮助你保持应用可用，并确保它能干净地退出，不会丢失请求或破坏数据。

让我们来详细讲解一下。

首先，Express 中的“健康检查”到底是什么？

健康检查是一个简单的路由，用来告诉你应用是否正在运行。

例如：

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

像负载均衡器或容器编排工具——比如 Kubernetes 或 AWS Elastic Beanstalk——可以访问这个路由来判断你的应用是否健康。

如果路由返回 200 状态码，说明你的服务被认为是健康的。提醒一下——200 HTTP 状态码意味着一切正常！

如果没有响应，编排工具可以自动重启服务。

那么什么是“优雅关闭”？

优雅关闭是指当你的应用即将停止时发生的事情，比如服务器重启或部署时。

你不会立即切断连接，而是给应用时间完成正在进行的工作。

在 Express 中，你可以监听 `SIGTERM`，这是一个系统信号，表示进程应该终止。

示例如下：

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

这告诉 Express 停止接受新连接，并等待现有连接完成后再退出。

你还应该处理 `SIGINT`，这是在本地开发时按 Ctrl+C 停止应用时发送的信号。

虽然 `SIGTERM` 通常由生产环境中的 Kubernetes 等编排工具使用，`SIGINT` 则覆盖了日常开发者的工作流程。如果不处理它，按 Ctrl+C 会立即杀死进程，跳过你设置的任何清理工作。

因为这两个信号都应该触发相同的优雅关闭行为，开发者通常会监听这两个信号。

现在，把它们整合起来。

想象你的应用运行在容器中。Kubernetes 发送 `SIGTERM` 来关闭它以进行更新。

有了基本的优雅关闭，你的应用停止接受新连接，完成当前请求。所有工作完成后，它断开数据库连接并干净地退出。没有数据丢失，没有状态损坏。

但有个问题——负载均衡器不知道你的应用正在关闭，所以它可能继续向你发送新流量。这时下面的额外模式就派上用场了。

现在，我给你一个小彩蛋。我们来看“在关闭前标记应用为不健康”。

在关闭之前，你可以更新健康检查返回 503 状态。这样，负载均衡器看到不健康的响应，就会在关闭期间停止将新流量路由到你的应用，正如上面代码所示。

提醒：`503` 状态码表示“服务不可用”。它表明服务器暂时无法处理请求，这正是优雅关闭期间发生的情况。

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
