# --description--

每當你造訪網站或使用 API 時，你的瀏覽器和伺服器會不斷地互相溝通。

這段對話中最重要的部分之一，就是 HTTP 回應狀態碼。

它們是簡短的數字訊息——像是 200、404 或 500——用來告訴客戶端（終端用戶的裝置，例如筆電或手機）在發出請求後發生了什麼事。

你已經學過其中一些，但讓我們依照類別來拆解，讓你能準確回想它們的意義以及何時使用。

## 1xx – 資訊性回應

以 1 開頭的狀態碼在日常開發中很少見，但它們表示請求已被接收且處理程序正在繼續，例如上傳檔案時。

常見範例：

* 100 Continue – 伺服器告訴客戶端：「好的，你可以繼續傳送剩下的請求內容。」
* 101 Switching Protocols – 客戶端送出 `Upgrade` 標頭，因此伺服器正在切換協定。你最常在使用 Websockets 時看到這個。

## 2xx – 成功回應

這表示請求已成功接收、理解並處理。

* 200 OK – 請求成功。這是最常見的成功回應。
* 201 Created – 新資源已成功建立，例如註冊新使用者後。
* 204 No Content – 請求成功，但沒有回應主體要回傳。常用於 `DELETE` 請求。

## 3xx – 重新導向訊息

這告訴客戶端：「你要找的資源在別處。」

* 301 Moved Permanently – 資源有新的永久 URL。
* 302 Found – 資源暫時位於不同的 URL。
* 304 Not Modified – 用於快取。告訴客戶端沒有變更，可以使用快取版本。

## 4xx – 客戶端錯誤

當客戶端送出無效請求時會發生這些錯誤。

* 400 Bad Request – 請求格式錯誤或無效。
* 401 Unauthorized – 需要驗證（你必須登入）。
* 403 Forbidden – 驗證成功，但你沒有權限。
* 404 Not Found – 請求的資源不存在。沒錯，這是網路上最具代表性的錯誤。

## 5xx – 伺服器錯誤

這表示伺服器端發生了問題。

* 500 Internal Server Error – 通用的「發生錯誤」訊息。
* 502 Bad Gateway – 作為代理的伺服器從另一台伺服器收到無效回應。
* 503 Service Unavailable – 伺服器暫時過載或維護中。

現在你已熟悉一些最常見的 HTTP 狀態碼，讓我們來看幾個在 Express 中的實際範例。

假設你正在建置一個 API。如果使用者送出無效資料，你可以回應 400 狀態碼，表示「錯誤的請求」。你也可以回傳字串 "Bad Request" 給客戶端，讓他們清楚知道 400 狀態碼的意義：

```js
res.status(400).send("Bad Request");
```

如果一切順利，你可以回傳 200 狀態碼。你也可以附帶訊息。這裡我們回傳一些 JSON，裡面有簡單的 "Success" 訊息：

```js
res.status(200).json({ message: "Success" });
```

如果伺服器發生錯誤，你可以回傳 500 狀態碼，表示內部伺服器錯誤，並附上清楚的錯誤訊息：

```js
res.status(500).send("Internal Server Error");
```

這些狀態碼不只是給機器用的。它們幫助開發者偵錯問題，也幫助使用者（和其他開發者）了解發生了什麼事。
