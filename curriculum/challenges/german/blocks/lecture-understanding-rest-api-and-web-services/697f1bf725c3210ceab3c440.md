# --description--

Webservices und REST-APIs treiben die meisten Anwendungen und Websites an, die Sie täglich nutzen. Sie ermöglichen es verschiedenen Anwendungen, miteinander zu kommunizieren und Daten über ein Netzwerk auszutauschen, meist über das Internet.

Eine API (Application Programming Interface) ist eine allgemeine Möglichkeit für Software, mit anderer Software zu kommunizieren. Ein Webservice ist eine spezielle Art von API, die Webtechnologien wie HTTP verwendet, um Daten über ein Netzwerk, üblicherweise das Internet, auszutauschen. Deshalb ist jeder Webservice eine API – aber nicht jede API ist ein Webservice. REST-APIs sind eine häufige Art von Webservice, die einer bestimmten Reihe von Regeln folgen, wie Daten angefragt und zurückgegeben werden.

Diese Lektion konzentriert sich auf die grundlegende Funktionsweise von Webservices, und die nächste Lektion wird detaillierter darauf eingehen, wie REST-APIs funktionieren.

Sie können sich einen Webservice als eine Brücke von Software zu Software vorstellen. Er hat keine Benutzeroberfläche und ist nicht dafür gedacht, direkt verwendet zu werden. Stattdessen konzentriert er sich vollständig darauf, Daten zwischen Anwendungen auszutauschen.

Damit ein System als Webservice gilt, muss es folgende Eigenschaften haben:

* **Läuft auf einem Server**: Ein Webservice muss online gehostet werden wie eine Website, aber anstelle von Seiten, die Benutzer sehen können, macht er Daten oder Funktionen verfügbar
    
* **Maschine-zu-Maschine-Kommunikation**: Ein Webservice ist nicht dafür gemacht, dass Menschen ihn direkt ansehen, sondern damit mobile Apps und Web-Apps ihn nutzen können, um Daten oder Funktionen bereitzustellen, die der Benutzer schließlich sehen kann
    
* **Sprachenunabhängig**: Ein Webservice kann in jeder Sprache geschrieben sein, zum Beispiel Java oder C#, und jede Anwendung sollte mit ihm interagieren können, selbst wenn sie in einer anderen Sprache geschrieben ist
    
* **Plattformunabhängig**: Es spielt keine Rolle, auf welcher Plattform die Maschinen laufen. Zum Beispiel sollte ein Webservice, der auf Linux läuft, nicht verhindern, dass ein Gerät mit Windows mit ihm kommuniziert
    
* **Verwendung standardisierter Webprotokolle**: Ein Webservice sollte typischerweise HTTP/HTTPS verwenden, um Daten über das Internet zu übertragen, FTP für Dateiübertragungen und SMTP für E-Mails
    

**REST** und **SOAP** sind zwei der bekanntesten Arten von Webservices, obwohl es auch andere Ansätze gibt, wie **XML-RPC**.

**REST**, was für **Representational State Transfer (REST)** steht, ist heute mit Abstand am weitesten verbreitet. Die meisten modernen Anwendungen setzen auf RESTful-Webservices, weil sie einfach, flexibel sind und natürlich mit dem Web funktionieren.

REST-APIs machen Ressourcen verfügbar, die üblicherweise durch URL-Pfade wie `/users` oder `/users/20` dargestellt werden. Wenn ein Client eine Ressource anfragt, gibt der Server eine Darstellung dieser Ressource zurück. Diese Darstellung ist oft in JSON, kann aber auch in Formaten wie XML oder sogar einfachem Text vorliegen.

Clients können das gewünschte Format über HTTP-Kopfzeilen angeben, wie zum Beispiel die `Accept`-Kopfzeile. Ein Client kann beispielsweise eine JSON-Antwort anfordern, indem er den passenden `Accept`-Wert setzt.

```bash
Accept: application/json
```

Und so können Sie angeben, dass Sie XML möchten:

```bash
Accept: application/xml
```

Die Ressource, die Sie zurückbekommen, kann so aussehen, wenn sie JSON ist:

```json
{
  "id": 20,
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "createdAt": "2025-09-22T10:30:00Z"
}
```

Und so, wenn sie XML ist:

```xml
<user>
  <id>20</id>
  <name>John Doe</name>
  <email>john@example.com</email>
  <role>admin</role>
  <createdAt>2025-09-22T10:30:00Z</createdAt>
</user>
```

SOAP steht für **Simple Object Access Protocol**. Es verwendet ausschließlich das XML-Datenformat und ist am häufigsten in Unternehmenssystemen, die strengere Standards und Sicherheit benötigen.

So könnte SOAP-Daten aussehen:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
   <soapenv:Body>
      <getUser>
         <userId>20</userId>
      </getUser>
   </soapenv:Body>
</soapenv:Envelope>
```

XML-RPC verwendet XML, um seine Anfragen und Antworten zu kodieren. Das macht es leichtgewichtig, aber weniger flexibel als REST.

UDDI ist kein Datenaustauschdienst, sondern ein XML-basiertes Verzeichnis, in dem Unternehmen ihre verfügbaren Webservices auflisten, entdecken und beschreiben können.
