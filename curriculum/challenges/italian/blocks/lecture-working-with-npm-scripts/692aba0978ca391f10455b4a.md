# --description--

Scopriamo il registro npm, dove puoi condividere i tuoi pacchetti JavaScript con il mondo.

Il **registro npm** è un database di pacchetti pubblici e privati.

Sviluppatori e organizzazioni possono pubblicare pacchetti in questo registro per condividere i loro strumenti con le loro comunità e membri del team, e in molti casi, con tutti gli altri sviluppatori Node.js.

I pacchetti possono essere pubblici o privati. I pacchetti pubblici possono essere consultati da chiunque. I pacchetti privati possono essere consultati solo da utenti specifici o membri di un'organizzazione.

Nel registro npm, puoi condividere ciò che hai creato con il mondo con pochi clic. Suona bene, vero?

Vediamo come puoi pubblicare un pacchetto passo dopo passo.

## Passaggio 1: crea un account

Il primo passaggio è creare un account npm gratuito. Per creare il tuo account:

* Vai a [https://www.npmjs.com/signup](https://www.npmjs.com/signup).
    
* Compila il modulo. Dovrai inserire il tuo nome utente, email e password.
    

Nota che, secondo la pagina di registrazione, "il tuo indirizzo email sarà aggiunto ai metadati dei pacchetti che pubblichi, quindi potrebbe essere visibile pubblicamente."

## Passaggio 2: **accedi**

Una volta che hai un account npm, dovrai accedere dal terminale eseguendo questo comando:

```bash
npm login
```

Poi dovrai autenticarti con le informazioni del tuo account.

## **Passaggio 3: crea un file** `package.json`

Prima di pubblicare il tuo pacchetto, devi creare un file `package.json` con questo comando:

```bash
npm init
```

(Se il tuo progetto ne ha già uno, puoi andare direttamente al **passaggio 4**).

Dovrai inserire queste informazioni nel tuo file `package.json`:

* **name**: di solito è il nome del tuo progetto/directory. Se prevedi di rendere pubblico il pacchetto, questo nome deve essere unico, a meno che tu non usi gli scope (ne parleremo tra poco).
    
* **version**: dovresti iniziare con `"1.0.0"`, il valore predefinito. Rappresenta la versione iniziale del pacchetto e puoi aggiornarla man mano che apporti modifiche.
    
* **main**: il punto di ingresso del progetto. Per impostazione predefinita è `index.js`.
    
* **author**: puoi scrivere il tuo nome o il tuo nickname preferito.
    
* **license**: scegli una licenza specifica. Alcuni esempi sono MIT e ISC.
    
* **description**: un breve riassunto del pacchetto.
    

Quando scegli il nome del pacchetto, assicurati che non sia già stato preso nel registro npm. Per farlo, puoi andare all’URL seguente, sostituendo `<your-package-name>` con il nome del tuo pacchetto:

```bash
https://www.npmjs.com/package/<your-package-name>
```

Se non esiste un pacchetto a quell’URL, il nome è disponibile e puoi usarlo.

Se invece il nome è già preso e vuoi davvero mantenerlo, puoi usare ciò che si chiama **scope** per pubblicarlo.

### **Scope**

Quando crei un account utente o organizzazione nel registro npm, ti viene assegnato uno scope con il tuo nome utente o il nome della tua organizzazione.

Gli scope sono utili per evitare conflitti di nomi perché puoi pubblicare il pacchetto sotto il tuo scope, anche se esiste già un altro pacchetto con lo stesso nome.

Il nome di un pacchetto con scope inizia con il nome dello scope. Per esempio, in `package.json`, troveresti questa coppia chiave-valore:

```bash
"name": "@<scope-name>/<your-package-name>"
```

Puoi fare questa modifica direttamente nel file se necessario, oppure puoi creare il file `package.json` con lo scope fin dall’inizio aggiungendo `--scope=@<scope-name>` al comando `npm init`.

```bash
npm init --scope=@<scope-name>
```

Il nome dello scope può essere il tuo nome utente se usi un account personale, o il nome della tua organizzazione se usi un account organizzativo.

## Passaggio 4: crea README.md

Il README del tuo pacchetto è un file markdown dove descrivi cosa fa il pacchetto e come usarlo.

Alcuni esempi di contenuti importanti che potresti voler includere sono:

* Lo scopo del pacchetto.
    
* Come installarlo.
    
* Dipendenze importanti e considerazioni.
    
* Come iniziare.
    
* Frammenti di codice di esempio.
    
* Linee guida per i contributi (se accetti contributi).
    
* Collegamento a un file changelog dove tieni traccia delle modifiche fatte nelle diverse versioni.
    

La struttura del README è flessibile, puoi adattarla alle tue esigenze.

## **Passaggio 5:** `.npmignore`

Il file `.npmignore` indica al registro npm quali file includere e quali ignorare quando pubblichi il pacchetto. È molto importante per evitare di condividere informazioni sensibili come chiavi private, password e dati personali.

In questo file puoi scrivere pattern per specificare quali file, directory e tipi di file devono essere ignorati. Segue le stesse regole di pattern matching dei file `.gitignore`.

Se hai lavorato con Git, potresti pensare che questo file sia molto simile a `.gitignore`, giusto? Infatti lo è. Secondo la documentazione npm, "Se non c’è un file `.npmignore`, ma c’è un file `.gitignore`, allora npm ignorerà ciò che è indicato nel file `.gitignore`."

Devi anche sapere che npm cerca i file `.npmignore` e `.gitignore` in tutte le sottodirectory del pacchetto, non solo nella directory principale.

Per impostazione predefinita, questi percorsi sono ignorati automaticamente, quindi non devi aggiungerli a `.npmignore`:

* `.*.swp`
    
* `._*`
    
* `.DS_Store`
    
* `.git`
    
* `.gitignore`
    
* `.hg`
    
* `.npmignore`
    
* `.npmrc`
    
* `.lock-wscript`
    
* `.svn`
    
* `.wafpickle-*`
    
* `config.gypi`
    
* `CVS`
    
* `npm-debug.log`
    
La directory `node_modules` è anch’essa ignorata, tranne le dipendenze bundle.

## **Passaggio 6: pubblica il tuo pacchetto**

Prima di pubblicare il pacchetto nel registro npm, dovresti testarlo eseguendo questo comando:

```bash
npm install <full-path-to-package-directory>
```

Quando sei pronto a pubblicarlo, il comando dipenderà dal tipo di pacchetto: con scope o senza scope.

Se il pacchetto è senza scope, esegui questo comando:

```bash
npm publish
```

Se il pacchetto ha uno scope, sarà pubblicato come pacchetto privato per impostazione predefinita (se questa funzionalità è disponibile nel tuo account). Se vuoi renderlo pubblico, devi aggiungere `--access public` al comando.

```bash
npm publish --access public
```

Per vedere il tuo pacchetto pubblico, vai a questo URL:

```bash
https://npmjs.com/package/<your-package-name>
```

Dovresti vedere un’indicazione della visibilità del pacchetto (pubblico o privato) sotto il nome del pacchetto.

## **Passaggio 7 (opzionale): installa il tuo pacchetto**

Ora che hai tutto pronto, puoi installare il pacchetto in un progetto Node.js con questo comando:

```bash
npm install <your-package-name>
```

## Aggiornare il pacchetto

Se devi apportare modifiche al pacchetto dopo averlo pubblicato, puoi aggiornare la versione nel file `package.json`.

Questa è la convenzione standard per il versionamento semantico:

* **Prima release:** `1.0.0`.
    
* **Patch:** incrementa la terza cifra. Per esempio, da `1.0.0` a `1.0.1`.
    
* **Minor:** incrementa la cifra centrale e azzera l’ultima. Per esempio, da `1.0.0` a `1.1.0`.
    
* **Major:** incrementa la prima cifra e azzera la seconda e la terza. Per esempio, da `1.1.0` a `2.0.0`.
    
Ottimo lavoro. Ora sai come pubblicare un pacchetto nel registro npm e come aggiornarlo per correggere bug e aggiungere nuove funzionalità.
