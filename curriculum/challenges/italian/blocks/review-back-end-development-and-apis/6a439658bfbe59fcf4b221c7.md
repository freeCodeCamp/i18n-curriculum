## Lavorare con Node.js e l'architettura event-driven

- **Node.js**: Un ambiente di esecuzione JavaScript open source e multipiattaforma che ti permette di eseguire codice JavaScript al di fuori del browser. È ampiamente usato per costruire server web, API e sistemi back-end.
- **Limitazioni lato client**: JavaScript lato client ha un accesso limitato al file system, supporto ridotto per logiche applicative complesse e potenziali rischi di sicurezza (ad esempio, non devi esporre le credenziali del database nel codice lato client).
- **Oggetto globale nel browser vs Node.js**: Nel browser, l'oggetto globale è `window`, che espone la manipolazione del DOM, i cookie e gli eventi del browser. In Node l'oggetto globale è `global`, che espone moduli integrati per file, rete e interazione con il sistema operativo.
- **Accesso al DOM**: Il browser fornisce accesso all'API DOM, mentre Node no. Al contrario, Node può consultare quasi tutte le risorse di sistema, incluso il file system, cosa che il browser non può fare.
- **Controllo della versione**: Puoi scegliere quale versione di Node eseguire sul tuo server. Non hai controllo sulla versione del browser che usano gli utenti.
- **Architettura non bloccante e event-driven**: Node usa un singolo thread e un event loop che gestisce efficientemente molte richieste simultanee e operazioni I/O senza bloccare, ideale per applicazioni in tempo reale.
- **Callback**: Funzioni che definiscono cosa succede una volta completata un'operazione asincrona. L'ampio uso di callback è parte del motivo per cui il codice asincrono in Node può essere più difficile da leggere e fare il debug.
- **Vantaggi di Node nel back-end**: Un unico linguaggio (JavaScript) per front-end e back-end, architettura non bloccante adatta ad alta concorrenza, vasto ecosistema npm, gratuito e open source.
- **Svantaggi di Node nel back-end**: Single-threaded, quindi compiti intensivi per la CPU (elaborazione immagini, crittografia, matematica complessa) possono bloccare l'event loop, l'uso intensivo di codice asincrono e basato su callback può ridurre la leggibilità, i pacchetti npm variano in qualità e manutenzione.
- **NVM (Node Version Manager)**: Uno strumento per gestire e passare tra più versioni di Node.js sulla stessa macchina. Su macOS/Linux si installa con:

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **Installare la versione LTS di Node**: `nvm install --lts`
- **Cambiare versione di Node**: `nvm use 20`
- **Elencare tutte le versioni di Node installate**: `nvm ls`
- **Impostare una versione predefinita di Node (macOS/Linux)**: `nvm alias default 22.20.0`
- **nvm-windows**: Un progetto separato per gestire le versioni di Node su Windows (non WSL). Scarica l'installer dal repository ufficiale ed esegui `nvm install lts` poi `nvm use <version>`.
- **Eseguire un file JavaScript con Node**: `node app.js`

## Lavorare con i moduli core di Node

### Il modulo `fs`

- **Scopo**: Fornisce metodi per creare, leggere, scrivere, aggiungere in coda ed eliminare file e directory.
- **Tre stili di utilizzo**: Basato su callback (asincrono), basato su promise (`fs/promises`) e sincrono (suffisso `Sync`). I metodi sincroni bloccano l'event loop e sono adatti solo per script.
- **`writeFile()`**: Scrive contenuto in un file, creandolo se non esiste:

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**: Aggiunge contenuto in coda a un file esistente senza sovrascriverlo.
- **`readFile()`**: Legge il contenuto di un file. Passa `"utf8"` come codifica per ottenere una stringa, oppure omettendo ottieni un `Buffer`.
- **`unlink()`**: Elimina un file. Lancia `ENOENT` se il file non esiste.

### Il modulo `Buffer`

- **Scopo**: Gestisce dati binari (file, immagini, stream di rete) direttamente in memoria.
- **`Buffer.from()`**: Crea un buffer da una stringa:

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**: Crea un buffer di `n` byte inizializzato a zero:

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **Troncamento**: Scrivere più dati di quelli che il buffer può contenere tronca silenziosamente l'eccesso.
- **`Buffer.byteLength(str)`**: Restituisce il numero di byte necessari per memorizzare una stringa.

### Il modulo `crypto`

- **Scopo**: Fornisce strumenti crittografici come hashing, HMAC, crittografia/decrittografia, firme digitali e generazione sicura di numeri casuali.
- **`createHash(algorithm)`**: Crea un hash unidirezionale (irreversibile). Usa `.update(data).digest("hex")` per fornire i dati da hashare e ottenere l'hash come stringa esadecimale:

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**: Simile a `createHash` ma richiede una chiave segreta, adatto per verificare integrità e autenticazione dei dati.
- **`createCipheriv()` / `createDecipheriv()`**: Crittografano e decrittografano dati. Entrambi richiedono un algoritmo, una chiave e un vettore di inizializzazione (IV):

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**: Genera byte casuali crittograficamente sicuri, più sicuri di `Math.random()` per token e chiavi.
- **`randomInt(min, max)`**: Genera un intero casuale sicuro nell'intervallo dato. Utile per OTP.
- **`sign()` / `verify()`**: Creano e verificano firme digitali usando una coppia di chiavi privata/pubblica.

### Il modulo `os`

- **Scopo**: Espone informazioni sul sistema operativo su cui gira Node.
- **`os.platform()`**: Restituisce l'identificatore del sistema operativo, es. `'darwin'`, `'win32'`, `'linux'`.
- **`os.arch()`**: Restituisce l'architettura della CPU, es. `'x64'`, `'arm64'`.
- **`os.type()`**: Restituisce il nome ufficiale del sistema operativo, es. `'Darwin'`.
- **`os.release()`**: Restituisce la versione del kernel del sistema operativo.
- **`os.cpus()`**: Restituisce un array di oggetti che descrivono ogni core logico della CPU (modello, velocità, tempi).
- **`os.uptime()`**: Restituisce il tempo di attività del sistema in secondi.
- **`os.totalmem()` / `os.freemem()`**: Restituiscono la memoria totale e disponibile in byte.
- **`os.userInfo()`**: Restituisce un oggetto con `uid`, `gid`, `username`, `homedir` e `shell` dell'utente corrente.
- **`os.networkInterfaces()`**: Restituisce oggetti delle interfacce di rete con indirizzo, famiglia, MAC e dettagli CIDR.

### Il modulo `path`

- **Scopo**: Fornisce utility per lavorare con percorsi di file e directory in modo indipendente dalla piattaforma.
- **`__filename`**: Variabile globale, percorso assoluto del file corrente.
- **`__dirname`**: Variabile globale, percorso assoluto della directory contenente il file corrente.
- **`path.basename(p)`**: Restituisce l'ultimo componente di un percorso (il nome del file).
- **`path.dirname(p)`**: Restituisce la parte di directory di un percorso.
- **`path.extname(p)`**: Restituisce l'estensione del file, es. `'.js'`.
- **`path.join(...segments)`**: Unisce segmenti di percorso in un percorso normalizzato e corregge slash errati.
- **`path.resolve(...segments)`**: Risolve segmenti in un percorso assoluto partendo dalla directory di lavoro corrente.
- **`path.parse(p)`**: Restituisce un oggetto con proprietà `root`, `dir`, `base`, `ext` e `name`.
- **`path.format(obj)`**: Costruisce una stringa di percorso da un oggetto con proprietà `dir`, `name` e `ext`.

### Il modulo `process`

- **Scopo**: Espone informazioni e controllo sul processo Node.js corrente. È un oggetto globale, quindi non serve importarlo.
- **`process.env`**: Oggetto contenente tutte le variabili d'ambiente. Consulta valori specifici come `process.env.NODE_ENV`.
- **`process.argv`**: Array degli argomenti da linea di comando. L'indice 0 è l'eseguibile Node, l'indice 1 è il percorso dello script, gli indici 2+ sono argomenti utente.
- **`process.cwd()`**: Restituisce la directory di lavoro corrente.
- **Eventi del processo**:

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**: Genera un avviso personalizzato intercettabile da un listener `warning`.

### Il modulo `stream`

- **Scopo**: Gestisce grandi quantità di dati a pezzi invece di caricare tutto in memoria in una volta.
- **Quattro tipi di stream**: Readable (legge dati a pezzi), Writable (scrive dati a pezzi), Duplex (entrambi), Transform (un Duplex che modifica i dati mentre scorrono).
- **Piping degli stream**:

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **Evento `finish`**: Scatta su uno stream scrivibile quando tutti i dati sono stati svuotati.
- **Evento `error`**: Scatta quando uno stream incontra un problema.

## Introduzione a npm

- **npm**: Comprende tre parti: il sito web `npmjs.com`, il registro pubblico (un database di pacchetti) e lo strumento CLI installato con Node.js.
- **Scopo**: Permette di installare, gestire e pubblicare pacchetti JavaScript riutilizzabili. npm è il gestore di pacchetti più usato.
- **Alternative**: Yarn, PNPM, Bun.

### Il file `package.json`

- **Scopo**: Il manifesto di configurazione npm. Tiene traccia del nome del progetto, versione, descrizione, punto di ingresso (`main`), autore, licenza, script e dipendenze.
- **Creare `package.json` interattivamente**: `npm init`
- **Creare con valori predefiniti**: `npm init -y`
- **`dependencies`**: Pacchetti necessari per far girare l'app in produzione (es. `express`, `react`).
- **`devDependencies`**: Pacchetti necessari solo durante sviluppo e test (es. `jest`, `nodemon`, `eslint`).
- **`license`**: Termini legali per l'utilizzo. Valori comuni: `MIT`, `ISC`, `GPL`.

### Versionamento semantico (SemVer)

- **Formato**: `MAJOR.MINOR.PATCH`, es. `4.17.21`.
- **MAJOR**: Incrementato per cambiamenti incompatibili.
- **MINOR**: Incrementato per nuove funzionalità retrocompatibili.
- **PATCH**: Incrementato per correzioni di bug retrocompatibili.
- **Caret `^`**: Permette aggiornamenti a versioni minor e patch ma non major. Esempio: `^4.17.21` accetta `4.18.0` ma non `5.0.0`.
- **Tilde `~`**: Permette aggiornamenti solo a patch. Esempio: `~1.2.3` accetta `1.2.4` ma non `1.3.0`.
- **Asterisco `*`**: Corrisponde a qualsiasi versione, adatto solo per test, mai per produzione.
- **Versione esatta**: Nessun prefisso, es. `1.2.3` fissa esattamente quella versione.

### Installare e rimuovere dipendenze

- **Installare un pacchetto**: `npm install express` (abbreviazione: `npm i express`)
- **Installare come devDependency**: `npm install nodemon -D`
- **Disinstallare un pacchetto**: `npm uninstall chalk`
- **`node_modules/`**: Cartella creata dopo l'installazione contenente il codice effettivo dei pacchetti. Non va committata nel controllo versione.
- **`package-lock.json`**: File auto-generato che blocca le versioni esatte di ogni pacchetto installato (incluse dipendenze figlie). Va sempre committato e non modificato manualmente.
- **Installare una versione specifica**: `npm install express@4.21.2`
- **Visualizzare versioni disponibili**: `npm view express versions`
- **Controllare pacchetti obsoleti**: `npm outdated`
- **Aggiornare un singolo pacchetto** (entro il range consentito): `npm update express`
- **Forzare l'installazione dell'ultima versione**: `npm install express@latest`

### Pubblicare nel registro npm

- **Login**: `npm login`
- **Pacchetti scoped**: Usa `@username/package-name` per evitare collisioni di nomi.
- **`.npmignore`**: Elenca i file da escludere dal pacchetto pubblicato (simile a `.gitignore`).
- **Pubblicare un pacchetto non scoped**: `npm publish`
- **Pubblicare un pacchetto scoped pubblicamente**: `npm publish --access public`

## Lavorare con gli script npm

- **Script npm**: Comandi personalizzati definiti nell'oggetto `"scripts"` di `package.json`. Automatizzano attività ripetitive come avviare un server, eseguire test, linting e build.
- **Definire uno script**:

```json
"scripts": {
  "start": "node app.js"
}
```

- **Eseguire uno script**: `npm run start`
- **Abbreviazioni per nomi di script integrati** (`start`, `test`, `stop`, `restart`): `npm start`
- **Comandi sequenziali con `&&`** (il comando successivo parte solo se il precedente ha successo):

```json
"start": "npm run build && node server.js"
```

- **Comandi sequenziali con `;`** (il comando successivo parte sempre):

```json
"start": "npm run build; node server.js"
```

- **Comandi concorrenti con `&`**: `"dev": "npm run start:server & npm run start:client"`
- **Passare argomenti a uno script** (usa il separatore `--`): `npm start -- 8000`; gli argomenti sono accessibili tramite `process.argv[2]`.

### Moduli CommonJS

- **CommonJS**: Il sistema di moduli originale di Node.js. Carica moduli in modo sincrono.
- **Importare** con `require()`:

```js
const { multiply } = require('./math');
```

- **Esportare** con `module.exports`:

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- CommonJS è il default quando `package.json` ha `"type": "commonjs"` (o nessun campo `"type"`), oppure il file ha estensione `.cjs`.

### Moduli ES (ESM)

- **Moduli ES (ESM)**: Il formato di modulo moderno e standardizzato. Caricati in modo asincrono.
- **Esportazione nominata**: `export function multiply(a, b) { return a * b; }`
- **Esportazione di default**: `export default function multiply(a, b) { return a * b; }`
- **Importare**: `import { multiply } from './math.mjs';`
- ESM si usa quando il file ha estensione `.mjs` o `package.json` contiene `"type": "module"`.

## Comprendere come funzionano HTTP, DNS e TCP/IP

### Server e modello client-server

- **Modello client-server**: Un'architettura di sistema distribuito dove i componenti sono client (processi che inviano richieste, es. un browser) o server.
- **Server**: Un computer o programma software che riceve richieste dai client e restituisce dati o servizi su una rete.
- **Tipi di server**: Server web (HTML/CSS/risorse), server applicativi (logica di business), server database, server mail, server file, server proxy.

### DNS

- **DNS (Domain Name System)**, spesso chiamato "rubrica telefonica di internet": Traduce nomi di dominio leggibili dall'uomo (es. `freecodecamp.org`) in indirizzi IP.
- **Gerarchia dei nomi di dominio** (da destra a sinistra): Dominio di primo livello (TLD, es. `.org`) → Dominio di secondo livello (es. `freecodecamp`) → Sottodominio (es. `www`).
- **Indirizzo IP**: Identificatore numerico unico per un dispositivo in rete. Esempio IPv4: `127.0.0.1`. Esempio IPv6: `0:0:0:0:0:0:0:1`.
- **Processo di risoluzione DNS**: Browser → Resolver ricorsivo → Server root → Server TLD → Server autorevole → restituisce indirizzo IP → Resolver ricorsivo memorizza nella cache e consegna al client.

### TCP/IP e HTTP

- **IP (Internet Protocol)**: Instrada pacchetti dati da sorgente a destinazione usando indirizzi IP. Senza connessione, ogni pacchetto è inviato indipendentemente.
- **TCP (Transmission Control Protocol)**: Suddivide i dati in segmenti numerati, li ricompone a destinazione, verifica errori e richiede ritrasmissione se necessario. Garantisce consegna affidabile e ordinata.
- **HTTP (HyperText Transfer Protocol)**: Protocollo di livello applicazione che definisce come client e server scambiano contenuti web.
- **HTTPS**: Versione sicura di HTTP dove i dati sono criptati usando SSL/TLS.

### Struttura di una richiesta HTTP

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **Linea di richiesta**: Metodo HTTP + percorso/URI + versione HTTP.
- **Intestazioni**: Coppie chiave-valore di metadati.
- **Corpo** (opzionale): Inviato con `POST`, `PUT`, `PATCH`, `DELETE` per trasportare dati.

### Struttura di una risposta HTTP

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **Linea di stato**: Versione HTTP + codice di stato + frase di motivo.
- **Intestazioni**: Metadati sulla risposta (tipo contenuto, lunghezza, direttive di caching).
- **Corpo** (opzionale): I dati richiesti.

### Categorie di codici di stato HTTP

- **1xx Informativi**: Richiesta ricevuta, elaborazione continua. Esempio: `101 Switching Protocols`.
- **2xx Successo**: `200 OK`, `201 Created`, `204 No Content`.
- **3xx Reindirizzamento**: `301 Moved Permanently`, `302 Found`, `304 Not Modified`.
- **4xx Errori client**: `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.
- **5xx Errori server**: `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable`.

## Comprendere il modello richiesta-risposta HTTP

- **Metodi HTTP**: `GET` (recuperare), `POST` (creare), `PUT` (aggiornamento completo), `PATCH` (aggiornamento parziale), `DELETE` (rimuovere).
- **Ciclo richiesta-risposta**: Il client invia la richiesta → Il server valida l'autenticazione, controlla metodo e URL, interroga il database → Il server costruisce la risposta (codice di stato + corpo opzionale) → Il client gestisce la risposta.

### Risorse restituite nelle risposte HTTP

Ogni risposta include un'intestazione `Content-Type` che indica al browser come gestire la risorsa:

- **HTML**: `text/html`
- **CSS**: `text/css`
- **JavaScript**: `application/javascript`
- **JSON**: `application/json`
- **PNG/JPEG/WebP**: `image/png`, `image/jpeg`, `image/webp`
- **MP4/MP3**: `video/mp4`, `audio/mpeg`
- **PDF**: `application/pdf`
- **WOFF2**: `font/woff2`

### Invio di moduli HTML

- **Attributo `action`**: Specifica l'URL a cui inviare i dati del modulo.
- **Attributo `method`**: Specifica il metodo HTTP (`GET` o `POST`). HTML non supporta nativamente `PUT`, `PATCH` o `DELETE`, usa invece Fetch API o il pacchetto `method-override` in Express.
- **Validazione lato client**: Usa gli attributi `required` e `pattern` per feedback immediato. Valida sempre anche lato server.
- **Modulo `GET`**: I dati sono aggiunti all'URL come parametri di query (es. moduli di ricerca).
- **Modulo `POST`**: I dati sono inviati nel corpo della richiesta (es. moduli di login e registrazione).

## Comprendere il modello degli standard web

### Organismi di standardizzazione

- **WHATWG**: Mantiene HTML e DOM come "standard vivente" continuamente aggiornato. Le decisioni sono guidate dai principali team dei motori browser (Chromium, Gecko, WebKit).
- **W3C (World Wide Web Consortium)**: Organismo principale di standardizzazione. Supervisiona CSS, SVG, linee guida per l'accessibilità (WCAG) e numerose API web.
- **ECMA International / TC39**: Ecma pubblica la specifica ufficiale ECMAScript. TC39 sviluppa il linguaggio attraverso un processo di proposta in cinque fasi (Stage 0 idea grezza → Stage 4 specifica completa).
- **Khronos Group**: Si concentra su grafica, calcolo e media. Produce WebGL e WebGL 2 per rendering 3D accelerato GPU nei browser.

### Creazione degli standard web

1. **Identificazione di un bisogno**: Uno sviluppatore o un'organizzazione incontra un problema e lo documenta.
2. **Proposta e discussione iniziale**: Issue informale su GitHub o documento esplicativo condiviso per valutare interesse. In TC39 è Stage 0/Stage 1.
3. **Scrittura della specifica**: Documento formale preciso che copre ogni caso limite, condizione di errore e implicazione di sicurezza.
4. **Revisione, obiezioni e iterazione**: Bozza revisionata pubblicamente dai vendor browser, funzionalità che un browser importante rifiuta raramente diventano standard.
5. **Implementazione e test**: I vendor implementano dietro flag di funzionalità, una suite di test condivisa assicura comportamento identico su tutti i browser.
6. **Finalizzazione**: W3C la chiama Raccomandazione, TC39 la chiama Stage 4.

### Ciclo di vita di una funzionalità standard web

1. **Proposta**: Problema identificato e documentato.
2. **Incubazione**: Discussione aperta, molte proposte muoiono qui.
3. **Specificazione**: Specifica formale scritta con regole esatte.
4. **Test di interoperabilità**: La suite W3C Web Platform Tests verifica la coerenza cross-browser.
5. **Finalizzazione**: La funzionalità viene rilasciata senza flag come parte della piattaforma web di base.
6. **Deprecazione**: Funzionalità superate o insicure marcate per rimozione, ma i browser le mantengono per retrocompatibilità.

### Principi chiave degli standard web

- **Apertura**: Gli standard sono sviluppati pubblicamente e implementati senza royalty.
- **Accessibilità**: Integrata in HTML, CSS e comportamento del browser, WCAG fornisce linee guida formali.
- **Interoperabilità**: Una pagina costruita secondo standard dovrebbe comportarsi identicamente su tutti i browser.
- **Retrocompatibilità**: Le vecchie pagine devono continuare a funzionare nei browser moderni.
- **Privacy e sicurezza**: Nuove API che potrebbero esporre dati utente sono sottoposte a rigorosi controlli.
- **Stratificazione**: HTML per struttura, CSS per presentazione, JavaScript per comportamento. Ogni livello può essere adottato indipendentemente.
- **Progressive enhancement**: Le funzionalità devono funzionare prima per tutti, poi migliorare per utenti con browser più capaci.

## Comprendere REST API e servizi web

### Servizi web

- **API (Application Programming Interface)**: Meccanismo generale per software di comunicare con altro software.
- **Servizio web**: Tipo specifico di API che usa protocolli web (HTTP/HTTPS) per scambiare dati su rete. Ogni servizio web è un'API, ma non ogni API è un servizio web.
- **Caratteristiche**: Gira su un server, abilita comunicazione macchina-macchina, indipendente da linguaggio e piattaforma, usa protocolli web standard.
- **REST**: Trasferimento di stato rappresentazionale, lo stile di servizio web più usato. Semplice, flessibile, funziona naturalmente con HTTP.
- **SOAP**: Simple Object Access Protocol, usa solo XML, comune in sistemi enterprise che richiedono standard rigorosi.
- **XML-RPC**: Usa XML per codificare richieste e risposte leggere.

### Architettura REST

- **Senza stato**: Ogni richiesta è gestita indipendentemente. Il server non conserva stato di sessione tra richieste. Ogni richiesta deve includere tutte le informazioni necessarie (es. token di autenticazione a ogni chiamata).
- **Risorse**: Tutto è una risorsa identificata da un URL, es. `/products` per una collezione, `/products/1` per un elemento specifico.
- **Metodi HTTP in REST**: `GET` (recuperare), `POST` (creare), `PUT`/`PATCH` (aggiornare), `DELETE` (rimuovere).
- **Formati di risposta**: JSON è il più comune ma sono supportati anche XML e testo semplice. I client specificano il formato tramite l'intestazione `Accept`:

```bash
Accept: application/json
```

### Microservizi

- **Monolite**: Un grande codice base dove UI, database e API risiedono insieme. Difficile da mantenere e scalare con l'aumentare della complessità.
- **Microservizi**: Approccio architetturale dove una grande applicazione è suddivisa in piccoli servizi indipendenti, ognuno responsabile di una singola funzionalità di business. I servizi comunicano tramite API o code di messaggi.
- **Vantaggi**: Sviluppo e distribuzione indipendenti più veloci, migliore isolamento dei guasti, servizi diversi possono usare linguaggi o framework diversi.
- **Svantaggi**: Debugging più complesso (i problemi si estendono tra servizi), maggiore overhead infrastrutturale, aumento della comunicazione di rete.

## Lavorare con Express

- **Express.js**: Un framework web minimale e flessibile costruito sopra Node.js. Semplifica la creazione di server web e API con molto meno codice boilerplate rispetto al modulo `http` nativo.
- **Un server Express di base**:

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**: Funzione factory che crea un'istanza di applicazione Express (`app`).
- **`app.listen(port, callback)`**: Avvia il server e inizia ad ascoltare le richieste HTTP sulla porta specificata.
- **`req`**: L'oggetto richiesta, contiene URL, intestazioni, parametri, corpo e dati di query.
- **`res`**: L'oggetto risposta, usato per inviare dati al client.
- **Middleware**: Codice che gira tra la ricezione di una richiesta e l'invio di una risposta.

## Comprendere il routing in Express.js

- **Routing**: Definisce come un'app risponde quando un client visita un URL specifico con un metodo HTTP specifico.
- **`app.get/post/put/delete(path, handler)`**: Registra gestori per i rispettivi metodi HTTP.
- **Rotta statica**: Un pattern URL fisso, es. `app.get("/home", handler)`.
- **Rotta dinamica con parametri**: Usa il segnaposto `:paramName`. Catturato tramite `req.params`:

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**: Parametri di rotta (segmenti `:id`).
- **`req.query`**: Parametri della stringa di query (`?key=value`).
- **`req.body`**: Corpo della richiesta (richiede middleware per parsing del corpo).
- **`res.send(data)`**: Invia una risposta (testo, HTML, JSON o binario).
- **`res.json(obj)`**: Invia una risposta JSON, imposta automaticamente `Content-Type: application/json`.
- **`res.status(code).send(msg)`**: Imposta il codice di stato e invia una risposta.
- **`res.redirect(url)`**: Reindirizza il client a un altro URL.
- **`res.render(view, data)`**: Renderizza una vista di un motore template e invia HTML.

### Gestori di rotta concatenabili con `app.route()`

Raggruppa più metodi HTTP per lo stesso percorso per ridurre ripetizioni:

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

Crea una mini-app modulare per raggruppare rotte correlate:

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

### Servire file statici

- **`express.static(directory)`**: Middleware integrato che serve file da una directory specificata:

```js
app.use(express.static(path.join(__dirname, "public")));
```

- Servire da più directory con un percorso di mount: `app.use("/images", express.static(path.join(__dirname, "images")));`

## Comprendere la gestione degli errori e i controlli di salute

### Gestione degli errori in Express

- **Middleware per la gestione degli errori**: Identificato da quattro parametri `(err, req, res, next)`. Sempre posizionato alla fine dello stack di middleware:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **Errori asincroni in Express 4**: Devono essere catturati manualmente con `try/catch` e inoltrati con `next(err)`:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **Errori asincroni in Express 5**: Inoltrati automaticamente al middleware di gestione errori, non serve `next(err)` manuale.
- **Gestore 404**: Posizionato dopo tutte le rotte ma prima del gestore errori:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Debugging e logging

- **Debug integrato**: Abilita impostando la variabile d'ambiente `DEBUG`: `DEBUG=express:* node index.js`
- **`morgan`**: Middleware popolare per il logging delle richieste HTTP. Registra metodo, URL, codice di stato e tempo di risposta:

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### Controlli di salute e spegnimenti graduali

- **Rotta di controllo dello stato di salute**: Restituisce `200 OK` per segnalare che l'app è in esecuzione. Usata da bilanciatori di carico e orchestratori di container:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**: Segnale inviato quando un processo deve terminare. Gestiscilo per permettere alle richieste in corso di completarsi:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**: Segnale inviato premendo Ctrl+C durante lo sviluppo locale. Gestiscilo allo stesso modo di `SIGTERM`.
- **Segnalare lo stato di malfunzionamento prima dello spegnimento**: Restituisci un `503` dall'endpoint di controllo dello stato durante lo spegnimento per impedire al bilanciatore di carico di indirizzare nuovo traffico:

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## Middleware di Express

- **Middleware**: Una funzione che viene eseguita durante il ciclo di vita di una richiesta. Ha accesso a `req`, `res` e `next`. Chiama `next()` per passare il controllo al middleware successivo, omettendolo si termina il ciclo.
- **Middleware a livello di applicazione**: Associato all'intera app tramite `app.use()`. Viene eseguito per ogni richiesta in arrivo.
- **Middleware a livello di router**: Associato a una specifica istanza di `express.Router()` tramite `router.use()`. Si applica solo alle rotte gestite da quel router.
- **Middleware per la gestione degli errori**: Identificato da quattro parametri `(err, req, res, next)`. Posizionato alla fine della catena di middleware.
- **`express.json()`**: Analizza le richieste in arrivo con payload JSON. Rende i dati disponibili su `req.body`.
- **`express.urlencoded({ extended: true })`**: Analizza i dati dei moduli codificati in URL.
- **`express.static('public')`**: Serve file statici da una directory.
- **`cors`**: Abilita il Cross-Origin Resource Sharing. Installa con `npm install cors`, usa con `app.use(cors())`.
- **`morgan`**: Registra le richieste HTTP. Usa `app.use(morgan('tiny'))`.
- I middleware vengono applicati nell'ordine in cui sono registrati con `app.use()`.

## Comprendere i WebSocket

- **WebSocket**: Un protocollo che crea una connessione persistente, full-duplex (bidirezionale) tra client e server, abilitando la comunicazione in tempo reale.
- **Problema di HTTP per il tempo reale**: HTTP richiede che il client interroghi ripetutamente il server per nuovi dati, quindi è lento e inefficiente per app di chat, tabelloni live e giochi multiplayer.
- **Handshake WebSocket**: Ogni connessione WebSocket inizia con una richiesta HTTP. Il client richiede un upgrade e il server risponde con `101 Switching Protocols`. Dopo di che la connessione rimane aperta finché una delle due parti non invia un segnale di chiusura.
- **Casi d'uso**: Applicazioni di chat, giochi multiplayer online, dashboard sportive live, feed di borsa, strumenti collaborativi.

### La libreria `ws`

- Node.js non ha un server WebSocket integrato; la libreria `ws` è il modo comune per implementarne uno.

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**: Invia un messaggio a un client specifico.
- **Broadcast a tutti i client connessi**:

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**: Proprietà di ogni connessione. `WebSocket.OPEN` indica che la connessione è attiva.

### Architettura Pub/Sub

- **Pub/Sub (Publish/Subscribe)**: Un pattern di messaggistica in cui **publisher** inviano messaggi a **topic**, e **subscriber** ricevono messaggi dai topic a cui si sono iscritti. Publisher e subscriber non comunicano direttamente.
- **Vantaggi rispetto alla comunicazione diretta**: Disaccoppia i componenti, un publisher non deve sapere chi sono i suoi subscriber.
- **Pub/Sub vs WebSocket**: I WebSocket sono il canale di comunicazione (la strada). Pub/Sub è il pattern di messaggistica che determina chi riceve quali messaggi (il sistema del traffico).

## Introduzione all'autenticazione e autorizzazione

### Autenticazione vs autorizzazione

- **Autenticazione**: Verifica l'identità, risponde alla domanda “Chi sei?”. Deve avvenire prima dell'autorizzazione.
- **Autorizzazione**: Determina i permessi, risponde alla domanda “Cosa ti è permesso fare?”.
- **Tre fattori di autenticazione**:
  - **Qualcosa che sai**: Password, PIN, risposta a domanda di sicurezza.
  - **Qualcosa che hai**: Smartphone che riceve un codice SMS, token di app autenticatrice.
  - **Qualcosa che sei**: Impronta digitale, riconoscimento facciale (biometria).
- **Ruoli comuni nell'autorizzazione**: Utente standard (self-service limitato), Editor (gestione contenuti), Admin (accesso completo al sistema).
- Nel codice back-end, l'autorizzazione è solitamente applicata tramite middleware che verifica ruoli o permessi prima di concedere l'accesso.

### Cross-Site Request Forgery (CSRF)

- **CSRF**: Un attacco in cui un sito malevolo inganna il browser di un utente autenticato per fare una richiesta indesiderata a un server back-end, sfruttando l'allegato automatico dei cookie da parte del browser.
- **Flusso dell'attacco**: L'utente effettua il login su `trusted-bank.com` → visita `evil-hacker-site.com` → il sito malevolo invia un modulo nascosto alla banca → il server della banca vede il cookie di sessione valido ed esegue la richiesta.
- **`SameSite=Strict`**: Il browser non invia mai il cookie se la richiesta proviene da un sito terzo.
- **`SameSite=Lax`**: Il browser consente i cookie solo quando l'utente naviga in modo sicuro verso il sito target (cliccando un link), ma li blocca nelle sottomissioni di moduli cross-site.
- **Token CSRF**: Il server genera un token casuale e imprevedibile incorporato nei moduli. Il client deve includere il token in ogni richiesta che modifica lo stato. Un sito malevolo non può leggere il token dal frontend, quindi le sue richieste contraffatte falliranno la validazione.

### Token web JSON (JWT)

- **JWT**: Un token firmato digitalmente emesso dal server dopo un login riuscito. Il client lo conserva e lo invia con ogni richiesta successiva, abilitando l'autenticazione senza stato (non serve memorizzare sessioni lato server).
- **Struttura del JWT**: Tre parti codificate Base64URL separate da punti:
  - **Header**: Tipo di token (`JWT`) e algoritmo di firma (es. `HS256`).
  - **Payload**: Dichiarazioni sull'utente (id utente, username, ruolo). Non è criptato, quindi non memorizzare mai password o dati sensibili qui.
  - **Firma**: Hash di header + payload + chiave segreta del server. Previene manomissioni.

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

- Se un utente manomette il payload, la firma non corrisponderà e il server rifiuterà la richiesta con `401 Unauthorized`.

### Passport.js

- **Passport.js**: Middleware di autenticazione per Node.js. Standardizza il flusso di autenticazione e gestisce i dettagli di basso livello così gli sviluppatori possono concentrarsi sulla logica di business.
- **Strategie**: Plugin modulari che implementano metodi di autenticazione specifici. Oltre 500 strategie disponibili: `passport-local` (username/password), `passport-jwt` (JWT), `passport-google-oauth20` (login Google), `passport-facebook`, `passport-github2`.
- **Integrazione con Express**:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **Uso come middleware di rotta**:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- Dopo l'autenticazione riuscita, Passport memorizza un dato utente nel cookie di sessione e allega l'oggetto utente autenticato a `req.user`.

### Helmet.js

- **Helmet.js**: Una raccolta di middleware per la sicurezza che protegge le app Express impostando, modificando o rimuovendo automaticamente intestazioni HTTP di risposta.

```js
const helmet = require('helmet');
app.use(helmet());
```

- **Rimuove `X-Powered-By`**: Impedisce agli attaccanti di conoscere la tecnologia del server.
- **Imposta `Content-Security-Policy`**: Limita da dove possono essere caricati script, immagini e stili, mitigando attacchi XSS.
- **Imposta `X-Frame-Options: SAMEORIGIN`**: Previene il clickjacking bloccando l'incorporamento del sito in iframe esterni.
- **Imposta `Strict-Transport-Security`**: Costringe i browser a usare HTTPS, proteggendo i dati dall'intercettazione.

## Comprendere sicurezza e privacy nelle applicazioni web

### Sicurezza vs privacy

- **Sicurezza**: I meccanismi tecnici di difesa che proteggono sistemi e dati da accessi non autorizzati. La sicurezza fornisce la serratura sulla porta.
- **Privacy**: La gestione di come i dati personali vengono raccolti, usati e condivisi. La privacy fornisce la tenda sulla finestra.
- Puoi avere sicurezza senza privacy (un database blindato può comunque vendere i tuoi dati), ma non puoi avere privacy senza sicurezza (dati non protetti non possono rimanere privati).

### Same-Origin Policy e CORS

- **Origin**: Definita da tre componenti: il protocollo (ad esempio, `https`), il dominio (ad esempio, `example.com`) e la porta (ad esempio, `443`). Tutti e tre devono corrispondere affinché due URL condividano la stessa origin.
- **Same-origin policy**: Una regola di sicurezza del browser che blocca gli script di un'origine dal leggere dati da un'origine diversa, impedendo ai siti malevoli di rubare dati da altre schede aperte.
- **CORS (Cross-Origin Resource Sharing)**: Un meccanismo che permette ai server di rilassare la same-origin policy per origini fidate impostando intestazioni HTTP.
- **In Express (sviluppo)**: `app.use(cors())` permette tutte le origini.
- **In Express (produzione)**: Passa un oggetto `corsOptions` con una lista di origini consentite `origin`:

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**: HTTP con crittografia fornita da TLS (Transport Layer Security). TLS ha sostituito il vecchio protocollo SSL.
- **Tre garanzie di HTTPS**:
  - **Crittografia**: I dati inviati tra client e server sono illeggibili a terzi.
  - **Autenticazione**: Un certificato digitale da un'autorità di certificazione affidabile conferma che il server è chi dice di essere.
  - **Integrità dei dati**: I messaggi non possono essere alterati in transito senza essere rilevati.
- HTTPS è richiesto per le API moderne del browser come geolocalizzazione, notifiche push e accesso alla fotocamera.

### Perché si verificano problemi di sicurezza

- **Pressione per consegne rapide**: Team che corrono troppo per rispettare le scadenze saltano le revisioni di sicurezza e rilasciano vulnerabilità.
- **Integrazione di sistemi complessi**: Il rischio della supply chain nasce quando librerie di terze parti o servizi cloud introducono vulnerabilità non scritte dal team di sviluppo.
- **Gestione della configurazione scadente**: Lasciare impostazioni predefinite o di fabbrica (password admin predefinite, porte aperte) espone inutilmente i sistemi.
- **Errore umano**: Attacchi di phishing sfruttano personale che clicca link o allegati malevoli e consegna credenziali.
- **Fusioni organizzative**: Combinare codebase e infrastrutture diverse mette in luce politiche di sicurezza non allineate e vulnerabilità legacy.

### Cookie: memorizzazione, flag di sicurezza e privacy

- **Come sono memorizzati i cookie**: Il server invia un'intestazione `Set-Cookie`. Il browser salva il file su disco e lo restituisce automaticamente a ogni richiesta successiva a quell'origine.
- **Flag `HttpOnly`**: Blocca JavaScript dal leggere il valore del cookie, prevenendo il furto di sessione via XSS.
- **Flag `Secure`**: Restringe il cookie alle connessioni HTTPS, proteggendolo dall'intercettazione su reti pubbliche.
- **`SameSite=Strict`**: Impedisce al browser di inviare il cookie in qualsiasi richiesta cross-site, bloccando completamente gli attacchi CSRF.
- **`SameSite=Lax`**: Permette i cookie nelle navigazioni di primo livello (es. clic su link) ma li blocca nelle sottomissioni di moduli e chiamate API cross-site.
- **Cookie di prima parte**: Limitati al sito che stai visitando.
- **Cookie di terze parti**: Impostati da reti pubblicitarie su migliaia di siti per tracciare comportamenti e creare profili utente. I browser moderni li bloccano per impostazione predefinita.
- **Conformità GDPR**: Le leggi sulla privacy richiedono banner di consenso ai cookie che danno agli utenti il diritto legale di rifiutare cookie di tracciamento non essenziali.

### Minacce comuni alla sicurezza e mitigazioni

- **Phishing**: Email fraudolente che si spacciano per aziende affidabili per rubare credenziali. Le mitigazioni includono formazione sulla sicurezza e protocolli di autenticazione email per bloccare messaggi malevoli prima che arrivino nelle caselle.
- **Malware** (virus, spyware, ransomware): Il ransomware cripta file e chiede un riscatto per sbloccarli. Le mitigazioni includono aggiornamenti automatici e software antimalware.
- **Credential stuffing**: Attaccanti testano milioni di combinazioni rubate di username e password su siti diversi. La mitigazione principale è l'autenticazione a più fattori (MFA), che richiede un secondo passaggio di verifica come OTP o codice da app autenticatrice.
- **DDoS (Distributed Denial of Service)**: Attaccanti inondano un server con traffico falso per farlo crashare e bloccare utenti legittimi. Le mitigazioni includono servizi cloud di filtraggio traffico come Cloudflare, Akamai e AWS Shield.
- **Minacce interne**: Dipendenti malevoli o negligenti che rubano o perdono dati. Le mitigazioni includono l'applicazione del **principio del minimo privilegio** (i lavoratori accedono solo a ciò che serve per il loro lavoro) e il monitoraggio dei log di sistema per attività insolite.

### Content Security Policy e Permissions-Policy

- **CSP (Content Security Policy)**: Un'intestazione di risposta HTTP che autorizza domini sorgente approvati per script, stili e immagini. Quando un browser riceve un'intestazione CSP, blocca qualsiasi risorsa caricata da un dominio non in lista. La direttiva `script-src` specifica quali domini possono eseguire JavaScript, fermando attacchi di iniezione XSS.
- **Permissions-Policy**: Un'intestazione HTTP che abilita o disabilita funzionalità del browser e hardware del dispositivo per una pagina e per eventuali iframe incorporati. Ha sostituito la vecchia intestazione `Feature-Policy`.
  - `camera=(self)` limita l'accesso alla webcam solo all'origine della pagina.
  - `geolocation=()` disabilita completamente la geolocalizzazione per tutti gli utenti.
- Combinando entrambe le intestazioni si ottiene una protezione a più livelli: CSP previene l'iniezione di codice e Permissions-Policy previene accessi hardware non autorizzati.

### Concetti fondamentali di privacy per l'utente

- **PII (Personally Identifiable Information)**: Qualsiasi dato che può identificare una persona direttamente o indirettamente.
  - **Identificatori diretti**: Nome completo, indirizzo di casa, numero di identificazione nazionale.
  - **Identificatori indiretti**: Indirizzo IP, numero seriale del dispositivo, cronologia precisa della posizione.
- **Riservatezza**: Mantenere i PII al sicuro da accessi non autorizzati tramite crittografia e controlli di accesso.
- **Minimizzazione dei dati**: Raccogliere solo il minimo indispensabile di PII necessario per completare un'attività specifica.
- **Limitazione dello scopo**: I dati raccolti per una ragione non possono essere riutilizzati (ad esempio venduti a reti pubblicitarie) senza nuovo consenso dell'utente.
- **Tracciamento**: Monitoraggio continuo del comportamento online dell'utente tramite script, fingerprinting del dispositivo e cronologia del browser.
- **Diritto all'oblio (cancellazione dati)**: Il diritto legale di richiedere la cancellazione permanente di tutti i dati personali da database aziendali dopo la chiusura di un account.

### Il file `.env` e `dotenv`

- **File `.env`**: Un file di testo semplice nella radice del progetto che memorizza segreti di configurazione come chiavi API, password del database e numeri di porta come coppie chiave-valore. Aggiungilo sempre a `.gitignore` per tenere i segreti fuori dal controllo versione.
- **Pacchetto `dotenv`**: Legge il file `.env` all'avvio e inietta i valori in `process.env`:

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **Due benefici chiave**: Le credenziali restano fuori dal codice sorgente (sicurezza), e lo stesso codice può comportarsi diversamente in sviluppo, staging e produzione cambiando il file `.env` su ogni server (flessibilità).

### Leggi regionali sulla privacy e conformità

- **GDPR (Regolamento generale sulla protezione dei dati)**: Protegge le persone nell'UE. Richiede consenso esplicito prima del tracciamento, senza caselle preselezionate. Concede agli utenti il diritto di vedere e cancellare definitivamente i propri dati. Prevede pesanti sanzioni finanziarie per le violazioni.
- **CCPA (California Consumer Privacy Act) / CPRA (California Privacy Rights Act)**: Protegge i residenti in California. Si concentra sui diritti di opt-out più che sull'opt-in. Richiede un link visibile “Do Not Sell My Personal Information” nella homepage.
- **COPPA (Children's Online Privacy Protection Act)**: Legge federale USA che protegge i bambini sotto i 13 anni. Richiede una chiara informativa sulla privacy e il consenso verificabile dei genitori prima di raccogliere dati personali da un bambino.
- **DPA (Data Protection Act)**: Il principale quadro normativo sulla privacy dati del Regno Unito. Opera insieme al GDPR UK e concede agli utenti il diritto di accedere e cancellare dati personali.
- Altri paesi hanno le proprie leggi: Nigeria ha la NDPA (Nigeria Data Protection Act) e India ha il DPDP Act (Digital Personal Data Protection Act).
- **Passaggi di conformità per team globali**:
  - Costruire banner di consenso consapevoli della posizione geografica che mostrano l'avviso cookie appropriato in base alla posizione dell'utente.
