# --description--

Wir haben die `.gitignore`-Datei bereits kurz in einer vorherigen Lektion angesprochen. Aber was genau ist diese Datei, und was sollten Sie darin eintragen?

Die `.gitignore`-Datei ist eine spezielle Art von Datei, die mit Git-Operationen zusammenhängt. Der Name deutet darauf hin, dass diese Datei Git mitteilt, Dinge zu ignorieren, und das ist der häufigste Anwendungsfall. Tatsächlich bewirkt sie aber, dass Git aufhört, eine Datei zu verfolgen. Dieser Unterschied ist wichtig und wird später noch erläutert.

Was sollten Sie also tatsächlich in diese Datei eintragen? Nun, Sie sollten Dinge auflisten, die Sie nicht committen möchten.

Das Erste wären Ihre Geheimdateien. Das könnte zum Beispiel eine `.env`-Datei sein, in der Sie Ihre API-Schlüssel für Ihre Anwendung gespeichert haben. Es könnte eine private GPG-Schlüsseldatei sein, die Ihre Anwendung für die Authentifizierung benötigt. Alles, was sensible Informationen enthält, die Sie nicht öffentlich machen möchten, sollte mit `.gitignore` ignoriert werden. Selbst wenn Ihr Repository privat ist, sollten Sie so handeln, als wäre es öffentlich – behandeln Sie es so, als könnte es jeder sehen, und achten Sie darauf, keine Geheimnisse zu committen.

Schauen wir uns an, wie unsere Datei jetzt aussehen könnte:

```sh
# Secrets
.env
github.key
```

Wir haben Git angewiesen, die `.env`-Datei zu ignorieren, in der wir unsere Geheimnisse aufbewahren. Außerdem haben wir Git angewiesen, eine `github.key`-Datei zu ignorieren, die einen privaten GPG-Schlüssel enthält.

Als Nächstes sollten Sie die Pakete oder Abhängigkeiten berücksichtigen, die Ihr Projekt verwendet. Zum Beispiel installieren Sie bei einem Node.js-Projekt wahrscheinlich Pakete von npm. Diese Pakete landen in einem `node_modules`-Ordner in Ihrem Projekt, damit Ihr Code sie importieren und verwenden kann.

Der `node_modules`-Ordner wird jedoch sehr groß, und Sie möchten auf keinen Fall alle diese Pakete committen und die Änderungen bei jedem Update verfolgen. Stattdessen committen Sie Ihre `package.json`-Datei und die relevante Lockdatei (z. B. `package-lock.json`, `pnpm-lock.yaml`). Diese Dateien sagen npm, welche Pakete und Versionen installiert werden sollen, sodass ein Mitwirkender dieselben Abhängigkeiten installieren kann, ohne sie ins Repository zu committen.

Fügen wir also `node_modules` zu unserer `.gitignore`-Datei hinzu:

```sh
# Secrets
.env
github.key

# Packages
node_modules
```

Kommen wir nun zu einer wichtigen Syntaxregel für `.gitignore`-Dateien. Wenn Sie einen Eintrag wie `node_modules` hinzufügen, ignoriert Git alles, was diesem Muster entspricht. Das bedeutet, es ignoriert das `node_modules`-Verzeichnis im Stammverzeichnis Ihres Projekts, aber auch das `node_modules` im `client`-Verzeichnis Ihres Projekts.

Wenn Sie nur etwas im Stammverzeichnis ignorieren möchten, können Sie die Zeile mit einem Schrägstrich voranstellen. `node_modules` würde zu `/node_modules`. Das würde das `node_modules`-Verzeichnis im Stamm ignorieren, aber nicht das `client/node_modules`-Verzeichnis. Das wollen wir nicht, also lassen wir unseren Eintrag einfach als `node_modules`.

Als Nächstes sollten Sie Ihre Build-Ausgaben ignorieren. Zum Beispiel kann das Kompilieren eines TypeScript-Projekts zu einem `dist`-Ordner mit dem JavaScript-Ausgabe-Code führen. Sie möchten diese Ausgabe nicht committen, da sie die Git-Historie aufbläht. Stattdessen sollten Sie diese Ausgabe ignorieren – Mitwirkende können sie selbst reproduzieren, indem sie denselben Build-Befehl ausführen.

Ignorieren wir also den `dist`-Ordner:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
```

Manchmal kann eine fehlplatzierte Datei dazu führen, dass der TypeScript-Compiler abstürzt, was dazu führt, dass unsere JavaScript-Ausgabe überall landet. Da wir ausschließlich TypeScript schreiben, verwenden wir Glob-Syntax, um alle JavaScript-Dateien zu ignorieren:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js
```

Andere häufig ignorierte Ausgaben sind das `coverage`-Verzeichnis für die Ergebnisse der Unit-Test-Abdeckung, ein `.angular`-Verzeichnis, wenn Sie Angular verwenden, ein `.next`-Verzeichnis, wenn Sie NextJS verwenden, und so weiter.

Für den Moment bleiben wir bei dem, was wir hier haben. Wenn Sie ein Framework verwenden, das Dateien generiert, sollten Sie in dessen Dokumentation nachsehen, was Sie zur `.gitignore` hinzufügen müssen.

Die vierte Gruppe von Dingen, die Sie ignorieren sollten, sind IDE-Dateien. Jetbrains-Produkte verwenden zum Beispiel einen `.idea`-Ordner für IDE-spezifische Konfigurationen. Visual Studio Code verwendet einen `.vscode`-Ordner.

Diese Dateien sind oft gut zu ignorieren, da sie Dateien enthalten, die spezifisch für Ihre lokale Umgebung sind und andere Mitwirkende nicht benötigen (weil sie ihre eigenen Konfigurationen verwenden). Fügen wir also das Visual Studio Code-Verzeichnis hinzu, da wir es als IDE verwenden.

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
```

Unser `.vscode`-Ordner enthält jedoch eine `settings.json`-Datei, die einige arbeitsbereichsspezifische Dinge enthält, die für unsere Mitwirkenden nützlich sein könnten – diese Datei ermöglicht es ihnen, ihre Erweiterungen und Umgebung so zu konfigurieren, dass sie besser unserer eigenen entsprechen.

Verwenden wir also Negationssyntax, um diese Datei „nicht mehr zu ignorieren“:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json
```

Wenn Sie MacOS verwenden, kennen Sie vielleicht die `.DS_STORE`-Datei. Diese Datei enthält Metadaten, die Finder verwendet, um zu bestimmen, wie das Verzeichnis dargestellt wird (z. B. mit Miniaturansichten oder als Liste). Windows-Benutzer sehen möglicherweise eine versteckte `Thumbs.db`-Datei, die Miniaturansichten für das Verzeichnis zwischenspeichert.

Es ist generell eine gute Idee, diese Dateien zu ignorieren, da es sich um persönliche Einstellungen handelt, die keine Auswirkungen auf das Projekt haben. Fügen wir beide hinzu:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json

# System Files
.DS_STORE
Thumbs.db
```

Ein kurzer Hinweis zum Ignorieren von bereits committeten Dateien. Angenommen, Sie haben versehentlich Ihr `dist`-Verzeichnis früher committet, möchten es jetzt aber entfernen und ignorieren. Sie können es nicht einfach zu Ihrer `.gitignore`-Datei hinzufügen – das lässt bereits committete Inhalte nicht verschwinden, sondern sagt Git nur, die Datei nicht mehr zu verfolgen. Wenn Sie also eine bereits committete Datei ignorieren, verfolgt Git keine Änderungen mehr, aber auch nicht das Löschen der Datei.

Stattdessen müssen Sie die Datei zuerst löschen. Erstellen Sie einen Commit mit dieser Löschung. Dann können Sie den Eintrag zu Ihrer `.gitignore` hinzufügen und einen zweiten Commit mit dieser Aktualisierung erstellen. Von nun an wird Git die Datei ignorieren, wenn sie wieder erstellt wird, und sie wird nicht mehr in Ihren Commits auftauchen.

Schließlich, wenn Sie an einem Projekt in einer anderen Sprache arbeiten, wie Python, unterscheiden sich die zu ignorierenden Elemente von dem, was wir hier behandelt haben. GitHub pflegt ein sehr hilfreiches Repository mit `.gitignore`-Vorlagen für verschiedene Sprachen und Frameworks. Sie finden diese Liste unter <a href="https://github.com/github/gitignore" target="_blank">https://github.com/github/gitignore</a>.

Damit sind Sie jetzt bereit, Dateien zu vermeiden, die Sie nicht in Ihren Repositories committen möchten!
