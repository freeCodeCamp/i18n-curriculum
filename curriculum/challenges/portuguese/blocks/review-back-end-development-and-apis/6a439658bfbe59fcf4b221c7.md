## Trabalhando com Node.js e arquitetura orientada a eventos

- **Node.js**: Um ambiente de execução JavaScript open-source e multiplataforma que permite rodar código JavaScript fora do navegador. É amplamente usado para construir servidores web, APIs e sistemas back-end.
- **Limitações do lado do cliente**: JavaScript no cliente tem acesso restrito ao sistema de arquivos, suporte limitado para lógica complexa de aplicação e riscos potenciais de segurança (por exemplo, você não deve expor credenciais do banco de dados no código do cliente).
- **Objeto global no navegador vs. Node.js**: No navegador, o objeto global é `window`, que expõe manipulação do DOM, cookies e eventos do navegador. No Node, o objeto global é `global`, que expõe módulos embutidos para arquivos, rede e interação com o sistema operacional.
- **Acesso ao DOM**: O navegador fornece acesso à API do DOM, mas o Node não. Por outro lado, o Node pode acessar quase todos os recursos do sistema, incluindo o sistema de arquivos, enquanto o navegador não pode.
- **Controle de versão**: Você pode escolher qual versão do Node roda no seu servidor. Você não tem controle sobre qual versão do navegador seus usuários usam.
- **Arquitetura não bloqueante e orientada a eventos**: O Node usa uma única thread e um loop de eventos que gerencia eficientemente muitas requisições simultâneas e operações de E/S sem bloquear, tornando-o ideal para aplicações em tempo real.
- **Callbacks**: Funções que definem o que acontece quando uma operação assíncrona é concluída. A forte dependência delas é parte do motivo pelo qual o código assíncrono em Node pode ser mais difícil de ler e depurar.
- **Vantagens do Node no back-end**: Uma única linguagem (JavaScript) no front-end e back-end, arquitetura não bloqueante adequada para alta concorrência, grande ecossistema npm, gratuito e open-source.
- **Desvantagens do Node no back-end**: Single-threaded, então tarefas intensivas de CPU (processamento de imagens, criptografia, matemática complexa) podem bloquear o loop de eventos, forte dependência de código assíncrono/baseado em callbacks pode reduzir a legibilidade, pacotes npm variam em qualidade e manutenção.
- **NVM (Node Version Manager)**: Uma ferramenta para gerenciar e alternar entre múltiplas versões do Node.js na mesma máquina. No macOS/Linux instale com:

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **Instalando a versão LTS do Node**: `nvm install --lts`
- **Alternando versões do Node**: `nvm use 20`
- **Listando todas as versões do Node instaladas**: `nvm ls`
- **Definindo uma versão padrão do Node (macOS/Linux)**: `nvm alias default 22.20.0`
- **nvm-windows**: Um projeto separado para gerenciar versões do Node no Windows (não WSL). Baixe o instalador do repositório oficial e execute `nvm install lts` e depois `nvm use <version>`.
- **Executando um arquivo JavaScript com Node**: `node app.js`

## Trabalhando com módulos core do Node

### O módulo `fs`

- **Propósito**: Fornece métodos para criar, ler, escrever, anexar e excluir arquivos e diretórios.
- **Três estilos de uso**: Baseado em callback (assíncrono), baseado em promise (`fs/promises`) e síncrono (sufixo `Sync`). Métodos síncronos bloqueiam o loop de eventos e são indicados apenas para scripts.
- **`writeFile()`**: Escreve conteúdo em um arquivo, criando-o se não existir:

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**: Anexa conteúdo a um arquivo existente sem sobrescrevê-lo.
- **`readFile()`**: Lê o conteúdo do arquivo. Passe `"utf8"` como codificação para obter uma string, ou omita para obter um `Buffer`.
- **`unlink()`**: Exclui um arquivo. Lança `ENOENT` se o arquivo não existir.

### O módulo `Buffer`

- **Propósito**: Manipula dados binários (arquivos, imagens, fluxos de rede) diretamente na memória.
- **`Buffer.from()`**: Cria um buffer a partir de uma string:

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**: Cria um buffer preenchido com zeros de `n` bytes:

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **Truncamento**: Escrever mais dados do que o buffer pode armazenar trunca silenciosamente o excesso.
- **`Buffer.byteLength(str)`**: Retorna o número de bytes necessários para armazenar uma string.

### O módulo `crypto`

- **Propósito**: Fornece ferramentas criptográficas como hashing, HMAC, criptografia/descriptografia, assinaturas digitais e geração segura de números aleatórios.
- **`createHash(algorithm)`**: Cria um hash unidirecional (irreversível). Use `.update(data).digest("hex")` para fornecer os dados a serem hasheados e retornar o hash como string hexadecimal:

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**: Semelhante ao `createHash` mas requer uma chave secreta, adequado para verificar integridade dos dados e autenticação.
- **`createCipheriv()` / `createDecipheriv()`**: Criptografam e descriptografam dados. Ambos requerem um algoritmo, uma chave e um vetor de inicialização (IV):

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**: Gera bytes aleatórios criptograficamente seguros, mais seguro que `Math.random()` para tokens e chaves.
- **`randomInt(min, max)`**: Gera um inteiro aleatório seguro no intervalo dado. Útil para OTPs.
- **`sign()` / `verify()`**: Criam e verificam assinaturas digitais usando par de chaves privada/pública.

### O módulo `os`

- **Propósito**: Expõe informações sobre o sistema operacional onde o Node está rodando.
- **`os.platform()`**: Retorna o identificador do SO, ex: `'darwin'`, `'win32'`, `'linux'`.
- **`os.arch()`**: Retorna a arquitetura da CPU, ex: `'x64'`, `'arm64'`.
- **`os.type()`**: Retorna o nome oficial do SO, ex: `'Darwin'`.
- **`os.release()`**: Retorna a versão do kernel do SO.
- **`os.cpus()`**: Retorna um array de objetos descrevendo cada núcleo lógico da CPU (modelo, velocidade, tempos).
- **`os.uptime()`**: Retorna o tempo de atividade do sistema em segundos.
- **`os.totalmem()` / `os.freemem()`**: Retornam a memória total e disponível do sistema em bytes.
- **`os.userInfo()`**: Retorna um objeto com `uid`, `gid`, `username`, `homedir` e `shell` do usuário atual.
- **`os.networkInterfaces()`**: Retorna objetos das interfaces de rede com endereço, família, MAC e detalhes CIDR.

### O módulo `path`

- **Propósito**: Fornece utilitários para trabalhar com caminhos de arquivos e diretórios de forma independente da plataforma.
- **`__filename`**: Variável global, caminho absoluto do arquivo atual.
- **`__dirname`**: Variável global, caminho absoluto do diretório que contém o arquivo atual.
- **`path.basename(p)`**: Retorna o último componente de um caminho (nome do arquivo).
- **`path.dirname(p)`**: Retorna a parte do diretório de um caminho.
- **`path.extname(p)`**: Retorna a extensão do arquivo, ex: `'.js'`.
- **`path.join(...segments)`**: Junta segmentos de caminho em um caminho normalizado e corrige barras erradas.
- **`path.resolve(...segments)`**: Resolve segmentos em um caminho absoluto a partir do diretório de trabalho atual.
- **`path.parse(p)`**: Retorna um objeto com propriedades `root`, `dir`, `base`, `ext` e `name`.
- **`path.format(obj)`**: Constrói uma string de caminho a partir de um objeto com `dir`, `name` e `ext`.

### O módulo `process`

- **Propósito**: Expõe informações e controle sobre o processo Node.js atual. É um objeto global, não requer importação.
- **`process.env`**: Objeto contendo todas as variáveis de ambiente. Acesse valores específicos como `process.env.NODE_ENV`.
- **`process.argv`**: Array de argumentos da linha de comando. Índice 0 é o executável Node, índice 1 é o caminho do script, índices 2+ são argumentos do usuário.
- **`process.cwd()`**: Retorna o diretório de trabalho atual.
- **Eventos do processo**:

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**: Dispara um aviso customizado que pode ser capturado por um listener de `warning`.

### O módulo `stream`

- **Propósito**: Processa grandes quantidades de dados em pedaços ao invés de carregar tudo na memória de uma vez.
- **Quatro tipos de stream**: Readable (lê dados em pedaços), Writable (escreve dados em pedaços), Duplex (ambos), Transform (um Duplex que modifica dados enquanto passam).
- **Encadeamento de streams**:

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **Evento `finish`**: Disparado em uma stream gravável quando todos os dados foram enviados.
- **Evento `error`**: Disparado quando uma stream encontra um problema.

## Introdução ao npm

- **npm**: Composto por três partes: o site `npmjs.com`, o registro público (um banco de dados de pacotes) e a ferramenta CLI instalada com o Node.js.
- **Propósito**: Permite instalar, gerenciar e publicar pacotes JavaScript reutilizáveis. npm é o gerenciador de pacotes mais usado.
- **Alternativas**: Yarn, PNPM, Bun.

### O arquivo `package.json`

- **Propósito**: O manifesto de configuração do npm. Rastreia nome do projeto, versão, descrição, ponto de entrada (`main`), autor, licença, scripts e dependências.
- **Criar `package.json` interativamente**: `npm init`
- **Criar com padrões**: `npm init -y`
- **`dependencies`**: Pacotes necessários para rodar a aplicação em produção (ex: `express`, `react`).
- **`devDependencies`**: Pacotes usados apenas no desenvolvimento e testes (ex: `jest`, `nodemon`, `eslint`).
- **`license`**: Termos legais para uso. Valores comuns: `MIT`, `ISC`, `GPL`.

### Versionamento semântico (SemVer)

- **Formato**: `MAJOR.MINOR.PATCH`, ex: `4.17.21`.
- **MAJOR**: Incrementado para mudanças incompatíveis.
- **MINOR**: Incrementado para novos recursos compatíveis com versões anteriores.
- **PATCH**: Incrementado para correções de bugs compatíveis.
- **Caret `^`**: Permite atualizações para versões minor e patch, mas não major. Exemplo: `^4.17.21` aceita `4.18.0` mas não `5.0.0`.
- **Tilde `~`**: Permite atualizações apenas para patch. Exemplo: `~1.2.3` aceita `1.2.4` mas não `1.3.0`.
- **Asterisco `*`**: Aceita qualquer versão, adequado apenas para testes, nunca produção.
- **Versão exata**: Sem prefixo, ex: `1.2.3` fixa exatamente essa versão.

### Instalando e removendo dependências

- **Instalar um pacote**: `npm install express` (abreviação: `npm i express`)
- **Instalar como devDependency**: `npm install nodemon -D`
- **Desinstalar um pacote**: `npm uninstall chalk`
- **`node_modules/`**: Pasta criada após instalação contendo o código real dos pacotes. Não deve ser commitada no controle de versão.
- **`package-lock.json`**: Arquivo gerado automaticamente que bloqueia as versões exatas de todos os pacotes instalados (incluindo dependências filhas). Sempre commit este arquivo e não edite manualmente.
- **Instalar uma versão específica**: `npm install express@4.21.2`
- **Ver versões disponíveis**: `npm view express versions`
- **Verificar pacotes desatualizados**: `npm outdated`
- **Atualizar um pacote específico** (dentro do intervalo permitido): `npm update express`
- **Forçar instalação da última versão**: `npm install express@latest`

### Publicando no registro npm

- **Login**: `npm login`
- **Pacotes com escopo**: Use `@username/package-name` para evitar colisão de nomes.
- **`.npmignore`**: Lista arquivos a excluir do pacote publicado (semelhante ao `.gitignore`).
- **Publicar pacote sem escopo**: `npm publish`
- **Publicar pacote com escopo publicamente**: `npm publish --access public`

## Trabalhando com scripts npm

- **Scripts npm**: Comandos personalizados definidos no objeto `"scripts"` do `package.json`. Automatizam tarefas repetitivas como iniciar servidor, rodar testes, linting e build.
- **Definindo um script**:

```json
"scripts": {
  "start": "node app.js"
}
```

- **Executando um script**: `npm run start`
- **Abreviação para scripts embutidos** (`start`, `test`, `stop`, `restart`): `npm start`
- **Comandos sequenciais com `&&`** (próximo comando roda só se o anterior tiver sucesso):

```json
"start": "npm run build && node server.js"
```

- **Comandos sequenciais com `;`** (próximo comando sempre roda):

```json
"start": "npm run build; node server.js"
```

- **Comandos concorrentes com `&`**: `"dev": "npm run start:server & npm run start:client"`
- **Passando argumentos para um script** (use separador `--`): `npm start -- 8000`; argumentos acessíveis via `process.argv[2]`.

### Módulos CommonJS

- **CommonJS**: O sistema original de módulos do Node.js. Carrega módulos de forma síncrona.
- **Importando** com `require()`:

```js
const { multiply } = require('./math');
```

- **Exportando** com `module.exports`:

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- CommonJS é o padrão quando o `package.json` tem `"type": "commonjs"` (ou não tem o campo `"type"`), ou o arquivo tem extensão `.cjs`.

### Módulos ES (ESM)

- **ES Modules**: O formato moderno e padronizado de módulos. Carregados de forma assíncrona.
- **Exportação nomeada**: `export function multiply(a, b) { return a * b; }`
- **Exportação padrão**: `export default function multiply(a, b) { return a * b; }`
- **Importação**: `import { multiply } from './math.mjs';`
- ESM é usado quando o arquivo tem extensão `.mjs` ou o `package.json` contém `"type": "module"`.

## Entendendo como HTTP, DNS e TCP/IP funcionam

### Servidores e o modelo cliente-servidor

- **Modelo cliente-servidor**: Arquitetura de sistema distribuído onde os componentes são clientes (processos que enviam requisições, ex: navegador) ou servidores.
- **Servidor**: Computador ou programa que recebe requisições dos clientes e retorna dados ou serviços pela rede.
- **Tipos de servidores**: Servidores web (HTML/CSS/assets), servidores de aplicação (lógica de negócio), servidores de banco de dados, servidores de email, servidores de arquivos, servidores proxy.

### DNS

- **DNS (Domain Name System)**, frequentemente chamado de "lista telefônica da internet": Traduz nomes de domínio legíveis por humanos (ex: `freecodecamp.org`) em endereços IP.
- **Hierarquia de nomes de domínio** (da direita para a esquerda): Domínio de topo (TLD, ex: `.org`) → Domínio de segundo nível (ex: `freecodecamp`) → Subdomínio (ex: `www`).
- **Endereço IP**: Identificador numérico único para um dispositivo na rede. Exemplo IPv4: `127.0.0.1`. Exemplo IPv6: `0:0:0:0:0:0:0:1`.
- **Processo de resolução DNS**: Navegador → Resolvedor Recursivo → Servidores Raiz → Servidor TLD → Servidor Autoritativo → retorna endereço IP → Resolvedor Recursivo armazena em cache e entrega ao cliente.

### TCP/IP e HTTP

- **IP (Internet Protocol)**: Roteia pacotes de dados da origem ao destino usando endereços IP. Sem conexão, cada pacote é enviado independentemente.
- **TCP (Transmission Control Protocol)**: Divide dados em segmentos numerados, reagrupa no destino, verifica erros e solicita retransmissão se necessário. Garante entrega confiável e ordenada.
- **HTTP (HyperText Transfer Protocol)**: Protocolo da camada de aplicação que define como clientes e servidores trocam conteúdo web.
- **HTTPS**: Versão segura do HTTP onde os dados são criptografados usando SSL/TLS.

### Estrutura da requisição HTTP

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **Linha de requisição**: Método HTTP + caminho/URI + versão HTTP.
- **Cabeçalhos**: Pares chave-valor de metadados.
- **Corpo** (opcional): Enviado com `POST`, `PUT`, `PATCH`, `DELETE` para transportar dados.

### Estrutura da resposta HTTP

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **Linha de status**: Versão HTTP + código de status + frase de motivo.
- **Cabeçalhos**: Metadados sobre a resposta (tipo de conteúdo, comprimento, diretivas de cache).
- **Corpo** (opcional): Os dados solicitados.

### Categorias de códigos de status HTTP

- **1xx Informativo**: Requisição recebida, processamento continua. Exemplo: `101 Switching Protocols`.
- **2xx Sucesso**: `200 OK`, `201 Created`, `204 No Content`.
- **3xx Redirecionamento**: `301 Moved Permanently`, `302 Found`, `304 Not Modified`.
- **4xx Erros do cliente**: `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.
- **5xx Erros do servidor**: `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable`.

## Entendendo o modelo requisição-resposta HTTP

- **Métodos HTTP**: `GET` (recuperar), `POST` (criar), `PUT` (atualização completa), `PATCH` (atualização parcial), `DELETE` (remover).
- **Ciclo requisição-resposta**: Cliente envia requisição → Servidor valida autenticação, verifica método e URL, consulta banco de dados → Servidor constrói resposta (código de status + corpo opcional) → Cliente processa resposta.

### Assets retornados em respostas HTTP

Toda resposta inclui um cabeçalho `Content-Type` que informa ao navegador como tratar o asset:

- **HTML**: `text/html`
- **CSS**: `text/css`
- **JavaScript**: `application/javascript`
- **JSON**: `application/json`
- **PNG/JPEG/WebP**: `image/png`, `image/jpeg`, `image/webp`
- **MP4/MP3**: `video/mp4`, `audio/mpeg`
- **PDF**: `application/pdf`
- **WOFF2**: `font/woff2`

### Envio de formulários HTML

- **Atributo `action`**: Especifica a URL para onde os dados do formulário são enviados.
- **Atributo `method`**: Especifica o método HTTP (`GET` ou `POST`). HTML não suporta nativamente `PUT`, `PATCH` ou `DELETE`, use a Fetch API ou o pacote `method-override` no Express.
- **Validação no cliente**: Use os atributos `required` e `pattern` para feedback imediato. Sempre valide também no servidor.
- **Formulário `GET`**: Dados anexados à URL como parâmetros de consulta (ex: formulários de busca).
- **Formulário `POST`**: Dados enviados no corpo da requisição (ex: formulários de login e cadastro).

## Entendendo o modelo de padrões web

### Organizações de padrões

- **WHATWG**: Mantém HTML e DOM como um "padrão vivo" continuamente atualizado. Decisões são tomadas por grandes equipes de motores de navegador (Chromium, Gecko, WebKit).
- **W3C (World Wide Web Consortium)**: Principal órgão de padrões. Supervisiona CSS, SVG, diretrizes de acessibilidade (WCAG) e várias APIs web.
- **ECMA International / TC39**: Ecma publica a especificação oficial do ECMAScript. TC39 desenvolve a linguagem por meio de um processo de cinco estágios (Estágio 0 ideia inicial → Estágio 4 especificação completa).
- **Khronos Group**: Focado em gráficos, computação e mídia. Produz WebGL e WebGL 2 para renderização 3D acelerada por GPU em navegadores.

### Criando padrões web

1. **Identificação de necessidade**: Um desenvolvedor ou organização encontra uma dificuldade e documenta.
2. **Proposta e discussão inicial**: Issue informal no GitHub ou explicação para medir interesse. No TC39 isso é Estágio 0/1.
3. **Escrita da especificação**: Documento formal preciso cobrindo todos os casos, erros e implicações de segurança.
4. **Revisão, objeção e iteração**: Rascunho revisado publicamente por fornecedores de navegador; recursos recusados por um navegador principal raramente viram padrão.
5. **Implementação e testes**: Fornecedores implementam atrás de flag de recurso; suíte de testes compartilhada garante comportamento idêntico.
6. **Finalização**: W3C chama de Recomendação, TC39 chama de Estágio 4.

### Ciclo de vida de um recurso padrão web

1. **Proposta**: Problema identificado e documentado.
2. **Incubação**: Discussão aberta, muitas propostas morrem aqui.
3. **Especificação**: Especificação formal escrita com regras exatas.
4. **Testes de interoperabilidade**: Suíte W3C Web Platform Tests verifica consistência entre navegadores.
5. **Finalização**: Recurso lançado sem flags como parte da plataforma web base.
6. **Depreciação**: Recursos substituídos ou inseguros marcados para remoção, mas navegadores mantêm para compatibilidade retroativa.

### Princípios chave dos padrões web

- **Abertura**: Padrões são desenvolvidos abertamente e implementados sem royalties.
- **Acessibilidade**: Incorporada em HTML, CSS e comportamento do navegador; WCAG fornece diretrizes formais.
- **Interoperabilidade**: Uma página construída conforme padrão deve funcionar igual em todos os navegadores.
- **Compatibilidade retroativa**: Páginas antigas devem continuar funcionando em navegadores modernos.
- **Privacidade e segurança**: Novas APIs que podem expor dados do usuário passam por rigorosa análise.
- **Camadas**: HTML para estrutura, CSS para apresentação, JavaScript para comportamento. Cada camada pode ser adotada independentemente.
- **Aprimoramento progressivo**: Recursos devem funcionar para todos primeiro, depois melhorar para usuários com navegadores mais capazes.

## Entendendo REST APIs e web services

### Web services

- **API (Application Programming Interface)**: Mecanismo geral para software se comunicar com outro software.
- **Web service**: Tipo específico de API que usa protocolos web (HTTP/HTTPS) para trocar dados pela rede. Todo web service é uma API, mas nem toda API é um web service.
- **Características**: Roda em servidor, permite comunicação máquina a máquina, independente de linguagem e plataforma, usa protocolos web padrão.
- **REST**: Transferência de estado representacional, estilo de web service mais usado. Simples, flexível, funciona naturalmente com HTTP.
- **SOAP**: Protocolo simples de acesso a objetos, usa apenas XML, comum em sistemas empresariais que precisam de padrões rígidos.
- **XML-RPC**: Usa XML para codificar requisições e respostas leves.

### Arquitetura REST

- **Sem estado**: Cada requisição é tratada independentemente. O servidor não armazena estado de sessão entre requisições. Cada requisição deve incluir todas as informações necessárias (ex: token de autenticação em toda chamada).
- **Recursos**: Tudo é um recurso identificado por uma URL, ex: `/products` para coleção, `/products/1` para item específico.
- **Métodos HTTP no REST**: `GET` (recuperar), `POST` (criar), `PUT`/`PATCH` (atualizar), `DELETE` (remover).
- **Formatos de resposta**: JSON é o mais comum, mas XML e texto simples também são suportados. Clientes especificam formato via cabeçalho `Accept`:

```bash
Accept: application/json
```

### Microsserviços

- **Monólito**: Uma grande base de código onde UI, banco de dados e API residem juntos. Difícil de manter e escalar conforme cresce a complexidade.
- **Microsserviços**: Abordagem arquitetural onde uma grande aplicação é dividida em pequenos serviços independentes, cada um responsável por uma única capacidade de negócio. Serviços se comunicam via APIs ou filas de mensagens.
- **Vantagens**: Desenvolvimento e implantação independentes mais rápidos, melhor isolamento de falhas, diferentes serviços podem usar linguagens ou frameworks diferentes.
- **Desvantagens**: Depuração mais complexa (problemas abrangem serviços), maior overhead de infraestrutura, aumento da comunicação em rede.

## Trabalhando com Express

- **Express.js**: Um framework web minimalista e flexível construído sobre Node.js. Simplifica a criação de servidores web e APIs com muito menos boilerplate que o módulo nativo `http`.
- **Um servidor Express básico**:

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**: Função fábrica que cria uma instância de aplicação Express (`app`).
- **`app.listen(port, callback)`**: Inicia o servidor e começa a escutar requisições HTTP na porta especificada.
- **`req`**: Objeto de requisição, contém URL, cabeçalhos, params, corpo e dados de consulta.
- **`res`**: Objeto de resposta, usado para enviar dados de volta ao cliente.
- **Middleware**: Código que roda entre receber uma requisição e enviar uma resposta.

## Entendendo roteamento no Express.js

- **Roteamento**: Define como uma aplicação responde quando um cliente visita uma URL específica com um método HTTP específico.
- **`app.get/post/put/delete(path, handler)`**: Registra manipuladores para os respectivos métodos HTTP.
- **Rota estática**: Um padrão de URL fixo, ex: `app.get("/home", handler)`.
- **Rota dinâmica com parâmetros**: Usa marcador `:paramName`. Capturado via `req.params`:

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**: Parâmetros de rota (`:id` segmentos).
- **`req.query`**: Parâmetros da string de consulta (`?key=value`).
- **`req.body`**: Corpo da requisição (requer middleware para análise do corpo).
- **`res.send(data)`**: Envia uma resposta (texto, HTML, JSON ou binário).
- **`res.json(obj)`**: Envia uma resposta JSON, define automaticamente `Content-Type: application/json`.
- **`res.status(code).send(msg)`**: Define o código de status e envia uma resposta.
- **`res.redirect(url)`**: Redireciona o cliente para outra URL.
- **`res.render(view, data)`**: Renderiza uma view de template engine e envia HTML.

### Manipuladores de rota encadeáveis com `app.route()`

Agrupa múltiplos métodos HTTP para o mesmo caminho para reduzir repetição:

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

Cria uma mini-aplicação modular para agrupar rotas relacionadas:

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

### Servindo arquivos estáticos

- **`express.static(directory)`**: Middleware embutido que serve arquivos de um diretório especificado:

```js
app.use(express.static(path.join(__dirname, "public")));
```

- Servir de múltiplos diretórios com caminho de montagem: `app.use("/images", express.static(path.join(__dirname, "images")));`

## Entendendo tratamento de erros e health checks

### Tratamento de erros no Express

- **Middleware de tratamento de erros**: Identificado por quatro parâmetros `(err, req, res, next)`. Sempre colocado no final da pilha de middleware:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **Erros assíncronos no Express 4**: Devem ser capturados manualmente com `try/catch` e encaminhados com `next(err)`:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **Erros assíncronos no Express 5**: Encaminhados automaticamente para middleware de erro, não é necessário `next(err)` manual.
- **Manipulador 404**: Coloque após todas as rotas mas antes do middleware de erro:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Depuração e logging

- **Depuração embutida**: Habilite definindo a variável de ambiente `DEBUG`: `DEBUG=express:* node index.js`
- **`morgan`**: Middleware popular de logging de requisições HTTP. Registra método, URL, código de status e tempo de resposta:

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### Health checks e desligamentos suaves

- **Rota de verificação de integridade**: Retorna `200 OK` para sinalizar que o app está funcionando. Usada por balanceadores de carga e orquestradores de container:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**: Sinal enviado quando um processo deve terminar. Trate-o para permitir que requisições em andamento sejam concluídas:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**: Sinal enviado ao pressionar Ctrl+C durante o desenvolvimento local. Trate da mesma forma que `SIGTERM`.
- **Marcar como não saudável antes do desligamento**: Retorne um `503` no endpoint de verificação de integridade durante o desligamento para impedir que o balanceador de carga direcione novo tráfego:

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## Middleware do Express

- **Middleware**: Uma função que executa durante o ciclo de vida de uma requisição. Tem acesso a `req`, `res` e `next`. Chame `next()` para passar o controle para o próximo middleware, omita para encerrar o ciclo.
- **Middleware em nível de aplicação**: Vinculado ao app inteiro via `app.use()`. Executa para toda requisição recebida.
- **Middleware em nível de roteador**: Vinculado a uma instância específica de `express.Router()` via `router.use()`. Aplica-se apenas às rotas desse roteador.
- **Middleware de tratamento de erros**: Identificado por quatro parâmetros `(err, req, res, next)`. Colocado no final da pilha.
- **`express.json()`**: Analisa requisições com payload JSON. Disponibiliza os dados em `req.body`.
- **`express.urlencoded({ extended: true })`**: Analisa dados de formulário codificados em URL.
- **`express.static('public')`**: Serve arquivos estáticos de um diretório.
- **`cors`**: Habilita Cross-Origin Resource Sharing. Instale com `npm install cors`, use com `app.use(cors())`.
- **`morgan`**: Registra requisições HTTP. Use `app.use(morgan('tiny'))`.
- Middleware é aplicado na ordem em que é registrado com `app.use()`.

## Entendendo WebSockets

- **WebSockets**: Um protocolo que cria uma conexão persistente e full-duplex (bidirecional) entre cliente e servidor, permitindo comunicação em tempo real.
- **Problema do HTTP para tempo real**: HTTP exige que o cliente faça polling repetido ao servidor para novos dados, tornando-o lento e ineficiente para apps de chat, placares ao vivo e jogos multiplayer.
- **Handshake do WebSocket**: Toda conexão WebSocket começa com uma requisição HTTP. O cliente solicita um upgrade e o servidor responde com `101 Switching Protocols`. Depois disso, a conexão permanece aberta até que qualquer lado envie um sinal de fechamento.
- **Casos de uso**: Aplicativos de chat, jogos multiplayer online, dashboards esportivos ao vivo, feeds do mercado financeiro, ferramentas colaborativas.

### A biblioteca `ws`

- Node.js não possui servidor WebSocket embutido; a biblioteca `ws` é a forma comum de implementar um.

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**: Envia uma mensagem para um cliente específico.
- **Broadcast para todos os clientes conectados**:

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**: Propriedade em toda conexão. `WebSocket.OPEN` significa que a conexão está ativa.

### Arquitetura Pub/Sub

- **Pub/Sub (Publicar/Assinar)**: Um padrão de mensagens onde **publicadores** enviam mensagens para **tópicos**, e **assinantes** recebem mensagens dos tópicos aos quais se inscreveram. Publicadores e assinantes não se comunicam diretamente.
- **Vantagens sobre comunicação direta**: Desacopla componentes, um publicador não precisa saber quem são seus assinantes.
- **Pub/Sub vs. WebSockets**: WebSockets são o canal de comunicação (a estrada). Pub/Sub é o padrão de mensagens que determina quem recebe quais mensagens (o sistema de tráfego).

## Introdução à autenticação e autorização

### Autenticação vs. autorização

- **Autenticação**: Verifica identidade, responde à pergunta "Quem é você?", deve ocorrer antes da autorização.
- **Autorização**: Determina permissões, responde à pergunta "O que você tem permissão para fazer?".
- **Três fatores de autenticação**:
  - **Algo que você sabe**: Senha, PIN, resposta a pergunta de segurança.
  - **Algo que você tem**: Smartphone recebendo código SMS, token de app autenticador.
  - **Algo que você é**: Impressão digital, reconhecimento facial (biometria).
- **Papéis comuns na autorização**: Usuário padrão (autoatendimento limitado), Editor (gerenciamento de conteúdo), Admin (acesso total ao sistema).
- No código backend, a autorização geralmente é aplicada via middleware que verifica papéis ou permissões antes de conceder acesso.

### Falsificação de requisição entre sites (CSRF)

- **CSRF**: Um ataque onde um site malicioso engana o navegador de um usuário autenticado para fazer uma requisição indesejada a um servidor backend, explorando o envio automático de cookies pelo navegador.
- **Fluxo do ataque**: Usuário faz login em `trusted-bank.com` → visita `evil-hacker-site.com` → site malicioso envia um formulário oculto para o banco → servidor do banco vê cookie de sessão válido e executa a requisição.
- **`SameSite=Strict`**: Navegador nunca envia o cookie se a requisição vier de um site terceiro.
- **`SameSite=Lax`**: Navegador permite cookies apenas quando o usuário navega com segurança para o site alvo (clicando em um link), mas bloqueia em envios de formulário cross-site.
- **Tokens CSRF**: O servidor gera um token aleatório e imprevisível incorporado em formulários. O cliente deve incluir o token em toda requisição que altera estado. Um site malicioso não pode ler o token do frontend, então suas requisições forjadas falham na validação.

### Tokens web JSON (JWT)

- **JWT**: Um token assinado digitalmente emitido pelo servidor após login bem-sucedido. O cliente o armazena e envia em todas as requisições subsequentes, permitindo autenticação sem estado (sem armazenamento de sessão no servidor).
- **Estrutura do JWT**: Três partes codificadas em Base64URL separadas por pontos:
  - **Cabeçalho**: Tipo do token (`JWT`) e algoritmo de assinatura (ex.: `HS256`).
  - **Payload**: Declarações sobre o usuário (id, nome, papel). Não é criptografado, então nunca armazene senhas ou dados sensíveis aqui.
  - **Assinatura**: Hash do cabeçalho + payload + chave secreta do servidor. Evita adulteração.

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

- Se um usuário adulterar o payload, a assinatura não corresponderá e o servidor rejeitará a requisição com `401 Unauthorized`.

### Passport.js

- **Passport.js**: Middleware de autenticação para Node.js. Padroniza o fluxo de autenticação e lida com detalhes de baixo nível para que desenvolvedores foquem na lógica de negócio.
- **Estratégias**: Plugins modulares que implementam métodos específicos de autenticação. Mais de 500 estratégias disponíveis: `passport-local` (usuário/senha), `passport-jwt` (JWTs), `passport-google-oauth20` (login Google), `passport-facebook`, `passport-github2`.
- **Integração com Express**:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **Uso como middleware de rota**:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- Após autenticação bem-sucedida, Passport armazena um dado do usuário no cookie de sessão e anexa o objeto do usuário autenticado em `req.user`.

### Helmet.js

- **Helmet.js**: Uma coleção de middleware de segurança que protege apps Express configurando, modificando ou removendo automaticamente cabeçalhos HTTP de resposta.

```js
const helmet = require('helmet');
app.use(helmet());
```

- **Remove `X-Powered-By`**: Impede que atacantes saibam a tecnologia do servidor.
- **Define `Content-Security-Policy`**: Restringe de onde scripts, imagens e estilos podem ser carregados, mitigando ataques XSS.
- **Define `X-Frame-Options: SAMEORIGIN`**: Evita clickjacking bloqueando seu site de ser incorporado em iframes externos.
- **Define `Strict-Transport-Security`**: Obriga navegadores a usarem HTTPS, protegendo dados contra interceptação.

## Entendendo segurança e privacidade em aplicações web

### Segurança vs. privacidade

- **Segurança**: Mecanismos técnicos de defesa que protegem sistemas e dados contra acesso não autorizado. Segurança é a trava na porta.
- **Privacidade**: Governança sobre como dados pessoais são coletados, usados e compartilhados. Privacidade é a cortina na janela.
- Você pode ter segurança sem privacidade (um banco de dados trancado pode vender seus dados), mas não pode ter privacidade sem segurança (dados desprotegidos não permanecem privados).

### Política de mesma origem e CORS

- **Origem**: Definida por três componentes: protocolo (ex.: `https`), domínio (ex.: `example.com`) e porta (ex.: `443`). Todos os três devem coincidir para que duas URLs compartilhem a mesma origem.
- **Política de mesma origem**: Regra de segurança do navegador que bloqueia scripts de uma origem de ler dados de outra, prevenindo que sites maliciosos roubem dados de outras abas abertas.
- **CORS (Cross-Origin Resource Sharing)**: Mecanismo que permite aos servidores relaxar a política de mesma origem para origens confiáveis definindo cabeçalhos HTTP.
- **No Express (desenvolvimento)**: `app.use(cors())` permite todas as origens.
- **No Express (produção)**: Passe um objeto `corsOptions` com uma lista de origens permitidas (`origin`):

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**: HTTP com criptografia fornecida por TLS (Transport Layer Security). TLS substituiu o protocolo SSL antigo.
- **Três garantias do HTTPS**:
  - **Criptografia**: Dados enviados entre cliente e servidor ficam ilegíveis para terceiros.
  - **Autenticação**: Um certificado digital de uma Autoridade Certificadora confiável confirma que o servidor é quem diz ser.
  - **Integridade dos dados**: Mensagens não podem ser alteradas em trânsito sem detecção.
- HTTPS é obrigatório para APIs modernas do navegador como geolocalização, notificações push e acesso à câmera.

### Por que ocorrem problemas de segurança

- **Pressão por entrega rápida**: Equipes que correm para cumprir prazos pulam revisões de segurança e lançam vulnerabilidades.
- **Integração complexa de sistemas**: Risco na cadeia de suprimentos surge quando bibliotecas de terceiros ou serviços em nuvem introduzem vulnerabilidades que a equipe de desenvolvimento não escreveu.
- **Má gestão de configuração**: Deixar configurações padrão ou de fábrica (senhas admin padrão, portas abertas) expõe sistemas desnecessariamente.
- **Erro humano**: Ataques de phishing exploram funcionários que clicam em links ou anexos maliciosos e entregam credenciais.
- **Fusões organizacionais**: Combinar diferentes bases de código e infraestrutura expõe políticas de segurança incompatíveis e vulnerabilidades legadas.

### Cookies: armazenamento, flags de segurança e privacidade

- **Como cookies são armazenados**: O servidor envia um cabeçalho `Set-Cookie`. O navegador salva o arquivo no disco e o retorna automaticamente em toda requisição subsequente para aquela origem.
- **Flag `HttpOnly`**: Bloqueia o JavaScript de ler o valor do cookie, prevenindo roubo de sessão via XSS.
- **Flag `Secure`**: Restringe o cookie a conexões HTTPS, protegendo-o contra interceptação em redes públicas.
- **`SameSite=Strict`**: Impede que o navegador envie o cookie em qualquer requisição cross-site, bloqueando totalmente ataques CSRF.
- **`SameSite=Lax`**: Permite cookies em navegações de nível superior (ex.: clique em link) mas bloqueia em envios de formulário e chamadas API cross-site.
- **Cookies de primeira parte**: Escopados ao site que você está visitando.
- **Cookies de terceiros**: Definidos por redes de publicidade em milhares de sites para rastrear comportamento e criar perfis. Navegadores modernos os bloqueiam por padrão.
- **Conformidade com GDPR**: Leis de privacidade exigem banners de consentimento de cookies que dão aos usuários o direito legal de rejeitar cookies de rastreamento não essenciais.

### Ameaças comuns de segurança e mitigação

- **Phishing**: E-mails fraudulentos que se passam por empresas confiáveis para roubar credenciais. Mitigações incluem treinamento de conscientização em segurança e implantação de protocolos de autenticação de e-mail para bloquear mensagens maliciosas antes de chegarem às caixas de entrada.
- **Malware** (vírus, spyware, ransomware): Ransomware criptografa arquivos e exige pagamento para desbloqueá-los. Mitigações incluem atualizações automáticas e software antimalware.
- **Credential stuffing**: Atacantes testam milhões de combinações roubadas de usuário e senha em vários sites. A principal mitigação é autenticação multifator (MFA), que exige uma segunda etapa de verificação como OTP ou código de app autenticador.
- **DDoS (Distributed Denial of Service)**: Atacantes inundam um servidor com tráfego falso para derrubá-lo e bloquear usuários legítimos. Mitigações incluem serviços de filtragem de tráfego baseados em nuvem como Cloudflare, Akamai e AWS Shield.
- **Ameaças internas**: Funcionários maliciosos ou descuidados que roubam ou vazam dados. Mitigações incluem aplicar o **princípio do menor privilégio** (funcionários acessam apenas o que o trabalho exige) e monitorar logs do sistema para atividades incomuns.

### Content Security Policy e Permissions-Policy

- **CSP (Content Security Policy)**: Um cabeçalho HTTP de resposta que lista domínios aprovados para scripts, estilos e imagens. Quando o navegador recebe um cabeçalho CSP, bloqueia qualquer recurso carregado de um domínio não listado. A diretiva `script-src` especifica quais domínios podem executar JavaScript, impedindo ataques de injeção XSS.
- **Permissions-Policy**: Um cabeçalho HTTP que habilita ou desabilita recursos do navegador e hardware do dispositivo para uma página e quaisquer iframes incorporados. Substituiu o cabeçalho antigo `Feature-Policy`.
  - `camera=(self)` restringe o acesso à webcam apenas à origem da própria página.
  - `geolocation=()` desabilita geolocalização completamente para todos os usuários.
- Combinar ambos os cabeçalhos oferece proteção em camadas: CSP previne injeção de código e Permissions-Policy previne acesso não autorizado a hardware.

### Conceitos fundamentais de privacidade do usuário

- **PII (Informações Pessoais Identificáveis)**: Qualquer dado que possa identificar uma pessoa direta ou indiretamente.
  - **Identificadores diretos**: Nome completo, endereço residencial, número de identificação nacional.
  - **Identificadores indiretos**: Endereço IP, número de série do dispositivo, histórico preciso de localização.
- **Confidencialidade**: Manter PII segura contra acesso não autorizado por meio de criptografia e controles de acesso.
- **Minimização de dados**: Coletar apenas o mínimo absoluto de PII necessário para completar uma tarefa específica.
- **Limitação de propósito**: Dados coletados para um motivo não podem ser reutilizados (por exemplo, vendidos para redes de publicidade) sem novo consentimento do usuário.
- **Rastreamento**: Monitoramento contínuo do comportamento do usuário online usando scripts, fingerprinting de dispositivo e logs de histórico do navegador.
- **Direito ao esquecimento (apagamento de dados)**: Direito legal de solicitar a exclusão permanente de todos os registros pessoais dos bancos de dados de uma empresa após encerrar uma conta.

### O arquivo `.env` e o `dotenv`

- **Arquivo `.env`**: Um arquivo de texto simples na raiz do projeto que armazena segredos de configuração como chaves de API, senhas de banco de dados e números de porta como pares chave-valor. Sempre adicione ao `.gitignore` para manter segredos fora do controle de versão.
- **Pacote `dotenv`**: Lê o arquivo `.env` na inicialização e injeta os valores em `process.env`:

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **Dois benefícios principais**: Credenciais ficam fora do código fonte (segurança) e a mesma base de código pode se comportar de forma diferente em desenvolvimento, staging e produção trocando o arquivo `.env` em cada servidor (flexibilidade).

### Leis regionais de privacidade e conformidade

- **GDPR (Regulamento Geral sobre a Proteção de Dados)**: Protege indivíduos na UE. Exige consentimento explícito antes do rastreamento, sem caixas pré-marcadas. Concede aos usuários o direito de visualizar e excluir permanentemente seus dados. Impõe pesadas multas por violações.
- **CCPA (Lei de Privacidade do Consumidor da Califórnia) / CPRA (Lei de Direitos de Privacidade da Califórnia)**: Protege residentes da Califórnia. Foca em direitos de opt-out em vez de opt-in. Exige um link visível "Não vender minhas informações pessoais" na página inicial.
- **COPPA (Lei de Proteção à Privacidade Online das Crianças)**: Lei federal dos EUA que protege crianças menores de 13 anos. Exige política de privacidade clara e consentimento parental verificável antes de coletar dados pessoais de uma criança.
- **DPA (Lei de Proteção de Dados)**: Principal estrutura de privacidade de dados do Reino Unido. Funciona junto com o GDPR do Reino Unido e concede aos usuários o direito de acessar e excluir dados pessoais.
- Outros países têm suas próprias leis: Nigéria tem a NDPA (Lei de Proteção de Dados da Nigéria) e Índia tem a DPDP Act (Lei de Proteção de Dados Pessoais Digitais).
- **Passos para conformidade em equipes globais**:
