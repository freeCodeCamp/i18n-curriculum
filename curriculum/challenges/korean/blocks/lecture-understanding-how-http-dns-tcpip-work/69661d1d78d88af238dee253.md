# --description--

오늘날 우리가 아는 디지털 세계는 장치들이 서로 통신하며 필요에 따라 데이터를 주고받는다는 개념을 기반으로 합니다.

그런데 이 과정이 어떻게 내부적으로 작동하는지 생각해 본 적 있나요?

인터넷을 탐색할 때 네트워크를 통해 끊임없이 데이터를 요청하고 전송합니다. 이는 정보가 적절한 장치에 올바르게 전송되고 수신되도록 하는 표준 규칙과 절차에 기반한 상세한 과정 덕분에 가능합니다.

이러한 규칙과 절차의 집합을 **프로토콜**이라고 합니다.

프로토콜은 기본적으로 두 개 이상의 장치가 특정 표준, 예를 들어 교환될 데이터 형식을 모두 준수하도록 하여 서로 통신할 수 있게 합니다.

프로토콜은 데이터 교환 과정에 참여하는 모든 장치가 서로를 이해할 수 있도록 보장합니다. 비유하자면 모두가 “같은 언어를 말하는 법”을 아는 셈입니다.

핵심적으로 TCP/IP는 인터넷을 가능하게 하는 프로토콜입니다.

**TCP/IP**는 데이터가 네트워크를 통해 어떻게 패키징되고 전송되며 라우팅되고 수신되는지를 결정하는 통신 프로토콜 모음입니다. 정보가 신뢰성 있게 전달되도록 보장합니다.

특히 **IP**는 **Internet Protocol**의 약자로, 데이터가 네트워크 내에서 출발지에서 올바른 목적지로 가도록 보장하는 프로토콜입니다.

왜 이것이 중요할까요? 인터넷은 전 세계적으로 연결된 장치들의 네트워크임을 기억하세요. 장치가 요청을 보내려면 먼저 데이터를 어디로 보낼지 알아야 합니다. 이것이 IP 주소, 즉 인터넷 프로토콜(IP)의 기반 목적입니다. IP 주소는 네트워크 내 장치를 고유하게 식별하여 출발지 장치가 데이터를 어디로 보내야 할지 알게 합니다.

인터넷 프로토콜 자체는 연결 지향적이지 않습니다. 즉, 각 데이터 단위(“패킷”이라고 부름)가 네트워크를 통해 독립적으로 목적지로 전송되며, 사전에 연결을 설정하거나 올바른 순서로 전달될지 보장하지 않습니다. 패킷이 목적지에 도달하더라도 출발지에는 전혀 알리지 않습니다.

이때 TCP 같은 프로토콜이 매우 중요해집니다.

**TCP**는 **Transmission Control Protocol**로 알려져 있으며, 데이터를 작은 세그먼트로 나눕니다. 이 기본 데이터 단위들은 번호가 매겨지고 목적지에 도착하면 특정 알고리즘을 사용해 완전하고 올바른지 재조립합니다. 이 알고리즘은 오류와 손실된 패킷도 검사하여 필요 시 출발지에 재전송을 요청해 정보가 정확히 수신되도록 합니다.

Transmission Control Protocol과 Internet Protocol은 함께 인터넷을 통해 데이터가 신뢰성 있게 전송되도록 필요한 규칙과 표준을 정합니다.

TCP/IP 외에도 현대 인터넷에서 매우 중요한 또 다른 프로토콜이 있습니다: HTTP입니다.

HTTP는 애플리케이션 레벨에서 작동합니다.

**HTTP**는 **HyperText Transfer Protocol**의 약자로, 클라이언트(예: 브라우저)와 서버 간 통신을 위한 규칙과 표준을 정의합니다.

브라우저에서 웹사이트에 접속하려 할 때, 브라우저는 특정 자원과 관련 자산을 요청하는 HTTP 요청을 서버에 보냅니다. 서버는 HTTP 요청을 받고 로직을 실행한 뒤 요청한 콘텐츠나 요청이 실패한 이유를 담은 HTTP 응답을 클라이언트에 반환합니다.

HTTP 요청과 HTTP 응답은 매우 유사한 구조를 가집니다:

* 요청 라인(시작 라인)
    
* 헤더
    
* 본문(선택 사항)

## **HTTP 요청**

HTTP 요청의 **요청 라인**은 세 부분으로 나뉩니다:

* 요청되는 동작을 설명하는 메서드. 예를 들어, `GET`는 데이터를 가져오고 `POST`은 데이터를 제출합니다. 다른 일반적인 메서드로는 `PUT`, `PATCH`, `DELETE`, `HEAD`가 있습니다.
    
* 클라이언트가 요청하는 특정 자원을 설명하는 경로나 URI.
    
* 통신 과정에 사용되는 HTTP 프로토콜의 버전. 예를 들어 `HTTP/1.1`.

다음은 루트 `GET` 경로에서 `/` 메서드와 HTTP 버전 1.1을 사용한 예시입니다:

`GET / HTTP/1.1`

**헤더**는 요청에 관한 키-값 쌍 정보를 담고 있습니다.

예를 들어: `Content-Type: application/json`.

헤더 아래에는 요청에 관한 추가 정보인 메타데이터가 완성되었음을 나타내는 빈 줄이 있습니다.

마지막으로 **본문**이 있습니다. 선택 사항이며, 클라이언트가 서버에 정보를 보내야 할 때, 예를 들어 `POST` 메서드로 새 데이터를 제출할 때 전송됩니다.

다음은 HTTP 요청 예시입니다:

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

## **HTTP 응답**

HTTP 응답도 매우 유사한 구조를 가집니다:

* 상태 라인
    
* 헤더
    
* 본문
    

먼저, **상태 라인**은 요청 결과에 관한 정보를 담고 있습니다. 보통 다음을 포함합니다:

* 통신 과정에 사용되는 HTTP 버전. 예를 들어 `HTTP/1.1`.
    
* 요청 결과를 나타내는 상태 코드.
    
* 상태 코드를 간단히 설명하는 사람이 읽을 수 있는 이유 구문. 무슨 일이 있었는지 이해하는 데 도움이 됩니다.
    

일반적인 상태 코드는 다음과 같습니다:

* `200 OK`
    
* `201 CREATED`
    
* `400 BAD REQUEST`
    
* `404 NOT FOUND`
    
* `403 FORBIDDEN`
    
* `500 INTERNAL SERVER ERROR`
    

상태 코드는 매우 많으며, 왼쪽부터 첫 번째 숫자에 따라 분류됩니다.

* `1xx`는 정보성 응답입니다.
    
* `2xx`는 성공적인 응답입니다.
    
* `3xx`는 리다이렉션 메시지입니다.
    
* `4xx`는 클라이언트 오류 응답입니다.
    
* `5xx`는 서버 오류 응답입니다.
    

HTTP 응답의 상태 라인 아래에는 응답에 관한 메타데이터가 담긴 **헤더**가 있습니다. 예를 들어 `Content-Type: text/html`. 여기에는 콘텐츠 길이, 날짜, 위치 및 클라이언트가 유용하게 여길 추가 정보도 포함될 수 있습니다.

마지막으로 HTTP 응답 **본문**이 있습니다. 클라이언트가 요청한 데이터나 요청이 실패했을 경우 무슨 일이 있었는지 설명하는 내용을 담습니다. 선택 사항이지만 대부분 응답에 포함됩니다.

다음은 HTTP 응답 예시입니다:

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

HTTP에 대해 알게 되었으니 HTTPS에 대해 더 궁금할 수 있습니다.

HTTPS는 브라우저와 서버 간에 전송되는 데이터가 보안을 위해 암호화되는 “보안” 버전의 HTTP라고 생각할 수 있습니다. 대부분의 현대 웹 트래픽은 HTTPS를 사용합니다.

TCP/IP와 HTTP 같은 프로토콜 덕분에 월드 와이드 웹과 인터넷 전체가 우리가 매일 의존하는 필수 도구가 되었습니다. 이 덕분에 장치들이 네트워크를 통해 제대로 통신하고 전 세계에서 데이터를 교환할 수 있습니다.
