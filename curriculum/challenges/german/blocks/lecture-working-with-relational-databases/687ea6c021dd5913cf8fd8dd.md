# --description--

Eine relationale Datenbank ist eine Sammlung von Daten, die in Tabellen organisiert sind. Diese Tabellen bestehen aus Zeilen und Spalten.

Zeilen stellen einzelne Datensätze dar. Wenn Sie beispielsweise Benutzerdaten speichern, könnte jede Zeile einen bestimmten Benutzer darstellen.

Spalten repräsentieren die Attribute oder Felder, die jeden Datensatz beschreiben. Die Tabelle könnte Spalten für gängige Benutzerattribute enthalten, wie Name und Personalausweisnummer.

Eine der wichtigsten Eigenschaften relationaler Datenbanken ist, dass sie es ermöglichen, verschiedene Informationsstücke zu verbinden, indem Tabellen über gemeinsame Attribute verknüpft werden. So lässt sich leichter erkennen, wie sie zueinander in Beziehung stehen.

Relationale Datenbanken benötigen ein Schema. Das Schema definiert die Gesamtstruktur der Datenbank, einschließlich ihrer Tabellen, Spalten, Datentypen, Beziehungen und Einschränkungen.

Alle Tabellen in einer relationalen Datenbank haben einen Primärschlüssel, der ein eindeutiger Bezeichner für jede Zeile ist. Sie ermöglichen Beziehungen zwischen Tabellen über Fremdschlüssel. In den kommenden Lektionen lernen Sie mehr über Primärschlüssel, Fremdschlüssel und deren Funktionsweise. Für den Moment sollten Sie nur wissen, dass sie grundlegend sind, um Datensätze über Tabellen hinweg zu verknüpfen.

Zum Beispiel nehmen wir an, Sie erstellen eine Datenbank für eine Organisation, die ein Wildtierschutzzentrum für stark gefährdete Tiere betreibt.

Eine der Tabellen könnte allgemeine Informationen über die Tiere speichern, die Sie `animals` nennen könnten. Jede Zeile könnte ein Tier darstellen, während jede Spalte ein Attribut dieses Tieres repräsentiert, wie Art, Größe, Gewicht, Alter und andere einzigartige Merkmale.

Um jedes Tier eindeutig zu identifizieren, würden Sie typischerweise eine `id`-Spalte zur Tabelle hinzufügen. Diese wäre der Primärschlüssel der Tabelle, weil sie für jedes Tier einzigartig ist. Die ID jedes Tieres könnte verwendet werden, um seinen Datensatz in dieser Tabelle mit verwandten Datensätzen in anderen Tabellen zu verknüpfen.

Sie könnten auch eine `species`-Tabelle haben, um Informationen über jede Art zu speichern, wie ihren Lebensraum und ihren Schutzstatus. Jedes Tier in der `animals`-Tabelle könnte eine Art aus dieser Tabelle über eine Art-ID referenzieren.

Außerdem könnten Sie eine Tabelle erstellen, um die tierärztlichen Aufzeichnungen der Tiere zu speichern. So kann das Zentrum deren Gesundheit, Behandlungen und Medikamente verfolgen.

Sobald Sie dieses allgemeine Schema haben, könnten Sie diese Tabellen mit der `animals`-Tabelle verknüpfen, die als zentraler Punkt für tierbezogene Daten fungiert.

So müssen Sie nicht alle Informationen über ein Tier in derselben Tabelle speichern. Stattdessen können Sie verwandte Daten in separaten Tabellen speichern und sie über Beziehungen verknüpfen. Durch das Erstellen dieser Beziehungen können Sie mit einer einzigen Abfrage die spezifischen Daten abrufen, die Sie benötigen.

Eine Abfrage ist eine Anfrage nach bestimmten Daten aus der Datenbank. Beispiele für Abfragen, die Sie im Kontext unseres Beispiels stellen könnten, wären: alle Tiere finden, die eine tierärztliche Untersuchung benötigen, die am stärksten gefährdeten Arten im Zentrum finden, die Anzahl der in Gefangenschaft geborenen Tiere ermitteln und mehr. Sie können die Abfragen an Ihre Bedürfnisse anpassen.

Unser Beispiel bezog sich auf den Wildtierschutz, aber relationale Datenbanken haben ein breites Anwendungsspektrum in Branchen wie Gesundheitswesen, Bauwesen, Wirtschaft, Gaming, Bildung, Regierung, E-Commerce, soziale Medien und mehr.

Sie bieten mehrere Vorteile, darunter Skalierbarkeit. Sie können große Datensätze verarbeiten, weshalb sie perfekt für komplexe, reale Anwendungen sind. Außerdem gewährleisten sie Datenintegrität durch Primärschlüssel, Fremdschlüssel und Datentypen, was sicherstellt, dass die Daten konsistent und genau sind.

Neben relationalen Datenbanken können Sie auch mit nicht-relationalen Datenbanken arbeiten.

Der Hauptunterschied zwischen relationalen und nicht-relationalen Datenbanken liegt in der Art, wie die Daten gespeichert werden. Nicht-relationale Datenbanken, auch NoSQL-Datenbanken genannt, sind flexibler. Sie speichern Daten in einzelnen Dateien, die nicht miteinander verbunden sind. Ihr Datenmodell ist ebenfalls flexibler.

Während einige nicht-relationale Datenbanken ein einfaches Schema oder Datenmodell haben können, ist es oft weniger starr als das in relationalen Datenbanken definierte Schema. Sie können bei Bedarf Datenfelder hinzufügen, ändern oder entfernen.

Die Entscheidung, ob Sie eine relationale oder nicht-relationale Datenbank verwenden, hängt von verschiedenen Faktoren ab, darunter die Art der Daten und die spezifischen Anforderungen der Anwendung. Sie sollten die Vor- und Nachteile dieser beiden Ansätze abwägen, um die beste Lösung für Ihre Anwendung zu wählen.
