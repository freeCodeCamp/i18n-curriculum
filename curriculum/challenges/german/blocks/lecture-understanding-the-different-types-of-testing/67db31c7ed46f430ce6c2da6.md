# --description--

Mocking ist der Prozess, echte Daten durch falsche Daten zu ersetzen, die das Verhalten realer Komponenten simulieren. Dadurch können Tester und Entwickler bestimmte Komponenten in der Anwendung testen, ohne auf externe Abhängigkeiten angewiesen zu sein.

Ein häufiger Anwendungsfall für das Mocking von Daten ist der Umgang mit API-Aufrufen. Stellen Sie sich vor, Sie haben in Ihrer Anwendung einen solchen fetch-Aufruf:

```js
export const postService = {
  async getPosts() {
    const response = await fetch("https://api.example.com/posts");
    return response.json();
  },
};
```

Wenn Sie die `postService`-Funktion testen müssen, könnten Sie die API-Antwort mocken, anstatt ständig API-Aufrufe zu tätigen, um die Daten abzurufen. Das Abrufen von Daten über eine API dauert Zeit und hängt von externen Faktoren wie der Netzwerkverfügbarkeit und Serverantworten ab. Das Mocken der API-Antwort entfernt diese externen Faktoren und ermöglicht es Testern, mit einem vorhersehbaren, kontrollierten Datensatz zu arbeiten.

Hier ist ein Beispiel für eine gemockte API-Antwort:

```json
[
  {
    "id": 1,
    "title": "Understanding Async/Await in JavaScript",
    "content": "Async/Await makes asynchronous code look and behave like synchronous code...",
    "author": "Jane Doe",
    "created_at": "2025-04-10T14:32:00Z"
  },
  {
    "id": 2,
    "title": "10 Tips for Writing Clean Code",
    "content": "Clean code is not just about code formatting. It involves naming, architecture...",
    "author": "John Smith",
    "created_at": "2025-04-08T09:21:00Z"
  },
  {
    "id": 3,
    "title": "Exploring React 19 Features",
    "content": "React 19 comes with exciting features like new hooks and better performance...",
    "author": "Alex Lee",
    "created_at": "2025-04-07T18:47:00Z"
  }
]
```

Nachdem wir nun verstanden haben, was Mocking ist, vergleichen wir es mit Stubbing. Stubs sind Objekte, die vordefinierte Antworten oder Dummy-Daten für ein erwartetes Verhalten in einer Anwendung zurückgeben. Zum Beispiel können Sie das Verhalten einer Datenbankverbindung in Ihren Tests stubben, ohne auf eine tatsächliche Datenbankverbindung angewiesen zu sein.

Hier ist ein Beispiel für einen Stub in Jest:

```js
import { postService } from "./postService";
import { db } from "./db";

jest.mock("./db", () => ({
  db: {
    fetchPosts: jest.fn(),
  },
}));

test("should return stubbed posts", async () => {
  const fakePosts = [
    { id: 1, title: "Stubbed Post", content: "This is a stubbed post." },
  ];

  db.fetchPosts.mockResolvedValue(fakePosts);

  const posts = await postService.getPosts();

  expect(posts).toEqual(fakePosts);
});
```

`db.fetchPosts.mockResolvedValue(fakePosts);` ist in diesem Beispiel der Stub.

Die letzte Komponente, die wir besprechen, sind Fakes. Fakes sind vereinfachte Versionen realer Komponenten ohne die Komplexität oder Nebenwirkungen des Originals. Zum Beispiel können Sie eine Datenbank faken, indem Sie die Daten im Speicher speichern, anstatt mit der echten Datenbank zu interagieren. So können Sie Datenbankoperationen im Speicher nachahmen, was viel schneller ist als der Umgang mit der echten Datenbank. Ein weiteres häufiges Beispiel ist die Verwendung von Fakes beim Arbeiten mit Dateisystemen. Sie können ein Fake-Dateisystem anstelle des echten erstellen, wenn es ums Testen geht.

Wenn Sie weiterhin komplexere Softwareanwendungen entwickeln, sollten Sie in Ihren Tests dort, wo es sinnvoll ist, Mocks, Stubs oder Fakes verwenden.
