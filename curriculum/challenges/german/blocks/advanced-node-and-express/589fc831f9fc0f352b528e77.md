# --description--

Derzeit können Sie nicht feststellen, wer mit Ihrem Websocket verbunden ist. Während `req.user` das Benutzerobjekt enthält, ist das nur der Fall, wenn Ihr Benutzer mit dem Webserver interagiert, und bei Websockets haben Sie keine `req` (Anfrage) und daher keine Benutzerdaten. Eine Möglichkeit, das Problem zu lösen, zu wissen, wer mit Ihrem Websocket verbunden ist, besteht darin, das Cookie zu parsen und zu decodieren, das die Passport-Session enthält, und es dann zu deserialisieren, um das Benutzerobjekt zu erhalten. Glücklicherweise gibt es ein Paket auf npm, das genau dafür gedacht ist und eine einst komplexe Aufgabe einfach macht!

`passport.socketio@~3.7.0`, `connect-mongo@~3.2.0` und `cookie-parser@~1.4.5` wurden bereits als Abhängigkeiten hinzugefügt. Fordern Sie sie als `passportSocketIo`, `MongoStore` und `cookieParser` an. Außerdem müssen wir einen neuen Memory Store initialisieren, von `express-session`, den wir zuvor angefordert haben. Es sollte so aussehen:

```js
const MongoStore = require('connect-mongo')(session);
const URI = process.env.MONGO_URI;
const store = new MongoStore({ url: URI });
```

Jetzt müssen wir Socket.IO nur noch mitteilen, dass es diesen verwenden soll, und die Optionen setzen. Stellen Sie sicher, dass dies vor dem bestehenden Socket-Code und nicht im bestehenden Verbindungs-Listener hinzugefügt wird. Für Ihren Server sollte es so aussehen:

```js
io.use(
  passportSocketIo.authorize({
    cookieParser: cookieParser,
    key: 'express.sid',
    secret: process.env.SESSION_SECRET,
    store: store,
    success: onAuthorizeSuccess,
    fail: onAuthorizeFail
  })
);
```

Beachten Sie, dass die Konfiguration der Passport-Authentifizierung für Socket.IO der Art und Weise sehr ähnlich ist, wie wir die `session`-Middleware für die API konfiguriert haben. Das liegt daran, dass sie dieselbe Authentifizierungsmethode verwenden sollen – die Session-ID aus einem Cookie holen und validieren.

Als wir zuvor die `session`-Middleware konfigurierten, haben wir den Cookie-Namen für die Session (`key`) nicht explizit gesetzt. Das liegt daran, dass das `session`-Paket den Standardwert verwendet hat. Da wir jetzt ein weiteres Paket hinzugefügt haben, das Zugriff auf denselben Wert aus den Cookies benötigt, müssen wir den `key`-Wert in beiden Konfigurationsobjekten explizit setzen.

Fügen Sie unbedingt den `key` mit dem Cookie-Namen zur `session`-Middleware hinzu, die mit dem Socket.IO-Schlüssel übereinstimmt. Fügen Sie außerdem die `store`-Referenz zu den Optionen hinzu, in der Nähe der Stelle, an der wir `saveUninitialized: true` setzen. Das ist notwendig, um Socket.IO mitzuteilen, welche Session zugeordnet werden soll.

<hr>

Definieren Sie nun die `success` und `fail` Callback-Funktionen:

```js
function onAuthorizeSuccess(data, accept) {
  console.log('successful connection to socket.io');

  accept(null, true);
}

function onAuthorizeFail(data, message, error, accept) {
  if (error) throw new Error(message);
  console.log('failed connection to socket.io:', message);
  accept(null, false);
}
```

Das Benutzerobjekt ist jetzt auf Ihrem Socket-Objekt als `socket.request.user` zugänglich. Zum Beispiel können Sie jetzt Folgendes hinzufügen:

```js
console.log('user ' + socket.request.user.username + ' connected');
```

Es wird in der Server-Konsole protokollieren, wer verbunden ist!
