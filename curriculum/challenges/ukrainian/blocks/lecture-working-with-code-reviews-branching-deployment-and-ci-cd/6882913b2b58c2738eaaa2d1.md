# --description--

Ми коротко торкалися файлу `.gitignore` у попередньому уроці. Але що саме це за файл і що в нього слід додавати?

Файл `.gitignore` — це спеціальний тип файлу, пов’язаний з операціями Git. Назва підказує, що цей файл використовується, щоб сказати Git ігнорувати певні речі, і це найпоширеніше використання. Але насправді він каже Git припинити відстежувати файл. Це важлива відмінність, яку ми пояснимо пізніше.

Що ж насправді слід додавати до цього файлу? Ви маєте перелічити те, що не хочете комітити.

Перш за все, це ваші файли з секретами. Це може бути, наприклад, файл `.env`, де ви зберігаєте ключі API для вашого застосунку. Це може бути приватний файл ключа GPG, який ваш застосунок використовує для автентифікації. Все, що містить конфіденційну інформацію, яку ви не хочете розголошувати, слід ігнорувати за допомогою `.gitignore`. Навіть якщо ваш репозиторій приватний, слід поводитися так, ніби він публічний — вважайте, що його може бачити будь-хто, і дбайте про те, щоб не комітити секрети.

Давайте подивимося, як зараз може виглядати наш файл:

```sh
# Secrets
.env
github.key
```

Ми сказали Git ігнорувати файл `.env`, де зберігаємо наші секрети. Також ми сказали Git ігнорувати файл `github.key`, який містить приватний ключ GPG.

Наступне, що варто врахувати — це пакунки або залежності, які використовує ваш проєкт. Наприклад, коли ви працюєте з проєктом Node.js, ви, ймовірно, встановлюєте пакунки з npm. Ці пакунки потрапляють у папку `node_modules` у вашому проєкті, щоб ваш код міг їх імпортувати та використовувати.

Але папка `node_modules` стає надто великою, і ви точно не хочете комітити всі ці пакунки та відстежувати зміни щоразу, коли оновлюєте один із них. Замість цього ви комітите файл `package.json` і відповідний lockfile (наприклад, `package-lock.json`, `pnpm-lock.yaml`). Ці файли кажуть npm, які пакунки та версії встановлювати, щоб інші учасники могли встановити ті ж залежності, не комітячи їх у репозиторій.

Давайте додамо `node_modules` до нашого файлу `.gitignore`:

```sh
# Secrets
.env
github.key

# Packages
node_modules
```

Тепер торкнемося важливого синтаксичного моменту для файлів `.gitignore`. Коли ви додаєте запис на кшталт `node_modules`, Git ігнорує все, що відповідає цьому шаблону. Це означає, що він ігноруватиме директорію `node_modules` у корені вашого проєкту, але також і `node_modules` у папці `client` вашого проєкту.

Якщо ви хочете ігнорувати щось лише в корені, можна додати префікс у вигляді косої риски. `node_modules` стане `/node_modules`. Це ігноруватиме кореневу директорію `node_modules`, але не `client/node_modules`. Нам це не потрібно, тож залишимо запис просто `node_modules`.

Наступне, що варто ігнорувати — це вихідні дані збірки. Наприклад, компіляція проєкту TypeScript може створити папку `dist` з JavaScript-вихідними файлами. Ви не хочете комітити цей вихід, бо це роздуває історію Git. Замість цього слід ігнорувати цей вихід — учасники можуть відтворити його самі, запустивши ту саму команду збірки.

Давайте ігнорувати папку `dist`:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
```

Іноді помилково розміщений файл може зламати компілятор TypeScript, через що JavaScript-вихідні файли розкидаються по різних місцях. Оскільки ми виключно пишемо на TypeScript, давайте використаємо синтаксис glob, щоб ігнорувати всі JavaScript-файли:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js
```

Інші поширені вихідні дані, які можна ігнорувати, — це директорія `coverage` для результатів покриття юніт-тестів, директорія `.angular`, якщо ви використовуєте Angular, директорія `.next`, якщо ви використовуєте NextJS, і так далі.

Поки що можна обмежитися тим, що ми маємо. Якщо ви використовуєте фреймворк, який генерує файли, перевірте в його документації, що саме потрібно додати до `.gitignore`.

Четверта група файлів для ігнорування — це файли IDE. Наприклад, продукти Jetbrains можуть використовувати папку `.idea` для конфігурацій IDE. Visual Studio Code використовує папку `.vscode`.

Ці файли часто варто ігнорувати, бо вони містять налаштування вашого локального середовища, які іншим учасникам не потрібні (адже вони користуються власними конфігураціями). Давайте додамо папку Visual Studio Code, бо саме цю IDE ми використовуємо.

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
```

Але наша папка `.vscode` містить файл `settings.json`, який включає деякі налаштування робочого простору, що можуть бути корисними для наших учасників — цей файл дозволяє їм налаштувати розширення та середовище, щоб краще відповідати нашому.

Тож давайте використаємо синтаксис заперечення, щоб "розігнорувати" цей файл:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json
```

Якщо ви користуєтеся MacOS, вам може бути знайомий файл `.DS_STORE`. Цей файл містить метадані, які Finder використовує, щоб визначити, як відтворювати директорію (наприклад, з мініатюрами або у вигляді списку). Аналогічно, користувачі Windows можуть бачити прихований файл `Thumbs.db`, який кешує мініатюри для директорії.

Зазвичай ці файли варто ігнорувати, бо це особисті налаштування, які не впливають на проєкт. Додамо обидва:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json

# System Files
.DS_STORE
Thumbs.db
```

Коротка порада щодо ігнорування файлів, які вже були закомічені. Припустимо, ви випадково закомітили папку `dist` раніше, але тепер хочете її видалити та ігнорувати. Ви не можете просто додати її до файлу `.gitignore` — це не видалить раніше закомічений вміст, а лише скаже Git припинити відстежувати файл. Тож якщо ви ігноруєте файл, який уже закомічений, Git не відстежуватиме зміни, але й не відстежуватиме його видалення.

Натомість спочатку потрібно видалити файл. Створіть коміт із цим видаленням. Потім додайте елемент до `.gitignore` і створіть другий коміт із цим оновленням. Надалі, коли цей файл знову з’явиться, Git ігноруватиме його, і він не з’явиться у ваших комітах.

Нарешті, якщо ви працюєте над проєктом іншою мовою, наприклад Python, елементи, які потрібно ігнорувати, відрізнятимуться від описаних тут. GitHub підтримує дуже корисний репозиторій шаблонів `.gitignore` для різних мов і фреймворків. Ви можете знайти цей список за адресою <a href="https://github.com/github/gitignore" target="_blank">https://github.com/github/gitignore</a>.

Отже, тепер ви готові уникати коміту файлів, які не хочете бачити у своїх репозиторіях!
