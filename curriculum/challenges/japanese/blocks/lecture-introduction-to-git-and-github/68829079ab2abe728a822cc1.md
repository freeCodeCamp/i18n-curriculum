# --description--

前のレッスンでは、GitHubで新しいリポジトリを作成する方法を学びました。しかし、すでにローカルにあるプロジェクトをGitHubに追加したい場合はどうすればよいでしょうか？このレッスンでは、既存のローカルリポジトリをGitHubにプッシュし、リモートリポジトリから変更をプルする方法を学びます。

まず、ファイルなしでGitHubに新しいリポジトリを作成してください。正しく行うと、リポジトリをクローンするために必要なURLが含まれたページにリダイレクトされます。

次にターミナルを開き、GitHubに追加したい既存のプロジェクトを選択します。このレッスンでは、`super-awesome-game`という名前のローカルプロジェクトを選び、その中に以下のファイルがあります：

```sh
jessicawilkins super-awesome-game >> ls
README.md  index.html script.js  styles.css
```

Gitを使ってこのプロジェクトのバージョン管理を設定するには、プロジェクトディレクトリで`git init`コマンドを実行する必要があります。これにより空のGitリポジトリが初期化され、Gitがこのプロジェクトの変更を追跡し始めます。正しく実行されると、ターミナルに似たような出力が表示されます：

```sh
jessicawilkins super-awesome-game >> git init
Initialized empty Git repository in /Users/jessicawilkins/workspace/freeCodeCamp/super-awesome-game/.git/
```

プロジェクトに空のGitリポジトリを初期化すると、新しい`.git`という隠しディレクトリが追加されます。追加されたディレクトリを確認するには、隠しファイルも含めてすべてのファイルとディレクトリを一覧表示する`ls -a`コマンドを実行してください：

```sh
jessicawilkins super-awesome-game >> ls -a
.          ..         .git       README.md  index.html script.js  styles.css
```

この`.git`ディレクトリを削除しないことが重要です。削除するとGitの履歴がすべて失われます。

空のGitリポジトリが初期化されたので、`git status`コマンドを実行できます。このコマンドは作業ディレクトリの現在の状態を表示するために使われ、ワークフローで頻繁に使用します。次のような出力が表示されるはずです：

```sh
jessicawilkins super-awesome-game >> git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	README.md
	index.html
	script.js
	styles.css

nothing added to commit but untracked files present (use "git add" to track)
```

`On branch main`は現在`main`ブランチにいることを意味します。Gitのブランチは変更を加えるための別の作業スペースです。`main`ブランチは実際のアプリケーションでは主に主要または本番ブランチを表します。開発チームは新機能やバグ修正のために複数のブランチを作成し、それらの変更を`main`ブランチにマージします。このプロセスについては将来のレッスンで詳しく学びます。

`Untracked files`はGitが現在追跡していないファイルを検出したことを意味します。

ファイルには5つの状態があります：

1. 「Untracked（未追跡）」：これはファイルがリポジトリに新しく、Gitがまだ「見て」いないことを意味します。
2. 「Modified（変更済み）」：このファイルは前回のコミットに存在し、まだコミットされていない変更があります。
3. 「Ignored（無視）」：Gitでは無視されたファイルは通常表示されませんが、IDEには表示される場合があります。無視されたファイルは通常`.gitignore`ファイルに含まれているため、Gitの操作から除外されます。
4. 「Deleted（削除済み）」：削除されたファイルは未追跡ファイルの逆で、以前存在していたファイルが削除されたことを意味します。
5. 「Renamed（名前変更）」：名前変更されたファイルは内容は変わらず、名前や場所が変更されたファイルです。場合によっては、少しの変更があっても名前変更とみなされることがあります。

リポジトリに未コミットの変更がある場合、まず「ステージ」する必要があります。「ステージングエリア」は`.git`ディレクトリ内の特別なファイルで、次のコミットに含まれる情報を保持します。コミットはリポジトリの現在の状態のスナップショットのようなものです。

1つずつファイルをステージしたい場合は、次のコマンドを使えます：

```sh
git add name-of-file
```

すべての未ステージの変更をまとめてステージしたい場合は、`git add .`を使えます。ピリオド（`.`）は現在のディレクトリのエイリアスです。`git add .`の使用を避けるよう警告する開発者もいます。意図しないファイルをステージしてしまう可能性があるためです。もしそうなった場合は、次のコマンドで「アンステージ」できます：

```sh
git reset name-of-file
```

一般的には、ステージする前に`git status`でどのファイルが追加されるかを確認していれば問題ありません。

ファイルをステージしたら、次にコミットする必要があります。`git commit`コマンドを実行するとコミットできます。これによりGit設定で指定した好みのエディターが開き、変更内容の詳細なメッセージを入力できます。短いメッセージを使いたい場合は、次のように`git commit -m`コマンドを使えます：

```sh
git commit -m "short message goes here"
```

`-m`フラグを使う場合でも、変更内容の文脈を示す説明的なメッセージを提供することをお勧めします。以下は良いメッセージの例です：

```sh
git commit -m "feat(api): implement JWT-based authentication"
git commit -m "fix(auth): correct token expiration time"
git commit -m "refactor: simplify login flow logic"
```

`feat(api)`と`fix(auth)`はConventional Commitsスタイルに従っており、コミットの変更タイプに関する文脈を提供します。

作業をコミットし、再度`git status`を実行すると、ターミナルに次のような出力が表示されます：

```sh
On branch main
nothing to commit, working tree clean
```

このプロジェクトの過去のすべてのコミットを表示するには、`git log`コマンドを実行します。これにより、著者、コミット日時、コミットメッセージ、コミットハッシュなどの情報とともにすべての過去のコミットが一覧表示されます。コミットハッシュはコミットの一意の識別子となる長い文字列です。

コミットが完了したら、変更をGitHubにプッシュできます。まずリモートリポジトリへの接続を設定する必要があります。これは前のレッスンで学びましたが、SSHを使った例をここで示します：

```sh
git remote add origin git@github.com:your-github-username/name-of-repo.git
```

最後のステップは、次のコマンドで変更をGitHubにプッシュすることです：

```sh
git push -u origin main
```

`-u`フラグは`--set-upstream`の省略形です。ローカルの`main`ブランチを`main`上のリモート`origin`ブランチに接続します。新しいブランチごとに一度だけ使えばよく、その後はブランチ名を指定せずに`git push`を実行できます。

正しく実行されると、ターミナルに似たような結果が表示されます：

```sh
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Delta compression using up to 8 threads
Compressing objects: 100% (2/2), done.
Writing objects: 100% (3/3), 253 bytes | 253.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0 (from 0)
To https://github.com/jdwilkin4/super-awesome-game.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

GitHubのリポジトリページを更新すると、新しい変更も確認できます。

他の人とプロジェクトを共同作業していてリポジトリから新しい変更をプルする必要がある場合は、`git pull`コマンドを使います。チームでの作業や共同作業については将来のレッスンで詳しく学びます。
