# --description--

El mundo digital tal como lo conocemos hoy se basa en la noción de que los dispositivos pueden comunicarse entre sí, enviando y recibiendo datos según sea necesario.

¿Pero alguna vez has pensado en cómo funciona este proceso detrás de escena?

Cuando navegas por internet, constantemente solicitas y envías datos a través de la red. Esto es posible gracias a un proceso detallado basado en reglas y procedimientos estándar que aseguran que la información se envíe y reciba correctamente por los dispositivos adecuados.

Ese conjunto de reglas y procedimientos se conoce como un **protocolo**.

Los protocolos básicamente permiten que dos o más dispositivos se comuniquen entre sí asegurándose de que todos sigan ciertos estándares, como el formato de los datos que se intercambiarán.

Los protocolos aseguran que todos los dispositivos involucrados en el proceso de intercambio de datos puedan entenderse entre sí. Figurativamente, todos sabrán cómo "hablar el mismo idioma".

En su esencia, TCP/IP son los protocolos que hacen posible internet.

**TCP/IP** es un conjunto de protocolos de comunicación que determinan cómo se empaquetan, envían, enrutan y reciben los datos a través de una red. Garantiza que la información se transporte de manera confiable.

Específicamente, el **IP**, donde IP significa **Internet Protocol**, es un protocolo que asegura que los datos vayan desde la fuente hasta el destino correcto en la red.

¿Por qué es esto importante? Recuerda que internet es una red global de dispositivos interconectados. Para que tu dispositivo envíe una solicitud, primero tiene que saber a dónde enviar esos datos. Ese es el propósito principal de las direcciones IP, la base del Protocolo de Internet (IP). Identifican de manera única a los dispositivos en la red, para que el dispositivo de origen sepa a dónde enviar los datos.

Por sí mismo, el Protocolo de Internet es sin conexión, lo que significa que cada unidad de datos (llamada “paquete”) se envía de forma independiente a su destino a través de la red, sin establecer previamente una conexión ni asegurar que se entregará correctamente o en el orden correcto. Si llegan a su destino, la fuente no recibe ninguna notificación.

Aquí es donde protocolos como TCP se vuelven realmente importantes.

El protocolo **TCP**, conocido como **Transmission Control Protocol**, divide los datos en pequeños segmentos. Estas unidades fundamentales de datos están numeradas y se reensamblan cuando llegan a su destino utilizando algoritmos específicos que detectan si los datos están completos y correctos. Estos algoritmos también verifican errores y paquetes perdidos, por lo que pueden pedir a la fuente que los reenvíe si es necesario, asegurando que la información se reciba correctamente.

Juntos, el Protocolo de Control de Transmisión y el Protocolo de Internet establecen las reglas y estándares necesarios para asegurar que los datos se transmitan de manera confiable a través de internet.

Además de TCP/IP, hay otro protocolo muy importante para el internet moderno: HTTP.

HTTP funciona a nivel de aplicación.

**HTTP** significa **HyperText Transfer Protocol**. Define las reglas y estándares para la comunicación entre el cliente (como tu navegador) y el servidor.

Cuando intentas acceder a un sitio web en tu navegador, el navegador envía una solicitud HTTP al servidor pidiendo ese recurso específico y sus activos relacionados. El servidor recibe la solicitud HTTP, ejecuta su lógica y devuelve una respuesta HTTP al cliente con el contenido solicitado o alguna información que indique por qué la solicitud no fue exitosa.

Las solicitudes HTTP y las respuestas HTTP tienen estructuras muy similares, incluyendo:

* Una línea de solicitud (la línea inicial)
    
* Encabezados
    
* Cuerpo (opcional)

## **Solicitudes HTTP**

La **línea de solicitud** de una petición HTTP se divide en tres partes principales:

* Un método, que describe la acción que se está solicitando. Por ejemplo, `GET` para recuperar datos y `POST` para enviar datos. Otros métodos comunes incluyen `PUT`, `PATCH`, `DELETE` y `HEAD`.
    
* Una ruta o URI, que describe el recurso específico que el cliente está solicitando.
    
* La versión del protocolo HTTP que se está usando para el proceso de comunicación. Por ejemplo, `HTTP/1.1`.

Este es un ejemplo con el método `GET` en la ruta raíz `/` y la versión HTTP 1.1:

`GET / HTTP/1.1`

El **header** contiene pares clave-valor con información sobre la solicitud.

Por ejemplo: `Content-Type: application/json`.

Debajo del encabezado, hay una línea vacía para indicar que los metadatos (información adicional sobre la solicitud) están completos.

Y finalmente, el **cuerpo**. Esto es opcional. Se envía cuando el cliente tiene que enviar información al servidor, como al enviar nuevos datos con el método `POST`.

Este es un ejemplo de una solicitud HTTP:

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

## **Respuestas HTTP**

Tienen una estructura muy similar:

* Una línea de estado
    
* Encabezado
    
* Cuerpo
    

Primero, la **línea de estado** contiene información sobre el resultado de la solicitud. Normalmente encontrarás esto:

* La versión de HTTP que se está usando para el proceso de comunicación. Por ejemplo, `HTTP/1.1`.
    
* El código de estado, que indica el resultado de la solicitud.
    
* Una frase de razón, una breve descripción legible para humanos del código de estado. Esto es útil para entender qué ocurrió.
    

Los códigos de estado comunes incluyen:

* `200 OK`
    
* `201 CREATED`
    
* `400 BAD REQUEST`
    
* `404 NOT FOUND`
    
* `403 FORBIDDEN`
    
* `500 INTERNAL SERVER ERROR`
    

Hay muchos códigos de estado. Se categorizan por su primer dígito de izquierda a derecha.

* `1xx` para respuestas informativas.
    
* `2xx` para respuestas exitosas.
    
* `3xx` para mensajes de redirección.
    
* `4xx` para respuestas de error del cliente.
    
* `5xx` para respuestas de error del servidor.
    

Debajo de la línea de estado de una respuesta HTTP, encontramos el **header** con metadatos sobre la respuesta. Por ejemplo, `Content-Type: text/html`. Esto también puede incluir la longitud del contenido, la fecha, la ubicación y otra información adicional que el cliente pueda encontrar útil.

Y finalmente, encontramos el **cuerpo** de la respuesta HTTP, que contiene los datos que el cliente estaba solicitando o una explicación de lo que ocurrió si la solicitud no fue exitosa. Esto es opcional, pero se incluye en la mayoría de las respuestas.

Este es un ejemplo de una respuesta HTTP:

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

Ahora que sabes sobre HTTP, podrías tener curiosidad por saber más sobre HTTPS.

Puedes pensar en HTTPS como una versión “segura” de HTTP, donde los datos enviados entre el navegador y el servidor están cifrados por motivos de seguridad. La mayoría del tráfico web moderno usa HTTPS.

Protocolos como TCP/IP y HTTP han hecho que la World Wide Web y el internet en general se conviertan en la herramienta esencial en la que la mayoría de nosotros confiamos todos los días. Gracias a ellos, los dispositivos pueden comunicarse correctamente a través de una red e intercambiar datos en todo el mundo.
