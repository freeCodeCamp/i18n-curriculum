# --description--

Die Same-Origin-Policy ist eine eingebaute Sicherheitsmaßnahme in Ihren Webbrowsern, die verhindert, dass eine Website private Daten von einer anderen Website ausliest. Diese Regel schützt Ihre persönlichen Konten vor Hackern.

Eine Origin besteht aus drei spezifischen Teilen: dem Webprotokoll, dem Domainnamen und der Portnummer.

Zum Beispiel ist in der Adresse `https://example.com:443` das Protokoll HTTPS, der Domainname `example.com` und die Portnummer `443`.

Browser verbergen die Portnummer oft, wenn sie die Standardnummer ist. HTTPS verwendet standardmäßig Port 443, und HTTP Port 80. Auch wenn sie verborgen ist, prüft der Browser sie weiterhin. Zwei Websites haben nur dann dieselbe Origin, wenn alle drei Teile exakt übereinstimmen.

Nach dieser strengen Regel blockiert ein Browser eine Website daran, eine andere Website auszuspionieren. Stellen Sie sich vor, Sie öffnen Ihr Online-Bankkonto in einem Browser-Tab und eine bösartige Website in einem zweiten Tab. Die Same-Origin-Policy verhindert, dass die bösartige Website Ihre Bankdaten stiehlt.

Moderne Websites müssen jedoch oft Daten zwischen verschiedenen Origins austauschen. Zum Beispiel könnte eine Frontend-Webseite Daten von einem separaten Backend-Server abrufen müssen. Die Same-Origin-Regeln in Browsern blockieren diese sicheren Anfragen, daher brauchten Entwickler eine Möglichkeit, Daten sicher zu teilen. Hier kommt CORS ins Spiel.

CORS ist ein System, das Servern erlaubt, den Zugriff durch Browserregeln zu öffnen. Es verwendet spezielle Header, um vertrauenswürdigen Seiten den Datenzugriff zu gewähren.

So läuft es ab, wenn Ihr Browser eine andere Website um Daten bittet:

* Es sendet einen `Origin`-Header mit, der die anfragende Website benennt
* Der empfangende Server prüft diesen Namen
* Wenn der Server der Website vertraut, antwortet er mit einem speziellen Berechtigungs-Header, der dem Browser mitteilt, dass die Website die eingehenden Daten lesen darf.

Sendet der Server diese Berechtigung nicht, blockiert der Browser die Daten sofort. Die Frontend-Website erhält dann eine Fehlermeldung statt der privaten Dateien.

In Node.js-Anwendungen mit Express wird CORS mit dem `cors`-Paket implementiert:

```javascript
const express = require('express');
const cors = require('cors');
const app = express();

// this enables CORS for all origins and all routes for development
app.use(cors());
```

In der Produktion müssen Sie möglicherweise ein Konfigurationsobjekt übergeben, um den Zugriff auf eine bestimmte Domain einzuschränken:

```javascript
const corsOptions = {
  origin: ['https://yourapp.com', 'https://admin.yourapp.com'],
  credentials: true
};

app.use(cors(corsOptions));
```

Wenn Sie reines Node.js verwenden, müssen Sie dies manuell handhaben, indem Sie bei jeder Anfrage die CORS-Antwortheader selbst setzen:

```javascript
res.setHeader('Access-Control-Allow-Origin', 'https://example.com');
res.setHeader('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');
res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization');
```
