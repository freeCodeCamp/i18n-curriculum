# --description--

Manchmal gibt es Situationen, in denen Sie mehr über einen Typ wissen als TypeScript. Hier kommen Typzusicherungen ins Spiel.

Hier ist ein Beispiel, das die `querySelector()`-Methode verwendet, um auf ein Element mit der `id` von `submit` zuzugreifen:

```ts
const submitBtn = document.querySelector("#submit");
```

Im Moment weiß TypeScript nur, dass dieses `submitBtn` irgendeine Art von `Element` oder möglicherweise `null` ist. TypeScript schaut nicht in den entsprechenden HTML-Code, um das herauszufinden.

Dies ist ein guter Moment, um eine Typzusicherung zu verwenden, um TypeScript mitzuteilen, um welchen Elementtyp es sich handelt. Hier ist ein Beispiel:

```ts
const submitBtn = document.querySelector("#submit") as HTMLButtonElement;
```

Jetzt behandelt TypeScript dieses Element aufgrund der Typzusicherung als `button`-Element.

Es gibt Grenzen bei der Verwendung des `as`-Schlüsselworts. Zum Beispiel können Sie solche Dinge nicht machen, ohne dass TypeScript einen Fehler wirft:

```ts
// Conversion of type 'string' to type 'number' may be a mistake because neither type sufficiently overlaps with the other. If this was intentional, convert the expression to 'unknown' first.
const age = "awesome" as number;
```

In diesem Beispiel macht es wenig Sinn zu versuchen, den String `"awesome"` als Zahlentyp zuzusichern. TypeScript erlaubt Typzusicherungen nur zwischen kompatiblen oder sich überschneidenden Typen.

Eine Möglichkeit, diesen Fehler zu vermeiden, wäre eine doppelte Zusicherung wie diese:

```ts
const age = "awesome" as unknown as number;
```

Obwohl dies in TypeScript funktioniert, wird es dennoch nicht empfohlen, da es logisch wenig Sinn ergibt.

Eine andere Möglichkeit, Typzusicherungen zu schreiben, ist die Verwendung der spitzen Klammer-Syntax wie diese:

```ts
const submitBtn = <HTMLButtonElement>document.querySelector("#submit");
```

Bei der Verwendung von `tsx`-Dateien müssen Sie bei dieser Syntax vorsichtig sein, da TypeScript sonst denkt, Sie wollen eine Komponente rendern, und einen Fehler wie diesen wirft:

```ts
// file: index.tsx
// This JSX tag requires 'React' to be in scope, but it could not be found.
// JSX element 'HTMLButtonElement' has no corresponding closing tag.
const submitBtn =  <HTMLButtonElement>document.querySelector("#submit");
```

In den meisten Situationen wird die erste Syntax mit dem `as`-Schlüsselwort der zweiten Option vorgezogen.
