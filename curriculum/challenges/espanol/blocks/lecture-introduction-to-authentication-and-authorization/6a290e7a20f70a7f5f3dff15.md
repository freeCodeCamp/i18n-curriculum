# --description--

En las dos últimas clases, aprendiste cómo verificar la identidad del usuario y mantener su sesión usando cookies. Sin embargo, enviar cookies automáticamente con cada solicitud introduce una vulnerabilidad de seguridad única llamada Cross-Site Request Forgery, CSRF.

CSRF es un ataque que obliga a un usuario autenticado a ejecutar acciones no deseadas en una aplicación web donde está actualmente conectado. En un ataque CSRF, un sitio web malicioso engaña al navegador de la víctima para que haga una solicitud fraudulenta a tu servidor backend, explotando la confianza del servidor en la cookie del navegador del usuario.

Para entender CSRF, considera este ejemplo paso a paso de una aplicación bancaria:

1. **La configuración:** Inicias sesión en el sitio web de tu banco, por ejemplo, `trusted-bank.com`. El banco valida tus credenciales y almacena un token de sesión en la cookie de tu navegador.

2. **La trampa:** Sin cerrar sesión en tu banco, abres una nueva pestaña del navegador y visitas un sitio web malicioso, por ejemplo, `evil-hacker-site.com`.

3. **El disparador:** El sitio web malicioso contiene un script oculto o un formulario que envía automáticamente una solicitud al backend del banco, como `POST trusted-bank.com/transfer?amount=1000&to=hacker`.

4. **La ejecución:** Como aún estás conectado a tu banco, tu navegador adjunta automáticamente la cookie de sesión del banco a esa solicitud. El servidor del banco ve una cookie válida, asume que hiciste la solicitud intencionalmente y transfiere tu dinero.

Lo más importante aquí es que el sitio web malicioso no puede leer tu cookie ni robar tus datos directamente. En cambio, simplemente se aprovecha de que tu navegador adjunta automáticamente cookies a cualquier solicitud que vaya al servidor objetivo.

Hay dos defensas principales: el **atributo SameSite de la cookie** y los **tokens CSRF**.

La defensa más simple es configurar el atributo `SameSite` en tus cookies. Esto indica al navegador cuándo está permitido enviar la cookie junto con solicitudes entre sitios.

- **SameSite=Strict:** El navegador nunca envía la cookie si la solicitud proviene de un sitio web de terceros.

- **SameSite=Lax:** El navegador permite cookies solo cuando el usuario navega de forma segura al sitio objetivo, como al hacer clic en un enlace estándar, pero las bloquea en envíos de formularios entre sitios o solicitudes API.

Para una protección robusta, los frameworks backend usan tokens CSRF. Cuando un usuario carga una página, el backend genera un token aleatorio, único e impredecible y lo adjunta al formulario frontend o a la sesión.

Cuando el cliente envía una solicitud, debe incluir este token en los encabezados personalizados de la solicitud o en los datos del formulario. El backend compara el token enviado con el token guardado en la sesión del usuario. Como un sitio malicioso no puede acceder a tu frontend para copiar este token secreto, cualquier solicitud entre sitios que forje carecerá de este token y fallará la validación.

Al implementar estas estrategias, aseguras que las solicitudes que llegan a tu backend sean acciones intencionales hechas por el usuario real, y no acciones no autorizadas forzadas por un atacante malicioso.
