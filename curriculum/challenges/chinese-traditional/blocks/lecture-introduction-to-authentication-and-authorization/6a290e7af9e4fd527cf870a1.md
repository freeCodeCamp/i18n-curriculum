# --description--

在上一堂課中，你學到了後端伺服器如何在登入時識別使用者。但他們登入後會發生什麼事？當使用者瀏覽到新頁面時，伺服器如何記住他們是誰？這就是 JSON Web Tokens（JWT）派上用場的地方。

用簡單的話說，JWT 就像伺服器在使用者成功登入後發給他們的數位鑰匙卡。使用者在每次後續請求時出示這張鑰匙卡，證明自己的身份，而不需要重新輸入密碼。

伺服器不再需要在自己的記憶體中記住每個作用中的使用者，而是透過一個簡單的四步驟流程，將責任轉移給客戶端：

1. **登入：** 使用者將他們的使用者名稱和密碼，或其他登入方式，傳送給伺服器。

2. **發行：** 伺服器將憑證與註冊時儲存的憑證進行驗證。如果正確，伺服器會產生 JWT 並回傳給使用者。

3. **儲存：** 客戶端接收 JWT 並將其儲存，通常是在瀏覽器的 cookie 或區域儲存中。

4. **請求：** 對於每個未來對受保護路由的請求，客戶端會自動帶上 JWT。伺服器驗證該語彙單元並授予存取權限。

JWT 看起來像是一串冗長且混亂的字元。它由三個不同的部分組成，並以兩個點分隔，如下所示：

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

第一部分是 **Header**，第二部分是 **Payload**，第三部分是 **Signature**。

**Header** 是一個經過編碼的 JSON 物件，告訴伺服器這是什麼型別的語彙單元（JWT），以及用來保護它的雜湊演算法，例如 HMAC SHA256。

**Payload** 是語彙單元的中間部分。它包含實際的資料，或稱為使用者的「claims」。這裡會儲存像是使用者 ID、使用者名稱或帳號角色（例如 admin 或 user）等資訊。

一個重要的安全提醒是，header 和 payload 只是被編碼（Base64），而非加密。這表示任何人都能輕易解碼並讀取裡面的資料。因此，切勿在 JWT 的 payload 中儲存像是原始密碼或信用卡號碼等敏感資料。

**Signature** 是語彙單元的最後且最關鍵的部分。伺服器將編碼後的 header 和 payload 與只有伺服器知道的秘密金鑰混合，然後進行雜湊以產生簽名。

簽名防止語彙單元被竄改。如果使用者試圖解碼他們的語彙單元，將角色從「user」改成「admin」，再重新編碼，簽名將不再符合被更改的資料。

當伺服器收到被修改的語彙單元時，它會嘗試用秘密金鑰重新產生簽名。因為 payload 已改變，計算出的簽名將與語彙單元的簽名不符，伺服器會立即以 401 Unauthorized 狀態碼拒絕該請求。

簡言之，JWT 允許無狀態的驗證。伺服器不需要在資料庫中儲存會話資料，它只要信任該語彙單元，因為簽名證明是伺服器自己產生的。
