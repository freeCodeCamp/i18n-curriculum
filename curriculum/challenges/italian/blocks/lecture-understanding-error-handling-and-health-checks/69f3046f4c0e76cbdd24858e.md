# --description--

Ti sei mai chiesto cosa succede quando qualcosa va storto nella tua app Express?

E se la connessione al database fallisce? O se un utente visita una rotta che non esiste?

Capire come funziona la gestione degli errori in Express è fondamentale se vuoi creare applicazioni web sicure e affidabili.

In questa lezione, vedremo come Express gestisce gli errori e come puoi intercettarli e rispondere usando esempi di codice reali.

## Come gestire errori semplici in Express

In Express, la gestione degli errori si basa sul middleware.

Proprio come le normali funzioni middleware, il middleware per la gestione degli errori ha accesso a `req`, `res` e `next`. Ma include un parametro extra all’inizio: `err`.

Ecco come si presenta la sintassi:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

Definisci questo middleware alla fine della tua catena di middleware.

Express sa che è un gestore di errori perché ha quattro parametri — quello nuovo, `err`, oltre ai soliti `req`, `res` e `next` usati per gli altri middleware.

In una lezione futura, imparerai di più sul middleware per la gestione degli errori e su come si inserisce nella più ampia catena di middleware di Express.

## Come gestire errori asincroni

Gestire errori asincroni dipende dalla versione di Express che stai usando.

Nota che, qualunque versione di Express usi, devi comunque definire il middleware per la gestione degli errori alla fine della catena, dopo tutte le tue rotte.

Ecco di nuovo il gestore degli errori:

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Express 4

In Express 4 e versioni precedenti, che molti progetti usano ancora, gli errori asincroni **non vengono catturati automaticamente**.

Se usi `async`/`await` e qualcosa lancia un errore dentro un gestore di rotta `async`, Express non lo catturerà a meno che tu non lo passi manualmente a `next()`.

Ecco come gestire correttamente gli errori asincroni in Express 4:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

Se dimentichi il blocco `try/catch` e `next(err)`, l’errore potrebbe:

* Far crashare la tua app
* O causare un warning di promise rejection non gestita

Per questo è essenziale avvolgere la logica asincrona in `try`/`catch` in Express 4.

Alcuni sviluppatori creano persino una funzione helper per evitare di ripetere `try/catch` ovunque:

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

Poi la usano così:

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

In questo modo non devi usare blocchi `try/catch` in tutte le tue rotte `async` ma puoi comunque inoltrare gli errori correttamente.

## Express 5

La versione attuale di Express migliora questo comportamento.

Da Express 5 in poi, se una rotta `async` lancia un errore o restituisce una promise rifiutata, Express lo inoltra automaticamente al middleware per la gestione degli errori.

Questo significa che questo esempio gestisce gli errori senza un `try/catch`:

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

Se `getUserFromDatabase()` lancia un errore, Express 5 lo catturerà e lo passerà automaticamente al tuo gestore di errori.

Non serve chiamare `next(err)` manualmente.

## Come gestire errori 404

Gli errori 404 sono un po’ diversi. Succedono quando nessuna rotta corrisponde alla richiesta.

Puoi gestirli con questo middleware:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**Importante**: questo deve andare dopo le tue rotte, ma prima del middleware per la gestione degli errori.

Infine, ecco alcune buone pratiche per la gestione degli errori in Express:

* Metti sempre il middleware per la gestione degli errori per ultimo.
* Gestisci separatamente gli errori 404.
* Usa `next(err)` per far risalire gli errori dalle rotte o dalla logica `async` in Express 4 e versioni precedenti.

In produzione, non inviare mai agli utenti gli stack trace. Registra i dettagli internamente e restituisci invece un messaggio amichevole.

La gestione degli errori in Express consiste nel configurare middleware che ascoltano i problemi e rispondono in modo chiaro e sicuro. Che si tratti di un bug nel codice o di una rotta mancante, la tua app non dovrebbe mai crashare senza un piano di riserva.
