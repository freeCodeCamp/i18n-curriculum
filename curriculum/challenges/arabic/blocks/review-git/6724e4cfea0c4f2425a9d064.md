## مقدمة إلى نظام التحكم في الإصدارات

- **التعريف**: يسمح لك نظام التحكم في الإصدارات بتتبع وإدارة التغييرات في مشروعك. من أمثلة أنظمة التحكم في الإصدارات المستخدمة في البرمجيات Git وSVN وMercurial.

## مزودو التحكم في الإصدارات السحابية

- **قائمة مزودي التحكم في الإصدارات السحابية**: GitHub وGitLab هما مثالان شائعان لمزودي التحكم في الإصدارات السحابية الذين يتيحون لفرق البرمجيات التعاون وإدارة المستودعات.

## تثبيت Git وإعداده

- **تثبيت Git**: للتحقق مما إذا كان Git مثبتًا بالفعل على جهازك، يمكنك تشغيل الأمر التالي في الطرفية:

```sh
git --version
```

إذا رأيت رقم إصدار، فهذا يعني أن Git مثبت. إذا لم يكن كذلك، فستحتاج إلى تثبيته.

في أنظمة Linux، غالبًا ما يأتي Git مثبتًا مسبقًا مع معظم التوزيعات. إذا لم يكن Git مثبتًا مسبقًا، يمكنك تثبيته باستخدام أوامر مدير الحزم مثل `sudo apt-get install git` أو `sudo pacman -S git`.

لمستخدمي Mac، يمكنك تثبيت Git عبر Homebrew باستخدام الأمر `brew install git`، أو يمكنك تنزيل مثبت التنفيذ من موقع Git.

لمستخدمي Windows، يمكنك تنزيل مثبت التنفيذ من موقع Git. أو، إذا قمت بإعداد Chocolatey، يمكنك تشغيل `choco install git.install` في PowerShell. لاحظ أنه على Windows، قد ترغب أيضًا في تنزيل Git Bash للحصول على بيئة شل شبيهة بيونيكس.

للتأكد من نجاح التثبيت، شغّل الأمر `git --version` مرة أخرى في الطرفية.

- **إعدادات Git**: تُستخدم `git config` لضبط متغيرات الإعداد التي تتحكم في كيفية عمل Git على جهازك. لعرض متغيرات الإعداد الحالية وأماكن تخزينها على نظامك، يمكنك تشغيل الأمر التالي:

```sh
git config --list --show-origin
```

الآن يجب أن ترى فقط إعدادات التكوين على مستوى النظام إذا قمت بتثبيت Git لأول مرة.

لتعيين اسم المستخدم، يمكنك تشغيل الأمر التالي:

```sh
git config --global user.name "Jane Doe"
```

يُستخدم العلم `--global` هنا لتعيين اسم المستخدم لجميع المشاريع على نظامك التي تستخدم Git. إذا كنت بحاجة إلى تجاوز اسم المستخدم لمشروع معين، يمكنك تشغيل الأمر في مجلد ذلك المشروع بدون العلم `--global`.

لتعيين عنوان البريد الإلكتروني للمستخدم، يمكنك تشغيل الأمر التالي:

```sh
git config --global user.email janedoe@example.com
```

يمكنك أيضًا تعيين محرر النصوص المفضل الذي تريد أن يستخدمه Git. إليك مثالًا على كيفية تعيين محررك المفضل إلى Emacs:

```sh
git config --global core.editor emacs
```

إذا اخترت عدم تعيين محرر مفضل، فسيستخدم Git المحرر الافتراضي لنظامك.

## البرمجيات مفتوحة المصدر مقابل مغلقة المصدر

- **التعريف**: "مفتوح المصدر" يعني أن الناس يمكنهم رؤية الشفرة التي تنشرها، اقتراح تغييرات، الإبلاغ عن مشكلات، وحتى تشغيل نسخة معدلة. "مغلق المصدر" يعني أن الأشخاص الوحيدين الذين يمكنهم رؤية المشروع والتفاعل معه هم من تمنحهم إذنًا صريحًا.

## GitHub

- **التعريف**: GitHub هو حل قائم على السحابة يوفر تخزين المشاريع التي تخضع للتحكم في الإصدارات في ما يسمى "المستودعات"، ويتيح ميزات التعاون لاستخدامها مع تلك المشاريع.
- **واجهة سطر أوامر GitHub**: تُستخدم هذه الأداة لأداء مهام خاصة بـ GitHub دون مغادرة سطر الأوامر. إذا لم تكن مثبتة لديك، يمكنك الحصول على تعليمات التثبيت من توثيق GitHub - ولكن يجب أن تكون متاحة في مدير الحزم الخاص بنظامك.
- **صفحات GitHub**: خيار لنشر المواقع الثابتة أو التطبيقات التي لا تتطلب خادم خلفي لمعالجة المنطق. أي التطبيقات التي تعمل بالكامل على جانب العميل أو في متصفح المستخدم يمكن نشرها بالكامل على هذه المنصة.
- **إجراءات GitHub**: ميزة تتيح لك أتمتة سير العمل مباشرة في مستودع GitHub الخاص بك، بما في ذلك البناء والاختبار والنشر.

## أوامر Git الشائعة

- **`git init`**: سيُنشئ هذا الأمر مستودع Git فارغًا حتى يبدأ Git بتتبع التغييرات في هذا المشروع. عند تهيئة مستودع Git فارغ لمشروع، ستُضاف مجلد مخفي جديد باسم `.git`. يحتوي هذا المجلد `.git` على معلومات مهمة لإدارة Git لمشروعك.
- **`git status`**: يُستخدم هذا الأمر لعرض الحالة الحالية لدليل العمل الخاص بك - ستستخدم هذا الأمر كثيرًا في سير عملك.
- **`git add`**: يُستخدم هذا الأمر لوضع تغييراتك في منطقة الإعداد (staging area). أي شيء في منطقة الإعداد سيُضاف في الالتزام التالي. إذا أردت إعداد كل التغييرات غير المُعدة، يمكنك استخدام `git add .` حيث النقطة (`.`) تمثل الدليل الحالي الذي تعمل فيه.
- **`git commit`**: يُستخدم هذا الأمر لتثبيت تغييراتك. الالتزام هو لقطة لحالة مشروعك في ذلك الوقت. عند تشغيل `git commit`، سيفتح المحرر المفضل الذي ضبطته في إعدادات Git. بعد فتح المحرر، يمكنك كتابة رسالة مفصلة عن تغييراتك. يمكنك أيضًا اختيار كتابة رسالة قصيرة باستخدام الأمر `git commit -m` كما يلي:

```sh
git commit -m "short message goes here"
```

- **`git log`**: يعرض جميع الالتزامات السابقة مع معلومات مفيدة مثل المؤلف، تاريخ الالتزام، رسالة الالتزام، ورمز الالتزام. رمز الالتزام هو سلسلة طويلة تُستخدم كمُعرف فريد للالتزام.
- **`git remote add`**: يُستخدم هذا الأمر لإعداد الاتصال بالمستودع البعيد.
- **`git push`**: يُستخدم هذا الأمر لدفع تغييراتك إلى مستودع بعيد.
- **`git pull`**: يُستخدم هذا الأمر لسحب أحدث التغييرات من المستودع البعيد إلى المستودع المحلي.
- **`git clone`**: يُستخدم هذا الأمر لاستنساخ مستودع. يعني أن لديك نسخة من المستودع تشمل التاريخ الكامل، كل الملفات/المجلدات، والالتزامات على جهازك المحلي.
- **`git remote -v`**: يعرض قائمة المستودعات البعيدة المرتبطة بمستودع Git المحلي.
- **`git branch`**: يعرض جميع الفروع المحلية.
- **`git fetch upstream`**: يأمر Git بجلب أحدث التغييرات من المستودع الأصلي (upstream).
- **`git merge upstream/main`**: يأمر Git بدمج أحدث التغييرات من فرع `main` في المستودع الأصلي إلى الفرع الحالي.
- **`git reset`**: يسمح لك بإعادة تعيين حالة الفرع الحالي. استخدام العلم `--hard` يجبر Git على مطابقة الملفات المحلية مع حالة الفرع، مما يضمن أن لديك بيئة نظيفة للعمل.
- **`git rebase`**: إعادة الأساس في Git هي طريقة لنقل أو دمج متتالية من الالتزامات من فرع إلى آخر.
  
## العمل مع الفروع

- **التعريف**: الفرع في Git هو مساحة عمل منفصلة يمكنك إجراء تغييرات فيها. غالبًا ما يمثل فرع `main` الفرع الأساسي أو فرع الإنتاج في التطبيقات الواقعية. تنشئ فرق المطورين فروعًا متعددة للميزات الجديدة وتصحيح الأخطاء ثم تدمج تلك التغييرات مرة أخرى في فرع `main`.
- **إنشاء فرع جديد**: لإنشاء فرع جديد، يمكنك تشغيل الأمر التالي:

```sh
git branch feature
```

لتبديل ذلك الفرع، يمكنك تشغيل الأمر التالي:

```sh
git checkout feature
```

يستخدم معظم المطورين الأمر المختصر لإنشاء وتبديل الفرع وهو كالتالي:

```sh
git checkout -b new-branch-name
```

أمر أحدث وبديل هو أمر `git switch`. إليك مثالًا لإنشاء والتبديل إلى فرع جديد:

```sh
git switch -c new-branch-name
```

- **استراتيجيات التفرع**: فرع `main` هو الفرع الافتراضي وعادة ما يكون مستقرًا. من الأفضل التفرع منه لإنشاء فروع جديدة لأشياء مثل تصحيح الأخطاء، الميزات الجديدة، أو أعمال أخرى متنوعة.
- **صراعات الدمج**: تحدث عندما يحاول Git دمج التغييرات تلقائيًا من فروع مختلفة لكنه لا يستطيع تحديد التغييرات التي يجب الاحتفاظ بها. عادةً ما يحدث ذلك عندما تكون هناك تغييرات متضاربة لنفس جزء الملف.

## خمس حالات لملف متتبع بواسطة Git

- **"غير متتبع"**: يعني أن الملف جديد في المستودع، وGit لم "يراه" من قبل.
- **"معدل"**: هذا الملف كان موجودًا في الالتزام السابق، وله تغييرات لم تُلتزم بعد.
- **"متجاهل"**: من المحتمل ألا ترى الملفات المتجاهلة في Git، لكن قد يكون لدى بيئة التطوير المتكاملة مؤشر لها. تُستثنى الملفات المتجاهلة من عمليات Git، عادةً لأنها مدرجة في ملف `.gitignore`.
- **"محذوف"**: الملف المحذوف هو عكس الملف غير المتتبع - هو ملف كان موجودًا سابقًا وتمت إزالته.
- **"معاد تسميته"**: الملف المعاد تسميته هو ملف لم تتغير محتوياته، لكن تم تعديل اسمه أو موقعه. في بعض الحالات، يمكن اعتبار الملف معاد التسمية حتى لو كان به تغييرات بسيطة.

## ملفات `.gitignore`

- **التعريف**: ملف `.gitignore` هو نوع خاص من الملفات المتعلقة بعمليات Git. الاسم يشير إلى أن هذا الملف يُستخدم لإخبار Git بتجاهل أشياء معينة، وهذا هو الاستخدام الشائع. لكنه في الواقع يخبر Git بالتوقف عن تتبع ملف معين.

## العمل مع المستودعات

- **التعريف**: المستودع يشبه حاوية لمشروع - إذا كنت تعمل على تطبيق، تحتفظ بملفات ذلك التطبيق معًا في مستودع. يمكن أن تكون المستودعات محلية على جهازك، أو بعيدة على خدمة مثل GitHub.
- **المستودعات العامة مقابل الخاصة**: يمكن لأي شخص عرض وتحميل المستودع العام. المستودع الخاص يمكن الوصول إليه فقط من قبلك ومن تمنحهم إذنًا صريحًا.
- **إنشاء مستودعات على GitHub**: لإنشاء مستودع جديد على GitHub، يمكنك النقر على زر `"New Repository"` واتباع واجهة GitHub لإعداد المستودع الجديد.
- **دفع المستودعات المحلية إلى GitHub**: إذا كان لديك مشروع محلي على جهازك، يمكنك دفع ذلك المستودع إلى GitHub. إليك نظرة عامة خطوة بخطوة على العملية:

1. تهيئة مستودع Git فارغ في مجلد المشروع (`git init`).
2. إجراء تغييرات على مشروعك.
3. تشغيل الأمر `git status` لرؤية كل التغييرات التي يتتبعها Git.
4. إعداد تغييراتك (`git add`).
5. الالتزام بتغييراتك (`git commit`).
6. إعداد الاتصال بالمستودع البعيد (`git remote add`).
7. دفع تغييراتك إلى GitHub (`git push`).

## طلبات السحب

- **طلبات السحب**: طلب السحب هو طلب لسحب التغييرات من فرعك إلى الفرع الهدف. تستخدم طلبات السحب عندما تريد المساهمة بتغييرات شفرة في مشروع. تتيح هذه الطريقة للمشرفين على المشروع مراجعة تغييراتك. يمكنهم ترك تعليقات، طرح أسئلة، واقتراح تعديلات. وبعد اكتمال المراجعة، يمكن الموافقة عليها ودمجها في الفرع الرئيسي.

## المساهمة في مستودعات أخرى

- **العملية**: هناك آلاف المشاريع التي يمكنك المساهمة فيها. إليك العملية الأساسية لكيفية المساهمة في مستودع آخر:

1. قراءة وثائق المساهمة
2. إيجاد مشكلة متاحة للعمل عليها
3. عمل فورك للمستودع
4. استنساخ نسخة الفورك الخاصة بك
5. إنشاء فرع جديد
6. إجراء التغييرات حسب المشكلة
7. إنشاء طلب سحب (Pull Request)
8. انتظار مراجعة طلب السحب

## العمل مع مفاتيح SSH وGPG

- **مفاتيح GPG**: مفاتيح GPG، أو Gnu Privacy Guard، تُستخدم عادة لتوقيع الملفات أو الالتزامات. يمكن لشخص ما استخدام مفتاح GPG العام الخاص بك للتحقق من أن توقيع الملف صادر عن مفتاحك وأن محتويات الملف لم تُعدل أو تُعبث بها.

لتوليد مفتاح GPG، ستحتاج إلى تشغيل:

```sh
gpg --full-generate-key
```

- **مفاتيح SSH**: مفاتيح SSH، أو Secure SHell، تُستخدم عادةً للمصادقة على اتصال بعيد بخادم - عبر أداة `ssh`. يمكنك أيضًا استخدام مفتاح SSH لتوقيع الالتزامات.

لمفتاح SSH، ستشغل:

```sh
ssh-keygen -t ed25519 -C "your_email@example.com"
```

`ed25519` هو خوارزمية توقيع مفتاح عام حديثة.

- **توقيع الالتزامات بمفاتيح GPG**: لتوقيع الالتزامات باستخدام مفتاح GPG، ستحتاج إلى رفع المفتاح العام، وليس الخاص، إلى حساب GitHub الخاص بك. لعرض مفاتيحك العامة، ستحتاج إلى تشغيل الأمر التالي:

```sh
gpg --list-secret-keys --keyid-format=long
```

ثم، للحصول على المفتاح العام، استخدم:

```sh
gpg --armor --export "<key id>"
```
  
بعد ذلك، خذ المعرف القصير الذي حصلت عليه من قائمة المفاتيح وشغل هذا الأمر لتعيينه كمفتاح توقيع git الخاص بك:

```sh
git config --global user.signingkey <your_gpg_key_id>
```

بعدها، يمكنك تمرير العلم `-S` إلى أمر `git commit` لتوقيع التزام محدد - ستحتاج إلى إدخال عبارة المرور الخاصة بك. بدلاً من ذلك، إذا أردت توقيع كل الالتزامات تلقائيًا، يمكنك تعيين إعداد autosign إلى `true`:

```sh
git config --global commit.gpgsign true
```

- **توقيع الالتزامات بمفاتيح SSH**: لتوقيع باستخدام مفتاح SSH، وهي ميزة جديدة نسبيًا على GitHub، ستحتاج أولًا إلى رفع المفتاح إلى حساب GitHub الخاص بك. ثم ستحتاج إلى تعيين وضع التوقيع في git لاستخدام SSH:

```sh
git config --global gpg.format ssh
```

ثم، لتعيين مفتاح التوقيع، ستمرر مسار الملف بدلًا من المعرف:

```sh
git config --global user.signingkey <path_to_your_ssh_keys>
```
