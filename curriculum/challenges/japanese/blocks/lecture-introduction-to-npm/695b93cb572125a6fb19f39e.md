# --description--

npmで依存関係をインストールするとき、特に既存のプロジェクトでは、npmは`package.json`ファイルを見てプロジェクトに必要なパッケージを確認します。

`package.json`の各依存関係には、このようなバージョン番号が含まれています：

```bash
"lodash": "^4.17.21"
```

前に学んだように、**キャレット (^)** はnpmが`4.18.0`や`4.17.22`のような新しいマイナーおよびパッチバージョンをインストールできるようにします。この柔軟性は便利ですが、問題を引き起こす場合もあります。

もし2人の開発者が異なるタイミングで`npm install`を実行すると、npmは同じパッケージのわずかに異なるバージョンをインストールするかもしれません。小さな違いでもバグや予期しない動作の原因になります。

`package-lock.json`ファイルがこれを解決します。これは`npm install`を初めて実行したときに自動的に作成され、インストールされたすべての依存関係の正確なバージョンを固定します。子依存関係（依存関係の依存関係）も含まれます。

これにより、プロジェクトに関わる全員が毎回同じバージョンのパッケージを使うことが保証されます。

こちらは`package.json`を依存関係に持つシンプルな`lodash`ファイルです：

```bash
{
  "name": "package-lock-json",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC",
  "type": "commonjs",
  "dependencies": {
    "lodash": "^4.17.21"
  }
}
```

そしてこちらが`package-lock.json`の対応するセクションです：

```bash
"node_modules/lodash": {
  "version": "4.17.21",
  "resolved": "https://registry.npmjs.org/lodash/-/lodash-4.17.21.tgz",
  "integrity": "sha512-v2kDEe57..."
}
```

違いに注目してください：

* `package.json`では、`lodash`が`^4.17.21`としてリストされています — 柔軟なルールです。
    
* `package-lock.json`では、`lodash`がバージョン`4.17.21`に固定されています。
    

誰がいつ`npm install`を実行しても、その正確なバージョンが得られます。

このように、誰がいつ`npm install`を実行しても、毎回同じバージョンのLodashが得られます。

これは、プロジェクトで使う依存関係の子依存関係にも当てはまります。

      "version": "1.0.0",
      "license": "ISC",
      "dependencies": {
        "express": "^5.2.1"
      }

    ...

Express自体のほかに、`accepts`、`mime-types`、`negotiator`のような他のパッケージもあるのがわかります。これらは子依存関係で、Expressが内部で動作するために依存しているパッケージです。

また、`dependencies`の`node_modules/express`ではキャレット記号を使った範囲指定がありますが、`node_modules/etags`のような各子依存関係のエントリは特定のバージョンに固定されています。

要するに、`package-lock.json`ファイルは全員が同じ状態を保ち、開発者のマシン、CIサーバー、デプロイ環境などで一貫性を確保します。

チーム全員が同じ環境を共有できるように、`package-lock.json`ファイルは必ずGitにコミットしてください。また、`package-lock.json`ファイルは手動で編集せず、依存関係が変わるたびにnpmが自動で更新します。
