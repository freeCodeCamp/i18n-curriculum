# --description--

У програмному забезпеченні залежність — це ситуація, коли один компонент або модуль у застосунку покладається на інший для правильного функціонування. Залежності є загальними у програмних застосунках, оскільки це дозволяє розробникам використовувати готові функції або інструменти, створені іншими.

Якщо ви хочете створити застосунок на React, вам потрібно буде встановити залежності React. Без цих залежностей жоден ваш код не працюватиме правильно, і застосунок відобразить список помилок.

Коли ви працюєте з більш складними проєктами, часто доводиться покладатися на багато залежностей. Якщо множина залежностей не буде добре керована або визначена в проєкті, це призведе до того, що називають пеклом залежностей.

Для управління залежностями програмного забезпечення в проєкті вам потрібно буде використовувати менеджер пакетів. Менеджер пакетів — це інструмент, який використовується для встановлення, оновлення та видалення залежностей. Багато популярних мов програмування, таких як JavaScript, Python, Ruby і Java, використовують менеджери пакетів.

У попередньому уроці ви коротко ознайомилися з одним популярним менеджером пакетів під назвою npm.

Щоб створити новий проєкт React за допомогою Vite і npm, ви можете виконати таку команду:

```sh
npm create vite@latest my-react-app -- --template react
```

Як ви пам’ятаєте з попередніх уроків, це створить увесь необхідний шаблонний код для запуску нового застосунку React. Перед тим як запустити застосунок, вам потрібно буде встановити залежності, виконавши `npm install` або коротко `npm i`.

Ви можете переглянути всі залежності у файлі `package.json`, який знаходиться в кореневій папці вашого проєкту.

Файл `package.json` — це ключовий конфігураційний файл у проєктах, який містить метадані про ваш проєкт, включно з його назвою, версією та залежностями. Він також визначає скрипти, інформацію про ліцензію та інші налаштування, що допомагають керувати проєктом і його залежностями.

Коли ви встановлюєте залежності, у ваш проєкт буде додано папку `node_modules`.

Папка `node_modules` — це місце, де зберігаються всі пакети та бібліотеки, необхідні вашому проєкту. Ця папка містить фактичний код залежностей, перелічених у файлі `package.json`, включно з прямими залежностями вашого проєкту та залежностями цих залежностей.

Дві основні залежності, необхідні для проєкту React, — це пакети `react` і `react-dom`:

```json
"dependencies": {
  "react": "^18.3.1",
  "react-dom": "^18.3.1"
}
```

Файл `package.json` міститиме поточні версії цих пакетів, які ви використовуєте. Якщо потрібно оновити будь-які пакети локально, ви можете виконати команду `npm update`. Або оновити всі пакети глобально, виконавши команду `npm update -g`.

Окрім файлу `package.json`, у вас також буде файл `package-lock.json`. Цей файл фіксує точні версії всіх пакетів, які використовує ваш проєкт. Коли ви оновлюєте пакет, нові версії також оновлюються у файлі блокування.

Ще один важливий аспект файлу `package.json` — це dev-залежності:

```json
"devDependencies": {
  "@eslint/js": "^9.17.0",
  "@types/react": "^18.3.18",
  "@types/react-dom": "^18.3.5",
  "@vitejs/plugin-react": "^4.3.4",
  "eslint": "^9.17.0",
  "eslint-plugin-react": "^7.37.2",
  "eslint-plugin-react-hooks": "^5.0.0",
  "eslint-plugin-react-refresh": "^0.4.16",
  "globals": "^15.14.0",
  "vite": "^6.0.5"
}
```

Dev-залежності — це пакети, які використовуються лише під час розробки, а не у виробництві. Прикладом може бути бібліотека для тестування, як-от Jest. Ви встановлюєте Jest як dev-залежність, бо використовуєте її лише для локального тестування проєкту, і вона не потрібна для запуску застосунку у виробництві.

Протягом більшої частини цього уроку ми зосереджувалися на npm. Але існують й інші менеджери пакетів, як-от yarn і pnpm. Тож який менеджер пакетів слід використовувати для вашого проєкту?

Коротка відповідь — це залежить.

Якщо ви приєднуєтеся до існуючого проєкту з командою, усі ці рішення вже прийняті, і ви використовуватимете існуючий менеджер пакетів. Якщо ж ви створюєте проєкт з нуля, вам потрібно буде дослідити переваги та недоліки кожного менеджера і вирішити, який краще підходить для ваших потреб.
