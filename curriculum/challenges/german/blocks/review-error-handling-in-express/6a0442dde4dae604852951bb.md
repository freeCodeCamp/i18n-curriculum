## HTTP-Antwortstatuscodes

- **Statuscodes** sind kurze numerische Meldungen, die dem Client mitteilen, was nach einer Anfrage passiert ist (zum Beispiel 200, 404 oder 500).

### 1xx — Informativ

- **100 Continue** — der Server teilt dem Client mit, den Rest der Anfrage zu senden.
- **101 Switching Protocols** — der Server wechselt das Protokoll (häufig bei WebSockets).

### 2xx — Erfolg

- **200 OK** — die Anfrage war erfolgreich.
- **201 Created** — eine neue Ressource wurde erfolgreich erstellt.
- **204 No Content** — die Anfrage war erfolgreich, aber es gibt keinen Antwortkörper (oft verwendet für `DELETE`).

### 3xx — Umleitung

- **301 Moved Permanently** — die Ressource hat eine neue dauerhafte URL.
- **302 Found** — die Ressource befindet sich vorübergehend unter einer anderen URL.
- **304 Not Modified** — der Client kann seine zwischengespeicherte Version verwenden; es hat sich nichts geändert.

### 4xx — Client-Fehler

- **400 Bad Request** — die Anfrage ist fehlerhaft oder ungültig.
- **401 Unauthorized** — Authentifizierung ist erforderlich.
- **403 Forbidden** — authentifiziert, aber keine Berechtigung vorhanden.
- **404 Not Found** — die angeforderte Ressource existiert nicht.

### 5xx — Server-Fehler

- **500 Internal Server Error** — ein allgemeiner serverseitiger Fehler.
- **502 Bad Gateway** — ein Proxy erhielt eine ungültige Antwort von einem anderen Server.
- **503 Service Unavailable** — der Server ist vorübergehend überlastet oder ausgefallen.

### Verwendung von Statuscodes in Express

- `res.status(400).send("Bad Request")` — sendet einen 400 mit einer Klartextnachricht.
- `res.status(200).json({ message: "Success" })` — sendet einen 200 mit einem JSON-Körper.
- `res.status(500).send("Internal Server Error")` — sendet einen 500 mit einer Nachricht.

## Fehlerbehandlung in Express

- **Fehlerbehandlungs-Middleware** hat vier Parameter: `(err, req, res, next)`. Die Signatur mit vier Parametern signalisiert Express, dass es sich um einen Fehler-Handler handelt.
- Platzieren Sie die Fehlerbehandlungs-Middleware **am Ende** des Middleware-Stacks, nach allen Routen.

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Asynchrone Fehlerbehandlung in Express 4

- Asynchrone Fehler werden in Express 4 **nicht automatisch abgefangen**.
- Verpacken Sie asynchrone Logik in `try/catch` und rufen Sie `next(err)` auf, um den Fehler an den Fehler-Handler weiterzuleiten.

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- Ein `asyncHandler`-Wrapper vermeidet es, `try/catch` in jeder Route zu wiederholen:

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Asynchrone Fehlerbehandlung in Express 5

- In Express 5 werden asynchrone Fehler und abgelehnte Promises **automatisch an den Fehler-Handler weitergeleitet** — kein manuelles `next(err)` mehr nötig.

### Behandlung von 404-Fehlern

- Fügen Sie eine Catch-All-Middleware **nach den Routen, aber vor dem Fehler-Handler** hinzu, um nicht gefundene Routen zu behandeln:

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Best Practices für die Fehlerbehandlung

- Platzieren Sie Fehlerbehandlungs-Middleware immer zuletzt.
- Behandeln Sie 404-Fehler getrennt von anderen Fehlern.
- Protokollieren Sie in der Produktion Fehlerdetails intern — senden Sie niemals Stacktraces an Benutzer.

## Debugging und Logging in Express

- **Eingebautes Debugging**: Setzen Sie die Umgebungsvariable `DEBUG`, um die eingebaute Debug-Ausgabe von Express zu aktivieren.
  - `DEBUG=express:* node index.js` — aktiviert alle Express-Debug-Nachrichten.
  - `DEBUG=express:router node index.js` — zeigt nur Router-bezogene Debug-Ausgaben.
  - PowerShell: `$env:DEBUG = "express:*"; node index.js`
  - CMD: `set DEBUG=express:* && node index.js`
- **`morgan`**: eine beliebte Middleware zum Protokollieren von HTTP-Anfragen (Methode, URL, Statuscode, Antwortzeit).

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- **Debugging** verfolgt, wie Express Anfragen und Middleware verarbeitet, ohne die Produktionsprotokolle zu überladen.
- **Logging** zeichnet auf, was bei Anfragen passiert — nützlich, um Bugs zu erkennen und den Traffic zu verstehen.

## Gesundheitsprüfungen und sanfte Abschaltungen

### Gesundheitsprüfungen

- Eine **Gesundheitsprüfungs-Route** meldet, ob die App korrekt läuft.
- Load Balancer und Orchestratoren (zum Beispiel Kubernetes, AWS Elastic Beanstalk) fragen diese Route ab, um zu entscheiden, ob der Dienst gesund ist.
- Eine **200**-Antwort bedeutet gesund; keine Antwort löst einen automatischen Neustart aus.

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### Sanfte Abschaltung

- Eine **sanfte Abschaltung** erlaubt der App, bestehende Anfragen zu beenden, bevor sie stoppt, anstatt Verbindungen sofort zu trennen.
- Hören Sie auf das `SIGTERM`-Signal (gesendet, wenn der Prozess beendet werden soll) und rufen Sie `server.close()` auf:

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- Hören Sie auf das `SIGINT`-Signal (gesendet, wenn der Entwickler während der lokalen Entwicklung Strg+C drückt) und rufen Sie dieselbe Abschaltlogik wie `SIGTERM` auf.

### Die App vor der Abschaltung als nicht gesund markieren

- Geben Sie während der Abschaltung eine **503** von der Gesundheitsprüfung zurück, damit der Load Balancer keine neuen Anfragen mehr an die App weiterleitet.

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
