# --description--

Semantische Versionierung, oder SemVer, ist ein System, das Versionsnummern eine klare Bedeutung gibt, sodass Entwickler schnell verstehen können, welche Art von Änderungen an der Software vorgenommen wurden.

SemVer verwendet dieses Format:

```bash
MAJOR.MINOR.PATCH
```

Zum Beispiel ist die beliebte Bibliothek `lodash` derzeit in der Version `4.17.21`.

Das bedeutet jeder Teil:

* MAJOR — Große Änderungen, die bestehenden Code brechen. Beispiel: Entfernen oder Umbenennen einer Funktion, von der bestehender Code abhängen könnte.
    
* MINOR — Neue Funktionen, die bestehenden Code nicht brechen. Beispiel: Hinzufügen einer neuen Funktion oder eines optionalen Parameters.
    
* PATCH — Kleine Fehlerbehebungen und Verbesserungen. Beispiel: Beheben eines Bugs oder Tippfehlers, ohne die Funktionsweise zu ändern.
    

<img src="https://cdn.freecodecamp.org/curriculum/lecture-transcripts/what-is-semantic-versioning-1.png" alt="In der Versionsnummer 4.17.21 ist 4 die Major-Version, sie ändert sich bei Breaking Changes, die 17 ist die Minor-Version, sie ändert sich bei nicht-brechenden Änderungen, und die 21 ist der Patch, sie ändert sich bei Bugfixes und kleinen Funktionen">

Angenommen, Sie haben ein Paket namens `my-package`. Hier ist ein Beispiel, wie sich seine Versionsnummer im Laufe der Zeit mit semantischer Versionierung ändern könnte:

| Version | Was sich geändert hat |
| --- | --- |
| 1.0.0 | Erste stabile Veröffentlichung |
| 1.1.0 | Neue optionale Funktion |
| 1.1.1 | Kleine Fehlerbehebung oder Funktion |
| 2.0.0 | Einführung von Breaking Changes |

Aber es gibt noch mehr zu Versionsnummern in npm. Wenn Sie ein Paket installieren, könnte die Version in Ihrer `package.json`-Datei so aussehen:

```json
"lodash": "^4.17.21" 
```

Sie haben bereits gelernt, dass `4` die Major-Version ist, `17` die Minor-Version und `21` die Patch-Version. Aber was bedeutet das Caret-Zeichen (`^`) davor?

Das Caret-Zeichen erlaubt npm, automatisch neuere Minor- und Patch-Versionen zu installieren (`4.x.x`), aber keine neueren Major-Versionen (`5.x.x`), weil Major-Updates Breaking Changes enthalten können.

Wenn die Major-Version `0` ist, wird das Caret restriktiver. Zum Beispiel würde `^0.2.3` Updates wie `0.2.4` erlauben, aber nicht `0.3.0`, da Major-Versionen, die mit `0` beginnen, als weniger stabil gelten.

Das Caret ist nicht das einzige Symbol, das `npm` und andere JavaScript-Paketmanager zur Verwaltung von Abhängigkeitsversionen verwenden.

Es gibt das Tilde-Symbol (`~`), das nur Updates der Patch-Version erlaubt. Also würde `~1.2.3` `1.2.4` und `1.2.9` akzeptieren, aber niemals `1.3.0`.

Sie können auch die Symbole größer als (`>`), kleiner als (`<`) und deren „oder gleich“-Varianten verwenden, um Versionen in einem Bereich zu akzeptieren. Zum Beispiel bedeutet `>=1.2.3 <2.0.0` jede Version von `1.2.3` bis, aber nicht einschließlich, `2.0.0`.

Das Sternchen-Symbol (`*`) entspricht jeder Version, es akzeptiert also alles. Das ist nur ideal zum Testen und nicht für die Produktion empfohlen.

Schließlich können Sie eine genaue Version festlegen, indem Sie die Versionsnummer ohne Präfix angeben. Zum Beispiel akzeptiert die Version `1.2.3` nur `1.2.3`.

Hier eine kurze Zusammenfassungstabelle:

| Symbol | Bedeutung | Beispielbereich |
| --- | --- | --- |
| `^` | Erlaubt Minor- und Patch-Updates | `^1.2.3` → `1.x.x` |
| `~` | Erlaubt nur Patch-Updates | `~1.2.3` → `1.2.x` |
| `>`, `<`, `>=`, `<=` | Benutzerdefinierter Bereich | `>=1.2.3` oder `<2.0.0` |
| **\*** | Jede Version | alle Versionen |
| keine | Exakte Version | `1.2.3` |
