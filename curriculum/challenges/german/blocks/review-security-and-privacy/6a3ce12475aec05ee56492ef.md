## Sicherheit vs Datenschutz

**Sicherheit** schützt Daten vor unbefugtem Zugriff, Diebstahl und böswilligen Schäden. Sie baut technische Verteidigungen wie Firewalls, Verschlüsselung und Antivirensoftware auf, um Angreifer von Ihren Systemen fernzuhalten.

**Datenschutz** steuert, wie Organisationen persönliche Daten sammeln, speichern, verwenden und teilen. Er gibt den Nutzenden die Kontrolle darüber, wer ihre Informationen sehen darf.

Die beiden Konzepte sind eng miteinander verbunden, aber unterschiedlich:

- Sie können Sicherheit ohne Datenschutz haben. Ein Unternehmen kann Ihre Daten in einer hochsicheren Datenbank speichern und sie dennoch ohne Ihre Zustimmung an Werbetreibende verkaufen. In diesem Fall ist die Sicherheit intakt, aber der Datenschutz wird verletzt.
- Sie können keinen echten Datenschutz ohne Sicherheit haben. Wenn ein Server mit privaten Daten gehackt wird, ist jedes Datenschutzversprechen sofort gebrochen.

Sicherheit bietet den technischen Schutzmechanismus, während Datenschutz die rechtlichen Rechte und Richtlinien zur Datennutzung regelt.

## HTTPS

**HTTP** überträgt Daten im Klartext, sodass sie für jeden lesbar sind, der sie abfängt. **HTTPS** (Hypertext Transfer Protocol Secure) verwendet **TLS (Transport Layer Security)**, um den Kommunikationskanal zwischen Browser und Server zu verschlüsseln. TLS hat das ältere **SSL (Secure Sockets Layer)**-Protokoll ersetzt, obwohl viele Webhosting-Plattformen heute noch SSL in ihren Einstellungen erwähnen.

Wichtige Vorteile von HTTPS:

- **Verschlüsselung:** Verschlüsselt Daten in unlesbaren Code. Nur der Browser und der Server besitzen die Schlüssel zum Entschlüsseln.
- **Authentifizierung:** Erfordert, dass der Seiteninhaber ein digitales Zertifikat von einer vertrauenswürdigen Organisation erhält, das die Legitimität der Website bestätigt.
- **Datenintegrität:** Verhindert, dass Daten während der Übertragung verändert werden; Sie erhalten genau das, was der Server gesendet hat.
- **Schutz bei öffentlichem WLAN:** Erstellt einen privaten Tunnel durch offene Netzwerke und verbirgt Ihren Datenverkehr vor Angreifern in der Nähe.

Moderne Browser warnen Nutzer, wenn eine Seite nur HTTP verwendet, und verlangen HTTPS für erweiterte Funktionen wie Geolokalisierung und Push-Benachrichtigungen.

## Same-Origin-Policy und CORS

Die **Same-Origin-Policy (SOP)** ist eine Browser-Sicherheitsregel, die verhindert, dass eine Webseite Daten von einer anderen Herkunft liest. Eine **Herkunft** besteht aus drei Teilen, die alle exakt übereinstimmen müssen:

- **Protokoll** (z. B. `https`)
- **Domainname** (z. B. `example.com`)
- **Portnummer** (z. B. `443`, der HTTPS-Standard, und `80`, der HTTP-Standard)

Diese Regel verhindert, dass bösartige Seiten, die in einem Browser-Tab geöffnet sind, Daten von einer Bankseite in einem anderen Tab stehlen.

**CORS (Cross-Origin Resource Sharing)** erlaubt es Servern, kontrolliertes Teilen von Daten über verschiedene Herkünfte hinweg mittels HTTP-Headern zu ermöglichen:

- Der Browser sendet bei jeder Anfrage an eine andere Herkunft einen `Origin`-Header mit.
- Wenn der Server dieser Herkunft vertraut, antwortet er mit einem `Access-Control-Allow-Origin`-Header.
- Der Browser erlaubt dann das Lesen der Antwortdaten; andernfalls blockiert er sie.

In Express übernimmt das `cors`-Paket diese Aufgabe:

```javascript
// development: allow all origins
app.use(cors());

// production: restrict to trusted domains
app.use(cors({ origin: ['https://yourapp.com', 'https://admin.yourapp.com'], credentials: true }));
```

## Cookies: Speicherung, Sicherheit und Datenschutz

**Cookies** sind kleine Textdateien, die Websites auf Ihrem Gerät speichern, um Informationen zwischen Besuchen zu merken. Der Server sendet einen `Set-Cookie`-Header; der Browser speichert die Datei und sendet sie bei jeder weiteren Anfrage automatisch zurück.

Sicherheitsflags zum Schutz von Cookies:

- **`HttpOnly`:** Verhindert, dass JavaScript das Cookie lesen kann, um den Diebstahl von Sitzungstokens durch XSS zu verhindern.
- **`Secure`:** Stellt sicher, dass das Cookie nur über HTTPS übertragen wird, um es vor Abfangen in öffentlichem WLAN zu schützen.
- **`SameSite=Strict`:** Verhindert, dass der Browser das Cookie bei Anfragen von anderen Seiten sendet, und blockiert so CSRF-Angriffe.
- **`SameSite=Lax`:** Erlaubt Cookies bei sicherer Navigation auf oberster Ebene, blockiert sie aber bei Formularübermittlungen und API-Anfragen von anderen Seiten.

**Datenschutzimplikationen:**

- **First-Party-Cookies** verfolgen Ihre Aktionen nur auf der Website, die Sie gerade besuchen.
- **Third-Party-Cookies** werden von Werbenetzwerken gesetzt und verfolgen Sie über völlig verschiedene Websites hinweg, um detaillierte Profile Ihrer Interessen zu erstellen.

Datenschutzgesetze wie die DSGVO verlangen von Websites, Cookie-Zustimmungsbanner anzuzeigen und Nutzenden das Recht zu geben, Tracking-Cookies abzulehnen. Moderne Browser blockieren standardmäßig Third-Party-Tracking-Cookies.

## Situationen, in denen Sicherheitsprobleme auftreten

Sicherheitslücken entstehen selten zufällig. Häufige Auslöser sind:

- **Eile beim Veröffentlichen:** Entwicklungsteams überspringen gründliche Code-Reviews und Sicherheitstests, um enge Fristen einzuhalten, was ungepatchte Bugs und fehlende Zugriffskontrollen hinterlässt.
- **Komplexe Systemintegration:** Jede externe Bibliothek, Drittanbieter-API oder Microservice erhöht das Risiko in der Lieferkette. Eine Schwachstelle in einer Abhängigkeit breitet sich auf die Hauptanwendung aus.
- **Schlechte Konfigurationsverwaltung:** Das Bereitstellen von Servern oder Cloud-Datenbanken mit werkseitigen Standardeinstellungen lässt oft leere Admin-Passwörter, offene Ports und öffentlichen Datenzugriff zu. Automatisierte Scanner suchen nach solchen Fehlkonfigurationen.
- **Menschliches Versagen:** Mitarbeitende, die durch Phishing-E-Mails, betrügerische SMS oder täuschende Anrufe angegriffen werden, können versehentlich Zugangsdaten preisgeben oder Daten an Angreifer übertragen.
- **Organisatorische Fusionen:** Das schnelle Verbinden zweier unterschiedlicher Firmennetzwerke schafft blinde Flecken wie verlorene Sichtbarkeit über alte Server, falsch konfigurierte Firewalls und zu großzügige Zugriffsrechte.

## Häufige Sicherheitsbedrohungen und Gegenmaßnahmen

| Bedrohung | Funktionsweise | Gegenmaßnahme |
|---|---|---|
| **Phishing** | Gefälschte E-Mails imitieren vertrauenswürdige Marken, um Zugangsdaten zu stehlen | Schulungen zur Sicherheitsbewusstseins; E-Mail-Authentifizierungsprotokolle (SPF, DKIM, DMARC) |
| **Malware (Ransomware, Spyware)** | Verschlüsselt oder stiehlt Dateien des Opfers; fordert Lösegeld | Automatisches Patchen; Antimalware-Software |
| **Credential stuffing** | Automatisierte Tools testen Millionen gestohlener Benutzername-/Passwort-Kombinationen auf Websites | Erzwingen von Multi-Faktor-Authentifizierung (MFA) |
| **DDoS** | Überflutet einen Server mit gefälschtem Datenverkehr, um ihn zum Absturz zu bringen | Cloud-basierte Traffic-Filterdienste (z. B. Cloudflare) |
| **Insider-Bedrohungen** | Unzufriedene oder nachlässige Mitarbeitende leaken oder stehlen Daten | Prinzip der minimalen Rechtevergabe; Überwachung von Systemprotokollen auf ungewöhnliche Zugriffe |

## Content Security Policy (CSP) und Permissions-Policy

**Content Security Policy (CSP)** ist ein HTTP-Antwortheader, der einschränkt, welche externen Ressourcen ein Browser auf einer Seite laden darf. Er schützt hauptsächlich vor **Cross-Site Scripting (XSS)**-Angriffen, indem er vertrauenswürdige Quell-Domains für Skripte, Styles und Bilder auf eine Whitelist setzt.

Versucht ein bösartiges Skript von einer nicht genehmigten Domain auszuführen, blockiert der Browser es und protokolliert einen Sicherheitsfehler.

**Permissions-Policy** (früher Feature-Policy) steuert den Zugriff auf Browserfunktionen und Gerätehardware wie Webcam, Mikrofon und Geolokalisierung. Sie können eine Funktion auf Ihre eigene Domain beschränken oder komplett deaktivieren:

```http
Permissions-Policy: camera=(self), geolocation=()
```

`camera=(self)`: nur Ihre eigene Domain darf auf die Kamera zugreifen.
`geolocation=()`: Geolokalisierung ist für alle auf der Seite deaktiviert.

Gemeinsam sperren CSP die Datenübertragung ab und verhindern Code-Injektionen; Permissions-Policy schützt Gerätehardware und sorgt für Datenschutzkonformität.

## Grundlegende Konzepte im Datenschutz für Nutzer

| Konzept | Definition |
|---|---|
| **PII (Personally Identifiable Information)** | Alle Daten, die eine Person direkt oder indirekt identifizieren können. Direkte Identifikatoren: Name, Adresse, Ausweisnummer. Indirekte Identifikatoren: IP-Adresse, Gerätenummer, Standortverlauf. |
| **Vertraulichkeit** | Sicherstellen, dass private Daten durch Verschlüsselung und Zugriffskontrollen unzugänglich für Unbefugte bleiben. |
| **Datenminimierung** | Nur die minimale Menge an PII sammeln, die zur Erfüllung einer bestimmten Aufgabe nötig ist. |
| **Zweckbindung** | Für einen Zweck erhobene Daten dürfen rechtlich nicht für einen anderen Zweck verwendet werden (z. B. Verkauf von Kontaktdaten an Werbetreibende). |
| **Tracking** | Kontinuierliche Überwachung der Online-Aktivitäten mittels Skripten, Geräte-Fingerprinting und Browser-Verlauf, um Nutzerprofile zu erstellen. |
| **Informierte Einwilligung** | Nutzer müssen aktiv der Datenerhebung zustimmen. Vorgekreuzte Kästchen gelten nicht als gültige Zustimmung. |
| **Recht auf Vergessenwerden** | Nutzer können rechtlich verlangen, dass eine Organisation alle ihre persönlichen Daten und Tracking-Historie dauerhaft löscht. |

## Regionale Datenschutzgesetze

| Gesetz | Zuständigkeit | Wichtige Anforderungen |
|---|---|---|
| **DSGVO** | Europäische Union | Strenge Opt-in-Zustimmung erforderlich; Nutzer haben Recht auf Zugriff und Löschung ihrer Daten; hohe Bußgelder bei Verstößen. |
| **CCPA / CPRA** | Kalifornien, USA | Opt-out-Modell; Unternehmen müssen einen Link „Do Not Sell My Personal Information“ anzeigen. |
| **COPPA** | USA (Bundesebene) | Verifizierbare elterliche Zustimmung erforderlich, bevor Daten von Kindern unter 13 Jahren gesammelt werden. |
| **DPA / UK DSGVO** | Vereinigtes Königreich | Rechtmäßige, faire und transparente Datenverarbeitung; Personen haben Zugriffs- und Löschrechte. |
| **NDPA** | Nigeria | Regelt Sammlung, Speicherung und Nutzung persönlicher Daten nigerianischer Einwohner. |
| **DPDP** | Indien | Regelt, wie persönliche Daten indischer Einwohner verwendet werden. |

**Praktische Schritte zur Einhaltung:**

1. **Standortabhängige Zustimmungsbanner:** Erkennen Sie den geografischen Standort des Nutzers und zeigen Sie Cookie-Banner entsprechend den lokalen Vorschriften an.
2. **Automatisiertes DSAR-System:** Bearbeiten Sie Anfragen zur Auskunft über personenbezogene Daten sicher, um den gesamten Datenverlauf eines Nutzers zu exportieren oder zu löschen.
3. **Datenminimierung und Altersprüfung:** Sammeln Sie nur das Notwendige und verwenden Sie Altersverifikation zum Schutz Minderjähriger.
4. **Datenschutzbeauftragte (DPO):** Bestimmen Sie eine Fachkraft, die die Datenschutzarchitektur überwacht und die Einhaltung der Vorschriften sicherstellt.

## Die `.env`-Datei und die `dotenv`-Bibliothek

Eine **`.env`-Datei** ist eine Textdatei, die im Stammverzeichnis eines Projekts liegt und sensible Konfigurationsdaten als Schlüssel-Wert-Paare speichert. Dazu gehören API-Schlüssel, Datenbankpasswörter und Portnummern. So bleiben sie außerhalb des Quellcodes und sind nicht öffentlich zugänglich.

```bash
PORT=3000
DB_HOST=localhost
API_KEY=xyz123secretkey
```

Fügen Sie `.env` immer zu `.gitignore` hinzu, damit sie nie in die Versionskontrolle gelangt.

Das **`dotenv`-Paket** liest die `.env`-Datei beim Start und injiziert die Schlüssel in `process.env`:

```javascript
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

Zwei wichtige Vorteile:

- **Sicherheit:** Sensible Zugangsdaten erscheinen nie im Quellcode oder in der Versionshistorie.
- **Flexibilität:** Dieselbe Codebasis läuft in Entwicklung, Test und Produktion, indem einfach die `.env`-Datei auf jedem Server ausgetauscht wird.
