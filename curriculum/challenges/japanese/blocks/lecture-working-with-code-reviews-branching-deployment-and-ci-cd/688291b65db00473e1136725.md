# --description--

CIはContinuous Integration（継続的インテグレーション）の略で、CDはContinuous Delivery（継続的デリバリー）またはContinuous Deployment（継続的デプロイ）の略です。これらのプロセスを組み合わせることで、変更が正しく動作することを確認した後に、自動的にアプリケーションへ変更をデプロイできます。

CIパイプラインでは、コードに対してリンターやテストを実行することがあります。CIパイプラインは通常、プルリクエストで実行され、変更がアプリを壊していないかを確認し、また`main` ブランチでも実行されて、デプロイの準備ができているかをチェックします。

CDパイプラインは、プレビュー環境を設定して変更をライブで確認できるようにすれば、プルリクエストでも実行される場合があります。しかし、`main` ブランチでも実行され、変更が承認されてマージされると継続的にデプロイされます。

では、これらはGitHub Actionsで実際にどのように動作するのでしょうか？まず、GitHub Actionsとは何かを理解する必要があります。GitHub Actionsは一時的なランナー（仮想の一時的なコンピューターのようなもの）を提供し、特定のコマンドを実行できます。

ActionsはYAMLファイルで設定します。異なるアクションを実行するために複数のファイルを持つことも可能です。例えば、CI用のファイルとCD用のファイルを分けることができます。

基本的なCIの例を見てみましょう：

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  lint:
    name: Lint and Test
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Source Files
        uses: actions/checkout@v4

      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10

      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

内容が多いので分解して説明します。最初のセクションを見てみましょう：

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```

`name` はアクション自体の名前を定義します。これはプルリクエストのステータスチェックのセクションなどに表示されます。`on` プロパティは「ワークフロートリガー」と呼ばれ、アクションを実行するきっかけとなるイベントを定義します。

この例では、コミットが`main` にプッシュされたとき、またプルリクエストが`main` をターゲットにしたときにアクションが実行されます。つまり、誰かがプルリクエストを作成したときと、マージしたときにCIが動作します。

`jobs` セクションは、アクションが実際に何をするかを定義する場所です。`lint:` キーはジョブの内部名を定義する任意のキーです。`name` プロパティはこのジョブの外部名を定義します。そして`runs-on` プロパティはアクションを実行する環境の種類を定義します。この場合は、最新のサポートされているUbuntuバージョンで実行しています。

`steps` プロパティが実際の処理を行う部分です。このブロックはアクションが実行すべきステップを定義しています。最初のブロックを見てみましょう：

```yml
      - name: Checkout Source Files
        uses: actions/checkout@v4
```

このステップには2つのプロパティがあります。`name` と`uses` プロパティです。この`uses` プロパティは特別で、アクションランナーにGitHubから公開されているアクションを取得して実行するよう指示します。`actions/checkout` アクションはリポジトリのクローン作成とセットアップを行います。

```yml
      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10
```

次の2つのブロックもサードパーティのアクションを使っています。これらのステップはNodeとpnpmパッケージマネージャーをセットアップします。どちらも`with` キーを持ち、これを使ってサードパーティアクションに引数を渡せます。これは関数に引数を渡すようなイメージです。

最後の4つのブロックが本当の処理部分です。

```yml
      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

`Install Dependencies` ブロックはnpmパッケージをインストールします。`Lint Source Files` ブロックはリンターを実行し、`Verify Build` ブロックはTypeScriptコードのコンパイルを確認し、`Run Tests` ブロックはユニットテストを実行します。

アクションで重要なのは、デフォルトではこれらのステップのどれかが失敗すると、残りのステップはスキップされ、実行全体が失敗としてマークされることです。

これらのYAMLファイルはどこに置けばGitHub Actionsで実行できるでしょうか？GitHubが見つけて利用できるように、リポジトリの`.github/workflows` ディレクトリに置く必要があります。

アクションには他にも多くの機能があり、ドキュメントを読んだりfreeCodeCampリポジトリの既存アクションを調べたりすることをお勧めします。ですが今は、GitHub ActionsとCI/CDの基本を十分理解できたはずです！
