# --description--

Il modello di richiesta-risposta HTTP è il pattern di comunicazione fondamentale che alimenta il World Wide Web (WWW). Ha molti componenti che lo rendono efficiente, così tutti possiamo consultare i materiali dal web.

Vediamo cos’è il modello di richiesta-risposta HTTP, quali sono i componenti che lo costituiscono e come questi componenti lavorano insieme per portare efficienza a ciò che alimenta il web.

Il modello di richiesta-risposta HTTP è un pattern di comunicazione nelle interazioni web in cui un client, solitamente un browser, invia una **richiesta HTTP** a un server, e il server elabora la richiesta e invia indietro una **risposta HTTP**.

Ecco i componenti che costituiscono il modello di richiesta-risposta HTTP:

* **Client**: un browser, un’app mobile o qualsiasi altro software che invia richieste HTTP al server e interpreta le risposte HTTP che riceve.
    
* **Richiesta**: il messaggio inviato dal client al server. Include il metodo, l’URL, le intestazioni e un corpo opzionale.
    
* **Metodo HTTP**: l’azione da eseguire — `GET` per recuperare dati, `POST` per inviare dati, `PATCH` e `PUT` per modificare dati, e `DELETE` per rimuovere dati.
    
* **URL (Uniform Resource Locator)**: la risorsa richiesta al server.
    
* **Intestazioni**: i metadati inviati con la richiesta o la risposta. Per esempio, `Content-Type` e `Authorization`.
    
* **Corpo**: i dati inviati con le richieste `POST`, `PUT`, `PATCH`, `DELETE`, spesso in formato JSON o dati di modulo.
    
* **Server**: il sistema back-end che elabora la richiesta e genera una risposta. Può essere scritto in qualsiasi linguaggio di programmazione che supporti la programmazione lato server.
    
* **Risposta**: il messaggio inviato indietro dal server. Contiene un codice di stato, intestazioni e un corpo opzionale.
    
* **Codice di stato o codice di risposta**: un codice numerico che indica successo, errori o reindirizzamenti.
    
* **Corpo della risposta**: i dati effettivi inviati indietro, tipicamente in formato HTML, JSON o XML.

Come lavorano insieme tutti questi componenti nel ciclo di richiesta-risposta HTTP?

1. **Il client avvia una richiesta**
    
    * Un browser, un’app mobile o un client API invia una richiesta HTTP destinata al server.
        
    * La richiesta include un metodo come `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, URL, intestazioni e a volte un corpo.
        
2. **La richiesta viene inviata al server**
    
    * La richiesta viaggia su internet fino al server, un sistema back-end scritto con un linguaggio di programmazione.
        
    * Il server riceve la richiesta ed estrae le informazioni rilevanti.
        
3. **Il server elabora la richiesta**
    
    * Il server controlla il metodo della richiesta e l’URL.
        
    * Potrebbe dover validare l’autenticazione (tramite intestazioni come authorization).
        
    * Se necessario, interagisce con un database per recuperare o modificare i dati.
        
4. **Il server prepara una risposta**
    
    * Il server genera una risposta con un codice di stato come 200, 404, 500 e così via.
        
    * La risposta include intestazioni, per esempio `Content-Type: application/json`.
        
    * Se applicabile, viene aggiunto un corpo della risposta in JSON, HTML o XML.
        
5. **La risposta viene inviata al client**
    
    * Il server invia la risposta indietro su internet.
        
    * Il client riceve la risposta ed elabora i dati.
        
6. **Il client gestisce la risposta**
    
    * Se la risposta ha successo (codice di stato 200), il client mostra i dati, per esempio renderizzandoli su una pagina web.
        
    * Se si verifica un errore (`404 Not Found`, `500 Internal Server Error` e così via), il client può mostrare un messaggio di errore.

Questo ciclo si ripete per ogni nuova richiesta che il client fa.
