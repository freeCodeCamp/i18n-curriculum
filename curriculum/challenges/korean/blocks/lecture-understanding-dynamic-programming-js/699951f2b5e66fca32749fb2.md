# --description--
  
동적 프로그래밍은 복잡한 문제를 더 단순한 하위 문제로 나누고 결과를 저장하여 중복 계산을 피하는 알고리즘 기법입니다. 이 방법은 보통 지수 시간 복잡도가 걸리는 문제를 다항 시간 내에 해결할 수 있도록 바꿉니다.

## 동적 프로그래밍의 핵심 원칙

동적 프로그래밍은 문제에 두 가지 핵심 조건이 있을 때 작동합니다.

- **중복되는 하위 문제**: 큰 문제를 해결할 때 같은 작은 문제가 여러 번 나타납니다. 이 하위 문제들을 반복해서 다시 계산하는 대신, 그 해답을 저장합니다.
    
- **최적 부분 구조**: 문제의 최적 해답은 하위 문제들의 최적 해답을 포함합니다. 즉, 작은 부분들의 최적 해답을 결합해 전체 최적 해답을 만들 수 있습니다.
    

이 개념들을 고전적인 "계단 오르기" 문제를 통해 살펴보겠습니다.

## 단순 재귀의 문제점

계단 오르기 문제를 생각해보세요: 계단이 `n`개 있고 한 번에 1칸 또는 2칸씩 오를 수 있습니다. 정상에 도달하는 서로 다른 방법은 몇 가지일까요?

```js
function climbStairsRecursive(n) {
  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Recursive rule: to reach step n, you can come from step (n-1) or step (n-2)
  return climbStairsRecursive(n - 1) + climbStairsRecursive(n - 2);
}
```

이 구현은 중복 계산이 많아 지수 시간 복잡도를 가집니다. `climbStairsRecursive(5)`을 계산할 때 다음과 같은 일이 일어납니다:

- `climbStairsRecursive(5)`가 `climbStairsRecursive(4)`과 `climbStairsRecursive(3)`를 호출합니다.
    
- `climbStairsRecursive(4)`가 `climbStairsRecursive(3)`과 `climbStairsRecursive(2)`를 호출합니다.
    
- `climbStairsRecursive(3)`은 **두 번** 계산됩니다.
    
- `climbStairsRecursive(3)`가 `climbStairsRecursive(2)`과 `climbStairsRecursive(1)`을 호출합니다.
    
- `climbStairsRecursive(2)`는 총 **3번** 계산됩니다.
    

`n=5`만 해도 5개의 고유 계산이 필요한데 9번의 함수 호출이 발생합니다. `n`가 커질수록 이 중복은 지수적으로 폭발합니다 — `climbStairsRecursive(30)`는 수백만 번의 함수 호출이 필요할 것입니다! 시간 복잡도는 `O(2^n)`이 되어 큰 `n` 값에 비효율적이고 실용적이지 않습니다.

## 동적 프로그래밍 해결법

동적 프로그래밍은 두 가지 주요 방법으로 중복 계산을 없앱니다:

### 메모이제이션 (탑다운 방식)

메모이제이션은 비용이 큰 함수 호출 결과를 저장하고 같은 입력이 다시 들어오면 저장된 결과를 반환합니다:

```js
function climbStairsMemo(n, memo = {}) {
  // If already calculated, return cached value
  if (memo[n] !== undefined) {
    return memo[n];
  }

  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Calculate once and store in memo for future use
  memo[n] =
    climbStairsMemo(n - 1, memo) +
    climbStairsMemo(n - 2, memo);

  return memo[n];
}
```

메모이제이션은 `1`부터 `n`까지 각 고유 값이 정확히 한 번만 계산되기 때문에 훨씬 효율적입니다. `climbStairsMemo(3)`이 다시 필요할 때 재계산하지 않고 `O(1)` 시간에 메모 딕셔너리에서 바로 조회합니다.

탑다운 방식으로 `climbStairsMemo(5)` 실행을 추적해 메모이제이션이 중복 작업을 어떻게 없애는지 살펴봅시다:

```md
Call: climbStairsMemo(5)
  memo = {} (empty)
  
  Call: climbStairsMemo(4) 
    memo = {} (empty)
    
    Call: climbStairsMemo(3)
      memo = {} (empty)
      
      Call: climbStairsMemo(2) → returns 2 (base case)
      Call: climbStairsMemo(1) → returns 1 (base case)
      
      Result: 2 + 1 = 3
      memo = {3: 3} (stored!)
    
    Call: climbStairsMemo(2) → returns 2 (base case)
    
    Result: 3 + 2 = 5
    memo = {3: 3, 4: 5} (stored!)
  
  Call: climbStairsMemo(3) → returns 3 (FROM MEMO - no recursion!)
  
  Result: 5 + 3 = 8
  memo = {3: 3, 4: 5, 5: 8}
```

**효율성 비교**

- **단순 재귀**: 9번 함수 호출, 중복 계산 많음
    
- **메모이제이션**: 5번 고유 계산만 수행 후 저장된 결과 재사용
    
- **시간 복잡도**: `O(2^n)`에서 `O(n)`로 감소, `n`개의 고유 계산만 수행
    
- **공간 복잡도**: 메모 저장소와 호출 스택에 `O(n)`
    
- **실제 효과**: `climbStairsMemo(30)`가 수백만 번의 호출에서 약 30번의 고유 계산으로 줄어듭니다!
    

### 테이블화 (바텀업 방식)

테이블화는 하위 문제들의 해답을 표에 채워가며 아래에서 위로 해결책을 만듭니다:

```js
function climbStairsTabulation(n) {
  if (n <= 2) {
    return n;
  }

  // Create an array to store solutions from 0 to n
  const dp = new Array(n + 1).fill(0);

  // Base cases
  dp[1] = 1; // 1 way to reach step 1
  dp[2] = 2; // 2 ways to reach step 2

  // Build up the solution iteratively
  for (let i = 3; i <= n; i++) {
    // Ways to reach step i = ways to reach (i-1) + ways to reach (i-2)
    dp[i] = dp[i - 1] + dp[i - 2];
  }

  return dp[n];
}
```

테이블화는 재귀를 완전히 없애고 가장 작은 하위 문제부터 목표까지 반복적으로 해결책을 만듭니다.

바텀업 방식을 실제로 살펴보며 `climbStairsTabulation(5)`을 체계적으로 만드는 과정을 봅시다:

```md
Initial state:
dp = [0, 1, 2, 0, 0, 0]
     [0, 1, 2, 3, 4, 5] ← indices (step numbers)

Step by step construction:

i = 3:
  dp[3] = dp[2] + dp[1] = 2 + 1 = 3
  dp = [0, 1, 2, 3, 0, 0]
  
i = 4:
  dp[4] = dp[3] + dp[2] = 3 + 2 = 5
  dp = [0, 1, 2, 3, 5, 0]
  
i = 5:
  dp[5] = dp[4] + dp[3] = 5 + 3 = 8
  dp = [0, 1, 2, 3, 5, 8]

Final result: dp[5] = 8
```

**테이블화의 주요 장점**

- **재귀 오버헤드 없음**: 메모이제이션과 달리 재귀 호출 스택이 없습니다.
    
- **예측 가능한 실행**: 1, 2, 3, 4, 5... 순서대로 값을 계산합니다.
    
- **캐시 친화적**: 연속된 배열 접근으로 메모리 사용 최적화
    
- **최적화 용이**: 마지막 두 값만 필요하므로 공간 복잡도를 `O(1)`로 줄일 수 있습니다.
    

```js
function climbStairsOptimized(n) {
  if (n <= 2) {
    return n;
  }

  // Store only the last two results to save space
  let prev2 = 1; // ways to reach step 1
  let prev1 = 2; // ways to reach step 2

  for (let i = 3; i <= n; i++) {
    const current = prev1 + prev2;

    // Shift values forward
    prev2 = prev1;
    prev1 = current;
  }

  return prev1;
}
```

**효율성 비교**

- **단순 재귀**: `n=5`에 대해 9번 함수 호출, 지수적 증가
    
- **테이블화**: `n=5`에 대해 3번 간단한 덧셈, 선형 증가
    
- **시간 복잡도**: `O(n)` 대신 `O(2^n)`
    
- **공간 복잡도**: 배열에 `O(n)`, 최적화 시 `O(1)`
    
- **예측 가능한 성능**: 큰 입력에도 스택 오버플로우 위험 없음
    

두 방법 모두 시간 복잡도를 지수적 `O(2^n)`에서 선형 `O(n)`으로 줄여, 문제를 밀리초 내에 해결할 수 있게 해줍니다. 큰 입력에선 수년을 기다리는 것과 큰 차이가 납니다.

## 실제 적용 사례

동적 프로그래밍은 컴퓨터 과학뿐 아니라 다양한 분야에 널리 쓰입니다:

- **경로 최적화**: GPS 시스템은 위치 간 최단 경로를 찾기 위해 동적 프로그래밍 알고리즘을 사용합니다.
    
- **텍스트 처리**: 맞춤법 검사기와 자동 완성 기능은 단어 간 편집 거리를 계산할 때 동적 프로그래밍을 활용합니다.
    
- **금융 모델링**: 투자 전략과 포트폴리오 최적화에 동적 프로그래밍 기법이 자주 쓰입니다.
    
- **자원 할당**: 배낭 문제와 변형들은 일정 관리, 예산 편성, 자원 관리에 나타납니다.
    

## 실전 예제: 동전 교환 문제

동전 교환 문제는 동적 프로그래밍의 두 핵심 원칙인 최적 부분 구조와 중복 하위 문제를 보여주는 고전 코딩 도전 과제입니다.

동전 교환 문제는 "목표 금액을 만들기 위해 필요한 최소 동전 개수는 얼마인가?"를 묻습니다.

동적 프로그래밍을 사용한 한 가지 해결법은 다음과 같습니다:

```js
function minCoins(amount, coins) {
  // Create dp array from 0 to amount, filled with Infinity
  const dp = new Array(amount + 1).fill(Infinity);

  // Base case: 0 coins needed to make amount 0
  dp[0] = 0;

  // Build up from 1 to target amount
  for (let i = 1; i <= amount; i++) {
    // Try each coin denomination
    for (let coin of coins) {
      // Only use coin if it is not larger than current amount
      if (coin <= i) {
        // Choose the minimum: current value OR solution for remaining amount + 1 coin
        dp[i] = Math.min(dp[i], dp[i - coin] + 1);
      }
    }
  }

  // Return result if possible, -1 if impossible (infinity)
  return dp[amount] === Infinity ? -1 : dp[amount];
}

/*
 Example usage:
coins = [1, 3, 4], amount = 6

dp[0] = 0
dp[1] = dp[0] + 1 = 1
dp[2] = dp[1] + 1 = 2
dp[3] = min(dp[2]+1, dp[0]+1) = min(2+1, 0+1) = 1
dp[4] = min(dp[3]+1, dp[1]+1, dp[0]+1) = min(1+1, 1+1, 0+1) = 1
dp[5] = min(dp[4]+1, dp[2]+1, dp[1]+1) = min(1+1, 2+1, 1+1) = 2
dp[6] = min(dp[5]+1, dp[3]+1, dp[2]+1) = min(2+1, 1+1, 2+1) = 2

Result: 2 coins (3 + 3)
*/
```

그리고 `coins = [1, 3, 4]`, `amount = 6`에 대해 동전 교환 알고리즘이 단계별로 어떻게 작동하는지 봅시다:

```md
Initial state:
dp = [0, ∞, ∞, ∞, ∞, ∞, ∞]
     [0, 1, 2, 3, 4, 5, 6] ← amounts

Building up the solution:

For amount = 1:
  Try coin 1: dp[1] = min(∞, dp[0] + 1) = min(∞, 0 + 1) = 1
  dp = [0, 1, ∞, ∞, ∞, ∞, ∞]

For amount = 2:
  Try coin 1: dp[2] = min(∞, dp[1] + 1) = min(∞, 1 + 1) = 2
  dp = [0, 1, 2, ∞, ∞, ∞, ∞]

For amount = 3:
  Try coin 1: dp[3] = min(∞, dp[2] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[3] = min(3, dp[0] + 1) = min(3, 0 + 1) = 1
  dp = [0, 1, 2, 1, ∞, ∞, ∞]

For amount = 4:
  Try coin 1: dp[4] = min(∞, dp[3] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[4] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  Try coin 4: dp[4] = min(2, dp[0] + 1) = min(2, 0 + 1) = 1
  dp = [0, 1, 2, 1, 1, ∞, ∞]

For amount = 5:
  Try coin 1: dp[5] = min(∞, dp[4] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[5] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  Try coin 4: dp[5] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, ∞]

For amount = 6:
  Try coin 1: dp[6] = min(∞, dp[5] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[6] = min(3, dp[3] + 1) = min(3, 1 + 1) = 2
  Try coin 4: dp[6] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, 2]

Final result: dp[6] = 2 (achieved with coins 3 + 3)
```

이 해결법은 동적 프로그래밍의 두 핵심 원칙을 보여줍니다. 중복 하위 문제가 있는데, 예를 들어 금액 6의 최소 동전을 찾으려면 5, 3, 2의 해답을 알아야 합니다. 이 하위 문제들은 다른 금액을 계산할 때도 반복됩니다. 최적 부분 구조도 있는데, 어떤 금액의 최적 해답은 더 작은 금액들의 최적 해답을 포함합니다. 예를 들어 금액 3의 최소 동전이 1개라면, 금액 6을 만드는 한 방법은 그 해답에 3원짜리 동전 하나를 더하는 것입니다.

DP가 없으면 모든 동전 조합을 시도해야 하므로 경우의 수가 지수적으로 늘어납니다. DP를 사용하면 체계적으로 해결책을 만듭니다:

- **시간 복잡도**: 지수적 대신 `O(amount × number of coins)`
    
- **공간 복잡도**: `O(amount)` 크기의 `dp` 배열
    
- **중복 작업 없음**: 각 하위 문제(각 금액의 최소 동전 수)는 한 번만 해결
    
- **결과 재사용**: 금액 3의 최소 동전을 알면, 이 정보를 활용해 더 큰 금액 계산에 사용
    

## 동적 프로그래밍을 언제 사용하나요?

동적 프로그래밍은 다음과 같은 경우에 효과적입니다:

- 문제를 중복되는 하위 문제로 나눌 수 있을 때
    
- 문제에 최적 부분 구조가 있을 때
    
- 단순 재귀 해결법이 반복 계산을 포함할 때
    
- 시간 복잡도를 줄이기 위해 공간 복잡도를 희생할 수 있을 때
    

일반적인 동적 프로그래밍 패턴은 최솟값/최댓값 찾기 같은 최적화 문제, 경우의 수 세기 문제, 더 작은 결정으로 나눌 수 있는 결정 문제 등이 있습니다.

동적 프로그래밍은 복잡한 문제를 하위 문제의 해답을 체계적으로 저장하고 재사용해 관리 가능한 문제로 바꿉니다. 이 기법을 이해하면 다양한 계산 문제를 효율적으로 해결할 수 있습니다.
