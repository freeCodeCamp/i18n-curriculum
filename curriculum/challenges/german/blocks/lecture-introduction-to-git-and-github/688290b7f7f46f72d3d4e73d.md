# --description--

Sie haben gelernt, wie man Commits erstellt und diese in Ihr eigenes Repository pusht, das Sie kontrollieren. Dasselbe können Sie auch für andere Projekte tun, die Sie nicht verwalten, aber es gibt einige zusätzliche Schritte.

Zunächst einmal haben Sie in der Regel keinen „Schreibzugriff“ auf das Projekt einer anderen Person. Das bedeutet, dass Sie keine Commits direkt dorthin pushen können. Stattdessen müssen Sie eine Kopie des Repositories erstellen, von der aus Sie arbeiten. Dieser Vorgang wird „Forken“ genannt.

Um ein Repository zu forken, besuchen Sie das Repository, zu dem Sie beitragen möchten, auf GitHub. Oben sehen Sie eine Schaltfläche „Fork“. Klicken Sie darauf, um zur Benutzeroberfläche für das Erstellen Ihres Forks weitergeleitet zu werden.

Sie sehen einen ähnlichen Bildschirm wie bei der Erstellung Ihres eigenen Repositories, aber mit weniger Optionen. Sie können einen Besitzer auswählen, entweder Ihr persönliches Konto oder eine Organisation, auf die Sie Zugriff haben, einen neuen Namen für den Fork und eine neue Beschreibung.

Sie haben auch die Möglichkeit, nur den `main` Branch (oder den für das Repository standardmäßig konfigurierten) zu kopieren. Das ist in der Regel eine gute Idee, es sei denn, Sie müssen zu einem bestimmten Feature-Branch beitragen. Mehr über Branches lernen Sie in einer zukünftigen Lektion. Lassen Sie dieses Kästchen vorerst angekreuzt.

Klicken Sie auf „Create fork“ und Ihr neues Repository wird erstellt, und Sie werden automatisch dorthin weitergeleitet.

Klonen Sie es anschließend auf Ihren lokalen Computer, genau wie bei Ihrem eigenen Repository, entweder per SSH oder mit der GitHub CLI.

Schauen wir uns nun die Remotes an, die Ihr lokaler Klon konfiguriert hat. Öffnen Sie ein Terminal im Verzeichnis des lokalen Repositories und führen Sie `git remote -v` aus, um eine vollständige Liste zu erhalten. Es sollte ungefähr so aussehen:

```sh
origin git@github.com:your-username/your-repo-name.git (fetch)
origin git@github.com:your-username/your-repo-name.git (pull)
```

Durch das Klonen eines Repositories von GitHub hat Git automatisch Ihren Fork als `origin` Remote konfiguriert. Wenn Sie jedoch von einem Fork aus arbeiten, müssen Sie Ihre Kopie häufig aktualisieren, um die Änderungen aus dem Original-Repository zu übernehmen.

Dazu fügen wir das Original-Repository als weiteren Remote hinzu. Gehen Sie zurück zur Seite, von der Sie das Repository geforkt haben, klicken Sie auf die Schaltfläche „Code“ und kopieren Sie die SSH-URL. Führen Sie dann den Befehl aus, um einen neuen upstream Remote zu erstellen:

```sh
git remote add upstream git@github.com:original-repo-username/repo-name.git
```

`upstream` ist der Name des Remotes – Sie können ihn anders benennen, aber `upstream` ist die übliche Konvention. Wenn Sie Ihre Remotes erneut ansehen, sehen Sie Ihren neuen upstream Remote:

```sh
origin git@github.com:your-username/your-repo-name.git (fetch)
origin git@github.com:your-username/your-repo-name.git (pull)
upstream git@github.com:original-repo-username/your-repo-name.git (fetch)
upstream git@github.com:original-repo-username/your-repo-name.git (pull)
```

Mit dem `upstream` Remote können Sie Dinge wie `git fetch upstream` und `git merge upstream/main` ausführen, um die neuesten Änderungen aus dem Originalprojekt in Ihren lokalen Fork zu holen.

Ein Fork ist immer noch ein Repository, und Sie erstellen Commits und pushen Änderungen auf die gleiche Weise. Bevor Sie jedoch damit beginnen, sollten Sie verstehen, dass es bestimmte Strategien gibt, die Sie befolgen sollten, wenn Sie Änderungen an einem fremden Projekt vornehmen. Mehr über diese Strategien lernen Sie in kommenden Lektionen.

Projekte, die ihren Quellcode öffentlich zugänglich machen, Ihnen erlauben, den Code zu verändern, und Code-Beiträge akzeptieren, nennt man „Open Source“-Projekte. Open Source bietet viele Vorteile, zum Beispiel, dass die Community Ihnen hilft, Bugs zu finden und zu beheben, Sie von den Beiträgen anderer lernen können und eine Entwickler-Community entsteht, die Sie zu einem Netzwerk ausbauen können.

Wenn Sie Änderungen an einem fremden Projekt vornehmen, gelten Sie als „Open Source Contributor“. Das Beitragen zu Open Source-Projekten kann eine hervorragende Vorschau darauf sein, wie es ist, in einem professionellen Entwicklerteam zu arbeiten. Einige der hier vorgestellten Strategien helfen Ihnen dabei und begleiten Sie in Ihre erste bezahlte Rolle.

Und falls Sie jemals Fragen zum Beitragen haben, haben die meisten Open Source-Projekte Richtlinien zum Beitragen, die einen Überblick über alles enthalten, was Sie wissen müssen. Wenn Sie dennoch unsicher sind, schadet es nie, ein Issue im Repository zu eröffnen, um Ihre Fragen zu stellen – oder, falls es einen Community-Bereich auf einer Plattform wie Discord gibt, treten Sie bei und stellen Sie Ihre Fragen dort.
