# --description--

El modelo de solicitud-respuesta HTTP es el patrón de comunicación fundamental que impulsa la World Wide Web (WWW). Tiene muchos componentes que lo hacen eficiente, para que todos podamos acceder a materiales desde la web.

Veamos qué es el modelo de solicitud-respuesta HTTP, los componentes que lo conforman y cómo esos componentes trabajan juntos para aportar eficiencia a lo que impulsa la web.

El modelo de solicitud-respuesta HTTP es un patrón de comunicación en las interacciones web en el que un cliente, usualmente un navegador web, envía una **HTTP request** a un servidor, y el servidor procesa la solicitud y envía de vuelta una **HTTP response**.

Aquí están los componentes que conforman el modelo de solicitud-respuesta HTTP:

* **Cliente**: Un navegador, aplicación móvil o cualquier otro software que envía solicitudes HTTP al servidor e interpreta las respuestas HTTP que recibe.
    
* **Request**: El mensaje enviado por el cliente al servidor. Incluye el método, URL, encabezados y un cuerpo opcional.
    
* **Método HTTP**: La acción a realizar — `GET` para obtener datos, `POST` para enviar datos, `PATCH` y `PUT` para modificar datos, y `DELETE` para eliminar datos.
    
* **URL (Uniform Resource Locator)**: El recurso que se está solicitando al servidor.
    
* **Headers**: Los metadatos enviados con la solicitud o respuesta. Por ejemplo, `Content-Type` y `Authorization`.
    
* **Cuerpo**: Los datos enviados con las solicitudes `POST`, `PUT`, `PATCH`, `DELETE`, a menudo en formato JSON o datos de formulario.
    
* **Servidor**: El sistema backend que procesa la solicitud y genera una respuesta. Puede estar escrito en cualquier lenguaje de programación que soporte la programación del lado del servidor.
    
* **Respuesta**: El mensaje enviado de vuelta por el servidor. Contiene un código de estado, encabezados y un cuerpo opcional.
    
* **Código de Estado o Código de Respuesta**: Un código numérico que indica éxito, errores o redirecciones.
    
* **Cuerpo de la Respuesta**: Los datos reales enviados de vuelta, típicamente en formato HTML, JSON o XML.

Entonces, ¿cómo funcionan todos estos componentes juntos en el ciclo de solicitud-respuesta HTTP?

1. **The Client Initiates a Request**
    
    * Un navegador, una aplicación móvil o un cliente API envía una solicitud HTTP destinada a llegar al servidor.
        
    * La solicitud incluye un método como `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, URL, encabezados y a veces un cuerpo.
        
2. **The Request is Sent to the Server**
    
    * La solicitud viaja por internet hasta el servidor, un sistema backend escrito con un lenguaje de programación.
        
    * El servidor recibe la solicitud y extrae la información relevante de ella.
        
3. **The Server Processes the Request**
    
    * El servidor verifica el método de la solicitud y la URL.
        
    * Puede que necesite validar la autenticación (a través de encabezados como authorization).
        
    * Si es necesario, interactúa con una base de datos para obtener o modificar los datos.
        
4. **The Server Prepares a Response**
    
    * El servidor genera una respuesta con un código de estado como 200, 404, 500, y así sucesivamente.
        
    * La respuesta incluye encabezados, por ejemplo, `Content-Type: application/json`.
        
    * Si aplica, se añade un cuerpo de respuesta en JSON, HTML o XML.
        
5. **The Response is Sent to the Client**
    
    * El servidor envía la respuesta de vuelta a través de internet.
        
    * El cliente recibe la respuesta y procesa los datos.
        
6. **The Client Handles the Response**
    
    * Si la respuesta es exitosa (código de estado 200), el cliente muestra los datos, por ejemplo, renderiza los datos en una página web.
        
    * Si ocurre un error (`404 Not Found`, `500 Internal Server Error`, y así sucesivamente), el cliente puede mostrar un mensaje de error.

Este ciclo se repite por cada nueva solicitud que haga el cliente.
