## WebSocket vs HTTP

- **HTTP** segue un modello di richiesta-risposta: il client invia una richiesta, il server invia una risposta, poi la connessione si chiude.
- **WebSocket** creano una connessione permanente e bidirezionale tra un client e un server che resta aperta finché una delle due parti non invia un segnale di chiusura.
- HTTP è adatto per caricare pagine o recuperare dati, ma diventa lento e inefficiente per applicazioni in tempo reale.
- Ogni connessione WebSocket inizia con una **stretta di mano HTTP**: il client richiede un upgrade, e il server risponde con `101 Switching Protocols`.

### Flusso di connessione WebSocket

1. Il client si connette al server tramite HTTP e richiede un upgrade.
2. Il server accetta e risponde con `101 Switching Protocols`.
3. La connessione resta aperta.
4. Entrambe le parti scambiano dati in qualsiasi momento finché la connessione si chiude.

### Comunicazione HTTP vs WebSocket

- **HTTP**: `Client → Request → Server → Response` (un ciclo, poi si chiude)
- **WebSocket**: `Client ↔ Server` (persistente, bidirezionale)

### Un'analogia utile

- **HTTP** è come scambiarsi lettere: ne scrivi una, aspetti una risposta, e la conversazione finisce.
- **WebSocket** è come una telefonata: entrambe le parti parlano in tempo reale per tutto il tempo necessario.

### Usi reali dei WebSocket

- Applicazioni di chat
- Giochi multiplayer online
- Aggiornamenti in tempo reale dei punteggi sportivi
- Cruscotti per il mercato azionario
- Strumenti collaborativi (es. Google Docs)

## Architettura Pub/Sub

- **Pub/Sub** (Publish/Subscribe) è un’architettura di messaggistica che permette a parti di un sistema di comunicare senza conoscersi direttamente.
- Ci sono tre parti principali: **publisher**, **topic** e **subscriber**.
  - Un **publisher** invia messaggi a un topic.
  - Un **topic** (detto anche canale o gruppo) funge da intermediario.
  - Un **subscriber** riceve messaggi dai topic a cui si è iscritto.

### Flusso dei messaggi Pub/Sub

1. Un subscriber si iscrive a un topic.
2. Un publisher invia un messaggio a quel topic.
3. Il sistema di messaggistica consegna il messaggio a ogni subscriber di quel topic.

### Comunicazione diretta vs Pub/Sub

- **Senza Pub/Sub**: `Application A → Application B` — fortemente accoppiata; una modifica in B può rompere A.
- **Con Pub/Sub**: `Application A → Topic → Subscribers` — A pubblica solo; non deve sapere chi riceve.

### Usi reali di Pub/Sub

- Applicazioni di chat con più stanze
- Giochi multiplayer online
- Sistemi di notifiche
- Cruscotti in tempo reale
- Applicazioni basate su eventi

### WebSocket vs Pub/Sub

- **WebSocket** fornisce il canale di comunicazione (la strada).
- **Pub/Sub** fornisce il pattern di messaggistica che determina chi riceve quali messaggi (il sistema del traffico).
- Molte applicazioni in tempo reale usano entrambi insieme: WebSocket mantiene la connessione aperta, mentre Pub/Sub instrada i messaggi ai subscriber giusti.

## Lavorare con WebSocket in Node.js

- Node.js non ha un server WebSocket integrato. La libreria `ws` è il pacchetto esterno standard usato per questo.
- Installa con: `npm install ws`

### Creare un server WebSocket

```js
const WebSocket = require("ws");

const server = new WebSocket.Server({ port: 3000 });
```

### Eventi WebSocket

| Evento | Attivazione |
|---|---|
| `connection` | Un nuovo client si connette al server |
| `message` | Un client connesso invia dati |
| `close` | Una connessione si chiude |
| `error` | Qualcosa va storto sulla connessione |

### Inviare un messaggio a un client

```js
socket.send("Hello from the server");
```

### Trasmettere a tutti i client connessi

- `server.clients` contiene tutti i client connessi.
- Controlla `readyState === WebSocket.OPEN` prima di inviare per evitare errori con client disconnessi.

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **Broadcasting** è il pattern di inviare un messaggio a più client connessi contemporaneamente — usato in app di chat, notifiche live, aggiornamenti sportivi e eventi di giochi multiplayer.
