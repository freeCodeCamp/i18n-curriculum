# --description--
  
La programmazione dinamica è una tecnica algoritmica che risolve problemi complessi suddividendoli in sottoproblemi più semplici e memorizzando i risultati per evitare calcoli ridondanti. Questo approccio trasforma problemi che normalmente richiederebbero un tempo esponenziale in problemi risolvibili in tempo polinomiale.

## Principi fondamentali della programmazione dinamica

La programmazione dinamica funziona quando in un problema sono presenti due condizioni chiave.

- **Sottoproblemi sovrapposti**: gli stessi sottoproblemi più piccoli compaiono più volte durante la risoluzione del problema più grande. Invece di ricalcolare questi sottoproblemi ripetutamente, memorizziamo le loro soluzioni.
    
- **Sottostruttura ottimale**: la soluzione ottimale del problema contiene soluzioni ottimali ai suoi sottoproblemi. Questo significa che possiamo costruire la soluzione migliore combinando le soluzioni migliori delle parti più piccole.
    

Esaminiamo questi concetti usando il classico problema del "salire le scale".

## Il problema della ricorsione ingenua

Considera il problema di salire le scale: stai salendo una scala con `n` gradini e puoi salire 1 o 2 gradini alla volta. Quanti modi distinti ci sono per arrivare in cima?

```js
function climbStairsRecursive(n) {
  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Recursive rule: to reach step n, you can come from step (n-1) or step (n-2)
  return climbStairsRecursive(n - 1) + climbStairsRecursive(n - 2);
}
```

Questa implementazione ha una complessità temporale esponenziale a causa di massicci calcoli ridondanti. Quando calcoliamo `climbStairsRecursive(5)`, succede questo:

- `climbStairsRecursive(5)` chiama `climbStairsRecursive(4)` e `climbStairsRecursive(3)`
    
- `climbStairsRecursive(4)` chiama `climbStairsRecursive(3)` e `climbStairsRecursive(2)`
    
- Ora `climbStairsRecursive(3)` viene calcolato **due volte**
    
- `climbStairsRecursive(3)` chiama `climbStairsRecursive(2)` e `climbStairsRecursive(1)`
    
- `climbStairsRecursive(2)` viene calcolato **3 volte in totale**
    

Solo per `n=5` facciamo 9 chiamate di funzione quando servirebbero solo 5 calcoli unici. Man mano che `n` cresce, questa ridondanza esplode esponenzialmente: `climbStairsRecursive(30)` richiederebbe oltre qualche milione di chiamate di funzione! La complessità temporale diventa `O(2^n)`, rendendo l’approccio inefficiente e poco pratico per valori più grandi di `n`.

## Soluzioni con la programmazione dinamica

La programmazione dinamica elimina questo calcolo ridondante attraverso due approcci principali:

### Memoizzazione (approccio top-down)

La memoizzazione memorizza i risultati di chiamate di funzione costose e restituisce il risultato memorizzato quando gli stessi input si ripresentano:

```js
function climbStairsMemo(n, memo = {}) {
  // If already calculated, return cached value
  if (memo[n] !== undefined) {
    return memo[n];
  }

  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Calculate once and store in memo for future use
  memo[n] =
    climbStairsMemo(n - 1, memo) +
    climbStairsMemo(n - 2, memo);

  return memo[n];
}
```

La memoizzazione è molto più efficiente perché ogni valore unico da `1` a `n` viene calcolato esattamente una volta. Quando serve di nuovo `climbStairsMemo(3)`, invece di ricalcolarlo (che causerebbe altre chiamate ricorsive), lo cerchiamo semplicemente nel nostro dizionario memo in tempo `O(1)`.

Seguiamo l’esecuzione di `climbStairsMemo(5)` con l’approccio top-down per vedere come la memoizzazione elimina il lavoro ridondante:

```md
Call: climbStairsMemo(5)
  memo = {} (empty)
  
  Call: climbStairsMemo(4) 
    memo = {} (empty)
    
    Call: climbStairsMemo(3)
      memo = {} (empty)
      
      Call: climbStairsMemo(2) → returns 2 (base case)
      Call: climbStairsMemo(1) → returns 1 (base case)
      
      Result: 2 + 1 = 3
      memo = {3: 3} (stored!)
    
    Call: climbStairsMemo(2) → returns 2 (base case)
    
    Result: 3 + 2 = 5
    memo = {3: 3, 4: 5} (stored!)
  
  Call: climbStairsMemo(3) → returns 3 (FROM MEMO - no recursion!)
  
  Result: 5 + 3 = 8
  memo = {3: 3, 4: 5, 5: 8}
```

**Confronto di efficienza**

- **Ricorsione ingenua**: fa 9 chiamate di funzione con calcoli ripetuti
    
- **Memoizzazione**: fa solo 5 calcoli unici, poi riusa i risultati memorizzati
    
- **Complessità temporale**: ridotta da `O(2^n)` a `O(n)` perché facciamo solo `n` calcoli unici
    
- **Complessità spaziale**: `O(n)` per la memorizzazione memo e lo stack delle chiamate
    
- **Impatto reale**: `climbStairsMemo(30)` scende da milioni di chiamate di funzione a circa 30 calcoli unici!
    

### Tabulazione (approccio bottom-up)

La tabulazione costruisce la soluzione dal basso verso l’alto, riempiendo una tabella con le soluzioni ai sottoproblemi:

```js
function climbStairsTabulation(n) {
  if (n <= 2) {
    return n;
  }

  // Create an array to store solutions from 0 to n
  const dp = new Array(n + 1).fill(0);

  // Base cases
  dp[1] = 1; // 1 way to reach step 1
  dp[2] = 2; // 2 ways to reach step 2

  // Build up the solution iteratively
  for (let i = 3; i <= n; i++) {
    // Ways to reach step i = ways to reach (i-1) + ways to reach (i-2)
    dp[i] = dp[i - 1] + dp[i - 2];
  }

  return dp[n];
}
```

La tabulazione elimina completamente la ricorsione costruendo la soluzione in modo iterativo dai sottoproblemi più piccoli fino all’obiettivo.

Vediamo l’approccio bottom-up in azione per costruire sistematicamente la soluzione. Ecco la costruzione iterativa per `climbStairsTabulation(5)`:

```md
Initial state:
dp = [0, 1, 2, 0, 0, 0]
     [0, 1, 2, 3, 4, 5] ← indices (step numbers)

Step by step construction:

i = 3:
  dp[3] = dp[2] + dp[1] = 2 + 1 = 3
  dp = [0, 1, 2, 3, 0, 0]
  
i = 4:
  dp[4] = dp[3] + dp[2] = 3 + 2 = 5
  dp = [0, 1, 2, 3, 5, 0]
  
i = 5:
  dp[5] = dp[4] + dp[3] = 5 + 3 = 8
  dp = [0, 1, 2, 3, 5, 8]

Final result: dp[5] = 8
```

**Vantaggi chiave della tabulazione**

- **Nessun overhead di ricorsione**: a differenza della memoizzazione, non c’è uno stack di chiamate ricorsive.
    
- **Esecuzione prevedibile**: calcoliamo i valori in un ordine prestabilito (1, 2, 3, 4, 5...).
    
- **Ottimizzazione della cache**: l’accesso sequenziale all’array ottimizza l’uso della memoria.
    
- **Facile da ottimizzare**: si può ridurre la complessità spaziale a `O(1)` perché servono solo gli ultimi due valori.
    

```js
function climbStairsOptimized(n) {
  if (n <= 2) {
    return n;
  }

  // Store only the last two results to save space
  let prev2 = 1; // ways to reach step 1
  let prev1 = 2; // ways to reach step 2

  for (let i = 3; i <= n; i++) {
    const current = prev1 + prev2;

    // Shift values forward
    prev2 = prev1;
    prev1 = current;
  }

  return prev1;
}
```

**Confronto di efficienza**

- **Ricorsione ingenua**: 9 chiamate di funzione per `n=5`, crescita esponenziale.
    
- **Tabulazione**: 3 semplici addizioni per `n=5`, crescita lineare.
    
- **Complessità temporale**: `O(n)` invece di `O(2^n)`.
    
- **Complessità spaziale**: `O(n)` per l’array, o `O(1)` con ottimizzazione.
    
- **Prestazioni prevedibili**: nessun rischio di overflow dello stack per input grandi.
    

Entrambi gli approcci riducono la complessità temporale da `O(2^n)` esponenziale a `O(n)` lineare, un miglioramento drastico che fa la differenza tra risolvere il problema in millisecondi o aspettare anni per input più grandi.

## Applicazioni nel mondo reale

La programmazione dinamica ha applicazioni diffuse nell’informatica e oltre:

- **Ottimizzazione delle rotte**: i sistemi GPS usano algoritmi di programmazione dinamica per trovare i percorsi più brevi tra località.
    
- **Elaborazione del testo**: correttori ortografici e funzionalità di completamento automatico spesso si basano sulla programmazione dinamica per calcolare le distanze di modifica tra parole.
    
- **Modellazione finanziaria**: strategie di investimento e ottimizzazione del portfolio impiegano frequentemente tecniche di programmazione dinamica.
    
- **Allocazione delle risorse**: il problema dello zaino e le sue varianti compaiono in pianificazione, budgeting e gestione delle risorse.
    

## Esempio pratico: problema del cambio monete

Il problema del cambio monete è una classica sfida di programmazione che, risolta con la programmazione dinamica, dimostra entrambi i principi chiave della PD: sottoproblemi sovrapposti e sottostruttura ottimale.

Il problema del cambio monete chiede: "Qual è il numero minimo di monete necessario per raggiungere un importo obiettivo?"

Ecco una soluzione usando la programmazione dinamica:

```js
function minCoins(amount, coins) {
  // Create dp array from 0 to amount, filled with Infinity
  const dp = new Array(amount + 1).fill(Infinity);

  // Base case: 0 coins needed to make amount 0
  dp[0] = 0;

  // Build up from 1 to target amount
  for (let i = 1; i <= amount; i++) {
    // Try each coin denomination
    for (let coin of coins) {
      // Only use coin if it is not larger than current amount
      if (coin <= i) {
        // Choose the minimum: current value OR solution for remaining amount + 1 coin
        dp[i] = Math.min(dp[i], dp[i - coin] + 1);
      }
    }
  }

  // Return result if possible, -1 if impossible (infinity)
  return dp[amount] === Infinity ? -1 : dp[amount];
}

/*
 Example usage:
coins = [1, 3, 4], amount = 6

dp[0] = 0
dp[1] = dp[0] + 1 = 1
dp[2] = dp[1] + 1 = 2
dp[3] = min(dp[2]+1, dp[0]+1) = min(2+1, 0+1) = 1
dp[4] = min(dp[3]+1, dp[1]+1, dp[0]+1) = min(1+1, 1+1, 0+1) = 1
dp[5] = min(dp[4]+1, dp[2]+1, dp[1]+1) = min(1+1, 2+1, 1+1) = 2
dp[6] = min(dp[5]+1, dp[3]+1, dp[2]+1) = min(2+1, 1+1, 2+1) = 2

Result: 2 coins (3 + 3)
*/
```

Ecco come funziona passo dopo passo l’algoritmo di programmazione dinamica per il problema del cambio monete con `coins = [1, 3, 4]`, `amount = 6`:

```md
Initial state:
dp = [0, ∞, ∞, ∞, ∞, ∞, ∞]
     [0, 1, 2, 3, 4, 5, 6] ← amounts

Building up the solution:

For amount = 1:
  Try coin 1: dp[1] = min(∞, dp[0] + 1) = min(∞, 0 + 1) = 1
  dp = [0, 1, ∞, ∞, ∞, ∞, ∞]

For amount = 2:
  Try coin 1: dp[2] = min(∞, dp[1] + 1) = min(∞, 1 + 1) = 2
  dp = [0, 1, 2, ∞, ∞, ∞, ∞]

For amount = 3:
  Try coin 1: dp[3] = min(∞, dp[2] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[3] = min(3, dp[0] + 1) = min(3, 0 + 1) = 1
  dp = [0, 1, 2, 1, ∞, ∞, ∞]

For amount = 4:
  Try coin 1: dp[4] = min(∞, dp[3] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[4] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  Try coin 4: dp[4] = min(2, dp[0] + 1) = min(2, 0 + 1) = 1
  dp = [0, 1, 2, 1, 1, ∞, ∞]

For amount = 5:
  Try coin 1: dp[5] = min(∞, dp[4] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[5] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  Try coin 4: dp[5] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, ∞]

For amount = 6:
  Try coin 1: dp[6] = min(∞, dp[5] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[6] = min(3, dp[3] + 1) = min(3, 1 + 1) = 2
  Try coin 4: dp[6] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, 2]

Final result: dp[6] = 2 (achieved with coins 3 + 3)
```

Questa soluzione dimostra entrambi i principi chiave della programmazione dinamica. Ha sottoproblemi sovrapposti perché trovare il numero minimo di monete per l’importo 6 richiede di conoscere le soluzioni per gli importi 5, 3 e 2. Questi stessi sottoproblemi compaiono anche nel calcolo di altri importi. Ha sottostruttura ottimale perché la soluzione ottimale per qualsiasi importo incorpora soluzioni ottimali per importi più piccoli. Se sappiamo che il numero minimo di monete per l’importo 3 è 1, allora un modo per fare 6 è usare quella soluzione più una moneta da 3.

Senza la programmazione dinamica, dovremmo provare ogni possibile combinazione di monete — un numero esponenziale di possibilità. Con la PD, costruiamo la soluzione sistematicamente:

- **Complessità temporale**: `O(amount × number of coins)` invece di esponenziale.
    
- **Complessità spaziale**: `O(amount)` per l’array `dp`.
    
- **Nessun lavoro ridondante**: ogni sottoproblema (trovare il numero minimo di monete per ogni importo) viene risolto esattamente una volta.
    
- **Risultati riutilizzabili**: una volta che conosciamo il numero minimo di monete per l’importo 3, usiamo questa conoscenza per tutti gli importi maggiori che ne possono beneficiare.
    

## Quando usare la programmazione dinamica

La programmazione dinamica è efficace quando:

- Il problema può essere suddiviso in sottoproblemi sovrapposti.
    
- Il problema presenta sottostruttura ottimale.
    
- Una soluzione ricorsiva ingenua comporterebbe calcoli ripetuti.
    
- Serve ottimizzare la complessità temporale a costo di una maggiore complessità spaziale.
    

I pattern comuni della programmazione dinamica includono problemi di ottimizzazione (trovare valori minimi/massimi), problemi di conteggio (numero di modi per ottenere qualcosa) e problemi decisionali che possono essere suddivisi in decisioni più piccole.

La programmazione dinamica trasforma problemi complessi in problemi gestibili memorizzando e riutilizzando sistematicamente le soluzioni ai sottoproblemi. Comprendere questa tecnica apre la porta a risolvere una vasta gamma di sfide computazionali in modo efficiente.
