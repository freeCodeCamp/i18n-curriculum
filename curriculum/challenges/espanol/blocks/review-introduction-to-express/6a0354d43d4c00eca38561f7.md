## Servicios web y APIs

- **API (interfaz de programación de aplicaciones)**: Una forma general para que el software se comunique con otro software.
- **Servicio web**: Un tipo específico de API que usa tecnologías web como HTTP para intercambiar datos a través de una red. Todo servicio web es una API, pero no toda API es un servicio web.
- **Características clave de un servicio web**: se ejecuta en un servidor, habilita la comunicación máquina a máquina, es independiente del lenguaje, es independiente de la plataforma y usa protocolos web estándar (HTTP/HTTPS, FTP, SMTP).
- **REST (transferencia de estado representacional)**: El tipo de servicio web más usado hoy en día. Expone recursos identificados por rutas URL como `/users` o `/users/20`. Las respuestas suelen estar en JSON, pero también pueden ser XML o texto plano.
- **SOAP (protocolo simple de acceso a objetos)**: Usa solo XML. Común en sistemas empresariales que necesitan estándares y seguridad más estrictos.
- **XML-RPC**: Usa XML para codificar solicitudes y respuestas. Ligero pero menos flexible que REST.
- **Encabezado `Accept`**: Usado por un cliente para indicar al servidor qué formato de respuesta desea, por ejemplo `Accept: application/json`.

## La arquitectura REST

- **Sin estado**: Cada solicitud debe incluir toda la información que el servidor necesita para procesarla. El servidor no recuerda solicitudes anteriores.
- **Recurso**: Un dato que puede ser accedido o modificado, identificado por una URL (por ejemplo, `/products/1`).
- **Métodos HTTP en REST**:
  - `GET` — recuperar un recurso
  - `POST` — crear un recurso nuevo
  - `PUT` / `PATCH` — actualizar un recurso existente
  - `DELETE` — eliminar un recurso

## Códigos de estado HTTP

Después de procesar una solicitud, el servidor responde con un **código de estado** que indica el resultado:

- `200` **(OK)**: La solicitud fue exitosa
- `201` **(Created)**: Se creó un recurso nuevo
- `400` **(Bad Request)**: El cliente envió datos inválidos
- `404` **(Not Found)**: El recurso solicitado no existe
- `500` **(Internal Server Error)**: Ocurrió un error en el servidor

### Cómo funciona una solicitud REST

1. El cliente envía una solicitud con un método, URL, encabezados opcionales y cuerpo opcional.
2. El servidor valida la solicitud e identifica el recurso.
3. El servidor realiza la acción (obtener, crear, actualizar o eliminar).
4. El servidor construye una respuesta con un código de estado y cuerpo opcional (usualmente JSON).
5. El servidor envía la respuesta — no se almacena estado del cliente.
6. El cliente verifica el código de estado y procesa los datos recibidos.

## Monolitos vs. microservicios

- **Monolito**: Una base de código grande y única donde la interfaz, base de datos y API conviven juntas. Funciona bien para apps pequeñas pero se vuelve difícil de mantener y escalar al crecer.
- **Microservicios**: Una arquitectura donde una app grande se divide en servicios más pequeños e independientes. Cada servicio maneja una capacidad de negocio (por ejemplo, productos, pedidos, pagos) y puede tener su propia base de código, base de datos y equipo.
- **Cómo se comunican los microservicios**: A través de APIs o sistemas basados en mensajes.
- **Ventajas de los microservicios**: desarrollo más rápido, mejor aislamiento de fallos y libertad para usar distintos lenguajes o frameworks por servicio.
- **Desafíos de los microservicios**: depuración más compleja, gestión de infraestructura más difícil y mayor sobrecarga de red entre servicios.

## ¿Qué es Express.js?

- **Express**: Un framework web minimalista y flexible construido sobre Node.js que facilita crear servidores web y APIs.
- **Instalar Express**: `npm install express`
- **Qué puedes construir con Express**: apps web, APIs RESTful, backends para SPA, servicios con mucho middleware y cualquier cosa del lado servidor con HTTP.
- **Middleware**: Código que se ejecuta entre recibir una solicitud y enviar una respuesta. Se usa para manejo de errores, validación de entrada, modificar datos de solicitud o respuesta y más. Se pueden encadenar múltiples funciones middleware.
- **Plugins**: Módulos externos de npm que extienden Express, usualmente implementados como middleware.

## Crear una app básica con Express

```javascript
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- `require("express")` — carga la librería Express.
- `express()` — función fábrica que crea y devuelve una nueva instancia de aplicación Express.
- `app` — el objeto principal del servidor con acceso a métodos como `.get()`, `.post()` y `.listen()`.
- `port` — el endpoint de red donde la app escucha. El puerto `3000` es común para desarrollo local.
- `app.listen(port, callback)` — inicia el servidor y comienza a escuchar solicitudes HTTP entrantes.

## Enrutamiento en Express

- **Ruta**: Una regla que indica a la app cómo responder cuando un cliente solicita una URL específica con un método HTTP específico.
- **Métodos de ruta**: `app.get()`, `app.post()`, `app.put()`, `app.delete()` — cada uno maneja solicitudes con el método HTTP correspondiente.
- **Ruta estática**: Una URL fija como `/home` o `/menu/drinks`.
- **Ruta dinámica (parámetro de ruta)**: Un marcador de posición en la URL que captura un valor, con prefijo `:`, por ejemplo `/post/:postId`.
- **`req.params`**: Un objeto que contiene los valores dinámicos capturados de la URL. Por ejemplo, una solicitud a `/post/42` da `req.params.postId === "42"`.

### Manejadores de ruta

- Un **manejador de ruta** es la función callback `(req, res) => { ... }` que se ejecuta cuando una solicitud coincide con una ruta.
- `req` — contiene información sobre la solicitud entrante (params, cadenas de consulta, encabezados, cuerpo).
- `res` — se usa para enviar una respuesta al cliente.

## Métodos de respuesta

| Método | Qué hace |
| --- | --- |
| `res.send()` | Envía una respuesta — texto plano, HTML, JSON o datos binarios. |
| `res.json()` | Envía una respuesta JSON; convierte automáticamente un objeto JavaScript. |
| `res.status(code)` | Establece el código de estado HTTP; usualmente encadenado con `res.send()` o `res.json()`. |
| `res.redirect(url)` | Redirecciona al cliente a una URL diferente. |
| `res.render(view, data)` | Renderiza una vista con motor de plantillas y envía el HTML resultante. |

## Manejadores de ruta encadenables con `app.route()`

- `app.route(path)` agrupa múltiples manejadores de métodos HTTP para la misma ruta para evitar repetirla.

```javascript
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

## Enrutamiento modular con `express.Router()`

- `express.Router()` crea un objeto router — como una mini app Express — donde puedes definir rutas y luego montar en la app principal.
- Monta un router con `app.use(path, router)`. Las rutas dentro del router son relativas a la ruta de montaje.
- **Beneficios**: separa responsabilidades, mantiene limpio el archivo principal y hace las rutas reutilizables y más fáciles de escalar.

```javascript
// userRoutes.js
const express = require("express");
const router = express.Router();
router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => { res.send(`User ${req.params.id}`); });
module.exports = router;

// app.js
app.use("/users", userRoutes); // mounts the router at /users
```

## Servir archivos estáticos

- **Archivos estáticos**: Recursos como imágenes, CSS y JavaScript del lado cliente que se sirven directamente sin procesamiento del servidor.
- `express.static(directory)` — middleware integrado que sirve archivos desde una carpeta especificada.
- Si no se encuentra un archivo solicitado, Express responde automáticamente con un 404.
- No se necesitan definiciones de ruta para archivos estáticos individuales.

```javascript
app.use(express.static(path.join(__dirname, "public")));
```

- Puedes servir desde múltiples directorios llamando `app.use()` varias veces, opcionalmente con una ruta de montaje:

```javascript
app.use(express.static(path.join(__dirname, "public")));
app.use("/images", express.static(path.join(__dirname, "images")));
```
