# --description--

経験豊富な開発者とオンラインコミュニティや対面で交流したことがあれば、「恐ろしいマージコンフリクト」という言葉を聞いたことがあるかもしれません。でも、それは具体的に何なのでしょうか？

前のレッスンで説明したような適切なブランチ戦略を使っている場合、同時に2つの異なる機能を2つの異なるブランチで作業していることがあります。または、自分が作業している間に同僚が別のブランチで作業しているかもしれません。どちらにせよ、2つの独立した変更が進行中です。しかし、時には両方のブランチが同じファイルを編集することがあります。そして、もし他のブランチが先に`main`にマージされると、自分のブランチの変更は遅れてしまいます。

ほとんどの場合、`main`を自分の機能ブランチにマージし直せば解決します。しかし、他のブランチの変更が自分のブランチの変更と競合すると、Gitはマージ時にどの変更を残すべきか判断できません。これが「マージコンフリクト」と呼ばれるものです。

では、どうやって解決するのでしょうか？小さくて単純なコンフリクトの場合、GitHubはウェブUIを提供してくれます。プルリクエストの下部に「Resolve conflicts」ボタンが表示されます。これをクリックすると、あなたのコードと競合しているコードが含まれたコンフリクトのあるファイルが表示されます。Gitは「マージコンフリクトマーカー」をあなたのコードと競合コードの周りに追加して、問題の箇所を特定しやすくします。以下のような感じです：

```sh
1 <<<<<<< feat/conflict
2 Here's some code from my first PR :)
3 =======
4 Here's some code that was merged to main that conflicts with the code on my PR
5 >>>>>>> main
```

開くマーカーは`<<<<<<< feat/conflict`で、現在のブランチの変更の開始を示しています。次に`Here's some code from my first PR :)`が見えます。その後に`=======`マーカーがあり、これは自分の変更の終わりと`main`ブランチの変更の開始を示しています。`main`で行われた変更は`Here's some code that was merged to main that conflicts with the code on my PR`の行です。そして最後に、コンフリクトの終わりを示す閉じる`>>>>>> main`マーカーがあります。

では、どうやって直すのでしょうか？マージコンフリクトの中でどの変更を残すか決める必要があります。両方の変更を残したい場合は、コンフリクトマーカーだけを削除して変更はそのままにします。片方のブランチの変更だけを残したい場合は、コンフリクトマーカーと残したくない方の変更を削除します。もしどの変更も気に入らなければ、開くマーカーから閉じるマーカーまで（含む）すべて削除できます。

今回は`main`ブランチの変更を削除して、自分の変更だけ残しましょう。こうなります：

```sh
1 Here's some code from my first PR :)
```

「Mark as resolved」ボタンをクリックすると、GitHubに解決済みであることを伝えます。もし他にファイルがあれば次のファイルに切り替わりますが、これが唯一のコンフリクトファイルなので、「Commit merge」ボタンが表示されます。そのボタンをクリックするとGitHubがマージコミットを作成します。プルリクエストに戻り、新しいマージコミットと解決済みのコンフリクトが確認できます。これで完了です！

しかし、もしコンフリクトが複雑すぎてGitHub上で直接編集できない場合はどうでしょう？

もしあなたのブランチにコミットが少なかったり、そのファイルを変更しているコミットが数個だけなら、`main`を自分のブランチにマージできるかもしれません：

```sh
git checkout feat/conflict
git fetch origin
git merge origin/main
```

Gitがこのマージを処理するとき、手動で解決すべきコンフリクトが発生します。Gitはどのファイルを修正すべきか通知してくれます。エディターでコンフリクトのあるファイルを開くと、先ほどと同じコンフリクトマーカーが見えます。VSCodeのおかげで便利なUIもあります。ファイル内のコンフリクトを同じ方法で修正してください。ただし今回はコミットを手動で作成する必要があります。通常のコミット作成と同じです：

```sh
git add .
git commit -m "chore: resolve conflicts"
```

これらのコマンドを実行すると、Gitが解決を受け入れたことが確認できます。

コミットやコンフリクトが多い場合は、リベースが役立つことがあります。マージではなく、リベースはブランチ上のすべてのコミットを取り出して、`main`の最新の変更の後に再適用します。Gitの用語では、これはブランチのベースを`main`の最新コミットに「リセット」することに相当します。

これを試すために新しいブランチを作成しましょう：

```sh
git checkout main
git pull
git checkout -b feat/rebase
```

練習のためにブランチにたくさんのコミットを作成します：

```sh
echo "change one" > README.md
git add .
git commit -m "change one"
echo "change two" > README.md
git add .
git commit -m "change two"
echo "change three" > README.md
git add .
git commit -m "change three"
```

次に、`main`で別の変更を加えてコンフリクトを作りましょう：

```sh
git checkout main
echo "change four" > README.md
git add .
git commit -m "change four"
```

そしてリベース用のブランチにチェックアウトします：

```sh
git checkout feat/rebase
```

次に`git rebase -i main`を実行します。これはブランチを`main`の現在の状態にリベースします。`-i`フラグはGitに対話モードでリベースを行うよう指示します。コミットの横に`pick`という単語が表示されます。`pick`はコミットをそのままリベースに使うことを意味します。ファイルを保存して閉じると、コンフリクトがあるというエラーメッセージが表示されます。

`change one`コミットが`change four`コミットと競合しているので解決が必要です。エディターでファイルを開くと、先ほどと同じUIが表示されます。自分の`change one`の変更を残しましょう。次に`git add .`を実行して変更を追加し、`git rebase --continue`を実行してリベースを続行する準備ができたことをGitに伝えます。最初のコミットのコンフリクトを解決したので、Gitは残りのコミットのリベースを自動的に処理できます。

最後に、履歴をきれいにするために3つの変更コミットを1つにまとめましょう。もう一度`git rebase -i main`を実行してエディターを開きます。最初のコマンドは`pick`のままにして、次の2つを`squash`に変更します。同じコンフリクトが再び起こるかもしれません。その場合は前回と同じ方法で解決してください。解決してリベースを続行すると（またはコンフリクトがなければすぐに）、新しいコミットメッセージエディターが表示されます。このメッセージが3つのコミットをまとめた新しいコミットになります。ここでは`change three complete`と名付けましょう。保存してエディターを閉じると、Gitがリベースを完了します。

コミット履歴を確認すると、1つの`change three complete`コミットがあり、`change four`コミットを正しくベースにしていることがわかります。これでリベースとコミットのまとめに成功しました！これでマージコンフリクトの達人になるためのツールが揃いました！

ブランチのコンフリクトを扱う際、マージとリベースの主な違いは何ですか？
