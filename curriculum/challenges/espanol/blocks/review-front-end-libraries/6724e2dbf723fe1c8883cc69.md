## Bibliotecas y frameworks de JavaScript

- Las bibliotecas y frameworks de JavaScript ofrecen soluciones rápidas a problemas comunes y aceleran el desarrollo al proporcionar código preconstruido.
- Las bibliotecas suelen estar más enfocadas en ofrecer soluciones para tareas específicas, como manipular el DOM, manejar eventos o gestionar solicitudes AJAX.
- Algunos ejemplos de bibliotecas de JavaScript son jQuery y React.
- Los frameworks, por otro lado, proporcionan una estructura más definida para construir aplicaciones. A menudo vienen con un conjunto de reglas y convenciones que los desarrolladores deben seguir.
- Ejemplos de frameworks incluyen Angular y Next.js, un meta framework para React.
- Las **aplicaciones de una sola página** (SPA) son aplicaciones web que cargan una única página HTML y actualizan dinámicamente esa página a medida que el usuario interactúa con la aplicación sin recargar toda la página.
- Las SPA usan JavaScript para gestionar el estado de la aplicación y renderizar contenido. Esto se hace a menudo usando frameworks que proporcionan excelentes herramientas para construir interfaces de usuario complejas.
- Algunos problemas relacionados con las SPA incluyen:

  - Los lectores de pantalla tienen dificultades con contenido actualizado dinámicamente.
  - La URL no cambia cuando el usuario navega dentro de la aplicación, lo que puede dificultar marcar, retroceder o compartir páginas específicas.
  - El tiempo de carga inicial puede ser lento si la aplicación es grande, ya que todos los recursos deben cargarse de antemano.

## React

- React es una biblioteca popular de JavaScript para construir interfaces de usuario y aplicaciones web.
- Un concepto central de React es la creación de componentes de UI reutilizables que pueden actualizarse y renderizarse de forma independiente a medida que cambian los datos.
- React permite a los desarrolladores describir cómo debe verse la UI según el estado de la aplicación. Luego React actualiza y renderiza los componentes correctos cuando los datos o el estado cambian.

## Componentes de React

- Los componentes son los bloques de construcción de las aplicaciones React que permiten a los desarrolladores dividir interfaces de usuario complejas en piezas más pequeñas y manejables.
- La UI se describe usando JSX, una extensión de la sintaxis de JavaScript, que permite escribir código similar a HTML dentro de JavaScript.
- Los componentes son básicamente funciones o clases JS que devuelven una parte de la UI.

Aquí tienes un ejemplo de un componente React simple que renderiza un mensaje de saludo:

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

Para usar el componente, simplemente puedes llamarlo así:

```jsx
  <Greeting />
```

## Importar y exportar componentes React

- Una exportación hace que un componente esté disponible para otros archivos. Una importación permite que un archivo use un componente que fue exportado en otro lugar.
- Hay dos tipos de exportaciones en React: exportaciones por defecto y exportaciones nombradas.

**Exportaciones por defecto**

- Un archivo solo puede tener una exportación por defecto. Es ideal para un archivo que contiene principalmente un solo componente.
- Puedes exportar un componente como exportación por defecto así:

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default City;
```

- También puedes exportar en la misma línea que la definición del componente:

```jsx
export default function City() {
  return <p>New York</p>;
}
```

- Para importar una exportación por defecto, usa la palabra clave `import` sin llaves:

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

**Exportaciones nombradas**

- Las exportaciones nombradas permiten que un archivo comparta múltiples componentes. A diferencia de las exportaciones por defecto, deben importarse usando el nombre exacto con el que fueron exportados.
- Puedes exportar múltiples componentes desde el mismo archivo así:

```jsx
// Animals.jsx
export function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- Para importar exportaciones nombradas, usa llaves con los nombres exactos exportados:

```jsx
import { Cat, Dog } from './Animals';
```

- Puedes renombrar una exportación nombrada durante la importación usando la palabra clave `as`:

```jsx
import { Cat as Kitty } from './Animals';
```

**Exportaciones mixtas por defecto y nombradas**

- Un archivo puede tener una exportación por defecto y múltiples exportaciones nombradas al mismo tiempo:

```jsx
// Animals.jsx
export default function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- Al importar exportaciones mixtas, la exportación por defecto va primero (sin llaves), seguida de las exportaciones nombradas (dentro de llaves):

```jsx
import Cat, { Dog } from './Animals';
```

## Configurar un proyecto React usando Vite

- Las herramientas de configuración de proyectos y CLIs ofrecen una forma rápida y sencilla de iniciar nuevos proyectos, permitiendo a los desarrolladores enfocarse en escribir código en lugar de lidiar con la configuración.
- Vite es una herramienta popular para configurar proyectos y puede usarse con React.
- Para crear un nuevo proyecto con Vite, puedes usar el siguiente comando en tu terminal:

```bash
npm create vite@latest my-react-app -- --template react
```

Este comando crea un nuevo proyecto React llamado `my-react-app` usando la plantilla React de Vite. En el directorio del proyecto, verás un archivo `package.json` con las dependencias y comandos listados.

- Para ejecutar el proyecto, navega al directorio del proyecto y ejecuta los siguientes comandos:

```bash
cd my-react-app # path to the project directory
npm install # installs the dependencies listed in the package.json file
```

- Una vez instaladas las dependencias, deberías notar una nueva carpeta en tu proyecto llamada `node_modules`.

- La carpeta `node_modules` es donde se almacenan todos los paquetes y bibliotecas requeridos por tu proyecto.

- Para ejecutar tu proyecto, usa el siguiente comando:

```bash
npm run dev
```

- Después, abre tu navegador y navega a `http://localhost:5173` para ver tu aplicación React en ejecución.

- Para ver realmente el código de la plantilla inicial, puedes entrar en tu proyecto dentro de la carpeta `src` y deberías ver el archivo `App.jsx`.

## Pasar props en componentes React

- En React, las props (abreviatura de propiedades) son una forma de pasar datos de un componente padre a un componente hijo. Este mecanismo es necesario para crear elementos de UI reutilizables y dinámicos.
- Las props pueden ser cualquier valor de JavaScript. Para pasar props de un padre a un hijo, agregas las props como atributos cuando usas el componente hijo en el JSX del padre. Aquí tienes un ejemplo simple:

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

Puedes pasar múltiples props usando el operador spread `(...)`, después de convertirlas en un objeto. Aquí un ejemplo:

```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

En este código, el operador spread `{...person}` convierte el objeto person en props individuales que se pasan al componente Child.

## Renderizado condicional en React

- El renderizado condicional en React permite crear interfaces de usuario dinámicas. Se usa para mostrar contenido diferente basado en ciertas condiciones o estados dentro de tu aplicación.
- Hay varias formas de renderizar contenido condicionalmente en React. Un enfoque común es usar el operador ternario. Aquí un ejemplo:

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- Otra forma de renderizar contenido condicionalmente es usar el operador lógico AND (`&&`). Esto es útil cuando quieres renderizar contenido solo si se cumple cierta condición. Aquí un ejemplo:

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

En el código anterior, el elemento `h1` solo se renderiza si el objeto user es verdadero.

También puedes usar una sentencia `if` directa así:

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```

## Renderizar listas en React

- Renderizar listas en React es una tarea común al construir interfaces de usuario.
- Las listas pueden renderizarse usando el método `map()` del array JS para iterar sobre un array de elementos y devolver un nuevo array de elementos JSX.
- Por ejemplo, si tienes un array de nombres que quieres renderizar como lista, puedes hacer lo siguiente:

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- Recuerda siempre proporcionar una clave única para cada elemento de la lista para ayudar a React a gestionar la actualización y renderizado. Con estas técnicas, puedes crear listas flexibles, eficientes y dinámicas en tus aplicaciones React.

## Estilos en línea en React

- Los estilos en línea en React permiten aplicar estilos CSS directamente a elementos JSX usando objetos de JavaScript.
- Para aplicar estilos en línea en React, puedes usar el atributo style en elementos JSX. El atributo style toma un objeto donde las claves son propiedades CSS en camelCase y los valores son los valores correspondientes. Aquí un ejemplo:

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

También puedes extraer los estilos en un objeto separado y referenciarlo en el atributo `style` así:

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Los estilos en línea soportan estilos dinámicos permitiendo aplicar estilos condicionalmente basados en props o estado. Aquí un ejemplo de cómo aplicar estilos condicionalmente basado en una prop:

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- En el código anterior, los estilos `color` y `fontSize` se establecen condicionalmente según la prop `isImportant`.

## Trabajar con eventos en React

- **Sistema de eventos sintéticos**: Esta es la forma en que React maneja eventos. Sirve como un envoltorio alrededor de eventos nativos como `click`, `keydown` y `submit`. Los manejadores de eventos en React usan la convención de nombres en camelCase. (Ej. `onClick`, `onSubmit`, etc.)

Aquí un ejemplo de usar el atributo `onClick` para un elemento `button` en React:

```jsx
function handleClick() {
  console.log("Button clicked!");
}

<button onClick={handleClick}>Click Me</button>;
```

En React, las funciones manejadoras de eventos suelen comenzar con el prefijo `handle` para indicar que son responsables de manejar eventos, como `handleClick` o `handleSubmit`.

Cuando una acción del usuario dispara un evento, React pasa un objeto Synthetic Event a tu manejador. Este objeto se comporta muy parecido al objeto Event nativo en JavaScript puro, proporcionando propiedades como `type`, `target` y `currentTarget`.

Para prevenir comportamientos por defecto como la recarga del navegador durante un evento `onSubmit`, por ejemplo, puedes llamar al método `preventDefault()`:

```jsx
function handleSubmit(event) {
  event.preventDefault();
  console.log("Form submitted!");
}

<form onSubmit={handleSubmit}>
  <input type="text" />
  <button>Submit</button>
</form>;
```

También puedes envolver una función manejadora en una función flecha así:

```jsx
function handleDelete(id) {
  console.log("Deleting item:", id);
}

<button onClick={() => handleDelete(1)}>Delete Item</button>;
```

## Trabajar con estado y el hook `useState`

- **Definición de estado**: En React, el estado es un objeto que contiene datos para un componente. Cuando el estado se actualiza, el componente se vuelve a renderizar. React trata el estado como inmutable, lo que significa que no debes modificarlo directamente.
- **Hook `useState()`**: El hook `useState` es una función que te permite declarar variables de estado en componentes funcionales. Aquí la sintaxis básica:

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

En la variable de estado tienes lo siguiente:

- `stateVariable` contiene el valor actual del estado
- `setStateFunction` (la función setter) actualiza la variable de estado
- `initialValue` establece el estado inicial

Aquí un ejemplo completo para un componente `Counter`:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>

      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```

## Renderizado y componentes React

- **Definición**: En React, el renderizado es el proceso por el cual los componentes aparecen en la interfaz de usuario (UI), usualmente el navegador. El proceso de renderizado consta de tres etapas: disparador, renderizado y compromiso.

La etapa de disparador ocurre cuando React detecta que algo ha cambiado y la interfaz de usuario podría necesitar actualizarse. Este cambio suele deberse a una actualización en el estado o las props.

Una vez que ocurre el disparador, React entra en la etapa de renderizado. Aquí, React reevalúa tus componentes y determina qué mostrar. Para hacer esto, React usa una copia ligera del DOM "real" llamada DOM virtual. Con el DOM virtual, React puede verificar rápidamente qué necesita cambiar en el componente.

La etapa de compromiso es donde React toma los cambios preparados del DOM virtual y los aplica al DOM real. En otras palabras, esta es la etapa donde ves el resultado final en pantalla.

## Actualizar objetos y arrays en estado

- **Actualizar objetos en estado**: Si necesitas actualizar un objeto en estado, primero debes crear un nuevo objeto o copiar uno existente, luego establecer el estado con ese nuevo objeto. Cualquier objeto puesto en estado debe considerarse de solo lectura. Aquí un ejemplo de establecer el nombre, edad y ciudad de un usuario. La función `handleChange` se usa para manejar actualizaciones de la información del usuario:

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **Actualizar arrays en estado**: Al actualizar arrays en estado, es importante no modificar directamente el array usando métodos como `push()` o `pop()`. En su lugar, debes crear un nuevo array al actualizar el estado:

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

Si quieres eliminar elementos de un array, debes usar el método `filter()`, que devuelve un nuevo array después de filtrar lo que quieres eliminar:

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```

## Referenciar valores usando refs

- **Atributo `ref`**: Puedes acceder a un nodo DOM en React usando el atributo `ref`. Aquí un ejemplo que muestra un `ref` para enfocar un elemento `input`. La propiedad `current` se usa para acceder al valor actual de ese `ref`:

```jsx
import { useRef } from "react";

const Focus = () => {
  const inputRef = useRef(null);

  const handleFocus = () => {
    if (inputRef.current) {
      inputRef.current.focus();
    }
  };

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Enter text" />
      <button onClick={handleFocus}>Focus Input</button>
    </div>
  );
};

export default Focus;
```

## Trabajar con el hook `useEffect`

- **Hook `useEffect()`**: En React, un efecto es cualquier cosa que sucede fuera del proceso de renderizado del componente. Es decir, cualquier cosa que React no maneja directamente como parte del renderizado de la UI. Ejemplos comunes incluyen obtener datos, actualizar el título de la pestaña del navegador, leer o escribir en el almacenamiento local del navegador, obtener la ubicación del usuario y mucho más. Estas operaciones interactúan con el mundo exterior y se conocen como efectos secundarios. React proporciona el hook `useEffect` para manejar esos efectos secundarios. `useEffect` te permite ejecutar una función después de que el componente renderiza o actualiza.

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [dependencies]);
```

La función efecto se ejecuta después de que el componente renderiza, mientras que el argumento opcional `dependencies` controla cuándo se ejecuta el efecto.

Ten en cuenta que `dependencies` puede ser un array de "valores reactivos" (estado, props, funciones, variables, etc.), un array vacío o puede omitirse por completo. Así es como esas opciones controlan el funcionamiento de `useEffect`:

- Si `dependencies` es un array que incluye uno o más valores reactivos, el efecto se ejecutará cada vez que cambien.

- Si `dependencies` es un array vacío, `useEffect` se ejecuta solo una vez cuando el componente se renderiza por primera vez.

- Si omites `dependencies`, el efecto se ejecuta cada vez que el componente renderiza o actualiza.

## Cómo crear hooks personalizados

- **Hooks personalizados**: Un hook personalizado te permite extraer lógica reutilizable de los componentes, como obtención de datos, gestión de estado, alternancia y efectos secundarios como seguimiento del estado en línea. En React, todos los hooks integrados comienzan con la palabra `use`, por lo que tu hook personalizado debe seguir la misma convención.

Aquí un ejemplo de creación de un hook `useDebounce`:

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```

## Trabajar con formularios en React

- **Entradas controladas**: Esto es cuando almacenas el valor del campo de entrada en el estado y lo actualizas mediante eventos `onChange`. Esto te da control completo sobre los datos del formulario y permite validación instantánea y renderizado condicional.

```jsx
import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **Entradas no controladas**: En lugar de manejar las entradas mediante el hook `useState`, las entradas no controladas en HTML mantienen su propio estado interno con ayuda del DOM. Dado que el DOM controla los valores de entrada, necesitas obtener los valores de los campos de entrada con un `ref`.

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```

## Trabajar con el hook `useActionState`

- **Acciones del servidor**: Son funciones que se ejecutan en el servidor para permitir el manejo de formularios directamente en el servidor sin necesidad de endpoints API. Aquí un ejemplo de una aplicación Next.js:

```js
"use server";

async function submitForm(formData) {
 const name = formData.get("name");
 return { message: `Hello, ${name}!` };
}
```

La directiva `"use server"` marca la función como una acción del servidor.

- **Hook `useActionState`**: Este hook actualiza el estado basado en el resultado de una presentación de formulario. Aquí la sintaxis básica del hook `useActionState`:

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state` es el estado actual que devuelve la acción.
- `action` es la función que dispara la acción del servidor.
- `isPending` es un booleano que indica si la acción está en ejecución o no.
- `actionFunction` es el parámetro que representa la acción del servidor.
- `initialState` es el parámetro que representa el punto de partida para el estado antes de que la acción se ejecute.
- `permalink` es una cadena opcional que contiene la URL única de la página que el formulario modifica.

## Obtención de datos en React

- **Opciones para obtener datos**: Hay muchas formas diferentes de obtener datos en React. Puedes usar la API Fetch nativa o herramientas de terceros como Axios o SWR.
- **Variables de estado comúnmente usadas al obtener datos**: Independientemente del método que elijas para obtener datos en React, hay algunas piezas de estado que necesitarás controlar. La primera es la propia data. La segunda rastrea si los datos aún se están obteniendo. La tercera es una variable de estado que captura cualquier error que pueda ocurrir durante el proceso de obtención de datos.

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

Dado que la obtención de datos es un efecto secundario, es mejor usar `Fetch API` dentro de un hook `useEffect`.

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

Luego puedes renderizar un mensaje de carga si la obtención de datos no está completa, un mensaje de error si hubo un error al obtener los datos, o los resultados.

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

Si quieres usar Axios, necesitas instalarlo e importarlo:

```bash
npm i axios
```

```js
import axios from "axios";
```

Luego puedes obtener los datos usando `axios.get`:

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);

useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await axios.get(
        "https://jsonplaceholder.typicode.com/users"
      );
      setData(res.data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

## Trabajar con el hook `useOptimistic`

- **Hook `useOptimistic`**: Este hook se usa para mantener las interfaces de usuario responsivas mientras se espera que una acción asíncrona se complete en segundo plano. Ayuda a manejar "actualizaciones optimistas" en la UI, una estrategia en la que proporcionas actualizaciones inmediatas a la UI basadas en el resultado esperado de una acción, como esperar una respuesta del servidor.

Aquí la sintaxis básica:

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState` es el estado temporal que se actualiza de inmediato para una mejor experiencia de usuario.
- `addOptimistic` es la función que aplica la actualización optimista antes de que el estado real cambie.
- `actualState` es el valor real del estado que proviene del resultado de una acción, como obtener datos de un servidor.
- `updateFunction` es la función que determina cómo debe actualizarse el estado optimista cuando se llama.

Aquí un ejemplo de uso del hook `useOptimistic` en un componente `TaskList`:

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**: Se usa para renderizar parte de la UI y marcar una actualización de estado como una transición no urgente. Esto permite que la UI sea responsiva durante actualizaciones costosas. Aquí la sintaxis básica:

```js
startTransition(action)
```

El `action` realiza una actualización de estado o dispara alguna lógica relacionada con la transición. Esto asegura que las actualizaciones urgentes de la UI (como escribir o hacer clic) no se bloqueen.

## React Router

- **Introducción**: React Router es una biblioteca de terceros que te permite agregar enrutamiento a tus aplicaciones React. Para comenzar, necesitarás instalar React Router en un proyecto React existente así:

```bash
npm i react-router
```

Luego, dentro del archivo `main.jsx` o `index.jsx`, deberás configurar la estructura de rutas así:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

Los componentes `path` y `element` se usan para vincular la URL y los componentes UI juntos. En este caso, configuramos una ruta para la página principal que apunta al componente `App`.

- **Múltiples vistas y configuración de rutas**: Es común en aplicaciones grandes tener múltiples vistas y rutas configuradas así:

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

La prop `index` en estos ejemplos representa la ruta por defecto para un segmento de ruta dado. Así, el componente `Home` se mostrará en la ruta raíz `/`, mientras que el componente `ProductsHome` se mostrará en la ruta `/products`.

- **Anidamiento de rutas**: Puedes anidar rutas dentro de otras rutas, lo que resulta en que la ruta del hijo se añada a la ruta del padre.

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

En el ejemplo anterior, la ruta para productos en tendencia será products/trending.

- **Segmentos dinámicos**: Un segmento dinámico es cuando cualquier parte de la ruta URL es dinámica.

```jsx
<Route path=":category" element={<Category />} />
```

En este ejemplo tenemos un segmento dinámico llamado `category`. Cuando un usuario navega a una URL como products/brass-instruments, la vista cambiará al componente `Category` y puedes obtener dinámicamente los datos apropiados basados en el segmento.

- **Hook `useParams`**: Este hook se usa para acceder a los parámetros dinámicos de una ruta URL.

```js
import { useParams } from "react-router";

export default function Category() {
  let params = useParams();
   {/* Accessing the category param: params.category */}
   {/* rest of code goes here */}
}
```

## Gestión de estado

- **Context API**: Contexto se refiere a cuando un componente padre pone información disponible para componentes hijos sin necesidad de pasarla explícitamente mediante props. `createContext` se usa para crear un objeto de contexto que representa el contexto que otros componentes leerán. `Provider` se usa para suministrar valores de contexto a los componentes hijos.

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**: Redux maneja la gestión de estado proporcionando una tienda central y control estricto sobre las actualizaciones de estado. Usa un patrón predecible con acciones, reductores y middleware. Las acciones son cargas de información que envían datos desde tu aplicación a la tienda Redux, a menudo disparadas por interacciones del usuario. Los reductores son funciones que especifican cómo debe cambiar el estado en respuesta a esas acciones, asegurando que el estado se actualice de forma inmutable. El middleware, por otro lado, actúa como puente entre el despacho de acciones y el reductor, permitiendo extender la funcionalidad de Redux (por ejemplo, registro, manejo de operaciones asíncronas) sin modificar el flujo central.

- **Zustand**: Esta solución de gestión de estado es ideal para aplicaciones de pequeña a mediana escala. Funciona usando un hook `useStore` para acceder al estado directamente en componentes y páginas. Esto te permite modificar y acceder a datos sin necesidad de acciones, reductores o un proveedor.

## Diferencias entre rendimiento real y percibido

- **Rendimiento percibido**: Es cómo los usuarios perciben el rendimiento de un sitio web. Es cómo lo evalúan en términos de capacidad de respuesta y fiabilidad. Es una medida subjetiva, por lo que es difícil cuantificarla, pero es muy importante, ya que la experiencia del usuario determina el éxito o fracaso de un sitio.
- **Rendimiento real**: Es el rendimiento objetivo y medible del sitio web. Se mide usando métricas como tiempo de carga de página, tiempo de respuesta del servidor y tiempo de renderizado. Estas mediciones están influenciadas por múltiples factores relacionados con la red y el código mismo.

## Técnicas para mejorar el rendimiento percibido

- **Carga diferida (lazy loading)**: Esta técnica reduce el tiempo de carga inicial al máximo cargando recursos no esenciales en segundo plano.
- **Minimizar retrasos en fuentes**: Si tu sitio tiene fuentes personalizadas, también deberías intentar minimizar los retrasos en la carga de fuentes, ya que esto puede causar parpadeos o mostrar la fuente de reserva mientras se carga la fuente personalizada. Una sugerencia es usar una fuente de reserva similar a la personalizada para que el cambio sea más sutil.
- **Uso de indicadores de carga**: Mostrar un indicador de carga para un proceso largo tan pronto como el usuario haga clic en un elemento puede ayudar a que el usuario se sienta conectado y comprometido con el proceso, haciendo que el tiempo de espera parezca más corto.

## Conceptos clave de rendimiento

- **Orden de fuente**: Se refiere a cómo están estructurados los elementos HTML en el documento. Esto determina qué se carga primero y puede impactar significativamente el rendimiento y la accesibilidad.

Algunas buenas prácticas para el orden de fuente incluyen:

- Colocar contenido crítico como encabezados, navegación o texto principal más arriba en la estructura HTML.
- Diferir scripts no esenciales como los de analíticas o widgets de terceros para que no bloqueen el renderizado.
- Usar mejora progresiva para asegurar que la experiencia básica funcione incluso antes de que se carguen estilos y scripts. La mejora progresiva es una forma de construir sitios y aplicaciones basada en la idea de que primero debe funcionar con HTML.

Aquí un ejemplo de buen orden de fuente, usando las mejores prácticas que acabamos de ver:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **Ruta crítica de renderizado**: Es la secuencia de pasos que sigue el navegador para convertir código en píxeles en pantalla.
- **Latencia**: Es el tiempo que tarda una solicitud en viajar entre el navegador y el servidor. En otras palabras, alta latencia equivale a páginas lentas.

Algunas formas de reducir la latencia incluyen:

- Usar CDN o redes de entrega de contenido para servir archivos desde ubicaciones más cercanas.
- Habilitar compresión usando cosas como Gzip para reducir tamaños de archivos.
- Optimizar imágenes y usar carga diferida.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## Mejorar INP

- **Definición**: INP (Interaction to Next Paint) evalúa la capacidad de respuesta general de una página midiendo el tiempo desde que un usuario interactúa, como un clic o pulsación de tecla, hasta la siguiente vez que el navegador actualiza la pantalla. Un INP más bajo indica una página más responsiva.

Aquí algunas formas de mejorar INP:

- Reducir el trabajo del hilo principal dividiendo tareas largas de JavaScript.
- Usar `requestIdleCallback()` para scripts no críticos. Esto encola una función para ser llamada durante periodos de inactividad del navegador.
- Diferir o cargar de forma diferida recursos pesados que se cubrieron antes.
- Optimizar manejadores de eventos. Si estos manejadores se ejecutan con demasiada frecuencia o realizan operaciones pesadas, pueden ralentizar la página y aumentar INP. La solución es el debouncing. El debouncing asegura que la función solo se ejecute después de que el usuario deje de escribir por un breve retraso, por ejemplo 300ms. Esto previene cálculos innecesarios y mejora el rendimiento.

## Cómo funciona el renderizado en el navegador

- **Cómo funciona el renderizado**: Primero el navegador analiza el HTML y construye el DOM. Luego procesa el CSS, construyendo el Modelo de Objetos CSS o CSSOM. Esta es otra estructura de árbol que dicta cómo deben estilizarse los elementos. Finalmente, el navegador pinta los píxeles en pantalla, renderizando cada elemento basado en los estilos y el diseño calculados. En páginas complejas, esto puede involucrar múltiples capas que se componen para formar la salida visual final.

## Cómo el rendimiento impacta la sostenibilidad

- **Información de fondo**: Internet representa alrededor del 2% de las emisiones globales de carbono — ¡lo mismo que la industria aérea! Cada byte transferido requiere electricidad, desde centros de datos hasta dispositivos de usuario. Archivos grandes y scripts ineficientes significan más consumo de energía. Un sitio web de alto rendimiento no solo es más rápido, también reduce el procesamiento y uso de energía innecesarios.

## Formas de reducir tiempos de carga de página

- **Optimizar recursos multimedia**: Imágenes y videos grandes son causas comunes de tiempos de carga lentos. Al optimizar estos recursos, puedes acelerar significativamente tu sitio. Esto incluye comprimir imágenes, usar formatos modernos como WebP y usar carga diferida para recursos.
- **Aprovechar el caché del navegador**: El caché permite que los navegadores almacenen partes de tu sitio localmente, reduciendo tiempos de carga para visitantes recurrentes.
- **Minificar y comprimir archivos**: Reducir el tamaño de tus archivos puede llevar a descargas más rápidas. Esto incluye reducir el tamaño de archivos transmitidos y minificar CSS y JavaScript.

## Mejorar "tiempo hasta usable"

- **Definición**: "Tiempo hasta usable" es el intervalo desde que un usuario solicita una página hasta que puede interactuar con ella de forma significativa. Para mejorar este tiempo puedes cargar tus recursos de forma diferida o minimizar recursos que bloquean el renderizado.

## Métricas clave para medir rendimiento

- **First Contentful Paint o FCP**: Mide qué tan rápido aparece el primer contenido — texto o imagen — en pantalla. Un buen FCP se considera un tiempo menor a 1.8 segundos, y un FCP pobre es más de 3 segundos. Puedes revisar tu FCP usando Chrome DevTools en la pestaña de rendimiento.
- **Tiempo total de bloqueo**: Muestra cuánto tiempo el hilo principal está bloqueado por tareas pesadas de JavaScript. Si el Tiempo Total de Bloqueo (TBT) es alto, los usuarios experimentan interacciones lentas. Para mejorar TBT, divide tareas largas y difiere scripts no esenciales.
- **Tasa de rebote**: Es el porcentaje de visitantes que se van sin interactuar. Si tu sitio tiene altas tasas de rebote, podría ser porque tu página es demasiado lenta.
- **Usuarios únicos**: Esta métrica rastrea cuántos visitantes individuales llegan a tu sitio. Para ver la tasa de rebote y usuarios únicos, puedes usar Google Analytics. Te permitirá monitorear estas métricas y mejorar el compromiso.

## Trabajar con APIs web de rendimiento

- **Definición**: Las APIs web de rendimiento permiten a los desarrolladores rastrear qué tan eficientemente carga y responde una página web directamente en el código. Estas APIs permiten medir tiempos de carga, rastrear retrasos en renderizado e interacción y analizar tiempo de ejecución de JavaScript.
- **`performance.now()`**: Esta API te da marcas de tiempo de alta precisión (en milisegundos) para medir cuánto tardan diferentes partes de tu sitio en cargar.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **API de temporización de rendimiento**: Esta API te da un desglose de cada etapa de carga de página, desde la búsqueda DNS hasta `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: Esta API escucha eventos de rendimiento como cambios de diseño, tareas largas e interacciones de usuario.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## Técnicas para mejorar el rendimiento CSS

- **Animaciones CSS**: Animar ciertas propiedades CSS, como dimensiones, posición y diseño, desencadena un proceso llamado "reflujo", durante el cual el navegador recalcula la posición y geometría de ciertos elementos en la página. Esto requiere un repintado, que es computacionalmente costoso. Por lo tanto, se recomienda reducir la cantidad de animaciones CSS tanto como sea posible o al menos dar al usuario la opción de activarlas o desactivarlas.

## Técnicas para mejorar el rendimiento JavaScript

- **División de código**: Dividir tu código JavaScript en módulos que realicen tareas críticas y no críticas también es útil. Así podrás precargar los críticos lo antes posible y diferir los no críticos para renderizar la página lo más rápido posible.
- **Manipulación del DOM**: Recuerda que la manipulación del DOM se refiere al proceso de cambiar dinámicamente el contenido de una página con JavaScript interactuando con el Modelo de Objetos del Documento (DOM). Manipular el DOM es computacionalmente costoso. Reducir la cantidad de manipulación del DOM en tu código JavaScript mejorará el rendimiento.

## Pruebas manuales y automatizadas

- **Pruebas manuales**: En las pruebas manuales, un tester revisa manualmente cada parte de la aplicación y prueba diferentes funciones para asegurarse de que funcionan correctamente. Si se encuentran errores, el tester los reporta al equipo de software para que los corrijan.
- **Pruebas automatizadas**: En las pruebas automatizadas, puedes automatizar tus pruebas escribiendo un programa separado que verifica si tu aplicación se comporta como se espera.

## Pruebas unitarias

- **Pruebas unitarias**: En las pruebas unitarias, pruebas cada función para asegurar que todo funciona como se espera. Las pruebas unitarias también pueden servir como documentación para tu aplicación porque representan el comportamiento esperado de tu código.
- **Principio de responsabilidad única**: Este principio recomienda mantener cada función pequeña y responsable de una sola cosa.
- **Frameworks comunes de pruebas JavaScript**: Algunos frameworks comunes incluyen Jest, Mocha y Vitest. Jest es un framework popular para pruebas unitarias.

Aquí un ejemplo de pruebas unitarias usando Jest.

Primero, puedes crear una función responsable de devolver una cadena formateada nueva:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

En un archivo `getFormattedWord.test.js` separado, puedes escribir algunas pruebas para verificar que la función funciona como se espera. El archivo `getFormattedWord.test.js` se verá así:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **Función `expect`**: La función `expect` se usa para probar un valor.
- **Matcher**: Un matcher es una función que verifica si el valor se comporta como se espera. En el ejemplo anterior, el matcher es `toBe()`. Jest tiene una variedad de matchers.

Para usar Jest, primero necesitas instalar el paquete `jest` usando `npm i jest`. También necesitarás agregar un script a tu archivo `package.json` así:

```js
"scripts": {
  "test": "jest"
},
```

Luego, puedes ejecutar el comando `npm run test` para correr tus pruebas.

## Ciclo de vida del desarrollo de software

- **Diferentes etapas del ciclo de vida del desarrollo de software**:
  - **Etapa de planificación**: El equipo de desarrollo recopila requisitos del trabajo propuesto de los interesados.
  - **Etapa de diseño**: El equipo de software desglosa los requisitos y decide las mejores aproximaciones para las soluciones.
  - **Etapa de implementación**: El equipo de software desglosa los requisitos en tareas manejables y las implementa.
  - **Etapa de pruebas**: Incluye pruebas manuales y automatizadas para el nuevo trabajo. A veces, el equipo prueba la aplicación durante toda la etapa de desarrollo para detectar y corregir problemas.
  - **Etapa de despliegue**: El equipo despliega los nuevos cambios a un entorno de construcción o pruebas.
  - **Etapa de mantenimiento**: Incluye corregir problemas que surjan de los clientes en la aplicación en producción.

- **Diferentes modelos del ciclo de vida del desarrollo de software**:
  - **Modelo cascada**: El modelo cascada es donde cada fase del ciclo debe completarse antes de que comience la siguiente.
  - **Modelo ágil**: El modelo ágil se enfoca en desarrollo iterativo dividiendo el trabajo en sprints.

## BDD y TDD

- **TDD**: Desarrollo guiado por pruebas es una metodología que enfatiza escribir pruebas primero. Escribir pruebas antes de construir características proporciona retroalimentación en tiempo real a los desarrolladores durante el proceso.
- **BDD**: Desarrollo guiado por comportamiento es el enfoque de alinear una serie de pruebas con objetivos de negocio. Los escenarios de prueba en BDD deben escribirse en un lenguaje que puedan entender tanto técnicos como no técnicos. Un ejemplo de esta sintaxis es Gherkin.
- **Frameworks de pruebas BDD**: Ejemplos incluyen Cucumber, JBehave y SpecFlow.

## Aserciones en pruebas unitarias

- **Aserción**: Las aserciones se usan para probar que el código se comporta como se espera.
- **Librerías de aserciones**: Chai es una librería de aserciones comúnmente usada. Otras librerías comunes de aserciones JavaScript son `should.js` y `expect.js`.

Aquí un ejemplo de una aserción usando Chai que verifica que el valor retornado por la función `addThreeAndFour` es igual al número 7:

## Mocking, faking y stubbing

- **Mocking**: Mocking es el proceso de reemplazar datos reales con datos falsos que simulan el comportamiento de componentes reales. Por ejemplo, podrías simular la respuesta de una API en pruebas en lugar de hacer llamadas continuas a la API para obtener datos.
- **Stubbing**: Los stubs son objetos que devuelven respuestas predefinidas o datos ficticios para un comportamiento esperado en una aplicación. Por ejemplo, puedes stubear el comportamiento de una conexión a base de datos en tus pruebas sin depender de una conexión real.
- **Faking**: Los fakes son versiones simplificadas de componentes reales sin la complejidad o efectos secundarios. Por ejemplo, puedes simular una base de datos almacenando datos en memoria en lugar de interactuar con la base real. Esto permite imitar operaciones de base de datos en memoria, que es mucho más rápido que tratar con la base real.

## Pruebas funcionales

- **Pruebas funcionales**: Verifican si las características y funciones de la aplicación funcionan como se espera. El objetivo es probar el sistema completo contra múltiples escenarios.
- **Pruebas no funcionales**: Se enfocan en aspectos como rendimiento y fiabilidad.
- **Pruebas de humo**: Son una verificación preliminar del sistema para detectar problemas básicos o críticos antes de comenzar pruebas más extensas.

## Pruebas de usabilidad

- **Pruebas de usabilidad**: Son cuando usuarios reales interactúan con la aplicación para descubrir si hay problemas de diseño, experiencia de usuario o funcionalidad. Se enfocan en la intuición del uso de la aplicación por parte de los usuarios.
- **Cuatro tipos comunes de pruebas de usabilidad**:
  - **Explorativas**: Los usuarios interactúan con diferentes funciones para entender mejor cómo funcionan.
  - **Comparativas**: Comparas la experiencia de usuario de tu aplicación con aplicaciones similares en el mercado.
  - **Evaluación**: Estudias qué tan intuitiva es la aplicación para usar.
  - **Validación**: Identificas problemas mayores que impidan usar la aplicación efectivamente.
- **Herramientas para pruebas de usabilidad**: Ejemplos incluyen Loop11, Maze, Userbrain, UserTesting y UXTweak.

## Pruebas de compatibilidad

- **Pruebas de compatibilidad**: El objetivo es asegurar que tu aplicación funcione en diferentes entornos computacionales.
- **Tipos de pruebas de compatibilidad**:
  - **Compatibilidad hacia atrás**: El software es compatible con versiones anteriores.
  - **Compatibilidad hacia adelante**: El software y sistemas funcionarán con versiones futuras.
  - **Compatibilidad de hardware**: El software funciona correctamente en diferentes configuraciones de hardware.
  - **Compatibilidad con sistemas operativos**: El software funciona en diferentes sistemas operativos, como macOS, Windows y distribuciones Linux como Ubuntu y Fedora.
  - **Compatibilidad de red**: El software funciona en diferentes condiciones de red, como velocidades, protocolos, configuraciones de seguridad, etc.
  - **Compatibilidad de navegador**: La aplicación web funciona consistentemente en diferentes navegadores, como Google Chrome, Safari, Firefox, etc.
  - **Compatibilidad móvil**: Es importante asegurar que tus aplicaciones funcionen en una variedad de dispositivos Android e iOS, incluyendo teléfonos y tablets.

## Pruebas de rendimiento

- **Pruebas de rendimiento**: Se prueba la velocidad, capacidad de respuesta, escalabilidad y estabilidad de una aplicación bajo diferentes cargas. El objetivo es resolver cualquier cuello de botella de rendimiento.
- **Tipos de pruebas de rendimiento**:
  - **Pruebas de carga**: Determinan cómo se comporta un sistema durante cargas normales y máximas.
  - **Pruebas de estrés**: Pruebas bajo cargas extremas para ver cómo responde el sistema.
  - **Pruebas de resistencia (soak testing)**: Pruebas con carga alta durante un período prolongado.
  - **Pruebas de picos**: Incrementos y decrementos dramáticos de carga para analizar reacciones del sistema.
  - **Pruebas de punto de ruptura (capacity testing)**: Incremento lento de carga hasta que el sistema falla o degrada.

## Pruebas de seguridad

- **Pruebas de seguridad**: Ayudan a identificar vulnerabilidades y debilidades.
- **Principios de seguridad**:
  - **Confidencialidad**: Protege contra la divulgación de información sensible a destinatarios no autorizados.
  - **Integridad**: Previene que usuarios maliciosos modifiquen información de usuarios.
  - **Autenticación**: Verifica la identidad del usuario para asegurar que puede usar el sistema.
  - **Autorización**: Determina qué acciones pueden realizar usuarios autenticados o qué partes del sistema pueden acceder.
  - **Disponibilidad**: Asegura que la información y servicios estén disponibles para usuarios autorizados cuando los necesiten.
  - **No repudio**: Asegura que tanto remitente como receptor tengan prueba de entrega y verificación de identidad, protegiendo contra negaciones.
- **Amenazas comunes de seguridad**:
  - **Cross-Site Scripting (XSS)**: Ataques donde un atacante inyecta scripts maliciosos en una página web y los ejecuta en el navegador de la víctima.
  - **Inyección SQL**: Permite a usuarios maliciosos inyectar código malicioso en una base de datos.
  - **Ataque de denegación de servicio (DoS)**: Usuarios maliciosos inundan un sitio con muchas solicitudes o tráfico, causando que el servidor se ralentice o caiga, haciendo el sitio inaccesible.
- **Categorías de herramientas de pruebas de seguridad**:
  - **Pruebas estáticas de seguridad de aplicaciones**: Evalúan el código fuente para identificar vulnerabilidades.
  - **Pruebas dinámicas de seguridad de aplicaciones**: Interactúan con el front-end para descubrir debilidades potenciales. No tienen acceso al código fuente.
- **Pruebas de penetración (pentest)**: Tipo de prueba de seguridad que simula ciberataques para identificar vulnerabilidades en el sistema.

## Pruebas A/B

- **Pruebas A/B**: Consisten en comparar dos versiones de una página o aplicación y estudiar cuál funciona mejor. También se conocen como pruebas bucket o split. Permiten tomar decisiones basadas en datos y mejorar continuamente la experiencia de usuario.
- **Herramientas para pruebas A/B**: Ejemplos incluyen GrowthBook y LaunchDarkly.

## Pruebas alfa y beta

Una vez que el desarrollo inicial y las pruebas de software están completas, es importante que la aplicación sea probada por testers y usuarios reales. Aquí entran las pruebas alfa y beta.

- **Pruebas alfa**: Se realizan por un grupo selecto de testers que revisan la aplicación para asegurar que no haya errores antes de su lanzamiento. Son parte de las pruebas de aceptación y usan técnicas de caja blanca y negra.
- **Pruebas beta**: La aplicación se pone a disposición de usuarios reales. Los usuarios pueden interactuar y dar retroalimentación. También es una forma de pruebas de aceptación de usuario.
- **Pruebas de aceptación**: Aseguran que la aplicación cumple con los requisitos de negocio y necesidades de usuarios antes del lanzamiento.
- **Pruebas de caja negra**: Se enfocan solo en el comportamiento esperado de la aplicación.
- **Pruebas de caja blanca**: El tester conoce los componentes internos y realiza pruebas sobre ellos.

## Pruebas de regresión

- **Regresión**: Se refiere a situaciones donde cambios nuevos rompen funcionalidades existentes sin querer.
- **Pruebas de regresión**: Ayudan a detectar problemas de regresión. Consisten en volver a ejecutar pruebas funcionales en partes de la aplicación para asegurar que todo sigue funcionando como se espera.
- **Herramientas para pruebas de regresión**: Puppeteer, Playwright, Selenium y Cypress.
- **Técnicas para pruebas de regresión**:
  - **Pruebas unitarias de regresión**: Lista de ítems que deben probarse cada vez que se implementan cambios o correcciones importantes.
  - **Pruebas parciales de regresión**: Enfoques dirigidos para asegurar que cambios nuevos no rompan aspectos específicos.
  - **Pruebas completas de regresión**: Ejecutan pruebas en todas las funcionalidades del código. Es la opción más detallada y consume más tiempo.
- **Repruebas**: Se usan para verificar problemas conocidos y asegurar que se hayan resuelto. En contraste, las pruebas de regresión buscan problemas desconocidos que puedan haber surgido por cambios recientes.

## Qué es TypeScript

- **JavaScript**: Es un lenguaje de tipado dinámico. Esto significa que las variables pueden recibir cualquier valor en tiempo de ejecución. El desafío es que la falta de seguridad de tipos puede introducir errores.

Por ejemplo, aunque tu función JavaScript espere un array, aún puedes llamarla con un número:

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

La salida `console` para el ejemplo anterior será `undefined`.

- **TypeScript**: Extiende JavaScript para incluir tipado estático, lo que ayuda a detectar errores causados por incompatibilidades de tipos antes de ejecutar el código.

Por ejemplo, puedes definir un tipo para el parámetro `array` así:

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

Esta definición indica a TypeScript que el parámetro `array` debe ser un array de cadenas. Luego, si llamas a `getRandomValue` y le pasas un número, obtendrás un error de compilador.

- **Compilador**: Primero debes compilar el código TypeScript a JavaScript regular. Al ejecutar el compilador, TypeScript evalúa tu código y lanza errores donde los tipos no coinciden.

## Tipos de datos en TypeScript

- **Tipos de datos primitivos en TypeScript**: Para los tipos primitivos `string`, `null`, `undefined`, `number`, `boolean` y `bigint`, TypeScript ofrece palabras clave de tipo correspondientes.

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **Array**: Puedes definir un array de tipo específico con dos sintaxis diferentes.

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **Objetos**: Puedes definir la estructura exacta de un objeto.

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

Si quieres un objeto con cualquier clave, pero donde todos los valores deben ser cadenas, hay dos formas de definirlo:

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **Otros tipos útiles en TypeScript**:
  - **`any`**: `any` indica que un valor puede tener cualquier tipo. Le dice al compilador que deje de preocuparse por el tipo de esa variable.
  - **`unknown`**: `unknown` indica que te importa el tipo del valor, pero no sabes cuál es. `unknown` es generalmente preferido sobre `any`.
  - **`void`**: Tipo especial que normalmente usas solo al definir funciones. Funciones sin valor de retorno usan tipo de retorno `void`.
  - **`never`**: Representa un tipo que nunca existirá.

- **Palabra clave `type`**: Es como `const`, pero en lugar de declarar una variable, declaras un tipo.

Es útil para declarar tipos personalizados como tipos unión o tipos que incluyen solo valores específicos:

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**: Las interfaces son como clases para tipos. Pueden implementar o extender otras interfaces, son específicamente tipos de objeto y generalmente se prefieren a menos que necesites una característica específica de una declaración `type`.

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **Definir tipo de retorno**: También puedes definir el tipo de retorno de la función.

El ejemplo define el valor de retorno como cadena. Si intentas retornar otra cosa, TypeScript dará error de compilador.

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```

## Genéricos

- **Definir tipo genérico**: Puedes definir un tipo genérico y referirte a él en tu función. Se puede pensar como un parámetro especial que proporcionas a una función para controlar el comportamiento de la definición de tipo de la función.

Aquí un ejemplo de definir un tipo genérico para una función:

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

La sintaxis `<T>` indica a TypeScript que defines un tipo genérico `T` para la función. `T` es un nombre común para tipos genéricos, pero puedes usar cualquier nombre.

Luego, indicas a TypeScript que el parámetro `array` es un array de valores que coinciden con el tipo genérico, y que el valor retornado es un solo elemento de ese mismo tipo.

- **Especificar argumento de tipo en llamada a función**: Puedes pasar un argumento de tipo a una llamada de función usando corchetes angulares entre el nombre de la función y sus parámetros.

Aquí un ejemplo de pasar un argumento de tipo a una llamada de función:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Esto indica a TypeScript que el elemento que esperas encontrar será un elemento input.

## Restricción de tipos (type narrowing)

- **Restricción por veracidad**: En el ejemplo, obtienes un error de compilador al intentar acceder a la propiedad `value` de `email` porque `email` podría ser `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Puedes usar una condición para confirmar que `email` es verdadero antes de acceder a la propiedad:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

Las comprobaciones de veracidad también pueden funcionar en dirección inversa:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

Lanzar un error termina la ejecución lógica de este código, lo que significa que cuando llegas a la llamada `console.log()`, TypeScript sabe que `email` no puede ser `null`.

- **Encadenamiento opcional**: El encadenamiento opcional `?.` también es una forma de restricción de tipos, bajo la misma premisa de que el acceso a la propiedad no puede ocurrir si el valor `email` es `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **Operador `typeof`**: Puedes usar una condición para verificar el tipo de una variable usando el operador `typeof`.

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **Palabra clave `instanceof`**: Si el objeto proviene de una clase, puedes usar la palabra clave `instanceof` para restringir el tipo.

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **Casting**: Cuando TypeScript no puede determinar automáticamente el tipo de un valor, como el resultado del método `request.json()` en el ejemplo, obtendrás un error de compilador. Una forma de resolverlo es haciendo un casting, pero esto debilita la capacidad de TypeScript para detectar errores potenciales.

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **Guardia de tipo**: En lugar de hacer casting, puedes escribir una guardia de tipo:

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

La sintaxis `user is User` indica que tu función devuelve un booleano que, cuando es true, significa que el valor `user` satisface la interfaz `User`.

## Archivo `tsconfig`

- **`tsconfig.json`**: La configuración del compilador de TypeScript vive en un archivo `tsconfig.json` en el directorio raíz de tu proyecto.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

Aquí descripciones de las opciones del compilador usadas en el ejemplo:

- **`compilerOptions`**: La propiedad `compilerOptions` controla cómo se comporta el compilador TypeScript.
- **`rootDir` y `outDir`**: `rootDir` y `outDir` indican a TypeScript qué directorio contiene tus archivos fuente y cuál debe contener el código JavaScript transpilado.
- **`lib`**: La propiedad `lib` determina qué definiciones de tipo usa el compilador y permite incluir soporte para versiones específicas de ES, DOM y más.
- **`module` y `moduleResolution`**: `module` y `moduleResolution` trabajan en conjunto para gestionar cómo tu paquete usa módulos — CommonJS o ECMAScript.
- **`esModuleInterop`**: `esModuleInterop` permite una interoperabilidad más fluida entre CommonJS y módulos ES creando automáticamente objetos namespace para importaciones.
- **`skipLibCheck`**: La opción `skipLibCheck` omite validar archivos `.d.ts` que no están referenciados por importaciones en tu código.
- **`strict`**: La bandera `strict` habilita varias comprobaciones, como asegurar manejo adecuado de tipos anulables y advertir cuando TypeScript recurre a `any`.
- **`exclude`**: La propiedad de nivel superior `exclude` indica al compilador que ignore esos archivos TypeScript durante la compilación.
