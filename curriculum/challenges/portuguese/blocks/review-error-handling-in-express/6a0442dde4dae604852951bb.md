## Códigos de status de resposta HTTP

- **Códigos de status** são mensagens numéricas curtas que informam ao cliente o que aconteceu após uma requisição (por exemplo, 200, 404 ou 500).

### 1xx — Informativo

- **100 Continue** — o servidor indica ao cliente para enviar o restante da requisição.
- **101 Switching Protocols** — o servidor está trocando de protocolo (comum em WebSockets).

### 2xx — Sucesso

- **200 OK** — a requisição foi bem-sucedida.
- **201 Created** — um novo recurso foi criado com sucesso.
- **204 No Content** — a requisição foi bem-sucedida, mas não há corpo na resposta (frequentemente usado para `DELETE`).

### 3xx — Redirecionamento

- **301 Moved Permanently** — o recurso tem uma nova URL permanente.
- **302 Found** — o recurso está temporariamente em uma URL diferente.
- **304 Not Modified** — o cliente pode usar sua versão em cache; nada mudou.

### 4xx — Erros do cliente

- **400 Bad Request** — a requisição está malformada ou inválida.
- **401 Unauthorized** — é necessária autenticação.
- **403 Forbidden** — autenticado, mas sem permissão.
- **404 Not Found** — o recurso solicitado não existe.

### 5xx — Erros do servidor

- **500 Internal Server Error** — falha genérica no servidor.
- **502 Bad Gateway** — um proxy recebeu uma resposta inválida de outro servidor.
- **503 Service Unavailable** — o servidor está temporariamente sobrecarregado ou indisponível.

### Usando códigos de status no Express

- `res.status(400).send("Bad Request")` — envia um 400 com uma mensagem em texto simples.
- `res.status(200).json({ message: "Success" })` — envia um 200 com um corpo JSON.
- `res.status(500).send("Internal Server Error")` — envia um 500 com uma mensagem.

## Tratamento de erros no Express

- **Middleware de tratamento de erros** tem quatro parâmetros: `(err, req, res, next)`. Essa assinatura com quatro parâmetros é o que indica ao Express que se trata de um manipulador de erros.
- Coloque o middleware de tratamento de erros **no final** da pilha de middleware, depois de todas as rotas.

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Tratamento de erros assíncronos no Express 4

- Erros assíncronos **não são capturados automaticamente** no Express 4.
- Envolva a lógica assíncrona em `try/catch` e chame `next(err)` para encaminhar o erro ao manipulador de erros.

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- Um wrapper `asyncHandler` evita repetir `try/catch` em cada rota:

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Tratamento de erros assíncronos no Express 5

- No Express 5, erros assíncronos e promises rejeitadas são **encaminhados automaticamente** para o manipulador de erros — não é necessário chamar `next(err)` manualmente.

### Tratando erros 404

- Adicione um middleware catch-all **depois das rotas, mas antes do manipulador de erros** para lidar com rotas não correspondentes:

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Melhores práticas para tratamento de erros

- Sempre coloque o middleware de tratamento de erros por último.
- Trate os erros 404 separadamente dos outros erros.
- Em produção, registre os detalhes dos erros internamente — nunca envie rastros de pilha para os usuários.

## Depuração e registro no Express

- **Depuração embutida**: defina a variável de ambiente `DEBUG` para habilitar a saída de depuração embutida do Express.
  - `DEBUG=express:* node index.js` — habilita todas as mensagens de depuração do Express.
  - `DEBUG=express:router node index.js` — mostra apenas a saída de depuração relacionada ao roteador.
  - PowerShell: `$env:DEBUG = "express:*"; node index.js`
  - CMD: `set DEBUG=express:* && node index.js`
- **`morgan`**: um middleware popular para registrar requisições HTTP (método, URL, código de status, tempo de resposta).

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- **Depuração** rastreia como o Express lida com requisições e middleware sem poluir os logs de produção.
- **Registro** grava o que está acontecendo nas requisições — útil para identificar bugs e entender o tráfego.

## Verificações de saúde e desligamentos suaves

### Verificações de saúde

- Uma **rota de verificação de saúde** informa se o app está funcionando corretamente.
- Balanceadores de carga e orquestradores (por exemplo, Kubernetes, AWS Elastic Beanstalk) fazem ping nessa rota para decidir se o serviço está saudável.
- Uma resposta **200** significa saudável; a ausência de resposta aciona uma reinicialização automática.

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### Desligamento suave

- Um **desligamento suave** permite que o app finalize as requisições existentes antes de parar, em vez de cortar conexões imediatamente.
- Escute o sinal `SIGTERM` (enviado quando o processo deve terminar) e chame `server.close()`:

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- Escute o sinal `SIGINT` (enviado quando o desenvolvedor pressiona Ctrl+C durante o desenvolvimento local) e chame a mesma lógica de desligamento que `SIGTERM`.

### Marcando o app como não saudável antes do desligamento

- Retorne um **503** na verificação de saúde durante o desligamento para que o balanceador de carga pare de direcionar tráfego novo para o app.

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
