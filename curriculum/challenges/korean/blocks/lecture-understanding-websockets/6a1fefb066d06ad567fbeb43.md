# --description--

웹소켓은 클라이언트와 서버가 실시간으로 통신할 수 있게 합니다.

보통 웹과 모바일 애플리케이션 같은 클라이언트 측 애플리케이션은 HTTP를 사용합니다. HTTP에서는 클라이언트가 요청을 보내고 서버가 응답을 보냅니다. 이것을 요청과 응답 주기라고 배웠습니다.

연결이 설정되고 데이터가 교환된 후에는 연결이 닫힙니다. 페이지를 불러오거나 데이터를 가져오거나 게시하는 대부분의 경우에는 이 방식이 잘 작동합니다. 하지만 실시간 애플리케이션에는 느리고 신뢰성이 떨어집니다.

예를 들어, 채팅 앱을 생각해 보세요. HTTP를 사용한다면 클라이언트가 서버에 주기적으로 데이터를 요청해야 하며, 메시지마다 연결이 열리고 데이터가 도착한 후 연결이 닫히는 과정을 반복해야 합니다.

이 문제를 웹소켓이 해결합니다. 웹소켓은 클라이언트와 서버 사이에 영구적인 연결을 만듭니다. 연결이 열리면 어느 쪽에서든 닫기 신호를 보내기 전까지 실시간 채널에서 언제든 데이터를 교환할 수 있습니다.

웹소켓은 HTTP와 다르지만, 모든 웹소켓 연결은 핸드셰이크라고 하는 과정에서 HTTP로 시작합니다.

클라이언트가 연결을 웹소켓으로 업그레이드해 달라고 요청하면 서버가 동의하고 `101 Switching Protocols` 응답을 보내 연결을 웹소켓으로 업그레이드합니다. 그 후 연결은 열린 상태로 유지되며 실시간 통신이 시작됩니다.

흐름은 다음과 같습니다:

1. 클라이언트가 HTTP로 서버에 연결
2. 서버가 연결을 받아들이고 웹소켓으로 업그레이드
3. 연결이 열린 상태로 유지
4. 연결이 닫힐 때까지 양쪽이 언제든 데이터 교환

이 흐름은 HTTP처럼 요청-응답이 아니라 양방향입니다.

```md
Client <---> Server
```

HTTP에서는 보통 요청-응답 모델을 따릅니다:

```md
Client ---> Request
Server ---> Response
```

웹소켓과 HTTP의 차이를 설명하자면, HTTP는 편지를 주고받는 것 같고 웹소켓은 전화 통화와 같습니다. 편지를 쓰고 답장을 기다리면 대화가 끝나지만, 전화 통화는 실시간으로 대화합니다.

웹소켓은 채팅 앱, 온라인 멀티플레이어 게임, 실시간 스포츠 업데이트, 주식 시장 대시보드, Google Docs 같은 협업 도구 등 실시간 애플리케이션에 사용됩니다.
