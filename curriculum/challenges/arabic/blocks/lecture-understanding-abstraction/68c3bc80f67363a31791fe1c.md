# --description--

بعد أن استعرضنا التغليف والوراثة وتعدد الأشكال، دعنا نناقش المفهوم الرئيسي التالي في البرمجة كائنية التوجه – التجريد.

التجريد هو عملية إخفاء تفاصيل التنفيذ المعقدة وإظهار الميزات الأساسية فقط لكائن أو نظام. فكر فيه على أنه تركيز على ما يفعله الشيء بدلاً من كيفية فعله.

التجريد ليس مقتصرًا على Python فقط. إنه مفهوم برمجي يمكن تنفيذه في العديد من اللغات التي تدعم البرمجة كائنية التوجه.

لتوضيح التجريد، تخيل أنك تقود سيارة. عندما تكون في مقعد السائق، تتفاعل مع الأجزاء الأساسية مثل عجلة القيادة، وذراع التغيير، ودواسات الوقود والفرامل. لا تحتاج إلى معرفة التفاصيل الدقيقة لكيفية عمل المحرك، أو كيفية تغيير التروس في ناقل الحركة، أو الفيزياء وراء نظام الفرامل، إذ إن كل ذلك هو تفاصيل تنفيذ معقدة.

هذا هو التجريد يعمل! فهو يوفر لك واجهة مبسطة للتفاعل مع نظام معقد.

في حالة السيارة مرة أخرى، الواجهة المبسطة هي عجلة القيادة والفرامل ودواسة الوقود، بينما النظام المعقد هو السيارة نفسها.

أما كيف ينفذ Python التجريد، فهو يتم عبر وحدة `abc`.

تقدم هذه الوحدة الصنف `ABC` (الذي يعني “الصنف الأساسي المجرد”) وزخرفة `@abstractmethod`.

`ABC` هو الصنف الذي يُراد الوراثة منه، لكن لا يمكنك إنشاء كائنات مباشرة منه. هو الذي يحدد واجهة مشتركة من الطرق والخصائص التي يجب على الأصناف الفرعية تنفيذها.

من ناحية أخرى، الطريقة المجردة هي طريقة معلنة في صنف أساسي مجرد (ABC) باستخدام زخرفة `@abstractmethod`. قد لا تحتوي على تنفيذ أو تحتوي على تنفيذ افتراضي بسيط. ومع ذلك، يجب على أي صنف فرعي تجاوزها ليُعتبر ملموسًا وقابلًا للإنشاء، حتى لو وُجد تنفيذ افتراضي.

إليك الصيغة النحوية الأساسية للصنف المجرد في Python:

```py
from abc import ABC, abstractmethod

# Define an abstract base class
class AbstractClass(ABC):
    @abstractmethod
    def abstract_method(self):
        pass

# Concrete subclass that implements the abstract method
class ConcreteClassOne(AbstractClass):
    def abstract_method(self):
        print('Implementation in ConcreteClassOne')

# Another concrete subclass
class ConcreteClassTwo(AbstractClass):
    def abstract_method(self):
        print('Implementation in ConcreteClassTwo')
```

إليك مثالًا أساسيًا:

```py
from abc import ABC, abstractmethod

class Animal(ABC): # Inherits from abstract base class
   @abstractmethod # Abstract method decorator
   def make_sound(self):  # The method subclasses must override
       pass

# Concrete class that will override the abstract method
class Dog(Animal):
   def make_sound(self):
       print('Woof!')

# Another concrete class that will override the abstract method
class Cat(Animal):
   def make_sound(self):
       print('Meow!')

# Another concrete class that will override the abstract method
class Monkey(Animal):
   def make_sound(self):
       print('Ooh ooh aah aah!')

# Create instances of each concrete class
animals = [Dog(), Cat(), Monkey()]

# Loop through the instances to call the make_sound method
for animal in animals:
   animal.make_sound()

# Output:
# Woof!
# Meow!
# Ooh ooh aah aah!
```

في هذا المثال:

- نستورد الصنف `ABC` والطريقة `abstractmethod` من وحدة `abc`.
- ثم ننشئ صنف `Animal` يرث من `ABC`، ونعلن طريقة مجردة `make_sound` يجب على كل صنف فرعي من `Animal` تجاوزها.
- ننشئ الأصناف الملموسة `Dog` و`Cat` و`Monkey`، التي يجب أن تتجاوز الطريقة المجردة `make_sound`.
- ننشئ كائنات من الأصناف الملموسة وندعو طريقتهم `make_sound` لنُظهر كيف ينفذ كل منها الطريقة المجردة `make_sound` بطريقته الخاصة.

تذكر أنه لا يمكنك إنشاء نسخة من صنف Animal. هذا ما يحدث إذا حاولت ذلك:

```py
dog = Animal() 
# TypeError: Can't instantiate abstract class Animal 
# without an implementation for abstract method 'make_sound'
```

ينطبق نفس القاعدة على الأصناف الفرعية التي لا توفر تنفيذًا للطريقة المجردة. حتى لو عرّفت صنفًا فرعيًا من Animal، لا يمكنك إنشاء كائن منه حتى يتجاوز make_sound. إليك مثالًا يوضح ذلك:

```py
class Bird(Animal):
    pass

bird = Bird()
# TypeError: Can't instantiate abstract class Bird 
# without an implementation for abstract method 'make_sound'
```

إليك مثالًا آخر، هذه المرة مع خاصية مثيل يمكنك تمريرها إلى كائنات الطرق الملموسة:

```py
from abc import ABC, abstractmethod

# The blueprint for any toy that can speak
class TalkingToy(ABC):
   def __init__(self, name):
       self.name = name
   @abstractmethod
   def speak(self):
       pass

class RobotToy(TalkingToy):
   def speak(self):
       print(f'{self.name} says beep boop! I am a robot!')

class TeddyBearToy(TalkingToy):
   def speak(self):
       print(f"{self.name} says hug me! I'm cuddly!")

class DinosaurToy(TalkingToy):
   def speak(self):
       print(f'{self.name} says ROOOOAR!')

# Create toys
rusty = RobotToy('Rusty')
fluffy = TeddyBearToy('Fluffy')
rex = DinosaurToy('Rex')

toys = [rusty, fluffy, rex]
for toy in toys:
   toy.speak()

# Output:
# Rusty says beep boop! I am a robot!
# Fluffy says hug me! I'm cuddly!
# Rex says ROOOOAR!
```

في هذا المثال:

- لدينا صنف أساسي مجرد `TalkingToy` يحدد مخططًا لأي لعبة يمكنها التحدث.
- تنفذ الأصناف الفرعية `RobotToy` و`TeddyBearToy` و`DinosaurToy` الطريقة `speak` بطريقتها الخاصة.
- عند إنشاء كائنات من هذه الأصناف الفرعية واستدعاء الطريقة `speak`، تتحدث كل لعبة بطريقتها الفريدة.

في الختام، يبسط التجريد في Python الأنظمة المعقدة بزيادة إعادة الاستخدام.

لقد رأيت كيف يمكنك إعادة استخدام طريقة واحدة من صنف مجرد عبر عدة أصناف فرعية مع إجبار كل صنف فرعي على توفير سلوكه الخاص.

يحافظ هذا الأسلوب على تنظيم شفرتك ومرونتها وسهولة صيانتها، خاصة مع نمو تطبيقك.
