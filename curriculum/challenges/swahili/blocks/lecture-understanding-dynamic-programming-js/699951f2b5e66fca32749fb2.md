# --description--
  
Programu ya mabadiliko ya hali ni mbinu ya algoriti inayotatua matatizo magumu kwa kuyavunja kuwa matatizo madogo madogo na kuhifadhi matokeo ili kuepuka mahesabu yanayojirudia. Njia hii hubadilisha matatizo ambayo kawaida yangechukua muda wa mabadiliko ya mlinganyo kuwa matatizo yanayoweza kutatuliwa kwa muda wa polinomu.

## Kanuni kuu za programu ya mabadiliko ya hali

Programu ya mabadiliko ya hali hufanya kazi pale ambapo kuna masharti mawili muhimu katika tatizo.

- **Matatizo madogo yanayojirudia**: Matatizo madogo madogo yanayofanana hujitokeza mara nyingi wakati wa kutatua tatizo kubwa. Badala ya kuhesabu matatizo haya madogo mara kwa mara, tunahifadhi suluhisho lao.
    
- **Muundo bora wa tatizo**: Suluhisho bora la tatizo lina suluhisho bora za matatizo madogo madogo. Hii ina maana tunaweza kujenga suluhisho bora kwa kuunganisha suluhisho bora za sehemu ndogo ndogo.
    

Tuchunguze dhana hizi kwa kutumia tatizo la kawaida la "kupanda ngazi".

## Tatizo la kutumia rekursi ya kawaida

Fikiria tatizo la kupanda ngazi: unapanda ngazi yenye hatua `n` na unaweza kupanda hatua 1 au 2 kwa wakati mmoja. Kuna njia ngapi tofauti za kufikia kilele?

```js
function climbStairsRecursive(n) {
  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Recursive rule: to reach step n, you can come from step (n-1) or step (n-2)
  return climbStairsRecursive(n - 1) + climbStairsRecursive(n - 2);
}
```

Utekelezaji huu una ugumu wa muda wa mabadiliko ya mlinganyo kwa sababu ya mahesabu mengi yanayojirudia. Wakati wa kuhesabu `climbStairsRecursive(5)`, yafuatayo hutokea:

- `climbStairsRecursive(5)` inaita `climbStairsRecursive(4)` na `climbStairsRecursive(3)`
    
- `climbStairsRecursive(4)` inaita `climbStairsRecursive(3)` na `climbStairsRecursive(2)`
    
- Sasa `climbStairsRecursive(3)` inahesabiwa **mara mbili**
    
- `climbStairsRecursive(3)` inaita `climbStairsRecursive(2)` na `climbStairsRecursive(1)`
    
- `climbStairsRecursive(2)` inahesabiwa **mara 3 jumla**
    

Kwa `n=5` pekee, tunafanya wito 9 za kitendakazi wakati tunahitaji mahesabu 5 tu ya kipekee. Kadiri `n` inavyoongezeka, kurudia hii kunazidi kwa mlinganyo - `climbStairsRecursive(30)` ingehitaji wito wa kitendakazi milioni kadhaa! Ugumu wa muda unakuwa `O(2^n)`, na kufanya kuwa hauna ufanisi na hauwezi kutumika kwa thamani kubwa za `n`.

## Suluhisho za programu ya mabadiliko ya hali

Programu ya mabadiliko ya hali huondoa mahesabu yanayojirudia kupitia mbinu kuu mbili:

### Kumbukumbu (Mbinu ya juu-chini)

Kumbukumbu huhifadhi matokeo ya wito za kitendakazi ghali na hurudisha matokeo yaliyohifadhiwa wakati ingizo sawa linapotokea tena:

```js
function climbStairsMemo(n, memo = {}) {
  // If already calculated, return cached value
  if (memo[n] !== undefined) {
    return memo[n];
  }

  // Base cases: 1 way for 1 step, 2 ways for 2 steps
  if (n <= 2) {
    return n;
  }

  // Calculate once and store in memo for future use
  memo[n] =
    climbStairsMemo(n - 1, memo) +
    climbStairsMemo(n - 2, memo);

  return memo[n];
}
```

Kumbukumbu ni ya ufanisi zaidi kwa sababu kila thamani ya kipekee kutoka `1` hadi `n` huhesabiwa mara moja tu. Tunapohitaji `climbStairsMemo(3)` tena, badala ya kuhesabu tena (ambayo ingehamasisha wito zaidi wa rekursi), tunatafuta tu katika kamusi yetu ya kumbukumbu kwa muda wa `O(1)`.

Tufuate utekelezaji wa `climbStairsMemo(5)` kwa mbinu ya juu-chini kuona jinsi kumbukumbu inavyopunguza kazi inayojirudia:

```md
Call: climbStairsMemo(5)
  memo = {} (empty)
  
  Call: climbStairsMemo(4) 
    memo = {} (empty)
    
    Call: climbStairsMemo(3)
      memo = {} (empty)
      
      Call: climbStairsMemo(2) → returns 2 (base case)
      Call: climbStairsMemo(1) → returns 1 (base case)
      
      Result: 2 + 1 = 3
      memo = {3: 3} (stored!)
    
    Call: climbStairsMemo(2) → returns 2 (base case)
    
    Result: 3 + 2 = 5
    memo = {3: 3, 4: 5} (stored!)
  
  Call: climbStairsMemo(3) → returns 3 (FROM MEMO - no recursion!)
  
  Result: 5 + 3 = 8
  memo = {3: 3, 4: 5, 5: 8}
```

**Ulinganisho wa ufanisi**

- **Rekursi ya kawaida**: Inafanya wito 9 za kitendakazi na mahesabu yanayojirudia
    
- **Kumbukumbu**: Inafanya mahesabu 5 ya kipekee tu, kisha inatumia tena matokeo yaliyohifadhiwa
    
- **Ugumu wa muda**: Umepunguzwa kutoka `O(2^n)` hadi `O(n)` kwa sababu tunafanya mahesabu `n` ya kipekee tu
    
- **Ugumu wa nafasi**: `O(n)` kwa ajili ya hifadhi ya kumbukumbu na stack ya wito
    
- **Athari halisi**: `climbStairsMemo(30)` hupungua kutoka kwa wito wa kitendakazi milioni nyingi hadi mahesabu 30 ya kipekee tu!
    

### Jedwali (Mbinu ya chini-juu)

Jedwali hujenga suluhisho kutoka chini kwenda juu, likijaza jedwali na suluhisho za matatizo madogo:

```js
function climbStairsTabulation(n) {
  if (n <= 2) {
    return n;
  }

  // Create an array to store solutions from 0 to n
  const dp = new Array(n + 1).fill(0);

  // Base cases
  dp[1] = 1; // 1 way to reach step 1
  dp[2] = 2; // 2 ways to reach step 2

  // Build up the solution iteratively
  for (let i = 3; i <= n; i++) {
    // Ways to reach step i = ways to reach (i-1) + ways to reach (i-2)
    dp[i] = dp[i - 1] + dp[i - 2];
  }

  return dp[n];
}
```

Jedwali huondoa kabisa rekursi kwa kujenga suluhisho kwa njia ya mfululizo kutoka matatizo madogo kabisa hadi lengo.

Tuwekeze mbinu ya chini-juu kwa vitendo kuona jinsi tunavyojenga suluhisho kwa mpangilio. Huu ni ujenzi wa mfululizo kwa `climbStairsTabulation(5)`:

```md
Initial state:
dp = [0, 1, 2, 0, 0, 0]
     [0, 1, 2, 3, 4, 5] ← indices (step numbers)

Step by step construction:

i = 3:
  dp[3] = dp[2] + dp[1] = 2 + 1 = 3
  dp = [0, 1, 2, 3, 0, 0]
  
i = 4:
  dp[4] = dp[3] + dp[2] = 3 + 2 = 5
  dp = [0, 1, 2, 3, 5, 0]
  
i = 5:
  dp[5] = dp[4] + dp[3] = 5 + 3 = 8
  dp = [0, 1, 2, 3, 5, 8]

Final result: dp[5] = 8
```

**Faida kuu za jedwali**

- **Hakuna mzigo wa rekursi**: Tofauti na kumbukumbu, hakuna stack ya wito za rekursi.
    
- **Utekelezaji unaotarajiwa**: Tunahesabu thamani kwa mpangilio uliowekwa (1, 2, 3, 4, 5...).
    
- **Rafiki kwa cache**: Upatikanaji wa safu ya data kwa mfululizo huongeza matumizi ya kumbukumbu.
    
- **Rahisi kuboresha**: Inaweza kupunguza ugumu wa nafasi hadi `O(1)` kwa sababu tunahitaji thamani mbili za mwisho tu.
    

```js
function climbStairsOptimized(n) {
  if (n <= 2) {
    return n;
  }

  // Store only the last two results to save space
  let prev2 = 1; // ways to reach step 1
  let prev1 = 2; // ways to reach step 2

  for (let i = 3; i <= n; i++) {
    const current = prev1 + prev2;

    // Shift values forward
    prev2 = prev1;
    prev1 = current;
  }

  return prev1;
}
```

**Ulinganisho wa ufanisi**

- **Rekursi ya kawaida**: Wito 9 za kitendakazi kwa `n=5`, ukuaji wa mabadiliko ya mlinganyo.
    
- **Jedwali**: Ongeza 3 rahisi kwa `n=5`, ukuaji wa mstari.
    
- **Ugumu wa muda**: `O(n)` badala ya `O(2^n)`.
    
- **Ugumu wa nafasi**: `O(n)` kwa safu ya data, au `O(1)` kwa uboreshaji.
    
- **Utendaji unaotarajiwa**: Hakuna hatari ya kuzidi stack kwa ingizo kubwa.
    

Mbinu zote mbili hupunguza ugumu wa muda kutoka `O(2^n)` wa mabadiliko ya mlinganyo hadi `O(n)` wa mstari, kuboresha kwa kiasi kikubwa kinachofanya tofauti kati ya kutatua tatizo kwa milisekunde dhidi ya kusubiri miaka kwa ingizo kubwa.

## Mfano wa vitendo: Tatizo la kubadilisha sarafu

Tatizo la kubadilisha sarafu ni changamoto ya kawaida ya kuandika programu ambayo, inapotatuliwa kwa kutumia programu ya mabadiliko ya hali, inaonyesha kanuni kuu mbili za DP: muundo bora wa tatizo na matatizo madogo yanayojirudia.

Tatizo la kubadilisha sarafu linauliza, "Ni sarafu ngapi za kiwango cha chini zinazohitajika kutengeneza kiasi kinacholengwa?"

Hapa kuna suluhisho moja kwa kutumia programu ya mabadiliko ya hali:

```js
function minCoins(amount, coins) {
  // Create dp array from 0 to amount, filled with Infinity
  const dp = new Array(amount + 1).fill(Infinity);

  // Base case: 0 coins needed to make amount 0
  dp[0] = 0;

  // Build up from 1 to target amount
  for (let i = 1; i <= amount; i++) {
    // Try each coin denomination
    for (let coin of coins) {
      // Only use coin if it is not larger than current amount
      if (coin <= i) {
        // Choose the minimum: current value OR solution for remaining amount + 1 coin
        dp[i] = Math.min(dp[i], dp[i - coin] + 1);
      }
    }
  }

  // Return result if possible, -1 if impossible (infinity)
  return dp[amount] === Infinity ? -1 : dp[amount];
}

/*
 Example usage:
coins = [1, 3, 4], amount = 6

dp[0] = 0
dp[1] = dp[0] + 1 = 1
dp[2] = dp[1] + 1 = 2
dp[3] = min(dp[2]+1, dp[0]+1) = min(2+1, 0+1) = 1
dp[4] = min(dp[3]+1, dp[1]+1, dp[0]+1) = min(1+1, 1+1, 0+1) = 1
dp[5] = min(dp[4]+1, dp[2]+1, dp[1]+1) = min(1+1, 2+1, 1+1) = 2
dp[6] = min(dp[5]+1, dp[3]+1, dp[2]+1) = min(2+1, 1+1, 2+1) = 2

Result: 2 coins (3 + 3)
*/
```

Na hapa ni jinsi algoriti ya kubadilisha sarafu ya programu ya mabadiliko ya hali inavyofanya kazi hatua kwa hatua kwa `coins = [1, 3, 4]`, `amount = 6`:

```md
Initial state:
dp = [0, ∞, ∞, ∞, ∞, ∞, ∞]
     [0, 1, 2, 3, 4, 5, 6] ← amounts

Building up the solution:

For amount = 1:
  Try coin 1: dp[1] = min(∞, dp[0] + 1) = min(∞, 0 + 1) = 1
  dp = [0, 1, ∞, ∞, ∞, ∞, ∞]

For amount = 2:
  Try coin 1: dp[2] = min(∞, dp[1] + 1) = min(∞, 1 + 1) = 2
  dp = [0, 1, 2, ∞, ∞, ∞, ∞]

For amount = 3:
  Try coin 1: dp[3] = min(∞, dp[2] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[3] = min(3, dp[0] + 1) = min(3, 0 + 1) = 1
  dp = [0, 1, 2, 1, ∞, ∞, ∞]

For amount = 4:
  Try coin 1: dp[4] = min(∞, dp[3] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[4] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  Try coin 4: dp[4] = min(2, dp[0] + 1) = min(2, 0 + 1) = 1
  dp = [0, 1, 2, 1, 1, ∞, ∞]

For amount = 5:
  Try coin 1: dp[5] = min(∞, dp[4] + 1) = min(∞, 1 + 1) = 2
  Try coin 3: dp[5] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  Try coin 4: dp[5] = min(2, dp[1] + 1) = min(2, 1 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, ∞]

For amount = 6:
  Try coin 1: dp[6] = min(∞, dp[5] + 1) = min(∞, 2 + 1) = 3
  Try coin 3: dp[6] = min(3, dp[3] + 1) = min(3, 1 + 1) = 2
  Try coin 4: dp[6] = min(2, dp[2] + 1) = min(2, 2 + 1) = 2
  dp = [0, 1, 2, 1, 1, 2, 2]

Final result: dp[6] = 2 (achieved with coins 3 + 3)
```

Suluhisho hili linaonyesha kanuni kuu mbili za programu ya mabadiliko ya hali. Lina matatizo madogo yanayojirudia kwa sababu kupata sarafu chache kwa kiasi 6 kunahitaji kujua suluhisho za kiasi 5, 3, na 2. Matatizo madogo haya yanajitokeza pia wakati wa kuhesabu kiasi kingine. Lina muundo bora wa tatizo kwa sababu suluhisho bora kwa kiasi chochote linajumuisha suluhisho bora za kiasi kidogo. Ikiwa tunajua sarafu chache kwa kiasi 3 ni 1, basi njia moja ya kutengeneza kiasi 6 ni kutumia suluhisho hilo pamoja na sarafu moja ya thamani 3.

Bila DP, tungehitaji kujaribu mchanganyiko wote wa sarafu - idadi ya mabadiliko ya mlinganyo. Kwa DP, tunajenga suluhisho kwa mpangilio:

- **Ugumu wa muda**: `O(amount × number of coins)` badala ya mabadiliko ya mlinganyo.
    
- **Ugumu wa nafasi**: `O(amount)` kwa safu ya `dp`.
    
- **Hakuna kazi inayojirudia**: Kila tatizo dogo (kupata sarafu chache kwa kila kiasi) linatatuliwa mara moja tu.
    
- **Matokeo yanayoweza kutumika tena**: Mara tunapojua sarafu chache kwa kiasi 3, tunatumia maarifa haya kwa kiasi kikubwa kinachoweza kufaidika.
    

## Lini utumie programu ya mabadiliko ya hali

Programu ya mabadiliko ya hali ni yenye ufanisi pale:

- Tatizo linaweza kuvunjwa kuwa matatizo madogo yanayojirudia.
    
- Tatizo lina muundo bora wa tatizo.
    
- Suluhisho la rekursi la kawaida lingehitaji mahesabu yanayojirudia.
    
- Unahitaji kuboresha ugumu wa muda kwa gharama ya ugumu wa nafasi.
    

Mifumo ya kawaida ya programu ya mabadiliko ya hali ni pamoja na matatizo ya uboreshaji (kupata thamani ya chini/kubwa), matatizo ya kuhesabu (idadi ya njia za kufanikisha jambo), na matatizo ya maamuzi yanayoweza kuvunjwa kuwa maamuzi madogo.

Programu ya mabadiliko ya hali hubadilisha matatizo magumu kuwa matatizo yanayoweza kudhibitiwa kwa kuhifadhi na kutumia tena suluhisho za matatizo madogo. Kuelewa mbinu hii hufungua mlango wa kutatua changamoto nyingi za kihesabu kwa ufanisi.
