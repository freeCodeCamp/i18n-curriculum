## JavaScript 라이브러리와 프레임워크

- JavaScript 라이브러리와 프레임워크는 자주 발생하는 문제에 대한 빠른 해결책을 제공하고, 미리 작성된 코드를 통해 개발 속도를 높입니다.
- 라이브러리는 일반적으로 DOM 조작, 이벤트 처리, AJAX 요청 관리와 같은 특정 작업에 대한 해결책 제공에 더 집중합니다.
- JavaScript 라이브러리의 예로는 jQuery와 React가 있습니다.
- 반면 프레임워크는 애플리케이션을 구축하기 위한 보다 명확한 구조를 제공합니다. 개발자가 따라야 하는 규칙과 관례 세트를 포함하는 경우가 많습니다.
- 프레임워크의 예로는 Angular와 React용 메타 프레임워크인 Next.js가 있습니다.
- **싱글 페이지 애플리케이션**(SPA)은 하나의 HTML 페이지를 로드하고, 사용자가 애플리케이션과 상호작용할 때 전체 페이지를 다시 로드하지 않고 해당 페이지를 동적으로 업데이트하는 웹 애플리케이션입니다.
- SPA는 애플리케이션 상태를 관리하고 콘텐츠를 렌더링하기 위해 JavaScript를 사용합니다. 이는 복잡한 사용자 인터페이스를 구축하는 데 훌륭한 도구를 제공하는 프레임워크를 사용해 이루어지는 경우가 많습니다.
- SPA와 관련된 몇 가지 문제는 다음과 같습니다:

  - 화면 읽기 프로그램이 동적으로 업데이트되는 콘텐츠를 처리하는 데 어려움을 겪습니다.
  - 사용자가 애플리케이션 내에서 이동해도 URL이 변경되지 않아 특정 페이지를 북마크하거나 뒤로 가기, 공유하기가 어렵습니다.
  - 애플리케이션이 크면 모든 자산을 처음에 로드해야 하므로 초기 로드 시간이 느릴 수 있습니다.

## React

- React는 사용자 인터페이스와 웹 애플리케이션을 구축하기 위한 인기 있는 JavaScript 라이브러리입니다.
- React의 핵심 개념은 데이터가 변경될 때 독립적으로 업데이트되고 렌더링할 수 있는 재사용 가능한 UI 컴포넌트를 만드는 것입니다.
- React는 애플리케이션 상태에 따라 UI가 어떻게 보여야 하는지 설명할 수 있게 합니다. 그런 다음 데이터나 상태가 변경되면 React가 적절한 컴포넌트를 업데이트하고 렌더링합니다.

## React 컴포넌트

- 컴포넌트는 React 애플리케이션의 기본 구성 요소로, 개발자가 복잡한 사용자 인터페이스를 더 작고 관리하기 쉬운 조각으로 나눌 수 있게 합니다.
- UI는 JSX를 사용해 설명하는데, JSX는 JavaScript 구문의 확장으로, 개발자가 JavaScript 내에서 HTML과 유사한 코드를 작성할 수 있게 합니다.
- 컴포넌트는 기본적으로 UI 조각을 반환하는 JS 함수나 클래스입니다.

다음은 인사 메시지를 렌더링하는 간단한 React 컴포넌트 예시입니다:

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

컴포넌트를 사용하려면, 다음과 같이 호출하세요:
    
```jsx
  <Greeting />
```

## React 컴포넌트 가져오기와 내보내기

- React 컴포넌트는 한 파일에서 내보내고 다른 파일에서 가져올 수 있습니다.
- `City`라는 파일에 `City.js`라는 컴포넌트가 있다고 가정해 보세요. `export` 키워드를 사용해 컴포넌트를 내보낼 수 있습니다:

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default City;
```

- 다른 파일에서 `City` 컴포넌트를 가져오려면 `import` 키워드를 사용하세요:

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

- `default` 키워드는 `City.js` 파일에서 기본 내보내기이기 때문에 사용합니다.

- 컴포넌트 정의와 같은 줄에서 컴포넌트를 내보내도록 선택할 수도 있습니다:

```jsx
export default function City() {
  return <p>New York</p>;
}
```

## Vite를 사용해 React 프로젝트 설정하기

- 프로젝트 설정 도구와 명령줄 인터페이스는 새 프로젝트를 빠르고 쉽게 시작할 수 있게 해주어, 개발자가 설정 작업 대신 코드 작성에 집중할 수 있도록 도와줍니다.  
- Vite는 인기 있는 프로젝트 설정 도구이며 React와 함께 사용할 수 있습니다.  
- Vite로 새 프로젝트를 만들려면 터미널에서 다음 명령어를 사용하세요:

```bash
npm create vite@latest my-react-app -- --template react
```

이 명령어는 Vite의 React 템플릿을 사용해 `my-react-app`라는 이름의 새 React 프로젝트를 만듭니다. 프로젝트 디렉터리에는 프로젝트 의존성과 명령어가 나열된 `package.json` 파일이 있습니다.

- 프로젝트를 실행하려면 프로젝트 디렉터리로 이동한 후 다음 명령어를 실행하세요:

```bash
cd my-react-app # path to the project directory
npm install # installs the dependencies listed in the package.json file
```

- 의존성이 설치되면 프로젝트 내에 `node_modules`라는 새 폴더가 생긴 것을 확인할 수 있습니다.

- `node_modules` 폴더는 프로젝트에 필요한 모든 패키지와 라이브러리가 저장되는 곳입니다.

- 프로젝트를 실행하려면 다음 명령어를 사용하세요:

```bash
npm run dev
```

- 그다음 브라우저를 열고 `http://localhost:5173`로 이동하면 React 애플리케이션이 실행되는 것을 볼 수 있습니다.

- 시작 템플릿의 코드를 실제로 보려면 프로젝트 내 `src` 폴더로 들어가면 `App.jsx` 파일을 확인할 수 있습니다.

## React 컴포넌트에서 프로프스 전달하기

- React에서 프로프스(속성의 줄임말)는 부모 컴포넌트에서 자식 컴포넌트로 데이터를 전달하는 방법입니다. 이 메커니즘은 재사용 가능하고 동적인 UI 요소를 만들 때 필요합니다.
- 프로프스는 어떤 JavaScript 값도 될 수 있습니다. 부모에서 자식 컴포넌트로 프로프스를 전달하려면, 부모의 JSX에서 자식 컴포넌트를 사용할 때 프로프스를 속성으로 추가하세요. 간단한 예는 다음과 같습니다:

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

- 전개 연산자 `(...)`를 사용해 여러 프로프스를 객체로 변환한 후 전달할 수 있습니다. 예시는 다음과 같습니다:

```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

- 이 코드에서 전개 연산자 `{...person}`는 person 객체를 개별 프로프스로 변환해 Child 컴포넌트에 전달합니다.

## React에서 조건부 렌더링

- React에서 조건부 렌더링은 동적인 사용자 인터페이스를 만들 수 있게 해줍니다. 애플리케이션 내 특정 조건이나 상태에 따라 다른 내용을 보여줄 때 사용합니다.
- React에서 조건부로 내용을 렌더링하는 방법은 여러 가지가 있습니다. 흔히 사용하는 방법 중 하나는 삼항 연산자를 사용하는 것입니다. 예시는 다음과 같습니다:

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- 조건부 렌더링을 하는 또 다른 방법은 논리 AND (`&&`) 연산자를 사용하는 것입니다. 특정 조건이 충족될 때만 내용을 렌더링하고 싶을 때 유용합니다. 예시는 다음과 같습니다:

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

위 코드에서 `h1` 요소는 user 객체가 참일 때만 렌더링됩니다.

직접 `if` 문을 이렇게 사용할 수도 있습니다:

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```

## React에서 리스트 렌더링하기

- React에서 리스트를 렌더링하는 것은 사용자 인터페이스를 만들 때 흔히 하는 작업입니다.  
- 리스트는 JS 배열 `map()` 메서드를 사용해 배열의 항목을 순회하며 새로운 JSX 요소 배열을 반환하는 방식으로 렌더링할 수 있습니다.  
- 예를 들어, 리스트로 렌더링하려는 이름 배열이 있다면 다음과 같이 할 수 있습니다:  

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- 각 리스트 항목에 고유한 키를 반드시 제공해 React가 업데이트와 렌더링 역할을 효율적으로 관리하도록 하세요. 이러한 기법을 사용하면 React 애플리케이션에서 유연하고 효율적이며 동적인 리스트를 만들 수 있습니다.

## React에서 인라인 스타일

- React에서 인라인 스타일은 JavaScript 객체를 사용해 CSS 스타일을 JSX 요소에 직접 적용할 수 있게 합니다.
- React에서 인라인 스타일을 적용하려면 JSX 요소에 style 속성을 사용하세요. style 속성은 키가 카멜 케이스로 된 CSS 속성이고 값이 해당 속성 값인 객체를 받습니다. 예시는 다음과 같습니다:

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

- 스타일을 별도의 객체로 분리한 후 `style` 속성에서 참조할 수도 있습니다:

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- 인라인 스타일은 프로프스나 상태에 따라 조건부로 스타일을 적용할 수 있어 동적 스타일링을 지원합니다. 프로프에 따라 조건부로 스타일을 적용하는 예시는 다음과 같습니다:

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- 위 코드에서 `color`과 `fontSize` 스타일은 `isImportant` 프로프에 따라 조건부로 설정됩니다.

## React에서 이벤트 다루기

- **합성 이벤트 시스템**: React가 이벤트를 처리하는 방식입니다. `click`, `keydown`, `submit` 같은 네이티브 이벤트를 감싸는 래퍼 역할을 합니다. React의 이벤트 핸들러는 카멜 케이스 명명 관례를 사용합니다. (예: `onClick`, `onSubmit` 등)

다음은 React에서 `onClick` 요소에 `button` 속성을 사용하는 예입니다:

```jsx
function handleClick() {
  console.log("Button clicked!");
}

<button onClick={handleClick}>Click Me</button>;
```

React에서 이벤트 핸들러 함수는 보통 `handle` 접두사로 시작하여 이벤트를 처리하는 함수임을 나타냅니다. 예를 들어 `handleClick`나 `handleSubmit` 같은 이름입니다.

사용자 동작으로 이벤트가 발생하면 React는 합성 이벤트 객체를 핸들러에 전달합니다. 이 객체는 일반 JavaScript의 네이티브 Event 객체처럼 동작하며 `type`, `target`, `currentTarget` 같은 속성을 제공합니다.

예를 들어 `onSubmit` 이벤트 중 브라우저 새로 고침 같은 기본 동작을 막으려면 `preventDefault()` 메서드를 호출하세요:

```jsx
function handleSubmit(event) {
  event.preventDefault();
  console.log("Form submitted!");
}

<form onSubmit={handleSubmit}>
  <input type="text" />
  <button>Submit</button>
</form>;
```

또한 핸들러 함수를 화살표 함수로 감쌀 수도 있습니다:

```jsx
function handleDelete(id) {
  console.log("Deleting item:", id);
}

<button onClick={() => handleDelete(1)}>Delete Item</button>;
```

## 상태와 `useState` 훅 다루기

- **상태 정의**: React에서 상태는 컴포넌트의 데이터를 담고 있는 객체입니다. 상태가 업데이트되면 컴포넌트가 다시 렌더링됩니다. React는 상태를 불변으로 취급하므로 직접 수정하지 않아야 합니다.  
- **`useState()` 훅**: `useState` 훅은 함수형 컴포넌트에서 상태 변수를 선언할 수 있게 해주는 함수입니다. 기본 구문은 다음과 같습니다:

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

상태 변수에는 다음이 포함됩니다:

- `stateVariable`는 현재 상태 값을 담고 있습니다
- `setStateFunction` (설정 함수)은 상태 변수를 업데이트합니다
- `initialValue`는 초기 상태를 설정합니다

다음은 `Counter` 컴포넌트의 완전한 예제입니다:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>

      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```

## 렌더링과 React 컴포넌트

- **정의**: React에서 렌더링은 컴포넌트가 사용자 인터페이스(UI), 보통은 브라우저에 나타나는 과정입니다. 렌더링 과정은 트리거, 렌더, 커밋의 세 단계로 구성됩니다.

트리거 단계는 React가 무언가가 변경되어 사용자 인터페이스(UI)를 업데이트해야 할 수도 있다고 감지할 때 발생합니다. 이 변경은 보통 상태(state)나 프로프스(props)의 업데이트 때문입니다.

트리거가 발생하면 React는 렌더 단계로 들어갑니다. 여기서 React는 컴포넌트를 다시 평가하고 무엇을 표시할지 결정합니다. 이를 위해 React는 "실제" DOM의 가벼운 복사본인 가상 DOM을 사용합니다. 가상 DOM을 통해 React는 컴포넌트에서 변경해야 할 부분을 빠르게 확인할 수 있습니다.

커밋 단계는 React가 가상 DOM에서 준비한 변경 사항을 실제 DOM에 적용하는 단계입니다. 다시 말해, 화면에서 최종 결과를 볼 수 있는 단계입니다.

## 상태에서 객체와 배열 업데이트하기

- **상태에서 객체 업데이트하기**: 상태에 있는 객체를 업데이트해야 할 때는 먼저 새 객체를 만들거나 기존 객체를 복사한 다음, 그 새 객체로 상태를 설정해야 합니다. 상태에 넣는 모든 객체는 읽기 전용으로 간주해야 합니다. 다음은 사용자의 이름, 나이, 도시를 설정하는 예시입니다. `handleChange` 함수는 사용자의 정보를 업데이트하는 데 사용됩니다:

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **상태에서 배열 업데이트하기**: 상태에 있는 배열을 업데이트할 때는 `push()`나 `pop()` 같은 메서드를 사용해 배열을 직접 수정하지 않는 것이 중요합니다. 대신 상태를 업데이트할 때는 새 배열을 만들어야 합니다:

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

배열에서 항목을 제거하려면, 제거할 항목을 필터링한 후 새 배열을 반환하는 `filter()` 메서드를 사용하세요:

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```

## Refs를 사용해 값 참조하기

- **`ref` 속성**: React에서 `ref` 속성을 사용해 DOM 노드에 접근할 수 있습니다. 다음은 `ref`를 사용해 `input` 요소에 포커스를 맞추는 예시입니다. `current` 속성은 해당 `ref`의 현재 값을 접근하는 데 사용됩니다:

```jsx
import { useRef } from "react";

const Focus = () => {
  const inputRef = useRef(null);

  const handleFocus = () => {
    if (inputRef.current) {
      inputRef.current.focus();
    }
  };

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Enter text" />
      <button onClick={handleFocus}>Focus Input</button>
    </div>
  );
};

export default Focus;
```

## `useEffect` 훅 사용하기

- **`useEffect()` 훅**: React에서 이펙트는 컴포넌트 렌더링 과정 외부에서 발생하는 모든 일을 뜻합니다. 즉, React가 UI 렌더링의 일부로 직접 처리하지 않는 모든 작업입니다. 흔한 예로는 데이터 가져오기, 브라우저 탭 제목 업데이트, 브라우저 로컬 스토리지 읽기 또는 쓰기, 사용자 위치 가져오기 등이 있습니다. 이러한 작업은 외부 세계와 상호작용하며 부수 효과라고 합니다. React는 이러한 부수 효과를 처리할 수 있도록 `useEffect` 훅을 제공합니다. `useEffect`는 컴포넌트가 렌더링되거나 업데이트된 후 함수를 실행하게 해줍니다.

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [dependencies]);
```

이펙트 함수는 컴포넌트가 렌더링된 후 실행되며, 선택적인 `dependencies` 인수가 이펙트 실행 시점을 제어합니다.

`dependencies`는 "반응 값"(state, props, 함수, 변수 등)의 배열일 수도 있고, 빈 배열일 수도 있으며, 완전히 생략할 수도 있습니다. 다음은 이 모든 옵션이 `useEffect` 작동 방식을 어떻게 제어하는지 설명합니다:

- `dependencies`가 하나 이상의 반응 값을 포함하는 배열이라면, 해당 값들이 변경될 때마다 이펙트가 실행됩니다.

- `dependencies`가 빈 배열이라면, `useEffect`는 컴포넌트가 처음 렌더링될 때 한 번만 실행됩니다.

- `dependencies`를 생략하면, 컴포넌트가 렌더링되거나 업데이트될 때마다 이펙트가 실행됩니다.

## 커스텀 훅 만드는 방법

- **커스텀 훅**: 커스텀 훅은 데이터 가져오기, 상태 관리, 토글, 온라인 상태 추적 같은 부수 효과 등 컴포넌트에서 재사용 가능한 로직을 추출할 수 있게 해줍니다. React에서는 모든 내장 훅이 `use`로 시작하므로, 커스텀 훅도 같은 관례를 따라야 합니다.

다음은 `useDebounce` 훅을 만드는 예시입니다:

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```

## React에서 폼 다루기

- **제어된 입력**: 입력 필드 값을 상태에 저장하고 `onChange` 이벤트를 통해 업데이트하는 경우입니다. 이렇게 하면 폼 데이터를 완전히 제어할 수 있고 즉각적인 유효성 검사와 조건부 렌더링이 가능합니다.

```jsx
import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **비제어 입력**: `useState` 훅을 통해 입력을 처리하는 대신, HTML의 비제어 입력은 DOM의 도움으로 자체 내부 상태를 유지합니다. DOM이 입력 값을 제어하기 때문에 `ref`을 사용해 입력 필드 값을 가져와야 합니다.

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```

## `useActionState` 훅 사용하기

- **서버 액션**: 서버에서 실행되는 함수로, API 엔드포인트 없이도 서버에서 바로 폼 처리를 할 수 있게 합니다. 다음은 Next.js 애플리케이션의 예시입니다:

```js
"use server";

async function submitForm(formData) {
 const name = formData.get("name");
 return { message: `Hello, ${name}!` };
}
```

`"use server"` 지시어는 해당 함수를 서버 액션으로 표시합니다.

- **`useActionState` 훅**: 이 훅은 폼 제출 결과에 따라 상태를 업데이트합니다. `useActionState` 훅의 기본 구문은 다음과 같습니다:

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state`는 액션이 반환하는 현재 상태입니다.
- `action`는 서버 액션을 실행하는 함수입니다.
- `isPending`는 액션이 현재 실행 중인지 여부를 나타내는 불리언입니다.
- `actionFunction` 매개변수는 서버 액션 자체입니다.
- `initialState`는 액션 실행 전 상태의 시작점을 나타내는 매개변수입니다.
- `permalink`는 폼이 수정하는 고유한 페이지 URL을 담은 선택적 문자열입니다.

## React에서 데이터 가져오기

- **데이터를 가져오는 방법**: React에서 데이터를 가져오는 방법은 여러 가지가 있습니다. 기본 Fetch API를 사용하거나 Axios, SWR 같은 서드파티 도구를 사용할 수 있습니다.  
- **데이터를 가져올 때 자주 사용하는 상태 변수**: React에서 어떤 방법으로 데이터를 가져오든, 관리해야 할 상태가 몇 가지 있습니다. 첫 번째는 데이터 자체입니다. 두 번째는 데이터가 아직 가져오는 중인지 추적하는 상태입니다. 세 번째는 데이터 가져오기 과정에서 발생할 수 있는 오류를 저장하는 상태 변수입니다.

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

데이터 가져오기는 부수 효과이므로, `Fetch API`를 `useEffect` 훅 안에서 사용하는 것이 가장 좋습니다.

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

그다음 데이터 가져오기가 완료되지 않았으면 로딩 메시지를, 오류가 발생했으면 오류 메시지를, 아니면 결과를 렌더링할 수 있습니다.

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

Axios를 사용하려면 설치하고 가져와야 합니다:

```bash
npm i axios
```

```js
import axios from "axios";
```

그다음 `axios.get`를 사용해 데이터를 가져올 수 있습니다:

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);

useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await axios.get(
        "https://jsonplaceholder.typicode.com/users"
      );
      setData(res.data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

`useSWR` 훅을 사용해 데이터를 가져오려면 먼저 설치하고 가져와야 합니다.

```bash
npm i swr
```

```js
import useSWR from "swr";
```

훅을 사용해 데이터를 가져오는 방법은 다음과 같습니다:

```jsx
import useSWR from "swr";

const fetcher = (url) => fetch(url).then((res) => res.json());

const FetchTodos = () => {
 const { data, error } = useSWR(
   "https://jsonplaceholder.typicode.com/todos",

 );

 if (!data) {

 }
 if (error) {
   return <h2>Error: {error.message}</h2>;
 }

 return (
   <>
     <h2>Todos</h2>
     <div>
       {data.map((todo) => (
         <h3 key={todo.id}>{todo.title}</h3>
       ))}
     </div>
   </>
 );
};

export default FetchTodos;
```

## `useOptimistic` 훅 사용하기

- **`useOptimistic` 훅**: 이 훅은 비동기 작업이 백그라운드에서 완료될 때까지 UI가 반응형을 유지하도록 도와줍니다. 서버 응답을 기다리는 것처럼 작업의 예상 결과를 기반으로 UI에 즉각적인 업데이트를 제공하는 "낙관적 업데이트" 전략을 관리하는 데 유용합니다.

기본 구문은 다음과 같습니다:

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState`은 더 나은 사용자 경험을 위해 즉시 업데이트되는 임시 상태입니다.
- `addOptimistic`은 실제 상태가 변경되기 전에 낙관적 업데이트를 적용하는 함수입니다.
- `actualState`은 서버에서 데이터를 가져오는 것과 같은 작업 결과로부터 오는 실제 상태 값입니다.
- `updateFunction`은 호출될 때 낙관적 상태가 어떻게 업데이트될지 결정하는 함수입니다.

다음은 `useOptimistic` 컴포넌트에서 `TaskList` 훅을 사용하는 예시입니다:

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**: UI의 일부를 렌더링하고 상태 업데이트를 비긴급 전환으로 표시하는 데 사용합니다. 이렇게 하면 비용이 많이 드는 업데이트 중에도 UI가 반응형을 유지할 수 있습니다. 기본 구문은 다음과 같습니다:

```js
startTransition(action)
```

`action`는 상태 업데이트를 수행하거나 전환 관련 로직을 트리거합니다. 이를 통해 타이핑이나 클릭 같은 긴급 UI 업데이트가 차단되지 않도록 보장합니다.

## `useMemo` 훅 사용하기

- **메모이제이션**: 특정 인수를 기준으로 비용이 많이 드는 함수 호출 결과를 캐시(기억)하는 최적화 기법입니다. 같은 인수가 다시 주어지면, 함수를 다시 계산하지 않고 캐시된 결과를 반환합니다.
- **`useMemo` 훅**: 이 훅은 계산된 값을 메모이제이션하는 데 사용합니다. 다음은 큰 배열을 정렬한 결과를 메모이제이션하는 예시입니다. `expensiveSortFunction`은 `largeArray`가 변경될 때만 실행됩니다.

```js
const memoizedSortedArray = useMemo(
  () => expensiveSortFunction(largeArray),
  [largeArray]
);
```

## `useCallback` 훅 사용하기

- **`useCallback` 훅**: 함수 참조를 메모이제이션하는 데 사용합니다.

```js
const handleClick = useCallback(() => {
 // code goes here
}, [dependency]);
``` 

- **`React.memo`**: 프로프가 변경되지 않았을 때 불필요한 재렌더링을 방지하기 위해 컴포넌트를 메모이제이션하는 데 사용합니다.

```js
const MemoizedComponent = React.memo(({ prop }) => {
 return (
   <>
     {/* Presentation */}
   </>
 )
});
```

## 의존성 관리 도구

- **의존성 정의**: 소프트웨어에서 의존성은 애플리케이션 내 한 컴포넌트나 모듈이 제대로 작동하려면 다른 컴포넌트나 모듈에 의존하는 경우를 말합니다. 의존성은 개발자가 다른 사람이 만든 미리 만들어진 함수나 도구를 사용할 수 있게 해주기 때문에 소프트웨어 애플리케이션에서 흔히 나타납니다. React 프로젝트에 필요한 두 가지 핵심 의존성은 `react`와 `react-dom` 패키지입니다.

```json
"dependencies": {
  "react": "^18.3.1",
  "react-dom": "^18.3.1"
}
```

- **패키지 관리자 정의**: 프로젝트에서 소프트웨어 의존성을 관리하려면 패키지 관리자를 사용해야 합니다. 패키지 관리자는 의존성 설치, 업데이트 및 제거에 사용하는 도구입니다. JavaScript, Python, Ruby, Java 같은 인기 있는 프로그래밍 언어들은 모두 패키지 관리자를 사용합니다. JavaScript의 인기 있는 패키지 관리자로는 npm, Yarn, pnpm이 있습니다.
- **`package.json` 파일**: 이 파일은 프로젝트에 관한 메타데이터를 포함하는 핵심 구성 파일로, 프로젝트 이름, 버전, 의존성 정보를 담고 있습니다. 또한 스크립트, 라이선스 정보, 프로젝트와 의존성 관리를 돕는 기타 설정도 정의합니다.
- **`package-lock.json` 파일**: 이 파일은 프로젝트에서 사용하는 모든 패키지의 정확한 버전을 고정합니다. 패키지를 업데이트하면 새로운 버전이 이 잠금 파일에도 반영됩니다.
- **`node_modules` 폴더**: 이 폴더는 `package.json` 파일에 나열된 의존성들의 실제 코드를 포함하며, 프로젝트의 직접 의존성과 그 의존성들의 의존성 모두를 담고 있습니다.
- **개발 의존성**: 개발 중에만 사용되고 실제 운영 환경에서는 사용하지 않는 패키지들입니다. 예를 들어 Jest 같은 테스트 라이브러리가 이에 해당합니다. Jest는 애플리케이션을 로컬에서 테스트할 때 필요하지만 운영 환경에서는 필요하지 않으므로 개발 의존성으로 설치합니다.

## React Router

- **소개**: React Router는 React 애플리케이션에 라우팅을 추가할 수 있게 해주는 서드파티 라이브러리입니다. 시작하려면 기존 React 프로젝트에 React Router를 이렇게 설치하세요:

```bash
npm i react-router
```

그다음 `main.jsx` 또는 `index.jsx` 파일 안에서 다음과 같이 라우트 구조를 설정해야 합니다:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

`path`과 `element`는 URL과 UI 컴포넌트를 연결하는 데 사용됩니다. 여기서는 홈페이지 라우트를 `App` 컴포넌트로 설정하고 있습니다.

- **여러 뷰와 라우트 설정**: 큰 애플리케이션에서는 여러 뷰와 라우트를 이렇게 설정하는 경우가 많습니다:

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

이 예제에서 `index` 프로프는 특정 경로 구간의 기본 라우트를 나타냅니다. 그래서 `Home` 컴포넌트는 루트 `/` 경로에서, `ProductsHome` 컴포넌트는 `/products` 경로에서 보여집니다.

- **라우트 중첩**: 라우트 안에 다른 라우트를 중첩할 수 있으며, 이 경우 자식 라우트의 경로가 부모 라우트 경로에 덧붙여집니다.

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

위 예제에서 트렌딩 상품 경로는 products/trending이 됩니다.

- **동적 세그먼트**: 동적 세그먼트는 URL 경로의 일부가 동적으로 변하는 부분입니다.

```jsx
<Route path=":category" element={<Category />} />
```

이 예제에서는 `category`라는 동적 세그먼트가 있습니다. 사용자가 products/brass-instruments 같은 URL로 이동하면 뷰가 `Category` 컴포넌트로 바뀌고, 세그먼트에 따라 적절한 데이터를 동적으로 가져올 수 있습니다.

- **`useParams` 훅**: 이 훅은 URL 경로에서 동적 매개변수에 접근하는 데 사용됩니다.

```js
import { useParams } from "react-router";

export default function Category() {
  let params = useParams();
   {/* Accessing the category param: params.category */}
   {/* rest of code goes here */}
}
```

## React 프레임워크

- **소개**: React 프레임워크는 라우팅, 이미지 최적화, 데이터 가져오기, 인증 등 다양한 기능을 제공합니다. 이는 특정 용도에 대해 별도의 프론트엔드와 백엔드 애플리케이션을 설정할 필요가 없을 수도 있다는 뜻입니다. React 프레임워크의 예로는 Next.js와 Remix가 있습니다.
- **Next.js 라우팅**: 이 라우팅 시스템은 동적 라우트, 병렬 라우트, 라우트 핸들러, 리다이렉트, 국제화 등을 지원합니다.

다음은 커스텀 요청 핸들러를 만드는 예시입니다:

```js
export async function GET() {
  const res = await fetch("https://example-api.com");
  const data = await res.json();

  return Response.json({ data });
}
```

- **Next.js 이미지 최적화**: `Image` 컴포넌트는 기본 HTML `img` 요소를 확장하여 페이지 로드 속도와 크기 최적화를 가능하게 합니다. 즉, 이미지는 뷰포트에 들어올 때만 로드되고 `Image` 컴포넌트는 각 기기에 맞는 크기의 이미지를 자동으로 제공합니다.

```jsx
import Image from "next/image";

export default function Page() {
  return (
    <Image src="link-to-image-goes-here" alt="descriptive-title-goes-here" />
  );
}
```

## 프로프 드릴링

- **정의**: 프로프 드릴링은 부모 컴포넌트에서 깊게 중첩된 자식 컴포넌트로 프로프스를 전달하는 과정으로, 일부 자식 컴포넌트가 해당 프로프스를 필요로 하지 않아도 전달하는 경우를 말합니다.

## 상태 관리

- **Context API**: Context는 부모 컴포넌트가 자식 컴포넌트에 프로프스를 통해 명시적으로 전달하지 않고도 정보를 제공할 때를 말합니다. `createContext`는 다른 컴포넌트가 읽을 컨텍스트를 나타내는 컨텍스트 객체를 생성하는 데 사용됩니다. `Provider`는 자식 컴포넌트에 컨텍스트 값을 공급하는 데 사용됩니다.

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**: Redux는 중앙 저장소와 상태 업데이트에 대한 엄격한 제어를 제공하여 상태 관리를 처리합니다. 액션, 리듀서, 미들웨어를 사용하는 예측 가능한 패턴을 따릅니다. 액션은 애플리케이션에서 Redux 저장소로 데이터를 보내는 정보의 페이로드로, 주로 사용자 상호작용에 의해 트리거됩니다. 리듀서는 이러한 액션에 따라 상태가 어떻게 변경되어야 하는지 지정하는 함수로, 상태가 불변 방식으로 업데이트되도록 보장합니다. 반면 미들웨어는 액션 디스패치와 리듀서 사이의 다리 역할을 하며, Redux의 핵심 흐름을 변경하지 않고 기능(예: 로깅, 비동기 작업 처리)을 확장할 수 있게 합니다.

- **Zustand**: 이 상태 관리 솔루션은 소규모에서 중간 규모 애플리케이션에 적합합니다. `useStore` 훅을 사용해 컴포넌트와 페이지에서 상태에 직접 접근할 수 있습니다. 이를 통해 액션, 리듀서, 프로바이더 없이도 데이터를 수정하고 접근할 수 있습니다.

## React 개발자 도구를 사용한 React 컴포넌트 디버깅

- **React Developer Tools**: Chrome, Firefox, Edge에서 React 컴포넌트를 검사하고 성능 문제를 파악할 수 있는 브라우저 확장 프로그램입니다. Safari에서는 `react-devtools` npm 패키지를 설치해야 합니다. React DevTools를 설치하고 브라우저에서 React 앱을 연 후, 브라우저 개발자 도구를 열면 React 디버깅을 위한 두 개의 추가 탭인 Components와 Profiler를 사용할 수 있습니다.
- **Components 탭**: 이 탭은 각 컴포넌트를 트리 뷰 형식으로 보여줍니다. 이 탭에서 할 수 있는 작업은 다음과 같습니다:
  - 앱의 컴포넌트 계층 구조를 확인하기
  - 프로프스, 상태, 컨텍스트 값을 실시간으로 확인하고 수정하기
  - 선택한 각 컴포넌트의 소스 코드를 확인하기
  - 컴포넌트 데이터를 콘솔에 기록하기
  - 컴포넌트의 DOM 요소를 검사하기
- **Profiler 탭**: 이 탭은 컴포넌트 성능 분석을 돕습니다. 컴포넌트 성능을 기록하여 불필요한 재렌더링을 찾아내고, 커밋 지속 시간을 확인하며, 느린 컴포넌트를 최적화할 수 있습니다.

## 실제 성능과 인지된 성능의 차이점

- **인지된 성능**: 사용자가 웹사이트의 성능을 인식하는 방식입니다. 반응 속도와 신뢰성 측면에서 평가하는 주관적인 척도이며, 수치로 정확히 측정하기 어렵지만 사용자 경험이 웹사이트의 성공 여부를 결정하기 때문에 매우 중요합니다.
- **실제 성능**: 웹사이트의 객관적이고 측정 가능한 성능입니다. 페이지 로드 시간, 서버 응답 시간, 렌더링 시간 같은 지표로 측정하며, 네트워크와 코드 자체에 관련된 여러 요인의 영향을 받습니다.

## 인지된 성능 향상을 위한 기법

- **지연 로딩**: 이 기법은 필수적이지 않은 리소스를 백그라운드에서 로딩하여 초기 로딩 시간을 최대한 줄입니다.  
- **폰트 지연 최소화**: 웹사이트에 커스텀 폰트가 있다면, 폰트 로딩 지연을 최소화해야 합니다. 그렇지 않으면 깜빡임 현상이나 커스텀 폰트가 로딩되는 동안 대체 폰트가 표시될 수 있습니다. 이를 위해 커스텀 폰트와 유사한 대체 폰트를 사용하는 것이 좋으며, 이렇게 하면 변경이 더 미묘하게 느껴집니다.  
- **로딩 표시기 사용**: 사용자가 요소를 클릭하는 즉시 긴 작업에 대해 로딩 표시기를 보여주면, 사용자가 작업에 연결되어 있다고 느끼고 기다리는 시간이 더 짧게 느껴집니다.

## 핵심 성능 개념

- **소스 순서**: HTML 요소가 문서 내에서 구조화된 방식을 말합니다. 이것은 무엇이 먼저 로드되는지를 결정하며 성능과 접근성에 큰 영향을 미칩니다.

소스 순서에 대한 몇 가지 모범 사례는 다음과 같습니다:

- 제목, 내비게이션 또는 주요 텍스트와 같은 중요한 콘텐츠를 HTML 구조에서 더 위쪽에 배치하세요.
- 분석용 스크립트나 서드파티 위젯과 같은 필수적이지 않은 스크립트는 렌더링을 차단하지 않도록 지연시키세요.
- 점진적 향상(progressive enhancement)을 사용해 스타일과 스크립트가 로드되기 전에도 핵심 경험이 작동하도록 하세요. 점진적 향상은 페이지가 우선 HTML로 작동하도록 만드는 아이디어를 기반으로 웹사이트와 애플리케이션을 구축하는 방법입니다.

다음은 방금 살펴본 모범 사례를 적용한 좋은 소스 순서 예시입니다:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **중요 렌더링 경로(Critical Rendering Path)**: 브라우저가 코드를 화면의 픽셀로 변환하기 위해 따르는 단계들의 수열입니다.
- **지연 시간(Latency)**: 브라우저와 서버 간에 요청이 이동하는 데 걸리는 시간입니다. 즉, 지연 시간이 길면 페이지가 느려집니다.

지연 시간을 줄이는 몇 가지 방법은 다음과 같습니다:

- CDN(Content Delivery Network)을 사용해 더 가까운 위치에서 파일을 제공하세요.
- Gzip 같은 압축을 활성화해 파일 크기를 줄이세요.
- 이미지를 최적화하고 지연 로딩(lazy loading)을 사용하세요.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## INP 개선하기

- **정의**: INP(Interaction to Next Paint)는 사용자가 클릭이나 키 입력 같은 상호작용을 한 시점부터 브라우저가 다음 화면을 업데이트할 때까지 걸리는 시간을 측정해 페이지의 전반적인 반응성을 평가합니다. INP가 낮을수록 더 반응성이 좋은 페이지입니다.

INP를 개선하는 방법은 다음과 같습니다:

- 긴 JavaScript 작업을 나누어 메인 스레드 작업을 줄이세요.
- 중요하지 않은 스크립트에는 `requestIdleCallback()`를 사용하세요. 이 함수는 브라우저가 유휴 상태일 때 호출되도록 예약합니다.
- 앞서 다룬 무거운 자산은 지연 로드하거나 미루세요.
- 이벤트 핸들러를 최적화하세요. 핸들러가 너무 자주 실행되거나 무거운 작업을 수행하면 페이지가 느려지고 INP가 증가할 수 있습니다. 이를 해결하는 방법은 디바운싱입니다. 디바운싱은 사용자가 입력을 멈춘 후 일정 시간(예: 300ms) 지나야 함수가 실행되도록 하여 불필요한 계산을 막고 성능을 향상합니다.

## 브라우저에서 렌더링이 작동하는 방식

- **렌더링 작동 방식**: 먼저 브라우저가 HTML을 파싱하여 DOM을 만듭니다. 다음으로 브라우저가 CSS를 처리하여 CSS Object Model, 즉 CSSOM을 구성합니다. 이것은 요소에 스타일을 적용하는 방식을 결정하는 또 다른 트리 구조입니다. 마지막으로 브라우저가 계산된 스타일과 레이아웃을 바탕으로 각 요소를 화면에 픽셀 단위로 그려 렌더링합니다. 복잡한 페이지에서는 여러 레이어가 합성되어 최종 시각적 결과물을 형성할 수도 있습니다.

## 성능이 지속 가능성에 미치는 영향

- **배경 정보**: 인터넷은 전 세계 탄소 배출량의 약 2%를 차지하는데, 이는 항공 산업과 같은 수준입니다! 데이터 센터에서 사용자 기기까지 모든 바이트 전송에는 전력이 필요합니다. 큰 파일과 비효율적인 스크립트는 더 많은 전력 소비를 의미합니다. 고성능 웹사이트는 단순히 더 빠른 것뿐만 아니라 불필요한 처리와 에너지 사용을 줄입니다.

## 페이지 로딩 시간을 줄이는 방법

- **미디어 자산 최적화**: 큰 이미지와 동영상은 로딩 속도를 느리게 하는 주요 원인입니다. 이러한 자산을 최적화하면 사이트 속도를 크게 높일 수 있습니다. 여기에는 이미지 압축, WebP 같은 최신 형식 사용, 자산에 대한 지연 로딩 적용 등이 포함됩니다.
- **브라우저 캐싱 활용**: 캐싱은 브라우저가 웹사이트 일부를 로컬에 저장하게 하여 재방문 시 로딩 시간을 줄여줍니다.
- **파일 축소 및 압축**: 파일 크기를 줄이면 다운로드 속도가 빨라집니다. 여기에는 전송 파일 크기 축소와 CSS 및 JavaScript 파일의 축소가 포함됩니다.

## "사용 가능 시간" 개선하기

- **정의**: "사용 가능 시간"은 사용자가 페이지를 요청한 시점부터 의미 있게 상호작용할 수 있을 때까지의 간격입니다. "사용 가능 시간"을 개선하려면 자산을 지연 로드하거나 렌더링을 차단하는 리소스를 최소화하세요.

## 성능 측정을 위한 주요 지표

- **First Contentful Paint 또는 FCP**: 화면에 첫 번째 콘텐츠(텍스트나 이미지)가 얼마나 빨리 나타나는지를 측정합니다. 좋은 FCP는 1.8초 미만으로 간주하며, 나쁜 FCP는 3초 이상입니다. Chrome 개발자 도구의 성능 탭에서 FCP를 확인할 수 있습니다.
- **Total Blocking Time**: 메인 스레드가 무거운 JavaScript 작업으로 얼마나 오래 차단되는지를 보여줍니다. Total Blocking Time(TBT)이 높으면 사용자가 느린 반응을 경험합니다. TBT를 개선하려면 긴 작업을 나누고 필수적이지 않은 스크립트는 지연시키세요.
- **이탈률**: 상호작용 없이 사이트를 떠나는 방문자의 비율입니다. 이탈률이 높으면 페이지가 너무 느릴 수 있습니다.
- **고유 사용자 수**: 사이트를 방문하는 개별 방문자 수를 추적하는 지표입니다. 이탈률과 고유 사용자 수는 Google Analytics를 사용해 확인할 수 있습니다. 이를 통해 지표를 모니터링하고 참여도를 높일 수 있습니다.

## 일반적인 성능 측정 도구

- **Chrome 개발자 도구**: Chrome 개발자 도구는 Google Chrome 내장 도구로, 실시간으로 성능을 분석하고 디버그할 수 있습니다. 개발자 도구는 로딩 시간, CPU 사용량, 렌더링 지연을 보여줍니다. 특히 First Contentful Paint(FCP), 즉 사용자가 처음으로 눈에 띄는 콘텐츠를 보는 속도를 측정하는 데 유용합니다. 웹사이트가 느리게 느껴진다면 개발자 도구가 병목 현상을 찾아내는 데 도움을 줍니다.
- **Lighthouse**: 성능, SEO, 접근성을 자동으로 검사하는 도구입니다.
- **WebPageTest**: 이 도구는 다양한 위치와 기기에서 사이트가 어떻게 로드되는지 테스트할 수 있습니다. 사이트의 Speed Index, Total Blocking Time 등 주요 성능 지표를 상세하게 분석해 줍니다. 전 세계 실제 사용자가 사이트를 어떻게 경험하는지 알고 싶다면 WebPageTest가 적합합니다.
- **PageSpeed Insights**: 이 도구는 웹사이트를 분석하고 모바일과 데스크톱 모두에 대해 빠른 개선 방안을 제안합니다. 사이트 속도를 저해하는 요소를 알려주고 이미지 최적화, 렌더링 차단 스크립트 제거, 서버 응답 시간 단축 같은 구체적인 권장 사항을 제공합니다. PageSpeed Insights는 Google이 사이트 성능을 어떻게 평가하는지 빠르고 쉽게 확인할 수 있는 방법입니다.
- **실제 사용자 모니터링**: RUM 도구는 실제 사용자 행동을 추적하여 방문자가 사이트를 어떻게 경험하는지 보여줍니다. 대표적인 RUM 도구로는 페이지 로드 시간과 이탈률을 추적하는 Google Analytics, 실시간 성능 문제를 모니터링하는 New Relic이나 Datadog이 있습니다. 실제 사용자 데이터를 원한다면 RUM 도구가 필수입니다.

## 성능 웹 API 활용하기

- **정의**: 성능 웹 API는 개발자가 웹페이지가 얼마나 효율적으로 로드되고 반응하는지 코드를 통해 직접 추적할 수 있게 해줍니다. 이 API들은 페이지 로드 시간 측정, 렌더링 및 상호작용 지연 추적, JavaScript 실행 시간 분석을 가능하게 합니다.
- **`performance.now()`**: 이 API는 사이트의 각 부분이 로드되는 데 걸리는 시간을 측정할 수 있도록 밀리초 단위의 고정밀 타임스탬프를 제공합니다.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: 이 API는 DNS 조회부터 `DOMContentLoaded`까지 페이지 로드의 모든 단계를 세분화해서 보여줍니다.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: 이 API는 레이아웃 이동, 긴 작업, 사용자 상호작용 같은 성능 이벤트를 감지합니다.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## CSS 성능 향상을 위한 기법

- **CSS 애니메이션**: 크기, 위치, 레이아웃 같은 특정 CSS 속성을 애니메이션하면 "리플로우"라는 과정이 발생하는데, 이 과정에서 브라우저가 페이지 내 일부 요소의 위치와 형태를 다시 계산합니다. 이로 인해 다시 그리기(repaint)가 필요한데, 이는 계산 비용이 많이 듭니다. 따라서 CSS 애니메이션 수를 가능한 한 줄이거나 최소한 사용자가 애니메이션을 켜고 끌 수 있는 옵션을 제공하는 것이 좋습니다.

## JavaScript 성능 향상을 위한 기법

- **코드 분할**: JavaScript 코드를 중요한 작업과 중요하지 않은 작업을 수행하는 모듈로 분할하는 것도 도움이 됩니다. 이렇게 하면 중요한 모듈은 가능한 빨리 미리 불러오고, 중요하지 않은 모듈은 페이지를 최대한 빠르게 렌더링하기 위해 나중으로 미룰 수 있습니다.
- **DOM 조작**: DOM 조작은 Document Object Model(DOM)과 상호작용하여 JavaScript로 페이지 내용을 동적으로 변경하는 과정을 의미합니다. DOM 조작은 계산 비용이 많이 듭니다. JavaScript 코드에서 DOM 조작을 줄이면 성능이 향상됩니다.

## CSS 프레임워크

- **CSS 프레임워크**: CSS 프레임워크는 작업 속도를 높이고, 웹사이트 전반에 걸쳐 일관된 시각적 스타일을 만들며, 여러 브라우저에서 디자인이 일관되게 보이도록 하고, CSS 코드를 더 체계적으로 관리할 수 있게 도와줍니다.
- **인기 있는 CSS 프레임워크**: 인기 있는 CSS 프레임워크로는 Tailwind CSS, 부트스트랩, Materialize, Foundation이 있습니다.
- **잠재적인 단점**:
  - 프레임워크에서 제공하는 CSS가 사용자 정의 CSS와 충돌할 수 있습니다.
  - 같은 프레임워크를 사용하는 다른 웹사이트와 비슷하게 보일 수 있습니다.
  - 큰 프레임워크는 성능 문제를 일으킬 수 있습니다.

## 두 가지 유형의 CSS 프레임워크

- **유틸리티 우선 CSS 프레임워크**: 이 프레임워크들은 여백, 패딩, 배경색 설정과 같은 특정 목적을 가진 작은 클래스들로 구성되어 있습니다. 필요에 따라 이러한 작은 클래스를 HTML 요소에 직접 할당할 수 있습니다. Tailwind CSS가 유틸리티 우선 CSS 프레임워크에 속합니다.

다음은 Tailwind CSS를 사용해 버튼을 스타일링하는 예시입니다.

```html
<button class="bg-blue-500 text-white font-bold py-2 px-4 rounded-full hover:bg-blue-700">
  Button
</button>
```

- **컴포넌트 기반 CSS 프레임워크**: 이 프레임워크들은 미리 정의된 스타일을 가진 미리 만들어진 컴포넌트를 제공하며, 이를 웹사이트에 추가할 수 있습니다. 컴포넌트는 CSS 프레임워크 공식 문서에서 확인할 수 있으며, 프로젝트에 복사해서 붙여넣으면 됩니다. Bootstrap이 컴포넌트 기반 CSS 프레임워크에 속합니다.

다음은 Bootstrap을 사용해 리스트 그룹을 만드는 예시입니다. HTML 요소에 작은 클래스를 적용하는 대신, HTML 구조를 포함한 전체 컴포넌트를 추가합니다.

```html
<div class="card" style="width: 25rem;">
  <ul class="list-group list-group-flush">
    <li class="list-group-item">HTML</li>
    <li class="list-group-item">CSS</li>
    <li class="list-group-item">JavaScript</li>
  </ul>
</div>
```

## Tailwind CSS

Tailwind은 유틸리티 우선 CSS 프레임워크입니다. 사용자 정의 CSS 규칙을 작성하는 대신, HTML에서 작은 유틸리티 클래스를 조합해 디자인을 만드세요.

#
## 반응형 디자인 유틸리티

Tailwind는 `sm:`, `md:`, `lg:` 같은 접두사를 사용해 다양한 화면 크기에서 스타일을 적용합니다.

```html
<div class="w-full md:w-1/2 lg:flex-row">
  Responsive layout
</div>
```

#
## 플렉스박스 유틸리티

`flex`, `flex-col`, `justify-around`, `items-center` 같은 클래스로 유연한 레이아웃을 쉽게 만들 수 있습니다.

```html
<div class="flex flex-col md:flex-row justify-around items-center">
  <p>Column on small screens</p>
  <p>Row on medium and larger screens</p>
</div>
```

#
## 그리드 유틸리티

Tailwind는 `grid`, `grid-cols-1`, `md:grid-cols-3` 같은 CSS Grid 유틸리티를 포함합니다.

```html
<div class="grid grid-cols-1 md:grid-cols-3 gap-8">
  <div class="bg-gray-100 p-4">Column 1</div>
  <div class="bg-gray-100 p-4">Column 2</div>
  <div class="bg-gray-100 p-4">Column 3</div>
</div>
```

#
## 간격 유틸리티

`mt-8`, `mx-auto`, `p-4`, `gap-4` 같은 유틸리티로 CSS를 작성하지 않고도 일관된 간격을 만들 수 있습니다.

```html
<div class="mt-8 p-4 bg-indigo-600 text-white">
  Spaced content
</div>
```

#
## 타이포그래피 유틸리티

`uppercase`, `font-bold`, `font-semibold`, `text-4xl` 같은 유틸리티로 텍스트 모양을 제어하세요.

`text-3xl`, `md:text-5xl` 같은 브레이크포인트에서 조정되는 폰트 크기를 설정할 수 있습니다.

```html
<h1 class="text-3xl md:text-5xl font-semibold text-center">
  Responsive Heading
</h1>
```

#
## 색상과 호버 상태

Tailwind는 `text-red-700`, `bg-indigo-600`, `bg-gray-100` 같은 다양한 색상 팔레트를 제공합니다.

`hover:bg-pink-600` 같은 클래스로 인터랙티브 효과를 간단히 만들 수 있습니다.

```html
<a href="#" class="bg-pink-500 hover:bg-pink-600 text-white px-4 py-2 rounded-md">
  Hover Me
</a>
```

#
## 테두리, 링, 효과

- **테두리**: `border-2 border-red-300`는 지정한 두께와 색상의 테두리를 추가합니다.
- **링**: `ring-1 ring-gray-300`는 포커스나 카드에 자주 쓰이는 윤곽선 같은 효과를 만듭니다.
- **둥근 모서리와 스케일링**: `rounded-md`, `rounded-xl`, `scale-105` 같은 클래스로 마무리감을 더하세요.

```html
<div class="p-6 rounded-xl ring-2 ring-fuchsia-500 scale-105">
  Highlighted card
</div>
```

#
## 그라디언트

Tailwind는 `bg-gradient-to-r from-fuchsia-500 to-indigo-600` 같은 그라디언트 유틸리티를 지원합니다.

```html
<div class="p-4 text-white bg-gradient-to-r from-fuchsia-500 to-indigo-600">
  Gradient background
</div>
```

## CSS 전처리기

- **CSS 전처리기**: CSS 전처리기는 표준 CSS를 확장하는 도구입니다. 확장된 구문으로 작성한 코드를 네이티브 CSS 파일로 컴파일합니다. 복잡한 프로젝트에서 더 깔끔하고 재사용 가능하며 반복을 줄이고 확장 가능한 CSS를 작성하는 데 도움이 됩니다.
- **기능**: CSS 전처리기가 제공할 수 있는 기능에는 변수, 믹스인, 중첩, 선택자 상속 등이 있습니다.
- **인기 있는 CSS 전처리기**: 인기 있는 CSS 전처리기로는 Sass, Less, Stylus가 있습니다.  
- **잠재적 단점**:
  - CSS 규칙을 표준 CSS로 컴파일하는 과정에서 오버헤드가 발생할 수 있습니다.
  - 컴파일된 코드는 디버그하기 어려울 수 있습니다.

## Sass

- **Sass**: 가장 인기 있는 CSS 전처리기 중 하나입니다. Sass는 "Syntactically Awesome Style Sheets"의 약자입니다.
- **Sass가 지원하는 기능**: Sass는 변수, 중첩된 CSS 규칙, 모듈, 믹스인, 상속, 기본 수학 연산을 위한 연산자 같은 기능을 지원합니다.

## Sass가 지원하는 두 가지 구문

다음은 들여쓰기 구문에서 변수를 정의하고 사용하는 예입니다.

```sass
$primary-color: #3498eb

header
  background-color: $primary-color
```

다음은 SCSS 구문에서 믹스인을 정의하는 예입니다. 이 경우 믹스인 이름은 `center-flex`이며, 플렉스박스를 사용해 요소를 가운데 정렬하는 세 가지 CSS 속성을 포함합니다.

@mixin center-flex {

section {

## 수동 및 자동 테스트

- **수동 테스트**: 수동 테스트에서는 테스터가 애플리케이션의 각 부분을 직접 확인하며 다양한 기능을 테스트해 제대로 작동하는지 확인합니다. 테스트 과정에서 버그가 발견되면 테스터가 소프트웨어 팀에 보고하여 수정할 수 있게 합니다.
- **자동 테스트**: 자동 테스트에서는 별도의 프로그램을 작성해 애플리케이션이 예상대로 작동하는지 자동으로 검사할 수 있습니다.

## 단위 테스트

- **단위 테스트**: 단위 테스트에서는 각 함수를 테스트하여 모든 것이 예상대로 작동하는지 확인합니다. 단위 테스트는 코드의 예상 동작을 나타내기 때문에 애플리케이션 문서의 한 형태로도 활용할 수 있습니다.
- **단일 책임 원칙**: 단일 책임 원칙은 각 함수가 작고 하나의 책임만 가지도록 유지할 것을 권장합니다.
- **일반적인 JavaScript 테스트 프레임워크**: 일반적인 테스트 프레임워크로는 Jest, Mocha, Vitest가 있습니다. Jest는 단위 테스트에 널리 사용되는 인기 있는 테스트 프레임워크입니다.

다음은 Jest를 사용한 단위 테스트 예시입니다.

먼저, 새로 포맷된 문자열을 반환하는 함수 하나를 만듭니다:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

별도의 `getFormattedWord.test.js` 파일에서 함수가 예상대로 작동하는지 확인하는 테스트를 작성할 수 있습니다. `getFormattedWord.test.js` 파일은 다음과 같습니다:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **`expect` 함수**: `expect` 함수는 값을 테스트하는 데 사용됩니다.
- **매처**: 매처는 값이 예상대로 동작하는지 확인하는 함수입니다. 위 예시에서 매처는 `toBe()` 입니다. Jest에는 다양한 매처가 있습니다.

Jest를 사용하려면 먼저 `jest` 패키지를 `npm i jest` 로 설치해야 합니다. 또한 `package.json` 파일에 다음과 같은 스크립트를 추가해야 합니다:

```js
"scripts": {
  "test": "jest"
},
```

그다음 `npm run test` 명령어를 실행하여 테스트를 실행할 수 있습니다.

## 소프트웨어 개발 수명 주기

- **소프트웨어 개발 수명 주기의 다양한 단계**:
  - **기획 단계**: 개발 팀이 이해관계자로부터 제안된 작업에 대한 요구사항을 수집합니다.
  - **설계 단계**: 소프트웨어 팀이 요구사항을 세분화하고 최적의 해결 방안을 결정합니다.
  - **구현 단계**: 소프트웨어 팀이 요구사항을 관리 가능한 작업으로 나누어 구현합니다.
  - **테스트 단계**: 수동 및 자동 테스트를 통해 새 작업을 검증합니다. 때로는 개발 전 과정에서 애플리케이션을 테스트하여 발생하는 문제를 발견하고 수정합니다.
  - **배포 단계**: 팀이 새 변경 사항을 빌드 또는 테스트 환경에 배포합니다.
  - **유지 관리 단계**: 운영 중인 애플리케이션에서 고객에게서 발생하는 문제를 수정합니다.

- **소프트웨어 개발 수명 주기의 다양한 모델**:
  - **워터폴 모델**: 각 수명 주기 단계가 완료되어야 다음 단계로 넘어갈 수 있는 모델입니다.
  - **애자일 모델**: 작업을 스프린트로 나누어 반복적으로 개발하는 데 중점을 둡니다.

## BDD와 TDD

- **TDD**: 테스트 주도 개발은 먼저 테스트를 작성하는 방법론입니다. 기능을 구현하기 전에 테스트를 작성하면 개발 과정에서 개발자에게 실시간 피드백을 제공합니다.
- **BDD**: 행동 주도 개발은 일련의 테스트를 비즈니스 목표와 일치시키는 접근법입니다. BDD의 테스트 시나리오는 기술자와 비기술자 모두가 이해할 수 있는 언어로 작성되어야 합니다. 이러한 구문의 예로는 Gherkin이 있습니다.
- **BDD 테스트 프레임워크**: BDD 테스트 프레임워크의 예로는 Cucumber, JBehave, SpecFlow가 있습니다.

## 단위 테스트에서의 어서션

다음은 Chai를 사용해 `addThreeAndFour` 함수의 반환값이 숫자 7과 같은지 확인하는 어서션 예시입니다:

assert.equal(addThreeAndFour(), 7);

- **모범 사례**: 어떤 어서션 라이브러리를 사용하든, 어떤 테스트가 왜 실패하는지 이해하는 데 도움이 되는 명확한 어서션과 실패 메시지를 작성하세요.

## 모킹, 페이킹, 스터빙

- **모킹**: 모킹은 실제 데이터를 실제 컴포넌트의 동작을 흉내 내는 가짜 데이터로 대체하는 과정입니다. 예를 들어, 테스트에서 데이터를 가져오기 위해 계속 API를 호출하는 대신 API 응답을 모킹할 수 있습니다.
- **스터빙**: 스터브는 애플리케이션에서 예상되는 동작에 대해 미리 정의된 응답이나 더미 데이터를 반환하는 객체입니다. 예를 들어, 실제 데이터베이스 연결에 의존하지 않고 테스트에서 데이터베이스 연결 동작을 스터빙할 수 있습니다.
- **페이킹**: 페이크는 복잡성이나 부작용 없이 실제 컴포넌트의 단순화된 버전입니다. 예를 들어, 실제 데이터베이스와 상호작용하는 대신 메모리에 데이터를 저장하여 데이터베이스를 페이킹할 수 있습니다. 이렇게 하면 실제 데이터베이스를 다루는 것보다 훨씬 빠르게 메모리에서 데이터베이스 작업을 흉내 낼 수 있습니다.

## 종단 간 테스트

- **종단 간 테스트**: 종단 간 테스트 또는 E2E는 사용자의 관점에서 실제 시나리오를 테스트합니다. 종단 간 테스트는 애플리케이션이 올바르게 작동하고 사용자에게 예측 가능하도록 돕습니다. 하지만 설정, 설계, 유지 관리에 시간이 많이 걸립니다.
- **종단 간 테스트 프레임워크**: Playwright는 마이크로소프트에서 개발한 인기 있는 종단 간 테스트 프레임워크입니다. 그 밖에 Cypress, Selenium, Puppeteer 같은 종단 간 테스트 도구도 있습니다.

다음은 Playwright를 사용한 freeCodeCamp 코드베이스의 E2E 테스트 예시입니다. `beforeEach` 훅은 각 테스트 전에 실행됩니다. 테스트는 기부자가 메뉴 바에 후원자 링크가 있고, 아바타 주위에 특별한 스타일의 테두리가 있는지 확인합니다:

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## 보안 테스트

- **보안 테스트**: 보안 테스트는 취약점과 약점을 식별하는 데 도움을 줍니다.
- **보안 원칙**:
  - **기밀성**: 의도된 수신자가 아닌 다른 수신자에게 민감한 정보가 유출되는 것을 방지합니다.
  - **무결성**: 악의적인 사용자가 사용자 정보를 변경하지 못하도록 합니다.
  - **인증**: 사용자의 신원을 확인하여 해당 시스템을 사용할 권한이 있는지 검증합니다.
  - **권한 부여**: 인증된 사용자가 수행할 수 있는 작업이나 접근할 수 있는 시스템 부분을 결정하는 과정입니다.
  - **가용성**: 권한이 있는 사용자가 필요할 때 정보와 서비스를 이용할 수 있도록 보장합니다.
  - **부인 방지**: 송신자와 수신자 모두에게 전달 증명과 송신자 신원 확인을 보장하여 송신자가 정보를 보냈다는 사실을 부인하지 못하게 합니다.
- **일반적인 보안 위협**:
  - **크로스 사이트 스크립팅(XSS)**: 공격자가 악성 스크립트를 웹 페이지에 주입하고 피해자의 브라우저 환경에서 실행할 때 발생합니다.
  - **SQL 인젝션**: 악의적인 사용자가 데이터베이스에 악성 코드를 주입할 수 있게 합니다.
  - **서비스 거부(DoS) 공격**: 악의적인 사용자가 다량의 요청이나 트래픽을 웹사이트에 몰아 서버를 느리게 하거나 다운시켜 사이트를 사용자에게 사용할 수 없게 만듭니다.
- **보안 테스트 도구 분류**:
  - **정적 애플리케이션 보안 테스트**: 애플리케이션의 소스 코드를 평가하여 보안 취약점을 식별합니다.
  - **동적 애플리케이션 보안 테스트**: 애플리케이션의 프런트엔드와 인터페이스하여 잠재적 보안 약점을 찾아냅니다. DAST 도구는 소스 코드에 접근하지 않습니다.
- **침투 테스트(pentest)**: 침투 테스트는 애플리케이션에 대한 모의 사이버 공격을 생성하여 시스템의 취약점을 식별하는 보안 테스트 유형입니다.

## A/B 테스트

- **A/B 테스트**: A/B 테스트는 페이지나 애플리케이션의 두 가지 버전을 비교하여 어느 쪽이 더 나은 성능을 보이는지 연구하는 방법입니다. 버킷 테스트 또는 분할 테스트라고도 합니다. A/B 테스트를 통해 보다 데이터 기반의 결정을 내리고 사용자 경험을 지속적으로 개선할 수 있습니다.
- **A/B 테스트 도구**: A/B 테스트에 사용할 수 있는 도구로는 GrowthBook과 LaunchDarkly가 있습니다.

## 알파 및 베타 테스트

초기 개발과 소프트웨어 테스트가 완료되면, 애플리케이션을 테스터와 실제 사용자가 테스트하는 것이 중요합니다. 여기서 알파 테스트와 베타 테스트가 등장합니다.

- **알파 테스트**: 알파 테스트는 선택된 테스터 그룹이 애플리케이션을 검토하여 출시 전에 버그가 없는지 확인하는 과정입니다. 알파 테스트는 인수 테스트의 일부이며 화이트 박스와 블랙 박스 테스트 기법을 모두 활용합니다.
- **베타 테스트**: 베타 테스트는 애플리케이션을 실제 사용자에게 제공하는 단계입니다. 사용자는 애플리케이션과 상호작용하며 피드백을 제공합니다. 베타 테스트 역시 사용자 인수 테스트의 한 형태입니다.
- **인수 테스트**: 인수 테스트는 소프트웨어 애플리케이션이 출시 전에 비즈니스 요구사항과 사용자 요구를 충족하는지 확인합니다.
- **블랙 박스 테스트**: 블랙 박스 테스트는 애플리케이션의 예상 동작에만 집중합니다.
- **화이트 박스 테스트**: 화이트 박스 테스트는 테스터가 내부 컴포넌트를 알고 이를 대상으로 테스트를 수행합니다.

## 회귀 테스트

- **회귀**: 회귀는 새로운 변경 사항이 의도치 않게 기존 기능을 망가뜨리는 상황을 말합니다.
- **회귀 테스트**: 회귀 테스트는 회귀 문제를 발견하는 데 도움을 줍니다. 회귀 테스트에서는 애플리케이션의 일부에 대해 기능 테스트를 다시 실행하여 모든 것이 여전히 예상대로 작동하는지 확인합니다.
- **회귀 테스트 도구**: 회귀 테스트를 수행하는 데 사용할 수 있는 도구로는 Puppeteer, Playwright, Selenium, Cypress가 있습니다.
- **회귀 테스트 기법**:
  - **단위 회귀 테스트**: 주요 변경이나 수정이 애플리케이션에 적용될 때마다 테스트해야 할 항목 목록을 갖고 테스트하는 방법입니다.
  - **부분 회귀 테스트**: 새로운 변경 사항이 애플리케이션의 특정 부분을 망가뜨리지 않았는지 확인하는 목표 지향적 접근법입니다.
  - **완전 회귀 테스트**: 코드베이스의 모든 기능에 대해 테스트를 실행합니다. 가장 시간이 많이 걸리고 세밀한 방법입니다.
- **재테스트**: 재테스트는 알려진 문제를 확인하고 해결되었는지 검증하는 데 사용합니다. 반면 회귀 테스트는 최근 코드베이스 변경으로 인해 발생했을 수 있는 알려지지 않은 문제를 찾는 데 집중합니다.

## TypeScript란 무엇인가요

- **JavaScript**: JavaScript는 동적 타입 언어입니다. 즉, 변수는 실행 시점에 어떤 값도 받을 수 있습니다. 동적 타입 언어의 어려움은 타입 안전성이 부족해 오류가 발생할 수 있다는 점입니다.

예를 들어, JavaScript 함수가 배열을 기대하더라도 숫자를 전달해 호출할 수 있습니다:

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

위 예제의 `console` 출력은 `undefined`입니다.

- **TypeScript**: TypeScript는 JavaScript 언어에 정적 타입을 추가하여 타입 불일치로 인한 오류를 코드 실행 전에 잡아내도록 도와줍니다.

예를 들어, `array` 매개변수의 타입을 다음과 같이 정의할 수 있습니다:

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

이 타입 정의는 TypeScript에 `array` 매개변수가 문자열 배열이어야 한다고 알려줍니다. 그런 다음 `getRandomValue`를 호출할 때 숫자를 전달하면 컴파일러 오류라는 오류가 발생합니다.

- **컴파일러**: TypeScript 코드를 일반 JavaScript로 먼저 컴파일해야 합니다. 컴파일러를 실행하면 TypeScript가 코드를 평가하고 타입이 맞지 않는 문제에 대해 오류를 발생시킵니다.

## TypeScript의 데이터 타입

- **TypeScript의 원시 데이터 타입**: 원시 데이터 타입인 `string`, `null`, `undefined`, `number`, `boolean`, `bigint`에 대해 TypeScript는 해당하는 타입 키워드를 제공합니다.

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **배열**: 특정 타입의 배열을 두 가지 구문으로 정의할 수 있습니다.

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **객체**: 객체의 정확한 구조를 정의할 수 있습니다.

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

모든 키는 임의지만 모든 값이 문자열이어야 하는 객체를 원한다면, 두 가지 방법으로 정의할 수 있습니다:

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **TypeScript의 기타 유용한 타입**:
  - **`any`**: `any`는 값이 어떤 타입이든 될 수 있음을 나타냅니다. 이 키워드는 컴파일러에게 해당 변수의 타입을 더 이상 신경 쓰지 말라고 알려줍니다.
  - **`unknown`**: `unknown`는 값의 타입을 신경 쓰지만 실제로 어떤 타입인지 모를 때 TypeScript에 알립니다. `unknown`가 일반적으로 `any`보다 선호됩니다.
  - **`void`**: 보통 함수 정의 시에만 사용하는 특별한 타입입니다. 반환값이 없는 함수는 반환 타입으로 `void`를 사용합니다.
  - **`never`**: 절대 존재하지 않는 타입을 나타냅니다.

- **`type` 키워드**: 이 키워드는 `const`와 비슷하지만 변수를 선언하는 대신 타입을 선언할 수 있습니다.

이는 유니언 타입이나 특정 값만 포함하는 타입 같은 사용자 정의 타입을 선언할 때 유용합니다:

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**: 인터페이스는 타입을 위한 클래스와 같습니다. 다른 인터페이스를 구현하거나 확장할 수 있고, 구체적으로 객체 타입이며, `type` 선언에서 제공하는 특정 기능이 필요하지 않다면 일반적으로 선호됩니다.

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **반환 타입 정의**: 함수의 _반환 타입_도 정의할 수 있습니다.

아래 예시는 반환값을 문자열로 정의합니다. 다른 값을 반환하려 하면 TypeScript가 컴파일 오류를 발생시킵니다.

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```

## 제네릭

- **제네릭 타입 정의하기**: 제네릭 타입을 정의하고 함수에서 참조할 수 있습니다. 이는 함수의 타입 정의 동작을 제어하기 위해 함수에 제공하는 특별한 매개변수로 생각할 수 있습니다.

다음은 함수에 제네릭 타입을 정의하는 예시입니다:

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

`<T>` 구문은 TypeScript에 함수에 대한 제네릭 타입 `T` 을 정의하고 있음을 알립니다. `T` 는 제네릭 타입에 흔히 쓰이는 이름이지만, 어떤 이름이든 사용할 수 있습니다.

그다음, `array` 매개변수가 제네릭 타입과 일치하는 값들의 배열임을 TypeScript에 알리고, 반환값은 같은 타입의 단일 요소임을 지정합니다.

- **함수 호출 시 타입 인수 지정하기**: 함수 이름과 매개변수 사이에 꺾쇠괄호를 사용해 함수 호출에 타입 인수를 전달할 수 있습니다.

다음은 함수 호출에 타입 인수를 전달하는 예시입니다:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

이것은 TypeScript에 찾으려는 요소가 입력 요소임을 알려줍니다.

## 타입 좁히기

- **진리값에 의한 좁히기**: 아래 예제에서, `value`이 `email`일 수도 있기 때문에 `email`의 `null` 속성에 접근하려 하면 컴파일러 오류가 발생합니다.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

조건문을 사용해 `email`이 진리값일 때만 속성에 접근할 수 있습니다:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

진리값 검사는 반대 방향으로도 작동할 수 있습니다:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

오류를 발생시키면 이 코드의 논리 실행이 종료되므로 `console.log()` 호출에 도달했을 때 TypeScript는 `email`이 `null`일 수 없다는 것을 압니다.

- **선택적 체이닝**: 선택적 체이닝 `?.`도 타입 좁히기의 한 형태로, `email` 값이 `null`일 경우 속성 접근이 불가능하다는 전제에 기반합니다.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **`typeof` 연산자**: `typeof` 연산자를 사용해 변수의 타입을 조건문으로 확인할 수 있습니다.

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **`instanceof` 키워드**: 객체가 클래스에서 온 경우 `instanceof` 키워드를 사용해 타입을 좁힐 수 있습니다.

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **캐스팅**: TypeScript가 아래 예제에서 `request.json()` 메서드 결과값의 타입을 자동으로 판단하지 못하면 컴파일러 오류가 발생합니다. 이를 해결하는 한 방법은 타입을 캐스팅하는 것이지만, 이렇게 하면 TypeScript가 잠재적 오류를 잡아내는 능력이 약해집니다.

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **타입 가드**: 타입을 캐스팅하는 대신 타입 가드를 작성할 수 있습니다:

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

`user is User` 구문은 함수가 불리언 값을 반환함을 나타내며, true일 때 `user` 값이 `User` 인터페이스를 만족함을 의미합니다.

## `tsconfig` 파일

- **`tsconfig.json`**: TypeScript 컴파일러 설정은 프로젝트 루트 디렉터리에 있는 `tsconfig.json` 파일에 있습니다.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

위 예제에서 사용된 컴파일러 옵션에 대한 설명은 다음과 같습니다:

- **`compilerOptions`**: `compilerOptions` 속성은 TypeScript 컴파일러 동작 방식을 제어하는 곳입니다.
- **`rootDir` 및 `outDir`**: `rootDir` 와 `outDir` 은 TypeScript에 소스 파일이 있는 디렉터리와 변환된 JavaScript 코드가 들어갈 디렉터리를 알려줍니다.
- **`lib`**: `lib` 속성은 컴파일러가 사용할 타입 정의를 결정하며, 특정 ES 릴리스, DOM 등 지원을 포함할 수 있게 합니다.
- **`module` 및 `moduleResolution`**: `module` 와 `moduleResolution` 는 패키지가 CommonJS 또는 ECMAScript 모듈을 어떻게 사용하는지 관리하는 데 함께 작동합니다.
- **`esModuleInterop`**: `esModuleInterop` 은 CommonJS와 ES 모듈 간 원활한 상호 운용성을 위해 가져오기(import)에 대해 네임스페이스 객체를 자동으로 생성합니다.
- **`skipLibCheck`**: `skipLibCheck` 옵션은 코드에서 가져오기로 참조되지 않은 `.d.ts` 파일 검증을 건너뜁니다.
- **`strict`**: `strict` 플래그는 널 가능 타입 처리 보장, TypeScript가 `any` 로 대체할 때 경고 등 여러 검사를 활성화합니다.
- **`exclude`**: 최상위 `exclude` 속성은 컴파일 시 이 TypeScript 파일들을 무시하도록 컴파일러에 지시합니다.
