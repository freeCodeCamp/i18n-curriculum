# --description--

소프트웨어 팀에서 일할 때는 새로운 제품과 서비스를 설계하고 구축하는 프로세스를 갖추는 것이 중요합니다. 소프트웨어 개발 생명 주기(SDLC)는 소프트웨어 팀이 애플리케이션을 구축하고 배포할 때 일반적으로 사용하는 방법입니다.

이 수업에서는 소프트웨어 개발 생명 주기의 여러 단계를 나누어 살펴보고, 그 과정에서 테스트가 어떤 역할을 하는지 알아보겠습니다.

소프트웨어 개발 생명 주기에서 사용되는 표준 단계는 다음과 같습니다:

- 기획 단계

- 설계 단계

- 구현 단계

- 테스트 단계

- 배포 단계

- 유지 관리 단계

이제 각 단계를 좀 더 자세히 살펴보겠습니다.

기획 단계에서는 개발 팀이 이해관계자로부터 제안된 작업에 대한 요구사항을 수집합니다. 이해관계자는 제품이나 서비스의 성공에 이해관계가 있는 사람을 말합니다. 이해관계자의 예로는 투자자, 소유자, 또는 다른 직원들이 있을 수 있습니다. 기획 단계에는 작업 비용을 추정하고 작업 완료 일정을 만드는 것도 포함됩니다.

다음 단계는 설계 단계입니다. 이 단계에서 소프트웨어 팀은 요구사항을 세분화하고 최적의 해결책을 결정합니다. 여기에는 작업을 돕기 위해 사용할 수 있는 외부 도구와 작업을 완료하기 위해 구축해야 할 내부 도구를 결정하는 것도 포함됩니다.

설계 단계가 끝나면 구현 단계가 이어집니다. 이 단계에서는 소프트웨어 팀이 요구사항을 관리 가능한 작업으로 나누어 각자 분담하여 구축합니다.

구현 단계가 완료되면 소프트웨어 팀은 테스트 단계로 넘어갑니다. 이 단계에서는 수동 및 자동 테스트를 통해 새 작업을 검증합니다. 때로는 개발 단계 전반에 걸쳐 애플리케이션을 테스트하여 발생하는 문제를 조기에 발견하고 수정하기도 합니다.

다음 단계는 배포 단계입니다. 이 단계에서는 팀이 새 변경 사항을 빌드 또는 테스트 환경에 배포합니다. 이를 통해 팀은 애플리케이션을 추가로 테스트하고 변경 사항이 운영 환경에 반영되기 전에 문제를 수정할 수 있습니다. 운영 애플리케이션은 최종 사용자가 사용하는 앱입니다.

마지막 단계는 유지 관리 단계입니다. 이 단계에서는 운영 애플리케이션에서 고객에게서 발생하는 문제를 수정합니다. 또한 소프트웨어 개선을 위해 시스템을 모니터링하는 작업도 포함됩니다.

마지막으로, 업계에서는 워터폴과 애자일 같은 다양한 소프트웨어 개발 생명 주기 모델을 사용합니다. 워터폴 모델은 생명 주기의 각 단계를 완료해야 다음 단계로 넘어갈 수 있는 방식입니다. 반면 애자일 모델은 작업을 스프린트로 나누어 반복적으로 개발하며, 팀이 빠르게 변경하고 작업을 점진적으로 전달하는 데 포커스를 둡니다.

소프트웨어 개발 생명 주기를 거치면서 개발 팀과 이해관계자는 요구사항, 비용 추정, 일정에 대해 같은 인식을 공유할 수 있습니다. 이는 전반적으로 더 효율적인 협업과 소통을 가능하게 합니다.
