# --description--

CSP（Content Security Policy）とPermissions-Policyは、ウェブアプリケーションを保護するために、読み込めるリソースやページがアクセスできるブラウザ機能を制限する強力なHTTPセキュリティヘッダーです。

Content Security Policyは主にリソースの読み込みに焦点を当てています。フロントエンドのスクリプト、スタイル、画像の安全フィルターとして機能します。

悪意のある攻撃者はしばしば脆弱なウェブサイトに有害なスクリプトを注入しようとします。強力なCSPは、信頼できるソースドメインの厳格なホワイトリストを設定することで、クロスサイトスクリプティング（XSS）攻撃を防ぎます。

サーバーがCSPヘッダーを送信すると、ブラウザは特定のディレクティブを読み取ります。例えば、`script-src`ディレクティブは、どのドメインがページ上で安全にJavaScriptを実行できるかをブラウザに伝えます。

ハッカーが承認されていない外部サーバーから悪意のあるスクリプトを注入しようとすると、ブラウザはポリシー違反を検出し、即座にスクリプトを破棄し、セキュリティエラーを記録します。

CSPは画像のソース、フォームの送信先、スタイルシートの起源も制限できます。

Permissions-Policyはブラウザ機能とデバイスハードウェアに焦点を当てています。これは旧Feature-Policyヘッダーに代わるものです。

このセキュリティ機構により、サイト所有者は自分のページや埋め込まれたiframeの特定のブラウザ機能を明示的に有効または無効にでき、第三者のスニペットによるユーザープライバシーの悪用を防げます。

Permissions-Policyヘッダーを使うと、ウェブカメラ、マイク、位置情報追跡などのハードウェアへのアクセスを制御できます。

例えば、ウェブサイトがユーザーの位置情報を必要としない場合、ポリシーを設定して位置情報APIを完全にブロックできます。たとえ攻撃者がページ上のスクリプトを侵害しても、ブラウザは追跡機能の起動を拒否します。

Permissions-Policyの実装は、機能名と許可された起源を定義する明確で構造化されたヘッダーフォーマットを使用しています。

ハードウェア機能を自分のドメインのみに厳密に制限するには、丸括弧で囲んだ`self`キーワードを使います。例えば`camera=(self)`です。全ユーザーのページ全体で機能を無効にしたい場合は、空のリストを使い、`geolocation=()`のように書きます。

両方のヘッダーを組み合わせることで、ウェブアプリケーションの堅牢な防御が実現します。CSPはデータ配信を制限しコード注入を防ぎ、Permissions-Policyは物理デバイスを守りプライバシー遵守を保証します。

現代のウェブ開発者は両方のツールを使って、非常に強固な閲覧環境を作り出しています。
