# --description--

In einer vorherigen Lektion wurden Sie erstmals mit Zusicherungen und Unit-Testing vertraut gemacht. Zur Wiederholung: Unit-Testing bedeutet, dass Sie die Ausgabe einzelner Funktionen oder Komponenten unabhängig voneinander überprüfen, im Gegensatz zum Testen der Endausgabe des gesamten Programms. Zusicherungen werden verwendet, um zu testen, ob der Code wie erwartet funktioniert.

In dieser Lektion werfen wir einen genaueren Blick auf Zusicherungen in Unit-Tests und erkunden einige bewährte Vorgehensweisen.

Eine häufig verwendete Zusicherungsbibliothek ist Chai. Diese Bibliothek wird auch im freeCodeCamp-Code verwendet. Immer wenn Sie einen Workshop oder ein Labor durchlaufen, läuft eine automatisierte Testsuite gegen Ihren Code, um sicherzustellen, dass Ihr Code so funktioniert, wie er soll.

Schauen wir uns einige Beispiele aus dem Full-Stack-Developer-Curriculum an.

Im Workshop Build a Calculator ist Schritt 1 das Testen einer Funktion namens `addTwoAndSeven`:

```js
assert.isFunction(addTwoAndSeven);
```

Diese Zusicherung prüft, ob der bereitgestellte Code eine Funktion mit dem Namen `addTwoAndSeven` enthält. Wenn der Benutzer ein Array, einen String oder etwas anderes erstellt, das keine Funktion ist, und es `addTwoAndSeven` zuweist, schlägt der Test fehl.

Schauen wir uns eine weitere Zusicherung aus einem späteren Schritt im Workshop an:

```js
assert.equal(addThreeAndFour(), 7);
```

Diese Zusicherung prüft, ob der Rückgabewert der Funktion `addThreeAndFour` gleich der Zahl `7` ist. Wenn der Rückgabewert nicht `7` entspricht, schlägt der Test fehl.

In Chai gibt es viele weitere Zusicherungen, zum Beispiel solche, die prüfen, ob ein Wert ein Boolean ist, ob ein Objekt eine bestimmte Eigenschaft hat oder ob ein Wert `null` ist. Andere gängige JavaScript-Zusicherungsbibliotheken sind should.js und expect.js.

Unabhängig davon, mit welcher Zusicherungsbibliothek Sie arbeiten, gibt es einige bewährte Vorgehensweisen, die Sie beachten sollten. Die erste ist, klare Zusicherungsnachrichten zu schreiben. Klare und aussagekräftige Zusicherungsnachrichten sollten die Absicht dahinter beschreiben. Andernfalls wird es schwierig, herauszufinden, welche Tests fehlschlagen.

Eine weitere bewährte Vorgehensweise ist, beschreibende Fehlermeldungen zu schreiben. Eine beschreibende Fehlermeldung sollte angeben, welche Bedingung wahr ist. Das hilft Ihnen dabei, herauszufinden, warum eine bestimmte Zusicherung fehlschlägt.

Während Sie weiter programmieren lernen, ist es wichtig, sich mit Tests vertraut zu machen und zu beginnen, Tests für Ihre persönlichen Projekte zu schreiben. Das wird Ihnen sehr dabei helfen, sich auf Ihren ersten Job als Entwickler vorzubereiten.
