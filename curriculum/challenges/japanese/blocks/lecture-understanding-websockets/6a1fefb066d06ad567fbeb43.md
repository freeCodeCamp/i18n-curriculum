# --description--

WebSocketはクライアントとサーバーがリアルタイムで通信できる仕組みです。

通常、ウェブやモバイルのクライアント側アプリはHTTPを使います。HTTPではクライアントがリクエストを送り、サーバーがレスポンスを返します。これはリクエストとレスポンスのサイクルと呼ばれ、これまでに学んだ通りです。

接続が確立されてデータが交換されると、その接続は閉じられます。これはページの読み込みやデータの取得・送信など、多くの用途でうまく機能します。しかし、リアルタイムアプリケーションには遅くて信頼性が低くなります。

例えばチャットアプリを考えてみてください。HTTPを使う場合、クライアントはサーバーに定期的にデータを問い合わせる必要があり、メッセージごとに接続が開かれ、データが届き、接続が閉じられます。

この問題を解決するのがWebSocketです。WebSocketはクライアントとサーバーの間に常時接続を作ります。一度接続が開かれると、どちらからでも閉じる信号が送られるまで、リアルタイムのチャネルでいつでもデータを交換できます。

WebSocketはHTTPとは異なりますが、すべてのWebSocket接続は最初にHTTPで始まります。これをハンドシェイクと呼びます。

クライアントは接続をWebSocketにアップグレードするようリクエストします。サーバーが同意すると、`101 Switching Protocols`レスポンスを送り、接続がWebSocketにアップグレードされます。その後は接続が開いたままになり、リアルタイム通信が始まります。

流れは次のようになります：

1. クライアントがHTTPでサーバーに接続する
2. サーバーが接続を受け入れ、WebSocketにアップグレードする
3. 接続が開いたままになる
4. 接続が閉じられるまで、両者がいつでもデータを交換する

この流れは双方向であり、HTTPのようなリクエスト・レスポンスではありません。

```md
Client <---> Server
```

HTTPでは通常、リクエスト・レスポンスモデルに従います：

```md
Client ---> Request
Server ---> Response
```

WebSocketとHTTPの違いを例えると、HTTPは手紙のやり取りのようなもので、手紙を書いて返事を待ち、会話が終わります。一方WebSocketは電話のように、リアルタイムで話し合うことができます。

WebSocketはチャットアプリ、オンラインマルチプレイヤーゲーム、ライブスポーツの更新、株価ダッシュボード、Google Docsのような共同作業ツールなど、リアルタイムアプリを支えています。
