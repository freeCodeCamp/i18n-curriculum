# --description--

**REST** 是代表性狀態轉移的縮寫。可以將它視為一組指導方針，描述網頁客戶端與伺服端應如何透過網際網路彼此通訊。

在本課程中，你將學習什麼是 REST、它的關鍵概念，以及它如何運作。

REST 定義了一種使用標準網路技術讓不同系統互動的方式。使用 REST，像是網頁瀏覽器、行動應用程式和 API 等客戶端會使用常見的 HTTP 方法如 `GET`、`POST`、`PUT`、`PATCH` 和 `DELETE` 與伺服器溝通，這些方法我們在先前的課程中已經討論過。

REST 的一些核心概念包括 **無狀態性**、**資源**、**HTTP methods** 和 **回應或狀態碼**。

**無狀態性** 意味著客戶端傳送到伺服端的每個請求都是獨立控制的。伺服端不會記住先前的請求。因此，每個請求必須包含處理該請求所需的所有資訊。 例如，如果使用者已通過驗證，他們的驗證 `token` 必須隨每個請求一起傳送。

在 REST 中，一切皆視為資源。資源是一段可以被存取或修改的資料。每個資源皆由 URL 識別，例如 `/products` 代表所有產品，或 `/products/1` 代表特定產品。

以下是 HTTP 方法、狀態碼的概述，以及 REST 架構運作的逐步範例。

## HTTP 方法

HTTP 方法定義對資源執行的動作：

* `GET`：擷取一個資源
    
* `POST`：創建一個新資源
    
* `PUT / PATCH`：更新現有資源
    
* `DELETE`：移除一個資源
    

## HTTP 狀態碼

在處理請求後，伺服器回應一個 **status code**，用以顯示結果：

* `200` **(OK)**：請求成功
    
* `201` **(已創建)**：已建立一個新資源
    
* `400` **（錯誤的請求）**：客戶端傳送了無效的資料
    
* `404` **（找不到）**：請求的資源不存在
    
* `500` **（內部伺服器錯誤）**：伺服器發生錯誤
    

## REST 的運作方式：逐步範例

### 1\. 客戶端發送請求

* 客戶端使用像是 `GET` 或 `POST` 的方法發送 HTTP 請求
    
* 該請求包含用於識別資源的 URL
    
* 它也可能包含標頭、詮釋資料和請求主體
    

### 2\. 伺服端接收請求

* 伺服端驗證請求的格式和身份驗證（如果需要）
    
* 它從 URL 中識別標的資源
    
* 它根據 HTTP `method` 判斷所請求的動作
    
* 它會檢查權限和存取控制元件
    

### 3\. 伺服端執行動作

* `GET`：擷取該資源
    
* `POST`：創建一個新的資源
    
* `PUT / PATCH`：更新資源
    
* `DELETE`：移除資源
    
* 執行任何其他必要的邏輯
    

### 4\. 伺服端建置回應

* 伺服端會選擇適當的狀態碼（例如，`200`、`201` 或 `404`）
    
* 它會格式化回應 `data`，通常為 **JSON**
    
* 它會添加相關的標頭，例如 `Content-Type` 或 `Cache-Control`
    

### 5\. 伺服端傳送回應

* 回應包含狀態碼、標頭和選用主體
    
* 伺服器上不會儲存任何客戶端狀態（無狀態）
    

### 6\. 客戶端控制代碼回應

* 客戶端檢查狀態碼以判斷成功或失敗
    
* 它會處理伺服器傳回的任何資料
