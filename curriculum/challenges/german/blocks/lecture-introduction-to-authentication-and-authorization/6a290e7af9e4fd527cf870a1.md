# --description--

In der vorherigen Vorlesung haben Sie gelernt, wie ein Backend-Server einen Benutzer während des Logins identifiziert. Aber was passiert, nachdem er sich eingeloggt hat? Wie merkt sich der Server, wer der Benutzer ist, wenn er zu einer neuen Seite navigiert? Hier kommen JSON Web Tokens (JWTs) ins Spiel.

Einfach gesagt ist ein JWT eine digitale Schlüsselkarte, die der Server einem Benutzer nach einem erfolgreichen Login ausstellt. Der Benutzer zeigt diese Schlüsselkarte bei jeder weiteren Anfrage dem Server, um seine Identität zu bestätigen, ohne das Passwort erneut eingeben zu müssen.

Anstatt dass der Server sich jeden aktiven Benutzer in seinem eigenen Speicher merkt, verlagert er die Verantwortung auf den Client und nutzt dafür einen einfachen vierstufigen Prozess:

1. **Login:** Der Benutzer sendet seinen Benutzernamen und sein Passwort oder andere Login-Daten an den Server.

2. **Ausstellen:** Der Server überprüft die Zugangsdaten mit denen, die bei der Anmeldung gespeichert wurden. Sind sie korrekt, generiert der Server ein JWT und sendet es an den Benutzer zurück.

3. **Speichern:** Der Client erhält das JWT und speichert es, typischerweise in einem Browser-Cookie oder im lokalen Speicher.

4. **Anfrage:** Bei jeder zukünftigen Anfrage an eine geschützte Route sendet der Client das JWT automatisch mit. Der Server überprüft das Token und gewährt Zugriff.

Ein JWT sieht aus wie eine lange, chaotische Zeichenkette. Es besteht aus drei klar unterscheidbaren Teilen, die durch zwei Punkte getrennt sind, so:

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

Der erste Teil ist die **Kopfzeile**, der zweite die **Nutzlast**, und der dritte Teil ist die **Signatur**.

Die **Kopfzeile** ist ein kodiertes JSON-Objekt, das dem Server mitteilt, um welchen Token-Typ es sich handelt (JWT) und welcher Hash-Algorithmus zur Sicherung verwendet wurde, zum Beispiel HMAC SHA256.

Die **Nutzlast** ist der mittlere Teil des Tokens. Sie enthält die eigentlichen Daten oder „Claims“ über den Benutzer. Hier speichern Sie Informationen wie die Benutzer-ID, den Benutzernamen oder die Kontorolle wie Admin oder Benutzer.

Ein wichtiger Sicherheitshinweis ist, dass Kopfzeile und Nutzlast nur kodiert (Base64), aber nicht verschlüsselt sind. Das bedeutet, jeder kann sie leicht dekodieren und die darin enthaltenen Daten lesen. Speichern Sie also niemals sensible Daten wie unverschlüsselte Passwörter oder Kreditkartennummern in einer JWT-Nutzlast.

Die **Signatur** ist der letzte und wichtigste Teil des Tokens. Der Server nimmt die kodierte Kopfzeile, die kodierte Nutzlast und vermischt sie mit einem geheimen Schlüssel, der nur dem Server bekannt ist. Dann erzeugt er daraus einen Hash, der die Signatur bildet.

Die Signatur verhindert Manipulationen. Wenn ein Benutzer versucht, sein Token zu dekodieren, seine Rolle von „user“ auf „admin“ zu ändern und es wieder zu kodieren, stimmt die Signatur nicht mehr mit den veränderten Daten überein.

Wenn der Server das veränderte Token erhält, versucht er, die Signatur mit seinem geheimen Schlüssel neu zu erstellen. Da sich die Nutzlast geändert hat, stimmt die berechnete Signatur nicht mit der Signatur des Tokens überein, und der Server lehnt die Anfrage sofort mit dem Statuscode 401 Unauthorized ab.

Kurz gesagt ermöglichen JWTs eine zustandslose Authentifizierung. Der Server muss keine Sitzungsdaten in einer Datenbank speichern, sondern vertraut einfach dem Token, weil die Signatur beweist, dass der Server es selbst erstellt hat.
