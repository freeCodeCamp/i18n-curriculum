# --description--

تحدثنا قليلاً عن استخدام الفروع للعمل على تغييرات معزولة لتجنب تلويث فرع `main` الخاص بك. لكن ماذا يعني ذلك، ولماذا هو مهم؟

إذا قمت بعمل تغييرات مباشرة على فرع `main` الخاص بك، وقبل المستودع الأصلي تغييرات أخرى ليست من عملك، سينتهي بك الأمر إلى ما يسمى "تاريخ متباعد". هذا يعني أن الالتزامات على فرع `main` الخاص بك لم تعد متوافقة مع الالتزامات على فرع `main` في المستودع الأصلي. وهذا يجعل مزامنة التغييرات الأصلية كابوسًا.

لنفترض أنك اتبعت ممارسات جيدة للفروع، ولم تكن هذه مشكلة. إذا تابعت دروسنا السابقة، يجب أن يكون لديك نسخة مستنسخة (fork) من مستودع قمت باستنساخه محليًا، ويجب أن يكون لديك مستودع بعيد upstream يشير إلى المستودع الأصلي.

إذا لم تفعل ذلك، فإليك تذكير سريع. ستحتاج إلى عمل نسخة مستنسخة من مستودع، واستنساخ نسختك محليًا، وتشغيل هذا الأمر:

```sh
git remote add upstream <url>
```

استبدل `<url>` بعنوان SSH للمستودع الأصلي، وليس نسختك المستنسخة.

مع وجود upstream مُعد بشكل صحيح، يمكنك تشغيل سلسلة الأوامر التالية لسحب أحدث التغييرات من upstream:

```sh
git checkout main
git fetch upstream
git merge upstream/main
```

ماذا تفعل هذه الأوامر؟ حسنًا، لقد تعلمت عن `git checkout`، لذا الأمر الأول يضمن أنك على فرع `main` محليًا.

الأمر الثاني، `git fetch upstream`، يأمر Git بجلب أحدث التغييرات الموجودة على المستودع البعيد upstream (وهو المستودع الأصلي). ولكن، على عكس الأمر pull، لن يقوم Git بتحديث فرعك تلقائيًا. هذا مهم لأن فرع `main` المحلي مرتبط بفرع `main` في المستودع الخاص بك (fork) وليس بفرع `main` في المستودع الأصلي! تنفيذ أمر git pull قد يؤدي إلى سلوك غير متوقع.

الأمر الأخير، `git merge upstream/main`، يخبر Git بدمج أحدث التغييرات من فرع `main` في المستودع البعيد upstream إلى فرعك الحالي - وهو فرع `main` المحلي، لأننا قمنا بتحديده.

ماذا لو فشل الدمج؟ ماذا لو قمت بالخطأ بعمل التزام على `main`، ولا يستطيع Git دمج التواريخ المتباعدة؟ حسنًا... بافتراض أنك لم تقدم أي طلبات سحب من فرع `main` الخاص بك، يمكنك إجراء إعادة تعيين صلبة لإجبار Git على تحديث فرعك.

```sh
git reset --hard upstream/main
```

أمر `git reset` يسمح لك بإعادة تعيين الحالة الحالية لفرع. تمرير الخيار `--hard` يخبر Git بإجبار الملفات المحلية لتطابق حالة الفرع. هذا يضمن أن لديك صفحة نظيفة للعمل منها. وتمرير `upstream/main` يخبر Git أنك تريد أن يطابق فرعك الحالي (وهو `main`) فرع `main` في المستودع البعيد upstream. كن حذرًا مع هذا الأمر - أي وقت تقوم فيه بتحديث تاريخ Git بالقوة، قد تحدث أشياء غريبة وغير مرغوبة.

بمجرد تحديث فرع `main` المحلي بشكل صحيح، من المهم دفع هذه الحالة إلى نسختك المستنسخة البعيدة (والتي يجب أن تكون `origin` إذا اتبعت دروسنا). للقيام بذلك، يمكنك تشغيل:

```sh
git push
```

هذا سيدفع الحالة الحالية إلى نسختك المستنسخة البعيدة. لكن إذا اضطررت إلى إجراء إعادة تعيين صلبة لتنظيف التاريخ، فمن الممكن أن يكون لدى نسختك المستنسخة البعيدة أيضًا تاريخ متباعد وقد يفشل الدفع.

يمكنك تصحيح ذلك عن طريق فرض الدفع:

```sh
git push --force
```

**كن حذرًا مع هذا الأمر**. تنفيذ دفع بالقوة يعيد كتابة تاريخ المستودع البعيد، وإذا قمت بشيء خاطئ، يصبح فك تشابك الدفع القسري كابوسًا.

دفع التغييرات المتزامنة إلى نسختك المستنسخة البعيدة مهم لأنه يساعدك على التأكد من أنه يمكنك استنساخ النسخة بأمان مع أحدث التغييرات إذا فقدت المستودع المحلي الحالي.

وبمجرد أن يكون فرع `main` متزامنًا، يمكنك إنشاء فرع جديد من تلك الحالة للبدء في العمل على مساهمة أخرى - مع أحدث نسخة من قاعدة الشفرة الأصلية!
