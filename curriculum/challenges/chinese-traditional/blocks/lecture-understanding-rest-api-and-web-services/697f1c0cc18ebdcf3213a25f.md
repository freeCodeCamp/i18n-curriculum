# --description--

過去，大多數應用程式，無論是大型還是小型，都是以單體架構建置。單體是指包含網頁應用程式中所有內容的一個龐大程式碼庫。

這個單體系統意味著 UI、資料庫 和 API 全部都存在於同一個程式碼庫中。

想像一個像 Amazon 的電子商務商店，產品頁面、結帳系統、付款處理、身份驗證和授權，以及支援它們所有的 API 和資料庫都位於同一個程式碼庫中。

單體架構對於小型應用程式和早期新創公司運作良好。然而，隨著應用程式成長及使用者需求增加，單體架構通常變得較難維護、更新較慢且難以擴展。某一區域的小變更可能需要重新部署整個系統，而擴展某個特性通常意味著必須擴展整個應用程式。

這些挑戰促使了微服務的興起，這是一種架構方法，旨在解決單體系統的限制。

在微服務架構中，一個大型應用程式被拆分成較小且獨立的服務。每個服務負責特定的商務能力，例如產品、訂單、驗證、付款等等。這些服務通常擁有自己的程式碼庫，甚至可能管理自己的資料庫，並且可以由完全不同的團隊負責開發。

這些服務透過 API 或基於 `message` 的系統相互通訊，使它們能夠作為單一應用程式協同運作。

透過這種方法，網頁和行動應用程式都更容易維護、更新和擴展。例如，一個電子商務平台可能會有獨立的服務來處理產品、購物車、使用者、訂單和付款。每個服務都可以獨立建置、部署和擴展，而不會影響其他服務。

微服務提供多項優點，包括更快速的開發、更好的故障隔離，以及更高的彈性。不同的服務可以使用不同的程式語言或框架—例如，搜尋服務可能使用 Java，產品服務可能使用 Node.js，認證服務可能使用 .NET。

然而，微服務也帶來新的挑戰。偵錯變得更複雜，因為問題可能跨越多個服務。部署和基礎建設管理更加困難，且由於服務之間頻繁的通訊，網路額外開銷也增加。
