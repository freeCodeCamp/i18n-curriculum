## Бібліотеки та фреймворки JavaScript

- Бібліотеки та фреймворки JavaScript пропонують швидкі рішення для загальних проблем і пришвидшують розробку, надаючи готовий код.
- Бібліотеки зазвичай більше зосереджені на вирішенні конкретних завдань, таких як маніпулювання DOM, обробка подій або керування AJAX-запитами.
- Декілька прикладів бібліотек JavaScript — це jQuery та React.
- Фреймворки, натомість, забезпечують більш чітку структуру для створення застосунків. Вони часто містять набір правил і конвенцій, яких розробник має дотримуватися.
- Прикладами фреймворків є Angular і Next.js — метафреймворк для React.
- **Односторінкові застосунки** (SPA) — це вебзастосунки, які завантажують одну HTML-сторінку та динамічно оновлюють цю сторінку під час взаємодії користувача із застосунком без повного перезавантаження сторінки.
- SPA використовують JavaScript для керування станом застосунку та відтворення вмісту. Це часто реалізується за допомогою фреймворків, які надають чудові інструменти для створення складних інтерфейсів користувача.
- Деякі проблеми, пов’язані з SPA:

  - Читачі екрана мають труднощі з динамічно оновлюваним вмістом.
  - URL не змінюється, коли користувач переміщується всередині застосунку, що ускладнює створення закладок, повернення назад або поширення конкретних сторінок.
  - Початковий час завантаження може бути повільним, якщо застосунок великий, оскільки всі ресурси потрібно завантажити одразу.
## React

- React — це популярна бібліотека JavaScript для створення користувацьких інтерфейсів і вебзастосунків.
- Основна ідея React полягає у створенні багаторазових UI-компонентів, які можуть оновлюватися та відтворюватися незалежно один від одного під час зміни даних.
- React дозволяє розробникам описувати, як має виглядати інтерфейс користувача залежно від стану застосунку. Далі React оновлює та відтворює потрібні компоненти, коли змінюються дані або стан.
## React-компоненти

- Компоненти — це будівельні блоки застосунків на React, які дозволяють розробникам розбивати складні інтерфейси користувача на менші, керовані частини.
- Інтерфейс користувача описується за допомогою JSX — розширення синтаксису JavaScript, яке дозволяє розробникам писати HTML-подібний код всередині JavaScript.
- Компоненти — це, по суті, JS-функції або класи, які повертають частину інтерфейсу користувача.

Ось приклад простого React-компонента, який відтворює вітальне повідомлення:

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

Щоб використати компонент, просто викличте:

```jsx
  <Greeting />
```
## Імпортування та експорт React компонентів

- React компоненти можна експортувати з одного файлу та імпортувати в інший файл.
- Припустимо, у вас є компонент з назвою `City` у файлі з назвою `City.js`. Ви можете експортувати компонент, використовуючи ключове слово `export`:

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default City;
```

- Щоб імпортувати компонент `City` в інший файл, ви можете використати ключове слово `import`:

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

- Використовується ключове слово `default`, оскільки це експорт за замовчуванням з файлу `City.js`.

- Ви також можете експортувати компонент у тому ж рядку, що й визначення компонента, ось так:

```jsx
export default function City() {
  return <p>New York</p>;
}
```
## Налаштування проєкту React за допомогою Vite

- Інструменти для налаштування проєктів та інтерфейси командного рядка надають швидкий і простий спосіб почати нові проєкти, дозволяючи розробникам зосередитися на написанні коду, а не на налаштуванні.
- Vite — це популярний інструмент для налаштування проєктів, який можна використовувати з React.
- Щоб створити новий проєкт з Vite, ви можете скористатися такою командою у своєму терміналі:

```bash
npm create vite@latest my-react-app -- --template react
```

Ця команда створює новий проєкт React з назвою `my-react-app`, використовуючи шаблон React від Vite. У папці проєкту ви побачите файл `package.json` зі списком залежностей проєкту та команд.

- Щоб запустити проєкт, перейдіть у папку проєкту та виконайте такі команди:

```bash
cd my-react-app # path to the project directory
npm install # installs the dependencies listed in the package.json file
```

- Після встановлення залежностей ви повинні побачити нову папку у своєму проєкті з назвою `node_modules`.

- Папка `node_modules` — це місце, де зберігаються всі пакунки та бібліотеки, необхідні для вашого проєкту.

- Щоб запустити свій проєкт, скористайтеся такою командою:

```bash
npm run dev
```

- Після цього відкрийте браузер і перейдіть за адресою `http://localhost:5173`, щоб побачити, як працює ваш застосунок React.

- Щоб переглянути код стартового шаблону, зайдіть у свій проєкт у папці `src` і ви побачите файл `App.jsx`.
## Передача пропсів у React-компонентах

- У React пропси (скорочено від properties) — це спосіб передавати дані від батьківського компонента до дочірнього. Такий механізм потрібен для створення багаторазових і динамічних елементів інтерфейсу.
- Пропси можуть бути будь-яким значенням JavaScript. Щоб передати пропси від батьківського до дочірнього компонента, додайте пропси як атрибути під час використання дочірнього компонента в JSX батьківського. Ось простий приклад:

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

Ви можете передати кілька пропсів за допомогою оператора розширення `(...)`, попередньо перетворивши їх на об’єкт. Ось приклад:
    
```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

У цьому коді оператор розширення `{...person}` перетворює об’єкт person на окремі пропси, які передаються до компонента Child.
## Умовне відтворення у React

- Умовне відтворення у React дозволяє створювати динамічні інтерфейси користувача. Його використовують, щоб показувати різний вміст залежно від певних умов або стану у вашому застосунку.
- Є кілька способів умовно відтворювати вміст у React. Один із загальних підходів — використовувати тернарний оператор. Ось приклад:

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- Ще один спосіб умовно відтворювати вміст — використовувати логічний оператор AND (`&&`). Це корисно, коли ви хочете відтворити вміст лише за виконання певної умови. Ось приклад:

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

У наведеному вище коді елемент `h1` відтворюється лише тоді, коли об’єкт user має істинне значення.

Ви також можете використати пряму інструкцію `if` таким чином:

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```
## Відтворення списків у React

- Відтворення списків у React — це загальне завдання під час створення користувацьких інтерфейсів.
- Списки можна відтворювати, використовуючи метод масиву JS `map()`, щоб ітерувати над масивом елементів і повертати новий масив JSX-елементів.
- Наприклад, якщо у вас є масив імен, які ви хочете відтворити у вигляді списку, ви можете зробити так:

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- Завжди пам’ятайте надавати унікальний ключ для кожного елемента списку, щоб React міг керувати оновленням і відтворенням ролей. Завдяки цим технікам ви зможете створювати гнучкі, ефективні та динамічні списки у своїх застосунках React.
## Рядкові стилі в React

- Рядкові стилі в React дозволяють застосовувати CSS-стилі безпосередньо до JSX-елементів за допомогою об’єктів JavaScript.
- Щоб застосувати рядкові стилі в React, ви можете використати атрибут style на JSX-елементах. Атрибут style приймає об’єкт, де ключі — це CSS-властивості у верблюжому регістрі, а значення — відповідні значення. Ось приклад:

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

Ви також можете винести стилі в окремий об’єкт і передати його в атрибут `style` таким чином:

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Рядкові стилі підтримують динамічне стилізування, дозволяючи умовно застосовувати стилі на основі пропсів або стану. Ось приклад, як можна умовно застосувати стилі залежно від пропса:

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- У наведеному вище коді стилі `color` та `fontSize` встановлюються умовно залежно від пропса `isImportant`.
## Робота з подіями в React

- **Синтетична система подій**: це спосіб React обробляти події. Вона є обгорткою над нативними подіями, такими як події `click`, `keydown` та `submit`. Обробники подій у React використовують конвенцію іменування у camelCase. (Наприклад, `onClick`, `onSubmit` тощо)

Ось приклад використання атрибута `onClick` для елемента `button` у React:

```jsx
function handleClick() {
  console.log("Button clicked!");
}

<button onClick={handleClick}>Click Me</button>;
```

У React функції-обробники подій зазвичай починаються з префікса `handle`, щоб показати, що вони відповідають за обробку подій, наприклад `handleClick` або `handleSubmit`.

Коли дія користувача викликає подію, React передає об’єкт Synthetic Event у ваш обробник. Цей об’єкт поводиться подібно до нативного об’єкта Event у чистому JavaScript, надаючи такі властивості, як `type`, `target` та `currentTarget`.

Щоб запобігти типовим діям, наприклад, оновленню браузера під час події `onSubmit`, ви можете викликати метод `preventDefault()`:

```jsx
function handleSubmit(event) {
  event.preventDefault();
  console.log("Form submitted!");
}

<form onSubmit={handleSubmit}>
  <input type="text" />
  <button>Submit</button>
</form>;
```

Ви також можете обгорнути функцію-обробник у стрілкову функцію ось так:

```jsx
function handleDelete(id) {
  console.log("Deleting item:", id);
}

<button onClick={() => handleDelete(1)}>Delete Item</button>;
```
## Робота зі станом і хуком `useState`

- **Визначення стану**: У React стан — це об’єкт, який містить дані для компонента. Коли стан оновлюється, компонент буде повторно відтворено. React розглядає стан як незмінний, тобто не слід змінювати його напряму.  
- **Хук `useState()`**: Хук `useState` — це функція, яка дозволяє оголосити змінні стану у функціональних компонентах. Ось базовий синтаксис:

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

У змінній стану ви маєте таке:

- `stateVariable` містить поточне значення стану
- `setStateFunction` (функція-сетер) оновлює змінну стану
- `initialValue` встановлює ініціальний стан

Ось повний приклад для компонента `Counter`:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>

      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```
## Відтворення та компоненти React

- **Визначення**: У React відтворення — це процес, під час якого компоненти з’являються в інтерфейсі користувача (UI), зазвичай у браузері. Процес відтворення складається з трьох етапів: тригер, відтворення та коміт.

Етап тригера відбувається, коли React виявляє, що щось змінилося і інтерфейс користувача (UI) можливо потрібно оновити. Зазвичай ця зміна відбувається через оновлення стану або пропсів.

Після спрацювання тригера React переходить до етапу відтворення. Тут React повторно оцінює ваші компоненти й визначає, що потрібно показати. Для цього React використовує легку копію "справжнього" DOM, яка називається віртуальним DOM. Завдяки віртуальному DOM React може швидко перевірити, що потрібно змінити в компоненті.

Етап коміту — це коли React бере підготовлені зміни з віртуального DOM і застосовує їх до справжнього DOM. Іншими словами, саме на цьому етапі ви бачите фінальний результат на екрані.
## Оновлення об’єктів і масивів у стані

- **Оновлення об’єктів у стані**: Якщо потрібно оновити об’єкт у стані, спочатку створіть новий об’єкт або скопіюйте наявний, а потім встановіть стан для цього нового об’єкта. Будь-який об’єкт, який ви додаєте до стану, слід вважати лише для читання. Ось приклад встановлення імені, віку та міста користувача. Функція `handleChange` використовується для обробки оновлень інформації користувача:

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **Оновлення масивів у стані**: Коли ви оновлюєте масиви у стані, важливо не змінювати масив напряму за допомогою таких методів, як `push()` або `pop()`. Замість цього створюйте новий масив під час оновлення стану:

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

Якщо потрібно видалити елементи з масиву, скористайтеся методом `filter()`, який повертає новий масив після фільтрації всього, що ви хочете видалити:

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```
## Посилання на значення за допомогою refs

- **Атрибут `ref`**: Ви можете отримати доступ до DOM-вузла в React, використовуючи атрибут `ref`. Ось приклад, який демонструє, як використати `ref`, щоб встановити фокус на `input` елементі. Властивість `current` використовується для доступу до поточного значення цього `ref`:

```jsx
import { useRef } from "react";

const Focus = () => {
  const inputRef = useRef(null);

  const handleFocus = () => {
    if (inputRef.current) {
      inputRef.current.focus();
    }
  };

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Enter text" />
      <button onClick={handleFocus}>Focus Input</button>
    </div>
  );
};

export default Focus;
```
## Робота з хуком `useEffect`

- **Хук `useEffect()`**: У React ефект — це будь-яка дія, що відбувається поза процесом відтворення компонента. Тобто все, чим React не керує безпосередньо під час відтворення інтерфейсу. До загальних прикладів належать отримання даних, оновлення заголовка вкладки браузера, читання або запис у локальне сховище браузера, отримання місцезнаходження користувача тощо. Ці операції взаємодіють із зовнішнім світом і відомі як побічні ефекти. React надає хук `useEffect`, щоб ви могли керувати цими побічними ефектами. `useEffect` дозволяє запускати функцію після відтворення або оновлення компонента.

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [dependencies]);
```

Функція ефекту виконується після відтворення компонента, а необов’язковий аргумент `dependencies` керує тим, коли саме виконується ефект.

Зверніть увагу, що `dependencies` може бути масивом «реактивних значень» (стан, пропси, функції, змінні тощо), порожнім масивом або бути повністю відсутнім. Ось як ці варіанти впливають на роботу `useEffect`:

- Якщо `dependencies` — це масив, що містить одне або кілька реактивних значень, ефект виконуватиметься щоразу, коли вони змінюються.

- Якщо `dependencies` — порожній масив, `useEffect` виконується лише один раз, коли компонент вперше відтворюється.

- Якщо ви не вказуєте `dependencies`, ефект виконується щоразу, коли компонент відтворюється або оновлюється.
## Як створювати кастомні хуки

- **Кастомні хуки**: Кастомний хук дозволяє винести повторно використовувану логіку з компонентів, наприклад, отримання даних, керування станом, перемикання та побічні ефекти, як-от відстеження онлайн-статусу. У React усі вбудовані хуки починаються зі слова `use`, тому ваш кастомний хук також має дотримуватися цієї конвенції.

Ось приклад створення хука `useDebounce`:

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```
## Робота з формами в React

- **Керовані поля введення**: Це коли ви зберігаєте значення поля введення у стані та оновлюєте його через `onChange` події. Це дає вам повний контроль над даними форми та дозволяє миттєво виконувати валідацію й умовно відтворювати компоненти.

```jsx
import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **Некеровані поля введення**: Замість обробки вхідних даних через хук `useState`, некеровані поля введення в HTML зберігають свій власний внутрішній стан за допомогою DOM. Оскільки DOM керує значеннями полів введення, вам потрібно отримувати значення полів введення за допомогою `ref`.

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```

- **Серверні дії**: Це функції, які запускаються на сервері, щоб обробляти форми безпосередньо на сервері без необхідності створювати API кінцеві точки. Ось приклад із застосунку Next.js:

```js

}
```

Директива `"use server"` позначає функцію як серверну дію.

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state` — це поточний стан, який повертає дія.

- `actionFunction` параметр — це сама серверна дія.

## Отримання даних у React

- **Варіанти отримання даних**: Існує багато різних способів отримати дані у React. Ви можете використовувати вбудований Fetch API або сторонні інструменти, такі як Axios чи SWR.
- **Загальні змінні стану при отриманні даних**: Незалежно від того, який спосіб отримання даних у React ви оберете, є кілька змінних стану, за якими потрібно стежити. Перша — це самі дані. Друга відстежує, чи дані ще завантажуються. Третя — це змінна стану, яка фіксує будь-які помилки, що можуть виникнути під час процесу отримання даних.

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

Оскільки отримання даних є побічним ефектом, найкраще використовувати `Fetch API` всередині хука `useEffect`.

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

Потім ви можете відтворити повідомлення про завантаження, якщо отримання даних ще не завершено, повідомлення про помилку, якщо сталася помилка під час отримання даних, або результати.

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

Якщо ви хочете використовувати Axios, потрібно спочатку встановити та імпортувати його:

```bash
npm i axios
```

```js
import axios from "axios";
```

Після цього ви можете отримати дані за допомогою `axios.get`:

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);

useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await axios.get(
        "https://jsonplaceholder.typicode.com/users"
      );
      setData(res.data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

Щоб отримати дані за допомогою хука `useSWR`, спочатку потрібно встановити та імпортувати його.

```bash

```

```js

```

```jsx
const fetcher = (url) => fetch(url).then((res) => res.json());

const FetchTodos = () => {
 const { data, error } = useSWR(

 );

 if (!data) {
   return <h2>Loading...</h2>;
 }
 if (error) {
   return <h2>Error: {error.message}</h2>;
 }

 return (
   <>
     <h2>Todos</h2>
     <div>
       {data.map((todo) => (
         <h3 key={todo.id}>{todo.title}</h3>
       ))}

 );
};

export default FetchTodos;
```

## Робота з хуком `useOptimistic`

- **Хук `useOptimistic`**: Цей хук використовується для того, щоб інтерфейс залишався адаптивним під час очікування завершення асинхронної дії у фоновому режимі. Він допомагає керувати "оптимістичними оновленнями" в інтерфейсі — це стратегія, коли ви одразу оновлюєте інтерфейс на основі очікуваного результату дії, наприклад, під час очікування відповіді від сервера.

Ось базовий синтаксис:

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState` — це тимчасовий стан, який оновлюється миттєво для кращого користувацького досвіду.
- `addOptimistic` — це функція, яка застосовує оптимістичне оновлення до фактичної зміни стану.
- `actualState` — це реальне значення стану, яке надходить у результаті дії, наприклад, отримання даних із сервера.
- `updateFunction` — це функція, яка визначає, як має оновлюватися оптимістичний стан під час виклику.

Ось приклад використання хука `useOptimistic` у компоненті `TaskList`:

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**: Це використовується для відтворення частини інтерфейсу та позначення оновлення стану як не термінового переходу. Це дозволяє інтерфейсу залишатися адаптивним під час ресурсомістких оновлень. Ось базовий синтаксис:

```js
startTransition(action)
```

`action` виконує оновлення стану або запускає певну логіку, пов’язану з переходом. Це гарантує, що термінові оновлення інтерфейсу (наприклад, введення тексту чи натискання) не блокуються.
## Робота з `useMemo` Hook

```js
const memoizedSortedArray = useMemo(
  () => expensiveSortFunction(largeArray),

);
```

## Інструменти для управління залежностями

- **Визначення залежності**: У програмному забезпеченні залежність — це коли один компонент або модуль у застосунку покладається на інший для коректної роботи. Залежності є загальними у програмних застосунках, оскільки дозволяють розробникам використовувати готові функції або інструменти, створені іншими. Дві основні залежності, необхідні для проєкту на React, — це пакети `react` та `react-dom`:

```json
"dependencies": {
  "react": "^18.3.1",
  "react-dom": "^18.3.1"
}
```

- **Визначення менеджера пакетів**: Щоб керувати програмними залежностями у проєкті, потрібно використовувати менеджер пакетів. Менеджер пакетів — це інструмент для встановлення, оновлення та видалення залежностей. Багато популярних мов програмування, таких як JavaScript, Python, Ruby та Java, використовують менеджери пакетів. Популярні менеджери пакетів для JavaScript — це npm, Yarn та pnpm.
- **Файл `package.json`**: Це ключовий конфігураційний файл у проєктах, який містить метадані про ваш проєкт, зокрема його назву, версію та залежності. Також у ньому визначаються скрипти, ліцензійна інформація та інші налаштування, які допомагають керувати проєктом і його залежностями.
- **Файл `package-lock.json`**: Цей файл фіксує точні версії всіх пакетів, які використовує ваш проєкт. Коли ви оновлюєте пакет, нові версії також оновлюються у файлі блокування.
- **Папка `node_modules`**: У цій папці міститься фактичний код залежностей, перелічених у вашому файлі `package.json`, включаючи як прямі залежності вашого проєкту, так і залежності цих залежностей.
- **Dev Dependencies**: Це пакети, які використовуються лише під час розробки, а не у продакшені. Прикладом може бути бібліотека для тестування, така як Jest. Ви встановлюєте Jest як dev dependency, оскільки вона потрібна для локального тестування застосунку, але не потрібна для запуску застосунку у продакшені.

## React Router

- **Вступ**: React Router — це стороння бібліотека, яка дозволяє додати маршрутизацію до ваших React-застосунків. Щоб почати, потрібно встановити React Router у вже існуючий React-проєкт ось так:

```bash
npm i react-router
```

Далі, у файлі `main.jsx` або `index.jsx`, потрібно налаштувати структуру маршрутів ось так:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

`path` та `element` використовуються для зв'язку URL та UI-компонентів. У цьому випадку ми налаштовуємо маршрут для домашньої сторінки, який вказує на компонент `App`.

- **Кілька виглядів і налаштування маршрутів**: У більших застосунках часто є кілька виглядів і маршрути налаштовані ось так:

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

Проп `index` у цих прикладах призначений для представлення маршруту за замовчуванням для певного сегмента шляху. Тож компонент `Home` буде показано на кореневому шляху `/`, а компонент `ProductsHome` буде показано на шляху `/products`.

- **Вкладені маршрути**: Ви можете вкладати маршрути один в один, у результаті чого шлях дочірнього маршруту додається до шляху батьківського маршруту.

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

У наведеному вище прикладі шлях для популярних товарів буде products/trending.

- **Динамічні сегменти**: Динамічний сегмент — це коли будь-яка частина шляху URL є динамічною.

```jsx
<Route path=":category" element={<Category />} />
```

У цьому прикладі ми маємо динамічний сегмент з назвою `category`. Коли користувач переходить за URL на кшталт products/brass-instruments, вигляд зміниться на компонент `Category`, і ви зможете динамічно отримати відповідні дані на основі цього сегмента.

- **Хук `useParams`**: Цей хук використовується для доступу до динамічних параметрів із шляху URL.

```js
import { useParams } from "react-router";

export default function Category() {
  let params = useParams();
   {/* Accessing the category param: params.category */}
   {/* rest of code goes here */}
}
```


- **Оптимізація зображень у Next.js**: Компонент `Image` розширює нативний HTML-елемент `img` і дозволяє пришвидшити завантаження сторінки та оптимізувати розмір. Це означає, що зображення завантажуватимуться лише тоді, коли потрапляють у вікно перегляду, а компонент `Image` автоматично підбиратиме зображення відповідного розміру для кожного пристрою.

```jsx
import Image from "next/image";

export default function Page() {
  return (
    <Image src="link-to-image-goes-here" alt="descriptive-title-goes-here" />
  );
}
```
## Prop Drilling

- **Визначення**: Prop drilling — це процес передавання пропсів від батьківського компонента до глибоко вкладених дочірніх елементів, навіть якщо деякі з дочірніх елементів не потребують цих пропсів.
## Керування станом

- **Context API**: Контекст означає, що батьківський компонент надає інформацію дочірнім елементам без необхідності явно передавати її через пропси. `createContext` використовується для створення об’єкта контексту, який представляє контекст, що буде зчитуватися іншими компонентами. `Provider` використовується для надання значень контексту дочірнім елементам.

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**: Redux керує станом, надаючи центральне сховище та суворий контроль над оновленнями стану. Він використовує передбачуваний шаблон з діями, редюсерами та підпрограмним забезпеченням. Дії — це корисне навантаження інформації, яке надсилає дані із вашого застосунку до сховища Redux, часто ініційоване взаємодією користувача. Редюсери — це функції, які визначають, як має змінюватися стан у відповідь на ці дії, забезпечуючи оновлення стану незмінним способом. Підпрограмне забезпечення, своєю чергою, виступає містком між відправленням дій та редюсером, дозволяючи розширювати функціональність Redux (наприклад, логування, обробка асинхронних операцій) без зміни основного потоку.

- **Zustand**: Це рішення для керування станом ідеально підходить для малих і середніх застосунків. Воно працює за допомогою хука `useStore`, щоб напряму отримувати доступ до стану в компонентах і на сторінках. Це дозволяє змінювати й отримувати дані без необхідності у діях, редюсерах чи провайдері.
## Налагодження компонентів React за допомогою React DevTools

- **React Developer Tools**: це розширення для браузера, яке ви можете використовувати у Chrome, Firefox та Edge для перегляду компонентів React і виявлення проблем із продуктивністю. Для Safari потрібно встановити пакет npm `react-devtools`. Після встановлення React DevTools і відкриття застосунку React у браузері відкрийте інструменти розробника браузера, щоб отримати доступ до двох додаткових вкладок для налагодження React — Components і Profiler.
- **Вкладка Components**: ця вкладка відображає кожен компонент у вигляді дерева. Ось що ви можете зробити на цій вкладці:
  - переглядати ієрархію компонентів застосунку
  - перевіряти та змінювати пропси, стани й значення контексту в реальному часі
  - переглядати вихідний код для кожного вибраного компонента
  - виводити дані компонента в консоль
  - переглядати DOM-елементи компонента
- **Вкладка Profiler**: ця вкладка допомагає аналізувати продуктивність компонентів. Ви можете записувати продуктивність компонентів, щоб виявляти зайві повторні відтворення, переглядати тривалість комітів і надалі оптимізувати повільні компоненти.
## React Server Components

- **Визначення**: React Server Components — це React-компоненти, які відтворюються виключно на сервері, надсилаючи клієнту лише фінальний HTML. Це означає, що такі компоненти можуть напряму отримувати доступ до серверних ресурсів і суттєво зменшують обсяг JavaScript, який надсилається у браузер.
## Відмінності між реальною та сприйнятою продуктивністю

- **Сприйнята продуктивність**: Це те, як користувачі сприймають продуктивність вебсайту. Вони оцінюють її з точки зору адаптивності та надійності. Це суб’єктивна оцінка, тому її важко виміряти, але вона дуже важлива, оскільки користувацький досвід визначає успіх або невдачу вебсайту.
- **Реальна продуктивність**: Це об’єктивна та вимірювана продуктивність вебсайту. Її вимірюють за допомогою таких метрик, як час завантаження сторінки, час відповіді сервера та час відтворення. На ці показники впливають численні фактори, пов’язані з мережею та самим кодом.
## Техніки покращення сприйманої продуктивності

- **Ліниве завантаження**: Ця техніка зменшує час початкового завантаження настільки, наскільки це можливо, завантажуючи другорядні ресурси у фоновому режимі.
- **Мінімізація затримок завантаження шрифтів**: Якщо на вашому сайті використовуються кастомні шрифти, варто також намагатися мінімізувати затримки завантаження шрифтів, оскільки це може призвести до мерехтіння або відображення запасного шрифту під час завантаження кастомного шрифту. Рекомендується використовувати запасний шрифт, схожий на кастомний, щоб у разі такої ситуації зміна виглядала менш помітною.
- **Використання індикаторів завантаження**: Відображення індикатора завантаження для тривалого процесу одразу після того, як користувач натискає на елемент, допомагає користувачу відчувати зв’язок і залученість до процесу, завдяки чому час очікування здається коротшим.
## Основні поняття продуктивності

- **Порядок у вихідному коді**: Це стосується того, як HTML-елементи структуровані у документі. Від цього залежить, що завантажується першим, і це може суттєво впливати на продуктивність і доступність.

Деякі найкращі практики щодо порядку у вихідному коді:

- Розміщуйте критичний вміст, наприклад заголовки, навігацію або основний текст, вище у структурі HTML.
- Відкладіть завантаження несуттєвих скриптів, таких як аналітика чи сторонні віджети, щоб вони не блокували відтворення сторінки.
- Використовуйте прогресивне покращення, щоб основний досвід роботи був доступний навіть до завантаження стилів і скриптів. Прогресивне покращення — це підхід до створення вебсайтів і застосунків, який базується на ідеї, що спочатку сторінка має працювати лише з HTML.

Ось приклад хорошого порядку у вихідному коді, з використанням найкращих практик, які ми щойно розглянули:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **Критичний шлях відтворення**: Це послідовність кроків, які браузер виконує, щоб перетворити код на пікселі на екрані.
- **Затримка**: Це час, який потрібен для того, щоб запит пройшов між браузером і сервером. Іншими словами, висока затримка означає повільні сторінки.

Деякі способи зменшення затримки:

- Використовуйте CDN (мережі доставки вмісту), щоб віддавати файли з ближчих локацій.
- Увімкніть стиснення, наприклад Gzip, щоб зменшити розмір файлів.
- Оптимізуйте зображення та використовуйте ледаче завантаження.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```
## Покращення INP

- **Визначення**: INP (Interaction to Next Paint) оцінює загальну адаптивність сторінки, вимірюючи час від взаємодії користувача, наприклад натискання або натискання клавіші, до наступного оновлення відображення браузером. Нижчий INP означає більш адаптивну сторінку.

Ось кілька способів покращити INP:

- Зменшіть навантаження на головний потік, розбиваючи довгі JavaScript-завдання на менші частини.
- Використовуйте `requestIdleCallback()` для неважливих скриптів. Це поставить функцію в чергу для виклику під час простою браузера.
- Відкладіть або ліниво завантажуйте важкі ресурси, про які йшлося раніше.
- Оптимізуйте обробники подій. Якщо ці обробники запускаються занадто часто або виконують важкі операції, це може сповільнити сторінку та збільшити INP. Рішенням для цього є дебаунсінг. Дебаунсінг гарантує, що функція виконується лише після того, як користувач перестає вводити дані протягом короткої затримки — наприклад, 300 мс. Це запобігає зайвим обчисленням і покращує продуктивність.
## Як працює відтворення у браузері

- **Як працює відтворення**: Спочатку браузер аналізує HTML і створює DOM. Далі браузер обробляє CSS, формуючи CSS Object Model, або CSSOM. Це ще одна структура дерева, яка визначає, як мають бути стилізовані елементи. Нарешті, браузер малює пікселі на екрані, відтворюючи кожен елемент на основі розрахованих стилів і макету. На складних сторінках це може включати кілька шарів, які композитуються разом, щоб утворити фінальні візуальні вихідні дані.
## Як продуктивність впливає на сталість

- **Довідкова інформація**: Інтернет відповідає приблизно за 2% світових викидів вуглецю — це стільки ж, скільки й авіаційна галузь! Кожен переданий байт потребує електроенергії — від дата-центрів до пристроїв користувачів. Більші файли та неефективні скрипти означають більше споживання енергії. Вебсайт із високою продуктивністю не лише працює швидше, а й зменшує зайве опрацювання та використання енергії.
## Способи зменшити час завантаження сторінки

- **Оптимізуйте медіаресурси**: Великі зображення та відео часто є причиною повільного завантаження. Оптимізуючи ці ресурси, ви можете суттєво прискорити свій сайт. Це включає стиснення зображень, використання сучасних форматів, таких як WebP, і застосування ледачого завантаження для ресурсів.
- **Використовуйте кешування браузера**: Кешування дозволяє браузерам зберігати частини вашого сайту локально, що зменшує час завантаження для повторних відвідувачів.
- **Мініфікуйте та стискайте файли**: Зменшення розміру ваших файлів може призвести до швидшого скачування. Це включає зменшення розміру передаваних файлів і мініфікацію CSS та JavaScript файлів.
## Покращення "часу до використання"

- **Визначення**: "час до використання" — це інтервал від моменту, коли користувач надсилає запит на сторінку, до моменту, коли він може повноцінно взаємодіяти з нею. Щоб покращити "час до використання", ви можете ліниво завантажувати свій ресурс або мінімізувати ресурси, що блокують відтворення.
## Основні метрики для вимірювання продуктивності

- **First Contentful Paint або FCP**: Вимірює, як швидко перший елемент вмісту — текст чи зображення — з’являється на екрані. Хорошим FCP вважається час менше 1,8 секунди, а поганим — понад 3 секунди. Ви можете перевірити свій FCP за допомогою Chrome DevTools, переглянувши вкладку performance.
- **Total Blocking Time**: Показує, як довго основний потік блокується важкими JavaScript-завданнями. Якщо Total Blocking Time (TBT) високий, користувачі відчувають повільну взаємодію. Щоб покращити TBT, розбивайте довгі завдання та відкладайте несуттєві скрипти.
- **Bounce Rate**: Це відсоток відвідувачів, які залишають сайт, не взаємодіючи з ним. Якщо на вашому сайті високий bounce rate, це може бути через повільне завантаження сторінки.
- **Унікальні користувачі**: Ця метрика відстежує, скільки окремих відвідувачів заходять на ваш сайт. Щоб переглянути bounce rate та унікальних користувачів, ви можете використовувати Google Analytics. Це дозволить вам відстежувати ці метрики та підвищувати залученість.
## Загальні інструменти для вимірювання продуктивності

- **Chrome DevTools**: Chrome DevTools — це вбудований інструмент у Google Chrome, який дозволяє аналізувати та налагоджувати продуктивність у реальному часі. Інструменти розробника показують час завантаження, використання центрального процесора та затримки відтворення. Особливо корисний для вимірювання First Contentful Paint (FCP) — наскільки швидко користувач бачить перший видимий вміст. Якщо ваш сайт здається повільним, інструменти розробника допоможуть знайти вузькі місця.
- **Lighthouse**: Це автоматизований інструмент, який перевіряє продуктивність, SEO та доступність.
- **WebPageTest**: Цей інструмент дозволяє перевірити, як ваш сайт завантажується з різних локацій і пристроїв. Він надає детальний розбір індексу швидкості, загального часу блокування та інших ключових показників продуктивності вашого сайту. Якщо ви хочете дізнатися, як реальні користувачі по всьому світу сприймають ваш сайт, WebPageTest — саме для цього.
- **PageSpeed Insights**: Цей інструмент аналізує ваш сайт і пропонує швидкі покращення як для мобільних пристроїв, так і для комп'ютерів. Він підкаже, що саме сповільнює ваш сайт, і дасть конкретні рекомендації, наприклад, оптимізувати зображення, видалити скрипти, що блокують відтворення, і скоротити час відповіді сервера. PageSpeed Insights — це швидкий і простий спосіб перевірити, як Google оцінює продуктивність вашого сайту.
- **Real User Monitoring**: RUM-інструменти відстежують реальну поведінку користувачів, показуючи, як справжні відвідувачі взаємодіють із вашим сайтом. Популярні RUM-інструменти включають Google Analytics, який відстежує час завантаження сторінки та показник відмов, а також New Relic чи Datadog, які моніторять проблеми продуктивності в реальному часі. Якщо вам потрібні дані саме від реальних користувачів, RUM-інструменти є незамінними.
## Робота з веб-API продуктивності

- **Визначення**: Веб-API продуктивності дозволяють розробникам відстежувати, наскільки ефективно завантажується та реагує вебсторінка, безпосередньо у коді. Ці API дають змогу вимірювати час завантаження сторінки, відстежувати затримки відтворення та взаємодії, а також аналізувати час виконання JavaScript.
- **`performance.now()`**: Цей API надає високоточні часові мітки (у мілісекундах), щоб вимірювати, скільки часу займає завантаження різних частин вашого сайту.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: Цей API надає розподіл кожного етапу завантаження сторінки — від пошуку DNS до `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: Цей API відстежує події продуктивності, такі як зсуви макету, тривалі завдання та взаємодії користувача.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```
## Техніки покращення продуктивності CSS

- **CSS-анімації**: Анімація певних CSS-властивостей, таких як розміри, позиція та макет, запускає процес, який називається «перерахунок потоку», під час якого браузер перераховує позицію та геометрію певних елементів на сторінці. Це вимагає перемальовування, що є ресурсомістким процесом. Тому рекомендується зменшити кількість CSS-анімацій настільки, наскільки це можливо, або принаймні надати користувачу можливість вмикати чи вимикати їх.
## Техніки покращення продуктивності JavaScript

- **Розбиття коду**: Розбивати свій JavaScript-код на модулі, які виконують критичні та некритичні завдання, також корисно. Таким чином ви зможете попередньо завантажити критичні модулі якомога швидше, а некритичні відкласти, щоб відтворити сторінку максимально швидко.
- **Маніпуляції з DOM**: Пам’ятайте, що маніпуляції з DOM — це процес динамічної зміни вмісту сторінки за допомогою JavaScript через взаємодію з Document Object Model (DOM). Маніпулювання DOM є обчислювально затратним. Зменшення кількості маніпуляцій з DOM у вашому JavaScript-коді покращить продуктивність.
## CSS-фреймворки

- **CSS-фреймворки**: CSS-фреймворки можуть пришвидшити ваш робочий процес, створити єдиний візуальний стиль для всього сайту, забезпечити однаковий вигляд дизайну у різних браузерах і допомогти краще організувати ваш CSS-код.
- **Популярні CSS-фреймворки**: Деякі з популярних CSS-фреймворків — це Tailwind CSS, Bootstrap, Materialize та Foundation.
- **Можливі недоліки**:
  - CSS, який надає фреймворк, може конфліктувати з вашим кастомним CSS.
  - Ваш сайт може виглядати схожим на інші сайти, які використовують той самий фреймворк.
  - Великі фреймворки можуть спричиняти проблеми з продуктивністю.
## Два типи CSS-фреймворків

- **CSS-фреймворки з орієнтацією на утиліти**: Такі фреймворки містять невеликі класи зі специфічним призначенням, наприклад, для встановлення полів, відступів або кольору фону. Ви можете призначати ці невеликі класи безпосередньо HTML-елементам за потреби. Tailwind CSS належить до CSS-фреймворків з орієнтацією на утиліти.

Ось приклад використання Tailwind CSS для стилізації кнопки.

```html
<button class="bg-blue-500 text-white font-bold py-2 px-4 rounded-full hover:bg-blue-700">
  Button
</button>
```

- **CSS-фреймворки на основі компонентів**: Такі фреймворки містять готові компоненти з наперед визначеними стилями, які ви можете додати на свій сайт. Компоненти доступні в офіційній документації CSS-фреймворку, і ви можете скопіювати та вставити їх у свій проєкт. Bootstrap належить до CSS-фреймворків на основі компонентів.

Ось приклад використання Bootstrap для створення групи списку. Замість застосування невеликих класів до своїх HTML-елементів ви додасте цілий компонент разом із HTML-структурою.

```html
<div class="card" style="width: 25rem;">
```

## Tailwind CSS

Tailwind — це фреймворк CSS з підходом utility-first. Замість написання кастомних CSS-правил ви створюєте свої дизайни, комбінуючи невеликі утилітарні класи прямо у своєму HTML.

### Утиліти для адаптивного дизайну

Tailwind використовує префікси, такі як `sm:`, `md:` та `lg:`, щоб застосовувати стилі для різних розмірів екрана.

```html
<div class="w-full md:w-1/2 lg:flex-row">
  Responsive layout
</div>
```

### Утиліти гнучкого контейнера

Класи на кшталт `flex`, `flex-col`, `justify-around` та `items-center` дозволяють легко створювати гнучкі макети.

```html
<div class="flex flex-col md:flex-row justify-around items-center">
  <p>Column on small screens</p>
  <p>Row on medium and larger screens</p>
</div>
```

### Утиліти сітки

Tailwind містить утиліти для CSS Grid, такі як `grid`, `grid-cols-1` та `md:grid-cols-3`.

```html
<div class="grid grid-cols-1 md:grid-cols-3 gap-8">
  <div class="bg-gray-100 p-4">Column 1</div>
  <div class="bg-gray-100 p-4">Column 2</div>
  <div class="bg-gray-100 p-4">Column 3</div>
</div>
```

### Утиліти відступів

Утиліти на кшталт `mt-8`, `mx-auto`, `p-4` та `gap-4` допомагають створювати послідовні відступи без написання CSS.

```html
<div class="mt-8 p-4 bg-indigo-600 text-white">
  Spaced content
</div>
```

### Утиліти типографіки

Утиліти на кшталт `uppercase`, `font-bold`, `font-semibold` та `text-4xl` керують виглядом тексту.

Ви можете встановлювати розміри шрифту, які змінюються на точках зупинки, наприклад `text-3xl` та `md:text-5xl`.

```html
<h1 class="text-3xl md:text-5xl font-semibold text-center">
  Responsive Heading
</h1>
```

### Кольори та стани при наведенні

Tailwind надає широку палітру кольорів, наприклад `text-red-700`, `bg-indigo-600` та `bg-gray-100`.

Класи на кшталт `hover:bg-pink-600` спрощують створення інтерактивних ефектів.

```html
<a href="#" class="bg-pink-500 hover:bg-pink-600 text-white px-4 py-2 rounded-md">
  Hover Me
</a>
```

### Кордони, кільця та ефекти

- **Кордони**: `border-2 border-red-300` додає кордони із заданою товщиною та кольорами.
- **Кільця**: `ring-1 ring-gray-300` створює ефекти, схожі на обриси, які часто використовують для фокусу або карток.
- **Округлені кути та масштабування**: класи на кшталт `rounded-md`, `rounded-xl` та `scale-105` додають завершеності.

```html
<div class="p-6 rounded-xl ring-2 ring-fuchsia-500 scale-105">
  Highlighted card
</div>
```

### Градієнти

Tailwind підтримує утиліти для градієнтів, такі як `bg-gradient-to-r from-fuchsia-500 to-indigo-600`.

```html
<div class="p-4 text-white bg-gradient-to-r from-fuchsia-500 to-indigo-600">
  Gradient background
</div>
```
## CSS-препроцесори

- **Функції**: Деякі функції, які можуть надавати CSS-препроцесори, — це змінні, міксини, вкладення та успадкування селекторів.
- **Популярні CSS-препроцесори**: Деякі з популярних CSS-препроцесорів — Sass, Less та Stylus.  
- **Можливі недоліки**:
  - Компіляція CSS-правил у стандартний CSS може створювати додаткове навантаження.
  - Скомпільований код може бути складним для налагодження.

- **Функції, які підтримує Sass**: Sass підтримує такі функції, як змінні, вкладені CSS-правила, модулі, міксини, успадкування та оператори для базових математичних операцій

Ось приклад оголошення та використання змінної у SCSS.

Ось приклад оголошення міксина у синтаксисі SCSS. У цьому випадку міксин називається `center-flex`. Він містить три CSS-властивості для центрування елементів за допомогою гнучкого контейнера.

## Ручне та автоматизоване тестування

- **Ручне тестування**: Під час ручного тестування тестувальник вручну перевіряє кожну частину застосунку та тестує різні функції, щоб переконатися, що все працює правильно. Якщо під час тестування виявляються якісь помилки, тестувальник повідомляє про ці помилки команді розробників, щоб їх можна було виправити.
- **Автоматизоване тестування**: Під час автоматизованого тестування ви можете автоматизувати свої тести, написавши окрему програму, яка перевіряє, чи поводиться ваш застосунок так, як очікується.
## Юніт-тестування

- **Юніт-тестування**: Під час юніт-тестування ви перевіряєте кожну функцію, щоб переконатися, що все працює як очікується. Юніт-тести також можуть слугувати формою документації для вашого застосунку, оскільки вони мають представляти очікувану поведінку для вашого коду.
- **Принцип єдиної відповідальності**: Принцип єдиної відповідальності рекомендує робити кожну функцію невеликою та відповідальною лише за одну задачу.
- **Загальні JavaScript-фреймворки для тестування**: Деякі загальні фреймворки для тестування включають Jest, Mocha та Vitest. Jest — популярний фреймворк для юніт-тестів.

Ось приклад юніт-тестів із використанням Jest.

Спочатку ви можете створити функцію, яка відповідає за повернення новоствореного відформатованого рядка:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

В окремому `getFormattedWord.test.js` файлі ви можете написати кілька тестів, щоб перевірити, чи функція працює як очікується. Файл `getFormattedWord.test.js` виглядатиме так:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **Функція `expect`**: Функція `expect` використовується для тестування значення.
- **Matcher**: Matcher — це функція, яка перевіряє, чи поводиться значення як очікується. У наведеному вище прикладі matcher — це `toBe()`. У Jest є різноманітні matchers.

Щоб використовувати Jest, спочатку потрібно встановити пакет `jest` за допомогою `npm i jest`. Також потрібно додати скрипт у свій `package.json` файл ось так:

```js
"scripts": {
  "test": "jest"
},
```

Потім ви можете виконати команду `npm run test`, щоб запустити свої тести.
## Життєвий цикл розробки програмного забезпечення

- **Різні етапи життєвого циклу розробки програмного забезпечення**:
  - **Етап планування**: Команда розробки збирає вимоги до запланованої роботи від зацікавлених сторін.
  - **Етап проєктування**: Команда програмістів деталізує вимоги та визначає найкращі підходи до їх вирішення.
  - **Етап реалізації**: Команда програмістів розбиває вимоги на керовані завдання та реалізує їх.
  - **Етап тестування**: Це включає ручне та автоматизоване тестування нової роботи. Іноді команда тестує застосунок протягом усього етапу розробки, щоб виявити й виправити будь-які проблеми, які виникають.
  - **Етап розгортання**: Команда розгортає нові зміни у середовищі збірки або тестування.
  - **Етап обслуговування**: Це включає виправлення будь-яких проблем, які виникають у клієнтів у робочому застосунку.

- **Різні моделі життєвого циклу розробки програмного забезпечення**:
  - **Каскадна модель**: Каскадна модель передбачає, що кожна фаза життєвого циклу має бути завершена перед початком наступної.
  - **Agile-модель**: Agile-модель фокусується на ітеративній розробці, розбиваючи роботу на спринти.
## BDD та TDD

- **TDD**: Розробка через тестування — це методологія, яка акцентує увагу на написанні тестів у першу чергу. Написання тестів до створення функцій забезпечує розробнику зворотний зв'язок у реальному часі під час розробки.
- **BDD**: Розробка, орієнтована на поведінку, — це підхід, що полягає у приведенні серії тестів у відповідність до бізнес-цілей. Сценарії тестування у BDD мають бути написані мовою, зрозумілою як технічним, так і нетехнічним людям. Прикладом такого синтаксису є Gherkin.
- **BDD-фреймворки для тестування**: Прикладами BDD-фреймворків для тестування є Cucumber, JBehave та SpecFlow.
## Твердження в модульному тестуванні

- **Твердження**: Твердження використовуються для перевірки того, що код поводиться так, як очікується.
- **Бібліотеки тверджень**: Chai — це часто використовувана бібліотека тверджень. Інші загальні бібліотеки тверджень для JavaScript — `should.js` і `expect.js`.

Ось приклад твердження з використанням Chai, яке перевіряє, що повернене значення з функції `addThreeAndFour` дорівнює числу 7:

```js
assert.equal(addThreeAndFour(), 7);
```

- **Найкращі практики**: Незалежно від того, яку бібліотеку тверджень ви використовуєте, слід писати зрозумілі повідомлення про твердження та помилки, які допоможуть зрозуміти, які тести не проходять і чому.
## Мокінг, фейкінг і стабінг

- **Мокінг**: Мокінг — це процес заміни реальних даних фальшивими, які імітують поведінку справжніх компонентів. Наприклад, ви можете змокати відповідь API під час тестування замість того, щоб постійно робити виклики API для отримання даних.
- **Стабінг**: Стаби — це об’єкти, які повертають наперед визначені відповіді або тестові дані для очікуваної поведінки у застосунку. Наприклад, ви можете застабити поведінку підключення до бази даних у своїх тестах, не покладаючись на справжнє підключення до бази даних.
- **Фейкінг**: Фейки — це спрощені версії справжніх компонентів без складності чи побічних ефектів. Наприклад, ви можете зімітувати базу даних, зберігаючи дані в пам’яті замість взаємодії зі справжньою базою даних. Це дозволить вам імітувати операції з базою даних у пам’яті, що буде значно швидше, ніж робота зі справжньою базою даних.
## Функціональне тестування

- **Функціональне тестування**: Функціональне тестування перевіряє, чи функції та можливості застосунку працюють так, як очікується. Мета функціонального тестування — перевірити систему в цілому на відповідність різним сценаріям.
- **Нефункціональне тестування**: Нефункціональне тестування зосереджується на таких аспектах, як продуктивність і надійність.
- **Димове тестування**: Димове тестування — це попередня перевірка системи на базові або критичні проблеми перед початком більш масштабного тестування.
## End-to-End тестування

- **End-to-End тестування**: End-to-End тестування, або E2E, перевіряє сценарії з реального життя з точки зору користувача. End-to-End тести допомагають переконатися, що ваш застосунок поводиться правильно та передбачувано для користувачів. Однак налаштування, проєктування та підтримка таких тестів займає багато часу.
- **Фреймворки для End-to-End тестування**: Playwright — популярний фреймворк для End-to-End тестування, розроблений Microsoft. Інші приклади інструментів для End-to-End тестування: Cypress, Selenium та Puppeteer.

Ось приклад E2E тестів із кодової бази freeCodeCamp, які використовують Playwright. Хук `beforeEach` буде виконуватися перед кожним із тестів. Тести перевіряють, що донор має посилання для підтримки в меню, а також спеціальний стилізований кордон навколо свого аватара:

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## Тестування зручності використання

- **Тестування зручності використання**: Тестування зручності використання — це коли справжні користувачі взаємодіють із застосунком, щоб виявити, чи є якісь проблеми з дизайном, користувацьким досвідом або функціональністю у застосунку. Тестування зручності використання зосереджене на тому, наскільки інтуїтивно зрозумілий застосунок для користувачів.
- **Чотири загальні типи тестування зручності використання**:
  - **Дослідницьке**: Дослідницьке тестування зручності використання передбачає, що користувачі взаємодіють із різними функціями застосунку, щоб краще зрозуміти, як вони працюють.
  - **Порівняльне**: Порівняльне тестування — це коли ви порівнюєте користувацький досвід свого застосунку з подібними застосунками на ринку.
  - **Оцінювальне**: Оцінювальне тестування — це коли ви досліджуєте, наскільки інтуїтивно зрозуміло використовувати застосунок.
  - **Валідаційне**: Валідаційне тестування — це коли ви визначаєте будь-які серйозні проблеми, які завадять користувачу ефективно використовувати застосунок.
- **Інструменти для тестування зручності використання**: Прикладами інструментів для тестування зручності використання є Loop11, Maze, Userbrain, UserTesting та UXTweak.
## Тестування сумісності

- **Тестування сумісності**: Мета тестування сумісності — переконатися, що ваш застосунок працює в різних обчислювальних середовищах.
- **Різновиди тестування сумісності**:
  - **Зворотна сумісність**: Зворотна сумісність означає, що програмне забезпечення сумісне з попередніми версіями.
  - **Пряма сумісність**: Пряма сумісність означає, що програмне забезпечення та системи працюватимуть із майбутніми версіями.
  - **Сумісність із апаратним забезпеченням**: Сумісність із апаратним забезпеченням — це здатність програмного забезпечення правильно працювати в різних апаратних конфігураціях.
  - **Сумісність із операційними системами**: Сумісність із операційними системами — це здатність програмного забезпечення працювати на різних операційних системах, таких як macOS, Windows і дистрибутивах Linux, наприклад Ubuntu та Fedora.
  - **Сумісність із мережею**: Сумісність із мережею означає, що програмне забезпечення може працювати в різних мережевих умовах, таких як різна швидкість мережі, протоколи, налаштування безпеки тощо.
  - **Сумісність із браузерами**: Сумісність із браузерами означає, що вебзастосунок може стабільно працювати в різних браузерах, таких як Google Chrome, Safari, Firefox тощо.
  - **Сумісність із мобільними пристроями**: Важливо переконатися, що ваші програмні застосунки працюють на різноманітних пристроях Android та iOS, включаючи телефони й планшети.
## Тестування продуктивності

- **Тестування продуктивності**: Під час тестування продуктивності ви перевіряєте швидкість, чутливість, масштабованість і стабільність застосунку під різними робочими навантаженнями. Мета — усунути будь-які вузькі місця продуктивності.
- **Різновиди тестування продуктивності**:
  - **Тестування навантаження**: Тестування навантаження визначає, як система поводиться під час звичайного та пікового навантаження.
  - **Стрес-тестування**: Стрес-тестування — це коли ви перевіряєте свій застосунок під екстремальними навантаженнями й оцінюєте, наскільки добре система реагує на підвищене навантаження.
  - **Тестування на витривалість (Soak Testing, Endurance Testing)**: Тестування на витривалість або soak testing — це різновид тестування навантаження, коли ви перевіряєте систему під підвищеним навантаженням протягом тривалого часу.
  - **Spike-тестування**: Spike-тестування — це коли ви різко збільшуєте й зменшуєте навантаження та аналізуєте реакцію системи на ці зміни.
  - **Тестування точки зупинки (тестування на місткість)**: Тестування точки зупинки або тестування на місткість — це коли ви поступово збільшуєте навантаження з часом до того моменту, коли система починає відмовляти або деградувати.
## Тестування безпеки

- **Тестування безпеки**: Тестування безпеки допомагає виявити вразливості та слабкі місця.
- **Принципи безпеки**:
  - **Конфіденційність**: Захищає від розголошення конфіденційної інформації іншим отримувачам, які не є цільовими.
  - **Цілісність**: Передбачає запобігання зміні інформації користувача зловмисниками.
  - **Автентифікація**: Передбачає перевірку особи користувача, щоб переконатися, що він має право використовувати цю систему.
  - **Авторизація**: Це процес визначення, які дії автентифіковані користувачі можуть виконувати або до яких частин системи їм дозволено доступ.
  - **Доступність**: Гарантує, що інформація та сервіси доступні для авторизованих користувачів тоді, коли вони їм потрібні.
  - **Незаперечність**: Гарантує, що і відправник, і отримувач мають докази доставки та підтвердження особи відправника. Захищає від заперечення факту відправлення інформації.
- **Загальні загрози безпеці**:
  - **Міжсайтове скриптування (XSS)**: Атаки XSS відбуваються, коли зловмисник впроваджує шкідливі скрипти у вебсторінку, а потім виконує їх у браузері жертви.
  - **SQL-інʼєкція**: SQL-інʼєкція дозволяє зловмисникам впроваджувати шкідливий код у базу даних.
  - **Атака типу відмова в обслуговуванні (DoS)**: Атака DoS — це коли зловмисники перевантажують вебсайт великою кількістю запитів або трафіку, змушуючи сервер сповільнюватися або навіть виходити з ладу, роблячи сайт недоступним для користувачів.
- **Категорії інструментів тестування безпеки**:
  - **Статичне тестування безпеки застосунків**: Ці інструменти аналізують вихідний код застосунку для виявлення вразливостей безпеки.
  - **Динамічне тестування безпеки застосунків**: Ці інструменти взаємодіють із фронтендом застосунку, щоб виявити потенційні слабкі місця безпеки. DAST-інструменти не мають доступу до вихідного коду.
- **Тестування на проникнення (pentest)**: Тестування на проникнення — це тип тестування безпеки, який передбачає моделювання кібератак на застосунок для виявлення вразливостей у системі.
## A/B тестування

- **A/B тестування**: A/B тестування полягає в порівнянні двох версій сторінки або застосунку та дослідженні, яка версія працює краще. Його також називають bucket або split тестуванням. A/B тестування дозволяє приймати більш обґрунтовані рішення на основі даних і постійно покращувати досвід користувачів.
- **Інструменти для A/B тестування**: Прикладами інструментів, які можна використовувати для A/B тестування, є GrowthBook та LaunchDarkly.
## Альфа- та бета-тестування

Після завершення початкової розробки та тестування програмного забезпечення важливо, щоб застосунок протестували тестувальники та реальні користувачі. Саме тут і з’являються альфа- та бета-тестування.

- **Альфа-тестування**: Альфа-тестування проводиться обраною групою тестувальників, які проходять застосунок, щоб переконатися, що немає помилок перед випуском на ринок. Альфа-тестування є частиною приймального тестування та використовує як техніки "white box", так і "black box".
- **Бета-тестування**: Бета-тестування — це коли застосунок стає доступним для реальних користувачів. Користувачі можуть взаємодіяти із застосунком і надавати відгуки. Бета-тестування також є формою приймального тестування користувачів.
- **Приймальне тестування**: Приймальне тестування гарантує, що програмний застосунок відповідає бізнес-вимогам і потребам користувачів перед випуском.
- **Black box тестування**: Black box тестування зосереджується лише на очікуваній поведінці застосунку.
- **White box тестування**: White box тестування передбачає, що тестувальник знає внутрішні компоненти та проводить тестування саме над ними.
## Регресійне тестування

- **Регресія**: Регресія означає ситуації, коли нові зміни ненавмисно ламають вже існуючу функціональність.
- **Регресійне тестування**: Регресійне тестування допомагає виявити проблеми регресії. Під час регресійного тестування ви повторно запускаєте функціональні тести для частин свого застосунку, щоб переконатися, що все працює як очікується.
- **Інструменти для регресійного тестування**: Інструменти, які можна використовувати для проведення регресійного тестування, включають Puppeteer, Playwright, Selenium та Cypress.
- **Техніки регресійного тестування**:
  - **Модульне регресійне тестування**: Це коли у вас є список елементів, які потрібно протестувати щоразу, коли у застосунок впроваджуються суттєві зміни або виправлення.
  - **Часткове регресійне тестування**: Це передбачає цільові підходи, щоб переконатися, що нові зміни не зламали певні аспекти застосунку.
  - **Повне регресійне тестування**: Це запуск тестів для всієї функціональності у кодовій базі. Це найбільш трудомісткий і детальний варіант.
- **Повторне тестування**: Повторне тестування використовується для перевірки відомих проблем і впевненості, що вони були вирішені. На відміну від цього, регресійне тестування шукає невідомі проблеми, які могли виникнути через нещодавні зміни у кодовій базі.
## Що таке TypeScript

- **JavaScript**: JavaScript — це мова з динамічною типізацією. Це означає, що змінні можуть отримувати будь-які значення під час виконання. Проблема динамічно типізованої мови полягає в тому, що відсутність перевірки типів може призводити до помилок.

Наприклад, навіть якщо ваша функція JavaScript очікує масив, ви все одно можете викликати її з числом:

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

Вихідні дані `console` для наведеного вище прикладу будуть `undefined`.

- **TypeScript**: TypeScript розширює мову JavaScript, додаючи статичну типізацію, що допомагає виявляти помилки, спричинені невідповідністю типів, ще до запуску вашого коду.

Наприклад, ви можете визначити тип для параметра `array` так:

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

Це визначення типу повідомляє TypeScript, що параметр `array` має бути масивом рядків. Тоді, коли ви викликаєте `getRandomValue` і передаєте йому число, ви отримаєте помилку, яка називається помилкою компілятора.

- **Компілятор**: Спочатку потрібно скомпілювати код TypeScript у звичайний JavaScript. Коли ви запускаєте компілятор, TypeScript перевіряє ваш код і видає помилку для будь-яких проблем, де типи не збігаються.
## Типи даних у TypeScript

- **Примітивні типи даних у TypeScript**: Для примітивних типів даних `string`, `null`, `undefined`, `number`, `boolean` та `bigint` TypeScript пропонує відповідні ключові слова типів.

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **Масив**: Ви можете визначити масив певного типу двома різними синтаксисами.

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **Об’єкти**: Ви можете визначити точну структуру об’єкта.

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

Якщо вам потрібен об’єкт з будь-якими ключами, але всі значення повинні бути рядками, є два способи це визначити:

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **Інші корисні типи у TypeScript**:
  - **`any`**: `any` означає, що значення може мати будь-який тип. Це вказує компілятору ігнорувати тип цієї змінної.
  - **`unknown`**: `unknown` повідомляє TypeScript, що ви _дбаєте_ про тип значення, але насправді не знаєте, який саме. `unknown` зазвичай рекомендується використовувати замість `any`.
  - **`void`**: Це спеціальний тип, який ви зазвичай використовуватимете лише при визначенні функцій. Функції, які не мають поверненого значення, використовують тип повернення `void`.
  - **`never`**: Він представляє тип, який ніколи не існуватиме.

- **Ключове слово `type`**: Це ключове слово схоже на `const`, але замість оголошення змінної ви можете оголосити тип.

Воно корисне для оголошення кастомних типів, таких як об’єднання типів або типів, що містять лише певні значення:

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**: Інтерфейси схожі на класи для типів. Вони можуть реалізовувати або розширювати інші інтерфейси, є виключно об’єктними типами та зазвичай рекомендуються, якщо вам не потрібна конкретна функція, яку пропонує оголошення `type`.

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **Визначення типу повернення**: Ви також можете визначити _тип повернення_ функції.

У наведеному нижче прикладі тип поверненого значення визначено як рядок. Якщо ви спробуєте повернути щось інше, TypeScript видасть помилку компіляції.

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```
## Узагальнення (generics)

- **Визначення узагальненого типу**: Ви можете визначити узагальнений тип і посилатися на нього у своїй функції. Це можна уявити як спеціальний параметр, який ви передаєте функції, щоб керувати поведінкою визначення типу цієї функції.

Ось приклад визначення узагальненого типу для функції:

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

Синтаксис `<T>` повідомляє TypeScript, що ви визначаєте узагальнений тип `T` для функції. `T` — це загальна назва для узагальнених типів, але ви можете використовувати будь-яку іншу назву.

Далі ви повідомляєте TypeScript, що параметр `array` є масивом значень, які відповідають узагальненому типу, а повернуте значення — це один елемент цього ж типу.

- **Вказування аргументу типу під час виклику функції**: Ви можете передати аргумент типу у виклику функції, використовуючи кутові дужки між назвою функції та її параметрами.

Ось приклад передачі аргументу типу у виклику функції:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Це повідомляє TypeScript, що елемент, який ви очікуєте знайти, буде елементом введення.
## Звуження типу

- **Звуження за істинністю**: У наведеному нижче прикладі ви отримаєте помилку компілятора при спробі отримати доступ до властивості `value` об’єкта `email`, оскільки `email` _може_ бути `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Ви можете використати умовну інструкцію, щоб переконатися, що `email` є _істинним_, перш ніж звертатися до властивості:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

Перевірки на істинність також можуть працювати у зворотному напрямку:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

Викидання помилки завершує логічне виконання цього коду, тобто коли ви дійдете до виклику `console.log()`, TypeScript знатиме, що `email` _не може_ бути `null`.

- **Опціональний ланцюжок**: Опціональний ланцюжок `?.` також є формою звуження типу, оскільки доступ до властивості не відбудеться, якщо значення `email` є `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **Оператор `typeof`**: Ви можете використати умовну перевірку типу змінної за допомогою оператора `typeof`.

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **Ключове слово `instanceof`**: Якщо об’єкт створено з класу, ви можете використати ключове слово `instanceof`, щоб звузити тип.

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **Кастинг**: Коли TypeScript не може автоматично визначити тип значення, наприклад, результату методу `request.json()` у наведеному нижче прикладі, ви отримаєте помилку компілятора. Один зі способів вирішення — кастинг типу, але це послаблює здатність TypeScript виявляти потенційні помилки.

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **Захисник типу**: Замість кастингу типу ви можете написати захисник типу:

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

Синтаксис `user is User` вказує, що ваша функція повертає булеве значення, яке, якщо істинне, означає, що значення `user` відповідає інтерфейсу `User`.
## `tsconfig` файл

- **`tsconfig.json`**: Налаштування компілятора TypeScript зберігаються у `tsconfig.json` файлі в кореневій директорії вашого проєкту.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

Ось описи параметрів компілятора, які використовуються у наведеному вище прикладі:

- **`compilerOptions`**: Властивість `compilerOptions` визначає, як поводиться компілятор TypeScript.
- **`rootDir` та `outDir`**: `rootDir` і `outDir` вказують TypeScript, у якій директорії знаходяться вихідні файли, а також у якій директорії має зберігатися транспільований JavaScript-код.
- **`lib`**: Властивість `lib` визначає, які типи визначень використовує компілятор, і дозволяє додати підтримку певних версій ES, DOM тощо.
- **`module` та `moduleResolution`**: `module` і `moduleResolution` працюють разом, щоб керувати тим, як ваш пакет використовує модулі — або CommonJS, або ECMAScript.
- **`esModuleInterop`**: `esModuleInterop` забезпечує кращу взаємодію між CommonJS та ES-модулями, автоматично створюючи об’єкти простору імен для імпортів.
- **`skipLibCheck`**: Опція `skipLibCheck` пропускає перевірку `.d.ts` файлів, які не імпортуються у вашому коді.
- **`strict`**: Прапорець `strict` увімкнює кілька перевірок, наприклад, правильну обробку nullable-типів і попередження, коли TypeScript повертається до `any`.
- **`exclude`**: Властивість верхнього рівня `exclude` повідомляє компілятору ігнорувати ці TypeScript-файли під час компіляції.
