## Вступ до систем контролю версій

- **Визначення**: Система контролю версій дозволяє відстежувати та керувати змінами у вашому проєкті. Прикладами систем контролю версій, які використовуються у програмному забезпеченні, є Git, SVN або Mercurial.

## Постачальники систем контролю версій на базі хмари

- **Список постачальників систем контролю версій на базі хмари**: GitHub і GitLab — популярні приклади постачальників систем контролю версій на базі хмари, які дозволяють командам розробників співпрацювати та керувати репозиторіями.

## Встановлення та налаштування Git

- **Встановлення Git**: Щоб перевірити, чи Git вже встановлено на вашому комп’ютері, ви можете виконати таку команду в терміналі:

```sh
git --version
```

Якщо ви побачите номер версії, це означає, що Git встановлено. Якщо ні — вам потрібно його встановити.

Для систем Linux Git часто встановлено за замовчуванням у більшості дистрибутивів. Якщо Git не встановлено, ви можете встановити його за допомогою команд вашого менеджера пакетів, таких як `sudo apt-get install git` або `sudo pacman -S git`.

Для користувачів Mac можна встановити Git через Homebrew за допомогою `brew install git`, або завантажити виконуваний інсталятор з офіційного сайту Git.

Для Windows можна завантажити виконуваний інсталятор з сайту Git. Або, якщо у вас налаштовано Chocolatey, ви можете виконати `choco install git.install` у PowerShell. Зверніть увагу, що на Windows також варто завантажити Git Bash, щоб мати середовище оболонки, схоже на Unix.

Щоб переконатися, що встановлення пройшло успішно, знову виконайте команду `git --version` у терміналі.

- **Налаштування Git**: `git config` використовується для встановлення змінних конфігурації, які відповідають за роботу Git на вашому комп’ютері. Щоб переглянути поточні налаштування та де вони зберігаються у вашій системі, виконайте таку команду:

```sh
git config --list --show-origin
```

Зараз ви повинні бачити лише системні налаштування, якщо ви щойно встановили Git.

Щоб встановити своє ім’я користувача, виконайте таку команду:

```sh
git config --global user.name "Jane Doe"
```

Прапорець `--global` використовується тут, щоб встановити ім’я користувача для всіх проєктів на вашій системі, які використовують Git. Якщо потрібно замістити ім’я користувача для конкретного проєкту, виконайте команду у каталозі цього проєкту без прапорця `--global`.

Щоб встановити адресу електронної пошти користувача, виконайте таку команду:

```sh
git config --global user.email janedoe@example.com
```

Ще одне налаштування, яке можна встановити, — це улюблений редактор, який Git буде використовувати. Ось приклад, як встановити Emacs як улюблений редактор:

```sh
git config --global core.editor emacs
```

Якщо ви не встановите улюблений редактор, Git за замовчуванням використовуватиме редактор вашої системи.

## Відкрите та закрите програмне забезпечення

- **Визначення**: «Відкрите програмне забезпечення» означає, що люди можуть бачити код, який ви публікуєте, пропонувати зміни, повідомляти про проблеми і навіть запускати змінену версію. «Закрите програмне забезпечення» означає, що бачити і взаємодіяти з проєктом можуть лише ті, кому ви явно надали доступ.

## GitHub

- **Визначення**: GitHub — це хмарне рішення, яке пропонує зберігання проєктів із контролем версій у так званих «репозиторіях» і надає функції для співпраці з цими проєктами.
- **GitHub CLI**: Цей інструмент використовується для виконання специфічних завдань GitHub без виходу з командного рядка. Якщо він не встановлений, інструкції з встановлення можна знайти в документації GitHub — але зазвичай він доступний у менеджері пакетів вашої системи.
- **GitHub Pages**: GitHub Pages — це опція для розгортання статичних сайтів або застосунків, які не потребують серверної логіки. Тобто застосунки, що працюють повністю на стороні клієнта або у браузері користувача, можна повністю розгорнути на цій платформі.
- **GitHub Actions**: GitHub Actions — це функція, яка дозволяє автоматизувати робочі процеси безпосередньо у вашому репозиторії GitHub, включно зі збіркою, тестуванням і розгортанням коду.

## Загальні команди Git

- **`git init`**: Ініціалізує порожній Git-репозиторій, щоб Git міг почати відстежувати зміни у цьому проєкті. Після ініціалізації у проєкті з’явиться нова прихована папка `.git`. Ця папка `.git` містить важливу інформацію для керування проєктом Git.
- **`git status`**: Ця команда показує поточний стан вашого робочого каталогу — ви будете часто використовувати цю команду у своєму робочому процесі.
- **`git add`**: Ця команда використовується для додавання змін у стадію підготовки. Все, що знаходиться у стадії підготовки, буде додано до наступного коміту. Щоб додати всі незапідготовлені зміни, можна використати `git add .`. Крапка (`.`) — це псевдонім для поточного каталогу, в якому ви перебуваєте.
- **`git commit`**: Ця команда використовується для фіксації змін. Коміт — це знімок стану проєкту на певний момент часу. Якщо виконати `git commit`, відкриється ваш улюблений редактор, встановлений у конфігурації Git. Після відкриття редактора ви можете надати детальний опис змін. Також можна надати коротке повідомлення, використовуючи команду `git commit -m` так:

```sh
git commit -m "short message goes here"
```

- **`git log`**: Показує список усіх попередніх комітів з корисною інформацією, як-от автор, дата коміту, повідомлення коміту та хеш коміту. Хеш коміту — це довгий рядок, який слугує унікальним ідентифікатором коміту.
- **`git remote add`**: Використовується для налаштування віддаленого з’єднання з вашим віддаленим репозиторієм.
- **`git push`**: Використовується для відправлення ваших змін у віддалений репозиторій.
- **`git pull`**: Використовується для отримання останніх змін з віддаленого репозиторію у ваш локальний репозиторій.
- **`git clone`**: Клонує репозиторій. Це означає, що ви отримаєте копію репозиторію. Ця копія включає історію репозиторію, усі файли/папки та коміти на вашому локальному пристрої.
- **`git remote -v`**: Показує список віддалених репозиторіїв, пов’язаних з вашим локальним Git-репозиторієм.
- **`git branch`**: Показує всі ваші локальні гілки.
- **`git fetch upstream`**: Команда, яка каже Git отримати останні зміни з вашого upstream-репозиторію (оригінального репозиторію).
- **`git merge upstream/main`**: Команда, яка каже Git об’єднати останні зміни з гілки `main` у upstream-репозиторії у вашу поточну гілку.
- **`git reset`**: Дозволяє скинути поточний стан гілки. Використання прапорця `--hard` змушує Git примусово привести локальні файли у відповідність зі станом гілки. Це гарантує, що у вас чистий початок для роботи.
- **`git rebase`**: Rebase у Git — це спосіб перемістити або об’єднати послідовність комітів з однієї гілки в іншу.

## Робота з гілками

- **Визначення**: Гілка у Git — це окремий робочий простір, де ви можете вносити зміни. Гілка `main` часто є основною або продуктивною гілкою у реальному застосунку. Команди розробників створюють кілька гілок для нових функцій і виправлень помилок, а потім об’єднують ці зміни назад у гілку `main`.
- **Створення нової гілки**: Щоб створити нову гілку, виконайте таку команду:

```sh
git branch feature
```

Щоб переключитися на цю гілку, виконайте таку команду:

```sh
git checkout feature
```

Більшість розробників використовують скорочену команду для створення та переключення гілки, яка виглядає так:

```sh
git checkout -b new-branch-name
```

Новішою альтернативою є команда `git switch`. Ось приклад створення та переходу на нову гілку:

```sh
git switch -c new-branch-name
```

- **Стратегії гілкування**: Ваша гілка `main` — це гілка за замовчуванням і зазвичай досить стабільна. Тому найкраще створювати нові гілки від неї для таких завдань, як виправлення помилок, нові функції або інша допоміжна робота.
- **Конфлікти злиття**: Виникають, коли Git намагається автоматично об’єднати зміни з різних гілок, але не може вирішити, які зміни залишити. Зазвичай це трапляється, коли є конфліктні зміни в одній і тій самій частині файлу.

## П’ять станів файлу, що відстежується Git

- **«Untracked» (не відстежується)**: Файл новий для репозиторію, і Git його раніше не «бачив».
- **«Modified» (змінений)**: Файл існував у попередньому коміті і має зміни, які ще не зафіксовані.
- **«Ignored» (ігнорується)**: Ви, ймовірно, не побачите ігноровані файли у Git, але ваша IDE може їх позначати. Ігноровані файли виключені з операцій Git, зазвичай тому, що вони включені у файл `.gitignore`.
- **«Deleted» (видалений)**: Видалений файл — це протилежність не відстежуваному файлу — це файл, який раніше існував, але був видалений.
- **«Renamed» (перейменований)**: Перейменований файл — це файл, вміст якого не змінився, але було змінено ім’я або розташування файлу. Іноді файл вважається перейменованим, навіть якщо має невеликі зміни.

## Файли `.gitignore`

- **Визначення**: Файл `.gitignore` — це спеціальний тип файлу, пов’язаний з операціями Git. Назва підказує, що цей файл використовується, щоб сказати Git ігнорувати певні речі, і це найпоширеніше використання. Але насправді він каже Git припинити відстежувати файл.

## Робота з репозиторіями

- **Визначення**: Репозиторій — це як контейнер для проєкту — якщо ви працюєте над застосунком, ви зберігаєте файли цього застосунку разом у репозиторії. Репозиторії можуть бути локальними на вашому комп’ютері або віддаленими на сервісі, як GitHub.
- **Публічні та приватні репозиторії**: Публічний репозиторій може переглядати і скачувати будь-хто. Приватний репозиторій доступний лише вам і тим, кому ви явно надали доступ.
- **Створення репозиторіїв на GitHub**: Щоб створити новий репозиторій на GitHub, натисніть кнопку `"New Repository"` і пройдіть через інтерфейс GitHub для налаштування нового репозиторію.
- **Відправлення локальних репозиторіїв на GitHub**: Якщо у вас є локальний проєкт на комп’ютері, ви можете відправити цей репозиторій на GitHub. Ось покроковий огляд процесу:

1. Ініціалізуйте порожній git-репозиторій у каталозі проєкту (`git init`).
2. Внесіть зміни у проєкт.
3. Виконайте команду `git status`, щоб побачити всі зміни, які відстежує git.
4. Додайте зміни у стадію підготовки (`git add`).
5. Зафіксуйте зміни (`git commit`).
6. Налаштуйте віддалене з’єднання (`git remote add`).
7. Відправте зміни на GitHub (`git push`).

## Запити на злиття

- **Запити на злиття**: Запит на злиття — це запит на внесення змін з вашої гілки у цільову гілку. Запити на злиття — це процес, який ви використовуєте, коли хочете зробити внесок у проєкт. Це дає змогу підтримувачам проєкту переглянути ваші зміни. Вони можуть залишати коментарі, ставити запитання і пропонувати правки. Після завершення повторення запит можна схвалити і об’єднати з основною гілкою.

## Внесок у інші репозиторії

- **Процес**: Існують тисячі проєктів, у які ви можете робити внесок. Ось базовий процес, як зробити внесок у чужий репозиторій:

1. Прочитайте документацію щодо внеску
2. Знайдіть доступну проблему для роботи
3. Форкніть репозиторій
4. Клонуйте вашу форкнуту копію репозиторію
5. Створіть нову гілку
6. Внесіть зміни відповідно до проблеми
7. Створіть запит на злиття (Pull Request)
8. Чекайте на повторення цього запиту

## Робота з SSH та GPG ключами

- **GPG ключі**: GPG, або Gnu Privacy Guard, ключі зазвичай використовуються для підписування файлів або комітів. Хтось може використати ваш публічний GPG ключ, щоб перевірити, що підпис файлу зроблено вашим ключем і що вміст файлу не було змінено або підроблено.

Щоб згенерувати GPG ключ, виконайте:

```sh
gpg --full-generate-key
```

- **SSH ключі**: SSH, або Secure SHell, ключі зазвичай використовуються для автентифікації віддаленого з’єднання з сервером — через утиліту `ssh`. Ви також можете використовувати SSH ключ для підписування комітів.

Для створення SSH ключа виконайте:

```sh
ssh-keygen -t ed25519 -C "your_email@example.com"
```

`ed25519` — це сучасний алгоритм підпису з відкритим ключем.

- **Підписування комітів за допомогою GPG ключів**: Щоб підписувати коміти своїм GPG ключем, потрібно завантажити публічний ключ, а не приватний, у свій акаунт GitHub. Щоб переглянути публічні ключі, виконайте таку команду:

```sh
gpg --list-secret-keys --keyid-format=long
```

Потім, щоб отримати публічний ключ, використайте:

```sh
gpg --armor --export "<key id>"
```

Потім візьміть короткий ідентифікатор, який ви отримали, і виконайте цю команду, щоб встановити його як ключ підпису git:

```sh
git config --global user.signingkey <your_gpg_key_id>
```

Потім ви можете додати прапорець `-S` до команди `git commit`, щоб підписати конкретний коміт — вам потрібно буде ввести свій пароль. Альтернативно, щоб підписувати всі коміти автоматично, можна встановити конфігурацію autosign у `true`:

```sh
git config --global commit.gpgsign true
```

- **Підписування комітів за допомогою SSH ключів**: Щоб підписувати за допомогою SSH ключа, що є відносно новою функцією на GitHub, спочатку потрібно завантажити ключ у свій акаунт GitHub. Потім потрібно встановити режим підпису git на використання SSH:

```sh
git config --global gpg.format ssh
```

Потім, щоб встановити ключ підпису, передайте шлях до файлу замість ідентифікатора:

```sh
git config --global user.signingkey <path_to_your_ssh_keys>
```
