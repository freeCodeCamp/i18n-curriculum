## Сервери та модель клієнт-сервер

- **Сервер**: Комп’ютер або програма, яка отримує запити від інших комп’ютерів і надсилає дані або послуги назад мережею. Термін охоплює як фізичне обладнання, так і програмне забезпечення, що на ньому працює.
- **Модель клієнт-сервер**: Розподілена архітектура, де клієнти надсилають запити, а сервери обробляють ці запити і повертають відповіді.
- **Клієнт**: Процес або застосунок (наприклад, веб-браузер), який ініціює запити до сервера.
- **Цикл запит-відповідь**: Клієнт надсилає запит; сервер обробляє його і повертає відповідь.

### Типи серверів

- **Веб-сервери**: Постачають веб-вміст (HTML, зображення, CSS) браузерам.
- **Сервери застосунків**: Обробляють бізнес-логіку та вхідні дані користувача.
- **Сервери баз даних**: Зберігають, отримують і керують даними.
- **Поштові сервери**: Надсилають, отримують і зберігають е-пошту.
- **Файлові сервери**: Керують зберіганням і спільним доступом до файлів у мережі.
- **Проксі-сервери**: Діють як посередники між клієнтами та іншими серверами.

## DNS (система доменних імен)

- **DNS**: Ієрархічна та децентралізована система імен, яка перетворює доменні імена, зрозумілі людині, на IP-адреси, часто називають «телефонною книгою» інтернету.
- **IP-адреса**: Унікальний числовий ідентифікатор пристрою в мережі (наприклад, `127.0.0.1` в IPv4, `0:0:0:0:0:0:0:1` в IPv6).
- **Ієрархія доменних імен** (читається справа наліво):
  - **TLD (домен верхнього рівня)**: Найправіший сегмент (наприклад, `.org`, `.com`).
  - **SLD (домен другого рівня)**: Зареєстроване ім’я (наприклад, `freecodecamp` у `freecodecamp.org`).
  - **Піддомен**: Визначає розділ основного домену (наприклад, `www`, `blog`, `api`).

### Процес розв’язання DNS

1. Браузер надсилає доменне ім’я до **рекурсивного резолвера** (зазвичай керованого провайдером інтернету).
2. Рекурсивний резолвер запитує **кореневі сервери**, які направляють його до відповідного сервера TLD.
3. **Сервер TLD** повертає адресу **авторитетного іменного сервера** для домену.
4. Авторитетний іменний сервер повертає **DNS-запис** із IP-адресою.
5. Рекурсивний резолвер передає IP-адресу клієнту і **кешує** запис для прискорення майбутніх пошуків.

## TCP/IP

- **Протокол**: Набір правил і стандартів, які дозволяють пристроям спілкуватися мережею.
- **TCP/IP**: Сукупність протоколів, що визначає, як дані пакуються, надсилаються, маршрутизуються та отримуються.
  - **IP (інтернет-протокол)**: Маршрутизує пакети даних від джерела до призначення за IP-адресами. Це безз’єднувальний протокол — кожен пакет рухається незалежно без гарантії доставки.
  - **TCP (протокол керування передачею)**: Розбиває дані на нумеровані сегменти, збирає їх на приймальному кінці, перевіряє на помилки та втрати, і за потреби запитує повторну передачу. Забезпечує надійну та впорядковану доставку.

## HTTP (протокол передачі гіпертексту)

- **HTTP**: Протокол рівня застосунку, який визначає правила комунікації між клієнтами і серверами у вебі.

- **HTTPS**: Захищена версія HTTP, де дані шифруються за допомогою SSL/TLS перед передачею.

### Структура HTTP-запиту

- **Рядок запиту**: `METHOD /path HTTP/version` (наприклад, `GET / HTTP/1.1`)
- **Заголовки**: Параметри метаданих у форматі ключ-значення (наприклад, `Content-Type: application/json`).
- **Тіло** (необов’язкове): Дані, що надсилаються серверу, використовуються з `POST`, `PUT` і `PATCH`.

**Загальні HTTP-методи:**

| Метод | Призначення |
| --- | --- |
| `GET` | Отримати дані |
| `POST` | Відправити нові дані |
| `PUT` | Замінити ресурс |
| `PATCH` | Частково оновити ресурс |
| `DELETE` | Видалити ресурс |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### Структура HTTP-відповіді

- **Рядок статусу**: версія HTTP, код статусу та пояснювальна фраза (наприклад, `HTTP/1.1 200 OK`).
- **Заголовки**: метадані про відповідь (наприклад, `Content-Type: text/html`).
- **Тіло** (необов’язкове): повернутий вміст або пояснення помилки.

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## Коди статусу HTTP

Коди статусу групуються за першою цифрою:

- **1xx** — інформаційні відповіді.
- **2xx** — успішні відповіді.
- **3xx** — повідомлення про перенаправлення.
- **4xx** — помилки клієнта.
- **5xx** — помилки сервера.

### Загальні коди

| Код | Назва | Значення |
| --- | --- | --- |
| `200` | OK | Запит успішний; вміст повернуто. |
| `201` | Created | Ресурс успішно створено. |
| `204` | No Content | Успіх, але немає вмісту для повернення. |
| `301` | Moved Permanently | Ресурс назавжди переміщено на нову URL-адресу. |
| `302` | Found | Ресурс тимчасово переміщено на іншу URL-адресу. |
| `304` | Not Modified | Ресурс не змінювався з останнього запиту; використовуйте кешовану версію. |
| `400` | Bad Request | Сервер не зрозумів запит (некоректний синтаксис або дані). |
| `401` | Unauthorized | Потрібні облікові дані для автентифікації. |
| `403` | Forbidden | Клієнт не має дозволу на ресурс, навіть з дійсними обліковими даними. |
| `404` | Not Found | Ресурс не існує на сервері. |
| `500` | Internal Server Error | Неочікувана помилка на сервері. |
| `502` | Bad Gateway | Сервер отримав некоректну відповідь від вищестоящого сервера. |
| `503` | Service Unavailable | Сервер тимчасово не може обробляти запити. |

## Модель HTTP запит-відповідь

Цикл запит-відповідь складається з таких компонентів, що працюють разом:

- **Клієнт**: браузер, мобільний застосунок або API-клієнт, який надсилає HTTP-запити.
- **HTTP-метод**: дія, яку потрібно виконати (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`).
- **URL**: адреса ресурсу, до якого звертаються.
- **Заголовки**: метадані для запиту або відповіді (наприклад, `Authorization`, `Content-Type`).
- **Тіло**: дані, що надсилаються з `POST`, `PUT`, `PATCH` або `DELETE` запитами (часто JSON або форма).
- **Сервер**: бекенд-система, що обробляє запити і повертає відповіді.
- **Код статусу**: числовий код, що вказує на успіх, помилку або перенаправлення.
- **Тіло відповіді**: фактичні дані, що повертаються, зазвичай HTML, JSON або XML.

### Шість кроків циклу

1. **Клієнт ініціює запит** з методом, URL, заголовками та необов’язковим тілом.
2. **Запит передається серверу** через інтернет.
3. **Сервер обробляє запит**: перевіряє метод, URL, автентифікацію, за потреби звертається до бази даних.
4. **Сервер готує відповідь** з кодом статусу, заголовками та необов’язковим тілом.
5. **Відповідь надсилається назад** клієнту.
6. **Клієнт обробляє відповідь**: відтворює дані при успіху (2xx) або показує повідомлення про помилку при невдачі (4xx/5xx).

## Ресурси HTTP-відповіді та типи вмісту

| Ресурс | Content-Type |
| --- | --- |
| HTML | `text/html` |
| CSS | `text/css` |
| JavaScript | `application/javascript` |
| JSON | `application/json` |
| XML | `application/xml` |
| JPEG | `image/jpeg` |
| PNG | `image/png` |
| GIF | `image/gif` |
| SVG | `image/svg+xml` |
| WebP | `image/webp` |
| MP4 | `video/mp4` |

| ZIP | `application/zip` |

## Надсилання HTML-форм

- Форми збирають і відправляють дані користувача за допомогою елемента `<form>` з `<input>` елементами.
- Кожен вхідний елемент повинен мати атрибут `name`, щоб сервер міг ідентифікувати дані.
- **Валідація на стороні клієнта**: використовуйте атрибути `required` і `pattern` для миттєвого зворотного зв’язку.
- **Валідація на стороні сервера**: завжди перевіряйте на сервері для безпеки, незалежно від перевірок на клієнті.
- Атрибут `action` вказує, куди надсилати дані форми.
- Атрибут `method` визначає, як їх надсилати (`GET` або `POST`; `PUT`, `PATCH`, `DELETE` потребують обхідних шляхів, таких як Fetch API або `method-override` в Express).

### Потік надсилання форми

1. Користувач заповнює і відправляє форму.
2. Браузер виконує валідацію на клієнті; якщо вона успішна, продовжує.
3. Браузер надсилає дані форми на URL у `action` з використанням вказаного `method`.
4. Сервер отримує і обробляє дані (зберігає в базі даних, автентифікує користувача тощо).
5. Сервер може виконати додаткову валідацію на стороні сервера.
6. Сервер відповідає повідомленням про успіх, помилкою або перенаправленням.
7. Користувач бачить зворотний зв’язок залежно від відповіді сервера.

## Модель веб-стандартів і організації стандартів

- **Веб-стандарти**: правила і специфікації, яких дотримуються браузери для відтворення сторінок і забезпечення веб-функцій.
- **WHATWG**: організація стандартів, що підтримує HTML і DOM як безперервно оновлюваний «живий стандарт», керований основними командами браузерних рушіїв.
- **W3C**: головна організація стандартів, що контролює CSS, SVG, керівництва з доступності (WCAG) і численні веб-API.
- **ECMA / TC39**: ECMA публікує специфікацію ECMAScript (стандарт JavaScript); TC39 розвиває функції мови через п’ятиетапний процес пропозицій (етапи 0–4).
- **Khronos Group**: відповідає за WebGL і WebGL 2, які забезпечують апаратне прискорення 3D-графіки у браузері.

## Процес створення веб-стандартів

1. **Визначення потреби**: розробник або інженер браузера документує реальну, широко поширену проблему.
2. **Пропозиція та раннє обговорення**: ідея викладається неформально для збору відгуків і оцінки зацікавленості виробників браузерів.
3. **Написання специфікації**: створюється точний, формальний документ, що охоплює всі крайні випадки, умови помилок і питання безпеки.
4. **Огляд, заперечення та ітерації**: проект публічно переглядається; виробники висловлюють заперечення, і специфікація редагується до досягнення консенсусу.
5. **Реалізація та тестування**: виробники браузерів реалізують функцію за прапорцем; спільний набір тестів перевіряє, що всі реалізації поводяться однаково.
6. **Фіналізація**: після проходження тестів кількома незалежними реалізаціями стандарт фіналізується (рекомендація W3C, етап 4 TC39 або оновлення живого стандарту WHATWG).

## Життєвий цикл функцій веб-стандартів

1. **Пропозиція**: хтось визначає проблему і записує її, щоб дізнатися, чи поділяють її інші.
2. **Інкубація**: ідея проходить стрес-тестування через відкриті обговорення; багато пропозицій відмирають, якщо вони надто вузькі або не задовольняють усіх.
3. **Специфікація**: пишеться формальна специфікація, що перетворює нечіткі концепції на точні, реалізовні правила.
4. **Тестування сумісності**: набір тестів W3C Web Platform Tests перевіряє, що кожен браузер реалізує функцію однаково.
5. **Фіналізація**: кілька успішних реалізацій означають, що функція стає частиною базової веб-платформи для користувачів.
6. **Виведення з експлуатації**: застарілі або проблемні функції позначають для припинення; видалення з браузерів відбувається повільно через зобов’язання зворотної сумісності.

## Ключові принципи веб-стандартів

- **Відкритість**: стандарти розробляються відкрито і реалізуються без ліцензійних платежів будь-ким.
- **Доступність**: веб створений для використання всіма, з доступністю, вбудованою в HTML, CSS і поведінку браузера.
- **Сумісність**: сторінка, створена за стандартом, має однаково працювати у всіх браузерах і операційних системах.
- **Зворотна сумісність**: нові можливості накладаються на існуючі, щоб старі сторінки продовжували працювати в сучасних браузерах.
- **Конфіденційність і безпека**: нові API, які можуть відкривати дані користувача, проходять суворий контроль; платформа захищає користувачів за замовчуванням.
- **Шарування**: HTML відповідає за структуру, CSS — за презентацію, а JavaScript — за поведінку; кожен шар працює незалежно.
- **Прогресивне покращення**: функції мають працювати для всіх спочатку, а потім покращуватися для користувачів із більш потужними браузерами або пристроями.
