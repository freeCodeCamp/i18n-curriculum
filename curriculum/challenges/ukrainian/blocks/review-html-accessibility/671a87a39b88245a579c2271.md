## Вступ до доступності

- **Керівництво з доступності вебвмісту**: Керівництво з доступності вебвмісту (WCAG) — це набір рекомендацій для підвищення доступності вебвмісту для людей з інвалідністю. Чотири принципи WCAG — POUR, що означає сприйнятливий, керований, зрозумілий і надійний.

## Допоміжні технології для доступності

- **Читачі екрана**: Програмне забезпечення, яке озвучує вміст екрану комп’ютера. Їх використовують люди, які сліпі або мають порушення зору, щоб отримати доступ до вебу.
- **Клавіатури з великим шрифтом або шрифтом Брайля**: Використовуються людьми з порушеннями зору для доступу до вебу.
- **Збільшувачі екрану**: Програми, які збільшують вміст екрану комп’ютера. Їх використовують люди з низьким зором для доступу до вебу.
- **Альтернативні пристрої вказівки**: Використовуються людьми з порушеннями рухливості для доступу до вебу. Це включає пристрої, такі як джойстики, трекболи та тачпади.
- **Розпізнавання голосу**: Використовується людьми з порушеннями рухливості для доступу до вебу. Дозволяє керувати комп’ютером за допомогою голосу.

## Інструменти аудиту доступності

- **Загальні інструменти доступності**: Google Lighthouse, Wave, IBM Equal Accessibility Checker і axe DevTools — це деякі з поширених інструментів доступності, які використовують для аудиту доступності вебсайту.

## Найкращі практики доступності

- **Правильна структура рівнів заголовків**: Ви повинні використовувати правильні рівні заголовків, щоб створити логічну структуру для свого вмісту. Це допомагає людям, які користуються допоміжними технологіями, краще розуміти вміст вашого вебсайту.
- **Доступність і таблиці**: При використанні таблиць слід використовувати елемент `th` для визначення заголовкових клітинок і елемент `td` для визначення клітинок з даними. Це допомагає людям, які користуються допоміжними технологіями, розуміти структуру таблиці. За допомогою елемента `caption` можна написати підпис (або заголовок) таблиці, щоб користувачі, особливо ті, хто використовує допоміжні технології, могли швидко зрозуміти призначення та вміст таблиці. Ви повинні розміщувати елемент `caption` безпосередньо після початкового тегу елемента `table`. Таким чином, читачі екрана та інші допоміжні технології можуть надати більше контексту, оголошуючи підпис перед читанням вмісту.
- **Важливість наявності пов’язаної мітки для вхідних даних**: Ви повинні використовувати елемент `label` для асоціації міток із вхідними даними форми. Це допомагає людям, які користуються допоміжними технологіями, розуміти призначення вхідних даних.
- **Важливість якісного тексту `alt`**: Ви повинні використовувати атрибут `alt` для надання текстової альтернативи зображенням. Це допомагає людям, які користуються допоміжними технологіями, розуміти вміст зображення.
- **Важливість якісного тексту посилань**: Ви повинні використовувати описовий текст посилань, щоб допомогти користувачам зрозуміти призначення посилання. Це допомагає людям, які користуються допоміжними технологіями, розуміти призначення посилання.
- **Найкращі практики для забезпечення доступності аудіо- та відеоконтенту**: Ви повинні надавати підписи та транскрипти для аудіо- та відеоконтенту, щоб зробити його доступним для людей з порушеннями слуху. Також слід надавати аудіоописи для відеоконтенту, щоб зробити його доступним для людей з порушеннями зору.
- **Атрибут `tabindex`**: Використовується для того, щоб зробити елементи фокусованими та визначити відносний порядок, у якому їх слід переміщувати за допомогою клавіатури. Важливо ніколи не використовувати атрибут `tabindex` зі значенням більше за 0. Замість цього слід використовувати значення 0 або -1.

:::interactive_editor

```html
<p tabindex="-1">Sorry, there was an error with your submission.</p>
```

:::

- **Атрибут `accesskey`**: Використовується для визначення клавіатурного скорочення для елемента. Це може допомогти користувачам з порушеннями рухливості легше орієнтуватися на вебсайті.

:::interactive_editor

```html
<button accesskey="s">Save</button>
<button accesskey="c">Cancel</button>
<a href="index.html" accesskey="h">Home</a>
```

:::

## WAI-ARIA, ролі та атрибути

- **WAI-ARIA**: Це ініціатива з доступності вебу — Accessible Rich Internet Applications. Це набір атрибутів, які можна додати до HTML-елементів для покращення доступності. Вони надають допоміжним технологіям додаткову інформацію про призначення та структуру вмісту.
- **Ролі ARIA**: Набір заздалегідь визначених ролей, які можна додати до HTML-елементів, щоб визначити їх призначення. Це допомагає людям, які користуються допоміжними технологіями, розуміти вміст вебсайту. Прикладами є `role="tab"`, `role="menu"` і `role="alert"`.

Існує шість основних категорій ролей ARIA:

- **Ролі структури документа**: Ці ролі визначають загальну структуру вебсторінки. Завдяки цим ролям допоміжні технології можуть розуміти зв’язки між різними розділами та допомагати користувачам орієнтуватися у вмісті.
- **Ролі віджетів**: Ці ролі визначають призначення та функціональність інтерактивних елементів, таких як смуги прокрутки.
- **Ролі орієнтирів**: Ці ролі класифікують і позначають основні розділи вебсторінки. Читачі екрана використовують їх для зручної навігації до важливих розділів сторінки.
- **Ролі живих регіонів**: Ці ролі визначають елементи з вмістом, який динамічно змінюється. Таким чином, читачі екрана та інші допоміжні технології можуть оголошувати зміни користувачам із порушеннями зору.
- **Ролі вікон**: Ці ролі визначають підвікна, такі як модальні діалоги. До цих ролей належать `alertdialog` і `dialog`.
- **Абстрактні ролі**: Ці ролі допомагають організувати документ. Вони призначені лише для внутрішнього використання браузером, а не розробниками, тому ви маєте знати про їх існування, але не повинні використовувати їх на своїх вебсайтах або вебзастосунках.

- **Атрибути `aria-label` і `aria-labelledby`**: Ці атрибути використовуються для надання елементу програмного (або доступного) імені, що допомагає людям, які користуються допоміжними технологіями (наприклад, читачами екрана), розуміти призначення елемента. Їх часто використовують, коли візуальна мітка елемента є зображенням або символом, а не текстом. `aria-label` дозволяє визначити ім’я безпосередньо в атрибуті, тоді як `aria-labelledby` дозволяє посилатися на існуючий текст на сторінці.

```html
<button aria-label="Search">
    <i class="fas fa-search"></i>
</button>
```

:::interactive_editor

```html
<input type="text" aria-labelledby="search-btn">
<button type="button" id="search-btn">Search</button>
```

:::

- **Атрибут `aria-hidden`**: Використовується для приховування елемента від допоміжних технологій, таких як читачі екрана. Наприклад, це можна застосувати для приховування декоративних зображень, які не містять значущого вмісту.

:::interactive_editor

```html
<button>
    <i class="fa-solid fa-gear" aria-hidden="true"></i>
    <span class="label">Settings</span>
</button>
```

:::

- **Атрибут `aria-describedby`**: Використовується для надання додаткової інформації про елемент шляхом асоціації його з іншим елементом, який містить цю інформацію. Це дає людям, які користуються читачами екрана, миттєвий доступ до додаткової інформації під час навігації до елемента. Зазвичай використовується для асоціації інструкцій з форматування з текстовим полем або повідомлення про помилку з полем після невдалої відправки форми.

:::interactive_editor

```html
<form>
    <label for="password">Password:</label>
    <input type="password" id="password" aria-describedby="password-help" />
    <p id="password-help">Your password must be at least 8 characters long.</p>
</form>
```

:::
