# --description--

在之前的课程中，你学习了如何向你自己的仓库发起 pull request。你还学习了如何合并该 pull request。但你之所以能做到这一点，是因为该仓库属于你——你对自己的仓库拥有维护者访问权限，这意味着你有权限合并 pull request。

然而，当你为别人的开源项目贡献时，你通常不会拥有那些权限。相反，你需要等待维护者通过代码审查流程并接受你的贡献。然后他们会处理合并。

但是代码审查进程是什么样的？

维护者采取的进程因人而异，且可能存在项目特定的方法。但通常，代码审查通常有三个步骤。

第一个是确认 CI，或持续集成，是否通过。你将在未来的课程中了解更多，所以现在你可以将它们视为用于验证代码是否符合样式指南并通过测试套件的检查。

如果 CI 失败，维护者通常会请求你更新拉取请求以解决失败。这些失败应包含某种报告，供你查看错误原因，但维护者也可能为你提供更具体的指导以解析问题。

一旦检查通过，维护者将查看你实际的代码更改。他们可以分析差异视图以查看你修改、添加或删除了什么。他们可能会询问你为何采取特定的方法，因此在准备讨论时，保持谨慎和深思熟虑是有益的。

他们也可能请求你进行额外的修改。我们稍后会讨论这是什么样子的。

最后一步通常是将你的更改拉取到本地。维护者随后可以使用你的修改运行应用，测试功能性，并验证没有任何功能损坏或异常行为。

如果一切看起来正常，维护者将批准你的 pull request——有时会带有 `"LGTM"` 评论，表示“looks good to me”。在你的 pull request 获得批准后，只要维护者准备好合并更改，就应将其合并。一些项目可能需要多个批准——例如，freeCodeCamp 需要两个。

但是如果你的 pull request 有问题怎么办？维护者可以执行审查，在审查中他们会请求更改。这些请求可以有几种形式。

有时，维护者可能只是留下一个带有他们请求的“评论”，可能指出你应该更改或更新的内容，但在如何实现上有更多自由。

或者，维护者也可能留下官方的 `"request changes"` 评论。通常，这样的评论会阻止拉取请求被合并。你需要进行所需的更改，然后请求更改的同一维护者需要重新审核并请求更多更改或批准你的工作。

当维护者有具体更改想法时，他们实际上可以通过 GitHub 的 UI 提出更改。当他们使用这样的直接建议时，GitHub 会给你一个按钮，直接将建议提交到你的分支中。然而，如果有多个这样的建议，你会想切换到文件视图。

从此视图中，你可以将所有建议（逐个）添加到一个 `batch` 中，并一次性提交它们。当建议很多时，这非常有用，因为逐个处理会导致大量提交。所有的 CI 检查都必须在每个提交上运行，这会减慢接受你的 PR 的进程。

如果请求的更改较大，你可能想要在局部的进行更改，将它们提交到你用于该 PR 的同一分支，然后推送上去。GitHub 会自动更新你的拉取请求以反映新的提交。

还有一些其他的注意事项我们想要说明。

首先，记住维护者也是人。开源项目很少能产生足够的收入让维护者全职工作，这意味着他们必须保持一份工作。对审核过程保持耐心非常重要——不要反复催促维护者重新审核，也不要试图催促他们加快审核进度。

贡献可以很令人兴奋！但不要急于完成整个过程。相反，要花时间。确保在本地测试你的更改——运行项目可能有的任何测试套件，运行开发服务器并手动确认你的更改是否有效，等等。并且始终遵循你正在参与的具体项目的贡献指南。

如果你牢记所有这些，并全力以赴，你的贡献应该会顺利进行，你的代码审查也应该会获得坚实的批准！
