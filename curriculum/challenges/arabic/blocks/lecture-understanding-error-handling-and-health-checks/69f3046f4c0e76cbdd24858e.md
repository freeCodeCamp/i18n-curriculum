# --description--

هل تساءلت يومًا ماذا يحدث عندما يحدث خطأ في تطبيق Express الخاص بك؟

ماذا لو فشل اتصال قاعدة البيانات؟ أو زار المستخدم مسارًا غير موجود؟

فهم كيفية التعامل مع الأخطاء في Express أمر ضروري إذا أردت بناء تطبيقات ويب آمنة وموثوقة.

في هذا الدرس، سنشرح كيف يتعامل Express مع الأخطاء، وكيف يمكنك التقاطها والرد عليها باستخدام أمثلة شفرة حقيقية.

## كيفية التعامل مع الأخطاء البسيطة في Express

في Express، يعتمد التعامل مع الأخطاء على البرمجيات الوسيطة.

تمامًا مثل دوال البرمجيات الوسيطة العادية، تمتلك برمجيات التعامل مع الأخطاء وصولًا إلى `req` و `res` و `next`. لكنها تتضمن معلمة إضافية في البداية: `err`.

إليك كيف تبدو الصيغة النحوية:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

تعرف هذه البرمجيات الوسيطة في نهاية سلسلة البرمجيات الوسيطة الخاصة بك.

يعرف Express أنها معالِج أخطاء لأنها تحتوي على أربعة معلمات — المعلمة الجديدة `err`، بالإضافة إلى المعلمات المعتادة `req` و `res` و `next` الخاصة بدوال البرمجيات الوسيطة الأخرى.

في درس قادم، ستتعلم المزيد عن برمجيات التعامل مع الأخطاء وكيف تندمج في سلسلة برمجيات Express الوسيطة الأوسع.

## كيفية التعامل مع الأخطاء غير المتزامنة

تعتمد معالجة الأخطاء غير المتزامنة على إصدار Express الذي تستخدمه.

لاحظ أنه، مهما كان إصدار Express الذي تستخدمه، ستظل بحاجة لتعريف برمجيات التعامل مع الأخطاء في نهاية السلسلة، بعد جميع المسارات.

إليك معالج الأخطاء مرة أخرى:

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Express 4

في Express 4 وما دونه، والذي لا تزال العديد من المشاريع تستخدمه، لا يتم التقاط الأخطاء غير المتزامنة تلقائيًا.

إذا كنت تستخدم `async`/`await` وحدث رمي خطأ داخل معالِج مسار `async`، فلن يلتقطه Express إلا إذا مررته يدويًا إلى `next()`.

إليك كيفية التعامل مع الأخطاء غير المتزامنة بشكل صحيح في Express 4:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

إذا نسيت كتلة `try/catch` و `next(err)`، فقد يؤدي الخطأ إلى:

* تعطل تطبيقك
* أو ظهور تحذير رفض وعد غير معالج

لهذا السبب، من الضروري تغليف المنطق غير المتزامن بـ `try`/`catch` في Express 4.

بعض المطورين ينشئون دالة مساعدة لتجنب تكرار `try/catch` في كل مكان:

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

ثم يستخدمونها هكذا:

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

هذا يجعل من غير الضروري استخدام كتل `try/catch` في جميع مسارات `async` مع الاستمرار في تمرير الأخطاء بشكل صحيح.

## Express 5

الإصدار الحالي من Express يحسن هذا السلوك.

اعتبارًا من Express 5، إذا رمى مسار `async` خطأ أو أعاد وعدًا مرفوضًا، يقوم Express تلقائيًا بتمريره إلى برمجيات التعامل مع الأخطاء.

مما يعني أن هذا المثال يتعامل مع الأخطاء بدون `try/catch`:

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

إذا رمى `getUserFromDatabase()` خطأ، سيلتقطه Express 5 ويمرره إلى معالج الأخطاء تلقائيًا.

لا حاجة لاستدعاء يدوي لـ `next(err)`.

## كيفية التعامل مع أخطاء 404

أخطاء 404 مختلفة قليلاً. تحدث عندما لا يطابق أي مسار الطلب.

يمكنك التعامل معها بهذه البرمجيات الوسيطة:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**مهم**: يجب أن توضع هذه بعد مساراتك، ولكن قبل برمجيات التعامل مع الأخطاء.

وأخيرًا، إليك بعض أفضل الممارسات للتعامل مع الأخطاء في Express:

* ضع دائمًا برمجيات التعامل مع الأخطاء في النهاية.
* تعامل مع أخطاء 404 بشكل منفصل.
* استخدم `next(err)` لتمرير الأخطاء من المسارات أو المنطق `async` في Express 4 وما دونه.

في بيئة الإنتاج، لا ترسل تتبع الأخطاء للمستخدمين. سجّل التفاصيل داخليًا وارجع رسالة سهلة الفهم بدلاً من ذلك.

التعامل مع الأخطاء في Express يتعلق بإعداد برمجيات وسيطة تستمع للمشاكل وترد بطريقة واضحة وآمنة. سواء كان خطأ برمجي في شفرتك أو مسار مفقود، يجب ألا يتعطل تطبيقك بدون خطة احتياطية.
