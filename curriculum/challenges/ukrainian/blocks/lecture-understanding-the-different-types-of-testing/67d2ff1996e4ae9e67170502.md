# --description--

Під час розробки застосунків у вас часто буде серія невеликих функцій, відповідальних за одну конкретну задачу. Це відоме як «принцип єдиної відповідальності». Коли у вас є серія невеликих функцій, найкраще тестувати ці функції, щоб переконатися, що все працює як очікується. У цьому уроці ми розглянемо, як створити модульний тест за допомогою популярного фреймворку для тестування Jest.

У цьому прикладі ми створимо функцію, яка відповідає за повернення ново відформатованого рядка:

```js
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

Ця `getFormattedWord` функція має параметр `str` і спочатку перевіряє, чи є `str` порожнім. Якщо так, повертається порожній рядок. Інакше повертається новий рядок, де перша літера є великою. Ми експортуємо цю функцію, щоб мати змогу використовувати її у файлі тестів.

У окремому `getFormattedWord.test.js` файлі ми можемо написати тести, щоб перевірити, чи функція виконує те, що від неї очікується.

На початку `getFormattedWord.test.js` файлу нам потрібно спочатку імпортувати функцію ось так:

```js
import { getFormattedWord } from "./getFormattedWord.js";
```

Потім потрібно встановити пакет Jest за допомогою `npm i jest`. Після цього ми можемо додати тест, щоб перевірити, що слово, складене з символів нижнього регістру, `hello`, правильно форматоване у `Hello`, яке починається з великої `H`:

```js
test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

Функція `expect` використовується для тестування значення. Вона поєднується з матчером — функцією, яка перевіряє, чи поводиться значення так, як очікується. У цьому випадку матчером є `toBe()`. Jest має різноманітні матчери, які допомагають тестувати істинність, рядки, числа та інше.

Один зі способів запуску тестів — додати npm-скрипт у ваш `package.json` файл. Ось приклад скрипта для фреймворку Jest:

```js
"scripts": {
  "test": "jest"
},
```

Потім просто запустіть `npm run test` у терміналі, щоб виконати тести. Якщо тести пройдуть, у терміналі ви побачите щось подібне:

```sh
PASS ./getFormattedWord.test.js
  ✓ capitalizes the first letter of a word (1 ms)

Test Suites: 1 passed, 1 total
Tests:       1 passed, 1 total
Snapshots:   0 total
Time:        0.103 s, estimated 1 s
Ran all test suites.
```

Щоб побачити, як виглядає провалений тест, ми можемо навмисно зламати тест, оновивши функцію до такого вигляду:

```js
export function getFormattedWord(str) {
  if (!str) return '';
  return "This is incorrect";
}
```

Тепер, коли ви запустите команду `npm run test`, з’явиться повідомлення про помилку, оскільки тест очікував іншого результату:

```sh
FAIL ./getFormattedWord.test.js
  ✕ capitalizes the first letter of a word (1 ms)

  ● capitalizes the first letter of a word

  expect(received).toBe(expected) // Object.is equality

  Expected: "Hello"
  Received: "This is incorrect"

    2 |
    3 | test('capitalizes the first letter of a word', () => {
    4 |   expect(getFormattedWord('hello')).toBe('Hello');
                                            ^
    5 | });
    6 |

    at Object.toBe (getFormattedWord.test.js:4:37)

Test Suites: 1 failed, 1 total
Tests:       1 failed, 1 total
Snapshots:   0 total
Time:        0.121 s, estimated 1 s
Ran all test suites.
```

Тепер ми можемо повернути нашу функцію до початкового вигляду тут:

```js
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

Додайте ще один тест, щоб перевірити випадки, коли аргумент не передається у `getFormattedWord`:

```js
test("returns an empty string when no argument is provided", () => {
  expect(getFormattedWord()).toBe("");
});
```

Існують інші тести, які ви можете додати, щоб зробити все більш надійним, але ці перші кілька тестів є хорошим вступом до модульного тестування.

Що стосується тестування JavaScript-застосунків, деякі загальні фреймворки для тестування включають Jest, Mocha та Vitest.

Модульне тестування важливе, бо допомагає виявляти більше помилок у ваших програмах і гарантує, що все працює як очікується. Воно також може слугувати формою документації для вашого застосунку, оскільки має представляти очікувану поведінку вашого коду.
