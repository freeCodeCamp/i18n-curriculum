# --description--

Семантичне версіонування, або SemVer, — це система, яка надає номерам версій чітке значення, щоб розробники могли швидко зрозуміти, які зміни були внесені до програмного забезпечення.

SemVer використовує такий формат:

```bash
MAJOR.MINOR.PATCH
```

Наприклад, популярна бібліотека `lodash` наразі має версію `4.17.21`.

Ось що означає кожна частина:

* MAJOR — великі зміни, які порушують сумісність із існуючим кодом. Приклад: видалення або перейменування функції, від якої може залежати існуючий код.
    
* MINOR — нові функції, які не порушують сумісність із існуючим кодом. Приклад: додавання нової функції або необов’язкового параметра.
    
* PATCH — невеликі виправлення та покращення. Приклад: виправлення помилки або опечатки без зміни роботи.
    

<img src="https://cdn.freecodecamp.org/curriculum/lecture-transcripts/what-is-semantic-versioning-1.png" alt="У номері версії 4.17.21, 4 — це major-версія, вона змінюється при порушенні сумісності, 17 — minor-версія, вона змінюється при незначних змінах, а 21 — patch, вона змінюється при виправленні помилок і невеликих функцій">

Припустимо, у вас є пакет під назвою `my-package`. Ось приклад того, як номер його версії може змінюватися з часом, використовуючи семантичне версіонування:

| Версія | Що змінилося |
| --- | --- |
| 1.0.0 | Початковий стабільний реліз |
| 1.1.0 | Нова необов’язкова функція |
| 1.1.1 | Невелике виправлення помилки або функції |
| 2.0.0 | Внесені порушення сумісності |

Але в npm номери версій мають ще більше значень. Коли ви встановлюєте пакет, версія у вашому `package.json` файлі може виглядати так:

```json
"lodash": "^4.17.21" 
```

Ви вже знаєте, що `4` — це major-версія, `17` — minor-версія, а `21` — patch-версія. Але що означає карет (`^`) на початку?

Символ карет дозволяє npm автоматично встановлювати новіші minor- та patch-версії (`4.x.x`), але не новіші major-версії (`5.x.x`), оскільки major-оновлення можуть містити порушення сумісності.

Коли major-версія дорівнює `0`, карет стає більш обмежувальним. Наприклад, `^0.2.3` дозволить оновлення на кшталт `0.2.4`, але не `0.3.0`, оскільки major-версії, що починаються з `0`, вважаються менш стабільними.

Карет — не єдиний символ, який `npm` та інші менеджери пакетів JavaScript використовують для керування версіями залежностей.

Існує символ тильда (`~`), який дозволяє оновлення лише patch-версії. Тож `~1.2.3` прийме `1.2.4` і `1.2.9`, але ніколи `1.3.0`.

Ви також можете використовувати символи більше (`>`), менше (`<`) та їхні варіанти з "або дорівнює", щоб приймати версії в певному діапазоні. Наприклад, `>=1.2.3 <2.0.0` означає будь-яку версію від `1.2.3` до, але не включно, `2.0.0`.

Символ зірочки (`*`) відповідає будь-якій версії, тому приймає все. Це підходить лише для тестування і не рекомендується для продакшену.

Нарешті, ви можете зафіксувати точну версію, не додаючи префікс до номера версії. Наприклад, версія `1.2.3` приймає лише `1.2.3`.

Ось коротка таблиця:

| Символ | Значення | Приклад діапазону |
| --- | --- | --- |
| `^` | Дозволити оновлення minor + patch | `^1.2.3` → `1.x.x` |
| `~` | Дозволити оновлення лише patch | `~1.2.3` → `1.2.x` |
| `>`, `<`, `>=`, `<=` | Кастомний діапазон | `>=1.2.3` або `<2.0.0` |
| **\*** | Будь-яка версія | усі версії |
| без символу | Точна версія | `1.2.3` |
