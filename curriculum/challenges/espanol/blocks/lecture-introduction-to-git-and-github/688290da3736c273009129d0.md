# --description--

En la lección anterior, creamos una rama de características, hicimos un commit de un archivo nuevo en esa rama y lo subimos a nuestro repositorio remoto. ¿Pero cuáles son los siguientes pasos? Necesitamos crear un pull request en GitHub.

Una pull request es, efectivamente, una solicitud para incorporar cambios desde tu rama hacia la rama objetivo. Las pull requests son el flujo que usas cuando quieres contribuir con cambios de código a un proyecto de código abierto. Este enfoque permite que los mantenedores del proyecto revisen tus cambios. Pueden dejar comentarios, hacer preguntas, sugerir ajustes y (con suerte) aprobar tus cambios y fusionarlos.

Vamos a ir a nuestro repositorio en GitHub de nuevo.

Ahora, después de que hayas subido tu nueva rama, deberías ver un banner elegante que dice que has subido a una rama, y un botón en el que puedes hacer clic para crear un pull request. Este botón omite un par de pasos y agiliza el proceso, así que lo vamos a ignorar por ahora. De esa manera estarás preparado para los casos en los que el banner no aparezca.

Haz clic en la pestaña de pull requests en la parte superior de la página del repositorio, luego ignora el banner nuevamente y haz clic en "New Pull Request". Serás llevado a una interfaz para preparar tu solicitud.

Ahora, hay algunos términos que necesitamos repasar aquí. Primero, tenemos los menús desplegables "base" y "compare". La "base" es tu objetivo para la fusión. Como queremos fusionar en `main`, podemos dejar esto como está.

El "compare" es lo que quieres fusionar. Queremos fusionar nuestra rama `feature`, así que deberíamos cambiar esto a `feature`. Esto también se conoce como la rama head, un término que querrás recordar.

Una vez que hagas esto, la interfaz de usuario se actualizará. Ahora debería mostrar los commits en `feature` que no están en `main`, indicando lo que estás solicitando fusionar. También verás lo que se llama un "diff", que es una representación visual de los cambios reales que has hecho.

Como todo lo que hemos hecho es agregar un archivo, el diff está bastante limpio. Para cambios más complejos, GitHub resaltará cosas como modificaciones de líneas, adiciones, eliminaciones y más. Entender cómo leer un diff puede ser un desafío, pero con algo de práctica te volverás bastante hábil en ello.

Por ahora, volvamos a nuestra pull request. Hay un pequeño enlace en la parte superior que dice "compare across forks". Si estuviéramos trabajando desde nuestro fork del repositorio freeCodeCamp, esto nos daría la capacidad de cambiar nuestro PR para apuntar al repositorio original de freeCodeCamp, en lugar de solo a la rama `main` dentro de nuestro fork.

Como no estamos trabajando con un fork, podemos ignorar esto. Sin embargo, es muy importante entender que lo que estamos haciendo aquí es principalmente para aprendizaje y exploración. No deberías abrir un PR en el proyecto de alguien que solo contenga cambios de práctica; esto crea ruido adicional para los mantenedores.

Ahora, como estamos satisfechos con nuestros cambios y todo está configurado, podemos hacer clic en el botón "Create Pull Request". Esto nos lleva a una nueva interfaz de usuario.

Hacia la parte inferior y fuera de vista están la misma lista de commits y la vista de diferencias que vimos antes. Nunca está de más revisar una última vez antes de crear tu PR, pero para esta exploración estás listo para continuar.

También verás un par de nuevas entradas. La primera entrada es el título para tu pull request. GitHub generará esto automáticamente basado en el commit que estás fusionando, si solo hay uno en el PR, o el nombre de la rama, si tienes múltiples commits.

La segunda entrada es para tu descripción. En la mayoría de los casos, esta no estará vacía. En cambio, los mantenedores de un proyecto pueden definir una plantilla que se prellena automáticamente. Si proporcionan una, debes respetar su flujo de trabajo y completar la plantilla.

En ambos casos, querrás leer la documentación de contribución del proyecto para asegurarte de entender y seguir sus directrices sobre cómo titular y completar tu PR.

Como eres el dueño de este repositorio, verás a la derecha que puedes establecer reviewers, assignees, labels, projects y milestones. No podrás editar estos, excepto para solicitar una revisión, cuando contribuyas al proyecto de otra persona. Pero si quieres explorar esas funciones, puedes configurarlas como quieras ahora, ya que este es tu repositorio.

Una vez que todo esté completado y configurado, haz clic en "Create pull request". ¡Finalmente, verás el resultado real!

Cuando haces un pull request en el proyecto de otra persona, debe pasar por una "revisión de código". Este es un proceso donde los mantenedores pueden evaluar tu pull request, solicitar cambios, hacer preguntas, y más. Una vez que los mantenedores lo han aprobado, pueden fusionarlo.

Como este es tu propio proyecto, no necesitas esperar una revisión. En cambio, puedes verificar el historial de commits en la pestaña Commits, los cambios en Files changed, y si estás satisfecho puedes hacer el merge.

Después de que se haya fusionado, querrás descargar estos cambios a tu repositorio local.

Regresa a tu terminal y asegúrate de que todavía estés en la carpeta del repositorio. Luego cambia a la rama `main` con `git checkout main`.

A continuación, descarga los últimos cambios con `git pull`.

Ahora, si miras el historial de commits con `git log`, verás tu commit. Pero también verás un mensaje `Merge pull request #1`. ¿Qué es eso?

Bueno, cuando haces merge de un pull request en GitHub hay tres estrategias que puedes usar:

El valor predeterminado es "Merge", que trae todos los commits de la rama head, luego crea otro commit para procesar y registrar la fusión.

Otra opción es "Squash and Merge", que toma TODOS los commits de la rama principal, los aplasta en un solo commit y fusiona ese nuevo commit directamente en la rama base sin crear un commit de fusión.

Finalmente, está "Rebase and Merge", que toma todos los commits, los reinicia para que "vengan después" del commit más reciente en la rama base, y luego fusiona el head sin un commit de fusión. El rebase es un tema complicado que aprenderás en profundidad más adelante, así que está bien si hoy no lo entiendes completamente.

Cada proyecto de código abierto elegirá la estrategia de merge que funcione mejor. Pero debido a estas diferencias en el comportamiento, siempre debes crear una nueva rama para hacer commit de tus contribuciones. Nunca hagas commit directamente en la rama `main` por defecto, o las cosas se complicarán cuando necesites traer los cambios desde el repositorio original.

Con todo eso, ¡has creado y fusionado exitosamente tu primer pull request!
