# --description--

¿Alguna vez te has preguntado qué pasa cuando algo sale mal en tu aplicación Express?

¿Qué sucede si falla la conexión a la base de datos? ¿O si un usuario visita una ruta que no existe?

Entender cómo funciona el manejo de errores en Express es fundamental si quieres construir aplicaciones web seguras y confiables.

En esta lección, veremos cómo Express maneja los errores y cómo puedes capturarlos y responder a ellos usando ejemplos reales de código.

## Cómo manejar errores simples en Express

En Express, el manejo de errores se basa en middleware.

Al igual que las funciones middleware normales, el middleware para manejo de errores tiene acceso a `req`, `res` y `next`. Pero incluye un parámetro extra al principio: `err`.

Así es como se ve la sintaxis:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

Defines este middleware al final de tu pila de middleware.

Express sabe que es un manejador de errores porque tiene cuatro parámetros — el nuevo, `err`, además de los típicos `req`, `res` y `next` que usan otras funciones middleware.

En una lección próxima, aprenderás más sobre el middleware para manejo de errores y cómo encaja en la pila más amplia de middleware de Express.

## Cómo manejar errores asíncronos

El manejo de errores asíncronos depende de la versión de Express que estés usando.

Ten en cuenta que, cualquiera que sea la versión de Express que uses, aún necesitas definir tu middleware para manejo de errores al final de la pila, después de todas tus rutas.

Aquí está el manejador de errores otra vez:

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Express 4

En Express 4 y versiones anteriores, que muchos proyectos aún usan, los errores asíncronos **no se capturan automáticamente**.

Si usas `async`/`await` y algo lanza un error dentro de un manejador de ruta `async`, Express no lo capturará a menos que lo pases manualmente a `next()`.

Así es como manejas correctamente los errores asíncronos en Express 4:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

Si olvidas el bloque `try/catch` y `next(err)`, el error puede:

* Provocar que tu aplicación se caiga
* O generar una advertencia de promesa rechazada no manejada

Por eso es esencial envolver la lógica asíncrona en `try`/`catch` en Express 4.

Algunos desarrolladores incluso crean una función auxiliar para evitar repetir `try/catch` en todas partes:

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

Luego la usan así:

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

Esto hace que no necesites usar bloques `try/catch` en todas tus rutas `async`, pero aún así reenvías los errores correctamente.

## Express 5

La versión actual de Express mejora este comportamiento.

A partir de Express 5, si un manejador de ruta `async` lanza un error o devuelve una promesa rechazada, Express lo reenvía automáticamente al middleware para manejo de errores.

Esto significa que este ejemplo maneja errores sin un `try/catch`:

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

Si `getUserFromDatabase()` lanza un error, Express 5 lo capturará y lo pasará a tu manejador de errores automáticamente.

No se requiere llamada manual a `next(err)`.

## Cómo manejar errores 404

Los errores 404 son un poco diferentes. Ocurren cuando ninguna ruta coincide con la solicitud.

Puedes manejarlos con este middleware:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**Importante**: Esto debe ir después de tus rutas, pero antes de tu middleware para manejo de errores.

Y finalmente, aquí tienes algunas buenas prácticas para el manejo de errores en Express:

* Siempre coloca tu middleware para manejo de errores al final.
* Maneja los errores 404 por separado.
* Usa `next(err)` para propagar errores desde rutas o lógica `async` en Express 4 y versiones anteriores.

En producción, no envíes rastros de pila a los usuarios. Registra los detalles internamente y devuelve un mensaje amigable para el usuario.

El manejo de errores en Express consiste en configurar middleware que detecte problemas y responda de forma clara y segura. Ya sea un error en tu código o una ruta faltante, tu aplicación nunca debería caerse sin un plan de respaldo.
