# --description--

이전 수업에서 GitHub에 새 저장소를 만드는 방법을 배웠습니다. 하지만 이미 로컬에 존재하는 프로젝트를 GitHub에 추가하고 싶다면 어떻게 해야 할까요? 이번 수업에서는 기존 로컬 저장소를 GitHub에 푸시하고 원격 저장소에서 변경 사항을 내려받는 방법을 배우겠습니다.

먼저 GitHub에서 파일이 없는 새 저장소를 만드세요. 제대로 완료하면 저장소를 클론하는 데 필요한 URL이 포함된 페이지로 리다이렉트됩니다.

이제 터미널로 이동해 GitHub에 추가하려는 기존 프로젝트를 선택하세요. 이번 수업에서는 `super-awesome-game`라는 로컬 프로젝트를 선택했으며, 그 안에 다음과 같은 파일들이 있습니다:

```sh
jessicawilkins super-awesome-game >> ls
README.md  index.html script.js  styles.css
```

이 프로젝트에 Git을 사용해 버전 관리를 설정하려면 프로젝트 디렉터리에서 `git init` 명령어를 실행해야 합니다. 이 명령어는 빈 Git 저장소를 초기화하여 Git이 이 프로젝트의 변경 사항을 추적할 수 있게 합니다. 제대로 완료되면 터미널에 비슷한 출력이 나타납니다:

```sh
jessicawilkins super-awesome-game >> git init
Initialized empty Git repository in /Users/jessicawilkins/workspace/freeCodeCamp/super-awesome-game/.git/
```

프로젝트에 빈 Git 저장소를 초기화하면 `.git`라는 숨김 디렉터리가 새로 추가됩니다. 이 디렉터리 추가를 확인하려면 `ls -a` 명령어를 실행해 숨김 파일과 디렉터리를 포함한 모든 파일과 디렉터리를 나열할 수 있습니다:

```sh
jessicawilkins super-awesome-game >> ls -a
.          ..         .git       README.md  index.html script.js  styles.css
```

이 `.git` 디렉터리를 삭제하지 않는 것이 중요합니다. 그렇지 않으면 모든 Git 기록이 사라집니다.

빈 Git 저장소가 초기화되었으니 이제 `git status` 명령어를 실행할 수 있습니다. 이 명령어는 작업 디렉터리의 현재 상태를 보여주며, 작업 흐름에서 자주 사용하게 됩니다. 다음과 비슷한 출력이 나타날 것입니다:

```sh
jessicawilkins super-awesome-game >> git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	README.md
	index.html
	script.js
	styles.css

nothing added to commit but untracked files present (use "git add" to track)
```

`On branch main`는 현재 `main` 브랜치에 있음을 의미합니다. Git의 브랜치는 변경 사항을 만들 수 있는 별도의 작업 공간입니다. `main` 브랜치는 실제 애플리케이션에서 주로 기본 또는 프로덕션 브랜치를 나타냅니다. 개발자 팀은 새로운 기능과 버그 수정을 위해 여러 브랜치를 만들고, 그런 다음 변경 사항을 `main` 브랜치에 병합합니다. 이 과정은 이후 수업에서 더 자세히 배우게 됩니다.

`Untracked files`는 Git이 현재 추적하지 않는 파일을 감지했음을 의미합니다.

파일이 있을 수 있는 다섯 가지 상태는 다음과 같습니다:

1. "Untracked": 이 파일은 저장소에 새로 추가된 파일로, Git이 이전에 본 적이 없습니다.
2. "Modified": 이 파일은 이전 커밋에 있었으며, 아직 커밋되지 않은 변경 사항이 있습니다.
3. "Ignored": Git에서는 무시된 파일을 보지 못할 수 있지만, IDE에서는 표시할 수 있습니다. 무시된 파일은 일반적으로 `.gitignore` 파일에 포함되어 Git 작업에서 제외됩니다.
4. "Deleted": 삭제된 파일은 이전에 존재했으나 제거된 파일입니다.
5. "Renamed": 이름이 변경된 파일은 내용은 같지만 이름이나 위치가 바뀐 파일입니다. 경우에 따라 약간의 변경이 있어도 이름이 변경된 것으로 간주할 수 있습니다.

커밋되지 않은 변경 사항이 있으면 먼저 "스테이지"해야 합니다. "스테이징 영역"은 `.git` 디렉터리에 있는 특별한 파일로, 다음 커밋에 포함될 정보를 담고 있습니다. 커밋은 저장소의 현재 상태를 스냅샷으로 저장하는 것입니다.

한 번에 한 파일씩 스테이지하려면 다음 명령어를 사용하세요:

```sh
git add name-of-file
```

모든 스테이지되지 않은 변경 사항을 한꺼번에 스테이지하려면 `git add .` 명령어를 사용할 수 있습니다. 점(`.`)은 현재 디렉터리를 가리키는 별칭입니다. 일부 개발자는 `git add .` 사용을 경고하는데, 의도하지 않은 파일까지 스테이지할 수 있기 때문입니다. 그런 경우 다음 명령어로 "언스테이지"할 수 있습니다:

```sh
git reset name-of-file
```

일반적으로 `git status` 명령어로 어떤 파일이 추가되는지 확인한 후 스테이지하면 문제가 없습니다.

파일을 스테이지한 후에는 커밋해야 합니다. `git commit` 명령어를 실행하면 설정한 기본 편집기가 열립니다. 편집기가 열리면 변경 사항에 대한 자세한 메시지를 작성할 수 있습니다. `git commit -m` 명령어를 사용하면 짧은 메시지를 제공할 수도 있습니다:

```sh
git commit -m "short message goes here"
```

`-m` 플래그를 사용할 때도 변경 사항에 대한 맥락을 제공하는 설명 메시지를 작성하는 것이 좋습니다. 좋은 메시지 예시는 다음과 같습니다:

```sh
git commit -m "feat(api): implement JWT-based authentication"
git commit -m "fix(auth): correct token expiration time"
git commit -m "refactor: simplify login flow logic"
```

`feat(api)`과 `fix(auth)`은 Conventional Commits 스타일을 따르며, 커밋의 변경 유형에 대한 맥락을 제공합니다.

작업을 커밋하고 `git status` 명령어를 다시 실행하면 터미널에 다음과 같은 출력이 나타납니다:

```sh
On branch main
nothing to commit, working tree clean
```

이 프로젝트의 이전 모든 커밋을 보려면 `git log` 명령어를 실행하세요. 작성자, 커밋 날짜, 메시지, 커밋 해시 등 유용한 정보를 포함한 모든 이전 커밋이 나열됩니다. 커밋 해시는 커밋을 고유하게 식별하는 긴 문자열입니다.

커밋 작업을 마쳤으면 이제 변경 사항을 GitHub에 푸시할 수 있습니다. 먼저 원격 저장소와의 연결을 설정해야 합니다. 이전 수업에서 배웠지만, SSH를 사용하는 방법을 다시 알려드립니다:

```sh
git remote add origin git@github.com:your-github-username/name-of-repo.git
```

마지막 단계는 다음 명령어로 변경 사항을 GitHub에 푸시하는 것입니다:

```sh
git push -u origin main
```

`-u` 플래그는 `--set-upstream`의 축약형입니다. 로컬 `main` 브랜치를 `main`의 원격 `origin` 브랜치에 연결합니다. 새 브랜치마다 한 번만 사용하면 되고, 이후에는 브랜치 이름 없이 `git push` 명령어만 실행하면 됩니다.

제대로 완료하면 터미널에 비슷한 결과가 나타납니다:

```sh
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Delta compression using up to 8 threads
Compressing objects: 100% (2/2), done.
Writing objects: 100% (3/3), 253 bytes | 253.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0 (from 0)
To https://github.com/jdwilkin4/super-awesome-game.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

GitHub 저장소 페이지를 새로 고침하면 새 변경 사항도 확인할 수 있습니다.

프로젝트에서 다른 사람과 협업하며 저장소에서 새 변경 사항을 내려받아야 할 때는 `git pull` 명령어를 사용하세요. 팀 작업과 협업에 대해서는 이후 수업에서 더 배우게 됩니다.
