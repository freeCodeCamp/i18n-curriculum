# --description--

在過去兩堂課中，你學會了如何驗證使用者身份並使用 cookie 維護他們的會話。然而，自動在每個請求中傳送 cookie 會引入一種獨特的安全漏洞，稱為跨站請求偽造（CSRF）。

CSRF 是一種攻擊，強迫已驗證的使用者在他們目前登入的網路應用程式上執行不想要的操作。在 CSRF 攻擊中，惡意網站會誘使受害者瀏覽器向你的後端伺服器發出偽造請求，利用伺服器對使用者瀏覽器 cookie 的信任。

為了理解 CSRF，請參考以下銀行應用程式的逐步範例：

1. **設定階段：** 你登入你的銀行網站，例如 `trusted-bank.com`。銀行驗證你的憑證並將會話標記存放在你的瀏覽器 cookie 中。

2. **陷阱：** 在未登出銀行的情況下，你開啟一個新的瀏覽器分頁並造訪一個惡意網站，例如 `evil-hacker-site.com`。

3. **觸發：** 惡意網站包含一個隱藏的腳本或表單，自動向銀行後端提交請求，例如 `POST trusted-bank.com/transfer?amount=1000&to=hacker`。

4. **執行：** 因為你仍然登入銀行，你的瀏覽器會自動將銀行會話 cookie 附加到該請求。銀行伺服器看到有效的 cookie，假設你是有意發出該請求，並轉帳你的錢。

這裡的關鍵是惡意網站無法直接讀取你的 cookie 或竊取你的資料。它只是利用瀏覽器會自動將 cookie 附加到任何發往標的伺服器的請求這個事實。

有兩種主要防禦方法：**SameSite Cookie 屬性**和**CSRF 語彙單元**。

最簡單的防禦是為你的 cookie 設定 `SameSite` 屬性。這告訴瀏覽器何時允許在跨站請求中傳送 cookie。

- **SameSite=Strict：** 如果請求來自第三方網站，瀏覽器絕不會傳送 cookie。

- **SameSite=Lax：** 瀏覽器只在使用者安全地導覽到標的網站時允許傳送 cookie，例如點擊一般連結，但會阻擋跨站表單提交或 API 請求。

為了強固的保護，後端框架會使用 CSRF 語彙單元。當使用者載入頁面時，後端會產生一個隨機、唯一且不可預測的語彙單元，並將它附加到前端表單或會話中。

當客戶端提交請求時，必須在自訂請求標頭或表單資料中包含此語彙單元。後端會將提交的語彙單元與儲存在使用者會話中的語彙單元比對。因為惡意網站無法存取你的前端來複製這個秘密語彙單元，任何他們偽造的跨站請求都會缺少此語彙單元而驗證失敗。

透過實作這些策略，你確保送達後端的請求是實際使用者有意發出的行為，而非惡意攻擊者強迫的未授權行為。
