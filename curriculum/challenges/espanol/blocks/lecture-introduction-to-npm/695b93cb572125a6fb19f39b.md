# --description--

El versionado semántico, o SemVer, es un sistema que da a los números de versión un significado claro, para que los desarrolladores puedan entender rápidamente qué tipo de cambios se hicieron en el software.

SemVer usa este formato:

```bash
MAJOR.MINOR.PATCH
```

Por ejemplo, la popular biblioteca `lodash` está actualmente en la versión `4.17.21`.

Esto es lo que significa cada parte:

* MAYOR — Cambios grandes que rompen código existente. Ejemplo: eliminar o renombrar una función de la que el código existente podría depender.
    
* MENOR — Nuevas características que no rompen el código existente. Ejemplo: agregar una nueva función o parámetro opcional.
    
* PATCH — Pequeñas correcciones y mejoras. Ejemplo: arreglar un error o un error tipográfico sin cambiar cómo funciona nada.
    

<img src="https://cdn.freecodecamp.org/curriculum/lecture-transcripts/what-is-semantic-versioning-1.png" alt="In the version number 4.17.21, 4 is the Major Version, it's changed for breaking changes, the 17 is the minor version, it's changed for non-breaking changes, and the 21 is the patch, it's changed for bug fixes and small features">

Supongamos que tienes un paquete llamado `my-package`. Aquí hay un ejemplo de cómo podría cambiar su número de versión con el tiempo usando versionado semántico:

| Version | What Changed |
| --- | --- |
| 1.0.0 | Initial stable release |
| 1.1.0 | New optional feature |
| 1.1.1 | Small bug fix or feature |
| 2.0.0 | Breaking changes introduced |

Pero hay más en los números de versión en npm. Cuando instalas un paquete, la versión en tu archivo `package.json` podría verse así:

```json
"lodash": "^4.17.21" 
```

Ya aprendiste que `4` es la versión mayor, `17` es la versión menor y `21` es la versión de parche. Pero, ¿qué significa el caret (`^`) al frente?

El símbolo caret permite que npm instale automáticamente versiones menores y parches más recientes (`4.x.x`), pero no versiones mayores más recientes (`5.x.x`), porque las actualizaciones mayores pueden incluir cambios incompatibles.

Cuando la versión mayor es `0`, el caret se vuelve más restrictivo. Por ejemplo, `^0.2.3` permitiría actualizaciones como `0.2.4`, pero no `0.3.0`, ya que las versiones mayores que comienzan con `0` se consideran menos estables.

El caret no es el único símbolo que `npm` y otros gestores de paquetes de JavaScript usan para gestionar versiones de dependencias.

Existe el símbolo de tilde (`~`) que permite actualizaciones solo a la versión de parche. Entonces, `~1.2.3` aceptaría `1.2.4` y `1.2.9`, pero nunca `1.3.0`.

También puedes usar los símbolos mayor que (`>`), menor que (`<`) y sus equivalentes de "o igual" para aceptar versiones dentro de un rango. Por ejemplo, `>=1.2.3 <2.0.0` significa cualquier versión desde `1.2.3` hasta, pero sin incluir, `2.0.0`.

El símbolo asterisco (`*`), coincide con cualquier versión, por lo que acepta todo. Esto es ideal solo para pruebas y no se recomienda para producción.

Finalmente, puedes bloquear una versión exacta no anteponiendo ningún símbolo al número de versión. Por ejemplo, la versión `1.2.3` acepta solo `1.2.3`.

Aquí tienes una tabla resumen rápida:

| Symbol | Meaning | Example Range |
| --- | --- | --- |
| `^` | Allow minor + patch updates | `^1.2.3` → `1.x.x` |
| `~` | Allow patch updates only | `~1.2.3` → `1.2.x` |
| `>`, `<`, `>=`, `<=` | Custom range | `>=1.2.3` or `<2.0.0` |
| **\*** | Any version | all versions |
| none | Exact version | `1.2.3` |
