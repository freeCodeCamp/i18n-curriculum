# --description--

Ви коли-небудь замислювалися, що відбувається, коли у вашому застосунку Express щось іде не так?

Що, якщо з’єднання з базою даних не вдається? Або користувач заходить на маршрут, якого не існує?

Розуміння того, як працює обробка помилок в Express, є критично важливим, якщо ви хочете створювати безпечні та надійні веб-застосунки.

У цьому уроці ми розглянемо, як Express обробляє помилки, і як ви можете їх ловити та реагувати на них за допомогою реальних прикладів коду.

## Як обробляти прості помилки в Express

В Express обробка помилок базується на підпрограмному забезпеченні.

Як і звичайні функції підпрограмного забезпечення, обробники помилок мають доступ до `req`, `res` та `next`. Але вони містять один додатковий параметр на початку: `err`.

Ось як виглядає синтаксис:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

Це підпрограмне забезпечення визначається в кінці стеку підпрограм.

Express розпізнає його як обробник помилок, бо він має чотири параметри — новий `err`, крім звичних `req`, `res` та `next`, які є у звичайних функціях підпрограмного забезпечення.

У наступному уроці ви дізнаєтеся більше про обробники помилок і як вони вписуються у загальний стек підпрограм Express.

## Як обробляти асинхронні помилки

Обробка асинхронних помилок залежить від версії Express, яку ви використовуєте.

Зверніть увагу, що незалежно від версії Express, яку ви використовуєте, вам все одно потрібно визначати обробник помилок у кінці стеку, після всіх маршрутів.

Ось цей обробник помилок ще раз:

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Express 4

У Express 4 і нижче, які досі використовують багато проєктів, асинхронні помилки **не ловляться автоматично**.

Якщо ви використовуєте `async`/`await` і всередині обробника маршруту `async` виникає помилка, Express її не спіймає, якщо ви вручну не передасте її в `next()`.

Ось як правильно обробляти асинхронні помилки в Express 4:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

Якщо ви забудете блок `try/catch` і `next(err)`, помилка може:

* Завалити ваш застосунок
* Або викликати попередження про необроблене відхилення промісу

Саме тому обгортання асинхронної логіки в `try`/`catch` є обов’язковим у Express 4.

Деякі розробники навіть створюють допоміжну функцію, щоб не повторювати `try/catch` скрізь:

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

Потім використовують її так:

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

Це дозволяє не використовувати блоки `try/catch` у всіх маршрутах `async`, але при цьому правильно передавати помилки.

## Express 5

Поточна версія Express покращує цю поведінку.

Починаючи з Express 5, якщо обробник маршруту `async` викидає помилку або повертає відхилений проміс, Express автоматично передає її обробнику помилок.

Отже, цей приклад обробляє помилки без `try/catch`:

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

Якщо `getUserFromDatabase()` викидає помилку, Express 5 її спіймає і автоматично передасть вашому обробнику помилок.

Виклик `next(err)` вручну не потрібен.

## Як обробляти помилки 404

Помилки 404 трохи інші. Вони виникають, коли жоден маршрут не відповідає запиту.

Ви можете обробити їх таким підпрограмним забезпеченням:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**Важливо**: це має йти після ваших маршрутів, але перед обробниками помилок.

І нарешті, ось кілька найкращих практик для обробки помилок в Express:

* Завжди розміщуйте обробники помилок останніми.
* Обробляйте помилки 404 окремо.
* Використовуйте `next(err)` для передачі помилок з маршрутів або асинхронної логіки `async` у Express 4 і нижче.

У продакшені не надсилайте користувачам трасування стека. Логінуйте деталі внутрішньо і повертайте дружнє повідомлення користувачу.

Обробка помилок в Express — це налаштування підпрограмного забезпечення, яке слухає проблеми і реагує на них чітко та безпечно. Чи це помилка в коді, чи відсутній маршрут — ваш застосунок ніколи не повинен аварійно завершувати роботу без запасного плану.
