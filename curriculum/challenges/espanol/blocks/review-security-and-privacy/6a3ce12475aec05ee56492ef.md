## Seguridad vs privacidad

**Seguridad** protege los datos contra accesos no autorizados, robos y daños maliciosos. Construye defensas técnicas como cortafuegos, encriptación y software antivirus para mantener a los actores malintencionados fuera de tus sistemas.

**Privacidad** controla cómo las organizaciones recopilan, almacenan, usan y comparten datos personales. Da a los usuarios el poder de decidir quién ve su información.

Los dos conceptos están estrechamente relacionados pero son distintos:

- Puedes tener seguridad sin privacidad. Una empresa puede almacenar tus datos en una base de datos altamente segura y aun así venderlos a anunciantes sin tu consentimiento. En ese caso, la seguridad está intacta, pero la privacidad se viola.
- No puedes tener verdadera privacidad sin seguridad. Si un servidor que contiene datos privados es vulnerado, cualquier promesa de privacidad se rompe inmediatamente.

La seguridad proporciona el mecanismo técnico de defensa, mientras que la privacidad regula los derechos legales y las políticas sobre el uso de datos.

## HTTPS

**HTTP** transfiere datos en texto plano, haciéndolos legibles para cualquiera que los intercepte. **HTTPS** (Hypertext Transfer Protocol Secure) usa **TLS (Transport Layer Security)** para encriptar el canal de comunicación entre un navegador y un servidor. TLS reemplazó al protocolo más antiguo **SSL (Secure Sockets Layer)**, aunque muchas plataformas de hosting web aún hacen referencia a SSL en sus configuraciones hoy en día.

Beneficios clave de HTTPS:

- **Encriptación:** Codifica los datos en un código ilegible. Solo el navegador y el servidor tienen las claves para descifrarlo.
- **Autenticación:** Requiere que el propietario del sitio obtenga un certificado digital de una organización confiable, demostrando que el sitio web es legítimo.
- **Integridad de datos:** Impide que alguien altere los datos en tránsito, recibes exactamente lo que el servidor envió.
- **Protección en Wi-Fi público:** Crea un túnel privado a través de redes abiertas, ocultando tu tráfico de atacantes cercanos.

Los navegadores modernos advierten a los usuarios cuando un sitio usa HTTP simple y requieren HTTPS para funcionalidades avanzadas como geolocalización y notificaciones push.

## Política de mismo origen y CORS

La **Política de mismo origen (SOP)** es una regla de seguridad del navegador que impide que una página web lea datos de un origen diferente. Un **origen** consta de tres partes que deben coincidir exactamente:

- **Protocolo** (por ejemplo, `https`)
- **Nombre de dominio** (por ejemplo, `example.com`)
- **Número de puerto** (por ejemplo, `443`, el predeterminado para HTTPS y `80`, el predeterminado para HTTP)

Esta política evita que sitios maliciosos abiertos en una pestaña del navegador roben datos de un sitio bancario abierto en otra pestaña.

**CORS (Cross-Origin Resource Sharing)** permite que los servidores opten por compartir datos entre orígenes controladamente usando encabezados HTTP:

- El navegador envía un encabezado `Origin` con cada solicitud entre orígenes.
- Si el servidor confía en ese origen, responde con un encabezado `Access-Control-Allow-Origin`.
- Entonces el navegador permite leer los datos de la respuesta; de lo contrario, los bloquea.

En Express, el paquete `cors` maneja esto:

```javascript
// development: allow all origins
app.use(cors());

// production: restrict to trusted domains
app.use(cors({ origin: ['https://yourapp.com', 'https://admin.yourapp.com'], credentials: true }));
```

## Cookies: almacenamiento, seguridad y privacidad

**Cookies** son pequeños archivos de texto que los sitios web guardan en tu dispositivo para recordar información entre visitas. El servidor envía un encabezado `Set-Cookie`; el navegador almacena el archivo y lo envía automáticamente en cada solicitud posterior.

Banderas de seguridad para proteger las cookies:

- **`HttpOnly`:** Bloquea que JavaScript lea la cookie, previniendo el robo de tokens de sesión vía XSS.
- **`Secure`:** Asegura que la cookie solo se transmita por HTTPS, protegiéndola de intercepciones en Wi-Fi público.
- **`SameSite=Strict`:** Impide que el navegador envíe la cookie en cualquier solicitud entre sitios, bloqueando ataques CSRF.
- **`SameSite=Lax`:** Permite cookies en navegación segura de nivel superior pero las bloquea en envíos de formularios y solicitudes API entre sitios.

**Implicaciones de privacidad:**

- **Cookies de primer partido** rastrean tus acciones solo en el sitio web que visitas.
- **Cookies de terceros** son colocadas por redes publicitarias y te rastrean a través de sitios web completamente diferentes, creando perfiles detallados de tus intereses.

Leyes de privacidad como GDPR requieren que los sitios web muestren banners de consentimiento de cookies y den a los usuarios el derecho a rechazar cookies de rastreo. Los navegadores modernos ahora bloquean por defecto las cookies de rastreo de terceros.

## Situaciones donde ocurren problemas de seguridad

Las vulnerabilidades de seguridad rara vez ocurren por casualidad. Situaciones comunes que las desencadenan incluyen:

- **Prisa por lanzar:** Los equipos de desarrollo omiten revisiones exhaustivas de código y pruebas de seguridad para cumplir con plazos estrictos, dejando bugs sin parchear y controles de acceso faltantes.
- **Integración de sistemas complejos:** Cada biblioteca externa, API de terceros o microservicio añade riesgo a la cadena de suministro. Una vulnerabilidad en una dependencia se propaga a la aplicación principal.
- **Mala gestión de configuración:** Desplegar servidores o bases de datos en la nube con configuraciones predeterminadas suele dejar contraseñas de administrador en blanco, puertos abiertos y acceso público a datos habilitado. Escáneres automáticos buscan estos sistemas mal configurados.
- **Error humano:** Empleados atacados por correos de phishing, mensajes SMS fraudulentos o llamadas engañosas pueden exponer credenciales o transferir datos a atacantes sin querer.
- **Fusiones organizacionales:** Conectar rápidamente dos redes corporativas diferentes crea puntos ciegos como pérdida de visibilidad sobre servidores antiguos, cortafuegos mal configurados y permisos de acceso excesivos.

## Amenazas comunes de seguridad y mitigaciones

| Amenaza | Cómo funciona | Mitigación |
|---|---|---|
| **Phishing** | Correos fraudulentos imitan marcas confiables para robar credenciales | Capacitación en conciencia de seguridad; protocolos de autenticación de correo (SPF, DKIM, DMARC) |
| **Malware (ransomware, spyware)** | Encripta o roba archivos de la víctima; exige pago | Parcheo automatizado; software antimalware |
| **Credential stuffing** | Herramientas automatizadas prueban millones de pares usuario/contraseña robados en sitios | Aplicar autenticación multifactor (MFA) |
| **DDoS** | Inunda un servidor con tráfico falso para colapsarlo | Servicios de filtrado de tráfico en la nube (p. ej., Cloudflare) |
| **Amenazas internas** | Empleados descontentos o descuidados filtran o roban datos | Principio de menor privilegio; monitorear registros del sistema por accesos inusuales |

## Política de seguridad de contenido (CSP) y Permissions-Policy

**Política de seguridad de contenido (CSP)** es un encabezado de respuesta HTTP que restringe qué recursos externos puede cargar un navegador en una página. Defiende principalmente contra ataques de **Cross-Site Scripting (XSS)** al permitir solo dominios fuente confiables para scripts, estilos e imágenes.

Si un script malicioso de un dominio no aprobado intenta ejecutarse, el navegador lo bloquea y registra un error de seguridad.

**Permissions-Policy** (antes Feature-Policy) controla el acceso a funcionalidades del navegador y hardware del dispositivo como cámara, micrófono y geolocalización. Puedes restringir una funcionalidad a tu propio dominio o deshabilitarla completamente:

```http
Permissions-Policy: camera=(self), geolocation=()
```

`camera=(self)`: solo tu propio dominio puede acceder a la cámara.
`geolocation=()`: la geolocalización está deshabilitada para todos en la página.

Juntas, CSP bloquea la entrega de datos y previene la inyección de código; Permissions-Policy protege el hardware del dispositivo y asegura el cumplimiento de privacidad.

## Conceptos fundamentales en la privacidad del usuario

| Concepto | Definición |
|---|---|
| **PII (Información personal identificable)** | Cualquier dato que pueda identificar directa o indirectamente a una persona. Identificadores directos: nombre, dirección, número de identificación. Identificadores indirectos: dirección IP, número de serie del dispositivo, historial de ubicación. |
| **Confidencialidad** | Garantizar que los datos privados permanezcan inaccesibles para partes no autorizadas mediante encriptación y controles de acceso. |
| **Minimización de datos** | Recopilar solo la cantidad mínima de PII necesaria para completar una tarea específica. |
| **Limitación de propósito** | Los datos recopilados para un motivo no pueden usarse legalmente para otro (p. ej., vender información de contacto a anunciantes). |
| **Rastreo** | Monitoreo continuo de la actividad en línea usando scripts, huellas digitales de dispositivos y registros del historial del navegador para crear perfiles de usuario. |
| **Consentimiento informado** | Los usuarios deben optar activamente por la recopilación de datos. Las casillas pre-marcadas no constituyen consentimiento válido. |
| **Derecho al olvido** | Los usuarios pueden solicitar legalmente que una organización elimine permanentemente todos sus datos personales y el historial de rastreo. |

## Leyes regionales de privacidad

| Ley | Jurisdicción | Requisitos clave |
|---|---|---|
| **GDPR** | Unión Europea | Consentimiento estricto opt-in requerido; los usuarios tienen derecho a acceder y eliminar sus datos; multas severas por violaciones. |
| **CCPA / CPRA** | California, EE. UU. | Modelo opt-out; las empresas deben mostrar un enlace "No vender mi información personal". |
| **COPPA** | EE. UU. (federal) | Consentimiento parental verificable requerido antes de recopilar datos de niños menores de 13 años. |
| **DPA / UK GDPR** | Reino Unido | Procesamiento de datos legal, justo y transparente; individuos tienen derechos de acceso y eliminación. |
| **NDPA** | Nigeria | Regula la recopilación, almacenamiento y uso de datos personales de residentes nigerianos. |
| **DPDP** | India | Regula cómo se usan los datos personales de residentes indios. |

**Pasos prácticos para el cumplimiento:**

1. **Banners de consentimiento con detección de ubicación:** Detecta la ubicación geográfica del usuario y muestra banners de cookies que cumplen con las regulaciones locales.
2. **Sistema automatizado DSAR:** Gestiona solicitudes de acceso de sujetos de datos para exportar o eliminar de forma segura todo el historial de datos de un usuario.
3. **Minimización de datos y control de edad:** Recopila solo lo necesario y usa verificación de edad para proteger a menores.
4. **Oficial de protección de datos (DPO):** Designa un experto para supervisar la arquitectura de privacidad y gestionar el cumplimiento regulatorio.

## El archivo `.env` y la biblioteca `dotenv`

Un **archivo `.env`** es un archivo de texto plano ubicado en la raíz de un proyecto que almacena datos sensibles de configuración como pares clave-valor. Estos incluyen claves API, contraseñas de base de datos y números de puerto. Los mantiene fuera del código fuente y lejos de repositorios públicos.

```bash
PORT=3000
DB_HOST=localhost
API_KEY=xyz123secretkey
```

Siempre agrega `.env` a `.gitignore` para que nunca se suba al control de versiones.

El **paquete `dotenv`** lee el archivo `.env` al iniciar y inyecta las claves en `process.env`:

```javascript
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

Dos beneficios clave:

- **Seguridad:** Las credenciales sensibles nunca aparecen en el código fuente ni en el historial de versiones.
- **Flexibilidad:** La misma base de código se ejecuta en desarrollo, pruebas y producción simplemente cambiando el archivo `.env` en cada servidor.
