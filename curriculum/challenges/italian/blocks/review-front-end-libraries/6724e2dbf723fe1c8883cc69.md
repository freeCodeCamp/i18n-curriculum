## Librerie e framework JavaScript

- Le librerie e i framework JavaScript offrono soluzioni rapide a problemi comuni e accelerano lo sviluppo fornendo codice predefinito.
- Le librerie sono generalmente più focalizzate nel fornire soluzioni a compiti specifici, come manipolare il DOM, gestire eventi o gestire richieste AJAX.
- Un paio di esempi di librerie JavaScript sono jQuery e React.
- I framework, invece, forniscono una struttura più definita per costruire applicazioni. Spesso includono un insieme di regole e convenzioni che gli sviluppatori devono seguire.
- Esempi di framework includono Angular e Next.js, un meta framework per React.
- Le **single-page application** (SPA) sono applicazioni web che caricano una singola pagina HTML e aggiornano dinamicamente quella pagina mentre l'utente interagisce con l’applicazione senza ricaricare l’intera pagina.
- Le SPA usano JavaScript per gestire lo stato dell’applicazione e renderizzare il contenuto. Questo viene spesso fatto usando framework che offrono ottimi strumenti per costruire interfacce utente complesse.
- Alcuni problemi legati alle SPA includono:

  - I lettori di schermo hanno difficoltà con contenuti aggiornati dinamicamente.
  - L’URL non cambia quando l’utente naviga all’interno dell’applicazione, il che può rendere difficile aggiungere ai preferiti, tornare indietro o condividere pagine specifiche.
  - Il tempo di caricamento iniziale può essere lento se l’applicazione è grande, poiché tutte le risorse devono essere caricate subito.

## React

- React è una popolare libreria JavaScript per costruire interfacce utente e applicazioni web.
- Un concetto fondamentale di React è la creazione di componenti UI riutilizzabili che possono aggiornarsi e renderizzarsi indipendentemente quando i dati cambiano.
- React permette agli sviluppatori di descrivere come dovrebbe apparire l’interfaccia utente in base allo stato dell’applicazione. React poi aggiorna e renderizza i componenti corretti quando i dati o lo stato cambiano.

## Componenti React

- I componenti sono i mattoni delle applicazioni React che permettono agli sviluppatori di suddividere interfacce utente complesse in pezzi più piccoli e gestibili.
- L’interfaccia utente viene descritta usando JSX, un’estensione della sintassi JavaScript, che permette agli sviluppatori di scrivere codice simile all’HTML all’interno di JavaScript.
- I componenti sono fondamentalmente funzioni JS o classi che restituiscono un pezzo di interfaccia utente.

Ecco un esempio di un semplice componente React che renderizza un messaggio di saluto:

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

Per usare il componente, puoi semplicemente chiamare:
    
```jsx
  <Greeting />
```

## Importare ed esportare componenti React

- Un export rende un componente disponibile ad altri file. Un import permette a un file di usare un componente esportato altrove.
- Ci sono due tipi di export in React: export predefiniti (default) e export nominati.

**Export predefiniti**

- Un file può avere solo un export predefinito. È ideale per un file che contiene principalmente un singolo componente.
- Puoi esportare un componente come export predefinito così:

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default City;
```

- Puoi anche esportare sulla stessa riga della definizione del componente:

```jsx
export default function City() {
  return <p>New York</p>;
}
```

- Per importare un export predefinito, usa la parola chiave `import` senza parentesi graffe:

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

**Export nominati**

- Gli export nominati permettono a un file di condividere più componenti. A differenza degli export predefiniti, devono essere importati usando esattamente il nome con cui sono stati esportati.
- Puoi esportare più componenti dallo stesso file così:

```jsx
// Animals.jsx
export function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- Per importare export nominati, usa le parentesi graffe con i nomi esatti esportati:

```jsx
import { Cat, Dog } from './Animals';
```

- Puoi rinominare un export nominato durante l’importazione usando la parola chiave `as`:

```jsx
import { Cat as Kitty } from './Animals';
```

**Export misti predefiniti e nominati**

- Un file può avere un export predefinito e più export nominati contemporaneamente:

```jsx
// Animals.jsx
export default function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- Quando importi export misti, l’export predefinito viene prima (senza parentesi), seguito dagli export nominati (dentro parentesi):

```jsx
import Cat, { Dog } from './Animals';
```

## Configurare un progetto React usando Vite

- Gli strumenti di configurazione progetto e le interfacce a riga di comando (CLI) offrono un modo rapido e semplice per iniziare nuovi progetti, permettendo agli sviluppatori di concentrarsi sul codice invece che sulla configurazione.
- Vite è uno strumento popolare per configurare progetti e può essere usato con React.
- Per creare un nuovo progetto con Vite, puoi usare questo comando nel terminale:

```bash
npm create vite@latest my-react-app -- --template react
```

Questo comando crea un nuovo progetto React chiamato `my-react-app` usando il template React di Vite. Nella directory del progetto vedrai un file `package.json` con elencate le dipendenze e i comandi del progetto.

- Per eseguire il progetto, vai nella directory del progetto ed esegui questi comandi:

```bash
cd my-react-app # path to the project directory
npm install # installs the dependencies listed in the package.json file
```

- Una volta installate le dipendenze, noterai una nuova cartella nel progetto chiamata `node_modules`.

- La cartella `node_modules` è dove sono archiviate tutte le librerie e i pacchetti richiesti dal progetto.

- Per eseguire il progetto, usa questo comando:

```bash
npm run dev
```

- Dopodiché, apri il browser e vai a `http://localhost:5173` per vedere la tua applicazione React in esecuzione.

- Per vedere effettivamente il codice del template iniziale, puoi entrare nella cartella `src` del progetto e dovresti vedere il file `App.jsx`.

## Passare props nei componenti React

- In React, le props (abbreviazione di proprietà) sono un modo per passare dati da un componente genitore a un componente figlio. Questo meccanismo serve per creare elementi UI riutilizzabili e dinamici.
- Le props possono essere qualsiasi valore JavaScript. Per passare props da un genitore a un figlio, aggiungi le props come attributi quando usi il componente figlio nel JSX del genitore. Ecco un esempio semplice:

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

Puoi passare più props usando l’operatore di espansione `(...)`, dopo averle convertite in un oggetto. Ecco un esempio:
    
```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

In questo codice, l’operatore di espansione `{...person}` converte l’oggetto person in singole props che vengono passate al componente Child.

## Rendering condizionale in React

- Il rendering condizionale in React permette di creare interfacce utente dinamiche. Serve per mostrare contenuti diversi in base a certe condizioni o stati nell’applicazione.
- Ci sono diversi modi per fare rendering condizionale in React. Un approccio comune è usare l’operatore ternario. Ecco un esempio:

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- Un altro modo per fare rendering condizionale è usare l’operatore logico AND (`&&`). Questo è utile quando vuoi renderizzare contenuti solo se una certa condizione è vera. Ecco un esempio:

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

Nel codice sopra, l’elemento `h1` viene renderizzato solo se l’oggetto user è truthy.

Puoi anche usare un’istruzione `if` diretta in questo modo:

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```

## Rendering di liste in React

- Renderizzare liste in React è un’attività comune quando si costruiscono interfacce utente.
- Le liste possono essere renderizzate usando il metodo `map()` dell’array JS per iterare su un array di elementi e restituire un nuovo array di elementi JSX.
- Per esempio, se hai un array di nomi che vuoi renderizzare come lista, puoi fare così:

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- Ricorda sempre di fornire una chiave unica per ogni elemento della lista per aiutare React a gestire aggiornamenti e rendering. Con queste tecniche puoi creare liste flessibili, efficienti e dinamiche nelle tue applicazioni React.

## Stili in linea in React

- Gli stili in linea in React permettono di applicare stili CSS direttamente agli elementi JSX usando oggetti JavaScript.
- Per applicare stili in linea in React, puoi usare l’attributo style sugli elementi JSX. L’attributo style prende un oggetto dove le chiavi sono proprietà CSS in camelCase e i valori sono i valori corrispondenti. Ecco un esempio:

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

Puoi anche estrarre gli stili in un oggetto separato e riferirlo nell’attributo `style` così:

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Gli stili in linea supportano lo stile dinamico permettendo di applicare condizionalmente stili basati su props o stato. Ecco un esempio di come applicare condizionalmente stili basati su una prop:

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Nel codice sopra, gli stili `color` e `fontSize` sono impostati condizionalmente in base alla prop `isImportant`.

## Gestire eventi in React

- **Sistema di eventi sintetici**: Questo è il modo in cui React gestisce gli eventi. Funziona come un wrapper attorno agli eventi nativi come `click`, `keydown` e `submit`. I gestori di eventi in React usano la convenzione camelCase per i nomi. (Es. `onClick`, `onSubmit`, ecc.)

Ecco un esempio di utilizzo dell’attributo `onClick` per un elemento `button` in React:

```jsx
function handleClick() {
  console.log("Button clicked!");
}

<button onClick={handleClick}>Click Me</button>;
```

In React, le funzioni gestore di eventi di solito iniziano con il prefisso `handle` per indicare che gestiscono eventi, come `handleClick` o `handleSubmit`.

Quando un’azione utente scatena un evento, React passa un oggetto Evento Sintetico al gestore. Questo oggetto si comporta molto come l’oggetto Evento nativo in JavaScript vanilla, fornendo proprietà come `type`, `target` e `currentTarget`.

Per prevenire comportamenti predefiniti come il refresh del browser durante un evento `onSubmit`, per esempio, puoi chiamare il metodo `preventDefault()`:

```jsx
function handleSubmit(event) {
  event.preventDefault();
  console.log("Form submitted!");
}

<form onSubmit={handleSubmit}>
  <input type="text" />
  <button>Submit</button>
</form>;
```

Puoi anche avvolgere una funzione gestore in una funzione freccia così:

```jsx
function handleDelete(id) {
  console.log("Deleting item:", id);
}

<button onClick={() => handleDelete(1)}>Delete Item</button>;
```

## Gestire lo stato e il hook `useState`

- **Definizione di stato**: In React, lo stato è un oggetto che contiene dati per un componente. Quando lo stato si aggiorna, il componente si ri-renderizza. React tratta lo stato come immutabile, quindi non dovresti modificarlo direttamente.
- **Hook `useState()`**: Il hook `useState` è una funzione che ti permette di dichiarare variabili di stato nei componenti funzionali. Ecco la sintassi base:

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

Nella variabile di stato hai:

- `stateVariable` contiene il valore corrente dello stato
- `setStateFunction` (la funzione setter) aggiorna la variabile di stato
- `initialValue` imposta lo stato iniziale

Ecco un esempio completo per un componente `Counter`:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>

      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```

## Rendering e componenti React

- **Definizione**: In React, il rendering è il processo con cui i componenti appaiono nell’interfaccia utente (UI), solitamente nel browser. Il processo di rendering consiste in tre fasi: trigger, render e commit.

La fase di trigger avviene quando React rileva che qualcosa è cambiato e l’interfaccia utente potrebbe aver bisogno di essere aggiornata. Questo cambiamento è spesso dovuto a un aggiornamento di stato o props.

Una volta avvenuto il trigger, React entra nella fase di render. Qui React rivaluta i componenti e decide cosa mostrare. Per farlo, React usa una copia leggera del DOM “reale” chiamata virtual DOM. Con il virtual DOM, React può rapidamente verificare cosa deve cambiare nel componente.

La fase di commit è dove React prende le modifiche preparate dal virtual DOM e le applica al DOM reale. In altre parole, questa è la fase in cui vedi il risultato finale sullo schermo.

## Aggiornare oggetti e array nello stato

- **Aggiornare oggetti nello stato**: Se devi aggiornare un oggetto nello stato, dovresti prima creare un nuovo oggetto o copiare un oggetto esistente, poi impostare lo stato con quel nuovo oggetto. Qualsiasi oggetto messo nello stato dovrebbe essere considerato di sola lettura. Ecco un esempio di impostazione del nome, età e città di un utente. La funzione `handleChange` viene usata per gestire gli aggiornamenti delle informazioni dell’utente:

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **Aggiornare array nello stato**: Quando aggiorni array nello stato, è importante non modificare direttamente l’array usando metodi come `push()` o `pop()`. Invece, dovresti creare un nuovo array quando aggiorni lo stato:

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

Se vuoi rimuovere elementi da un array, dovresti usare il metodo `filter()`, che restituisce un nuovo array dopo aver filtrato ciò che vuoi rimuovere:

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```

## Riferimenti ai valori usando i Ref

- **Attributo `ref`**: Puoi consultare un nodo DOM in React usando l’attributo `ref`. Ecco un esempio che mostra un `ref` per mettere il focus su un elemento `input`. La proprietà `current` viene usata per accedere al valore corrente di quel `ref`:

```jsx
import { useRef } from "react";

const Focus = () => {
  const inputRef = useRef(null);

  const handleFocus = () => {
    if (inputRef.current) {
      inputRef.current.focus();
    }
  };

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Enter text" />
      <button onClick={handleFocus}>Focus Input</button>
    </div>
  );
};

export default Focus;
```

## Lavorare con il hook `useEffect`

- **Hook `useEffect()`**: In React, un effetto è qualsiasi cosa che succede al di fuori del processo di rendering del componente. Cioè, qualsiasi cosa che React non gestisce direttamente come parte del rendering dell’interfaccia utente. Esempi comuni includono il recupero dati, aggiornare il titolo della scheda del browser, leggere o scrivere nella memoria locale del browser, ottenere la posizione dell’utente e molto altro. Queste operazioni interagiscono con il mondo esterno e sono chiamate effetti collaterali. React fornisce il hook `useEffect` per gestire questi effetti collaterali. `useEffect` ti permette di eseguire una funzione dopo che il componente è stato renderizzato o aggiornato.

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [dependencies]);
```

La funzione effetto viene eseguita dopo il rendering del componente, mentre l’argomento opzionale `dependencies` controlla quando l’effetto viene eseguito.

Nota che `dependencies` può essere un array di “valori reattivi” (stato, props, funzioni, variabili, ecc.), un array vuoto o essere omesso del tutto. Ecco come queste opzioni controllano il funzionamento di `useEffect`:

- Se `dependencies` è un array che include uno o più valori reattivi, l’effetto verrà eseguito ogni volta che cambiano.

- Se `dependencies` è un array vuoto, `useEffect` viene eseguito solo una volta quando il componente viene renderizzato la prima volta.

- Se ometti `dependencies`, l’effetto viene eseguito ogni volta che il componente viene renderizzato o aggiornato.

## Come creare hook personalizzati

- **Hook personalizzati**: Un hook personalizzato ti permette di estrarre logica riutilizzabile dai componenti, come il recupero dati, la gestione dello stato, il toggle e gli effetti collaterali come il monitoraggio dello stato online. In React, tutti gli hook integrati iniziano con la parola `use`, quindi il tuo hook personalizzato dovrebbe seguire la stessa convenzione.

Ecco un esempio di creazione di un hook `useDebounce`:

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```

## Lavorare con i moduli in React

- **Input controllati**: Questo è quando memorizzi il valore del campo di input nello stato e lo aggiorni tramite eventi `onChange`. Questo ti dà il controllo completo sui dati del modulo e permette validazione istantanea e rendering condizionale.

```jsx
import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **Input non controllati**: Invece di gestire gli input tramite il hook `useState`, gli input non controllati in HTML mantengono il proprio stato interno con l’aiuto del DOM. Poiché il DOM controlla i valori degli input, devi prelevare i valori dei campi di input con un `ref`.

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```

## Lavorare con il hook `useActionState`

- **Azioni server**: Sono funzioni che vengono eseguite sul server per permettere la gestione dei moduli direttamente sul server senza bisogno di endpoint API. Ecco un esempio da un’applicazione Next.js:

```js
"use server";

async function submitForm(formData) {
 const name = formData.get("name");
 return { message: `Hello, ${name}!` };
}
```

La direttiva `"use server"` marca la funzione come azione server.

- **Hook `useActionState`**: Questo hook aggiorna lo stato in base al risultato di un invio di modulo. Ecco la sintassi base del hook `useActionState`:

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state` è lo stato corrente che l’azione restituisce.
- `action` è la funzione che attiva l’azione server.
- `isPending` è un booleano che indica se l’azione è in esecuzione o meno.
- `actionFunction` è il parametro che rappresenta l’azione server stessa.
- `initialState` è il parametro che rappresenta il punto di partenza per lo stato prima che l’azione venga eseguita.
- `permalink` è una stringa opzionale che contiene l’URL unico della pagina che il modulo modifica.

## Recupero dati in React

- **Opzioni per recuperare dati**: Ci sono molti modi diversi per recuperare dati in React. Puoi usare la Fetch API nativa o strumenti di terze parti come Axios o SWR.
- **Variabili di stato comunemente usate durante il recupero dati**: Indipendentemente dal metodo scelto per recuperare dati in React, ci sono alcune variabili di stato da tenere sotto controllo. La prima è il dato stesso. La seconda tiene traccia se i dati sono ancora in fase di recupero. La terza è una variabile di stato che cattura eventuali errori che possono verificarsi durante il recupero dati.

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

Poiché il recupero dati è un effetto collaterale, è meglio usare `Fetch API` all’interno di un hook `useEffect`.

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

Poi puoi renderizzare un messaggio di caricamento se il recupero dati non è completo, un messaggio di errore se c’è stato un errore, o i risultati.

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

## Lavorare con il hook `useOptimistic`

- **Hook `useOptimistic`**: Questo hook serve a mantenere le interfacce utente responsivi mentre si aspetta che un’azione asincrona si completi in background. Aiuta a gestire gli “aggiornamenti ottimistici” nell’interfaccia, una strategia in cui fornisci aggiornamenti immediati basati sull’esito atteso di un’azione, come aspettare una risposta dal server.

Ecco la sintassi base:

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState` è lo stato temporaneo che si aggiorna subito per una migliore esperienza utente.
- `addOptimistic` è la funzione che applica l’aggiornamento ottimistico prima che lo stato reale cambi.
- `actualState` è il valore reale dello stato che deriva dal risultato di un’azione, come il recupero dati da un server.
- `updateFunction` è la funzione che determina come lo stato ottimistico deve aggiornarsi quando viene chiamata.

Ecco un esempio di utilizzo del hook `useOptimistic` in un componente `TaskList`:

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**: Serve a renderizzare una parte dell’interfaccia e a marcare un aggiornamento di stato come non urgente. Questo permette all’interfaccia di rimanere responsiva durante aggiornamenti costosi. Ecco la sintassi base:

```js
startTransition(action)
```

`action` esegue un aggiornamento di stato o attiva una logica legata alla transizione. Questo assicura che aggiornamenti urgenti dell’interfaccia (come digitare o cliccare) non vengano bloccati.

## React Router

- **Introduzione**: React Router è una libreria di terze parti che permette di aggiungere il routing alle applicazioni React. Per iniziare, devi installare React Router in un progetto React esistente così:

```bash

```

Poi, all’interno del file `main.jsx` o `index.jsx`, devi configurare la struttura delle rotte così:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(

    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

`path` e `element` vengono usati per collegare URL e componenti UI insieme. In questo caso, stiamo configurando una rotta per la homepage che punta al componente `App`.

- **Visualizzazioni multiple e configurazione delle rotte**: È comune in applicazioni più grandi avere più viste e rotte configurate così:

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

La prop `index` in questi esempi rappresenta la rotta predefinita per un dato segmento di percorso. Quindi il componente `Home` sarà mostrato al percorso radice `/`, mentre il componente `ProductsHome` sarà mostrato al percorso `/products`.

- **Annidare rotte**: Puoi annidare rotte dentro altre rotte, il che fa sì che il percorso della rotta figlia venga aggiunto al percorso della rotta genitore.

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

Nell’esempio sopra, il percorso per i prodotti di tendenza sarà products/trending.

- **Segmenti dinamici**: Un segmento dinamico è quando una parte del percorso URL è dinamica.

```jsx
<Route path=":category" element={<Category />} />
```

In questo esempio abbiamo un segmento dinamico chiamato `category`. Quando un utente naviga a un URL come products/brass-instruments, la vista cambia al componente `Category` e puoi recuperare dinamicamente i dati appropriati basandoti sul segmento.

## Gestione dello stato

- **Context API**: Il contesto si riferisce a quando un componente genitore rende informazioni disponibili ai componenti figli senza doverle passare esplicitamente tramite props. `createContext` viene usato per creare un oggetto contesto che rappresenta il contesto che altri componenti leggeranno. `Provider` viene usato per fornire valori di contesto ai componenti figli.

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**: Redux gestisce lo stato fornendo un archivio centrale e un controllo rigoroso sugli aggiornamenti di stato. Usa un pattern prevedibile con azioni, reducer e middleware. Le azioni sono payload di informazioni che inviano dati dall’applicazione all’archivio Redux, spesso attivate da interazioni utente. I reducer sono funzioni che specificano come lo stato deve cambiare in risposta a quelle azioni, assicurando che lo stato venga aggiornato in modo immutabile. Il middleware, invece, agisce come ponte tra il dispatch delle azioni e il reducer, permettendo di estendere le funzionalità di Redux (es. logging, gestione di operazioni asincrone) senza modificare il flusso principale.

- **Zustand**: Questa soluzione di gestione dello stato è ideale per applicazioni di piccole e medie dimensioni. Funziona usando un hook `useStore` per accedere direttamente allo stato nei componenti e nelle pagine. Questo ti permette di modificare e consultare i dati senza bisogno di azioni, reducer o provider.

## Differenze tra prestazioni reali e percepite

- **Prestazioni percepite**: È come gli utenti percepiscono le prestazioni di un sito web. È come lo valutano in termini di reattività e affidabilità. È una misura soggettiva, quindi difficile da quantificare, ma molto importante perché l’esperienza utente determina il successo o il fallimento di un sito.
- **Prestazioni reali**: Sono le prestazioni oggettive e misurabili del sito. Vengono misurate con metriche come il tempo di caricamento pagina, il tempo di risposta del server e il tempo di rendering. Queste misurazioni sono influenzate da molti fattori legati alla rete e al codice stesso.

## Tecniche per migliorare le prestazioni percepite

- **Lazy loading**: Questa tecnica riduce il tempo di caricamento iniziale caricando risorse non essenziali in background.
- **Minimizzare i ritardi nel caricamento dei caratteri**: Se il tuo sito usa caratteri personalizzati, dovresti cercare di minimizzare i ritardi nel caricamento dei font, perché questo può causare sfarfallii o mostrare il font di fallback mentre il font personalizzato si carica. Un suggerimento è usare un font di fallback simile a quello personalizzato, così il cambiamento sarà più sottile.
- **Uso di indicatori di caricamento**: Mostrare un indicatore di caricamento per un processo lungo appena l’utente clicca su un elemento può aiutare l’utente a sentirsi connesso e coinvolto, facendo sembrare il tempo di attesa più breve.

## Concetti fondamentali sulle prestazioni

- **Ordine sorgente**: Si riferisce a come gli elementi HTML sono strutturati nel documento. Questo determina cosa viene caricato prima e può influenzare significativamente prestazioni e accessibilità.

Alcune buone pratiche per l’ordine sorgente includono:

- Mettere contenuti critici come intestazioni, navigazione o testo principale più in alto nella struttura HTML.
- Rinviare script non essenziali come quelli per analytics o widget di terze parti, così non bloccano il rendering.
- Usare il progressive enhancement, per assicurare che l’esperienza base funzioni anche prima che stili e script vengano caricati. Il progressive enhancement è un modo di costruire siti e applicazioni basato sull’idea che la pagina dovrebbe funzionare con HTML prima di tutto.

Ecco un esempio di buon ordine sorgente, usando le buone pratiche appena viste:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **Percorso critico di rendering**: È la sequenza di passaggi che il browser segue per convertire il codice in pixel sullo schermo.
- **Latenza**: È il tempo che impiega una richiesta a viaggiare tra browser e server. In altre parole, alta latenza significa pagine lente.

Alcuni modi per ridurre la latenza includono:

- Usare CDN, o Content Delivery Network, per servire file da posizioni più vicine.
- Abilitare la compressione usando strumenti come Gzip per ridurre la dimensione dei file.
- Ottimizzare le immagini e usare il lazy loading.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## Migliorare INP

- **Definizione**: INP (Interaction to Next Paint) valuta la reattività complessiva di una pagina misurando il tempo da quando un utente interagisce, come un clic o una pressione di tasto, al prossimo aggiornamento del browser. Un INP più basso indica una pagina più reattiva.

Ecco alcuni modi per migliorare INP:

- Ridurre il lavoro del thread principale spezzando compiti JavaScript lunghi.
- Usare `requestIdleCallback()` per script non critici. Questo mette in coda una funzione da chiamare durante i periodi di inattività del browser.
- Rinviare o caricare in modo lazy risorse pesanti come visto prima.
- Ottimizzare i gestori di eventi. Se questi gestori vengono eseguiti troppo frequentemente o fanno operazioni pesanti, possono rallentare la pagina e aumentare INP. La soluzione è il debouncing. Il debouncing assicura che la funzione venga eseguita solo dopo che l’utente ha smesso di digitare per un breve ritardo — per esempio 300ms. Questo evita calcoli inutili e migliora le prestazioni.

## Come funziona il rendering nel browser

- **Come funziona il rendering**: Prima il browser analizza l’HTML e costruisce il DOM. Poi processa il CSS, costruendo il CSS Object Model, o CSSOM. Questa è un’altra struttura ad albero che determina come gli elementi devono essere stilizzati. Infine, il browser dipinge i pixel sullo schermo, renderizzando ogni elemento basandosi sugli stili e il layout calcolati. In pagine complesse, questo può coinvolgere più livelli che vengono composti insieme per formare l’output visivo finale.

## Come le prestazioni influenzano la sostenibilità

- **Informazioni di base**: Internet rappresenta circa il 2% delle emissioni globali di carbonio — pari all’industria aerea! Ogni byte trasferito richiede elettricità, dai data center ai dispositivi degli utenti. File più grandi e script inefficienti significano più consumo di energia. Un sito ad alte prestazioni non è solo più veloce, ma riduce anche elaborazioni e consumi energetici inutili.

## Modi per ridurre i tempi di caricamento della pagina

- **Ottimizzare le risorse multimediali**: Immagini e video grandi sono spesso la causa di caricamenti lenti. Ottimizzando queste risorse puoi velocizzare significativamente il sito. Questo include comprimere immagini, usare formati moderni come WebP e usare il lazy loading per le risorse.
- **Sfruttare la cache del browser**: La cache permette ai browser di memorizzare parti del sito localmente, riducendo i tempi di caricamento per i visitatori di ritorno.
- **Minificare e comprimere i file**: Ridurre la dimensione dei file porta a download più rapidi. Questo include ridurre la dimensione dei file trasmessi e minificare CSS e JavaScript.

## Migliorare il "tempo di usabilità"

- **Definizione**: Il "tempo di usabilità" è l’intervallo da quando un utente richiede una pagina a quando può interagire in modo significativo con essa. Per migliorare il "tempo di usabilità" puoi caricare in modo lazy le risorse o minimizzare le risorse che bloccano il rendering.

## Metriche chiave per misurare le prestazioni

- **First Contentful Paint o FCP**: Misura quanto velocemente il primo contenuto — testo o immagine — appare sullo schermo. Un buon FCP è considerato sotto 1,8 secondi, mentre un FCP scarso è sopra i 3 secondi. Puoi controllare il tuo FCP usando Chrome DevTools nella scheda performance.
- **Total Blocking Time**: Mostra per quanto tempo il thread principale è bloccato da compiti JavaScript pesanti. Se il Total Blocking Time (TBT) è alto, gli utenti sperimentano interazioni lente. Per migliorare TBT, spezza compiti lunghi e rinvia script non essenziali.
- **Bounce Rate**: È la percentuale di visitatori che lasciano senza interagire. Se il tuo sito ha un alto bounce rate potrebbe essere perché la pagina è troppo lenta.
- **Utenti unici**: Questa metrica traccia quanti visitatori individuali arrivano sul sito. Per vedere Bounce Rate e Utenti unici puoi usare Google Analytics. Ti permette di monitorare queste metriche e migliorare l’engagement.

## Strumenti comuni per misurare le prestazioni

- **Chrome DevTools**: Chrome DevTools è uno strumento integrato in Google Chrome che ti permette di analizzare e fare il debug delle prestazioni in tempo reale. DevTools mostra tempi di caricamento, uso della CPU e ritardi nel rendering. È particolarmente utile per misurare il First Contentful Paint, cioè quanto velocemente un utente vede il primo contenuto visibile. Se il tuo sito sembra lento, DevTools ti aiuta a individuare i colli di bottiglia.
- **Lighthouse**: È uno strumento automatizzato che verifica prestazioni, SEO e accessibilità.
- **WebPageTest**: Questo strumento ti permette di testare come il sito si carica da diverse località e dispositivi. Fornisce un’analisi dettagliata di Speed Index, Total Blocking Time e altre metriche chiave. Se vuoi sapere come gli utenti reali sperimentano il sito a livello globale, WebPageTest è lo strumento giusto.
- **PageSpeed Insights**: Analizza il sito e suggerisce miglioramenti rapidi per mobile e desktop. Ti dice cosa rallenta il sito e fornisce raccomandazioni specifiche come ottimizzare immagini, rimuovere script che bloccano il rendering e ridurre i tempi di risposta del server. PageSpeed Insights è un modo veloce e semplice per vedere come Google valuta le prestazioni del sito.

Se vuoi dati da utenti reali, gli strumenti RUM sono essenziali.

## Lavorare con le Performance Web API

- **Definizione**: Le Performance Web API permettono agli sviluppatori di tracciare quanto efficientemente una pagina web si carica e risponde direttamente nel codice. Queste API ti permettono di misurare i tempi di caricamento pagina, tracciare ritardi di rendering e interazione e analizzare il tempo di esecuzione di JavaScript.
- **`performance.now()`**: Questa API ti dà timestamp ad alta precisione (in millisecondi) per misurare quanto tempo impiegano diverse parti del sito a caricarsi.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: Questa API fornisce una suddivisione di ogni fase del caricamento pagina, dalla risoluzione DNS a `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: Questa API ascolta eventi di performance come spostamenti di layout, compiti lunghi e interazioni utente.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## Tecniche per migliorare le prestazioni CSS

- **Animazioni CSS**: Animare certe proprietà CSS, come dimensioni, posizione e layout, attiva un processo chiamato “reflow”, durante il quale il browser ricalcola posizione e geometria di certi elementi nella pagina. Questo richiede un repaint, che è computazionalmente costoso. Perciò si consiglia di ridurre il numero di animazioni CSS il più possibile o almeno dare all’utente la possibilità di attivarle o disattivarle.

## Tecniche per migliorare le prestazioni JavaScript

- **Code splitting**: Dividere il codice JavaScript in moduli che eseguono compiti critici e non critici è utile. Così puoi precaricare quelli critici il prima possibile e rinviare quelli non critici per rendere la pagina più veloce.
- **Manipolazione del DOM**: Ricorda che la manipolazione del DOM è il processo di cambiare dinamicamente il contenuto di una pagina con JavaScript interagendo con il Document Object Model (DOM). Manipolare il DOM è computazionalmente costoso. Ridurre la quantità di manipolazione del DOM nel codice JavaScript migliora le prestazioni.

## Framework CSS

- **Framework CSS**: I framework CSS possono velocizzare il tuo flusso di lavoro, creare uno stile visivo uniforme su un sito, rendere il design coerente su più browser e mantenere il codice CSS più organizzato.
- **Framework CSS popolari**: Alcuni framework CSS popolari sono Tailwind CSS, Bootstrap, Materialize e Foundation.
- **Possibili svantaggi**:

  - Il tuo sito potrebbe sembrare simile ad altri siti che usano lo stesso framework.
  - I framework grandi potrebbero causare problemi di prestazioni.

Puoi assegnare queste piccole classi direttamente agli elementi HTML secondo necessità. Tailwind CSS è classificato come framework CSS utility-first.

Ecco un esempio di uso di Bootstrap per creare un gruppo di liste. Invece di applicare piccole classi agli elementi HTML, aggiungerai l’intero componente, inclusa la struttura HTML.

<div class="card" style="width: 25rem;">

## Preprocessori CSS

## Test manuali e automatizzati

- **Test manuale**: Nel test manuale, un tester esamina manualmente ogni parte dell’applicazione e prova diverse funzionalità per assicurarsi che funzionino correttamente. Se vengono trovati bug, il tester li segnala al team software per la correzione.
- **Test automatizzato**: Nel test automatizzato, puoi automatizzare i test scrivendo un programma separato che verifica se l’applicazione si comporta come previsto.

## Test unitari

- **Test unitari**: Nei test unitari, testi ogni funzione per assicurarti che tutto funzioni come previsto. I test unitari possono anche servire come documentazione per l’applicazione perché rappresentano il comportamento atteso del codice.
- **Principio di responsabilità singola**: Consiglia di mantenere ogni funzione piccola e responsabile di una sola cosa.
- **Framework di test JavaScript comuni**: Alcuni framework comuni sono Jest, Mocha e Vitest. Jest è un framework popolare per test unitari.

Ecco un esempio di test unitari usando Jest.

Prima, puoi creare una funzione responsabile di restituire una stringa formattata:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

In un file `getFormattedWord.test.js` separato, puoi scrivere alcuni test per verificare che la funzione funzioni come previsto. Il file `getFormattedWord.test.js` sarà così:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **Funzione `expect`**: La funzione `expect` serve a testare un valore.
- **Matcher**: Il matcher è una funzione che verifica se il valore si comporta come previsto. Nell’esempio sopra, il matcher è `toBe()`. Jest ha una varietà di matcher.

Per usare Jest, devi prima installare il pacchetto `jest` usando `npm i jest`. Devi anche aggiungere uno script al file `package.json` così:

```js
"scripts": {
  "test": "jest"
},
```

Poi puoi eseguire il comando `npm run test` per far partire i test.

## Ciclo di vita dello sviluppo software

- **Diverse fasi del ciclo di vita dello sviluppo software**:
  - **Fase di pianificazione**: Il team di sviluppo raccoglie i requisiti dal committente.
  - **Fase di progettazione**: Il team software scompone i requisiti e decide le migliori soluzioni.
  - **Fase di implementazione**: Il team software scompone i requisiti in attività gestibili e le implementa.
  - **Fase di test**: Include test manuali e automatizzati per il nuovo lavoro. A volte il team testa l’applicazione durante tutto lo sviluppo per individuare e correggere problemi.
  - **Fase di distribuzione**: Il team distribuisce le nuove modifiche in un ambiente di build o test.
  - **Fase di manutenzione**: Include la correzione di problemi segnalati dai clienti nell’applicazione in produzione.

- **Diversi modelli del ciclo di vita dello sviluppo software**:
  - **Modello a cascata**: Ogni fase deve essere completata prima che inizi la successiva.
  - **Modello Agile**: Si concentra sullo sviluppo iterativo suddividendo il lavoro in sprint.

## BDD e TDD

- **TDD**: Lo sviluppo guidato dai test è una metodologia che enfatizza la scrittura dei test prima. Scrivere test prima di costruire funzionalità fornisce feedback in tempo reale agli sviluppatori durante lo sviluppo.
- **BDD**: Lo sviluppo guidato dal comportamento è l’approccio di allineare una serie di test con gli obiettivi di business. Gli scenari di test in BDD dovrebbero essere scritti in un linguaggio comprensibile sia da persone tecniche che non tecniche. Un esempio di sintassi è Gherkin.
- **Framework di test BDD**: Esempi includono Cucumber, JBehave e SpecFlow.

## Asserzioni nei test unitari

- **Asserzione**: Le asserzioni servono a testare che il codice si comporti come previsto.
- **Librerie di asserzioni**: Chai è una libreria di asserzioni comunemente usata. Altre librerie comuni JavaScript sono `should.js` e `expect.js`.

Ecco un esempio di asserzione usando Chai che verifica che il valore di ritorno dalla funzione `addThreeAndFour` sia uguale al numero 7:

```js
assert.equal(addThreeAndFour(), 7);
```

- **Buone pratiche**: Qualunque libreria di asserzioni usi, dovresti scrivere messaggi chiari di asserzione e fallimento che ti aiutino a capire quali test falliscono e perché.

## Mocking, faking e stubbing

- **Mocking**: Il mocking è il processo di sostituire dati reali con dati falsi che simulano il comportamento di componenti reali. Per esempio, potresti simulare la risposta API nei test invece di fare chiamate API continue per recuperare dati.
- **Stubbing**: Gli stub sono oggetti che restituiscono risposte predefinite o dati fittizi per un comportamento atteso in un’applicazione. Per esempio, puoi stubbare il comportamento di una connessione database nei test senza dover usare un database reale.
- **Faking**: I fake sono versioni semplificate di componenti reali senza la complessità o gli effetti collaterali. Per esempio, puoi simulare un database memorizzando i dati in memoria invece di interagire con il database reale. Questo ti permette di imitare operazioni database in memoria, molto più veloce che usare il database reale.

## Test funzionali

- **Test funzionali**: Verificano se le funzionalità e le funzioni dell’applicazione funzionano come previsto. L’obiettivo è testare il sistema nel suo insieme contro più scenari.
- **Test non funzionali**: Si concentrano su aspetti come prestazioni e affidabilità.
- **Smoke testing**: È un controllo preliminare del sistema per problemi basilari o critici prima di test più estesi.

## Test end-to-end

- **Test end-to-end**: Testano scenari reali dal punto di vista dell’utente. Aiutano a garantire che l’applicazione si comporti correttamente e sia prevedibile per gli utenti. Tuttavia, richiedono tempo per essere configurati, progettati e mantenuti.

test("The menu should have a supporters link", async ({ page }) => {

## Test di usabilità

- **Test di usabilità**: Consiste nell’avere utenti reali che interagiscono con l’applicazione per scoprire se ci sono problemi di design, esperienza utente o funzionalità. Si concentra sull’intuitività dell’applicazione per gli utenti.
- **Quattro tipi comuni di test di usabilità**:
  - **Esplorativo**: Gli utenti interagiscono con le diverse funzionalità per capire meglio come funzionano.
  - **Comparativo**: Confronti l’esperienza utente della tua applicazione con applicazioni simili sul mercato.
  - **Valutativo**: Studi quanto è intuitiva l’applicazione da usare.
  - **Validazione**: Identifichi problemi importanti che impedirebbero all’utente di usare efficacemente l’applicazione.
- **Strumenti per test di usabilità**: Esempi includono Loop11, Maze, Userbrain, UserTesting e UXTweak.

## Test di compatibilità

- **Test di compatibilità**: L’obiettivo è assicurarsi che l’applicazione funzioni in diversi ambienti di calcolo.
- **Tipi diversi di test di compatibilità**:
  - **Compatibilità retroattiva**: Il software è compatibile con versioni precedenti.
  - **Compatibilità futura**: Il software e i sistemi funzionano con versioni future.
  - **Compatibilità hardware**: Il software funziona correttamente su diverse configurazioni hardware.
  - **Compatibilità sistemi operativi**: Il software funziona su diversi sistemi operativi, come macOS, Windows e distribuzioni Linux come Ubuntu e Fedora.
  - **Compatibilità di rete**: Il software funziona in diverse condizioni di rete, come velocità, protocolli, impostazioni di sicurezza, ecc.
  - **Compatibilità browser**: L’applicazione web funziona in modo coerente su diversi browser, come Google Chrome, Safari, Firefox, ecc.
  - **Compatibilità mobile**: È importante assicurarsi che le applicazioni funzionino su vari dispositivi Android e iOS, inclusi telefoni e tablet.

## Test di prestazioni

- **Test di prestazioni**: Testano velocità, reattività, scalabilità e stabilità di un’applicazione sotto diversi carichi di lavoro. L’obiettivo è risolvere qualsiasi collo di bottiglia nelle prestazioni.
- **Tipi diversi di test di prestazioni**:
  - **Load testing**: Determina come un sistema si comporta durante carichi normali e di picco.
  - **Stress testing**: Testa l’applicazione con carichi estremi per vedere come risponde.
  - **Soak testing (Endurance testing)**: Tipo di load testing dove si testa il sistema con carichi elevati per un periodo prolungato.
  - **Spike testing**: Aumenta e diminuisce drasticamente i carichi e analizza le reazioni del sistema.
  - **Breakpoint testing (Capacity testing)**: Incrementa lentamente il carico fino al punto in cui il sistema inizia a fallire o degradare.

## Test di sicurezza

- **Test di sicurezza**: Aiutano a identificare vulnerabilità e debolezze.
- **Principi di sicurezza**:
  - **Confidenzialità**: Protegge contro la divulgazione di informazioni sensibili a destinatari non autorizzati.
  - **Integrità**: Previene che utenti malintenzionati modifichino informazioni utente.
  - **Autenticazione**: Verifica l’identità dell’utente per assicurarsi che possa usare il sistema.
  - **Autorizzazione**: Determina quali azioni gli utenti autenticati possono eseguire o quali parti del sistema possono accedere.
  - **Disponibilità**: Assicura che informazioni e servizi siano disponibili agli utenti autorizzati quando servono.
  - **Non ripudio**: Garantisce che mittente e destinatario abbiano prova della consegna e verifica dell’identità del mittente. Protegge contro la negazione dell’invio.
- **Minacce comuni alla sicurezza**:
  - **Cross-Site Scripting (XSS)**: Attacchi in cui un aggressore inietta script malevoli in una pagina web ed esegue nel contesto del browser della vittima.
  - **SQL Injection**: Permette a utenti malevoli di iniettare codice malevolo in un database.
  - **Denial-of-Service (DoS)**: Attacchi in cui utenti malevoli inondano un sito con richieste o traffico elevato, causando rallentamenti o crash del server, rendendo il sito non disponibile.
- **Categorie di strumenti per test di sicurezza**:
  - **Static Application Security Testing**: Valutano il codice sorgente per identificare vulnerabilità.
  - **Dynamic Application Security Testing**: Interagiscono con il front-end dell’applicazione per scoprire debolezze. Non hanno accesso al codice sorgente.
- **Test di penetrazione (pentest)**: Tipo di test di sicurezza che simula attacchi informatici per identificare vulnerabilità nel sistema.

## A/B Testing

- **A/B Testing**: Consiste nel confrontare due versioni di una pagina o applicazione e studiare quale versione funziona meglio. È anche chiamato bucket o split testing. Permette decisioni basate sui dati e migliorare continuamente l’esperienza utente.
- **Strumenti per A/B Testing**: Esempi includono GrowthBook e LaunchDarkly.

## Test alpha e beta

Dopo lo sviluppo iniziale e i test software, è importante far testare l’applicazione da tester e utenti reali. Qui entrano in gioco i test alpha e beta.

- **Test alpha**: Viene fatto da un gruppo selezionato di tester che esaminano l’applicazione per assicurarsi che non ci siano bug prima del rilascio sul mercato. Fa parte dei test di accettazione e usa tecniche di white box e black box testing.
- **Test beta**: L’applicazione viene resa disponibile a utenti reali. Gli utenti possono interagire e fornire feedback. È anche una forma di test di accettazione utente.
- **Test di accettazione**: Assicura che il software soddisfi i requisiti di business e le esigenze degli utenti prima del rilascio.
- **Black box testing**: Si concentra solo sul comportamento atteso dell’applicazione.
- **White box testing**: Il tester conosce i componenti interni e li testa.

## Test di regressione

- **Regressione**: Situazioni in cui nuove modifiche rompono involontariamente funzionalità esistenti.
- **Test di regressione**: Aiuta a individuare problemi di regressione. Consiste nel rieseguire test funzionali su parti dell’applicazione per assicurarsi che tutto funzioni ancora come previsto.
- **Strumenti per test di regressione**: Puppeteer, Playwright, Selenium e Cypress.
- **Tecniche per test di regressione**: 
  - **Unit regression testing**: Lista di elementi da testare ogni volta che si fanno modifiche o correzioni importanti.
  - **Partial regression testing**: Approcci mirati per assicurarsi che nuove modifiche non rompano aspetti specifici.
  - **Complete regression testing**: Testa tutte le funzionalità nel codice. È il più dettagliato e dispendioso in termini di tempo.
- **Retesting**: Serve a verificare problemi noti e assicurarsi che siano risolti. A differenza del test di regressione, che cerca problemi sconosciuti causati da modifiche recenti.

## Cos’è TypeScript

- **JavaScript**: JavaScript è un linguaggio a tipizzazione dinamica. Questo significa che le variabili possono ricevere qualsiasi valore a runtime. La sfida di un linguaggio a tipizzazione dinamica è che la mancanza di sicurezza di tipo può introdurre errori.

Per esempio, anche se la tua funzione JavaScript si aspetta un array, puoi comunque chiamarla con un numero:

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

L’output `console` per l’esempio sopra sarà `undefined`.

- **TypeScript**: TypeScript estende JavaScript includendo la tipizzazione statica, che aiuta a catturare errori causati da mismatch di tipo prima di eseguire il codice.

Per esempio, puoi definire un tipo per il parametro `array` così:

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

Questa definizione dice a TypeScript che il parametro `array` deve essere un array di stringhe. Quando chiami `getRandomValue` passando un numero, ottieni un errore di compilazione.

- **Compilatore**: Devi prima compilare il codice TypeScript in JavaScript normale. Quando esegui il compilatore, TypeScript valuta il codice e segnala errori per problemi di tipo.

## Tipi di dati in TypeScript

- **Tipi primitivi in TypeScript**: Per i tipi primitivi `string`, `null`, `undefined`, `number`, `boolean` e `bigint`, TypeScript offre parole chiave di tipo corrispondenti.

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **Array**: Puoi definire un array di tipo specifico con due sintassi diverse.

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **Oggetti**: Puoi definire la struttura esatta di un oggetto.

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

Se vuoi un oggetto con chiavi qualsiasi, ma dove tutti i valori devono essere stringhe, ci sono due modi per definirlo:

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **Altri tipi utili in TypeScript**:
  - **`any`**: `any` indica che un valore può avere qualsiasi tipo. Dice al compilatore di non preoccuparsi del tipo di quella variabile.
  - **`unknown`**: `unknown` dice a TypeScript che ti interessa il tipo del valore, ma non sai quale sia. `unknown` è generalmente preferito a `any`.
  - **`void`**: Tipo speciale usato tipicamente per definire funzioni. Funzioni senza valore di ritorno usano `void` come tipo di ritorno.
  - **`never`**: Rappresenta un tipo che non esisterà mai.

- **Parola chiave `type`**: Simile a `const`, ma invece di dichiarare una variabile, dichiari un tipo.

È utile per dichiarare tipi personalizzati come union o tipi che includono solo valori specifici:

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**: Le interfacce sono come classi per tipi. Possono implementare o estendere altre interfacce, sono specifiche per tipi oggetto e sono generalmente preferite a meno che non serva una caratteristica specifica offerta da una dichiarazione `type`.

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **Definire il tipo di ritorno**: Puoi anche definire il tipo di ritorno della funzione.

L’esempio sotto definisce il valore di ritorno come stringa. Se provi a restituire altro, TypeScript segnala un errore di compilazione.

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```

## Generics

- **Definire un tipo generico**: Puoi definire un tipo generico e riferirti a esso nella funzione. Puoi pensarlo come un parametro speciale che fornisci a una funzione per controllare il comportamento della definizione di tipo della funzione.

Ecco un esempio di definizione di un tipo generico per una funzione:

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

La sintassi `<T>` dice a TypeScript che stai definendo un tipo generico `T` per la funzione. `T` è un nome comune per tipi generici, ma puoi usare qualsiasi nome.

Poi dici a TypeScript che il parametro `array` è un array di valori che corrispondono al tipo generico, e che il valore restituito è un singolo elemento dello stesso tipo.

- **Specificare l’argomento di tipo nella chiamata di funzione**: Puoi passare un argomento di tipo a una chiamata di funzione usando parentesi angolari tra il nome della funzione e i parametri.

Ecco un esempio di passaggio di un argomento di tipo a una chiamata di funzione:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Questo dice a TypeScript che l’elemento che ti aspetti di trovare sarà un elemento input.

## Type narrowing

- **Narrowing tramite truthiness**: Nell’esempio sotto, ottieni un errore di compilazione tentando di accedere alla proprietà `value` di `email` perché `email` potrebbe essere `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Puoi usare una condizione per confermare che `email` è truthy prima di accedere alla proprietà:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

I controlli di truthiness possono funzionare anche al contrario:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

Lanciare un errore termina l’esecuzione logica di questo codice, il che significa che quando raggiungi la chiamata `console.log()`, TypeScript sa che `email` non può essere `null`.

- **Optional chaining**: L’optional chaining `?.` è anche una forma di type narrowing, con lo stesso principio che l’accesso alla proprietà non può avvenire se il valore `email` è `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **Operatore `typeof`**: Puoi usare una condizione per controllare il tipo di una variabile usando l’operatore `typeof`.

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **Parola chiave `instanceof`**: Se l’oggetto proviene da una classe, puoi usare la parola chiave `instanceof` per restringere il tipo.

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **Casting**: Quando TypeScript non riesce a determinare automaticamente il tipo di un valore, come il risultato del metodo `request.json()` nell’esempio sotto, ottieni un errore di compilazione. Un modo per risolvere è fare il casting del tipo, ma questo indebolisce la capacità di TypeScript di catturare potenziali errori.

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **Type guard**: Invece di fare casting, puoi scrivere un type guard:

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

La sintassi `user is User` indica che la funzione restituisce un booleano che, se true, significa che il valore `user` soddisfa l’interfaccia `User`.

## File `tsconfig`

- **`tsconfig.json`**: Le impostazioni del compilatore TypeScript vivono in un file `tsconfig.json` nella directory principale del progetto.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

Ecco le descrizioni delle opzioni del compilatore usate nell’esempio sopra:

- **`compilerOptions`**: La proprietà `compilerOptions` controlla il comportamento del compilatore TypeScript.
- **`rootDir` e `outDir`**: `rootDir` e `outDir` indicano a TypeScript quale directory contiene i file sorgente e quale deve contenere il codice JavaScript traspilato.
- **`lib`**: La proprietà `lib` determina quali definizioni di tipo usa il compilatore e permette di includere supporto per specifiche versioni ES, il DOM e altro.
- **`module` e `moduleResolution`**: `module` e `moduleResolution` lavorano insieme per gestire come il pacchetto usa i moduli — CommonJS o ECMAScript.
- **`esModuleInterop`**: `esModuleInterop` permette una migliore interoperabilità tra CommonJS e moduli ES creando automaticamente oggetti namespace per gli import.
- **`skipLibCheck`**: L’opzione `skipLibCheck` salta la validazione dei file `.d.ts` che non sono referenziati da import nel codice.
- **`strict`**: Il flag `strict` abilita diversi controlli, come la gestione corretta dei tipi nullable e avvisi quando TypeScript ricade su `any`.
- **`exclude`**: La proprietà di livello superiore `exclude` dice al compilatore di ignorare questi file TypeScript durante la compilazione.
