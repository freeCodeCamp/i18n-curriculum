# --description--

لقد تطرقنا بإيجاز إلى ملف `.gitignore` في درس سابق. لكن ما هو هذا الملف بالضبط، وماذا يجب أن تضع فيه؟

ملف `.gitignore` هو نوع خاص من الملفات المرتبطة بعمليات Git. الاسم يوحي بأن هذا الملف يُستخدم لإخبار Git بتجاهل أشياء معينة، وهذا هو الاستخدام الشائع. لكن ما يفعله فعليًا هو أنه يخبر Git بالتوقف عن تتبع ملف معين. هذا التمييز مهم، وسيتم توضيحه لاحقًا.

ماذا يجب أن تضع فعليًا في هذا الملف؟ يجب أن تسرد الأشياء التي لا تريد الالتزام بها.

الأول هو ملفات الأسرار الخاصة بك. قد يكون هذا مثل ملف `.env`، حيث خزنت مفاتيح API التي يستخدمها تطبيقك. قد يكون ملف مفتاح GPG خاص يحتاج تطبيقك لاستخدامه في التوثيق. أي شيء يحتوي على معلومات حساسة لا تريد تسريبها للعامة يجب تجاهله باستخدام `.gitignore`. حتى لو كان مستودعك خاصًا، يجب أن تتصرف كما لو كان عامًا — اعتبر أن أي شخص يمكنه رؤيته، وكن حذرًا بعدم الالتزام بالأسرار.

لنلقِ نظرة على كيف قد يبدو ملفنا الآن:

```sh
# Secrets
.env
github.key
```

لقد أخبرنا Git بتجاهل ملف `.env`، حيث نحتفظ بأسرارنا. وأخبرناه أيضًا بتجاهل ملف `github.key`، الذي يحتوي على مفتاح GPG خاص.

الشيء التالي الذي يجب أخذه في الاعتبار هو الحزم، أو التبعيات، التي يستخدمها مشروعك. على سبيل المثال، عندما تعمل على مشروع Node.js، من المحتمل أنك تثبت حزمًا من npm. هذه الحزم تذهب إلى مجلد `node_modules` في مشروعك، حتى يتمكن شفرتك من استيرادها واستخدامها.

لكن مجلد `node_modules` يصبح ضخمًا جدًا، وبالتأكيد لا تريد الالتزام بكل تلك الحزم وتتبع التغييرات في كل مرة تحدث فيها تحديثًا. بدلًا من ذلك، تلتزم بملف `package.json` وملف القفل المناسب (مثل `package-lock.json` أو `pnpm-lock.yaml`). هذه الملفات تخبر npm ما الحزم والإصدارات التي يجب تثبيتها، حتى يتمكن المساهم من تثبيت نفس التبعيات التي تستخدمها دون الحاجة إلى الالتزام بها في المستودع.

لنقم بإضافة `node_modules` إلى ملف `.gitignore` الخاص بنا:

```sh
# Secrets
.env
github.key

# Packages
node_modules
```

الآن، دعنا نتحدث عن نقطة مهمة في صيغة ملفات `.gitignore`. عندما تضيف إدخالًا مثل `node_modules`، سيتجاهل Git أي شيء يطابق هذا النمط. هذا يعني أنه سيتجاهل مجلد `node_modules` في جذر مشروعك، ولكنه سيتجاهل أيضًا مجلد `node_modules` في مجلد `client` داخل مشروعك.

إذا أردت تجاهل شيء ما في الجذر فقط، يمكنك إضافة بادئة شرطة مائلة للأمام إلى السطر. يصبح `node_modules` هكذا: `/node_modules`. هذا سيتجاهل مجلد `node_modules` في الجذر فقط، وليس مجلد `client/node_modules`. نحن لا نريد ذلك، لذا دعنا نترك إدخالنا كما هو `node_modules` فقط.

الشيء التالي الذي تريد أخذه في الاعتبار هو تجاهل مخرجات البناء. على سبيل المثال، قد يؤدي تجميع مشروع TypeScript إلى إنشاء مجلد `dist` لمخرجات JavaScript. لا تريد الالتزام بهذه المخرجات لأنها تزيد من حجم سجل Git. بدلًا من ذلك، تريد تجاهل هذه المخرجات — يمكن للمساهمين إعادة إنتاجها بأنفسهم عن طريق تشغيل نفس أمر البناء.

لنقم بتجاهل مجلد `dist`:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
```

أحيانًا، قد يتسبب ملف موضوع في مكان خاطئ في تعطل مترجم TypeScript، مما يؤدي إلى انتشار مخرجات JavaScript في أماكن غير منظمة. بما أننا نكتب TypeScript حصريًا، دعنا نستخدم صيغة glob لتجاهل كل ملفات JavaScript:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js
```

بعض المخرجات الشائعة الأخرى التي قد تريد تجاهلها هي مجلد `coverage` لنتائج تغطية اختبارات الوحدة، ومجلد `.angular` إذا كنت تستخدم Angular، ومجلد `.next` إذا كنت تستخدم NextJS، وهكذا.

في الوقت الحالي، يمكننا الالتزام بما لدينا هنا. إذا كنت تستخدم إطار عمل يولد ملفات، يجب أن تتحقق من توثيقه لمعرفة ما يجب إضافته إلى `.gitignore`.

المجموعة الرابعة من الأشياء التي يجب تجاهلها هي ملفات بيئة التطوير المتكاملة (IDE). على سبيل المثال، قد تستخدم منتجات Jetbrains مجلد `.idea` لإعدادات خاصة بالبيئة. يستخدم Visual Studio Code مجلد `.vscode`.

غالبًا ما يكون من الجيد تجاهل هذه الملفات، لأنها تحتوي على إعدادات خاصة ببيئة العمل المحلية التي لا يحتاجها المساهمون الآخرون (لأن لديهم إعداداتهم الخاصة). دعنا نضيف مجلد Visual Studio Code، لأنه هو بيئة التطوير التي نستخدمها.

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
```

لكن مجلد `.vscode` يحتوي على ملف `settings.json`، الذي يشمل بعض الإعدادات الخاصة بمساحة العمل والتي قد تكون مفيدة لمساهمينا — هذا الملف يسمح لهم بضبط الإضافات وبيئة العمل لتكون مشابهة لبيئتنا.

لذا دعنا نستخدم صيغة النفي "لإلغاء تجاهل" ذلك الملف:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json
```

إذا كنت تستخدم MacOS، قد تكون على دراية بملف `.DS_STORE`. يحتوي هذا الملف على بيانات وصفية يستخدمها Finder لتحديد كيفية عرض المجلد (مثل الصور المصغرة أو العرض كقائمة). وبالمثل، قد يرى مستخدمو Windows ملفًا مخفيًا باسم `Thumbs.db`، الذي يخزن الصور المصغرة للمجلد.

من الجيد عمومًا تجاهل هذه الملفات، لأنها إعدادات شخصية لا تؤثر على المشروع. دعنا نضيف كلاهما:

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json

# System Files
.DS_STORE
Thumbs.db
```

ملاحظة سريعة حول تجاهل الملفات التي تم الالتزام بها سابقًا. لنفترض أنك التزمت بمجلد `dist` عن طريق الخطأ في السابق، والآن تريد حذفه وتجاهله. لا يمكنك فقط إضافته إلى ملف `.gitignore` — هذا لا يجعل المحتوى الذي تم الالتزام به يختفي، بل يخبر Git بالتوقف عن تتبع الملف. لذا إذا تجاهلت ملفًا تم الالتزام به، فلن يتتبع Git أي تغييرات عليه، لكنه لن يتتبع حذف الملف أيضًا.

بدلًا من ذلك، يجب عليك أولًا حذف الملف. أنشئ التزامًا بهذا الحذف. ثم يمكنك إضافة العنصر إلى `.gitignore` وإنشاء التزام ثاني بهذا التحديث. من الآن فصاعدًا، عندما يُعاد إنشاء ذلك الملف، سيتجاهله Git ولن يظهر في التزاماتك.

أخيرًا، إذا كنت تعمل على مشروع بلغة مختلفة، مثل Python، فالعناصر التي تحتاج إلى تجاهلها ستكون مختلفة عما غطيناه هنا. يحتفظ GitHub بمستودع مفيد جدًا لقوالب `.gitignore` للغات وأُطُر عمل مختلفة. يمكنك العثور على هذه القائمة في <a href="https://github.com/github/gitignore" target="_blank">https://github.com/github/gitignore</a>.

مع ذلك، أنت الآن جاهز لتجنب الالتزام بالملفات التي لا تريدها في مستودعاتك!
