# --description--

Кожна комп’ютерна програма, яка запускається на вашому пристрої, має певний набір інструкцій, які виконуються у визначеному порядку для завершення завдання.

Завданням може бути сортування множини чисел, редагування зображення, відстеження запасів або навіть запуск вашої улюбленої відеогри.

Саме тут на допомогу приходять алгоритми. **Алгоритм** — це набір однозначних інструкцій для розв’язання проблеми або виконання завдання.

Ви можете уявити алгоритми як «рецепти». Коли ви готуєте, рецепти перераховують усі інгредієнти, які вам знадобляться, і надають покрокові інструкції, як приготувати страву.

Аналогічно, алгоритми — це «рецепти», які точно кажуть комп’ютерам, що потрібно зробити і як це зробити.

Алгоритми мають дві ключові характеристики:

* Вони не можуть виконуватися нескінченно. Вони повинні завершитися за скінченну кількість кроків.
    
* Кожен крок має бути точним і однозначним.
    

Вони можуть мати нуль, один або більше вхідних даних і генерувати один або більше вихідних даних.

Кроки алгоритму не залежать від жодної мови програмування.

Але щоб фактично запустити їх на комп’ютері, потрібно реалізувати їх у мові програмування, наприклад Python або JavaScript.

Якщо алгоритм правильний, вихідні дані для будь-яких дійсних вхідних даних повинні відповідати очікуваним вихідним даним.

Окрім правильності, алгоритми мають бути ще й ефективними.

Ефективність алгоритму можна вимірювати за часом його виконання та обсягом пам’яті, необхідної для завершення завдання.

Знання ефективності алгоритму дуже важливе, бо дає уявлення про те, наскільки добре він працюватиме зі збільшенням розміру вхідних даних.

Наприклад, сортування 15 цілих чисел — це не те саме, що сортування 1 мільйона цілих чисел.

Якщо процес зростає за розміром і складністю, і алгоритм не є достатньо ефективним, ви можете отримати дуже повільну програму, яка навіть може призвести до збою всієї системи.

Саме тому дуже важливо розробляти та обирати максимально ефективні алгоритми.

Ось тут і стає в нагоді нотація Big O.

Нотація Big O описує найгірший випадок продуктивності або темп зростання алгоритму зі збільшенням розміру вхідних даних.

Темп зростання алгоритму означає, як зростають ресурси, які він потребує, зі збільшенням розміру вхідних даних.

Нотація Big O зосереджується на найгіршому випадку, бо це дуже важливо для розуміння, наскільки ефективним може бути алгоритм навіть у найгіршому сценарії, незалежно від вхідних даних.

Повертаючись до прикладу з сортуванням, інтуїтивно сортування 1 мільйона цілих чисел має займати більше часу і ресурсів, ніж сортування 15 чисел.

Але наскільки більше?

Це залежить від алгоритму, який ви оберете для сортування.

Нотація Big O не дасть вам точного числа, щоб описати ефективність алгоритму, але дасть уявлення про те, як він масштабується зі збільшенням розміру вхідних даних, базуючись на кількості операцій, які виконує алгоритм.

У нотації Big O зазвичай позначають розмір вхідних даних літерою `n`. Наприклад, якщо вхідні дані — це список, `n` позначатиме кількість елементів у цьому списку.

Константні множники та члени нижчого порядку не враховуються при визначенні часової складності алгоритму за кількістю операцій. Це тому, що зі збільшенням розміру `n` вплив цих менших членів у загальній кількості операцій стає дедалі меншим.

Член, який домінуватиме у загальній поведінці алгоритму, буде членом найвищого порядку з `n`, розміром вхідних даних.

Наприклад, якщо алгоритм виконує `7n + 20` операцій для завершення, вплив константи `20` на кінцевий результат буде дедалі меншим зі збільшенням `n`. Член `7n` буде домінувати і визначатиме загальну поведінку та ефективність алгоритму.

Іншим прикладом є алгоритм, який потребує `20n² + 15n + 7` операцій для завершення. Член `20n²` буде домінувати зі збільшенням `n`, тому цей алгоритм матиме квадратичну часову складність, бо домінуючий член має `n²`.

Квадратична часова складність — це один із багатьох типів часових складностей, які можна зустріти у світі алгоритмів.

Давайте розглянемо деякі з найпоширеніших.

**`O(1)`** відома як «постійна часова складність». Коли алгоритм має постійну часову складність, він виконується за однаковий час, незалежно від розміру вхідних даних.

Наприклад, перевірка, чи є число парним або непарним, завжди займатиме однаковий час, незалежно від самого числа.

```js
function checkEvenOrOdd(number) {
  if (number % 2 === 0) {
    return "Even";
  } else {
    return "Odd";
  }
}
```

**`O(log n)`** відома як «логарифмічна часова складність». Це означає, що час, необхідний алгоритму, повільно зростає зі збільшенням розміру вхідних даних. Це характерно для задач, у яких розмір проблеми багаторазово зменшується на сталу частку.

Наприклад, популярний алгоритм пошуку під назвою Binary Search має `O(log n)` часову складність у найгіршому випадку. Це тому, що він виключає половину залишкових елементів при кожному порівнянні, що робить його загалом ефективнішим.

**`O(n)`** відома як «лінійна часова складність». Час виконання алгоритмів з такою складністю зростає пропорційно до розміру вхідних даних.

Наприклад, `for` цикл, який ітерує всі елементи списку, виконає більше ітерацій зі збільшенням кількості елементів списку. Якщо список подвоїться за розміром, кількість операцій приблизно подвоїться.

```js
for (const grade of grades) { // grades is an array
  console.log(grade);
}
```

**`O(n log n)`** відома як «лог-лінійна часова складність». Це поширена часова складність ефективних алгоритмів сортування, таких як Merge Sort і Quick Sort.

**`O(n²)`** відома як «квадратична часова складність». Час виконання цих алгоритмів зростає квадратично відносно розміру вхідних даних, що зазвичай не є ефективним для реальних задач.

Вкладені цикли — це типовий приклад квадратичної часової складності. Внутрішній цикл виконає `n` ітерацій для кожної з `n` ітерацій зовнішнього циклу, що дає `n` у квадраті ітерацій.

```js
for (let i = 0; i < n; i++) {
  for (let j = 0; j < n; j++) {
    console.log("Hello, World!");
  }
}
```

Інші часові складності включають «експоненціальну часову складність», позначену як `O(2^n)`, та «факторіальну часову складність», позначену як `O(n!)`. Обидві неефективні для реальних сценаріїв.

На цьому графіку ви можете порівняти зростання математичних функцій, які представляють найпоширеніші часові складності. Уявіть, що вісь x (горизонтальна) — це розмір вхідних даних, а вісь y (вертикальна) — час виконання алгоритму.

Ви бачите, що квадратична часова складність (`O(n²)`) (жовтий) зростає набагато швидше за інші, тоді як постійна часова складність (`O(1)`) (червоний) залишається сталою, навіть якщо вхідні дані збільшуються.

Чудово. Поки що ви дізналися про нотацію Big O у контексті часових вимог, але цю нотацію також можна застосувати до вимог до пам’яті.

У цьому контексті вона описує, як зростає обсяг пам’яті, необхідний алгоритму, зі збільшенням розміру вхідних даних.

Алгоритми з «постійною просторовою складністю» `O(1)` завжди потребують сталого обсягу пам’яті, навіть якщо вхідні дані збільшуються.

Прикладом може бути алгоритм, який створює і зберігає в пам’яті лише кілька змінних.

Навпаки, обсяг пам’яті, необхідний алгоритмам з «лінійною просторовою складністю» `O(n)`, зростає пропорційно зі збільшенням розміру вхідних даних.

Прикладом такого алгоритму є той, що створює і зберігає копію списку довжиною `n`.

І нарешті, вимоги до пам’яті алгоритму з «квадратичною просторовою складністю» `O(n²)` зростають квадратично зі збільшенням розміру вхідних даних.

Прикладом є створення двовимірної матриці, розміри якої визначаються розміром вхідних даних, що зберігає всі можливі пари.

Алгоритми — це будівельні блоки комп’ютерних програм, а нотація Big O — потужний фреймворк для аналізу їх ефективності, базуючись на тому, як їх часові та просторові вимоги у найгіршому випадку масштабуються зі збільшенням розміру вхідних даних. Розуміння їх ефективності дуже важливе для розробки програмного забезпечення, яке працює ефективно у реальних сценаріях.
