# --description--

이전 수업에서 자신의 저장소에 풀 리퀘스트를 만드는 방법을 배웠습니다. 또한 그 풀 리퀘스트를 병합하는 방법도 배웠죠. 하지만 그건 저장소가 본인 소유여서 가능했습니다. 본인 저장소에 대한 유지 관리자 접근 권한이 있어 풀 리퀘스트를 병합할 수 있었던 겁니다.

하지만 다른 사람의 오픈 소스 프로젝트에 기여할 때는 보통 그런 권한이 없습니다. 대신 유지 관리자들이 코드 검토 과정을 거쳐 기여를 승인할 때까지 기다려야 합니다. 그들이 병합 작업을 처리하죠.

그렇다면 코드 검토 과정은 어떻게 진행될까요?

유지 관리자마다 다르고 프로젝트별로도 접근 방식이 다를 수 있습니다. 하지만 일반적으로 코드 검토는 보통 세 단계로 이루어집니다.

첫 번째는 CI, 즉 지속적 통합이 통과하는지 확인하는 것입니다. 이 부분은 앞으로 수업에서 더 배우게 될 테니, 지금은 코드가 스타일 가이드라인을 준수하고 테스트 스위트를 통과하는지 확인하는 검사라고 생각하세요.

CI가 실패하면 유지 관리자들은 보통 실패를 해결하도록 풀 리퀘스트를 업데이트해 달라고 요청합니다. 실패 보고서가 있어서 무엇이 잘못됐는지 확인할 수 있어야 하며, 유지 관리자들이 문제 해결을 위한 구체적인 안내를 제공할 수도 있습니다.

검사가 통과하면 유지 관리자들은 실제 코드 변경 사항을 살펴봅니다. 변경된 부분, 추가된 부분, 삭제된 부분을 diff 뷰에서 분석할 수 있죠. 특정 접근 방식을 선택한 이유에 대해 질문할 수도 있으니, 대화를 준비할 때 신중하고 생각을 잘 정리하는 것이 좋습니다.

추가 수정을 요청할 수도 있습니다. 그 부분은 조금 후에 설명하겠습니다.

마지막 단계는 보통 변경 사항을 로컬로 내려받는 것입니다. 유지 관리자들은 애플리케이션을 변경 사항과 함께 실행해 기능을 테스트하고, 문제가 없거나 예상치 못한 동작이 없는지 확인합니다.

모든 것이 괜찮으면 유지 관리자가 풀 리퀘스트를 승인합니다. 때로는 "LGTM"이라는 댓글을 남기는데, 이는 "looks good to me"의 약자입니다. 풀 리퀘스트가 승인되면 유지 관리자들이 준비되는 대로 변경 사항을 병합합니다. 일부 프로젝트는 한 명 이상의 승인을 요구하기도 하는데, 예를 들어 freeCodeCamp는 두 명의 승인을 필요로 합니다.

그런데 만약 풀 리퀘스트에 문제가 있다면 어떻게 될까요? 유지 관리자들은 변경 요청을 포함한 검토를 할 수 있습니다. 이런 요청은 몇 가지 형태로 나타납니다.

때로는 유지 관리자가 단순히 "댓글"로 요청을 남겨, 변경하거나 업데이트해야 할 부분을 알려주지만 구현 방법은 자유롭게 할 수 있게 합니다.

또는 유지 관리자들이 공식적인 "변경 요청" 댓글을 남길 수도 있습니다. 보통 이런 댓글은 풀 리퀘스트가 병합되는 것을 막습니다. 원하는 변경 사항을 반영해야 하며, 변경 요청을 한 유지 관리자가 다시 검토해서 추가 변경을 요청하거나 작업을 승인합니다.

유지 관리자들이 구체적인 변경 사항을 생각해 두었다면 GitHub UI를 통해 직접 제안할 수도 있습니다. 이런 직접 제안을 사용하면 GitHub가 제안을 브랜치에 바로 커밋할 수 있는 버튼을 제공합니다. 하지만 이런 제안이 여러 개라면 파일 뷰로 전환하는 것이 좋습니다.

파일 뷰에서는 모든 제안을 하나씩 "배치"에 추가해 한 번에 커밋할 수 있습니다. 제안이 많을 때 개별적으로 처리하면 커밋이 많아져 CI 검사가 각 커밋마다 실행되어 PR 승인 과정이 느려질 수 있으므로 매우 유용합니다.

요청된 변경 사항이 많다면 로컬에서 변경을 하고, PR에 사용한 동일한 브랜치에 커밋한 뒤 푸시하는 것이 좋습니다. GitHub가 자동으로 풀 리퀘스트를 새 커밋으로 업데이트합니다.

몇 가지 추가로 알아둘 점이 있습니다.

먼저, 유지 관리자도 사람이라는 점을 기억하세요. 오픈 소스 프로젝트는 유지 관리자들이 전업으로 일할 만큼 수익을 내지 못하는 경우가 많아 다른 일을 병행해야 합니다. 검토 과정에 인내심을 가지는 것이 중요하며, 반복해서 재검토를 요청하거나 검토를 서두르게 하지 마세요.

기여는 흥미진진할 수 있습니다! 하지만 서두르지 말고 천천히 하세요. 로컬에서 변경 사항을 테스트하고, 프로젝트에 테스트 스위트가 있다면 실행해 보고, 개발 서버를 실행해 수동으로 변경 사항이 잘 작동하는지 확인하세요. 그리고 항상 작업하는 특정 프로젝트의 기여 지침을 따르세요.

이 모든 점을 명심하고 최선을 다하면 기여가 순조롭게 진행되고 코드 검토가 확실한 승인이 될 것입니다!
