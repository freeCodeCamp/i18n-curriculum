# --description--

HTTP 請求-回應模型是驅動全球資訊網（WWW）的基本通訊樣式。它包含許多組件，使其高效，因此我們都能從網路存取資料。

讓我們來看看 HTTP 請求-回應模型是什麼、組成它的組件，以及這些組件如何協同運作以提升驅動網路的效率。

HTTP 請求-回應模型是在網頁互動中一種通訊樣式，其中客戶端，通常是網頁瀏覽器，向伺服端發送 **HTTP request**，而伺服端處理該請求並回傳 **HTTP response**。

以下是組成 HTTP 請求-回應模型的組件：

* **客戶端**：瀏覽器、行動應用程式或任何其他軟體，會向伺服器發送 HTTP 請求並解析收到的 HTTP 回應。
    
* **Request**：由客戶端傳送到伺端的訊息。它包含方法、URL、標頭和選用的主體。
    
* **HTTP 方法**：要執行的動作 — 使用 `GET` 來擷取資料，`POST` 來傳送資料，`PATCH` 和 `PUT` 來修改資料，以及 `DELETE` 來刪除資料。
    
* **URL（統一資源定位器）**：從伺服端請求的資源。
    
* **Headers**：隨請求或回應傳送的詮釋資料。 例如，`Content-Type` 和 `Authorization`。
    
* **Body**：隨 `POST`、`PUT`、`PATCH`、`DELETE` 請求傳送的資料，通常為 JSON 或表單資料格式。
    
* **伺服端**：處理請求並產生回應的後端系統。它可以使用任何支援伺服端程式設計的程式語言撰寫。
    
* **Response**：伺服端回傳的訊息。它包含狀態碼、標頭和選用的主體。
    
* **狀態碼或回應碼**：表示成功、錯誤或重新導向的數字碼。
    
* **Response Body**：實際回傳的資料，通常為 HTML、JSON 或 XML 格式。

那麼這些組件如何在 HTTP 請求-回應週期中協同運作？

1. **客戶端發起請求**
    
    * 瀏覽器、行動應用程式或 API 客戶端會傳送一個 HTTP 請求，目的是要到達伺服器。
        
    * 請求包含像是 `GET`、`POST`、`PUT`、`PATCH`、`DELETE` 的方法、URL、標頭，有時還有主體。
        
2. **請求已送至伺服器**
    
    * 請求透過網際網路傳送到伺服器，一個使用程式語言撰寫的後端系統。
        
    * 伺服器接收請求並從中擷取相關資訊。
        
3. **伺服端處理程序請求**
    
    * 伺服端檢查請求的 `method` 和 URL。
        
    * 可能需要驗證身份驗證（透過像是 authorization 的標頭）。
        
    * 如果需要，它會與資料庫交談以擷取或修改資料。
        
4. **伺服端準備回應**
    
    * 伺服端會產生帶有狀態碼（例如 200、404、500 等）的回應。
        
    * 回應包含標頭，例如 `Content-Type: application/json`。
        
    * 如果適用，會新增一個回應主體，格式為 JSON、HTML 或 XML。
        
5. **回應已傳送到客戶端**
    
    * 伺服端透過網際網路回傳回應。
        
    * 客戶端接收回應並處理資料。
        
6. **客戶端控制代碼回應**
    
    * 如果回應成功（狀態碼 200），客戶端會顯示資料，例如，在網頁上呈現資料。
        
    * 如果發生錯誤（`404 Not Found`、`500 Internal Server Error` 等），客戶端可能會顯示錯誤訊息。

此循環會針對客戶端所發出的每個新請求重複執行。
