# --description--

In einer vorherigen Lektion haben Sie gelernt, wie man ein neues Repository auf GitHub erstellt. Aber was ist, wenn Sie ein bestehendes Projekt lokal haben, das Sie zu GitHub hinzufügen möchten? In dieser Lektion lernen Sie, wie Sie bestehende lokale Repositories zu GitHub pushen und Änderungen von entfernten Repositories herunterladen.

Beginnen Sie damit, ein neues Repository auf GitHub ohne Dateien zu erstellen. Wenn alles richtig gemacht wurde, sollten Sie auf eine Seite weitergeleitet werden, die die URL enthält, die Sie zum Klonen Ihres Repositories benötigen.

Navigieren Sie nun zu Ihrem Terminal und wählen Sie ein bestehendes Projekt aus, das Sie zu GitHub hinzufügen möchten. In dieser Lektion wähle ich ein lokales Projekt namens `super-awesome-game` mit den folgenden Dateien darin:

```sh
jessicawilkins super-awesome-game >> ls
README.md  index.html script.js  styles.css
```

Um die Versionskontrolle für dieses Projekt mit Git einzurichten, müssen Sie im Projektverzeichnis den Befehl `git init` ausführen. Dadurch wird ein leeres Git-Repository initialisiert, sodass Git beginnen kann, Änderungen für dieses Projekt zu verfolgen. Wenn alles richtig gemacht wurde, sollten Sie eine ähnliche Ausgabe im Terminal sehen:

```sh
jessicawilkins super-awesome-game >> git init
Initialized empty Git repository in /Users/jessicawilkins/workspace/freeCodeCamp/super-awesome-game/.git/
```

Wenn Sie ein leeres Git-Repository für ein Projekt initialisieren, wird ein neues verstecktes Verzeichnis `.git` hinzugefügt. Um die Hinzufügung des Verzeichnisses zu sehen, können Sie den Befehl `ls -a` ausführen, der alle Dateien und Verzeichnisse auflistet, einschließlich der versteckten:

```sh
jessicawilkins super-awesome-game >> ls -a
.          ..         .git       README.md  index.html script.js  styles.css
```

Es ist wichtig, dieses Verzeichnis `.git` nicht zu löschen, sonst gehen alle Ihre Git-Historie verloren.

Nachdem das leere Git-Repository initialisiert wurde, können Sie den Befehl `git status` ausführen. Dieser Befehl zeigt den aktuellen Zustand Ihres Arbeitsverzeichnisses an – Sie werden diesen Befehl in Ihrem Workflow oft verwenden. Sie sollten eine Ausgabe ähnlich dieser sehen:

```sh
jessicawilkins super-awesome-game >> git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	README.md
	index.html
	script.js
	styles.css

nothing added to commit but untracked files present (use "git add" to track)
```

`On branch main` bedeutet, dass Sie sich auf dem Branch `main` befinden. Ein Branch in Git ist ein separater Arbeitsbereich, in dem Sie Änderungen vornehmen können. Der Branch `main` stellt in einer realen Anwendung oft den Haupt- oder Produktionsbranch dar. Entwicklerteams erstellen mehrere Branches für neue Funktionen und Bugfixes und führen diese Änderungen dann wieder in den Branch `main` zusammen. Mehr über diesen Prozess lernen Sie in einer späteren Lektion.

`Untracked files` bedeutet, dass Git Dateien erkannt hat, die es derzeit nicht verfolgt.

Eine Datei kann sich in fünf Zuständen befinden:

1. „Untracked“: Das bedeutet, dass die Datei neu im Repository ist und Git sie noch nicht „gesehen“ hat.
2. „Modified“: Diese Datei existierte im vorherigen Commit und hat Änderungen, die noch nicht committet wurden.
3. „Ignored“: Ignorierte Dateien sehen Sie wahrscheinlich nicht in Git, aber Ihre IDE könnte sie anzeigen. Ignorierte Dateien sind von Git-Operationen ausgeschlossen, typischerweise weil sie in der Datei `.gitignore` aufgeführt sind.
4. „Deleted“: Eine gelöschte Datei ist das Gegenteil einer untracked Datei – es ist eine Datei, die zuvor existierte und entfernt wurde.
5. „Renamed“: Eine umbenannte Datei ist eine Datei, deren Inhalt unverändert ist, aber deren Name oder Speicherort geändert wurde. In manchen Fällen kann eine Datei als umbenannt gelten, auch wenn sie kleine Änderungen enthält.

Wenn Sie uncommittete Änderungen in Ihrem Repo haben, müssen Sie diese zuerst „stagen“. Ein „Staging-Bereich“ ist eine spezielle Datei in Ihrem Verzeichnis `.git`, die Informationen enthält, die in Ihren nächsten Commit eingehen. Ein Commit ist effektiv eine Momentaufnahme des aktuellen Zustands Ihres Repositories.

Wenn Sie eine Datei nach der anderen stagen möchten, können Sie den folgenden Befehl verwenden:

```sh
git add name-of-file
```

Wenn Sie alle unstaged Änderungen stagen möchten, können Sie `git add .` verwenden. Der Punkt (`.`) ist ein Alias für das aktuelle Verzeichnis, in dem Sie sich befinden. Einige Entwickler warnen davor, `git add .` zu verwenden, weil Sie möglicherweise Dateien stagen, die Sie nicht beabsichtigt hatten. Falls das passiert, können Sie sie mit folgendem Befehl „unstagen“:

```sh
git reset name-of-file
```

Im Allgemeinen sollten Sie jedoch, solange Sie mit `git status` überwachen, welche Dateien hinzugefügt werden, bevor Sie sie stagen, keine Probleme haben.

Sobald Sie Ihre Dateien gestaged haben, müssen Sie diese committen. Sie können Ihre Änderungen committen, indem Sie den Befehl `git commit` ausführen. Dadurch wird Ihr bevorzugter Editor geöffnet, den Sie in der Git-Konfiguration festgelegt haben. Sobald der Editor geöffnet ist, können Sie eine ausführliche Nachricht zu Ihren Änderungen schreiben. Sie können auch eine kürzere Nachricht angeben, indem Sie den Befehl `git commit -m` wie folgt verwenden:

```sh
git commit -m "short message goes here"
```

Wenn Sie das Flag `-m` verwenden, sollten Sie dennoch eine beschreibende Nachricht angeben, um den Kontext der Änderungen zu erläutern. Hier sind einige Beispiele für gute Nachrichten, die Sie verwenden könnten:

```sh
git commit -m "feat(api): implement JWT-based authentication"
git commit -m "fix(auth): correct token expiration time"
git commit -m "refactor: simplify login flow logic"
```

Die `feat(api)` und `fix(auth)` folgen dem Stil der Conventional Commits, der verwendet wird, um den Kontext der Arten von Änderungen in diesem Commit anzugeben.

Nachdem Sie Ihre Arbeit committet und `git status` erneut ausgeführt haben, sollten Sie die folgende Ausgabe im Terminal sehen:

```sh
On branch main
nothing to commit, working tree clean
```

Um alle Ihre vorherigen Commits für dieses Projekt anzuzeigen, können Sie den Befehl `git log` ausführen. Dieser listet alle vorherigen Commits mit hilfreichen Informationen wie Autor, Datum des Commits, Commit-Nachricht und Commit-Hash auf. Der Commit-Hash ist eine lange Zeichenfolge, die als eindeutiger Bezeichner für einen Commit dient.

Nachdem Sie Ihre Arbeit committet haben, können Sie Ihre Änderungen jetzt zu GitHub pushen. Zuerst müssen Sie die Verbindung zum entfernten Repository einrichten. Das haben Sie in der vorherigen Lektion gelernt, hier eine Erinnerung mit SSH:

```sh
git remote add origin git@github.com:your-github-username/name-of-repo.git
```

Der letzte Schritt ist, Ihre Änderungen mit diesem Befehl zu GitHub zu pushen:

```sh
git push -u origin main
```

Das Flag `-u` ist eine Kurzform für `--set-upstream`. Es verbindet Ihren lokalen Branch `main` mit dem entfernten Branch `main` auf `origin`. Sie müssen es nur einmal pro neuem Branch verwenden. Danach können Sie einfach `git push` ausführen, ohne den Branch-Namen anzugeben.

Wenn alles richtig gemacht wurde, sollten Sie ein ähnliches Ergebnis im Terminal sehen:

```sh
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Delta compression using up to 8 threads
Compressing objects: 100% (2/2), done.
Writing objects: 100% (3/3), 253 bytes | 253.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0 (from 0)
To https://github.com/jdwilkin4/super-awesome-game.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

Sie sollten die neuen Änderungen auch auf Ihrer GitHub-Repository-Seite sehen, wenn Sie die Seite aktualisieren.

Wenn Sie mit anderen an einem Projekt zusammenarbeiten und neue Änderungen aus dem Repository herunterladen müssen, können Sie den Befehl `git pull` verwenden. Mehr über Teamarbeit und Zusammenarbeit lernen Sie in einer späteren Lektion.
