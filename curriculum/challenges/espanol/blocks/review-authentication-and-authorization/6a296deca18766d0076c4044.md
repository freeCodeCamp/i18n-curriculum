## Autenticación vs autorización

**La autenticación** es el proceso de verificar la identidad de un usuario. Responde a *"¿Es esta persona quien dice ser?"* Cuando un usuario inicia sesión, el servidor backend compara sus credenciales con los registros almacenados durante el registro.

Los desarrolladores backend usan tres categorías principales de factores de autenticación:

- **Algo que sabes:** Una contraseña, PIN o respuesta a una pregunta de seguridad.
- **Algo que tienes:** Un código SMS en un smartphone o un token temporal de una app autenticadora.
- **Algo que eres:** Datos biométricos como una huella digital o un escaneo facial.

**La autorización** toma el control después de la autenticación y decide qué puede hacer el usuario autenticado. Responde a *"¿Qué permisos tiene esta identidad?"* Los roles comunes incluyen:

- **Usuario estándar:** Puede editar su propio perfil, pero no puede ver datos privados de otros usuarios.
- **Editor:** Puede escribir y publicar contenido, pero no puede eliminar la base de código.
- **Administrador:** Tiene permiso total para modificar bases de datos, eliminar usuarios y cambiar configuraciones del sistema.

**La autenticación siempre debe ocurrir primero, luego la autorización.** No puedes decidir qué permisos otorgar hasta saber quién es el usuario.

## Tokens web JSON (JWT)

Un JWT es una tarjeta digital que el servidor emite a un usuario tras un inicio de sesión exitoso. El usuario la presenta en cada solicitud posterior para probar su identidad sin volver a ingresar la contraseña.

El flujo de cuatro pasos del JWT:

1. **Inicio de sesión:** El usuario envía credenciales al servidor.
2. **Emisión:** El servidor verifica las credenciales y genera un JWT, luego lo envía de vuelta.
3. **Almacenamiento:** El cliente guarda el JWT, típicamente en una cookie del navegador o almacenamiento local.
4. **Solicitud:** En cada solicitud futura a una ruta protegida, el cliente envía el JWT y el servidor lo verifica.

Un JWT consta de tres partes separadas por dos puntos:

- **Encabezado:** JSON codificado que declara el tipo de token (JWT) y el algoritmo de hash usado, como HMAC SHA256.
- **Carga útil:** JSON codificado que contiene afirmaciones sobre el usuario, como el id, nombre de usuario o rol. El encabezado y la carga útil están codificados en Base64, no encriptados, así que nunca almacenes datos sensibles como contraseñas aquí.
- **Firma:** Creada al hashear el encabezado y la carga útil codificados junto con una clave secreta conocida solo por el servidor. Si la carga útil es alterada, la firma no coincidirá y el servidor rechazará la solicitud con una respuesta `401 Unauthorized`.

Los JWT habilitan **autenticación sin estado**. El servidor no necesita almacenar datos de sesión en una base de datos porque la firma prueba que creó el token.

## Falsificación de solicitudes entre sitios (CSRF)

CSRF es un ataque que obliga a un usuario autenticado a ejecutar acciones no deseadas en una aplicación web sin saberlo. Un sitio malicioso engaña al navegador de la víctima para enviar una solicitud fraudulenta a un servidor backend, aprovechando que los navegadores adjuntan automáticamente cookies a cualquier solicitud dirigida al servidor objetivo.

**Flujo de ataque ejemplo:**

1. Inicias sesión, por ejemplo, en `trusted-bank.com`. El banco almacena un token de sesión en la cookie de tu navegador.
2. Sin cerrar sesión, visitas `evil-hacker-site.com`.
3. El sitio malicioso envía silenciosamente una solicitud a `trusted-bank.com/transfer?amount=1000&to=hacker`.
4. Tu navegador adjunta automáticamente la cookie de sesión bancaria, por lo que el servidor procesa la solicitud.

El sitio malicioso no puede leer tu cookie. Simplemente se aprovecha de que tu navegador adjunta cookies automáticamente a cualquier solicitud dirigida al servidor objetivo.

**Defensas contra CSRF:**

- **`SameSite=Strict`:** El navegador nunca envía la cookie si la solicitud proviene de un sitio web de terceros.
- **`SameSite=Lax`:** Permite cookies en navegación segura (clic en un enlace), pero las bloquea en envíos de formularios y solicitudes API entre sitios.
- **Tokens CSRF:** Cuando un usuario carga una página, el servidor genera un token aleatorio, único e impredecible y lo adjunta al formulario frontend o sesión. El cliente debe incluirlo en los encabezados de la solicitud o en los datos del formulario. Como un sitio malicioso no puede acceder al frontend para copiar el token, las solicitudes falsificadas fallan la validación.

## Passport.js

Passport.js es un middleware de autenticación para Node.js que estandariza cómo autenticar solicitudes usando plugins modulares llamados **Estrategias**. Con más de 500 estrategias disponibles, puedes agregar cualquier método de autenticación instalando el paquete correspondiente:

- `passport-local`: nombre de usuario y contraseña contra tu propia base de datos.
- `passport-jwt`: valida JSON Web Tokens en rutas API protegidas.
- `passport-google-oauth20`: iniciar sesión con cuentas de Google.
- `passport-github2`: iniciar sesión con cuentas de GitHub.

Instala el paquete principal y una estrategia:

```bash
npm install passport passport-local
```

Integra Passport en tu app Express antes de definir rutas:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database look-up and password verification logic
}));
```

Protege una ruta de inicio de sesión con Passport como middleware:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

Después de una autenticación exitosa, Passport maneja automáticamente la **gestión de sesiones** (almacena datos del usuario en la cookie de sesión) y la **decoración de solicitudes** (adjunta el objeto usuario a `req.user` para usar en manejadores de rutas posteriores).

## Helmet.js

Helmet.js es una colección de middleware de seguridad para Node.js que protege tu aplicación configurando, modificando o eliminando automáticamente encabezados HTTP de respuesta que Express expone por defecto.

Instálalo y aplícalo en un solo paso:

```bash
npm install helmet
```

```js
const helmet = require('helmet');
app.use(helmet());
```

Colocar `app.use(helmet())` al inicio de tu pila de middleware aplica estas protecciones a cada respuesta:

- **X-Powered-By eliminado:** Express anuncia su presencia por defecto con `X-Powered-By: Express`. Helmet elimina este encabezado, evitando que atacantes apunten a vulnerabilidades específicas de versión.
- **Content-Security-Policy configurado:** Restringe de dónde el navegador puede cargar scripts, imágenes y estilos, defendiendo contra ataques Cross-Site Scripting (XSS).
- **X-Frame-Options configurado a SAMEORIGIN:** Evita que tu sitio sea incrustado en un iframe externo, bloqueando ataques de clickjacking.
- **Strict-Transport-Security configurado:** Obliga a los navegadores a comunicarse con tu servidor solo por HTTPS, protegiendo datos de intercepciones en redes no confiables.
