# --description--

이전 강의에서 백엔드 서버가 로그인 중에 사용자를 어떻게 식별하는지 배웠습니다. 그렇다면 로그인 후에는 어떻게 될까요? 사용자가 새 페이지로 이동할 때 서버는 어떻게 그 사용자가 누구인지 기억할까요? 바로 이때 JSON 웹 토큰(JWT)이 등장합니다.

쉽게 말해 JWT는 서버가 로그인에 성공한 사용자에게 발급하는 디지털 키카드입니다. 사용자는 이후 모든 요청에서 이 키카드를 서버에 제시하여 비밀번호를 다시 입력하지 않고도 자신의 신원을 증명합니다.

서버가 모든 활성 사용자를 자체 메모리에 기억하는 대신, 간단한 네 단계 과정을 통해 책임을 클라이언트로 넘깁니다:

1. **로그인:** 사용자가 사용자 이름과 비밀번호 또는 다른 로그인 수단을 서버에 보냅니다.

2. **발급:** 서버는 가입 시 저장된 자격 증명과 입력된 자격 증명을 비교해 확인합니다. 맞으면 서버가 JWT를 생성해 사용자에게 보냅니다.

3. **저장:** 클라이언트는 JWT를 받아 보통 브라우저 쿠키나 로컬 저장소에 저장합니다.

4. **요청:** 이후 보호된 라우트에 대한 모든 요청마다 클라이언트가 자동으로 JWT를 함께 보냅니다. 서버는 토큰을 검증하고 접근을 허용합니다.

JWT는 길고 복잡한 문자열처럼 보입니다. 두 개의 점으로 구분된 세 개의 뚜렷한 부분으로 구성되어 있습니다:

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

첫 번째는 **헤더**, 두 번째는 **페이로드**, 세 번째는 **서명**입니다.

**헤더**는 인코딩된 JSON 객체로, 서버에 토큰 유형(JWT)과 보안을 위해 사용된 해싱 알고리즘(HMAC SHA256 등)을 알려줍니다.

**페이로드**는 토큰의 중간 부분으로, 사용자에 관한 실제 데이터 또는 "클레임"을 담고 있습니다. 여기에는 사용자 ID, 사용자 이름, 관리자나 사용자 같은 계정 역할 정보를 저장합니다.

중요한 보안 사항은 헤더와 페이로드는 암호화된 것이 아니라 단지 인코딩(Base64)되었다는 점입니다. 즉, 누구나 쉽게 디코딩해 내부 데이터를 읽을 수 있습니다. 따라서 원본 비밀번호나 신용카드 번호 같은 민감한 정보를 JWT 페이로드에 절대 저장하지 마세요.

**서명**은 토큰의 마지막이자 가장 중요한 부분입니다. 서버는 인코딩된 헤더와 페이로드를 서버만 아는 비밀 키와 함께 섞어 해싱해 서명을 만듭니다.

서명은 변조를 막아줍니다. 사용자가 토큰을 디코딩해 자신의 역할을 "user"에서 "admin"으로 바꾸고 다시 인코딩하면, 서명이 변경된 데이터와 맞지 않게 됩니다.

서버가 수정된 토큰을 받으면 비밀 키로 서명을 다시 만들려고 시도합니다. 페이로드가 바뀌었기 때문에 계산된 서명이 토큰의 서명과 일치하지 않아 서버가 즉시 401 Unauthorized 상태 코드로 요청을 거부합니다.

요약하면, JWT는 무상태 인증을 가능하게 합니다. 서버는 데이터베이스에 세션 데이터를 저장할 필요 없이, 서명이 서버가 직접 생성했음을 증명하기 때문에 토큰을 신뢰합니다.
