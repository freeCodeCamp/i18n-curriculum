# --description--

In SQL, le chiavi primarie e le chiavi esterne si usano per stabilire relazioni tra tabelle.

Iniziamo con le chiavi primarie.

Una chiave primaria è una colonna o un insieme di colonne che identifica in modo univoco ogni record (riga) in una tabella. Questa restrizione garantisce che nessun record nella tabella abbia lo stesso valore per la chiave primaria.

Per questo motivo una tabella può avere solo una chiave primaria.

I valori nella colonna della chiave primaria non possono essere `NULL`, quindi avranno sempre un valore valido.

In PostgreSQL, per rendere una colonna la chiave primaria della tua tabella, devi solo aggiungere `PRIMARY KEY` dopo il tipo di dato:

```sql
column_name data_type PRIMARY KEY
```

Questo è un esempio di tabella `students`:

```sql
CREATE TABLE students (
  student_id SERIAL PRIMARY KEY,
  name VARCHAR(100)
);
```

Ogni studente avrà il proprio `student_id`, e questo ID sarà la chiave primaria della tabella. Il tipo `SERIAL` è utile qui perché garantisce che il `student_id` avrà sempre un valore univoco.

Una chiave primaria composta serve quando una tabella non ha una singola colonna unica per identificare la riga. In questo caso puoi usare una combinazione di due o più colonne come chiave primaria che, insieme, sono uniche.

Per farlo, indica quali colonne sono la chiave primaria composta quando crei la tabella così:

```sql
CREATE TABLE table_name (
  column1 data_type column_constraint,
  column2 data_type column_constraint,
  column3 data_type column_constraint,
  ...
  PRIMARY KEY (column1, column2)
);
```

Immagina di avere una tabella con colonne `student_id` e `course_id` che ti dicono quali studenti sono iscritti a varie classi.

Ogni studente sarà iscritto a diverse classi, e ogni classe avrà diversi studenti iscritti. Quindi nessuna delle due colonne è unica — ma non avrai mai lo stesso studente iscritto alla stessa classe più di una volta. Quindi puoi usare la combinazione delle due colonne come chiave primaria:

```sql
CREATE TABLE course_enrollments (
  student_id INT,
  course_id INT,
  ...
  PRIMARY KEY (student_id, course_id),
);
```

Le chiavi primarie sono essenziali per garantire l’unicità e l’integrità dei dati, per ottimizzare il recupero dei dati e per stabilire relazioni con altre tabelle in un database relazionale.

Questo ci porta al tipo di chiave successivo che tratteremo in questa lezione: le chiavi esterne.

Una chiave esterna è una colonna (o un insieme di colonne) in una tabella che fa riferimento alla chiave primaria di un’altra tabella. Una tabella può avere più chiavi esterne.

In questo esempio, abbiamo due tabelle, `customers` e `orders`:

```sql
CREATE TABLE customers (
  customer_id SERIAL PRIMARY KEY,
  first_name VARCHAR(100) NOT NULL,
  ...
);

CREATE TABLE orders (
  order_id SERIAL PRIMARY KEY,
  customer_id INTEGER,
  ...
  FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);
```

La chiave primaria della tabella `customers` è `customer_id` e la chiave primaria della tabella `orders` è `order_id`.

Ma nota che c’è anche una colonna `customer_id` nella tabella `orders`.

Perché stiamo archiviando informazioni sui clienti nella tabella `orders`?

Lo facciamo per creare una relazione tra le tabelle `customers` e `orders`. Ogni ordine memorizzerà l’ID del cliente che lo ha inviato.

Per creare la relazione tra le tabelle, renderemo la colonna `customer_id` nella tabella `orders` una chiave esterna che fa riferimento alla colonna `customer_id` nella tabella `customers`:

```sql
FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
```

I valori in una colonna chiave esterna devono corrispondere ai valori della chiave primaria nella tabella di riferimento oppure essere `NULL` (se la chiave esterna permette `NULL`).

Perché questo è utile?

Garantendo che queste colonne in entrambe le tabelle corrispondano esattamente, le chiavi esterne e primarie impediscono la creazione di record "orfani", cioè record che fanno riferimento a un record inesistente in un’altra tabella.

Le chiavi primarie e le chiavi esterne sono fondamentali per i database relazionali. Alimentano l’aspetto "relazionale" di questi database, permettendoti di modellare dati del mondo reale in modo accurato.
