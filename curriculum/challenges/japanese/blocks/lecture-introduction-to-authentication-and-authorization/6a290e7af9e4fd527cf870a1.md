# --description--

前の講義では、バックエンドサーバーがログイン時にユーザーをどのように識別するかを学びました。しかし、ログイン後はどうなるのでしょうか？ユーザーが新しいページに移動したときにサーバーはどのようにして彼らを覚えているのでしょうか？ここでJSON Web Token（JWT）が登場します。

簡単に言うと、JWTはサーバーがログイン成功後にユーザーに発行するデジタルキーのようなものです。ユーザーはそのキーを以降のすべてのリクエストでサーバーに提示し、パスワードを再入力せずに自分の身元を証明します。

サーバーがすべてのアクティブユーザーを自分のメモリに覚えておく代わりに、責任をクライアントに移し、シンプルな4つのステップで処理します。

1. **ログイン:** ユーザーはユーザー名とパスワード、または他のログイン手段をサーバーに送信します。

2. **発行:** サーバーは登録時に保存された認証情報と照合します。正しければ、JWTを生成してユーザーに返します。

3. **保存:** クライアントはJWTを受け取り、通常はブラウザのクッキーやローカルストレージに保存します。

4. **リクエスト:** 保護されたルートへの将来のすべてのリクエストで、クライアントは自動的にJWTを送信します。サーバーはトークンを検証し、アクセスを許可します。

JWTは長くて複雑な文字列のように見えます。これは2つのドットで区切られた3つの異なる部分で構成されています。例えば次のように：

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

最初の部分が**ヘッダー**、2番目が**ペイロード**、3番目が**署名**です。

**ヘッダー**はエンコードされたJSONオブジェクトで、トークンの種類（JWT）や、HMAC SHA256のようなどのハッシュアルゴリズムが使われているかをサーバーに伝えます。

**ペイロード**はトークンの中央部分で、ユーザーに関する実際のデータ、つまり「クレーム」が含まれています。ここにはユーザーID、ユーザー名、管理者やユーザーなどのアカウント役割の情報を保存します。

重要なセキュリティ上の注意点として、ヘッダーとペイロードはエンコード（Base64）されているだけで暗号化されていません。つまり誰でも簡単にデコードして中のデータを読むことができます。したがって、生のパスワードやクレジットカード番号のような機密情報をJWTのペイロードに保存してはいけません。

**署名**はトークンの最後で最も重要な部分です。サーバーはエンコードされたヘッダーとペイロードを、サーバーだけが知る秘密鍵と組み合わせてハッシュ化し、署名を作成します。

署名は改ざんを防ぎます。ユーザーがトークンをデコードして、役割を「user」から「admin」に変更し再エンコードしようとしても、署名は変更されたデータと一致しなくなります。

サーバーが改ざんされたトークンを受け取ると、秘密鍵を使って署名を再生成しようとします。ペイロードが変わっているため、計算された署名はトークンの署名と一致せず、サーバーは即座に401 Unauthorizedのステータスコードでリクエストを拒否します。

要するに、JWTはステートレス認証を可能にします。サーバーはデータベースにセッションデータを保存する必要がなく、署名がサーバー自身がトークンを作成したことを証明するため、トークンを信頼するだけで済みます。
