## Códigos de estado de respuesta HTTP

- Los **códigos de estado** son mensajes numéricos breves que indican al cliente qué ocurrió después de hacer una solicitud (por ejemplo, 200, 404 o 500).

### 1xx — Informativos

- **100 Continue** — el servidor indica al cliente que envíe el resto de la solicitud.
- **101 Switching Protocols** — el servidor está cambiando protocolos (comúnmente visto con WebSockets).

### 2xx — Éxito

- **200 OK** — la solicitud fue exitosa.
- **201 Created** — se creó un nuevo recurso con éxito.
- **204 No Content** — la solicitud fue exitosa pero no hay cuerpo en la respuesta (a menudo usado para `DELETE`).

### 3xx — Redirección

- **301 Moved Permanently** — el recurso tiene una nueva URL permanente.
- **302 Found** — el recurso está temporalmente en una URL diferente.
- **304 Not Modified** — el cliente puede usar su versión en caché; nada ha cambiado.

### 4xx — Errores del cliente

- **400 Bad Request** — la solicitud está mal formada o es inválida.
- **401 Unauthorized** — se requiere autenticación.
- **403 Forbidden** — autenticado, pero no tiene permiso.
- **404 Not Found** — el recurso solicitado no existe.

### 5xx — Errores del servidor

- **500 Internal Server Error** — fallo genérico del lado del servidor.
- **502 Bad Gateway** — un proxy recibió una respuesta inválida de otro servidor.
- **503 Service Unavailable** — el servidor está temporalmente sobrecargado o caído.

### Uso de códigos de estado en Express

- `res.status(400).send("Bad Request")` — envía un 400 con un mensaje en texto plano.
- `res.status(200).json({ message: "Success" })` — envía un 200 con un cuerpo JSON.
- `res.status(500).send("Internal Server Error")` — envía un 500 con un mensaje.

## Manejo de errores en Express

- El **middleware para manejo de errores** tiene cuatro parámetros: `(err, req, res, next)`. La firma con cuatro parámetros es lo que indica a Express que este es un manejador de errores.
- Coloca el middleware para manejo de errores **al final** de la pila de middleware, después de todas las rutas.

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Manejo de errores asíncronos en Express 4

- Los errores asíncronos **no se capturan automáticamente** en Express 4.
- Envuelve la lógica asíncrona en `try/catch` y llama a `next(err)` para enviar el error al manejador de errores.

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- Un envoltorio `asyncHandler` evita repetir `try/catch` en cada ruta:

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Manejo de errores asíncronos en Express 5

- En Express 5, los errores asíncronos y las promesas rechazadas se **envían automáticamente** al manejador de errores — no se necesita `next(err)` manual.

### Manejo de errores 404

- Agrega un middleware catch-all **después de las rutas pero antes del manejador de errores** para manejar rutas no coincidentes:

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Buenas prácticas para el manejo de errores

- Siempre coloca el middleware para manejo de errores al final.
- Maneja los 404 por separado de otros errores.
- En producción, registra los detalles de los errores internamente — nunca envíes rastros de pila a los usuarios.

## Depuración y registro en Express

- **Depuración integrada**: configura la variable de entorno `DEBUG` para habilitar la salida de depuración integrada de Express.
  - `DEBUG=express:* node index.js` — habilita todos los mensajes de depuración de Express.
  - `DEBUG=express:router node index.js` — muestra solo la salida de depuración relacionada con el enrutador.
  - PowerShell: `$env:DEBUG = "express:*"; node index.js`
  - CMD: `set DEBUG=express:* && node index.js`
- **`morgan`**: un middleware popular para registrar solicitudes HTTP (método, URL, código de estado, tiempo de respuesta).

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- La **depuración** rastrea cómo Express maneja solicitudes y middleware sin saturar los registros de producción.
- El **registro** documenta lo que sucede en las solicitudes — útil para detectar bugs y entender el tráfico.

## Verificaciones de estado y apagados controlados

### Verificaciones de estado

- Una **ruta de verificación de estado** informa si la app está funcionando correctamente.
- Los balanceadores de carga y orquestadores (por ejemplo, Kubernetes, AWS Elastic Beanstalk) hacen ping a esta ruta para decidir si el servicio está saludable.
- Una respuesta **200** significa que está saludable; la ausencia de respuesta provoca un reinicio automático.

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### Apagado controlado

- Un **apagado controlado** permite que la app termine las solicitudes existentes antes de detenerse, en lugar de cortar las conexiones inmediatamente.
- Escucha la señal `SIGTERM` (enviada cuando el proceso debe terminar) y llama a `server.close()`:

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- Escucha la señal `SIGINT` (enviada cuando el desarrollador presiona Ctrl+C durante el desarrollo local) y llama a la misma lógica de apagado que `SIGTERM`.

### Marcar la app como no saludable antes del apagado

- Devuelve un **503** desde la verificación de estado durante el apagado para que el balanceador de carga deje de enviar tráfico nuevo a la app.

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
