# --description--

어서션과 단위 테스트는 이전 수업에서 처음 소개했습니다. 복습하자면, 단위 테스트는 전체 프로그램의 최종 결과를 테스트하는 것과 달리 개별 함수나 컴포넌트의 출력을 독립적으로 확인하는 것입니다. 어서션은 코드가 예상대로 작동하는지 테스트하는 데 사용합니다.

이번 수업에서는 단위 테스트에서 어서션을 좀 더 자세히 살펴보고 몇 가지 모범 사례를 탐구하겠습니다.

자주 사용되는 어서션 라이브러리 중 하나가 Chai입니다. 이 라이브러리는 freeCodeCamp 코드베이스 내에서도 사용됩니다. 워크숍이나 실습을 진행할 때마다 자동화된 테스트 스위트가 코드에 대해 실행되어 코드가 의도한 대로 작동하는지 확인합니다.

Full-Stack Developer Curriculum에서 가져온 몇 가지 예제를 살펴보겠습니다.

Build a Calculator 워크숍에서 1단계는 `addTwoAndSeven`라는 함수를 테스트하는 것입니다:

```js
assert.isFunction(addTwoAndSeven);
```

이 어서션은 제공된 코드에 `addTwoAndSeven`라는 이름의 함수가 포함되어 있는지 확인합니다. 사용자가 배열, 문자열 또는 함수가 아닌 다른 것을 `addTwoAndSeven`에 할당하면 테스트가 실패합니다.

워크숍 후반부의 또 다른 어서션을 살펴보겠습니다:

```js
assert.equal(addThreeAndFour(), 7);
```

이 어서션은 `addThreeAndFour` 함수의 반환값이 숫자 `7`와 같은지 확인합니다. 반환값이 `7`와 같지 않으면 테스트가 실패합니다.

Chai에는 값이 불리언인지, 객체에 특정 속성 이름이 있는지, 값이 `null`인지 확인하는 등 더 많은 어서션이 내장되어 있습니다. 다른 일반적인 JavaScript 어서션 라이브러리로는 should.js와 expect.js가 있습니다.

어떤 어서션 라이브러리를 사용하든 알아두어야 할 몇 가지 모범 사례가 있습니다. 첫 번째는 명확한 어서션 메시지를 작성하는 것입니다. 명확하고 의미 있는 어서션 메시지는 그 의도를 설명해야 합니다. 그렇지 않으면 어떤 테스트가 실패했는지 찾기 어려워집니다.

또 다른 모범 사례는 설명적인 실패 메시지를 작성하는 것입니다. 설명적인 실패 메시지는 어떤 조건이 참인지 명시해야 합니다. 이는 특정 어서션이 왜 실패했는지 파악하는 데 도움이 됩니다.

코딩을 계속 배우면서 테스트에 익숙해지고 개인 프로젝트에 대한 테스트를 작성하기 시작하는 것이 중요합니다. 이렇게 하면 첫 개발자 직장을 준비하는 데 큰 도움이 됩니다.
