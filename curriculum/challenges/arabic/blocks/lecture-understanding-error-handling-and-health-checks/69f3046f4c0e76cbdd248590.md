# --description--

هل تساءلت يومًا كيف تحافظ على استقرار تطبيق Express وجعله جاهزًا للإنتاج، خاصةً عندما يكون تحت حمل ثقيل أو أثناء الإغلاق؟

هنا تأتي أهمية فحوصات الصحة وعمليات الإغلاق السلس. هذان النمطان يساعدانك على إبقاء التطبيق متاحًا، ويضمنان خروجه بشكل نظيف دون فقدان الطلبات أو تلف البيانات.

لنشرح الأمر.

أولًا، ما هو "فحص الصحة" في Express؟

فحص الصحة هو مسار بسيط يخبرك ما إذا كان تطبيقك يعمل ويستجيب.

على سبيل المثال:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

أدوات مثل موازنات التحميل أو منظمي الحاويات - مثل Kubernetes أو AWS Elastic Beanstalk - يمكنها إرسال طلبات إلى هذا المسار لتقرر ما إذا كان تطبيقك صحيًا.

إذا أعاد المسار رمز الحالة 200، يُعتبر خدمتك صحية. للتذكير - رمز الحالة 200 في HTTP يعني أن كل شيء على ما يرام!

إذا لم يستجب، يمكن للمنظم إعادة تشغيل الخدمة تلقائيًا.

وماذا عن "الإغلاق السلس"؟

الإغلاق السلس هو ما يحدث عندما يكون تطبيقك على وشك التوقف، مثل إعادة تشغيل الخادم أو أثناء مرحلة التشغيل.

بدلًا من قطع الاتصالات فورًا، تعطي التطبيق وقتًا لإنهاء ما يقوم به.

في Express، يمكنك الاستماع إلى `SIGTERM`، وهو إشارة نظام تُرسل عندما يجب إنهاء عملية.

إليك مثالًا:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

هذا يخبر Express بالتوقف عن قبول اتصالات جديدة والانتظار حتى تنتهي الاتصالات الحالية قبل الخروج.

يجب أيضًا التعامل مع `SIGINT`، وهي الإشارة التي تُرسل عند الضغط على Ctrl+C لإيقاف التطبيق أثناء التطوير المحلي.

بينما يُستخدم `SIGTERM` عادةً من قبل منظمي الحاويات مثل Kubernetes في الإنتاج، يغطي `SIGINT` سير عمل المطور اليومي. بدون التعامل معه، يؤدي الضغط على Ctrl+C إلى إنهاء العملية فورًا متجاوزًا أي تنظيف قمت بإعداده.

نظرًا لأن كلا الإشارتين يجب أن تفعلا نفس سلوك الإغلاق السلس، يستمع المطورون عادةً إلى كلاهما.

الآن، دعنا نجمع كل شيء معًا

تخيل أن تطبيقك يعمل داخل حاوية. يرسل Kubernetes إشارة `SIGTERM` لإيقافه من أجل تحديث.

مع الإغلاق السلس الأساسي فقط، يتوقف تطبيقك عن قبول اتصالات جديدة وينهي الطلبات الحالية. بمجرد الانتهاء من كل العمل، ينفصل عن قاعدة البيانات ويخرج بشكل نظيف. لا فقدان بيانات، ولا حالة تالفة.

لكن هناك مشكلة — موازن التحميل لا يعرف أن تطبيقك يغلق، لذا قد يستمر في إرسال حركة مرور جديدة إليه. هنا يأتي دور النمط الإضافي أدناه.

الآن، لدي لك مكافأة صغيرة. سننظر في "وضع علامة على التطبيق كغير صحي قبل الإغلاق".

قبل الإغلاق، يمكنك تحديث فحص الصحة ليُعيد رمز الحالة 503. بهذه الطريقة، يرى موازن التحميل الاستجابة غير الصحية ويتوقف عن توجيه حركة مرور جديدة إلى تطبيقك خلال فترة الإغلاق، كما هو موضح في الشفرة أعلاه.

تذكير: رمز الحالة `503` يعني "الخدمة غير متوفرة". يشير إلى أن الخادم غير قادر مؤقتًا على معالجة الطلبات، وهذا بالضبط ما يحدث أثناء الإغلاق السلس.

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
