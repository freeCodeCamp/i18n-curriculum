# --description--

¿Alguna vez te has preguntado cómo mantener tu app de Express estable y lista para producción, especialmente cuando está bajo mucha carga o se está apagando?

Ahí es donde entran los chequeos de salud y los apagados ordenados. Estos dos patrones te ayudan a mantener tu app disponible y aseguran que se cierre limpiamente sin perder solicitudes ni corromper datos.

Vamos a desglosarlo.

Primero, ¿qué es un "chequeo de salud" en Express?

Un chequeo de salud es una ruta simple que indica si tu app está activa y funcionando.

Por ejemplo:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

Herramientas como balanceadores de carga u orquestadores de contenedores — piensa en Kubernetes o AWS Elastic Beanstalk — pueden hacer ping a esta ruta para decidir si tu app está saludable.

Si la ruta devuelve un código de estado 200, tu servicio se considera saludable. Como recordatorio: un código HTTP 200 significa que todo está bien.

Si no responde, el orquestador puede reiniciar el servicio automáticamente.

¿Y qué es un "apagado ordenado"?

Un apagado ordenado es lo que sucede cuando tu app está a punto de detenerse, como cuando un servidor se reinicia o se realiza un despliegue.

En lugar de cortar las conexiones inmediatamente, le das tiempo a la app para terminar lo que está haciendo.

En Express, puedes escuchar `SIGTERM`, que es una señal del sistema enviada cuando un proceso debe terminar.

Aquí tienes un ejemplo:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

Esto le indica a Express que deje de aceptar nuevas conexiones y espere a que las existentes terminen antes de salir.

También deberías manejar `SIGINT`, que es la señal que se envía cuando presionas Ctrl+C para detener tu app durante el desarrollo local.

Mientras que `SIGTERM` se usa típicamente por orquestadores como Kubernetes en producción, `SIGINT` cubre el flujo de trabajo diario del desarrollador. Sin manejarla, presionar Ctrl+C mata el proceso inmediatamente, saltándose cualquier limpieza que hayas configurado.

Como ambas señales deberían activar el mismo comportamiento de apagado ordenado, los desarrolladores suelen escuchar ambas.

Ahora, juntémoslo todo

Imagina que tu app se ejecuta en un contenedor. Kubernetes envía una `SIGTERM` para apagarla y actualizarla.

Con solo un apagado ordenado básico, tu app deja de aceptar nuevas conexiones y termina las solicitudes actuales. Una vez que todo el trabajo termina, se desconecta de la base de datos y sale limpiamente. Sin pérdida de datos ni estado corrupto.

Pero hay un problema: el balanceador de carga no sabe que tu app se está apagando, así que podría seguir enviando tráfico nuevo hacia ella. Ahí es donde entra el patrón extra que veremos a continuación.

Ahora, tengo un pequeño extra para ti. Vamos a ver "Marcar la app como no saludable antes del apagado".

Antes de apagar, puedes actualizar tu chequeo de salud para que devuelva un estado 503. Así, el balanceador de carga ve la respuesta no saludable y deja de enviar tráfico nuevo a tu app durante la ventana de apagado, como se muestra en el código anterior.

Recordatorio: un código `503` significa "Servicio no disponible". Señala que el servidor no puede manejar solicitudes temporalmente, que es exactamente lo que pasa durante un apagado ordenado.

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
