# --description--

**REST** steht für Representational State Transfer (REST). Man kann es sich als eine Reihe von Richtlinien vorstellen, die beschreiben, wie Web-Clients und Server über das Internet miteinander kommunizieren sollten.

In dieser Lektion lernen Sie, was REST ist, seine Schlüsselkonzepte und wie es funktioniert.

REST definiert eine Methode, wie verschiedene Systeme mithilfe standardisierter Webtechnologien interagieren. Mit REST kommunizieren Clients wie Webbrowser, mobile Apps und APIs mit einem Server unter Verwendung gängiger HTTP-Methoden wie `GET`, `POST`, `PUT`, `PATCH` und `DELETE`, die wir in einer vorherigen Lektion besprochen haben.

Zu den Kernkonzepten von REST gehören **Zustandslosigkeit**, **Ressourcen**, **HTTP-Methoden** und **Antwort- oder Statuscodes**.

**Zustandslosigkeit** bedeutet, dass jede Anfrage, die ein Client an den Server sendet, unabhängig behandelt wird. Der Server merkt sich keine vorherigen Anfragen. Deshalb muss jede Anfrage alle Informationen enthalten, die zur Verarbeitung nötig sind. Wenn ein Benutzer beispielsweise authentifiziert ist, muss sein Authentifizierungstoken mit jeder Anfrage gesendet werden.

In REST wird alles als Ressource behandelt. Eine Ressource ist ein Datenstück, auf das zugegriffen oder das verändert werden kann. Jede Ressource wird durch eine URL identifiziert, wie `/products` für alle Produkte oder `/products/1` für ein bestimmtes Produkt.

Nachfolgend finden Sie eine Übersicht über HTTP-Methoden, Statuscodes und ein Schritt-für-Schritt-Beispiel, wie die REST-Architektur funktioniert.

## HTTP-Methoden

HTTP-Methoden definieren die Aktion, die auf einer Ressource ausgeführt werden soll:

* `GET`: Eine Ressource abrufen
    
* `POST`: Eine neue Ressource erstellen
    
* `PUT / PATCH`: Eine bestehende Ressource aktualisieren
    
* `DELETE`: Eine Ressource entfernen
    

## HTTP-Statuscodes

Nach der Verarbeitung einer Anfrage antwortet der Server mit einem **Statuscode**, der das Ergebnis angibt:

* `200` **(OK)**: Die Anfrage war erfolgreich
    
* `201` **(Created)**: Eine neue Ressource wurde erstellt
    
* `400` **(Bad Request)**: Der Client hat ungültige Daten gesendet
    
* `404` **(Not Found)**: Die angeforderte Ressource existiert nicht
    
* `500` **(Internal Server Error)**: Auf dem Server ist ein Fehler aufgetreten
    

## Wie REST funktioniert: Ein Schritt-für-Schritt-Beispiel

### 1\. Client sendet eine Anfrage

* Der Client sendet eine HTTP-Anfrage mit einer Methode wie `GET` oder `POST`
    
* Die Anfrage enthält eine URL, die die Ressource identifiziert
    
* Sie kann auch Kopfzeilen, Metadaten und einen Anfragekörper enthalten
    

### 2\. Server empfängt die Anfrage

* Der Server überprüft das Format der Anfrage und die Authentifizierung (falls erforderlich)
    
* Er identifiziert die Zielressource anhand der URL
    
* Er bestimmt die angeforderte Aktion basierend auf der HTTP-Methode
    
* Er prüft Berechtigungen und Zugriffskontrolle
    

### 3\. Server führt die Aktion aus

* `GET`: Die Ressource abrufen
    
* `POST`: Eine neue Ressource erstellen
    
* `PUT / PATCH`: Die Ressource aktualisieren
    
* `DELETE`: Die Ressource entfernen
    
* Alle weiteren notwendigen Logiken werden ausgeführt
    

### 4\. Server erstellt die Antwort

* Der Server wählt einen passenden Statuscode aus (zum Beispiel `200`, `201` oder `404`)
    
* Er formatiert die Antwortdaten, meist als **JSON**
    
* Er fügt relevante Kopfzeilen hinzu, wie `Content-Type` oder `Cache-Control`
    

### 5\. Server sendet die Antwort

* Die Antwort enthält einen Statuscode, Kopfzeilen und optional einen Körper
    
* Es wird kein Client-Zustand auf dem Server gespeichert (Zustandslosigkeit)
    

### 6\. Client verarbeitet die Antwort

* Der Client prüft den Statuscode, um Erfolg oder Fehler zu erkennen
    
* Er verarbeitet alle vom Server zurückgegebenen Daten
