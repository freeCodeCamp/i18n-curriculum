# --description--

Web 服务和 REST API 支持你每天使用的大多数应用和网站。它们允许不同的应用程序通过网络（最常见的是互联网）相互通信并共享数据。

API（应用编程接口）是软件与其他软件通信的通用方式。Web 服务是一种特定类型的 API，使用诸如 `HTTP` 之类的 Web 技术，通过网络（通常是互联网）交换数据。因此，每个 Web 服务都是 API——但并非每个 API 都是 Web 服务。REST API 是一种常见的 Web 服务类型，遵循一套特定的规则来请求和返回数据。

本课将重点介绍什么是网络服务的高级概念，下一课将详细讲解 `REST API` 的工作原理。

你可以将网络服务视为一个软件与软件之间的桥梁。它没有用户界面，也不打算被直接使用。相反，它完全专注于在应用之间交换数据。

任何系统要被视为网络服务，必须具备以下特征：

* **运行在服务器上**：网络服务必须像网站一样在线托管，但它不是向用户展示页面，而是公开 `data` 或某些 `functionality`
    
* **机器对机器通信**：网络服务不是为人类直接查看而设计的，而是供移动应用和网络应用使用，以便它们可以暴露一些用户最终可以查看的 `data` 或 `functionality`。
    
* **语言无关**：一个网络服务可以用任何语言编写，例如 Java 或 C#，任何应用都应该能够与其交互，即使它是用不同的语言编写的
    
* **与平台无关**：机器运行在哪个平台无关紧要。例如，运行在 Linux 上的网络服务不应阻止运行 Windows 的设备与其通信。
    
* **使用标准的网络协议**：网络服务通常应使用 HTTP/HTTPS 在互联网传输数据，使用 FTP 进行文件传输，使用 SMTP 发送电子邮件
    

**REST** 和 **SOAP** 是两种最知名的网络服务 `type`，尽管还存在其他方法，例如 **XML-RPC**。

**REST**，即 **Representational State Transfer**，是迄今为止最广泛使用的。大多数现代应用依赖于 RESTful 网络服务，因为它们简单、灵活，并且与网络自然兼容。

REST API 暴露资源，这些资源通常由诸如 `/users` 或 `/users/20` 的 URL 路径表现。当客户请求资源时，服务器会返回该资源的表现。该表现通常是 JSON 格式，但也可以是 XML 或纯文本等格式。

客户可以通过使用超文本传输协议头部（例如 `Accept` 头部）来指定他们想要的形式。例如，客户可以通过设置适当的 `Accept` 值来请求 JSON 响应。

```bash
Accept: application/json
```

你可以通过这种方式指定你想要 XML：

```bash
Accept: application/xml
```

如果资源是 JSON，你得到的返回值可能如下所示：

```json
{
  "id": 20,
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "createdAt": "2025-09-22T10:30:00Z"
}
```

如果这是 XML：

```xml
<user>
  <id>20</id>
  <name>John Doe</name>
  <email>john@example.com</email>
  <role>admin</role>
  <createdAt>2025-09-22T10:30:00Z</createdAt>
</user>
```

SOAP 代表 **简单对象访问协议**。它仅使用 XML 数据格式，并且在需要更严格标准和安全性的企业级系统中最为常见。

下面是 SOAP `data` 的示例：

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
   <soapenv:Body>
      <getUser>
         <userId>20</userId>
      </getUser>
   </soapenv:Body>
</soapenv:Envelope>
```

XML-RPC 使用 XML 来编码其 `requests` 和 `responses`。这使其轻量但比 REST 灵活性较差。 

至于 UDDI，它不是一个数据交换服务，而是一个基于 XML 的目录，企业可以在其中列出、发现和描述其可用的 Web 服务。
