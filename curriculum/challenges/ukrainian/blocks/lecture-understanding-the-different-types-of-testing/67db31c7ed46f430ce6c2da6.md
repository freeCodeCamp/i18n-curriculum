# --description--

Мокінг — це процес заміни реальних даних на хибні, які імітують поведінку реальних компонентів. Це дозволяє тестувальникам і розробникам перевіряти конкретні компоненти застосунку без необхідності покладатися на зовнішні залежності.

Загальний випадок використання мокінгу даних — це робота з API-запитами. Уявіть, що у вашому застосунку є такий виклик fetch:

```js
export const postService = {
  async getPosts() {
    const response = await fetch("https://api.example.com/posts");
    return response.json();
  },
};
```

Якщо потрібно протестувати функцію `postService`, ви можете замокати відповідь API замість того, щоб робити безперервні виклики API для отримання даних. Отримання даних з API займає час і залежить від зовнішніх факторів, таких як доступність мережі та відповіді сервера. Мокінг відповіді API усуває ці зовнішні фактори і дозволяє тестувальникам працювати з передбачуваним, контрольованим набором даних.

Тепер, коли ми зрозуміли, що таке мокінг, порівняймо його зі стабінгом. Стаби — це об’єкти, які повертають заздалегідь визначені відповіді або фіктивні дані для очікуваної поведінки у застосунку. Наприклад, ви можете стабати поведінку підключення до бази даних у ваших тестах без необхідності покладатися на реальне підключення до бази даних.

Ось приклад стабу в Jest:

```js
import { postService } from "./postService";
import { db } from "./db";

jest.mock("./db", () => ({
  db: {
    fetchPosts: jest.fn(),
  },
}));

test("should return stubbed posts", async () => {
  const fakePosts = [
    { id: 1, title: "Stubbed Post", content: "This is a stubbed post." },
  ];

  db.fetchPosts.mockResolvedValue(fakePosts);

  const posts = await postService.getPosts();

  expect(posts).toEqual(fakePosts);
});
```

`db.fetchPosts.mockResolvedValue(fakePosts);` — це стаб у цьому прикладі.

Останній компонент, який варто обговорити, — це фейки. Фейки — це спрощені версії реальних компонентів без складності чи побічних ефектів справжніх. Наприклад, ви можете зімітувати базу даних, зберігаючи дані в пам’яті замість взаємодії з реальною базою даних. Це дозволить вам імітувати операції з базою даних у пам’яті, що буде значно швидше, ніж робота з реальною базою даних. Ще один поширений приклад — використання фейків при роботі з файловими системами. Ви можете створити фейкову файлову систему замість реальної для тестування.

Коли ви продовжуєте створювати більш складні програмні застосунки, розглядайте можливість використання моків, стабів або фейків там, де це доречно у ваших тестах.
