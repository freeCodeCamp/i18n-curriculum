# --description--

あなたのデバイスで動作するすべてのコンピュータプログラムは、特定の順序で実行される一連の指示を持っており、それによってタスクを完了します。

そのタスクは、数値の並べ替え、画像の修正、在庫の追跡、あるいはお気に入りのビデオゲームの実行などさまざまです。

ここでアルゴリズムが登場します。**アルゴリズム**とは、問題を解決したりタスクを実行したりするための明確な指示のセットです。

アルゴリズムは「レシピ」のようなものと考えることができます。料理をするとき、レシピには必要な材料がすべて記載されており、料理の準備方法がステップごとに示されています。

同様に、アルゴリズムはコンピューターに何をどうすべきかを正確に伝える「レシピ」と考えられます。

アルゴリズムには2つの重要な特徴があります：

* 無限に続くことはできません。有限のステップ数で終了する必要があります。
    
* 各ステップは正確かつ明確でなければなりません。
    

アルゴリズムは、入力がゼロ、1つ、または複数あり、出力も1つ以上生成する場合があります。

アルゴリズムのステップは、どのプログラミング言語にも依存しません。

しかし、実際にコンピューターで動かすには、PythonやJavaScriptのようなプログラミング言語で実装する必要があります。

アルゴリズムが正しければ、有効な入力に対する出力は期待される結果と一致するはずです。

正しいだけでなく、アルゴリズムは効率的であることも重要です。

アルゴリズムの効率は、実行にかかる時間や、タスクを完了するために必要なメモリの容量で測定できます。

アルゴリズムの効率を知ることは非常に重要です。なぜなら、入力サイズが大きくなるにつれてどの程度性能が維持できるかの目安になるからです。

たとえば、15個の整数を並べ替えるのと100万個の整数を並べ替えるのは同じではありません。

処理の規模や複雑さが増すにつれて、アルゴリズムがそれに対応できるほど効率的でなければ、非常に遅いプログラムになったり、システム全体がクラッシュすることもあります。

だからこそ、できるだけ効率的なアルゴリズムを開発し選ぶことが重要です。

ここでビッグO記法が非常に重要になります。

ビッグO記法は、入力サイズが増加したときのアルゴリズムの最悪ケースの性能や成長率を表します。

アルゴリズムの成長率とは、入力サイズが大きくなるにつれて必要なリソースがどのように増加するかを指します。

ビッグO記法は最悪ケースの性能に注目します。これは、入力に関係なく最悪の状況でもアルゴリズムがどれだけ効率的かを理解するために重要です。

先ほどの並べ替えの例に戻ると、100万個の整数を並べ替えるのは直感的に15個の整数を並べ替えるよりも時間とリソースがかかるはずです。

しかし、どのくらい多くかかるかは選ぶアルゴリズムによります。

ビッグO記法はアルゴリズムの効率を正確な数値で示すわけではありませんが、入力サイズが増えたときにどのようにスケールするかを、アルゴリズムが実行する操作数に基づいて示します。

ビッグO記法では、通常入力サイズを`n`で表します。たとえば、入力がリストの場合、`n`はそのリストの要素数を表します。

定数係数や低次の項は、操作数に基づくアルゴリズムの時間計算量を求める際に考慮されません。なぜなら、`n`のサイズが大きくなるにつれて、これらの小さな項の影響はどんどん小さくなるからです。

アルゴリズムの全体的な挙動を支配するのは、`n`、つまり入力サイズに関する最高次の項です。

たとえば、あるアルゴリズムが完了するのに`7n + 20`回の操作を行う場合、定数`20`の影響は`n`が大きくなるにつれて小さくなります。`7n`の項が支配的になり、これがアルゴリズムの全体的な挙動と効率を決定します。

別の例として、完了に`20n² + 15n + 7`回の操作が必要なアルゴリズムがあります。`20n²`が大きくなるにつれて`n`の項が支配的になるため、このアルゴリズムは二次の時間計算量を持つことになります。なぜなら支配的な項が`n²`だからです。

二次の時間計算量は、アルゴリズムの世界で見られる多くの時間計算量の一つです。

ここで、最も一般的なものをいくつか学びましょう。

**`O(1)`**は「定数時間計算量」と呼ばれます。定数時間計算量のアルゴリズムは、入力サイズに関係なく常に同じ時間で実行されます。

たとえば、数が偶数か奇数かをチェックする処理は、数自体に関係なく常に同じ時間がかかります。

```js
function checkEvenOrOdd(number) {
  if (number % 2 === 0) {
    return "Even";
  } else {
    return "Odd";
  }
}
```

**`O(log n)`**は「対数時間計算量」と呼ばれます。これは、入力サイズが増加してもアルゴリズムの実行時間がゆっくり増加することを意味します。問題のサイズが一定の割合で繰り返し減少する場合に一般的です。

たとえば、バイナリサーチという有名な探索アルゴリズムは`O(log n)`の最悪ケース時間計算量を持ちます。これは比較ごとに残りの要素の半分を除外するため、全体的に効率的です。

**`O(n)`**は「線形時間計算量」と呼ばれます。この時間計算量のアルゴリズムは、入力サイズに比例して実行時間が増加します。

たとえば、リストのすべての要素を反復する`for`ループは、リストの要素数が増えると反復回数も増えます。リストのサイズが2倍になれば、操作回数もほぼ2倍になります。

```js
for (const grade of grades) { // grades is an array
  console.log(grade);
}
```

**`O(n log n)`**は「対数線形時間計算量」と呼ばれます。これは、マージソートやクイックソートのような効率的なソートアルゴリズムに一般的な時間計算量です。

**`O(n²)`**は「二次時間計算量」と呼ばれます。これらのアルゴリズムの実行時間は入力サイズの二乗に比例して増加し、実際の問題には一般的に効率的ではありません。

ネストされたループは二次時間計算量の一般的な例です。内側のループは外側の`n`回の各反復に対して`n`回反復し、結果として`n`の二乗回の反復が行われます。

```js
for (let i = 0; i < n; i++) {
  for (let j = 0; j < n; j++) {
    console.log("Hello, World!");
  }
}
```

その他の時間計算量には、「指数時間計算量」`O(2^n)`や「階乗時間計算量」`O(n!)`があります。どちらも実際のシナリオでは非効率的です。

このグラフでは、最も一般的な時間計算量を表す数学関数の成長を比較できます。x軸（横軸）は入力サイズ、y軸（縦軸）はアルゴリズムの実行時間と考えてください。

二次時間計算量（`O(n²)`）（黄色）が他のものよりもはるかに速く増加し、定数時間計算量（`O(1)`）（赤）は入力が大きくなっても一定であることがわかります。

素晴らしいです。ここまでで、ビッグO記法を時間の要件の観点から学びましたが、この記法は空間の要件の文脈にも適用できます。

この場合、アルゴリズムが必要とするメモリ空間が入力サイズの増加に伴ってどのように増加するかを表します。

「定数空間計算量」`O(1)`のアルゴリズムは、入力が大きくなっても常に一定のメモリ空間を必要とします。

たとえば、メモリにいくつかの変数だけを作成して保存するアルゴリズムがこれに当たります。

対照的に、「線形空間計算量」`O(n)`のアルゴリズムは、入力サイズが増えるにつれて必要な空間も比例して増加します。

たとえば、長さ`n`のリストのコピーを作成して保存するアルゴリズムがこれに当たります。

最後に、「二次空間計算量」`O(n²)`のアルゴリズムは、入力サイズが増えるにつれて必要な空間が二乗的に増加します。

たとえば、入力サイズで決まる次元の2次元行列を作成し、すべての可能なペアを保存する場合がこれに当たります。

アルゴリズムはコンピュータプログラムの構成要素であり、ビッグO記法は最悪ケースの時間と空間の要件が入力サイズの増加に伴ってどのようにスケールするかに基づいて、アルゴリズムの効率を分析する強力なフレームワークです。効率を理解することは、実際のシナリオで効率的に動作するソフトウェアを開発するために非常に重要です。
