# --description--

이전 수업에서 기능 브랜치를 만들고, 그 브랜치에 새 파일을 커밋한 후 원격 저장소에 푸시했습니다. 그렇다면 다음 단계는 무엇일까요? GitHub에서 풀 리퀘스트를 만들어야 합니다.

풀 리퀘스트는 본질적으로 자신의 브랜치에서 대상 브랜치로 변경 사항을 끌어오도록 요청하는 것입니다. 풀 리퀘스트는 오픈 소스 프로젝트에 코드 변경을 기여할 때 사용하는 흐름입니다. 이 방식은 프로젝트 유지 관리자가 변경 사항을 검토할 수 있게 해줍니다. 유지 관리자는 주석을 남기고, 질문을 하며, 수정 제안을 할 수 있고, (운이 좋으면) 변경 사항을 승인하고 병합할 수 있습니다!

다시 GitHub의 저장소로 가봅시다.

새 브랜치를 푸시한 후에는 브랜치에 푸시했다는 멋진 배너와 풀 리퀘스트를 만들 수 있는 버튼이 보일 것입니다. 이 버튼은 몇 단계를 건너뛰고 과정을 간소화하지만, 지금은 무시할 것입니다. 이렇게 하면 배너가 나타나지 않는 경우에도 대비할 수 있습니다.

저장소 페이지 상단의 풀 리퀘스트 탭을 클릭한 다음, 배너를 무시하고 "New Pull Request"를 클릭하세요. 그러면 요청을 준비하는 UI로 이동합니다.

여기서 몇 가지 용어를 알아야 합니다. 먼저 "base"와 "compare" 드롭다운이 있습니다. "base"는 병합 대상입니다. `main`에 병합하려고 하므로 그대로 두면 됩니다.

"compare"는 병합하려는 브랜치입니다. `feature` 브랜치를 병합하려고 하므로 `feature`로 변경해야 합니다. 이것은 헤드 브랜치라고도 하며, 기억해 두면 좋습니다.

이렇게 하면 UI가 새로 고침됩니다. 이제 `feature`에 있지만 `main`에는 없는 커밋들이 표시되어 병합 요청 내용을 나타냅니다. 또한 "diff"라고 하는 실제 변경 사항의 시각적 표현도 볼 수 있습니다.

우리는 파일을 추가하는 것만 했기 때문에 diff가 꽤 깔끔합니다. 더 복잡한 변경 사항의 경우 GitHub는 줄 수정, 추가, 삭제 등을 강조 표시합니다. diff를 읽는 법을 이해하는 것은 어려울 수 있지만, 연습하면 능숙해질 것입니다.

이제 풀 리퀘스트로 돌아가 봅시다. 상단에 "compare across forks"라는 작은 링크가 있습니다. 만약 freeCodeCamp 저장소의 포크에서 작업 중이라면, 이 링크를 통해 PR 대상 저장소를 원본 freeCodeCamp 저장소로 변경할 수 있습니다. 포크 내의 `main` 브랜치만이 아니라요.

우리는 포크에서 작업하지 않으므로 이 부분은 무시해도 됩니다. 하지만 여기서 하는 작업은 주로 학습과 탐색용이라는 점을 꼭 이해하세요. 연습용 변경 사항만 포함된 PR을 다른 사람 프로젝트에 열면 유지 관리자가 불필요한 작업을 하게 됩니다.

변경 사항에 만족하고 모든 설정이 완료되었으면 "Create Pull Request" 버튼을 클릭하세요. 그러면 새로운 UI로 이동합니다.

화면 아래쪽에는 이전에 본 커밋 목록과 diff 뷰가 있습니다. PR을 만들기 전에 한 번 더 확인하는 것이 좋지만, 이번 탐색에서는 바로 진행해도 됩니다.

새로운 입력란도 몇 개 보일 것입니다. 첫 번째 입력란은 풀 리퀘스트 제목입니다. GitHub는 PR에 커밋이 하나뿐이면 그 커밋을, 여러 개면 브랜치 이름을 기반으로 자동 생성합니다.

두 번째 입력란은 설명란입니다. 대부분 비어 있지 않습니다. 프로젝트 유지 관리자가 미리 채워지는 템플릿을 정의할 수 있습니다. 템플릿이 제공되면 그 워크플로우를 존중해 작성하세요.

두 경우 모두 PR 제목과 내용을 작성하는 방법에 대해 프로젝트의 기여 문서를 읽어 지침을 따르세요.

이 저장소를 소유하고 있으므로 오른쪽에서 검토자, 담당자, 레이블, 프로젝트, 마일스톤을 설정할 수 있습니다. 다른 사람 프로젝트에 기여할 때는 검토 요청 외에는 편집할 수 없습니다. 하지만 이 저장소에서는 원하는 대로 설정해 기능을 탐색할 수 있습니다.

모든 항목을 작성하고 설정했으면 "Create pull request"를 클릭하세요. 드디어 실제 결과를 볼 수 있습니다!

다른 사람 프로젝트에 풀 리퀘스트를 만들면 "코드 리뷰" 과정을 거쳐야 합니다. 유지 관리자가 PR을 평가하고 변경을 요청하거나 질문할 수 있습니다. 승인되면 병합할 수 있습니다.

이것은 자신의 프로젝트이므로 리뷰를 기다릴 필요가 없습니다. 커밋 탭에서 커밋 기록을, 변경된 파일 탭에서 변경 사항을 다시 확인하고 만족하면 병합하세요.

병합 후에는 변경 사항을 로컬 저장소로 가져와야 합니다.

터미널로 돌아가 저장소 폴더에 있는지 확인하세요. 그런 다음 `main` 브랜치로 `git checkout main` 하세요.

다음으로 `git pull` 명령어로 최신 변경 사항을 가져오세요.

이제 `git log` 명령어로 커밋 기록을 보면 자신의 커밋이 보일 것입니다. 하지만 `Merge pull request #1` 메시지도 보일 텐데, 이게 무엇일까요?

GitHub에서 풀 리퀘스트를 병합할 때 사용할 수 있는 세 가지 전략이 있습니다:

기본값은 "Merge"로, 헤드 브랜치의 모든 커밋을 가져온 다음 병합을 처리하고 기록하는 또 다른 커밋을 만듭니다.

다른 옵션은 "Squash and Merge"로, 헤드 브랜치의 모든 커밋을 하나로 합쳐서 병합 커밋 없이 바로 베이스 브랜치에 병합합니다.

마지막으로 "Rebase and Merge"가 있는데, 모든 커밋을 베이스 브랜치의 최신 커밋 뒤로 재설정한 후 병합 커밋 없이 헤드를 병합합니다. 리베이스는 복잡한 주제로 나중에 자세히 배우게 되니 오늘은 이해하지 못해도 괜찮습니다.

각 오픈 소스 프로젝트는 자신에게 맞는 병합 전략을 선택합니다. 하지만 이런 차이 때문에 항상 새 브랜치를 만들어 기여하는 커밋을 해야 합니다. 기본 `main` 브랜치에 직접 커밋하면 원본 저장소에서 변경 사항을 가져올 때 복잡해집니다.

이제 첫 번째 풀 리퀘스트를 성공적으로 만들고 병합했습니다!
