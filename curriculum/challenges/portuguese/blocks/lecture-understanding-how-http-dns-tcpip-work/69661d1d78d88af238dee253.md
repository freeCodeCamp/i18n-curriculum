# --description--

O mundo digital como o conhecemos hoje é baseado na noção de que dispositivos podem se comunicar entre si, enviando e recebendo dados conforme necessário.

Mas você já pensou em como esse processo funciona nos bastidores?

Quando você navega na internet, você constantemente solicita e envia dados através da rede. Isso é possível graças a um processo detalhado baseado em regras e procedimentos padrão que garantem que a informação seja enviada e recebida corretamente pelos dispositivos apropriados.

Esse conjunto de regras e procedimentos é conhecido como um **protocolo**.

Protocolos basicamente permitem que dois ou mais dispositivos se comuniquem entre si garantindo que todos sigam certos padrões, como o formato dos dados que serão trocados.

Protocolos garantem que todos os dispositivos envolvidos no processo de troca de dados possam se entender. Figurativamente, todos saberão “falar a mesma língua”.

No seu núcleo, TCP/IP são os protocolos que tornam a internet possível.

**TCP/IP** é um conjunto de protocolos de comunicação que determinam como os dados são empacotados, enviados, roteados e recebidos através de uma rede. Ele garante que a informação seja transportada de forma confiável.

Especificamente, o **IP**, onde IP significa **Internet Protocol**, é um protocolo que garante que os dados vão da origem para o destino correto na rede.

Por que isso é importante? Lembre-se de que a internet é uma rede global de dispositivos interconectados. Para que seu dispositivo envie uma solicitação, primeiro ele precisa saber para onde enviar esses dados. Esse é o principal propósito dos endereços IP, a base do Protocolo de Internet (IP). Eles identificam exclusivamente os dispositivos na rede, para que o dispositivo de origem saiba para onde enviar os dados.

Por si só, o Protocolo de Internet é sem conexão, o que significa que cada unidade de dados (chamada de “pacote”) é enviada independentemente para seu destino através da rede, sem estabelecer previamente uma conexão ou garantir que será entregue corretamente ou na ordem certa. Se eles chegarem ao destino, a origem não é notificada de forma alguma.

É aqui que protocolos como TCP se tornam realmente importantes.

O protocolo **TCP**, conhecido como **Transmission Control Protocol**, divide os dados em pequenos segmentos. Essas unidades fundamentais de dados são numeradas e remontadas quando chegam ao destino usando algoritmos específicos que detectam se os dados estão completos e corretos. Esses algoritmos também verificam erros e pacotes perdidos, para que possam solicitar à origem que os reenvie se necessário, garantindo que a informação seja recebida corretamente.

Juntos, o Transmission Control Protocol e o Internet Protocol definem as regras e os padrões necessários para garantir que os dados sejam transmitidos de forma confiável pela internet.

Além do TCP/IP, existe outro protocolo muito importante para a internet moderna: HTTP.

HTTP funciona no nível da aplicação.

**HTTP** significa **HyperText Transfer Protocol**. Ele define as regras e padrões para a comunicação entre o cliente (como seu navegador) e o servidor.

Quando você tenta acessar um site no seu navegador, o navegador envia uma requisição HTTP para o servidor solicitando aquele recurso específico e seus ativos relacionados. O servidor recebe a requisição HTTP, executa sua lógica e retorna uma resposta HTTP para o cliente com o conteúdo solicitado ou alguma informação indicando por que a requisição não foi bem-sucedida.

As requisições HTTP e as respostas HTTP têm estruturas muito semelhantes, incluindo:

* Uma linha de requisição (a linha inicial)
    
* Cabeçalhos
    
* Corpo (opcional)

## **Requisições HTTP**

A **linha de requisição** de uma requisição HTTP é dividida em três partes principais:

* Um método, que descreve a ação que está sendo solicitada. Por exemplo, `GET` para recuperar dados e `POST` para enviar dados. Outros métodos comuns incluem `PUT`, `PATCH`, `DELETE` e `HEAD`.
    
* Um caminho ou URI, que descreve o recurso específico que o cliente está solicitando.
    
* A versão do protocolo HTTP que está sendo usada para o processo de comunicação. Por exemplo, `HTTP/1.1`.

Este é um exemplo com o método `GET` no caminho raiz `/` e versão HTTP 1.1:

`GET / HTTP/1.1`

O **header** contém pares chave-valor com informações sobre a requisição.

Por exemplo: `Content-Type: application/json`.

Abaixo do cabeçalho, há uma linha vazia para indicar que os metadados (informações adicionais sobre a requisição) estão completos.

E finalmente, o **body**. Isso é opcional. É enviado quando o cliente precisa enviar informações para o servidor, como ao enviar novos dados com o método `POST`.

Este é um exemplo de uma requisição HTTP:

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

## **Respostas HTTP**

Eles têm uma estrutura muito semelhante:

* Uma linha de status
    
* Cabeçalho
    
* Corpo
    

Primeiro, a **linha de status** contém informações sobre o resultado da requisição. Você geralmente encontrará isto:

* A versão do HTTP sendo usada para o processo de comunicação. Por exemplo, `HTTP/1.1`.
    
* O código de status, que indica o resultado da requisição.
    
* Uma frase de motivo, uma breve descrição legível por humanos do código de status. Isso é útil para entender o que aconteceu.
    

Códigos de status comuns incluem:

* `200 OK`
    
* `201 CREATED`
    
* `400 BAD REQUEST`
    
* `404 NOT FOUND`
    
* `403 FORBIDDEN`
    
* `500 INTERNAL SERVER ERROR`
    

Existem muitos códigos de status. Eles são categorizados pelo seu primeiro dígito da esquerda para a direita.

* `1xx` para respostas informativas.
    
* `2xx` para respostas bem-sucedidas.
    
* `3xx` para mensagens de redirecionamento.
    
* `4xx` para respostas de erro do cliente.
    
* `5xx` para respostas de erro do servidor.
    

Abaixo da linha de status de uma resposta HTTP, encontramos o **header** com metadados sobre a resposta. Por exemplo, `Content-Type: text/html`. Isso também pode incluir o tamanho do conteúdo, data, localização e informações adicionais que o cliente pode achar úteis.

E finalmente, encontramos o **body** da resposta HTTP, que contém os dados que o cliente estava solicitando ou uma explicação do que aconteceu se a solicitação não foi bem-sucedida. Isso é opcional, mas está incluído na maioria das respostas.

Este é um exemplo de uma resposta HTTP:

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

Agora que você sabe sobre HTTP, pode estar curioso para saber mais sobre HTTPS.

Você pode pensar no HTTPS como uma versão “segura” do HTTP, onde os dados enviados entre o navegador e o servidor são criptografados para fins de segurança. A maior parte do tráfego web moderno usa HTTPS.

Protocolos como TCP/IP e HTTP fizeram da World Wide Web e da internet como um todo a ferramenta essencial da qual a maioria de nós depende todos os dias. Graças a eles, os dispositivos podem se comunicar corretamente em uma rede e trocar dados ao redor do mundo.
