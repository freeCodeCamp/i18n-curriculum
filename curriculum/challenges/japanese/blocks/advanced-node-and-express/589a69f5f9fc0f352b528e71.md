# --description--

GitHub認証の設定の最後の部分は、戦略自体を作成することです。`passport-github@~1.1.0`はすでに依存関係として追加されているので、`auth.js`ファイル内で`GitHubStrategy`としてこのようにrequireしてください：`const GitHubStrategy = require('passport-github').Strategy;`。環境変数を使うために`dotenv`をrequireして設定するのも忘れないでください。

GitHub戦略を設定するには、Passportにインスタンス化された`GitHubStrategy`を使うよう指示する必要があります。これは2つの引数を受け取ります：オブジェクト（`clientID`、`clientSecret`、`callbackURL`を含む）と、ユーザーが認証に成功したときに呼び出される関数です。この関数はユーザーが新規かどうかを判定し、ユーザーのデータベースオブジェクトに最初に保存するフィールドを決めます。これは多くの戦略で共通していますが、特定の戦略のGitHub READMEに記載されているように、より多くの情報が必要な場合もあります。例えば、Googleは*スコープ*も必要で、これはどのような情報をリクエストしているかを決め、ユーザーにそのアクセスを承認させます。

今回実装している戦略は、GitHubアカウントとOAuth 2.0トークンを使ってユーザーを認証します。アプリケーション作成時に取得したクライアントIDとシークレットは、戦略作成時のオプションとして渡されます。戦略はまた`verify`コールバックを必要とし、これはアクセストークンとオプションの更新トークン、さらに認証されたユーザーのGitHubプロフィールを含む`profile`を受け取ります。`verify`コールバックは認証を完了するためにユーザーを渡して`cb`を呼び出さなければなりません。

この時点での新しい戦略は次のようになっているはずです：

```js
passport.use(new GitHubStrategy({
  clientID: process.env.GITHUB_CLIENT_ID,
  clientSecret: process.env.GITHUB_CLIENT_SECRET,
  callbackURL: /*INSERT CALLBACK URL ENTERED INTO GITHUB HERE*/
},
  function(accessToken, refreshToken, profile, cb) {
    console.log(profile);
    //Database logic here with callback containing your user object
  }
));
```

まだ認証は成功しませんし、データベースのロジックとコールバックがないと実際にエラーがスローされますが、試してみるとGitHubプロフィールがコンソールにログされるはずです！

正しくできたと思ったらページを提出してください。エラーが出る場合は、<a href="https://forum.freecodecamp.org/t/advanced-node-and-express/567135#implementation-of-social-authentication-ii-4" target="_blank" rel="noopener noreferrer nofollow">ここまで完成したプロジェクトを確認できます</a>。
