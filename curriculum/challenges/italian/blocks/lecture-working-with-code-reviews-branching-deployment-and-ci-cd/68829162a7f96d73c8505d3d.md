# --description--

Se hai frequentato sviluppatori esperti, sia in una comunità online sia di persona, probabilmente hai sentito parlare dei "temuti conflitti di merge". Ma cosa sono esattamente?

Quando usi una strategia di branching corretta come abbiamo visto nelle lezioni precedenti, potresti lavorare su due funzionalità diverse su due branch diversi contemporaneamente. Oppure potresti lavorare su qualcosa mentre un collega lavora su un altro branch. In ogni caso, ci sono due modifiche indipendenti in corso. Ma a volte entrambi i branch modificano lo stesso file. E se l'altro branch viene unito in `main` per primo, le modifiche del tuo branch risultano indietro.

Nella maggior parte dei casi, questo si risolve unendo `main` di nuovo nel tuo branch di funzionalità. Ma quando le modifiche dell'altro branch confliggono con quelle del tuo, Git non sa quali modifiche vuoi mantenere durante il merge. Questo si chiama "conflitto di merge".

Ma come si risolve? Per conflitti più piccoli e semplici, GitHub ti offre un’interfaccia web per risolverli. Un pulsante "Resolve conflicts" apparirà in fondo a una pull request se ci sono conflitti. Se ci clicchi, vedrai il file con i conflitti che include il tuo codice e il codice in conflitto. Git aggiungerà dei "marcatori di conflitto di merge" intorno al tuo codice e a quello in conflitto per aiutarti a identificare il problema. Assomiglia a questo:

```sh
1 <<<<<<< feat/conflict
2 Here's some code from my first PR :)
3 =======
4 Here's some code that was merged to main that conflicts with the code on my PR
5 >>>>>>> main
```

Il marcatore di apertura è `<<<<<<< feat/conflict`, e indica l’inizio delle modifiche che il nostro branch corrente sta facendo. Poi vediamo `Here's some code from my first PR :)`. Dopo c’è il marcatore `=======`, che indica la fine delle nostre modifiche e l’inizio delle modifiche del branch `main`. Possiamo vedere che le modifiche fatte su `main` sono la riga `Here's some code that was merged to main that conflicts with the code on my PR`. E dopo c’è il marcatore di chiusura `>>>>>> main` che mostra la fine dei conflitti.

Ma come lo risolviamo? Devi decidere quali modifiche vuoi mantenere come parte del conflitto di merge. Se vuoi mantenere entrambe le modifiche, puoi rimuovere solo i marcatori di conflitto lasciando le modifiche invariate. In alternativa, se vuoi mantenere solo le modifiche di un branch, eliminerai i marcatori di conflitto e le modifiche dell’altro branch (quelle che non vuoi mantenere). Infine, se non ti piace nessuna delle modifiche, puoi rimuovere tutto dal marcatore di apertura a quello di chiusura (inclusi).

Rimuoviamo le modifiche del branch `main`, ma manteniamo le nostre. Ecco cosa otteniamo ora:

```sh
1 Here's some code from my first PR :)
```

Clicca il pulsante "Mark as resolved", che dice a GitHub che sei soddisfatto della risoluzione. Se ci fossero altri file, GitHub passerebbe al prossimo. Ma dato che questo è il nostro unico file in conflitto, vedrai apparire un pulsante "Commit merge". Cliccalo e GitHub creerà un commit di merge per te. Verrai riportato alla pull request, dove vedrai il nuovo commit di merge e i conflitti risolti. E siamo a posto!

Ma se non puoi modificare i conflitti direttamente su GitHub perché sono troppo complessi?

Se il tuo branch ha pochi commit, o solo un paio di commit modificano quel file, potresti riuscire a unire `main` nel tuo branch:

```sh
git checkout feat/conflict
git fetch origin
git merge origin/main
```

Mentre Git esegue questo merge, dovrai risolvere manualmente i conflitti. Git ti notificherà quali file devono essere sistemati. Quando apriamo i file con conflitti nel nostro editor, vediamo gli stessi marcatori di conflitto, con alcune aggiunte utili all’interfaccia grazie a VSCode. Procedi a risolvere i conflitti nel file allo stesso modo. Ma questa volta dovremo creare manualmente il commit. È come creare qualsiasi altro commit:

```sh
git add .
git commit -m "chore: resolve conflicts"
```

Dopo aver eseguito questi comandi, vediamo che Git ha accettato la nostra risoluzione.

A volte, quando hai molti commit o conflitti, un rebase può essere utile. Invece di unire le modifiche, un rebase prende tutti i commit del tuo branch e li riapplica in modo che vengano dopo le modifiche più recenti su `main`. In termini Git, questo "resetta" la base del tuo branch all’ultimo commit su `main`.

Per sperimentare questo, ci serve un nuovo branch:

```sh
git checkout main
git pull
git checkout -b feat/rebase
```

Generiamo una serie di commit per il nostro branch, così possiamo esercitarci con il rebase:

```sh
echo "change one" > README.md
git add .
git commit -m "change one"
echo "change two" > README.md
git add .
git commit -m "change two"
echo "change three" > README.md
git add .
git commit -m "change three"
```

Ora facciamo un’altra modifica su `main` per creare un conflitto:

```sh
git checkout main
echo "change four" > README.md
git add .
git commit -m "change four"
```

E passiamo al nostro branch di rebase:

```sh
git checkout feat/rebase
```

Poi esegui `git rebase -i main`. Questo farà il rebase del nostro branch sullo stato attuale del branch `main`. La flag `-i` dice a Git di eseguire un rebase in modalità interattiva. Vedrai i tuoi commit con la parola `pick` accanto. `pick` dice a Git di usare il commit così com’è per il rebase. Salva e chiudi il file, e riceverai un messaggio di errore che indica che ci sono conflitti.

Il nostro commit `change one` confligge con il commit `change four`, e dobbiamo risolverlo. Apri il file nel tuo editor, e vedrai la stessa interfaccia che abbiamo avuto con i conflitti prima. Manteniamo le nostre modifiche `change one`. Poi dovrai eseguire `git add .` per aggiungere le modifiche, e `git rebase --continue` per dire a Git che siamo pronti a continuare il rebase. Poiché abbiamo risolto il conflitto nel primo commit, Git può gestire automaticamente il rebase degli altri commit.

Ora, come ultimo passaggio di pulizia, schiacciamo i nostri tre commit di modifica in un unico commit per una storia più pulita. Esegui di nuovo `git rebase -i main` per aprire l’editor. Lascia il primo comando come `pick` ma cambia i successivi due in `squash`. Potresti incontrare di nuovo lo stesso conflitto. Se succede, risolvilo come prima. Una volta risolto e continuato il rebase (o subito se non c’era conflitto), vedrai apparire un nuovo editor per il messaggio di commit. Questo sarà il tuo nuovo commit, contenente le modifiche dei tre commit che abbiamo schiacciato. Chiamiamolo `change three complete`. Salva e chiudi l’editor, e Git completerà il rebase.

Se controlliamo la cronologia dei commit ora, vediamo che abbiamo un singolo commit `change three complete`, e che è correttamente basato sul commit `change four`. Questo significa che hai completato con successo un rebase con commit schiacciati! Ora hai gli strumenti per diventare un campione nella risoluzione dei conflitti di merge!

## --text--

Qual è la differenza principale tra il merging e il rebasing quando si gestiscono conflitti tra branch?

## --answers--

Il merging crea un nuovo commit, mentre il rebasing riscrive la storia per mettere i tuoi commit dopo quelli dell’altro branch.

---

Il rebasing è più veloce del merging.
