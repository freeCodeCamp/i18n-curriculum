## Authentifizierung vs. Autorisierung

**Authentifizierung** ist der Prozess, die Identität eines Benutzers zu überprüfen. Sie beantwortet die Frage *„Ist diese Person wirklich die, die sie vorgibt zu sein?“* Wenn sich ein Benutzer anmeldet, vergleicht der Backend-Server seine Zugangsdaten mit den bei der Registrierung gespeicherten Daten.

Backend-Entwickler verwenden drei Hauptkategorien von Authentifizierungsfaktoren:

- **Etwas, das Sie wissen:** Ein Passwort, eine PIN oder die Antwort auf eine Sicherheitsfrage.
- **Etwas, das Sie haben:** Ein SMS-Code auf einem Smartphone oder ein temporärer Token aus einer Authentifizierungs-App.
- **Etwas, das Sie sind:** Biometrische Daten wie ein Fingerabdruck oder eine Gesichtserkennung.

**Autorisierung** übernimmt nach der Authentifizierung und entscheidet, was der authentifizierte Benutzer tun darf. Sie beantwortet die Frage *„Welche Berechtigungen hat diese Identität?“* Übliche Rollen sind:

- **Standardbenutzer:** Kann das eigene Profil bearbeiten, aber nicht die privaten Daten anderer Benutzer einsehen.
- **Redakteur:** Kann Inhalte schreiben und veröffentlichen, aber nicht den Code löschen.
- **Administrator:** Hat volle Berechtigung, Datenbanken zu ändern, Benutzer zu löschen und systemweite Einstellungen zu ändern.

**Authentifizierung muss immer zuerst erfolgen, dann die Autorisierung.** Sie können nicht entscheiden, welche Berechtigungen Sie vergeben, bevor Sie wissen, wer der Benutzer ist.

## JSON Web Tokens (JWTs)

Ein JWT ist eine digitale Schlüsselkarte, die der Server einem Benutzer nach erfolgreicher Anmeldung ausstellt. Der Benutzer legt sie bei jeder weiteren Anfrage vor, um seine Identität zu beweisen, ohne das Passwort erneut eingeben zu müssen.

Der vierstufige JWT-Ablauf:

1. **Anmeldung:** Der Benutzer sendet Zugangsdaten an den Server.
2. **Ausgabe:** Der Server überprüft die Zugangsdaten, erstellt ein JWT und sendet es zurück.
3. **Speicherung:** Der Client speichert das JWT, typischerweise in einem Browser-Cookie oder im lokalen Speicher.
4. **Anfrage:** Bei jeder zukünftigen Anfrage an eine geschützte Route sendet der Client das JWT mit, und der Server überprüft es.

Ein JWT besteht aus drei Teilen, die durch zwei Punkte getrennt sind:

- **Kopfzeile:** Kodiertes JSON, das den Token-Typ (JWT) und den verwendeten Hash-Algorithmus angibt, z. B. HMAC SHA256.
- **Nutzlast:** Kodiertes JSON mit Angaben über den Benutzer, wie Benutzer-ID, Benutzername oder Rolle. Kopfzeile und Nutzlast sind Base64-kodiert, nicht verschlüsselt, speichern Sie hier also niemals sensible Daten wie Passwörter.
- **Signatur:** Entsteht durch das Hashen der kodierten Kopfzeile und Nutzlast zusammen mit einem geheimen Schlüssel, der nur dem Server bekannt ist. Wird die Nutzlast manipuliert, stimmt die Signatur nicht mehr, und der Server lehnt die Anfrage mit einer `401 Unauthorized`-Antwort ab.

JWTs ermöglichen **zustandslose Authentifizierung**. Der Server muss keine Sitzungsdaten in einer Datenbank speichern, da die Signatur beweist, dass er den Token erstellt hat.

## Cross-Site Request Forgery (CSRF) – Angriff durch Webseitenübergreifende Anfragefälschung

CSRF ist ein Angriff, bei dem ein authentifizierter Benutzer unbewusst unerwünschte Aktionen auf einer Webanwendung ausführt. Eine bösartige Website täuscht den Browser des Opfers, eine gefälschte Anfrage an einen Backend-Server zu senden, indem sie ausnutzt, dass Browser automatisch Cookies an jede Anfrage an den Zielserver anhängen.

**Beispiel für einen Angriffsablauf:**

1. Sie melden sich beispielsweise bei `trusted-bank.com` an. Die Bank speichert ein Sitzungstoken in Ihrem Browser-Cookie.
2. Ohne sich abzumelden, besuchen Sie `evil-hacker-site.com`.
3. Die bösartige Seite sendet stillschweigend eine Anfrage an `trusted-bank.com/transfer?amount=1000&to=hacker`.
4. Ihr Browser hängt automatisch Ihr Bank-Sitzungscookie an, sodass der Server die Anfrage verarbeitet.

Die bösartige Seite kann Ihr Cookie nicht auslesen. Sie nutzt lediglich aus, dass Ihr Browser automatisch Cookies an jede Anfrage an den Zielserver anhängt.

**Abwehrmaßnahmen gegen CSRF:**

- **`SameSite=Strict`:** Der Browser sendet das Cookie niemals, wenn die Anfrage von einer Drittanbieter-Website stammt.
- **`SameSite=Lax`:** Erlaubt Cookies bei sicherer Navigation (Klicken auf einen Link), blockiert sie jedoch bei Formularübermittlungen und API-Anfragen von Drittseiten.
- **CSRF-Tokens:** Wenn ein Benutzer eine Seite lädt, generiert der Server ein zufälliges, einzigartiges und unvorhersehbares Token und hängt es an das Frontend-Formular oder die Sitzung an. Der Client muss es in den Anfrage-Headern oder Formulardaten mitsenden. Da eine bösartige Seite keinen Zugriff auf das Frontend hat, um das Token zu kopieren, schlagen gefälschte Anfragen bei der Validierung fehl.

## Passport.js

Passport.js ist eine Authentifizierungs-Middleware für Node.js, die standardisiert, wie Sie Anfragen mit modularen Plugins namens **Strategien** authentifizieren. Mit über 500 verfügbaren Strategien können Sie jede Authentifizierungsmethode hinzufügen, indem Sie das entsprechende Paket installieren:

- `passport-local`: Benutzername und Passwort gegen Ihre eigene Datenbank.
- `passport-jwt`: Validiert JSON Web Tokens auf geschützten API-Routen.
- `passport-google-oauth20`: Anmeldung mit Google-Konten.
- `passport-github2`: Anmeldung mit GitHub-Konten.

Installieren Sie das Kernpaket und eine Strategie:

```bash
npm install passport passport-local
```

Binden Sie Passport in Ihre Express-Anwendung ein, bevor Sie Routen definieren:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database look-up and password verification logic
}));
```

Schützen Sie eine Login-Route mit Passport als Middleware:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

Nach erfolgreicher Authentifizierung übernimmt Passport automatisch die **Sitzungsverwaltung** (speichert Benutzerdaten im Sitzungscookie) und die **Anfrage-Dekoration** (hängt das Benutzerobjekt an `req.user` für die Verwendung in nachfolgenden Routen-Handlern).

## Helmet.js

Helmet.js ist eine Sicherheits-Middleware-Sammlung für Node.js, die Ihre Anwendung schützt, indem sie automatisch HTTP-Antwort-Header setzt, ändert oder entfernt, die Express standardmäßig bereitstellt.

Installieren und wenden Sie es in einem Schritt an:

```bash
npm install helmet
```

```js
const helmet = require('helmet');
app.use(helmet());
```

Wenn Sie `app.use(helmet())` an den Anfang Ihres Middleware-Stacks setzen, gelten diese Schutzmaßnahmen für jede Antwort:

- **X-Powered-By entfernt:** Express gibt standardmäßig mit `X-Powered-By: Express` seine Präsenz preis. Helmet entfernt diesen Header, um Angreifer daran zu hindern, versionsspezifische Schwachstellen anzugreifen.
- **Content-Security-Policy gesetzt:** Beschränkt, von welchen Quellen der Browser Skripte, Bilder und Styles laden darf, und schützt so vor Cross-Site-Scripting (XSS)-Angriffen.
- **X-Frame-Options auf SAMEORIGIN gesetzt:** Verhindert, dass Ihre Seite in einem externen iframe eingebettet wird, und blockiert so Clickjacking-Angriffe.
- **Strict-Transport-Security gesetzt:** Erzwingt, dass Browser nur über HTTPS mit Ihrem Server kommunizieren, und schützt Daten vor Abfangen in unsicheren Netzwerken.
