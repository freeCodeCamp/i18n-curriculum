# --description--

アサーションとユニットテストは前のレッスンで初めて紹介しました。復習として、ユニットテストとは、プログラム全体の最終出力をテストするのではなく、個々の関数やコンポーネントの出力を独立してチェックすることです。アサーションはコードが期待通りに動作しているかをテストするために使われます。

このレッスンでは、ユニットテストのアサーションを詳しく見ていき、いくつかのベストプラクティスを探ります。

よく使われるアサーションライブラリの一つにChaiがあります。これはfreeCodeCampのコードベース内でも使われているライブラリです。ワークショップやラボを進めるとき、コードに対して自動テストスイートが実行され、コードが期待通りに動作しているかを確認します。

Full-Stack Developer Curriculumからのいくつかの例を見てみましょう。

Build a Calculatorワークショップのステップ1では、`addTwoAndSeven`という関数のテストをしています。

```js
assert.isFunction(addTwoAndSeven);
```

このアサーションは、提供されたコードに`addTwoAndSeven`という名前の関数が含まれているかをチェックします。もしユーザーが配列や文字列、または関数以外の何かを`addTwoAndSeven`に代入すると、テストは失敗します。

ワークショップの後半にある別のアサーションも見てみましょう。

```js
assert.equal(addThreeAndFour(), 7);
```

このアサーションは、`addThreeAndFour`関数の戻り値が数値の`7`と等しいかをチェックします。戻り値が`7`と等しくなければ、テストは失敗します。

Chaiには他にも、値がブール値かどうか、オブジェクトに特定のプロパティ名があるか、値が`null`かどうかをチェックするアサーションがたくさんあります。その他の一般的なJavaScriptのアサーションライブラリにはshould.jsやexpect.jsがあります。

どのアサーションライブラリを使う場合でも、いくつかのベストプラクティスを知っておくことが重要です。まず、明確なアサーションメッセージを書くことです。明確で意味のあるメッセージは、その意図を説明します。そうでないと、どのテストが失敗しているのか見つけにくくなります。

もう一つのベストプラクティスは、説明的な失敗メッセージを書くことです。説明的な失敗メッセージは、どの条件が真であるかを示します。これにより、なぜ特定のアサーションが失敗しているのかを理解しやすくなります。

コード学習を続ける中で、テストに慣れ親しみ、個人のプロジェクトに対してテストを書くことを始めることが大切です。これにより、最初の開発者の仕事に備えるのに大いに役立ちます。
