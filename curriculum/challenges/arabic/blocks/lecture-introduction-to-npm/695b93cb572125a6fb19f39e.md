# --description--

عندما تثبت التبعيات باستخدام npm، خاصة في مشروع قائم، ينظر npm إلى ملف `package.json` الخاص بك ليرى الحزم التي يحتاجها مشروعك.

كل تبعية في `package.json` تتضمن رقم إصدار مثل هذا:

```bash
"lodash": "^4.17.21"
```

كما تعلمت سابقًا، يسمح **مؤشر الإدراج (^)** لـ npm بتثبيت إصدارات فرعية وتصحيحية أحدث مثل `4.18.0` أو `4.17.22`. هذه المرونة مفيدة، لكنها قد تؤدي أيضًا إلى مشاكل.

إذا قام مطوّران بتشغيل `npm install` في أوقات مختلفة، قد يثبت npm إصدارات مختلفة قليلاً من نفس الحزمة. حتى الاختلافات الصغيرة قد تسبب أخطاء برمجية أو سلوك غير متوقع.

يحل ملف `package-lock.json` هذه المشكلة. يُنشأ تلقائيًا في المرة الأولى التي تشغل فيها `npm install`، ويُثبت الإصدارات الدقيقة لكل تبعية تم تثبيتها — بما في ذلك كل التبعيات الفرعية، وهي التبعيات الخاصة بالتبعيات.

هذا يضمن أن كل من يعمل على المشروع يحصل على نفس إصدارات الحزم في كل مرة.

إليك ملف `package.json` بسيط مع `lodash` كتبعٍ:

```bash
{
  "name": "package-lock-json",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC",
  "type": "commonjs",
  "dependencies": {
    "lodash": "^4.17.21"
  }
}
```

وهنا القسم المطابق من `package-lock.json`:

```bash
"node_modules/lodash": {
  "version": "4.17.21",
  "resolved": "https://registry.npmjs.org/lodash/-/lodash-4.17.21.tgz",
  "integrity": "sha512-v2kDEe57..."
}
```

لاحظ الفرق:

* في `package.json`، تُدرج `lodash` كـ `^4.17.21` — قاعدة مرنة.
    
* في `package-lock.json`، تُثبت `lodash` على الإصدار `4.17.21`.
    

بغض النظر عن وقت تشغيل أي شخص لـ `npm install`، سيحصل على هذا الإصدار بالضبط.

بهذه الطريقة، بغض النظر عن وقت تشغيل أي شخص لـ `npm install`، سيحصل على نفس إصدار Lodash في كل مرة.

لاحظ أن هذا ينطبق أيضًا على التبعيات الفرعية، وهي تبعيات أي تبعيات تستخدمها في مشروعك.

لاحظ أيضًا أنه، بينما تستخدم `dependencies` في `node_modules/express` أحرف المؤشر (caret) للنطاقات، إذا وجدت الإدخال لكل تبعية فرعية مثل `node_modules/etags`، فهي مثبتة على إصدار محدد.

باختصار، يحافظ ملف `package-lock.json` على توافق الجميع ويضمن الاتساق عبر أجهزة المطورين، وخوادم التكامل المستمر، ومرحلة التشغيل، والبيئات الأخرى.

من المهم دائمًا الالتزام بملف `package-lock.json` في Git حتى يشارك فريقك نفس الإعداد. أيضًا، لا تعدل ملف `package-lock.json` يدويًا — سيقوم npm بتحديثه تلقائيًا كلما تغيرت التبعيات.
