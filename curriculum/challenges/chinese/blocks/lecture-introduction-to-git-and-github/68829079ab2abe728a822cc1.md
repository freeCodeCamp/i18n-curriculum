# --description--

在之前的课程中，你学习了如何在 GitHub 上创建一个新仓库。但如果你有一个现有的局部项目，想要将其添加到 GitHub 呢？在本课中，你将学习如何将现有的局部仓库推送到 GitHub，以及如何从远程仓库拉取更改。

首先在 GitHub 上新建一个不包含任何文件的仓库。正确完成后，你将被重定向到一个包含克隆你的仓库所需 URL 的页面。

现在导航到你的终端并选择你希望添加到 GitHub 的现有项目。在本课中，我选择了一个名为 `super-awesome-game` 的局部项目，里面包含以下文件：

```sh
jessicawilkins super-awesome-game >> ls
README.md  index.html script.js  styles.css
```

要使用 Git 为此项目设置版本控件，你需要在项目目录中运行 `git init` 命令。这将初始化一个空的 Git 仓库，以便 Git 可以开始跟踪此项目的更改。正确完成后，你应该在终端中看到类似的输出：

```sh
jessicawilkins super-awesome-game >> git init
Initialized empty Git repository in /Users/jessicawilkins/workspace/freeCodeCamp/super-awesome-game/.git/
```

当你为一个项目初始化一个空的 Git 仓库时，会添加一个新的 `.git` 隐藏目录。要查看该目录的添加情况，你可以运行 `ls -a` 命令，该命令会列出所有文件和目录，包括隐藏的：

```sh
jessicawilkins super-awesome-game >> ls -a
.          ..         .git       README.md  index.html script.js  styles.css
```

重要的是不要删除那个 `.git` 目录，否则你所有的 Git 历史记录将会丢失。

现在空的 Git 仓库已经初始化，你可以运行 `git status` 命令。该命令用于显示你工作目录的当前状态——你将在工作流程中频繁使用此命令。你应该看到类似如下的输出：

```sh
jessicawilkins super-awesome-game >> git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	README.md
	index.html
	script.js
	styles.css

nothing added to commit but untracked files present (use "git add" to track)
```

`On branch main` 意味着你当前在 `main` 分支。Git 中的分支是一个独立的工作区，你可以在其中进行更改。`main` 分支通常表现为真实应用中的主分支或生产分支。开发团队会为新特色和错误修复创建多个分支，然后将这些更改合并回 `main` 分支。你将在后续课程中学习更多关于该进程的内容。

`Untracked files` 意味着 Git 检测到当前未跟踪的文件。

一个文件可以处于五种状态：

1. “Untracked”：这意味着该文件是仓库中新建的，且 Git 之前未“见过”它。
2. “Modified”：此 `file` 存在于上一个 `commit` 中，且有未提交的更改。
3. "Ignored"：你可能不会在 Git 中看到被忽略的文件，但你的 IDE 可能会有相应的指示器。被忽略的文件会被排除在 Git 操作之外，通常是因为它们包含在 `.gitignore` 文件中。
4. "Deleted"：已删除的文件是未跟踪文件的相反——它是之前存在但已被移除的文件。
5. “Renamed”：重命名的文件是指内容未更改，但文件的名称或位置被修改的文件。在某些情况下，即使文件有少量更改，也可以被视为重命名。

当你的仓库中有未提交的更改时，你需要先“暂存”它们。“暂存区”是 `.git` 目录中的一个特殊文件，包含将被包含在你下一次提交中的信息。提交实际上是你仓库当前状态的一个屏幕截图。

如果你想一次暂存一个文件，那么你可以使用以下命令：

```sh
git add name-of-file
```

如果你想暂存所有未暂存的更改，那么你可以使用 `git add .`。句点（`.`）是你当前所在目录的别名。一些开发者会警告不要使用 `git add .`，因为你可能会最终暂存你不打算暂存的文件。如果发生这种情况，你可以使用以下命令将它们“取消暂存”：

```sh
git reset name-of-file
```

不过，通常只要你在暂存文件之前使用 `git status` 监控哪些文件被添加，你就应该没问题。

一旦你暂存了你的文件，接着你需要提交它们。你可以通过运行 `git commit` 命令来提交你的更改。这将打开你在 Git 配置中设置的首选编辑器。编辑器打开后，你可以提供详细的更改消息。你也可以选择使用 `git commit -m` 命令来提供较短的消息，示例如下：

```sh
git commit -m "short message goes here"
```

如果你打算使用 `-m` 标记，你仍然需要提供一个描述性消息以为更改提供上下文。以下是一些你可能使用的良好消息示例：

```sh
git commit -m "feat(api): implement JWT-based authentication"
git commit -m "fix(auth): correct token expiration time"
git commit -m "refactor: simplify login flow logic"
```

`feat(api)` 和 `fix(auth)` 遵循 Conventional Commits 风格，用于提供该提交中更改类型的上下文。

在你提交你的工作后，再次运行 `git status`，你应该在终端中看到以下输出：

```sh
On branch main
nothing to commit, working tree clean
```

要查看此项目的所有先前提交，你可以运行 `git log` 命令。此命令将列出所有先前的提交，并提供有用的信息，如作者、提交日期、提交消息和提交哈希。提交哈希是一个长字串，作为提交的唯一标识符。

完成提交你的工作后，你现在可以将更改推送到 GitHub。你首先需要设置到远程仓库的远程连接。你在上一课中学习过这个内容，但这里用 SSH 提醒你一下：

```sh
git remote add origin git@github.com:your-github-username/name-of-repo.git
```

最后一步是使用此命令将你的更改推送到 GitHub：

```sh
git push -u origin main
```

`-u` 标记是 `--set-upstream` 的简写。它将你的局部的 `main` 分支连接到 `origin` 上的远程 `main` 分支。你每个新建分支只需使用一次。之后，你可以直接运行 `git push`，无需指定分支名称。

完成正确后，你应该在终端看到类似的结果：

```sh
Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Delta compression using up to 8 threads
Compressing objects: 100% (2/2), done.
Writing objects: 100% (3/3), 253 bytes | 253.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0 (from 0)
To https://github.com/jdwilkin4/super-awesome-game.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

当你刷新页面时，你还应该在你的 GitHub 仓库页面上看到新建的更改。

如果你正在与其他人协作一个项目，并且需要从仓库拉取新的更改，你可以使用 `git pull` 命令。你将在以后的课程中学习更多关于团队合作和协作的内容。
