# --description--

객체지향 프로그래밍, 즉 OOP는 개발자가 코드 내 모든 것을 실제 세계의 객체처럼 다루는 프로그래밍 스타일입니다.

클래스는 객체를 생성하기 위한 청사진과 같습니다. 클래스에서 생성된 모든 객체는 데이터를 정의하는 속성과 객체의 동작을 정의하는 메서드를 가집니다.

이전 수업에서 클래스를 만드는 방법을 배웠습니다. 구문을 다시 확인해 보세요:

```py
class ClassName:
   def __init__(self, parameters):
       attribute = value

   def method_name(self):
       # method logic
```

다음은 `__init__` 특수 메서드를 사용해 클래스에서 객체가 생성될 때마다 brand와 color 속성을 초기화하는 클래스 예시입니다:

```py
class Car:
   def __init__(self, brand, color):
       self.brand = brand
       self.color = color

# create two objects from the Car class
car1 = Car('Toyota', 'red')
car2 = Car('Lambo', 'green')

print('Car 1 Brand:', car1.brand) # Car 1 Brand: Toyota
print('Car 1 Color:', car1.color) # Car 1 Color: red

print('Car 2 Brand:', car2.brand) # Car 2 Brand: Lambo
print('Car 2 Color:', car2.color) # Car 2 Color: green
```

객체지향 프로그래밍에는 코드를 효과적으로 조직하고 관리하는 데 도움이 되는 네 가지 핵심 원칙이 있습니다. 그것은 캡슐화, 상속, 다형성, 추상화입니다.

이 수업의 나머지 부분은 캡슐화가 어떻게 작동하는지에 집중합니다.

캡슐화는 객체의 속성과 메서드를 하나의 단위인 클래스 안에 묶는 것입니다.

캡슐화를 통해 객체의 내부 상태를 단순한 공개 메서드와 속성 집합 뒤에 숨길 수 있습니다. 이 공개 메서드와 속성은 문처럼 작동합니다. 그 문 뒤에는 데이터 변경 방식과 누가 볼 수 있는지를 제어하는 비공개 속성과 메서드가 있습니다.

예를 들어 지갑 잔액을 추적하고 싶다고 가정해 보세요. 지갑에 돈을 입금하거나 출금할 수 있게 하되, 아무도 잔액을 직접 조작할 수 없어야 합니다.

이 경우 `deposit()`과 `withdraw()`를 공개 메서드로 만들고, 잔액은 `_balance` 속성 아래에 숨길 수 있습니다:

```py
class Wallet:
   def __init__(self, balance):
       self._balance = balance # For internal use by convention

   def deposit(self, amount):
       if amount > 0:
           self._balance += amount # Add to the balance safely

   def withdraw(self, amount):
       if 0 < amount <= self._balance:
           self._balance -= amount # Remove from the balance safely
```

관례상, 속성과 메서드 이름 앞에 단일 밑줄을 붙이면 내부 용도로만 사용한다는 의미입니다. 캡슐화 원칙에 어긋나므로 클래스 외부에서 직접 접근해서는 안 되며, 그렇지 않으면 버그가 발생할 수 있습니다.

단일 밑줄 접두사는 단지 관례일 뿐이지만, 속성과 메서드 이름 앞에 이중 밑줄을 붙이면 클래스 외부에서 접근하는 것을 효과적으로 막아 해당 속성과 메서드를 비공개로 만듭니다.

```py
class Wallet:
   def __init__(self, balance):
       self.__balance = balance # Private attribute

   def deposit(self, amount):
       if amount > 0:
           self.__balance += amount # Add to the balance safely

   def withdraw(self, amount):
       if 0 < amount <= self.__balance:
           self.__balance -= amount # Remove from the balance safely

account = Wallet(500)
print(account.__balance) # AttributeError: 'Wallet' object has no attribute '__balance'
```

현재 `__balance` 값을 얻으려면 get_balance 메서드를 정의할 수 있습니다. 예를 들어:

```py
class Wallet:
   def __init__(self, balance):
       self.__balance = balance

   def deposit(self, amount):
       if amount > 0:
           self.__balance += amount

   def withdraw(self, amount):
       if 0 < amount <= self.__balance:
           self.__balance -= amount
  
   def get_balance(self):
       return self.__balance


acct_one = Wallet(100)
acct_one.deposit(50)
print(acct_one.get_balance()) # 150

acct_two = Wallet(450)
acct_two.withdraw(28)
print(acct_two.get_balance()) # 422

acct_two.deposit(150)
print(acct_two.get_balance()) # 572
```

또한 모든 입금 또는 출금 금액이 양수인지 확인하는 비공개 `__validate` 메서드를 정의할 수 있습니다:

```py
class Wallet:
   def __init__(self):
       self.__balance = 0

   def __validate(self, amount):
       if amount < 0:
           raise ValueError('Amount must be positive')

   def deposit(self, amount):
       self.__validate(amount)
       self.__balance += amount

   def withdraw(self, amount):
       self.__validate(amount)
       if amount > self.__balance:
           raise ValueError('Insufficient funds')
       self.__balance -= amount

   def get_balance(self):
       return self.__balance

acct_one = Wallet()
acct_one.deposit(3)
print(acct_one.get_balance()) # 3

acct_one.deposit(50)
print(acct_one.get_balance()) # 53

acct_one.deposit(-4)  # ValueError: Amount must be positive
acct_one.withdraw(-8) # ValueError: Amount must be positive
acct_one.withdraw(58) # ValueError: Insufficient funds
```

보시다시피 `__validate` 메서드는 비공개이며, `deposit()`과 `withdraw()` 공개 메서드 내에서 항상 금액이 유효한지 확인하기 위해 내부에서 실행됩니다.

앞으로의 수업에서 이중 밑줄 접두사가 붙은 속성이 어떻게 작동하는지 더 배우게 됩니다.

요약하자면, 캡슐화는 내부 데이터를 명확한 공개 메서드 뒤에 잠급니다. 이렇게 하면 클래스가 조작으로부터 안전해지고 유효성 검사를 한 곳에 집중시킬 수 있습니다. 외부 코드는 노출한 인터페이스만 건드리므로 코드를 자유롭게 업데이트하거나 확장할 수 있습니다.
