# --description--

Simular es el proceso de reemplazar datos reales con datos falsos que simulan el comportamiento de componentes reales. Esto permite a los evaluadores y desarrolladores probar componentes específicos en la aplicación sin depender de dependencias externas.

Un caso común de uso para simular datos sería al trabajar con llamadas a API. Imagina que tienes una llamada de obtención como esta en tu aplicación:

```js
export const postService = {
  async getPosts() {
    const response = await fetch("https://api.example.com/posts");
    return response.json();
  },
};
```

Si necesitas probar la función `postService`, podrías simular la respuesta de la API en lugar de hacer llamadas API continuas para obtener los datos. Obtener datos de una API lleva tiempo y depende de factores externos como la disponibilidad de red y las respuestas del servidor. Simular la respuesta de la API elimina esos factores externos y permite a los evaluadores trabajar con un conjunto de datos predecible y controlado.

Aquí tienes un ejemplo de respuesta de API simulada:

```json
[
  {
    "id": 1,
    "title": "Understanding Async/Await in JavaScript",
    "content": "Async/Await makes asynchronous code look and behave like synchronous code...",
    "author": "Jane Doe",
    "created_at": "2025-04-10T14:32:00Z"
  },
  {
    "id": 2,
    "title": "10 Tips for Writing Clean Code",
    "content": "Clean code is not just about code formatting. It involves naming, architecture...",
    "author": "John Smith",
    "created_at": "2025-04-08T09:21:00Z"
  },
  {
    "id": 3,
    "title": "Exploring React 19 Features",
    "content": "React 19 comes with exciting features like new hooks and better performance...",
    "author": "Alex Lee",
    "created_at": "2025-04-07T18:47:00Z"
  }
]
```

Ahora que entendemos qué es la simulación, comparemos con la sustitución. Los sustitutos son objetos que devuelven respuestas predefinidas o datos simulados para un comportamiento esperado en una aplicación. Por ejemplo, puedes simular el comportamiento para una conexión de base de datos en tus pruebas sin tener que depender de una conexión de base de datos real.

Aquí tienes un ejemplo de un sustituto en Jest:

```js
import { postService } from "./postService";
import { db } from "./db";

jest.mock("./db", () => ({
  db: {
    fetchPosts: jest.fn(),
  },
}));

test("should return stubbed posts", async () => {
  const fakePosts = [
    { id: 1, title: "Stubbed Post", content: "This is a stubbed post." },
  ];

  db.fetchPosts.mockResolvedValue(fakePosts);

  const posts = await postService.getPosts();

  expect(posts).toEqual(fakePosts);
});
```

`db.fetchPosts.mockResolvedValue(fakePosts);` es el sustituto en este ejemplo.

El último componente a discutir son los falsos. Los falsos son versiones simplificadas de componentes reales sin la complejidad o efectos secundarios de lo real. Por ejemplo, puedes simular una base de datos almacenando los datos en memoria en lugar de interactuar con la base de datos real. Esto te permitirá imitar operaciones de bases de datos en memoria, lo que será mucho más rápido que lidiar con la base de datos real. Otro ejemplo común sería usar falsos al trabajar con sistemas de archivos. Puedes crear un sistema de archivos falso en lugar del real para realizar pruebas.

A medida que continúas construyendo aplicaciones de software más complejas, considera usar simulaciones, sustitutos o falsos donde sea apropiado en tus pruebas.
