# --description--

你是否曾想過如何讓你的 Express 應用程式在高負載或關閉時保持穩定且適合生產環境？

這就是健康檢查和優雅關閉的用武之地。這兩種樣式幫助你保持應用程式的可用性，並確保它能乾淨地退出，不會丟失請求或損壞資料。

讓我們來拆解說明。

首先，什麼是 Express 中的「健康檢查」？

健康檢查是一個簡單的路由，用來告訴你應用程式是否正在運作。

舉例來說：

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

像是負載平衡器或容器編排工具——想像 Kubernetes 或 AWS Elastic Beanstalk——可以透過這個路由來判斷你的應用程式是否健康。

如果路由回傳 200 狀態碼，代表你的服務被視為健康。提醒你，HTTP 200 狀態碼表示一切正常！

如果沒有回應，編排工具就會自動重新啟動服務。

那麼什麼是「優雅關閉」？

優雅關閉是指當你的應用程式即將停止時發生的事情，比如伺服器重新啟動或部署時。

你不會立刻切斷連線，而是給應用程式時間完成正在進行的工作。

在 Express 中，你可以監聽 `SIGTERM`，這是一個系統信號，當處理程序應該終止時會發送。

範例如下：

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

這告訴 Express 停止接受新的連線，並等待現有連線完成後再退出。

你也應該處理 `SIGINT`，這是你在本地開發時按 Ctrl+C 停止應用程式時發送的信號。

雖然 `SIGTERM` 通常由像 Kubernetes 這樣的編排工具在生產環境中使用，`SIGINT` 則涵蓋日常開發者的工作流程。如果不處理它，按 Ctrl+C 會立即終止處理程序，跳過你設定的任何清理工作。

因為這兩個信號都應該觸發相同的優雅關閉行為，開發者通常會同時監聽這兩個信號。

現在，讓我們把它們整合起來。

想像你的應用程式正在容器中執行。Kubernetes 發送 `SIGTERM` 來關閉它以進行更新。

有了基本的優雅關閉，你的應用程式會停止接受新連線，並完成目前的請求。所有工作完成後，它會斷開與資料庫的連線並乾淨地退出。沒有資料遺失，也沒有狀態損壞。

但有個問題——負載平衡器不知道你的應用程式正在關閉，所以它可能會繼續將新流量導向你的應用程式。這時候下面的額外樣式就派上用場了。

現在，我有個小驚喜要給你。我們將看看「在關閉前標記應用程式為不健康」。

在關閉之前，你可以更新健康檢查路由回傳 503 狀態碼。這樣負載平衡器就會看到不健康的回應，並在關閉期間停止將新流量導向你的應用程式，如上面程式碼所示。

提醒：`503` 狀態碼代表「服務不可用」。它表示伺服器暫時無法處理請求，這正是優雅關閉期間發生的情況。

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
