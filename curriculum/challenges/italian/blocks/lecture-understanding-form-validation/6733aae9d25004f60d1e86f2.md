# --interactive--

In una lezione precedente, hai imparato come usare HTML per limitare i valori che gli utenti possono inviare nel tuo modulo. Ma a volte non basta. Se vuoi fare qualcosa di più complesso, come mostrare messaggi di errore personalizzati all’utente, dovrai usare JavaScript.

Alcuni elementi HTML, come gli elementi `textarea` e `input`, espongono un’API di validazione dei vincoli (Constraint Validation API). Questa API ti permette di verificare che il valore fornito dall’utente per quell’elemento superi qualsiasi validazione a livello HTML che hai scritto, come la lunghezza minima o il controllo del pattern.

Ma come si usa effettivamente? Supponiamo che tu voglia che i dipendenti di un’azienda inviino messaggi di feedback tramite un modulo come questo:

:::interactive_editor

```html
<link rel="stylesheet" href="styles.css" />
<form>
  <label>Enter your email: </label>
  <input required type="email" />

  <label>Enter your feedback: </label>
  <textarea required placeholder="Your feedback here..."></textarea>

  <button type="submit">Submit Feedback</button>
</form>
```

Stiamo usando l’input email `input` che include una validazione integrata per controllare validazioni di base come se l’input contiene il simbolo at (`@`).

Ma cosa succede se l’utente inserisce un indirizzo email come `example@email.com`? Questo supererebbe la validazione di base, ma vogliamo essere più specifici accettando solo email aziendali.

Qui possiamo usare l’attributo `pattern` per specificare che l’indirizzo email deve terminare con un dominio aziendale. Ecco come sarà l’esempio aggiornato:

Nell’esempio sopra, abbiamo selezionato il nostro input dal DOM e aggiunto un listener per l’evento input.

Sappiamo che `e.target` si riferisce all’elemento che ha scatenato l’evento. In questo caso, il nostro `input`. Ma cos’è il metodo `checkValidity()`?

Fa parte della Constraint Validation API. Il metodo `checkValidity()` restituisce `true` se l’elemento supera tutte le validazioni HTML (basate sui suoi attributi), e `false` se fallisce.

Quando proviamo con un input non valido, vediamo che `false` viene registrato nella console. Ora che sappiamo che l’input è non valido, segnaliamo l’invalidità:

const input = document.querySelector("input");

  if (!e.target.checkValidity()) {
    e.target.reportValidity();
  }
})

E come risultato, vedrai il messaggio di errore del browser "Please match the requested format."

Segnala lo stato di errore immediatamente, invece di aspettare che inviamo il modulo. Ma usa ancora il messaggio predefinito. Questo perché il metodo `reportValidity` dice solo al browser che l’input è non valido. Il browser sceglie ancora come mostrare il motivo dell’errore. Qui entra in gioco il metodo `setCustomValidity`.

```js
const input = document.querySelector("input");

input.addEventListener("input", (e) => {
  if (!e.target.checkValidity()) {
    e.target.setCustomValidity(
      "You must use a company email address that ends in @sampleCompany.com"
    );
  }
});
```

Questo metodo accetta un messaggio di errore personalizzato, che viene mostrato all’utente. Come risultato, vedrai il messaggio di errore personalizzato `You must use a company email address that ends in @sampleCompany.com`.

La proprietà `validity` è un’istanza dell’oggetto `ValidityState`. Ecco un esempio di come potrebbe apparire l’oggetto nel browser:

```js
ValidityState {
  badInput: false,
  customError: false,
  patternMismatch: true,
  rangeOverflow: false,
  rangeUnderflow: false,
  stepMismatch: false,
  tooLong: false,
  tooShort: false,
  typeMismatch: true,
  valueMissing: false,
  valid: true
}
```

Ci sono diverse proprietà utili che contengono un valore booleano di `true` o `false`.

Alcune di queste proprietà utili che puoi esplorare da solo sono la proprietà `valueMissing` che è `true` quando un campo di input obbligatorio viene lasciato vuoto. Oppure `patternMismatch` che è `true` se il valore non corrisponde al pattern specificato dall’espressione regolare.

Dopo questa lezione, ti incoraggio a sperimentare con gli esempi dati e a esplorare di più le diverse proprietà di validità.
