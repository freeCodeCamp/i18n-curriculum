## HTTP 回應狀態碼

- **狀態碼** 是簡短的數字訊息，用來告訴客戶端在發出請求後發生了什麼事（例如 200、404 或 500）。

### 1xx — 資訊性狀態碼

- **100 Continue** — 伺服器告訴客戶端繼續傳送剩餘的請求。
- **101 Switching Protocols** — 伺服器正在切換協定（常見於 WebSockets）。

### 2xx — 成功狀態碼

- **200 OK** — 請求成功。
- **201 Created** — 成功創建了新的資源。
- **204 No Content** — 請求成功但沒有回應主體（常用於 `DELETE`）。

### 3xx — 重新導向狀態碼

- **301 Moved Permanently** — 資源已永久移動到新 URL。
- **302 Found** — 資源暫時位於不同的 URL。
- **304 Not Modified** — 客戶端可以使用快取版本；內容未更動。

### 4xx — 客戶端錯誤狀態碼

- **400 Bad Request** — 請求格式錯誤或無效。
- **401 Unauthorized** — 需要驗證身份。
- **403 Forbidden** — 已驗證身份，但沒有權限。
- **404 Not Found** — 請求的資源不存在。

### 5xx — 伺服器錯誤狀態碼

- **500 Internal Server Error** — 一般性的伺服器端錯誤。
- **502 Bad Gateway** — 代理伺服器從另一個伺服器收到無效回應。
- **503 Service Unavailable** — 伺服器暫時過載或停機。

### 在 Express 中使用狀態碼

- `res.status(400).send("Bad Request")` — 傳送 400 並附帶純文字訊息。
- `res.status(200).json({ message: "Success" })` — 傳送 200 並附帶 JSON 主體。
- `res.status(500).send("Internal Server Error")` — 傳送 500 並附帶訊息。

## Express 中的錯誤處理

- **錯誤處理中介層** 有四個參數：`(err, req, res, next)`。四個參數的標記是告訴 Express 這是錯誤處理常式的訊號。
- 將錯誤處理中介層 **放在中介層堆疊的最後面**，在所有路由之後。

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Express 4 非同步錯誤處理

- Express 4 不會 **自動捕捉** 非同步錯誤。
- 將非同步邏輯包裝在 `try/catch` 中，並呼叫 `next(err)` 將錯誤轉送給錯誤處理常式。

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- 使用 `asyncHandler` 包裝器可以避免在每個路由中重複 `try/catch`：

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Express 5 非同步錯誤處理

- 在 Express 5 中，非同步錯誤和拒絕的 Promise 會 **自動轉送** 給錯誤處理常式 — 不需要手動 `next(err)`。

### 處理 404 錯誤

- 在路由之後但錯誤處理常式之前，新增一個捕捉所有路由的中介層來處理未匹配的路由：

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### 錯誤處理最佳實踐

- 永遠將錯誤處理中介層放在最後。
- 對 404 錯誤與其他錯誤分開處理。
- 在生產環境中，內部記錄錯誤細節 — 絕不將堆疊追蹤傳送給使用者。

## Express 中的偵錯與記錄

- **內建偵錯**：設定 `DEBUG` 環境變數以啟用 Express 內建的偵錯輸出。
  - `DEBUG=express:* node index.js` — 啟用所有 Express 偵錯訊息。
  - `DEBUG=express:router node index.js` — 只顯示與路由器相關的偵錯輸出。
  - PowerShell：`$env:DEBUG = "express:*"; node index.js`
  - CMD：`set DEBUG=express:* && node index.js`
- **`morgan`**：一個流行的中介層，用於記錄 HTTP 請求（方法、URL、狀態碼、回應時間）。

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- **偵錯** 追蹤 Express 如何處理請求和中介層，且不會干擾生產環境的日誌。
- **記錄** 紀錄請求過程中發生的事情 — 有助於發現錯誤和了解流量狀況。

## 健康檢查與優雅關閉

### 健康檢查

- **健康檢查路由** 用來回報應用程式是否正常運作。
- 負載平衡器和編排工具（例如 Kubernetes、AWS Elastic Beanstalk）會定期 ping 這個路由，以判斷服務是否健康。
- 回應 **200** 表示健康；若無回應則會自動重新啟動。

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### 優雅關閉

- **優雅關閉** 讓應用程式在停止前完成現有請求，而不是立即切斷連線。
- 監聽 `SIGTERM` 信號（當行程應該終止時發送），並呼叫 `server.close()`：

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- 監聽 `SIGINT` 信號（開發者在本地開發時按下 Ctrl+C 發送），並呼叫與 `SIGTERM` 相同的關閉邏輯。

### 在關閉前標記應用程式為不健康

- 在關閉期間，從健康檢查回傳 **503**，讓負載平衡器停止將新流量導向該應用程式。

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
