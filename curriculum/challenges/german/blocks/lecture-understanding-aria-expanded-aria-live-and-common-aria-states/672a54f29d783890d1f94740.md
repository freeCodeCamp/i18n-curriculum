# --interactive--

Semantische Formularsteuerelemente wie `input`, `select`, `textarea`, `button` und `fieldset` haben eingebaute Zustände, die an unterstützende Technologien übermittelt werden.

Zum Beispiel könnten Sie das `disabled`-Attribut verwenden, um eine Schaltfläche zu deaktivieren, oder das `checked`-Attribut, um anzuzeigen, dass ein Kontrollkästchen aktiviert ist.

Wenn Sie jedoch ein benutzerdefiniertes Steuerungselement erstellen, müssen Sie ARIA-Attribute verwenden, um den Zustand des Steuerungselements an unterstützende Technologien zu übermitteln.

In dieser Lektion besprechen wir einige häufig verwendete ARIA-Zustände, die Sie bei benutzerdefinierten Steuerungselementen verwenden können.

Der erste ARIA-Zustand, den wir besprechen, ist `aria-selected`. Dieser Zustand wird verwendet, um anzuzeigen, dass ein Element ausgewählt ist. Sie können diesen Zustand bei benutzerdefinierten Steuerungen wie einer Registerkartenoberfläche, einer Listbox oder einem Raster verwenden.

Hier ist ein Beispiel, wie Sie `aria-selected` bei einer benutzerdefinierten Registerkartensteuerung verwenden können:

:::interactive_editor

```html
<link rel="stylesheet" href="styles.css">
<div role="tablist">
  <button role="tab" aria-selected="true">Tab 1</button>
  <button role="tab" aria-selected="false">Tab 2</button>
  <button role="tab" aria-selected="false">Tab 3</button>
</div>
<script src="index.js"></script>
```

```css
[role="tablist"] {
  display: flex;
  border-bottom: 2px solid #ddd;
  gap: 0.25rem;
  font-family: system-ui, sans-serif;
}

[role="tab"] {
  appearance: none;
  border: none;
  background: none;
  padding: 0.5rem 1rem;
  cursor: pointer;
  font-size: 1rem;
  color: #444;
  border-radius: 4px 4px 0 0;
  transition: background-color 0.2s, color 0.2s;
}

[role="tab"]:hover {
  background-color: #f3f3f3;
}

[role="tab"][aria-selected="true"] {
  background-color: #fff;
  color: #0078d4;
  border: 2px solid #0078d4;
  border-bottom: 2px solid #fff; 
  font-weight: 600;
  position: relative;
  z-index: 1;
}

[role="tab"]:focus {
  outline: 2px solid #0078d4;
  outline-offset: 2px;
}

```

```js
document.addEventListener("click", (event) => {
  const clickedTab = event.target.closest('[role="tab"]');
  if (!clickedTab) return;

  const tablist = clickedTab.closest('[role="tablist"]');
  const tabs = tablist.querySelectorAll('[role="tab"]');

  tabs.forEach((tab) => {
    const isSelected = tab === clickedTab;
    tab.setAttribute("aria-selected", isSelected);
    tab.tabIndex = isSelected ? 0 : -1;
  });
});

document.addEventListener("keydown", (event) => {
  const activeTab = document.activeElement;
  if (activeTab.getAttribute("role") !== "tab") return;

  const tablist = activeTab.closest('[role="tablist"]');
  const tabs = Array.from(tablist.querySelectorAll('[role="tab"]'));
  const index = tabs.indexOf(activeTab);

  let newIndex = index;
  if (event.key === "ArrowRight") newIndex = (index + 1) % tabs.length;
  if (event.key === "ArrowLeft") newIndex = (index - 1 + tabs.length) % tabs.length;

  if (newIndex !== index) {
    tabs[newIndex].focus();
    tabs[newIndex].click();
  }
});

```

:::

Registerkarten werden verwendet, um mehrere Inhaltsbereiche auf begrenztem Raum anzuzeigen. Der `aria-selected`-Zustand wird verwendet, um anzuzeigen, welche Registerkarte aktuell ausgewählt ist.

Wenn der Benutzer eine Registerkarte auswählt, wird der `aria-selected`-Zustand der ausgewählten Registerkarte auf `true` gesetzt, und der `aria-selected`-Zustand der anderen Registerkarten wird auf `false` gesetzt.

Ein weiterer häufiger ARIA-Zustand ist `aria-disabled`. Dieser Zustand wird verwendet, um anzuzeigen, dass ein Element nur für Personen mit unterstützenden Technologien, wie Screenreadern, deaktiviert ist. Es ist wichtig zu beachten, dass `aria-disabled` das Element nicht tatsächlich deaktiviert. Es liegt an Ihnen als Entwickler, das Element so aussehen und sich so verhalten zu lassen, als wäre es deaktiviert. Dieses Attribut wird auch häufig bei nativen HTML-Elementen anstelle des `disabled`-Attributs verwendet. Welche Variante Sie wählen, hängt vom Kontext ab, in dem die Schaltfläche verwendet wird.

Hier ist ein Beispiel, wie Sie `aria-disabled` bei einer benutzerdefinierten Bearbeitungsschaltfläche verwenden können:

:::interactive_editor

```html
<link rel="stylesheet" href="styles.css">
<div role="button" tabindex="-1" aria-disabled="true">Edit</div>
```

```css
[role="button"] {
  display: inline-block;
  background-color: #0078d4;
  color: #fff;
  padding: 0.5rem 1rem;
  border-radius: 6px;
  font-family: system-ui, sans-serif;
  font-size: 1rem;
  text-align: center;
  cursor: pointer;
  user-select: none;
  transition: background-color 0.2s, transform 0.1s, opacity 0.2s;
}

[role="button"]:not([aria-disabled="true"]):hover {
  background-color: #005fa3;
}

[role="button"]:not([aria-disabled="true"]):focus {
  outline: 2px solid #005fa3;
  outline-offset: 2px;
}

[role="button"]:not([aria-disabled="true"]):active {
  transform: scale(0.97);
}

[role="button"][aria-disabled="true"] {
  opacity: 0.5;
  pointer-events: none;
  cursor: not-allowed;
  background-color: #b0b0b0;
  color: #f2f2f2;
}
```

:::

Das `aria-disabled`-Attribut wird verwendet, um Screenreader-Nutzern mitzuteilen, dass die Bearbeitungsschaltfläche deaktiviert ist und nicht bedient werden kann. Auch hier wird die Schaltfläche nicht tatsächlich deaktiviert. Wenn Sie `aria-disabled` verwenden, müssen Sie Styling und JavaScript hinzufügen, damit sich die Steuerung wie eine deaktivierte Schaltfläche verhält und so aussieht.

In den meisten Fällen werden Sie wahrscheinlich das native Button-Element verwenden, aber es gibt Situationen, in denen Sie eine benutzerdefinierte Steuerung benötigen. Deshalb ist es wichtig zu wissen, wie Sie den Zustand der Steuerung an unterstützende Technologien übermitteln.

Der nächste ARIA-Zustand, den wir besprechen, ist `aria-haspopup`. Dieser Zustand wird verwendet, um anzuzeigen, dass ein interaktives Element beim Aktivieren ein Popup-Element auslöst. Sie können das `aria-haspopup`-Attribut nur verwenden, wenn das Popup eine der folgenden Rollen hat: `menu`, `listbox`, `tree`, `grid` oder `dialog`. Der Wert von `aria-haspopup` muss entweder eine dieser Rollen oder `true` sein, was standardmäßig die Rolle `menu` annimmt.

Hier ist ein Beispiel für ein Dateieditor-Menü, das `aria-haspopup` verwendet:

:::interactive_editor

```html
<link rel="stylesheet" href="styles.css">
<button id="menubutton" aria-haspopup="menu" aria-controls="filemenu" aria-expanded="false">File</button>
<ul id="filemenu" role="menu" aria-labelledby="menubutton" hidden>
  <li role="menuitem" tabindex="-1">Open</li>
  <li role="menuitem" tabindex="-1">New</li>
  <li role="menuitem" tabindex="-1">Save</li>
  <li role="menuitem" tabindex="-1">Delete</li>
</ul>
```

```css

  border: none;

  border-radius: 4px;
  cursor: pointer;
  font-size: 14px;
  position: relative;
}

#menubutton:hover,
#menubutton:focus {
  background-color: #005ea2;
  outline: none;
}

#filemenu {
  list-style: none;
  padding: 4px 0;
  margin: 4px 0 0;
  border: 1px solid #ccc;
  border-radius: 4px;
  background-color: #fff;
  width: 160px;
  position: absolute;
  z-index: 1000;
  box-shadow: 0 4px 6px rgba(0,0,0,0.1);
}

#filemenu[hidden] {
  display: none;
}

#filemenu [role="menuitem"] {
  display: block;
  padding: 8px 12px;
  font-size: 14px;
  color: #333;
  cursor: pointer;
}

#filemenu [role="menuitem"]:hover,
#filemenu [role="menuitem"]:focus {
  background-color: #e5f1fb;
  outline: none;
}

#filemenu [role="menuitem"]:focus-visible {
  box-shadow: inset 0 0 0 2px #0078d4;
}

```

:::

Der `aria-haspopup`-Zustand wird verwendet, um anzuzeigen, dass die `File`-Menüschaltfläche beim Aktivieren ein Popup-Menü öffnet. Screenreader-Nutzer erhalten diese zusätzliche Information, wenn sie zur Schaltfläche navigieren.

Sie müssen JavaScript verwenden, um das Popup-Menü anzuzeigen und zu verbergen sowie eine ordnungsgemäße Tastaturunterstützung für die Interaktion mit dem Menü zu implementieren. Bitte beachten Sie auch, dass die ARIA-Rolle `menu` sich auf einen sehr spezifischen Menütetyp bezieht. Sie bezeichnet in der Regel eine Liste von Aktionen, die der Benutzer ausführen kann, ähnlich einem Menü in einer Desktop-Anwendung. Sie umfasst nicht die häufigeren Verwendungen von „Menüs“, wie Navigationsmenüs. Realistisch gesehen werden die meisten „Menüs“, die Sie im Web erstellen, keine ARIA-Menüs sein, und Sie werden `aria-haspopup` nicht mit ihnen verwenden.

Der nächste ARIA-Zustand, den wir besprechen, ist `aria-required`. Das `aria-required`-Attribut wird verwendet, um anzuzeigen, dass ein Feld ausgefüllt werden muss, bevor das Formular abgesendet wird.

Hier ist ein Beispiel für die Arbeit mit dem `aria-required`-Attribut bei einem benutzerdefinierten Formularsteuerelement.

:::interactive_editor

```html
<link rel="stylesheet" href="styles.css">
<div id="name-label">Full Name*</div>
<div role="textbox" contenteditable aria-labelledby="name-label" aria-required="true" id="name"></div>
```

```css
#name-label {
  font-family: system-ui, sans-serif;
  font-size: 0.95rem;
  font-weight: 600;
  margin-bottom: 0.25rem;
  color: #333;
}

[role="textbox"] {
  display: block;
  width: 100%;
  min-height: 2rem;
  padding: 0.5rem 0.75rem;
  border: 1.5px solid #ccc;
  border-radius: 4px;
  font-family: system-ui, sans-serif;
  font-size: 1rem;
  color: #222;
  background-color: #fff;
  line-height: 1.4;
  transition: border-color 0.2s, box-shadow 0.2s;
}

[role="textbox"]:hover {
  border-color: #999;
}

[role="textbox"]:focus {
  outline: none;
  border-color: #0078d4;
  box-shadow: 0 0 0 3px rgba(0, 120, 212, 0.25);
}

[role="textbox"]:empty::before {
  content: attr(data-placeholder);
  color: #aaa;
  pointer-events: none;
}

[role="textbox"][aria-required="true"] {
  border-left: 3px solid #e81123;
  padding-left: calc(0.75rem - 3px);
}
```

:::

Wir müssen das `contenteditable`-Attribut verwenden, damit Nutzer ihre Eingabe eingeben können. Außerdem verwenden wir das `aria-required`-Attribut mit dem Wert `true`, um anzuzeigen, dass dieses benutzerdefinierte Formularsteuerelement erforderlich ist.

Um das Formularsteuerelement wie ein normales Formularsteuerelement aussehen zu lassen, müssten Sie CSS hinzufügen. Außerdem müssten Sie JavaScript hinzufügen, um zu verhindern, dass das Formular ohne Inhalt abgesendet wird.

Wenn die Bezeichnung bereits das Wort `required` enthält, sollten Sie das `aria-required`-Attribut weglassen. So wird sichergestellt, dass Screenreader das Wort „erforderlich“ nur einmal ansagen.

In den meisten Fällen werden Sie wahrscheinlich die nativen `label`- und `form`-Elemente mit dem `required`-Attribut verwenden. Wenn Sie jedoch ein benutzerdefiniertes Formularsteuerelement erstellen müssen, ist es wichtig, das `aria-required`-Attribut bei Bedarf hinzuzufügen.

Zusätzlich kann das `aria-required`-Attribut auch bei nativen Formulareingaben verwendet werden, wie den `input`-, `textarea`- und `select`-Elementen. Dies wird oft dem nativen required-Attribut vorgezogen, da das required-Attribut potenzielle Nutzbarkeits- und Barrierefreiheitsprobleme haben kann, insbesondere mit der standardmäßigen Fehlerbehandlung des Browsers. Letztlich müssen Sie testen, um zu bestimmen, welches Attribut für Ihre Situation am besten geeignet ist.

Der letzte ARIA-Zustand, den wir besprechen, ist `aria-checked`. Dieses Attribut wird verwendet, um anzuzeigen, ob ein Element sich im aktivierten Zustand befindet. Es wird am häufigsten bei der Erstellung benutzerdefinierter Kontrollkästchen, Optionsschaltflächen, Schalter und Listboxen verwendet.

Hier ist ein Beispiel, wie Sie `aria-checked` bei einer benutzerdefinierten Kontrollkästchensteuerung verwenden können:

:::interactive_editor

```html
<link rel="stylesheet" href="styles.css">
<div role="checkbox" aria-checked="true" tabindex="0">Checkbox</div>
```

```css
[role="checkbox"] {
  display: inline-flex;
  align-items: center;
  gap: 0.5rem;
  font-family: system-ui, sans-serif;
  font-size: 1rem;
  cursor: pointer;
  user-select: none;
  color: #222;
}

[role="checkbox"]::before {
  content: "";
  display: inline-block;
  width: 1rem;
  height: 1rem;
  border: 2px solid #666;
  border-radius: 4px;
  background-color: #fff;
  transition: all 0.2s ease;
  box-sizing: border-box;
}

[role="checkbox"]:hover::before {
  border-color: #0078d4;
}

[role="checkbox"]:focus::before {
  outline: 2px solid #0078d4;
  outline-offset: 2px;
}

[role="checkbox"][aria-checked="true"]::before {
  background-color: #0078d4;
  border-color: #0078d4;
  background-image: url("data:image/svg+xml;utf8,<svg xmlns='http://www.w3.org/2000/svg' width='14' height='14' viewBox='0 0 14 14'><path fill='white' d='M5.2 10.4L2 7.2l1.1-1.1 2.1 2.1L10.9 2.5 12 3.6z'/></svg>");
  background-repeat: no-repeat;
  background-position: center;
}

[role="checkbox"][aria-disabled="true"] {
  opacity: 0.5;
  pointer-events: none;
  cursor: not-allowed;
}

```

:::

Native Kontrollkästchenelemente haben einen eingebauten `checked`-Zustand, der an unterstützende Technologien übermittelt wird. Wenn Sie jedoch eine benutzerdefinierte Kontrollkästchensteuerung erstellen, müssen Sie das `aria-checked`-Attribut verwenden, um den Zustand anzuzeigen.

Wenn der Benutzer mit der benutzerdefinierten Kontrollkästchensteuerung interagiert, müssen Sie den `aria-checked`-Zustand verwenden, um den neuen Zustand des Kontrollkästchens widerzuspiegeln. Wenn das Kontrollkästchen aktiviert ist, wird das `aria-checked`-Attribut auf `true` gesetzt. Wenn es deaktiviert ist, wird es auf `false` gesetzt.

Native Elemente bieten in der Regel bessere Unterstützung und eingebaute Barrierefreiheitsfunktionen.

Wenn Sie jedoch benutzerdefinierte Steuerungen erstellen müssen, ist die Verwendung von ARIA-Attributen unerlässlich, um den Zustand dieser Steuerungen effektiv an unterstützende Technologien zu übermitteln.

Testen Sie wie immer Ihre Arbeit, um sicherzustellen, dass ARIA-Attribute korrekt angewendet werden und die benutzerdefinierte Steuerung sowohl barrierefrei als auch benutzerfreundlich funktioniert.
