# --description--

GET 외에도 또 다른 일반적인 HTTP 동사가 있는데, 그것은 POST입니다. POST는 HTML 폼으로 클라이언트 데이터를 전송할 때 기본으로 사용하는 메서드입니다. REST 관례에서는 POST를 데이터베이스에 새 항목(새 사용자 또는 새 블로그 게시물)을 생성하기 위해 데이터를 전송할 때 사용합니다. 이 프로젝트에는 데이터베이스가 없지만, POST 요청을 처리하는 방법을 배우게 됩니다.

이런 종류의 요청에서는 데이터가 URL에 나타나지 않고 요청 본문에 숨겨져 있습니다. 본문은 HTTP 요청의 일부로, 페이로드라고도 합니다. 데이터가 URL에 보이지 않는다고 해서 비공개라는 뜻은 아닙니다. 왜 그런지 보려면 HTTP POST 요청의 원시 내용을 살펴보세요:

```http
POST /path/subpath HTTP/1.0
From: john@example.com
User-Agent: someBrowser/1.0
Content-Type: application/x-www-form-urlencoded
Content-Length: 20

name=John+Doe&age=25
```

보시다시피, 본문은 쿼리 문자열처럼 인코딩되어 있습니다. 이것은 HTML 폼에서 사용하는 기본 형식입니다. Ajax를 사용하면 더 복잡한 구조의 데이터를 처리하기 위해 JSON도 사용할 수 있습니다. 또 다른 인코딩 유형으로는 multipart/form-data가 있는데, 이 형식은 바이너리 파일 업로드에 사용됩니다. 이 연습에서는 URL 인코딩된 본문을 사용할 것입니다. POST 요청에서 오는 데이터를 파싱하려면 `body-parser` 패키지를 사용해야 합니다. 이 패키지는 다양한 형식의 데이터를 디코딩할 수 있는 일련의 미들웨어를 사용할 수 있게 해줍니다.

# --instructions--

`body-parser`는 이미 설치되어 있고 프로젝트의 `package.json` 파일에 있습니다. `require`를 `myApp.js` 파일 상단에 불러와서 `bodyParser`라는 변수에 저장하세요. URL 인코딩된 데이터를 처리하는 미들웨어는 `bodyParser.urlencoded({extended: false})`에서 반환됩니다. 이전 메서드 호출에서 반환된 함수를 `app.use()`에 전달하세요. 평소처럼, 미들웨어는 그것에 의존하는 모든 라우트보다 먼저 마운트해야 합니다.

**참고:** `extended`는 `body-parser`에 어떤 파싱을 사용할지 알려주는 설정 옵션입니다. `extended=false`일 때는 기본 인코딩인 `querystring` 라이브러리를 사용합니다. `extended=true`일 때는 `qs` 라이브러리를 사용해 파싱합니다.

`extended=false`를 사용할 때 값은 문자열이나 배열만 될 수 있습니다. `querystring`를 사용할 때 반환되는 객체는 기본 JavaScript `Object`를 프로토타입으로 상속하지 않으므로 `hasOwnProperty`, `toString` 같은 함수는 사용할 수 없습니다. 확장된 버전은 더 많은 데이터 유연성을 제공하지만 JSON에 비해 부족합니다.
