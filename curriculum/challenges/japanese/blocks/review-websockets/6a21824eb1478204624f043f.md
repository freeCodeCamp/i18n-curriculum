## WebSocketsとHTTPの違い

- **HTTP**はリクエスト・レスポンスモデルに従います：クライアントがリクエストを送り、サーバーがレスポンスを返し、その後接続が閉じられます。
- **WebSockets**はクライアントとサーバー間に恒久的な双方向接続を作り、どちらかが終了信号を送るまで接続が開いたままになります。
- HTTPはページの読み込みやデータ取得に適していますが、リアルタイムアプリケーションには遅く非効率になります。
- すべてのWebSocket接続は**HTTPハンドシェイク**から始まります：クライアントがアップグレードをリクエストし、サーバーが`101 Switching Protocols`で応答します。

### WebSocket接続の流れ

1. クライアントがHTTP経由でサーバーに接続し、アップグレードをリクエストします。
2. サーバーが受け入れ、`101 Switching Protocols`で応答します。
3. 接続は開いたままになります。
4. 両者は接続が閉じられるまでいつでもデータを交換します。

### HTTPとWebSocketの通信の違い

- **HTTP**：`Client → Request → Server → Response`（1サイクルで終了）
- **WebSocket**：`Client ↔ Server`（持続的で双方向）

### わかりやすい例え

- **HTTP**は手紙のやり取りのようなもの：1通書いて返事を待ち、会話が終わります。
- **WebSockets**は電話のようなもの：両者が必要なだけリアルタイムで話し続けます。

### WebSocketsの実際の利用例

- チャットアプリ
- オンラインマルチプレイヤーゲーム
- ライブスポーツのスコア更新
- 株価ダッシュボード
- 共同編集ツール（例：Google Docs）

## Pub/Subアーキテクチャ

- **Pub/Sub**（パブリッシュ/サブスクライブ）は、システムの各部分が直接知らなくても通信できるメッセージングアーキテクチャです。
- 主な3つの要素があります：**パブリッシャー**、**トピック**、**サブスクライバー**。
  - **パブリッシャー**はメッセージをトピックに送ります。
  - **トピック**（チャネルやグループとも呼ばれます）は仲介役をします。
  - **サブスクライバー**は購読したトピックからメッセージを受け取ります。

### Pub/Subのメッセージの流れ

1. サブスクライバーがトピックを購読します。
2. パブリッシャーがそのトピックにメッセージを送ります。
3. メッセージングシステムがそのトピックのすべてのサブスクライバーにメッセージを届けます。

### 直接通信とPub/Subの違い

- **Pub/Subなし**：`Application A → Application B` — 密結合で、Bの変更がAを壊す可能性があります。
- **Pub/Subあり**：`Application A → Topic → Subscribers` — Aはただ公開するだけで、誰が受け取るかを知る必要はありません。

### Pub/Subの実際の利用例

- 複数ルームのチャットアプリ
- オンラインマルチプレイヤーゲーム
- 通知システム
- ライブダッシュボード
- イベント駆動型アプリケーション

### WebSocketsとPub/Subの違い

- **WebSockets**は通信チャネル（道路）を提供します。
- **Pub/Sub**は特定のメッセージを誰が受け取るかを決めるメッセージングパターン（交通システム）を提供します。
- 多くのリアルタイムアプリは両方を組み合わせて使います：WebSocketsで接続を開いたままにし、Pub/Subでメッセージを適切なサブスクライバーにルーティングします。

## Node.jsでのWebSocketsの扱い方

- Node.jsには組み込みのWebSocketサーバーはありません。`ws`ライブラリが標準の外部パッケージとして使われます。
- インストールは：`npm install ws`

### WebSocketサーバーの作成

```js
const WebSocket = require("ws");

const server = new WebSocket.Server({ port: 3000 });
```

### WebSocketのイベント

| イベント | トリガー |
|---|---|
| `connection` | 新しいクライアントがサーバーに接続したとき |
| `message` | 接続済みクライアントがデータを送信したとき |
| `close` | 接続が閉じられたとき |
| `error` | 接続で問題が発生したとき |

### クライアントにメッセージを送る

```js
socket.send("Hello from the server");
```

### すべての接続クライアントにブロードキャストする

- `server.clients`にはすべての接続クライアントが保持されています。
- `readyState === WebSocket.OPEN`を確認して、切断されたクライアントへのエラーを避けてください。

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **ブロードキャスト**は複数の接続クライアントに一度にメッセージを送るパターンで、チャットアプリ、ライブ通知、スポーツ更新、マルチプレイヤーゲームイベントで使われます。
