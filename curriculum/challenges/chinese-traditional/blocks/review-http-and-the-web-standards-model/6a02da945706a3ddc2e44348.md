## 伺服器與客戶端/伺服器架構

- **伺服器**：接收其他電腦請求並透過網路回傳資料或服務的電腦或程式。此詞同時指實體硬體與其上運行的軟體。
- **客戶端/伺服器架構**：一種分散式架構，客戶端發送請求，伺服器處理請求並回傳回應。
- **客戶端**：發起對伺服器請求的程序或應用程式（例如，網頁瀏覽器）。
- **請求-回應循環**：客戶端發送請求；伺服器處理並回傳回應。

### 伺服器類型

- **網頁伺服器**：將網頁內容（HTML、圖片、CSS）傳送給瀏覽器。
- **應用程式伺服器**：處理商務邏輯和使用者輸入。
- **資料庫伺服器**：儲存、擷取和管理資料。
- **郵件伺服器**：發送、接收和儲存電子郵件。
- **檔案伺服器**：管理網路上的檔案儲存與分享。
- **代理伺服器**：作為客戶端與其他伺服器之間的中介。

## DNS（網域名稱系統）

- **DNS**：一個階層式且去中心化的命名系統，將人類可讀的網域名稱轉換成 IP 位址，常被稱為網際網路的「電話簿」。
- **IP 位址**：網路中裝置的唯一數字識別碼（例如，`127.0.0.1` 在 IPv4，`0:0:0:0:0:0:0:1` 在 IPv6）。
- **網域名稱階層**（由右至左讀）：
  - **頂級網域（TLD）**：最右邊的部分（例如，`.org`、`.com`）。
  - **次級網域（SLD）**：註冊名稱（例如，`freecodecamp` 在 `freecodecamp.org` 中）。
  - **子網域**：識別主網域的某個區段（例如，`www`、`blog`、`api`）。

### DNS 決議流程

1. 瀏覽器將網域名稱送至 **遞迴解析器**（通常由 ISP 管理）。
2. 遞迴解析器查詢 **根伺服器**，根伺服器指引至相關的 TLD 伺服器。
3. **TLD 伺服器** 回傳該網域的 **權威名稱伺服器** 位址。
4. 權威名稱伺服器回傳包含 IP 位址的 **DNS 紀錄**。
5. 遞迴解析器將 IP 位址傳給客戶端，並 **快取** 該紀錄以加速未來查詢。

## TCP/IP

- **協定**：一組規則和標準，讓裝置能在網路上通訊。
- **TCP/IP**：一套協定，決定資料如何封包、傳送、路由和接收。
  - **IP（網際網路協定）**：使用 IP 位址將資料封包從來源路由到目的地。它是無連線的—每個封包獨立傳送，無交付保證。
  - **TCP（傳輸控制協定）**：將資料拆成編號區段，在目的地重新組合，檢查錯誤和遺失封包，必要時請求重傳。確保可靠且有序的傳送。

## HTTP（超文字傳輸協定）

- **HTTP**：一種應用層協定，定義網頁上客戶端與伺服器間通訊的規則。

- **HTTPS**：HTTP 的安全版本，資料在傳輸前使用 SSL/TLS 加密。

### HTTP 請求結構

- **請求行**：`METHOD /path HTTP/version`（例如，`GET / HTTP/1.1`）
- **標頭**：鍵值對形式的詮釋資料（例如，`Content-Type: application/json`）。
- **主體**（選用）：傳送給伺服器的資料，用於 `POST`、`PUT` 和 `PATCH`。

**常見 HTTP 方法：**

| 方法 | 目的 |
| --- | --- |
| `GET` | 擷取資料 |
| `POST` | 提交新資料 |
| `PUT` | 替換資源 |
| `PATCH` | 部分更新資源 |
| `DELETE` | 移除資源 |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### HTTP 回應結構

- **狀態行**：HTTP 版本、狀態碼和原因短語（例如，`HTTP/1.1 200 OK`）。
- **標頭**：關於回應的詮釋資料（例如，`Content-Type: text/html`）。
- **主體**（選用）：回傳的內容或錯誤說明。

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## HTTP 狀態碼

狀態碼依首位數字分組：

- **1xx** — 資訊性回應。
- **2xx** — 成功回應。
- **3xx** — 重新導向訊息。
- **4xx** — 客戶端錯誤回應。
- **5xx** — 伺服器錯誤回應。

### 常見狀態碼

| 狀態碼 | 名稱 | 意義 |
| --- | --- | --- |
| `200` | OK | 請求成功；回傳內容。 |
| `201` | Created | 資源成功建立。 |
| `204` | No Content | 成功但無內容回傳。 |
| `301` | Moved Permanently | 資源永久移動到新 URL。 |
| `302` | Found | 資源暫時移動到其他 URL。 |
| `304` | Not Modified | 資源自上次請求後未變更；使用快取版本。 |
| `400` | Bad Request | 伺服器無法理解請求（語法或資料錯誤）。 |
| `401` | Unauthorized | 需要驗證憑證。 |
| `403` | Forbidden | 即使有有效憑證，客戶端無權存取資源。 |
| `404` | Not Found | 伺服器上找不到資源。 |
| `500` | Internal Server Error | 伺服器發生意外錯誤。 |
| `502` | Bad Gateway | 伺服器從上游伺服器收到無效回應。 |
| `503` | Service Unavailable | 伺服器暫時無法處理請求。 |

## HTTP 請求-回應模型

請求-回應循環由以下元件協同運作：

- **客戶端**：瀏覽器、行動應用程式或 API 客戶端，發送 HTTP 請求。
- **HTTP 方法**：要執行的動作（`GET`、`POST`、`PUT`、`PATCH`、`DELETE`）。
- **URL**：被請求資源的位址。
- **標頭**：請求或回應的詮釋資料（例如，`Authorization`、`Content-Type`）。
- **主體**：用於 `POST`、`PUT`、`PATCH` 或 `DELETE` 請求的資料（通常是 JSON 或表單資料）。
- **伺服器**：處理請求並回傳回應的後端系統。
- **狀態碼**：數字代碼，表示成功、錯誤或重新導向。
- **回應主體**：實際回傳的資料，通常是 HTML、JSON 或 XML。

### 循環的六個步驟

1. **客戶端發起請求**，包含方法、URL、標頭和選用主體。
2. **請求透過網際網路傳送至伺服器**。
3. **伺服器處理請求**：檢查方法、URL、驗證，必要時查詢資料庫。
4. **伺服器準備回應**，包含狀態碼、標頭和選用主體。
5. **回應傳回給客戶端**。
6. **客戶端處理回應**：成功（2xx）時呈現資料，失敗（4xx/5xx）時顯示錯誤訊息。

## HTTP 回應資源與 Content-Type

`Content-Type` 標頭告訴瀏覽器如何處理回應中的資源。

| 資源 | Content-Type |
| --- | --- |
| HTML | `text/html` |
| CSS | `text/css` |
| JavaScript | `application/javascript` |
| JSON | `application/json` |
| XML | `application/xml` |
| JPEG | `image/jpeg` |
| PNG | `image/png` |
| GIF | `image/gif` |
| SVG | `image/svg+xml` |
| WebP | `image/webp` |
| MP4 | `video/mp4` |
| MP3 | `audio/mpeg` |
| PDF | `application/pdf` |
| ZIP | `application/zip` |
| WOFF2 | `font/woff2` |

## HTML 表單提交

- 表單使用 `<form>` 元素搭配 `<input>` 元素收集並提交使用者資料。
- 每個輸入欄位需有 `name` 屬性，讓伺服器識別資料。
- **客戶端驗證**：使用 `required` 和 `pattern` 屬性提供即時回授。
- **伺服器端驗證**：無論客戶端檢查如何，為安全起見必須在伺服器端驗證。
- `action` 屬性指定表單資料送往何處。
- `method` 屬性指定送出方式（`GET` 或 `POST`；`PUT`、`PATCH`、`DELETE` 需使用 Fetch API 或 Express 中的 `method-override` 等替代方案）。

### 表單提交流程

1. 使用者填寫並提交表單。
2. 瀏覽器執行客戶端驗證；若通過則繼續。
3. 瀏覽器使用指定的 `action` 將表單資料送至 `method` 的 URL。
4. 伺服器接收並處理資料（儲存於資料庫、驗證使用者等）。
5. 伺服器可能執行額外的伺服器端驗證。
6. 伺服器回應成功訊息、錯誤或重新導向。
7. 使用者根據伺服器回應看到回授。

## 網頁標準模型與標準組織

- **網頁標準**：瀏覽器遵循的規則和規範，用以呈現頁面並啟用網頁功能。
- **WHATWG**：維護 HTML 和 DOM 的標準組織，持續更新「活文件」標準，由主要瀏覽器引擎團隊推動。
- **W3C**：主要標準組織，監督 CSS、SVG、無障礙指引（WCAG）和多項網頁 API。
- **ECMA / TC39**：ECMA 發布 ECMAScript 規範（JavaScript 標準）；TC39 透過五階段提案流程（階段 0–4）推動語言新功能。
- **Khronos Group**：負責 WebGL 和 WebGL 2，將 GPU 加速的 3D 呈現帶入瀏覽器。

## 創建網頁標準的流程

1. **識別需求**：開發者或瀏覽器工程師記錄一個真實且廣泛存在的問題。
2. **提案與早期討論**：非正式撰寫想法，收集回饋並評估瀏覽器廠商興趣。
3. **撰寫規範**：草擬精確且正式的文件，涵蓋所有邊界狀況、錯誤條件和安全考量。
4. **審查、異議與迭代**：公開審查草案；廠商提出異議並修訂規範，直到達成共識。
5. **實作與測試**：瀏覽器廠商在旗標下實作功能；共享測試套件驗證所有實作行為一致。
6. **定案**：多個獨立實作通過測試後，標準正式定案（W3C 推薦、TC39 階段 4 或 WHATWG 活文件更新）。

## 網頁標準功能的生命週期

1. **提案**：有人發現問題並記錄，確認是否有其他人也遇到。
2. **孵化**：透過公開討論檢驗想法；若過於小眾或無解，許多提案會在此階段終止。
3. **規範**：撰寫正式規範，將模糊概念轉成精確且可實作的規則。
4. **互通性測試**：W3C 網頁平台測試套件驗證所有瀏覽器一致實作該功能。
5. **定案**：多個通過測試的實作代表功能成為基線網頁平台的一部分，開始對使用者提供服務。
6. **棄用**：被取代或有問題的功能標記為淘汰；由於向後相容義務，瀏覽器移除過程緩慢。

## 網頁標準的核心原則

- **開放性**：標準在公開環境開發，任何人皆可免版稅實作。
- **無障礙性**：網頁設計為所有人可用，無障礙性內建於 HTML、CSS 和瀏覽器行為中。
- **互通性**：依標準建置的頁面應在所有瀏覽器和作業系統上表現一致。
- **向後相容**：新功能建立於既有功能之上，確保舊頁面在現代瀏覽器仍能運作。
- **隱私與安全**：可能暴露使用者資料的新 API 受到嚴格審查；平台預設保護使用者。
- **分層**：HTML 處理結構，CSS 處理呈現，JavaScript 處理行為—各層獨立運作。
- **漸進增強**：功能必須先對所有人有效，再針對更強大瀏覽器或裝置提供增強體驗。
