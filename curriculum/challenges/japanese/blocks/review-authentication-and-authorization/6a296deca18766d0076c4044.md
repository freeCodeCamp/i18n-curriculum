## 認証と認可の違い

**認証**とはユーザーの身元を確認するプロセスです。*「この人は本人か？」*という問いに答えます。ユーザーがログインすると、バックエンドサーバーは登録時に保存された情報と照合します。

バックエンド開発者は認証要素を主に3つのカテゴリに分けて使います：

- **知っているもの：** パスワード、PIN、またはセキュリティ質問の答え。
- **持っているもの：** スマートフォンのSMSコードや認証アプリの一時トークン。
- **本人であるもの：** 指紋や顔認証スキャンなどの生体情報。

**認可**は認証の後に行われ、認証されたユーザーが何をできるかを決定します。*「この身元にはどんな権限があるか？」*に答えます。一般的な役割は以下の通りです：

- **標準ユーザー：** 自分のプロフィールは編集できるが、他のユーザーのプライベートデータは見られません。
- **編集者：** コンテンツの作成と公開はできるが、コードベースの削除はできません。
- **管理者：** データベースの変更、ユーザーの削除、システム全体の設定変更が可能です。

**認証は必ず最初に行い、その後に認可が続きます。** ユーザーが誰かを知らなければ、どんな権限を与えるか決められません。

## JSON Web Tokens（JWT）

JWTはログイン成功後にサーバーがユーザーに発行するデジタルキーです。ユーザーは以降のリクエストでこれを提示し、パスワードを再入力せずに身元を証明します。

JWTの4つのステップの流れ：

1. **ログイン：** ユーザーが認証情報をサーバーに送信します。
2. **発行：** サーバーが認証情報を検証し、JWTを生成して返します。
3. **保存：** クライアントはJWTをブラウザのクッキーやローカルストレージに保存します。
4. **リクエスト：** 保護されたルートへのすべてのリクエストでクライアントはJWTを送信し、サーバーが検証します。

JWTは2つのドットで区切られた3つの部分から成ります：

- **ヘッダー：** トークンタイプ（JWT）と使用されるハッシュアルゴリズム（例：HMAC SHA256）を宣言するエンコードされたJSON。
- **ペイロード：** ユーザーID、ユーザー名、役割などのクレームを含むエンコードされたJSON。ヘッダーとペイロードはBase64エンコードされており暗号化されていないため、パスワードなどの機密情報は絶対に保存しないでください。
- **署名：** エンコードされたヘッダーとペイロードをサーバーだけが知る秘密鍵でハッシュ化して作成。ペイロードが改ざんされると署名が一致せず、サーバーは`401 Unauthorized`レスポンスでリクエストを拒否します。

JWTは**ステートレス認証**を可能にします。署名がトークンを作成したことを証明するため、サーバーはデータベースにセッションデータを保存する必要がありません。

## クロスサイトリクエストフォージェリ（CSRF）

CSRFは認証されたユーザーに気づかれずにウェブアプリで望まない操作を実行させる攻撃です。悪意あるサイトが被害者のブラウザを騙して、ターゲットサーバーに不正なリクエストを送らせます。ブラウザはターゲットサーバーへのリクエストに自動的にクッキーを付加する性質を悪用します。

**攻撃の例の流れ：**

1. 例えば`trusted-bank.com`にログインします。銀行はブラウザのクッキーにセッショントークンを保存します。
2. ログアウトせずに`evil-hacker-site.com`を訪問します。
3. 悪意あるサイトが`trusted-bank.com/transfer?amount=1000&to=hacker`へのリクエストを静かに送信します。
4. ブラウザは銀行のセッションクッキーを自動的に付加するため、サーバーはリクエストを処理します。

悪意あるサイトはクッキーを読み取れません。ただ、ブラウザがターゲットサーバーへのリクエストにクッキーを自動で付加することに便乗しているだけです。

**CSRFへの防御策：**

- **`SameSite=Strict`:** リクエストが第三者サイト由来の場合、ブラウザはクッキーを送信しません。
- **`SameSite=Lax`:** 安全なナビゲーション（リンククリック）ではクッキーを許可しますが、クロスサイトのフォーム送信やAPIリクエストではブロックします。
- **CSRFトークン：** ユーザーがページを読み込むと、サーバーはランダムで一意かつ予測不可能なトークンを生成し、フロントエンドのフォームやセッションに付加します。クライアントはリクエストヘッダーやフォームデータにこれを含める必要があります。悪意あるサイトはフロントエンドにアクセスできないためトークンをコピーできず、不正リクエストはバリデーションに失敗します。

## Passport.js

Passport.jsはNode.js用の認証ミドルウェアで、**Strategies**と呼ばれるモジュールプラグインを使って認証方法を標準化します。500以上の戦略があり、関連パッケージをインストールすれば任意の認証方法を追加できます：

- `passport-local`: 自分のデータベースに対するユーザー名とパスワード認証。
- `passport-jwt`: 保護されたAPIルートでJSON Web Tokensを検証。
- `passport-google-oauth20`: Googleアカウントでログイン。
- `passport-github2`: GitHubアカウントでログイン。

コアパッケージと戦略をインストールしてください：

```bash
npm install passport passport-local
```

ルートを定義する前にPassportをExpressアプリに組み込みます：

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database look-up and password verification logic
}));
```

Passportをミドルウェアとして使い、ログインルートを保護してください：

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

認証成功後、Passportは自動的に**セッション管理**（ユーザーデータをセッションクッキーに保存）と**リクエスト装飾**（`req.user`にユーザーオブジェクトを付加し、後続のルートハンドラで利用可能にする）を行います。

## Helmet.js

Helmet.jsはNode.js用のセキュリティミドルウェア集で、Expressがデフォルトで公開するHTTPレスポンスヘッダーを自動的に設定、変更、または削除してアプリケーションを保護します。

インストールして一度に適用してください：

```bash
npm install helmet
```

```js
const helmet = require('helmet');
app.use(helmet());
```

ミドルウェアスタックの最上部に`app.use(helmet())`を置くと、すべてのレスポンスにこれらの保護が適用されます：

- **X-Powered-Byを削除：** Expressはデフォルトで`X-Powered-By: Express`を付加しますが、Helmetはこれを削除し、攻撃者がバージョン固有の脆弱性を狙うのを防ぎます。
- **Content-Security-Policyを設定：** ブラウザがスクリプト、画像、スタイルを読み込む場所を制限し、クロスサイトスクリプティング（XSS）攻撃を防ぎます。
- **X-Frame-OptionsをSAMEORIGINに設定：** サイトが外部iframeに埋め込まれるのを防ぎ、クリックジャッキング攻撃をブロックします。
- **Strict-Transport-Securityを設定：** ブラウザにHTTPSのみでサーバーと通信させ、信頼できないネットワークでのデータ傍受を防ぎます。
