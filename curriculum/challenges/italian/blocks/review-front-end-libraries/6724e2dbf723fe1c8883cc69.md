## Librerie e framework JavaScript

- Le librerie e i framework JavaScript offrono soluzioni rapide a problemi comuni e velocizzano lo sviluppo fornendo codice già pronto.
- Le librerie sono generalmente più focalizzate nel fornire soluzioni per attività specifiche, come manipolare il DOM, gestire eventi o amministrare richieste AJAX.
- Alcuni esempi di librerie JavaScript sono jQuery e React.
- I framework, invece, offrono una struttura più definita per costruire applicazioni. Spesso includono un insieme di regole e convenzioni che gli sviluppatori devono seguire.
- Esempi di framework sono Angular e Next.js, un meta framework per React.
- Le **single-page application** (SPA) sono applicazioni web che caricano una sola pagina HTML e aggiornano dinamicamente quella pagina mentre l’utente interagisce con l’applicazione, senza ricaricare l’intera pagina.
- Le SPA usano JavaScript per gestire lo stato dell’applicazione e renderizzare il contenuto. Questo spesso viene fatto tramite framework che offrono ottimi strumenti per costruire interfacce utente complesse.
- Alcuni problemi legati alle SPA includono:

  - I lettori di schermo hanno difficoltà con il contenuto aggiornato dinamicamente.
  - L’URL non cambia quando l’utente naviga all’interno dell’applicazione, il che può rendere difficile aggiungere ai preferiti, tornare indietro o condividere pagine specifiche.
  - Il tempo di caricamento iniziale può essere lento se l’applicazione è grande, poiché tutte le risorse devono essere caricate subito.
## React

- React è una popolare libreria JavaScript per creare interfacce utente e applicazioni web.
- Un concetto fondamentale di React è la creazione di componenti UI riutilizzabili che possono aggiornarsi e renderizzarsi in modo indipendente quando i dati cambiano.
- React permette agli sviluppatori di descrivere come dovrebbe apparire l’interfaccia utente in base allo stato dell’applicazione. React poi aggiorna e renderizza i componenti giusti quando i dati o lo stato cambiano.
## Componenti React

- I componenti sono i mattoni fondamentali delle applicazioni React che permettono agli sviluppatori di suddividere interfacce utente complesse in parti più piccole e gestibili.
- L'interfaccia utente viene descritta usando JSX, un'estensione della sintassi JavaScript, che consente agli sviluppatori di scrivere codice simile a HTML all'interno di JavaScript.
- I componenti sono fondamentalmente funzioni o classi JS che restituiscono una parte dell'interfaccia utente.

Ecco un esempio di un semplice componente React che renderizza un messaggio di saluto:

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

Per usare il componente, puoi semplicemente chiamare:

```jsx
  <Greeting />
```
## Importazione ed esportazione dei componenti React

- I componenti React possono essere esportati da un file e importati in un altro file.
- Supponiamo che tu abbia un componente chiamato `City` in un file chiamato `City.js`. Puoi esportare il componente usando la parola chiave `export`:

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default City;
```

- Per importare il componente `City` in un altro file, puoi usare la parola chiave `import`:

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

- La parola chiave `default` viene usata perché è l'esportazione predefinita dal file `City.js`.

- Puoi anche scegliere di esportare il componente sulla stessa riga della definizione del componente in questo modo:

```jsx
export default function City() {
  return <p>New York</p>;
}
```
## Configurare un progetto React usando Vite

- Gli strumenti di configurazione dei progetti e le interfacce a riga di comando offrono un modo rapido e semplice per iniziare nuovi progetti, permettendo agli sviluppatori di concentrarsi sulla scrittura del codice invece che sulla configurazione.

Questo comando crea un nuovo progetto React chiamato `my-react-app` usando il template React di Vite. Nella cartella del progetto, vedrai un file `package.json` con le dipendenze del progetto e i comandi elencati al suo interno.

```bash

npm install # installs the dependencies listed in the package.json file
```

- Una volta installate le dipendenze, dovresti notare una nuova cartella nel tuo progetto chiamata `node_modules`.

- La cartella `node_modules` è dove vengono archiviati tutti i pacchetti e le librerie richiesti dal tuo progetto.

- Per avviare il tuo progetto, usa il seguente comando:

```bash
npm run dev
```

- Dopo di ciò, apri il browser e vai su `http://localhost:5173` per vedere la tua applicazione React in esecuzione.

- Per vedere effettivamente il codice del template iniziale, puoi entrare nel tuo progetto nella cartella `src` e dovresti vedere il file `App.jsx`.
## Passaggio delle props nei componenti React

- In React, le props (abbreviazione di proprietà) sono un modo per passare dati da un componente genitore a un componente figlio. Questo meccanismo è necessario per creare elementi dell'interfaccia utente riutilizzabili e dinamici.
- Le props possono essere qualsiasi valore JavaScript. Per passare le props da un genitore a un figlio, aggiungi le props come attributi quando usi il componente figlio nel JSX del genitore. Ecco un semplice esempio:

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

Puoi passare più props usando l'operatore di espansione `(...)`, dopo averle convertite in un oggetto. Ecco un esempio:
    
```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

In questo codice, l'operatore di espansione `{...person}` converte l'oggetto person in singole props che vengono passate al componente Child.
## Renderizzazione condizionale in React

- La renderizzazione condizionale in React ti permette di creare interfacce utente dinamiche. Viene usata per mostrare contenuti diversi in base a determinate condizioni o stati all'interno della tua applicazione.
- Esistono diversi modi per renderizzare contenuto in modo condizionale in React. Un approccio comune è usare l'operatore ternario. Ecco un esempio:

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- Un altro modo per renderizzare contenuto in modo condizionale è usare l'operatore logico AND (`&&`). Questo è utile quando vuoi renderizzare un contenuto solo se una certa condizione è soddisfatta. Ecco un esempio:

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

Nel codice qui sopra, l'elemento `h1` viene renderizzato solo se l'oggetto user è truthy.

Puoi anche usare direttamente un'istruzione `if` in questo modo:

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```
## Renderizzare liste in React

- Renderizzare liste in React è un'attività comune quando si costruiscono interfacce utente.
- Puoi renderizzare liste usando il metodo `map()` degli array JS per iterare su un array di elementi e restituire un nuovo array di elementi JSX.
- Ad esempio, se hai un array di nomi che vuoi visualizzare come lista, puoi fare così:

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- Ricorda sempre di fornire una chiave unica per ogni elemento della lista per aiutare React a gestire i ruoli di aggiornamento e rendering. Con queste tecniche puoi creare liste flessibili, efficienti e dinamiche nelle tue applicazioni React.
## Definire lo stile in linea in React

- Definire lo stile in linea in React ti permette di applicare stili CSS direttamente agli elementi JSX usando oggetti JavaScript.
- Per applicare stili in linea in React, puoi usare l'attributo style sugli elementi JSX. L'attributo style accetta un oggetto in cui le chiavi sono proprietà CSS in camelCase e i valori sono i valori corrispondenti. Ecco un esempio:

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

Puoi anche estrarre gli stili in un oggetto separato e poi richiamarlo nell'attributo `style` in questo modo:

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Gli stili in linea supportano la definizione dinamica dello stile permettendoti di applicare stili in modo condizionale in base alle props o allo stato. Ecco un esempio di come puoi applicare stili in modo condizionale in base a una prop:

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Nel codice qui sopra, gli stili `color` e `fontSize` vengono impostati in modo condizionale in base alla prop `isImportant`.
## Lavorare con gli eventi in React

- **Synthetic Event System**: Questo è il modo in cui React gestisce gli eventi. Funziona come un wrapper attorno agli eventi nativi come gli eventi `click`, `keydown` e `submit`. I gestori di eventi in React usano la convenzione di denominazione camel case. (Es. `onClick`, `onSubmit`, ecc.)

Ecco un esempio di utilizzo dell'attributo `onClick` per un elemento `button` in React:

```jsx
function handleClick() {
  console.log("Button clicked!");
}

<button onClick={handleClick}>Click Me</button>;
```

In React, le funzioni gestore degli eventi di solito iniziano con il prefisso `handle` per indicare che sono responsabili della gestione degli eventi, come `handleClick` o `handleSubmit`.

Quando un'azione dell'utente scatena un evento, React passa un oggetto Synthetic Event al tuo gestore. Questo oggetto si comporta in modo simile all'oggetto Event nativo in JavaScript puro, fornendo proprietà come `type`, `target` e `currentTarget`.

Per impedire comportamenti predefiniti come l'aggiornamento del browser durante un evento `onSubmit`, ad esempio, puoi chiamare il metodo `preventDefault()`:

```jsx
function handleSubmit(event) {
  event.preventDefault();
  console.log("Form submitted!");
}

<form onSubmit={handleSubmit}>
  <input type="text" />
  <button>Submit</button>
</form>;
```

Puoi anche racchiudere una funzione gestore in una funzione freccia come questa:

```jsx
function handleDelete(id) {
  console.log("Deleting item:", id);
}

<button onClick={() => handleDelete(1)}>Delete Item</button>;
```
## Lavorare con lo stato e l'hook `useState`

- **Definizione di stato**: In React, lo stato è un oggetto che contiene dati per un componente. Quando lo stato viene aggiornato, il componente verrà renderizzato di nuovo. React considera lo stato come immutabile, il che significa che non dovresti modificarlo direttamente.  
- **Hook `useState()`**: L'hook `useState` è una funzione che ti permette di dichiarare variabili di stato nei componenti funzionali. Ecco una sintassi di base:

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

Nella variabile di stato hai quanto segue:

- `stateVariable` contiene il valore attuale dello stato
- `setStateFunction` (la funzione setter) aggiorna la variabile di stato
- `initialValue` imposta lo stato iniziale

Ecco un esempio completo per un componente `Counter`:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>

      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```
## Renderizzazione e componenti React

- **Definizione**: In React, la renderizzazione è il processo attraverso cui i componenti appaiono nell'interfaccia utente (UI), di solito nel browser. Il processo di renderizzazione si compone di tre fasi: trigger, renderizzazione e commit.

La fase di trigger avviene quando React rileva che qualcosa è cambiato e l'interfaccia utente (UI) potrebbe dover essere aggiornata. Questo cambiamento è spesso dovuto a un aggiornamento dello stato o delle props.

Una volta che si verifica il trigger, React entra nella fase di renderizzazione. Qui React rivaluta i tuoi componenti e decide cosa mostrare. Per farlo, React utilizza una copia leggera del DOM "reale" chiamata virtual DOM. Con il virtual DOM, React può controllare rapidamente cosa deve cambiare nel componente.

La fase di commit è quella in cui React prende le modifiche preparate dal virtual DOM e le applica al DOM reale. In altre parole, questa è la fase in cui vedi il risultato finale sullo schermo.
## Aggiornare oggetti e array nello stato

- **Aggiornare oggetti nello stato**: Se devi aggiornare un oggetto nello stato, prima crea un nuovo oggetto o copia un oggetto esistente, poi imposta lo stato su quel nuovo oggetto. Qualsiasi oggetto inserito nello stato dovrebbe essere considerato di sola lettura. Ecco un esempio di come impostare nome, età e città di un utente. La funzione `handleChange` viene usata per gestire gli aggiornamenti delle informazioni dell'utente:

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **Aggiornare array nello stato**: Quando aggiorni gli array nello stato, è importante non modificare direttamente l'array usando metodi come `push()` o `pop()`. Invece, dovresti creare un nuovo array quando aggiorni lo stato:

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

Se vuoi rimuovere elementi da un array, dovresti usare il metodo `filter()`, che restituisce un nuovo array dopo aver filtrato ciò che vuoi rimuovere:

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```
## Consultare i valori usando i ref

- **Attributo `ref`**: Puoi consultare un nodo DOM in React usando l'attributo `ref`. Ecco un esempio che mostra come usare `ref` per impostare il focus su un elemento `input`. La proprietà `current` viene usata per consultare il valore attuale di quel `ref`:

```jsx
import { useRef } from "react";

const Focus = () => {
  const inputRef = useRef(null);

  const handleFocus = () => {
    if (inputRef.current) {
      inputRef.current.focus();
    }
  };

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Enter text" />
      <button onClick={handleFocus}>Focus Input</button>
    </div>
  );
};

export default Focus;
```
## Lavorare con l'hook `useEffect`

- **Hook `useEffect()`**: In React, un effetto è qualsiasi cosa che avviene esternamente al processo di renderizzazione del componente. Cioè, tutto ciò che React non gestisce direttamente come parte del render della UI. Esempi comuni includono il recupero di dati, l'aggiornamento del titolo della scheda del browser, la lettura o scrittura nello storage locale del browser, l'ottenimento della posizione dell'utente e molto altro. Queste operazioni interagiscono con il mondo esterno e sono chiamate effetti collaterali. React fornisce l'hook `useEffect` per permetterti di gestire questi effetti collaterali. `useEffect` ti consente di eseguire una funzione dopo che il componente è stato renderizzato o aggiornato.

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [dependencies]);
```

La funzione effetto viene eseguita dopo il render del componente, mentre l'argomento opzionale `dependencies` controlla quando l'effetto viene eseguito.

Nota che `dependencies` può essere un array di "valori reattivi" (stato, props, funzioni, variabili, ecc.), un array vuoto oppure può essere omesso del tutto. Ecco come tutte queste opzioni controllano il funzionamento di `useEffect`:

- Se `dependencies` è un array che include uno o più valori reattivi, l'effetto verrà eseguito ogni volta che questi cambiano.

- Se `dependencies` è un array vuoto, `useEffect` viene eseguito solo una volta quando il componente viene renderizzato per la prima volta.

- Se ometti `dependencies`, l'effetto viene eseguito ogni volta che il componente viene renderizzato o aggiornato.
## Come creare hook personalizzati

- **Hook personalizzati**: Un hook personalizzato ti permette di estrarre logica riutilizzabile dai componenti, come il recupero di dati, la gestione dello stato, il toggle e gli effetti collaterali come il tracciamento dello stato online. In React, tutti gli hook integrati iniziano con la parola `use`, quindi anche il tuo hook personalizzato dovrebbe seguire la stessa convenzione.

Ecco un esempio di creazione di un hook `useDebounce`:

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```
## Lavorare con i moduli in React

- **Input controllati**: In questo caso memorizzi il valore del campo di input nello stato e lo aggiorni tramite eventi `onChange`. Questo ti dà il pieno controllo sui dati del modulo e permette una validazione istantanea e un rendering condizionale.

```jsx
import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **Input non controllati**: Invece di gestire gli input tramite il hook `useState`, gli input non controllati in HTML mantengono il proprio stato interno grazie al DOM. Poiché è il DOM a controllare i valori degli input, devi recuperare i valori dei campi di input con un `ref`.

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```
## Lavorare con il `useActionState` Hook

- **Server Actions**: Queste sono funzioni che vengono eseguite sul server per gestire i moduli direttamente sul server senza la necessità di endpoint API. Ecco un esempio da un'applicazione Next.js:

```js
"use server";

async function submitForm(formData) {
 const name = formData.get("name");
 return { message: `Hello, ${name}!` };
}
```

La direttiva `"use server"` indica che la funzione è una server action.

- **`useActionState` Hook**: Questo hook aggiorna lo stato in base all'esito dell'invio di un modulo. Ecco la sintassi di base dell'hook `useActionState`:

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state` è lo stato attuale restituito dall'azione.
- `action` è la funzione che avvia la server action.
- `isPending` è un booleano che indica se l'azione è attualmente in esecuzione oppure no.
- Il parametro `actionFunction` è la server action stessa.
- `initialState` è il parametro che rappresenta il punto di partenza per lo stato prima che l'azione venga eseguita.
- `permalink` è una stringa opzionale che contiene l'URL univoco della pagina che il modulo modifica.
## Recupero dei dati in React

- **Opzioni per recuperare i dati**: Esistono molti modi diversi per recuperare i dati in React. Puoi usare la Fetch API nativa oppure strumenti di terze parti come Axios o SWR.
- **Variabili di stato comunemente usate durante il recupero dei dati**: Indipendentemente dal metodo scelto per recuperare i dati in React, ci sono alcune variabili di stato di cui dovrai tenere traccia. La prima è il dato stesso. La seconda serve a tracciare se i dati sono ancora in fase di recupero. La terza è una variabile di stato che cattura eventuali errori che potrebbero verificarsi durante il processo di recupero dei dati.

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

Poiché il recupero dei dati è un effetto collaterale, è meglio usare il `Fetch API` all'interno di un hook `useEffect`.

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

Poi puoi renderizzare un messaggio di caricamento se il recupero dei dati non è completo, un messaggio di errore se si è verificato un errore durante il recupero dei dati, oppure i risultati.

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

Se vuoi usare Axios, devi prima installarlo e importarlo:

```bash
npm i axios
```

```js
import axios from "axios";
```

Poi puoi recuperare i dati usando `axios.get`:

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);

useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await axios.get(
        "https://jsonplaceholder.typicode.com/users"
      );
      setData(res.data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();

```

## Lavorare con l'hook `useOptimistic`

- **Hook `useOptimistic`**: Questo hook viene usato per mantenere le interfacce utente responsivi mentre si attende il completamento di un'azione asincrona in background. Aiuta a gestire gli "aggiornamenti ottimistici" nell'interfaccia, una strategia in cui fornisci aggiornamenti immediati all'interfaccia in base al risultato atteso di un'azione, come quando si attende una risposta dal server.

Ecco la sintassi di base:

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState` è lo stato temporaneo che si aggiorna subito per offrire una migliore esperienza utente.
- `addOptimistic` è la funzione che applica l'aggiornamento ottimistico prima che lo stato reale cambi.
- `actualState` è il valore reale dello stato che deriva dal risultato di un'azione, come il recupero di dati da un server.
- `updateFunction` è la funzione che determina come aggiornare lo stato ottimistico quando viene chiamata.

Ecco un esempio di utilizzo dell'hook `useOptimistic` in un componente `TaskList`:

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**: Questo viene usato per renderizzare una parte dell'interfaccia e segnare un aggiornamento di stato come una transizione non urgente. Questo permette all'interfaccia di restare responsiva durante aggiornamenti costosi. Ecco la sintassi di base:

```js
startTransition(action)
```

`action` esegue un aggiornamento di stato o attiva una logica legata alla transizione. Questo garantisce che gli aggiornamenti urgenti dell'interfaccia (come digitare o fare clic) non vengano bloccati.

## React Router

- **Introduzione**: React Router è una libreria di terze parti che ti permette di aggiungere il routing alle tue applicazioni React. Per iniziare, devi installare React Router in un progetto React esistente in questo modo:

```bash
npm i react-router
```

Poi, all'interno del file `main.jsx` o `index.jsx`, devi configurare la struttura delle rotte così:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

`path` e `element` vengono usati per collegare l'URL e i componenti dell'interfaccia utente. In questo caso, stiamo configurando una rotta per la homepage che punta al componente `App`.

- **Viste multiple e configurazione delle rotte**: È comune nelle applicazioni più grandi avere viste e rotte multiple configurate in questo modo:

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

La prop `index` in questi esempi serve a rappresentare la rotta predefinita per un determinato segmento di percorso. Quindi il componente `Home` verrà mostrato al percorso root `/` mentre il componente `ProductsHome` verrà mostrato al percorso `/products`.

- **Annidare le rotte**: Puoi annidare rotte all'interno di altre rotte, il che fa sì che il percorso della rotta figlia venga aggiunto in coda al percorso della rotta genitore.

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

Nell'esempio sopra, il percorso per i prodotti di tendenza sarà products/trending.

- **Segmenti dinamici**: Un segmento dinamico è una parte del percorso URL che è dinamica.

```jsx
<Route path=":category" element={<Category />} />
```

In questo esempio abbiamo un segmento dinamico chiamato `category`. Quando una persona naviga verso un URL come products/brass-instruments, la vista cambierà al componente `Category` e puoi recuperare dinamicamente i dati appropriati in base al segmento.

- **Hook `useParams`**: Questo hook viene usato per consultare i parametri dinamici da un percorso URL.

```js
import { useParams } from "react-router";

export default function Category() {
```

## Gestione dello stato

- **Context API**: Il context si riferisce a quando un componente genitore rende disponibili delle informazioni ai componenti figli senza doverle passare esplicitamente tramite le props. `createContext` viene usato per creare un oggetto context che rappresenta il context che altri componenti leggeranno. `Provider` viene usato per fornire i valori del context ai componenti figli.

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**: Redux gestisce la gestione dello stato fornendo un archivio centrale e un controllo rigoroso sugli aggiornamenti dello stato. Utilizza un pattern prevedibile con azioni, reducer e middleware. Le azioni sono payload di informazioni che inviano dati dalla tua applicazione all'archivio Redux, spesso attivate dalle interazioni dell'utente. I reducer sono funzioni che specificano come lo stato dovrebbe cambiare in risposta a quelle azioni, garantendo che lo stato venga aggiornato in modo immutabile. Il middleware, invece, agisce come un ponte tra il dispatch delle azioni e il reducer, permettendoti di estendere la funzionalità di Redux (ad esempio, logging, gestione di operazioni asincrone) senza modificare il flusso principale.

- **Zustand**: Questa soluzione per la gestione dello stato è ideale per applicazioni di piccole e medie dimensioni. Funziona usando un hook `useStore` per consultare direttamente lo stato nei componenti e nelle pagine. Questo ti permette di modificare e consultare i dati senza dover usare azioni, reducer o un provider.

## Differenze tra prestazioni reali e percepite

- **Prestazioni percepite**: sono il modo in cui le persone percepiscono le prestazioni di un sito web. È come valutano il sito in termini di reattività e affidabilità. Si tratta di una misurazione soggettiva, quindi è difficile quantificarla, ma è molto importante perché l'esperienza utente determina il successo o il fallimento di un sito web.
- **Prestazioni reali**: sono le prestazioni oggettive e misurabili del sito web. Si misurano usando metriche come il tempo di caricamento della pagina, il tempo di risposta del server e il tempo di rendering. Queste misurazioni sono influenzate da molteplici fattori legati sia alla rete sia al codice stesso.
## Tecniche per migliorare le prestazioni percepite

- **Lazy Loading**: Questa tecnica riduce il tempo di caricamento iniziale il più possibile caricando in background le risorse non essenziali.
- **Minimizzare i ritardi di caricamento dei caratteri tipografici**: Se il tuo sito web utilizza caratteri tipografici personalizzati, dovresti anche cercare di ridurre al minimo i ritardi di caricamento dei caratteri, poiché questo può causare sfarfallii o la visualizzazione del carattere tipografico di fallback mentre quello personalizzato viene caricato. Un suggerimento è usare un carattere tipografico di fallback simile a quello personalizzato, così nel caso in cui ciò accada il cambiamento sarà più delicato.
- **Uso di indicatori di caricamento**: Mostrare un indicatore di caricamento per un processo che richiede tempo non appena l’utente fa clic su un elemento può aiutare l’utente a sentirsi coinvolto e connesso al processo, facendo percepire il tempo di attesa come più breve.
## Concetti fondamentali sulle prestazioni

- **Ordine del sorgente**: Si riferisce al modo in cui gli elementi HTML sono strutturati nel documento. Questo determina cosa viene caricato per primo e può avere un impatto significativo sulle prestazioni e sull'accessibilità.

Alcune buone pratiche per l'ordine del sorgente includono:

- Posizionare il contenuto critico come intestazioni, navigazione o testo principale più in alto nella struttura HTML.
- Posticipare gli script non essenziali come quelli per l'analisi dei dati o widget di terze parti, in modo che non blocchino il rendering.
- Utilizzare il progressive enhancement, per garantire che l'esperienza principale funzioni anche prima che vengano caricati stili e script. Il progressive enhancement è un modo di costruire siti web e applicazioni basato sull'idea che la pagina debba funzionare prima di tutto con l'HTML.

Ecco un esempio di buon ordine del sorgente, utilizzando le buone pratiche che abbiamo appena visto:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **Critical Rendering Path**: Questa è la sequenza di passaggi che il browser segue per convertire il codice in pixel sullo schermo.
- **Latenza**: È il tempo che impiega una richiesta a viaggiare tra il browser e il server. In altre parole, una latenza elevata equivale a pagine lente.

Alcuni modi per ridurre la latenza includono:

- Utilizzare CDN, ovvero Content Delivery Network, per fornire i file da posizioni più vicine.
- Abilitare la compressione usando strumenti come Gzip per ridurre la dimensione dei file.
- Ottimizzare le immagini e utilizzare il caricamento differito (lazy loading).

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```
## Migliorare INP

- **Definizione**: INP (Interaction to Next Paint) valuta la reattività complessiva di una pagina misurando il tempo che intercorre tra l'interazione dell'utente, come un clic o la pressione di un tasto, e il successivo aggiornamento della visualizzazione da parte del browser. Un valore INP più basso indica una pagina più responsiva.

Ecco alcuni modi per migliorare l'INP:

- Riduci il lavoro sul thread principale suddividendo le attività JavaScript lunghe.
- Usa `requestIdleCallback()` per gli script non critici. Questo metterà in coda una funzione da chiamare durante i periodi di inattività del browser.
- Rimanda o carica in modo lazy le risorse pesanti, come visto in precedenza.
- Ottimizza i gestori degli eventi. Se questi gestori vengono eseguiti troppo spesso o svolgono operazioni pesanti, possono rallentare la pagina e aumentare l'INP. La soluzione a questo problema è il debouncing. Il debouncing assicura che la funzione venga eseguita solo dopo che l'utente ha smesso di digitare per un breve intervallo di tempo, ad esempio 300 ms. Questo evita calcoli non necessari e migliora le prestazioni.
## Come funziona il rendering nel browser

- **Come funziona il rendering**: Prima il browser analizza l'HTML e costruisce il DOM. Successivamente, il browser elabora il CSS, costruendo il CSS Object Model, o CSSOM. Questo è un altro albero che determina come gli elementi devono essere stilizzati. Infine, il browser disegna i pixel sullo schermo, renderizzando ogni elemento in base agli stili e al layout calcolati. Nelle pagine complesse, questo può coinvolgere più livelli che vengono composti insieme per creare l'output visivo finale.
## Come le prestazioni influenzano la sostenibilità

- **Informazioni di base**: Internet rappresenta circa il 2% delle emissioni globali di carbonio, ovvero quanto l'industria aerea! Ogni byte trasferito richiede elettricità, dai data center ai dispositivi degli utenti. File più grandi e script inefficienti comportano un maggiore consumo di energia. Un sito web ad alte prestazioni non è solo più veloce, ma riduce anche l'elaborazione superflua e l'uso di energia.
## Modi per ridurre i tempi di caricamento della pagina

- **Ottimizza le risorse multimediali**: Immagini e video di grandi dimensioni sono cause comuni di tempi di caricamento lenti. Ottimizzando queste risorse, puoi velocizzare notevolmente il tuo sito. Questo include operazioni come comprimere le immagini, usare formati moderni come WebP e applicare il caricamento ritardato per le risorse.
- **Sfrutta la cache del browser**: La cache permette ai browser di archiviare localmente parti del tuo sito, riducendo i tempi di caricamento per i visitatori che tornano.
- **Minimizza e comprimi i file**: Ridurre la dimensione dei tuoi file può portare a scaricamenti più rapidi. Questo include la riduzione della dimensione dei file trasmessi e la minimizzazione di file CSS e JavaScript.
## Migliorare il "time to usable"

- **Definizione**: il "time to usable" è l'intervallo che passa da quando un utente effettua una richiesta per una pagina a quando può interagire in modo significativo con essa. Per migliorare il "time to usable" puoi caricare le tue risorse in modo lazy o ridurre al minimo le risorse che bloccano il render.
## Metriche chiave per misurare le prestazioni

- **First Contentful Paint o FCP**: misura quanto velocemente il primo elemento di contenuto—testo o immagine—appare sullo schermo. Un buon FCP è considerato un tempo inferiore a 1,8 secondi, mentre un FCP scarso è superiore a 3 secondi. Puoi controllare il tuo FCP usando gli strumenti per sviluppatori di Chrome e verificando la scheda delle prestazioni.
- **Total Blocking Time**: mostra per quanto tempo il thread principale è bloccato da attività JavaScript pesanti. Se il Total Blocking Time (TBT) è alto, le interazioni risultano lente per l’utente. Per migliorare il TBT, suddividi le attività lunghe e rimanda l’esecuzione degli script non essenziali.
- **Bounce Rate**: è la percentuale di visitatori che abbandonano il sito senza interagire. Se il tuo sito ha un bounce rate elevato, potrebbe essere perché la pagina è troppo lenta.
- **Utenti unici**: questa metrica tiene traccia di quanti visitatori individuali arrivano sul tuo sito. Per visualizzare il bounce rate e gli utenti unici, puoi usare Google Analytics. Ti permetterà di monitorare queste metriche e migliorare il coinvolgimento.

## Lavorare con le API Web delle prestazioni

- **Definizione**: Le API Web delle prestazioni permettono agli sviluppatori di monitorare quanto efficientemente una pagina viene caricata e risponde, direttamente nel codice. Queste API consentono di misurare i tempi di caricamento della pagina, tracciare i ritardi di renderizzazione e interazione e analizzare il tempo di esecuzione di JavaScript.
- **`performance.now()`**: Questa API fornisce timestamp ad alta precisione (in millisecondi) per misurare quanto tempo impiegano a caricarsi le diverse parti del tuo sito.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: Questa API offre una suddivisione di ogni fase del caricamento della pagina, dalla ricerca DNS a `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: Questa API ascolta eventi di prestazione come spostamenti di layout, attività lunghe e interazioni dell’utente.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```
## Tecniche per migliorare le prestazioni del CSS

- **Animazioni CSS**: Animare alcune proprietà CSS, come dimensioni, posizione e layout, attiva un processo chiamato "reflow", durante il quale il browser ricalcola la posizione e la geometria di alcuni elementi sulla pagina. Questo richiede un repaint, che è costoso dal punto di vista computazionale. Per questo motivo, si consiglia di ridurre il più possibile il numero di animazioni CSS o almeno di dare all'utente la possibilità di attivarle o disattivarle.
## Tecniche per migliorare le prestazioni di JavaScript

- **Code Splitting**: Dividere il tuo codice JavaScript in moduli che svolgono attività critiche e non critiche è anch'esso utile. In questo modo, potrai precaricare quelli critici il prima possibile e rimandare quelli non critici per renderizzare la pagina il più velocemente possibile.
- **Manipolazione del DOM**: Ricorda che la manipolazione del DOM si riferisce al processo di modifica dinamica del contenuto di una pagina con JavaScript interagendo con il Document Object Model (DOM). Manipolare il DOM richiede molte risorse computazionali. Riducendo la quantità di manipolazione del DOM nel tuo codice JavaScript migliorerai le prestazioni.

## Test manuale e automatico

## Test unitari

- **Test unitari**: Nei test unitari, testi ogni funzione per assicurarti che tutto funzioni come previsto. I test unitari possono anche servire come una forma di documentazione per la tua applicazione perché dovrebbero rappresentare il comportamento atteso per il tuo codice.
- **Principio della singola responsabilità**: Il principio della singola responsabilità raccomanda di mantenere ogni funzione piccola e responsabile di una sola cosa.
- **Framework di test JavaScript comuni**: Alcuni framework di test comuni includono Jest, Mocha e Vitest. Jest è un framework di test molto usato per i test unitari.

Ecco un esempio di test unitari usando Jest.

Per prima cosa, puoi creare una funzione che si occupa di restituire una nuova stringa formattata:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

In un file `getFormattedWord.test.js` separato, puoi scrivere alcuni test per verificare che la funzione funzioni come previsto. Il file `getFormattedWord.test.js` sarà simile a questo:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **Funzione `expect`**: La funzione `expect` viene usata per testare un valore.
- **Matcher**: Un matcher è una funzione che controlla se il valore si comporta come previsto. Nell'esempio sopra, il matcher è `toBe()`. Jest offre una varietà di matcher.

Per usare Jest, devi prima installare il pacchetto `jest` usando `npm i jest`. Dovrai anche aggiungere uno script al tuo file `package.json` in questo modo:

```js
"scripts": {
  "test": "jest"
},
```

Poi puoi eseguire il comando `npm run test` per avviare i tuoi test.
## Ciclo di vita dello sviluppo software

## Mocking, faking e stubbing

- **Mocking**: Il mocking consiste nel sostituire dati reali con dati falsi che simulano il comportamento di componenti reali. Ad esempio, puoi fare il mocking della risposta di un'API durante i test invece di effettuare chiamate API continue per recuperare i dati.
- **Stubbing**: Gli stub sono oggetti che restituiscono risposte predefinite o dati fittizi per un comportamento atteso in un'applicazione. Ad esempio, puoi creare uno stub per il comportamento di una connessione al database nei tuoi test senza dover dipendere da una connessione reale al database.
- **Faking**: I fake sono versioni semplificate di componenti reali, senza la complessità o gli effetti collaterali. Ad esempio, puoi creare un fake di un database memorizzando i dati in memoria invece di interagire con il database reale. Questo ti permette di simulare le operazioni di database in memoria, che saranno molto più veloci rispetto a lavorare con il database reale.

## Test di usabilità

- **Test di usabilità**: Il test di usabilità consiste nel far interagire utenti reali con l'applicazione per scoprire se ci sono problemi di design, esperienza utente o funzionalità nell'app. Il test di usabilità si concentra sull'intuitività dell'applicazione per gli utenti.
- **Quattro tipi comuni di test di usabilità**:
  - **Esplorativo**: Il test di usabilità esplorativo prevede che gli utenti interagiscano con le diverse funzionalità dell'applicazione per capire meglio come funzionano.
  - **Comparativo**: Il test comparativo consiste nel confrontare l'esperienza utente della tua applicazione con quella di applicazioni simili presenti sul mercato.
  - **Valutativo**: Il test valutativo serve a studiare quanto sia intuitivo usare l'applicazione.
  - **Di validazione**: Il test di validazione serve a individuare eventuali problemi gravi che impedirebbero all'utente di usare l'applicazione in modo efficace.
- **Strumenti per il test di usabilità**: Esempi di strumenti per il test di usabilità includono Loop11, Maze, Userbrain, UserTesting e UXTweak.
## Test di compatibilità

- **Test di compatibilità**: L'obiettivo del test di compatibilità è assicurarsi che la tua applicazione funzioni in diversi ambienti informatici.
- **Tipi diversi di test di compatibilità**:
  - **Compatibilità all'indietro**: La compatibilità all'indietro indica quando il software è compatibile con versioni precedenti.
  - **Compatibilità in avanti**: La compatibilità in avanti indica quando il software e i sistemi funzioneranno con versioni future.
  - **Compatibilità hardware**: La compatibilità hardware è la capacità del software di funzionare correttamente in diverse configurazioni hardware.
  - **Compatibilità con i sistemi operativi**: La compatibilità con i sistemi operativi è la capacità del software di funzionare su diversi sistemi operativi, come macOS, Windows e distribuzioni Linux come Ubuntu e Fedora.
  - **Compatibilità di rete**: La compatibilità di rete significa che il software può funzionare in diverse condizioni di rete, come velocità di rete diverse, protocolli, impostazioni di sicurezza, ecc.
  - **Compatibilità con i browser**: La compatibilità con i browser significa che l'applicazione web può funzionare in modo coerente su browser diversi, come Google Chrome, Safari, Firefox, ecc.
  - **Compatibilità mobile**: È importante assicurarsi che le tue applicazioni software funzionino su una varietà di dispositivi Android e iOS, inclusi telefoni e tablet.
## Test delle prestazioni

- **Test delle prestazioni**: Nei test delle prestazioni, verifichi la velocità, la reattività, la scalabilità e la stabilità di un'applicazione sotto carichi di lavoro diversi. L'obiettivo è risolvere qualsiasi tipo di collo di bottiglia delle prestazioni.
- **Tipi diversi di test delle prestazioni**:
  - **Test di carico**: Il test di carico determina come si comporta un sistema durante i periodi di carico normali e di picco.
  - **Test di stress**: Il test di stress consiste nel testare la tua applicazione con carichi estremi e vedere come il sistema risponde a un carico maggiore.
  - **Test di durata (Soak Testing)**: Il test di durata, o soak testing, è un tipo di test di carico in cui il sistema viene testato con un carico elevato per un periodo di tempo prolungato.
  - **Test a picco (Spike Testing)**: Il test a picco consiste nell'aumentare e diminuire drasticamente i carichi e analizzare le reazioni del sistema ai cambiamenti.
  - **Test di capacità (Breakpoint Testing)**: Il test di capacità, o breakpoint testing, consiste nell'incrementare lentamente il carico nel tempo fino al punto in cui il sistema inizia a fallire o a degradarsi.
## Test di sicurezza

- **Test di sicurezza**: I test di sicurezza aiutano a individuare vulnerabilità e punti deboli.
- **Principi di sicurezza**:
  - **Riservatezza**: Protegge dalla divulgazione di informazioni sensibili a destinatari diversi da quelli previsti.
  - **Integrità**: Consiste nell'impedire che utenti malintenzionati modifichino le informazioni degli utenti.
  - **Autenticazione**: Consiste nel verificare l'identità dell'utente per assicurarsi che sia autorizzato a usare quel sistema.
  - **Autorizzazione**: È il processo che determina quali azioni gli utenti autenticati possono eseguire o a quali parti del sistema possono consultare.
  - **Disponibilità**: Garantisce che informazioni e servizi siano disponibili agli utenti autorizzati quando ne hanno bisogno.
  - **Non ripudio**: Garantisce che sia il mittente sia il destinatario abbiano la prova della consegna e la verifica dell'identità del mittente. Protegge dal rischio che il mittente neghi di aver inviato le informazioni.
- **Minacce di sicurezza comuni**:
  - **Cross-Site Scripting (XSS)**: Gli attacchi XSS avvengono quando un attaccante inietta script dannosi in una pagina web ed esegue questi script nel contesto del browser della vittima.
  - **SQL Injection**: L'SQL injection permette a utenti malintenzionati di iniettare codice dannoso in un database.
  - **Attacco Denial-of-Service (DoS)**: Un attacco DoS si verifica quando utenti malintenzionati sovraccaricano un sito web con un numero elevato di richieste o traffico, causando il rallentamento o il crash del server e rendendo il sito non disponibile agli utenti.
- **Categorie di strumenti per il test di sicurezza**:
  - **Static Application Security Testing**: Questi strumenti valutano il codice sorgente di un'applicazione per individuare vulnerabilità di sicurezza.
  - **Dynamic Application Security Testing**: Questi strumenti interagiscono con il front-end dell'applicazione per scoprire potenziali punti deboli di sicurezza. Gli strumenti DAST non hanno accesso al codice sorgente.
- **Test di penetrazione (pentest)**: Il test di penetrazione è un tipo di test di sicurezza che consiste nel simulare attacchi informatici sull'applicazione per individuare eventuali vulnerabilità nel sistema.
## Test A/B

- **Test A/B**: Il test A/B consiste nel confrontare due versioni di una pagina o di un'applicazione e analizzare quale versione ottiene risultati migliori. È conosciuto anche come bucket testing o split testing. Il test A/B permette di prendere decisioni più basate sui dati e di migliorare continuamente l'esperienza utente.
- **Strumenti per il test A/B**: Esempi di strumenti da usare per il test A/B includono GrowthBook e LaunchDarkly.
## Test di alpha e beta

Una volta completati lo sviluppo iniziale e i test del software, è importante che l'applicazione venga testata da tester e utenti reali. Qui entrano in gioco i test di alpha e beta.

- **Test di alpha**: Il test di alpha viene eseguito da un gruppo selezionato di tester che analizzano l'applicazione per assicurarsi che non ci siano bug prima che venga rilasciata sul mercato. Il test di alpha fa parte dei test di accettazione e utilizza sia tecniche di white box che di black box.
- **Test di beta**: Il test di beta consiste nel rendere l'applicazione disponibile agli utenti reali. Gli utenti possono interagire con l'applicazione e fornire feedback. Anche il test di beta è una forma di test di accettazione da parte degli utenti.
- **Test di accettazione**: Il test di accettazione garantisce che l'applicazione software soddisfi i requisiti aziendali e le esigenze degli utenti prima del rilascio.
- **Test di black box**: Il test di black box si concentra solo sul comportamento atteso dell'applicazione.
- **Test di white box**: Il test di white box prevede che il tester conosca i componenti interni e svolga test su di essi.
## Test di regressione

- **Regressione**: La regressione si riferisce a situazioni in cui nuove modifiche interrompono involontariamente funzionalità già esistenti.
- **Test di regressione**: Il test di regressione aiuta a individuare problemi di regressione. Nel test di regressione, esegui nuovamente i test funzionali su parti della tua applicazione per assicurarti che tutto funzioni ancora come previsto.
- **Strumenti per il test di regressione**: Gli strumenti che puoi usare per eseguire test di regressione includono Puppeteer, Playwright, Selenium e Cypress.
- **Tecniche per il test di regressione**: 
  - **Unit regression testing**: Qui hai un elenco di elementi che devono essere testati ogni volta che vengono implementate modifiche importanti o correzioni nell'app.
  - **Partial regression testing**: Questo prevede approcci mirati per assicurarsi che le nuove modifiche non abbiano compromesso aspetti specifici dell'applicazione.
  - **Complete regression testing**: Esegue test su tutte le funzionalità presenti nel codice. Questa è l'opzione più dettagliata e che richiede più tempo.
- **Retesting**: Il retesting viene usato per verificare problemi noti e assicurarsi che siano stati risolti. Al contrario, il test di regressione cerca problemi sconosciuti che potrebbero essere stati introdotti da modifiche recenti al codice.
## Che cos'è TypeScript

- **JavaScript**: JavaScript è un linguaggio tipizzato dinamicamente. Questo significa che le variabili possono ricevere qualsiasi valore durante l'esecuzione. La sfida di un linguaggio tipizzato dinamicamente è che la mancanza di sicurezza sui tipi può introdurre errori.

Ad esempio, anche se la tua funzione JavaScript si aspetta un array, puoi comunque chiamarla passando un numero:

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

L'output `console` per l'esempio sopra sarà `undefined`.

- **TypeScript**: TypeScript estende il linguaggio JavaScript includendo il tipaggio statico, che aiuta a individuare errori causati da incompatibilità di tipo prima di eseguire il codice.

Ad esempio, puoi definire un tipo per il parametro `array` come segue:

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

Questa definizione di tipo dice a TypeScript che il parametro `array` deve essere un array di stringhe. Poi, quando chiami `getRandomValue` e gli passi un numero, ottieni un errore chiamato errore del compilatore.

- **Compilatore**: Prima devi compilare il codice TypeScript in JavaScript normale. Quando esegui il compilatore, TypeScript valuterà il tuo codice e lancerà un errore per qualsiasi problema in cui i tipi non corrispondono.
## Tipi di dati in TypeScript

- **Tipi di dati primitivi in TypeScript**: Per i tipi di dati primitivi `string`, `null`, `undefined`, `number`, `boolean` e `bigint`, TypeScript offre le corrispondenti parole chiave di tipo.

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **Array**: Puoi definire un array di un tipo specifico con due sintassi diverse.

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **Oggetti**: Puoi definire la struttura esatta di un oggetto.

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

Se vuoi un oggetto con qualsiasi chiave, ma dove tutti i valori devono essere stringhe, ci sono due modi per definirlo:

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **Altri tipi utili in TypeScript**:
  - **`any`**: `any` indica che un valore può avere qualsiasi tipo. Comunica al compilatore di non preoccuparsi più del tipo di quella variabile.
  - **`unknown`**: `unknown` dice a TypeScript che ti interessa il tipo del valore, ma in realtà non sai quale sia. `unknown` è generalmente preferito rispetto a `any`.
  - **`void`**: Questo è un tipo speciale che userai tipicamente solo quando definisci funzioni. Le funzioni che non hanno un valore di ritorno usano un tipo di ritorno `void`.
  - **`never`**: Rappresenta un tipo che non esisterà mai.

- **Parola chiave `type`**: Questa parola chiave è simile a `const`, ma invece di dichiarare una variabile, puoi dichiarare un tipo.

È utile per dichiarare tipi personalizzati come tipi unione o tipi che includono solo valori specifici:

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**: Le interfacce sono come le classi per i tipi. Possono implementare o estendere altre interfacce, sono specificamente tipi oggetto e sono generalmente preferite a meno che tu non abbia bisogno di una funzionalità specifica offerta da una dichiarazione `type`.

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **Definizione del tipo di ritorno**: Puoi anche definire il _tipo di ritorno_ della funzione.

Nell'esempio qui sotto il valore di ritorno è definito come una stringa. Se provi a restituire qualsiasi altra cosa, TypeScript restituirà un errore del compilatore.

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```
## Generics

- **Definizione di un tipo generico**: Puoi definire un tipo generico e farvi riferimento nella tua funzione. Puoi considerarlo come un parametro speciale che fornisci a una funzione per controllare il comportamento della definizione del tipo della funzione.

Ecco un esempio di definizione di un tipo generico per una funzione:

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

La sintassi `<T>` dice a TypeScript che stai definendo un tipo generico `T` per la funzione. `T` è un nome comune per i tipi generici, ma puoi usare qualsiasi nome.

Poi, dici a TypeScript che il parametro `array` è un array di valori che corrispondono al tipo generico, e che il valore restituito è un singolo elemento di quello stesso tipo.

- **Specificare l'argomento di tipo nella chiamata di funzione**: Puoi passare un argomento di tipo a una chiamata di funzione usando le parentesi angolari tra il nome della funzione e i suoi parametri.

Ecco un esempio di passaggio di un argomento di tipo a una chiamata di funzione:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Questo dice a TypeScript che l'elemento che ti aspetti di trovare sarà un elemento di input.
## Restrizione del tipo

- **Restrizione tramite veridicità**: Nell'esempio qui sotto, ottieni un errore del compilatore quando provi a consultare la proprietà `value` di `email` perché `email` _potrebbe_ essere `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Puoi usare un'istruzione condizionale per confermare che `email` sia _truthy_ prima di consultare la proprietà:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

I controlli di veridicità possono funzionare anche nella direzione opposta:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

Lanciare un errore interrompe l'esecuzione logica di questo codice, il che significa che quando arrivi alla chiamata `console.log()`, TypeScript sa che `email` _non può_ essere `null`.

- **Concatenamento opzionale**: Il concatenamento opzionale `?.` è anch'esso una forma di restrizione del tipo, basata sullo stesso principio che la consultazione della proprietà non può avvenire se il valore `email` è `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **Operatore `typeof`**: Puoi usare una condizione per controllare il tipo della variabile usando l'operatore `typeof`.

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **Parola chiave `instanceof`**: Se l'oggetto proviene da una classe, puoi usare la parola chiave `instanceof` per restringere il tipo.

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **Casting**: Quando TypeScript non riesce a determinare automaticamente il tipo di un valore, come il risultato del metodo `request.json()` nell'esempio qui sotto, otterrai un errore del compilatore. Un modo per risolvere questo problema è effettuare il casting del tipo, ma così facendo si indebolisce la capacità di TypeScript di rilevare potenziali errori.

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **Type guard**: Invece di effettuare il casting del tipo, puoi scrivere un type guard:

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

La sintassi `user is User` indica che la tua funzione restituisce un valore booleano, che quando è true significa che il valore `user` soddisfa l'interfaccia `User`.
## `tsconfig` File

- **`tsconfig.json`**: Le impostazioni del compilatore di TypeScript si trovano in un file `tsconfig.json` nella directory principale del tuo progetto.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

Ecco le descrizioni delle opzioni del compilatore usate nell'esempio sopra:

- **`compilerOptions`**: La proprietà `compilerOptions` è dove controlli il comportamento del compilatore TypeScript.
- **`rootDir` e `outDir`**: `rootDir` e `outDir` indicano a TypeScript quale directory contiene i tuoi file sorgente e quale directory dovrebbe contenere il codice JavaScript traspilato.
- **`lib`**: La proprietà `lib` determina quali definizioni di tipo usa il compilatore e ti permette di includere il supporto per specifiche versioni di ES, il DOM e altro ancora.
- **`module` e `moduleResolution`**: `module` e `moduleResolution` lavorano insieme per gestire come il tuo pacchetto usa i moduli, sia CommonJS che ECMAScript.
- **`esModuleInterop`**: `esModuleInterop` permette una migliore interoperabilità tra CommonJS e moduli ES creando automaticamente oggetti namespace per le importazioni.
- **`skipLibCheck`**: L'opzione `skipLibCheck` salta la validazione dei file `.d.ts` che non sono referenziati da importazioni nel tuo codice.
- **`strict`**: La flag `strict` abilita diversi controlli, come assicurare la corretta gestione dei tipi nullable e avvisare quando TypeScript ricorre a `any`.
- **`exclude`**: La proprietà `exclude` a livello superiore indica al compilatore di ignorare questi file TypeScript durante la compilazione.
