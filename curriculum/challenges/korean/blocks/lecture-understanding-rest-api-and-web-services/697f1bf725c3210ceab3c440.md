# --description--

웹 서비스와 REST API는 매일 사용하는 대부분의 애플리케이션과 웹사이트를 구동합니다. 이들은 서로 다른 애플리케이션이 네트워크, 보통은 인터넷을 통해 통신하고 데이터를 공유할 수 있게 합니다.

API(Application Programming Interface)는 소프트웨어가 다른 소프트웨어와 소통하는 일반적인 방법입니다. 웹 서비스는 HTTP 같은 웹 기술을 사용해 네트워크, 보통은 인터넷을 통해 데이터를 교환하는 특정 유형의 API입니다. 그래서 모든 웹 서비스는 API이지만, 모든 API가 웹 서비스인 것은 아닙니다. REST API는 데이터 요청과 응답 방식에 관한 특정 규칙을 따르는 일반적인 웹 서비스 유형입니다.

이 수업에서는 웹 서비스가 무엇인지 높은 수준에서 다루고, 다음 수업에서는 REST API가 어떻게 작동하는지 더 자세히 설명합니다.

웹 서비스를 소프트웨어 간 다리로 생각할 수 있습니다. 사용자 인터페이스가 없으며 직접 사용하도록 설계되지 않았습니다. 대신 애플리케이션 간 데이터 교환에만 집중합니다.

시스템이 웹 서비스로 간주되려면 다음 특성을 가져야 합니다:

* **서버에서 실행됨**: 웹 서비스는 웹사이트처럼 온라인에 호스팅되어야 하지만, 사용자가 볼 수 있는 페이지 대신 데이터를 노출하거나 기능을 제공합니다
    
* **기계 간 통신**: 웹 서비스는 사람이 직접 보는 용도가 아니라 모바일 앱과 웹 앱이 데이터를 소비하거나 기능을 사용할 수 있도록 만듭니다
    
* **언어 독립적**: 웹 서비스는 Java나 C# 같은 어떤 언어로 작성될 수 있으며, 다른 언어로 작성되었더라도 어떤 애플리케이션이든 상호작용할 수 있어야 합니다
    
* **플랫폼 독립적**: 기계가 어떤 플랫폼에서 실행되는지는 중요하지 않습니다. 예를 들어, Linux에서 실행되는 웹 서비스가 Windows 기기와 통신하는 것을 막아서는 안 됩니다
    
* **표준 웹 프로토콜 사용**: 웹 서비스는 보통 HTTP/HTTPS를 사용해 인터넷에서 데이터를 전송하고, FTP는 파일 전송, SMTP는 이메일에 사용합니다
    

**REST**와 **SOAP**은 가장 잘 알려진 웹 서비스 유형이며, **XML-RPC** 같은 다른 방식도 존재합니다.

**REST**는 **표현 상태 전송**의 약자로, 오늘날 가장 널리 사용됩니다. 대부분 현대 애플리케이션은 RESTful 웹 서비스를 사용하며, 이는 간단하고 유연하며 웹과 자연스럽게 작동하기 때문입니다.

REST API는 보통 `/users` 또는 `/users/20` 같은 URL 경로로 나타내는 리소스를 노출합니다. 클라이언트가 리소스를 요청하면 서버는 그 리소스의 표현을 반환합니다. 이 표현은 보통 JSON 형식이지만 XML이나 일반 텍스트 형식일 수도 있습니다.

클라이언트는 `Accept` 헤더 같은 HTTP 헤더를 사용해 원하는 형식을 지정할 수 있습니다. 예를 들어, 클라이언트는 적절한 `Accept` 값을 설정해 JSON 응답을 요청할 수 있습니다.

```bash
Accept: application/json
```

XML을 원한다면 이렇게 지정할 수 있습니다:

```bash
Accept: application/xml
```

받는 리소스가 JSON이라면 다음과 같을 수 있습니다:

```json
{
  "id": 20,
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "createdAt": "2025-09-22T10:30:00Z"
}
```

XML이라면 다음과 같습니다:

```xml
<user>
  <id>20</id>
  <name>John Doe</name>
  <email>john@example.com</email>
  <role>admin</role>
  <createdAt>2025-09-22T10:30:00Z</createdAt>
</user>
```

SOAP는 **Simple Object Access Protocol**의 약자입니다. XML 데이터 형식만 사용하며, 더 엄격한 표준과 보안이 필요한 기업용 시스템에서 가장 흔히 사용됩니다.

SOAP 데이터 예시는 다음과 같습니다:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
   <soapenv:Body>
      <getUser>
         <userId>20</userId>
      </getUser>
   </soapenv:Body>
</soapenv:Envelope>
```

XML-RPC는 요청과 응답을 인코딩하는 데 XML을 사용합니다. 이로 인해 가볍지만 REST보다 유연성이 떨어집니다.

UDDI는 데이터 교환 서비스가 아니라, 기업이 제공하는 웹 서비스를 목록화하고 발견하며 설명할 수 있는 XML 기반 디렉터리입니다.
