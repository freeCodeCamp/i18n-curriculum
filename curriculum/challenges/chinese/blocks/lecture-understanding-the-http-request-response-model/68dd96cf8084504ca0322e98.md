# --description--

超文本传输协议 `request-response` 模型是驱动万维网（WWW）的基本通信模式。它包含许多组件，使其高效，因此我们都能访问网络上的资料。

让我们来看看超文本传输协议 `request-response model` 是什么，组成它的 `component`，以及这些 `component` 如何协同工作以提升驱动网络的效率。

超文本传输协议请求-响应模型是在网页交互中一种通信模式，其中客户（通常是网页浏览器）向服务器发送一个 **HTTP request**，服务器处理该请求并返回一个 **HTTP response**。

以下是构成超文本传输协议请求-响应模型的组件：

* **客户**：发送超文本传输协议请求到服务器并解释收到的超文本传输协议响应的浏览器、移动应用或任何其他软件。
    
* **请求**：客户发送给服务器的消息。它包括方法、URL、头部和可选的体。
    
* **超文本传输协议 方法**：要执行的操作——`GET` 用于检索数据，`POST` 用于发送数据，`PATCH` 和 `PUT` 用于修改数据，`DELETE` 用于删除数据。
    
* **URL（统一资源定位符）**：从服务器请求的资源。
    
* **头部**：随请求或响应发送的元数据。例如，`Content-Type` 和 `Authorization`。
    
* **体**：随 `POST`、`PUT`、`PATCH`、`DELETE` 请求发送的数据，通常为 JSON 或表单数据形式。
    
* **服务器**：处理请求并生成响应的后端系统。它可以用任何支持服务器端编程的编程语言编写。
    
* **响应**：服务器返回的消息。它包含状态码、头部和可选的体。
    
* **状态码或响应码**：表示成功、误差或重定向的数字代码。
    
* **响应正文**：实际返回的数据，通常为超文本标记语言、JSON 或 XML 形式。

那么所有这些组件如何在超文本传输协议请求-响应周期中协同工作？

1. **客户发起请求**
    
    * 浏览器、移动应用或 API 客户发送一个旨在到达服务器的超文本传输协议请求。
        
    * 请求包括像 `GET`、`POST`、`PUT`、`PATCH`、`DELETE` 这样的方法、URL、头部，有时还包括体。
        
2. **请求已发送到服务器**
    
    * 该 `request` 通过 `internet` 传输到 `server`，这是一个用 `programming language` 编写的后端系统。
        
    * 服务器接收请求并从中提取相关信息。
        
3. **服务器处理请求**
    
    * 服务器查看 `request` 方法和 URL。
        
    * 它可能需要通过诸如 `authorization` 之类的头部来验证认证。
        
    * 如果需要，它会与数据库交互以获取或修改数据。
        
4. **服务器准备一个响应**
    
    * 服务器生成带有类似 200、404、500 等状态码的响应。
        
    * 响应包括头部，例如，`Content-Type: application/json`。
        
    * 如果适用，将添加 JSON、超文本标记语言 或 XML 格式的响应正文。
        
5. **响应被发送到客户**
    
    * 服务器通过互联网发送响应。
        
    * 客户接收 `Response` 并处理 `data`。
        
6. **客户处理器处理响应**
    
    * 如果响应成功（状态码 200），客户将显示数据，例如，在网页上渲染数据。
        
    * 如果发生错误（`404 Not Found`、`500 Internal Server Error` 等），客户可能会显示错误信息。

此周期会针对客户发出的每个新建请求重复执行。
