# --description--

소프트웨어에서 의존성이란 애플리케이션 내 한 컴포넌트나 모듈이 제대로 작동하려면 다른 컴포넌트나 모듈에 의존하는 경우를 말합니다. 의존성은 개발자가 다른 사람이 만든 미리 만들어진 함수나 도구를 사용할 수 있게 해주기 때문에 소프트웨어 애플리케이션에서 흔히 발생합니다.

React 애플리케이션을 구축하려면 React 의존성을 설치해야 합니다. 이 의존성이 없으면 코드가 제대로 작동하지 않고 애플리케이션에 여러 오류가 표시됩니다.

더 복잡한 프로젝트를 작업할 때는 여러 의존성에 의존해야 하는 경우가 많습니다. 프로젝트 내 의존성 집합이 잘 관리되거나 정의되지 않으면 의존성 지옥이라고 불리는 상황이 발생할 수 있습니다.

프로젝트에서 소프트웨어 의존성을 관리하려면 패키지 관리자를 사용해야 합니다. 패키지 관리자는 의존성 설치, 업데이트, 제거에 사용하는 도구입니다. JavaScript, Python, Ruby, Java 같은 인기 있는 프로그래밍 언어들은 모두 패키지 관리자를 사용합니다.

이전 수업에서 npm이라는 인기 있는 패키지 관리자를 간단히 소개했습니다.

Vite와 npm을 사용해 새 React 프로젝트를 만들려면 다음 명령어를 실행하세요:

```sh
npm create vite@latest my-react-app -- --template react
```

이전 수업에서 기억하듯이, 이 명령어는 새 React 애플리케이션을 시작하는 데 필요한 모든 보일러플레이트 코드를 생성합니다. 애플리케이션을 실행하기 전에 `npm install` 또는 간단히 `npm i` 명령어를 실행해 의존성을 설치해야 합니다.

프로젝트 루트 디렉터리에 있는 `package.json` 파일에서 모든 의존성을 확인할 수 있습니다.

`package.json` 파일은 프로젝트의 이름, 버전, 의존성을 포함한 메타데이터를 담고 있는 핵심 구성 파일입니다. 또한 스크립트, 라이선스 정보, 프로젝트와 의존성 관리를 돕는 기타 설정도 정의합니다.

의존성을 설치하면 `node_modules` 폴더가 프로젝트에 추가됩니다.

`node_modules` 폴더는 프로젝트에 필요한 모든 패키지와 라이브러리가 저장되는 곳입니다. 이 폴더에는 `package.json` 파일에 나열된 의존성의 실제 코드가 들어 있으며, 프로젝트의 직접 의존성과 그 의존성의 의존성 모두 포함됩니다.

React 프로젝트에 필요한 두 가지 핵심 의존성은 `react`과 `react-dom` 패키지입니다:

```json
"dependencies": {
  "react": "^18.3.1",
  "react-dom": "^18.3.1"
}
```

`package.json` 파일에는 현재 사용 중인 해당 패키지들의 버전이 나열됩니다. 로컬에서 패키지를 업데이트하려면 `npm update` 명령어를 실행하세요. 또는 모든 패키지를 전역으로 업데이트하려면 `npm update -g` 명령어를 실행하면 됩니다.

`package.json` 파일 외에도 `package-lock.json` 파일도 있습니다. 이 파일은 프로젝트가 사용하는 모든 패키지의 정확한 버전을 고정합니다. 패키지를 업데이트하면 잠금 파일에도 새 버전이 반영됩니다.

`package.json` 파일의 또 다른 중요한 부분은 개발 의존성입니다:

```json
"devDependencies": {
  "@eslint/js": "^9.17.0",
  "@types/react": "^18.3.18",
  "@types/react-dom": "^18.3.5",
  "@vitejs/plugin-react": "^4.3.4",
  "eslint": "^9.17.0",
  "eslint-plugin-react": "^7.37.2",
  "eslint-plugin-react-hooks": "^5.0.0",
  "eslint-plugin-react-refresh": "^0.4.16",
  "globals": "^15.14.0",
  "vite": "^6.0.5"
}
```

개발 의존성은 개발할 때만 사용하고 실제 운영 환경에서는 사용하지 않는 패키지입니다. 예를 들어 Jest 같은 테스트 라이브러리가 있습니다. Jest는 프로젝트를 로컬에서 테스트할 때만 사용하므로 개발 의존성으로 설치하며, 운영 환경에서는 필요하지 않습니다.

이번 수업 대부분은 npm에 집중했지만 yarn과 pnpm 같은 다른 패키지 관리자도 있습니다. 그렇다면 어떤 패키지 관리자를 프로젝트에 사용해야 할까요?

간단한 답은 상황에 따라 다릅니다.

기존 팀 프로젝트에 참여한다면 이미 결정된 패키지 관리자를 사용하게 됩니다. 새 프로젝트를 처음부터 만든다면 각 관리자의 장단점을 조사해 필요에 맞는 것을 선택해야 합니다.
