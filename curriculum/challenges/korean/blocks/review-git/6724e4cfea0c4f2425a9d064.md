## 버전 관리 소개

- **정의**: 버전 관리 시스템은 프로젝트의 변경 사항을 추적하고 관리할 수 있게 해줍니다. 소프트웨어에서 사용되는 버전 관리 시스템의 예로는 Git, SVN, Mercurial이 있습니다.

## 클라우드 기반 버전 관리 제공자

- **클라우드 기반 버전 관리 제공자 목록**: GitHub와 GitLab은 소프트웨어 팀이 협업하고 저장소를 관리할 수 있게 해주는 인기 있는 클라우드 기반 버전 관리 제공자입니다.

## Git 설치 및 설정

- **Git 설치**: 터미널에서 다음 명령어를 실행하여 Git이 이미 설치되어 있는지 확인할 수 있습니다:

```sh
git --version
```

버전 번호가 보이면 Git이 설치된 것입니다. 그렇지 않으면 설치해야 합니다.

리눅스 시스템에서는 대부분의 배포판에 Git이 기본 설치되어 있습니다. 만약 Git이 설치되어 있지 않다면, `sudo apt-get install git` 또는 `sudo pacman -S git` 같은 패키지 관리자 명령어로 설치할 수 있습니다.

Mac 사용자는 `brew install git`를 통해 Homebrew로 Git을 설치하거나 Git 웹사이트에서 실행 파일 설치 프로그램을 다운로드할 수 있습니다.

Windows 사용자는 Git 웹사이트에서 실행 파일 설치 프로그램을 다운로드할 수 있습니다. 또는 Chocolatey를 설정했다면 PowerShell에서 `choco install git.install` 명령어를 실행할 수 있습니다. Windows에서는 Unix와 유사한 셸 환경을 위해 Git Bash도 다운로드하는 것이 좋습니다.

설치가 제대로 되었는지 확인하려면 터미널에서 다시 `git --version` 명령어를 실행하세요.

- **Git 설정**: `git config`는 Git이 시스템에서 어떻게 작동할지 설정하는 구성 변수를 설정하는 데 사용됩니다. 현재 설정 변수와 저장 위치를 보려면 다음 명령어를 실행하세요:

```sh
git config --list --show-origin
```

Git을 처음 설치했다면 지금은 시스템 수준 구성 설정만 보일 것입니다.

사용자 이름을 설정하려면 다음 명령어를 실행하세요:

```sh
git config --global user.name "Jane Doe"
```

여기서 `--global` 플래그는 시스템 내 모든 Git 프로젝트에 대해 사용자 이름을 설정하는 데 사용됩니다. 특정 프로젝트에서 사용자 이름을 재정의하려면 `--global` 플래그 없이 해당 프로젝트 디렉터리에서 명령어를 실행하면 됩니다.

사용자 이메일 주소를 설정하려면 다음 명령어를 실행하세요:

```sh
git config --global user.email janedoe@example.com
```

Git이 사용할 선호하는 편집기를 설정할 수도 있습니다. 예를 들어 Emacs를 선호 편집기로 설정하는 방법은 다음과 같습니다:

```sh
git config --global core.editor emacs
```

선호 편집기를 설정하지 않으면 Git은 시스템 기본 편집기를 사용합니다.

## 오픈 소스와 클로즈드 소스 소프트웨어

- **정의**: "오픈 소스"는 사람들이 게시한 코드를 보고, 변경을 제안하고, 문제를 보고하며, 수정된 버전을 실행할 수 있다는 뜻입니다. "클로즈드 소스"는 프로젝트를 볼 수 있고 상호작용할 수 있는 사람이 명시적으로 허가한 사람들뿐이라는 뜻입니다.

## GitHub

- **정의**: GitHub는 버전 관리된 프로젝트를 "저장소"라는 공간에 저장하고, 해당 프로젝트와 함께 사용할 협업 기능을 제공하는 클라우드 기반 솔루션입니다.
- **GitHub CLI**: 이 도구는 명령줄을 벗어나지 않고 GitHub 관련 작업을 수행하는 데 사용됩니다. 설치되어 있지 않다면 GitHub 문서에서 설치 방법을 확인할 수 있으며, 시스템 패키지 관리자에서 사용할 수 있을 것입니다.
- **GitHub Pages**: GitHub Pages는 백엔드 서버가 필요 없는 정적 사이트나 애플리케이션을 배포하는 옵션입니다. 즉, 클라이언트 측에서만 실행되거나 사용자의 브라우저에서 실행되는 애플리케이션을 완전히 배포할 수 있습니다.
- **GitHub Actions**: GitHub Actions는 GitHub 저장소 내에서 직접 워크플로를 자동화할 수 있는 기능으로, 코드 빌드, 테스트, 배포를 포함합니다.

## 자주 사용하는 Git 명령어

- **`git init`**: 이 명령어는 빈 Git 저장소를 초기화하여 Git이 프로젝트 변경 사항을 추적할 수 있게 합니다. 빈 Git 저장소를 초기화하면 새로운 `.git` 숨김 디렉터리가 추가됩니다. 이 `.git` 디렉터리는 Git이 프로젝트를 관리하는 데 중요한 정보를 담고 있습니다.
- **`git status`**: 이 명령어는 작업 디렉터리의 현재 상태를 보여줍니다. 워크플로에서 자주 사용할 것입니다.
- **`git add`**: 이 명령어는 변경 사항을 스테이징합니다. 스테이징 영역에 있는 모든 항목은 다음 커밋에 포함됩니다. 모든 스테이징되지 않은 변경 사항을 스테이징하려면 `git add .`을 사용할 수 있습니다. 점(`.`)은 현재 디렉터리의 별칭입니다.
- **`git commit`**: 이 명령어는 변경 사항을 커밋합니다. 커밋은 특정 시점의 프로젝트 상태 스냅샷입니다. `git commit`을 실행하면 Git 설정에서 지정한 선호 편집기가 열립니다. 편집기가 열리면 변경 사항에 대한 자세한 메시지를 작성할 수 있습니다. `git commit -m` 명령어를 사용하면 짧은 메시지를 제공할 수도 있습니다. 예시는 다음과 같습니다:

```sh
git commit -m "short message goes here"
```

- **`git log`**: 이 명령어는 작성자, 커밋 날짜, 커밋 메시지, 커밋 해시 같은 유용한 정보를 포함해 이전 커밋들을 모두 나열합니다. 커밋 해시는 커밋을 고유하게 식별하는 긴 문자열입니다.
- **`git remote add`**: 이 명령어는 원격 저장소와의 연결을 설정하는 데 사용됩니다.
- **`git push`**: 이 명령어는 변경 사항을 원격 저장소에 푸시합니다.
- **`git pull`**: 이 명령어는 원격 저장소에서 최신 변경 사항을 로컬 저장소로 가져옵니다.
- **`git clone`**: 이 명령어는 저장소를 복제합니다. 즉, 저장소의 복사본을 로컬 장치에 갖게 됩니다. 이 복사본에는 저장소 기록, 모든 파일/폴더, 커밋이 포함됩니다.
- **`git remote -v`**: 이 명령어는 로컬 Git 저장소와 연결된 원격 저장소 목록을 보여줍니다.
- **`git branch`**: 이 명령어는 모든 로컬 브랜치를 나열합니다.
- **`git fetch upstream`**: 이 명령어는 업스트림 원격(원본 저장소)에 있는 최신 변경 사항을 가져오라고 Git에 지시합니다.
- **`git merge upstream/main`**: 이 명령어는 업스트림 원격의 `main` 브랜치에서 최신 변경 사항을 현재 브랜치에 병합하라고 Git에 지시합니다.
- **`git reset`**: 이 명령어는 브랜치의 현재 상태를 재설정할 수 있습니다. `--hard` 플래그를 전달하면 로컬 파일이 브랜치 상태와 강제로 일치하도록 합니다. 이렇게 하면 작업할 깨끗한 상태를 확보할 수 있습니다.
- **`git rebase`**: Git에서 리베이스는 한 브랜치의 커밋 수열을 다른 브랜치 위로 이동하거나 결합하는 방법입니다.

## 브랜치 작업

- **정의**: Git의 브랜치는 변경 사항을 만들 수 있는 별도의 작업 공간입니다. `main` 브랜치는 실제 애플리케이션에서 주로 기본 또는 프로덕션 브랜치를 나타냅니다. 개발자 팀은 새로운 기능과 버그 수정을 위해 여러 브랜치를 만들고, 그 변경 사항을 `main` 브랜치에 병합합니다.
- **새 브랜치 만들기**: 새 브랜치를 만들려면 다음 명령어를 실행하세요:

```sh
git branch feature
```

해당 브랜치로 전환하려면 다음 명령어를 실행하세요:

```sh
git checkout feature
```

대부분 개발자는 브랜치를 만들고 전환하는 단축 명령어를 사용합니다. 다음과 같습니다:

```sh
git checkout -b new-branch-name
```

새롭고 대안적인 명령어는 `git switch`입니다. 새 브랜치를 만들고 전환하는 예시는 다음과 같습니다:

```sh
git switch -c new-branch-name
```

- **브랜칭 전략**: `main` 브랜치는 기본 브랜치이며 보통 안정적입니다. 버그 수정, 새로운 기능, 기타 작업을 위해 이 브랜치에서 분기하는 것이 가장 좋습니다.
- **병합 충돌**: Git이 서로 다른 브랜치의 변경 사항을 자동으로 병합하려 할 때 어떤 변경 사항을 유지할지 결정하지 못하면 발생합니다. 보통 같은 파일 부분에 충돌하는 변경 사항이 있을 때 발생합니다.

## Git이 추적하는 파일의 다섯 가지 상태

- **"Untracked"**: 이 파일은 저장소에 새로 추가된 파일이며 Git이 이전에 "본" 적이 없습니다.
- **"Modified"**: 이 파일은 이전 커밋에 존재했으며 아직 커밋되지 않은 변경 사항이 있습니다.
- **"Ignored"**: Git에서는 무시된 파일을 보지 못할 가능성이 높지만 IDE에서는 표시할 수 있습니다. 무시된 파일은 일반적으로 `.gitignore` 파일에 포함되어 Git 작업에서 제외됩니다.
- **"Deleted"**: 삭제된 파일은 추적되지 않은 파일과 반대입니다. 이전에 존재했지만 삭제된 파일입니다.
- **"Renamed"**: 이름이 변경된 파일은 내용은 변경되지 않았지만 파일 이름이나 위치가 수정된 파일입니다. 경우에 따라 약간의 변경이 있어도 파일이 이름 변경된 것으로 간주될 수 있습니다.

## `.gitignore` 파일

- **정의**: `.gitignore` 파일은 Git 작업과 관련된 특별한 유형의 파일입니다. 이름에서 알 수 있듯이 Git에게 무시할 항목을 알려주는 데 사용되며, 일반적인 용도입니다. 하지만 실제로는 Git이 파일 추적을 중지하도록 지시합니다.

## 저장소 작업

- **정의**: 저장소는 프로젝트를 위한 컨테이너와 같습니다. 앱 작업 시 해당 앱의 파일을 저장소에 함께 보관합니다. 저장소는 컴퓨터 로컬에 있거나 GitHub 같은 서비스의 원격에 있을 수 있습니다.
- **공개 저장소와 비공개 저장소**: 공개 저장소는 누구나 보고 다운로드할 수 있습니다. 비공개 저장소는 본인과 명시적으로 접근 권한을 부여한 사람만 접근할 수 있습니다.
- **GitHub에서 저장소 만들기**: GitHub에서 새 저장소를 만들려면 `"New Repository"` 버튼을 클릭하고 GitHub UI를 따라 새 저장소 설정을 진행하세요.
- **로컬 저장소를 GitHub에 푸시하기**: 컴퓨터에 로컬 프로젝트가 있다면 해당 저장소를 GitHub에 푸시할 수 있습니다. 과정은 다음과 같습니다:

1. 프로젝트 디렉터리에서 빈 git 저장소를 초기화합니다 (`git init`).
2. 프로젝트를 변경합니다.
3. `git status` 명령어를 실행해 Git이 추적하는 모든 변경 사항을 확인합니다.
4. 변경 사항을 스테이징합니다 (`git add`).
5. 변경 사항을 커밋합니다 (`git commit`).
6. 원격 연결을 설정합니다 (`git remote add`).
7. 변경 사항을 GitHub에 푸시합니다 (`git push`).

## 풀 리퀘스트

- **풀 리퀘스트**: 풀 리퀘스트는 자신의 브랜치에서 대상 브랜치로 변경 사항을 병합해 달라는 요청입니다. 코드 변경 사항을 프로젝트에 기여할 때 사용하는 흐름입니다. 프로젝트 유지 관리자가 변경 사항을 검토할 수 있게 하며, 댓글을 남기거나 질문하고 수정 제안을 할 수 있습니다. 검토가 완료되면 승인되어 메인 브랜치에 병합됩니다.

## 다른 저장소에 기여하기

- **과정**: 기여할 수 있는 수천 개의 프로젝트가 있습니다. 다른 저장소에 기여하는 기본 과정은 다음과 같습니다:

1. 기여 문서를 읽습니다.
2. 작업할 이슈를 찾습니다.
3. 저장소를 포크합니다.
4. 포크한 저장소를 복제합니다.
5. 새 브랜치를 만듭니다.
6. 이슈에 따라 변경 사항을 만듭니다.
7. PR(풀 리퀘스트)을 만듭니다.
8. PR 검토를 기다립니다.

## SSH 및 GPG 키 작업

- **GPG 키**: GPG 또는 Gnu Privacy Guard 키는 보통 파일이나 커밋에 서명하는 데 사용됩니다. 다른 사람이 공개 GPG 키를 사용해 파일 서명이 본인의 키에서 왔고 내용이 변경되지 않았음을 확인할 수 있습니다.

GPG 키를 생성하려면 다음을 실행하세요:

```sh
gpg --full-generate-key
```

- **SSH 키**: SSH 또는 Secure SHell 키는 보통 `ssh` 유틸리티를 통해 서버에 원격 연결을 인증하는 데 사용됩니다. SSH 키로 커밋에 서명할 수도 있습니다.

SSH 키를 생성하려면 다음을 실행하세요:

```sh
ssh-keygen -t ed25519 -C "your_email@example.com"
```

`ed25519`는 최신 공개키 서명 알고리즘입니다.

- **GPG 키로 커밋 서명하기**: GPG 키로 커밋에 서명하려면 개인 키가 아닌 공개 키를 GitHub 계정에 업로드해야 합니다. 공개 키 목록을 보려면 다음을 실행하세요:

```sh
gpg --list-secret-keys --keyid-format=long
```

그다음 공개 키를 얻으려면 다음을 사용하세요:

```sh
gpg --armor --export "<key id>"
```

목록에서 얻은 짧은 ID를 사용해 Git 서명 키로 설정하려면 다음 명령어를 실행하세요:

```sh
git config --global user.signingkey <your_gpg_key_id>
```

그다음 `-S` 플래그를 `git commit` 명령어에 전달해 특정 커밋에 서명할 수 있습니다. 암호를 입력해야 합니다. 또는 모든 커밋에 자동으로 서명하려면 autosign 구성을 `true`로 설정하세요:

```sh
git config --global commit.gpgsign true
```

- **SSH 키로 커밋 서명하기**: SSH 키로 서명하는 것은 GitHub에서 비교적 새로운 기능입니다. 먼저 키를 GitHub 계정에 업로드해야 합니다. 그다음 Git 서명 모드를 SSH로 설정하세요:

```sh
git config --global gpg.format ssh
```

서명 키를 설정할 때는 ID 대신 파일 경로를 전달하세요:

```sh
git config --global user.signingkey <path_to_your_ssh_keys>
```
