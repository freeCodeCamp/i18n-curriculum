# --description--

Il versionamento semantico, o SemVer, è un sistema che assegna ai numeri di versione un significato chiaro, così gli sviluppatori possono capire rapidamente che tipo di modifiche sono state apportate al software.

SemVer usa questo formato:

```bash
MAJOR.MINOR.PATCH
```

Per esempio, la libreria popolare `lodash` è attualmente alla versione `4.17.21`.

Ecco cosa significa ogni parte:

* MAJOR — Grandi cambiamenti che rompono il codice esistente. Esempio: rimuovere o rinominare una funzione da cui il codice esistente potrebbe dipendere.
    
* MINOR — Nuove funzionalità che non rompono il codice esistente. Esempio: aggiungere una nuova funzione o un parametro opzionale.
    
* PATCH — Piccole correzioni e miglioramenti. Esempio: correggere un bug o un errore di battitura senza cambiare il funzionamento.
    

<img src="https://cdn.freecodecamp.org/curriculum/lecture-transcripts/what-is-semantic-versioning-1.png" alt="Nel numero di versione 4.17.21, 4 è la versione major, cambia per modifiche che rompono il codice, 17 è la versione minor, cambia per modifiche non distruttive, e 21 è la patch, cambia per correzioni di bug e piccole funzionalità">

Supponiamo che tu abbia un pacchetto chiamato `my-package`. Ecco un esempio di come il suo numero di versione potrebbe cambiare nel tempo usando il versionamento semantico:

| Versione | Cosa è cambiato |
| --- | --- |
| 1.0.0 | Prima release stabile |
| 1.1.0 | Nuova funzionalità opzionale |
| 1.1.1 | Piccola correzione di bug o funzionalità |
| 2.0.0 | Introdotte modifiche che rompono il codice |

Ma c’è di più nei numeri di versione in npm. Quando installi un pacchetto, la versione nel tuo file `package.json` potrebbe apparire così:

```json
"lodash": "^4.17.21" 
```

Hai già imparato che `4` è la versione major, `17` è la minor e `21` è la patch. Ma cosa significa il caret (`^`) davanti?

Il simbolo caret permette a npm di installare automaticamente versioni minor e patch più recenti (`4.x.x`), ma non versioni major più recenti (`5.x.x`), perché gli aggiornamenti major possono includere modifiche che rompono il codice.

Quando la versione major è `0`, il caret diventa più restrittivo. Per esempio, `^0.2.3` permetterebbe aggiornamenti come `0.2.4`, ma non `0.3.0`, dato che le versioni major che iniziano con `0` sono considerate meno stabili.

Il caret non è l’unico simbolo che `npm` e altri gestori di pacchetti JavaScript usano per gestire le versioni delle dipendenze.

C’è il simbolo tilde (`~`) che permette aggiornamenti solo alla versione patch. Quindi, `~1.2.3` accetterebbe `1.2.4` e `1.2.9`, ma mai `1.3.0`.

Puoi anche usare i simboli maggiore (`>`), minore (`<`) e i loro equivalenti "o uguale" per accettare versioni in un intervallo. Per esempio, `>=1.2.3 <2.0.0` significa qualsiasi versione da `1.2.3` fino a, ma non inclusa, `2.0.0`.

Il simbolo asterisco (`*`) corrisponde a qualsiasi versione, quindi accetta tutto. Questo è ideale solo per test e non è consigliato per la produzione.

Infine, puoi bloccare una versione esatta non usando alcun simbolo prima del numero di versione. Per esempio, la versione `1.2.3` accetta solo `1.2.3`.

Ecco una tabella riassuntiva veloce:

| Simbolo | Significato | Esempio di intervallo |
| --- | --- | --- |
| `^` | Permette aggiornamenti minor + patch | `^1.2.3` → `1.x.x` |
| `~` | Permette aggiornamenti solo patch | `~1.2.3` → `1.2.x` |
| `>`, `<`, `>=`, `<=` | Intervallo personalizzato | `>=1.2.3` o `<2.0.0` |
| **\*** | Qualsiasi versione | tutte le versioni |
| nessuno | Versione esatta | `1.2.3` |
