# --description--

Quando visiti un sito web, la prima cosa che fai è digitare un indirizzo web nel browser.

Per esempio, per andare su freeCodeCamp, inseriresti `freecodecamp.org` nella barra degli indirizzi del browser.

Questo è noto come nome di dominio. È un identificatore unico e leggibile dall’uomo per quella risorsa su internet.

I nomi di dominio sono strettamente legati al concetto di DNS, che è l’argomento principale di questa lezione. Iniziamo parlando un po’ della gerarchia dei nomi di dominio, così puoi capire come funziona un DNS dietro le quinte.

La gerarchia dei nomi di dominio si legge da destra a sinistra, passando dalla categoria più generale a quella più specifica.

La struttura principale di un nome di dominio è, da destra a sinistra:

* **Dominio di primo livello (TLD):** l’ultima parte del nome di dominio, situata a destra. Nel caso di `freecodecamp.org`, sarebbe `.org`. Altri esempi includono `.com` e `.edu`.
    
* **Dominio di secondo livello (SLD):** il nome registrato dalla persona o organizzazione. Sarebbe `freecodecamp` in `freecodecamp.org`. Questo è ciò che di solito noti per primo quando vedi un indirizzo web.
    
* **Dominio di terzo livello (sottodominio):** usato per identificare una sezione specifica del dominio principale. Per esempio, `www`, che sta per “World Wide Web.” Oggi i server web moderni gestiscono automaticamente il routing dei domini, quindi puoi accedere a un sito web con o senza inserire `www`. Altri esempi di sottodomini includono `blog.`, `shop.`, `support.` e `api.`.
    

I nomi di dominio sono molto utili per noi umani perché sono facili da ricordare e digitare nel browser. Tuttavia, come sai, i computer lavorano con i numeri. Il tuo browser deve trasformare il nome di dominio che hai inserito in qualcosa noto come indirizzo IP per trovare il sito web che stai cercando.

Un **indirizzo IP** è una sequenza unica di numeri che identifica un computer su una rete, come internet.

Gli indirizzi IP sono essenziali per permettere ai client di inviare richieste ai server corretti e ricevere le risposte appropriate. Un esempio di indirizzo IPv4 è `127.0.0.1`. L’indirizzo IP equivalente in uno standard più recente chiamato IPv6 sarebbe `0:0:0:0:0:0:0:1`.

C’è molto da imparare sugli indirizzi IP, e potresti volerli esplorare più a fondo mentre continui a studiare. Ma per ora, devi solo capire che sono gli indirizzi numerici unici usati per identificare i computer su una rete.

Partendo dai concetti di nomi di dominio e indirizzi IP, ora possiamo arrivare al concetto di DNS.

DNS sta per **Domain Name System**. Viene solitamente chiamato la “rubrica telefonica” di internet.

Il DNS traduce il nome di dominio che scrivi nel browser nel suo indirizzo IP corrispondente. È un sistema di denominazione gerarchico e decentralizzato per computer e dispositivi connessi a internet o a una rete specifica.

Quando l’utente inserisce un nome di dominio nel browser, come `freecodecamp.org`, il DNS associa quel nome all’indirizzo IP del server corrispondente. Una volta identificato il server, la richiesta del client viene inviata al server. Il server gestisce quella richiesta in base a come è stato programmato e infine invia una risposta al client. Questo completa il ciclo richiesta-risposta.

Tutto questo è possibile grazie al DNS che ha effettuato quella conversione iniziale all’indirizzo IP obiettivo.

Addentriamoci nei dettagli del processo di risoluzione DNS.

## Il processo di risoluzione DNS

* La risoluzione DNS funziona inviando una query attraverso una catena di server DNS.

* Quando tu (l’utente) inserisci un nome di dominio nel browser, il **client** invia una richiesta a un server DNS locale. Questo server DNS locale è noto come **Recursive Resolver**. Di solito è gestito dal tuo provider di servizi internet (ISP).
    
* Il Recursive Resolver inoltra quella richiesta ai **Root Server**. Questi sono server generali che non hanno informazioni dirette sugli indirizzi IP. Funzionano indirizzando il Recursive Resolver verso un server più specifico, precisamente verso il server del dominio di primo livello (TLD) per il particolare dominio di primo livello della richiesta. Per esempio, il server responsabile di tutti i domini `.org`.
    
* Il Recursive Resolver quindi interroga il **server del dominio di primo livello (TLD)**. Questo server restituisce l’indirizzo del server autorevole specifico che contiene i record per il dominio richiesto (per esempio, `freecodecamp.org`).
    
* Il Recursive Resolver avvia una query finale al corrispondente **server autorevole**. Se il **record DNS** viene trovato, viene restituito al Recursive Resolver.
    
* Quando il Recursive Resolver riceve l’indirizzo IP, lo consegna al client, così puoi iniziare a navigare il sito web.

Il server autorevole restituisce **record DNS**, non solo indirizzi IP. I record DNS sono elementi di testo strutturati che contengono informazioni importanti su un dominio, come informazioni sui sottodomini, per quanto tempo le informazioni devono essere memorizzate nella cache e informazioni necessarie per eseguire il servizio richiesto.

Per ottimizzare il processo, il Recursive Resolver memorizza temporaneamente nella cache il record DNS per velocizzare il processo. In questo modo, non farai la stessa richiesta ripetutamente in un breve periodo di tempo. Questo rende il processo complessivo molto più efficiente.

Come puoi vedere, i server DNS sono essenziali perché forniscono il passaggio iniziale e critico di tradurre i nomi di dominio in indirizzi IP. Questo processo di traduzione permette a ogni client di localizzare e connettersi al server corretto. Senza questo sistema di directory, il modello client-server, la spina dorsale del web moderno, non funzionerebbe.
