# --description--

每当你访问一个网站或使用 API 时，你的浏览器和服务器都会不断地相互通信。

这段对话中最重要的部分之一就是 HTTP 响应状态码。

它们是简短的数字消息——比如 200、404 或 500——告诉客户（最终用户的设备，比如笔记本或手机）请求之后发生了什么。

你已经学过其中一些了，但让我们按类别来拆解它们，这样你就能准确回忆它们的含义以及何时使用。

## 1xx – 信息性响应

以 1 开头的状态码在日常开发中很少见，但它们表示请求已被接收且处理正在继续，比如上传文件时。

常见示例：

* 100 Continue – 服务器告诉客户，“好的，你可以发送剩余的请求内容。”
* 101 Switching Protocols – 客户发送了 `Upgrade` 头部，因此服务器正在切换协议。你最常在使用 Websockets 时看到这个。

## 2xx – 成功响应

这些表示请求已成功接收、理解并处理。

* 200 OK – 请求成功。这是最常见的成功响应。
* 201 Created – 新资源已成功创建，例如注册新用户后。
* 204 No Content – 请求成功，但没有响应正文返回。通常用于 `DELETE` 请求。

## 3xx – 重定向消息

这些告诉客户：“你要找的资源在别处。”

* 301 Moved Permanently – 资源有了新的永久 URL。
* 302 Found – 资源暂时在不同的 URL。
* 304 Not Modified – 用于缓存。告诉客户内容没有变化，可以使用缓存版本。

## 4xx – 客户端错误

当客户端发送无效请求时会发生这些错误。

* 400 Bad Request – 请求格式错误或无效。
* 401 Unauthorized – 需要认证（你需要登录）。
* 403 Forbidden – 认证成功，但你没有权限。
* 404 Not Found – 请求的资源不存在。没错，这是网络上最具标志性的错误。

## 5xx – 服务器错误

这些表示服务器端出了问题。

* 500 Internal Server Error – 通用的“出了点问题”消息。
* 502 Bad Gateway – 作为代理的服务器从另一个服务器收到无效响应。
* 503 Service Unavailable – 服务器暂时过载或正在维护。

现在你已经熟悉了一些最常见的 HTTP 状态码，让我们来看一些 Express 中的实际示例。

假设你正在创建一个 API。如果用户发送了无效数据，你可以用 400 状态码响应，表示“错误请求”。你还可以将字串 "Bad Request" 发送回客户，让它清楚 400 响应码的含义：

```js
res.status(400).send("Bad Request");
```

如果一切顺利，你可以发送 200 响应。如果你愿意，可以包含一条消息。这里我们发送了一些 JSON，带有简单的“Success”消息：

```js
res.status(200).json({ message: "Success" });
```

如果服务器崩溃了，你可以发送 500 状态码，表示发生了内部服务器错误，并附上关于错误的清晰消息：

```js
res.status(500).send("Internal Server Error");
```

这些状态码不仅仅是给机器看的。它们帮助开发者调试问题，也帮助用户（和其他开发者）理解发生了什么。
