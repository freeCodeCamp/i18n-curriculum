# --description--

Sie haben bereits gelernt, wie Sie Änderungen committen und in Ihr Remote-Repository pushen, aber das alles geschah auf Ihrem Standard-`main`-Branch. Das funktioniert bei kleineren Projekten, aber wenn Sie zu anderen Open-Source-Projekten beitragen oder Ihr Projekt wächst, sind Branches unverzichtbar.

Wenn Sie zum Beispiel ein Repository forken und Commits auf dem `main`-Branch Ihres Forks machen, können Sie Änderungen in Ihrem Fork haben, die das Original-Repository nicht akzeptiert hat. Das macht es sehr schwierig, Ihren Branch mit den akzeptierten Änderungen aus dem Fork zu aktualisieren. Oder wenn Sie an Ihrem eigenen Projekt arbeiten, wird das Wechseln zwischen verschiedenen Aufgaben viel schwieriger, wenn Sie keine Branches verwenden. Stattdessen müssen Sie manuell nachverfolgen, welche Änderungen zu welcher Funktion oder Fehlerbehebung gehören.

Aber was genau ist ein Branch? Betrachten Sie Ihren `main`-Branch als eine Autobahn, wobei jeder Commit ein Auto ist, das auf die Autobahn fährt. Vielleicht möchten Sie sich etwas ansehen, wie eine neue Funktion erstellen, und müssen dafür kurz von der Autobahn abfahren.

Ein Branch ist im Grunde eine Abzweigung von Ihrer Haupt-Historie, auf der Sie frei mit Änderungen für eine neue Funktion oder eine Fehlerbehebung experimentieren können. Wenn Sie mit Ihren Änderungen zufrieden sind, können Sie wieder auf die Autobahn zurückkehren, indem Sie den Branch in `main` zusammenführen. Wenn Sie nicht zufrieden sind, können Sie den Branch löschen und zum `main` zurückwechseln, ohne die Änderungen zu übernehmen.

Bevor Sie einen neuen Branch erstellen, sehen Sie sich Ihre Branches mit dem `git branch`-Befehl an. Die Ausgabe könnte so aussehen:

```sh
* main
```

Im Moment haben wir nur den einen Branch, unseren Standard-`main`-Branch. Das Sternchen (`*`) bedeutet, dass dies der Branch ist, den Sie gerade „ausgecheckt“ haben. Dazu lernen Sie gleich noch mehr.

Sie können einen neuen Branch mit dem `git branch`-Befehl erstellen, mit einem Argument, um den Namen des neuen Branches anzugeben:

```sh
git branch feature
```

Das erstellt einen neuen Branch namens `feature`. Schauen wir uns unsere Branches noch einmal mit `git branch` an:

```sh
  feature
* main
```

Wir sehen jetzt unseren neuen Branch, aber wir sind immer noch auf dem `main`-Branch. Wir müssen zu unserem neuen Branch wechseln, indem wir ihn „auschecken“. Wenn Sie einen neuen Branch auschecken, sagen Sie Git, dass es die Dateien genau in dem Zustand laden soll, in dem sie zu diesem Zeitpunkt waren.

Lassen Sie uns unseren neuen Branch auschecken mit:

```sh
git checkout feature
```

oder

```sh
git switch feature
```

Und führen Sie `git branch` erneut aus, um zu sehen:

```sh
* feature
  main
```

Beachten Sie, dass das Sternchen jetzt zum Branch verschoben wurde, den Sie gerade „ausgecheckt“ haben.

Oft möchten Sie, wenn Sie einen neuen Branch erstellen, ihn sofort auschecken. Anstatt den Branch zu erstellen und ihn dann mit zwei Befehlen auszuchecken, können Sie in einem Befehl einen Branch erstellen und zu ihm wechseln mit:

```sh
git checkout -b feature
```

oder

```sh
git switch -c feature
```

Jetzt haben wir also einen neuen Branch erstellt und ausgecheckt. Dieser Branch ist eine Kopie des `main`-Branches, er enthält den gleichen Code, den `main` zum Zeitpunkt der Branch-Erstellung hatte. Nun können wir beliebigen Code zu diesem Branch hinzufügen, ohne den `main`-Branch zu beeinflussen. Wir sind im Grunde von der Autobahn abgefahren, um eine Funktion zu erstellen. Wenn Sie Ihren `git status` prüfen würden, sähe das ungefähr so aus:

```sh
On branch feature
nothing to commit, working tree clean
```

Erstellen Sie eine neue `feature.md`-Datei, indem Sie Folgendes ausführen:

```sh
echo "This is our new feature" > feature.md
```

Und jetzt sieht der Status so aus:

```sh
On branch feature
Untracked files:
  (use "git add <file>..." to include in what will be committed)
	feature.md

nothing added to commit but untracked files present (use "git add" to track)
```

Als Nächstes ist es Zeit, Ihre neue Funktion zu committen. Fügen Sie zuerst die Datei zur Staging-Area hinzu:

```sh
git add feature.md
```

Dann committen Sie die Änderungen:

```sh
git commit -m "my new feature"
```

Schließlich pushen Sie Ihren `feature`-Branch in Ihr Repository:

```sh
git push -u origin feature
```

Das `-u`-Flag verbindet Ihren lokalen `feature`-Branch mit dem Remote-Branch, sodass zukünftige Pushes nur noch mit `git push` ausgeführt werden können.

Als Nächstes lernen wir, wie man einen Pull Request für unsere neue Funktion erstellt!
