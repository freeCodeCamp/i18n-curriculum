## 安全性與隱私

**安全性** 保護資料免於未經授權的存取、竊取和惡意破壞。它建立技術防禦機制，如防火牆、加密和防毒軟體，以阻擋壞人入侵你的系統。

**隱私** 控制組織如何收集、儲存、使用和分享個人資料。它賦予使用者決定誰能看到他們資訊的權力。

這兩個概念密切相關但又不同：

- 你可以有安全性但沒有隱私。一家公司可以將你的資料存放在高度安全的資料庫中，卻在未經你同意的情況下將資料賣給廣告商。這種情況下，安全性完好，但隱私被侵犯。
- 沒有安全性就不可能有真正的隱私。如果存放私人資料的伺服器遭到入侵，任何隱私承諾都會立即破裂。

安全性提供技術防禦機制，而隱私則管理資料使用的法律權利和政策。

## HTTPS

**HTTP** 以純文字傳輸資料，使任何攔截者都能讀取。**HTTPS**（超文本傳輸安全協定）使用 **TLS（傳輸層安全性）** 加密瀏覽器與伺服器之間的通訊通道。TLS 取代了較舊的 **SSL（安全插座層）** 協定，儘管許多網站主機平台今天仍在設定中提及 SSL。

HTTPS 的主要利益：

- **加密：** 將資料混淆成無法閱讀的程式碼。只有瀏覽器和伺服器持有解密金鑰。
- **驗證：** 要求網站擁有者從受信任組織取得數位憑證，證明網站的合法性。
- **資料完整性：** 防止任何人在傳輸過程中更改資料，你收到的資料與伺服器發送的完全相同。
- **公共 Wi-Fi 保護：** 在開放網路中建立私人通道，隱藏你的流量免受附近攻擊者窺視。

現代瀏覽器會在網站使用純 HTTP 時警告使用者，並要求 HTTPS 以啟用進階功能，如地理定位和推播通知。

## 同源政策與 CORS

**同源政策（SOP）** 是瀏覽器的安全規則，防止網頁讀取不同來源的資料。**來源** 包含三個部分，必須完全相符：

- **協定**（例如 `https`）
- **網域名稱**（例如 `example.com`）
- **埠號**（例如 `443`，HTTPS 預設；`80`，HTTP 預設）

此政策阻止惡意網站在一個瀏覽器分頁中偷取另一分頁中銀行網站的資料。

**CORS（跨來源資源共享）** 允許伺服器透過 HTTP 標頭選擇性地開放跨來源資料共享：

- 瀏覽器在每個跨來源請求中送出 `Origin` 標頭。
- 如果伺服器信任該來源，會回應 `Access-Control-Allow-Origin` 標頭。
- 瀏覽器接著允許讀取回應資料，否則會阻擋。

在 Express 中，`cors` 套件負責處理這件事：

```javascript
// development: allow all origins
app.use(cors());

// production: restrict to trusted domains
app.use(cors({ origin: ['https://yourapp.com', 'https://admin.yourapp.com'], credentials: true }));
```

## Cookies：儲存、安全性與隱私

**Cookies** 是網站在你的裝置上儲存的小型文字檔，用來記住訪問間的資訊。伺服器送出 `Set-Cookie` 標頭；瀏覽器儲存該檔案，並在每次後續請求時自動送回。

保護 cookies 的安全旗標：

- **`HttpOnly`：** 阻止 JavaScript 讀取 cookie，防止透過 XSS 竊取會話令牌。
- **`Secure`：** 確保 cookie 僅透過 HTTPS 傳輸，保護其在公共 Wi-Fi 上不被攔截。
- **`SameSite=Strict`：** 防止瀏覽器在任何跨站請求中送出 cookie，阻擋 CSRF 攻擊。
- **`SameSite=Lax`：** 允許在安全的頂層導覽中使用 cookie，但阻擋跨站表單提交和 API 請求。

**隱私影響：**

- **第一方 cookie** 僅追蹤你在正在訪問的網站上的行為。
- **第三方 cookie** 由廣告網路放置，追蹤你在完全不同網站上的行為，建立詳細的興趣檔案。

像 GDPR 這樣的隱私法規要求網站顯示 cookie 同意橫幅，並賦予使用者拒絕追蹤 cookie 的權利。現代瀏覽器預設阻擋第三方追蹤 cookie。

## 安全問題常見發生情境

安全漏洞很少是偶然發生。常見觸發情境包括：

- **急於發布：** 開發團隊為趕期限跳過完整程式碼審查和安全測試，留下未修補的漏洞和缺失的存取控制。
- **複雜系統整合：** 每個外部函式庫、第三方 API 或微服務都增加供應鏈風險。一個相依項的漏洞會擴散到主應用程式。
- **組態管理不善：** 使用預設出廠設定部署伺服器或雲端資料庫，常留下空白管理員密碼、開放埠口和公開資料存取。自動掃描器會尋找這些錯誤組態系統。
- **人為錯誤：** 員工遭釣魚郵件、詐騙簡訊或欺騙電話攻擊，可能不小心洩露憑證或將資料轉給攻擊者。
- **組織合併：** 快速連接兩個不同企業網路會產生盲點，如舊伺服器失去監控、錯誤組態的防火牆和過度存取權限。

## 常見安全威脅與緩解措施

| 威脅 | 運作方式 | 緩解措施 |
|---|---|---|
| **釣魚攻擊** | 偽造郵件模仿可信品牌以竊取憑證 | 安全意識訓練；電子郵件驗證協定（SPF、DKIM、DMARC） |
| **惡意軟體（勒索軟體、間諜軟體）** | 加密或竊取受害者檔案；勒索贖金 | 自動修補；防惡意軟體軟體 |
| **憑證填充攻擊** | 自動工具測試數百萬組被盜用的使用者名稱/密碼組合 | 強制多因素驗證（MFA） |
| **分散式阻斷服務攻擊（DDoS）** | 以假流量淹沒伺服器使其當機 | 雲端流量過濾服務（例如 Cloudflare） |
| **內部威脅** | 不滿或疏忽員工洩漏或竊取資料 | 最小權限原則；監控系統日誌異常存取 |

## 內容安全政策（CSP）與 Permissions-Policy

**內容安全政策（CSP）** 是一種 HTTP 回應標頭，限制瀏覽器在頁面上可載入的外部資源。它主要防禦 **跨站指令碼攻擊（XSS）**，透過白名單信任的腳本、樣式和圖片來源網域。

如果未經批准的網域嘗試執行惡意腳本，瀏覽器會阻擋並記錄安全錯誤。

**Permissions-Policy**（前稱 Feature-Policy）控制瀏覽器功能和裝置硬體的存取，如攝影機、麥克風和地理定位。你可以限制功能只在自己的網域使用，或完全停用：

```http
Permissions-Policy: camera=(self), geolocation=()
```

`camera=(self)`：只有你自己的網域可存取攝影機。
`geolocation=()`：頁面上所有人皆停用地理定位。

CSP 鎖定資料傳遞並防止程式碼注入；Permissions-Policy 保護裝置硬體並強制隱私合規。

## 使用者隱私的基本概念

| 概念 | 定義 |
|---|---|
| **PII（個人可識別資訊）** | 任何能直接或間接識別個人的資料。直接識別字：姓名、地址、身分證號。間接識別字：IP 位址、裝置序號、位置歷史。 |
| **機密性** | 透過加密和存取控制確保私人資料不被未授權方取得。 |
| **資料最小化** | 僅收集完成特定任務所需的最少 PII。 |
| **目的限制** | 為一個理由收集的資料不得非法用於其他用途（例如賣給廣告商）。 |
| **追蹤** | 使用腳本、裝置指紋和瀏覽器歷史紀錄持續監控線上活動以建立使用者檔案。 |
| **知情同意** | 使用者必須主動選擇同意資料收集。預先勾選的方框不構成有效同意。 |
| **被遺忘權** | 使用者可合法要求組織永久刪除所有個人資料和追蹤紀錄。 |

## 地區性隱私法規

| 法規 | 管轄區 | 主要要求 |
|---|---|---|
| **GDPR** | 歐盟 | 嚴格的主動同意要求；使用者有權存取和刪除資料；違規重罰。 |
| **CCPA / CPRA** | 美國加州 | 採用退出機制；企業必須顯示「請勿出售我的個人資訊」連結。 |
| **COPPA** | 美國（聯邦） | 收集 13 歲以下兒童資料前需可驗證的家長同意。 |
| **DPA / 英國 GDPR** | 英國 | 合法、公平且透明的資料處理；個人有存取和刪除權利。 |
| **NDPA** | 尼日利亞 | 管理尼日利亞居民個人資料的收集、儲存和使用。 |
| **DPDP** | 印度 | 管理印度居民個人資料的使用。 |

**實務合規步驟：**

1. **地理位置感知同意橫幅：** 偵測使用者地理位置並顯示符合當地法規的 cookie 橫幅。
2. **自動化 DSAR 系統：** 處理資料主體存取請求，安全匯出或刪除使用者的完整資料歷史。
3. **資料最小化與年齡門檻：** 僅收集必要資料並使用年齡驗證保護未成年人。
4. **資料保護專員（DPO）：** 指派專家監督隱私架構並管理法規遵循。

## `.env` 檔案與 `dotenv` 函式庫

**`.env` 檔案** 是放在專案根目錄的純文字檔，儲存敏感設定資料的鍵值對，包括 API 金鑰、資料庫密碼和埠號。它將這些資料從原始碼和公開倉庫中隔離。

```bash
PORT=3000
DB_HOST=localhost
API_KEY=xyz123secretkey
```

務必將 `.env` 加入 `.gitignore`，避免提交到版本控制。

**`dotenv` 套件** 在啟動時讀取 `.env` 檔案，並將金鑰注入 `process.env`：

```javascript
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

兩大關鍵利益：

- **安全性：** 敏感憑證不會出現在原始碼或版本歷史中。
- **彈性：** 同一份程式碼庫可在開發、測試和生產環境執行，只需在每台伺服器上替換 `.env` 檔案。
