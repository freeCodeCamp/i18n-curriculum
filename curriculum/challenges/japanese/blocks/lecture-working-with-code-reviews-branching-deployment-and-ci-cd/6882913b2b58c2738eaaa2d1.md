# --description--

前のレッスンで`.gitignore`ファイルに少し触れましたが、このファイルとは具体的に何で、何を入れるべきなのでしょうか？

`.gitignore`ファイルはGitの操作に関連する特別な種類のファイルです。名前から、このファイルはGitに無視するよう指示するためのものだと想像できますが、実際にはGitにファイルの追跡をやめるよう伝えるものです。この違いは重要で、後ほど詳しく説明します。

実際にこのファイルに何を入れるべきでしょうか？それは、コミットしたくないものをリストアップすることです。

まずは秘密情報のファイルです。例えば、アプリケーションが利用するAPIキーを保存している`.env`ファイルかもしれません。認証に使うプライベートなGPGキーのファイルかもしれません。公開されたくない機密情報が含まれるものは`.gitignore`で無視すべきです。リポジトリがプライベートであっても、誰でも見られるものとして扱い、秘密情報をコミットしないように注意してください。

現在の`.env`ファイルの例を見てみましょう。

```sh
# Secrets
.env
github.key
```

秘密情報を保存している`.env`ファイルをGitに無視するよう伝えています。また、プライベートなGPGキーを含む`github.key`ファイルも無視するようにしています。

次に考慮すべきは、プロジェクトで使うパッケージや依存関係です。例えばNode.jsのプロジェクトでは、npmからパッケージをインストールすることが多いでしょう。これらのパッケージはプロジェクト内の`node_modules`フォルダーに入るため、コードがそれらをインポートして利用できます。

しかし`node_modules`フォルダーは非常に巨大になりがちで、すべてのパッケージをコミットして更新ごとに追跡するのは望ましくありません。代わりに`package.json`ファイルと関連するロックファイル（例：`package-lock.json`、`pnpm-lock.yaml`）をコミットします。これらのファイルはnpmにどのパッケージとバージョンをインストールすべきかを伝えるため、貢献者は依存関係をリポジトリにコミットせずに同じものをインストールできます。

では、`node_modules`を`.gitignore`ファイルに追加しましょう。

```sh
# Secrets
.env
github.key

# Packages
node_modules
```

ここで`.gitignore`ファイルの重要な構文について触れます。`node_modules`のようなエントリを追加すると、そのパターンに一致するものはすべてGitに無視されます。つまり、プロジェクトのルートにある`node_modules`ディレクトリは無視されますが、`client`ディレクトリ内の`node_modules`も無視されます。

ルートだけを無視したい場合は、行の先頭にスラッシュを付けます。`node_modules`は`/node_modules`になります。これによりルートの`node_modules`ディレクトリは無視されますが、`client/node_modules`ディレクトリは無視されません。これは望ましくないので、エントリは`node_modules`のままにしておきましょう。

次に無視すべきはビルドの出力物です。例えばTypeScriptプロジェクトをコンパイルすると、JavaScriptの出力が`dist`フォルダーに生成されることがあります。この出力はGitの履歴を膨らませるためコミットしたくありません。代わりに無視し、貢献者は同じビルドコマンドを実行して自分で再現できます。

では`dist`フォルダーを無視しましょう。

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
```

時には誤って置かれたファイルがTypeScriptコンパイラを壊し、JavaScriptの出力が散らばってしまうことがあります。私たちはTypeScriptだけを書いているので、グロブ構文を使ってすべてのJavaScriptファイルを無視しましょう。

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js
```

他に無視することが多い出力物としては、ユニットテストのカバレッジ結果の`coverage`ディレクトリ、Angularを使っている場合の`.angular`ディレクトリ、NextJSを使っている場合の`.next`ディレクトリなどがあります。

今はここまでで十分です。フレームワークがファイルを生成する場合は、そのドキュメントで`.gitignore`に何を追加すべきか確認してください。

4つ目の無視対象はIDEのファイルです。例えばJetbrains製品はIDE固有の設定を`.idea`フォルダーに保存します。Visual Studio Codeは`.vscode`フォルダーを使います。

これらはローカル環境固有のファイルを含むため、他の貢献者は自分の設定を使うので無視するのが良いでしょう。私たちのIDEはVisual Studio Codeなので、そのディレクトリを追加しましょう。

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
```

ただし`.vscode`フォルダーには`settings.json`ファイルが含まれており、これは拡張機能や環境設定を私たちのものに近づけるために貢献者に役立つかもしれません。

そこで否定構文を使ってそのファイルだけ「無視しない」ようにしましょう。

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json
```

MacOSを使っているなら`.DS_STORE`ファイルに馴染みがあるかもしれません。このファイルはFinderがディレクトリの表示方法（サムネイルやリストなど）を決めるためのメタデータを含みます。同様にWindowsユーザーは隠し`Thumbs.db`ファイルを見かけることがあります。これはディレクトリのサムネイルをキャッシュします。

これらは個人設定でプロジェクトに影響しないため、無視するのが一般的です。両方を追加しましょう。

```sh
# Secrets
.env
github.key

# Packages
node_modules

# Outputs
dist
*.js

# IDE
.vscode
!.vscode/settings.json

# System Files
.DS_STORE
Thumbs.db
```

以前にコミットしたファイルを無視する場合の注意点です。例えば`dist`ディレクトリを誤ってコミットしてしまい、今は削除して無視したいとします。`.gitignore`ファイルに追加するだけでは、以前にコミットした内容は消えません。Gitはファイルの追跡をやめますが、削除も追跡しません。

そのため、まずファイルを削除し、その削除をコミットします。次に`.gitignore`に追加して2回目のコミットを作成します。以降、そのファイルが再作成されてもGitは無視し、コミットに表示されません。

最後に、Pythonなど別の言語でプロジェクトをしている場合、無視すべき項目はここで扱ったものとは異なります。GitHubは様々な言語やフレームワーク向けの`.gitignore`テンプレートを便利にまとめたリポジトリを管理しています。リストは<a href="https://github.com/github/gitignore" target="_blank">https://github.com/github/gitignore</a>で確認できます。

これでリポジトリに含めたくないファイルをコミットしない準備が整いました！
