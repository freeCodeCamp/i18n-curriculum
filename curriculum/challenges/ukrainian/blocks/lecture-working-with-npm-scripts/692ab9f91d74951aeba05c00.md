# --description--

Тепер ви дізнаєтеся про npm скрипти.

Перед тим як почати, давайте повторимо кілька важливих понять:

* **npm** — це інструмент командного рядка, який ви можете використовувати для встановлення та керування JavaScript пакетами.
    
* **пакет** — це директорія, яка логічно групує пов’язаний код.
    
* **скрипт** — це послідовність інструкцій або команд, написаних мовою сценаріїв.
    

**npm скрипти** — це кастомні команди, які ви можете визначити у файлі `package.json` вашого застосунку.

Вони корисні для автоматизації завдань і робочих процесів, які ви будете використовувати багаторазово під час розробки, тестування та розгортання вашого проєкту.

Наприклад, одним із варіантів використання npm скриптів є кастомізація `npm start` для запуску всього вашого середовища розробки. Замість того, щоб вручну писати всі команди по одній щоразу, коли потрібно запустити середовище розробки, ви можете просто виконати `npm start`, і весь процес буде виконано автоматично.

Ви також можете використовувати npm скрипти для автоматизації лінтингу у всій кодовій базі та для запуску тестових наборів, серед інших випадків.

Вони дуже потужні, тож давайте розглянемо їх основи.

## Як створити файл package.json

Щоб мати змогу створювати та запускати кастомні npm скрипти, вам потрібен файл `package.json`.

Файл `package.json` є необхідним для проєктів Node.js. Цей файл містить метадані, залежності та скрипти проєкту.

Нагадаємо, щоб створити файл `package.json`, потрібно виконати команду `npm init` у кореневій директорії вашого проєкту. Потім ви можете вручну ввести назву пакета, версію, опис та інші деталі або залишити ці поля порожніми, щоб використати значення за замовчуванням.

Після завершення цього процесу у кореневій директорії вашого проєкту з’явиться файл `package.json`.

Якщо ви залишили всі значення за замовчуванням, при відкритті `package.json` у вашому редакторі коду ви побачите такі пари ключ-значення:

```json
{
  "name": "<project_name>",
  "version": "1.0.0",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "author": "",
  "license": "ISC",
  "description": ""
}
```

Зверніть увагу на цю пару ключ-значення:

```json
"scripts": {
  "test": "echo \"Error: no test specified\" && exit 1"
}
```

Саме тут ви вказуватимете свої npm скрипти, під ключем `scripts`.

За замовчуванням там буде один скрипт — тестовий скрипт:

```json
"test": "echo \"Error: no test specified\" && exit 1"
```

Це приклад npm скрипта, визначеного у `package.json`.

Ось базовий синтаксис для визначення npm скрипта:

```json
"<script_name>": "<commands>"
```

Зліва від двокрапки знаходиться ім’я скрипта. Загальні імена включають `run`, `start`, `test`, `build`, `dev` та `lint`. Ці імена виступають як «ярлики» для відповідних команд праворуч.

Справа від двокрапки — це самі команди, які виконуються.

Наприклад:

```json
"scripts": {
  "start": "node app.js"
}
```

Щоб запустити скрипт, введіть це у командному рядку:

```bash
npm run <script_name>
```

Ось як можна запускати будь-які кастомні скрипти, визначені у `package.json`.

Іноді можна використовувати скорочену версію без слова `run` для виконання скриптів:

```bash
npm <script_name>
```

Але це працює лише для певних npm скриптів з іменами `start`, `test`, `stop` та `restart`.

Повертаючись до нашого прикладу, якщо у `package.json` є таке:

```json
"scripts": {
  "start": "node app.js"
}
```

Щоб запустити npm скрипт, потрібно ввести `npm run start` у командному рядку. При цьому фактично виконується команда `node app.js`.

Оскільки цей npm скрипт має скорочену версію, ви також можете використати `npm start`, і результат буде той самий.

У цьому випадку ви просто замінюєте просту команду `node app.js` на іншу `npm start`. Але уявіть, що у вас є послідовність складних команд, які потрібно виконати послідовно або паралельно, і ви можете просто використати `npm start`, щоб запустити всі одразу. Це суттєво заощадить ваш час, правда? Ось де справжня сила npm скриптів.

Ось короткий огляд деяких найпоширеніших npm скриптів:

* `npm run`: використовується для запуску кастомного npm скрипта.
    
* `npm start`: використовується для запуску основного застосунку або сервера розробки.
    
* `npm lint`: використовується для запуску перевірки стилю коду, щоб перевірити, чи відповідає ваш код стандартам стилю та найкращим практикам.
    
* `npm test`: використовується для запуску тестового набору проєкту.
    
Існує багато інших npm скриптів для різних цілей, але це одні з найпоширеніших, які ви побачите.

Також загальною конвенцією є використання двокрапки для групування пов’язаних скриптів під одним простором імен. Наприклад, `start:server` та `start:ui`.

## Як запускати команди послідовно

Якщо потрібно виконати кілька команд послідовно, у вас є два варіанти. Ви можете використати логічний оператор AND (`&&`) або крапку з комою (`;`). Розглянемо їхні відмінності.

Логічний оператор AND запускає наступну команду лише якщо перша виконалась успішно.

Наприклад:

```json
"scripts": {
  "build": "webpack --mode production",
  "start": "npm run build && node server.js"
}
```

Зверніть увагу, що у команді `start` використовується логічний оператор AND:

```json
"start": "npm run build && node server.js"
```

Тут `npm start` спочатку намагається виконати `npm run build`, і лише якщо цей скрипт виконається успішно, буде виконано `node server.js`.

Це дуже поширений підхід, бо він забезпечує надійність і запобігає марнуванню часу та ресурсів на наступні команди, які можуть не виконатися або викликати помилки, якщо попередня команда не вдалася.

Альтернативно, ви можете використати оператор крапки з комою, щоб запускати команди послідовно, незалежно від того, чи була попередня команда успішною.

У нашому прикладі ця проста зміна суттєво вплине на те, як виконуватимуться команди:

```json
"scripts": {
  "build": "webpack --mode production",
  "start": "npm run build; node server.js"
}
```

Знову зверніть увагу на команду `start`:

```json
"start": "npm run build; node server.js"
```

Тепер, оскільки ми використовуємо крапку з комою замість оператора `&&`, команда `node server.js` завжди виконуватиметься після `npm run build`, навіть якщо `node server.js` не була успішною.

```json
"start": "npm run build; node server.js"
```

## Як запускати команди паралельно

Якщо потрібно запускати команди паралельно (одночасно), можна використати оператор `&`.

```json
"scripts": {
  "start:server": "node server/index.js",
  "start:client": "npm run build:watch",
  "dev": "npm run start:server & npm run start:client"
}
```

Однак слід знати, що цей варіант запускає процеси у фоновому режимі.

Це означає, що якщо ви запускаєте їх у тій самій сесії термінала, ви можете побачити їх об’єднаний вихід, який може бути складним для розуміння.

Інші варіанти для паралельного запуску команд включають інструменти, такі як `concurrently` та `npm-run-all`.

## Як передавати аргументи npm скриптам

Якщо потрібно передати аргументи безпосередньо команді, яка виконується у npm скрипті, ви можете вказати їх у команді, написавши два дефіси (`--`) перед аргументами.

`--` повідомляє npm, що все, що йде після нього, буде аргументами для команд, які виконує npm скрипт.

Ось загальний синтаксис:

```bash
npm run <script_name> -- <argument_1> <argument_2> <argument_3>
```

Наприклад, уявімо, що ви визначили цей скрипт `npm start`:

```json
{
  "name": "project-example",
  "scripts": {
    "start": "node app.js"
  }
}
```

Як можна вказати порт, на якому ви хочете запустити сервер, якщо ви використовуєте `npm start`?

Припустимо, ми хочемо передати порт безпосередньо як аргумент командного рядка. Ви б написали так:

```bash
npm start -- 8000
```

У вашому коді ви можете отримати доступ до цих аргументів у масиві `process.argv` за індексами:

```bash
process.argv[0] # The path to the node executable.
process.argv[1] # The script file being executed.
process.argv[2] # The first argument (8000).
```

Отже, ви можете отримати і використати номер порту там, де потрібно, за допомогою `process.argv[2]`. Ось сила аргументів командного рядка у поєднанні з npm скриптами.

Як бачите, npm скрипти — це необхідні інструменти для спрощення робочого процесу розробки. Вони допомагають автоматизувати та перетворювати складні та багаторазові послідовності команд у прості та послідовні ярлики, які ви можете використовувати у своїй щоденній роботі.
