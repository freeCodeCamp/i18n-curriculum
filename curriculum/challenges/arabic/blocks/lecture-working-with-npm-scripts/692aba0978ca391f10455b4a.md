# --description--

دعنا نتعرف على سجل npm، حيث يمكنك مشاركة حزم JavaScript الخاصة بك مع العالم.

**سجل npm** هو قاعدة بيانات للحزم العامة والخاصة.

يمكن للمطورين والمنظمات نشر الحزم في هذا السجل لمشاركة أدواتهم مع مجتمعاتهم وأعضاء فرقهم، وفي كثير من الحالات، مع جميع مطوري Node.js الآخرين.

يمكن أن تكون الحزم عامة أو خاصة. يمكن لأي شخص الوصول إلى الحزم العامة. أما الحزم الخاصة فلا يمكن الوصول إليها إلا من قبل مستخدمين محددين أو أعضاء في منظمة.

في سجل npm، يمكنك مشاركة ما بنيته مع العالم ببضع نقرات فقط. يبدو رائعًا، أليس كذلك؟

دعنا نرى كيف يمكنك نشر حزمة خطوة بخطوة.

## الخطوة 1: إنشاء حساب

الخطوة الأولى هي إنشاء حساب npm مجاني. لإنشاء حسابك:

* اذهب إلى [https://www.npmjs.com/signup](https://www.npmjs.com/signup).
    
* املأ النموذج. ستحتاج إلى إدخال اسم المستخدم، البريد الإلكتروني، وكلمة المرور.
    

يرجى ملاحظة أنه، وفقًا لصفحة التسجيل، "سيتم إضافة عنوان بريدك الإلكتروني إلى البيانات الوصفية للحزم التي تنشرها، لذا قد يُرى علنًا."

## الخطوة 2: **تسجيل الدخول**

بمجرد أن يكون لديك حساب npm، ستحتاج إلى تسجيل الدخول في الطرفية الخاصة بك بتشغيل هذا الأمر:

```bash
npm login
```

بعدها ستحتاج إلى المصادقة باستخدام معلومات حسابك.

## **الخطوة 3: إنشاء ملف** `package.json`

قبل نشر حزمك، ستحتاج إلى إنشاء ملف `package.json` بهذا الأمر:

```bash
npm init
```

(إذا كان مشروعك يحتوي على واحد بالفعل، يمكنك الانتقال مباشرة إلى **الخطوة 4**).

ستحتاج إلى إدخال هذه المعلومات في ملف `package.json` الخاص بك:

* **name**: عادةً ما يكون اسم مشروعك/الدليل. إذا كنت تخطط لجعل حزمك عامة، يجب أن يكون هذا الاسم فريدًا، إلا إذا استخدمت نطاقات (سنغطي هذا بعد قليل).
    
* **version**: يجب أن تبدأ بـ `"1.0.0"`، القيمة الافتراضية. هذا يمثل الإصدار الأولي لحزمتك ويمكنك تحديثه مع إجراء تغييرات على الحزمة.
    
* **main**: نقطة الدخول لمشروعك. هذا هو `index.js` افتراضيًا.
    
* **author**: يمكنك كتابة اسمك أو المعرف المفضل لديك.
    
* **license**: اختر ترخيصًا محددًا. بعض الأمثلة تشمل MIT و ISC.
    
* **description**: ملخص موجز لحزمتك.
    

عند اختيار اسم لحزمتك، تأكد من أنه غير مستخدم بالفعل في سجل npm. للقيام بذلك، يمكنك الذهاب إلى الرابط التالي، مع استبدال `<your-package-name>` باسم حزمتك:

```bash
https://www.npmjs.com/package/<your-package-name>
```

إذا لم تكن هناك حزمة على ذلك الرابط، فإن اسم الحزمة متاح ويمكنك استخدامه.

ومع ذلك، إذا كان الاسم مستخدمًا بالفعل وترغب حقًا في الاحتفاظ به، يمكنك استخدام ما يُعرف بـ **النطاق** لنشره.

### **النطاقات**

عندما تنشئ حساب مستخدم أو منظمة في سجل npm، يتم منحك نطاقًا باسم المستخدم الخاص بك أو باسم المنظمة.

النطاقات مفيدة لتجنب تعارض الأسماء لأنك تستطيع نشر حزمتك تحت نطاقك، حتى لو كانت هناك حزمة أخرى بنفس الاسم موجودة بالفعل.

يبدأ اسم الحزمة ذات النطاق باسم النطاق. على سبيل المثال، في `package.json`، ستجد زوج المفتاح والقيمة هذا:

```bash
"name": "@<scope-name>/<your-package-name>"
```

يمكنك إجراء هذا التغيير مباشرة في الملف إذا احتجت، أو يمكنك إنشاء ملف `package.json` مع النطاق من البداية بإضافة `--scope=@<scope-name>` إلى أمر `npm init`.

```bash
npm init --scope=@<scope-name>
```

يمكن أن يكون اسم النطاق هو اسم المستخدم إذا كنت تستخدم حسابًا شخصيًا، أو اسم المنظمة إذا كنت تستخدم حسابًا تنظيميًا.

## الخطوة 4: إنشاء README.md

ملف README الخاص بحزمتك هو ملف ماركداون تصف فيه ما تفعله حزمتك وكيفية استخدامها.

بعض الأمثلة على المحتوى المهم الذي قد ترغب في تضمينه هي:

* الغرض منها.
    
* كيفية تثبيتها.
    
* التبعيات والاعتبارات المهمة.
    
* كيفية البدء.
    
* مقتطفات برمجية نموذجية.
    
* إرشادات المساهمة (إذا كنت تقبل المساهمات).
    
* رابط إلى ملف سجل التغييرات حيث تتعقب التغييرات التي تمت في الإصدارات المختلفة.
    

هيكل ملف README مرن. يمكنك تكييفه ليناسب احتياجاتك.

## **الخطوة 5:** `.npmignore`

يخبر ملف `.npmignore` سجل npm بالملفات التي يجب تضمينها وتلك التي يجب تجاهلها عند نشر حزمتك. هذا مهم جدًا لمنع مشاركة معلومات حساسة مثل المفاتيح الخاصة، كلمات المرور، والمعلومات الشخصية.

في هذا الملف، يمكنك كتابة أنماط لتحديد الملفات، الأدلة، وأنواع الملفات التي يتم تجاهلها. يتبع نفس قواعد مطابقة الأنماط مثل ملفات `.gitignore`.

إذا كنت قد عملت مع Git، قد تعتقد أن هذا الملف يشبه جدًا `.gitignore`، وهذا صحيح. وفقًا لوثائق npm، "إذا لم يكن هناك ملف `.npmignore`، ولكن يوجد ملف `.gitignore`، فإن npm سيتجاهل الأشياء التي يطابقها ملف `.gitignore`."

يجب أن تعرف أيضًا أن npm يبحث عن ملفات `.npmignore` و `.gitignore` في جميع المجلدات الفرعية لحزمتك، وليس فقط في المجلد الجذر.

افتراضيًا، يتم تجاهل هذه المسارات تلقائيًا، لذا لا تحتاج إلى إضافتها إلى `.npmignore`:

* `.*.swp`
    
* `._*`
    
* `.DS_Store`
    
* `.git`
    
* `.gitignore`
    
* `.hg`
    
* `.npmignore`
    
* `.npmrc`
    
* `.lock-wscript`
    
* `.svn`
    
* `.wafpickle-*`
    
* `config.gypi`
    
* `CVS`
    
* `npm-debug.log`
    

يتم أيضًا تجاهل دليل `node_modules`، باستثناء التبعيات المجمعة.

## **الخطوة 6: نشر حزمتك**

قبل نشر حزمتك إلى سجل npm، يجب اختبارها بتشغيل هذا الأمر:

```bash
npm install <full-path-to-package-directory>
```

بمجرد أن تكون جاهزًا للنشر، سيعتمد الأمر على نوع الحزمة: ذات نطاق أو بدون نطاق.

إذا كانت حزمتك بدون نطاق، شغّل هذا الأمر:

```bash
npm publish
```

إذا كانت حزمتك ذات نطاق، فستُنشر كحزمة خاصة افتراضيًا (إذا كانت هذه الميزة متاحة في حسابك). إذا أردت جعلها عامة، ستحتاج إلى إضافة `--access public` إلى الأمر.

```bash
npm publish --access public
```

لمشاهدة حزمتك العامة، اذهب إلى هذا الرابط:

```bash
https://npmjs.com/package/<your-package-name>
```

يجب أن ترى مؤشرًا لمستوى الظهور لحزمتك (عامة أو خاصة) أسفل اسم الحزمة.

## **الخطوة 7 (اختياري): تثبيت حزمتك**

الآن بعد أن أصبحت جاهزًا، يمكنك تثبيت حزمتك في مشروع Node.js بهذا الأمر:

```bash
npm install <your-package-name>
```

## تحديث حزمتك

إذا احتجت إلى إجراء تغييرات على حزمتك بعد نشرها، يمكنك تحديث إصدار الحزمة في `package.json`.

هذا هو الاصطلاح القياسي للإصدار الدلالي:

* **الإصدار الأول:** `1.0.0`.
    
* **إصدار تصحيحي:** زيادة الرقم الثالث. على سبيل المثال، التحديث من `1.0.0` إلى `1.0.1`.
    
* **إصدار ثانوي:** زيادة الرقم الأوسط وإعادة تعيين الرقم الأخير إلى صفر. على سبيل المثال، التحديث من `1.0.0` إلى `1.1.0`.
    
* **إصدار رئيسي:** زيادة الرقم الأول وإعادة تعيين الرقمين الأوسط والأخير إلى صفر. على سبيل المثال، التحديث من `1.1.0` إلى `2.0.0`.
    
عمل رائع. الآن تعرف كيف تنشر حزمتك إلى سجل npm وكيف تحدثها لإصلاح الأخطاء البرمجية وإضافة ميزات جديدة.
