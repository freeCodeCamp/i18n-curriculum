# --description--

Веб-сервіси та REST API забезпечують роботу більшості застосунків і вебсайтів, які ви використовуєте щодня. Вони дозволяють різним застосункам спілкуватися між собою та обмінюватися даними через мережу, найчастіше через інтернет.

API (інтерфейс програмування застосунків) — це загальний спосіб для програмного забезпечення взаємодіяти з іншим програмним забезпеченням. Веб-сервіс — це конкретний тип API, який використовує веб-технології, такі як HTTP, для обміну даними через мережу, зазвичай інтернет. Через це кожен веб-сервіс є API — але не кожен API є веб-сервісом. REST API — це поширений тип веб-сервісу, який дотримується певного набору правил щодо того, як запити і відповіді з даними мають відбуватися.

Цей урок зосередиться на загальному розумінні, що таке веб-сервіси, а наступний розгляне детальніше, як працюють REST API.

Ви можете уявити веб-сервіс як міст між програмним забезпеченням. Він не має користувацького інтерфейсу і не призначений для прямого використання. Натомість він повністю сфокусований на обміні даними між застосунками.

Щоб система вважалася веб-сервісом, вона має мати такі характеристики:

* **Працює на сервері**: веб-сервіс має бути розміщений онлайн, як вебсайт, але замість сторінок, які бачать користувачі, він відкриває доступ до даних або певної функції
    
* **Комунікація машина-машина**: веб-сервіс не призначений для прямого перегляду людьми, а для споживання мобільними та веб-застосунками, щоб вони могли відкрити доступ до даних або функцій, які в кінцевому результаті бачить користувач
    
* **Незалежність від мови програмування**: веб-сервіс може бути написаний будь-якою мовою, наприклад Java або C#, і будь-який застосунок має мати змогу взаємодіяти з ним, навіть якщо він написаний іншою мовою
    
* **Незалежність від платформи**: не має значення, на якій платформі працюють машини. Наприклад, веб-сервіс, що працює на Linux, не повинен заважати пристрою з Windows спілкуватися з ним
    
* **Використання стандартних веб-протоколів**: веб-сервіс зазвичай має використовувати HTTP/HTTPS для передачі даних через інтернет, FTP для передачі файлів і SMTP для електронної пошти
    

**REST** і **SOAP** — це два найвідоміші типи веб-сервісів, хоча існують й інші підходи, наприклад **XML-RPC**.

**REST**, що означає **передача репрезентативного стану**, є найбільш поширеним сьогодні. Більшість сучасних застосунків покладаються на RESTful веб-сервіси, бо вони прості, гнучкі і природно працюють з вебом.

REST API відкривають ресурси, які зазвичай представлені шляхами URL, такими як `/users` або `/users/20`. Коли клієнт робить запит до ресурсу, сервер повертає представлення цього ресурсу. Це представлення часто у форматі JSON, але також може бути у форматах XML або навіть простого тексту.

Клієнти можуть вказати бажаний формат, використовуючи HTTP-заголовки, наприклад заголовок `Accept`. Наприклад, клієнт може запросити відповідь у форматі JSON, встановивши відповідне значення `Accept`.

```bash
Accept: application/json
```

А ви можете вказати, що хочете XML так:

```bash
Accept: application/xml
```

Ресурс, який ви отримаєте, може виглядати так, якщо це JSON:

```json
{
  "id": 20,
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "createdAt": "2025-09-22T10:30:00Z"
}
```

А так, якщо це XML:

```xml
<user>
  <id>20</id>
  <name>John Doe</name>
  <email>john@example.com</email>
  <role>admin</role>
  <createdAt>2025-09-22T10:30:00Z</createdAt>
</user>
```

SOAP означає **Simple Object Access Protocol**. Він використовує лише XML-формат даних і найпоширеніший у корпоративних системах, які потребують суворіших стандартів і безпеки.

Ось як можуть виглядати дані SOAP:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
   <soapenv:Body>
      <getUser>
         <userId>20</userId>
      </getUser>
   </soapenv:Body>
</soapenv:Envelope>
```

XML-RPC використовує XML для кодування своїх запитів і відповідей. Це робить його легким, але менш гнучким, ніж REST.

Щодо UDDI, це не сервіс обміну даними, а XML-орієнтований каталог, де бізнеси можуть розміщувати, знаходити та описувати свої доступні веб-сервіси.
