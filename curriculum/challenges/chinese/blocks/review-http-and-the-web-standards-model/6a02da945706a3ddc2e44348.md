## 服务器和客户端-服务器模型

- **服务器**：接收其他计算机请求并通过网络发送数据或服务的计算机或程序。该术语既指物理硬件，也指运行其上的软件。
- **客户端-服务器模型**：一种分布式架构，客户端发送请求，服务器处理这些请求并返回响应。
- **客户端**：发起对服务器请求的进程或应用（例如，网页浏览器）。
- **请求-响应周期**：客户端发送请求；服务器处理请求并返回响应。

### 服务器类型

- **网络服务器**：向浏览器传送网页内容（超文本标记语言、图像、CSS）。
- **应用服务器**：处理业务逻辑和用户输入。
- **数据库服务器**：存储、检索和管理数据。
- **邮件服务器**：发送、接收和保存电子邮件。
- **文件服务器**：管理网络中的文件存储和共享。
- **代理服务器**：作为客户端和其他服务器之间的中介。

## DNS（域名系统）

- **DNS**：一种分层且去中心化的命名系统，将人类可读的域名转换为 IP 地址，常被称为互联网的“电话簿”。
- **IP 地址**：网络中设备的唯一数字标识符（例如，`127.0.0.1` 在 IPv4 中，`0:0:0:0:0:0:0:1` 在 IPv6 中）。
- **域名层次结构**（从右向左读取）：
  - **顶级域（TLD）**：最右侧的部分（例如，`.org`，`.com`）。
  - **二级域（SLD）**：注册名称（例如，`freecodecamp` 在 `freecodecamp.org` 中）。
  - **子域**：标识主域的某个部分（例如，`www`，`blog`，`api`）。

### DNS 解析过程

1. 浏览器将域名发送给**递归解析器**（通常由 ISP 管理）。
2. 递归解析器查询**根服务器**，根服务器将其指向相关的顶级域服务器。
3. **顶级域服务器**返回该域的**权威名称服务器**地址。
4. 权威名称服务器返回包含 IP 地址的**DNS 记录**。
5. 递归解析器将 IP 地址传递给客户端，并**缓存**该记录以加快未来查询速度。

## TCP/IP

- **协议**：允许设备通过网络通信的一组规则和标准。
- **TCP/IP**：一套协议，决定数据如何打包、发送、路由和接收。
  - **IP（互联网协议）**：使用 IP 地址将数据包从源头路由到目的地。它是无连接的——每个数据包独立传输，没有交付保证。
  - **TCP（传输控制协议）**：将数据分割成编号段，在目的地重新组合，检查错误和丢包，并在需要时请求重传。确保可靠、有序的传输。

## HTTP（超文本传输协议）

- **HTTP**：一种应用层协议，定义了客户端和服务器之间在网络上的通信规则。

- **HTTPS**：HTTP 的安全版本，数据在传输前使用 SSL/TLS 加密。

### HTTP 请求结构

- **请求行**：`METHOD /path HTTP/version`（例如，`GET / HTTP/1.1`）
- **头部**：键值对元数据（例如，`Content-Type: application/json`）。
- **正文**（可选）：发送给服务器的数据，常用于 `POST`、`PUT` 和 `PATCH`。

**常见 HTTP 方法：**

| 方法 | 目的 |
| --- | --- |
| `GET` | 获取数据 |
| `POST` | 提交新数据 |
| `PUT` | 替换资源 |
| `PATCH` | 部分更新资源 |
| `DELETE` | 删除资源 |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### HTTP 响应结构

- **状态行**：HTTP 版本、状态码和原因短语（例如，`HTTP/1.1 200 OK`）。
- **头部**：响应的元数据（例如，`Content-Type: text/html`）。
- **正文**（可选）：返回的内容或错误说明。

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## HTTP 状态码

状态码按首位数字分组：

- **1xx** — 信息性响应。
- **2xx** — 成功响应。
- **3xx** — 重定向消息。
- **4xx** — 客户端错误响应。
- **5xx** — 服务器错误响应。

### 常见状态码

| 代码 | 名称 | 含义 |
| --- | --- | --- |
| `200` | OK | 请求成功；返回内容。 |
| `201` | Created | 资源创建成功。 |
| `204` | No Content | 成功但无内容返回。 |
| `301` | Moved Permanently | 资源永久移动到新 URL。 |
| `302` | Found | 资源临时移动到另一个 URL。 |
| `304` | Not Modified | 资源自上次请求未更改；使用缓存版本。 |
| `400` | Bad Request | 服务器无法理解请求（语法或数据无效）。 |
| `401` | Unauthorized | 需要认证凭据。 |
| `403` | Forbidden | 即使凭据有效，客户端无权限访问资源。 |
| `404` | Not Found | 服务器上不存在资源。 |
| `500` | Internal Server Error | 服务器发生意外错误。 |
| `502` | Bad Gateway | 服务器从上游服务器收到无效响应。 |
| `503` | Service Unavailable | 服务器暂时无法处理请求。 |

## HTTP 请求-响应模型

请求-响应周期由以下组件协同工作：

- **客户端**：发送 HTTP 请求的浏览器、移动应用或 API 客户端。
- **HTTP 方法**：要执行的操作（`GET`、`POST`、`PUT`、`PATCH`、`DELETE`）。
- **URL**：请求资源的地址。
- **头部**：请求或响应的元数据（例如，`Authorization`、`Content-Type`）。
- **正文**：随 `POST`、`PUT`、`PATCH` 或 `DELETE` 请求发送的数据（通常是 JSON 或表单数据）。
- **服务器**：处理请求并返回响应的后端系统。
- **状态码**：指示成功、错误或重定向的数字代码。
- **响应正文**：实际返回的数据，通常是超文本标记语言、JSON 或 XML。

### 请求-响应周期的六个步骤

1. **客户端发起请求**，包含方法、URL、头部和可选正文。
2. **请求通过互联网传送到服务器**。
3. **服务器处理请求**：检查方法、URL、认证，并在需要时查询数据库。
4. **服务器准备响应**，包含状态码、头部和可选正文。
5. **响应发送回客户端**。
6. **客户端处理响应**：成功时（2xx）渲染数据，失败时（4xx/5xx）显示错误信息。

## HTTP 响应资源和内容类型

`Content-Type` 头部告诉浏览器如何处理响应中的资源。

| 资源 | 内容类型 |
| --- | --- |
| 超文本标记语言 | `text/html` |
| CSS | `text/css` |
| JavaScript | `application/javascript` |
| JSON | `application/json` |
| XML | `application/xml` |
| JPEG | `image/jpeg` |
| PNG | `image/png` |
| GIF | `image/gif` |
| SVG | `image/svg+xml` |
| WebP | `image/webp` |
| MP4 | `video/mp4` |
| MP3 | `audio/mpeg` |
| PDF | `application/pdf` |
| ZIP | `application/zip` |
| WOFF2 | `font/woff2` |

## HTML 表单提交

- 表单使用 `<form>` 元素和 `<input>` 元素收集并提交用户数据。
- 每个输入都需要 `name` 属性，以便服务器识别数据。
- **客户端验证**：使用 `required` 和 `pattern` 属性提供即时反馈。
- **服务器端验证**：无论客户端验证如何，出于安全考虑总是在服务器端验证。
- `action` 属性指定发送表单数据的目标地址。
- `method` 属性指定发送方式（`GET` 或 `POST`；`PUT`、`PATCH`、`DELETE` 需要使用 Fetch API 或 Express 中的 `method-override` 等变通方法）。

### 表单提交流程

1. 用户填写并提交表单。
2. 浏览器执行客户端验证；验证通过则继续。
3. 浏览器使用指定的 `action` 将表单数据发送到 `method` 中的 URL。
4. 服务器接收并处理数据（保存到数据库、认证用户等）。
5. 服务器可能执行额外的服务器端验证。
6. 服务器返回成功消息、错误或重定向。
7. 用户根据服务器响应看到反馈。

## 网络标准模型和标准组织

- **网络标准**：浏览器遵循的规则和规格，用于渲染页面和启用网络功能。
- **WHATWG**：维护超文本标记语言和文档对象模型的标准组织，作为持续更新的“活标准”，由主要浏览器引擎团队推动。
- **W3C**：主要标准组织，负责 CSS、SVG、无障碍指南（WCAG）和众多网络 API。
- **ECMA / TC39**：ECMA 发布 ECMAScript 规格（JavaScript 标准）；TC39 通过五阶段提案流程（阶段 0–4）推进语言特性。
- **Khronos Group**：负责 WebGL 和 WebGL 2，为浏览器带来 GPU 加速的 3D 渲染。

## 制定网络标准的流程

1. **识别需求**：开发者或浏览器工程师记录一个真实且广泛存在的问题。
2. **提案和早期讨论**：非正式撰写想法，收集反馈并评估浏览器厂商兴趣。
3. **编写规格说明**：起草精确、正式的文档，涵盖所有边缘情况、错误条件和安全影响。
4. **审查、反对和迭代**：公开审查草案；厂商提出异议，规格不断修订直至达成共识。
5. **实现和测试**：浏览器厂商在标记下实现功能；共享测试套件验证所有实现行为一致。
6. **最终定稿**：多个独立实现通过测试后，标准被定稿（W3C 推荐、TC39 阶段 4 或 WHATWG 活标准更新）。

## 网络标准功能的生命周期

1. **提案**：有人发现问题并记录，看看是否有其他人也遇到。
2. **孵化**：通过公开讨论对想法进行压力测试；许多提案在此阶段因过于小众或无满意方案而终止。
3. **规格说明**：编写正式规格，将模糊概念转化为精确、可实现的规则。
4. **互操作性测试**：W3C 网络平台测试套件验证每个浏览器对该功能的实现是否一致。
5. **最终定稿**：多个通过测试的实现意味着该功能作为基础网络平台的一部分发布给用户。
6. **弃用**：被替代或有问题的功能被标记为废弃；由于向后兼容义务，浏览器中移除过程缓慢。

## 网络标准的关键原则

- **开放性**：标准在公开环境中制定，任何人都可以免费实现。
- **无障碍**：网络设计为人人可用，无障碍功能内置于超文本标记语言、CSS 和浏览器行为中。
- **互操作性**：符合标准的页面应在所有浏览器和操作系统中表现一致。
- **向后兼容**：新功能建立在现有功能之上，确保旧页面在现代浏览器中继续工作。
- **隐私和安全性**：可能暴露用户数据的新 API 受到严格审查；平台默认保护用户。
- **分层**：超文本标记语言处理结构，CSS 处理表现，JavaScript 处理行为——各层独立工作。
- **渐进增强**：功能必须首先对所有人有效，然后为拥有更强浏览器或设备的用户提供增强体验。
