# --description--

Express 애플리케이션에서 문제가 발생하면 어떻게 되는지 궁금했던 적 있나요?

데이터베이스 연결이 실패하면 어떻게 될까요? 아니면 사용자가 존재하지 않는 라우트를 방문하면요?

Express에서 오류 처리 방식에 대해 이해하는 것은 안전하고 신뢰할 수 있는 웹 애플리케이션을 만들려면 매우 중요합니다.

이번 수업에서는 Express가 오류를 어떻게 처리하는지, 그리고 실제 코드 예제를 통해 오류를 어떻게 잡고 응답하는지 살펴보겠습니다.

## Express에서 간단한 오류 처리 방법

Express에서 오류 처리는 미들웨어를 기반으로 합니다.

일반 미들웨어 함수처럼, 오류 처리 미들웨어도 `req`, `res`, `next`에 접근할 수 있습니다. 하지만 맨 앞에 하나의 매개변수가 더 포함되어 있습니다: `err`.

구문은 다음과 같습니다:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

이 미들웨어는 미들웨어 스택의 맨 끝에 정의하세요.

Express는 매개변수가 네 개인 것을 보고 오류 핸들러임을 인식합니다 — 일반 미들웨어 함수의 `err`, `req`, `res` 매개변수 외에 새 매개변수 `next`가 추가된 것입니다.

앞으로의 수업에서 오류 처리 미들웨어가 Express 미들웨어 스택에서 어떻게 작동하는지 더 배우게 될 것입니다.

## 비동기 오류 처리 방법

비동기 오류 처리는 사용하는 Express 버전에 따라 다릅니다.

어떤 버전을 사용하든, 모든 라우트 뒤에 오류 처리 미들웨어를 스택의 끝에 정의해야 한다는 점은 변하지 않습니다.

다시 한 번 오류 핸들러는 다음과 같습니다:

```js
// App routes
// ...

// Error-handling middleware
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

## Express 4

많은 프로젝트가 아직 사용하는 Express 4 이하 버전에서는 비동기 오류가 **자동으로 잡히지 않습니다**.

`async`/`await`를 사용하고 `async` 라우트 핸들러 내부에서 오류가 발생하면, 수동으로 `next()`에 전달하지 않는 한 Express가 이를 잡지 못합니다.

Express 4에서 비동기 오류를 올바르게 처리하는 방법은 다음과 같습니다:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err); // Manually pass the error to the error handler
  }
});
```

`try/catch` 블록과 `next(err)`를 잊으면 오류가 다음과 같은 결과를 초래할 수 있습니다:

* 애플리케이션이 충돌하거나
* 처리되지 않은 프로미스 거부 경고가 발생

그래서 Express 4에서는 비동기 로직을 `try`/`catch`로 감싸는 것이 필수적입니다.

일부 개발자는 `try/catch`를 반복하지 않도록 도우미 함수를 만들기도 합니다:

```js
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

그리고 이렇게 사용합니다:

```js
app.get("/user", asyncHandler(async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
}));
```

이렇게 하면 모든 `try/catch` 라우트에서 `async` 블록을 사용할 필요 없이 오류를 제대로 전달할 수 있습니다.

## Express 5

현재 Express 버전은 이 동작을 개선했습니다.

Express 5부터는 `async` 라우트에서 오류가 발생하거나 거부된 프로미스를 반환하면, Express가 자동으로 오류 처리 미들웨어로 전달합니다.

즉, 이 예제는 `try/catch` 없이도 오류를 처리합니다:

```js
app.get("/user", async (req, res) => {
  const user = await getUserFromDatabase();
  res.send(user);
});
```

`getUserFromDatabase()`가 오류를 발생시키면 Express 5가 이를 잡아 자동으로 오류 핸들러에 전달합니다.

수동 `next(err)` 호출이 필요 없습니다.

## 404 오류 처리 방법

404 오류는 조금 다릅니다. 요청과 일치하는 라우트가 없을 때 발생합니다.

다음 미들웨어로 처리할 수 있습니다:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

**중요**: 이 미들웨어는 라우트 뒤에, 오류 처리 미들웨어 앞에 위치해야 합니다.

마지막으로, Express에서 오류 처리 시 모범 사례는 다음과 같습니다:

* 오류 처리 미들웨어는 항상 맨 마지막에 배치하세요.
* 404 오류는 별도로 처리하세요.
* Express 4 이하에서는 라우트나 `next(err)` 로직에서 발생한 오류를 `async`로 전달하세요.

운영 환경에서는 스택 추적을 사용자에게 보내지 마세요. 내부에 상세 내용을 기록하고 사용자에게는 친절한 메시지를 반환하세요.

Express에서 오류 처리는 문제를 감지하고 명확하고 안전하게 대응하는 미들웨어를 설정하는 것입니다. 코드의 버그든, 누락된 라우트든, 애플리케이션이 백업 계획 없이 충돌해서는 안 됩니다.
