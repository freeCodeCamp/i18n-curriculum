## Servidores e o modelo cliente-servidor

- **Servidor**: Um computador ou programa que recebe requisições de outros computadores e envia dados ou serviços de volta pela rede. O termo se refere tanto ao hardware físico quanto ao software que está rodando nele.
- **Modelo cliente-servidor**: Uma arquitetura distribuída onde clientes enviam requisições e servidores as processam e retornam respostas.
- **Cliente**: O processo ou aplicação (por exemplo, um navegador web) que inicia requisições a um servidor.
- **Ciclo requisição-resposta**: O cliente envia uma requisição; o servidor a processa e retorna uma resposta.

### Tipos de servidor

- **Servidores web**: Entregam conteúdo web (HTML, imagens, CSS) para navegadores.
- **Servidores de aplicação**: Lidam com a lógica de negócio e processam a entrada do usuário.
- **Servidores de banco de dados**: Armazenam, recuperam e gerenciam dados.
- **Servidores de e-mail**: Enviam, recebem e armazenam e-mails.
- **Servidores de arquivos**: Gerenciam armazenamento e compartilhamento de arquivos em uma rede.
- **Servidores proxy**: Atuam como intermediários entre clientes e outros servidores.

## DNS (Sistema de nomes de domínio)

- **DNS**: Um sistema hierárquico e descentralizado de nomes que traduz nomes de domínio legíveis por humanos em endereços IP, frequentemente chamado de "lista telefônica" da internet.
- **Endereço IP**: Um identificador numérico único para um dispositivo em uma rede (por exemplo, `127.0.0.1` no IPv4, `0:0:0:0:0:0:0:1` no IPv6).
- **Hierarquia de nomes de domínio** (lida da direita para a esquerda):
  - **TLD (domínio de topo)**: O segmento mais à direita (por exemplo, `.org`, `.com`).
  - **SLD (domínio de segundo nível)**: O nome registrado (por exemplo, `freecodecamp` em `freecodecamp.org`).
  - **Subdomínio**: Identifica uma seção do domínio principal (por exemplo, `www`, `blog`, `api`).

### Processo de resolução DNS

1. O navegador envia o nome do domínio para o **resolvedor recursivo** (geralmente gerenciado pelo provedor de internet).
2. O resolvedor recursivo consulta os **servidores raiz**, que o direcionam para o servidor TLD relevante.
3. O **servidor TLD** retorna o endereço do **servidor de nomes autoritativo** para o domínio.
4. O servidor de nomes autoritativo retorna o **registro DNS** contendo o endereço IP.
5. O resolvedor recursivo entrega o endereço IP ao cliente e **armazena em cache** o registro para acelerar buscas futuras.

## TCP/IP

- **Protocolo**: Um conjunto de regras e padrões que permitem que dispositivos se comuniquem em uma rede.
- **TCP/IP**: Um conjunto de protocolos que determina como os dados são empacotados, enviados, roteados e recebidos.
  - **IP (Protocolo de Internet)**: Roteia pacotes de dados da origem ao destino usando endereços IP. É sem conexão — cada pacote viaja independentemente sem garantia de entrega.
  - **TCP (Protocolo de Controle de Transmissão)**: Divide os dados em segmentos numerados, os reagrupa no destino, verifica erros e pacotes perdidos, e solicita retransmissão se necessário. Garante entrega confiável e ordenada.

## HTTP (Protocolo de transferência de hipertexto)

- **HTTP**: Um protocolo de nível de aplicação que define as regras para comunicação entre clientes e servidores na web.

- **HTTPS**: Uma versão segura do HTTP onde os dados são criptografados usando SSL/TLS antes da transmissão.

### Estrutura da requisição HTTP

- **Linha de requisição**: `METHOD /path HTTP/version` (por exemplo, `GET / HTTP/1.1`)
- **Cabeçalhos**: Pares de metadados chave-valor (por exemplo, `Content-Type: application/json`).
- **Corpo** (opcional): Dados enviados ao servidor, usados com `POST`, `PUT` e `PATCH`.

**Métodos HTTP comuns:**

| Método | Propósito |
| --- | --- |
| `GET` | Buscar dados |
| `POST` | Enviar novos dados |
| `PUT` | Substituir um recurso |
| `PATCH` | Atualizar parcialmente um recurso |
| `DELETE` | Remover um recurso |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### Estrutura da resposta HTTP

- **Linha de status**: Versão HTTP, código de status e frase de motivo (por exemplo, `HTTP/1.1 200 OK`).
- **Cabeçalhos**: Metadados sobre a resposta (por exemplo, `Content-Type: text/html`).
- **Corpo** (opcional): O conteúdo retornado ou explicação de erro.

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## Códigos de status HTTP

Os códigos de status são agrupados pelo seu primeiro dígito:

- **1xx** — Respostas informativas.
- **2xx** — Respostas de sucesso.
- **3xx** — Mensagens de redirecionamento.
- **4xx** — Respostas de erro do cliente.
- **5xx** — Respostas de erro do servidor.

### Códigos comuns

| Código | Nome | Significado |
| --- | --- | --- |
| `200` | OK | Requisição bem-sucedida; conteúdo retornado. |
| `201` | Created | Recurso criado com sucesso. |
| `204` | No Content | Sucesso, mas sem conteúdo para retornar. |
| `301` | Moved Permanently | Recurso movido permanentemente para uma nova URL. |
| `302` | Found | Recurso movido temporariamente para outra URL. |
| `304` | Not Modified | Recurso não modificado desde a última requisição; use a versão em cache. |
| `400` | Bad Request | Servidor não conseguiu entender a requisição (sintaxe ou dados inválidos). |
| `401` | Unauthorized | Credenciais de autenticação necessárias. |
| `403` | Forbidden | Cliente não tem permissão para o recurso, mesmo com credenciais válidas. |
| `404` | Not Found | Recurso não existe no servidor. |
| `500` | Internal Server Error | Erro inesperado no servidor. |
| `502` | Bad Gateway | Servidor recebeu uma resposta inválida de um servidor upstream. |
| `503` | Service Unavailable | Servidor temporariamente incapaz de atender requisições. |

## O modelo de requisição-resposta HTTP

O ciclo de requisição-resposta consiste nestes componentes trabalhando juntos:

- **Cliente**: Navegador, app móvel ou cliente API que envia requisições HTTP.
- **Método HTTP**: A ação a ser realizada (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`).
- **URL**: O endereço do recurso solicitado.
- **Cabeçalhos**: Metadados para a requisição ou resposta (por exemplo, `Authorization`, `Content-Type`).
- **Corpo**: Dados enviados com requisições `POST`, `PUT`, `PATCH` ou `DELETE` (frequentemente JSON ou dados de formulário).
- **Servidor**: O sistema back-end que processa requisições e retorna respostas.
- **Código de status**: Código numérico indicando sucesso, erro ou redirecionamento.
- **Corpo da resposta**: Os dados efetivamente retornados, tipicamente HTML, JSON ou XML.

### Os seis passos do ciclo

1. **Cliente inicia uma requisição** com método, URL, cabeçalhos e corpo opcional.
2. **Requisição viaja até o servidor** pela internet.
3. **Servidor processa a requisição**: verifica método, URL, autenticação e consulta banco de dados se necessário.
4. **Servidor prepara uma resposta** com código de status, cabeçalhos e corpo opcional.
5. **Resposta é enviada de volta** ao cliente.
6. **Cliente trata a resposta**: renderiza dados em caso de sucesso (2xx) ou exibe mensagem de erro em falha (4xx/5xx).

## Envio de formulários HTML

- Formulários coletam e enviam dados do usuário usando o elemento `<form>` com elementos `<input>`.
- Cada entrada precisa de um atributo `name` para que o servidor identifique os dados.
- **Validação no cliente**: Use os atributos `required` e `pattern` para feedback imediato.
- **Validação no servidor**: Sempre valide no servidor por segurança, independentemente das verificações no cliente.
- O atributo `action` especifica para onde enviar os dados do formulário.
- O atributo `method` especifica como enviá-los (`GET` ou `POST`; `PUT`, `PATCH`, `DELETE` exigem soluções alternativas como a Fetch API ou `method-override` no Express).

### Fluxo de envio de formulário

1. Usuário preenche e envia o formulário.
2. Navegador executa validação no cliente; prossegue se válido.
3. Navegador envia dados do formulário para a URL em `action` usando o método `method` especificado.
4. Servidor recebe e processa os dados (armazenar no banco, autenticar usuário etc.).
5. Servidor pode executar validação adicional no servidor.
6. Servidor responde com mensagem de sucesso, erro ou redirecionamento.
7. Usuário vê o feedback baseado na resposta do servidor.

## Modelo de padrões web e órgãos de padronização

- **Padrões web**: Regras e especificações que os navegadores seguem para renderizar páginas e habilitar recursos web.
- **WHATWG**: Órgão de padrões que mantém o HTML e o DOM como um "padrão vivo" continuamente atualizado, conduzido pelas principais equipes de motores de navegador.
- **W3C**: Organização principal de padrões, responsável por CSS, SVG, diretrizes de acessibilidade (WCAG) e várias APIs web.
- **ECMA / TC39**: ECMA publica a especificação ECMAScript (o padrão JavaScript); TC39 avança recursos da linguagem por meio de um processo de proposta em cinco etapas (estágio 0–4).
- **Khronos Group**: Responsável pelo WebGL e WebGL 2, que trazem renderização 3D acelerada por GPU para o navegador.

## Processo para criar padrões web

1. **Identificação de uma necessidade**: Um desenvolvedor ou engenheiro de navegador documenta um problema real e amplamente compartilhado.
2. **Proposta e discussão inicial**: A ideia é escrita informalmente para coletar feedback e avaliar o interesse dos fornecedores de navegador.
3. **Redação da especificação**: Um documento preciso e formal é elaborado cobrindo todos os casos extremos, condições de erro e implicações de segurança.
4. **Revisão, objeção e iteração**: O rascunho é revisado publicamente; fornecedores levantam objeções e a especificação é revisada até consenso.
5. **Implementação e testes**: Fornecedores implementam o recurso atrás de uma flag; um conjunto de testes compartilhado verifica se todas as implementações se comportam idênticas.
6. **Finalização**: Quando múltiplas implementações independentes passam nos testes, o padrão é finalizado (Recomendação W3C, estágio 4 do TC39 ou atualização do padrão vivo WHATWG).

## Ciclo de vida dos recursos de padrões web

1. **Proposta**: Alguém identifica um problema e o escreve para descobrir se outros compartilham.
2. **Incubação**: A ideia é testada por meio de discussão aberta; muitas propostas morrem aqui se forem muito específicas ou nenhuma solução satisfizer todos.
3. **Especificação**: Uma especificação formal é escrita que transforma conceitos vagos em regras exatas e implementáveis.
4. **Testes de interoperabilidade**: O conjunto de testes W3C Web Platform Tests verifica se todos os navegadores implementam o recurso de forma idêntica.
5. **Finalização**: Múltiplas implementações aprovadas significam que o recurso é disponibilizado aos usuários como parte da plataforma web básica.
6. **Descontinuação**: Recursos substituídos ou problemáticos são marcados para descontinuação; a remoção dos navegadores é lenta devido a obrigações de compatibilidade retroativa.

## Princípios chave dos padrões web

- **Abertura**: Os padrões são desenvolvidos abertamente e implementados sem royalties por qualquer pessoa.
- **Acessibilidade**: A web é projetada para ser usada por todos, com acessibilidade incorporada em HTML, CSS e comportamento do navegador.
- **Interoperabilidade**: Uma página construída conforme o padrão deve se comportar idêntica em todos os navegadores e sistemas operacionais.
- **Compatibilidade retroativa**: Novas capacidades são adicionadas sobre as existentes para que páginas antigas continuem funcionando em navegadores modernos.
- **Privacidade e segurança**: Novas APIs que podem expor dados do usuário enfrentam rigorosa análise; a plataforma protege os usuários por padrão.
- **Camadas**: HTML cuida da estrutura, CSS da apresentação e JavaScript do comportamento — cada camada funciona independentemente.
- **Aprimoramento progressivo**: Recursos devem funcionar para todos primeiro, depois melhorar para usuários com navegadores ou dispositivos mais capazes.
