## HTTPレスポンスステータスコード

- **ステータスコード**は、リクエスト後にクライアントに何が起きたかを伝える短い数字のメッセージです（例：200、404、500など）。

### 1xx — 情報

- **100 Continue** — サーバーがクライアントにリクエストの残りを送信するよう指示します。
- **101 Switching Protocols** — サーバーがプロトコルを切り替えています（WebSocketでよく見られます）。

### 2xx — 成功

- **200 OK** — リクエストが成功しました。
- **201 Created** — 新しいリソースが正常に作成されました。
- **204 No Content** — リクエストは成功しましたが、レスポンスボディはありません（`DELETE`でよく使われます）。

### 3xx — リダイレクト

- **301 Moved Permanently** — リソースのURLが恒久的に変更されました。
- **302 Found** — リソースが一時的に別のURLにあります。
- **304 Not Modified** — クライアントはキャッシュされたバージョンを使えます。変更はありません。

### 4xx — クライアントエラー

- **400 Bad Request** — リクエストが不正または不正確です。
- **401 Unauthorized** — 認証が必要です。
- **403 Forbidden** — 認証はされていますが、権限がありません。
- **404 Not Found** — リクエストされたリソースは存在しません。

### 5xx — サーバーエラー

- **500 Internal Server Error** — 一般的なサーバー側の失敗です。
- **502 Bad Gateway** — プロキシが別のサーバーから無効なレスポンスを受け取りました。
- **503 Service Unavailable** — サーバーが一時的に過負荷または停止しています。

### Expressでのステータスコードの使用

- `res.status(400).send("Bad Request")` — プレーンテキストメッセージで400を送信します。
- `res.status(200).json({ message: "Success" })` — JSONボディで200を送信します。
- `res.status(500).send("Internal Server Error")` — メッセージ付きで500を送信します。

## Expressにおけるエラー処理

- **エラー処理ミドルウェア**は4つのパラメータを持ちます：`(err, req, res, next)`。この4パラメータのシグネチャがExpressにエラーハンドラであることを示します。
- エラー処理ミドルウェアは、すべてのルートの後、ミドルウェアスタックの**最後に配置**してください。

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Express 4の非同期エラー処理

- Express 4では非同期エラーは**自動的にキャッチされません**。
- 非同期ロジックを`try/catch`でラップし、エラーをエラーハンドラに渡すために`next(err)`を呼び出してください。

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- `asyncHandler`ラッパーを使うと、すべてのルートで`try/catch`を繰り返す必要がなくなります：

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Express 5の非同期エラー処理

- Express 5では、非同期エラーや拒否されたPromiseは**自動的にエラーハンドラに渡されます**。手動で`next(err)`する必要はありません。

### 404エラーの処理

- ルートの後、エラーハンドラの前にキャッチオールミドルウェアを追加して、マッチしないルートを処理します：

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### エラー処理のベストプラクティス

- エラー処理ミドルウェアは必ず最後に配置してください。
- 404エラーは他のエラーとは別に処理してください。
- 本番環境ではエラーの詳細を内部でログに記録し、スタックトレースをユーザーに送信しないでください。

## Expressでのデバッグとログ記録

- **組み込みのデバッグ**：`DEBUG`環境変数を設定するとExpressの組み込みデバッグ出力が有効になります。
  - `DEBUG=express:* node index.js` — すべてのExpressデバッグメッセージを有効にします。
  - `DEBUG=express:router node index.js` — ルーター関連のデバッグ出力のみ表示します。
  - PowerShell: `$env:DEBUG = "express:*"; node index.js`
  - CMD: `set DEBUG=express:* && node index.js`
- **`morgan`**：HTTPリクエスト（メソッド、URL、ステータスコード、レスポンスタイム）をログに記録する人気のミドルウェアです。

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- **デバッグ**はExpressがリクエストやミドルウェアをどのように処理しているかを追跡し、本番ログを煩雑にしません。
- **ログ記録**はリクエスト全体の状況を記録し、バグの発見やトラフィックの理解に役立ちます。

## ヘルスチェックとグレースフルシャットダウン

### ヘルスチェック

- **ヘルスチェックルート**はアプリが正常に動作しているかを報告します。
- ロードバランサーやオーケストレーター（例：Kubernetes、AWS Elastic Beanstalk）がこのルートをポーリングしてサービスの健全性を判断します。
- **200**のレスポンスは正常を意味し、レスポンスがないと自動的に再起動されます。

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### グレースフルシャットダウン

- **グレースフルシャットダウン**は、接続を即座に切断するのではなく、既存のリクエストを完了させてからアプリを停止させます。
- プロセス終了時に送信される`SIGTERM`シグナルを監視し、`server.close()`を呼び出してください：

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- ローカル開発中にCtrl+Cが押されたときに送信される`SIGINT`シグナルを監視し、`SIGTERM`と同じシャットダウン処理を呼び出してください。

### シャットダウン前にアプリを不健康としてマークする

- シャットダウン中はヘルスチェックで**503**を返し、ロードバランサーが新しいトラフィックをアプリにルーティングしないようにします。

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
