# --description--

**REST** تعني نقل الحالة التمثيلية. فكر فيها كمجموعة من الإرشادات التي تصف كيف يجب أن تتواصل عملاء وخوادم الويب مع بعضها عبر الإنترنت.

في هذا الدرس، ستتعلم ما هو REST، مفاهيمه الأساسية، وكيف يعمل.

تعرف REST طريقة لتفاعل الأنظمة المختلفة باستخدام تقنيات الويب القياسية. مع REST، يتواصل العملاء مثل متصفحات الويب، تطبيقات الهواتف، وواجهات برمجة التطبيقات مع الخادم باستخدام طرق HTTP الشائعة مثل `GET`، `POST`، `PUT`، `PATCH`، و`DELETE`، التي ناقشناها في درس سابق.

تشمل بعض المفاهيم الأساسية لـ REST **العدمية الحالة**، **الموارد**، **طرق HTTP**، و**رموز الاستجابة أو الحالة**.

**العدمية الحالة** تعني أن كل طلب يرسله العميل إلى الخادم يُعالج بشكل مستقل. لا يحتفظ الخادم بذاكرة للطلبات السابقة. لذلك، يجب أن يتضمن كل طلب كل المعلومات اللازمة لمعالجته. على سبيل المثال، إذا كان المستخدم مُوثَّقًا، يجب إرسال رمز التوثيق مع كل طلب.

في REST، يُعامل كل شيء كمورد. المورد هو قطعة بيانات يمكن الوصول إليها أو تعديلها. يُحدد كل مورد بواسطة عنوان URL، مثل `/products` لكل المنتجات، أو `/products/1` لمنتج معين.

فيما يلي نظرة عامة على طرق HTTP، رموز الحالة، ومثال خطوة بخطوة لكيفية عمل بنية REST.

## طرق HTTP

تحدد طرق HTTP الإجراء الذي يُنفذ على المورد:

* `GET`: استرجاع مورد
    
* `POST`: إنشاء مورد جديد
    
* `PUT / PATCH`: تحديث مورد موجود
    
* `DELETE`: حذف مورد
    

## رموز حالة HTTP

بعد معالجة الطلب، يرد الخادم برمز حالة يشير إلى النتيجة:

* `200` **(OK)**: تم تنفيذ الطلب بنجاح
    
* `201` **(Created)**: تم إنشاء مورد جديد
    
* `400` **(Bad Request)**: أرسل العميل بيانات غير صحيحة
    
* `404` **(Not Found)**: المورد المطلوب غير موجود
    
* `500` **(Internal Server Error)**: حدث خطأ في الخادم
    

## كيف يعمل REST: مثال خطوة بخطوة

### 1\. يرسل العميل طلبًا

* يرسل العميل طلب HTTP باستخدام طريقة مثل `GET` أو `POST`
    
* يتضمن الطلب عنوان URL يحدد المورد
    
* قد يتضمن أيضًا أقسامًا علوية، بيانات وصفية، وجسم الطلب
    

### 2\. يستقبل الخادم الطلب

* يتحقق الخادم من صحة تنسيق الطلب والتوثيق (إذا كان مطلوبًا)
    
* يحدد المورد المستهدف من عنوان URL
    
* يحدد الإجراء المطلوب بناءً على طريقة HTTP
    
* يفحص الأذونات والتحكم في الوصول
    

### 3\. ينفذ الخادم الإجراء

* `GET`: استرجاع المورد
    
* `POST`: إنشاء مورد جديد
    
* `PUT / PATCH`: تحديث المورد
    
* `DELETE`: حذف المورد
    
* تنفيذ أي منطق ضروري آخر
    

### 4\. يبني الخادم الاستجابة

* يختار الخادم رمز حالة مناسب (مثل `200`، `201`، أو `404`)
    
* ينسق بيانات الاستجابة، عادةً بصيغة **JSON**
    
* يضيف الأقسام العلوية ذات الصلة مثل `Content-Type` أو `Cache-Control`
    

### 5\. يرسل الخادم الاستجابة

* تتضمن الاستجابة رمز الحالة، الأقسام العلوية، وجسم اختياري
    
* لا يُخزن أي حالة للعميل على الخادم (العدمية الحالة)
    

### 6\. يتعامل العميل مع الاستجابة

* يتحقق العميل من رمز الحالة لتحديد النجاح أو الفشل
    
* يعالج أي بيانات أُرسلت من الخادم
