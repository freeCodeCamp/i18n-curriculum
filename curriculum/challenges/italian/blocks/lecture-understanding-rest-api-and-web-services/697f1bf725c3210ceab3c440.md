# --description--

I web service e le API REST alimentano la maggior parte delle app e dei siti web che usi ogni giorno. Permettono a diverse applicazioni di comunicare tra loro e condividere dati su una rete, solitamente internet.

Un’API (Application Programming Interface) è un modo generale per il software di comunicare con un altro software. Un web service è un tipo specifico di API che usa tecnologie web, come HTTP, per scambiare dati su una rete, di solito internet. Per questo motivo, ogni web service è un’API, ma non ogni API è un web service. Le API REST sono un tipo comune di web service che seguono un insieme specifico di regole su come i dati vengono richiesti e restituiti.

Questa lezione si concentrerà su cosa sono i web service a un livello generale, mentre la prossima entrerà più nel dettaglio su come funzionano le API REST.

Puoi pensare a un web service come a un ponte software-software. Non ha un’interfaccia utente e non è pensato per essere usato direttamente. Invece, si concentra interamente sullo scambio di dati tra applicazioni.

Perché un sistema sia considerato un web service, deve avere le seguenti caratteristiche:

* **Funziona su un server**: un web service deve essere ospitato online come un sito web, ma invece di pagine visibili agli utenti, espone dati o qualche funzionalità
    
* **Comunicazione macchina-macchina**: un web service non è fatto per essere visto direttamente dagli esseri umani, ma per essere consumato da app mobili e app web, così da esporre dati o funzionalità che l’utente può infine visualizzare
    
* **Indipendente dal linguaggio**: un web service può essere scritto in qualsiasi linguaggio come Java o C#, e qualsiasi app dovrebbe poter interagire con esso, anche se è scritto in un linguaggio diverso
    
* **Indipendente dalla piattaforma**: non importa su quale piattaforma girano le macchine. Per esempio, un web service che gira su Linux non dovrebbe impedire a un dispositivo con Windows di comunicare con esso
    
* **Usa protocolli web standard**: un web service dovrebbe tipicamente usare HTTP/HTTPS per trasferire dati su internet, FTP per il trasferimento di file e SMTP per le email
    

**REST** e **SOAP** sono due dei tipi di web service più conosciuti, anche se esistono altri approcci, come **XML-RPC**.

**REST**, che sta per **trasferimento di stato rappresentazionale**, è di gran lunga il più usato oggi. La maggior parte delle applicazioni moderne si basa su web service RESTful perché sono semplici, flessibili e funzionano in modo naturale con il web.

Le API REST espongono risorse, che di solito sono rappresentate da percorsi URL come `/users` o `/users/20`. Quando un client richiede una risorsa, il server restituisce una rappresentazione di quella risorsa. Questa rappresentazione è spesso in JSON, ma può anche essere in formati come XML o anche testo semplice.

I client possono specificare il formato desiderato usando intestazioni HTTP, come l’intestazione `Accept`. Per esempio, un client può richiedere una risposta JSON impostando il valore `Accept` appropriato.

```bash
Accept: application/json
```

E puoi specificare che vuoi XML in questo modo:

```bash
Accept: application/xml
```

La risorsa che ricevi può apparire così se è JSON:

```json
{
  "id": 20,
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "createdAt": "2025-09-22T10:30:00Z"
}
```

E così, se è XML:

```xml
<user>
  <id>20</id>
  <name>John Doe</name>
  <email>john@example.com</email>
  <role>admin</role>
  <createdAt>2025-09-22T10:30:00Z</createdAt>
</user>
```

SOAP sta per **Simple Object Access Protocol**. Usa solo il formato dati XML ed è il più comune nei sistemi aziendali che richiedono standard e sicurezza più rigorosi.

Ecco come potrebbero apparire i dati SOAP:

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
   <soapenv:Body>
      <getUser>
         <userId>20</userId>
      </getUser>
   </soapenv:Body>
</soapenv:Envelope>
```

XML-RPC usa XML per codificare le sue richieste e risposte. Questo lo rende leggero ma meno flessibile di REST.

Per quanto riguarda UDDI, non è un servizio di scambio dati ma una directory basata su XML dove le aziende possono elencare, scoprire e descrivere i loro web service disponibili.
