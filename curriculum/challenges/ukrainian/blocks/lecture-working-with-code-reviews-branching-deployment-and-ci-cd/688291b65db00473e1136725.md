# --description--

CI означає безперервну інтеграцію, а CD — безперервну доставку або безперервне розгортання. Разом ці процеси дозволяють автоматично розгортати зміни у вашому застосунку після того, як переконаються, що ці зміни працюють.

У CI-пайплайні ви можете запускати лінтер і тести для коду. CI-пайплайни зазвичай запускаються на запитах на злиття, щоб переконатися, що внесені зміни не порушують роботу застосунку, а також на гілці `main`, щоб переконатися, що вона готова до розгортання.

CD-пайплайн також може запускатися на запитах на злиття, якщо ви налаштуєте середовище попереднього перегляду, щоб можна було побачити зміни вживу. Але ви також запускаєте його на вашій гілці `main`, щоб безперервно розгортати зміни, коли їх приймають і об’єднують.

Але як це насправді працює з GitHub Actions? Спершу потрібно зрозуміти, що таке GitHub Actions. GitHub Actions надає тимчасові ранери (уявіть їх як тимчасовий комп’ютер), які дозволяють виконувати конкретні команди.

Дії налаштовуються за допомогою YAML-файлу. Ви можете мати кілька файлів, щоб вказати різні дії для запуску. Наприклад, у вас може бути один файл для CI і інший для CD.

Давайте розглянемо базовий приклад CI:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  lint:
    name: Lint and Test
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Source Files
        uses: actions/checkout@v4

      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10

      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

Це багато інформації, тож розберемо її по частинах. Розглянемо перший розділ:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```

`name` визначає назву самої дії. Вона з’являється в таких областях, як розділ перевірок статусу у запиті на злиття. Властивості `on` визначають так звані «тригери робочого процесу» — це події, які запускають виконання вашої дії.

У цьому прикладі ми запускаємо дію, коли коміти надсилаються до `main`, і коли запит на злиття спрямований до `main`. Це означає, що CI запускається, коли хтось створює запит на злиття, і знову, коли ми його об’єднуємо.

Розділ `jobs` — це місце, де ви визначаєте, що саме робить ваша дія. Ключ `lint:` — це довільний ключ, який визначає внутрішню назву для цієї задачі. Властивість `name` визначає зовнішню назву для цієї конкретної задачі. А властивість `runs-on` визначає, в якому середовищі має запускатися ваша дія — у цьому випадку ми запускаємо її в останній підтримуваній версії Ubuntu.

Властивість `steps` — це місце, де відбувається магія. Цей блок визначає фактичні кроки, які має виконати ваша дія. Розглянемо перший блок там:

```yml
      - name: Checkout Source Files
        uses: actions/checkout@v4
```

Цей крок має дві властивості: `name`, які має мати крок, і властивість `uses`. Властивість `uses` особлива тим, що вона каже раннеру дії отримати публічну дію з GitHub і запустити її, замість того, щоб виконувати команду. Дія `actions/checkout` відповідає за клонування та налаштування репозиторію у вашому раннері.

```yml
      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10
```

Наступні два блоки також використовують сторонні дії. Ці кроки налаштовують Node і менеджер пакетів pnpm. Обидва мають ключ `with`, який можна використовувати для передачі аргументів цим стороннім діям. Уявіть це як передачу аргументів у функцію.

Останні чотири блоки — це місце, де відбувається справжня магія.

```yml
      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

У нас є блок `Install Dependencies`, який встановлює пакети npm. Блок `Lint Source Files` запускає лінтер, блок `Verify Build` перевіряє, чи можна скомпілювати код на Typescript, а блок `Run Tests` запускає юніт-тести.

Важливо знати, що за замовчуванням, якщо один із цих кроків не вдається, решта кроків пропускається, а весь запуск позначається як невдалий.

Куди потрібно поміщати ваші YAML-файли, щоб можна було запускати ці дії? Вони мають бути у каталозі `.github/workflows` у вашому репозиторії, щоб GitHub міг їх знайти і використати.

Існує багато додаткових можливостей дій, і я рекомендую вам прочитати документацію та дослідити існуючі дії в проєктах, таких як репозиторій freeCodeCamp. Але наразі ви маєте достатнє розуміння основ GitHub Actions і CI/CD!
