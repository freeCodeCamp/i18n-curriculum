## Бібліотеки та фреймворки JavaScript

- Бібліотеки та фреймворки JavaScript пропонують швидкі рішення для поширених проблем і пришвидшують розробку, надаючи готовий код.
- Бібліотеки зазвичай більше сфокусовані на вирішенні конкретних завдань, таких як маніпуляції з DOM, обробка подій або керування AJAX-запитами.
- Прикладами бібліотек JavaScript є jQuery та React.
- Фреймворки, натомість, забезпечують більш визначену структуру для створення застосунків. Вони часто мають набір правил і конвенцій, яких розробники повинні дотримуватися.
- Прикладами фреймворків є Angular та Next.js, мета-фреймворк для React.
- **Односторінкові застосунки** (SPA) — це вебзастосунки, які завантажують одну HTML-сторінку і динамічно оновлюють її під час взаємодії користувача без перезавантаження всієї сторінки.
- SPA використовують JavaScript для керування станом застосунку та відтворення вмісту. Часто це робиться за допомогою фреймворків, які надають чудові інструменти для створення складних інтерфейсів користувача.
- Деякі проблеми, пов’язані з SPA:

  - Читачі екрана мають труднощі з динамічно оновлюваним вмістом.
  - URL не змінюється при навігації всередині застосунку, що ускладнює додавання в закладки, повернення назад або спільний доступ до конкретних сторінок.
  - Початкове завантаження може бути повільним, якщо застосунок великий, оскільки всі ресурси потрібно завантажити одразу.

## React

- React — популярна бібліотека JavaScript для створення інтерфейсів користувача та вебзастосунків.
- Основна концепція React — створення багаторазових UI-компонентів, які можуть оновлюватися та відтворюватися незалежно при зміні даних.
- React дозволяє розробникам описувати, як має виглядати UI на основі стану застосунку. React потім оновлює і відтворює потрібні компоненти при зміні даних або стану.

## Компоненти React

- Компоненти — це будівельні блоки React-застосунків, які дозволяють розбивати складні інтерфейси на менші, керовані частини.
- UI описується за допомогою JSX — розширення синтаксису JavaScript, що дозволяє писати код, схожий на HTML, всередині JavaScript.
- Компоненти — це, по суті, JS-функції або класи, які повертають частину UI.

Ось приклад простого React-компонента, який відтворює привітальне повідомлення:

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

Щоб використати компонент, просто викличте:

```jsx
  <Greeting />
```

## Імпорт і експорт компонентів React

- Експорт робить компонент доступним для інших файлів. Імпорт дозволяє файлу використовувати компонент, який був експортований в іншому місці.
- В React існують два типи експорту: експорт за замовчуванням і іменований експорт.

**Експорт за замовчуванням**

- Файл може мати лише один експорт за замовчуванням. Це ідеально для файлу, який містить переважно один компонент.
- Ви можете експортувати компонент як експорт за замовчуванням так:

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default City;
```

- Також можна експортувати на тій же лінії, що й визначення компонента:

```jsx
export default function City() {
  return <p>New York</p>;
}
```

- Щоб імпортувати експорт за замовчуванням, використовуйте ключове слово `import` без фігурних дужок:

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

**Іменовані експорти**

- Іменовані експорти дозволяють файлу ділитися кількома компонентами. На відміну від експорту за замовчуванням, їх потрібно імпортувати з точними іменами, з якими вони були експортовані.
- Ви можете експортувати кілька компонентів з одного файлу так:

```jsx
// Animals.jsx
export function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- Щоб імпортувати іменовані експорти, використовуйте фігурні дужки з точними іменами:

```jsx
import { Cat, Dog } from './Animals';
```

- Ви можете перейменувати іменований експорт під час імпорту, використовуючи ключове слово `as`:

```jsx
import { Cat as Kitty } from './Animals';
```

**Змішаний експорт за замовчуванням і іменований**

- Файл може одночасно мати один експорт за замовчуванням і кілька іменованих експортів:

```jsx
// Animals.jsx
export default function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- При імпорті змішаних експортів експорт за замовчуванням йде першим (без дужок), а потім іменовані експорти (всередині дужок):

```jsx
import Cat, { Dog } from './Animals';
```

## Налаштування проєкту React за допомогою Vite

- Інструменти налаштування проєктів і CLI надають швидкий і простий спосіб почати нові проєкти, дозволяючи розробникам зосередитися на написанні коду, а не на конфігурації.
- Vite — популярний інструмент налаштування проєктів, який можна використовувати з React.
- Щоб створити новий проєкт з Vite, використайте таку команду в терміналі:

```bash
npm create vite@latest my-react-app -- --template react
```

Ця команда створює новий React-проєкт з назвою `my-react-app` за шаблоном React від Vite. У директорії проєкту ви побачите файл `package.json` з переліком залежностей і команд.

- Щоб запустити проєкт, перейдіть у директорію проєкту і виконайте такі команди:

```bash
cd my-react-app # path to the project directory
npm install # installs the dependencies listed in the package.json file
```

- Після встановлення залежностей у вашому проєкті з’явиться нова папка `node_modules`.

- Папка `node_modules` містить усі пакети та бібліотеки, необхідні для вашого проєкту.

- Щоб запустити проєкт, використайте команду:

```bash
npm run dev
```

- Потім відкрийте браузер і перейдіть за адресою `http://localhost:5173`, щоб побачити працюючий React-застосунок.

- Щоб переглянути код стартового шаблону, зайдіть у папку `src` у вашому проєкті, де ви знайдете файл `App.jsx`.

## Передача пропсів у компонентах React

- У React пропси (скорочення від properties) — це спосіб передати дані від батьківського компонента до дочірнього. Цей механізм потрібен для створення багаторазових і динамічних UI-елементів.
- Пропси можуть бути будь-яким значенням JavaScript. Щоб передати пропси від батька до дитини, додайте їх як атрибути при використанні дочірнього компонента в JSX батька. Ось простий приклад:

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

Ви можете передати кілька пропсів, використовуючи оператор розширення `(...)`, попередньо перетворивши їх на об’єкт. Ось приклад:

```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

У цьому коді оператор розширення `{...person}` перетворює об’єкт person на окремі пропси, які передаються компоненту Child.

## Умовне відтворення в React

- Умовне відтворення в React дозволяє створювати динамічні інтерфейси користувача. Його використовують, щоб показувати різний вміст залежно від певних умов або станів у застосунку.
- Існує кілька способів умовного відтворення в React. Один із поширених — використання тернарного оператора. Ось приклад:

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- Інший спосіб — використання логічного оператора AND (`&&`). Це корисно, коли потрібно відтворити вміст лише за умови, що певна умова істинна. Ось приклад:

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

У наведеному коді елемент `h1` відтворюється лише якщо об’єкт user істинний.

Також можна використати прямий оператор `if` таким чином:

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```

## Відтворення списків у React

- Відтворення списків у React — поширене завдання при створенні інтерфейсів користувача.
- Списки можна відтворювати за допомогою методу масиву `map()`, який ітерує масив елементів і повертає новий масив JSX-елементів.
- Наприклад, якщо у вас є масив імен, які потрібно відобразити у вигляді списку, зробіть так:

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- Завжди пам’ятайте додавати унікальний ключ для кожного елемента списку, щоб React міг ефективно керувати оновленнями та відтворенням. Використовуючи ці техніки, ви створите гнучкі, ефективні та динамічні списки у своїх React-застосунках.

## Рядкові стилі в React

- Рядкові стилі в React дозволяють застосовувати CSS-стилі безпосередньо до JSX-елементів за допомогою JavaScript-об’єктів.
- Щоб застосувати рядкові стилі, використовуйте атрибут style на JSX-елементах. Атрибут style приймає об’єкт, де ключі — CSS-властивості у верблюжому регістрі, а значення — відповідні значення. Ось приклад:

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

Ви також можете винести стилі в окремий об’єкт і посилатися на нього в атрибуті `style` так:

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Рядкові стилі підтримують динамічне стилювання, дозволяючи умовно застосовувати стилі залежно від пропсів або стану. Ось приклад умовного застосування стилів на основі пропса:

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- У наведеному коді стилі `color` і `fontSize` умовно встановлюються залежно від пропса `isImportant`.

## Робота з подіями в React

- **Синтетична система подій**: Це спосіб React обробляти події. Вона є обгорткою над нативними подіями, такими як `click`, `keydown` і `submit`. Обробники подій у React використовують верблюжий регістр у назвах (наприклад, `onClick`, `onSubmit` тощо).

Ось приклад використання атрибута `onClick` для елемента `button` у React:

```jsx
function handleClick() {
  console.log("Button clicked!");
}

<button onClick={handleClick}>Click Me</button>;
```

У React функції-обробники подій зазвичай починаються з префікса `handle`, щоб позначити, що вони відповідають за обробку подій, наприклад `handleClick` або `handleSubmit`.

Коли користувач виконує дію, що викликає подію, React передає обробнику об’єкт Synthetic Event. Цей об’єкт поводиться подібно до нативного об’єкта Event у ванільному JavaScript, надаючи властивості, такі як `type`, `target` і `currentTarget`.

Щоб запобігти стандартній поведінці, наприклад оновленню сторінки браузера під час події `onSubmit`, можна викликати метод `preventDefault()`:

```jsx
function handleSubmit(event) {
  event.preventDefault();
  console.log("Form submitted!");
}

<form onSubmit={handleSubmit}>
  <input type="text" />
  <button>Submit</button>
</form>;
```

Також можна обгорнути функцію-обробник у стрілкову функцію так:

```jsx
function handleDelete(id) {
  console.log("Deleting item:", id);
}

<button onClick={() => handleDelete(1)}>Delete Item</button>;
```

## Робота зі станом і хуком `useState`

- **Визначення стану**: У React стан — це об’єкт, який містить дані для компонента. Коли стан оновлюється, компонент повторно відтворюється. React вважає стан незмінним, тобто його не слід змінювати напряму.
- **Хук `useState()`**: Хук `useState` — це функція, яка дозволяє оголошувати змінні стану у функціональних компонентах. Ось базовий синтаксис:

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

У змінній стану є:

- `stateVariable` — поточне значення стану
- `setStateFunction` (функція-сеттер) — оновлює змінну стану
- `initialValue` — встановлює початковий стан

Ось повний приклад для компонента `Counter`:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>

      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```

## Відтворення та компоненти React

- **Визначення**: У React відтворення — це процес, під час якого компоненти з’являються в інтерфейсі користувача (UI), зазвичай у браузері. Процес відтворення складається з трьох етапів: тригер, відтворення і коміт.

Етап тригера відбувається, коли React виявляє зміну, і UI може потребувати оновлення. Зазвичай це пов’язано з оновленням стану або пропсів.

Після тригера React переходить до етапу відтворення. Тут React повторно оцінює компоненти і визначає, що потрібно показати. Для цього React використовує легку копію «реального» DOM — віртуальний DOM. Завдяки віртуальному DOM React швидко перевіряє, що потрібно змінити в компоненті.

Етап коміту — це коли React застосовує підготовлені зміни з віртуального DOM до реального DOM. Іншими словами, це момент, коли ви бачите кінцевий результат на екрані.

## Оновлення об’єктів і масивів у стані

- **Оновлення об’єктів у стані**: Якщо потрібно оновити об’єкт у стані, спочатку створіть новий об’єкт або скопіюйте існуючий, а потім встановіть стан для нового об’єкта. Будь-який об’єкт у стані слід вважати лише для читання. Ось приклад встановлення імені, віку та міста користувача. Функція `handleChange` використовується для обробки оновлень інформації користувача:

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **Оновлення масивів у стані**: При оновленні масивів у стані важливо не змінювати масив напряму за допомогою методів `push()` або `pop()`. Замість цього створюйте новий масив при оновленні стану:

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

Якщо потрібно видалити елементи з масиву, використовуйте метод `filter()`, який повертає новий масив після фільтрації:

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```

## Використання рефів для посилань на значення

- **Атрибут `ref`**: Ви можете отримати доступ до DOM-вузла в React за допомогою атрибута `ref`. Ось приклад використання `ref` для фокусування на елементі `input`. Властивість `current` використовується для доступу до поточного значення цього `ref`:

```jsx
import { useRef } from "react";

const Focus = () => {
  const inputRef = useRef(null);

  const handleFocus = () => {
    if (inputRef.current) {
      inputRef.current.focus();
    }
  };

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Enter text" />
      <button onClick={handleFocus}>Focus Input</button>
    </div>
  );
};

export default Focus;
```

## Робота з хуком `useEffect`

- **Хук `useEffect()`**: В React ефект — це все, що відбувається поза процесом відтворення компонента. Тобто все, що React не обробляє безпосередньо під час відтворення UI. Звичайні приклади — отримання даних, оновлення заголовка вкладки браузера, читання або запис у локальне сховище, визначення місцезнаходження користувача тощо. Ці операції взаємодіють із зовнішнім світом і називаються побічними ефектами. React надає хук `useEffect`, щоб керувати цими побічними ефектами. `useEffect` дозволяє запускати функцію після відтворення або оновлення компонента.

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [dependencies]);
```

Функція ефекту виконується після відтворення компонента, а необов’язковий аргумент `dependencies` контролює, коли ефект запускається.

Зверніть увагу, що `dependencies` може бути масивом «реактивних значень» (стан, пропси, функції, змінні тощо), порожнім масивом або зовсім відсутнім. Ось як ці варіанти впливають на роботу `useEffect`:

- Якщо `dependencies` — масив, що містить одне або кілька реактивних значень, ефект запускається щоразу при їх зміні.

- Якщо `dependencies` — порожній масив, `useEffect` запускається лише один раз при першому відтворенні компонента.

- Якщо `dependencies` пропущений, ефект запускається щоразу при відтворенні або оновленні компонента.

## Як створювати кастомні хуки

- **Кастомні хуки**: Кастомний хук дозволяє винести багаторазову логіку з компонентів, наприклад отримання даних, керування станом, перемикання та побічні ефекти, як-от відстеження онлайн-статусу. У React усі вбудовані хуки починаються зі слова `use`, тому ваш кастомний хук має дотримуватися цієї конвенції.

Ось приклад створення кастомного хука `useDebounce`:

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```

## Робота з формами в React

- **Контрольовані поля вводу**: Це коли значення поля вводу зберігається у стані і оновлюється через події `onChange`. Це дає повний контроль над даними форми і дозволяє миттєву валідацію та умовне відтворення.

```jsx
import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **Неконтрольовані поля вводу**: Замість керування через хук `useState`, неконтрольовані поля в HTML зберігають власний внутрішній стан за допомогою DOM. Оскільки DOM керує значеннями полів, потрібно отримувати їх за допомогою `ref`.

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```

## Робота з хуком `useActionState`

- **Серверні дії**: Це функції, які виконуються на сервері, дозволяючи обробляти форми без необхідності API-ендпоінтів. Ось приклад із застосунку Next.js:

```js
"use server";

async function submitForm(formData) {
 const name = formData.get("name");
 return { message: `Hello, ${name}!` };
}
```

Директива `"use server"` позначає функцію як серверну дію.

- **Хук `useActionState`**: Цей хук оновлює стан на основі результату відправлення форми. Ось базовий синтаксис хука `useActionState`:

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state` — поточний стан, який повертає дія.
- `action` — функція, що запускає серверну дію.
- `isPending` — булевий прапорець, який показує, чи виконується дія зараз.
- `actionFunction` — параметр, що є самою серверною дією.
- `initialState` — параметр, який представляє початковий стан перед запуском дії.
- `permalink` — необов’язковий рядок, що містить унікальний URL сторінки, яку змінює форма.

## Отримання даних у React

- **Варіанти отримання даних**: Існує багато способів отримати дані в React. Можна використовувати нативний Fetch API або сторонні інструменти, як Axios чи SWR.
- **Поширені змінні стану при отриманні даних**: Незалежно від способу отримання даних, потрібно відслідковувати кілька станів. Перший — це самі дані. Другий — чи триває отримання даних. Третій — змінна стану для збереження помилок, які можуть виникнути під час отримання.

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

Оскільки отримання даних — це побічний ефект, найкраще використовувати `Fetch API` всередині хука `useEffect`.

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

Потім можна відобразити повідомлення про завантаження, якщо дані ще не отримані, повідомлення про помилку, якщо сталася помилка, або результати.

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

Якщо хочете використовувати Axios, його потрібно встановити і імпортувати:

```bash
npm i axios
```

```js
import axios from "axios";
```

Потім можна отримати дані за допомогою `axios.get`:

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);

useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await axios.get(
        "https://jsonplaceholder.typicode.com/users"
      );
      setData(res.data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```



```jsx
const FetchTodos = () => {

 );

 if (!data) {

 }
 if (error) {

 }

 return (
   <>
     <h2>Todos</h2>
     <div>

 );
};

export default FetchTodos;
```

## Робота з хуком `useOptimistic`

- **Хук `useOptimistic`**: Цей хук використовується для підтримки адаптивності UI під час очікування завершення асинхронної дії у фоновому режимі. Він допомагає керувати «оптимістичними оновленнями» UI — стратегією, коли ви одразу оновлюєте UI на основі очікуваного результату дії, наприклад очікування відповіді сервера.

Ось базовий синтаксис:

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState` — тимчасовий стан, який оновлюється одразу для кращого користувацького досвіду.
- `addOptimistic` — функція, що застосовує оптимістичне оновлення перед фактичною зміною стану.
- `actualState` — реальне значення стану, отримане в результаті дії, наприклад отримання даних із сервера.
- `updateFunction` — функція, що визначає, як має оновлюватися оптимістичний стан при виклику.

Ось приклад використання хука `useOptimistic` у компоненті `TaskList`:

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**: Використовується для відтворення частини UI і позначення оновлення стану як не термінового переходу. Це дозволяє UI залишатися адаптивним під час дорогих оновлень. Ось базовий синтаксис:

```js
startTransition(action)
```

`action` виконує оновлення стану або запускає логіку, пов’язану з переходом. Це гарантує, що термінові оновлення UI (наприклад, введення тексту або кліки) не блокуються.

## Робота з хуком `useMemo`

- **Мемоізація**: Це оптимізаційна техніка, коли результат дорогих викликів функцій кешується (запам’ятовується) на основі конкретних аргументів. Якщо ті самі аргументи передаються знову, повертається кешований результат замість повторного обчислення.
- **Хук `useMemo`**: Використовується для мемоізації обчислених значень. Ось приклад мемоізації результату сортування великого масиву. `expensiveSortFunction` виконається лише при зміні `largeArray`:

```js
const memoizedSortedArray = useMemo(
  () => expensiveSortFunction(largeArray),
  [largeArray]
);
```

```js
 )
```

## Інструменти управління залежностями

- **Визначення залежності**: У програмному забезпеченні залежність — це коли один компонент або модуль у застосунку залежить від іншого для коректної роботи. Залежності поширені, бо дозволяють розробникам використовувати готові функції або інструменти, створені іншими. Дві основні залежності для React-проєкту — пакети `react` і `react-dom`:

```json
"dependencies": {
  "react": "^18.3.1",
  "react-dom": "^18.3.1"
}
```

- **Визначення менеджера пакетів**: Для керування залежностями у проєкті потрібен менеджер пакетів. Це інструмент для встановлення, оновлення та видалення залежностей. Багато популярних мов програмування, як JavaScript, Python, Ruby і Java, використовують менеджери пакетів. Популярні менеджери для JavaScript — npm, Yarn і pnpm.
- **Файл `package.json`**: Ключовий конфігураційний файл у проєктах, що містить метадані про проєкт, включно з назвою, версією та залежностями. Також визначає скрипти, ліцензії та інші налаштування для керування проєктом і залежностями.
- **Файл `package-lock.json`**: Цей файл фіксує точні версії всіх пакетів, які використовує ваш проєкт. При оновленні пакета нові версії оновлюються і в цьому файлі.
- **Папка `node_modules`**: Містить фактичний код залежностей, перелічених у файлі `package.json`, включно з прямими залежностями проєкту та їхніми залежностями.
- **Dev-залежності**: Пакети, які використовуються лише під час розробки, а не у продакшені. Наприклад, бібліотека для тестування Jest. Її встановлюють як dev-залежність, бо вона потрібна для локального тестування, але не для роботи застосунку в продакшені.

## React Router

- **Вступ**: React Router — стороння бібліотека, що дозволяє додати маршрутизацію у React-застосунки. Спочатку потрібно встановити React Router у існуючий React-проєкт так:

```bash
npm i react-router
```

Потім у файлі `main.jsx` або `index.jsx` налаштуйте структуру маршрутів так:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

`path` і `element` використовуються для зв’язування URL і UI-компонентів. У цьому випадку налаштовуємо маршрут для головної сторінки, що вказує на компонент `App`.

- **Кілька переглядів і налаштування маршрутів**: У більших застосунках часто налаштовують кілька переглядів і маршрутів так:

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

Проп `index` у цих прикладах означає маршрут за замовчуванням для певного сегмента шляху. Тож компонент `Home` відобразиться за кореневим шляхом `/`, а компонент `ProductsHome` — за шляхом `/products`.

- **Вкладені маршрути**: Маршрути можна вкладати один в одного, що призводить до додавання шляху дочірнього маршруту до шляху батьківського.

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

У наведеному прикладі шлях для популярних товарів буде products/trending.

- **Динамічні сегменти**: Динамічний сегмент — це частина URL, яка є змінною.

```jsx
<Route path=":category" element={<Category />} />
```

У цьому прикладі є динамічний сегмент `category`. Коли користувач переходить за адресою, наприклад, products/brass-instruments, відображається компонент `Category`, і можна динамічно отримати відповідні дані на основі сегмента.

- **Хук `useParams`**: Використовується для доступу до динамічних параметрів із URL.

```js
import { useParams } from "react-router";

export default function Category() {
  let params = useParams();
   {/* Accessing the category param: params.category */}
   {/* rest of code goes here */}
}
```

## Передача пропсів глибоко вниз (prop drilling)

- **Визначення**: Prop drilling — це процес передачі пропсів від батьківського компонента до глибоко вкладених дочірніх, навіть якщо деякі з них не потребують цих пропсів.

## Керування станом

- **Context API**: Контекст — це коли батьківський компонент робить інформацію доступною для дочірніх без явної передачі через пропси. `createContext` використовується для створення об’єкта контексту, який представляє контекст, що інші компоненти можуть читати. `Provider` використовується для надання значень контексту дочірнім компонентам.

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**: Redux керує станом, надаючи центральне сховище і суворий контроль оновлень стану. Він використовує передбачуваний патерн з діями, редюсерами та middleware. Дії — це корисне навантаження інформації, що надсилає дані від застосунку до сховища Redux, часто викликані взаємодією користувача. Редюсери — функції, що визначають, як змінюється стан у відповідь на дії, забезпечуючи незмінність стану. Middleware виступає мостом між відправленням дій і редюсером, дозволяючи розширювати функціональність Redux (наприклад, логування, обробка асинхронних операцій) без зміни основного потоку.

- **Zustand**: Це рішення для керування станом ідеальне для малих і середніх застосунків. Воно працює через хук `useStore`, що дозволяє напряму отримувати стан у компонентах і сторінках. Це дає змогу змінювати і отримувати дані без потреби в діях, редюсерах або провайдерах.

## Налагодження компонентів React за допомогою React DevTools

- **Інструменти розробника React**: Це розширення для браузера, яке можна використовувати у Chrome, Firefox і Edge для інспектування компонентів React і виявлення проблем з продуктивністю. Для Safari потрібно встановити пакет `react-devtools` через npm. Після встановлення React DevTools і відкриття React-застосунку у браузері відкрийте інструменти розробника, щоб отримати доступ до двох додаткових вкладок для налагодження React — Components і Profiler.
- **Вкладка Components**: Відображає ієрархію компонентів у вигляді дерева. Тут можна:
  - переглядати ієрархію компонентів застосунку
  - перевіряти і змінювати пропси, стани та значення контексту в реальному часі
  - переглядати вихідний код вибраного компонента
  - виводити дані компонента у консоль
  - інспектувати DOM-елементи компонента
- **Вкладка Profiler**: Допомагає аналізувати продуктивність компонентів. Можна записувати продуктивність, щоб виявити непотрібні повторні відтворення, переглядати тривалість комітів і оптимізувати повільні компоненти.

## React Server Components

- **Визначення**: React Server Components — це компоненти React, які відтворюються виключно на сервері, надсилаючи клієнту лише кінцевий HTML. Це означає, що такі компоненти можуть напряму отримувати доступ до серверних ресурсів і суттєво зменшувати обсяг JavaScript, що надсилається браузеру.

## Відмінності між реальною та сприйманою продуктивністю

- **Сприймана продуктивність**: Це те, як користувачі оцінюють продуктивність сайту з точки зору адаптивності та надійності. Це суб’єктивна оцінка, яку важко кількісно виміряти, але вона дуже важлива, бо досвід користувача визначає успіх або провал сайту.
- **Реальна продуктивність**: Об’єктивна і вимірювана продуктивність сайту. Вимірюється за метриками, як час завантаження сторінки, час відповіді сервера і час відтворення. На ці показники впливають різні фактори, пов’язані з мережею і самим кодом.

## Техніки покращення сприйманої продуктивності

- **Ліниве завантаження**: Ця техніка максимально зменшує початковий час завантаження, завантажуючи неважливі ресурси у фоновому режимі.
- **Мінімізація затримок шрифтів**: Якщо на сайті використовуються кастомні шрифти, варто мінімізувати затримки їх завантаження, щоб уникнути мерехтіння або показу запасного шрифту під час завантаження кастомного. Рекомендується використовувати запасний шрифт, схожий на кастомний, щоб зміна була менш помітною.
- **Використання індикаторів завантаження**: Показ індикатора завантаження під час тривалих процесів одразу після кліку користувача допомагає відчувати зв’язок із процесом і скорочує суб’єктивний час очікування.

## Основні поняття продуктивності

- **Порядок джерел**: Це спосіб структурування HTML-елементів у документі. Визначає, що завантажується першим і суттєво впливає на продуктивність і доступність.

Деякі найкращі практики для порядку джерел:

- Розміщення критичного вмісту, як заголовки, навігація або основний текст, вище у структурі HTML.
- Відкладення неважливих скриптів, наприклад для аналітики або сторонніх віджетів, щоб вони не блокували відтворення.
- Використання прогресивного покращення, щоб базовий досвід працював навіть до завантаження стилів і скриптів. Прогресивне покращення — це підхід до створення сайтів і застосунків, що спочатку працюють з HTML.

Ось приклад хорошого порядку джерел із застосуванням описаних практик:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **Критичний шлях відтворення**: Послідовність кроків, які браузер виконує, щоб перетворити код у пікселі на екрані.
- **Затримка (Latency)**: Час, який потрібен запиту, щоб пройти між браузером і сервером. Висока затримка означає повільне завантаження сторінок.

Способи зменшення затримки:

- Використання CDN (мереж доставки контенту) для подачі файлів із ближчих локацій.
- Увімкнення стиснення, наприклад Gzip, для зменшення розміру файлів.
- Оптимізація зображень і використання лінивого завантаження.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## Покращення INP

- **Визначення**: INP (Interaction to Next Paint) оцінює загальну адаптивність сторінки, вимірюючи час від взаємодії користувача (наприклад, клік або натискання клавіші) до наступного оновлення браузером відображення. Нижчий INP означає більш адаптивну сторінку.

Ось кілька способів покращити INP:

- Зменшити навантаження на головний потік, розбиваючи довгі JavaScript-завдання.
- Використовувати `requestIdleCallback()` для неважливих скриптів. Це поставить функцію у чергу для виклику під час простоїв браузера.
- Відкладене або ліниве завантаження важких ресурсів, як описано раніше.
- Оптимізувати обробники подій. Якщо вони запускаються надто часто або виконують важкі операції, це уповільнює сторінку і збільшує INP. Рішення — дебаунсинг. Дебаунсинг гарантує, що функція виконується лише після того, як користувач припиняє вводити текст на короткий час (наприклад, 300 мс). Це запобігає зайвим обчисленням і покращує продуктивність.

## Як працює відтворення у браузері

- **Як працює відтворення**: Спочатку браузер парсить HTML і будує DOM. Потім обробляє CSS, створюючи CSS Object Model (CSSOM) — ще одну структуру дерева, що визначає, як стилізувати елементи. Нарешті браузер малює пікселі на екрані, відтворюючи кожен елемент на основі обчислених стилів і макету. У складних сторінках це може включати кілька шарів, які композитяться разом для формування кінцевого візуального результату.

## Вплив продуктивності на сталий розвиток

- **Фонова інформація**: Інтернет відповідає приблизно за 2% світових викидів вуглецю — це стільки ж, скільки авіаційна галузь! Кожен байт, що передається, потребує електроенергії — від дата-центрів до пристроїв користувачів. Великі файли і неефективні скрипти означають більше споживання енергії. Високопродуктивний сайт не лише швидший, а й зменшує непотрібні обчислення і витрати енергії.

## Способи зменшення часу завантаження сторінки

- **Оптимізація медіаресурсів**: Великі зображення і відео часто спричиняють повільне завантаження. Оптимізуючи ці ресурси, можна суттєво пришвидшити сайт. Це включає стиснення зображень, використання сучасних форматів, як WebP, і ліниве завантаження.
- **Використання кешування браузера**: Кешування дозволяє браузерам зберігати частини сайту локально, скорочуючи час завантаження для повторних відвідувачів.
- **Мінімізація і стиснення файлів**: Зменшення розміру файлів призводить до швидшого завантаження. Це включає мінімізацію CSS і JavaScript, а також стиснення переданих файлів.

## Покращення "time to usable"

- **Визначення**: "time to usable" — це інтервал від моменту запиту сторінки користувачем до моменту, коли він може ефективно взаємодіяти з нею. Щоб покращити цей показник, можна використовувати ліниве завантаження ресурсів або мінімізувати ресурси, що блокують відтворення.

## Ключові метрики для вимірювання продуктивності

- **First Contentful Paint (FCP)**: Вимірює, як швидко на екрані з’являється перший видимий вміст — текст або зображення. Хороший FCP — менше 1,8 секунди, поганий — понад 3 секунди. Перевірити FCP можна у Chrome DevTools на вкладці продуктивності.
- **Total Blocking Time (TBT)**: Показує, скільки часу головний потік заблокований важкими JavaScript-завданнями. Якщо TBT високий, користувачі відчувають затримки у взаємодії. Щоб покращити TBT, розбивайте довгі завдання і відкладайте неважливі скрипти.
- **Bounce Rate**: Відсоток відвідувачів, які покидають сайт, не взаємодіючи з ним. Високий показник може свідчити про повільне завантаження сторінки.
- **Унікальні користувачі**: Метрика, що відслідковує кількість окремих відвідувачів сайту. Для перегляду Bounce Rate і унікальних користувачів можна використовувати Google Analytics, який допомагає моніторити ці показники і покращувати залучення.

## Поширені інструменти для вимірювання продуктивності

- **Chrome DevTools**: Вбудований інструмент у Google Chrome для аналізу і налагодження продуктивності в реальному часі. DevTools показує час завантаження, використання CPU і затримки відтворення. Особливо корисний для вимірювання First Contentful Paint (FCP) — як швидко користувач бачить перший видимий вміст. Якщо сайт повільний, DevTools допоможе знайти вузькі місця.
- **Lighthouse**: Автоматизований інструмент для перевірки продуктивності, SEO і доступності.
- **WebPageTest**: Дозволяє тестувати, як сайт завантажується з різних локацій і пристроїв. Надає детальний розбір Speed Index, Total Blocking Time та інших ключових метрик. Якщо хочете дізнатися, як реальні користувачі бачать ваш сайт у світі, WebPageTest — ваш інструмент.
- **PageSpeed Insights**: Аналізує сайт і пропонує швидкі покращення для мобільних і десктопних версій. Показує, що уповільнює сайт, і дає конкретні рекомендації, як оптимізувати зображення, видалити скрипти, що блокують відтворення, і зменшити час відповіді сервера. PageSpeed Insights — швидкий і простий спосіб перевірити, як Google бачить продуктивність вашого сайту.
- **Real User Monitoring (RUM)**: Інструменти відстежують поведінку реальних користувачів, показуючи, як вони сприймають сайт. Популярні RUM-інструменти — Google Analytics (відстежує час завантаження і Bounce Rate), New Relic і Datadog (моніторинг продуктивності в реальному часі). Якщо потрібні дані від реальних користувачів, RUM-інструменти незамінні.

## Робота з Performance Web API

- **Визначення**: Performance Web API дозволяють розробникам відстежувати, наскільки ефективно завантажується і реагує вебсторінка безпосередньо в коді. Ці API дають змогу вимірювати час завантаження сторінки, відстежувати затримки відтворення і взаємодії, а також аналізувати час виконання JavaScript.

- **`performance.now()`**: Цей API надає високоточні часові позначки (у мілісекундах) для вимірювання, скільки часу займає завантаження різних частин сайту.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: Цей API дає розбивку кожного етапу завантаження сторінки — від DNS-запиту до `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: Цей API слухає події продуктивності, такі як зсуви макету, довгі завдання і взаємодії користувача.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## Техніки покращення продуктивності CSS

- **CSS-анімації**: Анімація певних CSS-властивостей, як розміри, позиція і макет, запускає процес «перепланування» (reflow), під час якого браузер перераховує позиції і геометрію елементів на сторінці. Це вимагає повторного малювання (repaint), що є ресурсомістким. Тому рекомендується максимально зменшувати кількість CSS-анімацій або принаймні давати користувачу можливість увімкнути чи вимкнути їх.

## Техніки покращення продуктивності JavaScript

- **Розбиття коду (Code Splitting)**: Розбиття JavaScript-коду на модулі, які виконують критичні та некритичні завдання, також допомагає. Так ви зможете завантажувати критичні модулі якомога раніше, а некритичні відкладаєте, щоб сторінка відтворювалася швидше.
- **Маніпуляції з DOM**: Пам’ятайте, що маніпуляції з DOM — це процес динамічної зміни вмісту сторінки за допомогою JavaScript через Document Object Model. Маніпуляції з DOM є ресурсомісткими. Зменшення їх кількості у вашому JavaScript-коді покращить продуктивність.

## CSS-фреймворки

- **CSS-фреймворки**: CSS-фреймворки можуть пришвидшити роботу, створити єдиний візуальний стиль на сайті, забезпечити консистентність дизайну в різних браузерах і зробити CSS-код більш організованим.
- **Популярні CSS-фреймворки**: Серед популярних — Tailwind CSS, Bootstrap, Materialize і Foundation.
- **Можливі недоліки**:
  - CSS від фреймворку може конфліктувати з вашим кастомним CSS.
  - Ваш сайт може виглядати схожим на інші сайти, що використовують той самий фреймворк.
  - Великі фреймворки можуть спричиняти проблеми з продуктивністю.

## Два типи CSS-фреймворків

- **Utility-first CSS-фреймворки**: Ці фреймворки мають невеликі класи з конкретними призначеннями, наприклад для встановлення поля, відступу або кольору фону. Ви можете призначати ці класи безпосередньо HTML-елементам за потреби. Tailwind CSS належить до utility-first CSS-фреймворків.

Ось приклад використання Tailwind CSS для стилізації кнопки.

```html
<button class="bg-blue-500 text-white font-bold py-2 px-4 rounded-full hover:bg-blue-700">
  Button
</button>
```

- **Компонентні CSS-фреймворки**: Ці фреймворки мають готові компоненти з попередньо визначеними стилями, які можна додати на сайт. Компоненти доступні в офіційній документації CSS-фреймворку, і їх можна копіювати і вставляти у проєкт. Bootstrap належить до компонентних CSS-фреймворків.

Ось приклад використання Bootstrap для створення групи списку. Замість додавання маленьких класів до HTML-елементів, ви додаєте весь компонент, включно з HTML-структурою.

```html
<div class="card" style="width: 25rem;">
  <ul class="list-group list-group-flush">
    <li class="list-group-item">HTML</li>
    <li class="list-group-item">CSS</li>
    <li class="list-group-item">JavaScript</li>
  </ul>
</div>
```

## Tailwind CSS

Tailwind — utility-first CSS-фреймворк. Замість написання кастомних CSS-правил ви створюєте дизайн, комбінуючи маленькі утилітарні класи безпосередньо у HTML.

### Утиліти адаптивного дизайну

Tailwind використовує префікси, як `sm:`, `md:` і `lg:`, щоб застосовувати стилі на різних розмірах екрана.

```html
<div class="w-full md:w-1/2 lg:flex-row">
  Responsive layout
</div>
```

### Утиліти Flexbox

## CSS-препроцесори

- **CSS-препроцесор**: Інструмент, що розширює стандартний CSS. Компілірує код із розширеним синтаксисом у нативний CSS-файл. Допомагає писати чистіший, багаторазовий, менш повторюваний і масштабований CSS для складних проєктів.
- **Особливості**: Деякі можливості препроцесорів — змінні, міксіни, вкладеність і успадкування селекторів.
- **Популярні CSS-препроцесори**: Sass, Less і Stylus.
- **Можливі недоліки**:
  - Компіляція CSS може створювати додаткове навантаження.
  - Скомпільований код може бути складним для налагодження.

## Sass

- **Sass**: Один із найпопулярніших CSS-препроцесорів. Sass означає «Syntactically Awesome Style Sheets».
- **Підтримувані можливості Sass**: Змінні, вкладені CSS-правила, модулі, міксіни, успадкування і оператори для базових математичних операцій.

## Два синтаксиси Sass

- **Синтаксис SCSS**: SCSS (Sassy CSS) розширює базовий синтаксис CSS. Це найпоширеніший синтаксис для Sass. Файли SCSS мають розширення `.scss`.

Ось приклад оголошення і використання змінної в SCSS.

```scss
$primary-color: #3498eb;

header {
  background-color: $primary-color;
}
```

- **Відступний синтаксис**: Відступний синтаксис — це оригінальний синтаксис Sass, також відомий як «Sass syntax».

Ось приклад оголошення і використання змінної у відступному синтаксисі.

```sass
$primary-color: #3498eb

header
  background-color: $primary-color
```

### Міксіни

- **Міксіни**: Дозволяють групувати кілька CSS-властивостей і їх значень під одним ім’ям і повторно використовувати цей блок CSS-коду по всьому стилю.

Ось приклад оголошення міксіну у синтаксисі SCSS. У цьому випадку міксін називається `center-flex`. Він містить три CSS-властивості для центрування елементів за допомогою flexbox.

```scss
@mixin center-flex {
  display: flex;
  justify-content: center;
  align-items: center;
}
```

Ось приклад використання оголошеного міксіну.

```scss
section {
  @include center-flex;
  height: 500px;
  background-color: #3289a8;
}
```

## Ручне і автоматизоване тестування

- **Ручне тестування**: Тестувальник вручну перевіряє кожну частину застосунку і тестує різні функції, щоб переконатися, що все працює правильно. Якщо під час тестування виявляються помилки, тестувальник повідомляє про них команду розробників для виправлення.
- **Автоматизоване тестування**: Ви можете автоматизувати тести, написавши окрему програму, яка перевіряє, чи поводиться застосунок відповідно до очікувань.

## Модульне тестування

- **Модульне тестування**: Тестування кожної функції окремо, щоб переконатися, що все працює як треба. Модульні тести також слугують документацією, бо описують очікувану поведінку коду.
- **Принцип єдиної відповідальності**: Рекомендує тримати кожну функцію маленькою і відповідальною за одну задачу.
- **Поширені фреймворки для тестування JavaScript**: Jest, Mocha і Vitest. Jest — популярний фреймворк для модульних тестів.

Ось приклад модульних тестів із Jest.

Спочатку створіть функцію, яка повертає новий відформатований рядок:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

У окремому файлі `getFormattedWord.test.js` напишіть тести, щоб перевірити, чи функція працює правильно. Файл `getFormattedWord.test.js` виглядає так:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **Функція `expect`**: Функція `expect` використовується для тестування значення.
- **Матчер**: Матчер — це функція, що перевіряє, чи поводиться значення очікувано. У наведеному прикладі матчер — `toBe()`. Jest має багато різних матчерів.

Щоб використовувати Jest, спочатку встановіть пакет `jest` за допомогою `npm i jest`. Також додайте скрипт у файл `package.json` так:

```js
"scripts": {
  "test": "jest"
},
```

Потім виконайте команду `npm run test`, щоб запустити тести.

## Життєвий цикл розробки програмного забезпечення

- **Різні етапи життєвого циклу розробки ПЗ**:
  - **Етап планування**: Команда розробників збирає вимоги до майбутньої роботи від зацікавлених сторін.
  - **Етап проєктування**: Команда розробників розбиває вимоги і визначає найкращі підходи для рішень.
  - **Етап реалізації**: Команда розбиває вимоги на керовані завдання і реалізує їх.
  - **Етап тестування**: Включає ручне і автоматизоване тестування нової роботи. Іноді команда тестує застосунок протягом усього етапу розробки, щоб виявити і виправити проблеми.
  - **Етап розгортання**: Команда розгортає нові зміни у середовищі збірки або тестування.
  - **Етап обслуговування**: Включає виправлення проблем, що виникають у користувачів у продакшені.

- **Різні моделі життєвого циклу розробки ПЗ**:
  - **Водоспадна модель**: Кожен етап життєвого циклу має бути завершений перед початком наступного.
  - **Гнучка модель (Agile)**: Орієнтована на ітеративну розробку, розбиваючи роботу на спринти.

## BDD і TDD

- **TDD**: Розробка через тестування — методологія, що наголошує на написанні тестів перед розробкою функцій. Це дає розробникам зворотний зв’язок у реальному часі під час розробки.
- **BDD**: Розробка через поведінку — підхід, що узгоджує серію тестів із бізнес-цілями. Сценарії тестів у BDD мають писатися мовою, зрозумілою як технічним, так і нетехнічним особам. Приклад такої мови — Gherkin.
- **Фреймворки для BDD**: Приклади — Cucumber, JBehave і SpecFlow.

## Твердження в модульному тестуванні

- **Твердження**: Використовуються для перевірки, що код поводиться очікувано.
- **Бібліотеки тверджень**: Chai — поширена бібліотека тверджень. Інші популярні бібліотеки для JavaScript — `should.js` і `expect.js`.

Ось приклад твердження з Chai, що перевіряє, чи повернене значення функції `addThreeAndFour` дорівнює числу 7:

```js
assert.equal(addThreeAndFour(), 7);
```

- **Найкращі практики**: Незалежно від бібліотеки тверджень, слід писати чіткі повідомлення про успіх і помилки, щоб розуміти, які тести не проходять і чому.

## Мокінг, фейкінг і стабінг

- **Мокінг**: Процес заміни реальних даних на фальшиві, які імітують поведінку реальних компонентів. Наприклад, можна мокати відповідь API під час тестування замість постійних викликів API для отримання даних.
- **Стабінг**: Стабами називають об’єкти, що повертають заздалегідь визначені відповіді або фейкові дані для очікуваної поведінки в застосунку. Наприклад, можна стабити поведінку підключення до бази даних у тестах без реального підключення.
- **Фейкінг**: Фейки — спрощені версії реальних компонентів без складності чи побічних ефектів. Наприклад, можна фейкати базу даних, зберігаючи дані в пам’яті замість взаємодії з реальною базою. Це дозволяє імітувати операції з базою в пам’яті, що значно швидше.

## Функціональне тестування

- **Функціональне тестування**: Перевіряє, чи функції і можливості застосунку працюють як очікується. Мета — тестувати систему цілком за різними сценаріями.
- **Нефункціональне тестування**: Зосереджується на продуктивності і надійності.
- **Smoke testing**: Попередня перевірка системи на базові або критичні проблеми перед більш глибоким тестуванням.

## End-to-End тестування

- **End-to-End тестування**: Тестування реальних сценаріїв з точки зору користувача. Допомагає переконатися, що застосунок працює правильно і передбачувано для користувачів. Однак налаштування, проєктування і підтримка таких тестів займає багато часу.
- **Фреймворки для End-to-End тестування**: Playwright — популярний фреймворк від Microsoft. Інші приклади — Cypress, Selenium і Puppeteer.

Ось приклад E2E-тестів із кодової бази freeCodeCamp, де хук `beforeEach` запускається перед кожним тестом. Тести перевіряють, що у меню є посилання на підтримку донорів і спеціальний стилізований кордон навколо аватара:

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## Тестування зручності використання

- **Тестування зручності**: Коли реальні користувачі взаємодіють із застосунком, щоб виявити проблеми з дизайном, UX або функціональністю. Зосереджується на інтуїтивності застосунку для користувачів.
- **Чотири поширені типи тестування зручності**:
  - **Дослідницьке**: Користувачі взаємодіють з різними функціями, щоб краще зрозуміти, як вони працюють.
  - **Порівняльне**: Порівнює UX вашого застосунку з подібними на ринку.
  - **Оцінювальне**: Вивчає, наскільки інтуїтивно застосунок використовувати.
  - **Валідаційне**: Виявляє основні проблеми, що можуть завадити ефективному використанню застосунку.
- **Інструменти для тестування зручності**: Приклади — Loop11, Maze, Userbrain, UserTesting і UXTweak.

## Тестування сумісності

- **Тестування сумісності**: Мета — переконатися, що застосунок працює в різних обчислювальних середовищах.
- **Різні типи тестування сумісності**:
  - **Зворотна сумісність**: Коли ПЗ сумісне з попередніми версіями.
  - **Прогресивна сумісність**: Коли ПЗ і системи працюватимуть з майбутніми версіями.
  - **Сумісність з апаратним забезпеченням**: Здатність ПЗ працювати на різних конфігураціях обладнання.
  - **Сумісність з операційними системами**: Здатність ПЗ працювати на різних ОС, як macOS, Windows, Linux (Ubuntu, Fedora).
  - **Сумісність з мережею**: Здатність ПЗ працювати в різних мережевих умовах — швидкості, протоколи, налаштування безпеки тощо.
  - **Сумісність з браузерами**: Здатність вебзастосунку працювати однаково в різних браузерах — Google Chrome, Safari, Firefox тощо.
  - **Сумісність з мобільними пристроями**: Важливо, щоб застосунки працювали на різних Android і iOS пристроях, включно з телефонами і планшетами.

## Тестування продуктивності

- **Тестування продуктивності**: Перевіряє швидкість, адаптивність, масштабованість і стабільність застосунку під різним навантаженням. Мета — усунути будь-які вузькі місця продуктивності.
- **Різні типи тестування продуктивності**:
  - **Тестування навантаження**: Визначає, як система поводиться під нормальним і піковим навантаженням.
  - **Стрес-тестування**: Перевіряє застосунок при екстремальних навантаженнях і аналізує реакцію системи.
  - **Тестування витривалості (Soak testing)**: Тестування системи під підвищеним навантаженням протягом тривалого часу.
  - **Тестування сплеску (Spike testing)**: Різке збільшення і зменшення навантаження з аналізом реакції системи.
  - **Тестування точки зупинки (Breakpoint testing)**: Поступове збільшення навантаження до моменту, коли система починає відмовляти або деградувати.

## Тестування безпеки

- **Тестування безпеки**: Допомагає виявити вразливості і слабкі місця.
- **Принципи безпеки**:
  - **Конфіденційність**: Захист від розголошення чутливої інформації неавторизованим особам.
  - **Цілісність**: Запобігання несанкціонованим змінам інформації.
  - **Автентифікація**: Перевірка особи користувача для підтвердження права доступу.
  - **Авторизація**: Визначення, які дії дозволені автентифікованим користувачам.
  - **Доступність**: Забезпечення доступу до інформації і сервісів для авторизованих користувачів у потрібний час.
  - **Незаперечність**: Гарантія, що відправник і отримувач мають докази доставки і підтвердження особи відправника, захищаючи від заперечення відправлення.
- **Поширені загрози безпеці**:
  - **Cross-Site Scripting (XSS)**: Атаки, коли зловмисник вставляє шкідливі скрипти у вебсторінку і виконує їх у браузері жертви.
  - **SQL Injection**: Дозволяє зловмисникам вставляти шкідливий код у базу даних.
  - **Атака відмови в обслуговуванні (DoS)**: Зловмисники перевантажують сайт великою кількістю запитів, що уповільнює або зупиняє сервер, роблячи сайт недоступним.
- **Категорії інструментів для тестування безпеки**:
  - **Статичне тестування безпеки застосунків (SAST)**: Аналізує вихідний код для виявлення вразливостей.
  - **Динамічне тестування безпеки застосунків (DAST)**: Взаємодіє з фронтендом застосунку для виявлення потенційних слабких місць. DAST не має доступу до вихідного коду.
- **Тестування на проникнення (pentest)**: Імітація кібератак на застосунок для виявлення вразливостей.

## A/B тестування

- **A/B тестування**: Порівняння двох версій сторінки або застосунку, щоб визначити, яка працює краще. Також відоме як спліт-тестування. Дозволяє приймати рішення на основі даних і постійно покращувати UX.
- **Інструменти для A/B тестування**: Приклади — GrowthBook і LaunchDarkly.

## Альфа- і бета-тестування

Після початкової розробки і тестування важливо, щоб застосунок протестували тестувальники і реальні користувачі. Тут на допомогу приходять альфа- і бета-тестування.

- **Альфа-тестування**: Проводиться обраною групою тестувальників, які перевіряють застосунок на наявність помилок перед випуском на ринок. Альфа-тестування є частиною приймального тестування і використовує як білобокове, так і чорнобокове тестування.
- **Бета-тестування**: Застосунок стає доступним реальним користувачам, які можуть взаємодіяти з ним і надавати відгуки. Це також форма приймального тестування користувачами.
- **Приймальне тестування**: Переконується, що ПЗ відповідає бізнес-вимогам і потребам користувачів перед випуском.
- **Чорнобокове тестування**: Зосереджується лише на очікуваній поведінці застосунку.
- **Білобокове тестування**: Тестувальник знає внутрішні компоненти і проводить тести на них.

## Регресійне тестування

- **Регресія**: Ситуації, коли нові зміни випадково порушують існуючу функціональність.
- **Регресійне тестування**: Допомагає виявити проблеми регресії. Під час нього повторно запускають функціональні тести на частинах застосунку, щоб переконатися, що все працює як раніше.
- **Інструменти для регресійного тестування**: Puppeteer, Playwright, Selenium і Cypress.
- **Техніки регресійного тестування**:
  - **Модульне регресійне тестування**: Перелік елементів, які потрібно тестувати при кожних великих змінах або виправленнях.
  - **Часткове регресійне тестування**: Цільове тестування, щоб переконатися, що нові зміни не порушили конкретні аспекти застосунку.
  - **Повне регресійне тестування**: Запуск тестів на всі функції в кодовій базі. Найбільш трудомісткий і детальний варіант.
- **Повторне тестування**: Використовується для перевірки відомих проблем і підтвердження їх виправлення. На відміну від регресійного тестування, яке шукає невідомі проблеми, що могли з’явитися через останні зміни.

## Що таке TypeScript

- **JavaScript**: JavaScript — мова з динамічною типізацією. Це означає, що змінні можуть отримувати будь-які значення під час виконання. Проблема динамічної типізації — відсутність типобезпеки, що може призводити до помилок.

Наприклад, навіть якщо ваша функція JavaScript очікує масив, ви все одно можете викликати її з числом:

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

Вивід `console` для наведеного прикладу буде `undefined`.

- **TypeScript**: TypeScript розширює JavaScript, додаючи статичну типізацію, що допомагає виявляти помилки типів до запуску коду.

Наприклад, можна визначити тип для параметра `array` так:

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

Це визначення типу каже TypeScript, що параметр `array` має бути масивом рядків. Якщо викликати `getRandomValue` з числом, виникне помилка компіляції.

- **Компілятор**: Спочатку потрібно скомпілювати код TypeScript у звичайний JavaScript. Під час компіляції TypeScript перевіряє код і видає помилки, якщо типи не співпадають.

## Типи даних у TypeScript

- **Примітивні типи даних у TypeScript**: Для примітивних типів `string`, `null`, `undefined`, `number`, `boolean` і `bigint` TypeScript пропонує відповідні ключові слова типів.

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **Масиви**: Можна визначити масив певного типу двома різними синтаксисами.

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **Об’єкти**: Можна визначити точну структуру об’єкта.

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

Якщо потрібен об’єкт з будь-якими ключами, але всі значення мають бути рядками, є два способи визначення:

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **Інші корисні типи в TypeScript**:
  - **`any`**: `any` означає, що значення може мати будь-який тип. Це каже компілятору не звертати уваги на тип змінної.
  - **`unknown`**: `unknown` каже TypeScript, що ви _турбуєтеся_ про тип значення, але фактично не знаєте, який він. `unknown` зазвичай кращий за `any`.
  - **`void`**: Спеціальний тип, який зазвичай використовують для визначення функцій без поверненого значення. Використовують тип повернення `void`.
  - **`never`**: Представляє тип, який ніколи не існує.

- **Ключове слово `type`**: Це схоже на `const`, але замість оголошення змінної можна оголосити тип.

Корисне для оголошення кастомних типів, як об’єднання типів або типів із конкретними значеннями:

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**: Інтерфейси схожі на класи для типів. Вони можуть реалізовувати або розширювати інші інтерфейси, є об’єктними типами і зазвичай кращі, якщо не потрібна специфічна функція `type`.

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **Визначення типу повернення**: Можна також визначити _тип повернення_ функції.

Приклад нижче визначає повернене значення як рядок. Якщо спробувати повернути щось інше, TypeScript видасть помилку компіляції.

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```

## Дженерики

- **Визначення дженеричного типу**: Можна визначити дженеричний тип і посилатися на нього у функції. Це можна уявити як спеціальний параметр, який контролює поведінку типу функції.

Ось приклад визначення дженеричного типу для функції:

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

Синтаксис `<T>` каже TypeScript, що ви визначаєте дженеричний тип `T` для функції. `T` — поширене ім’я для дженериків, але можна використовувати будь-яке.

Потім ви кажете TypeScript, що параметр `array` — це масив значень типу `T`, а повернене значення — один елемент того ж типу.

- **Вказання аргументу типу при виклику функції**: Можна передати аргумент типу у виклик функції, використовуючи кутові дужки між ім’ям функції і параметрами.

Ось приклад передачі аргументу типу у виклик функції:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Це каже TypeScript, що елемент, який ви очікуєте знайти, буде елементом введення.

## Звуження типів

- **Звуження за істинністю**: У прикладі нижче виникає помилка компіляції при спробі доступу до властивості `value` об’єкта `email`, бо `email` _може_ бути `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Можна використати умовний оператор, щоб переконатися, що `email` є _істинним_ перед доступом до властивості:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

Перевірки істинності можуть працювати і в зворотному напрямку:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

Викидання помилки завершує логічне виконання коду, тому коли виконується виклик `console.log()`, TypeScript знає, що `email` _не може_ бути `null`.

- **Опціональний ланцюжок**: Опціональний ланцюжок `?.` — це також форма звуження типів, за тією ж логікою, що доступ до властивості не відбувається, якщо значення `email` є `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **Оператор `typeof`**: Можна використати умовний оператор для перевірки типу змінної за допомогою оператора `typeof`.

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **Ключове слово `instanceof`**: Якщо об’єкт походить з класу, можна використати ключове слово `instanceof` для звуження типу.

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **Кастинг**: Коли TypeScript не може автоматично визначити тип значення, наприклад результат методу `request.json()` у прикладі нижче, виникає помилка компіляції. Один із способів вирішення — кастинг типу, але це послаблює здатність TypeScript виявляти помилки.

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **Захист типу (Type Guard)**: Замість кастингу можна написати захист типу:

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

Синтаксис `user is User` означає, що ваша функція повертає булеве значення, яке при true означає, що значення `user` задовольняє інтерфейс `User`.

## Файл `tsconfig`

- **`tsconfig.json`**: Налаштування компілятора TypeScript зберігаються у файлі `tsconfig.json` у корені проєкту.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

Ось опис опцій компілятора, використаних у наведеному прикладі:

- **`compilerOptions`**: Властивість `compilerOptions` контролює поведінку компілятора TypeScript.
- **`rootDir` і `outDir`**: `rootDir` і `outDir` вказують TypeScript, де знаходяться вихідні файли і куди має записуватися транспільований JavaScript.
- **`lib`**: Властивість `lib` визначає, які типи використовує компілятор, і дозволяє додати підтримку конкретних версій ES, DOM тощо.
- **`module` і `moduleResolution`**: `module` і `moduleResolution` працюють разом, щоб керувати використанням модулів — CommonJS або ECMAScript.
- **`esModuleInterop`**: `esModuleInterop` забезпечує кращу сумісність між CommonJS і ES-модулями, автоматично створюючи простори імен для імпортів.
- **`skipLibCheck`**: Опція `skipLibCheck` пропускає перевірку файлів `.d.ts`, які не імпортуються у вашому коді.
- **`strict`**: Прапорець `strict` вмикає кілька перевірок, наприклад правильне оброблення nullable типів і попередження, коли TypeScript повертається до `any`.
- **`exclude`**: Верхньорівнева властивість `exclude` каже компілятору ігнорувати ці файли TypeScript під час компіляції.
