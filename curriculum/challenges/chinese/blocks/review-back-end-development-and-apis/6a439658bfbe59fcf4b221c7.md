## 使用 Node.js 和事件驱动架构

- **Node.js**：一个开源的跨平台 JavaScript 运行时环境，允许你在浏览器外运行 JavaScript 代码。它被广泛用于构建网络服务器、API 和后端系统。
- **客户端限制**：客户端 JavaScript 对文件系统访问有限，支持复杂应用逻辑有限，并存在潜在的安全风险（例如，不能在客户端代码中暴露数据库凭据）。
- **浏览器与 Node.js 的全局对象**：浏览器中的全局对象是 `window`，它暴露了 DOM 操作、cookie 和浏览器事件。Node 中的全局对象是 `global`，它暴露了用于文件、网络和操作系统交互的内置模块。
- **DOM 访问**：浏览器提供对 DOM API 的访问，而 Node 不提供。相反，Node 可以访问几乎所有系统资源，包括文件系统，而浏览器不能。
- **版本控制**：你可以选择服务器上运行的 Node 版本，但无法控制用户使用的浏览器版本。
- **非阻塞事件驱动架构**：Node 使用单线程和事件循环，高效处理大量同时的请求和 I/O 操作而不阻塞，非常适合实时应用。
- **回调函数**：定义异步操作完成后发生的事情。大量依赖回调是异步 Node 代码难以阅读和调试的原因之一。
- **Node 在后端的优势**：前后端统一语言（JavaScript）、非阻塞架构适合高并发、庞大的 npm 生态系统、免费且开源。
- **Node 在后端的劣势**：单线程，CPU 密集型任务（图像处理、加密、复杂数学）可能阻塞事件循环，过度依赖异步/回调代码降低可读性，npm 包质量和维护参差不齐。
- **NVM（Node 版本管理器）**：一个管理和切换同一台机器上多个 Node.js 版本的工具。在 macOS/Linux 上安装命令：

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **安装 Node 的 LTS 版本**：`nvm install --lts`
- **切换 Node 版本**：`nvm use 20`
- **列出所有已安装的 Node 版本**：`nvm ls`
- **设置默认 Node 版本（macOS/Linux）**：`nvm alias default 22.20.0`
- **nvm-windows**：Windows（非 WSL）上的独立 Node 版本管理项目。从官方仓库下载安装程序，运行 `nvm install lts`，然后 `nvm use <version>`。
- **用 Node 运行 JavaScript 文件**：`node app.js`

## 使用 Node 核心模块

### `fs` 模块

- **用途**：提供创建、读取、写入、追加和删除文件及目录的方法。
- **三种使用方式**：基于回调（异步）、基于 Promise（`fs/promises`）和同步（带 `Sync` 后缀）。同步方法会阻塞事件循环，仅适合脚本使用。
- **`writeFile()`**：写入内容到文件，若文件不存在则创建：

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**：向已有文件追加内容，不覆盖原有内容。
- **`readFile()`**：读取文件内容。传入编码参数 `"utf8"` 返回字符串，省略则返回 `Buffer`。
- **`unlink()`**：删除文件。若文件不存在则抛出 `ENOENT` 错误。

### `Buffer` 模块

- **用途**：直接在内存中处理二进制数据（文件、图像、网络流）。
- **`Buffer.from()`**：从字符串创建缓冲区：

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**：创建一个长度为 `n` 的零填充缓冲区：

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **截断**：写入数据超过缓冲区容量时，超出部分会被静默截断。
- **`Buffer.byteLength(str)`**：返回存储字符串所需的字节数。

### `crypto` 模块

- **用途**：提供加密工具，如哈希、HMAC、加密/解密、数字签名和安全随机数生成。
- **`createHash(algorithm)`**：创建单向哈希（不可逆）。使用 `.update(data).digest("hex")` 提供数据并返回十六进制字符串哈希：

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**：类似 `createHash`，但需要密钥，适合验证数据完整性和认证。
- **`createCipheriv()` / `createDecipheriv()`**：加密和解密数据。均需算法、密钥和初始化向量（IV）：

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**：生成加密安全的随机字节，比 `Math.random()` 更安全，适用于令牌和密钥。
- **`randomInt(min, max)`**：生成指定范围内的安全随机整数，适合一次性密码（OTP）。
- **`sign()` / `verify()`**：使用私钥/公钥对创建和验证数字签名。

### `os` 模块

- **用途**：暴露 Node 运行的操作系统信息。
- **`os.platform()`**：返回操作系统标识，如 `'darwin'`、`'win32'`、`'linux'`。
- **`os.arch()`**：返回 CPU 架构，如 `'x64'`、`'arm64'`。
- **`os.type()`**：返回官方操作系统名称，如 `'Darwin'`。
- **`os.release()`**：返回操作系统内核版本字符串。
- **`os.cpus()`**：返回描述每个逻辑 CPU 核心的对象数组（型号、速度、时间）。
- **`os.uptime()`**：返回系统运行时间（秒）。
- **`os.totalmem()` / `os.freemem()`**：返回系统总内存和可用内存（字节）。
- **`os.userInfo()`**：返回当前用户的 `uid`、`gid`、`username`、`homedir` 和 `shell` 对象。
- **`os.networkInterfaces()`**：返回网络接口对象，包含地址、类型、MAC 和 CIDR 详情。

### `path` 模块

- **用途**：提供跨平台处理文件和目录路径的实用函数。
- **`__filename`**：全局变量，当前文件的绝对路径。
- **`__dirname`**：全局变量，当前文件所在目录的绝对路径。
- **`path.basename(p)`**：返回路径的最后一部分（文件名）。
- **`path.dirname(p)`**：返回路径的目录部分。
- **`path.extname(p)`**：返回文件扩展名，如 `'.js'`。
- **`path.join(...segments)`**：连接路径片段，规范化路径并修正错误的斜线。
- **`path.resolve(...segments)`**：将路径片段解析为从当前工作目录开始的绝对路径。
- **`path.parse(p)`**：返回包含 `root`、`dir`、`base`、`ext` 和 `name` 属性的对象。
- **`path.format(obj)`**：从包含 `dir`、`name` 和 `ext` 属性的对象构建路径字符串。

### `process` 模块

- **用途**：暴露当前 Node.js 进程的信息和控制。它是全局对象，无需导入。
- **`process.env`**：包含所有环境变量的对象。访问特定值如 `process.env.NODE_ENV`。
- **`process.argv`**：命令行参数数组。索引 0 是 Node 可执行文件，索引 1 是脚本路径，索引 2 及以后是用户参数。
- **`process.cwd()`**：返回当前工作目录。
- **进程事件**：

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**：触发自定义警告，可被 `warning` 监听器捕获。

### `stream` 模块

- **用途**：分块处理大量数据，而不是一次性加载到内存。
- **四种流类型**：可读流（分块读取数据）、可写流（分块写入数据）、双工流（同时读写）、转换流（双工流，流经时修改数据）。
- **流的管道连接**：

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **`finish` 事件**：当可写流所有数据刷新完毕时触发。
- **`error` 事件**：流遇到问题时触发。

## npm 简介

- **npm**：由三个部分组成：网站 `npmjs.com`、公共注册表（包数据库）和随 Node.js 安装的命令行工具。
- **用途**：让你安装、管理和发布可复用的 JavaScript 包。npm 是最广泛使用的包管理器。
- **备选方案**：Yarn、PNPM、Bun。

### `package.json` 文件

- **用途**：npm 的配置清单。记录项目名称、版本、描述、入口点（`main`）、作者、许可证、脚本和依赖。
- **交互式创建 `package.json`**：`npm init`
- **默认创建**：`npm init -y`
- **`dependencies`**：应用运行时必需的包（如 `express`、`react`）。
- **`devDependencies`**：仅开发和测试时需要的包（如 `jest`、`nodemon`、`eslint`）。
- **`license`**：使用的法律条款。常见值：`MIT`、`ISC`、`GPL`。

### 语义化版本控制（SemVer）

- **格式**：`MAJOR.MINOR.PATCH`，例如 `4.17.21`。
- **MAJOR**：破坏性变更时递增。
- **MINOR**：新增向后兼容功能时递增。
- **PATCH**：向后兼容的错误修复时递增。
- **插入符号 `^`**：允许更新次版本和补丁版本，但不允许主版本更新。例如：`^4.17.21` 接受 `4.18.0`，但不接受 `5.0.0`。
- **波浪符 `~`**：只允许更新补丁版本。例如：`~1.2.3` 接受 `1.2.4`，但不接受 `1.3.0`。
- **星号 `*`**：匹配任何版本，仅适合测试，绝不用于生产。
- **精确版本**：无前缀，如 `1.2.3`，固定为该版本。

### 安装和移除依赖

- **安装包**：`npm install express`（简写：`npm i express`）
- **安装为开发依赖**：`npm install nodemon -D`
- **卸载包**：`npm uninstall chalk`
- **`node_modules/`**：安装后生成的文件夹，包含实际包代码。不提交版本控制。
- **`package-lock.json`**：自动生成的文件，锁定所有安装包的精确版本（包括子依赖）。必须提交，不要手动编辑。
- **安装指定版本**：`npm install express@4.21.2`
- **查看可用版本**：`npm view express versions`
- **检查过时包**：`npm outdated`
- **更新单个包**（在允许范围内）：`npm update express`
- **强制安装最新版本**：`npm install express@latest`

### 发布到 npm 注册表

- **登录**：`npm login`
- **作用域包**：使用 `@username/package-name` 避免名称冲突。
- **`.npmignore`**：列出发布包时排除的文件（类似 `.gitignore`）。
- **发布无作用域包**：`npm publish`
- **公开发布作用域包**：`npm publish --access public`

## 使用 npm 脚本

- **npm 脚本**：在 `package.json` 的 `"scripts"` 对象中定义的自定义命令。自动化重复任务，如启动服务器、运行测试、代码检查和构建。
- **定义脚本**：

```json
"scripts": {
  "start": "node app.js"
}
```

- **运行脚本**：`npm run start`
- **内置脚本名的简写**（`start`、`test`、`stop`、`restart`）：`npm start`
- **使用 `&&` 的顺序命令**（前一个成功后执行下一个）：

```json
"start": "npm run build && node server.js"
```

- **使用 `;` 的顺序命令**（无论前一个是否成功都执行下一个）：

```json
"start": "npm run build; node server.js"
```

- **使用 `&` 的并发命令**：`"dev": "npm run start:server & npm run start:client"`
- **向脚本传递参数**（使用 `--` 分隔）：`npm start -- 8000`；参数可通过 `process.argv[2]` 访问。

### CommonJS 模块

- **CommonJS**：Node.js 最初的模块系统。同步加载模块。
- **导入** 使用 `require()`：

```js
const { multiply } = require('./math');
```

- **导出** 使用 `module.exports`：

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- 当 `package.json` 中有 `"type": "commonjs"`（或无 `"type"` 字段），或文件扩展名为 `.cjs` 时，默认使用 CommonJS。

### ES 模块（ESM）

- **ES 模块（ESM）**：现代标准化模块格式。异步加载。
- **命名导出**：`export function multiply(a, b) { return a * b; }`
- **默认导出**：`export default function multiply(a, b) { return a * b; }`
- **导入**：`import { multiply } from './math.mjs';`
- 当文件扩展名为 `.mjs` 或 `package.json` 包含 `"type": "module"` 时使用 ESM。

## 理解 HTTP、DNS 和 TCP/IP 的工作原理

### 服务器与客户端-服务器模型

- **客户端-服务器模型**：一种分布式系统架构，组件分为客户端（发送请求的进程，如浏览器）和服务器。
- **服务器**：接收客户端请求并通过网络返回数据或服务的计算机或软件程序。
- **服务器类型**：网络服务器（HTML/CSS/资源）、应用服务器（业务逻辑）、数据库服务器、邮件服务器、文件服务器、代理服务器。

### DNS

- **DNS（域名系统）**，常被称为“互联网的电话簿”：将人类可读的域名（如 `freecodecamp.org`）转换为 IP 地址。
- **域名层级结构**（从右到左）：顶级域（TLD，如 `.org`）→ 二级域（如 `freecodecamp`）→ 子域（如 `www`）。
- **IP 地址**：网络中设备的唯一数字标识符。IPv4 示例：`127.0.0.1`。IPv6 示例：`0:0:0:0:0:0:0:1`。
- **DNS 解析过程**：浏览器 → 递归解析器 → 根服务器 → 顶级域服务器 → 权威名称服务器 → 返回 IP 地址 → 递归解析器缓存并交付给客户端。

### TCP/IP 和 HTTP

- **IP（互联网协议）**：使用 IP 地址将数据包从源头路由到目的地。无连接，每个数据包独立发送。
- **TCP（传输控制协议）**：将数据拆分为编号段，在目的地重新组装，检查错误并请求重传，确保可靠有序传输。
- **HTTP（超文本传输协议）**：定义客户端和服务器如何交换网页内容的应用层协议。
- **HTTPS**：HTTP 的安全版本，使用 SSL/TLS 加密数据。

### HTTP 请求结构

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **请求行**：HTTP 方法 + 路径/URI + HTTP 版本。
- **请求头**：键值对元数据。
- **请求体**（可选）：`POST`、`PUT`、`PATCH`、`DELETE` 请求携带数据。

### HTTP 响应结构

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **状态行**：HTTP 版本 + 状态码 + 状态描述。
- **响应头**：响应的元数据（内容类型、长度、缓存指令）。
- **响应体**（可选）：请求的数据。

### HTTP 状态码类别

- **1xx 信息性**：请求已接收，继续处理。例如：`101 Switching Protocols`。
- **2xx 成功**：`200 OK`、`201 Created`、`204 No Content`。
- **3xx 重定向**：`301 Moved Permanently`、`302 Found`、`304 Not Modified`。
- **4xx 客户端错误**：`400 Bad Request`、`401 Unauthorized`、`403 Forbidden`、`404 Not Found`。
- **5xx 服务器错误**：`500 Internal Server Error`、`502 Bad Gateway`、`503 Service Unavailable`。

## 理解 HTTP 请求-响应模型

- **HTTP 方法**：`GET`（获取）、`POST`（创建）、`PUT`（完整更新）、`PATCH`（部分更新）、`DELETE`（删除）。
- **请求-响应周期**：客户端发送请求 → 服务器验证认证、检查方法和 URL、查询数据库 → 服务器构建响应（状态码 + 可选响应体）→ 客户端处理响应。

### HTTP 响应中返回的资源

每个响应都包含一个 `Content-Type` 头，告诉浏览器如何处理资源：

- **超文本标记语言**：`text/html`
- **CSS**：`text/css`
- **JavaScript**：`application/javascript`
- **JSON**：`application/json`
- **PNG/JPEG/WebP**：`image/png`，`image/jpeg`，`image/webp`
- **MP4/MP3**：`video/mp4`，`audio/mpeg`
- **PDF**：`application/pdf`
- **字体**：`font/woff2`

### HTML 表单提交

- **`action` 属性**：指定表单数据发送的 URL。
- **`method` 属性**：指定 HTTP 方法（`GET` 或 `POST`）。HTML 原生不支持 `PUT`、`PATCH` 或 `DELETE`，可使用 Fetch API 或 Express 的 `method-override` 包。
- **客户端验证**：使用 `required` 和 `pattern` 属性提供即时反馈。服务器端也必须验证。
- **`GET` 表单**：数据附加在 URL 查询参数中（如搜索表单）。
- **`POST` 表单**：数据放在请求体中（如登录和注册表单）。

## 理解 Web 标准模型

### 标准组织

- **WHATWG**：维护 HTML 和 DOM，作为持续更新的“活标准”。决策由主要浏览器引擎团队（Chromium、Gecko、WebKit）驱动。
- **W3C（万维网联盟）**：主要标准组织。监管 CSS、SVG、无障碍指南（WCAG）和众多 Web API。
- **ECMA 国际 / TC39**：发布官方 ECMAScript 规范。TC39 通过五阶段提案流程开发语言（阶段 0 初步想法 → 阶段 4 完全规范）。
- **Khronos Group**：专注图形、计算和媒体。制作 WebGL 和 WebGL 2，实现浏览器中的 GPU 加速 3D 渲染。

### 制定 Web 标准

1. **识别需求**：开发者或组织遇到问题并记录。
2. **提案和早期讨论**：通过非正式 GitHub 议题或说明文档收集兴趣。在 TC39 中为阶段 0/阶段 1。
3. **编写规范**：精确的正式文档，涵盖所有边缘情况、错误条件和安全影响。
4. **审查、异议和迭代**：浏览器厂商公开审查草案，主要浏览器拒绝实现的功能很少成为标准。
5. **实现和测试**：厂商在功能标志下实现，共享测试套件确保所有浏览器行为一致。
6. **定稿**：W3C 称为推荐标准，TC39 称为阶段 4。

### Web 标准功能的生命周期

1. **提案**：识别问题并撰写。
2. **孵化**：公开讨论，许多提案在此阶段夭折。
3. **规范**：编写正式规范，包含精确规则。
4. **互操作性测试**：W3C Web 平台测试套件验证跨浏览器一致性。
5. **定稿**：功能无标志发布，成为基础 Web 平台一部分。
6. **废弃**：被替代或不安全的功能标记为移除，但浏览器为兼容性保留。

### Web 标准的关键原则

- **开放性**：标准公开制定，免费实现。
- **无障碍**：内嵌于 HTML、CSS 和浏览器行为中，WCAG 提供正式指南。
- **互操作性**：符合标准的页面应在所有浏览器表现一致。
- **向后兼容**：旧页面必须在现代浏览器中继续工作。
- **隐私和安全**：可能暴露用户数据的新 API 受到严格审查。
- **分层**：HTML 负责结构，CSS 负责表现，JavaScript 负责行为。各层可独立采用。
- **渐进增强**：功能必须先对所有用户有效，再为更强浏览器用户提供增强。

## 理解 REST API 和 Web 服务

### Web 服务

- **API（应用程序编程接口）**：软件与软件通信的一般机制。
- **Web 服务**：一种使用 Web 协议（HTTP/HTTPS）通过网络交换数据的特定类型 API。所有 Web 服务都是 API，但并非所有 API 都是 Web 服务。
- **特征**：运行于服务器，支持机器间通信，语言无关，平台无关，使用标准 Web 协议。
- **REST**：表现层状态转移，是最广泛使用的 Web 服务风格。简单、灵活，自然适配 HTTP。
- **SOAP**：简单对象访问协议，仅使用 XML，常见于需要严格标准的企业系统。
- **XML-RPC**：使用 XML 编码轻量级请求和响应。

### REST 架构

- **无状态性**：每个请求独立处理。服务器不存储请求间的会话状态。每个请求必须包含所有必要信息（例如，每次调用都带认证令牌）。
- **资源**：一切皆资源，由 URL 标识，如 `/products` 表示集合，`/products/1` 表示具体项。
- **REST 中的 HTTP 方法**：`GET`（获取）、`POST`（创建）、`PUT`/`PATCH`（更新）、`DELETE`（删除）。
- **响应格式**：JSON 最常见，也支持 XML 和纯文本。客户端通过 `Accept` 头指定格式：

```bash
Accept: application/json
```

### 微服务

- **单体应用**：一个庞大代码库，UI、数据库和 API 均在一起。随着复杂度增加，难以维护和扩展。
- **微服务**：将大型应用拆分为小型独立服务，每个负责单一业务能力。服务通过 API 或消息队列通信。
- **优点**：独立开发和部署更快，故障隔离更好，不同服务可用不同语言或框架。
- **缺点**：调试更复杂（问题跨服务），基础设施开销更高，网络通信增加。

## 使用 Express

- **Express.js**：基于 Node.js 的极简且灵活的 Web 框架。比原生 `http` 模块减少大量公式化代码，简化构建服务器和 API。
- **一个基本的 Express 服务器**：

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**：工厂函数，创建 Express 应用实例（`app`）。
- **`app.listen(port, callback)`**：启动服务器，监听指定端口的 HTTP 请求。
- **`req`**：请求对象，包含 URL、头部、参数、请求体和查询数据。
- **`res`**：响应对象，用于向客户端发送数据。
- **中间件**：在接收请求和发送响应之间执行的代码。

## 理解 Express.js 路由

- **路由**：定义应用如何响应客户端访问特定 URL 和 HTTP 方法。
- **`app.get/post/put/delete(path, handler)`**：为相应 HTTP 方法注册处理程序。
- **静态路由**：固定 URL 模式，如 `app.get("/home", handler)`。
- **带参数的动态路由**：使用 `:paramName` 占位符，通过 `req.params` 捕获：

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**：路由参数（`:id` 片段）。
- **`req.query`**：查询字符串参数（`?key=value`）。
- **`req.body`**：请求体（需中间件解析）。
- **`res.send(data)`**：发送响应（文本、HTML、JSON 或二进制）。
- **`res.json(obj)`**：发送 JSON 响应，自动设置 `Content-Type: application/json`。
- **`res.status(code).send(msg)`**：设置状态码并发送响应。
- **`res.redirect(url)`**：重定向客户端到另一个 URL。
- **`res.render(view, data)`**：渲染模板引擎视图并发送 HTML。

### 使用 `app.route()` 链式路由处理器

将同一路径的多个 HTTP 方法分组，减少重复：

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

创建模块化小型应用，用于分组相关路由：

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

### 提供静态文件服务

- **`express.static(directory)`**：内置中间件，从指定目录提供文件服务：

```js
app.use(express.static(path.join(__dirname, "public")));
```

- 通过挂载路径从多个目录提供服务：`app.use("/images", express.static(path.join(__dirname, "images")));`

## 理解错误处理和健康检查

### Express 中的错误处理

- **错误处理中间件**：有四个参数 `(err, req, res, next)`，总是放在中间件栈末尾：

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **Express 4 异步错误**：必须手动用 `try/catch` 捕获并用 `next(err)` 转发：

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **Express 5 异步错误**：自动转发到错误处理中间件，无需手动调用 `next(err)`。
- **404 处理器**：放在所有路由之后，错误处理中间件之前：

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### 调试和日志记录

- **内置调试**：通过设置环境变量 `DEBUG` 启用：`DEBUG=express:* node index.js`
- **`morgan`**：流行的第三方 HTTP 请求日志中间件。记录方法、URL、状态码和响应时间：

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### 健康检查和优雅关闭

- **健康检查路由**：返回 `200 OK` 表示应用正在运行。由负载均衡器和容器编排器使用：

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**：进程应终止时发送的信号。处理它以允许正在进行的请求完成：

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**：在本地开发时按 Ctrl+C 发送的信号。处理方式与 `SIGTERM` 相同。
- **在关闭前标记为不健康**：在关闭期间从健康检查端点返回 `503`，以阻止负载均衡器路由新流量：

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## Express 中间件

- **中间件**：在请求生命周期中执行的函数。它可以访问 `req`、`res` 和 `next`。调用 `next()` 以将控制权传递给下一个中间件，省略则结束该周期。
- **应用级中间件**：通过 `app.use()` 绑定到整个应用。对每个传入请求都运行。
- **路由器级中间件**：通过 `router.use()` 绑定到特定的 `express.Router()` 实例。仅应用于该路由器处理的路由。
- **错误处理中间件**：由四个参数 `(err, req, res, next)` 识别。放置在中间件栈的底部。
- **`express.json()`**：解析带有 JSON 负载的传入请求。使数据可通过 `req.body` 访问。
- **`express.urlencoded({ extended: true })`**：解析 URL 编码的表单数据。
- **`express.static('public')`**：从目录提供静态文件。
- **`cors`**：启用跨源资源共享。通过 `npm install cors` 安装，使用 `app.use(cors())`。
- **`morgan`**：记录 HTTP 请求。使用 `app.use(morgan('tiny'))`。
- 中间件按注册顺序通过 `app.use()` 应用。

## 理解 WebSockets

- **WebSockets**：一种协议，在客户端和服务器之间创建持久的全双工（双向）连接，实现实时通信。
- **HTTP 在实时通信中的问题**：HTTP 需要客户端反复轮询服务器以获取新数据，因此对于聊天应用、实时记分板和多人游戏来说速度慢且效率低。
- **WebSocket 握手**：每个 WebSocket 连接都以 HTTP 请求开始。客户端请求升级，服务器响应 `101 Switching Protocols`。之后连接保持打开，直到任一方发送关闭信号。
- **使用场景**：聊天应用、在线多人游戏、实时体育仪表盘、股票市场信息流、协作工具。

### `ws` 库

- Node.js 没有内置的 WebSocket 服务器；`ws` 库是实现 WebSocket 服务器的常用方式。

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**：向特定客户端发送消息。
- **向所有已连接客户端广播**：

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**：每个连接的属性。`WebSocket.OPEN` 表示连接处于活动状态。

### 发布/订阅架构

- **发布/订阅（Pub/Sub）**：一种消息模式，**发布者**向**主题**发送消息，**订阅者**接收他们订阅的主题消息。发布者和订阅者之间不直接通信。
- **相较于直接通信的优势**：解耦组件，发布者无需知道其订阅者是谁。
- **发布/订阅与 WebSockets**：WebSockets 是通信通道（道路），发布/订阅是决定谁接收哪些消息的消息模式（交通系统）。

## Introduction to Authentication and Authorization

### 认证与授权的区别

- **认证**：验证身份，回答“你是谁？”的问题，必须在授权之前完成。
- **授权**：确定权限，回答“你被允许做什么？”的问题。
- **三种认证因素**：
  - **你知道的东西**：密码、PIN、密保答案。
  - **你拥有的东西**：接收短信验证码的智能手机、认证器应用令牌。
  - **你本身的东西**：指纹、面部识别（生物识别）。
- **授权中的常见角色**：标准用户（有限自助服务）、编辑（内容管理）、管理员（完全系统访问）。
- 在后端代码中，授权通常通过中间件实现，检查用户角色或权限后才授予访问。

### 跨站请求伪造（CSRF）

- **CSRF**：一种攻击，恶意网站诱使已认证用户的浏览器向后端服务器发出未经授权的请求，利用浏览器自动附带的 Cookie。
- **攻击流程**：用户登录 `trusted-bank.com` → 访问 `evil-hacker-site.com` → 恶意网站提交隐藏表单给银行 → 银行服务器看到有效的会话 Cookie 并执行请求。
- **`SameSite=Strict`**：如果请求来自第三方网站，浏览器绝不发送 Cookie。
- **`SameSite=Lax`**：浏览器仅在用户安全导航到目标站点（点击链接）时允许 Cookie，但阻止跨站表单提交。
- **CSRF 令牌**：服务器生成随机且不可预测的令牌嵌入表单。客户端必须在每个状态变更请求中包含该令牌。恶意网站无法从前端读取令牌，因此伪造请求会验证失败。

### JSON Web 令牌（JWT）

- **JWT**：服务器在登录成功后签发的数字签名令牌。客户端存储并随每个后续请求发送，实现无状态认证（无需服务器端会话存储）。
- **JWT 结构**：由三个用点分隔的 Base64URL 编码部分组成：
  - **头部**：令牌类型（`JWT`）和签名算法（例如 `HS256`）。
  - **负载**：关于用户的声明（用户 ID、用户名、角色）。未加密，切勿存储密码或敏感数据。
  - **签名**：头部 + 负载 + 服务器密钥的哈希，防止篡改。

### Passport.js

- **Passport.js**：Node.js 的认证中间件。标准化认证流程，处理底层细节，让开发者专注业务逻辑。
- **策略**：实现特定认证方法的模块化插件。超过 500 种策略可用：`passport-local`（用户名/密码）、`passport-jwt`（JWT）、`passport-google-oauth20`（谷歌登录）、`passport-facebook`、`passport-github2`。
- **集成到 Express**：

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **作为路由中间件使用**：

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- 认证成功后，Passport 将一部分用户数据存储在会话 Cookie 中，并将认证用户对象附加到 `req.user`。

### Helmet.js

- **Helmet.js**：一组安全中间件，通过自动设置、修改或移除 HTTP 响应头来保护 Express 应用。

```js
const helmet = require('helmet');
app.use(helmet());
```

- **移除 `X-Powered-By`**：防止攻击者获知服务器技术。
- **设置 `Content-Security-Policy`**：限制脚本、图像和样式的加载来源，减轻 XSS 攻击。
- **设置 `X-Frame-Options: SAMEORIGIN`**：防止点击劫持，阻止网站被外部 iframe 嵌入。
- **设置 `Strict-Transport-Security`**：强制浏览器使用 HTTPS，保护数据不被截获。

## 理解 Web 应用中的安全性与隐私

### 安全性与隐私

- **安全性**：保护系统和数据免受未授权访问的技术防御机制。安全性提供门锁。
- **隐私**：管理个人数据的收集、使用和共享。隐私提供窗帘。
- 你可以有安全性而无隐私（受保护的数据库仍可能出售数据），但没有安全性就不可能有隐私（未保护的数据无法保持私密）。

### 同源策略与 CORS

- **源**：由协议（如 `https`）、域名（如 `example.com`）和端口（如 `443`）三部分定义。三者都匹配时，两个 URL 才共享同源。
- **同源策略**：浏览器安全规则，阻止一个源的脚本读取不同源的数据，防止恶意网站窃取其他打开标签页的数据。
- **CORS（跨源资源共享）**：允许服务器通过设置 HTTP 头放宽同源策略，信任特定源。
- **在 Express（开发环境）**：`app.use(cors())` 允许所有源。
- **在 Express（生产环境）**：传入带有 `origin` 白名单的 `corsOptions` 对象：

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**：基于 TLS（传输层安全）加密的 HTTP。TLS 取代了旧的 SSL 协议。
- **HTTPS 的三大保障**：
  - **加密**：客户端和服务器之间传输的数据对第三方不可读。
  - **认证**：受信任证书颁发机构颁发的数字证书确认服务器身份。
  - **数据完整性**：消息在传输中不能被篡改而不被检测。
- 现代浏览器 API（如地理位置、推送通知和摄像头访问）要求使用 HTTPS。

### 安全问题产生的原因

- **快速交付压力**：团队为赶进度跳过安全审查，导致漏洞发布。
- **复杂系统集成**：第三方库或云服务引入开发团队未编写的漏洞，形成供应链风险。
- **配置管理不善**：保留默认或出厂设置（默认管理员密码、开放端口）导致系统暴露。
- **人为错误**：钓鱼攻击利用员工点击恶意链接或附件并泄露凭证。
- **组织合并**：不同代码库和基础设施合并暴露安全策略不匹配和遗留漏洞。

### Cookie：存储、安全标记与隐私

- **Cookie 的存储方式**：服务器发送 `Set-Cookie` 头，浏览器保存到磁盘，并在后续对该源的请求中自动返回。
- **`HttpOnly` 标记**：阻止 JavaScript 读取 Cookie 值，防止基于 XSS 的会话劫持。
- **`Secure` 标记**：限制 Cookie 仅通过 HTTPS 连接发送，防止公共网络截获。
- **`SameSite=Strict`**：阻止浏览器在任何跨站请求中发送 Cookie，完全阻止 CSRF 攻击。
- **`SameSite=Lax`**：允许顶层导航（如点击链接）时发送 Cookie，但阻止跨站表单提交和 API 调用。
- **第一方 Cookie**：限定于你访问的网站。
- **第三方 Cookie**：由广告网络在成千上万个网站设置，用于跟踪行为和构建用户画像。现代浏览器默认阻止它们。
- **GDPR 合规**：隐私法规要求显示 Cookie 同意横幅，赋予用户拒绝非必要跟踪 Cookie 的合法权利。

### 常见安全威胁及缓解措施

- **钓鱼**：冒充可信公司发送欺诈邮件窃取凭证。缓解措施包括安全意识培训和部署邮件认证协议，阻止恶意邮件进入收件箱。
- **恶意软件**（病毒、间谍软件、勒索软件）：勒索软件加密文件并索要赎金。缓解措施包括自动更新和反恶意软件软件。
- **凭证填充**：攻击者在不同网站测试数百万被盗用户名和密码组合。主要缓解措施是多因素认证（MFA），要求第二步验证，如一次性密码或认证器代码。
- **DDoS（分布式拒绝服务）**：攻击者用假流量淹没服务器，导致崩溃并阻止合法用户访问。缓解措施包括基于云的流量过滤服务，如 Cloudflare、Akamai 和 AWS Shield。
- **内部威胁**：恶意或疏忽员工窃取或泄露数据。缓解措施包括执行**最小权限原则**（员工仅访问其工作所需资源）和监控系统日志异常活动。

### 内容安全策略和权限策略

- **CSP（内容安全策略）**：HTTP 响应头，白名单批准的脚本、样式和图像源。浏览器收到 CSP 头时，阻止从未列入名单的域加载资源。`script-src` 指令指定允许执行 JavaScript 的域，阻止 XSS 注入攻击。
- **Permissions-Policy**：HTTP 头，启用或禁用页面及其嵌入 iframe 的浏览器功能和设备硬件。取代旧的 `Feature-Policy` 头。
  - `camera=(self)` 限制摄像头访问仅限页面自身源。
  - `geolocation=()` 完全禁用所有用户的地理位置。
- 两个头结合提供分层保护：CSP 防止代码注入，Permissions-Policy 防止未授权硬件访问。

### 基本用户隐私概念

- **PII（个人身份信息）**：任何能直接或间接识别个人的数据。
  - **直接标识符**：全名、家庭地址、身份证号码。
  - **间接标识符**：IP 地址、设备序列号、精确位置历史。
- **机密性**：通过加密和访问控制保护 PII 免受未授权访问。
- **数据最小化**：仅收集完成特定任务所需的最少 PII。
- **目的限制**：为一个目的收集的数据不得未经用户同意用于其他用途（例如出售给广告网络）。
- **跟踪**：使用脚本、设备指纹和浏览器历史日志持续监控用户在线行为。
- **被遗忘权（数据删除）**：用户关闭账户后，有权要求公司永久删除所有个人记录的法律权利。

### `.env` 文件和 `dotenv`

- **`.env` 文件**：项目根目录下的纯文本文件，存储配置密钥，如 API 密钥、数据库密码和端口号，采用键值对形式。务必将其添加到 `.gitignore`，避免密钥进入版本控制。
- **`dotenv` 包**：启动时读取 `.env` 文件，并将值注入 `process.env`：

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **两个关键好处**：凭证不写入源代码（安全性），同一代码库可通过在不同服务器替换 `.env` 文件，在开发、测试和生产环境表现不同（灵活性）。

### 区域隐私法律与合规

- **GDPR（通用数据保护条例）**：保护欧盟个人。要求跟踪前明确获得用户同意，禁止预先勾选同意框。赋予用户查看和永久删除数据的权利。违规罚款严厉。
- **CCPA（加州消费者隐私法案）/ CPRA（加州隐私权法案）**：保护加州居民。侧重于选择退出权而非选择加入。要求主页显示明显的“禁止出售我的个人信息”链接。
- **COPPA（儿童在线隐私保护法案）**：美国联邦法，保护 13 岁以下儿童。要求明确隐私政策和可验证的家长同意，方可收集儿童个人数据。
- **DPA（数据保护法案）**：英国主要数据隐私框架。与英国 GDPR 配合，赋予用户访问和删除个人数据的权利。
- 其他国家有各自法律：尼日利亚有 NDPA（尼日利亚数据保护法），印度有 DPDP 法案（数字个人数据保护法案）。
- **全球团队合规步骤**：
  - 构建基于用户地理位置显示相应 Cookie 通知的定位同意横幅。
  - 实施自动化 DSAR（数据主体访问请求）系统，依法查找、导出或永久删除用户数据。
  - 采用数据最小化和年龄验证门禁，防止未授权跟踪未成年人。
  - 指定数据保护官（DPO）监督隐私架构，作为监管机构的主要联系人。
