## Server e modello client-server

- **Server**: Un computer o programma che riceve richieste da altri computer e invia dati o servizi in risposta sulla rete. Il termine indica sia l'hardware fisico sia il software in esecuzione.
- **Modello client-server**: Un'architettura distribuita in cui i client inviano richieste e i server le gestiscono restituendo risposte.
- **Client**: Il processo o l'applicazione (ad esempio un browser web) che avvia le richieste verso un server.
- **Ciclo richiesta-risposta**: Il client invia una richiesta; il server la elabora e restituisce una risposta.

### Tipi di server

- **Server web**: Forniscono contenuti web (HTML, immagini, CSS) ai browser.
- **Server applicativi**: Gestiscono la logica di business e processano gli input degli utenti.
- **Server database**: Conservano, recuperano e gestiscono dati.
- **Server di posta**: Invia, riceve e archivia email.
- **Server file**: Gestiscono l'archiviazione e la condivisione di file in rete.
- **Server proxy**: Agiscono da intermediari tra client e altri server.

## DNS (Domain Name System)

- **DNS**: Un sistema di denominazione gerarchico e decentralizzato che traduce nomi di dominio leggibili dall'uomo in indirizzi IP, spesso chiamato la "rubrica telefonica" di internet.
- **Indirizzo IP**: Un identificatore numerico unico per un dispositivo in rete (ad esempio, `127.0.0.1` in IPv4, `0:0:0:0:0:0:0:1` in IPv6).
- **Gerarchia dei nomi di dominio** (lettura da destra a sinistra):
  - **TLD (Top-Level Domain)**: Il segmento più a destra (ad esempio, `.org`, `.com`).
  - **SLD (Second-Level Domain)**: Il nome registrato (ad esempio, `freecodecamp` in `freecodecamp.org`).
  - **Sottodominio**: Identifica una sezione del dominio principale (ad esempio, `www`, `blog`, `api`).

### Processo di risoluzione DNS

1. Il browser invia il nome di dominio al **resolver ricorsivo** (di solito gestito dall'ISP).
2. Il resolver ricorsivo interroga i **server root**, che lo indirizzano al server TLD pertinente.
3. Il **server TLD** restituisce l'indirizzo del **server dei nomi autorevole** per il dominio.
4. Il server dei nomi autorevole restituisce il **record DNS** contenente l'indirizzo IP.
5. Il resolver ricorsivo consegna l'indirizzo IP al client e **memorizza nella cache** il record per velocizzare le future ricerche.

## TCP/IP

- **Protocollo**: Un insieme di regole e standard che permettono ai dispositivi di comunicare su una rete.
- **TCP/IP**: Una suite di protocolli che determina come i dati vengono impacchettati, inviati, instradati e ricevuti.
  - **IP (Internet Protocol)**: Instrada i pacchetti di dati dalla sorgente alla destinazione usando indirizzi IP. È senza connessione — ogni pacchetto viaggia indipendentemente senza garanzia di consegna.
  - **TCP (Transmission Control Protocol)**: Suddivide i dati in segmenti numerati, li ricompone a destinazione, verifica errori e pacchetti persi, e richiede ritrasmissioni se necessario. Garantisce una consegna affidabile e ordinata.

## HTTP (HyperText Transfer Protocol)

- **HTTP**: Un protocollo a livello applicativo che definisce le regole per la comunicazione tra client e server sul web.

- **HTTPS**: Una versione sicura di HTTP in cui i dati sono criptati usando SSL/TLS prima della trasmissione.

### Struttura di una richiesta HTTP

- **Riga di richiesta**: `METHOD /path HTTP/version` (ad esempio, `GET / HTTP/1.1`)
- **Intestazioni**: Coppie chiave-valore di metadati (ad esempio, `Content-Type: application/json`).
- **Corpo** (opzionale): Dati inviati al server, usati con `POST`, `PUT` e `PATCH`.

**Metodi HTTP comuni:**

| Metodo | Scopo |
| --- | --- |
| `GET` | Recuperare dati |
| `POST` | Inviare nuovi dati |
| `PUT` | Sostituire una risorsa |
| `PATCH` | Aggiornare parzialmente una risorsa |
| `DELETE` | Rimuovere una risorsa |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### Struttura di una risposta HTTP

- **Riga di stato**: Versione HTTP, codice di stato e frase di motivo (ad esempio, `HTTP/1.1 200 OK`).
- **Intestazioni**: Metadati sulla risposta (ad esempio, `Content-Type: text/html`).
- **Corpo** (opzionale): Il contenuto restituito o la spiegazione di un errore.

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## Codici di stato HTTP

I codici di stato sono raggruppati in base alla loro prima cifra:

- **1xx** — Risposte informative.
- **2xx** — Risposte di successo.
- **3xx** — Messaggi di reindirizzamento.
- **4xx** — Errori lato client.
- **5xx** — Errori lato server.

### Codici comuni

| Codice | Nome | Significato |
| --- | --- | --- |
| `200` | OK | Richiesta riuscita; contenuto restituito. |
| `201` | Created | Risorsa creata con successo. |
| `204` | No Content | Successo ma nessun contenuto da restituire. |
| `301` | Moved Permanently | Risorsa spostata permanentemente a un nuovo URL. |
| `302` | Found | Risorsa spostata temporaneamente a un altro URL. |
| `304` | Not Modified | Risorsa invariata dall'ultima richiesta; usa la versione in cache. |
| `400` | Bad Request | Il server non ha capito la richiesta (sintassi o dati non validi). |
| `401` | Unauthorized | Credenziali di autenticazione richieste. |
| `403` | Forbidden | Il client non ha permessi per la risorsa, anche con credenziali valide. |
| `404` | Not Found | La risorsa non esiste sul server. |
| `500` | Internal Server Error | Errore imprevisto sul server. |
| `502` | Bad Gateway | Il server ha ricevuto una risposta non valida da un server a monte. |
| `503` | Service Unavailable | Il server non è temporaneamente in grado di gestire le richieste. |

## Il modello richiesta-risposta HTTP

Il ciclo richiesta-risposta consiste in questi componenti che lavorano insieme:

- **Client**: Browser, app mobile o client API che inviano richieste HTTP.
- **Metodo HTTP**: L'azione da eseguire (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`).
- **URL**: L'indirizzo della risorsa richiesta.
- **Intestazioni**: Metadati per la richiesta o la risposta (ad esempio, `Authorization`, `Content-Type`).
- **Corpo**: Dati inviati con richieste `POST`, `PUT`, `PATCH` o `DELETE` (spesso JSON o dati di modulo).
- **Server**: Il sistema back-end che elabora le richieste e restituisce risposte.
- **Codice di stato**: Codice numerico che indica successo, errore o reindirizzamento.
- **Corpo della risposta**: I dati effettivamente restituiti, tipicamente HTML, JSON o XML.

### I sei passaggi del ciclo

1. **Il client avvia una richiesta** con metodo, URL, intestazioni e corpo opzionale.
2. **La richiesta viaggia verso il server** attraverso internet.
3. **Il server elabora la richiesta**: verifica metodo, URL, autenticazione e interroga un database se necessario.
4. **Il server prepara una risposta** con codice di stato, intestazioni e corpo opzionale.
5. **La risposta viene inviata al client**.
6. **Il client gestisce la risposta**: renderizza i dati in caso di successo (2xx) o mostra un messaggio di errore in caso di fallimento (4xx/5xx).

## Invio di moduli HTML

- I moduli raccolgono e inviano dati utente usando l'elemento `<form>` con elementi `<input>`.
- Ogni input necessita di un attributo `name` affinché il server possa identificare i dati.
- **Validazione lato client**: Usa gli attributi `required` e `pattern` per un feedback immediato.
- **Validazione lato server**: Verifica sempre sul server per motivi di sicurezza, indipendentemente dai controlli client.
- L'attributo `action` specifica dove inviare i dati del modulo.
- L'attributo `method` specifica come inviarli (`GET` o `POST`; `PUT`, `PATCH`, `DELETE` richiedono soluzioni alternative come Fetch API o `method-override` in Express).

### Flusso di invio del modulo

1. L'utente compila e invia il modulo.
2. Il browser esegue la validazione lato client; procede se valida.
3. Il browser invia i dati del modulo all'URL in `action` usando il metodo `method` specificato.
4. Il server riceve ed elabora i dati (archivia in database, autentica l'utente, ecc.).
5. Il server può eseguire ulteriori validazioni lato server.
6. Il server risponde con un messaggio di successo, un errore o un reindirizzamento.
7. L'utente riceve un feedback basato sulla risposta del server.

## Modello degli standard web e organismi di standardizzazione

- **Standard web**: Regole e specifiche che i browser seguono per renderizzare le pagine e abilitare funzionalità web.
- **WHATWG**: L'organismo che mantiene HTML e il DOM come uno standard "vivo" continuamente aggiornato, guidato dai principali team di motori browser.
- **W3C**: L'organizzazione principale per gli standard, che supervisiona CSS, SVG, linee guida per l'accessibilità (WCAG) e numerose API web.
- **ECMA / TC39**: ECMA pubblica la specifica ECMAScript (lo standard JavaScript); TC39 sviluppa nuove funzionalità del linguaggio attraverso un processo di proposta in cinque fasi (Stage 0–4).
- **Khronos Group**: Responsabile di WebGL e WebGL 2, che portano il rendering 3D accelerato GPU nel browser.

## Il processo di creazione degli standard web

1. **Individuazione di un bisogno**: Uno sviluppatore o ingegnere browser documenta un problema reale e ampiamente condiviso.
2. **Proposta e discussione iniziale**: L'idea viene scritta informalmente per raccogliere feedback e valutare l'interesse dei vendor browser.
3. **Redazione della specifica**: Si crea un documento preciso e formale che copre tutti i casi limite, condizioni di errore e implicazioni di sicurezza.
4. **Revisione, obiezioni e iterazioni**: La bozza viene revisionata pubblicamente; i vendor sollevano obiezioni e la specifica viene modificata fino al consenso.
5. **Implementazione e test**: I vendor implementano la funzionalità dietro un flag; una suite di test condivisa verifica che tutte le implementazioni si comportino identicamente.
6. **Finalizzazione**: Quando più implementazioni indipendenti superano i test, lo standard viene finalizzato (Raccomandazione W3C, Stage 4 TC39 o aggiornamento dello standard vivo WHATWG).

## Ciclo di vita delle funzionalità degli standard web

1. **Proposta**: Qualcuno individua un problema e lo scrive per capire se altri lo condividono.
2. **Incubazione**: L'idea viene sottoposta a discussione aperta; molte proposte muoiono qui se troppo di nicchia o senza soluzioni soddisfacenti.
3. **Specificazione**: Si scrive una specifica formale che trasforma concetti vaghi in regole esatte e implementabili.
4. **Test di interoperabilità**: La suite W3C Web Platform Tests verifica che ogni browser implementi la funzionalità in modo identico.
5. **Finalizzazione**: Più implementazioni superano i test e la funzionalità viene distribuita agli utenti come parte della piattaforma web di base.
6. **Deprecazione**: Funzionalità superate o problematiche vengono segnate per la dismissione; la rimozione dai browser è lenta per motivi di compatibilità retroattiva.

## Principi chiave degli standard web

- **Apertura**: Gli standard sono sviluppati pubblicamente e implementati senza royalty da chiunque.
- **Accessibilità**: Il web è progettato per essere utilizzabile da tutti, con accessibilità integrata in HTML, CSS e comportamento del browser.
- **Interoperabilità**: Una pagina costruita secondo lo standard dovrebbe comportarsi identicamente su tutti i browser e sistemi operativi.
- **Compatibilità retroattiva**: Le nuove funzionalità si aggiungono sopra quelle esistenti così le vecchie pagine continuano a funzionare nei browser moderni.
- **Privacy e sicurezza**: Le nuove API che potrebbero esporre dati utente sono sottoposte a rigorosi controlli; la piattaforma protegge gli utenti per impostazione predefinita.
- **Stratificazione**: HTML gestisce la struttura, CSS la presentazione e JavaScript il comportamento — ogni livello funziona indipendentemente.
- **Progressive enhancement**: Le funzionalità devono funzionare prima per tutti, poi migliorare per chi ha browser o dispositivi più avanzati.
