# --description--

نموذج طلب-استجابة HTTP هو نمط الاتصال الأساسي الذي يشغل شبكة الويب العالمية (WWW). يحتوي على العديد من المكونات التي تجعله فعالًا، حتى نتمكن جميعًا من الوصول إلى المواد من الويب.

لننظر إلى ما هو نموذج طلب-استجابة HTTP، والمكونات التي يتألف منها، وكيف تعمل هذه المكونات معًا لتحقيق الكفاءة في تشغيل الويب.

نموذج طلب-استجابة HTTP هو نمط اتصال في التفاعلات على الويب حيث يرسل العميل، عادة متصفح ويب، **طلب HTTP** إلى الخادم، ويعالج الخادم الطلب ويرسل **استجابة HTTP** مرة أخرى.

إليك المكونات التي تشكل نموذج طلب-استجابة HTTP:

* **العميل**: متصفح، تطبيق جوال، أو أي برنامج آخر يرسل طلبات HTTP إلى الخادم ويفسر استجابات HTTP التي يتلقاها.
    
* **الطلب**: الرسالة التي يرسلها العميل إلى الخادم. تشمل الطريقة، عنوان URL، الأقسام العلوية، وجسم اختياري.
    
* **طريقة HTTP**: الإجراء الذي يجب تنفيذه — `GET` لاسترجاع البيانات، `POST` لإرسال البيانات، `PATCH` و `PUT` لتعديل البيانات، و `DELETE` لحذف البيانات.
    
* **URL (محدد موقع الموارد الموحد)**: المورد المطلوب من الخادم.
    
* **الأقسام العلوية**: البيانات الوصفية المرسلة مع الطلب أو الاستجابة. مثل `Content-Type` و `Authorization`.
    
* **الجسم**: البيانات المرسلة مع طلبات `POST`، `PUT`، `PATCH`، `DELETE`، غالبًا بصيغة JSON أو بيانات نموذج.
    
* **الخادم**: النظام الخلفي الذي يعالج الطلب ويولد استجابة. يمكن كتابته بأي لغة برمجة تدعم البرمجة على جانب الخادم.
    
* **الاستجابة**: الرسالة التي يرسلها الخادم مرة أخرى. تحتوي على رمز حالة، أقسام علوية، وجسم اختياري.
    
* **رمز الحالة أو رمز الاستجابة**: رمز رقمي يشير إلى النجاح، الأخطاء، أو عمليات إعادة التوجيه.
    
* **جسم الاستجابة**: البيانات الفعلية المرسلة، عادة بصيغة HTML، JSON، أو XML.

فكيف تعمل كل هذه المكونات معًا في دورة طلب-استجابة HTTP؟

1. **العميل يبدأ الطلب**
    
* متصفح، تطبيق جوال، أو عميل API يرسل طلب HTTP موجه للوصول إلى الخادم.
        
* يشمل الطلب طريقة مثل `GET`، `POST`، `PUT`، `PATCH`، `DELETE`، عنوان URL، أقسام علوية، وأحيانًا جسمًا.
        
2. **يُرسل الطلب إلى الخادم**
    
* ينتقل الطلب عبر الإنترنت إلى الخادم، وهو نظام خلفي مكتوب بلغة برمجة.
        
* يستقبل الخادم الطلب ويستخرج المعلومات ذات الصلة منه.
        
3. **يعالج الخادم الطلب**
    
* يتحقق الخادم من طريقة الطلب وعنوان URL.
        
* قد يحتاج إلى التحقق من التوثيق (عبر الأقسام العلوية مثل authorization).
        
* إذا لزم الأمر، يتفاعل مع قاعدة بيانات لجلب أو تعديل البيانات.
        
4. **يُعد الخادم الاستجابة**
    
* يولد الخادم استجابة برمز حالة مثل 200، 404، 500، وهكذا.
        
* تتضمن الاستجابة أقسامًا علوية، مثل `Content-Type: application/json`.
        
* إذا كان ذلك مناسبًا، يُضاف جسم استجابة بصيغة JSON، HTML، أو XML.
        
5. **يُرسل الاستجابة إلى العميل**
    
* يرسل الخادم الاستجابة مرة أخرى عبر الإنترنت.
        
* يستقبل العميل الاستجابة ويعالج البيانات.
        
6. **يتعامل العميل مع الاستجابة**
    
* إذا كانت الاستجابة ناجحة (رمز الحالة 200)، يعرض العميل البيانات، مثل عرض البيانات على صفحة ويب.
        
* إذا حدث خطأ (`404 Not Found`، `500 Internal Server Error`، وهكذا)، قد يعرض العميل رسالة خطأ.

تتكرر هذه الدورة مع كل طلب جديد يرسله العميل.
