# --description--

O npm precisa de uma forma de armazenar detalhes importantes sobre seu projeto – coisas como quais pacotes ele usa, a versão desses pacotes e quais comandos ele pode executar. Ele faz isso com o arquivo `package.json`.

Pense no arquivo `package.json` como um arquivo de configuração npm para o projeto. É um arquivo JSON simples que descreve seu projeto usando pares chave-valor para acompanhar coisas como:

* o nome e a versão do projeto
    
* o autor
    
* o arquivo principal que deve ser executado primeiro
    
* a licença
    
* o repositório de código, palavras-chave e descrições
    
* quaisquer dependências que o projeto precisa
    

Aqui está como um arquivo `package.json` pode ser:

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "description": "A simple NPM project",
  "main": "index.js",
  "scripts": {
    "start": "node index.js",
    "test": "echo \"Running tests...\""
  },
  "repository": {
    "type": "git",
    "url": "git+https://github.com/username/project.git"
  },
  "keywords": ["fCC", "js"],
  "author": "Kolade Chris",
  "license": "ISC",
  "dependencies": {
    "react": "^19.2.1"
  },
  "devDependencies": {
    "cypress": "^15.7.1"
  }
}
```

O campo `main` geralmente é definido como `index.js` e informa ao Node.js qual arquivo executar primeiro.

O campo `license` é usado para especificar os termos legais sob os quais outras pessoas podem usar, modificar e distribuir o código no seu pacote. Existem muitas licenças diferentes como `MIT`, `ISC` ou `GPL`, que possuem termos e permissões diferentes. O padrão é `ISC`, a Licença do Internet Systems Consortium, que é uma licença de código aberto muito permissiva.

À medida que seu projeto cresce e você instala pacotes, o npm os adiciona em uma de duas seções:

* `dependencies`, que são pacotes que seu app precisa para rodar em produção
    
* `devDependencies`, que são pacotes que você precisa apenas durante o desenvolvimento
    

Por exemplo, se seu projeto usa React para sua UI, isso vai em `dependencies` porque é necessário para seu projeto funcionar depois que ele for implantado. Uma ferramenta como Cypress, que é usada apenas para testes end-to-end antes de implantar em produção, pertence a `devDependencies`.

Existem algumas maneiras fáceis de criar um arquivo `package.json`. A forma mais comum é executar o seguinte comando:

```bash
npm init
```

Você deve executar isso na pasta raiz do seu projeto. Este comando inicia um processo de configuração interativo onde o npm faz uma série de perguntas. Suas respostas são usadas para preencher campos como o nome do projeto, versão, descrição, scripts, palavras-chave, autor e licença.

Se preferir pular as perguntas e criar um arquivo `package.json` com valores padrão, você pode executar:

```bash
npm init -y
```

ou

```bash
npm init --yes
```

Isso cria instantaneamente um arquivo `package.json` para você com valores padrão em muitos campos.

Geralmente é melhor usar `npm init` sem a flag `-y` ou `--yes` para que você possa preencher cada campo e personalizar as informações do seu projeto. Se você não responder a uma pergunta, o npm simplesmente usa um valor padrão. Por exemplo, se você deixar a pergunta da licença em branco, o npm definirá como `ISC` por padrão.

Aqui está um exemplo do que você verá ao executar `npm init`:


Você pode notar que `dependencies` e `devDependencies` ainda não estão listados. Isso porque essas seções só são adicionadas depois que você instala seu primeiro pacote.

Você também notará que o campo `repository` está ausente. Embora o npm tenha perguntado sobre ele durante o `npm init`, ele não o preenche automaticamente depois. Por exemplo, quando você executa comandos como `git remote add origin` ou `git push`, o npm não atualizará o campo repository para você — você precisa adicioná-lo manualmente se quiser que ele seja incluído.
