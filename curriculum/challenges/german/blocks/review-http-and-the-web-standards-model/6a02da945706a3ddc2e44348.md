## Server und das Client-Server-Modell

- **Server**: Ein Computer oder Programm, das Anfragen von anderen Computern empfängt und Daten oder Dienste über das Netzwerk zurücksendet. Der Begriff bezieht sich sowohl auf die physische Hardware als auch auf die darauf laufende Software.
- **Client-Server-Modell**: Eine verteilte Architektur, bei der Clients Anfragen senden und Server diese Anfragen bearbeiten und Antworten zurückgeben.
- **Client**: Der Prozess oder die Anwendung (zum Beispiel ein Webbrowser), die Anfragen an einen Server initiiert.
- **Anfrage-Antwort-Zyklus**: Der Client sendet eine Anfrage; der Server verarbeitet sie und gibt eine Antwort zurück.

### Servertypen

- **Webserver**: Liefern Webinhalte (HTML, Bilder, CSS) an Browser.
- **Application-Server**: Verarbeiten Geschäftslogik und Benutzereingaben.
- **Datenbankserver**: Speichern, rufen ab und verwalten Daten.
- **Mailserver**: Senden, empfangen und speichern E-Mails.
- **Dateiserver**: Verwalten Dateispeicherung und -freigabe im Netzwerk.
- **Proxy-Server**: Agieren als Vermittler zwischen Clients und anderen Servern.

## DNS (Domain Name System)

- **DNS**: Ein hierarchisches und dezentrales Namenssystem, das menschenlesbare Domainnamen in IP-Adressen übersetzt, oft als „Telefonbuch“ des Internets bezeichnet.
- **IP-Adresse**: Eine eindeutige numerische Kennung für ein Gerät in einem Netzwerk (zum Beispiel `127.0.0.1` in IPv4, `0:0:0:0:0:0:0:1` in IPv6).
- **Domainnamen-Hierarchie** (von rechts nach links lesen):
  - **TLD (Top-Level-Domain)**: Der ganz rechte Abschnitt (zum Beispiel `.org`, `.com`).
  - **SLD (Second-Level-Domain)**: Der registrierte Name (zum Beispiel `freecodecamp` in `freecodecamp.org`).
  - **Subdomain**: Identifiziert einen Abschnitt der Hauptdomain (zum Beispiel `www`, `blog`, `api`).

### DNS-Auflösungsprozess

1. Der Browser sendet den Domainnamen an den **rekursiven Resolver** (meist vom ISP verwaltet).
2. Der rekursive Resolver fragt **Root-Server** ab, die ihn an den zuständigen TLD-Server weiterleiten.
3. Der **TLD-Server** gibt die Adresse des **autoritativen Nameservers** für die Domain zurück.
4. Der autoritative Nameserver liefert den **DNS-Eintrag** mit der IP-Adresse zurück.
5. Der rekursive Resolver übergibt die IP-Adresse an den Client und **cached** den Eintrag, um zukünftige Abfragen zu beschleunigen.

## TCP/IP

- **Protokoll**: Ein Satz von Regeln und Standards, die es Geräten ermöglichen, über ein Netzwerk zu kommunizieren.
- **TCP/IP**: Eine Protokollfamilie, die bestimmt, wie Daten verpackt, gesendet, geroutet und empfangen werden.
  - **IP (Internet Protocol)**: Leitet Datenpakete von der Quelle zum Ziel anhand von IP-Adressen. Es ist verbindungslos – jedes Paket reist unabhängig ohne Zustellgarantie.
  - **TCP (Transmission Control Protocol)**: Zerlegt Daten in nummerierte Segmente, setzt sie am Ziel wieder zusammen, prüft auf Fehler und verlorene Pakete und fordert bei Bedarf eine erneute Übertragung an. Gewährleistet eine zuverlässige, geordnete Zustellung.

## HTTP (HyperText Transfer Protocol)

- **HTTP**: Ein Anwendungsprotokoll, das die Regeln für die Kommunikation zwischen Clients und Servern im Web definiert.

- **HTTPS**: Eine sichere Version von HTTP, bei der Daten vor der Übertragung mit SSL/TLS verschlüsselt werden.

### Aufbau einer HTTP-Anfrage

- **Anfragezeile**: `METHOD /path HTTP/version` (zum Beispiel `GET / HTTP/1.1`)
- **Header**: Schlüssel-Wert-Metadatenpaare (zum Beispiel `Content-Type: application/json`).
- **Body** (optional): Daten, die an den Server gesendet werden, verwendet bei `POST`, `PUT` und `PATCH`.

**Gängige HTTP-Methoden:**

| Methode | Zweck |
| --- | --- |
| `GET` | Daten abrufen |
| `POST` | Neue Daten absenden |
| `PUT` | Eine Ressource ersetzen |
| `PATCH` | Eine Ressource teilweise aktualisieren |
| `DELETE` | Eine Ressource entfernen |

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

### Aufbau einer HTTP-Antwort

- **Statuszeile**: HTTP-Version, Statuscode und Begründung (zum Beispiel `HTTP/1.1 200 OK`).
- **Header**: Metadaten zur Antwort (zum Beispiel `Content-Type: text/html`).
- **Body** (optional): Der zurückgegebene Inhalt oder eine Fehlererklärung.

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

## HTTP-Statuscodes

Statuscodes werden nach ihrer ersten Ziffer gruppiert:

- **1xx** — Informationsantworten.
- **2xx** — Erfolgreiche Antworten.
- **3xx** — Umleitungsnachrichten.
- **4xx** — Client-Fehlerantworten.
- **5xx** — Server-Fehlerantworten.

### Gängige Codes

| Code | Name | Bedeutung |
| --- | --- | --- |
| `200` | OK | Anfrage erfolgreich; Inhalt zurückgegeben. |
| `201` | Created | Ressource erfolgreich erstellt. |
| `204` | No Content | Erfolg, aber kein Inhalt zum Zurücksenden. |
| `301` | Moved Permanently | Ressource dauerhaft an eine neue URL verschoben. |
| `302` | Found | Ressource vorübergehend an eine andere URL verschoben. |
| `304` | Not Modified | Ressource seit letzter Anfrage unverändert; zwischengespeicherte Version verwenden. |
| `400` | Bad Request | Server konnte die Anfrage nicht verstehen (ungültige Syntax oder Daten). |
| `401` | Unauthorized | Authentifizierungsdaten erforderlich. |
| `403` | Forbidden | Client hat keine Berechtigung für die Ressource, auch mit gültigen Anmeldedaten. |
| `404` | Not Found | Ressource existiert nicht auf dem Server. |
| `500` | Internal Server Error | Unerwarteter Fehler auf dem Server. |
| `502` | Bad Gateway | Server erhielt eine ungültige Antwort von einem vorgelagerten Server. |
| `503` | Service Unavailable | Server vorübergehend nicht in der Lage, Anfragen zu bearbeiten. |

## Das HTTP-Anfrage-Antwort-Modell

Der Anfrage-Antwort-Zyklus besteht aus diesen zusammenwirkenden Komponenten:

- **Client**: Browser, mobile App oder API-Client, der HTTP-Anfragen sendet.
- **HTTP-Methode**: Die auszuführende Aktion (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`).
- **URL**: Die Adresse der angeforderten Ressource.
- **Header**: Metadaten für die Anfrage oder Antwort (zum Beispiel `Authorization`, `Content-Type`).
- **Body**: Daten, die mit `POST`, `PUT`, `PATCH` oder `DELETE` Anfragen gesendet werden (oft JSON oder Formulardaten).
- **Server**: Das Backend-System, das Anfragen verarbeitet und Antworten zurückgibt.
- **Statuscode**: Numerischer Code, der Erfolg, Fehler oder Umleitung anzeigt.
- **Antwort-Body**: Die tatsächlich zurückgegebenen Daten, typischerweise HTML, JSON oder XML.

### Die sechs Schritte des Zyklus

1. **Client initiiert eine Anfrage** mit Methode, URL, Headern und optionalem Body.
2. **Anfrage reist über das Internet zum Server**.
3. **Server verarbeitet die Anfrage**: prüft Methode, URL, Authentifizierung und fragt bei Bedarf eine Datenbank ab.
4. **Server bereitet eine Antwort vor** mit Statuscode, Headern und optionalem Body.
5. **Antwort wird an den Client gesendet**.
6. **Client verarbeitet die Antwort**: rendert Daten bei Erfolg (2xx) oder zeigt eine Fehlermeldung bei Fehler (4xx/5xx) an.

## HTML-Formulareinsendungen

- Formulare sammeln und senden Benutzerdaten mithilfe des `<form>`-Elements mit `<input>`-Elementen.
- Jede Eingabe benötigt ein `name`-Attribut, damit der Server die Daten identifizieren kann.
- **Clientseitige Validierung**: Verwenden Sie die `required`- und `pattern`-Attribute für sofortiges Feedback.
- **Serverseitige Validierung**: Validieren Sie immer auf dem Server aus Sicherheitsgründen, unabhängig von clientseitigen Prüfungen.
- Das `action`-Attribut gibt an, wohin die Formulardaten gesendet werden.
- Das `method`-Attribut gibt an, wie die Daten gesendet werden (`GET` oder `POST`; `PUT`, `PATCH`, `DELETE` erfordern Workarounds wie die Fetch API oder `method-override` in Express).

### Ablauf der Formulareinsendung

1. Benutzer füllt das Formular aus und sendet es ab.
2. Browser führt clientseitige Validierung durch; bei Erfolg wird fortgefahren.
3. Browser sendet Formulardaten an die URL in `action` mit der angegebenen `method`.
4. Server empfängt und verarbeitet die Daten (speichert in der Datenbank, authentifiziert Benutzer usw.).
5. Server kann zusätzliche serverseitige Validierung durchführen.
6. Server antwortet mit einer Erfolgsmeldung, einem Fehler oder einer Umleitung.
7. Benutzer sieht eine Rückmeldung basierend auf der Serverantwort.

## Web-Standards-Modell und Standardisierungsgremien

- **Webstandards**: Die Regeln und Spezifikationen, denen Browser folgen, um Seiten darzustellen und Webfunktionen zu ermöglichen.
- **WHATWG**: Das Standardisierungsgremium, das HTML und das DOM als kontinuierlich aktualisierten „lebenden Standard“ pflegt, angetrieben von den großen Browser-Engine-Teams.
- **W3C**: Die primäre Standardisierungsorganisation, die CSS, SVG, Barrierefreiheitsrichtlinien (WCAG) und zahlreiche Web-APIs überwacht.
- **ECMA / TC39**: ECMA veröffentlicht die ECMAScript-Spezifikation (den JavaScript-Standard); TC39 entwickelt Sprachfeatures durch einen fünfstufigen Vorschlagsprozess (Stufe 0–4).
- **Khronos Group**: Verantwortlich für WebGL und WebGL 2, die GPU-beschleunigtes 3D-Rendering im Browser ermöglichen.

## Der Prozess zur Erstellung von Webstandards

1. **Bedarf erkennen**: Ein Entwickler oder Browseringenieur dokumentiert ein reales, weit verbreitetes Problem.
2. **Vorschlag und erste Diskussion**: Die Idee wird informell niedergeschrieben, um Feedback zu sammeln und das Interesse der Browseranbieter zu prüfen.
3. **Spezifikation schreiben**: Ein präzises, formales Dokument wird erstellt, das alle Randfälle, Fehlerbedingungen und Sicherheitsaspekte abdeckt.
4. **Überprüfung, Einwände und Iteration**: Der Entwurf wird öffentlich überprüft; Anbieter bringen Einwände vor, und die Spezifikation wird überarbeitet, bis ein Konsens erreicht ist.
5. **Implementierung und Testen**: Browseranbieter implementieren die Funktion hinter einer Flagge; eine gemeinsame Testsuite überprüft, dass alle Implementierungen identisch funktionieren.
6. **Finalisierung**: Sobald mehrere unabhängige Implementierungen die Tests bestehen, wird der Standard finalisiert (W3C-Empfehlung, TC39 Stufe 4 oder ein Update des WHATWG-Lebendstandards).

## Der Lebenszyklus von Webstandards-Funktionen

1. **Vorschlag**: Jemand erkennt ein Problem und schreibt es auf, um herauszufinden, ob andere es teilen.
2. **Inkubation**: Die Idee wird durch offene Diskussionen auf die Probe gestellt; viele Vorschläge sterben hier, wenn sie zu speziell sind oder keine Lösung alle zufriedenstellt.
3. **Spezifikation**: Eine formale Spezifikation wird geschrieben, die vage Konzepte in genaue, implementierbare Regeln verwandelt.
4. **Interoperabilitätstests**: Die W3C Web Platform Tests Suite überprüft, dass jeder Browser die Funktion identisch implementiert.
5. **Finalisierung**: Mehrere erfolgreiche Implementierungen bedeuten, dass die Funktion als Teil der Basis-Webplattform an Nutzer ausgeliefert wird.
6. **Veraltung**: Veraltete oder problematische Funktionen werden zur Einstellung markiert; die Entfernung aus Browsern erfolgt langsam wegen der Verpflichtung zur Rückwärtskompatibilität.

## Zentrale Grundsätze der Webstandards

- **Offenheit**: Standards werden offen entwickelt und von jedem lizenzfrei implementiert.
- **Barrierefreiheit**: Das Web ist so gestaltet, dass es von allen nutzbar ist, mit Barrierefreiheit in HTML, CSS und Browserverhalten integriert.
- **Interoperabilität**: Eine nach Standard erstellte Seite sollte in allen Browsern und Betriebssystemen identisch funktionieren.
- **Rückwärtskompatibilität**: Neue Funktionen bauen auf bestehenden auf, sodass alte Seiten in modernen Browsern weiterhin funktionieren.
- **Datenschutz und Sicherheit**: Neue APIs, die Benutzerdaten verfügbar machen könnten, werden streng geprüft; die Plattform schützt Nutzer standardmäßig.
- **Schichtung**: HTML übernimmt die Struktur, CSS die Darstellung und JavaScript das Verhalten – jede Schicht arbeitet unabhängig.
- **Progressive Verbesserung**: Funktionen müssen zuerst für alle funktionieren und sich dann für Nutzer mit leistungsfähigeren Browsern oder Geräten verbessern.
