## Einführung in die Versionskontrolle

- **Definition**: Ein Versionskontrollsystem ermöglicht es Ihnen, Änderungen in Ihrem Projekt zu verfolgen und zu verwalten. Beispiele für Versionskontrollsysteme, die in der Softwareentwicklung verwendet werden, sind Git, SVN oder Mercurial.

## Cloud-basierte Anbieter von Versionskontrolle

- **Liste der cloud-basierten Anbieter von Versionskontrolle**: GitHub und GitLab sind beliebte Beispiele für cloud-basierte Anbieter von Versionskontrolle, die Softwareteams die Zusammenarbeit und Verwaltung von Repositories ermöglichen.

## Installation und Einrichtung von Git

- **Git installieren**: Um zu überprüfen, ob Git bereits auf Ihrem Rechner installiert ist, können Sie den folgenden Befehl im Terminal ausführen:

```sh
git --version
```

Wenn Sie eine Versionsnummer sehen, ist Git installiert. Falls nicht, müssen Sie es installieren.

Bei Linux-Systemen ist Git oft bereits bei den meisten Distributionen vorinstalliert. Falls Git nicht vorinstalliert ist, sollten Sie es mit Ihrem Paketmanager über Befehle wie `sudo apt-get install git` oder `sudo pacman -S git` installieren können.

Für Mac-Nutzer können Sie Git über Homebrew mit `brew install git` installieren oder den ausführbaren Installer von der Git-Website herunterladen.

Für Windows können Sie den ausführbaren Installer von der Git-Website herunterladen. Oder, wenn Sie Chocolatey eingerichtet haben, können Sie `choco install git.install` in PowerShell ausführen. Beachten Sie, dass Sie unter Windows eventuell auch Git Bash herunterladen möchten, um eine Unix-ähnliche Shell-Umgebung zur Verfügung zu haben.

Um sicherzustellen, dass die Installation funktioniert hat, führen Sie den `git --version`-Befehl erneut im Terminal aus.

- **Git-Konfigurationen**: `git config` wird verwendet, um Konfigurationsvariablen festzulegen, die bestimmen, wie Git auf Ihrem Rechner arbeitet. Um Ihre aktuellen Einstellungvariablen und deren Speicherort auf Ihrem System anzuzeigen, können Sie den folgenden Befehl ausführen:

```sh
git config --list --show-origin
```

Im Moment sollten Sie nur systemweite Konfigurationseinstellungen sehen, wenn Sie Git gerade zum ersten Mal installiert haben.

Um Ihren Benutzernamen festzulegen, können Sie den folgenden Befehl ausführen:

```sh
git config --global user.name "Jane Doe"
```

Das `--global`-Flag wird hier verwendet, um den Benutzernamen für alle Projekte auf Ihrem System festzulegen, die Git verwenden. Wenn Sie den Benutzernamen für ein bestimmtes Projekt überschreiben möchten, können Sie den Befehl in diesem Projektverzeichnis ohne das `--global`-Flag ausführen.

Um die Benutzer-E-Mail-Adresse festzulegen, können Sie den folgenden Befehl ausführen:

```sh
git config --global user.email janedoe@example.com
```

Eine weitere Konfiguration, die Sie festlegen können, ist der bevorzugte Editor, den Git verwenden soll. Hier ein Beispiel, wie Sie Ihren bevorzugten Editor auf Emacs setzen:

```sh
git config --global core.editor emacs
```

Wenn Sie keinen bevorzugten Editor festlegen, verwendet Git standardmäßig den Editor Ihres Systems.

## Open-Source- vs. Closed-Source-Software

- **Definition**: „Open-Source“ bedeutet, dass Menschen den von Ihnen veröffentlichten Code sehen, Änderungen vorschlagen, Probleme melden und sogar eine modifizierte Version ausführen können. „Closed-Source“ bedeutet, dass nur die Personen, denen Sie ausdrücklich Zugriff gewähren, das Projekt sehen und damit interagieren können.

## GitHub

- **Definition**: GitHub ist eine cloud-basierte Lösung, die das Speichern von versionierten Projekten in sogenannten „Repositories“ ermöglicht und Funktionen zur Zusammenarbeit mit diesen Projekten bietet.
- **GitHub CLI**: Dieses Tool wird verwendet, um GitHub-spezifische Aufgaben direkt in der Befehlszeile auszuführen. Wenn Sie es nicht installiert haben, finden Sie Anweisungen dazu in der GitHub-Dokumentation – es sollte aber in Ihrem Paketmanager verfügbar sein.
- **GitHub Pages**: GitHub Pages ist eine Option zum Bereitstellen statischer Websites oder Anwendungen, die keinen Back-End-Server für die Logik benötigen. Das heißt, Anwendungen, die vollständig clientseitig oder im Browser des Nutzers laufen, können vollständig auf dieser Plattform bereitgestellt werden.
- **GitHub Actions**: GitHub Actions ist eine Funktion, mit der Sie Workflows direkt in Ihrem GitHub-Repository automatisieren können, einschließlich Build, Test und Deployment Ihres Codes.

## Häufige Git-Befehle

- **`git init`**: Dies initialisiert ein leeres Git-Repository, damit Git Änderungen für dieses Projekt verfolgen kann. Wenn Sie ein leeres Git-Repository für ein Projekt initialisieren, wird ein neues verstecktes `.git`-Verzeichnis hinzugefügt. Dieses `.git`-Verzeichnis enthält wichtige Informationen, damit Git Ihr Projekt verwalten kann.
- **`git status`**: Dieser Befehl zeigt den aktuellen Zustand Ihres Arbeitsverzeichnisses an – Sie werden diesen Befehl häufig in Ihrem Workflow verwenden.
- **`git add`**: Dieser Befehl wird verwendet, um Ihre Änderungen zu stagen. Alles im Staging-Bereich wird für den nächsten Commit hinzugefügt. Wenn Sie alle unstaged Änderungen stagen möchten, können Sie `git add .` verwenden. Der Punkt (`.`) ist ein Alias für das aktuelle Verzeichnis, in dem Sie sich befinden.
- **`git commit`**: Dieser Befehl wird verwendet, um Ihre Änderungen zu committen. Ein Commit ist ein Schnappschuss des Projektzustands zu einem bestimmten Zeitpunkt. Wenn Sie `git commit` ausführen, öffnet sich Ihr bevorzugter Editor, den Sie in der Git-Konfiguration festgelegt haben. Sobald der Editor geöffnet ist, können Sie eine ausführliche Nachricht zu Ihren Änderungen eingeben. Sie können auch eine kürzere Nachricht mit dem `git commit -m`-Befehl angeben, zum Beispiel so:

```sh
git commit -m "short message goes here"
```

- **`git log`**: Dies listet alle vorherigen Commits mit hilfreichen Informationen wie Autor, Commit-Datum, Commit-Nachricht und Commit-Hash auf. Der Commit-Hash ist eine lange Zeichenfolge, die als eindeutiger Bezeichner für einen Commit dient.
- **`git remote add`**: Dieser Befehl wird verwendet, um die Remote-Verbindung zu Ihrem Remote-Repository einzurichten.
- **`git push`**: Dieser Befehl wird verwendet, um Ihre Änderungen in ein Remote-Repository hochzuladen.
- **`git pull`**: Dieser Befehl wird verwendet, um die neuesten Änderungen aus Ihrem Remote-Repository in Ihr lokales Repository herunterzuladen.
- **`git clone`**: Dieser Befehl klont ein Repository. Das bedeutet, Sie erhalten eine Kopie des Repositories. Diese Kopie enthält die Repository-Historie, alle Dateien/Ordner und Commits auf Ihrem lokalen Gerät.
- **`git remote -v`**: Dieser Befehl zeigt die Liste der Remote-Repositories an, die mit Ihrem lokalen Git-Repository verbunden sind.
- **`git branch`**: Dieser Befehl listet alle Ihre lokalen Branches auf.
- **`git fetch upstream`**: Dieser Befehl sagt Git, die neuesten Änderungen vom Upstream-Remote (dem Original-Repository) zu holen.
- **`git merge upstream/main`**: Dieser Befehl sagt Git, die neuesten Änderungen vom `main`-Branch im Upstream-Remote in Ihren aktuellen Branch zu mergen.
- **`git reset`**: Dieser Befehl erlaubt es Ihnen, den aktuellen Zustand eines Branches zurückzusetzen. Das Übergeben des `--hard`-Flags sagt Git, die lokalen Dateien zwangsweise an den Branch-Zustand anzupassen. So stellen Sie sicher, dass Sie eine saubere Ausgangsbasis haben.
- **`git rebase`**: Ein Rebase in Git ist eine Möglichkeit, eine Folge von Commits von einem Branch auf einen anderen zu verschieben oder zusammenzuführen.

## Arbeiten mit Branches

- **Definition**: Ein Branch in Git ist ein separater Arbeitsbereich, in dem Sie Änderungen vornehmen können. Der `main`-Branch stellt oft den Haupt- oder Produktionsbranch in einer realen Anwendung dar. Entwicklerteams erstellen mehrere Branches für neue Funktionen und Bugfixes und führen diese Änderungen dann wieder in den `main`-Branch zusammen.
- **Erstellen eines neuen Branches**: Um einen neuen Branch zu erstellen, können Sie den folgenden Befehl ausführen:

```sh
git branch feature
```

Um zu diesem Branch zu wechseln, können Sie den folgenden Befehl ausführen:

```sh
git checkout feature
```

Die meisten Entwickler verwenden den Kurzbefehlt zum Erstellen und Wechseln eines Branches, der folgendermaßen lautet:

```sh
git checkout -b new-branch-name
```

Ein neuerer und alternativer Befehl ist der `git switch`-Befehl. Hier ein Beispiel, um einen neuen Branch zu erstellen und zu wechseln:

```sh
git switch -c new-branch-name
```

- **Branching-Strategien**: Ihr `main`-Branch ist Ihr Standardbranch und normalerweise ziemlich stabil. Es ist daher am besten, von dort aus neue Branches für Bugfixes, neue Funktionen oder andere Arbeiten zu erstellen.
- **Merge-Konflikte**: Diese treten auf, wenn Git versucht, Änderungen aus verschiedenen Branches automatisch zusammenzuführen, aber nicht entscheiden kann, welche Änderungen beibehalten werden sollen. Das passiert meist, wenn es widersprüchliche Änderungen am gleichen Teil einer Datei gibt.

## Fünf Zustände einer von Git verfolgten Datei

- **„Untracked“**: Das bedeutet, dass die Datei neu im Repository ist und Git sie noch nicht „gesehen“ hat.
- **„Modified“**: Diese Datei existierte im vorherigen Commit und wurde geändert, aber die Änderungen wurden noch nicht committet.
- **„Ignored“**: Sie werden ignorierte Dateien wahrscheinlich nicht in Git sehen, aber Ihre IDE könnte sie anzeigen. Ignorierte Dateien sind von Git-Operationen ausgeschlossen, meist weil sie in der `.gitignore`-Datei aufgeführt sind.
- **„Deleted“**: Eine gelöschte Datei ist das Gegenteil einer untracked Datei – sie existierte vorher und wurde entfernt.
- **„Renamed“**: Eine umbenannte Datei ist eine Datei, deren Inhalt unverändert ist, aber deren Name oder Speicherort geändert wurde. In manchen Fällen wird eine Datei auch als umbenannt betrachtet, wenn sie nur kleine Änderungen enthält.

## `.gitignore`-Dateien

- **Definition**: Die `.gitignore`-Datei ist eine spezielle Art von Datei, die mit Git-Operationen zusammenhängt. Der Name deutet darauf hin, dass diese Datei Git sagt, Dinge zu ignorieren, und das ist der häufigste Anwendungsfall. Tatsächlich teilt sie Git aber mit, eine Datei nicht mehr zu verfolgen.

## Arbeiten mit Repositories

- **Definition**: Ein Repository ist wie ein Container für ein Projekt – wenn Sie an einer Anwendung arbeiten, würden Sie die Dateien dieser Anwendung zusammen in einem Repository aufbewahren. Repositories können lokal auf Ihrem Computer oder remote auf einem Dienst wie GitHub liegen.
- **Öffentliche vs. private Repositories**: Ein öffentliches Repository kann von jedem angesehen und heruntergeladen werden. Ein privates Repository kann nur von Ihnen und den Personen, denen Sie ausdrücklich Zugriff gewähren, aufgerufen werden.
- **Erstellen von Repositories auf GitHub**: Um ein neues Repository auf GitHub zu erstellen, können Sie auf die `"New Repository"`-Schaltfläche klicken und den GitHub-Benutzeroberflächenprozess zum Einrichten eines neuen Repositories durchlaufen.
- **Lokale Repositories zu GitHub pushen**: Wenn Sie ein lokales Projekt auf Ihrem Computer haben, können Sie dieses Repository zu GitHub pushen. Hier eine Schritt-für-Schritt-Übersicht des Prozesses:

1. Initialisieren Sie ein leeres Git-Repository im Projektverzeichnis (`git init`).
2. Nehmen Sie Änderungen an Ihrem Projekt vor.
3. Führen Sie den `git status`-Befehl aus, um alle von Git verfolgten Änderungen zu sehen.
4. Stagen Sie Ihre Änderungen (`git add`).
5. Committen Sie Ihre Änderungen (`git commit`).
6. Richten Sie die Remote-Verbindung ein (`git remote add`).
7. Pushen Sie Ihre Änderungen zu GitHub (`git push`).

## Pull Requests

- **Pull Requests**: Ein Pull Request ist eine Anfrage, Änderungen aus Ihrem Branch in den Zielbranch zu übernehmen. Pull Requests sind der Ablauf, den Sie verwenden, wenn Sie Codeänderungen zu einem Projekt beitragen möchten. Dieser Ansatz ermöglicht es den Projektverantwortlichen, Ihre Änderungen zu überprüfen. Sie können Kommentare hinterlassen, Fragen stellen und Anpassungen vorschlagen. Sobald der Überprüfungsprozess abgeschlossen ist, kann der Pull Request genehmigt und in den Hauptbranch zusammengeführt werden.

## Beiträge zu anderen Repositories

- **Prozess**: Es gibt Tausende von Projekten, zu denen Sie beitragen können. Hier ist der grundlegende Prozess, wie Sie zu einem anderen Repository beitragen:

1. Lesen Sie die Beitragsdokumentation.
2. Finden Sie ein verfügbares Issue, an dem Sie arbeiten möchten.
3. Forken Sie das Repository.
4. Klonen Sie Ihre geforkte Kopie des Repositories.
5. Erstellen Sie einen neuen Branch.
6. Nehmen Sie die Änderungen entsprechend dem Issue vor.
7. Erstellen Sie einen PR (Pull Request).
8. Warten Sie auf eine Überprüfung dieses PR.

## Arbeiten mit SSH- und GPG-Schlüsseln

- **GPG-Schlüssel**: GPG oder Gnu Privacy Guard-Schlüssel werden typischerweise verwendet, um Dateien oder Commits zu signieren. Jemand kann dann Ihren öffentlichen GPG-Schlüssel verwenden, um zu überprüfen, dass die Dateisignatur von Ihrem Schlüssel stammt und dass der Inhalt der Datei nicht verändert oder manipuliert wurde.

Um einen GPG-Schlüssel zu erzeugen, müssen Sie Folgendes ausführen:

```sh
gpg --full-generate-key
```

- **SSH-Schlüssel**: SSH oder Secure SHell-Schlüssel werden typischerweise verwendet, um eine Remote-Verbindung zu einem Server zu authentifizieren – über das `ssh`-Tool. Sie können einen SSH-Schlüssel auch verwenden, um Commits zu signieren.

Für einen SSH-Schlüssel führen Sie Folgendes aus:

```sh
ssh-keygen -t ed25519 -C "your_email@example.com"
```

`ed25519` ist ein moderner Algorithmus für öffentliche Schlüssel-Signaturen.

- **Commits mit GPG-Schlüsseln signieren**: Um Ihre Commits mit Ihrem GPG-Schlüssel zu signieren, müssen Sie Ihren öffentlichen Schlüssel (nicht den privaten) in Ihrem GitHub-Konto hochladen. Um Ihre öffentlichen Schlüssel aufzulisten, führen Sie Folgendes aus:

```sh
gpg --list-secret-keys --keyid-format=long
```

Dann, um den öffentlichen Schlüssel zu erhalten, verwenden Sie:

```sh
gpg --armor --export "<key id>"
```

Nehmen Sie dann die kurze ID, die Sie beim Auflisten der Schlüssel erhalten haben, und führen Sie diesen Befehl aus, um sie als Ihren Git-Signierschlüssel festzulegen:

```sh
git config --global user.signingkey <your_gpg_key_id>
```

Anschließend können Sie das `-S`-Flag an Ihren `git commit`-Befehl übergeben, um einen bestimmten Commit zu signieren – Sie müssen Ihre Passphrase eingeben. Alternativ können Sie, wenn Sie jeden Commit automatisch signieren möchten, die Autosign-Konfiguration auf `true` setzen:

```sh
git config --global commit.gpgsign true
```

- **Commits mit SSH-Schlüsseln signieren**: Um mit einem SSH-Schlüssel zu signieren, was eine relativ neue Funktion auf GitHub ist, müssen Sie zunächst den Schlüssel in Ihrem GitHub-Konto hochladen. Dann müssen Sie den Signiermodus für Git auf SSH setzen:

```sh
git config --global gpg.format ssh
```

Anschließend legen Sie den Signierschlüssel fest, indem Sie den Dateipfad anstelle einer ID übergeben:

```sh
git config --global user.signingkey <path_to_your_ssh_keys>
```
