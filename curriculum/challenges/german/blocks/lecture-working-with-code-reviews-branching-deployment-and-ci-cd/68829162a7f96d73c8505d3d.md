# --description--

Wenn Sie schon mit erfahrenen Entwicklerinnen und Entwicklern zu tun hatten, sei es in einer Online-Community oder persönlich, haben Sie wahrscheinlich schon von den „berüchtigten Merge-Konflikten“ gehört. Aber was genau sind sie?

Nun, wenn Sie eine vernünftige Branching-Strategie verwenden, wie wir sie in vorherigen Lektionen besprochen haben, arbeiten Sie möglicherweise gleichzeitig an zwei verschiedenen Funktionen in zwei verschiedenen Branches. Oder vielleicht arbeiten Sie an etwas, und eine Kollegin oder ein Kollege arbeitet an einem anderen Branch. So oder so, Sie haben zwei unabhängige Änderungen. Aber manchmal bearbeiten beide Branches dieselbe Datei. Und wenn der andere Branch zuerst in `main` zusammengeführt wird, liegen die Änderungen Ihres Branches jetzt zurück.

Meistens lässt sich das beheben, indem Sie `main` zurück in Ihren Feature-Branch zusammenführen. Aber wenn die Änderungen aus dem anderen Branch mit Ihrem Branch in Konflikt stehen, kann Git beim Zusammenführen nicht entscheiden, welche Änderungen Sie behalten möchten. Das nennt man einen „Merge-Konflikt“.

Aber wie können Sie ihn beheben? Für kleinere und einfachere Konflikte stellt GitHub eine Web-Benutzeroberfläche zur Verfügung. Ein Button „Konflikte lösen“ erscheint unten in einem Pull Request, wenn es Konflikte gibt. Wenn Sie darauf klicken, sehen Sie die Datei mit Konflikten, die Ihren Code und den konfliktierenden Code enthält. Git fügt „Merge-Konflikt-Markierungen“ um Ihren Code und den konfliktierenden Code hinzu, um das Problem zu kennzeichnen. Das sieht ungefähr so aus:

```sh
1 <<<<<<< feat/conflict
2 Here's some code from my first PR :)
3 =======
4 Here's some code that was merged to main that conflicts with the code on my PR
5 >>>>>>> main
```

Der öffnende Marker ist das `<<<<<<< feat/conflict` und zeigt den Beginn der Änderungen an, die unser aktueller Branch macht. Dann sehen wir das `Here's some code from my first PR :)`. Danach folgt der `=======`-Marker, der das Ende unserer Änderungen und den Beginn der Änderungen des `main`-Branches anzeigt. Wir sehen, dass die Änderungen auf `main` die Zeile `Here's some code that was merged to main that conflicts with the code on my PR` sind. Danach folgt der schließende `>>>>>> main`-Marker, der das Ende der Konflikte zeigt.

Aber wie beheben wir das? Sie müssen entscheiden, welche Änderungen Sie als Teil der Merge-Konflikte behalten möchten. Wenn Sie beide Änderungen behalten wollen, können Sie einfach nur die Konflikt-Markierungen entfernen und die Änderungen unverändert lassen. Alternativ, wenn Sie nur die Änderungen eines Branches behalten möchten, löschen Sie die Konflikt-Markierungen und die Änderungen des anderen Branches (die Sie nicht behalten möchten). Und schließlich, wenn Ihnen keine der Änderungen gefällt, können Sie alles vom öffnenden Marker bis zum schließenden Marker (einschließlich) entfernen.

Lassen Sie uns die Änderungen des `main`-Branches entfernen, aber unsere behalten. So sieht das jetzt aus:

```sh
1 Here's some code from my first PR :)
```

Klicken Sie auf den Button „Als gelöst markieren“, der GitHub mitteilt, dass Sie mit Ihrer Lösung zufrieden sind. Wenn es weitere Dateien gäbe, würde GitHub zur nächsten wechseln. Da dies aber unsere einzige konfliktbehaftete Datei ist, erscheint ein Button „Merge commit erstellen“. Klicken Sie darauf, und GitHub erstellt für Sie einen Merge-Commit. Sie werden zurück zum Pull Request geführt, wo Sie Ihren neuen Merge-Commit und die gelösten Konflikte sehen. Und schon sind wir fertig!

Aber was, wenn Sie die Konflikte nicht direkt auf GitHub bearbeiten können, weil sie zu komplex sind?

Nun, wenn Ihr Branch nur wenige Commits hat oder nur ein paar Commits diese Datei ändern, können Sie versuchen, `main` in Ihren Branch zu mergen:

```sh
git checkout feat/conflict
git fetch origin
git merge origin/main
```

Während Git diesen Merge verarbeitet, erhalten Sie Konflikte, die Sie manuell lösen müssen. Git informiert Sie, welche Dateien behoben werden müssen. Wenn wir die Dateien mit Konflikten in unserem Editor öffnen, sehen wir dieselben Konflikt-Markierungen, mit einigen hilfreichen UI-Ergänzungen dank VSCode. Gehen Sie vor und beheben Sie die Konflikte in der Datei auf dieselbe Weise. Diesmal müssen wir den Commit manuell erstellen. Das ist genauso wie bei jedem anderen Commit:

```sh
git add .
git commit -m "chore: resolve conflicts"
```

Nachdem Sie diese Befehle ausgeführt haben, sehen wir, dass Git unsere Lösung akzeptiert hat.

Manchmal, wenn Sie viele Commits oder Konflikte haben, kann ein Rebase hilfreich sein. Anstatt Änderungen zusammenzuführen, nimmt ein Rebase alle Commits auf Ihrem Branch und wendet sie erneut an, sodass sie nach den neuesten Änderungen auf `main` kommen. In Git-Begriffen „setzt“ das effektiv die Basis Ihres Branches auf den neuesten Commit auf `main` zurück.

Um das auszuprobieren, brauchen wir einen neuen Branch:

```sh
git checkout main
git pull
git checkout -b feat/rebase
```

Lassen Sie uns eine Menge Commits für unseren Branch erzeugen, damit wir das Rebasen üben können:

```sh
echo "change one" > README.md
git add .
git commit -m "change one"
echo "change two" > README.md
git add .
git commit -m "change two"
echo "change three" > README.md
git add .
git commit -m "change three"
```

Jetzt machen wir eine weitere Änderung auf `main`, um einen Konflikt zu erzeugen:

```sh
git checkout main
echo "change four" > README.md
git add .
git commit -m "change four"
```

Und checken unseren Rebase-Branch aus:

```sh
git checkout feat/rebase
```

Als Nächstes führen Sie `git rebase -i main` aus. Es wird unseren Branch auf den aktuellen Stand des `main`-Branches rebasen. Das `-i`-Flag sagt Git, den Rebase im interaktiven Modus durchzuführen. Sie sehen Ihre Commits mit dem Wort `pick` daneben. `pick` sagt Git, den Commit unverändert für den Rebase zu verwenden. Speichern und schließen Sie die Datei, und wir erhalten eine Fehlermeldung, die uns mitteilt, dass es Konflikte gibt.

Unser `change one`-Commit steht im Konflikt mit dem `change four`-Commit, und wir müssen das lösen. Öffnen Sie die Datei in Ihrem Editor, und Sie sehen dieselbe UI wie bei den vorherigen Konflikten. Lassen Sie uns unsere `change one`-Änderungen behalten. Dann müssen Sie `git add .` ausführen, um die Änderungen hinzuzufügen, und `git rebase --continue`, um Git mitzuteilen, dass wir mit dem Rebase fortfahren wollen. Da wir den Konflikt im ersten Commit gelöst haben, kann Git den Rest der Commits automatisch rebasen.

Zum Schluss noch ein bisschen Aufräumen: Lassen Sie uns unsere drei Änderungs-Commits zu einem einzigen Commit zusammenfassen, um eine sauberere Historie zu erhalten. Führen Sie `git rebase -i main` erneut aus, um zum Editor zu gelangen. Lassen Sie den ersten Befehl als `pick` stehen, ändern Sie aber die nächsten beiden in `squash`. Möglicherweise treten dieselben Konflikte erneut auf. Wenn ja, lösen Sie sie wie zuvor. Sobald Sie den Rebase gelöst und fortgesetzt haben (oder sofort, wenn es keine Konflikte gab), erscheint ein neuer Commit-Nachrichten-Editor. Diese Nachricht wird Ihr neuer Commit sein, der die Änderungen der drei zusammengefassten Commits enthält. Nennen wir ihn `change three complete`. Speichern und schließen Sie den Editor, und Git schließt den Rebase ab.

Wenn wir jetzt Ihre Commit-Historie prüfen, sehen wir, dass wir jetzt unseren einzelnen `change three complete`-Commit haben, der korrekt auf dem `change four`-Commit basiert. Das bedeutet, Sie haben erfolgreich einen Rebase mit zusammengefassten Commits durchgeführt! Und jetzt haben Sie die Werkzeuge, um ein Merge-Konflikt-Champion zu werden!

## --text--

Was ist der Hauptunterschied zwischen Mergen und Rebasen bei der Behandlung von Branch-Konflikten?

## --answers--

Mergen erzeugt einen neuen Commit, während Rebasen die Historie umschreibt, um Ihre Commits nach den Commits des anderen Branches zu platzieren.

---

Rebasen ist schneller als Mergen.
