# --description--

語意版本控制，或稱 SemVer，是一種賦予版本號明確意義的系統，讓開發者能快速了解對軟體所做的變更類型。

SemVer 使用此格式：

```bash
MAJOR.MINOR.PATCH
```

例如，流行的函式庫 `lodash` 目前的版本是 `4.17.21`。

以下是每個零件的意義：

* 重大變更 — 會破壞現有程式碼的大幅變動。例如：移除或重新命名現有程式碼可能依賴的函式（程式）。
    
* MINOR — 不會破壞現有程式碼的新特性。範例：新增函式或選用參數。
    
* PATCH — 小幅修正與改進。例如：修正錯誤或錯字，但不改變任何功能運作方式。
    

<img src="https://cdn.freecodecamp.org/curriculum/lecture-transcripts/what-is-semantic-versioning-1.png" alt="In the version number 4.17.21, 4 is the Major Version, it's changed for breaking changes, the 17 is the minor version, it's changed for non-breaking changes, and the 21 is the patch, it's changed for bug fixes and small features">

假設你有一個名為 `my-package` 的軟體包。以下是使用語意版本控制時，其版本號如何隨時間變化的範例：

| Version | What Changed |
| --- | --- |
| 1.0.0 | Initial stable release |
| 1.1.0 | New optional feature |
| 1.1.1 | Small bug fix or feature |
| 2.0.0 | Breaking changes introduced |

但是 npm 中的版本號還有更多內容。當你安裝一個軟體包時，你的 `package.json` 檔案中的版本可能會長這樣：

```json
"lodash": "^4.17.21" 
```

你已經知道 `4` 是主要版本，`17` 是次要版本，`21` 是修補版本。但前面的插入符號（`^`）代表什麼意思？

插入符號允許 npm 自動安裝較新的次要和修補版本（`4.x.x`），但不會安裝較新的主要版本（`5.x.x`），因為主要更新可能包含破壞性變更。

當主要版本為 `0` 時，插入符號會變得更嚴格。例如，`^0.2.3` 會允許像 `0.2.4` 這樣的更新，但不會允許 `0.3.0`，因為以 `0` 開頭的主要版本被視為較不穩定。

插入符號並不是 `npm` 和其他 JavaScript 軟體包管理工具用來管理相依性版本的唯一符號。

有一個波浪符號（`~`），它只允許更新修補版本。因此，`~1.2.3` 會接受 `1.2.4` 和 `1.2.9`，但絕不會接受 `1.3.0`。

你也可以使用大於（`>`）、小於（`<`）及其「或等於」等價條件來接受區間內的版本。例如，`>=1.2.3 <2.0.0` 表示從 `1.2.3` 到但不包含 `2.0.0` 的任何版本。

星號符號（`*`）會匹配任何版本，因此它接受所有內容。這僅適用於測試，不建議用於生產環境。

最後，你可以透過不在版本號前加任何符號來鎖定精確版本。例如，版本 `1.2.3` 僅接受 `1.2.3`。

以下是快速摘要表：

| Symbol | Meaning | Example Range |
| --- | --- | --- |
| `^` | Allow minor + patch updates | `^1.2.3` → `1.x.x` |
| `~` | Allow patch updates only | `~1.2.3` → `1.2.x` |
| `>`, `<`, `>=`, `<=` | Custom range | `>=1.2.3` or `<2.0.0` |
| **\*** | Any version | all versions |
| none | Exact version | `1.2.3` |
