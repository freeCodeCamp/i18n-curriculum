# --description--

Ti sei mai chiesto come mantenere la tua app Express stabile e pronta per la produzione, soprattutto quando è sotto carico pesante o sta per chiudere?

Ecco dove entrano in gioco i controlli di integrità e le chiusure graduali. Questi due pattern ti aiutano a mantenere la tua app disponibile e a garantirne l'uscita pulita senza perdere richieste o corrompere dati.

Vediamoli nel dettaglio.

Prima di tutto, cos’è un "controllo di integrità" in Express?

Un controllo di integrità è una rotta semplice che indica se la tua app è attiva e funzionante.

Per esempio:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK')
})
```

Strumenti come bilanciatori di carico o orchestratori di contenitori — pensa a Kubernetes o AWS Elastic Beanstalk — possono interrogare questa rotta per decidere se la tua app è sana.

Se la rotta restituisce un codice di stato 200, il servizio è considerato sano. Ricorda: un codice di stato HTTP 200 significa che tutto va bene!

Se non risponde, l’orchestratore può riavviare automaticamente il servizio.

E cos’è una "chiusura graduale"?

Una chiusura graduale avviene quando la tua app sta per fermarsi, ad esempio quando un server si riavvia o durante una distribuzione.

Invece di interrompere subito le connessioni, dai tempo all’app di completare quello che sta facendo.

In Express, puoi ascoltare `SIGTERM`, che è un segnale di sistema inviato quando un processo deve terminare.

Ecco un esempio:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...')
    // Close DB connections or other cleanup here
  })
})
```

Questo dice a Express di smettere di accettare nuove connessioni e aspettare che quelle esistenti finiscano prima di uscire.

Dovresti anche gestire `SIGINT`, che è il segnale inviato quando premi Ctrl+C per fermare la tua app durante lo sviluppo locale.

Mentre `SIGTERM` è tipicamente usato dagli orchestratori come Kubernetes in produzione, `SIGINT` copre il flusso di lavoro quotidiano dello sviluppatore. Senza gestirlo, premere Ctrl+C termina immediatamente il processo, saltando qualsiasi pulizia che hai impostato.

Poiché entrambi i segnali dovrebbero attivare lo stesso comportamento di chiusura graduale, gli sviluppatori di solito ascoltano entrambi.

Ora, mettiamo tutto insieme

Immagina che la tua app stia girando in un contenitore. Kubernetes invia un `SIGTERM` per spegnerla per un aggiornamento.

Con una semplice chiusura graduale, la tua app smette di accettare nuove connessioni e completa le richieste in corso. Una volta finito tutto, si disconnette dal database ed esce pulita. Nessun dato perso, nessuno stato corrotto.

Ma c’è un problema — il bilanciatore di carico non sa che la tua app sta chiudendo, quindi potrebbe continuare a inviare traffico. Qui entra in gioco il pattern bonus qui sotto.

Ora, ho un piccolo bonus per te. Vedremo come "segnalare che l’app non è sana prima della chiusura".

Prima di chiudere, puoi aggiornare il controllo di integrità per restituire uno stato 503. In questo modo, il bilanciatore di carico vede la risposta non sana e smette di indirizzare nuovo traffico alla tua app durante la finestra di chiusura, come mostrato nel codice sopra.

Ricorda: un codice di stato `503` significa "Servizio non disponibile". Indica che il server è temporaneamente incapace di gestire le richieste, che è esattamente quello che succede durante una chiusura graduale.

```js
let isShuttingDown = false

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(500).send('Shutting down')
  } else {
    res.status(200).send('OK')
  }
})

process.on('SIGTERM', () => {
  isShuttingDown = true

  server.close(() => {
    console.log('Graceful shutdown complete')
    // Perform cleanup
  })
})
```
