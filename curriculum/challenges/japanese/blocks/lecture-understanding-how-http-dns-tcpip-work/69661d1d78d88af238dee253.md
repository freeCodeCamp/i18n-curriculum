# --description--

今日私たちが知っているデジタル世界は、デバイス同士が必要に応じてデータを送受信しながら通信できるという考えに基づいています。

しかし、このプロセスが裏側でどのように機能しているか考えたことはありますか？

インターネットを閲覧するとき、ネットワークを通じて絶えずデータをリクエストし送信しています。これは、情報が適切なデバイスに正しく送受信されることを保証する標準的なルールと手順に基づく詳細なプロセスのおかげで可能になっています。

そのルールと手順のセットは**プロトコル**として知られています。

プロトコルは基本的に、複数のデバイスが通信する際に、交換されるデータの形式など特定の基準をすべて守ることで、互いに通信できるようにします。

プロトコルは、データ交換に関わるすべてのデバイスが互いに理解できるようにします。比喩的に言えば、すべてのデバイスが「同じ言語を話す」方法を知っているのです。

その中核となるのが、インターネットを可能にするTCP/IPというプロトコル群です。

**TCP/IP**は、データがどのようにパッケージ化され、送信され、ルーティングされ、ネットワークを通じて受信されるかを決定する通信プロトコルのスイートです。情報が信頼性をもって運ばれることを保証します。

特に、**IP**（Internet Protocolの略）は、データがネットワーク内の正しい宛先に届くことを保証するプロトコルです。

なぜこれが重要なのでしょうか？インターネットは相互接続されたデバイスのグローバルネットワークであることを思い出してください。デバイスがリクエストを送るには、まずどこにデータを送るかを知る必要があります。これがIPアドレスの主な役割であり、Internet Protocol（IP）の基盤です。IPアドレスはネットワーク上のデバイスを一意に識別し、送信元デバイスがデータの送信先を知ることを可能にします。

Internet Protocol自体はコネクションレスであり、各データ単位（「パケット」と呼ばれます）が接続を確立せずに独立してネットワークを通じて送信され、正しく届くかや順序が保証されません。届いたとしても送信元には通知されません。

ここでTCPのようなプロトコルが重要になります。

**TCP**（Transmission Control Protocol）は、データを小さなセグメントに分割します。これらの基本的なデータ単位には番号が付けられ、目的地に届くと特定のアルゴリズムで再構成されます。このアルゴリズムはデータが完全かつ正しいかを検出し、エラーや紛失したパケットをチェックして必要に応じて送信元に再送を要求し、情報が正しく受信されることを保証します。

Transmission Control ProtocolとInternet Protocolは一緒に、インターネット上でデータが信頼性をもって送信されるためのルールと基準を定めています。

TCP/IPに加えて、現代のインターネットで非常に重要なもう一つのプロトコルがあります：HTTPです。

HTTPはアプリケーションレベルで動作します。

**HTTP**は**HyperText Transfer Protocol**の略で、クライアント（例えばブラウザ）とサーバー間の通信のルールと基準を定義します。

ブラウザでウェブサイトにアクセスしようとすると、ブラウザはその特定のリソースと関連資産を要求するHTTPリクエストをサーバーに送信します。サーバーはHTTPリクエストを受け取り、処理を実行し、要求されたコンテンツやリクエストが成功しなかった理由を示す情報を含むHTTPレスポンスをクライアントに返します。

HTTPリクエストとHTTPレスポンスは非常に似た構造を持ち、以下を含みます：

* リクエストライン（開始行）
    
* ヘッダー
    
* ボディ（任意）

## **HTTPリクエスト**

HTTPリクエストの**リクエストライン**は主に3つの部分に分かれます：

* メソッドは、要求されているアクションを表します。例えば、`GET`はデータを取得し、`POST`はデータを提出します。他の一般的なメソッドには`PUT`、`PATCH`、`DELETE`、`HEAD`があります。
    
* パスまたはURIは、クライアントが要求している特定のリソースを示します。
    
* 通信に使用されるHTTPプロトコルのバージョン。例えば、`HTTP/1.1`。

これは、ルート`GET`パスで`/`メソッドを使い、HTTPバージョン1.1の例です：

`GET / HTTP/1.1`

**ヘッダー**にはリクエストに関するキーと値のペアが含まれます。

例えば：`Content-Type: application/json`。

ヘッダーの下には空行があり、メタデータ（リクエストに関する追加情報）が完了したことを示します。

最後に**ボディ**があります。これは任意で、クライアントがサーバーに情報を送信する必要がある場合に送られます。例えば、`POST`メソッドで新しいデータを提出するときです。

HTTPリクエストの例は以下の通りです：

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

## **HTTPレスポンス**

HTTPレスポンスも非常に似た構造を持ちます：

* ステータスライン
    
* ヘッダー
    
* ボディ
    

まず、**ステータスライン**にはリクエストの結果に関する情報が含まれます。通常、以下が含まれます：

* 通信に使用されるHTTPのバージョン。例えば、`HTTP/1.1`。
    
* ステータスコードはリクエストの結果を示します。
    
* 理由フレーズはステータスコードの簡潔な人間向け説明で、何が起きたか理解するのに役立ちます。
    

一般的なステータスコードには以下があります：

* `200 OK`
    
* `201 CREATED`
    
* `400 BAD REQUEST`
    
* `404 NOT FOUND`
    
* `403 FORBIDDEN`
    
* `500 INTERNAL SERVER ERROR`
    

ステータスコードは多数あり、左からの最初の数字で分類されます。

* `1xx`は情報提供のレスポンス。
    
* `2xx`は成功のレスポンス。
    
* `3xx`はリダイレクトのメッセージ。
    
* `4xx`はクライアントエラーのレスポンス。
    
* `5xx`はサーバーエラーのレスポンス。
    

HTTPレスポンスのステータスラインの下には、レスポンスに関するメタデータを含む**ヘッダー**があります。例えば、`Content-Type: text/html`。これにはコンテンツの長さ、日付、場所、クライアントに役立つ追加情報も含まれる場合があります。

最後にHTTPレスポンスの**ボディ**があり、クライアントが要求したデータや、リクエストが成功しなかった場合の説明が含まれます。これは任意ですが、ほとんどのレスポンスに含まれています。

HTTPレスポンスの例は以下の通りです：

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

HTTPについて理解したところで、HTTPSについても気になるかもしれません。

HTTPSはHTTPの「安全な」バージョンと考えられ、ブラウザとサーバー間で送信されるデータがセキュリティのために暗号化されています。現代のウェブトラフィックのほとんどはHTTPSを使用しています。

TCP/IPやHTTPのようなプロトコルのおかげで、ワールドワイドウェブやインターネット全体が、私たちが毎日頼りにする不可欠なツールとなりました。これらのおかげで、デバイスはネットワーク上で正しく通信し、世界中でデータを交換できるようになっています。
