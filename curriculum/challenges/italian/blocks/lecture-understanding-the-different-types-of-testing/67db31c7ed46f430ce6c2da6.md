# --description--

Il mocking è il processo di sostituire dati reali con dati falsi che simulano il comportamento di componenti reali. Questo permette a tester e sviluppatori di testare componenti specifici dell'applicazione senza dover dipendere da dipendenze esterne.

Un caso d'uso comune per il mocking dei dati è quando si gestiscono chiamate API. Immagina di avere una chiamata fetch come questa nella tua applicazione:

```js
export const postService = {
  async getPosts() {
    const response = await fetch("https://api.example.com/posts");
    return response.json();
  },
};
```

Se devi testare la funzione `postService`, potresti fare il mock della risposta API invece di effettuare chiamate API continue per recuperare i dati. Recuperare dati da un'API richiede tempo e dipende da fattori esterni come la disponibilità della rete e le risposte del server. Fare il mock della risposta API elimina questi fattori esterni e permette ai tester di lavorare con un insieme di dati prevedibile e controllato.

Ora che abbiamo capito cos'è il mocking, confrontiamolo con lo stubbing. Gli stub sono oggetti che restituiscono risposte predefinite o dati fittizi per un comportamento previsto in un'applicazione. Per esempio, puoi stubbare il comportamento di una connessione al database nei tuoi test senza dover dipendere da una connessione reale al database.

Ecco un esempio di stub in Jest:

```js
import { postService } from "./postService";
import { db } from "./db";

jest.mock("./db", () => ({
  db: {
    fetchPosts: jest.fn(),
  },
}));

test("should return stubbed posts", async () => {
  const fakePosts = [
    { id: 1, title: "Stubbed Post", content: "This is a stubbed post." },
  ];

  db.fetchPosts.mockResolvedValue(fakePosts);

  const posts = await postService.getPosts();

  expect(posts).toEqual(fakePosts);
});
```

`db.fetchPosts.mockResolvedValue(fakePosts);` è lo stub in questo esempio.

L'ultimo componente di cui parlare sono i fake. I fake sono versioni semplificate di componenti reali senza la complessità o gli effetti collaterali dell'elemento reale. Per esempio, puoi fare il fake di un database memorizzando i dati in memoria invece di interagire con il database reale. Questo ti permetterà di simulare operazioni sul database in memoria, che sarà molto più veloce rispetto a lavorare con il database reale. Un altro esempio comune è usare i fake quando si lavora con i file system. Puoi creare un file system fake al posto di quello reale durante i test.

Man mano che costruisci applicazioni software più complesse, considera di usare mock, stub o fake dove appropriato nei tuoi test.
