# --interactive--

Rendere accessibile un contenuto statico può essere relativamente semplice, ma il contenuto dinamico può risultare più complesso. Qui entra in gioco WAI-ARIA.

Vediamo cos’è WAI-ARIA, qual è il suo scopo, come funziona e alcuni esempi.

WAI-ARIA sta per Web Accessibility Initiative - Accessible Rich Internet Applications. È una specifica che migliora l’accessibilità per contenuti dinamici e componenti dell’interfaccia utente.

Nota che WCAG e WAI-ARIA non sono la stessa cosa. WCAG fornisce linee guida generali per l’accessibilità web, mentre WAI-ARIA offre regole specifiche per rendere accessibili contenuti dinamici e interattivi agli utenti di tecnologie assistive.

Quindi, lo scopo principale di WAI-ARIA è migliorare l’accessibilità per contenuti dinamici e componenti dell’interfaccia utente che non hanno equivalenti nativi in HTML.

WAI-ARIA funziona introducendo un insieme di attributi che puoi aggiungere agli elementi HTML per fornire informazioni semantiche aggiuntive. Questi attributi sono suddivisi in ruoli, stati e proprietà.

Il ruolo ARIA definisce lo scopo di un elemento all’interno di un sito web o di un’app web. Ecco un esempio di impostazione del ruolo su `button` per un elemento `div`.

```html
<div role="button">Click Me</div>
```

Facendo questo stai indicando alla tecnologia assistiva che l’elemento è un pulsante. Tuttavia, i ruoli non forniscono alcuna funzionalità. Dare a questo `div` un `role` di `button` non lo farà comportare come un pulsante. Per farlo apparire e comportarsi come un pulsante dovresti usare CSS e JavaScript per ottenere il risultato desiderato.

Ecco un esempio di utilizzo di HTML, CSS e JavaScript per creare un elemento `button` personalizzato.

**NOTA**: Non preoccuparti di capire il codice CSS e JavaScript. Imparerai questi linguaggi nei moduli futuri.

:::interactive_editor

```html
<link href="styles.css" rel="stylesheet">
<div id="custom-btn" role="button">Click Me</div>
<script src="index.js"></script>
```

```css
#custom-btn {
  display: inline-block;
  padding: 0.4em 1em;
  font-size: 1rem;
  font-family: sans-serif;
  color: buttontext;
  background-color: buttonface;
  border: 1px solid #ccc;
  border-radius: 4px;
  cursor: pointer;
  user-select: none;
  text-align: center;
}

#custom-btn:focus {
  outline: 2px solid Highlight;
  outline-offset: 2px;
}

#custom-btn:active {
  background-color: #ddd;
}
```

```js
const button = document.getElementById("custom-btn");

button.addEventListener("click", () => {
  alert("Button clicked!");
});

button.addEventListener("keydown", (e) => {
  if (e.key === "Enter" || e.key === " ") {
    e.preventDefault(); 
    button.click();
  }
});
```

:::

È sempre meglio usare l’elemento `button` o `input` nativo con `type="button"` invece.

Le proprietà ARIA forniscono dettagli aggiuntivi sugli elementi. Per esempio, la proprietà `aria-labelledby` ti permette di collegare un elemento a un’etichetta specifica:

```html
<h2 id="header-id">About freeCodeCamp</h2>
<button id="button-id" aria-labelledby="header-id button-id">Learn More</button>
```

Questo renderà gli elementi comprensibili e navigabili per gli utenti di tecnologie assistive.

Per ottenere il massimo da WAI-ARIA, cerca di usare HTML nativo ogni volta che è possibile, poiché generalmente offre più accessibilità di base.

Usa WAI-ARIA solo quando HTML non basta, e non dimenticare di testare con tecnologie assistive come i lettori di schermo, o di far testare il tuo lavoro a persone con disabilità. Inoltre, assicurati che gli stati e le proprietà WAI-ARIA si aggiornino in tempo reale con il contenuto. Evita di usare troppo ARIA, perché può spesso risultare confuso.
