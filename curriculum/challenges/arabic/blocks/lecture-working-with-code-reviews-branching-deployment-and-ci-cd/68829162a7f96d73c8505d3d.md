# --description--

إذا كنت تتعامل مع مطوِّرين ذوي خبرة، سواء في مجتمع عبر الإنترنت أو شخصيًا، فمن المحتمل أنك سمعت عن "تعارضات الدمج المخيفة". لكن ما هي بالضبط؟

عندما تستخدم استراتيجية تفرع صحيحة كما ناقشنا في الدروس السابقة، قد تعمل على ميزتين مختلفتين في فرعين مختلفين في نفس الوقت. أو ربما تعمل على شيء وزميلك يعمل على فرع آخر. في كلتا الحالتين، لديك تغييران مستقلان. لكن أحيانًا، يقوم كلا الفرعين بتحرير نفس الملف. وإذا تم دمج الفرع الآخر في `main` أولًا، تصبح تغييرات فرعك متأخرة.

غالبًا ما يمكن إصلاح هذا بدمج `main` مرة أخرى في فرع الميزة الخاص بك. لكن عندما تتعارض التغييرات من الفرع الآخر مع فرعك، لا يستطيع Git تحديد التغييرات التي تريد الاحتفاظ بها أثناء الدمج. وهذا ما يسمى "تعارض دمج".

كيف تصلح ذلك؟ بالنسبة للتعارضات الصغيرة والبسيطة، يوفر GitHub واجهة ويب لإصلاحها. يظهر زر "حل التعارضات" في أسفل طلب السحب إذا كان هناك تعارضات. إذا نقرت عليه، سترى الملف المتعارض الذي يحتوي على شفرتك والشفرة المتعارضة. يضيف Git "علامات تعارض الدمج" حول شفرتك والشفرة المتعارضة لمساعدتك على تحديد المشكلة. يبدو الأمر كالتالي:

```sh
1 <<<<<<< feat/conflict
2 Here's some code from my first PR :)
3 =======
4 Here's some code that was merged to main that conflicts with the code on my PR
5 >>>>>>> main
```

علامة البداية هي `<<<<<<< feat/conflict`، وتدل على بداية التغييرات التي يجريها فرعنا الحالي. ثم نرى `Here's some code from my first PR :)`. بعد ذلك علامة `=======` التي تشير إلى نهاية تغييراتنا وبداية تغييرات فرع `main`. نرى أن التغييرات التي أجريت على `main` هي السطر `Here's some code that was merged to main that conflicts with the code on my PR`. وبعد ذلك علامة الإغلاق `>>>>>> main` التي تظهر نهاية التعارضات.

كيف نصلحها؟ عليك أن تقرر أي التغييرات تريد الاحتفاظ بها كجزء من تعارضات الدمج. إذا أردت الاحتفاظ بالتغييرَين، يمكنك حذف علامات التعارض فقط وترك التغييرات كما هي. بدلاً من ذلك، إذا أردت الاحتفاظ بتغييرات فرع واحد فقط، تحذف علامات التعارض والتغييرات من الفرع الآخر (التي لا تريد الاحتفاظ بها). وأخيرًا، إذا لم تعجبك أي من التغييرات، يمكنك حذف كل شيء من علامة البداية إلى علامة النهاية (شاملة).

لنحذف تغييرات فرع `main`، ونحتفظ بتغييراتنا. هذا ما لدينا الآن:

```sh
1 Here's some code from my first PR :)
```

انقر على زر "وضع علامة كمُحلَّل"، الذي يخبر GitHub بأنك راضٍ عن الحل. إذا كان هناك ملفات إضافية، سينتقل GitHub إلى الملف التالي. لكن بما أن هذا هو الملف المتعارض الوحيد، سترى زر "تأكيد الدمج" يظهر. انقر عليه وسيُنشئ GitHub تأكيد دمج لك. ستعود إلى طلب السحب، حيث سترى تأكيد الدمج الجديد والتعارضات تم حلها. وهكذا نكون انتهينا!

ماذا لو لم تستطع تعديل التعارضات مباشرة على GitHub لأنها معقدة جدًا؟

إذا كان فرعك يحتوي على عدد قليل من التأكيدات، أو فقط بعض التأكيدات تعدل ذلك الملف، قد تتمكن من دمج `main` في فرعك:

```sh
git checkout feat/conflict
git fetch origin
git merge origin/main
```

أثناء معالجة Git لهذا الدمج، ستظهر تعارضات تحتاج إلى حلها يدويًا. سيخبرك Git بالملفات التي تحتاج إلى إصلاح. عند فتح الملفات المتعارضة في محررنا، نرى نفس علامات التعارض، مع بعض الإضافات المفيدة بفضل VSCode. تابع وأصلح التعارضات في الملف بنفس الطريقة. لكن هذه المرة، سنحتاج إلى إنشاء التأكيد يدويًا. الأمر نفسه كما في إنشاء أي تأكيد آخر:

```sh
git add .
git commit -m "chore: resolve conflicts"
```

بعد تنفيذ هذه الأوامر، نرى أن Git قبل الحل الذي قدمناه.

أحيانًا، عندما يكون لديك العديد من التأكيدات أو التعارضات، يكون إعادة الأساس (rebase) مفيدًا. بدلًا من دمج التغييرات، يأخذ إعادة الأساس كل التأكيدات على فرعك ويعيد تطبيقها بحيث تأتي بعد أحدث التغييرات على `main`. بمصطلحات Git، هذا يعيد "إعداد" أساس فرعك إلى أحدث تأكيد على `main`.

لتجربة هذا، نحتاج إلى فرع جديد:

```sh
git checkout main
git pull
git checkout -b feat/rebase
```

لننشئ مجموعة من التأكيدات على فرعنا، حتى نتمكن من ممارسة إعادة الأساس:

```sh
echo "change one" > README.md
git add .
git commit -m "change one"
echo "change two" > README.md
git add .
git commit -m "change two"
echo "change three" > README.md
git add .
git commit -m "change three"
```

الآن، لنقم بتغيير آخر على `main` لإنشاء تعارض:

```sh
git checkout main
echo "change four" > README.md
git add .
git commit -m "change four"
```

وننتقل إلى فرع إعادة الأساس الخاص بنا:

```sh
git checkout feat/rebase
```

بعد ذلك، شغّل الأمر `git rebase -i main`. سيعيد هذا فرعنا إلى الحالة الحالية لفرع `main`. العلامة `-i` تخبر Git بأداء إعادة الأساس في الوضع التفاعلي. سترى تأكيداتك مع كلمة `pick` بجانبها. `pick` تعني استخدام التأكيد كما هو لإعادة الأساس. احفظ وأغلق الملف، وستظهر رسالة خطأ تخبرنا بوجود تعارضات.

تأكيد `change one` يتعارض مع تأكيد `change four`، ويجب حل ذلك. افتح الملف في محررك، وسترى نفس واجهة المستخدم التي رأيناها مع التعارضات سابقًا. لنحتفظ بتغييرات `change one`. بعد ذلك، شغّل `git add .` لإضافة التغييرات، و`git rebase --continue` لإخبار Git بأننا جاهزون لمتابعة إعادة الأساس. بما أننا حللنا التعارض في التأكيد الأول، يمكن لـ Git التعامل تلقائيًا مع إعادة الأساس لبقية التأكيدات.

كخطوة تنظيف أخيرة، دعنا ندمج تأكيدات التغيير الثلاثة في تأكيد واحد لتاريخ أنظف. شغّل `git rebase -i main` مرة أخرى للدخول إلى المحرر. اترك الأمر الأول كـ `pick` وغيّر الأمرين التاليين إلى `squash`. قد تواجه نفس التعارض مرة أخرى. إذا حدث ذلك، حلّه بنفس الطريقة السابقة. بعد حل التعارض ومتابعة إعادة الأساس (أو فورًا إذا لم يكن هناك تعارض)، ستظهر نافذة محرر رسالة التأكيد الجديدة. ستكون هذه الرسالة لتأكيدك الجديد، الذي يحتوي على تغييرات التأكيدات الثلاثة التي دمجناها. لنسمّ هذا التأكيد `change three complete`. احفظ وأغلق المحرر، وسيكمل Git إعادة الأساس.

إذا راجعنا سجل التأكيدات الآن، نرى أن لدينا تأكيدًا واحدًا فقط `change three complete`، وهو مبني بشكل صحيح على تأكيد `change four`. هذا يعني أنك أكملت إعادة أساس ناجحة مع دمج التأكيدات! والآن لديك الأدوات التي تحتاجها لتصبح بطل تعارضات الدمج!
