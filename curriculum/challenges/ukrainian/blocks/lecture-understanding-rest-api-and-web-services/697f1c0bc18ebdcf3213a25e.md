# --description--

**REST** означає передачу репрезентативного стану. Уявіть це як набір рекомендацій, які описують, як веб-клієнти та сервери повинні взаємодіяти один з одним через інтернет.

У цьому уроці ви дізнаєтеся, що таке REST, його ключові концепції та як він працює.

REST визначає спосіб взаємодії різних систем за допомогою стандартних веб-технологій. За допомогою REST клієнти, такі як веб-браузери, мобільні застосунки та API, спілкуються із сервером, використовуючи загальні HTTP-методи, такі як `GET`, `POST`, `PUT`, `PATCH` та `DELETE`, які ми розглядали в попередньому уроці.

Деякі основні концепції REST включають **безстанність**, **ресурси**, **HTTP-методи** та **коди відповіді або статусу**.

**Безстанність** означає, що кожен запит, який клієнт надсилає серверу, обробляється незалежно. Сервер не зберігає інформацію про попередні запити. Через це кожен запит повинен містити всю інформацію, необхідну для його обробки. Наприклад, якщо користувач автентифікований, його токен автентифікації має надсилатися з кожним запитом.

У REST усе розглядається як ресурс. Ресурс — це частина даних, до якої можна отримати доступ або змінити її. Кожен ресурс ідентифікується URL-адресою, наприклад `/products` для всіх продуктів або `/products/1` для конкретного продукту.

Нижче наведено огляд HTTP-методів, кодів статусу та покроковий приклад того, як працює архітектура REST.

## HTTP-методи

HTTP-методи визначають дію, яку потрібно виконати над ресурсом:

* `GET`: Отримати ресурс
    
* `POST`: Створити новий ресурс
    
* `PUT / PATCH`: Оновити існуючий ресурс
    
* `DELETE`: Видалити ресурс
    

## Коди статусу HTTP

Після обробки запиту сервер відповідає **кодом статусу**, який вказує на результат:

* `200` **(OK)**: Запит успішний
    
* `201` **(Created)**: Створено новий ресурс
    
* `400` **(Bad Request)**: Клієнт надіслав некоректні дані
    
* `404` **(Not Found)**: Запитуваний ресурс не існує
    
* `500` **(Internal Server Error)**: На сервері сталася помилка
    

## Як працює REST: покроковий приклад

### 1\. Клієнт надсилає запит

* Клієнт надсилає HTTP-запит, використовуючи метод, наприклад `GET` або `POST`
    
* Запит містить URL, що ідентифікує ресурс
    
* Може також містити заголовки, метадані та тіло запиту
    

### 2\. Сервер отримує запит

* Сервер перевіряє формат запиту та автентифікацію (якщо потрібно)
    
* Визначає цільовий ресурс за URL
    
* Визначає дію, яку потрібно виконати, на основі HTTP-методу
    
* Перевіряє дозволи та контроль доступу
    

### 3\. Сервер виконує дію

* `GET`: Отримати ресурс
    
* `POST`: Створити новий ресурс
    
* `PUT / PATCH`: Оновити ресурс
    
* `DELETE`: Видалити ресурс
    
* Виконується будь-яка інша необхідна логіка
    

### 4\. Сервер формує відповідь

* Сервер обирає відповідний код статусу (наприклад, `200`, `201` або `404`)
    
* Форматує дані відповіді, зазвичай у форматі **JSON**
    
* Додає відповідні заголовки, такі як `Content-Type` або `Cache-Control`
    

### 5\. Сервер надсилає відповідь

* Відповідь містить код статусу, заголовки та необов’язкове тіло
    
* Стан клієнта не зберігається на сервері (безстанність)
    

### 6\. Клієнт обробляє відповідь

* Клієнт перевіряє код статусу, щоб визначити успіх чи помилку
    
* Обробляє будь-які дані, отримані від сервера
