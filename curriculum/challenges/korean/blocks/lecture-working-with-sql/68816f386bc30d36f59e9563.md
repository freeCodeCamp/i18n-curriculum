# --description--

표준화는 관계형 데이터베이스를 조직하는 과정으로, 데이터 중복을 줄이고 데이터 무결성을 향상시키기 위해 정해진 정규형을 사용하는 것입니다.

데이터베이스를 표준화하려면, 표준화 규칙에 따라 단순화할 수 있는 기회를 찾기 위해 표 구조 내 속성과 관계를 분석하세요. 그런 다음 데이터를 더 작고 집중된 표로 나누고, 기본 키와 외래 키를 사용해 이들 사이에 관계를 설정합니다. 이렇게 나눈 작은 표들은 원래 있던 모든 데이터를 저장하지만, 더 큰 표에 비해 관리, 조직, 작업이 훨씬 쉬워집니다.

데이터 중복을 최소화함으로써 표준화는 저장 공간을 줄이고 불일치를 방지합니다. 예를 들어, 고객 주소가 변경되면 모든 주문 기록마다 수정할 필요 없이 고객 표 한 곳에서만 업데이트하면 됩니다.

또한 기본 키와 외래 키 제약 조건으로 의존성을 강제해 데이터 무결성을 유지하는 데 도움을 줍니다. 이는 삽입, 수정, 삭제 이상 현상의 가능성을 줄입니다.

표준화된 데이터베이스는 이해하고 유지 관리하기 쉬워 잘 설계된 데이터베이스 시스템에 기여합니다.

표를 조직하기 위해 따라야 할 규칙 집합을 정규형이라고 합니다. 각 정규형은 이전 정규형을 기반으로 하며, 높은 정규형으로 갈수록 규칙이 더 엄격해집니다. 제1정규형(1NF)은 가장 기본적인 규칙을 가진 기초 정규형입니다.

표가 제1정규형(1NF)에 속하려면 다음 조건을 만족해야 합니다:

- 각 셀에는 하나의 값만 들어 있어야 합니다. 예를 들어, `students` 표에 여러 전화번호를 가진 학생이 있을 수 있습니다. 이 전화번호들을 한 셀에 쉼표로 구분해 저장하는 대신, `phone_numbers` 표를 별도로 만들어 외래 키가 `students` 표를 가리키게 하고 각 셀에는 하나의 값만 저장해야 합니다. 이렇게 하면 각 값이 원자적이고 설계가 표준화됩니다.
- 모든 레코드(행)는 고유해야 합니다. 기본 키가 이 요구사항을 강제하지만, 1NF는 중복 행이 없어야 한다고 명시적으로 요구합니다.
- 행과 열의 순서는 중요하지 않아야 합니다. 데이터는 물리적 위치에 의존하지 않아야 합니다.

본질적으로 제1정규형(1NF)은 값을 단순하고 원자적으로 만드는 데 초점을 맞춥니다.

제2정규형(2NF)은 1NF를 기반으로 하며, 1NF 요구사항을 모두 충족하는 것 외에 추가 제약 조건을 요구합니다.

2NF를 이해하려면 두 가지 핵심 개념인 슈퍼키와 후보 키에 익숙해져야 합니다. 슈퍼키는 표의 각 행을 고유하게 식별하는 속성 집합으로, 슈퍼키 내 모든 열에서 두 행이 동일한 값을 가질 수 없습니다.

예를 들어, `customers` 표에 세 개의 열이 있다고 가정해 보겠습니다:

```sql
customer_id | name | email
```

`customer_id`와 `email`가 각 레코드마다 고유해야 한다고 합시다. 이 표의 슈퍼키 예시는 다음과 같습니다:

- `customer_id`
- `{ customer_id, name }`, 이 조합은 `customer_id`이 단독으로 고유하므로 `name`의 고유성과 관계없이 행을 고유하게 식별합니다.
- `{ customer_id, name, email }`
- `email`, 이 열에도 고유 제약 조건을 추가했으므로 이메일로 각 레코드를 고유하게 식별할 수 있습니다.

함께 각 행을 식별하는 속성 집합은 모두 슈퍼키가 될 수 있습니다.

후보 키는 비슷하지만 약간 다릅니다. 후보 키는 표에서 각 레코드를 고유하게 식별할 수 있는 하나 이상의 열 집합입니다. 표는 여러 후보 키를 가질 수 있지만, 그중 하나를 기본 키로 선택해야 합니다.

두 개념이 비슷하게 들리죠? 차이점은 슈퍼키는 각 행을 고유하게 식별하는 데 필요하지 않은 추가 속성을 포함할 수 있지만, 후보 키는 행을 고유하게 식별하는 속성만 포함한다는 점입니다.

`customers` 예시에서 `customer_id`, `{ customer_id, name }`, `{ customer_id, name, email }`, `email`이 슈퍼키가 될 수 있다고 했습니다. 이 경우 후보 키는 `customer_id`과 `email`만 될 수 있는데, 후보 키는 행을 고유하게 식별하지 않는 `name` 같은 속성을 포함할 수 없기 때문입니다. 이것이 후보 키와 슈퍼키를 구분하는 최소성 원칙을 보여줍니다.

이제 이 개념에 익숙해졌으니 제2정규형(2NF)으로 돌아가겠습니다.

제2정규형(2NF)은 부분적 종속성을 해결하는 데 기반합니다. 부분적 종속성은 기본 키의 일부에만 의존하는 기본 키에 속하지 않은 속성이 있을 때 발생합니다(복합 기본 키는 여러 열로 구성된 기본 키입니다). 이는 데이터 중복과 수정 이상 현상을 초래할 수 있습니다.

예를 들어, 다음 열을 가진 `orders` 표가 있다고 합시다:

```sql
order_id | item_id | order_date | quantity | order_shipping_city
```

이 표에서 기본 키는 `order_id`과 `item_id`의 조합입니다. 같은 상품 ID가 다른 주문에 있을 수 있지만, 이 조합은 고유합니다. `order_id`과 `order_shipping_city` 사이에 부분적 종속성이 있음을 알 수 있습니다. `order_id`는 기본 키의 일부입니다. `order_shipping_city`은 `order_id`에 의존하는데, 같은 ID를 가진 모든 주문은 같은 배송 도시를 가집니다. 하지만 배송 도시는 `item_id`에 의존하지 않는데, 이 또한 기본 키의 일부입니다. 따라서 `order_shipping_city`는 전체 기본 키에 의존하지 않습니다.

이를 해결하려면 표를 `order_header`과 `order_items` 같은 두 개의 작은 표로 나눌 수 있습니다. `order_header` 표에는 주문에 대한 최상위 정보를 저장할 수 있습니다:

```sql
order_id | order_date | order_shipping_city
```

`order_items` 표에는 제출된 다양한 주문의 상품 정보를 저장할 수 있습니다:

```sql
order_id | item_id | quantity
```

이 변경으로 두 표 모두 제2정규형(2NF)에 속하게 됩니다.

다음은 제3정규형(3NF)입니다. 이 정규형은 제2정규형을 기반으로 합니다.

표가 제3정규형에 속하려면:

- 제2정규형에 속해야 합니다.
- 모든 비키 속성이 기본 키에 직접 의존해야 하며, 비키 속성이 다른 비키 속성에 의존해서는 안 됩니다.

제3정규형(3NF)을 이해하려면 추이적 종속성을 알아야 합니다. 추이적 종속성은 기본 키에 속하지 않은 속성이 또 다른 기본 키에 속하지 않은 속성에 의존하고, 그 속성이 다시 기본 키에 의존할 때 발생합니다.

예를 들어, `orders` 표를 다음 열로 수정해 보겠습니다:

```sql
order_id | customer_id | customer_city | city_postal_code | order_date | quantity
```

이 새 표의 기본 키는 각 행을 고유하게 식별하는 `order_id`입니다.

이 표에는 추이적 종속성이 있습니다:

```sql
order_id > customer_id > customer_city > city_postal_code
```

도시의 우편번호는 고객 도시로 결정되고, 고객 도시는 고객 ID로 결정되며, 고객 ID는 주문 ID로 결정됩니다. 기본 키에 속하지 않은 속성인 `city_postal_code`은 기본 키에 속하지 않은 `customer_city`에 의해 결정되지만, `customer_city`은 `order_id`를 거쳐 `customer_id`에 의해 결정됩니다. 이는 같은 도시에 사는 고객이 주문할 때마다 도시 우편번호가 반복 저장됨을 의미합니다. 이는 데이터 중복을 초래합니다. 또한 도시 우편번호가 변경되면 여러 레코드를 모두 수정해야 합니다.

추이적 종속성을 해결하려면 표를 여러 개로 나누어야 합니다. 먼저 `orders` 표:

```sql
order_id | customer_id | order_date | quantity
```

그다음 `customers` 표:

```sql
customer_id | city_name
```

그리고 `cities` 표:

```sql
city_name | city_postal_code
```

이렇게 하면 표에서 추이적 종속성이 제거됩니다.

마지막으로 보이스-코드 정규형(BCNF)이 있습니다. 이는 제3.5정규형으로도 알려져 있으며, 제3정규형이 해결하지 못하는 이상 현상을 다룹니다. 표가 보이스-코드 정규형(BCNF)에 속하려면:

- 제3정규형에 속해야 합니다.
- 함수적 종속성의 왼쪽 속성 집합이 모두 슈퍼키여야 합니다.

이 정규형의 목표는 다른 속성을 결정하는 모든 속성 또는 속성 집합이 슈퍼키(후보 키 또는 그 상위 집합)임을 보장하는 것입니다.

정규형을 이해하고 적용하는 것은 견고하고 효율적인 관계형 데이터베이스를 설계하는 데 필수적입니다. 데이터 중복을 제거하고 논리적 데이터 의존성을 보장함으로써 표준화는 데이터 무결성 향상, 저장 공간 절감, 데이터베이스 유지 관리 간소화로 이어집니다. 보통 제3정규형(3NF)에 도달하는 것을 목표로 하세요.
