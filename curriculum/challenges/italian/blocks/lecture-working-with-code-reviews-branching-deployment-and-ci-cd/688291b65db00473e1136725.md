# --description--

CI sta per Continuous Integration, e CD sta per Continuous Delivery o Continuous Deployment. Insieme, questi processi ti permettono di distribuire automaticamente le modifiche alla tua applicazione dopo aver verificato che funzionino correttamente.

In una pipeline CI, potresti eseguire il linter e i test sul codice. Le pipeline CI di solito vengono eseguite sulle pull request per assicurarsi che le modifiche apportate non rompano l’app, e anche sul ramo `main` per garantire che sia pronto per la distribuzione.

Una pipeline CD potrebbe essere eseguita anche sulle pull request, se configuri un ambiente di anteprima per poter vedere le modifiche in tempo reale. Ma la eseguirai anche sul tuo ramo `main` per distribuire continuamente le modifiche man mano che vengono accettate e unite.

Ma come funzionano effettivamente con GitHub Actions? Prima, dobbiamo capire cosa sono realmente GitHub Actions. GitHub Actions fornisce runner effimeri (pensa a un computer temporaneo) che ti permettono di eseguire comandi specifici.

Le Actions si configurano con un file YAML. Puoi avere più file per specificare diverse azioni da eseguire. Per esempio, potresti avere un file per il tuo CI e un altro per il tuo CD.

Diamo un’occhiata a un esempio base di CI:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  lint:
    name: Lint and Test
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Source Files
        uses: actions/checkout@v4

      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10

      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

È molto, quindi spezzettiamolo. Guardando la prima sezione:

```yml
name: Node.js CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
```

La proprietà `name` definisce il nome dell’azione stessa. Questo appare in aree come la sezione dei controlli di stato di una PR. Le proprietà `on` definiscono quelli che si chiamano “trigger del workflow” — questi sono gli eventi che faranno partire l’esecuzione della tua azione.

In questo esempio, eseguiamo l’azione quando vengono pushati commit su `main`, e quando una pull request ha come obiettivo `main`. Questo significa che il CI viene eseguito quando qualcuno crea una pull request, e di nuovo quando la uniamo.

La sezione `jobs` è dove definisci cosa fa effettivamente la tua azione. La chiave `lint:` è una chiave arbitraria che definisce il nome interno per quel lavoro. La proprietà `name` definisce il nome esterno per questo lavoro specifico. E la proprietà `runs-on` definisce in che tipo di ambiente deve girare la tua azione — in questo caso, la eseguiamo nell’ultima versione supportata di Ubuntu.

La proprietà `steps` è dove avviene la magia. Questo blocco definisce i passaggi effettivi che la tua azione deve compiere. Diamo un’occhiata al primo blocco lì:

```yml
      - name: Checkout Source Files
        uses: actions/checkout@v4
```

Questo passaggio ha due proprietà, la `name` che il passaggio deve avere, e una proprietà `uses`. Questa proprietà `uses` è speciale, perché dice al runner dell’azione di andare a prendere un’azione pubblica da GitHub e di eseguirla, invece di eseguire un comando. L’azione `actions/checkout` gestisce il clonare e configurare il repository all’interno del tuo runner.

```yml
      - name: Use Node.js v22
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10
```

I due blocchi successivi usano anch’essi azioni di terze parti. Questi passaggi configurano Node e il gestore di pacchetti pnpm. Entrambi hanno una chiave `with`, che puoi usare per passare argomenti a quelle azioni di terze parti. Pensalo come passare argomenti a una funzione.

Gli ultimi quattro blocchi sono dove succede davvero la magia.

```yml
      - name: Install Dependencies
        run: pnpm install

      - name: Lint Source Files
        run: pnpm run lint

      - name: Verify Build
        run: pnpm run build

      - name: Run Tests
        run: pnpm run test
```

Abbiamo un blocco `Install Dependencies` che installa i pacchetti npm. Il blocco `Lint Source Files` esegue il linter, il blocco `Verify Build` assicura che il codice Typescript possa essere compilato, e il blocco `Run Tests` esegue i test unitari.

La cosa importante da sapere sulle azioni è che, per impostazione predefinita, se uno di questi passaggi fallisce, quelli rimanenti vengono saltati e l’intera esecuzione viene segnata come fallita.

Dove devi mettere i tuoi file YAML per poter eseguire queste azioni? Devono andare in una directory `.github/workflows` nel tuo repository affinché GitHub li trovi e li usi.

Ci sono molte altre cose che le azioni possono fare, e ti incoraggio a leggere la documentazione ed esplorare le azioni esistenti in progetti come il repository di freeCodeCamp. Ma per ora, dovresti avere una buona comprensione delle basi di GitHub Actions e CI/CD!
