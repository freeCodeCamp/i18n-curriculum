# --description--

網路服務和 REST API 支援你每天使用的大多數應用程式和網站。它們允許不同的應用程式彼此通訊並透過網路（最常見的是網際網路）共享資料。

API（應用程式介面）是一種軟體與其他軟體溝通的通用方式。網路服務是一種特定型別的 API，使用像是 HTTP 的網路技術，在網路上（通常是網際網路）交換資料。基於此，每個網路服務都是 API，但並非每個 API 都是網路服務。REST API 是一種常見的網路服務，遵循一組特定規則來請求和傳回資料。

本課程將著重於網路服務的高層次介紹，下一課將更詳細說明 REST API 的運作方式。

你可以將網路服務視為軟體與軟體之間的橋樑。它沒有使用者介面，也不打算直接使用。相反地，它完全專注於應用程式之間交換資料。

任何系統要被視為網路服務，必須具備以下特性：

* **在伺服器上執行**：網路服務必須像網站一樣線上託管，但它不是提供使用者可見的頁面，而是公開資料或某些功能。
    
* **機器對機器通訊**：網路服務不是為了讓人類直接查看，而是為了讓行動應用程式和網頁應用程式使用，這樣它們才能公開一些使用者最終可以查看的資料或功能。
    
* **語言無關**：網路服務可以用任何語言撰寫，例如 Java 或 C#，而任何應用程式都應該能與它互動，即使它是用不同語言撰寫的
    
* **跨平台**：機器執行在哪個平台並不重要。 例如，執行於 Linux 的網路服務不應阻止執行於 Windows 的裝置與其通訊。
    
* **使用標準的網路協定**：網路服務通常應該使用 HTTP/HTTPS 在網際網路上傳輸資料，使用 FTP 傳輸檔案，以及使用 SMTP 傳送電子郵件
    

**REST** 和 **SOAP** 是兩種最知名的網路服務型別，儘管還有其他方法存在，例如 **XML-RPC**。

**REST**，代表 **Representational State Transfer**，迄今為止是最廣泛使用的。大多數現代應用程式依賴 RESTful 網路服務，因為它們簡單、靈活，且能自然地與網路協同運作。

REST API 揭露資源，這些資源通常由 URL 路徑表述，例如 `/users` 或 `/users/20`。當客戶端請求資源時，伺服端會傳回該資源的表現。這種表現通常是 JSON 格式，但也可以是 XML 或純文字等格式。

客戶端可以透過使用 HTTP 標頭來指定他們想要的格式，例如 `Accept` 標頭。舉例來說，客戶端可以透過設定適當的 `Accept` 值來請求 JSON 回應。

```bash
Accept: application/json
```

你可以用這種方式指定你想要 XML：

```bash
Accept: application/xml
```

如果是 JSON，回傳的資源可能看起來像這樣：

```json
{
  "id": 20,
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "createdAt": "2025-09-22T10:30:00Z"
}
```

如果是 XML：

```xml
<user>
  <id>20</id>
  <name>John Doe</name>
  <email>john@example.com</email>
  <role>admin</role>
  <createdAt>2025-09-22T10:30:00Z</createdAt>
</user>
```

SOAP 代表 **Simple Object Access Protocol**。它僅使用 XML 資料格式，並且是需要更嚴格標準和安全性的企業層級系統中最常見的。

以下是 SOAP 資料可能的樣貌：

```xml
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/">
   <soapenv:Body>
      <getUser>
         <userId>20</userId>
      </getUser>
   </soapenv:Body>
</soapenv:Envelope>
```

XML-RPC 使用 XML 來編碼其請求和回應。這使它輕量但比 REST 靈活度較低。 

至於 UDDI，它不是資料交換服務，而是一個基於 XML 的目錄，企業可以在其中列出、發現並描述其可用的網路服務。
