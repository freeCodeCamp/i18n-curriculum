# --description--

Na palestra anterior, você aprendeu como um servidor backend identifica um usuário durante o login. Mas o que acontece depois que ele faz login? Como o servidor lembra quem ele é quando navega para uma nova página? É aí que entram os JSON Web Tokens (JWTs).

Em termos simples, um JWT é um cartão-chave digital que o servidor emite para um usuário após um login bem-sucedido. O usuário apresenta esse cartão-chave ao servidor em cada requisição subsequente para provar sua identidade sem precisar digitar a senha novamente.

Em vez de o servidor lembrar cada usuário ativo em sua própria memória, ele transfere essa responsabilidade para o cliente usando um processo simples de quatro passos:

1. **Login:** O usuário envia seu nome de usuário e senha, ou outro meio de login, para o servidor.

2. **Emissão:** O servidor verifica as credenciais com as que foram armazenadas quando o usuário se cadastrou. Se estiverem corretas, o servidor gera um JWT e o envia de volta para o usuário.

3. **Armazenamento:** O cliente recebe o JWT e o armazena, normalmente em um cookie do navegador ou no armazenamento local.

4. **Requisição:** Para cada requisição futura a uma rota protegida, o cliente envia automaticamente o JWT junto. O servidor verifica o token e concede acesso.

Um JWT parece uma string longa e caótica de caracteres. Ele consiste em três partes distintas separadas por dois pontos, assim:

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

A primeira parte é o **cabeçalho**, a segunda é o **payload** e a terceira parte é a **assinatura**.

O **cabeçalho** é um objeto JSON codificado que informa ao servidor qual tipo de token é (JWT) e qual algoritmo de hash foi usado para protegê-lo, como HMAC SHA256.

O **payload** é a parte do meio do token. Ele contém os dados reais, ou "claims", sobre o usuário. É aqui que você armazena informações como o id do usuário, nome de usuário ou o papel da conta, como admin ou user.

Uma nota importante de segurança é que o cabeçalho e o payload são apenas codificados (Base64), não criptografados. Isso significa que qualquer um pode facilmente decodificá-los e ler os dados dentro. Portanto, nunca armazene dados sensíveis como senhas em texto puro ou números de cartão de crédito dentro do payload de um JWT.

A **assinatura** é a parte final e mais crucial do token. O servidor pega o cabeçalho codificado, o payload codificado e os mistura com uma chave secreta conhecida apenas pelo servidor. Depois, ele gera um hash para criar a assinatura.

A assinatura previne adulterações. Se um usuário tentar decodificar seu token, mudar seu papel de "user" para "admin" e reencodificá-lo, a assinatura não vai mais corresponder aos dados alterados.

Quando o servidor recebe o token modificado, ele tenta recriar a assinatura usando sua chave secreta. Como o payload mudou, a assinatura calculada não vai bater com a assinatura do token, e o servidor rejeita instantaneamente a requisição com um código 401 Unauthorized.

Em resumo, os JWTs permitem autenticação sem estado. O servidor não precisa armazenar dados de sessão em um banco de dados, ele simplesmente confia no token porque a assinatura prova que o próprio servidor o criou.
