# --description--

在前两节课中，你学习了如何验证用户同一性并使用 cookies 维护他们的会话。然而，自动随每个请求发送 cookies 会引入一种独特的安全漏洞，称为跨站请求伪造（CSRF）。

CSRF 是一种攻击，它迫使已认证的用户在他们当前登录的 web 应用上执行不想要的操作。在 CSRF 攻击中，恶意网站欺骗受害者浏览器向你的后端服务器发出欺诈请求，利用服务器对用户浏览器 cookie 的信任。

为了理解 CSRF，考虑以下银行应用的逐步示例：

1. **准备阶段：** 你登录你的银行网站，比如 `trusted-bank.com`。银行验证你的凭证并将会话令牌存储在你的浏览器 cookie 中。

2. **陷阱：** 你没有退出银行账户，打开一个新的浏览器标签页，访问一个恶意网站，比如 `evil-hacker-site.com`。

3. **触发：** 恶意网站包含一个隐藏的脚本或表单，自动向银行后端提交请求，比如 `POST trusted-bank.com/transfer?amount=1000&to=hacker`。

4. **执行：** 因为你仍然登录着银行，浏览器会自动将银行会话 cookie 附加到该请求中。银行服务器看到有效的 cookie，认为这是你有意发出的请求，于是转账给黑客。

这里的关键是，恶意网站无法直接读取你的 cookie 或窃取你的数据。它只是利用了浏览器自动将 cookie 附加到任何发往目标服务器的请求这一事实。

主要有两种防御措施：**SameSite Cookie 属性** 和 **CSRF 令牌**。

最简单的防御是为你的 cookies 设置 `SameSite` 属性。这告诉浏览器何时允许随跨站请求发送 cookie。

- **SameSite=Strict：** 如果请求来自第三方网站，浏览器绝不发送 cookie。

- **SameSite=Lax：** 浏览器只在用户安全导航到目标站点时允许发送 cookie，比如点击普通链接，但会阻止跨站表单提交或 API 请求时发送 cookie。

为了更强的保护，后端框架使用 CSRF 令牌。当用户加载页面时，后端生成一个随机、唯一且不可预测的令牌，并将其附加到前端表单或会话中。

当客户端提交请求时，必须在自定义请求头或表单数据中包含该令牌。后端将提交的令牌与保存在用户会话中的令牌进行比较。因为恶意网站无法访问你的前端来复制这个秘密令牌，任何伪造的跨站请求都缺少该令牌，验证失败。

通过实施这些策略，你确保到达后端的请求是用户的有意操作，而不是恶意方强迫的未授权操作。
