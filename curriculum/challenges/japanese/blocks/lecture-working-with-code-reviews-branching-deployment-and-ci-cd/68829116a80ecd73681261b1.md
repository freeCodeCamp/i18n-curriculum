# --description--

ブランチを使って変更を分離し、`main` ブランチを汚さないように作業することについて少し話しました。でも、それはどういう意味で、なぜ重要なのでしょうか？

もし変更を直接 `main` ブランチにコミットし、上流リポジトリがあなたのものではない他の変更を受け入れた場合、「分岐した履歴」と呼ばれる状態になります。これは、`main` ブランチのコミットが上流の `main` ブランチのコミットと一致しなくなることを意味します。そして、上流の変更を同期するのが非常に難しくなります。

では、良いブランチ運用をしていてこの問題がないとしましょう。前のレッスンに従っていれば、リポジトリをフォークしローカルにクローンし、上流リモートが元のリポジトリを指しているはずです。

もしまだ設定していなければ、ここで簡単におさらいします。リポジトリをフォークし、フォークをローカルにクローンして、次のコマンドを実行してください。

```sh
git remote add upstream <url>
```

`<url>` は元のリポジトリのSSH URLに置き換えてください。フォークのURLではありません。

正しく設定された上流リモートがあれば、次の一連のコマンドで最新の上流の変更を取り込めます。

```sh
git checkout main
git fetch upstream
git merge upstream/main
```

これらは何をするのでしょう？`git checkout` について学んだので、最初のコマンドはローカルで `main` ブランチにいることを確認します。

2つ目のコマンド `git fetch upstream` は、Gitに上流リモート（元のリポジトリ）にある最新の変更を取得するよう指示します。ただし、pullとは異なり、Gitは自動的にブランチを更新しません。これは重要です。なぜなら、ローカルの `main` ブランチはフォークの `main` ブランチに接続されていて、元のリポジトリの `main` ブランチには接続されていないからです！git pull を実行すると予期しない動作になる場合があります。

最後のコマンド `git merge upstream/main` は、上流リモートの `main` ブランチから最新の変更を現在のブランチ（チェックアウトしたローカルの `main` ブランチ）にマージするようGitに指示します。

もしマージに失敗したらどうしますか？誤って `main` にコミットしていて、Gitが分岐した履歴をマージできない場合は？プルリクエストを `main` ブランチから作成していなければ、ハードリセットでGitにブランチを強制的に更新させることができます。

```sh
git reset --hard upstream/main
```

`git reset` コマンドはブランチの現在の状態をリセットできます。`--hard` フラグを渡すと、ローカルのファイルをブランチの状態に強制的に合わせます。これで作業のためのきれいな状態が確保されます。さらに `upstream/main` を渡すと、現在のブランチ（`main`）を上流リモートの `main` ブランチに合わせることをGitに伝えます。このコマンドは注意して使ってください。Gitの履歴を強制的に更新すると、思わぬトラブルが起きる場合があります。

ローカルの `main` ブランチが正しく更新できたら、その状態をリモートのフォーク（レッスンに従っていれば `origin`）にプッシュすることが重要です。次のコマンドを実行してください。

```sh
git push
```

これで現在の状態がリモートのフォークにプッシュされます。ただし、履歴をきれいにするためにハードリセットをした場合、リモートのフォークも分岐した履歴を持っていてプッシュが失敗する可能性があります。

その場合は強制プッシュで解決できます。

```sh
git push --force
```

**このコマンドは慎重に使ってください**。強制プッシュはリモートの履歴を書き換えるため、誤って使うと取り返しのつかないことになります。

同期した変更をリモートのフォークにプッシュするのは、もしローカルリポジトリを失った場合でも最新の変更を含むフォークを安全にクローンできるようにするために重要です。

`main` ブランチが同期できたら、その状態から新しいブランチを作成して、元のコードベースの最新バージョンで別の貢献作業を始められます！
