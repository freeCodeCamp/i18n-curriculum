# --description--

你在之前的课程中首次接触了断言和单元测试。但作为复习，单元测试是指你独立查看各个函数或组件的输出，而不是测试整个程序的最终输出。断言用于测试代码是否按预期运行。

在本课中，我们将更深入地了解单元测试中的断言，并探讨一些最佳实践。

一个常用的断言库是 Chai。这是 freeCodeCamp 代码库中使用的相同库。每当你完成一个研讨会或实验时，都会有一个自动化测试套件对你的代码进行运行，以确保你的代码表现符合预期。

让我们来看几个来自 Full-Stack Developer Curriculum 的示例。

在 Build a Calculator 研讨会中，第 1 步是测试一个名为 `addTwoAndSeven` 的函数：

```js
assert.isFunction(addTwoAndSeven);
```

该断言检查所提供的代码是否包含名为 `addTwoAndSeven` 的函数。如果用户创建一个数组、字串或任何其他非函数的内容并赋值给 `addTwoAndSeven`，那么测试将失败。

让我们来看一下研讨会后期的另一个断言：

```js
assert.equal(addThreeAndFour(), 7);
```

此断言查看 `addThreeAndFour` 函数的返回值是否相等于数字 `7`。如果返回值不相等于 `7`，则测试将失败。

Chai 内置了许多断言，比如查看一个值是否是布尔值，查看一个对象是否有某个属性名，或者查看一个值是否为 `null`。其他常见的 JavaScript 断言库有 should.js 和 expect.js。

无论你使用哪个断言库，都有一些最佳实践需要注意。首先是编写清晰的断言消息。清晰且有意义的断言消息应描述其背后的意图。否则，将很难定位哪些测试失败。

另一个最佳实践是编写描述性的失败消息。描述性的失败消息应说明哪个条件为真。这将帮助你弄清楚为什么特定的断言失败。

随着你继续学习编写代码，熟悉 `testing` 并开始为你的个人 `project` 编写测试非常重要。这样做将极大地帮助你为你的第一个开发者工作做好准备。
