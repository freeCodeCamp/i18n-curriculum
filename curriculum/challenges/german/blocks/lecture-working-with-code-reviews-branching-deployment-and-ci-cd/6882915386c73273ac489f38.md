# --description--

Wir haben in den vorherigen Lektionen bereits ein wenig über Branches gesprochen. Sie haben gelernt, wie man Branches erstellt, warum Sie nicht direkt in `main` committen sollten und Ähnliches. Aber wann und warum sollten Sie eigentlich Branches erstellen?

Zuerst sprechen wir darüber, warum Sie nicht einfach eine Menge Commits direkt in `main` pushen sollten. Ihr Standard-Branch sollte zuverlässig stabil sein – das heißt, zu jedem Zeitpunkt sollte ich in der Lage sein, Ihren aktuellen `main`-Branch herunterzuladen, Ihre Anwendung zu bauen, auszuführen und zu testen. Wenn Sie unvollständige Funktionen oder WIP-Commits pushen, ist das nicht immer gewährleistet.

Außerdem ist es sehr üblich, an mehreren Funktionen gleichzeitig zu arbeiten. Wenn Sie teilweise Arbeit für mehrere Funktionen gleichzeitig in `main` pushen, wird es viel schwieriger, ein mögliches Problem zu debuggen, da Sie keine isolierten Änderungen haben. Außerdem ist der Kontextwechsel (also das Wechseln zwischen verschiedenen Aufgaben) nicht so sauber.

Wann sollten Sie also einen Branch erstellen? Eine Faustregel ist, für jede isolierte Arbeitseinheit einen Branch zu erstellen. Wenn Sie also an einer neuen OAuth-Funktion arbeiten, sollte diese Arbeit in einem eigenen Branch erfolgen. Wenn Sie dann einen Bug in Ihrer Passwort-Authentifizierung beheben müssen, sollte diese Arbeit in einem eigenen Branch stattfinden. Müssen Sie die Benutzerprofilseite neu gestalten? Das ist ein weiterer Branch.

Beachten Sie, dass Sie Ihre Branches immer von `main` aus erstellen sollten. Wenn Sie also an einem Branch für Ihre OAuth-Funktion arbeiten, würden Sie `main` auschecken, bevor Sie einen neuen Branch für die Passwort-Fehlerbehebung erstellen. Wenn Sie stattdessen den Branch für die Fehlerbehebung direkt von Ihrem OAuth-Branch aus erstellen, würden die Commits Ihrer OAuth-Arbeit mit übernommen und die Historie wird unübersichtlich.

Aber wie sollten Sie Ihre Branches benennen? Das liegt ganz bei Ihnen (oder dem Projekt, zu dem Sie beitragen), aber hier sind einige gängige Konventionen:

`scope/description` – wobei `scope` etwas wie `feat` oder `fix` ist und `description` eine sehr kurze Erklärung der Änderungen im Branch. Zum Beispiel: `feat/oauth-support`.

`scope/issue/description` – ähnlich wie die vorherige Konvention, aber mit der Nummer des zugehörigen Issues. Zum Beispiel: `fix/25/password-auth`.

`username/scope/description` – Für Projekte, bei denen alle direkt an Branches im Original-Repository arbeiten, anstatt es zu forken (was bei privaten internen Repositories üblich ist), kann es hilfreich sein, den Benutzernamen am Anfang des Branches einzufügen. Zum Beispiel: `naomi-lgbt/feat/profile-redesign`.

Kommen wir zu ein paar weiteren Hinweisen zu Branching-Strategien.

Für lang andauernde Feature-Arbeiten, wie eine Beta-Funktion, die viele Entwicklerstunden erfordert, möchten Sie vielleicht eine Feature-Flag verwenden, um die Logik selektiv zu aktivieren, anstatt einen Branch zu nutzen. Je länger ein Branch ungemerged bleibt, desto wahrscheinlicher sind Konflikte.

Sie möchten jedoch möglicherweise dedizierte Branches für Deployment-Umgebungen oder Releases pflegen. Zum Beispiel verwendet das freeCodeCamp-Repository `prod-staging` und `prod-current`, um Deployments auf Staging- und Produktionsservern auszulösen. Das ermöglicht dem Team, schneller zu iterieren, indem es zusammenführt, sobald Dinge fertig und genehmigt sind, ohne für jeden Merge ein Deployment auslösen zu müssen. Stattdessen werden Deployments durch das Zusammenführen des neuesten `main` in diese Produktions-Branches erstellt.

Sie möchten auch „Release-Branches“ pflegen, also Branches, die eine Hauptversion Ihres Produkts darstellen. So haben Sie für jede Hauptversion einen sauberen Zustand, und wenn Sie einen kritischen Sicherheitspatch in eine frühere Release-Version zurückportieren müssen, können Sie das tun, ohne Änderungen der neuen Version einzubeziehen.

Und schließlich sollten Sie vermeiden, Ihre Branches direkt mit der lokalen CLI in `main` zusammenzuführen. Erstellen Sie stattdessen einen Pull Request mit Ihren Änderungen und führen Sie den Merge über diesen Ablauf durch. So haben Sie die Möglichkeit, sicherzustellen, dass alle Tests in Ihrer CI (die Sie in einer zukünftigen Lektion kennenlernen) bestanden werden, und können die Diff-Ansicht überprüfen, um sicherzugehen, dass nur die gewünschten Änderungen zusammengeführt werden.

Mit diesen Tipps können Sie erfolgreich mit Branches arbeiten!
