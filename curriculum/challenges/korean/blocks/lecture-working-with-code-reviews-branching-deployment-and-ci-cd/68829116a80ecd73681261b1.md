# --description--

분기(branch)를 사용해 격리된 변경 작업을 하여 `main` 브랜치를 더럽히지 않는 방법에 대해 조금 이야기했습니다. 그런데 이게 무슨 뜻이고 왜 중요한 걸까요?

만약 `main` 브랜치에 직접 변경 사항을 커밋하고, 원격 저장소(upstream repository)에서 당신 것이 아닌 다른 변경 사항을 받아들인다면, "분기 이력(divergent history)"이라는 문제가 발생합니다. 이는 `main` 브랜치의 커밋들이 원격 저장소의 `main` 브랜치 커밋들과 더 이상 일치하지 않는다는 뜻입니다. 이렇게 되면 원격 저장소 변경 사항을 동기화하는 일이 매우 어려워집니다.

그렇다면 좋은 분기 관리 방식을 잘 따라왔다고 가정해 봅시다. 이전 수업을 따라왔다면, 로컬에 복제한(fork) 저장소가 있고, 원본 저장소를 가리키는 upstream 원격 저장소가 있을 것입니다.

만약 아직 설정하지 않았다면, 간단히 다시 알려 드리겠습니다. 저장소를 포크하고, 포크한 저장소를 로컬에 복제한 뒤, 다음 명령어를 실행하세요:

```sh
git remote add upstream <url>
```

`<url>` 부분에는 포크가 아닌 원본 저장소의 SSH URL을 넣으세요.

upstream이 제대로 설정되면, 다음 명령어들을 실행해 최신 원본 저장소 변경 사항을 가져올 수 있습니다:

```sh
git checkout main
git fetch upstream
git merge upstream/main
```

이 명령어들은 무엇을 할까요? `git checkout`에 대해 배웠으니, 첫 번째 명령어는 로컬에서 `main` 브랜치에 있는지 확인합니다.

두 번째 명령어인 `git fetch upstream`는 Git에게 원본 저장소(upstream remote)에서 최신 변경 사항을 가져오라고 지시합니다. 하지만 git pull과 달리, Git은 자동으로 브랜치를 업데이트하지 않습니다. 이 점이 중요합니다. 로컬 `main` 브랜치는 포크의 `main` 브랜치와 연결되어 있고, 원본 저장소의 `main` 브랜치와 연결되어 있지 않기 때문입니다! git pull을 하면 예상치 못한 동작이 발생할 수 있습니다.

마지막 명령어인 `git merge upstream/main`는 Git에게 upstream 원격 저장소의 `main` 브랜치에서 최신 변경 사항을 현재 브랜치(우리가 체크아웃한 로컬 `main` 브랜치)로 병합하라고 지시합니다.

그런데 만약 병합이 실패하면 어떻게 할까요? 실수로 `main` 브랜치에 커밋했고 Git이 분기된 이력을 병합할 수 없다면? 만약 `main` 브랜치에서 풀 리퀘스트를 만들지 않았다면, 강제 리셋(hard reset)을 사용해 Git이 브랜치를 업데이트하도록 할 수 있습니다.

```sh
git reset --hard upstream/main
```

`git reset` 명령어는 브랜치의 현재 상태를 리셋할 수 있게 해줍니다. `--hard` 플래그를 주면 Git이 로컬 파일을 브랜치 상태와 강제로 일치시키도록 합니다. 이렇게 하면 작업할 깨끗한 상태가 보장됩니다. 그리고 `upstream/main` 플래그는 현재 브랜치(`main`)를 upstream 원격 저장소의 `main` 브랜치와 일치시키도록 합니다. 이 명령어는 주의해서 사용하세요. Git 이력을 강제로 업데이트하면 이상하고 불행한 일이 발생할 수 있습니다.

로컬 `main` 브랜치를 올바르게 업데이트했다면, 이 상태를 원격 포크(수업을 따라왔다면 `origin`일 것입니다)에 푸시하는 것이 중요합니다. 이렇게 하려면 다음 명령어를 실행하세요:

```sh
git push
```

이 명령어는 현재 상태를 원격 포크에 푸시합니다. 하지만 강제 리셋으로 이력을 정리했다면, 원격 포크에도 분기된 이력이 있을 수 있어 푸시가 실패할 수 있습니다.

이럴 때는 강제 푸시를 사용해 문제를 해결할 수 있습니다:

```sh
git push --force
```

**이 명령어는 매우 조심해서 사용하세요**. 강제 푸시는 원격 이력을 사실상 다시 쓰는 것이며, 잘못 사용하면 풀기 어려운 골칫거리가 됩니다.

동기화된 변경 사항을 원격 포크에 푸시하는 것은, 기존 로컬 저장소를 잃었을 때 최신 변경 사항이 반영된 포크를 안전하게 복제할 수 있도록 도와주기 때문에 중요합니다.

`main` 브랜치가 동기화되면, 그 상태에서 새 브랜치를 만들어 최신 원본 코드베이스를 기반으로 다른 기여 작업을 시작할 수 있습니다!
