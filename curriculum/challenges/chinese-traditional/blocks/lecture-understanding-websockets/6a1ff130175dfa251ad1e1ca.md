# --description--

Pub/Sub 代表 **發布/訂閱**。

它是一種訊息架構，允許系統中不同的部分彼此通訊，而不需要直接知道對方。

乍看之下，這聽起來可能很複雜，但你已經在日常生活中見過類似的系統。

想想 YouTube。

當創作者／頻道管理員上傳新影片時，他們不會直接把影片傳給每個訂閱者。相反地，他們會將影片發布到自己的頻道。任何訂閱該頻道的人都會收到新影片上傳的通知。

創作者不需要知道訂閱者是誰，訂閱者也不需要直接聯絡創作者。平台會處理中間的所有流程。

Pub/Sub 的運作方式類似。主要有三個部分：

- 發布者
- 主題（或頻道，或群組，就像 WhatsApp 上的）
- 訂閱者

**發布者** 發送訊息。**訂閱者** 接收訊息。**主題** 則是它們之間的中介。

流程如下：

1. 訂閱者訂閱一個主題。
2. 發布者向該主題發送訊息。
3. 訊息系統將訊息傳送給該主題的每個訂閱者。

舉例來說，想像一個體育新聞應用程式。使用者可以訂閱一個主題或他們喜歡的球隊，比如 `football-updates` 或 `chelsea fc`。

每當比賽發生事件，例如進球或紅牌，系統就會向該主題發布訊息。當切爾西足球俱樂部進球或失球時，所有訂閱 `chelsea fc` 的使用者會立即收到更新。

發布者不需要知道訂閱者是誰，它只需要知道要將訊息發布到哪裡。

這種分離是 Pub/Sub 架構最大的優點之一。沒有 Pub/Sub 時，系統通常直接通訊。流程可能是這樣：

`Application A → Application B`

這樣運作沒錯，但會造成應用程式間不必要的相依性。如果應用程式 B 改變或無法使用，應用程式 A 可能就無法正常運作。

使用 Pub/Sub，流程變成：

`Application A → Topic → Subscribers`

現在應用程式 A 只發布訊息，不需要知道誰會接收。

另一個例子是有多個聊天室或群組的聊天應用程式。Room A 的使用者訂閱 `room-a`，Room B 的使用者訂閱 `room-b`。

當有人在 Room A 發送訊息時，應用程式會將該訊息發布到 `room-a`。只有訂閱 `room-a` 的使用者會收到訊息，Room B 的使用者則不會看到。

這樣系統更容易維護、擴充和擴展。

這也是為什麼 Pub/Sub 常出現在聊天應用程式、多玩家遊戲、通知系統、即時儀表板和事件驅動應用程式中的原因之一。

同時也要理解 Pub/Sub 和 WebSockets 並不相同。WebSockets 提供通訊通道，讓資料能在客戶端和伺服端之間即時傳輸。Pub/Sub 則提供訊息樣式，決定誰應該接收特定訊息。

你可以這麼想：

- WebSockets 是道路。
- Pub/Sub 是決定車輛行駛路線的交通系統。

許多現代即時應用程式會同時使用這兩種技術。WebSockets 保持連線開啟，而 Pub/Sub 確保訊息送達正確的訂閱者。

Pub/Sub 的核心概念是發布者將訊息發送到主題，訂閱者監聽主題。訊息系統負責在它們之間傳遞訊息。
