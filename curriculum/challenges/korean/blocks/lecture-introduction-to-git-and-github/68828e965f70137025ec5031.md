# --description--

논문과 같이 중요한 변경을 한 후에 `Thesis-Draft`, `Thesis-Final`, `Thesis-Final-For-Real`, `Thesis-Absolute-Final-DO-NOT-CHANGE` 같은 이름으로 파일을 저장한 적이 있다면, 아주 기본적인 버전 관리 방식을 사용한 것입니다. 그런데 더 나은 방법이 있다고 하면 어떨까요?

Git, SVN, Mercurial 같은 도구를 들어본 적이 있을 겁니다. 이들은 버전 관리 시스템이며, 모든 개발자의 작업 흐름에서 핵심적인 컴포넌트입니다. 버전 관리 시스템은 프로젝트의 변경 사항을 추적하고 관리할 수 있게 해줍니다. 이 수업에서는 Git에 집중할 것입니다.

Git은 프로젝트의 편집 기록을 관리할 수 있는 강력한 도구를 제공합니다. 앞으로 수업에서 Git 사용법과 제공하는 기능을 더 배우게 될 것입니다. 지금은 꼭 알아야 할 내용을 알려드리겠습니다.

Git은 커밋을 만들 수 있게 해주는데, 커밋은 코드베이스의 특정 상태를 스냅샷으로 저장한 것입니다. 그런 다음 커밋을 비교해 어떤 변경이 있었는지 확인하거나, 잘못된 변경이 포함된 커밋을 되돌리는 등의 작업을 할 수 있습니다.

또한 "브랜치"를 만들 수 있는데, 브랜치는 코드에서 서로 다른 경로라고 생각하면 됩니다. 브랜치는 당신이 걷는 길이라면, 커밋은 그 길을 따라 있는 이정표와 같습니다. 브랜치를 사용하면 새로운 기능을 독립적으로 작업할 수 있어서, 작업이 잘못되어도 브랜치를 삭제해 핵심 코드베이스에 영향을 주지 않을 수 있습니다.

이러한 기능 덕분에 버전 관리는 다른 개발자와 협업할 때 필수적인 요소가 됩니다. 여러 사람이 같은 프로젝트에서 작업할 때, 브랜치는 각자가 자신의 코드를 독립적으로 작업할 수 있게 해 서로의 변경 사항이 충돌하지 않도록 합니다.

보통 Git 같은 버전 관리 시스템을 GitHub나 Gitlab 같은 버전 관리 제공자와 함께 사용합니다. 이들은 추가 협업 도구와 프로젝트를 클라우드에 백업할 수 있는 기능을 제공합니다.

하지만 클라우드 기반 저장소가 없어도 버전 관리 시스템은 작업 흐름에 큰 변화를 가져올 수 있습니다. 더 이상 `THIS-IS-THE-FINAL-COPY-FOREVER` 같은 파일을 만들 필요가 없습니다.

이 시스템을 사용하는 데 관한 멋진 정보가 많으니, 다음 수업에서 깊이 있게 다뤄보겠습니다!
