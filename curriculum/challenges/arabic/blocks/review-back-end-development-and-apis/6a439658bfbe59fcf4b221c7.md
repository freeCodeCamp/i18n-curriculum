## العمل مع Node.js والهندسة المعتمدة على الأحداث

- **Node.js**: بيئة تشغيل جافاسكريبت مفتوحة المصدر ومتعددة المنصات تتيح لك تشغيل شفرة جافاسكريبت خارج المتصفح. تُستخدم على نطاق واسع لبناء خوادم الويب، وواجهات برمجة التطبيقات، وأنظمة الواجهة الخلفية.
- **قيود جانب العميل**: جافاسكريبت على جانب العميل لها وصول محدود إلى نظام الملفات، ودعم محدود للمنطق المعقد للتطبيق، ومخاطر أمنية محتملة (مثلًا، يجب ألا تكشف بيانات اعتماد قاعدة البيانات في شفرة جانب العميل).
- **الفرق بين الكائن العالمي في المتصفح وNode.js**: في المتصفح، الكائن العالمي هو `window`، الذي يتيح التلاعب بـ DOM، وملفات تعريف الارتباط، وأحداث المتصفح. في Node، الكائن العالمي هو `global`، الذي يتيح وحدات مدمجة للتعامل مع الملفات، والشبكات، والتفاعل مع نظام التشغيل.
- **الوصول إلى DOM**: المتصفح يوفر وصولًا إلى واجهة برمجة تطبيقات DOM، لكن Node لا يفعل ذلك. بالمقابل، يمكن لـ Node الوصول إلى معظم موارد النظام، بما في ذلك نظام الملفات، بينما المتصفح لا يستطيع.
- **التحكم في الإصدار**: يمكنك اختيار إصدار Node الذي يعمل على الخادم الخاص بك. لا يمكنك التحكم في إصدار المتصفح الذي يستخدمه المستخدمون.
- **الهندسة غير المحجوزة والمعتمدة على الأحداث**: يستخدم Node خيطًا واحدًا وحلقة أحداث تدير بكفاءة العديد من الطلبات والعمليات الإدخالية/الإخراجية المتزامنة دون حجب، مما يجعله مثاليًا للتطبيقات الزمن الحقيقي.
- **الاستدعاءات الراجعة (Callbacks)**: دوال تحدد ما يحدث عند اكتمال عملية غير متزامنة. الاعتماد الكبير عليها يجعل شفرة Node غير المتزامنة أصعب في القراءة وتصحيح الأخطاء.
- **مزايا Node في الواجهة الخلفية**: لغة واحدة (جافاسكريبت) عبر الواجهة الأمامية والخلفية، هندسة غير محجوزة مناسبة للتعامل مع عدد كبير من الاتصالات، نظام بيئي ضخم لـ npm، مجاني ومفتوح المصدر.
- **عيوب Node في الواجهة الخلفية**: خيط واحد، لذا المهام المكثفة لوحدة المعالجة المركزية (معالجة الصور، التشفير، الرياضيات المعقدة) قد تحجب حلقة الأحداث، الاعتماد الكبير على الشفرة غير المتزامنة/الاستدعاءات الراجعة قد يقلل من قابلية القراءة، حزم npm تختلف في الجودة والصيانة.
- **NVM (مدير إصدارات Node)**: أداة لإدارة والتبديل بين إصدارات متعددة من Node.js على نفس الجهاز. على macOS/Linux يتم التثبيت باستخدام:

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **تثبيت إصدار LTS من Node**: `nvm install --lts`
- **التبديل بين إصدارات Node**: `nvm use 20`
- **عرض جميع إصدارات Node المثبتة**: `nvm ls`
- **تعيين إصدار Node افتراضي (macOS/Linux)**: `nvm alias default 22.20.0`
- **nvm-windows**: مشروع منفصل لإدارة إصدارات Node على ويندوز (ليس WSL). حمّل المثبت من المستودع الرسمي وشغّل `nvm install lts` ثم `nvm use <version>`.
- **تشغيل ملف جافاسكريبت باستخدام Node**: `node app.js`

## العمل مع وحدات Node الأساسية

### وحدة `fs`

- **الغرض**: توفر طرقًا لإنشاء وقراءة وكتابة وإضافة وحذف الملفات والمجلدات.
- **ثلاثة أنماط استخدام**: مع استدعاءات راجعة (غير متزامن)، مع وعود (`fs/promises`)، ومتزامن (لاحقة `Sync`). الطرق المتزامنة تحجب حلقة الأحداث ومناسبة فقط للسكريبتات.
- **`writeFile()`**: يكتب محتوى في ملف، وينشئه إذا لم يكن موجودًا:

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**: يضيف محتوى إلى ملف موجود دون استبداله.
- **`readFile()`**: يقرأ محتوى الملف. مرر `"utf8"` كترميز للحصول على سلسلة نصية، أو احذفه للحصول على `Buffer`.
- **`unlink()`**: يحذف ملفًا. يرمي `ENOENT` إذا لم يكن الملف موجودًا.

### وحدة `Buffer`

- **الغرض**: تتعامل مع البيانات الثنائية (الملفات، الصور، تدفقات الشبكة) مباشرة في الذاكرة.
- **`Buffer.from()`**: ينشئ مخزنًا مؤقتًا من سلسلة نصية:

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**: ينشئ مخزنًا مؤقتًا مملوءًا بالأصفار بحجم `n` بايت:

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **الاقتطاع**: كتابة بيانات أكثر مما يمكن للمخزن المؤقت استيعابه تقطع الزائد بصمت.
- **`Buffer.byteLength(str)`**: يعيد عدد البايتات اللازمة لتخزين سلسلة نصية.

### وحدة `crypto`

- **الغرض**: توفر أدوات تشفير مثل التجزئة، HMAC، التشفير/فك التشفير، التوقيعات الرقمية، وتوليد أرقام عشوائية آمنة.
- **`createHash(algorithm)`**: ينشئ تجزئة أحادية الاتجاه (غير قابلة للعكس). استخدم `.update(data).digest("hex")` لتزويد البيانات بالتجزئة وإرجاعها كسلسلة سداسية عشرية:

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**: مثل `createHash` لكنه يتطلب مفتاحًا سريًا، مناسب للتحقق من سلامة البيانات والتوثيق.
- **`createCipheriv()` / `createDecipheriv()`**: تشفير وفك تشفير البيانات. كلاهما يتطلب خوارزمية، مفتاح، ومتجه تهيئة (IV):

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**: يولد بايتات عشوائية آمنة تشفيرياً، أكثر أمانًا من `Math.random()` للرموز والمفاتيح.
- **`randomInt(min, max)`**: يولد عددًا صحيحًا عشوائيًا آمنًا ضمن النطاق المحدد. مفيد لأكواد التحقق (OTP).
- **`sign()` / `verify()`**: إنشاء والتحقق من التوقيعات الرقمية باستخدام زوج مفاتيح خاص/عام.

### وحدة `os`

- **الغرض**: تعرض معلومات عن نظام التشغيل الذي يعمل عليه Node.
- **`os.platform()`**: يعيد معرف نظام التشغيل، مثل `'darwin'`، `'win32'`، `'linux'`.
- **`os.arch()`**: يعيد بنية وحدة المعالجة المركزية، مثل `'x64'`، `'arm64'`.
- **`os.type()`**: يعيد الاسم الرسمي لنظام التشغيل، مثل `'Darwin'`.
- **`os.release()`**: يعيد سلسلة إصدار نواة نظام التشغيل.
- **`os.cpus()`**: يعيد مصفوفة من الكائنات تصف كل نواة منطقية لوحدة المعالجة المركزية (النموذج، السرعة، الأوقات).
- **`os.uptime()`**: يعيد مدة تشغيل النظام بالثواني.
- **`os.totalmem()` / `os.freemem()`**: يعيدان إجمالي وذاكرة النظام المتاحة بالبايت.
- **`os.userInfo()`**: يعيد كائنًا يحتوي على `uid`، `gid`، `username`، `homedir`، و`shell` للمستخدم الحالي.
- **`os.networkInterfaces()`**: يعيد كائنات واجهات الشبكة مع تفاصيل العنوان، العائلة، MAC، وCIDR.

### وحدة `path`

- **الغرض**: توفر أدوات للعمل مع مسارات الملفات والمجلدات بطريقة مستقلة عن النظام.
- **`__filename`**: متغير عالمي، المسار المطلق للملف الحالي.
- **`__dirname`**: متغير عالمي، المسار المطلق للمجلد الذي يحتوي الملف الحالي.
- **`path.basename(p)`**: يعيد آخر مكون من المسار (اسم الملف).
- **`path.dirname(p)`**: يعيد جزء المجلد من المسار.
- **`path.extname(p)`**: يعيد امتداد الملف، مثل `'.js'`.
- **`path.join(...segments)`**: يدمج أجزاء المسار في مسار موحد ويصحح الشرط المائلة الخاطئة.
- **`path.resolve(...segments)`**: يحل الأجزاء إلى مسار مطلق يبدأ من دليل العمل الحالي.
- **`path.parse(p)`**: يعيد كائنًا يحتوي على خصائص `root`، `dir`، `base`، `ext`، و`name`.
- **`path.format(obj)`**: يبني سلسلة مسار من كائن يحتوي على `dir`، `name`، و`ext`.

### وحدة `process`

- **الغرض**: تعرض معلومات وتحكم في عملية Node.js الحالية. هو كائن عالمي، لذا لا حاجة للاستيراد.
- **`process.env`**: كائن يحتوي على جميع متغيرات البيئة. يمكن الوصول إلى قيم محددة مثل `process.env.NODE_ENV`.
- **`process.argv`**: مصفوفة من معطيات سطر الأوامر. الفهرس 0 هو تنفيذ Node، الفهرس 1 هو مسار السكريبت، الفهارس 2+ هي معطيات المستخدم.
- **`process.cwd()`**: يعيد دليل العمل الحالي.
- **أحداث العملية**:

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**: يطلق تحذيرًا مخصصًا يمكن التقاطه بواسطة مستمع `warning`.

### وحدة `stream`

- **الغرض**: تعالج كميات كبيرة من البيانات على أجزاء بدلاً من تحميل كل شيء في الذاكرة دفعة واحدة.
- **أنواع التدفقات الأربعة**: قابلة للقراءة (قراءة البيانات على أجزاء)، قابلة للكتابة (كتابة البيانات على أجزاء)، ثنائية الاتجاه (كلاهما)، تحويل (ثنائية الاتجاه تعدل البيانات أثناء مرورها).
- **توصيل التدفقات (Piping)**:

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **حدث `finish`**: يُطلق على تدفق قابل للكتابة عندما يتم تفريغ كل البيانات.
- **حدث `error`**: يُطلق عندما تواجه التدفق مشكلة.

## مقدمة إلى npm

- **npm**: يتألف من ثلاثة أجزاء: موقع الويب `npmjs.com`، السجل العام (قاعدة بيانات الحزم)، وأداة واجهة سطر الأوامر المثبتة مع Node.js.
- **الغرض**: يتيح لك تثبيت وإدارة ونشر حزم جافاسكريبت قابلة لإعادة الاستخدام. npm هو أكثر مديري الحزم استخدامًا.
- **البدائل**: Yarn، PNPM، Bun.

### ملف `package.json`

- **الغرض**: ملف تكوين npm. يتتبع اسم المشروع، الإصدار، الوصف، نقطة الدخول (`main`)، المؤلف، الترخيص، السكريبتات، والتبعيات.
- **إنشاء `package.json` تفاعليًا**: `npm init`
- **إنشاء بالإعدادات الافتراضية**: `npm init -y`
- **`dependencies`**: الحزم المطلوبة لتشغيل التطبيق في الإنتاج (مثل `express`، `react`).
- **`devDependencies`**: الحزم المطلوبة فقط أثناء التطوير والاختبار (مثل `jest`، `nodemon`، `eslint`).
- **`license`**: الشروط القانونية للاستخدام. القيم الشائعة: `MIT`، `ISC`، `GPL`.

### الإصدار الدلالي (Semantic Versioning - SemVer)

- **الصيغة**: `MAJOR.MINOR.PATCH`، مثلًا `4.17.21`.
- **MAJOR**: يُزاد عند تغييرات غير متوافقة.
- **MINOR**: يُزاد عند ميزات جديدة متوافقة مع الإصدارات السابقة.
- **PATCH**: يُزاد عند إصلاحات أخطاء متوافقة مع الإصدارات السابقة.
- **علامة Caret `^`**: تسمح بتحديثات للإصدار الثانوي والفرعي لكن ليس الرئيسي. مثال: `^4.17.21` يقبل `4.18.0` لكن ليس `5.0.0`.
- **علامة Tilde `~`**: تسمح بتحديثات للإصدار الفرعي فقط. مثال: `~1.2.3` يقبل `1.2.4` لكن ليس `1.3.0`.
- **علامة النجمة `*`**: تطابق أي إصدار، مناسبة للاختبار فقط، ليست للإنتاج.
- **الإصدار الدقيق**: بدون بادئة، مثل `1.2.3` يثبت على هذا الإصدار بالضبط.

### تثبيت وإزالة التبعيات

- **تثبيت حزمة**: `npm install express` (اختصار: `npm i express`)
- **تثبيت كـ devDependency**: `npm install nodemon -D`
- **إلغاء تثبيت حزمة**: `npm uninstall chalk`
- **`node_modules/`**: مجلد يُنشأ بعد التثبيت يحتوي على شفرة الحزمة الفعلية. لا يُضاف إلى التحكم في الإصدار.
- **`package-lock.json`**: ملف يُولد تلقائيًا يقفل الإصدارات الدقيقة لكل حزمة مثبتة (بما في ذلك التبعيات الفرعية). يجب دائمًا الالتزام به وعدم تعديله يدويًا.
- **تثبيت إصدار محدد**: `npm install express@4.21.2`
- **عرض الإصدارات المتاحة**: `npm view express versions`
- **التحقق من الحزم القديمة**: `npm outdated`
- **تحديث حزمة واحدة** (ضمن النطاق المسموح): `npm update express`
- **فرض تثبيت أحدث إصدار**: `npm install express@latest`

### النشر إلى سجل npm

- **تسجيل الدخول**: `npm login`
- **الحزم ذات النطاق (Scoped packages)**: استخدم `@username/package-name` لتجنب تعارض الأسماء.
- **`.npmignore`**: يسرد الملفات التي يجب استثناؤها من الحزمة المنشورة (مماثل لـ `.gitignore`).
- **نشر حزمة غير ذات نطاق**: `npm publish`
- **نشر حزمة ذات نطاق بشكل عام**: `npm publish --access public`

## العمل مع سكريبتات npm

- **سكريبتات npm**: أوامر مخصصة تُعرّف في كائن `"scripts"` داخل `package.json`. تُستخدم لأتمتة المهام المتكررة مثل بدء الخادم، تشغيل الاختبارات، التدقيق، والبناء.
- **تعريف سكريبت**:

```json
"scripts": {
  "start": "node app.js"
}
```

- **تشغيل سكريبت**: `npm run start`
- **اختصار لأسماء السكريبتات المدمجة** (`start`، `test`، `stop`، `restart`): `npm start`
- **أوامر متسلسلة باستخدام `&&`** (يُشغل الأمر التالي فقط إذا نجح السابق):

```json
"start": "npm run build && node server.js"
```

- **أوامر متسلسلة باستخدام `;`** (يُشغل الأمر التالي دائمًا):

```json
"start": "npm run build; node server.js"
```

- **أوامر متزامنة باستخدام `&`**: `"dev": "npm run start:server & npm run start:client"`
- **تمرير معلمات إلى السكريبت** (استخدم الفاصل `--`): `npm start -- 8000`؛ المعلمات متاحة عبر `process.argv[2]`.

### وحدات CommonJS

- **CommonJS**: نظام الوحدات الأصلي في Node.js. يحمل الوحدات بشكل متزامن.
- **الاستيراد** باستخدام `require()`:

```js
const { multiply } = require('./math');
```

- **التصدير** باستخدام `module.exports`:

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- CommonJS هو النظام الافتراضي عندما يحتوي `package.json` على `"type": "commonjs"` (أو لا يحتوي على حقل `"type"`)، أو عندما يكون امتداد الملف `.cjs`.

### وحدات ES (ESM)

- **وحدات ES (ES Modules)**: صيغة الوحدات الحديثة والمعيارية. تُحمّل بشكل غير متزامن.
- **تصدير مسمى**: `export function multiply(a, b) { return a * b; }`
- **تصدير افتراضي**: `export default function multiply(a, b) { return a * b; }`
- **الاستيراد**: `import { multiply } from './math.mjs';`
- تُستخدم وحدات ES عندما يكون امتداد الملف `.mjs` أو يحتوي `package.json` على `"type": "module"`.

## فهم كيفية عمل HTTP وDNS وTCP/IP

### الخوادم ونموذج العميل-الخادم

- **نموذج العميل-الخادم**: هيكلية نظام موزع حيث تكون المكونات إما عملاء (عمليات ترسل طلبات، مثل متصفح الويب) أو خوادم.
- **الخادم**: جهاز كمبيوتر أو برنامج يستقبل الطلبات من العملاء ويرد بالبيانات أو الخدمات عبر الشبكة.
- **أنواع الخوادم**: خوادم الويب (HTML/CSS/الموارد)، خوادم التطبيقات (المنطق التجاري)، خوادم قواعد البيانات، خوادم البريد، خوادم الملفات، خوادم البروكسي.

### نظام أسماء النطاقات (DNS)

- **DNS (نظام أسماء النطاقات)**، يُطلق عليه غالبًا "دليل الهاتف للإنترنت": يترجم أسماء النطاقات القابلة للقراءة البشرية (مثل `freecodecamp.org`) إلى عناوين IP.
- **هيكلية أسماء النطاقات** (من اليمين إلى اليسار): النطاق الأعلى (TLD، مثل `.org`) → النطاق الثاني (مثل `freecodecamp`) → النطاق الفرعي (مثل `www`).
- **عنوان IP**: معرف رقمي فريد لجهاز على الشبكة. مثال IPv4: `127.0.0.1`. مثال IPv6: `0:0:0:0:0:0:0:1`.
- **عملية حل DNS**: المتصفح → محلل متكرر → خوادم الجذر → خادم TLD → خادم الاسم الموثوق → يعيد عنوان IP → المحلل المتكرر يخزن مؤقتًا ويرسله إلى العميل.

### TCP/IP وHTTP

- **IP (بروتوكول الإنترنت)**: يوجه حزم البيانات من المصدر إلى الوجهة باستخدام عناوين IP. اتصال غير متصل، تُرسل كل حزمة بشكل مستقل.
- **TCP (بروتوكول التحكم في الإرسال)**: يقسم البيانات إلى أجزاء مرقمة، يعيد تجميعها في الوجهة، يتحقق من الأخطاء، ويطلب إعادة الإرسال إذا لزم الأمر. يضمن تسليمًا موثوقًا ومرتبًا.
- **HTTP (بروتوكول نقل النص الفائق)**: بروتوكول طبقة التطبيقات يحدد كيفية تبادل المحتوى بين العملاء والخوادم.
- **HTTPS**: نسخة آمنة من HTTP حيث تُشفّر البيانات باستخدام SSL/TLS.

### هيكل طلب HTTP

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **سطر الطلب**: طريقة HTTP + المسار/URI + إصدار HTTP.
- **الرؤوس**: أزواج بيانات وصفية من المفتاح والقيمة.
- **الجسم** (اختياري): يُرسل مع `POST`، `PUT`، `PATCH`، `DELETE` لحمل البيانات.

### هيكل استجابة HTTP

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **سطر الحالة**: إصدار HTTP + رمز الحالة + عبارة السبب.
- **الرؤوس**: بيانات وصفية عن الاستجابة (نوع المحتوى، الطول، توجيهات التخزين المؤقت).
- **الجسم** (اختياري): البيانات المطلوبة.

### فئات رموز حالة HTTP

- **1xx معلوماتية**: تم استلام الطلب، المعالجة مستمرة. مثال: `101 Switching Protocols`.
- **2xx نجاح**: `200 OK`، `201 Created`، `204 No Content`.
- **3xx إعادة توجيه**: `301 Moved Permanently`، `302 Found`، `304 Not Modified`.
- **4xx أخطاء العميل**: `400 Bad Request`، `401 Unauthorized`، `403 Forbidden`، `404 Not Found`.
- **5xx أخطاء الخادم**: `500 Internal Server Error`، `502 Bad Gateway`، `503 Service Unavailable`.

## فهم نموذج طلب-استجابة HTTP

- **طرق HTTP**: `GET` (استرجاع)، `POST` (إنشاء)، `PUT` (تحديث كامل)، `PATCH` (تحديث جزئي)، `DELETE` (حذف).
- **دورة الطلب-الاستجابة**: يرسل العميل طلبًا → يتحقق الخادم من المصادقة، يراجع الطريقة وURL، يستعلم قاعدة البيانات → يبني الخادم الاستجابة (رمز الحالة + جسم اختياري) → يعالج العميل الاستجابة.

### الموارد المعادة في استجابات HTTP

كل استجابة تتضمن رأس `Content-Type` يخبر المتصفح بكيفية التعامل مع المورد:

- **HTML**: `text/html`
- **CSS**: `text/css`
- **JavaScript**: `application/javascript`
- **JSON**: `application/json`
- **PNG/JPEG/WebP**: `image/png`, `image/jpeg`, `image/webp`
- **MP4/MP3**: `video/mp4`, `audio/mpeg`
- **PDF**: `application/pdf`
- **WOFF2**: `font/woff2`

### إرسال نماذج HTML

- **خاصية `action`**: تحدد عنوان URL الذي تُرسل إليه بيانات النموذج.
- **خاصية `method`**: تحدد طريقة HTTP (`GET` أو `POST`). HTML لا يدعم أصلاً `PUT`، `PATCH`، أو `DELETE`، استخدم Fetch API أو حزمة `method-override` في Express بدلاً من ذلك.
- **التحقق من صحة جهة العميل**: استخدم خاصيتي `required` و`pattern` لتوفير ردود فورية. تحقق دائمًا على الخادم أيضًا.
- **نموذج `GET`**: تُضاف البيانات إلى URL كمعلمات استعلام (مثل نماذج البحث).
- **نموذج `POST`**: تُرسل البيانات في جسم الطلب (مثل نماذج تسجيل الدخول والتسجيل).

## فهم نموذج معايير الويب

### هيئات المعايير

- **WHATWG**: تحافظ على HTML وDOM كمعيار "حي" يتم تحديثه باستمرار. اتخاذ القرار يقوده فرق محركات المتصفحات الكبرى (Chromium، Gecko، WebKit).
- **W3C (كونسورتيوم الويب العالمي)**: الهيئة الأساسية للمعايير. تشرف على CSS، SVG، إرشادات إمكانية الوصول (WCAG)، والعديد من واجهات برمجة التطبيقات للويب.
- **ECMA International / TC39**: تنشر Ecma المواصفة الرسمية لـ ECMAScript. تطور TC39 اللغة عبر عملية اقتراح من خمس مراحل (المرحلة 0 فكرة أولية → المرحلة 4 مواصفة كاملة).
- **مجموعة Khronos**: تركز على الرسومات، الحوسبة، والوسائط. تنتج WebGL وWebGL 2 لعرض ثلاثي الأبعاد معزز بواسطة GPU في المتصفحات.

### إنشاء معايير الويب

1. **تحديد الحاجة**: يواجه مطوّر أو منظمة مشكلة ويكتبها.
2. **الاقتراح والنقاش المبكر**: قضية غير رسمية على GitHub أو شرح يُشارك لقياس الاهتمام. في TC39 هذه المرحلة 0/1.
3. **كتابة المواصفة**: وثيقة رسمية دقيقة تغطي كل الحالات، الأخطاء، والاعتبارات الأمنية.
4. **المراجعة والاعتراض والتكرار**: تُراجع المسودة علنًا من قبل بائعي المتصفحات، الميزات التي يرفض متصفح رئيسي تنفيذها نادرًا ما تصبح معايير.
5. **التنفيذ والاختبار**: ينفذ البائعون خلف علامة ميزة، ويضمن اختبار مشترك سلوكًا متطابقًا عبر المتصفحات.
6. **الإنهاء**: تسميها W3C توصية، وتسميها TC39 المرحلة 4.

### دورة حياة ميزة معايير الويب

1. **الاقتراح**: تحديد المشكلة وكتابتها.
2. **الحضانة**: نقاش مفتوح، تموت العديد من الاقتراحات هنا.
3. **المواصفة**: كتابة المواصفة الرسمية مع قواعد دقيقة.
4. **اختبار التوافقية**: مجموعة اختبارات منصة الويب من W3C تتحقق من التوافق عبر المتصفحات.
5. **الإنهاء**: تُطرح الميزة بدون علامات كجزء من منصة الويب الأساسية.
6. **الإلغاء**: تُعلّم الميزات المستبدلة أو غير الآمنة للحذف، لكن المتصفحات تحتفظ بها للتماشي مع الإصدارات السابقة.

### المبادئ الأساسية لمعايير الويب

- **الانفتاح**: تُطوّر المعايير بشكل مفتوح وتُنفذ بدون رسوم ترخيص.
- **إمكانية الوصول**: مدمجة في HTML وCSS وسلوك المتصفح، توفر WCAG إرشادات رسمية.
- **التوافقية**: يجب أن تتصرف الصفحة المبنية وفق المعيار بشكل متطابق عبر جميع المتصفحات.
- **التوافق مع الإصدارات السابقة**: يجب أن تستمر الصفحات القديمة في العمل على المتصفحات الحديثة.
- **الخصوصية والأمان**: تواجه واجهات برمجة التطبيقات الجديدة التي قد تكشف بيانات المستخدم تدقيقًا صارمًا.
- **التدرج**: HTML للهيكل، CSS للعرض، JavaScript للسلوك. يمكن اعتماد كل طبقة بشكل مستقل.
- **التحسين التدريجي**: يجب أن تعمل الميزات للجميع أولًا، ثم تتحسن للمستخدمين ذوي المتصفحات الأكثر قدرة.

## فهم واجهات برمجة التطبيقات (REST) وخدمات الويب

### خدمات الويب

- **واجهة برمجة التطبيقات (API)**: آلية عامة لتواصل البرمجيات مع بعضها البعض.
- **خدمة الويب**: نوع محدد من واجهات برمجة التطبيقات يستخدم بروتوكولات الويب (HTTP/HTTPS) لتبادل البيانات عبر الشبكة. كل خدمة ويب هي API، لكن ليس كل API هي خدمة ويب.
- **الخصائص**: تعمل على خادم، تتيح التواصل بين الآلات، غير معتمدة على لغة أو منصة، تستخدم بروتوكولات ويب قياسية.
- **REST**: نقل الحالة التمثيلية، أكثر أساليب خدمات الويب استخدامًا. بسيطة ومرنة وتعمل بشكل طبيعي مع HTTP.
- **SOAP**: بروتوكول الوصول البسيط للكائنات، يستخدم XML فقط، شائع في أنظمة المؤسسات التي تحتاج معايير صارمة.
- **XML-RPC**: يستخدم XML لترميز الطلبات والاستجابات الخفيفة.

### هندسة REST

- **عديم الحالة**: يُعالج كل طلب بشكل مستقل. لا يخزن الخادم حالة الجلسة بين الطلبات. يجب أن يتضمن كل طلب كل المعلومات اللازمة (مثل رمز توثيق في كل استدعاء).
- **الموارد**: كل شيء هو مورد يُعرف بواسطة عنوان URL، مثل `/products` لمجموعة، و`/products/1` لعنصر محدد.
- **طرق HTTP في REST**: `GET` (استرجاع)، `POST` (إنشاء)، `PUT`/`PATCH` (تحديث)، `DELETE` (حذف).
- **صيغ الاستجابة**: JSON هي الأكثر شيوعًا، لكن XML والنص العادي مدعومان أيضًا. يحدد العميل الصيغة عبر رأس `Accept`:

```bash
Accept: application/json
```

### الخدمات المصغرة

- **النظام الأحادي (Monolith)**: قاعدة شفرة واحدة كبيرة تضم واجهة المستخدم، وقاعدة البيانات، وواجهة برمجة التطبيقات معًا. يصعب صيانته وتوسيعه مع زيادة التعقيد.
- **الخدمات المصغرة**: نهج معماري يقسم التطبيق الكبير إلى خدمات صغيرة مستقلة، كل منها مسؤول عن وظيفة تجارية واحدة. تتواصل الخدمات عبر واجهات برمجة التطبيقات أو قوائم الرسائل.
- **المزايا**: تطوير وتشغيل أسرع بشكل مستقل، عزل أفضل للأخطاء، يمكن للخدمات استخدام لغات أو أُطُر مختلفة.
- **العيوب**: تصحيح أخطاء أكثر تعقيدًا (المشاكل تمتد عبر الخدمات)، زيادة في البنية التحتية، زيادة في التواصل الشبكي.

## العمل مع Express

- **Express.js**: إطار عمل ويب بسيط ومرن مبني على Node.js. يبسط بناء خوادم الويب وواجهات برمجة التطبيقات مع تقليل القوالب الجاهزة مقارنة بوحدة `http` الأصلية.
- **خادم Express أساسي**:

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**: دالة مصنع تنشئ مثيل تطبيق Express (`app`).
- **`app.listen(port, callback)`**: يبدأ الخادم ويبدأ الاستماع لطلبات HTTP على المنفذ المحدد.
- **`req`**: كائن الطلب، يحتوي على URL، رؤوس، معلمات، جسم، وبيانات الاستعلام.
- **`res`**: كائن الاستجابة، يُستخدم لإرسال البيانات إلى العميل.
- **البرمجيات الوسيطة (Middleware)**: شفرة تُنفذ بين استقبال الطلب وإرسال الاستجابة.

## فهم التوجيه في Express.js

- **التوجيه**: يحدد كيف يستجيب التطبيق عندما يزور العميل عنوان URL معين بطريقة HTTP محددة.
- **`app.get/post/put/delete(path, handler)`**: يسجل معالجات لطرق HTTP المقابلة.
- **مسار ثابت**: نمط URL ثابت، مثل `app.get("/home", handler)`.
- **مسار ديناميكي بمعلمات**: يستخدم عنصر نائب `:paramName`. يتم التقاطه عبر `req.params`:

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**: معلمات المسار (`:id` أجزاء).
- **`req.query`**: معلمات سلسلة الاستعلام (`?key=value`).
- **`req.body`**: جسم الطلب (يتطلب برمجيات وسيطة لتحليل الجسم).
- **`res.send(data)`**: يرسل استجابة (نص، HTML، JSON، أو ثنائي).
- **`res.json(obj)`**: يرسل استجابة JSON، ويضبط تلقائيًا رأس `Content-Type: application/json`.
- **`res.status(code).send(msg)`**: يضبط رمز الحالة ويرسل استجابة.
- **`res.redirect(url)`**: يعيد توجيه العميل إلى عنوان URL آخر.
- **`res.render(view, data)`**: يعرض قالب محرك القوالب ويرسل HTML.

### معالجات المسار القابلة للتسلسل باستخدام `app.route()`

يجمع عدة طرق HTTP لنفس المسار لتقليل التكرار:

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

ينشئ تطبيقًا مصغرًا معياريًا لتجميع المسارات ذات الصلة:

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

### تقديم الملفات الثابتة

- **`express.static(directory)`**: برمجيات وسيطة مدمجة تقدم الملفات من دليل محدد:

```js
app.use(express.static(path.join(__dirname, "public")));
```

- تقديم من عدة أدلة مع مسار تثبيت: `app.use("/images", express.static(path.join(__dirname, "images")));`

## فهم معالجة الأخطاء وفحوصات الصحة

### معالجة الأخطاء في Express

- **برمجيات وسيطة لمعالجة الأخطاء**: تُعرف بأربعة معلمات `(err, req, res, next)`. توضع دائمًا في نهاية سلسلة البرمجيات الوسيطة:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **أخطاء Express 4 غير المتزامنة**: يجب التقاطها يدويًا باستخدام `try/catch` وتمريرها عبر `next(err)`:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **أخطاء Express 5 غير المتزامنة**: تُمرر تلقائيًا إلى برمجيات معالجة الأخطاء، لا حاجة لـ `next(err)` يدويًا.
- **معالج 404**: يوضع بعد كل المسارات وقبل معالج الأخطاء:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### تصحيح الأخطاء وتسجيل الأحداث

- **تصحيح مدمج**: فعّل بتعيين متغير البيئة `DEBUG`: `DEBUG=express:* node index.js`
- **`morgan`**: برمجيات وسيطة شهيرة لتسجيل طلبات HTTP. تسجل الطريقة، URL، رمز الحالة، ووقت الاستجابة:

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### فحوصات الصحة والإغلاق السلس

- **مسار فحص الصحة**: يعيد `200 OK` للإشارة إلى أن التطبيق يعمل. يستخدمه موزعو الحمل ومنظمو الحاويات:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**: إشارة تُرسل عند وجوب إنهاء عملية. عالجها للسماح للطلبات الجارية بالانتهاء:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**: إشارة تُرسل عند الضغط على Ctrl+C أثناء التطوير المحلي. عالجها بنفس طريقة `SIGTERM`.
- **وضع علامة عدم الصحة قبل الإغلاق**: أعد `503` من نقطة فحص الصحة أثناء الإغلاق لمنع موزع الحمل من توجيه حركة مرور جديدة:

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## برمجيات Express الوسيطة

- **البرمجيات الوسيطة**: دالة تُنفذ خلال دورة حياة الطلب. لها وصول إلى `req`، `res`، و`next`. استدعِ `next()` لتمرير التحكم للبرمجيات الوسيطة التالية، أو احذفها لإنهاء الدورة.
- **برمجيات وسيطة على مستوى التطبيق**: مرتبطة بالتطبيق كاملًا عبر `app.use()`. تُنفذ لكل طلب وارد.
- **برمجيات وسيطة على مستوى الموجِّه**: مرتبطة بمثيل `express.Router()` محدد عبر `router.use()`. تُطبق فقط على المسارات التي يعالجها ذلك الموجِّه.
- **برمجيات وسيطة لمعالجة الأخطاء**: تُعرف بأربعة معلمات `(err, req, res, next)`. توضع في أسفل السلسلة.
- **`express.json()`**: يحلل الطلبات الواردة ذات الحمولة JSON. يجعل البيانات متاحة على `req.body`.
- **`express.urlencoded({ extended: true })`**: يحلل بيانات النماذج المشفرة URL.
- **`express.static('public')`**: يقدم ملفات ثابتة من دليل.
- **`cors`**: يفعّل مشاركة الموارد عبر الأصول المختلفة. ثبت باستخدام `npm install cors`، واستخدم مع `app.use(cors())`.
- **`morgan`**: يسجل طلبات HTTP. استخدم `app.use(morgan('tiny'))`.
- تُطبق البرمجيات الوسيطة بالترتيب الذي تُسجل به عبر `app.use()`.

## فهم WebSockets

- **WebSockets**: بروتوكول ينشئ اتصالًا دائمًا ثنائي الاتجاه (مزدوج) بين العميل والخادم، مما يتيح التواصل في الوقت الحقيقي.
- **مشكلة HTTP مع الوقت الحقيقي**: يتطلب HTTP من العميل الاستطلاع المتكرر للخادم للحصول على بيانات جديدة، لذا فهو بطيء وغير فعال لتطبيقات الدردشة، لوحات النتائج الحية، والألعاب متعددة اللاعبين.
- **مصافحة WebSocket**: يبدأ كل اتصال WebSocket بطلب HTTP. يطلب العميل ترقية ويرد الخادم بـ `101 Switching Protocols`. بعد ذلك يبقى الاتصال مفتوحًا حتى يرسل أحد الطرفين إشارة إغلاق.
- **حالات الاستخدام**: تطبيقات الدردشة، الألعاب متعددة اللاعبين عبر الإنترنت، لوحات الرياضة الحية، تغذية سوق الأسهم، أدوات التعاون.

### مكتبة `ws`

- لا يحتوي Node.js على خادم WebSocket مدمج؛ مكتبة `ws` هي الطريقة الشائعة لتنفيذ واحد.

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**: إرسال رسالة إلى عميل محدد.
- **البث إلى جميع العملاء المتصلين**:

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**: خاصية على كل اتصال. `WebSocket.OPEN` تعني أن الاتصال نشط.

### هندسة النشر/الاشتراك (Pub/Sub)

- **النشر/الاشتراك (Pub/Sub)**: نمط رسائل حيث يرسل **الناشرون** رسائل إلى **المواضيع**، ويتلقى **المشتركون** الرسائل من المواضيع التي اشتركوا بها. الناشرون والمشتركون لا يتواصلون مباشرة.
- **المزايا مقارنة بالتواصل المباشر**: يفصل المكونات، لا يحتاج الناشر لمعرفة من هم المشتركون.
- **Pub/Sub مقابل WebSockets**: WebSockets هي قناة الاتصال (الطريق). Pub/Sub هو نمط الرسائل الذي يحدد من يستقبل أي رسائل (نظام المرور).

## مقدمة في التوثيق والتفويض

### التوثيق مقابل التفويض

- **التوثيق**: يتحقق من الهوية، يجيب على سؤال "من أنت؟"، يجب أن يحدث قبل التفويض.
- **التفويض**: يحدد الأذونات، يجيب على سؤال "ما الذي يسمح لك بفعله؟".
- **ثلاثة عوامل توثيق**:
  - **شيء تعرفه**: كلمة مرور، رقم تعريف شخصي، إجابة سؤال أمان.
  - **شيء تملكه**: هاتف ذكي يستقبل رمز SMS، رمز تطبيق المصادقة.
  - **شيء أنت عليه**: بصمة إصبع، التعرف على الوجه (البيومترية).
- **الأدوار الشائعة في التفويض**: مستخدم عادي (خدمة ذاتية محدودة)، محرر (إدارة المحتوى)، مسؤول (وصول كامل للنظام).
- في شفرة الخادم، يُفرض التفويض عادة عبر برمجيات وسيطة تتحقق من أدوار المستخدم أو الأذونات قبل منح الوصول.

### تزوير طلبات المواقع عبر المواقع (CSRF)

- **CSRF**: هجوم يخدع فيه موقع خبيث متصفح مستخدم موثق لإرسال طلب غير مرغوب فيه إلى خادم خلفي، مستغلاً إرفاق المتصفح التلقائي لملفات تعريف الارتباط.
- **تدفق الهجوم**: المستخدم يسجل الدخول إلى `trusted-bank.com` → يزور `evil-hacker-site.com` → يرسل الموقع الخبيث نموذجًا مخفيًا إلى البنك → يرى خادم البنك ملف تعريف الجلسة صالح وينفذ الطلب.
- **`SameSite=Strict`**: المتصفح لا يرسل ملف تعريف الارتباط إذا جاء الطلب من موقع طرف ثالث.
- **`SameSite=Lax`**: يسمح المتصفح بملفات تعريف الارتباط فقط عند التنقل الآمن إلى الموقع الهدف (النقر على رابط)، لكنه يمنعها في إرسال النماذج عبر المواقع.
- **رموز CSRF**: يولد الخادم رمزًا عشوائيًا غير متوقع مضمنًا في النماذج. يجب على العميل تضمين الرمز في كل طلب يغير الحالة. لا يمكن للموقع الخبيث قراءة الرمز من الواجهة الأمامية، لذا تفشل طلباته المزورة في التحقق.

### رموز الويب JSON (JWT)

- **JWT**: رمز موقع رقميًا يصدره الخادم بعد تسجيل دخول ناجح. يخزنه العميل ويرسله مع كل طلب لاحق، مما يتيح توثيقًا عديم الحالة (لا حاجة لتخزين جلسة على الخادم).
- **هيكل JWT**: ثلاثة أجزاء مشفرة بـ Base64URL مفصولة بنقاط:
  - **الرأس**: نوع الرمز (`JWT`) وخوارزمية التوقيع (مثل `HS256`).
  - **الحمولة**: مطالبات عن المستخدم (معرف المستخدم، اسم المستخدم، الدور). غير مشفرة، فلا تخزن كلمات المرور أو البيانات الحساسة هنا.
  - **التوقيع**: تجزئة الرأس + الحمولة + مفتاح سر الخادم. يمنع التلاعب.

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

- إذا عبث المستخدم بالحمولة، لن يتطابق التوقيع ويرفض الخادم الطلب مع `401 Unauthorized`.

### Passport.js

- **Passport.js**: برمجيات وسيطة للتوثيق في Node.js. يوحد تدفق التوثيق ويتعامل مع التفاصيل منخفضة المستوى ليترك المطورين يركزون على منطق الأعمال.
- **الاستراتيجيات**: ملحقات معيارية تنفذ طرق توثيق محددة. أكثر من 500 استراتيجية متاحة: `passport-local` (اسم المستخدم/كلمة المرور)، `passport-jwt` (JWTs)، `passport-google-oauth20` (تسجيل دخول جوجل)، `passport-facebook`، `passport-github2`.
- **الربط مع Express**:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **الاستخدام كبرمجيات وسيطة للمسار**:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- بعد التوثيق الناجح، يخزن Passport قطعة من بيانات المستخدم في ملف تعريف الجلسة ويرفق كائن المستخدم الموثق إلى `req.user`.

### Helmet.js

- **Helmet.js**: مجموعة برمجيات وسيطة أمنية تحمي تطبيقات Express بضبط، تعديل، أو إزالة رؤوس استجابة HTTP تلقائيًا.

```js
const helmet = require('helmet');
app.use(helmet());
```

- **يزيل `X-Powered-By`**: يمنع المهاجمين من معرفة تقنية الخادم.
- **يضبط `Content-Security-Policy`**: يقيّد مصادر تحميل السكريبتات، الصور، والأنماط، مما يخفف هجمات XSS.
- **يضبط `X-Frame-Options: SAMEORIGIN`**: يمنع النقر الاحتيالي عبر حظر تضمين موقعك في إطارات خارجية.
- **يضبط `Strict-Transport-Security`**: يجبر المتصفحات على استخدام HTTPS، مما يحمي البيانات من الاعتراض.

## فهم الأمان والخصوصية في تطبيقات الويب

### الأمان مقابل الخصوصية

- **الأمان**: آليات الدفاع التقنية التي تحمي الأنظمة والبيانات من الوصول غير المصرح به. الأمان يوفر القفل على الباب.
- **الخصوصية**: إدارة كيفية جمع البيانات الشخصية واستخدامها ومشاركتها. الخصوصية توفر الستار على النافذة.
- يمكنك أن تمتلك أمانًا بدون خصوصية (قاعدة بيانات مغلقة قد تبيع بياناتك)، لكن لا يمكنك أن تمتلك خصوصية بدون أمان (البيانات غير المحمية لا تبقى خاصة).

### سياسة نفس الأصل وCORS

- **الأصل**: يُعرف بثلاثة مكونات: البروتوكول (مثل `https`)، النطاق (مثل `example.com`)، والمنفذ (مثل `443`). يجب أن تتطابق الثلاثة ليشارك عنوانان نفس الأصل.
- **سياسة نفس الأصل**: قاعدة أمان في المتصفح تمنع السكريبتات على أصل من قراءة بيانات من أصل مختلف، مما يمنع المواقع الخبيثة من سرقة البيانات من علامات تبويب أخرى مفتوحة.
- **CORS (مشاركة الموارد عبر الأصول المختلفة)**: آلية تسمح للخوادم بتخفيف سياسة نفس الأصل للأصول الموثوقة عبر ضبط رؤوس HTTP.
- **في Express (التطوير)**: `app.use(cors())` يسمح بكل الأصول.
- **في Express (الإنتاج)**: مرر كائن `corsOptions` مع قائمة سماح للأصول في خاصية `origin`:

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**: HTTP مع تشفير مقدم بواسطة TLS (أمان طبقة النقل). استبدل TLS البروتوكول القديم SSL.
- **ثلاث ضمانات لـ HTTPS**:
  - **التشفير**: البيانات المرسلة بين العميل والخادم غير قابلة للقراءة من قبل أطراف ثالثة.
  - **التوثيق**: شهادة رقمية من سلطة شهادات موثوقة تؤكد أن الخادم هو من يدعي.
  - **سلامة البيانات**: لا يمكن تعديل الرسائل أثناء النقل دون اكتشاف.
- HTTPS مطلوب لواجهات برمجة التطبيقات الحديثة في المتصفح مثل تحديد الموقع الجغرافي، الإشعارات الفورية، والوصول إلى الكاميرا.

### لماذا تحدث مشاكل الأمان

- **ضغط التسليم السريع**: الفرق التي تتحرك بسرعة كبيرة للوفاء بالمواعيد النهائية تتخطى مراجعات الأمان وترسل ثغرات.
- **تكامل النظام المعقد**: خطر سلسلة التوريد ينشأ عندما تقدم مكتبات الطرف الثالث أو خدمات السحابة ثغرات لم يكتبها فريق التطوير.
- **سوء إدارة التهيئة**: ترك الإعدادات الافتراضية أو المصنعية (كلمات مرور المسؤول الافتراضية، المنافذ المفتوحة) يعرض الأنظمة بلا داعٍ.
- **خطأ بشري**: هجمات التصيد تستغل الموظفين الذين ينقرون على روابط أو مرفقات خبيثة ويسلمون بيانات الاعتماد.
- **الاندماجات التنظيمية**: دمج قواعد شفرة وبنى تحتية مختلفة يكشف سياسات أمان غير متطابقة وثغرات موروثة.

### ملفات تعريف الارتباط: التخزين، علامات الأمان، والخصوصية

- **كيفية تخزين ملفات تعريف الارتباط**: يرسل الخادم رأس `Set-Cookie`. يحفظ المتصفح الملف على القرص ويعيده تلقائيًا مع كل طلب لاحق لنفس الأصل.
- **علامة `HttpOnly`**: تمنع جافاسكريبت من قراءة قيمة ملف تعريف الارتباط، مما يمنع سرقة الجلسة عبر XSS.
- **علامة `Secure`**: تقيد ملف تعريف الارتباط للاتصالات عبر HTTPS فقط، تحميه من الاعتراض على الشبكات العامة.
- **`SameSite=Strict`**: تمنع المتصفح من إرسال ملف تعريف الارتباط في أي طلب عبر المواقع، مما يحجب هجمات CSRF تمامًا.
- **`SameSite=Lax`**: يسمح بملفات تعريف الارتباط في التنقلات على المستوى الأعلى (مثل النقر على رابط) لكنه يمنعها في إرسال النماذج عبر المواقع واستدعاءات API.
- **ملفات تعريف الارتباط من الطرف الأول**: مخصصة للموقع الذي تزوره.
- **ملفات تعريف الارتباط من الطرف الثالث**: يضعها شبكات الإعلانات عبر آلاف المواقع لتتبع السلوك وبناء ملفات تعريف المستخدمين. المتصفحات الحديثة تحجبها افتراضيًا.
- **الامتثال لـ GDPR**: قوانين الخصوصية تتطلب لافتات موافقة على ملفات تعريف الارتباط تمنح المستخدمين الحق القانوني في رفض ملفات التتبع غير الضرورية.

### التهديدات الأمنية الشائعة والتدابير الوقائية

- **التصيد الاحتيالي**: رسائل بريد إلكتروني مزيفة تنتحل شركات موثوقة لسرقة بيانات الاعتماد. تشمل التدابير التدريب على الوعي الأمني ونشر بروتوكولات توثيق البريد الإلكتروني لحجب الرسائل الخبيثة قبل وصولها لصناديق البريد.
- **البرمجيات الخبيثة** (فيروسات، برامج تجسس، برامج فدية): برامج الفدية تشفر الملفات وتطلب فدية لفكها. تشمل التدابير التحديثات التلقائية وبرامج مكافحة البرمجيات الخبيثة.
- **حشو بيانات الاعتماد**: يختبر المهاجمون ملايين تركيبات أسماء المستخدمين وكلمات المرور المسروقة عبر مواقع مختلفة. التدبير الأساسي هو المصادقة متعددة العوامل (MFA)، التي تتطلب خطوة تحقق ثانية مثل رمز OTP أو رمز تطبيق المصادقة.
- **هجمات الحرمان من الخدمة الموزعة (DDoS)**: يفيض المهاجمون الخادم بحركة مرور مزيفة لتعطيله ومنع المستخدمين الشرعيين. تشمل التدابير خدمات تصفية حركة المرور السحابية مثل Cloudflare، Akamai، وAWS Shield.
- **التهديدات الداخلية**: موظفون خبيثون أو مهملون يسرقون أو يسرّبون البيانات. تشمل التدابير تطبيق **مبدأ أقل الامتيازات** (يصل العاملون فقط لما يتطلبه عملهم) ومراقبة سجلات النظام للنشاط غير المعتاد.

### سياسة أمان المحتوى وPermissions-Policy

- **CSP (سياسة أمان المحتوى)**: رأس استجابة HTTP يحدد قائمة بيضاء لمصادر السكريبتات، الأنماط، والصور المسموح بها. عندما يستلم المتصفح رأس CSP، يمنع تحميل أي مورد من نطاق غير مدرج. توجيه `script-src` يحدد النطاقات التي يمكنها تنفيذ جافاسكريبت، مما يوقف هجمات حقن XSS.
- **Permissions-Policy**: رأس HTTP يفعّل أو يعطل ميزات المتصفح وأجهزة الجهاز لصفحة وأي إطارات مضمنة. استبدل رأس `Feature-Policy` القديم.
  - `camera=(self)` يقيد الوصول إلى كاميرا الويب لأصل الصفحة فقط.
  - `geolocation=()` يعطل تحديد الموقع الجغرافي تمامًا لكل المستخدمين.
- الجمع بين الرأسين يوفر حماية متعددة الطبقات: CSP يمنع حقن الشفرة وPermissions-Policy يمنع الوصول غير المصرح به للأجهزة.

### مفاهيم الخصوصية الأساسية للمستخدم

- **المعلومات الشخصية القابلة للتعريف (PII)**: أي بيانات يمكنها تحديد هوية شخص بشكل مباشر أو غير مباشر.
  - **المعرفات المباشرة**: الاسم الكامل، عنوان المنزل، رقم الهوية الوطنية.
  - **المعرفات غير المباشرة**: عنوان IP، الرقم التسلسلي للجهاز، سجل الموقع الدقيق.
- **السرية**: الحفاظ على أمان المعلومات الشخصية من الوصول غير المصرح به عبر التشفير وضوابط الوصول.
- **تقليل البيانات**: جمع الحد الأدنى الضروري فقط من المعلومات الشخصية لإتمام مهمة محددة.
- **تحديد الغرض**: لا يمكن إعادة استخدام البيانات التي جُمعت لغرض معين (مثل بيعها لشبكات الإعلانات) دون موافقة جديدة من المستخدم.
- **التتبع**: المراقبة المستمرة لسلوك المستخدم عبر الإنترنت باستخدام السكربتات، وبصمة الجهاز، وسجلات تاريخ المتصفح.
- **الحق في النسيان (محو البيانات)**: الحق القانوني في طلب حذف جميع السجلات الشخصية من قواعد بيانات الشركة بشكل دائم بعد إغلاق الحساب.

### ملف `.env` وحزمة `dotenv`

- **ملف `.env`**: ملف نصي عادي في جذر المشروع يخزن أسرار التكوين مثل مفاتيح API، وكلمات مرور قاعدة البيانات، وأرقام المنافذ كأزواج المفتاح-القيمة. أضفه دائمًا إلى `.gitignore` لمنع تسرب الأسرار في نظام التحكم بالإصدار.
- **حزمة `dotenv`**: تقرأ ملف `.env` عند بدء التشغيل وتحقن القيم في `process.env`:

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **فائدتان رئيسيتان**: تبقى بيانات الاعتماد خارج شفرة المصدر (الأمان)، ويمكن أن يتصرف نفس الكود بشكل مختلف عبر بيئات التطوير، والاختبار، والإنتاج بتبديل ملف `.env` على كل خادم (المرونة).

### قوانين الخصوصية الإقليمية والامتثال

- **اللائحة العامة لحماية البيانات (GDPR)**: تحمي الأفراد في الاتحاد الأوروبي. تتطلب موافقة صريحة قبل التتبع، ولا تسمح بالمربعات المحددة مسبقًا. تمنح المستخدمين حق عرض بياناتهم وحذفها نهائيًا. تفرض غرامات مالية كبيرة على المخالفين.
- **قانون خصوصية المستهلك في كاليفورنيا (CCPA) / قانون حقوق الخصوصية في كاليفورنيا (CPRA)**: يحمي سكان كاليفورنيا. يركز على حقوق الرفض بدلاً من الموافقة المسبقة. يتطلب رابطًا واضحًا "لا تبيع معلوماتي الشخصية" على الصفحة الرئيسية.
- **قانون حماية خصوصية الأطفال على الإنترنت (COPPA)**: قانون اتحادي أمريكي يحمي الأطفال دون 13 سنة. يتطلب سياسة خصوصية واضحة وموافقة أبوية قابلة للتحقق قبل جمع بيانات الطفل.
- **قانون حماية البيانات (DPA)**: الإطار الرئيسي لحماية البيانات في المملكة المتحدة. يعمل جنبًا إلى جنب مع GDPR البريطاني ويمنح المستخدمين حق الوصول إلى بياناتهم وحذفها.
- لدى دول أخرى قوانين خاصة بها: نيجيريا لديها قانون حماية البيانات (NDPA) والهند لديها قانون حماية البيانات الرقمية الشخصية (DPDP).
- **خطوات الامتثال للفرق العالمية**:
  - بناء لافتات موافقة تعتمد على الموقع الجغرافي تعرض إشعار ملفات تعريف الارتباط المناسب بناءً على موقع المستخدم.
  - تنفيذ نظام طلبات الوصول إلى البيانات (DSAR) آليًا للعثور على بيانات المستخدم أو تصديرها أو حذفها نهائيًا بناءً على طلب قانوني.
  - ممارسة تقليل البيانات واستخدام بوابات التحقق من العمر لمنع التتبع غير المصرح به للقُصّر.
  - تعيين مسؤول حماية البيانات (DPO) للإشراف على هيكل الخصوصية والعمل كجهة اتصال رئيسية للسلطات التنظيمية.
