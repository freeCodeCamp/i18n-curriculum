# --description--

경험 많은 개발자들과 온라인 커뮤니티나 직접 만나서 이야기해본 적이 있다면, 아마도 "무서운 병합 충돌"에 대해 들어본 적이 있을 거예요. 그런데 병합 충돌이 정확히 무엇일까요?

이전에 배운 적절한 브랜칭 전략을 사용하고 있다면, 두 개의 서로 다른 기능을 두 개의 다른 브랜치에서 동시에 작업하고 있을 수 있습니다. 또는 내가 어떤 작업을 하는 동안 동료가 다른 브랜치에서 작업하고 있을 수도 있죠. 어쨌든 두 개의 독립적인 변경 사항이 동시에 진행되고 있는 셈입니다. 그런데 가끔 두 브랜치가 같은 파일을 수정할 때가 있습니다. 만약 다른 브랜치가 `main`에 먼저 병합된다면, 내 브랜치의 변경 사항은 뒤처지게 됩니다.

대부분의 경우, `main`을 다시 내 기능 브랜치에 병합하면 해결할 수 있습니다. 하지만 다른 브랜치의 변경 사항이 내 브랜치와 충돌하면, Git은 병합할 때 어떤 변경 사항을 유지해야 할지 알 수 없습니다. 이것을 "병합 충돌"이라고 합니다.

그럼 어떻게 고칠 수 있을까요? 작은 충돌이나 간단한 충돌의 경우, GitHub가 웹 UI를 제공해 줍니다. 충돌이 있는 풀 리퀘스트 하단에 "Resolve conflicts" 버튼이 나타납니다. 이 버튼을 클릭하면 충돌이 있는 파일이 나타나고, 내 코드와 충돌하는 코드가 함께 표시됩니다. Git은 문제를 식별하기 쉽게 내 코드와 충돌하는 코드 주위에 "병합 충돌 표시자"를 추가합니다. 대략 이런 모양입니다:

```sh
1 <<<<<<< feat/conflict
2 Here's some code from my first PR :)
3 =======
4 Here's some code that was merged to main that conflicts with the code on my PR
5 >>>>>>> main
```

열리는 표시자는 `<<<<<<< feat/conflict`이며, 현재 브랜치가 만든 변경 사항의 시작을 나타냅니다. 그다음 `Here's some code from my first PR :)`이 보입니다. 그 뒤에 `=======` 표시자가 있는데, 이는 내 변경 사항의 끝과 `main` 브랜치 변경 사항의 시작을 나타냅니다. `main`에서 이루어진 변경 사항은 `Here's some code that was merged to main that conflicts with the code on my PR` 줄입니다. 그리고 마지막으로 충돌의 끝을 나타내는 닫는 `>>>>>> main` 표시자가 있습니다.

그럼 어떻게 고치나요? 병합 충돌에서 어떤 변경 사항을 유지할지 결정해야 합니다. 두 변경 사항을 모두 유지하고 싶다면, 충돌 표시자만 삭제하고 변경 사항은 그대로 두면 됩니다. 한쪽 브랜치의 변경 사항만 유지하고 싶다면, 충돌 표시자와 유지하지 않을 쪽의 변경 사항을 삭제하세요. 그리고 만약 어떤 변경 사항도 마음에 들지 않는다면, 열리는 표시자부터 닫는 표시자까지 모두 삭제할 수 있습니다(포함해서).

이번에는 `main` 브랜치의 변경 사항을 삭제하고 내 변경 사항만 유지해 보겠습니다. 지금 상태는 다음과 같습니다:

```sh
1 Here's some code from my first PR :)
```

"Mark as resolved" 버튼을 클릭하면 GitHub에 해결 완료를 알립니다. 만약 추가 파일이 있다면 GitHub가 다음 파일로 넘어갑니다. 하지만 이 파일이 유일한 충돌 파일이라면 "Commit merge" 버튼이 나타납니다. 이 버튼을 클릭하면 GitHub가 병합 커밋을 생성합니다. 풀 리퀘스트 화면으로 돌아가면 새 병합 커밋과 해결된 충돌을 확인할 수 있습니다. 이제 모두 완료되었습니다!

그런데 충돌이 너무 복잡해서 GitHub에서 직접 편집할 수 없다면 어떻게 할까요?

브랜치에 커밋이 적거나 해당 파일을 수정한 커밋이 몇 개뿐이라면, `main`을 내 브랜치에 병합할 수 있습니다:

```sh
git checkout feat/conflict
git fetch origin
git merge origin/main
```

Git이 이 병합을 처리하는 동안 수동으로 해결해야 할 충돌이 발생합니다. Git이 어떤 파일을 수정해야 하는지 알려줍니다. 충돌이 있는 파일을 편집기에서 열면, 이전과 같은 충돌 표시자가 나타나고 VSCode 덕분에 유용한 UI도 추가되어 있습니다. 같은 방식으로 충돌을 해결하세요. 이번에는 커밋을 수동으로 생성해야 합니다. 일반 커밋을 만드는 것과 같습니다:

```sh
git add .
git commit -m "chore: resolve conflicts"
```

이 명령어들을 실행하면 Git이 해결을 받아들인 것을 확인할 수 있습니다.

커밋이 많거나 충돌이 복잡할 때는 리베이스가 도움이 될 수 있습니다. 병합 대신 리베이스는 브랜치의 모든 커밋을 가져와 `main`의 최신 변경 사항 뒤에 다시 적용합니다. Git 용어로는, 이 작업이 `main`의 가장 최근 커밋을 기준으로 브랜치의 기반을 "재설정"하는 효과가 있습니다.

이 작업을 실습하려면 새 브랜치가 필요합니다:

```sh
git checkout main
git pull
git checkout -b feat/rebase
```

브랜치에 여러 커밋을 만들어 리베이스를 연습해 보겠습니다:

```sh
echo "change one" > README.md
git add .
git commit -m "change one"
echo "change two" > README.md
git add .
git commit -m "change two"
echo "change three" > README.md
git add .
git commit -m "change three"
```

이제 `main`에서 또 다른 변경을 만들어 충돌을 발생시켜 보겠습니다:

```sh
git checkout main
echo "change four" > README.md
git add .
git commit -m "change four"
```

그리고 리베이스 브랜치로 체크아웃합니다:

```sh
git checkout feat/rebase
```

다음으로 `git rebase -i main`를 실행하세요. 이 명령은 브랜치를 `main` 브랜치의 현재 상태 위에 리베이스합니다. `-i` 플래그는 Git에게 대화형 모드로 리베이스하라고 지시합니다. 커밋 옆에 `pick`라는 단어가 보일 것입니다. `pick`는 커밋을 그대로 사용하라는 뜻입니다. 파일을 저장하고 닫으면 충돌이 있다는 오류 메시지가 나타납니다.

`change one` 커밋이 `change four` 커밋과 충돌하므로 해결해야 합니다. 편집기에서 파일을 열면 이전 충돌 때와 같은 UI가 나타납니다. 내 `change one` 변경 사항을 유지하세요. 그런 다음 `git add .`를 실행해 변경 사항을 추가하고 `git rebase --continue`를 실행해 리베이스를 계속할 준비가 되었음을 Git에 알립니다. 첫 번째 커밋에서 충돌을 해결했으므로 Git이 나머지 커밋 리베이스를 자동으로 처리할 수 있습니다.

마지막으로, 깔끔한 기록을 위해 세 개의 변경 커밋을 하나로 합쳐 보겠습니다. 다시 `git rebase -i main`를 실행해 편집기를 엽니다. 첫 번째 명령은 `pick`로 두고, 다음 두 개는 `squash`로 변경하세요. 같은 충돌이 다시 발생할 수 있습니다. 발생하면 이전과 같은 방식으로 해결하세요. 충돌을 해결하고 리베이스를 계속하면(또는 충돌이 없으면 바로) 새 커밋 메시지 편집기가 나타납니다. 이 메시지가 세 개 커밋을 합친 새 커밋 메시지가 됩니다. `change three complete`라고 이름을 붙이고 저장 후 편집기를 닫으면 Git이 리베이스를 완료합니다.

커밋 기록을 확인하면 이제 단일 `change three complete` 커밋이 있고, `change four` 커밋을 기준으로 제대로 기반이 설정된 것을 볼 수 있습니다. 이렇게 해서 커밋을 합친 리베이스를 성공적으로 완료했습니다! 이제 병합 충돌 해결의 달인이 될 준비가 되었습니다!
