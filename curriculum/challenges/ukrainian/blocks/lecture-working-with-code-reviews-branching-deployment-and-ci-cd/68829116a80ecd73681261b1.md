# --description--

Ми трохи говорили про використання гілок для роботи над ізольованими змінами, щоб не забруднити вашу `main` гілку. Але що це означає і чому це важливо?

Якщо ви робите коміти безпосередньо у вашу `main` гілку, а upstream-репозиторій приймає інші зміни, які не ваші, у вас утворюється так званий «розбіжний історія». Це означає, що коміти у вашій `main` гілці більше не узгоджуються з комітами у upstream `main` гілці. І це робить синхронізацію змін з upstream справжнім кошмаром.

Отже, припустимо, ви дотримуєтеся хороших практик роботи з гілками, і ця проблема вас не турбує. Якщо ви слідували нашим попереднім урокам, у вас має бути форк репозиторію, який ви клонували локально, і upstream-remote, що вказує на оригінальний репозиторій.

Якщо ви цього не зробили, ось коротке нагадування. Вам потрібно форкнути репозиторій, клонувати свій форк локально і виконати цю команду:

```sh
git remote add upstream <url>
```

Замініть `<url>` на SSH URL оригінального репозиторію, а не вашого форку.

З правильно налаштованим upstream ви можете виконати таку послідовність команд, щоб отримати останні зміни з upstream:

```sh
git checkout main
git fetch upstream
git merge upstream/main
```

Що вони роблять? Ви вже вивчали `git checkout`, тому перша команда гарантує, що ви перебуваєте у своїй локальній `main` гілці.

Друга команда, `git fetch upstream`, каже Git отримати останні зміни з upstream-remote (який є оригінальним репозиторієм). Але, на відміну від pull, Git не оновить вашу гілку автоматично. Це важливо, бо ваша локальна `main` гілка пов’язана з `main` гілкою вашого форку — а не з `main` гілкою оригінального репозиторію! Виконання git pull може призвести до непередбачуваної поведінки.

Остання команда, `git merge upstream/main`, каже Git об’єднати останні зміни з `main` гілки upstream-remote у вашу поточну гілку — тобто у вашу локальну `main` гілку, бо саме її ми вибрали.

А що, якщо об’єднання не вдається? Що, якщо ви випадково зробили коміт у `main`, і Git не може об’єднати розбіжні історії? Припустимо, ви не створювали запитів на злиття з вашої `main` гілки, тоді можна зробити жорсткий скидання, щоб примусити Git оновити вашу гілку.

```sh
git reset --hard upstream/main
```

Команда `git reset` дозволяє скинути поточний стан гілки. Прапорець `--hard` каже Git примусово привести локальні файли у відповідність зі станом гілки. Це гарантує, що у вас чистий початок для роботи. А прапорець `upstream/main` каже Git, що ви хочете, щоб ваша поточна гілка (яка є `main`) відповідала `main` гілці upstream-remote. Будьте обережні з цією командою — будь-яке примусове оновлення історії Git може призвести до дивних і неприємних наслідків.

Коли ваша локальна `main` гілка буде правильно оновлена, важливо відправити цей стан на ваш віддалений форк (який має назву `origin`, якщо ви слідували нашим урокам). Для цього виконайте:

```sh
git push
```

Це відправить поточний стан на ваш віддалений форк. Але якщо вам довелося робити жорсткий скидання, можливо, у вашого віддаленого форку також розбіжна історія, і push не вдасться.

Ви можете виправити це, примусово виконавши push:

```sh
git push --force
```

**Будьте обережні з цією командою**. Примусовий push фактично переписує історію на віддаленому репозиторії, і якщо ви зробили щось неправильно, то розплутати наслідки такого push буде дуже складно.

Відправлення синхронізованих змін на ваш віддалений форк важливе, бо це допомагає гарантувати, що ви зможете безпечно клонувати форк з останніми змінами, якщо втратите локальний репозиторій.

І коли ваша `main` гілка синхронізована, ви можете створити нову гілку з цього стану, щоб почати роботу над новим внеском — з останньою версією оригінального коду!
