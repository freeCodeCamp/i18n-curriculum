# --description--

正規化とは、リレーショナルデータベースを既存の正規形に従って整理し、データの冗長性を減らしデータの整合性を向上させるプロセスです。

データベースを正規化するには、テーブル構造の属性と関係を分析し、正規化ルールに基づいて簡素化の機会を見つけます。そして、データをより小さく、より焦点を絞ったテーブルに分割し、主キーと外部キーを使ってそれらの間に関係を確立します。これらの小さなテーブルは元のすべてのデータを保持しますが、大きなテーブルと比べて管理、整理、操作がしやすくなります。

データの冗長性を最小限に抑えることで、正規化はストレージの節約と不整合の防止に役立ちます。例えば、顧客の住所が変わった場合、すべての注文レコードではなく、顧客テーブルの1か所だけを更新すれば済みます。

また、主キーと外部キーの制約によって依存関係が強制されるため、データの整合性も保たれます。これにより、挿入、更新、削除の異常が起こる可能性が減ります。

正規化されたデータベースは理解しやすくメンテナンスもしやすいため、設計の良いデータベースシステムに貢献します。

テーブルを整理するために従うべきルールのセットを正規形と呼びます。各正規形は前の正規形を基にしており、上位の正規形ほどルールが厳しくなります。第一正規形（1NF）は基本的なルールを持つ基礎的な正規形です。

テーブルが第一正規形（1NF）であるとは：

- 各セルには単一の値のみを含むべきです。例えば、`students` テーブルに複数の電話番号を持つ学生がいる場合、それらを1つのセルにカンマ区切りで保存するのではなく、`phone_numbers` テーブルを作成し、外部キーで `students` テーブルを参照し、各セルには1つの値だけを保存すべきです。これにより各値が原子性を持ち、設計が正規化されます。
- すべてのレコード（行）は一意でなければなりません。主キーがこの要件を強制しますが、1NFはテーブル内に重複行が存在しないことを明示的に要求します。
- 行と列の順序は重要ではありません。データは物理的な位置に依存してはいけません。

要するに、第一正規形（1NF）は値を単純かつ原子的にすることに焦点を当てています。

第二正規形（2NF）は1NFの要件を満たし、さらに追加の制約を課します。

2NFを理解するには、スーパーキーと候補キーという2つの重要な概念を知っておく必要があります。スーパーキーとは、テーブル内の各行を一意に識別する属性の集合であり、スーパーキーのすべての列の値が同じ行は存在しません。

例えば、`customers` テーブルに3つの列があるとします：

```sql
customer_id | name | email
```

`customer_id` と `email` が各レコードで一意でなければならないとしましょう。このテーブルのスーパーキーの例は：

- `customer_id`
- `{ customer_id, name }`（この組み合わせは `customer_id` が単独で一意なので、`name` の一意性に関係なく行を一意に識別します）
- `{ customer_id, name, email }`
- `email`（この列にも一意制約を追加しているため、メールで各レコードを一意に識別できます）

行を一意に識別する属性の集合はすべてスーパーキーになり得ます。

候補キーは似ていますが少し異なります。候補キーは、テーブル内の1つ以上の列の集合で、各レコードを一意に識別できます。テーブルには複数の候補キーが存在する場合がありますが、その中から1つを主キーとして選ぶ必要があります。

似ていますよね？違いは、スーパーキーは行を一意に識別するために必要ない追加の属性を含む場合があるのに対し、候補キーは行を一意に識別する属性だけを含む点です。

`customers` の例では、`customer_id`、`{ customer_id, name }`、`{ customer_id, name, email }`、`email` がスーパーキーになり得ると述べました。この場合、候補キーになり得るのは `customer_id` と `email` のみです。なぜなら候補キーは行を一意に識別しない属性（`name` のような）を含められないからです。これが候補キーとスーパーキーを区別する最小性の原則を示しています。

これらの概念に慣れたところで、第二正規形（2NF）に戻りましょう。

第二正規形（2NF）は部分関数従属の解消に基づいています。部分関数従属とは、複合主キーの一部にのみ依存する非主キー属性が存在する場合のことです（複合主キーは複数の列で構成される主キー）。これによりデータの冗長性や更新異常が発生します。

例えば、次の列を持つ `orders` テーブルがあるとします：

```sql
order_id | item_id | order_date | quantity | order_shipping_city
```

このテーブルの主キーは `order_id` と `item_id` の組み合わせです。同じ商品IDが異なる注文に存在する可能性があるためですが、その組み合わせは一意です。`order_id` と `order_shipping_city` の間に部分関数従属があることがわかります。`order_id` は主キーの一部です。`order_shipping_city` は `order_id` に依存しています。同じ注文IDのすべての注文は同じ配送先の都市を持つためです。しかし配送先の都市は `item_id` には依存していませんが、これも主キーの一部です。したがって `order_shipping_city` は主キー全体に依存していません。

これを修正するには、テーブルを `order_header` と `order_items` のような2つの小さなテーブルに分割できます。`order_header` テーブルには注文のトップレベル情報を保存できます：

```sql
order_id | order_date | order_shipping_city
```

`order_items` テーブルには提出された異なる注文の商品の情報を保存できます：

```sql
order_id | item_id | quantity
```

これらの変更により、両方のテーブルは第二正規形（2NF）になります。

そして第三正規形（3NF）に進みます。この正規形は第二正規形を基にしています。

第三正規形であるためには：

- 第二正規形であること
- すべての非キー属性が主キーに直接依存していること。つまり、非キー属性が他の非キー属性に依存してはいけません。

第三正規形（3NF）を理解するには推移的関数従属を理解する必要があります。推移的関数従属とは、主キーの一部でない属性が、主キーの一部でない別の属性に依存し、その属性がさらに主キーに依存している場合のことです。

例えば、`orders` テーブルを次の列に変更しましょう：

```sql
order_id | customer_id | customer_city | city_postal_code | order_date | quantity
```

この新しいテーブルの主キーは `order_id` で、各行を一意に識別します。

このテーブルには推移的関数従属があります：

```sql
order_id > customer_id > customer_city > city_postal_code
```

都市の郵便番号は顧客の都市によって決まり、顧客の都市は顧客IDによって決まり、顧客IDは注文IDによって決まります。`city_postal_code` は主キーの一部でない属性で、`customer_city` に依存しています。`customer_city` も主キーの一部ではありませんが、`customer_city` は `order_id` を経由して `customer_id` に依存しています。つまり、同じ都市に住む顧客が行ったすべての注文で都市の郵便番号が繰り返されます。これによりデータの冗長性が生じます。さらに、都市の郵便番号を変更すると複数のレコードを更新する必要があります。

推移的関数従属を解消するには、テーブルを複数のテーブルに分割する必要があります。まず `orders` テーブル：

```sql
order_id | customer_id | order_date | quantity
```

次に `customers` テーブル：

```sql
customer_id | city_name
```

そして `cities` テーブル：

```sql
city_name | city_postal_code
```

これによりテーブルから推移的関数従属がなくなります。

最後に、Boyce-Codd正規形（BCNF）があります。これは第三正規形の一部の異常を解決するため、正規形3.5とも呼ばれます。Boyce-Codd正規形（BCNF）であるためには：

- 第三正規形であること
- 関数従属の左辺がすべてスーパーキーであること

この正規形の目的は、ある属性または属性の集合が別の属性を決定する場合、その属性集合がスーパーキー（候補キーまたはそのスーパーセット）であることを保証することです。

正規形を理解し適用することは、堅牢で効率的なリレーショナルデータベースを設計するために不可欠です。データの冗長性を排除し論理的なデータ依存関係を確保することで、正規化はデータの整合性を向上させ、ストレージの必要量を減らし、データベースのメンテナンスを簡素化します。通常は第三正規形（3NF）まで達成することを目指すべきです。
