# --description--

I WebSocket permettono a un client e a un server di comunicare in tempo reale.

Normalmente, le applicazioni lato client come le app web e mobile usano HTTP. Con HTTP, il client invia una richiesta e il server risponde con una risposta. Questo si chiama ciclo di richiesta e risposta, come hai imparato.

Dopo che la connessione è stabilita e i dati sono scambiati, la connessione si chiude. Questo funziona bene per la maggior parte delle cose, come caricare una pagina, recuperare o inviare dati. Ma diventa lento e poco affidabile per le applicazioni in tempo reale.

Per esempio, pensa a un'app di chat. Se scegliessi di usare HTTP per questa, il client dovrebbe continuare a chiedere al server i dati di tanto in tanto, con la connessione che si apre, i dati che arrivano e la connessione che si chiude dopo ogni messaggio.

Questo è il problema che risolvono i WebSocket. Creano una connessione permanente tra client e server. Una volta aperta la connessione, entrambe le parti possono scambiare dati in qualsiasi momento in un canale in tempo reale finché uno dei due non invia un segnale di chiusura all'altro.

Anche se i WebSocket sono diversi da HTTP, ogni connessione WebSocket inizia con HTTP, in un processo chiamato handshake.

Il client quindi richiede che la connessione venga aggiornata a WebSocket. Quando il server accetta, invia una `101 Switching Protocols` risposta che aggiorna la connessione a WebSocket. Da quel momento la connessione resta aperta e inizia la comunicazione in tempo reale.

Il flusso è il seguente:

1. Il client si connette al server tramite HTTP
2. Il server accetta e aggiorna la connessione a WebSocket
3. La connessione resta aperta
4. Entrambe le parti scambiano dati in qualsiasi momento finché la connessione non si chiude

Questo flusso è bidirezionale, non a richiesta e risposta come HTTP.

```md
Client <---> Server
```

Con HTTP, il flusso segue tipicamente un modello di richiesta e risposta:

```md
Client ---> Request
Server ---> Response
```

Per capire come i WebSocket differiscono da HTTP, pensa a HTTP come scambiarsi lettere e a WebSocket come una telefonata. Scrivi una lettera, aspetti una risposta e la conversazione finisce. Ma con una telefonata tu e il destinatario parlate in tempo reale.

I WebSocket alimentano app in tempo reale come app di chat, giochi multiplayer online, aggiornamenti sportivi live, dashboard di borsa e strumenti collaborativi come Google Docs.
