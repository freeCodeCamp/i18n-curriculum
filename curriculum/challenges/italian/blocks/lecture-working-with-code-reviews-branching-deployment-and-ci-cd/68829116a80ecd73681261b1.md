# --description--

Abbiamo parlato un po’ di usare i branch per lavorare su modifiche isolate ed evitare di sporcare il tuo branch `main`. Ma cosa significa esattamente e perché è importante?

Se fai commit direttamente sul tuo branch `main` e il repository upstream accetta altre modifiche che non sono tue, finirai con quella che si chiama una "storia divergente". Questo significa che i commit sul tuo branch `main` non sono più allineati con i commit sul branch `main` upstream. E questo rende un incubo sincronizzare le modifiche upstream.

Quindi, supponiamo che tu abbia seguito buone pratiche di branching e questo non sia un problema. Se hai seguito le nostre lezioni precedenti, dovresti avere un fork di un repository che hai clonato localmente, e dovresti avere un remote upstream che punta al repository originale.

Se non l’hai fatto, ecco un rapido promemoria. Vuoi fare il fork di un repository, clonare il tuo fork localmente e eseguire questo comando:

```sh
git remote add upstream <url>
```

Sostituisci `<url>` con l’URL SSH del repository originale, NON il tuo fork.

Con un upstream configurato correttamente, puoi eseguire questa serie di comandi per scaricare le ultime modifiche upstream:

```sh
git checkout main
git fetch upstream
git merge upstream/main
```

Cosa fanno questi comandi? Beh, hai imparato a usare `git checkout`, quindi il primo comando assicura che tu sia sul tuo branch `main` localmente.

Il secondo comando, `git fetch upstream`, dice a Git di andare a prendere le ultime modifiche presenti sul tuo remote upstream (che è il repo originale). Tuttavia, a differenza di un pull, Git non aggiornerà automaticamente il tuo branch. Questo è importante, perché il tuo branch `main` localmente è collegato al branch `main` sul tuo fork - non al branch `main` sul repo originale! Eseguire un git pull potrebbe causare comportamenti imprevisti.

L’ultimo comando, `git merge upstream/main`, dice a Git di unire le ultime modifiche dal branch `main` nel remote upstream nel tuo branch corrente - che è il tuo branch `main` locale, perché è quello che hai controllato.

Ma cosa succede se la merge fallisce? E se hai fatto commit per errore sul branch `main` e Git non riesce a unire le storie divergenti? Beh... supponendo che tu non abbia fatto pull request dal tuo branch `main`, puoi fare un reset hard per forzare Git ad aggiornare il tuo branch.

```sh
git reset --hard upstream/main
```

Il comando `git reset` ti permette di resettare lo stato corrente di un branch. Passare il flag `--hard` dice a Git di forzare i file locali a corrispondere allo stato del branch. Questo assicura che tu abbia una base pulita da cui lavorare. E passando `upstream/main` dici a Git che vuoi che il tuo branch corrente (che è `main`) corrisponda al branch `main` del remote upstream. Usa questo comando con cautela - ogni volta che aggiorni forzatamente una storia Git, possono succedere cose strane e sfortunate.

Una volta che hai aggiornato correttamente il tuo branch `main` locale, è importante spingere quello stato sul tuo fork remoto (che dovrebbe essere `origin`, se hai seguito le nostre lezioni). Per farlo, puoi eseguire:

```sh
git push
```

Questo spingerà lo stato corrente sul tuo fork remoto. Ma se hai dovuto fare un reset hard per pulire la storia, è possibile che anche il tuo fork remoto abbia una storia divergente e il push fallirà.

Puoi risolvere questo problema forzando il push:

```sh
git push --force
```

**Fai attenzione con questo comando**. Fare un push forzato riscrive effettivamente la storia remota, e se hai fatto qualcosa di sbagliato, un push forzato diventa un incubo da sistemare.

Spingere le modifiche sincronizzate sul tuo fork remoto è importante perché ti aiuta a garantire che tu possa clonare in sicurezza il fork con le ultime modifiche se perdi il repository locale esistente.

E una volta che hai sincronizzato il tuo branch `main`, sei libero di creare un nuovo branch da quello stato per iniziare a lavorare su un altro contributo - con l’ultima versione del codice originale!
