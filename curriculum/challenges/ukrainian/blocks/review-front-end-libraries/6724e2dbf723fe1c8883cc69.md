## Бібліотеки та фреймворки JavaScript

- Бібліотеки та фреймворки JavaScript пропонують швидкі рішення для поширених проблем і пришвидшують розробку, надаючи готовий код.
- Бібліотеки зазвичай більше сфокусовані на вирішенні конкретних завдань, таких як маніпуляції з DOM, обробка подій або керування AJAX-запитами.
- Прикладами бібліотек JavaScript є jQuery та React.
- Фреймворки, натомість, забезпечують більш визначену структуру для створення застосунків. Вони часто мають набір правил і конвенцій, яких розробники повинні дотримуватися.
- Прикладами фреймворків є Angular та Next.js, мета-фреймворк для React.
- **Односторінкові застосунки** (SPA) — це вебзастосунки, які завантажують одну HTML-сторінку і динамічно оновлюють її під час взаємодії користувача без перезавантаження всієї сторінки.
- SPA використовують JavaScript для керування станом застосунку та відтворення вмісту. Часто це робиться за допомогою фреймворків, які надають чудові інструменти для створення складних інтерфейсів користувача.
- Деякі проблеми, пов’язані з SPA:

- Читачі екрана мають труднощі з динамічно оновлюваним вмістом.
  - URL не змінюється, коли користувач переходить у межах застосунку, що ускладнює створення закладок, повернення назад або спільний доступ до конкретних сторінок.
  - Початковий час завантаження може бути повільним, якщо застосунок великий, оскільки всі ресурси потрібно завантажити одразу.

## React

- React — популярна бібліотека JavaScript для створення інтерфейсів користувача та веб-застосунків.
- Основна концепція React — створення багаторазових UI-компонентів, які можуть оновлюватися та відтворюватися незалежно при зміні даних.
- React дозволяє розробникам описувати, як має виглядати UI на основі стану застосунку. Потім React оновлює та відтворює потрібні компоненти, коли дані або стан змінюються.

## Компоненти React

- Компоненти — це будівельні блоки React-застосунків, які дозволяють розбивати складні інтерфейси на менші, керовані частини.
- UI описується за допомогою JSX — розширення синтаксису JavaScript, що дозволяє писати код, схожий на HTML, всередині JavaScript.
- Компоненти — це, по суті, JS-функції або класи, які повертають частину UI.

Ось приклад простого компонента React, який відтворює привітальне повідомлення:

```jsx
function Greeting() {
  const name = 'Anna';
  return <h1>Welcome, {name}!</h1>;
}
```

Щоб використати компонент, ви можете просто викликати:
    
```jsx
  <Greeting />
```

## Імпорт і експорт компонентів React

- Експорт робить компонент доступним для інших файлів. Імпорт дозволяє файлу використовувати компонент, який був експортований в іншому місці.
- Існує два типи експорту в React: експорт за замовчуванням і іменований експорт.

**Експорт за замовчуванням**

- Файл може мати лише один експорт за замовчуванням. Це ідеально для файлу, який містить переважно один компонент.
- Ви можете експортувати компонент як експорт за замовчуванням ось так:

```jsx
// City.js
function City() {
  return <p>New York</p>;
}

export default City;
```

- Ви також можете експортувати на тій же лінії, де визначаєте компонент:

```jsx
export default function City() {
  return <p>New York</p>;
}
```

- Щоб імпортувати експорт за замовчуванням, використовуйте ключове слово `import` без фігурних дужок:

```jsx
// App.js
import City from './City';

function App() {
  return (
    <div>
      <h1>My favorite city is:</h1>
      <City />
    </div>
  );
}
```

**Іменовані експорти**

- Іменовані експорти дозволяють файлу ділитися кількома компонентами. На відміну від експорту за замовчуванням, їх потрібно імпортувати з точною назвою, з якою вони були експортовані.
- Ви можете експортувати кілька компонентів з одного файлу ось так:

```jsx
// Animals.jsx
export function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- Щоб імпортувати іменовані експорти, використовуйте фігурні дужки з точними іменами експорту:

```jsx
import { Cat, Dog } from './Animals';
```

- Ви можете перейменувати іменований експорт під час імпорту, використовуючи ключове слово `as`:

```jsx
import { Cat as Kitty } from './Animals';
```

**Змішаний експорт за замовчуванням і іменовані експорти**

- Файл може мати один експорт за замовчуванням і кілька іменованих експортів одночасно:

```jsx
// Animals.jsx
export default function Cat() {
  return <h2>Mr. Whiskers</h2>;
}

export function Dog() {
  return <h2>Fido</h2>;
}
```

- При імпорті змішаних експортів експорт за замовчуванням іде першим (без дужок), а іменовані експорти — після нього (всередині дужок):

```jsx
import Cat, { Dog } from './Animals';
```

## Налаштування проєкту React за допомогою Vite

- Інструменти налаштування проєктів і CLI забезпечують швидкий і простий спосіб почати нові проєкти, дозволяючи розробникам зосередитися на написанні коду, а не на конфігурації.
- Vite — популярний інструмент налаштування проєктів, який можна використовувати з React.
- Щоб створити новий проєкт за допомогою Vite, використайте таку команду у вашому терміналі:

```bash
npm create vite@latest my-react-app -- --template react
```

Ця команда створює новий проєкт React з назвою `my-react-app` за допомогою шаблону React від Vite. У директорії проєкту ви побачите файл `package.json` із переліком залежностей і команд.

- Щоб запустити проєкт, перейдіть у директорію проєкту і виконайте такі команди:

```bash
cd my-react-app # path to the project directory
npm install # installs the dependencies listed in the package.json file
```

- Після встановлення залежностей ви помітите нову папку у вашому проєкті з назвою `node_modules`.

- Папка `node_modules` містить усі пакети та бібліотеки, необхідні для вашого проєкту.

- Щоб запустити проєкт, використайте таку команду:

```bash
npm run dev
```

- Потім відкрийте браузер і перейдіть за адресою `http://localhost:5173`, щоб побачити, як працює ваш React-застосунок.

- Щоб переглянути код стартового шаблону, зайдіть у папку `src` у вашому проєкті, де ви знайдете файл `App.jsx`.

## Передача пропсів у компоненти React

- У React пропси (скорочено від properties) — це спосіб передати дані від батьківського компонента до дочірнього. Цей механізм потрібен для створення багаторазових і динамічних UI-елементів.
- Пропси можуть бути будь-яким значенням JavaScript. Щоб передати пропси від батька до дитини, додайте їх як атрибути під час використання дочірнього компонента в JSX батька. Ось простий приклад:

```jsx
// Parent component
function Parent() {
  const name = 'Anna';
  return <Child name={name} />;
}

// Child component
function Child(props) {
  return <h1>Hello, {props.name}!</h1>;
}
```

Ви можете передати кілька пропсів, використовуючи оператор розширення `(...)`, попередньо перетворивши їх на об’єкт. Ось приклад:
    
```jsx
// Parent component
function Parent() {
  const person = {
    name: 'Anna',
    age: 25,
    city: 'New York'
  };
  return <Child {...person} />;
}
```

У цьому коді оператор розширення `{...person}` перетворює об’єкт person на окремі пропси, які передаються компоненту Child.

## Умовне відтворення в React

- Умовне відтворення в React дозволяє створювати динамічні інтерфейси користувача. Його використовують, щоб показувати різний вміст залежно від певних умов або станів у застосунку.
- Існує кілька способів умовно відтворювати вміст у React. Один із поширених — використання тернарного оператора. Ось приклад:

```jsx
function Greeting({ isLoggedIn }) {
  return (
    <div>
      {isLoggedIn ? <h1>Welcome back!</h1> : <h1>Please log in</h1>}
    </div>
  );
}
```

- Інший спосіб умовного відтворення — використання логічного оператора AND (`&&`). Це корисно, коли потрібно відтворити вміст лише за умови, що певна умова виконується. Ось приклад:

```jsx
function Greeting({ user }) {
  return (
    <div>
      {user && <h1>Welcome, {user.name}!</h1>}
    </div>
  );
}
```

У наведеному коді елемент `h1` відтворюється лише якщо об’єкт user істинний.

Ви також можете використати пряму інструкцію `if` таким чином:

```js
function Greeting({ isLoggedIn }) {
  if (isLoggedIn) {
    return <h1>Welcome back!</h1>;
  }
  return <h1>Please sign in</h1>;
}
```

## Відтворення списків у React

- Відтворення списків у React — це поширене завдання при створенні інтерфейсів користувача.
- Списки можна відтворювати, використовуючи метод масиву JS `map()` для ітерації над масивом елементів і повернення нового масиву JSX-елементів.
- Наприклад, якщо у вас є масив імен, які потрібно відтворити у вигляді списку, ви можете зробити так:

```jsx
function NameList({ names }) {
  return (
    <ul>
      {names.map((name, index) => (
        <li key={`${name}-${index}`}>{name}</li>
      ))}
    </ul>
  );
}
```

- Завжди пам’ятайте надавати унікальний ключ для кожного елемента списку, щоб допомогти React керувати оновленням і відтворенням. Використовуючи ці техніки, ви можете створювати гнучкі, ефективні та динамічні списки у ваших React-застосунках.

## Рядкові стилі в React

- Рядкові стилі в React дозволяють застосовувати CSS-стилі безпосередньо до JSX-елементів за допомогою об’єктів JavaScript.
- Щоб застосувати рядкові стилі в React, використовуйте атрибут style на JSX-елементах. Атрибут style приймає об’єкт, де ключі — це CSS-властивості у верблюжому регістрі, а значення — відповідні значення. Ось приклад:

```js
function Greeting() {
  return (
    <h1
      style={{ color: 'blue', fontSize: '24px', backgroundColor: 'lightgray' }}
    >
      Hello, world!
    </h1>
  );
}

export default Greeting;
```

Ви також можете винести стилі в окремий об’єкт і посилатися на нього в атрибуті `style` ось так:

```jsx
function Greeting() {

  const styles = {
    color: 'blue',
    fontSize: '24px',
    backgroundColor: 'lightgray'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- Рядкові стилі підтримують динамічне стилізування, дозволяючи умовно застосовувати стилі на основі пропсів або стану. Ось приклад, як умовно застосувати стилі на основі пропса:

```jsx
function Greeting({ isImportant }) {

  const styles = {
    color: isImportant ? 'red' : 'black',
    fontSize: isImportant ? '24px' : '16px'
  };

  return <h1 style={styles}>Hello, world!</h1>;
}

export default Greeting;
```

- У наведеному коді стилі `color` і `fontSize` умовно встановлюються залежно від пропса `isImportant`.

## Робота з подіями у React

- **Синтетична система подій**: Це спосіб обробки подій у React. Вона слугує обгорткою над нативними подіями, такими як `click`, `keydown` та `submit`. Обробники подій у React використовують конвенцію іменування у camelCase. (Наприклад, `onClick`, `onSubmit` тощо)

Ось приклад використання атрибута `onClick` для елемента `button` у React:

```jsx
function handleClick() {
  console.log("Button clicked!");
}

<button onClick={handleClick}>Click Me</button>;
```

У React функції-обробники подій зазвичай починаються з префікса `handle`, щоб позначити, що вони відповідають за обробку подій, наприклад `handleClick` або `handleSubmit`.

Коли користувацька дія викликає подію, React передає обробнику об’єкт Synthetic Event. Цей об’єкт поводиться подібно до нативного об’єкта Event у чистому JavaScript, надаючи властивості, такі як `type`, `target` та `currentTarget`.

Щоб запобігти стандартній поведінці, наприклад оновленню сторінки браузера під час події `onSubmit`, можна викликати метод `preventDefault()`:

```jsx
function handleSubmit(event) {
  event.preventDefault();
  console.log("Form submitted!");
}

<form onSubmit={handleSubmit}>
  <input type="text" />
  <button>Submit</button>
</form>;
```

Також можна обгорнути функцію-обробник у стрілкову функцію ось так:

```jsx
function handleDelete(id) {
  console.log("Deleting item:", id);
}

<button onClick={() => handleDelete(1)}>Delete Item</button>;
```

## Робота зі станом і хуком `useState`

- **Визначення стану**: У React стан — це об’єкт, який містить дані для компонента. Коли стан оновлюється, компонент повторно відтворюється. React вважає стан незмінним, тобто його не слід змінювати напряму.
- **Хук `useState()`**: Хук `useState` — це функція, яка дозволяє оголошувати змінні стану у функціональних компонентах. Ось базовий синтаксис:

```js
const [stateVariable, setStateFunction] = useState(initialValue);
```

У змінній стану є такі елементи:

- `stateVariable` містить поточне значення стану
- `setStateFunction` (функція-присвоювач) оновлює змінну стану
- `initialValue` задає початковий стан

Ось повний приклад для компонента `Counter`:

```jsx
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>

      <button onClick={() => setCount(count - 1)}>Decrement</button>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

export default Counter;
```

## Відтворення та компоненти React

- **Визначення**: У React відтворення — це процес, під час якого компоненти з’являються в інтерфейсі користувача (UI), зазвичай у браузері. Процес відтворення складається з трьох етапів: тригер, відтворення та коміт.

Етап тригера відбувається, коли React виявляє зміну, і інтерфейс користувача (UI) може потребувати оновлення. Зазвичай ця зміна пов’язана з оновленням стану або пропсів.

Після тригера React переходить до етапу відтворення. Тут React повторно оцінює ваші компоненти і визначає, що потрібно показати. Для цього React використовує легку копію «справжнього» DOM, звану віртуальним DOM. Завдяки віртуальному DOM React може швидко перевірити, що потрібно змінити в компоненті.

Етап коміту — це момент, коли React застосовує підготовлені зміни з віртуального DOM до справжнього DOM. Іншими словами, це етап, коли ви бачите фінальний результат на екрані.

## Оновлення об’єктів і масивів у стані

- **Оновлення об’єктів у стані**: Якщо потрібно оновити об’єкт у стані, спочатку слід створити новий об’єкт або скопіювати існуючий, а потім задати стан для цього нового об’єкта. Будь-який об’єкт, який поміщають у стан, слід вважати лише для читання. Ось приклад встановлення імені, віку та міста користувача. Функція `handleChange` використовується для обробки оновлень інформації користувача:

```jsx
import { useState } from "react";

function Profile() {
  const [user, setUser] = useState({ name: "John Doe", age: 31, city: "LA" });

  const handleChange = (e) => {
    const { name, value } = e.target;

    setUser((prevUser) => ({...prevUser, [name]: value}));
  };

  return (
    <div>
      <h1>User Profile</h1>
      <p>Name: {user.name}</p>
      <p>Age: {user.age}</p>
      <p>City: {user.city}</p>

      <h2>Update User Age </h2>
      <input type="number" name="age" value={user.age} onChange={handleChange} />

      <h2>Update User Name </h2>
      <input type="text" name="name" value={user.name} onChange={handleChange} />

      <h2>Update User City </h2>
      <input type="text" name="city" value={user.city} onChange={handleChange} />
    </div>
  );
}

export default Profile;
```

- **Оновлення масивів у стані**: При оновленні масивів у стані важливо не змінювати масив безпосередньо за допомогою методів `push()` або `pop()`. Замість цього слід створити новий масив під час оновлення стану:

```jsx
const addItem = () => {
  const newItem = {
    id: items.length + 1,
    name: `Item ${items.length + 1}`,
  };

  // Creates a new array
  setItems((prevItems) => [...prevItems, newItem]);
};
```

Якщо потрібно видалити елементи з масиву, слід використовувати метод `filter()`, який повертає новий масив після фільтрації того, що потрібно видалити:

```jsx
const removeItem = (id) => {
  setItems((prevItems) => prevItems.filter((item) => item.id !== id));
};
```

## Посилання на значення за допомогою Ref

- **Атрибут `ref`**: Ви можете отримати доступ до DOM-вузла у React, використовуючи атрибут `ref`. Ось приклад, який демонструє `ref` для фокусу на елементі `input`. Властивість `current` використовується для доступу до поточного значення цього `ref`:

```jsx
import { useRef } from "react";

const Focus = () => {
  const inputRef = useRef(null);

  const handleFocus = () => {
    if (inputRef.current) {
      inputRef.current.focus();
    }
  };

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Enter text" />
      <button onClick={handleFocus}>Focus Input</button>
    </div>
  );
};

export default Focus;
```

## Робота з хуком `useEffect`

- **Хук `useEffect()`**: У React ефект — це все, що відбувається поза процесом відтворення компонента. Тобто все, що React не обробляє безпосередньо як частину відтворення UI. Загальні приклади включають отримання даних, оновлення заголовка вкладки браузера, читання або запис у локальне сховище браузера, визначення місцезнаходження користувача тощо. Ці операції взаємодіють із зовнішнім світом і відомі як побічні ефекти. React надає хук `useEffect`, щоб ви могли обробляти ці побічні ефекти. `useEffect` дозволяє запускати функцію після відтворення або оновлення компонента.

```js
import { useEffect } from "react";

useEffect(() => {
 // Your side effect logic (usually a function) goes here
}, [dependencies]);
```

Функція ефекту виконується після відтворення компонента, а необов’язковий аргумент `dependencies` контролює, коли ефект запускається.

Зверніть увагу, що `dependencies` може бути масивом «реактивних значень» (стан, пропси, функції, змінні тощо), порожнім масивом або зовсім відсутнім. Ось як усі ці варіанти контролюють роботу `useEffect`:

- Якщо `dependencies` — це масив, що містить одне або кілька реактивних значень, ефект запускається щоразу, коли вони змінюються.

- Якщо `dependencies` — порожній масив, `useEffect` запускається лише один раз, коли компонент вперше відтворюється.

- Якщо ви опускаєте `dependencies`, ефект запускається щоразу, коли компонент відтворюється або оновлюється.

## Як створювати кастомні хуки

- **Кастомні хуки**: Кастомний хук дозволяє винести повторно використовувану логіку з компонентів, наприклад отримання даних, керування станом, перемикання та побічні ефекти, як-от відстеження статусу онлайн. У React усі вбудовані хуки починаються зі слова `use`, тому ваш кастомний хук має дотримуватися цієї конвенції.

Ось приклад створення хуку `useDebounce`:

```jsx
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);

  return debouncedValue;
}

export { useDebounce };
```

## Робота з формами в React

- **Керовані поля введення**: Це коли ви зберігаєте значення поля введення у стані і оновлюєте його через `onChange` події. Це дає вам повний контроль над даними форми та дозволяє миттєву валідацію і умовне відтворення.

```jsx
import { useState } from "react";

function App() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <>
      <form onSubmit={handleSubmit}>
        <label htmlFor="name">Your name</label> <br />
        <input value={name} id="name" onChange={handleChange} type="text" />
        <button type="submit">Submit</button>
      </form>
    </>
  );
}

export default App;
```

- **Некеровані поля введення**: Замість обробки полів введення через `useState` хук, некеровані поля в HTML підтримують власний внутрішній стан за допомогою DOM. Оскільки DOM контролює значення полів введення, вам потрібно отримувати значення цих полів за допомогою `ref`.

```jsx
import { useRef } from "react";

function App() {
 const nameRef = useRef();

 const handleSubmit = (e) => {
   e.preventDefault();
   console.log(nameRef.current.value);
 };

 return (
   <form onSubmit={handleSubmit}>
     <label htmlFor="name">Your</label>{" "}
     <input type="text" ref={nameRef} id="name" />
     <button type="submit">Submit</button>
   </form>
 );
}

export default App;
```

## Робота з хуком `useActionState`

- **Серверні дії**: Це функції, які виконуються на сервері, щоб дозволити обробку форм безпосередньо на сервері без потреби в API кінцевих точках. Ось приклад із застосунку Next.js:

```js
"use server";

async function submitForm(formData) {
 const name = formData.get("name");
 return { message: `Hello, ${name}!` };
}
```

Директива `"use server"` позначає функцію як серверну дію.

- **Хук `useActionState`**: Цей хук оновлює стан залежно від результату відправлення форми. Ось базовий синтаксис хуку `useActionState`:

```js
const [state, action, isPending] = useActionState(actionFunction, initialState, permalink);
```

- `state` — це поточний стан, який повертає дія.
- `action` — функція, що запускає серверну дію.
- `isPending` — булевий параметр, який вказує, чи виконується дія зараз.
- Параметр `actionFunction` — сама серверна дія.
- `initialState` — параметр, що представляє початковий стан перед виконанням дії.
- `permalink` — необов’язковий рядок, який містить унікальну URL-адресу сторінки, яку змінює форма.

## Отримання даних у React

- **Варіанти отримання даних**: Існує багато способів отримати дані в React. Ви можете використовувати нативний Fetch API або сторонні інструменти, такі як Axios чи SWR.
- **Загальні змінні стану при отриманні даних**: Незалежно від того, який спосіб отримання даних ви оберете в React, вам потрібно відслідковувати кілька станів. Перший — це самі дані. Другий — чи ще триває отримання даних. Третій — змінна стану, яка зберігатиме будь-які помилки, що можуть виникнути під час отримання даних.

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);
```

Оскільки отримання даних є побічним ефектом, найкраще використовувати `Fetch API` всередині хуку `useEffect`.

```js
useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      
      if (!res.ok) {
        throw new Error("Network response was not ok");
      }

      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```

Потім ви можете відтворити повідомлення про завантаження, якщо отримання даних не завершено, повідомлення про помилку, якщо сталася помилка, або результати.

```jsx
if (loading) {
  return <p>Loading...</p>;
}

if (error) {
  return <p>{error.message}</p>;
}

return (
  <ul>
    {data.map((post) => (
      <li key={post.id}>{post.title}</li>
    ))}
  </ul>
);
```

Якщо хочете використовувати Axios, потрібно встановити та імпортувати його:

```bash
npm i axios
```

```js
import axios from "axios";
```

Потім можна отримувати дані за допомогою `axios.get`:

```js
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);

useEffect(() => {
  const fetchData = async () => {
    try {
      const res = await axios.get(
        "https://jsonplaceholder.typicode.com/users"
      );
      setData(res.data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };

  fetchData();
}, []);
```



```jsx
const FetchTodos = () => {

 );

 if (!data) {

 }
 if (error) {
   return <h2>Error: {error.message}</h2>;
 }

 return (
   <>
     <h2>Todos</h2>

 );
};

export default FetchTodos;
```

## Робота з хуком `useOptimistic`

- **Хук `useOptimistic`**: Цей хук використовується, щоб інтерфейси залишалися адаптивними, поки асинхронна дія виконується у фоновому режимі. Він допомагає керувати «оптимістичними оновленнями» в UI — стратегією, коли ви одразу оновлюєте інтерфейс, очікуючи результат дії, наприклад, відповіді сервера.

Ось базовий синтаксис:

```js
const [optimisticState, addOptimistic] = useOptimistic(actualState, updateFunction);
```

- `optimisticState` — тимчасовий стан, який оновлюється одразу для кращого досвіду користувача.
- `addOptimistic` — функція, що застосовує оптимістичне оновлення перед фактичними змінами стану.
- `actualState` — реальне значення стану, яке приходить із результату дії, наприклад, отримання даних із сервера.
- `updateFunction` — функція, що визначає, як має оновлюватися оптимістичний стан при виклику.

Ось приклад використання хуку `useOptimistic` у компоненті `TaskList`:

```js
"use client";

import { useOptimistic } from "react";

export default function TaskList({ tasks, addTask }) {
  const [optimisticTasks, addOptimisticTask] = useOptimistic(
    tasks,
    (state, newTask) => [...state, { text: newTask, pending: true }]
  );

  async function handleSubmit(e) {
    e.preventDefault();
    const formData = new FormData(e.target);

    addOptimisticTask(formData.get("task"));

    addTask(formData);
    e.target.reset();
  }

  return <>{/* UI */}</>;
}
```

- **`startTransition`**: Використовується для відтворення частини UI та позначення оновлення стану як не термінового переходу. Це дозволяє інтерфейсу залишатися адаптивним під час дорогих оновлень. Ось базовий синтаксис:

```js
startTransition(action)
```

`action` виконує оновлення стану або запускає логіку, пов’язану з переходом. Це гарантує, що термінові оновлення UI (наприклад, введення тексту або натискання) не блокуються.

## Робота з хуком `useMemo`

```js
const memoizedSortedArray = useMemo(
  () => expensiveSortFunction(largeArray),
  [largeArray]
);
```

## Інструменти управління залежностями

- **Визначення залежності**: У програмному забезпеченні залежність — це коли один компонент або модуль у застосунку покладається на інший для правильної роботи. Залежності поширені в програмних застосунках, бо дозволяють розробникам використовувати готові функції або інструменти, створені іншими. Дві основні залежності для проєкту React — це пакети `react` і `react-dom`:

```json
"dependencies": {
  "react": "^18.3.1",
  "react-dom": "^18.3.1"
}
```

- **Визначення менеджера пакетів**: Щоб керувати залежностями в проєкті, потрібно використовувати менеджер пакетів. Менеджер пакетів — це інструмент для встановлення, оновлення та видалення залежностей. Багато популярних мов програмування, таких як JavaScript, Python, Ruby і Java, використовують менеджери пакетів. Популярні менеджери пакетів для JavaScript — npm, Yarn і pnpm.
- **Файл `package.json`**: Це ключовий конфігураційний файл у проєктах, який містить метадані про ваш проєкт, включно з його назвою, версією та залежностями. Він також визначає скрипти, інформацію про ліцензію та інші налаштування, що допомагають керувати проєктом і його залежностями.
- **Файл `package-lock.json`**: Цей файл фіксує точні версії всіх пакетів, які використовує ваш проєкт. Коли ви оновлюєте пакет, нові версії також оновлюються у файлі блокування.
- **Папка `node_modules`**: Ця папка містить фактичний код залежностей, перелічених у вашому файлі `package.json`, включно з прямими залежностями проєкту та залежностями цих залежностей.
- **Залежності для розробки**: Це пакети, які використовуються лише під час розробки, а не в продакшені. Приклад — бібліотека для тестування Jest. Ви встановлюєте Jest як залежність для розробки, бо вона потрібна для локального тестування застосунку, але не потрібна для запуску застосунку в продакшені.

## React маршрутизатор

- **Вступ**: React Router — це стороння бібліотека, яка дозволяє додати маршрутизацію у ваші React-застосунки. Щоб почати, потрібно встановити React Router у існуючий проєкт React так:

```bash
npm i react-router
```

Потім у файлі `main.jsx` або `index.jsx` потрібно налаштувати структуру маршрутів так:

```jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { BrowserRouter, Routes, Route } from "react-router";
import App from "./App.jsx";

import "./index.css";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<App />} />
      </Routes>
    </BrowserRouter>
  </StrictMode>
);
```

`path` і `element` використовуються для зв’язування URL і UI-компонентів. У цьому випадку ми налаштовуємо маршрут для головної сторінки, який вказує на компонент `App`.

- **Кілька переглядів і налаштування маршрутів**: У більших застосунках часто налаштовують кілька переглядів і маршрутів так:

```jsx
<Routes>
  <Route index element={<Home />} />
  <Route path="about" element={<About />} />

  <Route path="products">
    <Route index element={<ProductsHome />} />
    <Route path=":category" element={<Category />} />
    <Route path=":category/:productId" element={<ProductDetail />} />
    <Route path="trending" element={<Trending />} />
  </Route>
</Routes>
```

Проп `index` у цих прикладах означає маршрут за замовчуванням для певного сегмента шляху. Тож компонент `Home` відображатиметься за кореневим шляхом `/`, а компонент `ProductsHome` — за шляхом `/products`.

- **Вкладені маршрути**: Ви можете вкладати маршрути всередину інших маршрутів, що призводить до додавання шляху дочірнього маршруту до шляху батьківського.

```jsx
<Route path="products">
  <Route path="trending" element={<Trending />} />
</Route>
```

У наведеному прикладі шлях для популярних товарів буде products/trending.

- **Динамічні сегменти**: Динамічний сегмент — це частина URL-шляху, яка є динамічною.

```jsx
<Route path=":category" element={<Category />} />
```

У цьому прикладі є динамічний сегмент `category`. Коли користувач переходить за URL, наприклад products/brass-instruments, перегляд змінюється на компонент `Category`, і ви можете динамічно отримувати відповідні дані на основі сегмента.

- **Хук `useParams`**: Цей хук використовується для доступу до динамічних параметрів із URL-шляху.

```js
import { useParams } from "react-router";

export default function Category() {
  let params = useParams();
   {/* Accessing the category param: params.category */}
   {/* rest of code goes here */}
}
```

```js
}
```

- **Оптимізація зображень у Next.js**: Компонент `Image` розширює нативний HTML-елемент `img` і дозволяє швидше завантажувати сторінки та оптимізувати розмір зображень. Це означає, що зображення завантажуються лише тоді, коли вони потрапляють у вікно перегляду, а компонент `Image` автоматично подає зображення правильного розміру для кожного пристрою.

```jsx
import Image from "next/image";

export default function Page() {
  return (
    <Image src="link-to-image-goes-here" alt="descriptive-title-goes-here" />
  );
}
``` 

## Проп дрилінг

- **Визначення**: Проп дрилінг — це процес передачі пропсів від батьківського компонента до глибоко вкладених дочірніх компонентів, навіть якщо деякі дочірні компоненти не потребують цих пропсів.

## Управління станом

- **Context API**: Контекст означає, що батьківський компонент робить інформацію доступною для дочірніх компонентів без необхідності явно передавати її через пропси. `createContext` використовується для створення об’єкта контексту, який представляє контекст, що інші компоненти будуть читати. `Provider` використовується для надання значень контексту дочірнім компонентам.

```js
import { useState, createContext } from "react";

const CounterContext = createContext();

const CounterProvider = ({ children }) => {
  const [count, setCount] = useState(0);

  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export { CounterContext, CounterProvider };
```

- **Redux**: Redux керує станом, надаючи центральне сховище і суворий контроль над оновленнями стану. Він використовує передбачуваний шаблон із діями, редюсерами та підпрограмним забезпеченням. Дії — це корисне навантаження інформації, яке надсилає дані з вашого застосунку до сховища Redux, часто викликане взаємодією користувача. Редюсери — це функції, які визначають, як має змінюватися стан у відповідь на ці дії, забезпечуючи оновлення стану без мутацій. Підпрограмне забезпечення виступає мостом між відправленням дій і редюсером, дозволяючи розширювати функціональність Redux (наприклад, логування, обробка асинхронних операцій) без зміни основного потоку.

- **Zustand**: Це рішення для управління станом ідеально підходить для застосунків малого та середнього масштабу. Воно працює за допомогою хуку `useStore`, який дає доступ до стану безпосередньо в компонентах і сторінках. Це дозволяє змінювати і отримувати дані без потреби в діях, редюсерах або провайдері.

- **Вкладка Components**: Ця вкладка відображає кожен компонент у вигляді дерева. Ось що можна робити у цій вкладці:

## Відмінності між реальною та сприйманою продуктивністю

- **Сприймана продуктивність**: Це те, як користувачі сприймають продуктивність вебсайту. Це їхня оцінка з точки зору відгуку та надійності. Це суб’єктивна міра, тому її важко кількісно визначити, але вона дуже важлива, оскільки користувацький досвід визначає успіх або провал сайту.
- **Реальна продуктивність**: Це об’єктивна та вимірювана продуктивність сайту. Вона вимірюється за допомогою метрик, таких як час завантаження сторінки, час відповіді сервера та час відтворення. На ці вимірювання впливають численні фактори, пов’язані з мережею та самим кодом.

## Техніки покращення сприйманої продуктивності

- **Ліниве завантаження**: Ця техніка максимально зменшує початковий час завантаження, завантажуючи несуттєві ресурси у фоновому режимі.
- **Мінімізація затримок завантаження шрифтів**: Якщо на вашому сайті є кастомні шрифти, варто також мінімізувати затримки їх завантаження, оскільки це може спричинити мерехтіння або показ запасного шрифту, поки кастомний шрифт завантажується. Рекомендація — використовувати запасний шрифт, схожий на кастомний, щоб у разі такого випадку зміна була більш непомітною.
- **Використання індикаторів завантаження**: Показ індикатора завантаження під час тривалого процесу одразу після того, як користувач натисне на елемент, допомагає користувачу відчути зв’язок і залученість у процес, роблячи час очікування коротшим.

## Основні поняття продуктивності

- **Порядок джерел**: Це стосується того, як HTML-елементи структуровані в документі. Це визначає, що завантажується першим і може суттєво впливати на продуктивність і доступність.

Деякі найкращі практики для порядку джерел включають:

- Розміщення критичного вмісту, такого як заголовки, навігація або основний текст, вище в структурі HTML.
- Відкладення несуттєвих скриптів, наприклад аналітики або віджетів сторонніх розробників, щоб вони не блокували відтворення.
- Використання прогресивного покращення, щоб забезпечити роботу основного досвіду навіть до завантаження стилів і скриптів. Прогресивне покращення — це спосіб створення сайтів і застосунків на основі ідеї, що спочатку сторінка має працювати з HTML.

Ось приклад хорошого порядку джерел із використанням найкращих практик, які ми щойно розглянули:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **Критичний шлях відтворення**: Це послідовність кроків, які браузер виконує, щоб перетворити код у пікселі на екрані.
- **Затримка**: Це час, який потрібен запиту, щоб пройти між браузером і сервером. Іншими словами, висока затримка означає повільні сторінки.

Деякі способи зменшення затримки включають:

- Використання CDN, або мереж доставки контенту, щоб обслуговувати файли з ближчих локацій.
- Увімкнення стиснення за допомогою таких інструментів, як Gzip, щоб зменшити розмір файлів.
- Оптимізація зображень і використання лінивого завантаження.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## Покращення INP

- **Визначення**: INP (Interaction to Next Paint) оцінює загальну відзивність сторінки, вимірюючи час від взаємодії користувача, наприклад натискання або натискання клавіші, до наступного оновлення відображення браузером. Нижчий INP означає більш відзивну сторінку.

Ось кілька способів покращити INP:

- Зменшити навантаження на головний потік, розбиваючи довгі завдання JavaScript.
- Використовувати `requestIdleCallback()` для несуттєвих скриптів. Це поставить функцію в чергу для виклику під час простоїв браузера.
- Відкладати або ліниво завантажувати важкі ресурси, про які йшлося раніше.
- Оптимізувати обробники подій. Якщо ці обробники запускаються надто часто або виконують важкі операції, вони можуть уповільнювати сторінку і збільшувати INP. Рішення — дебаунсинг. Дебаунсинг гарантує, що функція виконується лише після того, як користувач припиняє вводити текст на короткий час — наприклад, 300 мс. Це запобігає зайвим обчисленням і покращує продуктивність.

## Як відбувається відтворення в браузері

- **Як працює відтворення**: Спочатку браузер аналізує HTML і будує DOM. Потім браузер обробляє CSS, створюючи CSS Object Model, або CSSOM. Це ще одна дерево-подібна структура, яка визначає, як елементи мають бути стилізовані. Нарешті, браузер відтворює пікселі на екрані, відтворюючи кожен елемент на основі обчислених стилів і макету. На складних сторінках це може включати кілька шарів, які композитяться разом, формуючи фінальний візуальний результат.

## Як продуктивність впливає на сталий розвиток

- **Фонова інформація**: Інтернет відповідає приблизно за 2% світових викидів вуглецю — це стільки ж, скільки авіаційна галузь! Кожен байт, що передається, потребує електроенергії — від дата-центрів до пристроїв користувачів. Великі файли та неефективні скрипти означають більше споживання енергії. Високопродуктивний сайт не лише швидший, а й зменшує непотрібну обробку та використання енергії.

## Способи зменшення часу завантаження сторінки

- **Оптимізація медіаресурсів**: Великі зображення та відео часто є причиною повільного завантаження. Оптимізуючи ці ресурси, ви можете значно прискорити сайт. Це включає стиснення зображень, використання сучасних форматів, таких як WebP, і ліниве завантаження ресурсів.
- **Використання кешування браузера**: Кешування дозволяє браузерам зберігати частини вашого сайту локально, зменшуючи час завантаження для повторних відвідувачів.
- **Мінімізація та стиснення файлів**: Зменшення розміру файлів призводить до швидшого завантаження. Це включає зменшення розміру передаваних файлів і мінімізацію CSS та JavaScript.

## Покращення "time to usable"

- **Визначення**: "time to usable" — це інтервал від моменту запиту сторінки користувачем до моменту, коли він може ефективно взаємодіяти з нею. Щоб покращити "time to usable", можна ліниво завантажувати ресурси або мінімізувати ресурси, що блокують відтворення.

## Ключові метрики для вимірювання продуктивності

- **First Contentful Paint або FCP**: Вимірює, як швидко на екрані з’являється перший фрагмент вмісту — текст або зображення. Хороший FCP — це час менше 1,8 секунди, а поганий — понад 3 секунди. Ви можете перевірити свій FCP за допомогою Chrome DevTools у вкладці продуктивності.
- **Total Blocking Time**: Показує, скільки часу головний потік заблокований важкими завданнями JavaScript. Якщо Total Blocking Time (TBT) високий, користувачі відчувають затримки у взаємодії. Щоб покращити TBT, розбивайте довгі завдання і відкладайте несуттєві скрипти.
- **Bounce Rate**: Це відсоток відвідувачів, які покидають сайт, не взаємодіючи з ним. Якщо у вас високий Bounce Rate, можливо, сторінка завантажується занадто повільно.
- **Unique Users**: Ця метрика відстежує, скільки унікальних відвідувачів приходить на ваш сайт. Щоб переглянути Bounce Rate і Unique Users, можна використовувати Google Analytics. Це дозволить вам контролювати ці метрики і покращувати залученість.

## Загальні інструменти для вимірювання продуктивності

- **Chrome DevTools**: Chrome DevTools — це вбудований інструмент у Google Chrome, який дозволяє аналізувати та налагоджувати продуктивність у реальному часі. DevTools показує час завантаження, використання центрального процесора та затримки відтворення. Особливо корисний для вимірювання First Contentful Paint, або FCP, який показує, як швидко користувач бачить перший видимий вміст. Якщо ваш сайт здається повільним, DevTools допоможе виявити вузькі місця.
- **Lighthouse**: Це автоматизований інструмент, який перевіряє продуктивність, SEO та доступність.
- **WebPageTest**: Цей інструмент дозволяє тестувати, як ваш сайт завантажується з різних локацій і пристроїв. Він дає детальний розподіл таких метрик, як Speed Index, Total Blocking Time та інших ключових показників продуктивності. Якщо ви хочете знати, як реальні користувачі бачать ваш сайт у світі, WebPageTest — це ваш інструмент.
- **PageSpeed Insights**: Цей інструмент аналізує ваш сайт і пропонує швидкі покращення для мобільних і десктопних пристроїв. Він підкаже, що уповільнює сайт, і дасть конкретні рекомендації, наприклад оптимізувати зображення, видалити скрипти, що блокують відтворення, і зменшити час відповіді сервера. PageSpeed Insights — швидкий і простий спосіб перевірити, як Google бачить продуктивність вашого сайту.
- **Real User Monitoring**: Інструменти RUM відстежують реальну поведінку користувачів, показуючи, як справжні відвідувачі сприймають ваш сайт. Популярні RUM-інструменти включають Google Analytics, який відстежує час завантаження сторінок і Bounce Rate, а також New Relic або Datadog, які моніторять проблеми продуктивності в реальному часі. Якщо вам потрібні дані від реальних користувачів, інструменти RUM є необхідними.

## Робота з Performance Web API

- **Визначення**: Performance Web API дозволяють розробникам відстежувати, наскільки ефективно завантажується і реагує вебсторінка безпосередньо в коді. Ці API дають змогу вимірювати час завантаження сторінки, відстежувати затримки відтворення та взаємодії, а також аналізувати час виконання JavaScript.
- **`performance.now()`**: Цей API надає високоточні часові позначки (у мілісекундах) для вимірювання, скільки часу займають різні частини вашого сайту.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: Цей API дає розподіл кожного етапу завантаження сторінки — від DNS-пошуку до `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: Цей API слухає події продуктивності, такі як зсуви макету, довгі завдання та взаємодії користувача.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## Техніки покращення продуктивності CSS

- **CSS-анімації**: Анімація певних CSS-властивостей, таких як розміри, позиція та макет, запускає процес, який називається "reflow" — під час нього браузер повторно обчислює позицію та геометрію деяких елементів на сторінці. Це вимагає повторного відтворення, що є обчислювально дорогим. Тому рекомендується максимально зменшити кількість CSS-анімацій або принаймні надати користувачу можливість увімкнути чи вимкнути їх.

## Техніки покращення продуктивності JavaScript

- **Розбиття коду**: Розбиття вашого JavaScript-коду на модулі, які виконують критичні та некритичні завдання, також корисне. Так ви зможете завантажувати критичні модулі якомога раніше, а некритичні відкладати, щоб сторінка відтворювалася максимально швидко.
- **Маніпуляції з DOM**: Пам’ятайте, що маніпуляції з DOM — це процес динамічної зміни вмісту сторінки за допомогою JavaScript через взаємодію з Document Object Model (DOM). Маніпуляції з DOM є обчислювально дорогими. Зменшення кількості маніпуляцій з DOM у вашому JavaScript-коді покращить продуктивність.

## CSS-фреймворки

- **CSS-фреймворки**: CSS-фреймворки можуть пришвидшити роботу, створити єдиний візуальний стиль на сайті, забезпечити консистентність дизайну в різних браузерах і зробити CSS-код більш організованим.
- **Популярні CSS-фреймворки**: Серед популярних — Tailwind CSS, Bootstrap, Materialize і Foundation.
- **Можливі недоліки**:
  - CSS від фреймворку може конфліктувати з вашим кастомним CSS.
  - Ваш сайт може виглядати схожим на інші сайти, що використовують той самий фреймворк.
  - Великі фреймворки можуть спричиняти проблеми з продуктивністю.

## Два типи CSS-фреймворків

- **Utility-first CSS-фреймворки**: Ці фреймворки мають невеликі класи з конкретними призначеннями, наприклад для встановлення поля, відступу або кольору фону. Ви можете призначати ці класи безпосередньо HTML-елементам за потреби. Tailwind CSS належить до utility-first CSS-фреймворків.

Ось приклад використання Tailwind CSS для стилізації кнопки.

```html
<button class="bg-blue-500 text-white font-bold py-2 px-4 rounded-full hover:bg-blue-700">
  Button
</button>
```

- **Компонентні CSS-фреймворки**: Ці фреймворки мають готові компоненти з попередньо визначеними стилями, які можна додати на сайт. Компоненти доступні в офіційній документації CSS-фреймворку, і їх можна копіювати і вставляти у проєкт. Bootstrap належить до компонентних CSS-фреймворків.

Ось приклад використання Bootstrap для створення групи списку. Замість додавання маленьких класів до HTML-елементів, ви додаєте весь компонент, включно з HTML-структурою.

```html
<div class="card" style="width: 25rem;">
  <ul class="list-group list-group-flush">
    <li class="list-group-item">HTML</li>
    <li class="list-group-item">CSS</li>
    <li class="list-group-item">JavaScript</li>
  </ul>
</div>
```

## Tailwind CSS

Tailwind — utility-first CSS-фреймворк. Замість написання кастомних CSS-правил ви створюєте дизайн, комбінуючи маленькі утилітарні класи безпосередньо у HTML.

### Утиліти адаптивного дизайну

Tailwind використовує префікси, як `sm:`, `md:` і `lg:`, щоб застосовувати стилі на різних розмірах екрана.

```html
<div class="w-full md:w-1/2 lg:flex-row">
  Responsive layout
</div>
```

### Утиліти Flexbox

Класи, як `flex`, `flex-col`, `justify-around` і `items-center`, полегшують створення гнучких макетів.

```html
<div class="flex flex-col md:flex-row justify-around items-center">
  <p>Column on small screens</p>
  <p>Row on medium and larger screens</p>
</div>
```

### Утиліти Grid

Tailwind включає утиліти для CSS Grid, як `grid`, `grid-cols-1` і `md:grid-cols-3`.

```html
<div class="grid grid-cols-1 md:grid-cols-3 gap-8">
  <div class="bg-gray-100 p-4">Column 1</div>
  <div class="bg-gray-100 p-4">Column 2</div>
  <div class="bg-gray-100 p-4">Column 3</div>
</div>
```

### Утиліти відступів

Утиліти, як `mt-8`, `mx-auto`, `p-4` і `gap-4`, допомагають створювати послідовні відступи без написання CSS.

```html
<div class="mt-8 p-4 bg-indigo-600 text-white">
  Spaced content
</div>
```

### Утиліти типографіки

Утиліти, як `uppercase`, `font-bold`, `font-semibold` і `text-4xl`, керують виглядом тексту.

Ви можете встановлювати розміри шрифтів, які адаптуються на точках зупинки, як `text-3xl` і `md:text-5xl`.

```html
<h1 class="text-3xl md:text-5xl font-semibold text-center">
  Responsive Heading
</h1>
```

### Кольори та стани наведення

Tailwind пропонує широку палітру кольорів, як `text-red-700`, `bg-indigo-600` і `bg-gray-100`.

Класи, як `hover:bg-pink-600`, спрощують інтерактивні ефекти.

```html
<a href="#" class="bg-pink-500 hover:bg-pink-600 text-white px-4 py-2 rounded-md">
  Hover Me
</a>
```

### Кордони, кільця та ефекти

- **Кордони**: `border-2 border-red-300` додає кордони з вказаною товщиною і кольорами.
- **Кільця**: `ring-1 ring-gray-300` створює ефекти, схожі на обриси, часто використовувані для фокусу або карток.
- **Закруглені кути і масштабування**: Класи, як `rounded-md`, `rounded-xl` і `scale-105`, додають елегантності.

```html
<div class="p-6 rounded-xl ring-2 ring-fuchsia-500 scale-105">
  Highlighted card
</div>
```

### Градієнти

Tailwind підтримує утиліти для градієнтів, як `bg-gradient-to-r from-fuchsia-500 to-indigo-600`.

```html
<div class="p-4 text-white bg-gradient-to-r from-fuchsia-500 to-indigo-600">
  Gradient background
</div>
```

## CSS-препроцесори

- **CSS-препроцесор**: Інструмент, що розширює стандартний CSS. Компілірує код із розширеним синтаксисом у нативний CSS-файл. Допомагає писати чистіший, багаторазовий, менш повторюваний і масштабований CSS для складних проєктів.
- **Особливості**: Деякі можливості препроцесорів — змінні, міксіни, вкладеність і успадкування селекторів.
- **Популярні CSS-препроцесори**: Sass, Less і Stylus.
- **Можливі недоліки**:
  - Компіляція CSS може створювати додаткове навантаження.
  - Скомпільований код може бути складним для налагодження.

## Sass

- **Sass**: Один із найпопулярніших CSS-препроцесорів. Sass означає «Syntactically Awesome Style Sheets».
- **Підтримувані можливості Sass**: Змінні, вкладені CSS-правила, модулі, міксіни, успадкування і оператори для базових математичних операцій.

## Два синтаксиси Sass

- **Синтаксис SCSS**: SCSS (Sassy CSS) розширює базовий синтаксис CSS. Це найпоширеніший синтаксис для Sass. Файли SCSS мають розширення `.scss`.

Ось приклад оголошення і використання змінної в SCSS.

```scss
$primary-color: #3498eb;

```

- **Відступний синтаксис**: Відступний синтаксис — це оригінальний синтаксис Sass, також відомий як «Sass syntax».

Ось приклад оголошення і використання змінної у відступному синтаксисі.

```sass
$primary-color: #3498eb

header
  background-color: $primary-color
```

### Міксіни

- **Міксіни**: Дозволяють групувати кілька CSS-властивостей і їх значень під одним ім’ям і повторно використовувати цей блок CSS-коду по всьому стилю.

Ось приклад оголошення міксіну у синтаксисі SCSS. У цьому випадку міксін називається `center-flex`. Він містить три CSS-властивості для центрування елементів за допомогою flexbox.

```scss
@mixin center-flex {

}
```

Ось приклад використання оголошеного міксіну.

```scss
section {

  background-color: #3289a8;
}
```

## Ручне та автоматизоване тестування

- **Ручне тестування**: При ручному тестуванні тестувальник вручну проходить кожну частину застосунку і перевіряє різні функції, щоб переконатися, що вони працюють правильно. Якщо під час тестування виявляються помилки, тестувальник повідомляє про них команді розробників, щоб їх виправили.
- **Автоматизоване тестування**: При автоматизованому тестуванні ви можете автоматизувати свої тести, написавши окрему програму, яка перевіряє, чи поводиться застосунок так, як очікується.

## Модульне тестування

- **Модульне тестування**: При модульному тестуванні ви перевіряєте кожну функцію, щоб переконатися, що все працює як очікується. Модульні тести також можуть слугувати формою документації для вашого застосунку, оскільки вони мають представляти очікувану поведінку вашого коду.
- **Принцип єдиної відповідальності**: Принцип єдиної відповідальності рекомендує тримати кожну функцію маленькою і відповідальною за одну задачу.
- **Загальні фреймворки для тестування JavaScript**: Деякі загальні фреймворки для тестування включають Jest, Mocha та Vitest. Jest — популярний фреймворк для модульних тестів.

Ось приклад модульних тестів із використанням Jest.

Спершу ви можете створити функцію, яка відповідає за повернення ново відформатованого рядка:

```javascript
export function getFormattedWord(str) {
  if (!str) return '';
  return str.charAt(0).toUpperCase() + str.slice(1);
}
```

В окремому файлі `getFormattedWord.test.js` ви можете написати тести, щоб перевірити, чи працює функція як очікується. Файл `getFormattedWord.test.js` виглядатиме так:

```javascript
import { getFormattedWord } from "./getFormattedWord.js";
import { test, expect } from "jest";

test('capitalizes the first letter of a word', () => {
  expect(getFormattedWord('hello')).toBe('Hello');
});
```

- **Функція `expect`**: Функція `expect` використовується для тестування значення.
- **Матчер**: Матчер — це функція, яка перевіряє, чи поводиться значення так, як очікується. У наведеному прикладі матчер — `toBe()`. У Jest є різноманітні матчери.

Щоб використовувати Jest, спершу потрібно встановити пакет `jest` за допомогою `npm i jest`. Також потрібно додати скрипт у файл `package.json` ось так:

```js
"scripts": {
  "test": "jest"
},
```

Потім ви можете запустити команду `npm run test`, щоб виконати ваші тести.

## Життєвий цикл розробки програмного забезпечення

- **Різні етапи життєвого циклу розробки програмного забезпечення**:
  - **Етап планування**: Команда розробників збирає вимоги до запропонованої роботи від зацікавлених сторін.
  - **Етап проєктування**: Команда розробників розбиває вимоги і визначає найкращі підходи до рішень.
  - **Етап реалізації**: Команда розробників розбиває вимоги на керовані завдання і реалізує їх.
  - **Етап тестування**: Включає ручне та автоматизоване тестування нової роботи. Іноді команда тестує застосунок протягом усього етапу розробки, щоб виявити та виправити будь-які проблеми.
  - **Етап розгортання**: Команда розгортає нові зміни у збірці або тестовому середовищі.
  - **Етап обслуговування**: Включає виправлення будь-яких проблем, що виникають у користувачів у продуктивному застосунку.

- **Різні моделі життєвого циклу розробки програмного забезпечення**:
  - **Водоспадна модель**: Водоспадна модель передбачає, що кожен етап життєвого циклу має бути завершений перед початком наступного.
  - **Гнучка модель (Agile)**: Гнучка модель зосереджена на ітеративній розробці шляхом розбиття роботи на спринти.

## BDD та TDD

- **TDD**: Розробка через тестування — це методологія, яка наголошує на написанні тестів першою. Написання тестів перед створенням функцій дає розробникам зворотний зв’язок у реальному часі під час розробки.
- **BDD**: Розробка через поведінку — це підхід, який узгоджує серію тестів із бізнес-цілями. Сценарії тестів у BDD мають бути написані мовою, зрозумілою як технічним, так і нетехнічним особам. Приклад такої синтаксичної мови — Gherkin.
- **Фреймворки для BDD**: Прикладами фреймворків для BDD є Cucumber, JBehave та SpecFlow.

## Твердження в модульному тестуванні

- **Твердження**: Твердження використовуються для перевірки, що код поводиться як очікується.
- **Бібліотеки тверджень**: Chai — це часто використовувана бібліотека тверджень. Інші загальні бібліотеки тверджень для JavaScript — `should.js` та `expect.js`.

Ось приклад твердження з використанням Chai, який перевіряє, що повернене значення з функції `addThreeAndFour` дорівнює числу 7:

```js
assert.equal(addThreeAndFour(), 7);
```

- **Найкращі практики**: Незалежно від того, яку бібліотеку тверджень ви використовуєте, слід писати чіткі повідомлення про твердження та помилки, які допоможуть зрозуміти, які тести не проходять і чому.

## Мокінг, фейкінг і стабінг

- **Мокінг**: Мокінг — це процес заміни реальних даних на хибні, які імітують поведінку реальних компонентів. Наприклад, ви можете мокати відповідь API під час тестування замість безперервних викликів API для отримання даних.
- **Стабінг**: Стабси — це об’єкти, які повертають заздалегідь визначені відповіді або фіктивні дані для очікуваної поведінки в застосунку. Наприклад, ви можете стабати поведінку підключення до бази даних у тестах без необхідності покладатися на реальне підключення.
- **Фейкінг**: Фейки — це спрощені версії реальних компонентів без складності чи побічних ефектів. Наприклад, ви можете фейкувати базу даних, зберігаючи дані в пам’яті замість взаємодії з реальною базою даних. Це дозволить імітувати операції з базою даних у пам’яті, що буде значно швидше, ніж робота з реальною базою.

## Функціональне тестування

- **Функціональне тестування**: Функціональне тестування перевіряє, чи функції та функціональність застосунку працюють як очікується. Мета функціонального тестування — перевірити систему в цілому за різними сценаріями.
- **Нефункціональне тестування**: Нефункціональне тестування зосереджене на таких аспектах, як продуктивність і надійність.
- **Димове тестування**: Димове тестування — це попередня перевірка системи на базові або критичні проблеми перед початком більш масштабного тестування.

## Тестування від кінця до кінця

- **Тестування від кінця до кінця**: Тестування від кінця до кінця, або E2E, перевіряє реальні сценарії з точки зору користувача. Тести E2E допомагають переконатися, що ваш застосунок поводиться правильно і передбачувано для користувачів. Однак налаштування, проєктування та обслуговування таких тестів займає багато часу.
- **Фреймворки для тестування від кінця до кінця**: Playwright — популярний фреймворк для тестування від кінця до кінця, розроблений Microsoft. Інші приклади інструментів для E2E тестування — Cypress, Selenium та Puppeteer.

Ось приклад тестів E2E з кодової бази freeCodeCamp із використанням Playwright. Хук `beforeEach` запускається перед кожним тестом. Тести перевіряють, що донор має посилання на підтримку в меню, а також спеціальний стилізований кордон навколо аватара:

```js
test.beforeEach(async ({ page }) => {
  execSync("node ./tools/scripts/seed/seed-demo-user --set-true isDonating");
  await page.goto("/donate");
});

...

test("The menu should have a supporters link", async ({ page }) => {
  const menuButton = page.getByTestId("header-menu-button");
  const menu = page.getByTestId("header-menu");

  await expect(menuButton).toBeVisible();
  await menuButton.click();

  await expect(menu).toBeVisible();

  await expect(page.getByRole("link", { name: "Supporters" })).toBeVisible();
});

test("The Avatar should have a special border for donors", async ({ page }) => {
  const container = page.locator(".avatar-container");
  await expect(container).toHaveClass("avatar-container gold-border");
});
```

## Тестування зручності використання

- **Тестування зручності використання**: Тестування зручності використання — це коли реальні користувачі взаємодіють із застосунком, щоб виявити, чи є проблеми з дизайном, користувацьким досвідом або функціональністю. Тестування зручності зосереджене на інтуїтивності застосунку для користувачів.
- **Чотири загальні типи тестування зручності**:
  - **Дослідницьке**: Дослідницьке тестування зручності передбачає, що користувачі взаємодіють із різними функціями застосунку, щоб краще зрозуміти, як вони працюють.
  - **Порівняльне**: Порівняльне тестування — це коли ви порівнюєте користувацький досвід вашого застосунку з подібними застосунками на ринку.
  - **Оцінювальне**: Оцінювальне тестування вивчає, наскільки інтуїтивно зрозуміло користуватися застосунком.
  - **Валідаційне**: Валідаційне тестування визначає будь-які серйозні проблеми, які можуть завадити користувачу ефективно використовувати застосунок.
- **Інструменти для тестування зручності**: Прикладами інструментів для тестування зручності є Loop11, Maze, Userbrain, UserTesting та UXTweak.

## Тестування сумісності

- **Тестування сумісності**: Мета тестування сумісності — переконатися, що ваш застосунок працює в різних обчислювальних середовищах.
- **Різні типи тестування сумісності**:
  - **Зворотна сумісність**: Зворотна сумісність означає, що програмне забезпечення сумісне з попередніми версіями.
  - **Прогресивна сумісність**: Прогресивна сумісність означає, що програмне забезпечення та системи працюватимуть із майбутніми версіями.
  - **Сумісність з апаратним забезпеченням**: Сумісність з апаратним забезпеченням — це здатність програмного забезпечення правильно працювати в різних апаратних конфігураціях.
  - **Сумісність з операційними системами**: Сумісність з операційними системами — це здатність програмного забезпечення працювати на різних ОС, таких як macOS, Windows і дистрибутивах Linux, наприклад Ubuntu та Fedora.
  - **Сумісність із мережею**: Сумісність із мережею означає, що програмне забезпечення може працювати в різних мережевих умовах, таких як різні швидкості мережі, протоколи, налаштування безпеки тощо.
  - **Сумісність із браузерами**: Сумісність із браузерами означає, що вебзастосунок може стабільно працювати в різних браузерах, таких як Google Chrome, Safari, Firefox тощо.
  - **Сумісність із мобільними пристроями**: Важливо переконатися, що ваші програмні застосунки працюють на різних пристроях Android та iOS, включно з телефонами та планшетами.

## Тестування продуктивності

- **Тестування продуктивності**: При тестуванні продуктивності ви перевіряєте швидкість, відгук, масштабованість і стабільність застосунку під різним робочим навантаженням. Мета — усунути будь-які вузькі місця продуктивності.
- **Різні типи тестування продуктивності**:
  - **Тестування навантаження**: Тестування навантаження визначає, як система поводиться під нормальним і піковим навантаженням.
  - **Стрес-тестування**: Стрес-тестування — це перевірка застосунку під екстремальним навантаженням, щоб побачити, як система реагує на підвищене навантаження.
  - **Тестування витривалості (Soak Testing)**: Тестування витривалості — це тип тестування навантаження, коли систему перевіряють під підвищеним навантаженням протягом тривалого часу.
  - **Тестування сплеску**: Тестування сплеску — це різке збільшення та зменшення навантаження з аналізом реакції системи на зміни.
  - **Тестування точки зупинки (Breakpoint Testing)**: Тестування точки зупинки або тестування ємності — це поступове збільшення навантаження до моменту, коли система починає відмовляти або деградувати.

## Тестування безпеки

- **Тестування безпеки**: Тестування безпеки допомагає виявити вразливості та слабкі місця.
- **Принципи безпеки**:
  - **Конфіденційність**: Захищає від розголошення конфіденційної інформації іншим, хто не є призначеним отримувачем.
  - **Цілісність**: Запобігає зміні інформації користувача зловмисниками.
  - **Автентифікація**: Перевіряє особу користувача, щоб переконатися, що він має право користуватися системою.
  - **Авторизація**: Визначає, які дії дозволені автентифікованим користувачам або до яких частин системи вони мають доступ.
  - **Доступність**: Забезпечує доступність інформації та послуг для авторизованих користувачів у потрібний час.
  - **Незаперечність**: Гарантує, що і відправник, і отримувач мають докази доставки та підтвердження особи відправника. Захищає від заперечення відправником факту відправлення інформації.
- **Поширені загрози безпеці**:
  - **Cross-Site Scripting (XSS)**: Атаки XSS відбуваються, коли зловмисник впроваджує шкідливі скрипти на вебсторінку і виконує їх у контексті браузера жертви.
  - **SQL Injection**: SQL-ін’єкція дозволяє зловмисникам впроваджувати шкідливий код у базу даних.
  - **Атака відмови в обслуговуванні (DoS)**: Атака DoS — це коли зловмисники переповнюють сайт великою кількістю запитів або трафіку, через що сервер сповільнюється або падає, роблячи сайт недоступним для користувачів.
- **Категорії інструментів для тестування безпеки**:
  - **Статичне тестування безпеки застосунків (SAST)**: Ці інструменти аналізують вихідний код застосунку для виявлення вразливостей.
  - **Динамічне тестування безпеки застосунків (DAST)**: Ці інструменти взаємодіють із фронтендом застосунку, щоб виявити потенційні слабкі місця безпеки. DAST не має доступу до вихідного коду.
- **Тестування на проникнення (pentest)**: Тестування на проникнення — це тип тестування безпеки, що включає створення імітованих кібернападів на застосунок для виявлення вразливостей у системі.

## A/B тестування

- **A/B тестування**: A/B тестування передбачає порівняння двох версій сторінки або застосунку та вивчення, яка версія працює краще. Його також називають бакет-тестуванням або спліт-тестуванням. A/B тестування дозволяє приймати рішення на основі даних і постійно покращувати користувацький досвід.
- **Інструменти для A/B тестування**: Прикладами інструментів для A/B тестування є GrowthBook та LaunchDarkly.

## Альфа- та бета-тестування

Після завершення ініціальної розробки та тестування програмного забезпечення важливо, щоб застосунок протестували тестувальники та реальні користувачі. Саме тут вступають у гру альфа- та бета-тестування.

- **Альфа-тестування**: Альфа-тестування проводиться обраною групою тестувальників, які проходять застосунок, щоб переконатися, що немає помилок перед випуском на ринок. Альфа-тестування є частиною приймального тестування і використовує як методи білого, так і чорного ящика.
- **Бета-тестування**: Бета-тестування — це коли застосунок стає доступним для реальних користувачів. Користувачі можуть взаємодіяти із застосунком і надавати відгуки. Бета-тестування також є формою приймального тестування користувачем.
- **Приймальне тестування**: Приймальне тестування гарантує, що програмне забезпечення відповідає бізнес-вимогам і потребам користувачів перед випуском.
- **Тестування чорного ящика**: Тестування чорного ящика зосереджене лише на очікуваній поведінці застосунку.
- **Тестування білого ящика**: Тестування білого ящика передбачає, що тестувальник знає внутрішні компоненти і виконує тести на них.

## Регресійне тестування

- **Регресія**: Регресія — це ситуації, коли нові зміни ненавмисно порушують існуючу функціональність.
- **Регресійне тестування**: Регресійне тестування допомагає виявити проблеми регресії. При регресійному тестуванні ви повторно запускаєте функціональні тести на частинах застосунку, щоб переконатися, що все ще працює як очікується.
- **Інструменти для регресійного тестування**: Інструменти, які можна використовувати для регресійного тестування, включають Puppeteer, Playwright, Selenium та Cypress.
- **Техніки регресійного тестування**:
  - **Модульне регресійне тестування**: Це коли у вас є список елементів, які потрібно тестувати щоразу при внесенні великих змін або виправлень у застосунок.
  - **Часткове регресійне тестування**: Включає цілеспрямовані підходи, щоб переконатися, що нові зміни не порушили конкретні аспекти застосунку.
  - **Повне регресійне тестування**: Запускає тести на всі функції у вихідному коді. Це найтриваліший і найдетальніший варіант.
- **Повторне тестування**: Повторне тестування використовується для перевірки відомих проблем і переконання, що їх виправили. На відміну від нього, регресійне тестування шукає невідомі проблеми, які могли виникнути через останні зміни у вихідному коді.

## Що таке TypeScript

- **JavaScript**: JavaScript — це мова з динамічною типізацією. Це означає, що змінні можуть отримувати будь-які значення під час виконання. Проблема динамічно типізованої мови в тому, що відсутність типобезпеки може призводити до помилок.

Наприклад, навіть якщо ваша JavaScript-функція очікує масив, ви все одно можете викликати її з числом:

```javascript
const getRandomValue = (array) => {
  return array[Math.floor(Math.random() * array.length)];
}

console.log(getRandomValue(10));
```

Вихідні дані `console` для наведеного вище прикладу будуть `undefined`.

- **TypeScript**: TypeScript розширює мову JavaScript, додаючи статичну типізацію, що допомагає виявляти помилки, спричинені невідповідністю типів, ще до запуску коду.

Наприклад, ви можете визначити тип для параметра `array` так:

```typescript
const getRandomValue = (array: string[]) => {
  return array[Math.floor(Math.random() * array.length)];
}
```

Це визначення типу повідомляє TypeScript, що параметр `array` має бути масивом рядків. Тоді, коли ви викликаєте `getRandomValue` і передаєте число, ви отримуєте помилку компілятора.

- **Компілятор**: Спочатку потрібно скомпілювати код TypeScript у звичайний JavaScript. Під час запуску компілятора TypeScript перевірить ваш код і видасть помилку, якщо типи не співпадають.

## Типи даних у TypeScript

- **Примітивні типи даних у TypeScript**: Для примітивних типів даних `string`, `null`, `undefined`, `number`, `boolean` і `bigint` TypeScript пропонує відповідні ключові слова типів.

```typescript
let str: string = "Naomi";
let num: number = 42;
let bool: boolean = true;
let nope: null = null;
let nada: undefined = undefined;
```

- **Масив**: Ви можете визначити масив певного типу двома різними синтаксисами.

```typescript
const arrOne: string[] = ["Naomi"];
const arrTwo: Array<string> = ["Camperchan"];
```

- **Об’єкти**: Ви можете визначити точну структуру об’єкта.

```typescript
const obj: { a: string, b: number } = { a: "Naomi", b: 42 };
```

Якщо ви хочете об’єкт з будь-якими ключами, але всі значення мають бути рядками, існує два способи це визначити:

```typescript
const objOne: Record<string, string> = {};
const objTwo: { [key: string]: string } = {};
```

- **Інші корисні типи в TypeScript**:
  - **`any`**: `any` означає, що значення може мати будь-який тип. Це повідомляє компілятору не звертати уваги на тип цієї змінної.
  - **`unknown`**: `unknown` повідомляє TypeScript, що вам _важливий_ тип значення, але ви фактично не знаєте, який саме. `unknown` зазвичай переважає над `any`.
  - **`void`**: Це спеціальний тип, який зазвичай використовують лише при визначенні функцій. Функції, що не повертають значення, мають тип повернення `void`.
  - **`never`**: Представляє тип, який ніколи не існує.

- **Ключове слово `type`**: Це ключове слово схоже на `const`, але замість оголошення змінної ви можете оголосити тип.

Це корисно для оголошення кастомних типів, таких як об’єднані типи або типи, що включають лише певні значення:

```typescript
type stringOrNumber = string | number;
type bot = "camperchan" | "camperbot" | "naomi";
```

- **`interface`**: Інтерфейси схожі на класи для типів. Вони можуть реалізовувати або розширювати інші інтерфейси, є спеціально об’єктними типами і зазвичай переважають, якщо вам не потрібна конкретна функція, яку пропонує оголошення `type`.

```typescript
interface wowie {
  zowie: boolean;
  method: () => void;
}
```

- **Визначення типу повернення**: Ви також можете визначити _тип повернення_ функції.

Приклад нижче визначає повернене значення як рядок. Якщо ви спробуєте повернути щось інше, TypeScript видасть помилку компілятора.

```typescript
const getRandomValue = (array: string[]): string => {
  return array[Math.floor(Math.random() * array.length)];
}
```

## Дженерики

- **Визначення дженеричного типу**: Ви можете визначити дженеричний тип і посилатися на нього у функції. Це можна уявити як спеціальний параметр, який ви передаєте функції, щоб контролювати поведінку визначення типу функції.

Ось приклад визначення дженеричного типу для функції:

```typescript
const getRandomValue = <T>(array: T[]): T => {
  return array[Math.floor(Math.random() * array.length)];
}
const val = getRandomValue([1, 2, 4])
```

Синтаксис `<T>` повідомляє TypeScript, що ви визначаєте дженеричний тип `T` для функції. `T` — це поширена назва для дженеричних типів, але ви можете використовувати будь-яку назву.

Потім ви повідомляєте TypeScript, що параметр `array` — це масив значень, що відповідають дженеричному типу, а повернене значення — це один елемент того самого типу.

- **Вказання аргументу типу при виклику функції**: Ви можете передати аргумент типу у виклик функції, використовуючи кутові дужки між назвою функції та її параметрами.

Ось приклад передачі аргументу типу у виклик функції:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Це повідомляє TypeScript, що елемент, який ви очікуєте знайти, буде елементом форми.

## Звуження типів

- **Звуження за правдивістю**: У наведеному нижче прикладі ви отримуєте помилку компілятора, намагаючись отримати доступ до властивості `value` об’єкта `email`, бо `email` _може_ бути `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email.value);
```

Ви можете використати умовну інструкцію, щоб підтвердити, що `email` є _правдивим_, перш ніж отримувати доступ до властивості:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (email) {
  console.log(email.value);
}
```

Перевірки правдивості також можуть працювати у зворотному напрямку:

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
if (!email) {
  throw new ReferenceError("Could not find email element!")
}
console.log(email.value);
```

Викидання помилки припиняє логічне виконання цього коду, що означає, що коли ви дійдете до виклику `console.log()`, TypeScript знає, що `email` _не може_ бути `null`.

- **Опціональний ланцюжок**: Опціональний ланцюжок `?.` також є формою звуження типів, за тією ж логікою, що доступ до властивості не може відбутися, якщо значення `email` є `null`.

```typescript
const email = document.querySelector<HTMLInputElement>("#email");
console.log(email?.value);
```

- **Оператор `typeof`**: Ви можете використати умовну інструкцію, щоб перевірити тип змінної за допомогою оператора `typeof`.

```typescript
const myVal = Math.random() > 0.5 ? 222 : "222";
if (typeof myVal === "number") {
  console.log(myVal / 10);
}
```

- **Ключове слово `instanceof`**: Якщо об’єкт походить із класу, ви можете використати ключове слово `instanceof` для звуження типу.

```typescript
const email = document.querySelector("#email");

if (email instanceof HTMLInputElement) {
    console.log(email.value);
}
```

- **Кастинг**: Коли TypeScript не може автоматично визначити тип значення, наприклад, результат методу `request.json()` у наведеному прикладі, ви отримаєте помилку компілятора. Один зі способів вирішення — кастинг типу, але це послаблює здатність TypeScript виявляти потенційні помилки.

```typescript
interface User {
    name: string;
    age: number;
}

const printAge = (user: User) => 
  console.log(`${user.name} is ${user.age} years old!`)

const request = await fetch("url")
const myUser = await request.json() as User;
printAge(myUser);
```

- **Охоронець типу**: Замість кастингу типу ви можете написати охоронця типу:

```typescript
interface User {
    name: string;
    age: number;
}

const isValidUser = (user: unknown): user is User => {
  return !!user && 
    typeof user === "object" &&
    "name" in user &&
    "age" in user;
}
```

Синтаксис `user is User` вказує, що ваша функція повертає булеве значення, яке, якщо істинне, означає, що значення `user` задовольняє інтерфейс `User`.

## Файл `tsconfig`

- **`tsconfig.json`**: Налаштування компілятора TypeScript зберігаються у файлі `tsconfig.json` у кореневій директорії вашого проєкту.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./prod",
    "lib": ["ES2023"],
    "target": "ES2023",
    "module": "ES2022",
    "moduleResolution": "Node",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "strict": true
  },
  "exclude": ["test/"]
}
```

Ось описи параметрів компілятора, використаних у наведеному прикладі:

- **`compilerOptions`**: Властивість `compilerOptions` контролює поведінку компілятора TypeScript.
- **`rootDir` та `outDir`**: `rootDir` і `outDir` вказують TypeScript, у якій директорії знаходяться ваші вихідні файли, а в якій має бути згенерований JavaScript.
- **`lib`**: Властивість `lib` визначає, які визначення типів використовує компілятор, і дозволяє включати підтримку конкретних версій ES, DOM тощо.
- **`module` та `moduleResolution`**: `module` і `moduleResolution` працюють разом, щоб керувати тим, як ваш пакет використовує модулі — CommonJS або ECMAScript.
- **`esModuleInterop`**: `esModuleInterop` забезпечує кращу сумісність між CommonJS і ES-модулями, автоматично створюючи об’єкти простору імен для імпортів.
- **`skipLibCheck`**: Опція `skipLibCheck` пропускає перевірку файлів `.d.ts`, які не імпортуються у вашому коді.
- **`strict`**: Прапорець `strict` увімкне кілька перевірок, таких як правильна обробка nullable типів і попередження, коли TypeScript повертається до `any`.
- **`exclude`**: Властивість верхнього рівня `exclude` наказує компілятору ігнорувати ці файли TypeScript під час компіляції.
