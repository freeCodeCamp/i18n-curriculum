# --description--

Como aprendiste en lecciones anteriores, las bases de datos relacionales almacenan datos en tablas, que están formadas por filas y columnas.

El aspecto "relacional" de las "bases de datos relacionales" proviene del hecho de que los registros en diferentes tablas pueden tener distintos tipos de relaciones.

Hay cinco tipos principales de relaciones:

- uno a uno
- uno a muchos
- muchos a uno
- muchos a muchos
- Y relaciones autorreferenciales (también conocidas como relaciones recursivas).

Dos tablas tienen una relación **uno a uno** cuando cada registro en la primera tabla puede estar asociado con como máximo un registro en la segunda tabla, y viceversa.

Por ejemplo, digamos que una empresa solo asigna un vehículo por empleado.

En ese caso, la base de datos podría tener una tabla `employees` y una tabla `vehicles`, donde cada registro de empleado corresponde a un registro de vehículo, y solo uno, en la base de datos.

En contraste, las relaciones **uno a muchos** ocurren cuando un registro en una tabla puede estar relacionado con uno o más registros en otra tabla.

Por ejemplo, si tenemos una tabla `customers` y una tabla `orders`, cada cliente podría estar relacionado con uno o más pedidos, pero cada pedido solo puede estar relacionado con un cliente.

Esto es lo mismo que la relación **many-to-one**, pero desde la perspectiva opuesta. Uno o más pedidos pueden estar relacionados con un cliente.

Las relaciones **muchos a muchos** ocurren cuando un registro en una tabla puede estar relacionado con múltiples registros en otra tabla y viceversa.

Por ejemplo, analicemos una base de datos para una biblioteca. Para simplificar, asumamos que esta base de datos solo tiene dos tablas: `books` y `authors`.

Un autor puede escribir múltiples libros, y un solo libro puede ser escrito por múltiples autores.

Implementar este tipo de relación es un poco más complejo porque el modelo relacional en su núcleo no soporta directamente relaciones de muchos a muchos.

Para manejar esto, normalmente crearías una tabla intermedia, también conocida como tabla de unión.

Esta tabla resuelve estas limitaciones iniciales al transformar la relación muchos a muchos en dos relaciones uno a muchos.

Crea una relación uno a muchos entre la primera tabla y la tabla de unión y una relación uno a muchos entre la segunda tabla y la tabla de unión.

El nombre de la tabla de unión suele ser una combinación de las entidades. En el contexto de nuestro ejemplo de `books` y `authors`, puedes llamarla `books_authors`.

En la tabla de unión, asociarías a los autores y sus libros creando dos columnas: una columna para el id del autor (`author_id`) y otra columna para el id del libro (`book_id`).

Estos serían claves foráneas que hacen referencia a sus correspondientes claves primarias en las tablas `authors` y `books`, respectivamente.

Con esta tabla de unión, podrás consultar todos los autores de un libro dado o todos los libros escritos por un autor dado.

Usar una tabla de unión facilita consultar los datos, previene la redundancia y simplifica el esquema de la base de datos. Son muy útiles para implementar relaciones muchos a muchos.

Y finalmente, encontramos relaciones **auto-referenciadas**, que ocurren cuando los registros de una tabla pueden estar relacionados con otros registros en la misma tabla. Esto también se conoce como una relación recursiva.

Entender estas relaciones es esencial para diseñar y modelar bases de datos relacionales complejas y eficientes.
