## WebSockets مقابل HTTP

- **HTTP** يتبع نموذج طلب-استجابة: يرسل العميل طلبًا، ويرسل الخادم استجابة، ثم يُغلق الاتصال.
- **WebSockets** تنشئ اتصالًا دائمًا ثنائي الاتجاه بين العميل والخادم يبقى مفتوحًا حتى يرسل أحد الطرفين إشارة إغلاق.
- HTTP مناسب لتحميل الصفحات أو جلب البيانات، لكنه يصبح بطيئًا وغير فعال للتطبيقات الزمن الحقيقي.
- يبدأ كل اتصال WebSocket بـ **مصافحة HTTP**: يطلب العميل ترقية، ويرد الخادم بـ `101 Switching Protocols`.

### تدفق اتصال WebSocket

1. يتصل العميل بالخادم عبر HTTP ويطلب ترقية.
2. يقبل الخادم ويرد بـ `101 Switching Protocols`.
3. يبقى الاتصال مفتوحًا.
4. يتبادل الطرفان البيانات في أي وقت حتى يُغلق الاتصال.

### التواصل عبر HTTP مقابل WebSocket

- **HTTP**: `Client → Request → Server → Response` (دورة واحدة ثم يغلق)
- **WebSocket**: `Client ↔ Server` (دائم، ثنائي الاتجاه)

### تشبيه مفيد

- **HTTP** يشبه تبادل الرسائل: تكتب رسالة، تنتظر ردًا، ثم تنتهي المحادثة.
- **WebSockets** مثل مكالمة هاتفية: يتحدث الطرفان في الوقت الحقيقي طالما دعت الحاجة.

### استخدامات WebSockets في العالم الحقيقي

- تطبيقات الدردشة
- ألعاب متعددة اللاعبين عبر الإنترنت
- تحديثات نتائج الرياضة الحية
- لوحات بيانات سوق الأسهم
- أدوات التعاون (مثل Google Docs)

## بنية Pub/Sub

- **Pub/Sub** (النشر/الاشتراك) هي بنية رسائل تسمح لأجزاء النظام بالتواصل دون معرفة بعضها البعض مباشرة.
- هناك ثلاثة أجزاء رئيسية: **الناشرون**، **المواضيع**، و**المشتركون**.
  - **الناشر** يرسل رسائل إلى موضوع.
  - **الموضوع** (يسمى أيضًا قناة أو مجموعة) يعمل كوسيط.
  - **المشترك** يستقبل الرسائل من المواضيع التي اشترك بها.

### تدفق رسائل Pub/Sub

1. يشترك المشترك في موضوع.
2. يرسل الناشر رسالة إلى ذلك الموضوع.
3. يوزع نظام الرسائل الرسالة إلى كل مشترك في ذلك الموضوع.

### التواصل المباشر مقابل Pub/Sub

- **بدون Pub/Sub**: `Application A → Application B` — ارتباط وثيق؛ أي تغيير في B قد يكسر A.
- **مع Pub/Sub**: `Application A → Topic → Subscribers` — A ينشر فقط؛ لا يحتاج لمعرفة من يستقبل.

### استخدامات Pub/Sub في العالم الحقيقي

- تطبيقات الدردشة مع غرف متعددة
- ألعاب متعددة اللاعبين عبر الإنترنت
- أنظمة الإشعارات
- لوحات البيانات الحية
- التطبيقات المعتمدة على الأحداث

### WebSockets مقابل Pub/Sub

- **WebSockets** توفر قناة الاتصال (الطريق).
- **Pub/Sub** توفر نمط الرسائل الذي يحدد من يستقبل الرسائل المحددة (نظام المرور).
- تستخدم العديد من التطبيقات الزمن الحقيقي كلاهما معًا: تحافظ WebSockets على الاتصال مفتوحًا، بينما يوجه Pub/Sub الرسائل إلى المشتركين المناسبين.

## العمل مع WebSockets في Node.js

- لا يحتوي Node.js على خادم WebSocket مدمج. مكتبة `ws` هي الحزمة الخارجية القياسية المستخدمة لهذا الغرض.
- ثبتها باستخدام: `npm install ws`

### إنشاء خادم WebSocket

```js
const WebSocket = require("ws");

const server = new WebSocket.Server({ port: 3000 });
```

### أحداث WebSocket

| الحدث | المحفز |
|---|---|
| `connection` | يتصل عميل جديد بالخادم |
| `message` | يرسل عميل متصل بيانات |
| `close` | يُغلق اتصال |
| `error` | يحدث خطأ في الاتصال |

### إرسال رسالة إلى عميل

```js
socket.send("Hello from the server");
```

### البث إلى جميع العملاء المتصلين

- يحتفظ `server.clients` بجميع العملاء المتصلين.
- تحقق من `readyState === WebSocket.OPEN` قبل الإرسال لتجنب الأخطاء مع العملاء المنفصلين.

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **البث** هو نمط إرسال رسالة إلى عدة عملاء متصلين دفعة واحدة — يُستخدم في تطبيقات الدردشة، الإشعارات الحية، تحديثات الرياضة، وأحداث الألعاب متعددة اللاعبين.
