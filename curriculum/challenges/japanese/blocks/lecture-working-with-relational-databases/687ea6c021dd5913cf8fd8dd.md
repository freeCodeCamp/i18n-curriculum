# --description--

リレーショナルデータベースは、テーブルに整理されたデータの集合です。これらのテーブルは行と列で構成されています。

行は個々のレコードを表します。例えば、ユーザーデータを保存している場合、各行は特定のユーザーを表すことができます。

列は各レコードを説明する属性やフィールドを表します。テーブルには、名前や国民ID番号など、一般的なユーザー属性の列が含まれるかもしれません。

リレーショナルデータベースの重要な特徴の一つは、共通の属性を通じてテーブルをリンクし、異なる情報の断片を結びつけることができるため、それらがどのように関連しているかを簡単に把握できることです。

リレーショナルデータベースにはスキーマが必要です。スキーマはデータベース全体の構造を定義し、テーブル、列、データ型、関係性、制約を含みます。

リレーショナルデータベースのすべてのテーブルには主キーがあり、これは各行の一意の識別子です。主キーは外部キーを介してテーブル間の関係を可能にします。主キーや外部キー、それらの仕組みについては今後のレッスンで詳しく学びます。今は、これらがテーブル間のレコードをリンクするための基本であることだけ覚えておいてください。

例えば、絶滅危惧種の動物を保護する野生生物保護センターを運営する組織のためにデータベースを作成するとしましょう。

そのうちの一つのテーブルは動物の一般情報を保存し、`animals`と呼ぶことができます。各行は動物を表し、各列は種、サイズ、体重、年齢、その他の特徴など、その動物の属性を表します。

各動物を一意に識別するために、通常はテーブルに`id`列を追加します。これは各動物に対して一意なのでテーブルの主キーになります。各動物のIDは、このテーブルのレコードを他のテーブルの関連レコードにリンクするために使われます。

また、`species`というテーブルを作成して、種ごとの生息地や保護状況などの情報を保存することもできます。`animals`テーブルの各動物は、このテーブルの種IDを使って種を参照できます。

さらに、動物の獣医記録を保存するテーブルを作成することもできます。これにより、センターは動物の健康状態、治療、投薬を追跡できます。

このような一般的なスキーマができたら、これらのテーブルを動物関連データの中心となる`animals`テーブルにリンクできます。

こうすることで、動物に関するすべての情報を同じテーブルに保存する必要がなくなります。代わりに、関連するデータを別々のテーブルに保存し、関係性を通じてリンクできます。これらの関係を作ることで、単一のクエリで必要な特定のデータを取得できます。

クエリとは、データベースから特定のデータをリクエストすることです。例としては、獣医の健康診断が必要なすべての動物を見つけること、センターで最も絶滅危惧種の種を見つけること、飼育下で生まれた動物の数を調べることなどがあります。クエリはニーズに合わせてカスタマイズできます。

この例は野生生物保護に関するものでしたが、リレーショナルデータベースは医療、建設、ビジネス、ゲーム、教育、政府、eコマース、ソーシャルメディアなど、さまざまな業界で幅広く使われています。

リレーショナルデータベースにはスケーラビリティなどの利点があります。大規模なデータセットを扱えるため、複雑な実世界のアプリケーションに最適です。また、主キー、外部キー、データ型を通じてデータの整合性を保証し、データが一貫して正確であることを確保します。

リレーショナルデータベースに加えて、非リレーショナルデータベースも利用できます。

リレーショナルデータベースと非リレーショナルデータベースの主な違いは、データの保存方法です。非リレーショナルデータベースはNoSQLデータベースとも呼ばれ、より柔軟です。接続されていない個別のファイルにデータを保存します。データモデルもより柔軟です。

一部の非リレーショナルデータベースには基本的なスキーマやデータモデルがある場合もありますが、リレーショナルデータベースで定義されるスキーマほど厳密ではありません。必要に応じてデータフィールドを追加、変更、削除できます。

リレーショナルデータベースか非リレーショナルデータベースかを選ぶ判断は、データの性質やアプリケーションの具体的な要件など、さまざまな要因によります。これら二つのアプローチのトレードオフを評価して、アプリケーションに最適なものを選んでください。
