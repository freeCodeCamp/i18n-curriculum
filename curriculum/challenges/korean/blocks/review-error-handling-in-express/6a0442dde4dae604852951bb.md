## HTTP 응답 상태 코드

- **상태 코드**는 요청 후 클라이언트에게 무슨 일이 일어났는지 알려주는 짧은 숫자 메시지입니다(예: 200, 404, 500).

### 1xx — 정보

- **100 Continue** — 서버가 클라이언트에게 나머지 요청을 보내라고 알립니다.
- **101 Switching Protocols** — 서버가 프로토콜을 전환 중입니다(주로 WebSocket에서 볼 수 있음).

### 2xx — 성공

- **200 OK** — 요청이 성공했습니다.
- **201 Created** — 새 리소스가 성공적으로 생성되었습니다.
- **204 No Content** — 요청은 성공했지만 응답 본문이 없습니다(종종 `DELETE`에 사용됨).

### 3xx — 리다이렉션

- **301 Moved Permanently** — 리소스가 새 영구 URL로 이동했습니다.
- **302 Found** — 리소스가 일시적으로 다른 URL에 있습니다.
- **304 Not Modified** — 클라이언트가 캐시된 버전을 사용할 수 있습니다; 변경된 내용이 없습니다.

### 4xx — 클라이언트 오류

- **400 Bad Request** — 요청이 잘못되었거나 유효하지 않습니다.
- **401 Unauthorized** — 인증이 필요합니다.
- **403 Forbidden** — 인증은 되었지만 권한이 없습니다.
- **404 Not Found** — 요청한 리소스가 존재하지 않습니다.

### 5xx — 서버 오류

- **500 Internal Server Error** — 일반적인 서버 측 실패입니다.
- **502 Bad Gateway** — 프록시가 다른 서버에서 잘못된 응답을 받았습니다.
- **503 Service Unavailable** — 서버가 일시적으로 과부하되었거나 다운되었습니다.

### Express에서 상태 코드 사용하기

- `res.status(400).send("Bad Request")` — 400 상태 코드와 일반 텍스트 메시지를 보냅니다.
- `res.status(200).json({ message: "Success" })` — 200 상태 코드와 JSON 본문을 보냅니다.
- `res.status(500).send("Internal Server Error")` — 500 상태 코드와 메시지를 보냅니다.

## Express에서의 오류 처리

- **오류 처리 미들웨어**는 네 개의 매개변수를 가집니다: `(err, req, res, next)`. 네 개 매개변수 서명은 Express에 이 미들웨어가 오류 핸들러임을 알립니다.
- 오류 처리 미들웨어는 모든 라우트 뒤, 미들웨어 스택의 **마지막에** 배치하세요.

```javascript
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

### Express 4 비동기 오류 처리

- Express 4에서는 비동기 오류가 **자동으로 잡히지 않습니다**.
- 비동기 로직을 `try/catch`로 감싸고 `next(err)`를 호출해 오류를 오류 핸들러로 전달하세요.

```javascript
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- `asyncHandler` 래퍼를 사용하면 모든 라우트에서 `try/catch`를 반복하지 않아도 됩니다:

```javascript
function asyncHandler(fn) {
  return function (req, res, next) {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
}
```

### Express 5 비동기 오류 처리

- Express 5에서는 비동기 오류와 거부된 프로미스가 **자동으로** 오류 핸들러로 전달됩니다 — 수동 `next(err)`가 필요 없습니다.

### 404 오류 처리

- 라우트 뒤, 오류 핸들러 앞에 모든 경로를 처리하는 catch-all 미들웨어를 추가하세요:

```javascript
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### 오류 처리 모범 사례

- 오류 처리 미들웨어는 항상 마지막에 배치하세요.
- 404 오류는 다른 오류와 별도로 처리하세요.
- 운영 환경에서는 오류 세부 정보를 내부에 기록하고 사용자에게 스택 추적을 절대 보내지 마세요.

## Express에서 디버깅과 로깅

- **내장 디버깅**: `DEBUG` 환경 변수를 설정하면 Express 내장 디버그 출력을 활성화합니다.
  - `DEBUG=express:* node index.js` — 모든 Express 디버그 메시지를 활성화합니다.
  - `DEBUG=express:router node index.js` — 라우터 관련 디버그 출력만 표시합니다.
  - PowerShell: `$env:DEBUG = "express:*"; node index.js`
  - CMD: `set DEBUG=express:* && node index.js`
- **`morgan`**: HTTP 요청(메서드, URL, 상태 코드, 응답 시간)을 기록하는 인기 미들웨어입니다.

```javascript
const morgan = require('morgan');
app.use(morgan('dev'));
```

- **디버깅**은 Express가 요청과 미들웨어를 처리하는 과정을 추적하지만 운영 로그를 복잡하게 만들지 않습니다.
- **로깅**은 요청 전반에서 일어나는 일을 기록해 버그를 찾거나 트래픽을 이해하는 데 유용합니다.

## 상태 점검과 정상 종료

### 상태 점검

- **상태 점검 라우트**는 앱이 정상 작동하는지 보고합니다.
- 로드 밸런서와 오케스트레이터(예: Kubernetes, AWS Elastic Beanstalk)가 이 라우트를 호출해 서비스 상태를 판단합니다.
- **200** 응답은 정상 상태를 의미하며, 응답이 없으면 자동 재시작이 발생합니다.

```javascript
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

### 정상 종료

- **정상 종료**는 앱이 즉시 연결을 끊지 않고 기존 요청을 마치도록 합니다.
- 프로세스 종료 신호인 `SIGTERM`를 감지하고 `server.close()`를 호출하세요:

```javascript
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- 로컬 개발 중 Ctrl+C를 누르면 보내지는 `SIGINT` 신호를 감지하고 `SIGTERM`와 동일한 종료 로직을 호출하세요.

### 종료 전 앱을 비정상 상태로 표시하기

- 종료 중에는 상태 점검에서 **503**을 반환해 로드 밸런서가 새 트래픽을 앱으로 보내지 않도록 하세요.

```javascript
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => {
    console.log('Graceful shutdown complete');
  });
});
```
