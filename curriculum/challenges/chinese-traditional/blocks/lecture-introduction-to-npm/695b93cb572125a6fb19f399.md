# --description--

npm 需要一種方式來儲存關於你的專案的重要細節－例如它使用了哪些軟體包、這些軟體包的版本，以及它可以執行哪些指令。它是透過 `package.json` 檔案來做到這一點的。

將 `package.json` 檔案視為專案的 npm 組態檔。它是一個簡單的 JSON 檔案，使用鍵值對來描述你的專案，以便追蹤像是：

* 專案的名稱和版本
    
* 作者
    
* 應該先執行的主要檔案
    
* 授權條款
    
* 程式碼儲存庫、關鍵字和描述
    
* 專案所需的任何依賴項
    

以下是 `package.json` 檔案可能的樣子：

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "description": "A simple NPM project",
  "main": "index.js",
  "scripts": {
    "start": "node index.js",
    "test": "echo \"Running tests...\""
  },
  "repository": {
    "type": "git",
    "url": "git+https://github.com/username/project.git"
  },
  "keywords": ["fCC", "js"],
  "author": "Kolade Chris",
  "license": "ISC",
  "dependencies": {
    "react": "^19.2.1"
  },
  "devDependencies": {
    "cypress": "^15.7.1"
  }
}
```

`main` 欄位通常設定為 `index.js`，並告訴 Node.js 要先執行哪個檔案。

`license` 欄用來指定其他人可以使用、修改和散佈你軟體包中程式碼的法律條款。有許多不同的授權條款，例如 `MIT`、`ISC` 或 `GPL`，它們各自有不同的條款和權限。預設值是 `ISC`，即網際網路系統聯盟授權，這是一種非常寬鬆的開放原始碼授權。

隨著你的專案成長並安裝軟體包，npm 會將它們新增到兩個區段之一：

* `dependencies`，是你的應用程式在生產環境中需要執行的軟體包。
    
* `devDependencies`，這些是你只在開發期間需要的軟體包
    

例如，如果你的專案使用 React 作為其 UI，則它應該放在 `dependencies` 中，因為它是你的專案部署後能正常運作所必需的。像 Cypress 這樣的工具，只用於部署到生產環境前的端對端測試，則應該放在 `devDependencies` 中。

有幾種簡單的方法可以創建 `package.json` 檔案。最常見的方法是執行以下指令：

```bash
npm init
```

你應該在你的專案根目錄中執行這個指令。此指令會啟動一個互動式設定處理程序，npm 會問你一系列問題。你的回答會用來填寫像是專案名稱、版本、描述、指令碼、關鍵字、作者和授權等欄位。

如果你想跳過這些問題並使用預設值創建一個 `package.json` 檔案，你可以執行：

```bash
npm init -y
```

或

```bash
npm init --yes
```

這會立即為你創建一個帶有多個欄位預設值的 `package.json` 檔案。

通常最好使用不帶 `-y` 或 `--yes` 旗標的 `npm init`，這樣你可以逐一瀏覽每個欄並自訂你的專案資訊。如果你沒有回答某個問題，npm 就會使用預設值。例如，如果你將授權問題留空，npm 預設會將其設為 `ISC`。

你可能會注意到 `dependencies` 和 `devDependencies` 尚未列出。那是因為這些區段只有在你安裝第一個軟體包後才會被添加。

你也會注意到 `repository` 欄位遺失了。即使 npm 在 `npm init` 過程中詢問過，它之後也不會自動填入。例如，當你執行像 `git remote add origin` 或 `git push` 這類指令時，npm 不會幫你更新 `repository` 欄位——如果你想包含它，必須手動添加。
