# --description--

現在、誰がウェブソケットに接続しているかを判別できません。`req.user`にはユーザーオブジェクトが含まれていますが、それはユーザーがウェブサーバーとやり取りするときだけで、ウェブソケットの場合は`req`（リクエスト）がなく、したがってユーザーデータもありません。ウェブソケットに接続しているユーザーを知る問題を解決する方法の一つは、passportセッションを含むクッキーを解析・デコードし、それをデシリアライズしてユーザーオブジェクトを取得することです。幸いなことに、npmにはこの複雑な作業を簡単にするパッケージがあります！

`passport.socketio@~3.7.0`、`connect-mongo@~3.2.0`、`cookie-parser@~1.4.5`はすでに依存関係として追加されています。それぞれを`passportSocketIo`、`MongoStore`、`cookieParser`としてrequireしてください。また、以前にrequireした`express-session`から新しいメモリストアを初期化する必要があります。以下のようになります：

```js
const MongoStore = require('connect-mongo')(session);
const URI = process.env.MONGO_URI;
const store = new MongoStore({ url: URI });
```

あとはSocket.IOにこれを使うように指示し、オプションを設定するだけです。これは既存のソケットコードの前、既存の接続リスナーの中ではなく追加してください。サーバー側は以下のようになります：

```js
io.use(
  passportSocketIo.authorize({
    cookieParser: cookieParser,
    key: 'express.sid',
    secret: process.env.SESSION_SECRET,
    store: store,
    success: onAuthorizeSuccess,
    fail: onAuthorizeFail
  })
);
```

Socket.IOのPassport認証の設定は、API用に設定した`session`ミドルウェアの設定と非常に似ていることに注意してください。これは同じ認証方法を使うためです — クッキーからセッションIDを取得し、それを検証します。

以前、`session`ミドルウェアを設定したとき、セッションのクッキー名（`key`）を明示的に設定しませんでした。これは`session`パッケージがデフォルト値を使っていたためです。今回、同じクッキーの値にアクセスする別のパッケージを追加したので、両方の設定オブジェクトで`key`の値を明示的に設定する必要があります。

Socket.IOのキーに一致する`key`を`session`ミドルウェアにクッキー名付きで追加してください。また、`store`の参照をオプションに、`saveUninitialized: true`を設定している近くに追加してください。これはSocket.IOにどのセッションに関連付けるかを伝えるために必要です。

<hr>

次に、`success`と`fail`のコールバック関数を定義してください：

```js
function onAuthorizeSuccess(data, accept) {
  console.log('successful connection to socket.io');

  accept(null, true);
}

function onAuthorizeFail(data, message, error, accept) {
  if (error) throw new Error(message);
  console.log('failed connection to socket.io:', message);
  accept(null, false);
}
```

ユーザーオブジェクトは今やソケットオブジェクトの`socket.request.user`としてアクセス可能です。例えば、以下のように追加できます：

```js
console.log('user ' + socket.request.user.username + ' connected');
```

これで誰が接続したかがサーバーのコンソールにログとして表示されます！
