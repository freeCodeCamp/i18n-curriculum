# --description--

In einer vorherigen Lektion haben Sie gelernt, wie Sie einen Pull Request für Ihr eigenes Repository erstellen. Sie haben auch gelernt, wie Sie diesen Pull Request zusammenführen. Aber das konnten Sie nur, weil das Repository Ihnen gehört – Sie haben Maintainer-Zugriff auf Ihre eigenen Repositories, was bedeutet, dass Sie die Berechtigung haben, Pull Requests zusammenzuführen.

Wenn Sie jedoch zu einem Open-Source-Projekt von jemand anderem beitragen, haben Sie normalerweise diese Berechtigungen nicht. Stattdessen müssen Sie warten, bis die Maintainer den Code-Review-Prozess durchlaufen und Ihren Beitrag akzeptieren. Sie übernehmen dann das Zusammenführen.

Aber wie sieht der Code-Review-Prozess eigentlich aus?

Der Prozess, den Maintainer durchlaufen, variiert von Person zu Person, und es kann projektspezifische Vorgehensweisen geben. Im Allgemeinen gibt es jedoch typischerweise drei Schritte bei einem Code-Review.

Der erste Schritt ist zu bestätigen, dass die CI, also die Continuous Integration, erfolgreich ist. Mehr dazu lernen Sie in einer zukünftigen Lektion, aber vorerst können Sie sich darunter Prüfungen vorstellen, die sicherstellen, dass der Code den Stilrichtlinien entspricht und eine Testsuite besteht.

Wenn die CI fehlschlägt, fordern Maintainer in der Regel, dass Sie den Pull Request aktualisieren, um die Fehler zu beheben. Diese Fehler sollten eine Art Bericht enthalten, den Sie sich ansehen können, um zu verstehen, was schiefgelaufen ist. Die Maintainer können Ihnen aber auch spezifischere Hinweise geben, wie Sie das Problem lösen.

Sobald die Prüfungen bestanden sind, schauen sich die Maintainer Ihre tatsächlichen Codeänderungen an. Sie können die Diff-Ansicht analysieren, um zu sehen, was Sie geändert, hinzugefügt oder gelöscht haben. Sie könnten Fragen dazu stellen, warum Sie bestimmte Ansätze gewählt haben, daher ist es gut, wenn Sie Ihre Änderungen bewusst und durchdacht vorbereiten.

Sie könnten auch verlangen, dass Sie weitere Änderungen vornehmen. Dazu kommen wir gleich noch.

Der letzte Schritt ist normalerweise, Ihre Änderungen lokal herunterzuladen. Die Maintainer können dann die Anwendung mit Ihren Änderungen ausführen, die Funktionalität testen und überprüfen, ob nichts kaputt ist oder sich unerwartet verhält.

Wenn alles gut aussieht, wird ein Maintainer Ihren Pull Request genehmigen – manchmal mit einem „LGTM“-Kommentar, was für „looks good to me“ steht. Nachdem Ihr Pull Request genehmigt wurde, sollte er zusammengeführt werden, sobald die Maintainer bereit sind, die Änderungen zu übernehmen. Manche Projekte verlangen mehr als eine Genehmigung – freeCodeCamp zum Beispiel verlangt zwei.

Aber was, wenn mit Ihrem Pull Request etwas nicht stimmt? Maintainer können eine Überprüfung durchführen, bei der sie Änderungen anfordern. Diese Anfragen können in verschiedenen Formen kommen.

Manchmal hinterlässt ein Maintainer einfach einen „Kommentar“ mit seiner Anfrage, vielleicht mit dem Hinweis, dass Sie etwas ändern oder aktualisieren sollten, aber mit mehr Freiheit, wie Sie das umsetzen.

Oder Maintainer hinterlassen auch einen offiziellen „request changes“-Kommentar. Normalerweise blockiert ein solcher Kommentar das Zusammenführen des Pull Requests. Sie müssen die gewünschten Änderungen vornehmen, und dann muss derselbe Maintainer, der die Änderungen angefordert hat, den Pull Request erneut überprüfen und entweder weitere Änderungen anfordern oder Ihre Arbeit genehmigen.

Wenn Maintainer konkrete Änderungen im Sinn haben, können sie diese tatsächlich über die GitHub-Benutzeroberfläche vorschlagen. Wenn sie eine direkte Empfehlung so abgeben, gibt GitHub Ihnen einen Button, um den Vorschlag direkt in Ihren Branch zu übernehmen. Wenn es jedoch mehrere solche Vorschläge gibt, sollten Sie zur Dateiansicht wechseln.

In dieser Ansicht können Sie alle Vorschläge (einen nach dem anderen) zu einem „Batch“ hinzufügen und sie alle auf einmal committen. Das ist sehr nützlich, wenn es viele Vorschläge gibt, da das einzelne Umsetzen viele Commits erzeugen würde. Alle CI-Prüfungen müssten bei jedem einzelnen Commit laufen, was den Prozess zur Annahme Ihres PR verlangsamen kann.

Wenn die angeforderten Änderungen umfangreich sind, möchten Sie die Änderungen wahrscheinlich lokal vornehmen, sie in denselben Branch committen, den Sie für den PR verwendet haben, und dann hochpushen. GitHub aktualisiert Ihren Pull Request automatisch, um die neuen Commits widerzuspiegeln.

Es gibt noch ein paar weitere Hinweise, die wir ansprechen möchten.

Erstens: Denken Sie daran, dass Maintainer auch Menschen sind. Open-Source-Projekte generieren selten genug Einnahmen, damit Maintainer Vollzeit daran arbeiten können, was bedeutet, dass sie einen Job behalten müssen. Es ist wichtig, geduldig mit dem Review-Prozess zu sein – schreiben Sie Maintainer nicht ständig an, um eine erneute Überprüfung zu erbitten, und versuchen Sie nicht, sie bei ihrer Überprüfung zu hetzen.

Beiträge zu leisten kann aufregend sein! Aber hetzen Sie nicht durch den Prozess. Nehmen Sie sich Zeit. Testen Sie Ihre Änderungen lokal – führen Sie alle Testsuiten aus, die das Projekt hat, starten Sie den Entwicklungsserver und prüfen Sie manuell, ob Ihre Änderungen funktionieren, usw. Und folgen Sie immer den Beitragsrichtlinien des jeweiligen Projekts, an dem Sie arbeiten.

Wenn Sie all das beachten und Ihr Bestes geben, sollten Ihre Beiträge reibungslos verlaufen und Ihr Code-Review eine solide Genehmigung sein!
