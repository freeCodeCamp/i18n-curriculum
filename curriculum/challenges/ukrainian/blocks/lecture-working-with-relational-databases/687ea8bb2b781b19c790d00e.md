# --description--

У SQL первинні та зовнішні ключі використовують для встановлення зв’язків між таблицями.

Почнемо з первинних ключів.

Первинний ключ — це стовпець або набір стовпців, які унікально ідентифікують кожен запис (рядок) у таблиці. Це обмеження гарантує, що в таблиці не буде записів із однаковим значенням первинного ключа.

Саме тому таблиця може мати лише один первинний ключ.

Значення в стовпці первинного ключа не можуть бути `NULL`, тому вони завжди матимуть дійсне значення.

У PostgreSQL, щоб зробити стовпець первинним ключем таблиці, потрібно просто додати `PRIMARY KEY` після типу даних:

```sql
column_name data_type PRIMARY KEY
```

Ось приклад таблиці `students`:

```sql
CREATE TABLE students (
  student_id SERIAL PRIMARY KEY,
  name VARCHAR(100)
);
```

Кожен студент матиме свій власний `student_id`, і цей ідентифікатор буде первинним ключем таблиці. Тип `SERIAL` тут корисний, бо гарантує, що `student_id` завжди матиме унікальне значення.

Складений первинний ключ використовують, коли в таблиці немає одного унікального стовпця для ідентифікації рядка. У такому випадку можна використати комбінацію двох або більше стовпців як первинний ключ, які разом є унікальними.

Для цього позначте, які стовпці є складеним первинним ключем під час створення таблиці так:

```sql
CREATE TABLE table_name (
  column1 data_type column_constraint,
  column2 data_type column_constraint,
  column3 data_type column_constraint,
  ...
  PRIMARY KEY (column1, column2)
);
```

Уявіть, що у вас є таблиця зі стовпцями `student_id` і `course_id`, яка показує, у яких класах навчаються студенти.

Кожен студент буде записаний у кілька класів, і кожен клас матиме кількох студентів. Тож жоден зі стовпців не є унікальним — але ви ніколи не матимете однакового студента, записаного в той самий клас більше одного разу. Тому можна використати комбінацію двох стовпців як первинний ключ:

```sql
CREATE TABLE course_enrollments (
  student_id INT,
  course_id INT,
  ...
  PRIMARY KEY (student_id, course_id),
);
```

Первинні ключі є необхідними для забезпечення унікальності та цілісності даних, оптимізації їхнього отримання та встановлення зв’язків з іншими таблицями в реляційній базі даних.

Тепер перейдемо до наступного типу ключа, який розглянемо в цьому уроці: зовнішніх ключів.

Зовнішній ключ — це стовпець (або набір стовпців) у таблиці, який посилається на первинний ключ іншої таблиці. Таблиця може мати кілька зовнішніх ключів.

У цьому прикладі ми маємо дві таблиці, `customers` і `orders`:

```sql
CREATE TABLE customers (
  customer_id SERIAL PRIMARY KEY,
  first_name VARCHAR(100) NOT NULL,
  ...
);

CREATE TABLE orders (
  order_id SERIAL PRIMARY KEY,
  customer_id INTEGER,
  ...
  FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);
```

Первинним ключем таблиці `customers` є `customer_id`, а первинним ключем таблиці `orders` є `order_id`.

Але зверніть увагу, що в таблиці `customer_id` також є стовпець `orders`.

Чому ми зберігаємо інформацію про клієнтів у таблиці `orders`?

Це робиться для створення зв’язку між таблицями `customers` і `orders`. Кожне замовлення зберігатиме ідентифікатор клієнта, який його відправив.

Щоб створити зв’язок між таблицями, зробимо стовпець `customer_id` у таблиці `orders` зовнішнім ключем, який посилається на стовпець `customer_id` у таблиці `customers`:

```sql
FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
```

Значення у стовпці зовнішнього ключа мають або співпадати зі значеннями первинного ключа у таблиці, на яку посилаються, або бути `NULL` (якщо зовнішній ключ дозволяє `NULL`).

Чому це корисно?

Переконуючись, що ці стовпці в обох таблицях точно співпадають, зовнішні та первинні ключі запобігають створенню «сирітських» записів — записів, які посилаються на неіснуючий запис в іншій таблиці.

Первинні та зовнішні ключі є фундаментальними для реляційних баз даних. Вони забезпечують «реляційний» аспект цих баз, дозволяючи моделювати реальні дані точно і коректно.
