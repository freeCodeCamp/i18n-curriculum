# --description--

Pub/Sub 代表 **发布/订阅**。

这是一种消息架构，允许系统的不同部分相互通信，而无需直接了解彼此。

乍一看，这听起来可能很复杂，但你已经在日常生活中见过类似的系统。

想想 YouTube。

当创作者/频道管理员上传新视频时，他们不会直接发送给每个订阅者。相反，他们将视频发布到他们的频道。任何订阅了该频道的人都会收到新上传的通知。

创作者不需要知道订阅者是谁，订阅者也不需要直接联系创作者。平台处理中间的所有事情。

Pub/Sub 的工作方式类似。它涉及三个主要部分：

- 发布者
- 主题（或频道，或者像 WhatsApp 上的群组）
- 订阅者

**发布者** 发送消息。**订阅者** 接收消息。**主题** 充当它们之间的中间人。

流程如下：

1. 订阅者订阅一个主题。
2. 发布者向该主题发送消息。
3. 消息系统将消息传递给该主题的每个订阅者。

例如，想象一个体育新闻应用。用户可以订阅一个主题或他们选择的球队，比如 `football-updates` 或 `chelsea fc`。

每当发生比赛事件，比如进球或红牌，系统就会向该主题发布消息。当切尔西足球俱乐部进球或失球时，所有订阅了 `chelsea fc` 的用户都会立即收到更新。

发布者不需要知道订阅者是谁。它只需要知道将消息发布到哪里。

这种分离是 Pub/Sub 架构最大的优势之一。没有 Pub/Sub，系统通常直接通信。流程可能是这样的：

`Application A → Application B`

这种方式可行，但会在应用之间产生不必要的依赖。如果应用 B 发生变化或不可用，应用 A 可能无法正常工作。

使用 Pub/Sub，流程变为：

`Application A → Topic → Subscribers`

现在应用 A 只发布消息。它不需要知道谁接收消息。

另一个例子是带有多个聊天室或群组的聊天应用。房间 A 的用户订阅 `room-a`。房间 B 的用户订阅 `room-b`。

当有人在房间 A 发送消息时，应用将该消息发布到 `room-a`。只有 `room-a` 的订阅者会收到消息，房间 B 的用户永远看不到。

这样，系统更容易维护、扩展和扩容。

这也是 Pub/Sub 经常出现在聊天应用、多玩家游戏、通知系统、实时仪表盘和事件驱动应用中的原因之一。

还需要理解的是，Pub/Sub 和 WebSockets 并不是同一回事。WebSockets 提供了允许客户端和服务器之间实时传输数据的通信通道。Pub/Sub 提供了决定谁应该接收特定消息的消息模式。

你可以这样理解：

- WebSockets 是道路。
- Pub/Sub 是决定车辆去向的交通系统。

许多现代实时应用同时使用这两种技术。WebSockets 保持连接打开，而 Pub/Sub 确保消息送达正确的订阅者。

Pub/Sub 的核心思想是发布者向主题发送消息，订阅者监听主题。消息系统负责它们之间的传递。
