# --description--

Die digitale Welt, wie wir sie heute kennen, basiert auf der Vorstellung, dass Geräte miteinander kommunizieren können, indem sie bei Bedarf Daten senden und empfangen.

Aber haben Sie sich schon einmal gefragt, wie dieser Prozess hinter den Kulissen funktioniert?

Wenn Sie im Internet surfen, fordern Sie ständig Daten an und senden sie über das Netzwerk. Dies ist dank eines detaillierten Prozesses möglich, der auf standardisierten Regeln und Verfahren basiert, die sicherstellen, dass die Informationen korrekt von den entsprechenden Geräten gesendet und empfangen werden.

Dieses Regelwerk und diese Verfahren werden als **Protokoll** bezeichnet.

Protokolle ermöglichen es grundsätzlich zwei oder mehr Geräten, miteinander zu kommunizieren, indem sie sicherstellen, dass alle bestimmte Standards einhalten, wie zum Beispiel das Format der auszutauschenden Daten.

Protokolle sorgen dafür, dass alle an dem Datenaustausch beteiligten Geräte einander verstehen können. Bildlich gesprochen „sprechen“ sie alle dieselbe Sprache.

Im Kern sind TCP/IP die Protokolle, die das Internet überhaupt erst möglich machen.

**TCP/IP** ist eine Sammlung von Kommunikationsprotokollen, die bestimmen, wie Daten verpackt, gesendet, geroutet und über ein Netzwerk empfangen werden. Es stellt sicher, dass Informationen zuverlässig transportiert werden.

Genauer gesagt ist das **IP**, wobei IP für **Internet Protocol** steht, ein Protokoll, das sicherstellt, dass die Daten vom Ursprung zur richtigen Zieladresse im Netzwerk gelangen.

Warum ist das wichtig? Denken Sie daran, dass das Internet ein globales Netzwerk miteinander verbundener Geräte ist. Damit Ihr Gerät eine Anfrage senden kann, muss es zuerst wissen, wohin die Daten gesendet werden sollen. Genau dafür sind IP-Adressen da, die Grundlage des Internet Protocol (IP). Sie identifizieren die Geräte im Netzwerk eindeutig, sodass das Quellgerät weiß, wohin es die Daten senden muss.

Das Internet Protocol allein ist verbindungslos, was bedeutet, dass jede Dateneinheit (ein sogenanntes „Paket“) unabhängig an ihr Ziel im Netzwerk gesendet wird, ohne vorher eine Verbindung herzustellen oder sicherzustellen, dass sie korrekt oder in der richtigen Reihenfolge ankommt. Wenn die Pakete ihr Ziel erreichen, wird die Quelle darüber überhaupt nicht benachrichtigt.

Hier kommen Protokolle wie TCP ins Spiel.

Das **TCP**-Protokoll, bekannt als **Transmission Control Protocol**, zerlegt die Daten in kleine Segmente. Diese grundlegenden Dateneinheiten werden nummeriert und bei Erreichen ihres Ziels mit speziellen Algorithmen wieder zusammengesetzt, die prüfen, ob die Daten vollständig und korrekt sind. Diese Algorithmen erkennen auch Fehler und verlorene Pakete, sodass sie die Quelle bei Bedarf auffordern können, diese erneut zu senden. So wird sichergestellt, dass die Informationen korrekt empfangen werden.

Gemeinsam legen das Transmission Control Protocol und das Internet Protocol die Regeln und Standards fest, die notwendig sind, um eine zuverlässige Datenübertragung über das Internet zu gewährleisten.

Neben TCP/IP gibt es ein weiteres sehr wichtiges Protokoll für das moderne Internet: HTTP.

HTTP arbeitet auf der Anwendungsebene.

**HTTP** steht für **HyperText Transfer Protocol**. Es definiert die Regeln und Standards für die Kommunikation zwischen dem Client (wie Ihrem Browser) und dem Server.

Wenn Sie versuchen, eine Website in Ihrem Browser zu öffnen, sendet der Browser eine HTTP-Anfrage an den Server, in der er nach dieser bestimmten Ressource und ihren zugehörigen Assets fragt. Der Server empfängt die HTTP-Anfrage, führt seine Logik aus und gibt eine HTTP-Antwort an den Client zurück, die den angeforderten Inhalt oder eine Information enthält, warum die Anfrage nicht erfolgreich war.

HTTP-Anfragen und HTTP-Antworten haben sehr ähnliche Strukturen, die Folgendes umfassen:

* Eine Anforderungszeile (die Startzeile)
    
* Kopfzeilen
    
* Body (optional)

## **HTTP-Anfragen**

Die **Anforderungszeile** einer HTTP-Anfrage ist in drei Hauptteile gegliedert:

* Eine Methode, die beschreibt, welche Aktion angefordert wird. Zum Beispiel `GET` zum Abrufen von Daten und `POST` zum Absenden von Daten. Weitere gängige Methoden sind `PUT`, `PATCH`, `DELETE` und `HEAD`.
    
* Ein Pfad oder URI, der die spezifische Ressource beschreibt, die der Client anfordert.
    
* Die Version des HTTP-Protokolls, die für die Kommunikation verwendet wird. Zum Beispiel `HTTP/1.1`.

Dies ist ein Beispiel mit der `GET`-Methode auf dem Stamm-Pfad `/` und der HTTP-Version 1.1:

`GET / HTTP/1.1`

Die **Kopfzeile** enthält Schlüssel-Wert-Paare mit Informationen über die Anfrage.

Zum Beispiel: `Content-Type: application/json`.

Unterhalb der Kopfzeile befindet sich eine Leerzeile, die anzeigt, dass die Metadaten (zusätzliche Informationen zur Anfrage) vollständig sind.

Und schließlich der **Body**. Dieser ist optional. Er wird gesendet, wenn der Client Informationen an den Server übermitteln muss, zum Beispiel beim Absenden neuer Daten mit der `POST`-Methode.

Dies ist ein Beispiel für eine HTTP-Anfrage:

```md
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Connection: keep-alive
```

## **HTTP-Antworten**

Sie haben eine sehr ähnliche Struktur:

* Eine Statuszeile
    
* Kopfzeile
    
* Body
    

Zuerst enthält die **Statuszeile** Informationen über das Ergebnis der Anfrage. Üblicherweise finden Sie hier:

* Die HTTP-Version, die für die Kommunikation verwendet wird. Zum Beispiel `HTTP/1.1`.
    
* Den Statuscode, der das Ergebnis der Anfrage angibt.
    
* Eine Begründung, eine kurze, für Menschen lesbare Beschreibung des Statuscodes. Diese hilft zu verstehen, was passiert ist.
    

Gängige Statuscodes sind:

* `200 OK`
    
* `201 CREATED`
    
* `400 BAD REQUEST`
    
* `404 NOT FOUND`
    
* `403 FORBIDDEN`
    
* `500 INTERNAL SERVER ERROR`
    

Es gibt viele Statuscodes. Sie werden anhand ihrer ersten Ziffer von links nach rechts kategorisiert.

* `1xx` für Informationsantworten.
    
* `2xx` für erfolgreiche Antworten.
    
* `3xx` für Umleitungsnachrichten.
    
* `4xx` für Client-Fehlerantworten.
    
* `5xx` für Server-Fehlerantworten.
    

Unterhalb der Statuszeile einer HTTP-Antwort finden wir die **Kopfzeile** mit Metadaten zur Antwort. Zum Beispiel `Content-Type: text/html`. Diese kann auch die Inhaltslänge, das Datum, den Ort und weitere Informationen enthalten, die für den Client hilfreich sein können.

Und schließlich finden wir den HTTP-Antwort-**Body**, der die Daten enthält, die der Client angefordert hat, oder eine Erklärung, was passiert ist, falls die Anfrage nicht erfolgreich war. Dieser ist optional, wird aber in den meisten Antworten mitgesendet.

Dies ist ein Beispiel für eine HTTP-Antwort:

```md
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105
Connection: keep-alive

<!DOCTYPE html>
<html>
<head><title>Home</title></head>
<body><h1>Welcome!</h1></body>
</html>
```

Jetzt, wo Sie HTTP kennen, sind Sie vielleicht neugierig auf HTTPS.

Sie können HTTPS als eine „sichere“ Version von HTTP betrachten, bei der die zwischen Browser und Server gesendeten Daten aus Sicherheitsgründen verschlüsselt werden. Der Großteil des modernen Webverkehrs nutzt HTTPS.

Protokolle wie TCP/IP und HTTP haben das World Wide Web und das Internet insgesamt zu dem unverzichtbaren Werkzeug gemacht, auf das die meisten von uns jeden Tag angewiesen sind. Dank ihnen können Geräte über ein Netzwerk richtig kommunizieren und weltweit Daten austauschen.
