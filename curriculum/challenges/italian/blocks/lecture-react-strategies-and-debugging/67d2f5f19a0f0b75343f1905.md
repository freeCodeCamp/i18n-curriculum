# --description--

I React Server Components (RSC) sono una nuova tendenza che ha cambiato il modo in cui gli sviluppatori React affrontano le cose. Con gli RSC, più lavoro viene spostato sul server, il che porta molti vantaggi.

Diamo un’occhiata a cosa sono i server component, come funzionano e cosa ha portato all’introduzione dei server component.

I React Server Components sono componenti React che vengono renderizzati esclusivamente sul server, che invia solo l’HTML finale al client. Questo significa che quei componenti possono consultare direttamente risorse lato server e ridurre drasticamente la quantità di JavaScript inviato al browser.

Le app React hanno tradizionalmente usato un sistema di "client component" che gestisce tutto in una tipica app React, come il rendering, l’interattività e gli effetti collaterali. Il termine "client component" è stato usato raramente fino all’introduzione recente dei React Server Components.

Ma il sistema dei client component presenta alcuni svantaggi, come bundle JavaScript molto grandi e tempi di caricamento iniziale più lenti.

Framework React come Next.js e Gatsby hanno trovato soluzioni per spostare alcuni processi sul server per risolvere questi problemi, ma nessuna di queste era standardizzata. Se hai usato uno di questi framework, probabilmente hai sentito parlare di `getServerSideProps` e `getServerData`.

Poi sono arrivati i React Server Components, che ti permettono di eseguire alcuni componenti interamente sul server così puoi fare cose come il recupero dati e i calcoli prima che qualsiasi codice venga eseguito nel browser dell’utente.

I server component sono stati resi popolari per la prima volta e sono disponibili subito in Next.js. Altri framework come Remix e Gatsby stanno recuperando terreno, e c’è un plugin sperimentale per Vite chiamato `vite-plugin-react-server` che ti permette di buildare server component.

Ma come funzionano i server component?

Uno dei modi migliori per dimostrare i React Server Components è con il recupero dati.

Nei tradizionali client component React, lasci che il browser gestisca le richieste API. Poiché il recupero dati è un effetto collaterale, fai quella chiamata API in un hook `useEffect`.

È anche buona pratica impostare variabili di stato come loading, data e error così puoi indicare che i dati stanno caricando, mostrare i dati quando sono pronti o mostrare un errore nella tua app.

Con i React Server Components, puoi spostare l’intero componente sul server e recuperare i dati lì senza dover usare `useState` o `useEffect`:

```jsx
const Users = async () => {
  const res = await fetch("https://jsonplaceholder.typicode.com/users");
  const users = await res.json();

  return (
    <>
      <h1 className="text-4xl text-center mt-6">Users</h1>
      <ul className="text-center mt-3">
        {users.map((user) => (
          <li key={user.id}>{user.name}</li>
        ))}
      </ul>
    </>
  );
};

export default Users;
```

Poiché i React Server Components vengono eseguiti solo sul server, puoi semplicemente recuperare i dati da un’API e renderizzare una sola volta. Inoltre, dato che il recupero dati avviene sul server, più vicino alla fonte, la tua app potrebbe avere prestazioni migliori, specialmente per chi ha connessioni di rete lente.

Un grosso limite è che tutto il codice per i server component rimane sul server e non viene inviato al browser. Questo significa che non puoi usare React hooks con loro e non hanno accesso alle Web API o ai listener di eventi del browser. Quindi come puoi aggiungere interattività?

Nel router app di Next.js, tutti i componenti sono server component per impostazione predefinita. Se vuoi aggiungere interattività, devi contrassegnare il componente come client component con la direttiva `"use client"`.

Supponiamo che tu voglia trasformare l’esempio precedente in un client component. Ecco come puoi farlo:

```jsx
"use client";

import { useState, useEffect } from "react";

const Users2 = () => {
  const [status, setStatus] = useState({
    users: [],
    loading: true,
    error: null,
  });

  async function fetchUsers() {
    try {
      const res = await fetch("https://jsonplaceholder.typicode.com/users");
      const data = await res.json();
      setStatus((prevStatus) => ({
        ...prevStatus,
        users: data,
        loading: false,
      }));
    } catch (err) {
      setStatus((prevStatus) => ({
        ...prevStatus,
        error: err.message,
        loading: false,
      }));
    }
  }

  useEffect(() => {
    fetchUsers();
  }, []);

  if (status.loading) {
    return <p>Loading Users...</p>;
  }
  if (status.error) {
    return <p>Error getting users: {status.error}</p>;
  }

  return (
    <>
      <h1 className="text-4xl text-center mt-6">Users</h1>
      <ul className="text-center mt-3">
        {status.users.map((user) => (
          <li key={user.id}>{user.name}</li>
        ))}
      </ul>
    </>
  );
};

export default Users2;
```

Se vuoi aggiungere interattività come eventi di clic, il componente deve essere contrassegnato come client component:

```jsx
"use client";

import { useState } from "react";

const Counter = () => {
  const [count, setCount] = useState(0);

  return (
    <>
      <h1>Counter</h1>
      <button onClick={() => setCount(count + 1)}>Increment</button>
      <h2>{count}</h2>
      <button onClick={() => setCount(count - 1)}>Decrement</button>
    </>
  );
};

export default Counter;
```

Se non aggiungi la direttiva use client al componente, ottieni un errore con un messaggio che dice "Stai importando un server component che necessita di `useState`. Questo hook React funziona solo in un client component. Per risolvere, contrassegna il file (o il suo genitore) con la direttiva `"use client"`."

I principali vantaggi dei React Server Components sono che il recupero dati diventa più semplice, il codice è più facile da leggere e la complessità lato client si riduce.
