## Робота з Node.js та подієво-орієнтованою архітектурою

- **Node.js**: Відкрите, кросплатформне середовище виконання JavaScript, яке дозволяє запускати JavaScript-код поза браузером. Широко використовується для створення вебсерверів, API та back end систем.
- **Обмеження на стороні клієнта**: JavaScript на стороні клієнта має обмежений доступ до файлової системи, обмежену підтримку складної логіки застосунку та потенційні ризики безпеки (наприклад, не можна відкривати доступ до облікових даних бази даних у коді на стороні клієнта).
- **Глобальний об’єкт у браузері та Node.js**: У браузері глобальний об’єкт — це `window`, який відкриває доступ до маніпуляцій DOM, куків і подій браузера. У Node глобальний об’єкт — `global`, який відкриває вбудовані модулі для роботи з файлами, мережею та ОС.
- **Доступ до DOM**: Браузер надає доступ до DOM API, а Node — ні. Натомість Node має доступ майже до всіх системних ресурсів, включно з файловою системою, чого браузер не може.
- **Контроль версій**: Ви можете обирати, яка версія Node запускається на вашому сервері. Ви не контролюєте, яку версію браузера використовують ваші користувачі.
- **Неблокуюча, подієво-орієнтована архітектура**: Node використовує один потік і цикл подій, який ефективно обробляє багато одночасних запитів і операцій вводу/виводу без блокування, що робить його ідеальним для реального часу.
- **Функції зворотного виклику**: Функції, які визначають, що відбувається після завершення асинхронної операції. Велика залежність від них ускладнює читання та налагодження асинхронного коду Node.
- **Переваги Node на back end**: Одна мова (JavaScript) для front-end і back-end, неблокуюча архітектура, що підходить для високої конкуренції, велика екосистема npm, безкоштовний і відкритий.
- **Недоліки Node на back end**: Однопотоковий, тому ресурсоємні задачі (обробка зображень, криптографія, складна математика) можуть блокувати цикл подій, велика залежність від асинхронного/функцій зворотного виклику ускладнює читання, пакети npm різняться за якістю та обслуговуванням.
- **NVM (Node Version Manager)**: Інструмент для керування та перемикання між кількома версіями Node.js на одній машині. На macOS/Linux встановлюється за допомогою:

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **Встановлення LTS-версії Node**: `nvm install --lts`
- **Перемикання версій Node**: `nvm use 20`
- **Перегляд усіх встановлених версій Node**: `nvm ls`
- **Встановлення версії Node за замовчуванням (macOS/Linux)**: `nvm alias default 22.20.0`
- **nvm-windows**: Окремий проєкт для керування версіями Node на Windows (не WSL). Завантажте інсталятор з офіційного репозиторію і виконайте `nvm install lts`, потім `nvm use <version>`.
- **Запуск JavaScript-файлу через Node**: `node app.js`

## Робота з основними модулями Node

### Модуль `fs`

- **Призначення**: Надає методи для створення, читання, запису, додавання та видалення файлів і папок.
- **Три стилі використання**: На основі функцій зворотного виклику (асинхронний), на основі промісів (`fs/promises`) і синхронний (`Sync` у назві). Синхронні методи блокують цикл подій і підходять лише для скриптів.
- **`writeFile()`**: Записує вміст у файл, створюючи його, якщо він не існує:

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**: Додає вміст до існуючого файлу, не перезаписуючи його.
- **`readFile()`**: Зчитує вміст файлу. Передайте `"utf8"` як кодування, щоб отримати рядок, або пропустіть його, щоб отримати `Buffer`.
- **`unlink()`**: Видаляє файл. Викидає помилку `ENOENT`, якщо файл не існує.

### Модуль `Buffer`

- **Призначення**: Обробляє двійкові дані (файли, зображення, мережеві потоки) безпосередньо в пам’яті.
- **`Buffer.from()`**: Створює буфер із рядка:

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**: Створює буфер з `n` байт, заповнений нулями:

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **Обрізання**: Запис більше даних, ніж може вмістити буфер, тихо обрізає зайве.
- **`Buffer.byteLength(str)`**: Повертає кількість байт, необхідних для збереження рядка.

### Модуль `crypto`

- **Призначення**: Надає криптографічні інструменти, такі як хешування, HMAC, шифрування/дешифрування, цифрові підписи та безпечне генерування випадкових чисел.
- **`createHash(algorithm)`**: Створює односторонній хеш (незворотний). Використовуйте `.update(data).digest("hex")`, щоб передати дані для хешування і отримати хеш у шістнадцятковому форматі:

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**: Як `createHash`, але потребує секретного ключа, підходить для перевірки цілісності даних і автентифікації.
- **`createCipheriv()` / `createDecipheriv()`**: Шифрують і дешифрують дані. Обидва вимагають алгоритм, ключ і вектор ініціалізації (IV):

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**: Генерує криптографічно безпечні випадкові байти, безпечніші за `Math.random()` для токенів і ключів.
- **`randomInt(min, max)`**: Генерує безпечне випадкове ціле число в заданому діапазоні. Корисно для OTP.
- **`sign()` / `verify()`**: Створюють і перевіряють цифрові підписи за допомогою пари приватного/публічного ключів.

### Модуль `os`

- **Призначення**: Надає інформацію про операційну систему, на якій працює Node.
- **`os.platform()`**: Повертає ідентифікатор ОС, наприклад `'darwin'`, `'win32'`, `'linux'`.
- **`os.arch()`**: Повертає архітектуру центрального процесора, наприклад `'x64'`, `'arm64'`.
- **`os.type()`**: Повертає офіційну назву ОС, наприклад `'Darwin'`.
- **`os.release()`**: Повертає рядок версії ядра ОС.
- **`os.cpus()`**: Повертає масив об’єктів, що описують кожне логічне ядро CPU (модель, швидкість, часи).
- **`os.uptime()`**: Повертає час роботи системи в секундах.
- **`os.totalmem()` / `os.freemem()`**: Повертають загальну і доступну системну пам’ять у байтах.
- **`os.userInfo()`**: Повертає об’єкт з інформацією про поточного користувача: `uid`, `gid`, `username`, `homedir` і `shell`.
- **`os.networkInterfaces()`**: Повертає об’єкти мережевих інтерфейсів з адресою, сімейством, MAC і CIDR.

### Модуль `path`

- **Призначення**: Надає утиліти для роботи з шляхами до файлів і папок у кросплатформенний спосіб.
- **`__filename`**: Глобальна змінна, абсолютний шлях до поточного файлу.
- **`__dirname`**: Глобальна змінна, абсолютний шлях до папки, що містить поточний файл.
- **`path.basename(p)`**: Повертає останній компонент шляху (ім’я файлу).
- **`path.dirname(p)`**: Повертає директорію шляху.
- **`path.extname(p)`**: Повертає розширення файлу, наприклад `'.js'`.
- **`path.join(...segments)`**: Об’єднує сегменти шляху в нормалізований шлях і виправляє помилкові слеші.
- **`path.resolve(...segments)`**: Розв’язує сегменти в абсолютний шлях, починаючи з поточної робочої директорії.
- **`path.parse(p)`**: Повертає об’єкт з властивостями `root`, `dir`, `base`, `ext` і `name`.
- **`path.format(obj)`**: Створює рядок шляху з об’єкта з властивостями `dir`, `name` і `ext`.

### Модуль `process`

- **Призначення**: Надає інформацію та контроль над поточним процесом Node.js. Це глобальний об’єкт, тому імпорт не потрібен.
- **`process.env`**: Об’єкт, що містить усі змінні середовища. Доступ до конкретних значень, наприклад `process.env.NODE_ENV`.
- **`process.argv`**: Масив аргументів командного рядка. Індекс 0 — виконуваний файл Node, індекс 1 — шлях до скрипта, індекси 2+ — аргументи користувача.
- **`process.cwd()`**: Повертає поточну робочу директорію.
- **Події процесу**:

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**: Викликає власне попередження, яке можна перехопити за допомогою слухача `warning`.

### Модуль `stream`

- **Призначення**: Обробляє великі обсяги даних частинами, замість завантаження всього в пам’ять одразу.
- **Чотири типи потоків**: Readable (читання даних частинами), Writable (запис даних частинами), Duplex (обидва), Transform (Duplex, що змінює дані під час проходження).
- **З’єднання потоків (piping)**:

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **Подія `finish`**: Виникає на Writable-потоці, коли всі дані записані.
- **Подія `error`**: Виникає, коли у потоці сталася помилка.

## Вступ до npm

- **npm**: Складається з трьох частин: вебсайт `npmjs.com`, публічний реєстр (база даних пакетів) і CLI-інструмент, встановлений разом з Node.js.
- **Призначення**: Дозволяє встановлювати, керувати та публікувати повторно використовувані JavaScript-пакети. npm — найпопулярніший менеджер пакетів.
- **Альтернативи**: Yarn, PNPM, Bun.

### Файл `package.json`

- **Призначення**: Маніфест конфігурації npm. Відстежує назву проєкту, версію, опис, точку входу (`main`), автора, ліцензію, скрипти та залежності.
- **Створення `package.json` інтерактивно**: `npm init`
- **Створення з параметрами за замовчуванням**: `npm init -y`
- **`dependencies`**: Пакети, потрібні для роботи застосунку в продакшені (наприклад, `express`, `react`).
- **`devDependencies`**: Пакети, потрібні лише під час розробки і тестування (наприклад, `jest`, `nodemon`, `eslint`).
- **`license`**: Юридичні умови використання. Загальні значення: `MIT`, `ISC`, `GPL`.

### Семантичне версіонування (SemVer)

- **Формат**: `MAJOR.MINOR.PATCH`, наприклад, `4.17.21`.
- **MAJOR**: Збільшується при сумісних зламних змінах.
- **MINOR**: Збільшується при додаванні нових сумісних функцій.
- **PATCH**: Збільшується при сумісних виправленнях помилок.
- **Карет `^`**: Дозволяє оновлення мінорних і патч-версій, але не мажорних. Приклад: `^4.17.21` приймає `4.18.0`, але не `5.0.0`.
- **Тильда `~`**: Дозволяє оновлення лише патч-версії. Приклад: `~1.2.3` приймає `1.2.4`, але не `1.3.0`.
- **Зірочка `*`**: Відповідає будь-якій версії, підходить лише для тестування, ніколи для продакшену.
- **Точна версія**: Без префікса, наприклад `1.2.3` фіксує саме цю версію.

### Встановлення та видалення залежностей

- **Встановити пакет**: `npm install express` (скорочено: `npm i express`)
- **Встановити як devDependency**: `npm install nodemon -D`
- **Видалити пакет**: `npm uninstall chalk`
- **`node_modules/`**: Папка, створена після встановлення, що містить фактичний код пакетів. Не комітиться у систему контролю версій.
- **`package-lock.json`**: Автоматично згенерований файл, що фіксує точні версії всіх встановлених пакетів (включно з дочірніми залежностями). Завжди комітіть цей файл і не редагуйте вручну.
- **Встановити конкретну версію**: `npm install express@4.21.2`
- **Переглянути доступні версії**: `npm view express versions`
- **Перевірити застарілі пакети**: `npm outdated`
- **Оновити один пакет** (в межах дозволеного діапазону): `npm update express`
- **Примусово встановити останню версію**: `npm install express@latest`

### Публікація в реєстр npm

- **Вхід**: `npm login`
- **Пакети з областю (scoped)**: Використовуйте `@username/package-name`, щоб уникнути конфліктів імен.
- **`.npmignore`**: Перелік файлів, які виключаються з публікації пакета (подібно до `.gitignore`).
- **Публікація непрофільного пакета**: `npm publish`
- **Публікація профільного пакета публічно**: `npm publish --access public`

## Робота зі скриптами npm

- **Скрипти npm**: Користувацькі команди, визначені в об’єкті `"scripts"` файлу `package.json`. Вони автоматизують повторювані завдання, такі як запуск сервера, тестування, лінтинг і збірка.
- **Визначення скрипта**:

```json
"scripts": {
  "start": "node app.js"
}
```

- **Запуск скрипта**: `npm run start`
- **Скорочення для вбудованих імен скриптів** (`start`, `test`, `stop`, `restart`): `npm start`
- **Послідовні команди з `&&`** (наступна команда виконується лише якщо попередня успішна):

```json
"start": "npm run build && node server.js"
```

- **Послідовні команди з `;`** (наступна команда виконується завжди):

```json
"start": "npm run build; node server.js"
```

- **Паралельні команди з `&`**: `"dev": "npm run start:server & npm run start:client"`
- **Передача аргументів у скрипт** (використовуйте роздільник `--`): `npm start -- 8000`; аргументи доступні через `process.argv[2]`.

### Модулі CommonJS

- **CommonJS**: Початкова система модулів Node.js. Завантажує модулі синхронно.
- **Імпорт за допомогою `require()`**:

```js
const { multiply } = require('./math');
```

- **Експорт за допомогою `module.exports`**:

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- CommonJS є за замовчуванням, коли в `package.json` є поле `"type": "commonjs"` (або відсутнє поле `"type"`), або файл має розширення `.cjs`.

### ES-модулі (ESM)

- **ES-модулі**: сучасний, стандартизований формат модулів. Завантажуються асинхронно.
- **Іменований експорт**: `export function multiply(a, b) { return a * b; }`
- **Експорт за замовчуванням**: `export default function multiply(a, b) { return a * b; }`
- **Імпорт**: `import { multiply } from './math.mjs';`
- ESM використовується, коли файл має розширення `.mjs` або `package.json` містить `"type": "module"`.

## Розуміння роботи HTTP, DNS і TCP/IP

### Сервери та модель клієнт-сервер

- **Модель клієнт-сервер**: Архітектура розподіленої системи, де компоненти є або клієнтами (процеси, що надсилають запити, наприклад, веббраузер), або серверами.
- **Сервер**: Комп’ютер або програмне забезпечення, що приймає запити від клієнтів і повертає дані або послуги через мережу.
- **Типи серверів**: Вебсервери (HTML/CSS/ресурси), сервери застосунків (бізнес-логіка), сервери баз даних, поштові сервери, файлові сервери, проксі-сервери.

### DNS

- **DNS (система доменних імен)**, часто називають «телефонною книгою інтернету»: перетворює доменні імена, зрозумілі людині (наприклад, `freecodecamp.org`), у IP-адреси.
- **Ієрархія доменних імен** (справа наліво): домен верхнього рівня (TLD, наприклад, `.org`) → домен другого рівня (наприклад, `freecodecamp`) → піддомен (наприклад, `www`).
- **IP-адреса**: унікальний числовий ідентифікатор пристрою в мережі. Приклад IPv4: `127.0.0.1`. Приклад IPv6: `0:0:0:0:0:0:0:1`.
- **Процес розв’язання DNS**: браузер → рекурсивний резолвер → кореневі сервери → сервер TLD → авторитетний сервер імен → повертає IP-адресу → рекурсивний резолвер кешує і доставляє клієнту.

### TCP/IP та HTTP

- **IP (Інтернет-протокол)**: маршрутизує пакети даних від джерела до призначення за допомогою IP-адрес. Без встановлення з’єднання, кожен пакет надсилається окремо.
- **TCP (протокол керування передачею)**: розбиває дані на пронумеровані сегменти, збирає їх на приймальній стороні, перевіряє на помилки та запитує повторну передачу за потреби. Забезпечує надійну та впорядковану доставку.
- **HTTP (протокол передачі гіпертексту)**: протокол прикладного рівня, що визначає, як клієнти та сервери обмінюються веб-вмістом.
- **HTTPS**: захищена версія HTTP, де дані шифруються за допомогою SSL/TLS.

### Структура HTTP-запиту

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **Рядок запиту**: HTTP-метод + шлях/URI + версія HTTP.
- **Заголовки**: пари ключ-значення з метаданими.
- **Тіло** (необов’язкове): надсилається з `POST`, `PUT`, `PATCH`, `DELETE` для передачі даних.

### Структура HTTP-відповіді

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **Рядок статусу**: версія HTTP + код статусу + пояснювальна фраза.
- **Заголовки**: метадані про відповідь (тип вмісту, довжина, директиви кешування).
- **Тіло** (необов’язкове): запитані дані.

### Категорії кодів стану HTTP

- **1xx Інформаційні**: запит отримано, обробка триває. Приклад: `101 Switching Protocols`.
- **2xx Успіх**: `200 OK`, `201 Created`, `204 No Content`.
- **3xx Перенаправлення**: `301 Moved Permanently`, `302 Found`, `304 Not Modified`.
- **4xx Помилки клієнта**: `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.
- **5xx Помилки сервера**: `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable`.

## Розуміння моделі запит-відповідь HTTP

- **HTTP-методи**: `GET` (отримати), `POST` (створити), `PUT` (повне оновлення), `PATCH` (часткове оновлення), `DELETE` (видалити).
- **Цикл запит-відповідь**: клієнт надсилає запит → сервер перевіряє автентифікацію, метод і URL, виконує запит до бази даних → сервер формує відповідь (код статусу + необов’язкове тіло) → клієнт обробляє відповідь.

### Ресурси, що повертаються у відповідях HTTP

Кожна відповідь містить заголовок `Content-Type`, який повідомляє браузеру, як обробляти ресурс:

- **HTML**: `text/html`
- **CSS**: `text/css`
- **JavaScript**: `application/javascript`
- **JSON**: `application/json`
- **PNG/JPEG/WebP**: `image/png`, `image/jpeg`, `image/webp`
- **MP4/MP3**: `video/mp4`, `audio/mpeg`
- **PDF**: `application/pdf`
- **WOFF2**: `font/woff2`

### Надсилання HTML-форм

- **атрибут `action`**: вказує URL, куди надсилаються дані форми.
- **атрибут `method`**: вказує HTTP-метод (`GET` або `POST`). HTML не підтримує нативно `PUT`, `PATCH` або `DELETE`, замість цього використовуйте Fetch API або пакет `method-override` в Express.
- **Валідація на стороні клієнта**: використовуйте атрибути `required` і `pattern` для миттєвого зворотного зв’язку. Завжди виконуйте валідацію і на сервері.
- **Форма `GET`**: дані додаються до URL як параметри запиту (наприклад, пошукові форми).
- **Форма `POST`**: дані надсилаються в тілі запиту (наприклад, форми входу та реєстрації).

## Розуміння моделі веб-стандартів

### Організації зі стандартизації

- **WHATWG**: підтримує HTML і DOM як безперервно оновлюваний «живий стандарт». Рішення приймають основні команди браузерних рушіїв (Chromium, Gecko, WebKit).
- **W3C (Консорціум Всесвітньої павутини)**: головний орган стандартизації. Контролює CSS, SVG, керівництва з доступності (WCAG) та численні веб-API.
- **ECMA International / TC39**: Ecma публікує офіційну специфікацію ECMAScript. TC39 розробляє мову через п’ятиетапний процес пропозицій (етап 0 — ідея → етап 4 — повна специфікація).
- **Khronos Group**: зосереджена на графіці, обчисленнях і медіа. Створює WebGL і WebGL 2 для апаратного 3D-відтворення в браузерах.

### Створення веб-стандартів

1. **Виявлення потреби**: розробник або організація стикається з проблемою і записує її.
2. **Пропозиція та раннє обговорення**: неформальний GitHub-issue або пояснювальний документ для оцінки інтересу. У TC39 це етапи 0/1.
3. **Написання специфікації**: точний формальний документ, що охоплює всі крайні випадки, помилки та питання безпеки.
4. **Огляд, заперечення та ітерації**: чернетка публічно переглядається виробниками браузерів, функції, які відмовляються впроваджувати, рідко стають стандартами.
5. **Реалізація та тестування**: виробники впроваджують за прапорцем функції, спільний набір тестів гарантує однакову поведінку у всіх браузерах.
6. **Фіналізація**: W3C називає це рекомендацією, TC39 — етапом 4.

### Життєвий цикл функції веб-стандартів

1. **Пропозиція**: проблема виявлена і описана.
2. **Інкубація**: відкрите обговорення, багато пропозицій відмирають тут.
3. **Специфікація**: формальний документ з точними правилами.
4. **Тестування сумісності**: набір тестів W3C Web Platform Tests перевіряє кросбраузерну узгодженість.
5. **Фіналізація**: функція виходить без прапорців як частина базової веб-платформи.
6. **Застарівання**: замінені або небезпечні функції позначаються для видалення, але браузери зберігають їх для сумісності з минулим.

### Ключові принципи веб-стандартів

- **Відкритість**: стандарти розробляються відкрито і впроваджуються без ліцензійних платежів.
- **Доступність**: інтегрована в HTML, CSS і поведінку браузера, WCAG надає формальні керівництва.
- **Інтероперабельність**: сторінка, створена за стандартом, має однаково працювати у всіх браузерах.
- **Зворотна сумісність**: старі сторінки мають працювати в сучасних браузерах.
- **Конфіденційність і безпека**: нові API, які можуть відкривати дані користувачів, проходять суворий контроль.
- **Шарування**: HTML для структури, CSS для презентації, JavaScript для поведінки. Кожен шар можна впроваджувати окремо.
- **Прогресивне покращення**: функції мають спочатку працювати для всіх, а потім покращуватися для користувачів із сучаснішими браузерами.

## Розуміння REST API та веб-сервісів

### Веб-сервіси

- **API (інтерфейс програмування застосунків)**: загальний механізм для взаємодії програмного забезпечення з іншим ПЗ.
- **Веб-сервіс**: специфічний тип API, що використовує веб-протоколи (HTTP/HTTPS) для обміну даними по мережі. Кожен веб-сервіс — це API, але не кожен API — веб-сервіс.
- **Характеристики**: працює на сервері, забезпечує взаємодію між машинами, не залежить від мови чи платформи, використовує стандартні веб-протоколи.
- **REST**: передача репрезентативного стану, найпоширеніший стиль веб-сервісів. Простий, гнучкий, природно працює з HTTP.
- **SOAP**: простий протокол доступу до об’єктів, використовує лише XML, поширений у корпоративних системах із суворими стандартами.
- **XML-RPC**: використовує XML для кодування легких запитів і відповідей.

### Архітектура REST

- **Безстанність**: кожен запит обробляється незалежно. Сервер не зберігає стан сесії між запитами. Кожен запит має містити всю необхідну інформацію (наприклад, токен автентифікації у кожному виклику).
- **Ресурси**: все є ресурсом, ідентифікованим URL, наприклад, `/products` для колекції, `/products/1` для конкретного елемента.
- **HTTP-методи в REST**: `GET` (отримати), `POST` (створити), `PUT`/`PATCH` (оновити), `DELETE` (видалити).
- **Формати відповіді**: найпоширеніший — JSON, також підтримуються XML і plain text. Клієнти вказують формат через заголовок `Accept`:

```bash
Accept: application/json
```

### Мікросервіси

- **Моноліт**: один великий кодовий базис, де UI, база даних і API розміщені разом. Складно підтримувати і масштабувати з ростом складності.
- **Мікросервіси**: архітектурний підхід, коли великий застосунок розбивається на маленькі незалежні сервіси, кожен відповідає за одну бізнес-функцію. Сервіси взаємодіють через API або черги повідомлень.
- **Переваги**: швидша незалежна розробка і розгортання, краща ізоляція помилок, різні сервіси можуть використовувати різні мови або фреймворки.
- **Недоліки**: складніше налагодження (проблеми поширюються між сервісами), більші накладні витрати на інфраструктуру, збільшене мережеве спілкування.

## Робота з Express

- **Express.js**: мінімалістичний і гнучкий веб-фреймворк на базі Node.js. Спрощує створення веб-серверів і API з набагато меншим шаблонним кодом, ніж нативний модуль `http`.
- **Базовий сервер Express**:

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**: фабрична функція, що створює екземпляр застосунку Express (`app`).
- **`app.listen(port, callback)`**: запускає сервер і починає прослуховувати HTTP-запити на вказаному порту.
- **`req`**: об’єкт запиту, містить URL, заголовки, параметри, тіло і дані запиту.
- **`res`**: об’єкт відповіді, використовується для відправлення даних клієнту.
- **Middleware**: код, що виконується між отриманням запиту і відправленням відповіді.

## Розуміння маршрутизації в Express.js

- **Маршрутизація**: визначає, як застосунок реагує, коли клієнт заходить на конкретний URL з певним HTTP-методом.
- **`app.get/post/put/delete(path, handler)`**: реєструє обробники для відповідних HTTP-методів.
- **Статичний маршрут**: фіксований URL-шаблон, наприклад, `app.get("/home", handler)`.
- **Динамічний маршрут з параметрами**: використовує заповнювач `:paramName`. Захоплюється через `req.params`:

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**: параметри маршруту (`:id` сегменти).
- **`req.query`**: параметри рядка запиту (`?key=value`).
- **`req.body`**: тіло запиту (потребує middleware для парсингу тіла).
- **`res.send(data)`**: відправляє відповідь (текст, HTML, JSON або бінарні дані).
- **`res.json(obj)`**: відправляє JSON-відповідь, автоматично встановлює `Content-Type: application/json`.
- **`res.status(code).send(msg)`**: встановлює код статусу і відправляє відповідь.
- **`res.redirect(url)`**: перенаправляє клієнта на інший URL.
- **`res.render(view, data)`**: відтворює шаблон і відправляє HTML.

### Ланцюжкові обробники маршрутів з `app.route()`

Групує кілька HTTP-методів для одного шляху, щоб зменшити повторення:

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

Створює модульний міні-застосунок для групування пов’язаних маршрутів:

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

### Обслуговування статичних файлів

- **`express.static(directory)`**: вбудоване middleware, що обслуговує файли з вказаної директорії:

```js
app.use(express.static(path.join(__dirname, "public")));
```

- Обслуговування з кількох директорій з монтувальним шляхом: `app.use("/images", express.static(path.join(__dirname, "images")));`

## Розуміння обробки помилок і перевірок стану

### Обробка помилок в Express

- **Middleware для обробки помилок**: визначається чотирма параметрами `(err, req, res, next)`. Завжди розміщується в кінці стеку middleware:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **Асинхронні помилки в Express 4**: потрібно ловити вручну через `try/catch` і передавати далі з `next(err)`:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **Асинхронні помилки в Express 5**: автоматично передаються middleware для обробки помилок, ручний `next(err)` не потрібен.
- **Обробник 404**: розміщуйте після всіх маршрутів, але перед обробником помилок:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Налагодження і логування

- **Вбудоване налагодження**: увімкніть, встановивши змінну середовища `DEBUG`: `DEBUG=express:* node index.js`
- **`morgan`**: популярне стороннє middleware для логування HTTP-запитів. Логує метод, URL, код статусу і час відповіді:

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### Перевірки стану і коректне завершення роботи

- **Маршрут перевірки стану**: Повертає `200 OK`, щоб сигналізувати, що застосунок працює. Використовується балансувальниками навантаження та оркестраторами контейнерів:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**: Сигнал, який надсилається, коли процес потрібно завершити. Обробляйте його, щоб дозволити завершити поточні запити:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**: Сигнал, який надсилається при натисканні Ctrl+C під час локальної розробки. Обробляйте так само, як `SIGTERM`.
- **Позначення як нездоровий перед завершенням роботи**: Поверніть `503` з кінцевої точки перевірки стану під час завершення, щоб зупинити маршрутизацію нового трафіку балансувальником навантаження:

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## Express middleware

- **Middleware**: Функція, яка виконується під час життєвого циклу запиту. Має доступ до `req`, `res` і `next`. Викликайте `next()`, щоб передати керування наступному middleware, або пропустіть виклик, щоб завершити цикл.
- **Middleware рівня застосунку**: Прив’язане до всього застосунку через `app.use()`. Виконується для кожного вхідного запиту.
- **Middleware рівня маршрутизатора**: Прив’язане до конкретного екземпляра `express.Router()` через `router.use()`. Застосовується лише до маршрутів, оброблених цим маршрутизатором.
- **Middleware для обробки помилок**: Визначається чотирма параметрами `(err, req, res, next)`. Розміщується внизу стеку.
- **`express.json()`**: Аналізує вхідні запити з JSON-корисним навантаженням. Робить дані доступними у `req.body`.
- **`express.urlencoded({ extended: true })`**: Аналізує URL-кодовані дані форми.
- **`express.static('public')`**: Обслуговує статичні файли з каталогу.
- **`cors`**: Увімкнює Cross-Origin Resource Sharing. Встановіть через `npm install cors`, використовуйте з `app.use(cors())`.
- **`morgan`**: Логує HTTP-запити. Використовуйте `app.use(morgan('tiny'))`.
- Middleware застосовується в порядку реєстрації через `app.use()`.

## Розуміння WebSockets

- **WebSockets**: Протокол, який створює постійне двонаправлене (full-duplex) з’єднання між клієнтом і сервером, що дозволяє обмінюватися даними в режимі реального часу.
- **Проблема HTTP для реального часу**: HTTP вимагає, щоб клієнт багаторазово опитував сервер на наявність нових даних, тому він повільний і неефективний для чатів, живих табло та багатокористувацьких ігор.
- **Рукостискання WebSocket**: Кожне WebSocket-з’єднання починається з HTTP-запиту. Клієнт запитує оновлення, а сервер відповідає `101 Switching Protocols`. Після цього з’єднання залишається відкритим, доки одна зі сторін не надішле сигнал закриття.
- **Випадки використання**: Чат-застосунки, онлайн-багатокористувацькі ігри, живі спортивні табло, стрічки фондового ринку, інструменти для спільної роботи.

### Бібліотека `ws`

- У Node.js немає вбудованого WebSocket-сервера; бібліотека `ws` — це поширений спосіб його реалізації.

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**: Надіслати повідомлення конкретному клієнту.
- **Трансляція всім підключеним клієнтам**:

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**: Властивість кожного з’єднання. `WebSocket.OPEN` означає, що з’єднання активне.

### Архітектура Pub/Sub

- **Pub/Sub (Publish/Subscribe)**: Патерн обміну повідомленнями, де **видавці** надсилають повідомлення до **топіків**, а **підписники** отримують повідомлення з топіків, на які вони підписані. Видавці і підписники не спілкуються безпосередньо.
- **Переваги над прямим зв’язком**: Роз’єднує компоненти, видавець не повинен знати, хто його підписники.
- **Pub/Sub vs. WebSockets**: WebSockets — це канал зв’язку (дорога). Pub/Sub — це патерн повідомлень, який визначає, хто отримує які повідомлення (система руху).

## Вступ до автентифікації та авторизації

### Автентифікація проти авторизації

- **Автентифікація**: Підтверджує особу, відповідає на питання «Хто ви?», має відбуватися перед авторизацією.
- **Авторизація**: Визначає права, відповідає на питання «Що вам дозволено робити?». 
- **Три фактори автентифікації**:
  - **Щось, що ви знаєте**: Пароль, PIN, відповідь на секретне питання.
  - **Щось, що ви маєте**: Смартфон, який отримує SMS-код, токен додатка-аутентифікатора.
  - **Щось, що ви є**: Відбиток пальця, розпізнавання обличчя (біометрія).
- **Загальні ролі в авторизації**: Стандартний користувач (обмежене самообслуговування), редактор (керування вмістом), адміністратор (повний доступ до системи).
- У back end коді авторизація зазвичай реалізується через middleware, яке перевіряє ролі або права користувача перед наданням доступу.

### Міжсайтове підроблення запитів (CSRF)

- **CSRF**: Атака, коли зловмисний сайт обманює браузер автентифікованого користувача, щоб той зробив небажаний запит до back end сервера, використовуючи автоматичне приєднання cookie браузером.
- **Хід атаки**: Користувач входить на `trusted-bank.com` → відвідує `evil-hacker-site.com` → зловмисний сайт надсилає приховану форму до банку → сервер банку бачить дійсний cookie сесії і виконує запит.
- **`SameSite=Strict`**: Браузер ніколи не надсилає cookie, якщо запит походить із стороннього сайту.
- **`SameSite=Lax`**: Браузер дозволяє cookie лише при безпечній навігації користувача на цільовий сайт (натискання посилання), але блокує їх при крос-сайтових відправленнях форм.
- **CSRF токени**: Сервер генерує випадковий непередбачуваний токен, вбудований у форми. Клієнт повинен включати токен у кожен запит, що змінює стан. Зловмисний сайт не може прочитати токен з фронтенду, тому підроблені запити не пройдуть валідацію.

### JSON веб-токени (JWT)

- **JWT**: Цифрово підписаний токен, виданий сервером після успішного входу. Клієнт зберігає його і надсилає з кожним наступним запитом, забезпечуючи безстанову автентифікацію (без зберігання сесії на сервері).
- **Структура JWT**: Три частини, закодовані Base64URL і розділені крапками:
  - **Заголовок**: Тип токена (`JWT`) і алгоритм підпису (наприклад, `HS256`).
  - **Корисне навантаження**: Твердження про користувача (ідентифікатор, ім’я користувача, роль). Не шифрується, тому не зберігайте тут паролі чи конфіденційні дані.
  - **Підпис**: Хеш заголовка + корисного навантаження + секретного ключа сервера. Запобігає підробці.

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

- Якщо користувач змінить корисне навантаження, підпис не співпаде, і сервер відхилить запит з `401 Unauthorized`.

### Passport.js

- **Passport.js**: Middleware для автентифікації в Node.js. Стандартизує процес автентифікації і обробляє низькорівневі деталі, щоб розробники могли зосередитися на бізнес-логіці.
- **Стратегії**: Модульні плагіни, що реалізують конкретні методи автентифікації. Існує понад 500 стратегій: `passport-local` (ім’я користувача/пароль), `passport-jwt` (JWT), `passport-google-oauth20` (вхід через Google), `passport-facebook`, `passport-github2`.
- **Інтеграція з Express**:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **Використання як middleware маршруту**:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- Після успішної автентифікації Passport зберігає частину даних користувача в cookie сесії і додає автентифікований об’єкт користувача до `req.user`.

### Helmet.js

- **Helmet.js**: Колекція middleware для безпеки, яка захищає Express-застосунки, автоматично встановлюючи, змінюючи або видаляючи HTTP-заголовки відповіді.

```js
const helmet = require('helmet');
app.use(helmet());
```

- **Видаляє `X-Powered-By`**: Запобігає зловмисникам дізнатися технологію сервера.
- **Встановлює `Content-Security-Policy`**: Обмежує джерела скриптів, зображень і стилів, зменшуючи ризик XSS-атак.
- **Встановлює `X-Frame-Options: SAMEORIGIN`**: Запобігає клікджекингу, блокуючи вбудовування сайту в зовнішні iframe.
- **Встановлює `Strict-Transport-Security`**: Змушує браузери використовувати HTTPS, захищаючи дані від перехоплення.

## Розуміння безпеки та конфіденційності у веб-застосунках

### Безпека проти конфіденційності

- **Безпека**: Технічні механізми захисту систем і даних від несанкціонованого доступу. Безпека — це замок на дверях.
- **Конфіденційність**: Управління тим, як збираються, використовуються та поширюються персональні дані. Конфіденційність — це штора на вікні.
- Ви можете мати безпеку без конфіденційності (захищена база даних може продавати ваші дані), але не можете мати конфіденційність без безпеки (незахищені дані не можуть залишатися приватними).

### Політика same-origin і CORS

- **Origin**: Визначається трьома компонентами: протоколом (наприклад, `https`), доменом (наприклад, `example.com`) та портом (наприклад, `443`). Всі три мають співпадати, щоб два URL мали однаковий origin.
- **Політика same-origin**: Правило безпеки браузера, яке блокує скрипти з одного origin від читання даних з іншого origin, запобігаючи крадіжці даних з інших відкритих вкладок.
- **CORS (Cross-Origin Resource Sharing)**: Механізм, що дозволяє серверам послабити політику same-origin для довірених origin, встановлюючи відповідні HTTP-заголовки.
- **В Express (розробка)**: `app.use(cors())` дозволяє всі origin.
- **В Express (продакшн)**: Передайте об’єкт `corsOptions` з allowlist для `origin`:

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**: HTTP з шифруванням, забезпеченим TLS (Transport Layer Security). TLS замінив старіший протокол SSL.
- **Три гарантії HTTPS**:
  - **Шифрування**: Дані між клієнтом і сервером недоступні для читання третіми сторонами.
  - **Автентифікація**: Цифровий сертифікат від довіреного центру сертифікації підтверджує, що сервер є тим, за кого себе видає.
  - **Цілісність даних**: Повідомлення не можуть бути змінені під час передачі без виявлення.
- HTTPS потрібен для сучасних браузерних API, таких як геолокація, push-повідомлення та доступ до камери.

### Чому виникають проблеми з безпекою

- **Тиск швидкої доставки**: Команди, які надто поспішно виконують дедлайни, пропускають перевірки безпеки і випускають уразливості.
- **Складна інтеграція систем**: Ризик ланцюга постачання виникає, коли сторонні бібліотеки або хмарні сервіси вводять уразливості, які команда розробки не створювала.
- **Погане управління конфігурацією**: Залишення налаштувань за замовчуванням (паролі адміністратора, відкриті порти) необґрунтовано відкриває системи.
- **Людська помилка**: Фішингові атаки експлуатують співробітників, які клікають на шкідливі посилання або вкладення і передають облікові дані.
- **Об’єднання організацій**: Поєднання різних кодових баз і інфраструктури виявляє невідповідність політик безпеки та застарілі уразливості.

### Cookies: зберігання, прапорці безпеки та конфіденційність

- **Як зберігаються cookies**: Сервер надсилає заголовок `Set-Cookie`. Браузер зберігає файл на диск і автоматично повертає його з кожним наступним запитом до цього походження.
- **Прапорець `HttpOnly`**: Блокує доступ JavaScript до значення cookie, запобігаючи крадіжці сесії через XSS.
- **Прапорець `Secure`**: Обмежує cookie лише HTTPS-з’єднаннями, захищаючи від перехоплення в публічних мережах.
- **`SameSite=Strict`**: Запобігає надсиланню cookie при будь-яких крос-сайтових запитах, повністю блокуючи CSRF-атаки.
- **`SameSite=Lax`**: Дозволяє cookie при навігації верхнього рівня (наприклад, натискання посилання), але блокує їх при крос-сайтових відправленнях форм і API-запитах.
- **Cookies першої сторони**: Прив’язані до сайту, який ви відвідуєте.
- **Cookies третьої сторони**: Встановлюються рекламними мережами на тисячах сайтів для відстеження поведінки і створення профілів користувачів. Сучасні браузери за замовчуванням блокують їх.
- **Відповідність GDPR**: Закони про конфіденційність вимагають банери згоди на cookie, що дають користувачам право відмовитися від неважливих трекінгових cookie.

### Загрози безпеці та заходи захисту

- **Фішинг**: Шахрайські електронні листи, що видають себе за довірені компанії для крадіжки облікових даних. Заходи захисту включають навчання безпеці та впровадження протоколів автентифікації електронної пошти для блокування шкідливих повідомлень до їх потрапляння в поштову скриньку.
- **Шкідливе ПЗ** (віруси, шпигунське ПЗ, програми-вимагачі): Програми-вимагачі шифрують файли і вимагають викуп за розблокування. Заходи захисту включають автоматичні оновлення та антивірусне програмне забезпечення.
- **Credential stuffing**: Зловмисники тестують мільйони викрадених комбінацій імен користувачів і паролів на різних сайтах. Основний захист — багатофакторна автентифікація (MFA), що вимагає другого кроку перевірки, наприклад OTP або коду з додатка-аутентифікатора.
- **DDoS (розподілена відмова в обслуговуванні)**: Зловмисники заливають сервер фальшивим трафіком, щоб його зупинити і заблокувати легітимних користувачів. Заходи захисту включають хмарні сервіси фільтрації трафіку, такі як Cloudflare, Akamai і AWS Shield.
- **Внутрішні загрози**: Зловмисні або необережні співробітники, які крадуть або розголошують дані. Заходи захисту включають застосування **принципу найменших привілеїв** (працівники мають доступ лише до необхідного для роботи) і моніторинг системних журналів на предмет підозрілої активності.

### Політика безпеки вмісту (Content Security Policy) та Permissions-Policy

- **CSP (Content Security Policy)**: HTTP-заголовок відповіді, який дозволяє лише схвалені домени для скриптів, стилів і зображень. Коли браузер отримує CSP-заголовок, він блокує будь-які ресурси з доменів, не включених до списку. Директива `script-src` визначає, з яких доменів можна виконувати JavaScript, що зупиняє XSS-атаки.
- **Permissions-Policy**: HTTP-заголовок, який вмикає або вимикає браузерні функції та доступ до апаратного забезпечення для сторінки та вбудованих iframe. Замінює старіший заголовок `Feature-Policy`.
  - `camera=(self)` обмежує доступ до вебкамери лише до походження сторінки.
  - `geolocation=()` повністю вимикає геолокацію для всіх користувачів.
- Поєднання обох заголовків забезпечує багаторівневий захист: CSP запобігає ін’єкції коду, а Permissions-Policy — несанкціонованому доступу до апаратного забезпечення.

### Основні поняття конфіденційності користувача

- **PII (Personally Identifiable Information)**: Будь-які дані, що можуть ідентифікувати особу безпосередньо або опосередковано.
  - **Прямі ідентифікатори**: Повне ім’я, домашня адреса, національний ідентифікаційний номер.
  - **Опосередковані ідентифікатори**: IP-адреса, серійний номер пристрою, точна історія місцезнаходжень.
- **Конфіденційність**: Захист PII від несанкціонованого доступу через шифрування та контроль доступу.
- **Мінімізація даних**: Збирати лише мінімально необхідний обсяг PII для виконання конкретного завдання.
- **Обмеження цілей**: Дані, зібрані для однієї мети, не можна використовувати для іншої (наприклад, продавати рекламним мережам) без нової згоди користувача.
- **Відстеження**: Безперервний моніторинг поведінки користувача в інтернеті за допомогою скриптів, відбитків пристроїв і журналів історії браузера.
- **Право бути забутим (видалення даних)**: Законне право вимагати постійного видалення всіх персональних записів з баз даних компанії після закриття облікового запису.

### Файл `.env` і `dotenv`

- **Файл `.env`**: Текстовий файл у корені проєкту, який зберігає конфігураційні секрети, такі як ключі API, паролі до бази даних і номери портів у вигляді пар ключ-значення. Завжди додавайте його до `.gitignore`, щоб не потрапляв у систему контролю версій.
- **Пакет `dotenv`**: Зчитує файл `.env` під час запуску і додає значення до `process.env`:

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **Дві ключові переваги**: Облікові дані не зберігаються у вихідному коді (безпека), і одна кодова база може поводитися по-різному в середовищах розробки, тестування та продакшн, замінюючи файл `.env` на кожному сервері (гнучкість).

### Регіональні закони про конфіденційність і відповідність

- **GDPR (Загальний регламент захисту даних)**: Захищає осіб в ЄС. Вимагає явної згоди перед відстеженням, без попередньо встановлених прапорців. Дає користувачам право переглядати і назавжди видаляти свої дані. Передбачає великі фінансові штрафи за порушення.
- **CCPA (Закон Каліфорнії про конфіденційність споживачів) / CPRA (Закон Каліфорнії про права конфіденційності)**: Захищає мешканців Каліфорнії. Орієнтований на право відмови, а не на згоду. Вимагає видимого посилання «Не продавати мою особисту інформацію» на головній сторінці.
- **COPPA (Закон США про захист конфіденційності дітей в інтернеті)**: Федеральний закон США, що захищає дітей до 13 років. Вимагає чіткої політики конфіденційності та підтвердженої згоди батьків перед збором персональних даних дитини.
- **DPA (Закон про захист даних)**: Основна рамка конфіденційності Великої Британії. Працює разом з UK GDPR і надає користувачам право доступу та видалення персональних даних.
- Інші країни мають власні закони: Нігерія має NDPA (Закон про захист даних Нігерії), Індія — DPDP Act (Закон про цифровий захист персональних даних).
- **Кроки відповідності для глобальних команд**:
  - Створюйте банери згоди, що враховують місцезнаходження користувача і показують відповідне повідомлення про cookie.
  - Впроваджуйте автоматизовану систему DSAR (запит на доступ до даних), щоб знаходити, експортувати або назавжди видаляти дані користувача за його законним запитом.
  - Практикуйте мінімізацію даних і використовуйте перевірку віку, щоб запобігти несанкціонованому відстеженню неповнолітніх.
  - Призначте офіцера з захисту даних (DPO) для контролю архітектури конфіденційності і як основну контактну особу для регуляторів.
