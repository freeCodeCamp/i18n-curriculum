## Arbeiten mit Node.js und ereignisgesteuerter Architektur

- **Node.js**: Eine Open-Source, plattformübergreifende JavaScript-Laufzeitumgebung, die es Ihnen ermöglicht, JavaScript-Code außerhalb eines Browsers auszuführen. Sie wird häufig zum Erstellen von Webservern, APIs und Backend-Systemen verwendet.
- **Einschränkungen auf der Client-Seite**: Client-seitiges JavaScript hat eingeschränkten Zugriff auf das Dateisystem, begrenzte Unterstützung für komplexe Anwendungslogik und potenzielle Sicherheitsrisiken (z. B. dürfen Datenbankzugangsdaten nicht im Client-Code verfügbar gemacht werden).
- **Globales Objekt im Browser vs. Node.js**: Im Browser ist das globale Objekt `window`, das DOM-Manipulation, Cookies und Browser-Ereignisse verfügbar macht. In Node ist das globale Objekt `global`, das eingebaute Module für Dateien, Netzwerke und Betriebssysteminteraktion bereitstellt.
- **DOM-Zugriff**: Der Browser bietet Zugriff auf die DOM-API, Node jedoch nicht. Umgekehrt kann Node auf fast alle Systemressourcen zugreifen, einschließlich des Dateisystems, was im Browser nicht möglich ist.
- **Versionskontrolle**: Sie können wählen, welche Node-Version auf Ihrem Server läuft. Sie haben keine Kontrolle darüber, welche Browserversion Ihre Nutzer verwenden.
- **Nicht blockierende, ereignisgesteuerte Architektur**: Node verwendet einen einzelnen Thread und eine Ereignisschleife, die viele gleichzeitige Anfragen und I/O-Operationen effizient verarbeitet, ohne zu blockieren, was es ideal für Echtzeitanwendungen macht.
- **Callbacks**: Funktionen, die definieren, was passiert, wenn eine asynchrone Operation abgeschlossen ist. Die starke Abhängigkeit von ihnen macht asynchronen Node-Code oft schwerer lesbar und zu debuggen.
- **Vorteile von Node im Backend**: Eine einzige Sprache (JavaScript) für Frontend und Backend, nicht blockierende Architektur für hohe Parallelität, großes npm-Ökosystem, kostenlos und Open Source.
- **Nachteile von Node im Backend**: Single-threaded, daher können CPU-intensive Aufgaben (Bildverarbeitung, Kryptografie, komplexe Mathematik) die Ereignisschleife blockieren, starke Abhängigkeit von asynchronem/Callback-basiertem Code kann die Lesbarkeit verringern, npm-Pakete variieren in Qualität und Wartung.
- **NVM (Node Version Manager)**: Ein Werkzeug zum Verwalten und Wechseln zwischen mehreren Node.js-Versionen auf demselben Rechner. Unter macOS/Linux installieren mit:

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
```

- **Installation der LTS-Version von Node**: `nvm install --lts`
- **Wechseln der Node-Version**: `nvm use 20`
- **Auflisten aller installierten Node-Versionen**: `nvm ls`
- **Festlegen einer Standard-Node-Version (macOS/Linux)**: `nvm alias default 22.20.0`
- **nvm-windows**: Ein separates Projekt zur Verwaltung von Node-Versionen unter Windows (nicht WSL). Laden Sie den Installer aus dem offiziellen Repository herunter und führen Sie `nvm install lts` und dann `nvm use <version>` aus.
- **Ausführen einer JavaScript-Datei mit Node**: `node app.js`

## Arbeiten mit Node Core Modules

### Das `fs`-Modul

- **Zweck**: Bietet Methoden zum Erstellen, Lesen, Schreiben, Anhängen und Löschen von Dateien und Verzeichnissen.
- **Drei Nutzungsarten**: Callback-basiert (asynchron), Promise-basiert (`fs/promises`) und synchron (`Sync`-Suffix). Synchrone Methoden blockieren die Ereignisschleife und eignen sich nur für Skripte.
- **`writeFile()`**: Schreibt Inhalt in eine Datei und erstellt sie, falls sie nicht existiert:

```js
const fs = require("fs/promises");

async function writeToFile() {
  try {
    await fs.writeFile("article.md", "## Node `fs` Module", "utf8");
    console.log("File written to!");
  } catch (err) {
    console.error("Error writing to file:", err);
  }
}
```

- **`appendFile()`**: Hängt Inhalt an eine bestehende Datei an, ohne sie zu überschreiben.
- **`readFile()`**: Liest den Inhalt einer Datei. Übergeben Sie `"utf8"` als Kodierung, um einen String zu erhalten, oder lassen Sie es weg, um einen `Buffer` zu erhalten.
- **`unlink()`**: Löscht eine Datei. Wirft `ENOENT`, wenn die Datei nicht existiert.

### Das `Buffer`-Modul

- **Zweck**: Verarbeitet Binärdaten (Dateien, Bilder, Netzwerkstreams) direkt im Speicher.
- **`Buffer.from()`**: Erstellt einen Puffer aus einem String:

```js
const { Buffer } = require("buffer");

const myStrBuffer = Buffer.from("freeCodeCamp");
console.log(myStrBuffer); // <Buffer 66 72 65 65 43 6f 64 65 43 61 6d 70>
console.log(myStrBuffer.toString()); // freeCodeCamp
```

- **`Buffer.alloc(n)`**: Erstellt einen mit Nullen gefüllten Puffer mit `n` Bytes:

```js
const someBuffer = Buffer.alloc(10);
someBuffer.write("Hello fCC");
console.log(someBuffer.toString()); // Hello fCC
```

- **Abschneiden**: Wenn mehr Daten geschrieben werden als der Puffer aufnehmen kann, wird der Überschuss stillschweigend abgeschnitten.
- **`Buffer.byteLength(str)`**: Gibt die Anzahl der Bytes zurück, die zum Speichern eines Strings benötigt werden.

### Das `crypto`-Modul

- **Zweck**: Bietet kryptografische Werkzeuge wie Hashing, HMAC, Verschlüsselung/Entschlüsselung, digitale Signaturen und sichere Zufallszahlengenerierung.
- **`createHash(algorithm)`**: Erstellt einen Einweg-Hash (irreversibel). Verwenden Sie `.update(data).digest("hex")`, um die zu hashenden Daten bereitzustellen und den Hash als hexadezimale Zeichenkette zurückzugeben:

```js
const crypto = require("crypto");

const hashedPassword = crypto
  .createHash("sha256")
  .update("myStrongPassword")
  .digest("hex");
```

- **`createHmac(algorithm, secret)`**: Wie `createHash`, benötigt aber einen geheimen Schlüssel, geeignet zur Überprüfung der Datenintegrität und Authentifizierung.
- **`createCipheriv()` / `createDecipheriv()`**: Verschlüsseln und entschlüsseln Daten. Beide benötigen einen Algorithmus, einen Schlüssel und einen Initialisierungsvektor (IV):

```js
const cipher = crypto.createCipheriv("aes-256-cbc", key, iv);
let encrypted = cipher.update("Hello campers!", "utf8", "hex");
encrypted += cipher.final("hex");
```

- **`randomBytes(size)`**: Erzeugt kryptografisch sichere Zufallsbytes, sicherer als `Math.random()` für Tokens und Schlüssel.
- **`randomInt(min, max)`**: Erzeugt eine sichere Zufallszahl im angegebenen Bereich. Nützlich für OTPs.
- **`sign()` / `verify()`**: Erstellen und überprüfen digitale Signaturen mit einem privaten/öffentlichen Schlüsselpaar.

### Das `os`-Modul

- **Zweck**: Stellt Informationen über das Betriebssystem bereit, auf dem Node läuft.
- **`os.platform()`**: Gibt die OS-Kennung zurück, z. B. `'darwin'`, `'win32'`, `'linux'`.
- **`os.arch()`**: Gibt die CPU-Architektur zurück, z. B. `'x64'`, `'arm64'`.
- **`os.type()`**: Gibt den offiziellen OS-Namen zurück, z. B. `'Darwin'`.
- **`os.release()`**: Gibt die Kernel-Version des OS zurück.
- **`os.cpus()`**: Gibt ein Array von Objekten zurück, die jeden logischen CPU-Kern beschreiben (Modell, Geschwindigkeit, Zeiten).
- **`os.uptime()`**: Gibt die Systemlaufzeit in Sekunden zurück.
- **`os.totalmem()` / `os.freemem()`**: Gibt den gesamten bzw. verfügbaren Systemspeicher in Bytes zurück.
- **`os.userInfo()`**: Gibt ein Objekt mit `uid`, `gid`, `username`, `homedir` und `shell` des aktuellen Benutzers zurück.
- **`os.networkInterfaces()`**: Gibt Netzwerkinterfaces mit Adresse, Familie, MAC und CIDR-Details zurück.

### Das `path`-Modul

- **Zweck**: Bietet Hilfsfunktionen für die plattformunabhängige Arbeit mit Datei- und Verzeichnispfaden.
- **`__filename`**: Eine globale Variable, der absolute Pfad der aktuellen Datei.
- **`__dirname`**: Eine globale Variable, der absolute Pfad des Verzeichnisses der aktuellen Datei.
- **`path.basename(p)`**: Gibt den letzten Teil eines Pfads zurück (den Dateinamen).
- **`path.dirname(p)`**: Gibt den Verzeichnisteil eines Pfads zurück.
- **`path.extname(p)`**: Gibt die Dateiendung zurück, z. B. `'.js'`.
- **`path.join(...segments)`**: Verbindet Pfadsegmente zu einem normalisierten Pfad und korrigiert fehlerhafte Schrägstriche.
- **`path.resolve(...segments)`**: Löst Segmente zu einem absoluten Pfad auf, ausgehend vom aktuellen Arbeitsverzeichnis.
- **`path.parse(p)`**: Gibt ein Objekt mit den Eigenschaften `root`, `dir`, `base`, `ext` und `name` zurück.
- **`path.format(obj)`**: Baut einen Pfadstring aus einem Objekt mit `dir`, `name` und `ext` zusammen.

### Das `process`-Modul

- **Zweck**: Stellt Informationen und Steuerung über den aktuellen Node.js-Prozess bereit. Es ist ein globales Objekt, daher ist kein Import erforderlich.
- **`process.env`**: Objekt mit allen Umgebungsvariablen. Greifen Sie auf spezifische Werte wie `process.env.NODE_ENV` zu.
- **`process.argv`**: Array der Befehlszeilenargumente. Index 0 ist die Node-Executable, Index 1 der Skriptpfad, Index 2+ sind Benutzerargumente.
- **`process.cwd()`**: Gibt das aktuelle Arbeitsverzeichnis zurück.
- **Prozessereignisse**:

```js
process.on("exit", (code) => {
  console.log(`Process exiting with code: ${code}`);
});

process.on("uncaughtException", (err) => {
  console.error("Uncaught error:", err.message);
});
```

- **`process.emitWarning(message, type)`**: Löst eine benutzerdefinierte Warnung aus, die von einem `warning`-Listener abgefangen werden kann.

### Das `stream`-Modul

- **Zweck**: Verarbeitet große Datenmengen in Teilen, anstatt alles auf einmal in den Speicher zu laden.
- **Vier Stream-Typen**: Readable (liest Daten in Teilen), Writable (schreibt Daten in Teilen), Duplex (beides), Transform (ein Duplex, das Daten beim Durchfluss verändert).
- **Streams verketten (piping)**:

```js
const readInputFileStream = fs.createReadStream(inputFilePath);
const writeOutputFileStream = fs.createWriteStream(outputFilePath);

readInputFileStream.pipe(writeOutputFileStream);
writeOutputFileStream.on("finish", () => {
  console.log("All data has been written to the file");
});
```

- **`finish`-Ereignis**: Wird bei einem Writable-Stream ausgelöst, wenn alle Daten geschrieben wurden.
- **`error`-Ereignis**: Wird ausgelöst, wenn ein Stream auf ein Problem stößt.

## Einführung in npm

- **npm**: Besteht aus drei Teilen: der Website `npmjs.com`, dem öffentlichen Registry (einer Datenbank von Paketen) und dem CLI-Tool, das mit Node.js installiert wird.
- **Zweck**: Ermöglicht das Installieren, Verwalten und Veröffentlichen wiederverwendbarer JavaScript-Pakete. npm ist der am weitesten verbreitete Paketmanager.
- **Alternativen**: Yarn, PNPM, Bun.

### Die `package.json`-Datei

- **Zweck**: Das npm-Konfigurationsmanifest. Es verfolgt Projektname, Version, Beschreibung, Einstiegspunkt (`main`), Autor, Lizenz, Skripte und Abhängigkeiten.
- **Interaktive Erstellung von `package.json`**: `npm init`
- **Erstellung mit Standardwerten**: `npm init -y`
- **`dependencies`**: Pakete, die für den Betrieb der Anwendung in Produktion benötigt werden (z. B. `express`, `react`).
- **`devDependencies`**: Pakete, die nur während Entwicklung und Tests benötigt werden (z. B. `jest`, `nodemon`, `eslint`).
- **`license`**: Rechtliche Nutzungsbedingungen. Übliche Werte: `MIT`, `ISC`, `GPL`.

### Semantische Versionierung (SemVer)

- **Format**: `MAJOR.MINOR.PATCH`, z. B. `4.17.21`.
- **MAJOR**: Erhöht bei inkompatiblen Änderungen.
- **MINOR**: Erhöht bei neuen, abwärtskompatiblen Funktionen.
- **PATCH**: Erhöht bei abwärtskompatiblen Bugfixes.
- **Caret `^`**: Erlaubt Updates auf Minor- und Patch-Versionen, aber nicht Major. Beispiel: `^4.17.21` akzeptiert `4.18.0`, aber nicht `5.0.0`.
- **Tilde `~`**: Erlaubt nur Patch-Updates. Beispiel: `~1.2.3` akzeptiert `1.2.4`, aber nicht `1.3.0`.
- **Sternchen `*`**: Passt auf jede Version, nur für Tests geeignet, niemals für Produktion.
- **Exakte Version**: Kein Präfix, z. B. `1.2.3` fixiert genau diese Version.

### Installation und Entfernung von Abhängigkeiten

- **Paket installieren**: `npm install express` (Kurzform: `npm i express`)
- **Als devDependency installieren**: `npm install nodemon -D`
- **Paket deinstallieren**: `npm uninstall chalk`
- **`node_modules/`**: Ordner, der nach der Installation erstellt wird und den tatsächlichen Paketcode enthält. Wird nicht in die Versionskontrolle aufgenommen.
- **`package-lock.json`**: Automatisch generierte Datei, die die exakten Versionen aller installierten Pakete (inklusive Abhängigkeiten) sperrt. Diese Datei immer committen und nicht manuell bearbeiten.
- **Bestimmte Version installieren**: `npm install express@4.21.2`
- **Verfügbare Versionen anzeigen**: `npm view express versions`
- **Veraltete Pakete prüfen**: `npm outdated`
- **Einzelnes Paket aktualisieren** (innerhalb erlaubter Version): `npm update express`
- **Neueste Version erzwingen**: `npm install express@latest`

### Veröffentlichung im npm-Registry

- **Anmelden**: `npm login`
- **Scoped-Pakete**: Verwenden Sie `@username/package-name`, um Namenskollisionen zu vermeiden.
- **`.npmignore`**: Listet Dateien auf, die vom veröffentlichten Paket ausgeschlossen werden (ähnlich `.gitignore`).
- **Unscoped-Paket veröffentlichen**: `npm publish`
- **Scoped-Paket öffentlich veröffentlichen**: `npm publish --access public`

## Arbeiten mit npm-Skripten

- **npm-Skripte**: Benutzerdefinierte Befehle, definiert im `"scripts"`-Objekt der `package.json`. Sie automatisieren wiederkehrende Aufgaben wie Serverstart, Tests, Linting und Build.
- **Skript definieren**:

```json
"scripts": {
  "start": "node app.js"
}
```

- **Skript ausführen**: `npm run start`
- **Kurzform für eingebaute Skriptnamen** (`start`, `test`, `stop`, `restart`): `npm start`
- **Sequenzielle Befehle mit `&&`** (nächster Befehl läuft nur, wenn der vorherige erfolgreich war):

```json
"start": "npm run build && node server.js"
```

- **Sequenzielle Befehle mit `;`** (nächster Befehl läuft immer):

```json
"start": "npm run build; node server.js"
```

- **Parallele Befehle mit `&`**: `"dev": "npm run start:server & npm run start:client"`
- **Argumente an Skript übergeben** (mit `--`-Trenner): `npm start -- 8000`; Argumente sind über `process.argv[2]` zugänglich.

### CommonJS-Module

- **CommonJS**: Das ursprüngliche Node.js-Modulsystem. Lädt Module synchron.
- **Importieren** mit `require()`:

```js
const { multiply } = require('./math');
```

- **Exportieren** mit `module.exports`:

```js
function multiply(a, b) { return a * b; }
module.exports = { multiply };
```

- CommonJS ist Standard, wenn `package.json` den Eintrag `"type": "commonjs"` hat (oder kein `"type"`-Feld) oder die Datei die Endung `.cjs` besitzt.

### ES-Module (ESM)

- **ES-Module**: Das moderne, standardisierte Modulformat. Wird asynchron geladen.
- **Benannter Export**: `export function multiply(a, b) { return a * b; }`
- **Standardexport**: `export default function multiply(a, b) { return a * b; }`
- **Import**: `import { multiply } from './math.mjs';`
- ESM wird verwendet, wenn die Datei die Endung `.mjs` hat oder `package.json` den Eintrag `"type": "module"` enthält.

## Verständnis, wie HTTP, DNS und TCP/IP funktionieren

### Server und das Client-Server-Modell

- **Client-Server-Modell**: Eine verteilte Systemarchitektur, bei der Komponenten entweder Clients (Prozesse, die Anfragen senden, z. B. ein Webbrowser) oder Server sind.
- **Server**: Ein Computer oder Softwareprogramm, das Anfragen von Clients empfängt und Daten oder Dienste über ein Netzwerk zurückgibt.
- **Servertypen**: Webserver (HTML/CSS/Assets), Anwendungsserver (Geschäftslogik), Datenbankserver, Mailserver, Dateiserver, Proxyserver.

### DNS

- **DNS (Domain Name System)**, oft als "Telefonbuch des Internets" bezeichnet: Übersetzt menschenlesbare Domainnamen (z. B. `freecodecamp.org`) in IP-Adressen.
- **Domain-Hierarchie** (von rechts nach links): Top-Level-Domain (TLD, z. B. `.org`) → Second-Level-Domain (z. B. `freecodecamp`) → Subdomain (z. B. `www`).
- **IP-Adresse**: Eine eindeutige numerische Kennung für ein Gerät in einem Netzwerk. IPv4-Beispiel: `127.0.0.1`. IPv6-Beispiel: `0:0:0:0:0:0:0:1`.
- **DNS-Auflösungsprozess**: Browser → Rekursiver Resolver → Root-Server → TLD-Server → Autoritativer Nameserver → gibt IP-Adresse zurück → Rekursiver Resolver cached und liefert an Client.

### TCP/IP und HTTP

- **IP (Internet Protocol)**: Leitet Datenpakete vom Absender zum Empfänger anhand von IP-Adressen. Verbindungslos, jedes Paket wird unabhängig gesendet.
- **TCP (Transmission Control Protocol)**: Zerlegt Daten in nummerierte Segmente, setzt sie am Ziel wieder zusammen, prüft auf Fehler und fordert bei Bedarf eine erneute Übertragung an. Sorgt für zuverlässige, geordnete Zustellung.
- **HTTP (HyperText Transfer Protocol)**: Anwendungsprotokoll, das definiert, wie Clients und Server Webinhalte austauschen.
- **HTTPS**: Eine sichere Version von HTTP, bei der Daten mit SSL/TLS verschlüsselt werden.

### Aufbau einer HTTP-Anfrage

```http
GET / HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
```

- **Anforderungszeile**: HTTP-Methode + Pfad/URI + HTTP-Version.
- **Header**: Schlüssel-Wert-Metadatenpaare.
- **Body** (optional): Wird bei `POST`, `PUT`, `PATCH`, `DELETE` zum Übertragen von Daten gesendet.

### Aufbau einer HTTP-Antwort

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 105

<!DOCTYPE html>...
```

- **Statuszeile**: HTTP-Version + Statuscode + Begründungstext.
- **Header**: Metadaten zur Antwort (Inhaltstyp, Länge, Cache-Anweisungen).
- **Body** (optional): Die angeforderten Daten.

### Kategorien von HTTP-Statuscodes

- **1xx Informativ**: Anfrage empfangen, Verarbeitung läuft weiter. Beispiel: `101 Switching Protocols`.
- **2xx Erfolg**: `200 OK`, `201 Created`, `204 No Content`.
- **3xx Umleitung**: `301 Moved Permanently`, `302 Found`, `304 Not Modified`.
- **4xx Client-Fehler**: `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`.
- **5xx Server-Fehler**: `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable`.

## Verständnis des HTTP-Anfrage-Antwort-Modells

- **HTTP-Methoden**: `GET` (abrufen), `POST` (erstellen), `PUT` (vollständiges Update), `PATCH` (teilweises Update), `DELETE` (entfernen).
- **Anfrage-Antwort-Zyklus**: Client sendet Anfrage → Server prüft Authentifizierung, Methode und URL, fragt Datenbank ab → Server erstellt Antwort (Statuscode + optionaler Body) → Client verarbeitet Antwort.

### In HTTP-Antworten zurückgegebene Assets

Jede Antwort enthält einen `Content-Type`-Header, der dem Browser sagt, wie das Asset behandelt werden soll:

- **HTML**: `text/html`
- **CSS**: `text/css`
- **JavaScript**: `application/javascript`
- **JSON**: `application/json`
- **PNG/JPEG/WebP**: `image/png`, `image/jpeg`, `image/webp`
- **MP4/MP3**: `video/mp4`, `audio/mpeg`
- **PDF**: `application/pdf`
- **WOFF2**: `font/woff2`

### HTML-Formularübermittlungen

- **`action`-Attribut**: Gibt die URL an, an die Formulardaten gesendet werden.
- **`method`-Attribut**: Gibt die HTTP-Methode an (`GET` oder `POST`). HTML unterstützt nativ kein `PUT`, `PATCH` oder `DELETE`; verwenden Sie stattdessen die Fetch-API oder das `method-override`-Paket in Express.
- **Client-seitige Validierung**: Verwenden Sie `required` und `pattern` für sofortiges Feedback. Validieren Sie immer auch serverseitig.
- **`GET`-Formular**: Daten werden als Abfrageparameter an die URL angehängt (z. B. Suchformulare).
- **`POST`-Formular**: Daten werden im Anfrage-Body gesendet (z. B. Login- und Registrierungsformulare).

## Verständnis des Web-Standards-Modells

### Standardisierungsgremien

- **WHATWG**: Pflegt HTML und das DOM als kontinuierlich aktualisierten "lebenden Standard". Entscheidungen werden von den großen Browser-Engine-Teams (Chromium, Gecko, WebKit) getroffen.
- **W3C (World Wide Web Consortium)**: Haupt-Standardisierungsgremium. Verantwortlich für CSS, SVG, Barrierefreiheitsrichtlinien (WCAG) und zahlreiche Web-APIs.
- **ECMA International / TC39**: Ecma veröffentlicht die offizielle ECMAScript-Spezifikation. TC39 entwickelt die Sprache durch einen fünfstufigen Vorschlagsprozess (Stufe 0 grobe Idee → Stufe 4 vollständig spezifiziert).
- **Khronos Group**: Fokussiert auf Grafik, Berechnung und Medien. Produziert WebGL und WebGL 2 für GPU-beschleunigtes 3D-Rendering in Browsern.

### Erstellung von Webstandards

1. **Bedarf erkennen**: Ein Entwickler oder eine Organisation stößt auf ein Problem und dokumentiert es.
2. **Vorschlag und frühe Diskussion**: Informelles GitHub-Issue oder Erklärungsdokument zur Interessenabfrage. Bei TC39 entspricht dies Stufe 0/1.
3. **Spezifikation schreiben**: Ein präzises formales Dokument, das alle Randfälle, Fehlerbedingungen und Sicherheitsaspekte abdeckt.
4. **Überprüfung, Einwände und Iteration**: Entwurf wird öffentlich von Browseranbietern geprüft; Funktionen, die ein großer Browser nicht implementieren will, werden selten Standard.
5. **Implementierung und Test**: Anbieter implementieren hinter Feature-Flags, eine gemeinsame Testsuite stellt browserübergreifende Konsistenz sicher.
6. **Abschluss**: W3C nennt es Empfehlung, TC39 Stufe 4.

### Lebenszyklus eines Webstandards-Features

1. **Vorschlag**: Problem erkannt und dokumentiert.
2. **Inkubation**: Offene Diskussion, viele Vorschläge sterben hier.
3. **Spezifikation**: Formale Spezifikation mit genauen Regeln.
4. **Interoperabilitätstests**: W3C Web Platform Tests Suite überprüft browserübergreifende Konsistenz.
5. **Abschluss**: Feature wird ohne Flags als Teil der Basis-Webplattform ausgeliefert.
6. **Veraltung**: Veraltete oder unsichere Features werden zur Entfernung markiert, bleiben aber aus Kompatibilitätsgründen erhalten.

### Wichtige Grundsätze von Webstandards

- **Offenheit**: Standards werden offen entwickelt und lizenzfrei implementiert.
- **Barrierefreiheit**: In HTML, CSS und Browserverhalten integriert, WCAG bietet formale Richtlinien.
- **Interoperabilität**: Eine standardkonforme Seite sollte in allen Browsern identisch funktionieren.
- **Abwärtskompatibilität**: Alte Seiten müssen in modernen Browsern weiterhin funktionieren.
- **Datenschutz und Sicherheit**: Neue APIs, die Nutzerdaten offenlegen könnten, werden streng geprüft.
- **Schichtung**: HTML für Struktur, CSS für Darstellung, JavaScript für Verhalten. Jede Schicht kann unabhängig übernommen werden.
- **Progressive Verbesserung**: Features müssen zuerst für alle funktionieren und dann für Nutzer mit besseren Browsern verbessert werden.

## Verständnis von REST-APIs und Webservices

### Webservices

- **API (Application Programming Interface)**: Ein allgemeiner Mechanismus, damit Software mit anderer Software kommunizieren kann.
- **Webservice**: Ein spezieller API-Typ, der Webprotokolle (HTTP/HTTPS) nutzt, um Daten über ein Netzwerk auszutauschen. Jeder Webservice ist eine API, aber nicht jede API ist ein Webservice.
- **Merkmale**: Läuft auf einem Server, ermöglicht Maschine-zu-Maschine-Kommunikation, sprach- und plattformunabhängig, verwendet Standard-Webprotokolle.
- **REST**: Representational State Transfer, der am weitesten verbreitete Webservice-Stil. Einfach, flexibel, arbeitet natürlich mit HTTP.
- **SOAP**: Simple Object Access Protocol, verwendet nur XML, häufig in Unternehmenssystemen mit strengen Standards.
- **XML-RPC**: Verwendet XML zur Kodierung leichter Anfragen und Antworten.

### REST-Architektur

- **Zustandslosigkeit**: Jede Anfrage wird unabhängig behandelt. Der Server speichert keinen Sitzungszustand zwischen Anfragen. Jede Anfrage muss alle notwendigen Informationen enthalten (z. B. ein Auth-Token bei jedem Aufruf).
- **Ressourcen**: Alles ist eine Ressource, identifiziert durch eine URL, z. B. `/products` für eine Sammlung, `/products/1` für ein spezifisches Element.
- **HTTP-Methoden in REST**: `GET` (abrufen), `POST` (erstellen), `PUT`/`PATCH` (aktualisieren), `DELETE` (entfernen).
- **Antwortformate**: JSON ist am gebräuchlichsten, aber auch XML und Klartext werden unterstützt. Clients geben das Format über den `Accept`-Header an:

```bash
Accept: application/json
```

### Microservices

- **Monolith**: Eine große Codebasis, in der UI, Datenbank und API zusammenliegen. Schwer wartbar und skalierbar bei wachsender Komplexität.
- **Microservices**: Ein Architekturansatz, bei dem eine große Anwendung in kleine, unabhängige Dienste aufgeteilt wird, die jeweils eine einzelne Geschäftsfähigkeit verantworten. Dienste kommunizieren über APIs oder Nachrichtenwarteschlangen.
- **Vorteile**: Schnellere unabhängige Entwicklung und Deployment, bessere Fehlerisolierung, verschiedene Dienste können unterschiedliche Sprachen oder Frameworks verwenden.
- **Nachteile**: Komplexeres Debugging (Probleme erstrecken sich über Dienste), höherer Infrastrukturaufwand, mehr Netzwerkkommunikation.

## Arbeiten mit Express

- **Express.js**: Ein minimalistisches und flexibles Webframework, das auf Node.js aufbaut. Es vereinfacht das Erstellen von Webservern und APIs mit deutlich weniger Boilerplate-Code als das native `http`-Modul.
- **Ein einfacher Express-Server**:

```js
const express = require("express");
const app = express();
const port = 3000;

app.get("/", (req, res) => {
  res.send("Hello World!");
});

app.listen(port, () => {
  console.log(`Example app listening on port ${port}`);
});
```

- **`express()`**: Fabrikfunktion, die eine Express-Anwendungsinstanz (`app`) erstellt.
- **`app.listen(port, callback)`**: Startet den Server und beginnt, HTTP-Anfragen am angegebenen Port zu empfangen.
- **`req`**: Das Anfrageobjekt, enthält URL, Header, Parameter, Body und Query-Daten.
- **`res`**: Das Antwortobjekt, wird verwendet, um Daten an den Client zu senden.
- **Middleware**: Code, der zwischen dem Empfang einer Anfrage und dem Senden einer Antwort ausgeführt wird.

## Verständnis von Routing in Express.js

- **Routing**: Definiert, wie eine Anwendung reagiert, wenn ein Client eine bestimmte URL mit einer bestimmten HTTP-Methode besucht.
- **`app.get/post/put/delete(path, handler)`**: Registriert Handler für die jeweiligen HTTP-Methoden.
- **Statische Route**: Ein fester URL-Pfad, z. B. `app.get("/home", handler)`.
- **Dynamische Route mit Parametern**: Verwendet `:paramName` als Platzhalter. Erfasst über `req.params`:

```js
app.get("/post/:postId", (req, res) => {
  const postId = req.params.postId;
  res.send(`Viewing post with ID: ${postId}`);
});
```

- **`req.params`**: Routenparameter (`:id`-Segmente).
- **`req.query`**: Query-String-Parameter (`?key=value`).
- **`req.body`**: Anfrage-Body (benötigt Body-Parsing-Middleware).
- **`res.send(data)`**: Sendet eine Antwort (Text, HTML, JSON oder Binärdaten).
- **`res.json(obj)`**: Sendet eine JSON-Antwort, setzt automatisch `Content-Type: application/json`.
- **`res.status(code).send(msg)`**: Setzt den Statuscode und sendet eine Antwort.
- **`res.redirect(url)`**: Leitet den Client zu einer anderen URL um.
- **`res.render(view, data)`**: Rendert eine Template-Engine-Ansicht und sendet HTML.

### Verkettbare Routen-Handler mit `app.route()`

Gruppiert mehrere HTTP-Methoden für denselben Pfad, um Wiederholungen zu reduzieren:

```js
app
  .route("/user")
  .get((req, res) => { res.send("Fetching user data"); })
  .post((req, res) => { res.send("Creating a user"); })
  .put((req, res) => { res.send("Updating the user"); })
  .delete((req, res) => { res.send("Deleting the user"); });
```

### `express.Router()`

Erstellt eine modulare Mini-Anwendung zum Gruppieren verwandter Routen:

```js
const express = require("express");
const router = express.Router();

router.get("/", (req, res) => { res.send("List of users"); });
router.get("/:id", (req, res) => {
  res.send(`User details for user with id: ${req.params.id}`);
});

module.exports = router;
```

```js
const userRoutes = require("./userRoutes");
app.use("/users", userRoutes);
```

### Bereitstellung statischer Dateien

- **`express.static(directory)`**: Eingebaute Middleware, die Dateien aus einem angegebenen Verzeichnis bereitstellt:

```js
app.use(express.static(path.join(__dirname, "public")));
```

- Bereitstellung aus mehreren Verzeichnissen mit einem Mount-Pfad: `app.use("/images", express.static(path.join(__dirname, "images")));`

## Verständnis von Fehlerbehandlung und Gesundheitsprüfungen

### Fehlerbehandlung in Express

- **Fehlerbehandlungs-Middleware**: Erkannt an vier Parametern `(err, req, res, next)`. Wird immer am Ende des Middleware-Stacks platziert:

```js
app.use((err, req, res, next) => {
  console.error(err.message);
  res.status(500).send("Internal Server Error");
});
```

- **Express 4 asynchrone Fehler**: Müssen manuell mit `try/catch` abgefangen und mit `next(err)` weitergeleitet werden:

```js
app.get("/user", async (req, res, next) => {
  try {
    const user = await getUserFromDatabase();
    res.send(user);
  } catch (err) {
    next(err);
  }
});
```

- **Express 5 asynchrone Fehler**: Werden automatisch an die Fehlerbehandlungs-Middleware weitergeleitet, kein manuelles `next(err)` nötig.
- **404-Handler**: Nach allen Routen, aber vor dem Fehlerhandler platzieren:

```js
app.use((req, res, next) => {
  res.status(404).send("Sorry, that route does not exist.");
});
```

### Debugging und Logging

- **Eingebautes Debugging**: Aktivieren durch Setzen der Umgebungsvariable `DEBUG`: `DEBUG=express:* node index.js`
- **`morgan`**: Eine beliebte Drittanbieter-Middleware zum Protokollieren von HTTP-Anfragen. Protokolliert Methode, URL, Statuscode und Antwortzeit:

```js
const morgan = require('morgan');
app.use(morgan('dev'));
```

### Gesundheitsprüfungen und sanfte Abschaltungen

- **Health-Check-Route**: Gibt `200 OK` zurück, um zu signalisieren, dass die Anwendung läuft. Wird von Load Balancern und Container-Orchestratoren verwendet:

```js
app.get('/health', (req, res) => {
  res.status(200).send('OK');
});
```

- **`SIGTERM`**: Signal, das gesendet wird, wenn ein Prozess beendet werden soll. Behandeln Sie es so, dass laufende Anfragen abgeschlossen werden können:

```js
process.on('SIGTERM', () => {
  server.close(() => {
    console.log('Server closed. Cleaning up...');
  });
});
```

- **`SIGINT`**: Signal, das beim Drücken von Strg+C während der lokalen Entwicklung gesendet wird. Behandeln Sie es wie `SIGTERM`.
- **Vor dem Herunterfahren als nicht gesund markieren**: Geben Sie während des Herunterfahrens von der Health-Check-Route einen `503` zurück, um zu verhindern, dass der Load Balancer neuen Traffic weiterleitet:

```js
let isShuttingDown = false;

app.get('/health', (req, res) => {
  if (isShuttingDown) {
    res.status(503).send('Shutting down');
  } else {
    res.status(200).send('OK');
  }
});

process.on('SIGTERM', () => {
  isShuttingDown = true;
  server.close(() => { console.log('Graceful shutdown complete'); });
});
```

## Express-Middleware

- **Middleware**: Eine Funktion, die während des Lebenszyklus einer Anfrage ausgeführt wird. Sie hat Zugriff auf `req`, `res` und `next`. Rufen Sie `next()` auf, um die Kontrolle an die nächste Middleware weiterzugeben, oder lassen Sie es weg, um den Zyklus zu beenden.
- **Anwendungsweite Middleware**: Wird über `app.use()` an die gesamte Anwendung gebunden. Läuft bei jeder eingehenden Anfrage.
- **Router-spezifische Middleware**: Wird an eine bestimmte `express.Router()`-Instanz über `router.use()` gebunden. Gilt nur für Routen, die von diesem Router behandelt werden.
- **Fehlerbehandlungs-Middleware**: Erkennbar an vier Parametern `(err, req, res, next)`. Wird am Ende des Middleware-Stacks platziert.
- **`express.json()`**: Parst eingehende Anfragen mit JSON-Nutzlasten. Macht die Daten in `req.body` verfügbar.
- **`express.urlencoded({ extended: true })`**: Parst URL-kodierte Formulardaten.
- **`express.static('public')`**: Dient statische Dateien aus einem Verzeichnis aus.
- **`cors`**: Ermöglicht Cross-Origin Resource Sharing. Installation mit `npm install cors`, Verwendung mit `app.use(cors())`.
- **`morgan`**: Protokolliert HTTP-Anfragen. Verwendung mit `app.use(morgan('tiny'))`.
- Middleware wird in der Reihenfolge angewendet, in der sie mit `app.use()` registriert wurde.

## WebSockets verstehen

- **WebSockets**: Ein Protokoll, das eine persistente, bidirektionale (Full-Duplex) Verbindung zwischen Client und Server herstellt und so Echtzeitkommunikation ermöglicht.
- **Problem mit HTTP für Echtzeit**: HTTP erfordert, dass der Client den Server wiederholt nach neuen Daten abfragt, was langsam und ineffizient für Chat-Apps, Live-Scoreboards und Mehrspieler-Spiele ist.
- **WebSocket-Handschlag**: Jede WebSocket-Verbindung beginnt mit einer HTTP-Anfrage. Der Client fordert ein Upgrade an, und der Server antwortet mit `101 Switching Protocols`. Danach bleibt die Verbindung offen, bis eine Seite ein Schließsignal sendet.
- **Anwendungsfälle**: Chat-Anwendungen, Online-Mehrspieler-Spiele, Live-Sport-Dashboards, Börsendaten-Feeds, kollaborative Werkzeuge.

### Die `ws`-Bibliothek

- Node.js hat keinen eingebauten WebSocket-Server; die `ws`-Bibliothek ist der übliche Weg, einen zu implementieren.

```js
const WebSocket = require("ws");
const server = new WebSocket.Server({ port: 3000 });

server.on("connection", (socket) => {
  console.log("Client connected");

  socket.on("message", (message) => {
    console.log(message.toString());
  });

  socket.on("close", () => { console.log("Disconnected"); });
  socket.on("error", (error) => { console.error(error); });
});
```

- **`socket.send(msg)`**: Sendet eine Nachricht an einen bestimmten Client.
- **Broadcast an alle verbundenen Clients**:

```js
server.on("connection", (socket) => {
  socket.on("message", (message) => {
    server.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(message.toString());
      }
    });
  });
});
```

- **`readyState`**: Eigenschaft jeder Verbindung. `WebSocket.OPEN` bedeutet, dass die Verbindung aktiv ist.

### Pub/Sub-Architektur

- **Pub/Sub (Publish/Subscribe)**: Ein Nachrichtenmuster, bei dem **Publisher** Nachrichten an **Themen** senden und **Subscriber** Nachrichten von den Themen erhalten, denen sie folgen. Publisher und Subscriber kommunizieren nicht direkt.
- **Vorteile gegenüber direkter Kommunikation**: Entkoppelt Komponenten, ein Publisher muss nicht wissen, wer seine Subscriber sind.
- **Pub/Sub vs. WebSockets**: WebSockets sind der Kommunikationskanal (die Straße). Pub/Sub ist das Nachrichtenmuster, das bestimmt, wer welche Nachrichten erhält (das Verkehrssystem).

## Einführung in Authentifizierung und Autorisierung

### Authentifizierung vs. Autorisierung

- **Authentifizierung**: Überprüft die Identität, beantwortet die Frage „Wer sind Sie?“, muss vor der Autorisierung erfolgen.
- **Autorisierung**: Bestimmt Berechtigungen, beantwortet die Frage „Was dürfen Sie tun?“. 
- **Drei Authentifizierungsfaktoren**:
  - **Etwas, das Sie wissen**: Passwort, PIN, Antwort auf Sicherheitsfrage.
  - **Etwas, das Sie haben**: Smartphone, das einen SMS-Code empfängt, Token einer Authenticator-App.
  - **Etwas, das Sie sind**: Fingerabdruck, Gesichtserkennung (Biometrie).
- **Übliche Rollen bei der Autorisierung**: Standardbenutzer (eingeschränkter Selbstservice), Redakteur (Inhaltsverwaltung), Admin (voller Systemzugriff).
- Im Backend wird Autorisierung meist über Middleware durchgesetzt, die Benutzerrollen oder Berechtigungen prüft, bevor Zugriff gewährt wird.

### Cross-Site Request Forgery (CSRF)

- **CSRF**: Ein Angriff, bei dem eine bösartige Website den Browser eines authentifizierten Benutzers dazu bringt, eine unerwünschte Anfrage an einen Backend-Server zu senden, indem sie die automatische Cookie-Übermittlung des Browsers ausnutzt.
- **Angriffsablauf**: Benutzer meldet sich bei `trusted-bank.com` an → besucht `evil-hacker-site.com` → bösartige Seite sendet ein verstecktes Formular an die Bank → Bank-Server sieht gültiges Sitzungscookie und führt die Anfrage aus.
- **`SameSite=Strict`**: Browser sendet das Cookie niemals, wenn die Anfrage von einer Drittanbieter-Website stammt.
- **`SameSite=Lax`**: Browser erlaubt Cookies nur, wenn der Benutzer sicher zur Zielseite navigiert (z. B. durch Klicken eines Links), blockiert sie aber bei Cross-Site-Formularübermittlungen.
- **CSRF-Tokens**: Der Server erzeugt einen zufälligen, unvorhersehbaren Token, der in Formularen eingebettet wird. Der Client muss den Token bei jeder zustandsändernden Anfrage mitsenden. Eine bösartige Seite kann den Token im Frontend nicht auslesen, daher schlagen gefälschte Anfragen die Validierung fehl.

### JSON Web Tokens (JWT)

- **JWT**: Ein digital signiertes Token, das der Server nach erfolgreichem Login ausgibt. Der Client speichert es und sendet es mit jeder folgenden Anfrage mit, was zustandslose Authentifizierung ermöglicht (keine serverseitige Sitzungsspeicherung nötig).
- **JWT-Struktur**: Drei Base64URL-kodierte Teile, getrennt durch Punkte:
  - **Header**: Token-Typ (`JWT`) und Signaturalgorithmus (z. B. `HS256`).
  - **Payload**: Ansprüche über den Benutzer (Benutzer-ID, Benutzername, Rolle). Nicht verschlüsselt, daher niemals Passwörter oder sensible Daten hier speichern.
  - **Signatur**: Hash von Header + Payload + geheimer Serverschlüssel. Verhindert Manipulation.

```bash
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3OCIsIm5hbWUiOiJKYW5lIERvZSIsImFkbWluIjp0cnVlLCJpYXQiOjE1MTYyMzkwMjJ9.Lfu39gE9OI8rhPXbmJMorXTtVappkNMw9xq-M4HXttA
```

- Wenn ein Benutzer die Nutzlast manipuliert, stimmt die Signatur nicht mehr, und der Server lehnt die Anfrage mit `401 Unauthorized` ab.

### Passport.js

- **Passport.js**: Authentifizierungs-Middleware für Node.js. Standardisiert den Authentifizierungsablauf und übernimmt niedrige Details, damit Entwickler sich auf die Geschäftslogik konzentrieren können.
- **Strategien**: Modulare Plugins, die spezifische Authentifizierungsmethoden implementieren. Über 500 Strategien verfügbar: `passport-local` (Benutzername/Passwort), `passport-jwt` (JWTs), `passport-google-oauth20` (Google-Login), `passport-facebook`, `passport-github2`.
- **Integration in Express**:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database lookup and verification logic
}));
```

- **Verwendung als Routen-Middleware**:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

- Nach erfolgreicher Authentifizierung speichert Passport ein Stück Benutzerdaten im Sitzungscookie und hängt das authentifizierte Benutzerobjekt an `req.user` an.

### Helmet.js

- **Helmet.js**: Eine Sicherheits-Middleware-Sammlung, die Express-Anwendungen schützt, indem sie HTTP-Antwortheader automatisch setzt, ändert oder entfernt.

```js
const helmet = require('helmet');
app.use(helmet());
```

- **Entfernt `X-Powered-By`**: Verhindert, dass Angreifer die Servertechnologie erkennen.
- **Setzt `Content-Security-Policy`**: Beschränkt, von welchen Quellen Skripte, Bilder und Styles geladen werden dürfen, und mindert XSS-Angriffe.
- **Setzt `X-Frame-Options: SAMEORIGIN`**: Verhindert Clickjacking, indem es blockiert, dass Ihre Seite in externen iframes eingebettet wird.
- **Setzt `Strict-Transport-Security`**: Erzwingt, dass Browser HTTPS verwenden, und schützt Daten vor Abhören.

## Sicherheit und Datenschutz in Webanwendungen verstehen

### Sicherheit vs. Datenschutz

- **Sicherheit**: Die technischen Schutzmechanismen, die Systeme und Daten vor unbefugtem Zugriff schützen. Sicherheit ist das Schloss an der Tür.
- **Datenschutz**: Die Regelung, wie persönliche Daten gesammelt, verwendet und geteilt werden. Datenschutz ist der Vorhang am Fenster.
- Sie können Sicherheit ohne Datenschutz haben (eine abgeschottete Datenbank kann trotzdem Ihre Daten verkaufen), aber keinen Datenschutz ohne Sicherheit (ungeschützte Daten bleiben nicht privat).

### Same-Origin-Policy und CORS

- **Origin**: Definiert durch drei Komponenten: das Protokoll (z. B. `https`), die Domain (z. B. `example.com`) und den Port (z. B. `443`). Alle drei müssen übereinstimmen, damit zwei URLs dieselbe Origin haben.
- **Same-Origin-Policy**: Eine Browser-Sicherheitsregel, die Skripte einer Origin daran hindert, Daten von einer anderen Origin zu lesen, um zu verhindern, dass bösartige Seiten Daten aus anderen offenen Tabs stehlen.
- **CORS (Cross-Origin Resource Sharing)**: Ein Mechanismus, der es Servern erlaubt, die Same-Origin-Policy für vertrauenswürdige Origins durch HTTP-Header zu lockern.
- **In Express (Entwicklung)**: `app.use(cors())` erlaubt alle Origins.
- **In Express (Produktion)**: Übergeben Sie ein `corsOptions`-Objekt mit einer Erlaubnisliste für `origin`:

```js
const corsOptions = {
  origin: ["https://mysite.com", "https://api.mysite.com"]
};
app.use(cors(corsOptions));
```

### HTTPS

- **HTTPS**: HTTP mit Verschlüsselung durch TLS (Transport Layer Security). TLS hat das ältere SSL-Protokoll ersetzt.
- **Drei Garantien von HTTPS**:
  - **Verschlüsselung**: Daten, die zwischen Client und Server gesendet werden, sind für Dritte unlesbar.
  - **Authentifizierung**: Ein digitales Zertifikat einer vertrauenswürdigen Zertifizierungsstelle bestätigt, dass der Server echt ist.
  - **Datenintegrität**: Nachrichten können unterwegs nicht unbemerkt verändert werden.
- HTTPS ist Voraussetzung für moderne Browser-APIs wie Geolocation, Push-Benachrichtigungen und Kamerazugriff.

### Warum Sicherheitsprobleme auftreten

- **Druck durch schnelle Lieferung**: Teams, die zu schnell arbeiten, überspringen Sicherheitsprüfungen und liefern Schwachstellen aus.
- **Komplexe Systemintegration**: Lieferkettenrisiken entstehen, wenn Drittanbieter-Bibliotheken oder Cloud-Dienste Schwachstellen einführen, die das Entwicklungsteam nicht selbst geschrieben hat.
- **Schlechte Konfigurationsverwaltung**: Das Belassen von Standard- oder Werkseinstellungen (Standard-Admin-Passwörter, offene Ports) setzt Systeme unnötig Risiken aus.
- **Menschliches Versagen**: Phishing-Angriffe nutzen Mitarbeiter aus, die auf bösartige Links oder Anhänge klicken und Zugangsdaten preisgeben.
- **Organisationsfusionen**: Das Zusammenführen unterschiedlicher Codebasen und Infrastrukturen bringt unterschiedliche Sicherheitsrichtlinien und veraltete Schwachstellen zum Vorschein.

### Cookies: Speicherung, Sicherheitsflags und Datenschutz

- **Wie Cookies gespeichert werden**: Der Server sendet einen `Set-Cookie`-Header. Der Browser speichert die Datei auf der Festplatte und sendet sie bei jeder folgenden Anfrage an diese Origin automatisch mit.
- **`HttpOnly`-Flag**: Verhindert, dass JavaScript den Cookie-Wert ausliest, und schützt so vor XSS-basiertem Session-Diebstahl.
- **`Secure`-Flag**: Beschränkt das Cookie auf HTTPS-Verbindungen, schützt es vor Abhören in öffentlichen Netzwerken.
- **`SameSite=Strict`**: Verhindert, dass der Browser das Cookie bei Cross-Site-Anfragen sendet, blockiert CSRF-Angriffe vollständig.
- **`SameSite=Lax`**: Erlaubt Cookies bei Top-Level-Navigationen (z. B. Linkklick), blockiert sie aber bei Cross-Site-Formularübermittlungen und API-Aufrufen.
- **First-Party-Cookies**: Sind auf die besuchte Website beschränkt.
- **Third-Party-Cookies**: Werden von Werbenetzwerken über Tausende von Websites gesetzt, um Verhalten zu verfolgen und Nutzerprofile zu erstellen. Moderne Browser blockieren sie standardmäßig.
- **DSGVO-Konformität**: Datenschutzgesetze verlangen Cookie-Zustimmungsbanner, die Nutzern das Recht geben, nicht notwendige Tracking-Cookies abzulehnen.

### Häufige Sicherheitsbedrohungen und Gegenmaßnahmen

- **Phishing**: Gefälschte E-Mails, die vertrauenswürdige Unternehmen imitieren, um Zugangsdaten zu stehlen. Gegenmaßnahmen sind Sicherheitsschulungen und E-Mail-Authentifizierungsprotokolle, die bösartige Nachrichten blockieren, bevor sie den Posteingang erreichen.
- **Malware** (Viren, Spyware, Ransomware): Ransomware verschlüsselt Dateien und verlangt Lösegeld zur Entschlüsselung. Gegenmaßnahmen sind automatische Updates und Antimalware-Software.
- **Credential Stuffing**: Angreifer testen Millionen gestohlener Benutzername-Passwort-Kombinationen auf verschiedenen Websites. Die wichtigste Gegenmaßnahme ist Multi-Faktor-Authentifizierung (MFA), die eine zweite Verifizierung wie OTP oder Authenticator-App-Code erfordert.
- **DDoS (Distributed Denial of Service)**: Angreifer überfluten einen Server mit gefälschtem Traffic, um ihn zum Absturz zu bringen und legitime Nutzer zu blockieren. Gegenmaßnahmen sind cloudbasierte Traffic-Filterdienste wie Cloudflare, Akamai und AWS Shield.
- **Insider-Bedrohungen**: Böswillige oder nachlässige Mitarbeiter, die Daten stehlen oder weitergeben. Gegenmaßnahmen sind die Durchsetzung des **Prinzip der geringsten Privilegien** (Mitarbeiter erhalten nur Zugriff, den sie für ihre Arbeit benötigen) und die Überwachung von Systemprotokollen auf ungewöhnliche Aktivitäten.

### Content Security Policy und Permissions-Policy

- **CSP (Content Security Policy)**: Ein HTTP-Antwortheader, der genehmigte Quell-Domains für Skripte, Styles und Bilder auf eine Whitelist setzt. Wenn ein Browser einen CSP-Header erhält, blockiert er Ressourcen von Domains, die nicht auf der Liste stehen. Die Direktive `script-src` legt fest, welche Domains JavaScript ausführen dürfen und verhindert so XSS-Injektionen.
- **Permissions-Policy**: Ein HTTP-Header, der Browserfunktionen und Gerätehardware für eine Seite und eingebettete iframes aktiviert oder deaktiviert. Er ersetzt den älteren `Feature-Policy`-Header.
  - `camera=(self)` beschränkt den Webcam-Zugriff auf die eigene Origin der Seite.
  - `geolocation=()` deaktiviert die Geolokalisierung vollständig für alle Nutzer.
- Die Kombination beider Header bietet mehrschichtigen Schutz: CSP verhindert Codeinjektionen, Permissions-Policy verhindert unbefugten Hardwarezugriff.

### Grundlegende Datenschutzkonzepte für Nutzer

- **PII (Personally Identifiable Information)**: Alle Daten, die eine Person direkt oder indirekt identifizieren können.
  - **Direkte Identifikatoren**: Vollständiger Name, Wohnadresse, nationale Ausweisnummer.
  - **Indirekte Identifikatoren**: IP-Adresse, Gerätenummer, genaue Standorthistorie.
- **Vertraulichkeit**: Schutz von PII vor unbefugtem Zugriff durch Verschlüsselung und Zugriffskontrollen.
- **Datenminimierung**: Es werden nur die absolut notwendigen PII gesammelt, um eine bestimmte Aufgabe zu erfüllen.
- **Zweckbindung**: Für einen Zweck erhobene Daten dürfen nicht ohne erneute Zustimmung des Nutzers für andere Zwecke verwendet werden (z. B. Verkauf an Werbenetzwerke).
- **Tracking**: Kontinuierliche Überwachung des Nutzerverhaltens online mittels Skripten, Geräte-Fingerprinting und Browser-Verlauf.
- **Recht auf Vergessenwerden (Datenlöschung)**: Das gesetzliche Recht, nach Kontoschließung die dauerhafte Löschung aller persönlichen Daten aus den Datenbanken eines Unternehmens zu verlangen.

### Die `.env`-Datei und `dotenv`

- **`.env`-Datei**: Eine Klartextdatei im Projektstamm, die Konfigurationsgeheimnisse wie API-Schlüssel, Datenbankpasswörter und Portnummern als Schlüssel-Wert-Paare speichert. Fügen Sie sie immer zu `.gitignore` hinzu, um Geheimnisse aus der Versionskontrolle herauszuhalten.
- **`dotenv`-Paket**: Liest die `.env`-Datei beim Start und injiziert die Werte in `process.env`:

```js
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

- **Zwei Hauptvorteile**: Zugangsdaten bleiben aus dem Quellcode heraus (Sicherheit), und derselbe Code kann sich in Entwicklung, Staging und Produktion unterschiedlich verhalten, indem die `.env`-Datei auf jedem Server ausgetauscht wird (Flexibilität).

### Regionale Datenschutzgesetze und Compliance

- **DSGVO (Datenschutz-Grundverordnung)**: Schützt Personen in der EU. Erfordert explizite Opt-in-Zustimmung vor Tracking, keine vorausgewählten Kästchen erlaubt. Gewährt Nutzern das Recht, ihre Daten einzusehen und dauerhaft löschen zu lassen. Verstößt man dagegen, drohen hohe Geldstrafen.
- **CCPA (California Consumer Privacy Act) / CPRA (California Privacy Rights Act)**: Schützt Einwohner Kaliforniens. Legt den Fokus auf Opt-out-Rechte statt Opt-in. Erfordert einen sichtbaren Link „Do Not Sell My Personal Information“ auf der Startseite.
- **COPPA (Children's Online Privacy Protection Act)**: US-Bundesgesetz zum Schutz von Kindern unter 13 Jahren. Erfordert eine klare Datenschutzrichtlinie und überprüfbare elterliche Zustimmung vor der Erhebung persönlicher Daten eines Kindes.
- **DPA (Data Protection Act)**: Das wichtigste Datenschutzgesetz im Vereinigten Königreich. Arbeitet zusammen mit dem UK-DSGVO und gewährt Nutzern das Recht auf Zugriff und Löschung persönlicher Daten.
- Andere Länder haben eigene Gesetze: Nigeria hat das NDPA (Nigeria Data Protection Act) und Indien das DPDP Act (Digital Personal Data Protection Act).
- **Compliance-Schritte für globale Teams**:
  - Erstellen Sie standortabhängige Zustimmungsbanner, die basierend auf dem geografischen Standort des Nutzers die passende Cookie-Hinweismeldung anzeigen.
  - Implementieren Sie ein automatisiertes DSAR-System (Data Subject Access Request), um auf rechtliche Anfragen zur Datenfindung, -export oder -löschung zu reagieren.
  - Praktizieren Sie Datenminimierung und verwenden Sie Altersverifikationsmechanismen, um unerlaubtes Tracking von Minderjährigen zu verhindern.
  - Bestimmen Sie einen Datenschutzbeauftragten (DPO), der die Datenschutzarchitektur überwacht und als Hauptansprechpartner für Aufsichtsbehörden fungiert.
