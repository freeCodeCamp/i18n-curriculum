## Segurança vs privacidade

**Segurança** protege dados contra acesso não autorizado, roubo e danos maliciosos. Ela constrói defesas técnicas como firewalls, criptografia e antivírus para manter agentes mal-intencionados fora dos seus sistemas.

**Privacidade** controla como organizações coletam, armazenam, usam e compartilham dados pessoais. Dá aos usuários o poder de decidir quem vê suas informações.

Os dois conceitos são intimamente relacionados mas distintos:

- Você pode ter segurança sem privacidade. Uma empresa pode armazenar seus dados em um banco de dados altamente seguro e ainda vendê-los para anunciantes sem seu consentimento. Nesse caso, a segurança está intacta, mas a privacidade é violada.
- Você não pode ter verdadeira privacidade sem segurança. Se um servidor que guarda dados privados for invadido, qualquer promessa de privacidade é imediatamente quebrada.

A segurança fornece o mecanismo técnico de defesa, enquanto a privacidade governa os direitos legais e políticas sobre o uso dos dados.

## HTTPS

**HTTP** transfere dados em texto simples, tornando-os legíveis para qualquer um que os intercepte. **HTTPS** (Hypertext Transfer Protocol Secure) usa **TLS (Transport Layer Security)** para criptografar o canal de comunicação entre um navegador e um servidor. TLS substituiu o protocolo mais antigo **SSL (Secure Sockets Layer)**, embora muitas plataformas de hospedagem web ainda façam referência ao SSL em suas configurações hoje.

Principais benefícios do HTTPS:

- **Criptografia:** Embaralha os dados em código ilegível. Apenas o navegador e o servidor possuem as chaves para descriptografá-los.
- **Autenticação:** Exige que o dono do site obtenha um certificado digital de uma organização confiável, provando que o site é legítimo.
- **Integridade dos dados:** Impede que alguém altere os dados em trânsito, você recebe exatamente o que o servidor enviou.
- **Proteção em Wi-Fi público:** Cria um túnel privado através de redes abertas, escondendo seu tráfego de atacantes próximos.

Navegadores modernos avisam os usuários quando um site usa HTTP simples e exigem HTTPS para recursos avançados como geolocalização e notificações push.

## Política de mesma origem e CORS

A **Política de mesma origem (SOP)** é uma regra de segurança do navegador que impede que uma página web leia dados de uma origem diferente. Uma **origem** consiste em três partes que devem coincidir exatamente:

- **Protocolo** (ex.: `https`)
- **Nome de domínio** (ex.: `example.com`)
- **Número da porta** (ex.: `443`, o padrão do HTTPS e `80`, o padrão do HTTP)

Essa política impede que sites maliciosos abertos em uma aba do navegador roubem dados de um site bancário aberto em outra aba.

**CORS (Cross-Origin Resource Sharing)** permite que servidores optem por compartilhamento controlado de dados entre origens usando cabeçalhos HTTP:

- O navegador envia um cabeçalho `Origin` com toda requisição entre origens.
- Se o servidor confiar naquela origem, responde com um cabeçalho `Access-Control-Allow-Origin`.
- O navegador então permite que os dados da resposta sejam lidos; caso contrário, bloqueia.

No Express, o pacote `cors` cuida disso:

```javascript
// development: allow all origins
app.use(cors());

// production: restrict to trusted domains
app.use(cors({ origin: ['https://yourapp.com', 'https://admin.yourapp.com'], credentials: true }));
```

## Cookies: armazenamento, segurança e privacidade

**Cookies** são pequenos arquivos de texto que sites salvam no seu dispositivo para lembrar informações entre visitas. O servidor envia um cabeçalho `Set-Cookie`; o navegador armazena o arquivo e o envia automaticamente em toda requisição subsequente.

Flags de segurança para proteger cookies:

- **`HttpOnly`:** Bloqueia o JavaScript de ler o cookie, prevenindo roubo de token de sessão via XSS.
- **`Secure`:** Garante que o cookie seja transmitido apenas via HTTPS, protegendo-o de interceptação em Wi-Fi público.
- **`SameSite=Strict`:** Impede que o navegador envie o cookie em qualquer requisição entre sites, bloqueando ataques CSRF.
- **`SameSite=Lax`:** Permite cookies em navegação segura de nível superior mas bloqueia em envios de formulários e requisições API entre sites.

**Implicações para a privacidade:**

- **Cookies de primeira parte** rastreiam suas ações apenas no site que você está visitando.
- **Cookies de terceiros** são colocados por redes de publicidade e rastreiam você em sites totalmente diferentes, criando perfis detalhados dos seus interesses.

Leis de privacidade como GDPR exigem que sites exibam banners de consentimento de cookies e deem aos usuários o direito de rejeitar cookies de rastreamento. Navegadores modernos agora bloqueiam cookies de terceiros por padrão.

## Situações onde ocorrem problemas de segurança

Vulnerabilidades de segurança raramente acontecem por acaso. Situações comuns que as desencadeiam incluem:

- **Pressa para lançar:** Equipes de desenvolvimento pulam revisões completas de código e testes de segurança para cumprir prazos rígidos, deixando bugs sem correção e controles de acesso faltando.
- **Integração complexa de sistemas:** Cada biblioteca externa, API de terceiros ou microsserviço adiciona risco à cadeia de suprimentos. Uma vulnerabilidade em uma dependência se espalha para a aplicação principal.
- **Má gestão de configuração:** Implantar servidores ou bancos de dados na nuvem usando configurações padrão de fábrica frequentemente deixa senhas de administrador em branco, portas abertas e acesso público a dados habilitado. Scanners automáticos caçam esses sistemas mal configurados.
- **Erro humano:** Funcionários alvo de e-mails de phishing, SMS fraudulentos ou ligações enganosas podem expor credenciais ou transferir dados para atacantes inadvertidamente.
- **Fusões organizacionais:** Conectar duas redes corporativas diferentes rapidamente cria pontos cegos como perda de visibilidade sobre servidores antigos, firewalls mal configurados e permissões de acesso excessivas.

## Ameaças comuns à segurança e mitigação

| Ameaça | Como funciona | Mitigação |
|---|---|---|
| **Phishing** | E-mails fraudulentos imitam marcas confiáveis para roubar credenciais | Treinamento de conscientização em segurança; protocolos de autenticação de e-mail (SPF, DKIM, DMARC) |
| **Malware (ransomware, spyware)** | Criptografa ou rouba arquivos da vítima; exige pagamento | Aplicação automática de patches; software antimalware |
| **Credential stuffing** | Ferramentas automatizadas testam milhões de pares usuário/senha roubados em sites | Aplicar autenticação multifator (MFA) |
| **DDoS** | Enche um servidor com tráfego falso para derrubá-lo | Serviços de filtragem de tráfego baseados em nuvem (ex.: Cloudflare) |
| **Ameaças internas** | Funcionários insatisfeitos ou descuidados vazam ou roubam dados | Princípio do menor privilégio; monitorar logs do sistema para acessos incomuns |

## Content Security Policy (CSP) e Permissions-Policy

**Content Security Policy (CSP)** é um cabeçalho de resposta HTTP que restringe quais recursos externos um navegador pode carregar numa página. Defende principalmente contra ataques de **Cross-Site Scripting (XSS)** ao permitir apenas domínios confiáveis para scripts, estilos e imagens.

Se um script malicioso de um domínio não aprovado tentar rodar, o navegador o bloqueia e registra um erro de segurança.

**Permissions-Policy** (antigo Feature-Policy) controla o acesso a recursos do navegador e hardware do dispositivo como webcam, microfone e geolocalização. Você pode restringir um recurso ao seu próprio domínio ou desativá-lo completamente:

```http
Permissions-Policy: camera=(self), geolocation=()
```

`camera=(self)`: apenas seu próprio domínio pode acessar a câmera.
`geolocation=()`: geolocalização está desativada para todos na página.

Juntos, CSP bloqueia a entrega de dados e previne injeção de código; Permissions-Policy protege hardware do dispositivo e reforça conformidade com privacidade.

## Conceitos fundamentais em privacidade do usuário

| Conceito | Definição |
|---|---|
| **PII (Informação Pessoal Identificável)** | Qualquer dado que pode identificar direta ou indiretamente uma pessoa. Identificadores diretos: nome, endereço, número de ID. Identificadores indiretos: endereço IP, número de série do dispositivo, histórico de localização. |
| **Confidencialidade** | Garantir que dados privados permaneçam inacessíveis a partes não autorizadas por meio de criptografia e controles de acesso. |
| **Minimização de dados** | Coletar apenas a quantidade mínima de PII necessária para completar uma tarefa específica. |
| **Limitação de propósito** | Dados coletados para um motivo não podem ser usados legalmente para outro (ex.: vender informações de contato para anunciantes). |
| **Rastreamento** | Monitoramento contínuo da atividade online usando scripts, fingerprinting de dispositivo e logs de histórico do navegador para construir perfis de usuário. |
| **Consentimento informado** | Usuários devem optar ativamente pela coleta de dados. Caixas pré-marcadas não constituem consentimento válido. |
| **Direito ao esquecimento** | Usuários podem solicitar legalmente que uma organização exclua permanentemente todos seus dados pessoais e histórico de rastreamento. |

## Leis regionais de privacidade

| Lei | Jurisdição | Requisitos principais |
|---|---|---|
| **GDPR** | União Europeia | Consentimento estrito opt-in exigido; usuários têm direito de acessar e excluir seus dados; multas pesadas por violações. |
| **CCPA / CPRA** | Califórnia, EUA | Modelo opt-out; empresas devem exibir link "Não vender minhas informações pessoais". |
| **COPPA** | EUA (federal) | Consentimento parental verificável exigido antes de coletar dados de crianças menores de 13 anos. |
| **DPA / UK GDPR** | Reino Unido | Processamento legal, justo e transparente de dados; indivíduos têm direitos de acesso e exclusão. |
| **NDPA** | Nigéria | Regula coleta, armazenamento e uso de dados pessoais de residentes nigerianos. |
| **DPDP** | Índia | Regula como dados pessoais de residentes indianos são usados. |

**Passos práticos para conformidade:**

1. **Banners de consentimento sensíveis à localização:** Detectar a localização geográfica do usuário e exibir banners de cookies conforme regulamentos locais.
2. **Sistema automatizado DSAR:** Gerenciar pedidos de acesso do titular dos dados para exportar ou excluir com segurança todo o histórico de dados do usuário.
3. **Minimização de dados e controle de idade:** Coletar apenas o necessário e usar verificação de idade para proteger menores.
4. **Data Protection Officer (DPO):** Designar um especialista para supervisionar a arquitetura de privacidade e gerenciar conformidade regulatória.

## O arquivo `.env` e a biblioteca `dotenv`

Um **arquivo `.env`** é um arquivo de texto simples colocado na raiz de um projeto que armazena dados sensíveis de configuração como pares chave-valor. Incluem chaves de API, senhas de banco de dados e números de porta. Mantém esses dados fora do código fonte e longe de repositórios públicos.

```bash
PORT=3000
DB_HOST=localhost
API_KEY=xyz123secretkey
```

Sempre adicione `.env` ao `.gitignore` para que nunca seja enviado ao controle de versão.

O **pacote `dotenv`** lê o arquivo `.env` na inicialização e injeta as chaves em `process.env`:

```javascript
require('dotenv').config();

const port = process.env.PORT || 5000;
const apiKey = process.env.API_KEY;
```

Dois benefícios principais:

- **Segurança:** Credenciais sensíveis nunca aparecem no código fonte ou no histórico de versões.
- **Flexibilidade:** A mesma base de código roda em desenvolvimento, teste e produção simplesmente trocando o arquivo `.env` em cada servidor.
