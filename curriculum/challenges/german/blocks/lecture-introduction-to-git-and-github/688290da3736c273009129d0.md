# --description--

In der vorherigen Lektion haben wir einen Feature-Branch erstellt, eine neue Datei in diesen Branch committet und ihn in unser Remote-Repository gepusht. Aber wie geht es jetzt weiter? Wir müssen einen Pull Request auf GitHub erstellen.

Ein Pull Request ist im Grunde eine Anfrage, Änderungen aus Ihrem Branch in den Ziel-Branch zu übernehmen. Pull Requests sind der Ablauf, den Sie verwenden, wenn Sie Codeänderungen zu einem Open-Source-Projekt beitragen möchten. Dieser Ansatz ermöglicht es den Maintainer:innen des Projekts, Ihre Änderungen zu überprüfen. Sie können Kommentare hinterlassen, Fragen stellen, Anpassungen vorschlagen und (hoffentlich) Ihre Änderungen genehmigen und zusammenführen!

Gehen wir also wieder zu unserem Repository auf GitHub.

Nachdem Sie Ihren neuen Branch gepusht haben, sollten Sie ein schickes Banner sehen, das anzeigt, dass Sie zu einem Branch gepusht haben, und einen Button, auf den Sie klicken können, um einen Pull Request zu erstellen. Dieser Button überspringt ein paar Schritte und vereinfacht den Prozess, daher ignorieren wir ihn jetzt. So sind Sie auf Fälle vorbereitet, in denen das Banner nicht erscheint.

Klicken Sie auf den Tab Pull Requests oben auf der Repository-Seite, ignorieren Sie dann erneut das Banner und klicken Sie auf „New Pull Request“. Sie gelangen zu einer Benutzeroberfläche, um Ihre Anfrage vorzubereiten.

Hier gibt es einige Begriffe, die wir durchgehen müssen. Zuerst haben wir die Dropdown-Menüs „base“ und „compare“. Die „base“ ist Ihr Ziel für das Zusammenführen. Da wir in `main` zusammenführen wollen, können wir das so lassen.

Das „compare“ ist das, was Sie zusammenführen möchten. Wir wollen unseren `feature`-Branch zusammenführen, also sollten wir das auf `feature` ändern. Das wird auch als Head-Branch bezeichnet, ein Begriff, den Sie sich merken sollten.

Sobald Sie das tun, aktualisiert sich die Benutzeroberfläche. Sie sollte nun die Commits auf `feature` anzeigen, die nicht auf `main` sind, und damit zeigen, was Sie zusammenführen möchten. Außerdem sehen Sie eine sogenannte „diff“, eine visuelle Darstellung der tatsächlichen Änderungen, die Sie vorgenommen haben.

Da wir nur eine Datei hinzugefügt haben, ist die diff ziemlich übersichtlich. Bei komplexeren Änderungen hebt GitHub Dinge wie Zeilenänderungen, Hinzufügungen, Löschungen und mehr hervor. Zu verstehen, wie man eine diff liest, kann herausfordernd sein, aber mit etwas Übung werden Sie darin sehr sicher.

Kehren wir jetzt zu unserem Pull Request zurück. Oben gibt es einen kleinen Link mit der Aufschrift „compare across forks“. Wenn wir von unserem Fork des freeCodeCamp-Repositories arbeiten würden, könnten wir damit unseren PR so ändern, dass er auf das originale freeCodeCamp-Repo zielt, statt nur auf den `main`-Branch in unserem Fork.

Da wir nicht mit einem Fork arbeiten, können wir das ignorieren. Es ist jedoch sehr wichtig zu verstehen, dass das, was wir hier tun, hauptsächlich zum Lernen und Erkunden dient. Sie sollten keinen PR zu einem fremden Projekt öffnen, der nur Übungsänderungen enthält – das erzeugt unnötigen Aufwand für die Maintainer:innen.

Da wir mit unseren Änderungen zufrieden sind und alles konfiguriert ist, können wir auf den Button „Create Pull Request“ klicken. Dadurch gelangen wir zu einer neuen Benutzeroberfläche.

Ganz unten, außerhalb des sichtbaren Bereichs, befinden sich dieselbe Commit-Liste und die diff-Ansicht, die wir zuvor gesehen haben. Es schadet nie, vor dem Erstellen Ihres PR noch einmal alles zu überprüfen, aber für diese Erkundung können Sie jetzt weitermachen.

Sie sehen auch ein paar neue Eingabefelder. Das erste ist der Titel für Ihren Pull Request. GitHub generiert diesen automatisch basierend auf dem Commit, den Sie zusammenführen, wenn es nur einen im PR gibt, oder auf dem Branch-Namen, wenn Sie mehrere Commits haben.

Das zweite Feld ist für Ihre Beschreibung. In den meisten Fällen ist dieses nicht leer. Stattdessen können Maintainer:innen eines Projekts eine Vorlage definieren, die vorausgefüllt wird. Wenn sie eine bereitstellen, sollten Sie deren Workflow respektieren und die Vorlage ausfüllen.

In beiden Fällen sollten Sie die Beitragsdokumentation des Projekts lesen, um sicherzustellen, dass Sie deren Richtlinien für die Benennung und Ausfüllung Ihres PR verstehen und befolgen.

Da Sie dieses Repository besitzen, sehen Sie rechts, dass Sie Reviewer:innen, Zuweisungen, Bezeichnungen, Projekte und Meilensteine festlegen können. Wenn Sie zu einem fremden Projekt beitragen, können Sie diese nicht bearbeiten, außer um eine Überprüfung anzufordern. Wenn Sie diese Funktionen erkunden möchten, können Sie sie jetzt beliebig einstellen, da dies Ihr Repository ist.

Wenn alles ausgefüllt und konfiguriert ist, klicken Sie auf „Create pull request“. Endlich sehen Sie das tatsächliche Ergebnis!

Wenn Sie einen Pull Request zu einem fremden Projekt machen, muss dieser eine „Code Review“ durchlaufen. Dabei können die Maintainer:innen Ihren Pull Request bewerten, Änderungen anfordern, Fragen stellen und so weiter. Sobald die Maintainer:innen ihn genehmigt haben, können sie ihn zusammenführen.

Da dies Ihr eigenes Projekt ist, müssen Sie nicht auf eine Überprüfung warten. Stattdessen können Sie die Commit-Historie unter dem Tab „Commits“ und die Änderungen unter „Files changed“ noch einmal überprüfen und, wenn Sie zufrieden sind, zusammenführen.

Nachdem er zusammengeführt wurde, möchten Sie diese Änderungen in Ihr lokales Repository ziehen.

Gehen Sie zurück zu Ihrem Terminal und stellen Sie sicher, dass Sie sich noch im Ordner des Repositories befinden. Wechseln Sie dann mit `main` zum `git checkout main`-Branch.

Ziehen Sie als Nächstes die neuesten Änderungen mit `git pull` herunter.

Wenn Sie sich jetzt die Commit-Historie mit `git log` ansehen, sehen Sie Ihren Commit. Aber Sie sehen auch eine `Merge pull request #1`-Nachricht. Was ist das?

Nun, wenn Sie einen Pull Request auf GitHub zusammenführen, gibt es drei Strategien, die Sie verwenden können:

Die Standardstrategie ist „Merge“, die alle Commits vom Head-Branch übernimmt und dann einen weiteren Commit erstellt, um das Zusammenführen zu verarbeiten und zu protokollieren.

Eine andere Möglichkeit ist „Squash and Merge“, die ALLE Commits vom Head-Branch nimmt, sie zu einem einzigen Commit zusammenfasst und diesen neuen Commit direkt in den Base-Branch zusammenführt, ohne einen Merge-Commit zu erstellen.

Schließlich gibt es „Rebase and Merge“, die alle Commits nimmt, sie so zurücksetzt, dass sie „nach“ dem neuesten Commit im Base-Branch kommen, und dann den Head ohne Merge-Commit zusammenführt. Rebase ist ein komplexes Thema, das Sie später ausführlich lernen werden, daher ist es in Ordnung, wenn Sie es heute noch nicht ganz verstehen.

Jedes Open-Source-Projekt wählt die Merge-Strategie, die am besten passt. Aufgrund dieser Unterschiede im Verhalten sollten Sie jedoch IMMER einen neuen Branch erstellen, um Ihre Beiträge zu commiten. Comitten Sie niemals direkt in den Standard-`main`-Branch, sonst wird es unübersichtlich, wenn Sie die Änderungen aus dem Original-Repo holen müssen.

Damit haben Sie erfolgreich Ihren ersten Pull Request erstellt und zusammengeführt!
