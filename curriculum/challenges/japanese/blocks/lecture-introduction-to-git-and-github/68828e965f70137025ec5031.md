# --description--

論文のようなものに取り組んでいて、大きな変更を加えた後にコピーを保存し、ファイル名を`Thesis-Draft`、`Thesis-Final`、`Thesis-Final-For-Real`、`Thesis-Absolute-Final-DO-NOT-CHANGE`のように付けたことがあるなら、とても原始的なバージョン管理の方法を使っていることになります。でも、もっと良い方法があるとしたらどうでしょうか？

Git、SVN、Mercurialのようなツールを聞いたことがあるかもしれません。これらはバージョン管理システムであり、すべての開発者のワークフローにおいて重要なコンポーネントです。バージョン管理システムは、プロジェクトの変更を追跡し管理することを可能にします。このレッスンではGitに焦点を当てます。

Gitは、プロジェクトの編集履歴を管理する強力なツールを提供します。Gitの使い方や提供される機能については、今後のレッスンで詳しく学びます。今は、以下のことを知っておいてください。

Gitでは、コミットを作成できます。コミットはコードベースの特定の状態のスナップショットです。コミットを比較してどのような変更を加えたかを確認したり、不適切な変更があったコミットを元に戻したりできます。

また、「ブランチ」を作成することもできます。ブランチはコードの異なる道筋のようなものです。ブランチが歩く道なら、コミットはその道沿いの目印です。ブランチを使うと、新しい機能を独立して作業でき、もし作業がうまくいかなければブランチを削除してもコアのコードベースに影響を与えません。

これらの機能により、バージョン管理は他の開発者との協力に欠かせないものになります。複数の人が同じプロジェクトで作業するとき、ブランチを使うことでそれぞれが自分のコードを独立して作業でき、互いの変更を上書きし合うことを防げます。

GitHubやGitlabのようなバージョン管理プロバイダーとGitのようなバージョン管理システムを組み合わせることが多く、これにより追加の協力ツールやプロジェクトをクラウドにバックアップする機能が利用できます。

しかし、クラウドベースのストレージがなくても、バージョン管理システムはワークフローを大きく変えることができます。もう`THIS-IS-THE-FINAL-COPY-FOREVER`のようなファイルは必要ありません。

これらのシステムの使い方にはたくさんの面白い情報があるので、次のレッスンでじっくり学んでいきましょう！
