# --description--

In questa lezione, imparerai come aggiungere tipi per oggetti e array.

Iniziamo dando un’occhiata agli array.

Ecco un esempio di un array di linguaggi di programmazione:

```js
const programmingLanguages = ["Java", "C++", "Python"];
```

Se vuoi tipizzare esplicitamente quell’array come un array di stringhe, puoi scriverlo così `string[]`:

```ts
const programmingLanguages:string[] = ["Java", "C++", "Python"];
```

Come hai imparato prima, TypeScript sarà abbastanza intelligente da inferire il tipo come un array di stringhe se scrivi semplicemente questo:

```js
const programmingLanguages = ["Java", "C++", "Python"];
```

Ma tipizzarlo esplicitamente aiuta in situazioni in cui provi ad aggiungere un altro tipo all’array così:

```ts
// Type 'number' is not assignable to type 'string'.(2322)
const programmingLanguages:string[] = ["Java", 9, "Python"];
```

Un altro modo per tipizzare gli array è usare questa sintassi:

```ts
const programmingLanguages:Array<string> = ["Java", "C++", "Python"];
```

Fondamentalmente, queste due sintassi sono equivalenti, e la scelta tra le due è generalmente una questione di preferenza.

Ora, diamo un’occhiata a come tipizzare gli oggetti.

Ecco un esempio di un oggetto letterale:

```js
const developer = {
  firstName: "Jessica",
  isEmployed: true
}
```

Per tipizzare esplicitamente le proprietà dell’oggetto, puoi usare questa sintassi:

```ts
const developer: { firstName: string, isEmployed: boolean } = {
  firstName: "Jessica",
  isEmployed: true
}
```

Se provi ad aggiungere o rimuovere proprietà dall’oggetto, otterrai un errore come questo:

```ts
// Property 'isEmployed' is missing in type '{ firstName: string; }' but required in type '{ firstName: string; isEmployed: boolean; }'
const developer: { firstName: string, isEmployed: boolean } = {
  firstName: "Jessica",
}
```

Puoi rendere opzionali le proprietà di un oggetto usando `?` dopo il nome della proprietà così:

```ts
const developer: { firstName: string, isEmployed?: boolean } = {
  firstName: "Jessica",
}
```

TypeScript non mostrerà più un errore perché la proprietà `isEmployed` è stata marcata come opzionale.

Finora, tutti gli esempi di oggetti avevano proprietà restrittive. Questo significa che abbiamo elencato esplicitamente ogni proprietà che l’oggetto può avere:

```ts
const developer: { firstName: string, isEmployed: boolean } = {
  firstName: "Jessica",
  isEmployed: true
}
```

Ma cosa succede se non conosci tutti i nomi delle proprietà in anticipo? E se vuoi un oggetto che possa avere un numero qualsiasi di proprietà ma tutti i valori devono seguire un tipo specifico?

Ci sono due modi per farlo. Il primo è usare un `Record`, che è un tipo generico integrato in TypeScript.

Ecco la sintassi base:

```ts
Record<Keys, ValueType>
```

Il primo tipo rappresenta il tipo delle chiavi mentre il secondo rappresenta il tipo dei valori.

Ecco un esempio di utilizzo di un `Record` per un oggetto `userRoles`:

```ts
const userRoles: Record<string, string> = {
  admin: "full-access",
  editor: "limited-access",
  viewer: "read-only"
};
```

Questo oggetto `userRoles` può avere qualsiasi chiave stringa, e tutti i valori devono essere stringhe. Se provi ad assegnare un valore non stringa, otterrai un errore:

```ts
// Type 'number' is not assignable to type 'string'
const userRoles: Record<string, string> = {
  admin: 1
};
```

Il secondo modo per scrivere la stessa idea è con una firma di indice:

```ts
const obj: { [key: string]: string } = {};
```

La sintassi sopra dice che qualsiasi proprietà con chiave stringa deve avere un valore stringa. Ecco un esempio per un oggetto `settings`:

```ts
const settings: { [key: string]: string } = {
  theme: "dark",
  language: "en",
  layout: "grid"
};
```

Come prima, se hai tipi di valore errati, TypeScript mostrerà un messaggio di errore:

```ts
// Type 'boolean' is not assignable to type 'string'
const settings: { [key: string]: string } = {
  darkMode: true
};
```

Sia `Record` che le firme di indice sono fondamentalmente la stessa cosa. Quindi scegliere tra loro è soprattutto una questione di preferenza.
