## Differenze tra prestazioni reali e percepite

- **Prestazioni percepite**: sono il modo in cui gli utenti percepiscono le prestazioni di un sito web. È come lo valutano in termini di reattività e affidabilità. Si tratta di una misurazione soggettiva, quindi è difficile quantificarla, ma è molto importante, poiché l’esperienza utente determina il successo o il fallimento di un sito.
- **Prestazioni reali**: sono le prestazioni oggettive e misurabili del sito web. Vengono misurate usando metriche come il tempo di caricamento della pagina, il tempo di risposta del server e il tempo di rendering. Queste misurazioni sono influenzate da molteplici fattori legati alla rete e al codice stesso.

## Tecniche per migliorare le prestazioni percepite

- **Lazy loading**: questa tecnica riduce il tempo di caricamento iniziale il più possibile caricando in background le risorse non essenziali.
- **Minimizzare i ritardi nel caricamento dei caratteri tipografici**: se il tuo sito usa caratteri personalizzati, dovresti anche cercare di minimizzare i ritardi nel caricamento dei caratteri, perché questo potrebbe causare sfarfallii o la visualizzazione del carattere di fallback mentre il carattere personalizzato viene caricato. Un suggerimento è usare un carattere di fallback simile a quello personalizzato, così in caso succeda il cambiamento sarà più sottile.
- **Uso di indicatori di caricamento**: mostrare un indicatore di caricamento per un processo che dura a lungo non appena l’utente clicca su un elemento può aiutare l’utente a sentirsi connesso e coinvolto nel processo, facendo sembrare più breve il tempo di attesa.

## Concetti fondamentali sulle prestazioni

- **Ordine di sorgente**: si riferisce al modo in cui gli elementi HTML sono strutturati nel documento. Questo determina cosa viene caricato per primo e può influenzare significativamente le prestazioni e l’accessibilità.

Alcune buone pratiche per l’ordine di sorgente includono:

- Posizionare contenuti critici come intestazioni, navigazione o testo principale più in alto nella struttura HTML.
- Rinviare gli script non essenziali, come quelli per l’analisi o widget di terze parti, in modo che non blocchino il rendering.
- Usare il progressive enhancement, per garantire che l’esperienza base funzioni anche prima che stili e script vengano caricati. Il progressive enhancement è un modo di costruire siti e applicazioni basato sull’idea che la pagina debba funzionare prima con solo HTML.

Ecco un esempio di buon ordine di sorgente, usando le buone pratiche appena descritte:

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **Percorso critico di rendering**: è la sequenza di passaggi che il browser segue per convertire il codice in pixel sullo schermo.
- **Latenza**: è il tempo che impiega una richiesta per viaggiare tra il browser e il server. In altre parole, alta latenza significa pagine lente.

Alcuni modi per ridurre la latenza includono:

- Usare CDN, o Content Delivery Network, per servire file da posizioni più vicine.
- Abilitare la compressione usando strumenti come Gzip per ridurre la dimensione dei file.
- Ottimizzare le immagini e usare il lazy loading.

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## Migliorare INP

- **Definizione**: INP (Interaction to Next Paint) valuta la reattività complessiva di una pagina misurando il tempo che intercorre da quando un utente interagisce, ad esempio con un clic o una pressione di tasto, al momento in cui il browser aggiorna la visualizzazione successiva. Un INP più basso indica una pagina più reattiva.

Ecco alcuni modi per migliorare INP:

- Ridurre il lavoro sul thread principale suddividendo i compiti JavaScript lunghi.
- Usare `requestIdleCallback()` per gli script non critici. Questo metterà in coda una funzione da chiamare durante i periodi di inattività del browser.
- Rinviare o caricare in lazy asset pesanti, come visto prima.
- Ottimizzare i gestori di eventi. Se questi gestori vengono eseguiti troppo frequentemente o fanno operazioni pesanti, possono rallentare la pagina e aumentare INP. La soluzione è il debouncing. Il debouncing assicura che la funzione venga eseguita solo dopo che l’utente ha smesso di digitare per un breve ritardo — per esempio 300ms. Questo evita calcoli inutili e migliora le prestazioni.

## Come funziona il rendering nel browser

- **Come funziona il rendering**: prima il browser analizza l’HTML e costruisce il DOM. Poi il browser elabora il CSS, costruendo il CSS Object Model, o CSSOM. Questo è un’altra struttura ad albero che determina come gli elementi devono essere stilizzati. Infine, il browser dipinge i pixel sullo schermo, renderizzando ogni elemento in base agli stili e al layout calcolati. Nelle pagine complesse, questo può coinvolgere più livelli che vengono compositati insieme per formare l’output visivo finale.

## Come le prestazioni influenzano la sostenibilità

- **Informazioni di base**: Internet rappresenta circa il 2% delle emissioni globali di carbonio — lo stesso dell’industria aerea! Ogni byte trasferito richiede elettricità, dai data center ai dispositivi degli utenti. File più grandi e script inefficienti significano più consumo di energia. Un sito ad alte prestazioni non è solo più veloce, ma riduce anche l’elaborazione e il consumo energetico inutili.

## Modi per ridurre i tempi di caricamento della pagina

- **Ottimizzare le risorse multimediali**: immagini e video di grandi dimensioni sono spesso i colpevoli dei tempi di caricamento lenti. Ottimizzando queste risorse puoi velocizzare significativamente il sito. Questo include comprimere le immagini, usare formati moderni come WebP e usare il lazy loading per le risorse.
- **Sfruttare la cache del browser**: la cache permette ai browser di archiviare parti del tuo sito localmente, riducendo i tempi di caricamento per i visitatori di ritorno.
- **Minificare e comprimere i file**: ridurre la dimensione dei file può portare a download più rapidi. Questo include ridurre la dimensione dei file trasmessi e minificare i file CSS e JavaScript.

## Migliorare il "tempo per essere utilizzabile"

- **Definizione**: il "tempo per essere utilizzabile" è l’intervallo che va da quando un utente richiede una pagina a quando può interagire in modo significativo con essa. Per migliorare questo tempo puoi caricare in lazy le risorse o minimizzare le risorse che bloccano il rendering.

## Metriche chiave per misurare le prestazioni

- **First Contentful Paint o FCP**: misura quanto velocemente appare il primo contenuto — testo o immagine — sullo schermo. Un buon FCP è considerato un tempo inferiore a 1,8 secondi, mentre un FCP scarso è sopra i 3 secondi. Puoi controllare il tuo FCP usando Chrome DevTools e la scheda performance.
- **Total Blocking Time**: mostra per quanto tempo il thread principale è bloccato da compiti JavaScript pesanti. Se il Total Blocking Time (TBT) è alto, gli utenti sperimentano interazioni lente. Per migliorare TBT, suddividi i compiti lunghi e rinvia gli script non essenziali.
- **Bounce Rate**: è la percentuale di visitatori che lasciano senza interagire. Se il tuo sito ha un alto bounce rate potrebbe essere perché la pagina è troppo lenta.
- **Utenti unici**: questa metrica traccia quanti visitatori individuali arrivano sul tuo sito. Per vedere Bounce Rate e Utenti unici puoi usare Google Analytics. Ti permetterà di monitorare queste metriche e migliorare il coinvolgimento.

## Strumenti comuni per misurare le prestazioni

- **Chrome DevTools**: Chrome DevTools è uno strumento integrato in Google Chrome che ti permette di analizzare e eseguire il debug delle prestazioni in tempo reale. DevTools mostra i tempi di caricamento, l’utilizzo della CPU e i ritardi di rendering. È particolarmente utile per misurare il First Contentful Paint, o FCP, cioè quanto velocemente un utente vede il primo contenuto visibile. Se il tuo sito sembra lento, DevTools ti aiuterà a individuare i colli di bottiglia.
- **Lighthouse**: è uno strumento automatico che verifica prestazioni, SEO e accessibilità.
- **WebPageTest**: questo strumento ti permette di testare come il tuo sito si carica da diverse località e dispositivi. Fornisce un’analisi dettagliata dello Speed Index, Total Blocking Time e altre metriche chiave. Se vuoi sapere come gli utenti reali vivono il tuo sito a livello globale, WebPageTest è lo strumento giusto.
- **PageSpeed Insights**: questo strumento analizza il tuo sito e suggerisce miglioramenti rapidi sia per mobile che per desktop. Ti dice cosa rallenta il sito e fornisce raccomandazioni specifiche come ottimizzare immagini, rimuovere script che bloccano il rendering e ridurre i tempi di risposta del server. PageSpeed Insights è un modo veloce e semplice per verificare come Google vede le prestazioni del tuo sito.
- **Real User Monitoring**: gli strumenti RUM tracciano il comportamento reale degli utenti, mostrando come i visitatori vivono il sito. Tra i più popolari ci sono Google Analytics, che traccia i tempi di caricamento e il bounce rate, e New Relic o Datadog, che monitorano i problemi di prestazioni in tempo reale. Se vuoi dati dagli utenti reali, gli strumenti RUM sono essenziali.

## Lavorare con le Performance Web API

- **Definizione**: le Performance Web API permettono agli sviluppatori di tracciare quanto efficientemente una pagina web si carica e risponde direttamente nel codice. Queste API consentono di misurare i tempi di caricamento della pagina, monitorare i ritardi di rendering e interazione e analizzare il tempo di esecuzione di JavaScript.
- **`performance.now()`**: questa API fornisce timestamp ad alta precisione (in millisecondi) per misurare quanto tempo impiegano diverse parti del sito a caricarsi.

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**: questa API fornisce una suddivisione di ogni fase del caricamento della pagina, dalla ricerca DNS a `DOMContentLoaded`.

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**: questa API ascolta eventi di performance come spostamenti di layout, compiti lunghi e interazioni utente.

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## Tecniche per migliorare le prestazioni CSS

- **Animazioni CSS**: animare alcune proprietà CSS, come dimensioni, posizione e layout, attiva un processo chiamato "reflow", durante il quale il browser ricalcola la posizione e la geometria di alcuni elementi nella pagina. Questo richiede un repaint, che è computazionalmente costoso. Perciò si consiglia di ridurre il numero di animazioni CSS il più possibile o almeno dare all’utente la possibilità di attivarle o disattivarle.

## Tecniche per migliorare le prestazioni JavaScript

- **Code splitting**: dividere il codice JavaScript in moduli che eseguono compiti critici e non critici è utile. In questo modo puoi precaricare quelli critici il prima possibile e rinviare quelli non critici per rendere la pagina più veloce.
- **Manipolazione del DOM**: ricorda che la manipolazione del DOM si riferisce al processo di modificare dinamicamente il contenuto di una pagina con JavaScript interagendo con il Document Object Model (DOM). Manipolare il DOM è computazionalmente costoso. Ridurre la quantità di manipolazione del DOM nel codice JavaScript migliorerà le prestazioni.
