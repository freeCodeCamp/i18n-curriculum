# --description--

Il problema N+1 è un collo di bottiglia comune nelle prestazioni delle applicazioni basate su database. Se hai mai sperimentato un tempo di caricamento molto lungo mentre usavi un'applicazione, è molto probabile che sia legato al problema N+1. Questo problema si verifica quando molte piccole query vengono eseguite in sequenza per ottenere i dati richiesti.

Ma perché eseguire molte query in sequenza? Potresti farlo per interrogare una lista di record e informazioni aggiuntive su quei record. Eseguirai una query iniziale per ottenere la lista dei record e poi un'ulteriore query per ciascuno di quei record, per ottenere informazioni aggiuntive su di essi.

Anche se intuitivamente potresti pensare che eseguire molte query piccole e semplici sia più efficiente che eseguire una sola query grande e complessa, di solito non è così. Più query esegui, più lungo sarà l'intero processo perché devi inviare ogni query al server, trovare i dati nel database e poi ricevere i dati inviati dal server per quella query. Questo può avere un impatto molto significativo sulle prestazioni della tua applicazione.

Per mostrarti un esempio pratico, supponiamo che tu stia sviluppando un'applicazione per un servizio di consegna cibo e che esegua una query per ottenere i primi 50 ordini nel database. Prima, otterrai una lista con questi ordini da una tabella `orders`, come questa:

```sql
order_id | product   | quantity | customer_id
1        | pizza     | 2        | 3422
2        | salad     | 1        | 1255
3        | ice cream | 4        | 2344
4        | donuts    | 10       | 3455
.        | .         | .        | .
.        | .         | .        | .
.        | .         | .        | .
```

Ma cosa succede se hai anche bisogno di ottenere i dati dei clienti che hanno inviato questi ordini? Queste informazioni sarebbero archiviate in una tabella `customers` diversa.

Un modo per affrontare questo sarebbe ottenere prima la lista degli ordini e poi eseguire una query per ogni ordine per ottenere le informazioni dei clienti. Per ottenere la lista degli ordini, potresti eseguire una query come questa, per ottenere i primi 50 record dalla tabella `orders`:

```sql
SELECT * FROM orders LIMIT 50;
```

Da qui deriva l'1 in N+1. È la query iniziale che ti dà la lista dei record. Poi, dovrai eseguire un'altra query per ciascuno di questi record per ottenere le informazioni del cliente che ha inviato l'ordine. Per questo, potresti scrivere un loop in una funzione asincrona per elaborare sia gli ordini sia i clienti. La funzione `getCustomerData()` eseguirà una query SQL per ottenere i dati del cliente che ha effettuato un ordine specifico. Lo farà in sequenza per ogni ordine effettuato:

```js
for (const order of orders) {
  const customerId = order.customer_id;
  const customerData = await getCustomerData(customerId);
  // Process the customer's data.
}
```

Da qui deriva la N in N+1. N rappresenta il numero di query che verranno eseguite per ottenere dati aggiuntivi per ciascuno dei record.

Questo approccio può sembrare semplice e intuitivo, ma potresti rimanere sorpreso nel sapere che è esattamente ciò che non dovresti fare. Dovresti evitare di fare query in un loop perché eseguire molte query piccole richiede molto più tempo rispetto a eseguire una singola query più grande per ottenere tutti i dati.

Dovresti cercare di usare gli strumenti che SQL ti offre per ridurre il numero di query il più possibile. In questo caso, potremmo usare un'operazione `JOIN` per unire le tabelle `orders` e `customers`.

```sql
SELECT 
  orders.order_id,
  orders.product,
  orders.quantity,
  customers.customer_id,
  customers.name AS customer_name,
  customers.email AS customer_email,
  customers.address AS customer_address
FROM orders
JOIN customers ON orders.customer_id = customers.customer_id
WHERE orders.order_id IN (SELECT order_id FROM orders LIMIT 50)
```

Con l'operazione `JOIN` possiamo recuperare i dati dei primi 50 ordini nel database, inclusi i dati dei clienti, i loro nomi, email e indirizzi, con una sola query. Potrebbe sembrare più complesso, ma in realtà è più efficiente così.

Ora sai cosa comporta il problema N+1, quindi puoi identificarlo e prevenirlo per interagire con il tuo database in modo efficiente.
