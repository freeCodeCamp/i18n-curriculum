## 実際のパフォーマンスと体感パフォーマンスの違い

- **体感パフォーマンス**：これはユーザーがウェブサイトのパフォーマンスをどのように感じるかを指します。レスポンスの速さや信頼性の観点から評価されます。主観的な評価なので数値化は難しいですが、ユーザー体験がウェブサイトの成功や失敗を決めるため非常に重要です。
- **実際のパフォーマンス**：これはウェブサイトの客観的かつ測定可能なパフォーマンスです。ページの読み込み時間、サーバーレスポンス時間、レンダリング時間などの指標で測定されます。これらの測定値はネットワークやコード自体に関連する複数の要因に影響されます。

## 体感パフォーマンスを改善する手法

- **遅延読み込み（Lazy Loading）**：重要でないリソースをバックグラウンドで読み込むことで、初期の読み込み時間をできるだけ短縮する手法です。
- **フォント遅延の最小化**：カスタムフォントを使用している場合、フォントの読み込み遅延を最小限に抑えることが重要です。遅延があると、フォントのちらつきやフォールバックフォントが表示されることがあります。これを防ぐために、カスタムフォントに似たフォールバックフォントを使うと、変化がより自然になります。
- **読み込みインジケーターの使用**：ユーザーが要素をクリックした直後に長時間かかる処理の読み込みインジケーターを表示すると、ユーザーが処理に接続されていると感じ、待ち時間が短く感じられます。

## パフォーマンスの基本概念

- **ソース順序**：HTML要素がドキュメント内でどのように構成されているかを指します。これにより何が最初に読み込まれるかが決まり、パフォーマンスやアクセシビリティに大きな影響を与えます。

ソース順序のベストプラクティスの例：

- 見出し、ナビゲーション、メインテキストなどの重要なコンテンツをHTML構造の上位に配置する。
- アナリティクスやサードパーティのウィジェットなど、重要でないスクリプトは遅延させてレンダリングを妨げないようにする。
- プログレッシブエンハンスメントを使い、スタイルやスクリプトが読み込まれる前でもコア体験が機能するようにする。プログレッシブエンハンスメントは、まずHTMLでページを動作させることを基本にしたウェブサイトやアプリの構築手法です。

以下は、先ほどのベストプラクティスを使った良いソース順序の例です：

```html
<h1>Welcome to FastSite!</h1>
<p>Critical information loads first.</p>
<script src="slow-script.js" defer></script>
```

- **クリティカルレンダリングパス**：ブラウザがコードを画面上のピクセルに変換するためにたどる一連のステップです。
- **レイテンシー**：ブラウザとサーバー間でリクエストが往復するのにかかる時間です。つまり、レイテンシーが高いとページが遅くなります。

レイテンシーを減らす方法の例：

- CDN（コンテンツデリバリーネットワーク）を使い、ファイルをより近い場所から配信する。
- Gzipなどの圧縮を有効にしてファイルサイズを減らす。
- 画像の最適化や遅延読み込みを活用する。

```html
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy">
```

## INPの改善

- **定義**：INP（Interaction to Next Paint）は、ユーザーがクリックやキー入力などの操作をしてからブラウザが次に画面を更新するまでの時間を測定し、ページ全体のレスポンスの良さを評価します。INPが低いほどレスポンスが良いページです。

INPを改善する方法：

- 長いJavaScriptタスクを分割してメインスレッドの負荷を減らす。
- `requestIdleCallback()`を非クリティカルなスクリプトに使う。これはブラウザのアイドル時間に関数を呼び出すキューを作ります。
- 先に説明した重いアセットを遅延読み込みまたは遅延実行する。
- イベントハンドラを最適化する。頻繁に実行されたり重い処理を行うハンドラはページを遅くし、INPを増加させます。これにはデバウンスが効果的です。デバウンスは、ユーザーが入力をやめてから一定時間（例：300ms）経ってから関数を実行する仕組みで、不要な計算を防ぎパフォーマンスを向上させます。

## ブラウザでのレンダリングの仕組み

- **レンダリングの流れ**：まずブラウザがHTMLをパースしてDOMを構築します。次にCSSを処理してCSS Object Model（CSSOM）を作ります。これは要素のスタイルを決めるツリー構造です。最後に計算されたスタイルとレイアウトに基づいて画面にピクセルを描画し、各要素をレンダリングします。複雑なページでは複数のレイヤーを合成して最終的な表示を作ることもあります。

## パフォーマンスが持続可能性に与える影響

- **背景情報**：インターネットは世界の二酸化炭素排出量の約2％を占めており、これは航空業界と同じ規模です。データセンターからユーザーのデバイスまで、すべてのデータ転送には電力が必要です。大きなファイルや非効率なスクリプトは消費電力を増やします。高パフォーマンスなウェブサイトは単に速いだけでなく、不要な処理やエネルギー消費を減らします。

## ページ読み込み時間を短縮する方法

- **メディアアセットの最適化**：大きな画像や動画は読み込み遅延の主な原因です。これらを圧縮したり、WebPのような最新フォーマットを使ったり、遅延読み込みを活用することでサイトの速度を大幅に向上できます。
- **ブラウザキャッシュの活用**：キャッシュを使うとブラウザがウェブサイトの一部をローカルに保存し、再訪問時の読み込み時間を短縮します。
- **ファイルの圧縮と縮小**：ファイルサイズを減らすことでダウンロード時間が短くなります。これには送信ファイルの圧縮やCSS・JavaScriptの縮小が含まれます。

## 「使えるまでの時間」の改善

- **定義**：「使えるまでの時間」とは、ユーザーがページをリクエストしてから意味のある操作ができるようになるまでの時間です。これを改善するには、アセットを遅延読み込みしたり、レンダリングを妨げるリソースを最小限に抑えます。

## パフォーマンス測定の主要指標

- **First Contentful Paint（FCP）**：最初のテキストや画像などのコンテンツが画面に表示されるまでの速さを測ります。良いFCPは1.8秒未満、悪いFCPは3秒以上とされています。Chrome開発者ツールのパフォーマンスタブで確認できます。
- **Total Blocking Time（TBT）**：メインスレッドが重いJavaScriptタスクでブロックされている時間を示します。TBTが高いとユーザーの操作が鈍く感じられます。改善には長いタスクを分割し、重要でないスクリプトを遅延させます。
- **直帰率**：ユーザーが何の操作もせずに離脱する割合です。直帰率が高い場合、ページが遅すぎる可能性があります。
- **ユニークユーザー数**：サイトを訪れた個別の訪問者数を追跡します。直帰率やユニークユーザー数はGoogle Analyticsで確認でき、エンゲージメント改善に役立ちます。

## よく使われるパフォーマンス測定ツール

- **Chrome開発者ツール**：Google Chromeに組み込まれたツールで、リアルタイムにパフォーマンスを分析・デバッグできます。読み込み時間、CPU使用率、レンダリング遅延などを表示し、特にFCPの測定に便利です。サイトが遅いと感じたら、ボトルネックを見つけるのに役立ちます。
- **Lighthouse**：パフォーマンス、SEO、アクセシビリティを自動でチェックするツールです。
- **WebPageTest**：異なる場所やデバイスからのサイトの読み込み状況をテストできます。Speed IndexやTotal Blocking Timeなどの詳細なパフォーマンス指標を提供し、世界中の実際のユーザー体験を知るのに適しています。
- **PageSpeed Insights**：モバイルとデスクトップ両方のサイトを分析し、画像の最適化、レンダーブロックスクリプトの削除、サーバーレスポンス時間の短縮など具体的な改善案を提示します。Googleがサイトのパフォーマンスをどう見ているかを素早く簡単にチェックできます。
- **Real User Monitoring（RUM）**：実際のユーザーの行動を追跡し、リアルな体験を示します。代表的なRUMツールにはGoogle Analytics（ページ読み込み時間や直帰率の追跡）、New RelicやDatadog（リアルタイムのパフォーマンス問題の監視）があります。実ユーザーデータが欲しい場合はRUMツールが必須です。

## パフォーマンスWeb APIの活用

- **定義**：パフォーマンスWeb APIは、開発者がコード内でページの読み込みや応答の効率を直接追跡できるAPIです。ページ読み込み時間の測定、レンダリングやインタラクションの遅延の追跡、JavaScript実行時間の分析が可能です。
- **`performance.now()`**：このAPIはミリ秒単位の高精度タイムスタンプを提供し、サイトの各部分の読み込み時間を測定できます。

```js
const start = performance.now();  
// Run some code here  
const end = performance.now();  

console.log(`Execution time: ${end - start}ms`);
```

- **Performance Timing API**：DNSルックアップから`DOMContentLoaded`まで、ページ読み込みの各段階を詳細に分解して教えてくれるAPIです。

```js
const timing = performance.timing;  

const pageLoadTime = timing.loadEventEnd - timing.navigationStart;  
console.log(`Page load time: ${pageLoadTime}ms`);
```

- **`PerformanceObserver`**：レイアウトシフト、長時間タスク、ユーザーインタラクションなどのパフォーマンスイベントを監視するAPIです。

```js
const observer = new PerformanceObserver((list) => {  
  list.getEntries().forEach((entry) => {  
    console.log(`Long task detected: ${entry.duration}ms`);  
  });  
});  

observer.observe({ type: "longtask", buffered: true });
```

## CSSパフォーマンス改善の手法

- **CSSアニメーション**：幅や位置、レイアウトなど特定のCSSプロパティをアニメーションさせると、「リフロー」と呼ばれる処理が発生し、ブラウザが要素の位置や形状を再計算します。これには再描画が伴い、計算コストが高いです。したがって、CSSアニメーションはできるだけ減らすか、ユーザーがオン・オフを切り替えられるようにすることをお勧めします。

## JavaScriptパフォーマンス改善の手法

- **コード分割**：JavaScriptコードをクリティカルな処理とそうでない処理に分割することも効果的です。こうすることで、重要なコードをできるだけ早くプリロードし、非クリティカルなコードは遅延させてページのレンダリングを速くできます。
- **DOM操作**：DOM操作とは、JavaScriptでDocument Object Model（DOM）を操作してページの内容を動的に変更することです。DOM操作は計算コストが高いため、JavaScriptコード内のDOM操作を減らすことでパフォーマンスが向上します。
