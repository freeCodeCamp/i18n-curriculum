¿Cómo funciona el evento `submit` con los formularios?

Aprendamos sobre cómo funciona el evento `submit` con formularios HTML. Primero, necesitamos entender cómo enviar un formulario. Hay tres maneras en que se puede enviar un formulario.

La primera es cuando el usuario hace clic en un botón en el formulario que tiene el atributo `type` configurado como `submit`.

La segunda es cuando el usuario presiona la tecla Enter en cualquier campo de entrada editable del formulario.

La tercera es mediante una llamada de JavaScript a los métodos `requestSubmit()` o `submit()` del elemento de formulario.

Pero, ¿qué sucede cuando se envía un formulario? Hay algunas cosas que pasan, y el comportamiento del formulario depende de sus atributos.

El primer atributo que necesitamos analizar es el atributo `action`. El atributo `action` debe contener una URL o una ruta relativa para el dominio actual. Este valor determina dónde intenta enviar el formulario los datos. Si no configuras un atributo `action`, el formulario enviará datos a la URL de la página actual. Aquí hay un ejemplo de un formulario con un atributo `action` configurado a una URL específica:

```html
<form action="https://freecodecamp.org">
  <input type="number" id="input" placeholder="Enter a number" name="number" />
  <button type="submit">Submit</button>
</form>
```

Cuando este formulario se envía, enviará datos a la página principal de freeCodeCamp, la cual probablemente no hará nada.

Aquí hay otro formulario, que envía a una ruta relativa:

```html
<form action="/data">
  <input type="number" id="input" placeholder="Enter a number" name="number" />
  <button type="submit">Submit</button>
</form>
```

Este envía a la ruta `/data` en el dominio actual. En nuestro caso, enviaría a `http://127.0.0.1:5500/data`.

El segundo atributo para controlar cómo se comporta un envío es el atributo `method`. Este atributo acepta un método HTTP estándar, como `GET` o `POST`, y utiliza ese método cuando se hace la solicitud a la URL del `action`.

HTTP significa Protocolo de Transferencia de Hipertexto y se usa para transferir datos por la web.

Los métodos HTTP se utilizan para definir las acciones que se pueden realizar sobre los recursos, como `GET`, `POST`, `PUT`, `DELETE`, y así sucesivamente. Aprenderás más sobre estos métodos en los futuros videos de la conferencia.

Cuando un método no está configurado, el formulario usará por defecto una solicitud `GET`. Una solicitud `GET` se utiliza para recuperar datos de un recurso especificado sin realizar cambios en él, y los parámetros se suelen anexar a la URL en forma de cadena de consulta.

Los datos en el formulario serán codificados en URL como pares `name=value` y anexados a la URL del `action` como parámetros de consulta. Por ejemplo, al enviar el formulario desde nuestro ejemplo anterior con el valor `3342` para el campo `input`, el formulario intentaría `GET` la URL `http://127.0.0.1:5500/data?number=3342`.

Esto es ideal para algo como un formulario de búsqueda, donde tu usuario está consultando datos. Pero si deseas que tu usuario envíe nuevos datos, como en un formulario de registro, el método idiomático a utilizar sería el método `POST`. El método `POST` se utiliza para enviar datos al servidor para crear o actualizar un recurso.

Establezcamos el atributo `method` al valor de `POST`:

```xml
<form action="/data" method="POST">
  <input type="number" id="input" placeholder="Enter a number" name="number" />
  <button type="submit">Submit</button>
</form>
```

Cuando envías una solicitud `POST`, se puede incluir un cuerpo que contiene los datos para tu solicitud. Así que a diferencia de un `GET`, los datos no se anexan a la URL, y nuestro formulario envía la solicitud `POST` a `http://127.0.0.1:5500/data`. Los datos, en su lugar, se pueden encontrar en el cuerpo de la solicitud, aún como datos de formulario codificados en URL.

Los datos de formulario codificados en URL son cuando los datos del formulario se convierten en una cadena de pares clave-valor, como `name=John+Doe&email=john%40example.com`, donde los caracteres especiales se reemplazan con versiones codificadas para enviar los datos de manera segura por la web. Aprenderás más sobre esto en los futuros videos de la conferencia.

Pero quizá no quieras enviar los datos como una carga de formulario codificada en URL.

El elemento `form` acepta un atributo `enctype`, que representa el tipo de codificación a utilizar para los datos.

Este atributo solo acepta tres valores: `application/x-www-form-urlencoded` (que es el valor por defecto, enviando los datos como un cuerpo de formulario codificado en URL), `text/plain` (que envía los datos en formato de texto sin formato, en pares `name=value` separados por nuevas líneas), o `multipart/form-data`, que es específicamente para manejar formularios con una carga de archivo.

En este video, has aprendido los conceptos básicos sobre los envíos de formularios. En futuros videos, profundizaremos en el trabajo con formularios y cómo interactúan con el servidor.
