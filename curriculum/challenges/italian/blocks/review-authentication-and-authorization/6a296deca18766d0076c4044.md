## Autenticazione vs autorizzazione

**L'autenticazione** è il processo di verifica dell'identità di un utente. Risponde a *"Questa persona è davvero chi dice di essere?"* Quando un utente effettua il login, il server backend confronta le sue credenziali con i dati memorizzati durante la registrazione.

Gli sviluppatori backend usano tre categorie principali di fattori di autenticazione:

- **Qualcosa che conosci:** una password, un PIN o la risposta a una domanda di sicurezza.
- **Qualcosa che possiedi:** un codice SMS su uno smartphone o un token temporaneo da un'app di autenticazione.
- **Qualcosa che sei:** dati biometrici come un'impronta digitale o una scansione del volto.

**L'autorizzazione** interviene dopo l'autenticazione e decide cosa l'utente autenticato può fare. Risponde a *"Quali permessi ha questa identità?"* I ruoli comuni includono:

- **Utente standard:** può modificare il proprio profilo, ma non può consultare i dati privati di altri utenti.
- **Editor:** può scrivere e pubblicare contenuti, ma non può eliminare il codice.
- **Admin:** ha pieno accesso per modificare database, eliminare utenti e cambiare le impostazioni di sistema.

**L'autenticazione deve sempre avvenire prima, poi l'autorizzazione.** Non puoi decidere quali permessi concedere finché non sai chi è l'utente.

## JSON Web Token (JWT)

Un JWT è una chiave digitale che il server rilascia a un utente dopo un login riuscito. L'utente la presenta a ogni richiesta successiva per dimostrare la propria identità senza dover reinserire la password.

Il flusso in quattro passaggi del JWT:

1. **Login:** l'utente invia le credenziali al server.
2. **Emissione:** il server verifica le credenziali e genera un JWT, poi lo invia indietro.
3. **Archiviazione:** il client conserva il JWT, tipicamente in un cookie del browser o nel local storage.
4. **Richiesta:** per ogni richiesta futura a una rotta protetta, il client invia il JWT e il server lo verifica.

Un JWT è composto da tre parti separate da due punti:

- **Intestazione:** JSON codificato che dichiara il tipo di token (JWT) e l'algoritmo di hashing usato, come HMAC SHA256.
- **Payload:** JSON codificato contenente affermazioni sull'utente, come id, nome utente o ruolo. Intestazione e payload sono codificati in Base64, non criptati, quindi non memorizzare mai dati sensibili come password qui.
- **Firma:** creata hashando insieme intestazione e payload codificati con una chiave segreta conosciuta solo dal server. Se il payload viene manomesso, la firma non corrisponde e il server rifiuta la richiesta con una risposta `401 Unauthorized`.

I JWT permettono un'autenticazione **senza stato**. Il server non deve memorizzare dati di sessione in un database perché la firma dimostra che ha creato il token.

## Cross-Site Request Forgery (CSRF)

Il CSRF è un attacco che costringe un utente autenticato a eseguire inconsapevolmente azioni indesiderate su un'applicazione web. Un sito malevolo inganna il browser della vittima facendogli inviare una richiesta fraudolenta a un server backend, sfruttando il fatto che i browser allegano automaticamente i cookie a ogni richiesta verso il server di destinazione.

**Esempio di flusso d'attacco:**

1. Effettui il login, per esempio, su `trusted-bank.com`. La banca memorizza un token di sessione nel cookie del tuo browser.
2. Senza effettuare il logout, visiti `evil-hacker-site.com`.
3. Il sito malevolo invia silenziosamente una richiesta a `trusted-bank.com/transfer?amount=1000&to=hacker`.
4. Il tuo browser allega automaticamente il cookie di sessione della banca, quindi il server elabora la richiesta.

Il sito malevolo non può leggere il tuo cookie. Sfrutta solo il fatto che il browser allega automaticamente i cookie a ogni richiesta verso il server di destinazione.

**Difese contro il CSRF:**

- **`SameSite=Strict`:** il browser non invia mai il cookie se la richiesta proviene da un sito terzo.
- **`SameSite=Lax`:** permette i cookie su navigazioni sicure (cliccando un link), ma li blocca su invii di moduli cross-site e richieste API.
- **Token CSRF:** quando un utente carica una pagina, il server genera un token casuale, unico e imprevedibile e lo allega al modulo frontend o alla sessione. Il client deve includerlo negli header della richiesta o nei dati del modulo. Poiché un sito malevolo non può accedere al frontend per copiare il token, le richieste contraffatte falliscono la validazione.

## Passport.js

Passport.js è un middleware di autenticazione per Node.js che standardizza come autenticare le richieste usando plugin modulari chiamati **strategie**. Con oltre 500 strategie disponibili, puoi aggiungere qualsiasi metodo di autenticazione installando il pacchetto corrispondente:

- `passport-local`: nome utente e password contro il tuo database.
- `passport-jwt`: valida JSON Web Token su rotte API protette.
- `passport-google-oauth20`: login con account Google.
- `passport-github2`: login con account GitHub.

Installa il pacchetto core e una strategia:

```bash
npm install passport passport-local
```

Collega Passport alla tua app Express prima di definire le rotte:

```js
const passport = require('passport');
const LocalStrategy = require('passport-local').Strategy;

app.use(passport.initialize());

passport.use(new LocalStrategy((username, password, done) => {
  // database look-up and password verification logic
}));
```

Proteggi una rotta di login con Passport come middleware:

```js
app.post('/login', passport.authenticate('local'), (req, res) => {
  res.send(`Welcome back, ${req.user.username}!`);
});
```

Dopo l'autenticazione riuscita, Passport gestisce automaticamente la **gestione della sessione** (memorizza i dati utente nel cookie di sessione) e la **decorazione della richiesta** (allega l'oggetto utente a `req.user` per l'uso nei gestori di rotta successivi).

## Helmet.js

Helmet.js è una raccolta di middleware di sicurezza per Node.js che protegge la tua applicazione impostando, modificando o rimuovendo automaticamente le intestazioni HTTP di risposta che Express espone per impostazione predefinita.

Installalo e applicalo in un solo passaggio:

```bash
npm install helmet
```

```js
const helmet = require('helmet');
app.use(helmet());
```

Inserendo `app.use(helmet())` in cima alla tua catena di middleware applichi queste protezioni a ogni risposta:

- **X-Powered-By rimosso:** Express pubblicizza la sua presenza per impostazione predefinita con `X-Powered-By: Express`. Helmet rimuove questa intestazione, impedendo agli attaccanti di mirare a vulnerabilità specifiche di versione.
- **Content-Security-Policy impostata:** limita da dove il browser può caricare script, immagini e stili, difendendo dagli attacchi Cross-Site Scripting (XSS).
- **X-Frame-Options impostata su SAMEORIGIN:** impedisce che il tuo sito venga incorporato in un iframe esterno, bloccando attacchi di clickjacking.
- **Strict-Transport-Security impostata:** forza i browser a comunicare con il tuo server solo tramite HTTPS, proteggendo i dati dall'intercettazione su reti non affidabili.
