# --description--

Has aprendido a crear commits y subirlos a tu propio repositorio que controlas. Puedes hacer lo mismo para otros proyectos que no mantienes, pero hay algunos pasos adicionales.

Primero que nada, generalmente no tendrás "write access" al proyecto de otra persona. Esto significa que no puedes hacer push de commits directamente a ellos. En cambio, necesitarás crear una copia del repositorio para trabajar desde ella. Este proceso se llama "forking".

Para hacer un fork de un repositorio, visita el repositorio al que quieres contribuir en GitHub. En la parte superior, verás un botón que dice "Fork". Haz clic en este para ser llevado a la interfaz para crear tu fork.

Verás una pantalla similar a la que usaste para crear tu propio repositorio, pero con menos opciones. Puedes elegir un propietario, ya sea tu cuenta personal o una organización a la que tengas acceso, un nuevo nombre para el fork y una nueva descripción.

También tienes la opción de copiar solo la rama `main` (o la configurada por defecto para el repositorio). Esto generalmente es una buena idea, a menos que necesites contribuir a una rama de función específica. Aprenderás más sobre las ramas en una lección futura. Por ahora, deja esa casilla marcada.

Haz clic en "Create fork" y se creará tu nuevo repositorio, y serás llevado automáticamente a él.

A continuación, clónalo en tu computadora local tal como lo harías con tu propio repositorio, usando SSH o la CLI de GitHub.

Ahora, echemos un vistazo a los remotos que tiene configurados tu clon local. Abre una terminal en el directorio del repositorio local y ejecuta `git remote -v` para obtener una lista completa, debería verse algo así:

```sh
origin git@github.com:your-username/your-repo-name.git (fetch)
origin git@github.com:your-username/your-repo-name.git (pull)
```

Al clonar un repositorio desde GitHub, Git ha configurado automáticamente tu fork como el remoto `origin`. Sin embargo, cuando trabajas desde un fork, a menudo necesitarás actualizar tu copia para incorporar los cambios del repositorio original.

Para hacer esto, vamos a agregar el repositorio original como otro remoto. Regresa a la página de donde hiciste el fork del repositorio, haz clic en el botón "Code" y copia la URL SSH. Luego ejecuta el comando para crear un nuevo remoto upstream:

```sh
git remote add upstream git@github.com:original-repo-username/repo-name.git
```

`upstream` es el nombre del remoto - puedes nombrarlo de otra manera, pero `upstream` suele ser la convención. Si vuelves a mirar tus remotos, verás tu nuevo remoto upstream:

```sh
origin git@github.com:your-username/your-repo-name.git (fetch)
origin git@github.com:your-username/your-repo-name.git (pull)
upstream git@github.com:original-repo-username/your-repo-name.git (fetch)
upstream git@github.com:original-repo-username/your-repo-name.git (pull)
```

Con el remoto `upstream` podrás hacer cosas como `git fetch upstream` y `git merge upstream/main` para obtener los últimos cambios del proyecto original en tu fork local.

Un fork sigue siendo un repositorio, y tú creas commits y haces push de cambios de la misma manera. Sin embargo, antes de profundizar en eso, debes entender que hay ciertas estrategias que debes seguir al hacer cambios en el proyecto de otra persona. Aprenderás más sobre estas estrategias en las próximas lecciones.

Los proyectos que hacen su código fuente disponible al público, te permiten modificar el código y aceptan contribuciones de código se llaman proyectos "open source". Hay muchos beneficios en el open source, como permitir que la comunidad te ayude a identificar y corregir errores, aprender de las contribuciones que las personas hacen a tu trabajo y fomentar una comunidad de desarrolladores a la que puedes convertir en una red.

Cuando haces cambios en el proyecto de alguien, se te considera un "open source contributor". Contribuir a proyectos de código abierto puede ser una excelente forma de anticipar cómo es trabajar en un equipo profesional de desarrolladores. Y algunas de las estrategias aquí te ayudarán a hacer justamente eso, y a trasladarlo a tu primer trabajo remunerado.

Y si alguna vez tienes preguntas sobre cómo contribuir, la mayoría de los proyectos de código abierto tendrán directrices de contribución que contienen un resumen de todo lo que necesitas saber. Pero si aún no estás seguro, nunca está de más abrir un issue en el repositorio para hacer tus preguntas, o, si tienen un espacio comunitario en una plataforma como Discord, únete y haz tus preguntas allí.
